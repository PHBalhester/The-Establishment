# H057: Micro-Tax Edge Case Routing

**Status:** CONFIRMED
**Severity:** INFORMATIONAL
**Category:** Token Economics

## Summary

When total tax calculated from a swap is less than 4 lamports, the `split_distribution()` function routes 100% of the tax to the staking escrow, with zero contribution to the Carnage Fund (24%) or treasury (5%). This is an intentional design choice documented in the code to avoid splitting dust across three destinations, but creates a rare scenario where two of the three economic incentive systems receive no compensation.

## Code Evidence

- **File:** `programs/tax-program/src/helpers/tax_math.rs:86-90`

```rust
// Handle micro-tax edge case: if tax < 4 lamports, all to staking
// This avoids splitting dust across three destinations
if total_tax < 4 {
    return Some((total_tax, 0, 0));
}
```

- **Called from:** `programs/tax-program/src/instructions/swap_sol_buy.rs:116-118`, `swap_sol_sell.rs:289-310`

## Analysis

The micro-tax threshold at 4 lamports is chosen because it's the minimum amount that can be meaningfully split into three parts (1+1+2 = 4 lamports, still sub-dust). Below this threshold, the function routes all tax to staking (the primary revenue stream at 71%).

**Conditions triggering H057:**
- Total swap amount < 4 lamports (extremely rare on mainnet where slot costs ~5,000 lamports)
- Most likely on devnet/testnet with micro-transactions

**Economic impact:**
- Carnage Fund misses its 24% portion: 0 to ~1 lamport (negligible)
- Treasury misses its 5% portion: 0 to ~0.2 lamport (negligible)
- Staking receives 100% instead of 71% (but on micro amounts only)

**Verification:**
The invariant `staking + carnage + treasury == total_tax` is preserved: `total_tax + 0 + 0 == total_tax`. No funds are lost; they are simply consolidated into a single destination on micro-swaps.

## Verdict

**Not vulnerable.** The edge case is intentional, well-documented, and economically harmless. The routing decision makes sense: splitting sub-lamport amounts across three destinations would create three zero-amount transfers (invalid on Solana), so consolidation to the primary recipient (staking) is the correct behavior. This pattern is common in DeFi dust-handling and is not exploitable.

The design trades off perfect 71/24/5 split distribution to avoid spurious zero-amount outputs. On mainnet, transactions large enough to incur tax will almost always be >4 lamports due to reasonable minimum swap amounts.
