# H035: Sell-Side WSOL Delegate Authority Exploit

## Status

**FALSE POSITIVE.** The hypothesized exploit cannot occur. SPL Token's Transfer instruction enforces that the provided authority must either be the token account's `owner` OR the account's current `delegate` — and the instruction explicitly validates that the signed-by key matches whichever authority role is used. The attack premise conflates two independent questions: (1) who *can* sign a Transfer instruction, and (2) whether a third party can sign on the user's behalf without the user's transaction-level signature. The on-chain code correctly names the user as the required signer, and the Solana runtime enforces this regardless of any delegate state.

---

## Summary

`swap_sol_sell.rs` constructs a raw SPL Token Transfer (`3u8`) instruction at lines 289–310 to move the tax portion of WSOL from the user's WSOL ATA to the `wsol_intermediary` account. The hypothesis states that if the WSOL account has a delegate with remaining allowance, the delegate could complete this transfer. This is incorrect — SPL Token's Transfer instruction requires the authority account to be a signer in the transaction. Since the instruction names `ctx.accounts.user` as the required signer (`AccountMeta::new_readonly(ctx.accounts.user.key(), true)`), only a transaction signed by the user can satisfy that constraint. A delegate with an open allowance cannot substitute for the user as transaction signer unless the delegate also controls a separate approved transfer — neither of which harms the user here.

---

## Analysis

### SPL Token Transfer Semantics

The SPL Token Transfer instruction (discriminator `3u8`) accepts three accounts: `[source, destination, authority]`. The authority must be a signer. SPL Token resolves the authority in one of two ways:

1. **Owner authority**: `source.owner == authority.key` — accepted if `authority.is_signer`.
2. **Delegate authority**: `source.delegate == Some(authority.key)` AND `source.delegated_amount >= transfer_amount` — accepted if `authority.is_signer`.

Critically, in **both cases** the authority must be a transaction-level signer. SPL Token does not allow an unsigned delegate to drain a token account behind the owner's back. A delegate can only transfer tokens if the delegate key appears as a signer in the transaction — meaning the delegate holder must co-sign the transaction.

### The Constructed Instruction

```rust
// swap_sol_sell.rs:289-310
let transfer_tax_ix = Instruction {
    program_id: ctx.accounts.token_program_a.key(),
    accounts: vec![
        AccountMeta::new(ctx.accounts.user_token_a.key(), false),        // source (not signer)
        AccountMeta::new(ctx.accounts.wsol_intermediary.key(), false),   // destination (not signer)
        AccountMeta::new_readonly(ctx.accounts.user.key(), true),        // authority (IS signer)
    ],
    data: {
        let mut d = vec![3u8]; // Transfer discriminator
        d.extend_from_slice(&tax_amount.to_le_bytes());
        d
    },
};
invoke(
    &transfer_tax_ix,
    &[
        ctx.accounts.user_token_a.to_account_info(),
        ctx.accounts.wsol_intermediary.to_account_info(),
        ctx.accounts.user.to_account_info(),
        ctx.accounts.token_program_a.to_account_info(),
    ],
)?;
```

The instruction specifies `user` as the signing authority (`true` in `AccountMeta::new_readonly`). The account struct declares `user: Signer<'info>` (line 515), meaning Anchor validates that `user` is a transaction signer before the handler executes. The Solana runtime then enforces the signer constraint when `invoke` is called.

### The Delegate Scenario

Suppose the user's WSOL ATA has a delegate `D` with allowance `1000 WSOL`. The question is: can `D` exploit this to drain the user's WSOL?

**Scenario A — Attacker calls `swap_sol_sell` without user's transaction signature:** The Anchor `Signer<'info>` constraint on `user` fails immediately. The instruction never executes.

**Scenario B — User calls `swap_sol_sell` normally; delegate `D` has open allowance:** The tax transfer succeeds using `user` as the owner-authority. The delegate's existence is irrelevant — SPL Token sees the owner signing and performs the transfer. The delegate allowance is NOT consumed and NOT involved. The user loses only the correct tax amount.

**Scenario C — Delegate `D` constructs its own SPL Transfer to drain user WSOL:** This is unrelated to `swap_sol_sell`. If a delegate has been granted allowance by the user, that is a separate pre-existing user action. The Tax Program neither creates nor expands any delegate allowance. The Tax Program has no exposure here.

None of the three scenarios result in an exploit attributable to the Tax Program's code.

### Account Constraint Confirmation

The `SwapSolSell` struct at line 512–652 shows:

```rust
// Line 514-515
#[account(mut)]
pub user: Signer<'info>,

// Line 569-570
#[account(mut)]
pub user_token_a: InterfaceAccount<'info, TokenAccount>,
```

`user_token_a` has no `has_one = user` ownership constraint at the Anchor level. However, this does not enable an exploit because:

1. SPL Token's Transfer instruction validates `authority == source.owner` (or delegate). Since `user` is named as authority in the instruction, SPL Token will reject the transfer unless `user.key == user_token_a.owner`. If an attacker passes a `user_token_a` they do not own with themselves as `user`, the SPL Token call fails with `OwnerMismatch`.
2. If an attacker passes a `user_token_a` they *do* own (but it belongs to them), they are only transferring their own funds — no theft occurs.

The absence of an explicit `has_one = user` Anchor constraint on `user_token_a` does mean an attacker could pass an arbitrary WSOL account as `user_token_a`. If they do so with a WSOL account that `user` owns (the signer), the transfer is legitimate. If they pass one they do not own, SPL Token rejects it. The worst-case impact of the missing `has_one` constraint is that a user could accidentally pass the wrong WSOL account — an operational error, not an attacker-exploitable path.

### Comparison with `transfer_checked`

The token-economic context notes (line 58) that the raw `3u8` discriminator is used rather than `transfer_checked` (discriminator `12u8`). `transfer_checked` adds a mint and decimals validation step. For WSOL (which is standard SPL Token, not Token-2022), the absence of `transfer_checked` is acceptable since there is no transfer hook to invoke. The authority enforcement behavior is identical between `transfer` and `transfer_checked` — both require the authority to be a transaction signer.

## Code Evidence

**Primary code path under analysis:**

- `programs/tax-program/src/instructions/swap_sol_sell.rs:289-310` — raw SPL Transfer instruction construction
- `programs/tax-program/src/instructions/swap_sol_sell.rs:514-515` — `user: Signer<'info>` constraint enforcing transaction-level signature
- `programs/tax-program/src/instructions/swap_sol_sell.rs:569-570` — `user_token_a` account definition (no explicit `has_one` constraint)

**SPL Token behavior reference:**

SPL Token `process_transfer` (spl-token/src/processor.rs) calls `Self::validate_owner` which checks: if `source.delegate == Some(expected_authority)` AND `source.delegated_amount >= amount`, the delegate is accepted — but only if the delegate key is a signer. The `is_signer` flag is enforced by the Solana runtime before any program logic executes.

## Verdict

**NOT EXPLOITABLE.** The raw `3u8` SPL Transfer discriminator is correct for WSOL. The user is correctly named as the required authority signer in the instruction. SPL Token's Transfer instruction enforces that the named authority must be a transaction signer. The Anchor `Signer<'info>` constraint on `user` ensures this. A delegate with open allowance cannot sign on behalf of the user without controlling the user's private key. No attack vector exists in this code path.

The one minor observation — the absence of a `has_one = user` constraint on `user_token_a` — does not constitute a security vulnerability in isolation. Its only effect is that a caller could theoretically pass a different WSOL account, but SPL Token would reject the transfer unless the signer owns that account. A note-level recommendation is included below for defense-in-depth completeness.

---

## Recommendations

### R1 — Add explicit `has_one` or `constraint` for `user_token_a` ownership (NOTE severity)

Add a constraint to the Anchor struct to explicitly enforce that `user_token_a.owner == user.key()`. This makes the ownership relationship compile-time verifiable and auditable, removing any ambiguity:

```rust
#[account(
    mut,
    token::mint = mint_a,
    token::authority = user,
)]
pub user_token_a: InterfaceAccount<'info, TokenAccount>,
```

This eliminates the theoretical path where a caller passes an unrelated WSOL account. It has no impact on valid user behavior (users always pass their own WSOL ATA) but strengthens the defense-in-depth posture.

### R2 — Consider `transfer_checked` over raw `transfer` for consistency (NOTE severity)

While the behavior is equivalent for WSOL, replacing the raw `3u8` discriminator with `transfer_checked` (`12u8`) with explicit mint and decimals arguments would:
- Prevent any future mistake if the token program for this path changes to Token-2022
- Align with the rest of the codebase which uses `transfer_checked` for Token-2022 tokens
- Make the intent explicit in the instruction encoding

Neither recommendation affects the security of the current implementation. H035 is closed as a false positive.
