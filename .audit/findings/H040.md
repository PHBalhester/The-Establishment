## Summary

Six locations across the Tax Program and Epoch Program build raw SPL Token / Token-2022
instruction data by hand, embedding integer discriminator bytes rather than calling the
SDK's `instruction::transfer()`, `instruction::burn()` etc. factory functions. The concern
raised is that a future SPL Token encoding change would silently break these call sites.

Investigation confirms the discriminator values are correct for both `spl-token 8.0.0` and
`spl-token-2022 8.0.1` (the crate versions locked in `Cargo.lock`). It also confirms these
byte values have been stable since SPL Token's initial public deployment in 2020 and are
effectively frozen — they are part of Solana's published on-chain interface contract. The
risk is real in theory but negligible in practice. What is missing is any constant-level
assertion or comment linking the magic number to the SDK source, and the Burn path
(`8u8`) has no LiteSVM-level test that actually executes the CPI through a real token
program.

### Correctness Verification

Cross-referenced against the locked crate sources in
`~/.cargo/registry/src/.../spl-token-8.0.0/src/instruction.rs` and
`~/.cargo/registry/src/.../spl-token-2022-8.0.1/src/instruction.rs`.

Both files confirm:
- `Transfer { amount }` → `buf.push(3)` (line 601 / line 901)
- `Approve { amount }` → `buf.push(4)` (line 605 / line 905)
- `Burn { amount }` → `buf.push(8)` (line 613 / line 913)
- `CloseAccount` → `buf.push(9)` (line 625 / line 926)
- `SyncNative` → `buf.push(17)` (line 653 / line 953)
- `InitializeAccount3` → `buf.push(18)` (line 656 / line 956)

All six values are correct. Token-2022 uses the same byte assignments as SPL Token for
these legacy variants — the Token-2022 instruction set is a strict superset, preserving
the original ordinals.

### Historical Stability

The SPL Token instruction enum was defined in the Solana Program Library before mainnet
launch and has not changed the byte assignment of any existing variant since. The SPL
Token program itself is a fixed-address, immutable deployed program (`TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA`). Token-2022 was introduced later as a separate program
(`TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb`) with full discriminator compatibility
for the base instruction set. These integers are part of the wire format that every Solana
RPC, explorer, and off-chain tool relies on — changing them would be a breaking protocol
change requiring a network-wide coordination effort.

The concern in the original issue report ("if SPL Token encoding changes, these silently
break") is therefore theoretical, not an active risk on the Solana network as of 2026.

### Contrast with Anchor-Derived Discriminators

The codebase correctly handles the distinct case where discriminator stability is NOT
guaranteed: Anchor instruction selectors (sha256("global:instruction_name")[0..8]) are
stored as named constants with `#[test]` assertions that re-derive the hash at test time
(`DEPOSIT_REWARDS_DISCRIMINATOR`, `UPDATE_CUMULATIVE_DISCRIMINATOR`,
`SWAP_EXEMPT_DISCRIMINATOR` — see `tax-program/src/constants.rs:156`,
`epoch-program/src/constants.rs:117,177`). The SPL Token variants receive no analogous
assertion.

### Test Coverage Assessment

**Covered (indirectly via LiteSVM integration):**
- `3u8` Transfer — exercised by every sell test in `test_swap_sol_sell.rs` (tests
  `test_sell_crime_with_tax`, `test_sell_fraud_with_tax`, etc.). The WSOL tax transfer
  step fires this CPI through LiteSVM's real SPL Token program.
- `9u8` CloseAccount — same sell tests; close follows the transfer.
- `17u8` SyncNative — `fund_native_wsol()` helper in `test_swap_sol_sell.rs:567`.
- `18u8` InitializeAccount3 — used in WSOL account setup helpers across at least 8 test
  files (confirmed by search), including `test_swap_sol_sell.rs:514`,
  `test_swap_sol_buy.rs:558`, `test_cpi_access_control.rs:391`,
  `test_pool_initialization.rs:406`, `test_transfer_routing.rs:498,570`.

**Not covered by any LiteSVM or automated test:**
- `8u8` Burn — `burn_held_tokens()` in `carnage_execution.rs:462-547`. The only tests
  for the Burn path are `test_partition_hook_accounts.rs:127,150,361`, which test account
  routing logic only (no actual CPI to a token program is made). The integration test
  `carnage.test.ts` logs `carnageAction === 1 ? "Burn"` but does not contain a test case
  that drives the Burn branch of `execute_carnage_atomic` / `execute_carnage`.
- `4u8` Approve — `approve_delegate()` in `carnage_execution.rs:860-906`. This is called
  as a pre-step in the sell-from-carnage path. No unit or LiteSVM test exercises it.

The absence of Burn and Approve CPI test coverage is the meaningful gap here. If either
byte were wrong it would fail at runtime on devnet during the first forced-Burn epoch, not
be caught by the test suite.

## Verdict

**The discriminator values are correct and the risk of silent breakage from an SPL Token
encoding change is negligible** — these bytes are part of Solana's immutable on-chain
wire format and have not changed since mainnet genesis.

The real, actionable gap is narrower:

1. **Missing test coverage for `8u8` Burn and `4u8` Approve CPIs.** The Burn branch of
   Carnage execution is not exercised by any LiteSVM test. If `burn_held_tokens` were
   mistakenly refactored or if the byte were corrupted by a future merge conflict, the
   only detection point is a devnet run with a VRF outcome that triggers the Burn action
   (probabilistic, ~3.5% of Carnage events that target a token the fund already holds).

2. **No constant-level documentation linking the magic bytes to their SDK source.**
   The Anchor-discriminator constants (`DEPOSIT_REWARDS_DISCRIMINATOR`, etc.) have both
   a named constant and a `#[test]` that re-derives the value. The SPL Token
   discriminators have only a comment. Adding named constants (`SPL_TRANSFER_IX: u8 = 3`)
   with a unit test that cross-checks `spl_token::instruction::transfer(...)?.data[0]`
   would close the documentation and regression gap without any runtime cost.

Severity is lowered from the initially-flagged Tier 2 to **Low** because (a) the values
are verified correct, (b) historical stability is essentially guaranteed by on-chain
immutability, and (c) the main residual risk is test-coverage quality rather than a live
vulnerability.

## Recommendations

**R1 — Extract named constants for SPL byte values (cosmetic, Low effort).**
Define at the top of each relevant file:
```rust
const SPL_IX_TRANSFER: u8 = 3;        // TokenInstruction::Transfer
const SPL_IX_APPROVE: u8 = 4;         // TokenInstruction::Approve
const SPL_IX_BURN: u8 = 8;            // TokenInstruction::Burn
const SPL_IX_CLOSE_ACCOUNT: u8 = 9;   // TokenInstruction::CloseAccount
const SPL_IX_SYNC_NATIVE: u8 = 17;    // TokenInstruction::SyncNative
const SPL_IX_INIT_ACCOUNT3: u8 = 18;  // TokenInstruction::InitializeAccount3
```
Replace inline magic numbers with these constants throughout both programs. Add a
`#[test]` that asserts `spl_token::instruction::burn(...).unwrap().data[0] == SPL_IX_BURN`
etc. using the actual SDK factory, mirroring the pattern used for Anchor discriminators.

**R2 — Add a LiteSVM test for the Burn CPI path (Medium effort, Higher priority).**
Add a test in `epoch-program/tests/` that:
- Creates a Token-2022 mint + vault + `CarnageFundState` PDA.
- Calls `burn_held_tokens()` directly (or via a thin wrapper) through LiteSVM.
- Asserts the vault balance decreased and `carnage_state.held_amount == 0`.

This ensures the `8u8` byte is exercised end-to-end before a mainnet Carnage Burn event
would be the first real test.

**R3 — Add a LiteSVM test for the Approve CPI path (Low effort).**
Similarly test `approve_delegate()` in isolation through LiteSVM to confirm the `4u8`
byte and account ordering are correct.

**Priority: R2 > R3 > R1.** The constants refactor is cosmetic; the missing test coverage
for Burn and Approve is the only operationally relevant gap.
