# H016: swap_exempt minimum_output=0 Sandwich Within AMM

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** MEV, CPI
**Origin:** KB (EP-060)

## Summary

The hypothesis asserts that `minimum_output=0` in `swap_exempt` plus the Epoch Program's post-swap slippage floor creates a gap: an attacker could manipulate reserves between the floor calculation and the actual swap. Code-level analysis finds this gap does not exist. The reserve snapshot and the swap CPI are in strict sequential order within a **single instruction execution on a single transaction**. There is no observable inter-instruction window between them on Solana. The `minimum_output=0` at the AMM layer is an intentional design choice; the Epoch Program's own post-swap slippage check is the operative protection, and it uses reserves read immediately before the swap CPI in the same instruction body — not in a prior instruction or a prior slot.

## Analysis

### The Hypothesised Attack Vector

The attack requires:
1. Attacker manipulates reserves BETWEEN the floor calculation (reading reserves) and the actual swap CPI.
2. Floor is computed from pre-manipulation reserves (so it looks acceptable), but the actual swap executes against post-manipulation reserves (giving Carnage worse output).
3. Post-swap check sees "received >= floor" and passes because both floor and actual output reflect the manipulated state — but the floor itself was stale relative to the attack.

### Why the Gap Cannot Exist

On Solana, a single instruction executes atomically. There is no mechanism for another account state mutation to occur between two lines of Rust code in the same instruction handler. The sequence in `carnage_execution.rs` is:

```
line 285: read_pool_reserves(target_pool_info, ...) -> (reserve_sol, reserve_token)
...
line 304: execute_buy_swap(accounts, total_buy_amount, target, ...)  [swap CPI]
...
line 331–350: slippage floor check against reserve_sol, reserve_token + actual output
```

All three steps execute sequentially in the same instruction. The pool account passed to `read_pool_reserves` is the same `AccountInfo` pointer passed via `execute_buy_swap` → `execute_swap_exempt_cpi` → Tax::swap_exempt → AMM::swap_sol_pool. No other transaction can modify the pool's on-chain state between line 285 and line 304 within the same transaction — Solana's single-threaded instruction execution model guarantees this.

The comment in the code at lines 278–280 explicitly documents the ordering rationale:

> "Must read BEFORE the swap CPI because Solana runtime updates AccountInfo data in-place after CPI returns."

This note is about the opposite concern (not wanting to read *stale* post-CPI data), confirming the developer is aware of and controlling the read-before-write ordering.

### The Actual Relationship Between Floor and Swap

Both the floor calculation and the swap CPI operate on the same reserve state:

| Step | Reserve state visible |
|------|----------------------|
| `read_pool_reserves` (line 285) | Pre-swap (correct) |
| `execute_buy_swap` → AMM (line 304) | Pre-swap (AMM reads its own pool.reserve_a/b) |
| Post-swap vault delta check (lines 319–322) | Post-swap (vault balance delta) |
| Slippage floor check (lines 331–350) | Compares post-swap output against floor derived from pre-swap reserves |

The floor is derived from pre-swap reserves, and the actual output is the vault delta from the just-completed swap. The comparison is correctly formed: "did the actual swap produce at least X% of what a fair constant-product swap on these pre-swap reserves would have produced?" This is a standard post-swap slippage check and there is no ordering gap.

### What minimum_output=0 Actually Means

`MINIMUM_OUTPUT: u64 = 0` in `swap_exempt.rs:111` means the **AMM layer** imposes no slippage floor independently. This is architecturally correct:

- The AMM is a shared primitive gated to Tax Program exclusively (via `swap_authority` PDA with `seeds::program = TAX_PROGRAM_ID`). The AMM cannot be called by anyone other than the Tax Program.
- All slippage protection for the Carnage path is lifted one level up to the Epoch Program, which performs the check using its own reserve snapshot.
- For ordinary user swaps, the Tax Program enforces `MINIMUM_OUTPUT_FLOOR_BPS = 5000` (50% floor) on the user-supplied `minimum_output` before calling the AMM. The AMM receives a non-zero value from the Tax Program for user swaps.
- `minimum_output=0` only flows to the AMM when the Epoch Program is the caller (via `swap_exempt`). The Epoch Program replaces the AMM-level check with its own stronger post-swap check.

This is a layered but complete defence. There is no net zero-protection gap.

### Distinguishing H016 from H010

H010 (CONFIRMED HIGH) documents a real MEV vulnerability in the **fallback path** where reserves are read at execution time (after an attacker front-run can inflate them) rather than from a pre-committed snapshot. That finding stands. The key distinction:

- **H010 attack**: Attacker manipulates pool state *before the transaction containing execute_carnage is submitted*. The reserve read in that transaction reflects the already-manipulated pool. The floor is computed from the manipulated baseline. The Epoch Program's check passes because the attacker only extracted up to 25% of the (already-degraded) expected output. The 75% floor is sound as a floor but does not detect manipulation of the baseline itself.

- **H016 hypothesis**: Attacker manipulates pool state *between the reserve read and the swap CPI within the same instruction*. This is impossible on Solana — instructions are atomic. The hypothesis relies on a timing model that does not exist in the Solana execution environment.

H016 does not identify an additional vulnerability beyond H010. The `minimum_output=0` line is the AMM-level parameter; the protection layer is correctly located one level up (Epoch Program post-swap check). The gap described in the hypothesis (between floor calculation and swap) is closed by Solana's single-instruction atomicity.

## Atomicity Assessment

**The floor calculation and swap CPI are atomic with respect to any external state change.**

Solana's execution model:
- All instructions in a transaction execute serially, with each instruction seeing the state as modified by prior instructions in the same transaction.
- No other transaction can interleave with or modify shared account state mid-instruction.
- The only state changes visible mid-instruction are those caused by CPIs *within that instruction*, which return before the calling code continues.

In `execute_carnage_core()`:
1. `read_pool_reserves` reads pool data → binds `reserve_sol`, `reserve_token` on the stack.
2. `execute_buy_swap` issues the swap CPI → AMM modifies pool state → CPI returns.
3. Post-swap vault reload + delta calculation → actual output measured.
4. Slippage floor check compares actual output to floor derived from step-1 reserves.

No external actor can insert a pool state modification between steps 1 and 2. The reserves read in step 1 are the same reserves the AMM will see when it begins executing the swap (since neither the Epoch Program nor the Tax Program modify pool reserves between the two steps, and no other party can execute in that window).

**Conclusion:** The hypothesised "gap between floor calculation and swap" cannot exist within a single Solana instruction. The `minimum_output=0` at the AMM layer is an intentional delegation of slippage protection to the caller (Epoch Program), which exercises that protection correctly via the post-swap check.

## Verdict

**NOT VULNERABLE.**

The specific attack modelled in H016 — manipulating reserves between the floor calculation and the swap CPI within the same instruction — is not possible on Solana. Single-instruction atomicity closes this gap absolutely.

The `minimum_output=0` in `swap_exempt.rs:111` is correctly documented and architecturally sound: the Epoch Program owns the slippage floor for Carnage swaps, and its post-swap check (lines 331–350) provides that protection using the correct reserve snapshot.

The real MEV vulnerability in the Carnage path is the H010 finding (fallback path front-running across separate transactions). H016 adds no new vulnerability to that analysis.

## Recommendations

No code changes required for the H016 scenario. The existing architecture correctly delegates slippage enforcement from the AMM layer to the Epoch Program layer.

One minor improvement for documentation clarity: the comment at `swap_exempt.rs:109–111` already states "no slippage protection per Carnage_Fund_Spec Section 9.3", but the spec reference is to an internal spec document, not to the on-chain code. Adding an inline reference pointing to the Epoch Program's post-swap check (carnage_execution.rs:331–350) as the operative guard would help future auditors verify the layered protection is present and understand why `0` is safe here.

**Cross-reference:** See H010 for the confirmed MEV vulnerability in the Carnage fallback path (across-transaction front-running, not within-instruction ordering).
