# H061: start_curve Unchecked Addition (deadline overflow)

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** Arithmetic

## Summary

The `start_curve` instruction in the bonding curve program computes the deadline slot via `clock.slot + DEADLINE_SLOTS` without using checked arithmetic. However, analysis of the current parameters shows this cannot overflow in practice, and even theoretically the operation would only overflow near u64::MAX (approximately year 100,000 of network operation).

## Code Evidence

- **File:** `programs/bonding_curve/src/instructions/start_curve.rs:69-74`

```rust
let clock = Clock::get()?;
let curve = &mut ctx.accounts.curve_state;

curve.status = CurveStatus::Active;
curve.start_slot = clock.slot;
curve.deadline_slot = clock.slot + DEADLINE_SLOTS;  // <-- Unchecked addition
```

- **DEADLINE_SLOTS definition:** `programs/bonding_curve/src/constants.rs:102-109`

```rust
/// Deadline duration in slots.
///
/// Devnet: 27,000 slots (~3 hours)
/// Localnet: 500 slots
/// Mainnet: 432,000 slots (~48 hours at 400ms/slot)
#[cfg(feature = "devnet")]
pub const DEADLINE_SLOTS: u64 = 27_000;

#[cfg(feature = "localnet")]
pub const DEADLINE_SLOTS: u64 = 500;

#[cfg(not(any(feature = "devnet", feature = "localnet")))]
pub const DEADLINE_SLOTS: u64 = 432_000;
```

## Analysis

**Overflow analysis:**

- **Solana slot pace:** ~400ms per slot on mainnet (2.5 slots/second)
- **Current slot (March 2026):** ~246M slots (data from memory)
- **u64::MAX:** 18,446,744,073,709,551,615 slots (~1.8e19)
- **Headroom:** (1.8e19 - 2.46e8) / 2.5 ≈ 7.2e18 seconds ≈ 228 million years

**Practical overflow scenarios:**

1. **Mainnet (DEADLINE_SLOTS = 432,000):**
   - Required clock.slot: u64::MAX - 432,000 = 18,446,744,073,709,119,615
   - Time to overflow: ~228 million years at current pace
   - **Conclusion:** Cannot happen during Solana network lifetime

2. **Devnet (DEADLINE_SLOTS = 27,000):**
   - Devnet resets frequently (weekly or monthly)
   - Each reset starts at slot 0
   - **Conclusion:** Cannot happen (reset before overflow possible)

3. **Localnet (DEADLINE_SLOTS = 500):**
   - Test environment, not production
   - **Conclusion:** N/A

**Why unchecked is acceptable:**

The addition `clock.slot + DEADLINE_SLOTS` can only overflow if clock.slot is already past u64::MAX - 432,000. This represents a network state so far in the future that:

1. The Solana network would have undergone multiple protocol upgrades
2. Mainnet slot space might have been reorganized (e.g., slot rollover mechanism added)
3. The deadline_slot field itself might have changed to u128 or a different type
4. The bonding curve program would likely be retired or reimplemented

**Current practice in Dr. Fraudsworth codebase:**

The codebase consistently uses unchecked arithmetic for clock operations that cannot realistically overflow:
- Epoch Program: `consume_randomness.rs` uses `.checked_add(CARNAGE_DEADLINE_SLOTS)` (checked)
- Bonding Curve: `start_curve.rs` uses bare `+` (unchecked)

This inconsistency is minor but worth noting: the Epoch Program is more conservative with deadline arithmetic (checked), while the Bonding Curve is not.

**Comparable operations in the codebase:**

- `epoch-program/src/instructions/consume_randomness.rs:107`: Uses `checked_add` for deadline
- `epoch-program/src/instructions/force_carnage.rs:66`: Uses `checked_add` for deadline

Both Epoch Program deadline computations use `checked_add`, suggesting a project pattern preference for defensive arithmetic.

## Verdict

**Not vulnerable.** The unchecked addition cannot overflow during any realistic network timeline. However, for consistency with the Epoch Program's pattern and to be defensive, this should be changed to `checked_add` for stylistic reasons, not security reasons.

**Recommendation (stylistic only):**

```rust
let deadline_slot = clock.slot
    .checked_add(DEADLINE_SLOTS)
    .ok_or(CurveError::Overflow)?;
curve.deadline_slot = deadline_slot;
```

This change:
- Matches the Epoch Program's pattern (`consume_randomness.rs`)
- Makes the assumption (overflow impossible) explicit
- Provides defense-in-depth if DEADLINE_SLOTS is ever changed to an unbounded value
- Adds zero runtime cost (checked_add is a single CPU instruction)
- Fails gracefully (returns error instead of panicking or wrapping)

**Effort:** One line change, no logic change, no functional impact.

**Impact classification:** INFORMATIONAL — style improvement, not a bug fix.
