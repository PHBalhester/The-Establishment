# H042: BC initialize_curve Fix Verification

**Status:** CONFIRMED FIXED
**Severity:** N/A (RECHECK -- previously MEDIUM)
**Category:** Access Control
**Origin:** RECHECK (H003)

## Summary

The `initialize_curve` instruction now properly enforces authorization via the `BcAdminConfig` PDA. The instruction contains a `has_one = authority` constraint on the `BcAdminConfig` account, requiring the caller to be the same authority stored in the admin config. This fully addresses the prior H003 vulnerability where any signer could initialize curves.

## Code Evidence

**File:** `programs/bonding_curve/src/instructions/initialize_curve.rs:22-27`

```rust
/// BcAdminConfig PDA -- gates admin operations.
#[account(
    seeds = [BC_ADMIN_SEED],
    bump = admin_config.bump,
    has_one = authority @ CurveError::Unauthorized,
)]
pub admin_config: Account<'info, BcAdminConfig>,
```

**File:** `programs/bonding_curve/src/instructions/initialize_curve.rs:17-19`

```rust
/// Protocol authority. Must match BcAdminConfig.authority. Pays rent for all new accounts.
#[account(mut)]
pub authority: Signer<'info>,
```

The `has_one = authority` constraint enforces that the `authority` signer MUST match the `admin_config.authority` field. This is the standard Anchor pattern for admin gating.

## Verdict

**CONFIRMED FIXED.** The initialization front-running vulnerability (H003) has been resolved by implementing proper admin authority gating. The `initialize_curve` instruction can now only be called by the authority stored in `BcAdminConfig`, which is itself protected by the ProgramData upgrade-authority check in `initialize_bc_admin`. This creates a complete authorization chain preventing unauthorized curve initialization.

The fix aligns with the protocol's broader access control pattern used in other programs (AMM, Transfer Hook, etc.) where admin operations are gated through PDA-stored authority values validated via `has_one` constraints.
