# H048: Buy Path 50% Output Floor Adequacy (Recheck H014)

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** Token/Economic, Timing

## Summary

The 50% output floor (MINIMUM_OUTPUT_FLOOR_BPS = 5000) is correctly calculated and applied in the buy path at `swap_sol_buy.rs:106-111`, using live pool reserves read at instruction time. The floor calculation in `tax_math.rs:141-165` properly uses u128 intermediates and checked arithmetic. However, the 50% threshold is **intentionally generous** and permits up to ~50% value extraction via sandwich attacks. This is not a bug but an architectural decision: the floor is a protocol-level backstop for broken clients, while tight slippage protection is delegated to the user-specified `minimum_output` parameter.

## Code Evidence

### Floor Calculation

**File:** `programs/tax-program/src/helpers/tax_math.rs:141-165`
```rust
pub fn calculate_output_floor(
    reserve_in: u64,
    reserve_out: u64,
    amount_in: u64,
    floor_bps: u64,
) -> Option<u64> {
    if reserve_in == 0 || reserve_out == 0 || amount_in == 0 {
        return Some(0);
    }

    // Constant-product expected output:
    let numerator = (reserve_out as u128).checked_mul(amount_in as u128)?;
    let denominator = (reserve_in as u128).checked_add(amount_in as u128)?;
    let expected = numerator.checked_div(denominator)?;

    // Floor = expected * floor_bps / 10_000
    let floor = expected
        .checked_mul(floor_bps as u128)?
        .checked_div(10_000)?;

    // Checked cast
    u64::try_from(floor).ok()
}
```

**Constant:** `programs/tax-program/src/constants.rs:40`
```rust
const MINIMUM_OUTPUT_FLOOR_BPS: u64 = 5000;  // 50%
```

### Floor Enforcement in Buy Path

**File:** `programs/tax-program/src/instructions/swap_sol_buy.rs:105-111`
```rust
let (sol_reserve, token_reserve) = read_pool_reserves(&ctx.accounts.pool)?;
let output_floor = calculate_output_floor(
    sol_reserve,
    token_reserve,
    sol_to_swap,  // <-- Uses post-tax SOL amount
    MINIMUM_OUTPUT_FLOOR_BPS
).ok_or(error!(TaxError::TaxOverflow))?;

require!(
    minimum_output >= output_floor,
    TaxError::MinimumOutputFloorViolation
);
```

### Key Detail: Uses Live Reserves

The floor is calculated using `read_pool_reserves(&ctx.accounts.pool)?` which reads the **current AMM state** via raw byte offsets at instruction time. The pool reserves are not stale — they reflect the present pool state before the AMM CPI executes.

**File:** `programs/tax-program/src/helpers/pool_reader.rs:79-88` (reading logic)

pub fn read_pool_reserves(pool: &AccountInfo) -> Result<(u64, u64), TaxError> {
    let pool_data = pool.data.borrow();
    if pool_data.len() < 153 {

    let reserve_a = u64::from_le_bytes([...pool_data[137..145]...]);
    let reserve_b = u64::from_le_bytes([...pool_data[145..153]...]);

    Ok((reserve_a, reserve_b))
}

## Floor Correctness Analysis

### Is the Calculation Correct?

**YES.** The formula uses the standard constant-product expected output:
```
expected = reserve_out * amount_in / (reserve_in + amount_in)
floor = expected * 50% / 100%
```

This is mathematically correct for the x*y=k model. The u128 intermediates prevent overflow (verified: u64::MAX * u64::MAX < u128::MAX).

### Is 5000 BPS (50%) Too Generous?

**YES, by design.** The 50% floor is intentionally conservative to:

1. **Prevent pathological zero-slippage cases:** A user who sets `minimum_output = 0` is now protected to 50% of expected output (from 0% to 50%).

2. **Allow large swaps with thin liquidity:** The floor is loose enough that a large buy against a small reserve doesn't get rejected. A 100x impact to pool reserves would have output floor = expected * 50%, which still permits the swap to execute.

3. **Delegate tight protection to users:** Frontend/wallet should set `minimum_output` to 1-5% slippage tolerance. The 50% floor is a backstop, not the intended UX.

### Is 5000 BPS Too Restrictive?

**NO.** The threshold is generous enough that:
- Users setting `minimum_output` to 95% of expected output (5% slippage) will always pass the floor.
- A legitimate large swap against thin liquidity could still execute (e.g., 40% impact, output = 60% of expected, still passes 50% floor).
- Only sandwich-attacked swaps or misconfigured clients would hit the floor.

## Attack Surface

### Sandwich Attack with 50% Floor

**Attacker strategy:**
1. Front-run: large buy pushes token price up by moving pool reserves significantly.
2. User's buy executes with output diminished by the front-run MEV.
3. User's `minimum_output` is set to (conservatively) 50% floor = 50% of original expected.
4. Attacker back-runs: sells at the now-higher token price, pockets the spread.

**Max extraction:** The attacker can extract ~50% of the expected output IF the user set their minimum at the floor boundary. With realistic user behavior (setting minimum to 1-5% slippage), extraction is much lower (capped at the user's slippage tolerance + sandwich impact).

### Comparison to Sell Path

The buy path has **two layers** of slippage protection:
1. Tax Program output floor check (line 109)
2. User's minimum_output forwarded directly to AMM (line 262)

The sell path has only one effective layer (the post-CPI check), because the AMM is called with `minimum_amount_out = 0`. The buy path is **marginally stronger** but the practical protection level is identical because both depend on the user's `minimum_output` being set appropriately.

## Verdict

**CONFIRMED — Not Vulnerable by Design**

The 50% output floor is correctly implemented, uses live pool reserves, and employs proper checked arithmetic. The threshold is intentionally conservative to permit large swaps and act as a backstop for misconfigured clients. The real slippage protection is the user's `minimum_output` parameter, which the frontend controls.

**No code changes required.** The current design is sound and intentional.

## Recommendation

### For Users/Frontend

Set `minimum_output` to reflect desired slippage tolerance (typically 1-3%), not the protocol floor (50%). Example:
```
minimum_output = expected_output * 9700 / 10000  // 3% slippage tolerance
// This is much tighter than the protocol floor, providing meaningful protection.
```

### For Mainnet Deployment

Consider tightening the floor from 50% to 80-90% if mainnet pool liquidity is deep:
```rust
const MINIMUM_OUTPUT_FLOOR_BPS: u64 = 8000;  // 80%
```

This would reduce sandwich profitability while still permitting large swaps against thin liquidity. **Trade-off:** Smaller legitimate swaps might get rejected if pool depth is lower than expected.

Current 50% floor is safe; tightening is an optional UX improvement for mainnet with confirmed liquidity depth.
