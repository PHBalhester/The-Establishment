# H024: Single Switchboard Oracle Dependency — Protocol Halt

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** Oracle, Risk Management
**Origin:** KB (EP-023)

---

## Summary

The Epoch Program depends entirely on Switchboard On-Demand as its sole source of VRF randomness. There is no fallback oracle, no admin emergency bypass (on mainnet), and no oracle-independent path to advance an epoch. If Switchboard suffers an extended outage, the protocol enters a permanent holding state: tax rates freeze at their last-set values, staking yield stops accumulating, and Carnage cannot trigger. The `retry_epoch_vrf` mechanism — the only recovery path — resolves individual oracle-level failures but is itself entirely dependent on Switchboard infrastructure; it cannot recover from a network-wide Switchboard outage.

---

## Analysis

### The Recovery Path (`retry_epoch_vrf`) Does Not Help Against a Full Outage

`retry_epoch_vrf.rs` allows anyone to rebind a fresh Switchboard randomness account after 300 slots have elapsed since the original VRF request. The handler:

1. Requires `vrf_pending == true` (protocol is stuck mid-epoch-transition)
2. Validates `elapsed_slots > VRF_TIMEOUT_SLOTS` (300 slots, ~2 min)
3. Validates the **new** randomness account with an `owner = SWITCHBOARD_PROGRAM_ID` constraint
4. Applies the same freshness check (`slot_diff <= 1`) as the original trigger

Step 3 is the key constraint. A fresh randomness account must be created by the Switchboard program itself — it cannot be created by an admin or any other authority. The `retry_epoch_vrf` path replaces a stale pending randomness account with a new one, but that new account must still be fulfilled by a Switchboard oracle before `consume_randomness` can proceed. If Switchboard is globally down, the retry merely replaces one unfulfillable randomness account with another.

In other words: `retry_epoch_vrf` handles **per-oracle** failures gracefully (the MEMORY notes confirm this — VRF gateway rotation was tested, and retrying with a fresh account can land on a different working oracle). It does **not** handle **network-wide** Switchboard failure.

### No Mainnet Admin Override

The only instruction capable of advancing epoch state without a Switchboard reveal is `force_carnage`. This instruction:

- Is gated by `#[cfg(feature = "devnet")]` at both module (`mod.rs:7-8`) and instruction (`force_carnage.rs`) level
- Is explicitly excluded from mainnet builds
- Sets `carnage_pending` directly, but does not advance `current_epoch`, update tax rates, or clear `vrf_pending`

Even if `force_carnage` were available on mainnet, it would not unblock a stuck VRF. There is no `emergency_advance_epoch`, `admin_set_tax_rates`, or equivalent instruction anywhere in the program's instruction set (9 total mainnet instructions verified via `mod.rs`).

### Consequence of a Prolonged Outage

When the protocol is in `vrf_pending = true` state and Switchboard cannot fulfill the randomness:

- **Epoch progression halts**: `trigger_epoch_transition` rejects with `VrfAlreadyPending`; no new epoch can start
- **Tax rates freeze**: Existing `crime_buy_tax_bps`, `crime_sell_tax_bps`, `fraud_buy_tax_bps`, `fraud_sell_tax_bps` remain unchanged — swaps still work but at stale rates
- **Staking yield stops**: `update_cumulative` CPI is called inside `consume_randomness`; since `consume_randomness` cannot complete, no new reward epoch is committed to the Staking Program
- **Carnage cannot trigger**: Carnage is derived from VRF bytes in `consume_randomness`; a frozen epoch means permanent no-Carnage
- **Bounty drain stops**: No new bounties are paid from the Carnage vault

Swaps remain functional because the Tax Program reads `EpochState` for tax rates but does not require `vrf_pending == false`. The protocol is not fully dead, but a key part of its economic design — the VRF-randomised asymmetric tax rate game — becomes static.

If the outage occurs **before** `trigger_epoch_transition` is called (i.e., `vrf_pending == false`), the impact is lower: no pending state is stuck, but epoch transitions simply cannot begin until Switchboard recovers, causing epoch skips. The `trigger_epoch_transition` handler explicitly tolerates epoch skips (lines 142-149), so recovery is automatic once Switchboard comes back.

### Switchboard Reliability Context

Switchboard On-Demand is a mature production network with multiple independent oracle operators. A complete multi-oracle outage is a low-probability event. However, the protocol provides no mechanism for the team to mitigate it if it occurs, which elevates the finding above informational.

## Verdict

**CONFIRMED — MEDIUM severity.**

The single-oracle dependency is a real, acknowledged architectural constraint. A complete Switchboard network outage would freeze epoch progression and Carnage triggering indefinitely, with no on-chain recovery path available to the protocol team. Swaps remain functional at stale tax rates, so user funds are not at risk. The finding is real but severity is bounded because:

1. Switchboard On-Demand has strong real-world uptime guarantees
2. A partial outage (single oracle down) IS recoverable via `retry_epoch_vrf` with a fresh account
3. If the outage occurs before `trigger_epoch_transition`, the protocol recovers automatically on Switchboard restoration without admin intervention
4. User funds (tokens, SOL, staking balance) are not affected — only the epoch cadence and Carnage mechanics are halted

The only scenario requiring upgrade-authority intervention is a prolonged global Switchboard outage that coincides with a stuck `vrf_pending` state, which would require deploying an emergency bypass instruction via the upgrade authority (Squads multisig post-launch).

---

## Recommendations

**R1 (Medium priority) — Add an admin emergency epoch advance instruction** gated behind the Squads upgrade authority (or a dedicated emergency admin key). This instruction would clear `vrf_pending`, set a deterministic fallback randomness derivation (e.g., from `clock.slot` + protocol state hash), and advance the epoch. This should carry a timelock (e.g., only callable if `vrf_pending` has been stuck for > 10,000 slots / ~70 min mainnet) to prevent routine misuse.

**R2 (Low priority) — Document the outage scenario in protocol governance materials.** The Squads multisig holders should have a documented runbook: if `vrf_pending = true` for > N slots and all `retry_epoch_vrf` attempts fail, the recovery path is a timelocked program upgrade deploying R1.

**R3 (Informational) — Consider monitoring Switchboard uptime** in the crank runner and emitting an alert (e.g., Discord webhook) if `retry_epoch_vrf` fails repeatedly. This gives the team early warning before the situation becomes critical.

**R4 (Future / post-launch) — Evaluate secondary oracle integration.** A secondary VRF source (e.g., drand, or a simple admin-signed hash-chain as a last resort) could provide a fallback if Switchboard is down for an extended period. This is a significant architectural change and should be weighed against Switchboard's demonstrated reliability.
