# H019: Cross-Epoch Tax Rate Arbitrage via Delayed consume_randomness

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** Timing, Economic, MEV
**Origin:** Novel

---

## Summary

Tax rates in the Tax Program are read directly from EpochState at swap time. The Tax Program performs **no check** on `vrf_pending` or `taxes_confirmed` before using those rates. This creates two distinct windows of exploitable information asymmetry:

1. **VRF-pending window (OLD rates persist):** After `trigger_epoch_transition` advances the epoch number and sets `vrf_pending = true`, old tax rates remain in EpochState until `consume_randomness` completes. The Switchboard oracle reveals the VRF output approximately 3 slots (~1.2s) after commit. During this window, any watcher of Switchboard oracle state can compute the exact new tax rates before they land on-chain, and trade at the *old* (now-known-to-be-expiring) rates.

2. **Deliberate delay by non-cranking:** `consume_randomness` is permissionless — anyone may call it, but **no one is required to**. If the only entity running the crank abstains, the VRF-pending window extends indefinitely (up to `retry_epoch_vrf` recovery at 300 slots). During this window an attacker who knows the new rates can execute unlimited swaps at the old, economically stale rates.

Neither scenario enables the attacker to *choose* the new tax rates (anti-reroll binding prevents VRF manipulation), but they can *know the outcome before it is applied* and act accordingly.

## Analysis

### Timeline of an Epoch Transition

```
Slot N:   trigger_epoch_transition
          → epoch_state.current_epoch advances
          → vrf_pending = true, taxes_confirmed = false
          → OLD tax rates still in epoch_state fields (crime_buy_tax_bps etc.)
          → Switchboard commitIx bundled in same TX; oracle queued

Slot N+1 or N+2: Switchboard oracle posts revealIx on-chain
          → RandomnessAccountData.get_value() now returns the 32 VRF bytes
          → VRF result is PUBLIC on Solana (anyone can read the account)
          → Attacker can call derive_taxes() locally with the 32 bytes
          → New rates are fully known

          *** WINDOW BEGINS: new rates known, old rates still active ***

Slot N+~2: consume_randomness called (if crank is running)
          → epoch_state tax rate fields overwritten with new rates
          → vrf_pending = false, taxes_confirmed = true
          → WINDOW CLOSES

          If crank delays or abstains:
          Window extends for up to 300 slots (~2 min) before retry_epoch_vrf
```

### Why the Tax Program Does Not Defend Against This

The Tax Program's `swap_sol_buy` and `swap_sol_sell` instructions both execute this logic at lines 59-78 of `swap_sol_buy.rs`:

```rust
// Owner check only — no vrf_pending or taxes_confirmed check
require!(ctx.accounts.epoch_state.owner == &epoch_program, TaxError::InvalidEpochState);
let epoch_state = { ... EpochState::try_deserialize(...) };
require!(epoch_state.initialized, TaxError::InvalidEpochState);
let tax_bps = epoch_state.get_tax_bps(is_crime, true);
```

The `EpochState` struct (mirrored in `tax-program/src/state/epoch_state_reader.rs`) contains both `vrf_pending: bool` and `taxes_confirmed: bool`, but the Tax Program reads neither. It proceeds directly to `get_tax_bps()` regardless of epoch transition state.

### The VRF Reveal Is Public

The Switchboard randomness account (`epoch_state.pending_randomness_account`) is a public Solana account. Once the oracle executes `revealIx`, `RandomnessAccountData::get_value(slot)` returns the 32-byte result. Any node can:

1. Subscribe to Switchboard oracle transactions via RPC
2. Observe the reveal transaction
3. Call `derive_taxes(vrf_bytes, current_cheap_side)` locally
4. Know exact new rates: flip decision (byte 0), CRIME low/high (bytes 1-2), FRAUD low/high (bytes 3-4)

This computation is trivial — it is 5 conditional branches and two array lookups.

### Information Value: What Can the Attacker Extract?

The attacker knows two things before `consume_randomness` lands:

**A. Whether cheap_side flips (byte 0 < 192, 75% flip probability)**

If cheap_side is about to flip from CRIME to FRAUD:
- Old rates: CRIME buy = low (e.g. 3%), FRAUD buy = high (e.g. 12%)
- New rates will be: CRIME buy = high, FRAUD buy = low
- Alpha action: buy FRAUD at old high-tax rate? No — *sell* CRIME at old low-sell-tax rate before the sell tax on CRIME becomes the low rate on FRAUD. More specifically: the cheap token's sell tax is the *high* rate.

Actually the more precise framing:
- Cheap side: low buy tax, high sell tax
- After a flip: the *previously expensive* token becomes cheap (low buy, high sell), and vice versa

So if a flip is coming:
- **Buy the about-to-become-cheap token NOW** at the current high buy tax (1-14%)... wait, that's paying *more* now.
- **Sell the about-to-become-expensive token NOW** at its current low sell tax (1-4%), because after the flip it will have a high sell tax.

The concrete edge: if you hold the "currently cheap" token (low sell tax = e.g. 1-4%), and you know it's about to become "expensive" (sell tax becomes 11-14%), you can sell before `consume_randomness` to save 7-13 percentage points of sell tax.

**B. Exact magnitude of the new rates (bytes 1-4, each % 4)**

Knowing the new magnitude allows calculating the exact tax difference. For a 1000 SOL sell:
- Old sell tax: 3% (300 bps) → pays 30 SOL
- New sell tax (if flip + high magnitude): 14% (1400 bps) → would pay 140 SOL
- Alpha from early sell: 110 SOL on 1000 SOL position

### Attack Scenario: Rate-Flip Front-Run

```
Attacker holds 1000 SOL worth of the "cheap" token (CRIME is cheap, CRIME sell = 3%)

Slot N: trigger_epoch_transition fires
         - Attacker monitors Switchboard oracle queue

Slot N+1: Oracle reveals. Attacker reads VRF bytes:
         byte[0] = 50 (< 192 = flip → FRAUD becomes cheap)
         byte[1] = 3  → CRIME low = 400 bps
         byte[2] = 3  → CRIME high = 1400 bps (becomes new sell tax on CRIME)
         ...

         Attacker computes: "CRIME sell will go from 300 bps to 1400 bps"
         Attacker submits sell TX immediately

Slot N+~2: consume_randomness executes, CRIME sell tax becomes 1400 bps

Result: Attacker paid 300 bps (3%) on sell instead of 1400 bps (14%)
        On 1000 SOL position: 30 SOL paid vs 140 SOL counterfactual
        Alpha captured: 110 SOL (or 11% of position)
```

### Attack Scenario: Crank Delay with Amplified Window

If the attacker controls (or socially-engineers) a delay of `consume_randomness` — for example by running the crank themselves and simply not submitting `consume_randomness` for 60-100 slots — the window to trade at known-stale-but-favorable rates extends substantially.

Critically: the attacker need not delay the crank themselves. They only need to be *fast enough* to transact in the ~2-3 slot window between oracle reveal and when the honest crank calls `consume_randomness`. Given Jito bundles and MEV infrastructure on Solana, this is achievable in practice.

## Timing Window Assessment

| Scenario | Window Duration | Controllability | Exploitability |
|----------|----------------|-----------------|----------------|
| Normal crank latency | ~2-5 slots (~1-2s) | No (oracle-driven) | Low — requires pre-positioned capital |
| Crank delay (attacker is crank) | Up to 300 slots (~2 min) | Yes — attacker runs crank | Medium — attacker has ample time |
| Crank delay (no one cranks) | Up to 300 slots | Indirect (grief by not cranking) | Medium — anyone with capital benefits |
| After retry_epoch_vrf | Resets; new ~2-5 slots | No | Same as normal crank latency |

**Key observation:** The 300-slot retry window is the outer bound. If no one calls `consume_randomness` for 300 slots, `retry_epoch_vrf` allows re-requesting from a fresh oracle. But crucially, even the *retry* window can be exploited: the attacker simply executes their position during the 300-slot window and allows the retry to eventually resolve.

**Mainnet epoch duration:** 4500 slots (~30 min). The VRF-pending window represents at most 300/4500 = 6.7% of an epoch during which old rates persist.

## Economic Impact Quantification

### Per-Trade Alpha (Rate Flip Scenario)

| Token position | Old sell tax | New sell tax (worst case) | Alpha (per 1 SOL) |
|----------------|-------------|---------------------------|-------------------|
| Cheap token, max flip | 100 bps | 1400 bps | 0.013 SOL |
| Cheap token, typical flip | 300 bps | 1100 bps | 0.08 SOL |

For a 100 SOL sell: 0.8 SOL alpha in typical case, up to 13 SOL in extreme.

### Maximum Realistic Alpha Per Epoch Transition

Constraints:
- Pool reserves and slippage limit single-swap size
- 50% minimum output floor (`MINIMUM_OUTPUT_FLOOR_BPS = 5000`) limits sandwiching by others but does not limit the attacker's own position exit
- Attacker must hold a pre-positioned token balance — they cannot "create" the alpha from nothing

Realistic upper bound: A whale holding 10,000 SOL worth of the cheap token could capture ~800 SOL in alpha per epoch transition where a favorable flip is predicted (8% rate delta * 10,000 SOL). However:
- This requires pre-positioning an entire epoch in advance (holding ~10,000 SOL worth of one token)
- 75% of epochs involve a flip, but not all flips are large magnitude shifts
- The 30-minute epoch cycle means this "front-running window" is available to everyone who monitors Switchboard reveals

### Realistic Impact (Not Flash-Loan-Amplifiable)

This is NOT a flash loan attack because:
1. Position must be held across the epoch (not within a single TX)
2. Alpha is proportional to held balance, not borrowed capital
3. Swapping into a large position before the epoch boundary incurs the current tax

Realistic risk: Medium. A sophisticated actor with meaningful pre-positioned capital (100-1000 SOL equivalent) can consistently capture 1-13 SOL per favorable epoch transition by monitoring oracle reveals. Over 48 epochs/day this is recurring but bounded per event.

## Verdict

**CONFIRMED — MEDIUM severity.**

The vulnerability is real: VRF bytes are publicly visible on-chain immediately after oracle reveal, and the Tax Program applies no guard on `vrf_pending`/`taxes_confirmed` before using tax rates from EpochState. This creates a predictable ~2-5 slot window (extendable to 300 slots if the crank is withheld) during which a sophisticated actor can trade at known-stale rates.

**Why not HIGH:** The attack is bounded by pre-positioned capital (not flash-loan-amplifiable), the window is short under normal crank operation (~2s), and the maximum alpha per event is proportional to position size, not pool size. The 75% flip probability means ~25% of epoch transitions have *no* alpha (rates persist or shift to the attacker's detriment). Protocol economics are not broken; this is a recurring MEV edge, not a draining attack.

**Why not LOW:** The oracle reveal is fully public, computation is trivial, Jito bundles make 2-slot windows exploitable in practice, and the crank-delay path extends the window substantially for an operator-level attacker. The alpha is real SOL (not dust), recurring, and asymmetrically available to sophisticated actors over retail participants.

## Recommendations

**R1 (Recommended — Low Friction): Add `taxes_confirmed` check to Tax swap instructions**

In `swap_sol_buy.rs` and `swap_sol_sell.rs`, after deserializing EpochState, add:

```rust
// Prevent trading at stale rates during VRF-pending window
// taxes_confirmed = false means consume_randomness hasn't run yet this epoch
// Note: this blocks swaps for ~2-5 slots per epoch transition (~30 min)
// Users get a cleaner error instead of silently paying stale rates
require!(epoch_state.taxes_confirmed, TaxError::TaxRatesNotConfirmed);
```

**Trade-off:** This blocks all swaps for ~2-5 slots per epoch boundary (~30 minutes per epoch). On mainnet at 2.5 slots/sec this is ~1-2 seconds of unavailability per 30-minute epoch — a 0.07% downtime per epoch. Acceptable for security; should be documented in the UI.

**R2 (Alternative — No Downtime): Rate-change cooldown per user**

Instead of blocking all swaps, track `last_epoch_swapped` per user and disallow swapping until the epoch has been confirmed. This is more complex state management and trades off state size for availability.

**R3 (Process): Ensure crank is reliably deployed**

The 300-slot window (the "attacker is the crank" scenario) is the more serious vector. Ensure the Railway crank service reliably calls `consume_randomness` within 5 slots of oracle reveal. Add monitoring/alerting on crank latency. The protocol already has retry_epoch_vrf as a safety valve but should not rely on it as the primary path.

**R4 (Informational): Document the information model for users**

Tax rates are probabilistically determined at epoch boundary. Inform users in the UI that during the brief transition window (< 2 seconds normally), displayed rates are "pending update." This manages user expectations without requiring protocol changes.

**Priority:** R1 is strongly recommended as it closes the fundamental gap with minimal implementation cost. R3 is independently necessary for liveness and closes the 300-slot attacker-as-crank window.
