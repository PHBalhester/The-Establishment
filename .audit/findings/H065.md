# H065: Bonding Curve Solvency Buffer Adequacy

**Status:** CONFIRMED
**Severity:** LOW
**Category:** Arithmetic — Rounding & Dust Management

## Summary

The bonding curve's solvency buffer is set to 10 lamports (`SOLVENCY_BUFFER_LAMPORTS = 10`). This buffer is added to the rent-exempt minimum to prevent cumulative rounding dust from eroding the vault below rent-exempt. The value was derived from empirical testing (BOK Finding 1 on LiteSVM) and observes a maximum 5-lamport gap after hundreds of micro-buys/sells. A 10-lamport buffer is **adequate** and provides a 2x safety margin over the observed worst case.

## Code Evidence

**File:** `programs/bonding_curve/src/constants.rs:134-142`

```rust
/// Solvency buffer: additional lamports above rent-exempt minimum that the
/// vault must retain after any sell payout. Prevents cumulative rounding dust
/// from eroding the vault below rent-exempt.
///
/// BOK Finding 1 (MEDIUM): LiteSVM inv_bc_014b_sequential_sells found a
/// 5-lamport gap after hundreds of micro-buys/sells. Each calculate_sol_for_tokens()
/// floors the result; cumulative effect can erode vault by ~1 lamport per sell.
/// 10 lamports provides comfortable margin (2x worst observed case).
pub const SOLVENCY_BUFFER_LAMPORTS: u64 = 10;
```

**Rounding Analysis:** `programs/bonding_curve/src/context/02-arithmetic.md` (context audit) documents:

From `.audit/context/02-arithmetic.md` (grep result line 184):
> "Purpose: Compute 15% sell-back tax with ceil rounding (protocol-favored)."

And from arithmetic context (line 155):
> "2. `split_distribution(total_tax)`: Floor(71%), floor(24%), remainder for treasury. Micro-tax (<4 lamports) all to staking."

Each `calculate_sol_for_tokens()` call (bonding curve sell operation) floors the result, which means the vault receives 1 lamport less than the theoretical value. Over hundreds of operations, this accumulates.

**Usage Context:** The solvency buffer is maintained in the sell flow to ensure the vault never drops below rent-exempt + buffer. This is validated in code that checks remaining lamports before approving a sell.

## Verdict

**CONFIRMED ADEQUATE.** The 10-lamport buffer is empirically validated and provides sufficient margin:

1. **Root Cause Identified:** Floor rounding in `calculate_sol_for_tokens()` costs ~1 lamport per sell operation
2. **Worst Case Measured:** 5 lamports lost over 100+ sequential micro-sells in BOK Finding 1 (LiteSVM test `inv_bc_014b_sequential_sells`)
3. **Buffer Safety:** 10 lamports = 2x worst observed loss
4. **Rent-Exempt Minimum:** ~890,880 lamports for a 0-byte account (standard Solana)
5. **Final State:** After distribution, escrow PDA retains rent-exempt minimum (line 81 of distribute_tax_escrow.rs), plus the 10-lamport buffer as permanent dust

**Rounding Consistency Note:** All rounding decisions in the protocol favor the protocol (floor for user outputs, ceil for user inputs). This is correct—dust accumulates with the protocol, not with users, preventing extraction attacks.

**No Vulnerability.** The buffer is appropriately sized and well-documented.
