# S009: PoolState Layout Drift Between Tax and Epoch Readers

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** CPI, Arithmetic, Upgrade Coupling
**Inspired by:** H011

## Summary

Two independent programs (Tax Program and Epoch Program) contain separate, nearly-identical raw-byte reader functions for AMM PoolState reserves, both hardcoding offsets `[137..145]` (reserve_a) and `[145..153]` (reserve_b). H011 confirmed these offsets are currently correct and flagged the fragility as POTENTIAL. This finding investigates whether the two readers are byte-for-byte consistent with each other and whether they could drift apart independently — specifically focusing on the structural gap that an upgrade or fix to one reader would not automatically propagate to the other.

## Analysis

### Two Independent Implementations, Same Offsets

The two reader functions are:

**Tax Program:** `programs/tax-program/src/helpers/pool_reader.rs:57-97`
**Epoch Program:** `programs/epoch-program/src/helpers/carnage_execution.rs:825-851`

Direct comparison:

| Attribute | Tax Program `pool_reader.rs` | Epoch Program `carnage_execution.rs` |
|---|---|---|
| Function signature | `read_pool_reserves(pool_info: &AccountInfo)` | `read_pool_reserves(pool_info: &AccountInfo, wsol_mint_key: &Pubkey)` |
| Length check | `data.len() >= 153` | `data.len() >= 153` |
| Reserve A offset | `[137..145]` | `[137..145]` |
| Reserve B offset | `[145..153]` | `[145..153]` |
| mint_a read offset | `[9..41]` | `[9..41]` |
| Owner check | Yes — verifies `pool_info.owner == amm_program_id()` | No — relies on Anchor struct constraints |
| is_reversed logic | Compares `mint_a == native_mint()` | Compares `pool_mint_a == *wsol_mint_key` |
| Error types | `TaxError::*` | `EpochError::*` |

**Key finding: The raw byte offsets are identical in both implementations.** There is no current divergence in the data they extract. However, two structural differences create independent drift risk:

### Structural Difference 1: Owner Validation Asymmetry

The Tax Program's reader includes an explicit owner check:
```rust
// pool_reader.rs:61-64
require!(
    *pool_info.owner == amm_program_id(),
    TaxError::InvalidPoolOwner
);
```

The Epoch Program's reader does NOT perform this check — it trusts that Anchor struct constraints enforce pool ownership upstream. This was flagged in H011 as a defense-in-depth gap.

From a layout-drift perspective, this asymmetry means:
- If an upgrade patches the Tax Program reader to add a struct-hash version check, the Epoch Program reader would not benefit.
- If the Epoch Program reader is patched to add owner validation (as a defense-in-depth improvement), the Tax Program reader's existing validation would be a redundant but compatible divergence.

The risk is not symmetric: a security patch to the Tax Program reader that tightens validation could inadvertently reveal that the Epoch Program reader is more permissive. A developer patching one would need to manually remember to patch both.

### Structural Difference 2: WSOL Reference Parameter

The Tax Program resolves WSOL position via a locally-computed constant:
```rust
// pool_reader.rs:29-32 (native_mint())
fn native_mint() -> Pubkey {
    Pubkey::from_str("So11111111111111111111111111111111111111112").unwrap()
}
// ...
if mint_a == native_mint() {
    Ok((reserve_a, reserve_b))
} else {
    Ok((reserve_b, reserve_a))
}
```

The Epoch Program passes the WSOL key as a parameter:
```rust
// carnage_execution.rs:825-828
pub fn read_pool_reserves(
    pool_info: &AccountInfo,
    wsol_mint_key: &Pubkey,    // <-- parameter
) -> Result<(u64, u64)> {
// ...
if pool_mint_a == *wsol_mint_key {
    Ok((reserve_a, reserve_b))
} else {
    Ok((reserve_b, reserve_a))
}
```

Currently these are equivalent (the caller passes the NATIVE_MINT address as `wsol_mint_key`). But they represent different maintenance patterns:

- The Tax Program's reader is self-contained: WSOL identity is hardcoded.
- The Epoch Program's reader is parameterized: the caller's passed value determines ordering.

If a future WSOL migration or test environment substitution causes `wsol_mint_key` to differ from `native_mint()`, the two readers would compute different `(sol_reserve, token_reserve)` orderings for the same pool. One would return `(sol, token)` and the other `(token, sol)`.

### Drift Scenario 1: AMM Upgrade Breaks Both (Same Way)

If an AMM upgrade shifts PoolState field ordering (H011 core risk), both readers would break identically because they use the same offsets. This is the H011 scenario. The two readers are consistent with each other; the risk is that they are both consistently wrong after an upgrade.

### Drift Scenario 2: One Reader Is Patched, Not the Other

A developer fixing a bug in one reader (e.g., adding a version check, adjusting an offset for a new struct field, fixing the is_reversed logic) has no compile-time or test-time mechanism that forces them to update the other reader. The failure mode:

- Tax Program's reader is patched for a new PoolState layout
- Epoch Program's reader is missed
- Tax Program computes correct slippage floors; Epoch Program computes garbage slippage floors for Carnage
- Carnage swaps either always pass (floor = 0 from garbage data) or always fail (floor = u64::MAX)

The comment at `pool_reader.rs:6-8` partially acknowledges this:
```
//! Identical approach to epoch-program/src/instructions/execute_carnage_atomic.rs
//! (lines 930-956), adapted for Tax Program error types.
```

This comment was accurate when written but represents aspirational documentation rather than enforced coupling. The "identical approach" can diverge without any tooling catching it.

### Drift Scenario 3: WSOL Parameter vs. Constant Divergence

If the Epoch Program caller ever passes a non-NATIVE_MINT key as `wsol_mint_key` (e.g., in a devnet test, or during a WSOL2 migration), and the Tax Program's reader uses its hardcoded `native_mint()`, the two readers would assign different pool ordering for the same pool data. This could produce:
- Tax Program: reads (SOL, token) correctly
- Epoch Program: reads (token, SOL) reversed — swap amount and slippage computations receive swapped inputs

For a pool with SOL reserve = 100 SOL and CRIME reserve = 1B tokens:
- Correct: `(100 SOL, 1B tokens)` → slippage computes expected output in tokens
- Reversed: `(1B tokens, 100 SOL)` → expected output computed as if buying 100 base units with 1B units of input

The constant-product formula with reversed inputs would compute a nonsensical result, causing Carnage slippage to either always pass (expected = tiny number) or always fail.

### No Cross-Reader Tests

H011 noted that cross-crate tests exist for EpochState but NOT for PoolState. This applies equally to the two independent readers: there is no test that instantiates both readers with the same input data and asserts they return identical results. Such a test would catch both structural divergence and offset divergence.

## Verdict

**CONFIRMED — MEDIUM severity.**

The two readers are currently consistent with each other at the byte-offset level. No active vulnerability exists. The risk is structural:

1. No enforcement mechanism prevents them from drifting independently.
2. No cross-reader test validates they return identical results for the same input.
3. The cross-reference comment in `pool_reader.rs` already contains a stale file reference (`execute_carnage_atomic.rs` vs. `carnage_execution.rs`), demonstrating that documentation coupling without code coupling degrades naturally.
4. An upgrade that patches one reader for a PoolState layout change would not automatically update the other, leading to different slippage computations in Tax Program vs. Epoch Program for the same pools.

Severity is MEDIUM (not HIGH) because:
- Both readers are currently correct.
- The scenario requires an upgrade event to trigger.
- The impact, while HIGH if triggered, is future-facing and upgrade-dependent (same as H011).

This finding is the maintenance-coupling analogue of H011: H011 covers "what happens when AMM changes break both readers"; S009 covers "what happens when a fix to one reader is not applied to the other."

## Recommendations

### R1 — Consolidate into a Single Shared Utility (Recommended)

Extract `read_pool_reserves` into a shared internal crate (e.g., `programs/common/src/pool_reader.rs`) that both Tax Program and Epoch Program import. This makes it structurally impossible for the two readers to drift: there is only one implementation.

The shared crate should:
- Accept `amm_program_id` as a parameter (or via a feature flag for devnet/mainnet)
- Use the parameterized `wsol_mint_key` approach (more flexible than hardcoded `native_mint()`)
- Include the owner check from the Tax Program version

### R2 — Add a Cross-Reader Consistency Test

Add a test to `tests/cross-crate/src/lib.rs` that:
1. Creates a mock PoolState byte array with known reserve values
2. Passes it through both reader implementations
3. Asserts they return identical `(sol_reserve, token_reserve)` pairs

This test fails if either offset or ordering logic diverges between the two readers.

### R3 — Fix the Stale Cross-Reference Comment

Update `pool_reader.rs:6-8`:
```
// BEFORE:
//! Identical approach to epoch-program/src/instructions/execute_carnage_atomic.rs
//! (lines 930-956), adapted for Tax Program error types.

// AFTER:
//! Identical approach to epoch-program/src/helpers/carnage_execution.rs
//! (read_pool_reserves, lines 825-851), adapted for Tax Program error types.
//! IMPORTANT: If this function is updated, the Epoch Program counterpart MUST
//! be updated identically. See S009 audit finding.
```

This is a documentation fix, not a code fix. It does not prevent drift but reduces the probability of accidental divergence.

### R4 — Add DATA_LEN Assertion for PoolState (from H011 R2)

H011 R2 recommends adding `PoolState::INIT_SPACE == 216` assertion to the AMM program so that adding a field causes a compile error. If this is implemented, the compile failure would alert the developer to update both readers. This is a complementary fix to H011 that also partially mitigates S009.
