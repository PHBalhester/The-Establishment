# H033: Sell-Then-Buy Carnage Compound Slippage

| Field | Value |
|-------|-------|
| **Status** | Confirmed — Incorrectly Scoped (Theoretical / Not Exploitable as Described) |
| **Severity** | Informational |
| **Category** | MEV / Ordering |
| **Location** | `programs/epoch-program/src/helpers/carnage_execution.rs:134-399` |
| **Functions** | `execute_carnage_core`, `execute_sell_swap`, `execute_buy_swap` |

---

## Summary

The original finding hypothesises that when Carnage performs a `Sell` action (dispose held tokens) followed immediately by a buy of the target token, an attacker could insert a transaction between the sell and the buy to profit from the price impact of the sell. After code review, this scenario is **not possible**: the sell and buy steps both execute within a single call to `execute_carnage_core`, which is a single Solana instruction. Solana's execution model makes interleaving external transactions within a single instruction impossible. The compound slippage effect is real and acknowledged by the protocol, but it is addressed by the slippage floor mechanism and constitutes an intentional design trade-off rather than an exploitable vulnerability.

---

## Analysis

### Atomicity of Sell + Buy

`execute_carnage_core` is called by both `execute_carnage_atomic` (handler in `execute_carnage_atomic.rs`) and `execute_carnage` fallback (handler in `execute_carnage.rs`). The entire dispose-then-buy flow is a **linear sequence of CPI calls within one instruction**:

1. **Step 1 (`CarnageAction::Sell`)** — `execute_sell_swap` → CPI to `Tax::swap_exempt` → AMM → Token-2022 → Hook. WSOL proceeds land in `carnage_wsol`.
2. **Step 1.5** — `accounts.crime_vault.reload()` / `accounts.fraud_vault.reload()` to get fresh on-chain balances.
3. **Step 2** — `wrap_sol_to_wsol` wraps any remaining SOL, then `execute_buy_swap` → CPI to `Tax::swap_exempt` → AMM → Token-2022 → Hook.

All steps execute sequentially inside a single Solana instruction. No other transaction can be inserted between them because Solana processes one instruction fully before scheduling another. An external attacker **cannot** place a transaction between `execute_sell_swap` and `execute_buy_swap`.

### The Compound Price Impact is Real But Bounded

The finding identifies a genuine economic reality: a sell CPI depresses the pool price, and the immediately following buy CPI operates at that lower reserve ratio, receiving fewer tokens than it would against undisturbed reserves. This is the compound slippage effect. However:

- The slippage floor (`CARNAGE_SLIPPAGE_BPS_ATOMIC = 8500` / `CARNAGE_SLIPPAGE_BPS_FALLBACK = 7500`) is calculated against the **pre-sell pool reserves** (read at line 285 in `carnage_execution.rs`, before any CPI):
  ```
  let (reserve_sol, reserve_token) = read_pool_reserves(target_pool_info, &accounts.mint_a.key())?;
  ```
  This reserve snapshot is taken before `execute_sell_swap` modifies the pool, so the slippage check anchors the expected output to the **pre-sell state**. The buy must deliver at least 85% (atomic) or 75% (fallback) of what the pre-sell reserves would have yielded.
- The `Sell` action has a probability of approximately 2% (VRF byte 6 < 5 out of 256) conditional on Carnage triggering (~4.3%), giving a combined probability of ~0.09% per epoch. The held tokens being sold are from a prior epoch's buy, so their size is capped at `MAX_CARNAGE_SWAP_LAMPORTS = 1000 SOL` worth of tokens acquired at historical prices.

### Fallback Path Distinction

The finding is more relevant to the **fallback path** than the atomic path. On the fallback path (after the 50-slot lock window), carnage_target and carnage_action are visible on-chain. An attacker could observe a pending `Sell+Buy` Carnage and attempt to front-run `execute_carnage` itself — but this is the already-documented fallback MEV window (H-level finding from the MEV/Carnage context), not a novel compound slippage issue. On the **atomic path**, `execute_carnage_atomic` is bundled in the same transaction as `consume_randomness`, so the VRF outcome is unknown until that transaction is fully processed. Front-running the entire atomic transaction is infeasible.

### Why Slippage Floor Anchors to Pre-Sell Reserves

The reserve snapshot is taken at `carnage_execution.rs:285-288`, before `execute_sell_swap` at line 220 modifies the pool. After the sell CPI completes, the pool reserves have changed (SOL increases, tokens decrease). The subsequent buy uses the same `reserve_sol`/`reserve_token` variables — which reflect the pre-sell state — to compute `expected` tokens. The check at lines 331-349 then enforces `bought >= min_output` where `min_output = expected * slippage_bps / 10_000`. Because the sell itself pushes the pool away from the attacker-manipulated price, the bought amount for a reasonably-sized sell will still pass this check under normal conditions.

However, for a large sell (clearing a significant held position), the post-sell reserves will be materially different from the pre-sell reserves used to compute `expected`. If the sell is large relative to pool depth, `expected` could significantly overestimate what the buy will actually receive (because it was computed on pre-sell reserves), causing the slippage check to **pass even though effective compound slippage was very high**. This is not a security bug per se — the protocol intended to sell first and buy with proceeds — but it does mean the slippage floor is less protective on the compound Sell+Buy path than on a pure Buy path.

## Code Evidence

**Atomicity — single function, sequential CPIs:**
```rust
// carnage_execution.rs:177-245 (sell step)
CarnageAction::Sell => {
    if held_amount > 0 {
        execute_sell_swap(accounts, held_amount, held_token, ...)?;
        accounts.carnage_wsol.reload()?;
        ...
    }
}

// carnage_execution.rs:258-354 (buy step, same function)
let tokens_bought = if total_buy_amount > 0 {
    let (reserve_sol, reserve_token) = read_pool_reserves(target_pool_info, ...)?;  // pre-sell snapshot
    ...
    execute_buy_swap(accounts, total_buy_amount, target, ...)?;
    ...
};
```

**No external transaction can interleave these steps** because they are sequential operations within `execute_carnage_core`, which is called once per instruction invocation.

**Slippage floor reads pre-sell reserves:**
```rust
// carnage_execution.rs:285-288 — BEFORE execute_sell_swap
let (reserve_sol, reserve_token) = read_pool_reserves(target_pool_info, &accounts.mint_a.key())?;
```

**Slippage check (lines 331-349):**
```rust
let expected = (reserve_token as u128)
    .checked_mul(total_buy_amount as u128)
    .and_then(|n| n.checked_div((reserve_sol as u128).checked_add(total_buy_amount as u128)?))?;
let min_output = (expected as u128).checked_mul(slippage_bps as u128).and_then(|n| n.checked_div(10_000))?;
require!(bought >= min_output, EpochError::CarnageSlippageExceeded);
```

## Verdict

**Not exploitable as described.** The sell-then-buy sequence is atomic within a single Solana instruction. No attacker can insert a transaction between the sell CPI and the buy CPI.

A secondary observation is confirmed: the slippage floor on the compound Sell+Buy path is anchored to the **pre-sell pool state**, so after a large sell moves the pool, the `expected` figure slightly overstates the ideal buy output — meaning the floor offers marginally less protection on this path than it appears. However, this is bounded by: (a) the `Sell` action only occurring ~0.09% of epochs, (b) held token amounts being capped at prior-epoch buy amounts, and (c) the pool's k-invariant ensuring any sell price impact is symmetric and proportional to the sell size relative to pool depth. The gap between pre-sell `expected` and post-sell actual output is not an exploitable attack surface; it is a design property the team should be aware of.

The pre-existing fallback MEV finding (25% extraction window after the lock window) covers any front-running concern on the fallback path. That is a separate issue and not duplicated here.

---

## Recommendations

1. **No immediate action required.** The core claim (front-run between sell and buy) is not possible due to Solana's execution model. Mark this as Informational.

2. **Consider documenting compound slippage behaviour.** Add a code comment at the `read_pool_reserves` call in the Sell path noting that the slippage floor is computed against pre-sell reserves, and that on a large Sell+Buy path the actual expected output post-sell will be lower. This is correct behaviour, but should be explicit.

3. **For future tuning:** If the Carnage Fund ever accumulates very large held positions (approaching single-epoch pool depth), the compound slippage gap will widen. A post-sell reserve re-read (to compute expected from the actual post-sell state) would give a tighter, more accurate floor. This is a potential future hardening step if the Sell action becomes more frequent or larger in scale.
