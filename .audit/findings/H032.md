---

## Summary

The original concern was that the Tax Program reads `tax_bps` values from EpochState without performing an explicit in-Tax-Program range check (i.e., no `require!(tax_bps <= 10_000, ...)` immediately after calling `get_tax_bps()`). Investigation reveals that the threat is neutralized by two independent, complementary defenses: (1) the `calculate_tax()` helper explicitly rejects any `tax_bps > 10_000` by returning `None` (which maps to a hard error), and (2) the Epoch Program's `derive_taxes()` function can only produce values from two closed constant sets — `{100, 200, 300, 400}` and `{1100, 1200, 1300, 1400}` — making any out-of-range value structurally impossible under normal program execution. A 100% or >100% tax rate cannot arise.

---

## Analysis

### Threat Model Recap

The hypothesis: a bug in the Epoch Program (or a future upgrade) writes a `tax_bps` value greater than `10_000` (i.e., >100%) into EpochState. When the Tax Program reads that value and applies it, the entire user input could be consumed as tax, and `sol_to_swap` or `net_output` could underflow, resulting in fund loss or a protocol halt.

### Defense Layer 1 — `calculate_tax()` Range Check

`programs/tax-program/src/helpers/tax_math.rs` lines 34–38:

```rust
pub fn calculate_tax(amount_lamports: u64, tax_bps: u16) -> Option<u64> {
    // Validate bps range: 0-10000 is valid (0% to 100%)
    if tax_bps > 10_000 {
        return None;
    }
    // ...
}
```

The call site in both `swap_sol_buy.rs` (line 83–84) and `swap_sol_sell.rs` converts `None` into a hard `TaxError::TaxOverflow` via `.ok_or(error!(TaxError::TaxOverflow))?`. This means any `tax_bps > 10_000` causes the entire swap instruction to abort before any SOL moves. The user's funds are never touched.

This is a **complete, unconditional guard** against the out-of-range path. Even if EpochState were somehow corrupted with `tax_bps = 50_000`, the Tax Program would simply refuse to process any swap until the Epoch Program writes a valid rate.

### Defense Layer 2 — Epoch Program Derives Only Discrete Legal Values

`programs/epoch-program/src/helpers/tax_derivation.rs` lines 23–26:

```rust
const LOW_RATES:  [u16; 4] = [100, 200, 300, 400];
const HIGH_RATES: [u16; 4] = [1100, 1200, 1300, 1400];
```

`derive_taxes()` selects exclusively from these two constant arrays using `% 4` indexing (lines 89–94). The function cannot produce any value outside `{100, 200, 300, 400, 1100, 1200, 1300, 1400}`. This set is entirely within `[0, 10_000]`. The absolute maximum tax rate is `1_400` BPS = 14%. An out-of-range value could only appear in EpochState if:

- An Epoch Program upgrade intentionally introduced one, **or**
- The EpochState account data were directly written by a different program (which the Tax Program's owner check prevents).

### Defense Layer 3 — EpochState Owner + Discriminator Validation

`swap_sol_buy.rs` lines 59–72 and `swap_sol_sell.rs` lines 74–87 both perform:
1. `epoch_state.owner == &epoch_program` — rejects any account not owned by the Epoch Program.
2. `EpochState::try_deserialize(...)` — validates Anchor's 8-byte discriminator (sha256("account:EpochState")[0..8]); rejects accounts with wrong discriminator.
3. `require!(epoch_state.initialized, ...)` — rejects a default-zeroed account.

Together these three checks ensure the account is a genuine, Anchor-initialized EpochState belonging to the deployed Epoch Program. An attacker cannot substitute a fake account.

### Could a Legit Epoch Program Bug Produce Out-of-Range Values?

`consume_randomness.rs` writes to EpochState at lines 199–203:

```rust
epoch_state.crime_buy_tax_bps  = tax_config.crime_buy_tax_bps;
epoch_state.crime_sell_tax_bps = tax_config.crime_sell_tax_bps;
epoch_state.fraud_buy_tax_bps  = tax_config.fraud_buy_tax_bps;
epoch_state.fraud_sell_tax_bps = tax_config.fraud_sell_tax_bps;
```

These fields come directly from `derive_taxes()`, which is constrained to the constant sets above. There is no arithmetic post-processing between `derive_taxes()` and the EpochState write. No code path in the Epoch Program produces a tax rate outside `{100..1400}` BPS.

The `low_tax_bps` / `high_tax_bps` summary fields (lines 209–216) are computed as `min`/`max` of the four per-token rates — also bounded to the same set.

### Theoretical Upgrade Scenario

If the Epoch Program were upgraded by an attacker who held the upgrade authority, they could write arbitrary values. However:
- This is in scope for the upgrade authority threat model, not for a Tax Program bug.
- The Tax Program's `calculate_tax()` guard would still reject `tax_bps > 10_000`, capping damage at a protocol freeze rather than fund loss.
- Upgrade authority is gated behind Squads 2-of-3 multisig with timelock (Docs/mainnet-governance.md).

### Edge Case: `tax_bps = 10_000` (100%)

`calculate_tax()` explicitly allows `tax_bps == 10_000` (test `tax_max_valid_bps` confirms this). In the buy path this would result in `sol_to_swap = 0`, which is caught by `require!(sol_to_swap > 0, TaxError::InsufficientInput)` at `swap_sol_buy.rs:94`. In the sell path, the guard `require!(tax_amount < gross_output, TaxError::InsufficientOutput)` catches this. Neither case results in fund loss.

## Verdict

**Not Exploitable.** The concern is neutralized by defense-in-depth:

1. **Epoch Program cannot produce out-of-range values** — `derive_taxes()` draws exclusively from two closed constant arrays. The maximum producible tax rate is 1,400 BPS (14%).
2. **Tax Program unconditionally validates the range** — `calculate_tax()` returns `None` (→ hard abort) for any `tax_bps > 10_000`, preventing fund loss even in a hypothetical Epoch upgrade scenario.
3. **Cross-program account spoofing is blocked** — owner check + discriminator validation + initialized flag make fake EpochState injection impossible.

The original filing was based on the absence of an *explicit* `require!(tax_bps <= 10_000)` between `get_tax_bps()` and `calculate_tax()`. That check is present, just one function call deep inside `calculate_tax()`. The layered approach is actually more robust because the guard travels with the computation, not with a single call site that could be bypassed if a new call site were added.

---

## Recommendations

No security action required. The following are minor code-quality suggestions for defense-in-depth documentation:

1. **Optional: Add a doc comment** to `get_tax_bps()` in `epoch_state_reader.rs` noting that the returned value is always `<= 1_400` BPS due to Epoch Program derivation constraints. This makes the trust relationship explicit at the read site.

2. **Optional: Add an assertion in `derive_taxes()`** unit tests that the maximum possible rate across all 256^5 input combinations cannot exceed `1_400` BPS. The existing tests verify specific values but no test asserts the global upper bound explicitly.

3. **Consider** adding `const MAX_TAX_BPS: u16 = 1_400` to the Epoch Program's constants and asserting against it in `derive_taxes()` for documentation purposes — not for security, since the array indexing is already a complete bound.

None of these are required for mainnet safety.
