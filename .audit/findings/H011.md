# H011: Cross-Program Byte-Offset Corruption After AMM Upgrade

**Status:** POTENTIAL
**Severity:** HIGH
**Category:** CPI, Arithmetic
**Origin:** Novel

## Summary

Two independent programs (Tax Program and Epoch Program) read AMM PoolState reserve values via hardcoded byte offsets `[137..145]` and `[145..153]` with no version check, struct hash, or compile-time enforcement of layout stability. The offsets are **correct for the current PoolState layout**, but any AMM upgrade that changes field ordering or inserts fields before the reserves would silently feed garbage values into slippage floor calculations and tax enforcement, with no on-chain error.

## Analysis

### Offset Verification

The current PoolState struct in `programs/amm/src/state/pool.rs` serializes (Anchor/Borsh) as follows:

| Bytes      | Field          | Size  |
|------------|----------------|-------|
| `[0..8]`   | discriminator  | 8     |
| `[8]`      | pool_type      | 1     |
| `[9..41]`  | mint_a         | 32    |
| `[41..73]` | mint_b         | 32    |
| `[73..105]`| vault_a        | 32    |
| `[105..137]`| vault_b       | 32    |
| `[137..145]`| **reserve_a** | **8** |
| `[145..153]`| **reserve_b** | **8** |

This is independently confirmed by:
1. The struct definition's own space comment (`= 8 + 216 = 224 bytes total`)
2. The AMM's own `read_pool_state` test helper in `programs/amm/tests/test_cpi_access_control.rs:639-670`, which reads fields at the same relative offsets using a running `offset` counter
3. The `pool_reader.rs` file comment, which documents this exact layout

**The offsets are currently correct.** The vulnerability is not present in the current deployed state.

### What Makes This a POTENTIAL Risk

The fragility lies in two structural facts:

1. **No version guard exists.** PoolState contains no version field, magic number, or schema hash. Neither `pool_reader.rs` nor `carnage_execution.rs:read_pool_reserves` performs any check beyond `data.len() >= 153`. An upgraded PoolState that is, say, 180 bytes (due to a new field inserted before reserves) would pass the length check while returning completely wrong reserve values.

2. **No cross-crate compile-time enforcement.** The cross-crate test suite at `tests/cross-crate/src/lib.rs` validates EpochState serialization round-trips between the epoch-program and tax-program crates, but there is **no equivalent test for PoolState**. Tax Program and Epoch Program do not import the AMM crate — the offsets are just integer literals in comments and code. A developer changing PoolState would get no compile error and no test failure warning them that two other programs depend on the exact byte layout.

3. **Two independent read sites.** Both `tax-program/src/helpers/pool_reader.rs:79-88` and `epoch-program/src/helpers/carnage_execution.rs:835-842` contain duplicate hardcoded offsets. A fix to one would not automatically fix the other.

### Impact If Triggered

If the offsets become stale after an AMM upgrade:

- **Tax Program (`pool_reader.rs`)**: `read_pool_reserves` feeds the slippage floor in `swap_sol_buy` and `swap_sol_sell`. With garbage reserve values, the 50% output floor could be computed against nonsensical numbers. If `reserve_a` reads as 0 (e.g., because it now overlaps the `lp_fee_bps` u16 field), the constant-product formula returns 0 as the expected output, the 50% floor becomes 0, and the user's `minimum_output` parameter is the only remaining protection. An attacker who sets `minimum_output=0` could receive nothing from a swap and the program would not revert.

- **Epoch Program (`carnage_execution.rs`)**: `read_pool_reserves` feeds the Carnage slippage floor (75-85%). Same corruption scenario: a garbage reserve value produces a garbage expected output, and the slippage floor enforcement either passes trivially (floor=0) or reverts all Carnage swaps (floor=u64::MAX).

- **Scenario severity**: The upgrade authority is currently the deployer wallet, moving to a Squads 2-of-3 multisig. This means **the same entity that could cause the break also controls the fix** — a governance-aware attacker who captures the multisig could deliberately introduce a layout-breaking AMM upgrade to suppress the slippage floor, then sandwich all subsequent swaps.

### Realistic Probability

The attack requires compromising the AMM upgrade authority (Squads multisig after Phase 97). For an internal deployment mistake (innocent layout change), the probability is low but non-zero — Rust's type system does not protect Borsh serialization order, and a developer adding a field to PoolState for a future USDC pair feature (noted in project memory) could accidentally shift the reserve offsets without realizing the downstream impact.

## Code Evidence

- `programs/tax-program/src/helpers/pool_reader.rs:79-88` — Hardcoded offsets `[137..145]`, `[145..153]`
- `programs/epoch-program/src/helpers/carnage_execution.rs:835-842` — Same hardcoded offsets, independent copy
- `programs/amm/src/state/pool.rs:42-79` — Actual PoolState struct definition (no version field)
- `tests/cross-crate/src/lib.rs:1-167` — Cross-crate tests covering EpochState only; no PoolState layout test
- `programs/amm/tests/test_cpi_access_control.rs:639-670` — Third independent reader (test helper), confirms current offsets are correct

## Risk Assessment

**Who controls AMM upgrades?** Currently the deployer wallet; transitioning to Squads 2-of-3 multisig per Phase 97. Post-multisig, a unilateral upgrade requires 2 of 3 signers.

**Is this a realistic scenario?** As an external attack, low probability — requires multisig compromise. As an accidental internal break, moderate probability — particularly if the planned USDC pool pairs feature (see project memory) adds new fields to PoolState. There is no tooling or test that would catch the offset drift.

**Precedent:** The project memory already documents a `is_reversed` detection bug (`Canonical Mint Ordering Phase 52.1`) caused by exactly this kind of assumption about pool field layout being stable across deployments. That incident required a hotfix.

## Verdict

This is a **POTENTIAL** vulnerability, not an active one. Current offsets are verified correct. The risk is entirely future-facing: an AMM upgrade (intentional or accidental) that shifts PoolState field ordering would silently corrupt the slippage floor calculations in both Tax Program and Epoch Program, with no on-chain error and no compile-time warning. The 50% output floor and 75-85% Carnage floor are two of the protocol's most critical user-protection mechanisms; their silent bypass would be HIGH severity. Severity rating is HIGH based on impact if triggered, discounted to POTENTIAL given that it requires an upgrade event.

## Recommendations

1. **Add a PoolState layout test to the cross-crate suite** (`tests/cross-crate/src/lib.rs`). Instantiate a PoolState, serialize it, and assert `bytes[137..145]` equals the known `reserve_a` value and `bytes[145..153]` equals `reserve_a`. This would catch any future layout drift at compile/test time. This is the minimal-cost fix.

2. **Add a compile-time `DATA_LEN` assertion to PoolState** mirroring the pattern used for EpochState. Assert `PoolState::INIT_SPACE == 216` so that adding a field triggers a compile error that alerts the developer to update all offset readers.

3. **Add a struct-hash or magic constant** to the first bytes of PoolState data (after the discriminator), increment it on any breaking layout change, and have `pool_reader.rs` and `carnage_execution.rs` verify it before reading. This is a belt-and-suspenders measure for runtime safety.

4. **Consolidate the two identical `read_pool_reserves` functions** into a shared utility crate (or a single canonical location) to eliminate the duplicated fragility. Currently a fix at one site does not propagate to the other.
