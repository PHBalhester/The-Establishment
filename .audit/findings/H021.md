# H021: Cross-Program Upgrade Cascade
**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** CPI, Upgrade/Admin
**Origin:** RECHECK (H049)

## Summary

The protocol's 7 programs are tightly coupled via 15+ hardcoded cross-program ID constants, 8 shared PDA seed constants, 3 hardcoded Anchor instruction discriminators, and 2 programs that byte-read AMM PoolState at fixed offsets without version negotiation. A standard Solana in-place upgrade (same program ID, new binary) that changes any of these shared interfaces — struct field layout, instruction name, instruction account list — will silently break all dependent programs until they too are upgraded. There is no on-chain version check, no atomic multi-program upgrade path, and no emergency pause to halt traffic during the inconsistency window.

H049 (prior audit sos-002-20260307-f891646) confirmed this as MEDIUM. This recheck updates the analysis with current source state.

This test **would fail** if run and indicates that `sync-program-ids.ts` updated the function body for devnet but the test assertion was not regenerated.

### 3. Hardcoded Anchor Discriminators (3 coupling points)

Three cross-program CPI calls construct instruction data manually using hardcoded 8-byte sha256 discriminators:

| Discriminator | Value | Caller | Target Instruction |
|--------------|-------|--------|--------------------|
| `DEPOSIT_REWARDS_DISCRIMINATOR` | `[52, 249, 112, 72, 206, 161, 196, 1]` | Tax Program | `staking::deposit_rewards` |
| `UPDATE_CUMULATIVE_DISCRIMINATOR` | `[0x93, 0x84, 0xdb, 0x65, 0xa5, 0x17, 0x3d, 0x71]` | Epoch Program | `staking::update_cumulative` |
| `SWAP_EXEMPT_DISCRIMINATOR` | `[0xf4, 0x5f, 0x5a, 0x24, 0x99, 0xa0, 0x37, 0x0c]` | Epoch Program | `tax::swap_exempt` |

These are sha256(`"global:<instruction_name>"`)[0..8]. If the target instruction is **renamed** during an upgrade, the discriminator becomes invalid. On Solana, a CPI call with a wrong discriminator does not produce a clear error — Anchor will return an `InstructionMissing` or silently route to the wrong instruction if another instruction happens to share bytes. The discriminators are validated by unit tests in each `constants.rs`, but those tests only verify the hash computation is correct — they do not cross-check that the target program's instruction is still named the same.

Current tests confirm correctness for the present source state:
- `tax-program/src/constants.rs:208-222` (`test_deposit_rewards_discriminator`)
- `staking/src/constants.rs:185-200` (`test_deposit_rewards_discriminator`)
- `epoch-program/src/constants.rs:183-199` (`test_update_cumulative_discriminator`)
- `epoch-program/src/constants.rs:249-262` (`test_swap_exempt_discriminator`)

### 4. Struct Layout Coupling — Raw Byte Reads

Two programs read AMM `PoolState` reserves at compile-time fixed byte offsets without any version marker or struct hash. This is the highest-impact interface coupling because it can corrupt **all swap calculations** silently:

- `tax-program/src/helpers/pool_reader.rs:79-88` — reads `[137..145]` (reserve_a) and `[145..153]` (reserve_b)
- `epoch-program/src/helpers/carnage_execution.rs:825-851` — reads same offsets for Carnage slippage checks

Documented layout in `pool_reader.rs:11-18`:
```
[0..8]     discriminator
[8]        pool_type (1 byte)
[9..41]    mint_a (Pubkey)
[41..73]   mint_b (Pubkey)
[73..105]  vault_a (Pubkey)
[105..137] vault_b (Pubkey)
[137..145] reserve_a (u64)   ← hardcoded
[145..153] reserve_b (u64)   ← hardcoded
```

An AMM upgrade that inserts a new field before `reserve_a` (e.g., a `last_updated_slot: u64` after `vault_b`) would shift offsets by 8 bytes. `pool_reader.rs` would then read vault_b bytes 129–137 as `reserve_a` and bytes 137–145 (the actual `reserve_a`) as `reserve_b`. Tax program's slippage floor calculations would use wrong numbers; Carnage slippage checks would use wrong numbers. Both errors are **silent** — no panic, no error code.

The EpochState cross-program deserialization (Tax Program reads Epoch's `EpochState` via `try_deserialize`) is **better defended** — Anchor verifies the discriminator and the struct layout must match exactly. A layout mismatch at this boundary would produce an explicit deserialization error rather than silent corruption. An offset-backed byte test at `epoch-program/src/constants.rs:345-461` pins the EpochState serialized layout with field-level assertions.

### 5. In-Place Upgrade Inconsistency Window

The most realistic upgrade scenario is in-place (same program ID, new binary via BPFLoaderUpgradeable). This avoids the full ID cascade but creates a struct-mismatch window:

1. AMM upgraded → new PoolState layout live
2. Tax Program still compiled against old offsets → all buy/sell swaps read wrong reserves → wrong slippage floors applied for every swap
3. Epoch Program still compiled against old offsets → all Carnage slippage checks wrong
4. Window closes when Tax and Epoch are upgraded

There is no pause mechanism to halt swaps during this window. The only circuit breaker is the upgrade authority key, used for a subsequent emergency upgrade of Tax and Epoch — but compiling, deploying, and verifying two programs takes at minimum several minutes.

### 6. Upgrade Cascade Severity per Program (new-address scenario)

| Program Changed | Must Rebuild & Redeploy | Impact During Window |
|----------------|------------------------|---------------------|
| **AMM** | Tax, Epoch | All swaps fail; Carnage fails |
| **Tax Program** | AMM, Staking, Epoch | All swaps fail; staking rewards halt; Carnage fails |
| **Epoch Program** | Tax, Staking, BC | Epoch transitions fail; Carnage fails; staking rewards halt |
| **Staking** | Tax, Epoch | Reward deposits fail; cumulative updates fail |
| **Bonding Curve** | none | None |
| **Transfer Hook** | none | None |
| **Conversion Vault** | none | None |

### 7. Existing Mitigations (unchanged)

- In-place upgrades preserve program IDs — cascade only applies to new-address deployments.
- `sync-program-ids.ts` (build step 0/4) patches all ID references at build time.
- Discriminator unit tests in each `constants.rs` verify sha256 hash correctness.
- `deploy-all.sh` two-pass pipeline handles the chicken-and-egg problem.
- Shared PDA seeds are byte literals — visually verifiable.
- EpochState cross-program read uses full `try_deserialize` (discriminator + layout) — explicit errors on mismatch.
- `epoch-program/src/constants.rs:345-461` byte-layout test pins EpochState fields with offsets.

## Verdict

**CONFIRMED — MEDIUM severity, structural and unchanged from H049.**

The underlying architecture has not changed since H049: 15 hardcoded cross-program ID edges, 3 discriminator CPI couplings, and 2 raw-byte struct readers constitute a tightly coupled upgrade mesh with no atomic upgrade path and no pause mechanism. The only meaningful change since H049 is the addition of the EpochState byte-layout test (`epoch-program/src/constants.rs:345-461`), which pins EpochState field offsets and would catch layout drift for that one struct. The AMM PoolState raw-read coupling remains unpinned.

An additional active finding is the cluster contamination state: Tax Program's `amm_program_id()` is the mainnet value while AMM's `declare_id!` and Epoch's `amm_program_id()` are devnet values. This means `cargo test` in the staking crate would currently **fail** (`test_tax_program_id` asserts the mainnet Tax ID but the function body holds the devnet value). A naive `anchor build` without `sync-program-ids.ts` would produce a Tax binary that cannot CPI to the devnet AMM.

The cascade is manageable with proper tooling (the `build.sh` pipeline), but it is a structural fragility, not a defect with a code fix. The severity is MEDIUM because: (a) in-place upgrades do not trigger the ID cascade; (b) the most dangerous scenario (struct layout change causing silent data corruption) requires a deliberate developer action during upgrade; (c) there is no currently deployed exploitable state — this is a process/governance risk rather than an on-chain vulnerability.

## Recommendations

**SHORT-TERM (pre-mainnet, low effort):**
1. Add a cross-crate layout assertion for `PoolState` analogous to the existing `EpochState` byte-layout test. Add a const-assert or a `#[cfg(test)]` test in `tax-program` or `epoch-program` that serializes a test `PoolState` and verifies bytes [137..153] contain the expected values. This gives compile-time detection of AMM layout drift.
2. Fix the stale test assertion in `staking/src/constants.rs:138` — `test_tax_program_id()` asserts the mainnet Tax ID while the function holds the devnet ID. This test will fail on a `cargo test` run, masking other test failures.
3. Document the upgrade ordering requirement (Tax and Epoch must be upgraded immediately after any AMM struct change, within a single maintenance window) in the governance playbook.

**MEDIUM-TERM:**
4. Add a protocol-wide pause flag (a `is_paused` boolean in EpochState or a dedicated global PDA checked by Tax Program's swap entry points) to halt all swaps during multi-program upgrades. Without this, there is no safe window to upgrade interdependent programs.
5. Move AMM raw byte reads to a versioned interface: add a `layout_version: u8` field at a fixed offset (e.g., offset 8, before `pool_type`) in PoolState, and check it in `read_pool_reserves` before reading reserve offsets. This makes layout mismatches explicit rather than silent.

**LONG-TERM:**
6. Adopt Anchor's CPI crate pattern, importing `PoolState` directly from the AMM crate in Tax and Epoch programs. This provides compile-time guarantees on struct layout compatibility and eliminates the raw-byte read pattern entirely.
