# H009: Carnage Suppression via Optional Account Omission

**Status:** CONFIRMED
**Severity:** HIGH
**Category:** Access Control, State Machine, Economic
**Origin:** Novel

## Summary

The `carnage_state` account in `consume_randomness` is declared `Option<Account<'info, CarnageFundState>>` with no enforcement that it must be present. Any caller — including a MEV bot — can submit a valid `consume_randomness` transaction that omits this account, consuming the VRF and advancing epoch taxes while silently skipping the Carnage trigger check entirely. The code path is unambiguous and cannot be worked around by on-chain logic.

## Analysis

The instruction handler at `consume_randomness.rs:282` gates all Carnage logic behind a single `if let Some(ref carnage_state) = ctx.accounts.carnage_state` branch. When `carnage_state` is `None`, the program logs `"Carnage state not provided - skipping Carnage trigger check"` and returns `Ok(())` — completing the instruction successfully. There is no error, no revert, and no penalty.

The VRF value has already been fully consumed and the epoch taxes have already been updated by the time the optional check is reached (steps 3-7 happen unconditionally before step 9). Once `consume_randomness` succeeds, `epoch_state.vrf_pending` is cleared to `false` and `epoch_state.taxes_confirmed` is set to `true`. The VRF is spent. There is no mechanism to re-run the Carnage check for that VRF result after the fact.

The code comment explicitly justifies this as intentional: `"This allows backward compatibility and gradual rollout."` This means the vulnerability is not an accident — it is a deliberate design choice whose security implications were not fully modelled.

### Confirmation Checklist

| Question | Answer |
|---|---|
| Is `carnage_state` truly `Option`? | Yes — `pub carnage_state: Option<Account<'info, CarnageFundState>>` at line 80 |
| Is there any constraint requiring it be `Some`? | No — the `#[account(seeds = [CARNAGE_FUND_SEED], bump)]` annotation only constrains the PDA derivation IF the account is provided; it does not require it |
| Is the trigger check behind the `Some` guard? | Yes — all of step 9 (lines 282-327) is inside `if let Some(...) = ctx.accounts.carnage_state` |
| Is the VRF consumed regardless? | Yes — `vrf_result` is read and applied in steps 3-8 unconditionally |
| Does the instruction succeed without `carnage_state`? | Yes — returns `Ok(())` at line 330 regardless |
| Can the VRF be replayed? | No — `vrf_pending` is cleared at line 219; no retry path re-evaluates Carnage for a consumed VRF |

## Code Evidence

**Vulnerable declaration** (`programs/epoch-program/src/instructions/consume_randomness.rs:73-80`):
```rust
/// Carnage Fund state (for checking holdings and auto-expire).
/// Optional - if not provided, Carnage trigger check is skipped.
/// This allows backward compatibility and gradual rollout.
#[account(
    seeds = [CARNAGE_FUND_SEED],
    bump,
)]
pub carnage_state: Option<Account<'info, CarnageFundState>>,
```

**Entire Carnage branch gated behind the `Some` check** (`consume_randomness.rs:279-328`):
```rust
// === 9. Carnage trigger check ===
// Check VRF bytes 5-7 if carnage_state is provided.
if let Some(ref carnage_state) = ctx.accounts.carnage_state {
    if is_carnage_triggered(&vrf_result) {
        // ... set epoch_state.carnage_pending = true ...
    }
} else {
    msg!("Carnage state not provided - skipping Carnage trigger check");
}
```

**VRF is consumed and state is committed before the optional check** (`consume_randomness.rs:219-220`):
```rust
// === 7. Clear VRF pending state ===
epoch_state.vrf_pending = false;
epoch_state.taxes_confirmed = true;
```

## Attack Feasibility

**Preconditions:** The attacker needs only a funded Solana wallet (any amount sufficient to pay transaction fees, ~0.000005 SOL). No special permissions, no stake, no prior position required.

**Attack steps:**
1. Monitor the Solana mempool (or listen to `trigger_epoch_transition` events) to detect when `epoch_state.vrf_pending == true` and the Switchboard oracle has revealed its randomness.
2. Optionally preview the VRF result off-chain to check whether `vrf_result[5] < 11` (i.e., whether Carnage would trigger). The attack is equally effective even without this preview since suppressing Carnage unconditionally is profitable.
3. Construct a `consume_randomness` transaction that passes `carnage_state = None` (simply do not include the PDA in the accounts list).
4. Submit with high priority fee to land before the honest crank's transaction.

**Execution reality:** Solana mempools are not publicly observable in the same way as EVM, but:
- The attacker does not need to see the honest crank's TX in-flight. They only need to detect the oracle reveal on-chain (via polling or websocket subscription to the randomness account).
- Once the oracle reveals, the attacker and the crank both race. The attacker's TX is structurally simpler (no `carnage_state` account means fewer accounts, slightly smaller TX).
- If the attacker wins the race even once per Carnage-triggering epoch, that Carnage is permanently suppressed for that epoch.
- The VRF result cannot be recovered; `retry_epoch_vrf` creates a FRESH randomness account, giving the attacker another chance to suppress the next roll.

**Persistence of the attack:** An attacker who monitors on-chain state can suppress 100% of Carnage events indefinitely at a cost of approximately one transaction fee (~0.000005 SOL) per epoch suppression.

## Economic Impact

### Direct impact per suppressed Carnage event
- Carnage Fund accumulates 24% of all swap taxes. At steady-state protocol volume, this vault could hold tens to hundreds of SOL before a trigger.
- A single Carnage execution can deploy up to 1000 SOL into a buy.
- Suppressing one Carnage event with a triggering probability of 4.3% means the attacker prevents approximately 1000 SOL × 4.3% = 43 SOL expected value of buy pressure per epoch (using the maximum cap; real impact scales with vault balance).

### Indirect / systemic impact
- **PROFIT staker harm**: Carnage buys create token price support, increasing the value of staked positions. Suppression of buy pressure gradually devalues held tokens.
- **Protocol credibility harm**: Carnage is presented as the protocol's core distinguishing feature (unpredictable buy pressure counteracting sell pressure). If an adversary can suppress it, the entire narrative is undermined.
- **Asymmetric profitability**: Any actor who holds a short position on CRIME or FRAUD (or benefits from lower token prices, e.g., a competitor trying to cause a death spiral) has a direct financial incentive to run this suppression bot. The attack cost is essentially zero; the benefit scales with AUM.

### Carnage Fund vault growth during sustained suppression
- Each epoch that Carnage is suppressed, the vault continues to receive the 24% tax allocation from all swaps.
- A sustained suppression campaign causes the vault to grow indefinitely without ever deploying capital.
- This trapped SOL is essentially removed from the economic system, reducing yield to stakers and reducing price support.

## Verdict

**CONFIRMED — HIGH severity.**

The vulnerability is directly confirmed by source code inspection. The `carnage_state` being `Option` with no mandatory-presence enforcement is unambiguously the root cause. There is no compensating control: no signature check, no delayed execution, no second-chance mechanism, no on-chain slashing of suppressors.

The severity is assessed HIGH rather than CRITICAL because:
1. Carnage triggers with only 4.3% probability per epoch — in epochs where it would not have triggered anyway, this attack is a no-op.
2. The attack suppresses protocol value accrual rather than causing direct fund loss (the vault balance is not drained).
3. Execution requires continuous monitoring and active front-running, which is non-trivial operationally even if cheap financially.

It would be CRITICAL if the attacker could also drain the vault; they cannot. However, indefinite suppression of the protocol's core differentiating mechanic by anyone with a funded wallet is a significant security failure.

## Recommendations

### Fix (Required)
Make `carnage_state` mandatory by changing the account declaration from `Option<Account<'info, CarnageFundState>>` to `Account<'info, CarnageFundState>`. The "backward compatibility" rationale in the comment is no longer valid once the protocol is deployed to mainnet with Carnage Fund initialized.

```rust
// AFTER FIX:
#[account(
    seeds = [CARNAGE_FUND_SEED],
    bump,
)]
pub carnage_state: Account<'info, CarnageFundState>,
```

If a conditional Carnage check is still desired (e.g., before `initialize_carnage_fund` has been called), the proper pattern is to check whether the account has been initialized via a flag in `EpochState` rather than making the account optional in the instruction struct. Once initialized, the account must always be present.

### Defense in depth (Recommended)
Emit an on-chain event or error if `carnage_state` is not provided AND `epoch_state.carnage_initialized` is true, so that monitoring systems can detect suppression attempts even before a code fix is deployed.
