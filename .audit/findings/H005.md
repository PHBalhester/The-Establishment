# H005: Staking Escrow Rent Depletion Fix

**Status:** NOT VULNERABLE
**Severity:** N/A (Fix Verified)
**Category:** Token/Economic, State Machine
**Origin:** RECHECK (H012/S003 from Audit #2)

## Summary

The primary vulnerability from Audit #2 (H012/S003) has been correctly fixed. The `claim.rs` instruction now computes a rent-exempt minimum reservation via `Rent::get()?.minimum_balance(0)` and gates all claims against `available = escrow_balance - rent_exempt_min`, preventing any claim from draining the escrow PDA below the rent-exempt threshold. No concurrent claim race condition exists due to Solana's single-threaded account model. The `Rent::get()` sysvar call uses the current epoch's rent parameters, satisfying the concern about lamports_per_byte_year accuracy.

## Previous Finding

**H012 (Audit #2, CONFIRMED HIGH):** The `claim.rs` handler checked only `escrow_balance >= rewards_to_claim` without reserving the rent-exempt minimum (~890,880 lamports for a 0-byte account). This allowed claims that would leave the escrow between 0 and rent-exempt, causing the Solana runtime to either reject the transaction (claim freeze) or garbage-collect the PDA (permanent staking subsystem failure). No explicit rent floor guard was present anywhere in the staking program.

**S003 (Audit #2, CONFIRMED HIGH):** Supplemental analysis demonstrated the cascade: if the escrow were destroyed and recreated by `system_instruction::transfer` (System Program owned), subsequent `deposit_rewards` CPIs would succeed (AccountInfo owner not checked) but all future claims would be permanently bricked because the Staking Program cannot modify lamports of a System-owned account. The escrow account lacked an owner check in `deposit_rewards.rs`.

## Current Analysis

### Fix 1: Rent-Exempt Guard in claim.rs (Lines 101–121) — VERIFIED CORRECT

The current `claim.rs` implements the exact fix pattern recommended in H012:

```rust
let escrow_balance = ctx.accounts.escrow_vault.lamports();
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(0);

let available = escrow_balance
    .checked_sub(rent_exempt_min)
    .unwrap_or(0);

if available < rewards_to_claim {
    emit!(EscrowInsufficientAttempt { ... });
    return Err(StakingError::InsufficientEscrowBalance.into());
}
```

Key correctness properties:
- `Rent::get()?` fetches the live sysvar — uses current `lamports_per_byte_year`, no stale constant.
- `minimum_balance(0)` is the correct argument because `escrow_vault` is a 0-data-byte PDA (`space = 0` in `initialize_stake_pool.rs`).
- `.checked_sub(rent_exempt_min).unwrap_or(0)` is conservative: if the escrow is already below rent-exempt (e.g., due to external lamport drain), `available` returns 0 rather than wrapping, blocking all claims until the balance is topped up.
- The guard fires before any state mutation (CEI order maintained). State updates at lines 126–141 and the lamport transfer at lines 150–162 occur only after the guard passes.
- The post-transfer escrow balance is guaranteed to be `>= rent_exempt_min`: since `available >= rewards_to_claim` passed, and `available = escrow_balance - rent_exempt_min`, the result `escrow_balance - rewards_to_claim >= rent_exempt_min` is algebraically guaranteed.

### Fix 2: Concurrent Claims Race Condition (Hypothesis Item 2) — NOT APPLICABLE

The hypothesis asks whether multiple concurrent claims could race past the guard. On Solana, accounts are locked for the duration of a transaction. Two transactions both including `escrow_vault` as mutable cannot execute in the same slot concurrently — the runtime serializes them. The second transaction will see the post-first-transaction escrow balance when it executes. No race condition is possible at the protocol level.

### Fix 3: Rent Calculation Uses Current Sysvar (Hypothesis Item 3) — VERIFIED

`Rent::get()?` (line 105) is the standard Anchor/Solana mechanism for reading the current Rent sysvar. It returns the on-chain epoch-accurate value of `lamports_per_byte_year` and `exemption_threshold`. This is the same call used throughout the Solana ecosystem and is not a stale constant. The concern about outdated lamports_per_byte_year is not applicable here.

### S003 Cascade: deposit_rewards Owner Check — PARTIALLY ADDRESSED

The `deposit_rewards.rs` account struct still uses `AccountInfo<'info>` for `escrow_vault` (line 66) with no explicit `constraint = escrow_vault.owner == crate::id()` check. This means the destruction-and-recreation cascade described in S003 (System Program re-owning the escrow) theoretically remains possible if the escrow were somehow drained to zero through a path other than `claim.rs`.

However, with the H012 fix in place, the only lamport-reducing path into the escrow is `claim.rs`, which now hard-blocks below rent-exempt. There is no other instruction in the staking program that reduces escrow lamports. The escrow can only be drained to zero if:
1. An external actor sends lamports OUT of the escrow — impossible because only the owning program can reduce lamports on a program-owned account via `try_borrow_mut_lamports`, and no other staking instruction does so.
2. The `claim.rs` guard is bypassed — not possible given the algebraic guarantee proven above.

The S003 cascade is therefore unreachable in the current code. The owner check in `deposit_rewards.rs` remains a defense-in-depth improvement (belt-and-suspenders) but is not required for security given the primary fix.

## Verdict

The H012/S003 primary vulnerability is **FIXED**. The rent-exempt guard is correctly implemented using the live Rent sysvar, the correct `minimum_balance(0)` argument matching the 0-byte escrow account, and conservative saturating subtraction. The fix is algebraically sound and cannot be bypassed through concurrent execution. The `deposit_rewards.rs` missing owner check noted in S003 is a residual defense-in-depth gap but is not exploitable given the closed lamport-drain path. No new vulnerabilities were introduced by the fix.

## Recommendations

1. **No action required on claim.rs** — the fix is complete and correct.

2. **Optional defense-in-depth (Low priority):** Add an owner check to `deposit_rewards.rs` as recommended in S003:
   ```rust
   require!(
       ctx.accounts.escrow_vault.owner == &crate::id(),
       StakingError::InvalidEscrowOwner
   );
   ```
   This would cause a hard revert on any unexpected escrow ownership change rather than silent state corruption. A new `InvalidEscrowOwner` error variant would need to be added to `errors.rs`. This is a hardening measure, not a fix for an active vulnerability.

3. **Consider a regression test** (Low priority): A test asserting that `claim` fails with `InsufficientEscrowBalance` when `escrow_balance == rent_exempt_min + rewards - 1` and succeeds when `escrow_balance == rent_exempt_min + rewards` would lock in this invariant against future refactors.
