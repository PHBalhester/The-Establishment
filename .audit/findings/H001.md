# H001: BC Admin Authority Fix Verification

**Status:** NOT VULNERABLE
**Severity:** N/A (previously CRITICAL — now remediated)
**Category:** Access Control, Initialization
**Origin:** RECHECK (H001/H002/H010 from Audit #2)

---

## Summary

The critical bonding curve authority gap (H001/H002/H010 from Audit #2 — any signer could steal ~2000 SOL by calling `prepare_transition` + `withdraw_graduated_sol`) has been completely remediated. The BcAdminConfig PDA pattern with upgrade authority gating is correctly implemented across all 6 admin-gated instructions, `transfer_bc_admin` rejects `Pubkey::default()`, and the deployment gap (DEPLOY-GAP-01) has been closed in `initialize.ts`. No residual attack surface from the original findings remains.

---

## Previous Finding

Audit #2 (2026-03-08) raised three CRITICAL findings:

- **H001**: `withdraw_graduated_sol` accepted any `Signer<'info>` with no identity check; all SOL was transferred directly to the unchecked signer.
- **H002**: `prepare_transition` accepted any signer, enabling an attacker to force-graduate both curves and then immediately drain ~2000 SOL via `withdraw_graduated_sol` in one atomic transaction.
- **H010**: Confirmed the same-TX state visibility mechanic making the attack feasible; classified as "Graduation MEV Bundle" and cross-referenced all 6 affected instructions.

The root cause was that all 6 bonding curve admin instructions (`initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault`) had a bare `Signer<'info> authority` with no `has_one`, no `address =`, and no ProgramData constraint.

---

## Current Analysis

### 1. `initialize_bc_admin` — Upgrade Authority Gating

**File:** `programs/bonding_curve/src/instructions/initialize_bc_admin.rs`

The instruction follows the exact pattern recommended in H001's Option B and implemented by the AMM program:

```rust
#[account(
    constraint = program.programdata_address()? == Some(program_data.key())
)]
pub program: Program<'info, crate::program::BondingCurve>,

#[account(
    constraint = program_data.upgrade_authority_address == Some(authority.key())
)]
pub program_data: Account<'info, ProgramData>,
```

- Two-step validation: `program.programdata_address()? == Some(program_data.key())` prevents a fake `program_data` account being passed that claims any desired upgrade authority.
- `program_data.upgrade_authority_address == Some(authority.key())` enforces the signer is the actual upgrade authority.
- `Anchor init` on `admin_config` with seeds `[BC_ADMIN_SEED]` guarantees single initialization — re-initialization is impossible.
- The `admin` pubkey passed as an argument is stored in `admin_config.authority` and can differ from the upgrade authority (supports multisig delegation on first call).

**Verdict:** Initialization is correctly gated. No front-running possible because only the upgrade authority can call this instruction.

### 2. All 6 Admin-Gated Instructions — `has_one = authority` on BcAdminConfig

Every admin-gated instruction now follows the same pattern:

```rust
pub authority: Signer<'info>,

#[account(
    seeds = [BC_ADMIN_SEED],
    bump = admin_config.bump,
    has_one = authority @ CurveError::Unauthorized,
)]
pub admin_config: Account<'info, BcAdminConfig>,
```

Verified in all 6 instructions:

| Instruction | has_one = authority | File |
|---|---|---|
| `initialize_curve` | YES | `initialize_curve.rs:22-27` |
| `fund_curve` | YES | `fund_curve.rs:26-31` |
| `start_curve` | YES | `start_curve.rs:21-26` |
| `prepare_transition` | YES | `prepare_transition.rs:23-28` |
| `withdraw_graduated_sol` | YES | `withdraw_graduated_sol.rs:31-36` |
| `close_token_vault` | YES | `close_token_vault.rs:31-36` |

The `has_one = authority` Anchor constraint desugars to: `admin_config.authority == authority.key()`. Combined with `authority: Signer<'info>`, this means: only the key stored in `BcAdminConfig.authority` at deploy time can call any of these instructions.

The H001/H002/H010 attack scenario is now impossible:
- `prepare_transition` requires `admin_config.authority == attacker.key()`. Since `BcAdminConfig` is a PDA initialized by the upgrade authority with their pubkey as `admin`, any attacker's wallet will fail `has_one`.
- `withdraw_graduated_sol` has the same gate.
- An attacker cannot create their own `BcAdminConfig` because the PDA seeds are deterministic (`[b"bc_admin"]`) and Anchor `init` prevents re-creation of an existing account.

### 3. Deploy Window — Gap Between Deploy and `initialize_bc_admin`

The original H001 recommendation noted the risk of a window between program deployment and `BcAdminConfig` initialization (DEPLOY-GAP-01). This gap has been explicitly closed in the deploy pipeline.

**File:** `scripts/deploy/initialize.ts:1593-1626`

Step 17 (the first BC-related step after deployment, preceding `initialize_curve` at Step 18) calls `initializeBcAdmin` before any curve is initialized. The comment reads: _"This closes DEPLOY-GAP-01: BcAdminConfig was previously a manual step that could be forgotten, leaving curves without admin access control."_

The idempotency guard checks whether `bcAdminConfig` already exists and skips if so. Steps 18–25 (all curve operations including `initializeCurve`, `fundCurve`, `startCurve`) all pass `adminConfig: bcAdminConfig` in their account structs, meaning they would fail at the Anchor constraint level if BcAdminConfig did not exist.

**Residual risk (low):** The deploy pipeline is a TypeScript script, not an on-chain enforcement. An operator who manually calls instructions out of order (e.g., deploys the program but does not run `initialize.ts`) would have a window during which no BcAdminConfig exists. During this window, `initialize_curve` etc. cannot be called at all (they require `admin_config` to exist), so the vulnerability is not re-opened — the instructions simply fail until BcAdminConfig is created. The window does NOT allow an attacker to front-run because `initialize_bc_admin` is gated by upgrade authority. Only the deployer can close the window.

### 4. `transfer_bc_admin` — Rejects `Pubkey::default()`

**File:** `programs/bonding_curve/src/instructions/transfer_bc_admin.rs:21-24`

```rust
require!(
    new_authority != Pubkey::default(),
    CurveError::InvalidAuthority
);
```

The explicit guard prevents accidental transfer to the zero address (which would silently burn admin without the ceremony of `burn_bc_admin`). Anyone wishing to permanently revoke admin must call `burn_bc_admin` explicitly.

### 5. `burn_bc_admin` — Correct Irreversibility

**File:** `programs/bonding_curve/src/instructions/burn_bc_admin.rs`

`burn_bc_admin` sets `admin_config.authority = Pubkey::default()`. After this call, all admin-gated instructions fail permanently because `has_one = authority` cannot be satisfied (no wallet can sign as `Pubkey::default()`). This is the intended permanent-revocation path and is correctly distinguished from `transfer_bc_admin`.

### 6. One-Step Authority Transfer (Residual Low Risk)

As noted in the access control context (Risk #3), all authority transfers (including `transfer_bc_admin`) are single-step: there is no propose/accept pattern. If `transfer_bc_admin` is called with a wrong address (that is not `Pubkey::default()`), the admin is permanently lost with no recovery path. This is an operational risk, not an exploitability issue, and matches the pattern used by the AMM's `transfer_admin`. It is the same risk that exists in the AMM program and has been accepted as a known limitation.

## Verdict

The fix is complete and correctly closes the original H001/H002/H010 attack surface. The BcAdminConfig PDA pattern mirrors the AMM's proven `AdminConfig` approach. All 6 admin-gated instructions consistently use `has_one = authority` against the PDA. The deploy pipeline automation removes the human-error gap. No new issues were introduced by the fix.

The only residual concern is the single-step authority transfer pattern shared with the AMM program — this is an operational risk (accidental loss on mistaken transfer target) rather than an exploitability issue, and is a known accepted limitation across the codebase.

---

## Recommendations

1. **No changes required for the original H001/H002/H010 vulnerabilities** — the fix is complete.
2. **Consider two-step authority transfer (LOW priority):** For both `transfer_bc_admin` and `transfer_admin` (AMM), a propose/accept pattern would eliminate the accidental-loss risk. This is an industry best practice but not urgent given that the deployer intends to transfer to a Squads multisig (where the multisig itself provides operational safeguards).
3. **Confirm DEPLOY-GAP-01 status in MEMORY.md (housekeeping):** The project memory states "BcAdminConfig deploy pipeline: Not automated in initialize.ts. Carried to v1.4 (DEPLOY-GAP-01)." However, `initialize.ts:1602` explicitly states this gap was closed. The memory entry is stale and should be updated to reflect that DEPLOY-GAP-01 is resolved.
