# H004: Sell Path Slippage Fix Verification

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** Token/Economic, MEV
**Origin:** RECHECK (H008)

## Summary

The H008 finding from Audit #2 identified that `swap_sol_sell` passed `minimum_amount_out = 0` to the AMM CPI, disabling the AMM's own slippage protection on the sell path. The recommended fix (Option A: compute a gross floor from `minimum_output` and `tax_bps`) has been correctly implemented. The two-step slippage architecture is sound: the pre-CPI gross floor passed to the AMM ensures the AMM cannot execute at a price that would, after tax deduction, leave the user below their stated minimum.

## Previous Finding

H008 (Audit #2) confirmed that `swap_sol_sell.rs:147` passed a hardcoded `amm_minimum: u64 = 0` to the AMM CPI, rendering the AMM's `require!(amount_out >= minimum_amount_out)` check vacuous. The 50% pre-CPI output floor checked only the user's *parameter*, not actual execution. The post-CPI net check (`net_output >= minimum_output`) was the sole real guard, but an attacker who could calibrate the sandwich to keep `net_output` just above `minimum_output` could extract the full spread between fair price and the user's stated tolerance (up to ~50%).

## Current Analysis

The code has been updated to implement H008's Option A recommendation. The sell path now computes a `gross_floor` before the AMM CPI and passes it as `amm_minimum`:

```rust
// swap_sol_sell.rs:151-167
let gross_floor = if minimum_output > 0 && (tax_bps as u64) < bps_denom {
    let numerator = (minimum_output as u128)
        .checked_mul(bps_denom as u128)
        ...
    // Ceil division: (numerator + denominator - 1) / denominator
    let result = numerator
        .checked_add(denominator - 1)
        ...
        / denominator;
    u64::try_from(result)...
} else {
    0
};
let amm_minimum: u64 = gross_floor;
```

This ensures: if `gross_output < gross_floor`, the AMM reverts before Tax Program even sees the result. Since `gross_floor = ceil(minimum_output * 10000 / (10000 - tax_bps))`, the guarantee holds: any `gross_output >= gross_floor` will produce `net_output = gross_output - tax >= minimum_output`.

The post-CPI net check at line 265 (`require!(net_output >= minimum_output)`) remains as a defense-in-depth redundant guard.

The pre-CPI output floor check at lines 113-118 (50% floor via `calculate_output_floor`) also remains as the anti-zero-slippage bot gate.

### Reserve Ordering

`read_pool_reserves` at `pool_reader.rs:92-96` correctly detects canonical mint ordering via `is_reversed` logic (DEF-02): if `mint_a != NATIVE_MINT`, it swaps `reserve_a` and `reserve_b`. The sell call at line 112 correctly requests `(sol_reserve, token_reserve)` and passes them to `calculate_output_floor` as `(token_reserve, sol_reserve)` for the BtoA direction.

### Arithmetic Soundness

`calculate_output_floor` in `tax_math.rs:141-165` uses u128 intermediates throughout. With `reserve_out` and `amount_in` both bounded by u64::MAX, the product `reserve_out * amount_in` fits in u128 (max ~3.4e38 vs u128::MAX ~3.4e38 — technically saturating at near-max reserves, but the `checked_mul` would return `None` rather than overflow, producing a `TaxOverflow` error). The property tests at line 369-390 verify no overflow at u64::MAX/2 scale inputs.

The `gross_floor` calculation uses u128 intermediates with ceil division and a final `u64::try_from` cast, which will error if the result exceeds u64::MAX. At `tax_bps` approaching 10000 and large `minimum_output`, the numerator `minimum_output * 10000` could approach `u64::MAX * 10000` before the division — this is handled correctly because the intermediate is u128 (max ~1.8e23 * 10000 = 1.8e27, well within u128::MAX).

### Two-Step Architecture Assessment

The two-step design is now symmetric with the buy path:

| Path | Pre-CPI check | AMM minimum | Post-CPI check |
|------|---------------|-------------|----------------|
| Buy  | 50% floor on user minimum | user's minimum_output | none needed |
| Sell | 50% floor on user minimum | gross_floor (= ceil(min_out * 10000 / (10000 - tax_bps))) | net_output >= minimum_output |

The sell post-CPI check is redundant given the AMM's own guard will enforce `gross_output >= gross_floor`, and `net_output = gross_output - ceil(gross_output * tax_bps / 10000)`. The redundancy is benign and provides defense-in-depth.

### Remaining Observation: Manipulable Reserves in Pre-CPI Floor

The 50% pre-CPI output floor uses reserves read from the pool at the time of the instruction. These reserves reflect the current on-chain state, which could have been manipulated by a front-runner in a prior transaction within the same block. This was noted in H008 and remains true. However:

1. This only affects the *gate* that checks whether the user's `minimum_output` is reasonable, not the actual execution floor.
2. A front-runner who manipulates reserves before the instruction executes will cause `gross_floor` to be computed from those same manipulated reserves — but `gross_floor` is derived from `minimum_output`, not from the pool's reserves. The `gross_floor` calculation at lines 151-167 uses only `minimum_output` and `tax_bps` as inputs, making it independent of current pool reserves.
3. The 50% pre-CPI floor check could be made to reject a legitimate user's `minimum_output` (if the manipulated reserves make the expected output appear lower, lowering `output_floor`, which would not cause the check to fail — `output_floor` would be lower after a sell-side front-run). This is a non-exploitable edge case.

## Verdict

The vulnerability identified in H008 has been fully remediated. The `amm_minimum` passed to the AMM CPI is now a non-zero gross floor derived from the user's `minimum_output` and the current `tax_bps`, using ceiling division to ensure the AMM cannot execute a swap that, after tax, would leave the user below their stated minimum. This matches the buy-path protection pattern. Arithmetic is safe at all practical scales. The two-step (pre-CPI + post-CPI) slippage architecture is sound and symmetric.

The residual observation about manipulable reserves in the pre-CPI floor check is pre-existing and non-exploitable for the same reasons noted in H008: the 50% floor is designed as a bot/zero-slippage gate, not a sandwich defense in isolation, and the actual sandwich defense is now the `gross_floor` passed directly to the AMM.

## Recommendations

None required. The fix is correct and complete.

One optional hardening note: the comment at line 106 ("The floor protects against zero-slippage sandwich attacks BEFORE CPI. We verify the user's stated minimum is reasonable; the AMM then enforces actual_output >= minimum during execution.") is now slightly imprecise — the AMM enforces `gross_output >= gross_floor`, not `actual_output >= minimum`. This is a documentation nit with no security impact.
