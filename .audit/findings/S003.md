# S003: Stale Mainnet Mints Cause Silent Graduation Failure

**Status:** CONFIRMED
**Severity:** HIGH
**Category:** Configuration, Upgrade/Admin
**Inspired by:** H008 (POTENTIAL)

## Summary

H008 confirmed that the mainnet branches of all five mint-address functions across Bonding Curve and Conversion Vault contain stale Phase 69/95 devnet addresses rather than the finalized mainnet vanity mints. This strategy investigates the specific graduation failure path: when a bonding curve fills to capacity and the admin attempts `prepare_transition` followed by graduation, would the stale mint constants reject real mainnet tokens? Would the Conversion Vault reject real tokens on convert? The analysis confirms that deployment without `patch-mint-addresses.ts` produces a non-functional mainnet deployment, and identifies that the failure mode for graduation is financial (graduated SOL stranded) while the failure mode for the vault is operational (all conversions rejected).

### Graduation Failure Path: Bonding Curve

The graduation lifecycle depends on the Bonding Curve's mint address validation. Tracing through the typical graduation sequence:

**Step 1: `initialize_curve`**
The initialization constraint validates the token mint against `crime_mint()` or `fraud_mint()`:
```rust
constraint = token_mint.key() == crime_mint() || token_mint.key() == fraud_mint()
```

If the binary was compiled without `patch-mint-addresses.ts`, this check rejects the real mainnet CRIME/FRAUD mints (`cRiME...`, `FraUd...`) and would only accept the devnet addresses. Since those devnet addresses don't exist on mainnet, no bonding curve can ever be initialized.

**Outcome of Step 1 without patch**: `initialize_curve` fails at account constraint level with an Anchor `ConstraintRaw` error. No curve can be started. This is a closed failure — no funds are at risk, just a bricked program.

**Step 2: `purchase`**
Requires an initialized curve (blocked by Step 1 failure). If somehow curve was initialized with a non-mint account, subsequent purchase calls would fail at token operations.

**Step 3: `prepare_transition` and `withdraw_graduated_sol`**
These instructions require a fully filled, `FILLED`-status curve. Since no curve can initialize, these are unreachable.

**However** — consider the more dangerous scenario: a partial deployment where `initialize_curve` was run in an earlier build (before the mint constants were stale) and then the program is upgraded to the stale-mint version. In that case:
- The curve is already initialized with correct mint addresses stored on-chain in `CurveState`
- `purchase` and `sell` may still work (they validate against `CurveState.token_mint`, not the constant)
- `prepare_transition` and `withdraw_graduated_sol` depend on `admin_config.authority` matching — this doesn't reference the mint constant
- The mint constant is primarily used in initialization

Under a program upgrade scenario where the stale binary replaces a working one mid-curve, the primary risk shifts to the conversion vault (below). The bonding curve graduation itself may complete, but the conversion vault will reject the graduated tokens.

### Graduation Failure Path: Conversion Vault

The Conversion Vault's `convert` instruction validates all three mints. From `conversion-vault/src/constants.rs` and referencing the initialization:

The vault's `initialize` instruction validates that passed mint accounts match `crime_mint()`, `fraud_mint()`, and `profit_mint()`. With stale devnet addresses compiled in, the vault's initialization would reject real mainnet mints.

Even if the vault was initialized in a previous build (correct addresses stored in vault state), a program upgrade to the stale binary would cause `convert` to reference the stale constants in constraints during the upgrade epoch's build.

**Critical failure mode**: If the vault is deployed with stale constants AND a user attempts a conversion:
- The vault validates the input mint against its stored state (on-chain, correct from initialization)
- The vault may also validate against the compiled constant for additional constraint checking
- If any constraint validates `token_mint.key() == crime_mint()`, it compares the real `cRiME...` against the compiled `DtbD...` and rejects

This means conversions would silently fail at the constraint level. Worse, users may deposit tokens into the vault's ATA expecting conversion, only to have the instruction rejected — leaving tokens in limbo at the vault ATA (though they remain recoverable as the vault ATA owner is the vault PDA).

### Graduation Scenario: No Financial Loss But Complete Bricking

For the specific graduation question posed: "Would graduation fail? Would conversion vault reject real tokens?"

**Graduation answer**: Yes, graduation fails closed at the `initialize_curve` step if the program is deployed without `patch-mint-addresses.ts`. No SOL at risk, but the bonding curve program is entirely non-functional. Graduation cannot proceed at all.

**Conversion vault answer**: Yes, the vault rejects real mainnet tokens. Conversions fail at constraint validation. SOL and tokens are not lost (they stay in the caller's wallet), but the vault is non-functional.

**The more dangerous scenario** is a two-deployment sequence:
1. Correct mainnet deploy (with `patch-mint-addresses.ts`) — protocol works, users buy through bonding curve, vault fills
2. Program upgrade deployed WITHOUT re-running `patch-mint-addresses.ts` (stale binary)
3. `prepare_transition` or `withdraw_graduated_sol` may still work (not directly constrained by mint constants in their immediate logic)
4. But the conversion vault rejects attempts to convert faction tokens to PROFIT

In this scenario, graduated SOL has been withdrawn by the team (Step 3 succeeds), but users who expected to convert faction tokens via the vault are blocked. The graduated tokens are stranded on holders who cannot access the conversion mechanism.

### Build Pipeline as Single Point of Failure

The build pipeline (`build.sh`, `patch-mint-addresses.ts`) is the sole safeguard between the stale constants and a broken mainnet deployment. As confirmed by H008 and H012:

- `build.sh` runs `patch-mint-addresses.ts` in step [0b/4]
- If someone runs `anchor build` directly (bypassing `build.sh`), stale constants ship
- No `compile_error!` guards exist in any of the affected files
- The post-build verification (`verify-program-ids.ts`) checks program IDs for consistency, but does NOT verify that mint constants contain mainnet addresses

This means a developer running `anchor build` on mainnet without `build.sh` produces a silently broken binary.

### Compile-Error Guards: Absent

The H008 recommendation to add `compile_error!` guards has not been implemented. A correct guard would be:

```rust
#[cfg(not(any(feature = "devnet", feature = "localnet")))]
pub fn crime_mint() -> Pubkey {
    compile_error!("MAINNET: crime_mint() not patched. Run scripts/deploy/build.sh.");
}
```

Without this, the stale addresses compile silently as a mainnet binary.

## Verdict

**CONFIRMED — HIGH severity.**

The stale mainnet mint constants are a confirmed deployment risk. The failure mode for graduation is a complete operational brick: bonding curves cannot initialize, and conversion vault cannot process real mainnet tokens. No user funds are directly lost from a fresh deployment with stale constants (failures are closed), but:

1. A partial deployment scenario (upgrade to stale binary mid-operation) could strand users who cannot convert graduated faction tokens to PROFIT.
2. The absence of `compile_error!` guards means this risk is invisible at compile time.
3. A single developer bypassing `build.sh` and running `anchor build` directly would ship a broken mainnet binary with no compile-time warning.
4. The `verify-program-ids.ts` post-build check does not catch stale mint addresses.

The risk is categorized HIGH because (a) it blocks an entire revenue-generating mechanism (bonding curves / graduation / vault conversions), (b) a partial-deployment scenario can trap users who already purchased on the bonding curve, and (c) the safeguard (`patch-mint-addresses.ts`) is a single unguarded step that can be bypassed.

## Recommendations

1. **Add `compile_error!` guards to all five stale mainnet functions** (H008 recommendation, still unimplemented). These are the critical safety net. Without them, the stale address is a silent compile-time landmine.

2. **Add mint address verification to post-build step [3/4]** in `build.sh`: After `anchor build`, grep the compiled binary (or the patched source files) for the known devnet mint addresses. If any appear in a mainnet build context, abort with an error.

3. **Extend `verify-program-ids.ts` to also verify mint constants**: The script currently checks program IDs but not mint addresses. Add a check that reads the compiled constants and verifies them against an expected manifest (`deployments/mainnet-mints.json`).

4. **Document `patch-mint-addresses.ts` as a mandatory pre-deploy gate** in the deploy checklist (`Docs/mainnet-deploy-checklist.md`), with an explicit "verify this ran" step (e.g., grep for `cRiME` in `constants.rs` before running `build.sh`).

5. **Two-person review for mainnet builds**: Require two authorized team members to independently verify mint address constants in the compiled binary before any mainnet deploy or upgrade. This is the human backstop for the missing compile-time guard.
