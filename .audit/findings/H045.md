# H045: Dual-Curve Grief Verification

**Status:** NOT VULNERABLE
**Severity:** N/A (RECHECK -- previously LOW)
**Category:** State Machine / Economics
**Origin:** RECHECK (H031)

## Summary

The dual-curve grief attack described in H031 remains theoretically possible but economically constrained. The bonding curve coupling mechanism (where both curves must be `Filled` to graduate, or either being `Failed` triggers refund mode for both) has not changed. However, the attack cost (15% sell tax + buying opportunity for other participants) remains prohibitive enough to prevent practical exploitation.

## Code Evidence

**Coupling mechanism verified in:**

**File:** `programs/bonding_curve/src/instructions/prepare_transition.rs`

The instruction requires both curve states to be `Filled` for graduation:
```rust
constraint = crime_curve_state.status == CurveStatus::Filled @ CurveError::InvalidStatus,
constraint = fraud_curve_state.status == CurveStatus::Filled @ CurveError::InvalidStatus,
```

**Refund eligibility verified in:**

**File:** `programs/bonding_curve/src/state.rs:137-143`

The `is_refund_eligible()` function returns true if the partner curve is `Failed`, triggering refund mode even if one curve is `Filled`.

**Attack cost verified in:**

**File:** `programs/bonding_curve/src/instructions/sell.rs:174-179`

The 15% sell tax is non-refundable and deducted from gross SOL proceeds:
```rust
let tax_amount = calculate_tax(gross_sol, SELL_TAX_BPS)
    .ok_or(error!(CurveError::TaxOverflow))?;
let net_sol = gross_sol.checked_sub(tax_amount)...
```

**Wallet cap verified in:**

**File:** `programs/bonding_curve/src/instructions/purchase.rs`

`MAX_TOKENS_PER_WALLET = 20,000,000,000,000` (20M tokens) limits single-wallet accumulation.

## Verdict

**NOT VULNERABLE.** The dual-curve grief mechanism exists as designed but remains economically infeasible.

The prior H031 audit correctly identified that one curve failing forces both into refund mode due to the `prepare_transition` coupling and the `is_refund_eligible(partner_status=Failed)` logic. However, the attack requires:

1. **Substantial capital:** An attacker buying 100M tokens across 5 wallets requires ~217 SOL (at average curve prices).
2. **Non-recoverable tax cost:** Selling those 100M tokens to prevent filling incurs a 15% tax (~32.5 SOL loss).
3. **Price dip exploitation:** After the attacker sells, tokens become cheaper, creating a buying opportunity for legitimate participants to re-fill the gap before deadline.
4. **Timing constraint:** The attacker must sell near the deadline to minimize the re-fill window, but `deadline_slot` blocks both buys and sells at the same cutoff.

**Economic analysis (from H031):**
- Grief magnitude: Requires moving 100M tokens (21.7% of 460M target) to create a meaningful filling obstacle
- Capital requirement: ~217 SOL
- Sunk tax cost: ~32.5 SOL (15% of gross proceeds)
- Total cost to attacker: >250 SOL in sunk expenses
- Attacker profit: Zero (griefing only, no direct economic benefit)

This is a rational-actor economic defense: the cost to grief exceeds any benefit.

**Scenarios where grief might succeed:**
- A curve struggling to fill organically (e.g., stalled at 80% with hours remaining)
- Weak demand where legitimate buyers cannot re-fill the gap quickly
- Both curves equally underwater (forcing both into refund mode has marginal additional impact)

In healthy launch conditions with strong demand, the 48-hour window and price-dip opportunity make this grief impractical.

## No On-Chain Remediation Required

The current design is adequate. If grief becomes a concern in practice:

1. **Off-chain monitoring:** Alert the team if either curve's token_sold drops significantly near deadline
2. **Operational mitigation:** Incentivize re-filling via partnerships, social layer, or manual buys
3. **Design change (if desired):** Allow independent graduation per curve (requires protocol re-architecture)
