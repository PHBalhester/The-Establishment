# H046: taxes_confirmed Unchecked by Tax Program

**Status:** CONFIRMED (DESIGN CHOICE)
**Severity:** INFORMATIONAL
**Category:** Oracle / State Machine
**Origin:** RECHECK (H048)

## Summary

The Tax Program deliberately omits a `taxes_confirmed` check when reading EpochState to determine tax rates. During the VRF pending window (between `trigger_epoch_transition` and `consume_randomness`), swaps execute using the previous epoch's rates. This is an intentional design choice to prioritize protocol liveness over rate freshness, with negligible economic impact.

## VRF Pending Window Duration

**Normal case:** 2-3 slots (~0.8-1.2 seconds at 400ms/slot) during which `taxes_confirmed = false`

**Degraded case (oracle timeout):** Up to `VRF_TIMEOUT_SLOTS = 300` slots (~2 minutes) if Switchboard oracle fails to reveal within timeout

**Context:** Mainnet epoch length is `SLOTS_PER_EPOCH = 4,500` (~30 minutes), so even a 300-slot timeout represents only ~6.7% of an epoch duration

## Economic Impact Analysis

**Can an attacker exploit stale rates?**

The tax rates during the VRF pending window are the *previous* epoch's rates:
- Already public on-chain (anyone can read current EpochState)
- Already known to all market participants from the previous epoch
- Bounded to fixed ranges: low {100-400} bps, high {1100-1400} bps

**Worst-case MEV scenario:** An attacker monitors the Switchboard reveal TX in the mempool, sees the new VRF bytes, and trades at the old (unknown-to-them) rate before `consume_randomness` lands. The maximum tax rate difference is 1,300 bps (from 1% to 14%). On a 10 SOL trade, this would save 1.3 SOL in tax -- but requires:
1. Knowing the reveal will flip rates unfavorably
2. Landing the TX in the 1-2 slot window (under normal conditions)
3. The swap itself being economically rational ignoring tax savings
4. Finding sufficient counterparty liquidity in that window

The MEV opportunity is marginal and constrained by the protocol's minimum output floor (50% of expected).

## Design Rationale

This is documented as an intentional choice in ARCHITECTURE.md (Section 6, A-10):

> "taxes_confirmed intentionally unchecked by Tax -- Design choice: swaps use stale rates during VRF window"

The architecture prioritizes **liveness over freshness**:
- Blocking swaps during `vrf_pending = true` would create a DoS vector: an attacker could grief by triggering epoch transitions during high-activity periods
- VRF oracle failures would halt all trading until the oracle recovers (300+ slots per retry)
- The bounded rate range (1-14%) ensures stale rates are always within acceptable protocol parameters

## Alternative Designs Rejected

If Tax Program checked `taxes_confirmed`, it would need to either:

1. **Reject swaps** when `taxes_confirmed = false` → Creates liveness/DoS issues
2. **Use a default/fallback rate** → Introduces new complexity and potential manipulation vectors
3. **Use genesis rate** → Not meaningfully different from using stale rates

The current approach (use previous epoch's rates without explicit check) is the simplest and most robust.

## Verdict

**CONFIRMED as intentional design choice. Risk: INFORMATIONAL.**

The Tax Program deliberately omits a `taxes_confirmed` check to prioritize liveness. Swaps continue with the previous epoch's bounded rates during the brief VRF pending window. This is:

1. **Documented** in ARCHITECTURE.md (A-10)
2. **Implemented consistently** across both `swap_sol_buy` and `swap_sol_sell`
3. **Safe from economic manipulation** due to bounded rate ranges and minimal MEV window
4. **Correct design** given the trade-off between freshness and availability

Risk factors:
- Normal window: ~1-2 slots -- effectively zero economic impact
- Degraded window: ~300 slots -- protocol remains functional, attacker has no foreknowledge of new rates
- Worst case: Indefinite oracle failure -- protocol operates at stale but bounded rates (strictly better than halting)

**No remediation recommended.** The current behavior correctly prioritizes availability over rate freshness. The bounded rate range (1-14%) ensures that stale rates are never economically catastrophic.
