# H041: Epoch Skip Staking Reward Forfeiture

**Status:** FALSE POSITIVE — No Vulnerability
**Severity (as claimed):** Tier 2 (High)
**Severity (actual):** Informational
**Category:** Token Economic / Timing
**Auditor:** SOS Phase 2

---

## Summary

The hypothesis that epoch skipping causes staking reward forfeiture is incorrect. The protocol's reward accumulation architecture is inherently safe against epoch gaps: `deposit_rewards` continuously accumulates tax revenue into `StakePool.pending_rewards` (a persistent running total), and `update_cumulative` distributes the entire accumulated balance in a single call when the epoch eventually transitions — regardless of how many epochs were skipped. No rewards are lost.

---

## Analysis

### Claim Under Investigation

> If crank delays cause epoch jumps (e.g., 100 → 105), rewards for skipped epochs are silently forfeited. Stakers lose real yield.

### How Reward Accumulation Actually Works

The staking reward pipeline has two independent stages:

**Stage 1 — Continuous accumulation (`deposit_rewards`):**
Every tax-paying swap calls `deposit_rewards` via CPI from the Tax Program. This instruction simply increments `StakePool.pending_rewards` by the SOL amount deposited. This happens regardless of epoch state — it is not epoch-gated. The `pending_rewards` field is a running total that persists until explicitly cleared.

```rust
// deposit_rewards.rs:90-93
pool.pending_rewards = pool
    .pending_rewards
    .checked_add(amount)
    .ok_or(StakingError::Overflow)?;
```

**Stage 2 — Epoch-boundary distribution (`update_cumulative`):**
Called once per epoch transition via CPI from `consume_randomness`. This takes the entire current `pending_rewards` balance and adds it to the cumulative `rewards_per_token_stored`, then clears `pending_rewards` to zero.

```rust
// update_cumulative.rs:86-108
let rewards_added = pool.pending_rewards;

if rewards_added > 0 && pool.total_staked > 0 {
    let reward_per_token = (rewards_added as u128)
        .checked_mul(PRECISION)
        // ...
        .checked_div(pool.total_staked as u128)?;

    pool.rewards_per_token_stored = pool
        .rewards_per_token_stored
        .checked_add(reward_per_token)?;

    pool.total_distributed = pool.total_distributed.checked_add(rewards_added)?;
}

// Clear pending regardless of whether distribution happened
pool.pending_rewards = 0;
```

### Epoch Skip Trace (Epoch 100 → 105)

1. **Epochs 100–104**: The crank is absent. Swaps continue normally. Each swap fires `deposit_rewards`, incrementing `pending_rewards`. After 5 epochs of trading, `pending_rewards` may hold, e.g., 500 SOL worth of accumulated tax revenue.

2. **Epoch 105 trigger**: When the crank finally calls `trigger_epoch_transition`, the epoch number jumps directly to `expected_epoch = 105` (as documented in the code comment at `trigger_epoch_transition.rs:143-149`).

3. **`consume_randomness` at epoch 105**: Calls `update_cumulative` with `epoch = 105`. The double-update guard (`require!(epoch > pool.last_update_epoch)`) passes because 105 > 100. The handler then distributes the **entire** 500 SOL that has been sitting in `pending_rewards` across all 5 missed epochs.

4. **Result**: All rewards deposited during epochs 100–104 are distributed in the single epoch 105 `update_cumulative` call. Nothing is forfeited.

### The Double-Update Guard Does Not Block This

The guard `require!(epoch > pool.last_update_epoch, StakingError::AlreadyUpdated)` at `update_cumulative.rs:78-81` uses a strict greater-than comparison. An epoch jump from 100 to 105 passes this guard trivially (105 > 100). The guard only prevents calling `update_cumulative` twice for the same epoch, not across gaps.

### What the Code Comment Actually Says

The code itself documents this behavior explicitly:

```
// NOTE: Epoch skipping is acceptable. If the crank is delayed, expected_epoch
// may be > current_epoch + 1 (e.g., jumping from epoch 100 to epoch 105).
// This is harmless because:
// - No rewards accrue during gaps (staking tracks per-epoch)  ← MISLEADING COMMENT
// - Tax rates persist at last-set values (safe fallback)
// - Carnage only triggers from VRF (no implicit triggers from skips)
```

The comment "No rewards accrue during gaps (staking tracks per-epoch)" is technically misleading — rewards **do** accrue during gaps via `deposit_rewards` on every swap. What the comment likely means is that `update_cumulative` is not called per-missed-epoch (i.e., no epoch-by-epoch accounting for staking accrual), but the **SOL itself does accumulate** and is distributed in bulk when the transition finally occurs.

### Comparison to Claimed Behavior

| Claimed | Actual |
|---------|--------|
| Rewards for epochs 101–104 are silently forfeited | Rewards for all missed epochs sit in `pending_rewards` and are distributed at epoch 105 |
| Stakers lose real yield | Stakers receive all yield, just delayed and batched |
| `update_cumulative` is called once per epoch and misses gaps | `update_cumulative` distributes all accumulated `pending_rewards` regardless of how many epochs were skipped |

## Verdict

**FALSE POSITIVE.** The Synthetix-style reward model is specifically designed to be epoch-agnostic at the accumulation layer. `pending_rewards` is a persistent balance, not a per-epoch slot that gets cleared on miss. An epoch skip of N epochs causes exactly one `update_cumulative` call that distributes all N epochs' worth of accumulated tax revenue in a single batch. Stakers receive the same total yield regardless of how many epochs were skipped; the only effect is that distribution is delayed and batched rather than spread across individual epoch boundaries.

The misleading code comment at `trigger_epoch_transition.rs:146` ("No rewards accrue during gaps") may have been the origin of this hypothesis. The comment should be corrected to accurately describe the actual behavior.

---

## Recommendations

1. **Correct the misleading comment** at `trigger_epoch_transition.rs:146`. Change:
   > `// - No rewards accrue during gaps (staking tracks per-epoch)`

   To:
   > `// - Staking rewards accumulate continuously in pending_rewards; all are distributed in bulk on the next update_cumulative call`

2. **No code changes required.** The accumulation model correctly handles epoch gaps.

3. **Optional — add a test**: A unit or integration test that skips multiple epochs and verifies `pending_rewards` is fully distributed (rather than partially or zeroed) would document this invariant explicitly and prevent future confusion.

*Finding H041 closed: FALSE POSITIVE. No staker yield is forfeited during epoch skips.*
