# H030: Admin SOL Withdrawal Centralization — Graduated Bonding Curve Vaults

## Summary

The `withdraw_graduated_sol` and `close_token_vault` instructions in the Bonding Curve program allow the single `BcAdminConfig.authority` key to drain all accumulated SOL from any graduated curve's SOL vault and recover rent from token vaults. At graduation, each curve may hold up to ~1000 SOL. A compromised or malicious admin key can extract this entire balance with a single instruction. As of the current devnet state, the authority is held by the deployer wallet (single key); the Squads multisig transfer is planned but not yet executed on mainnet.

---

## Analysis

### Authority Model

`BcAdminConfig` is a singleton PDA (`seeds = [b"bc_admin"]`) that stores one field: `authority: Pubkey`. All admin-gated bonding curve operations, including `withdraw_graduated_sol` and `close_token_vault`, validate the signer via a `has_one = authority` Anchor constraint. There is no secondary check, no timelock, and no co-signer requirement at the program level.

The architecture document (`ARCHITECTURE.md`) confirms the `BcAdminConfig` authority is currently the deployer wallet and is planned to be transferred to a Squads 2-of-3 multisig vault PDA (proven on devnet in Phase 97), but this transfer has not yet occurred on mainnet.

### `withdraw_graduated_sol` — Direct Lamport Drain

The instruction uses direct lamport manipulation on the program-owned `sol_vault` PDA, crediting the `authority` signer:

```
// withdraw_graduated_sol.rs:88-89
**ctx.accounts.sol_vault.try_borrow_mut_lamports()? -= withdrawable;
**ctx.accounts.authority.try_borrow_mut_lamports()? += withdrawable;
```

The only state guard is:
```
constraint = curve_state.status == CurveStatus::Graduated @ CurveError::CurveNotGraduated
```

This is correct — it prevents premature withdrawal before graduation. However, once `Graduated` status is set (via `prepare_transition`, itself admin-gated), the instruction imposes no additional delay, no partial withdrawal limit, no beneficiary restriction beyond "the admin signer," and no event-based rate limiting. The full vault balance (up to ~1000 SOL, minus bare rent minimum) can be swept atomically.

The `withdrawable` calculation leaves a floor of `Rent::get()?.minimum_balance(0)` (890880 lamports, ~0.00089 SOL), which is negligible relative to the vault balance.

### `close_token_vault` — Rent Recovery to Admin

The instruction closes the token vault (a Token-2022 `TokenAccount`) via CPI after verifying `token_vault.amount == 0` (no remaining tokens). Rent lamports from the closed account credit the `authority` wallet.

```
// close_token_vault.rs:88-96
token_interface::close_account(CpiContext::new_with_signer(
    ctx.accounts.token_program.to_account_info(),
    token_interface::CloseAccount {
        account: ctx.accounts.token_vault.to_account_info(),
        destination: ctx.accounts.authority.to_account_info(),
        authority: ctx.accounts.curve_state.to_account_info(),
    },
    signer_seeds,
))?;
```

The `amount == 0` constraint is sound — the token vault must be empty before it can be closed. This instruction's individual SOL impact is small (single token account rent), but it confirms the admin can fully clean up curve accounts.

### Squads Multisig — Planned but Not Enforced On-Chain

The governance documentation (`Docs/mainnet-governance.md`) describes transferring `BcAdminConfig.authority` to the Squads vault PDA as part of Stage 3 of the mainnet deployment. Phase 97 proved the Squads flow on devnet (2-of-3, 900s timelock). However:

- The program-level code has no knowledge of Squads. There is no on-chain enforcement that the authority must be a multisig.
- If the Squads transfer has not been executed (or is delayed post-launch), the single deployer key retains full withdrawal capability.
- Even after Squads transfer, the 900s (15-minute) timelock window gives the community minimal reaction time if a multisig signer is compromised.

### What Is Protected (Scope Boundaries)

The following attack paths are NOT possible even with admin key compromise:
- Withdrawing from Active/Filled/Failed curves — `CurveStatus::Graduated` constraint blocks this.
- Withdrawing tokens from the vault while it holds a non-zero balance — `token_vault.amount == 0` blocks `close_token_vault`; token withdrawal is handled by the graduation pipeline, not by these instructions directly.
- Withdrawing from an AMM pool — these instructions are scoped to the bonding curve program only.
- Redirecting funds to an arbitrary third party — `authority` is the `Signer` and the destination; the admin can only send funds to their own wallet (cannot specify a separate `destination`).

### Economic Exposure

Two graduated curves (CRIME and FRAUD), each with up to ~1000 SOL vault balance. Theoretical maximum single-key theft from these two instructions: ~2000 SOL total (minus negligible rent minimums). This is a large but time-bounded exposure — the bonding curve is a bootstrap mechanism and graduation is a one-time event. Post-graduation, the vaults are intentionally swept into AMM pool seeding.

## Verdict

**Confirmed: Design-by-intent centralization with insufficient operational safeguards at the time of assessment.**

The state guards (Graduated status, empty token vault) are correctly implemented and prevent the worst-case scenario of early fund extraction. The intended use case — sweeping graduated vault SOL for AMM seeding — is legitimate and necessary. This is not a code defect.

However, the risk profile is elevated because:
1. The authority is a single hot key with no on-chain timelock or co-signer enforcement.
2. The Squads multisig transfer is a manual deployment step that can be missed, delayed, or accidentally skipped.
3. Users who buy on the bonding curve trust that graduation proceeds correctly; a compromised key between graduation and AMM seeding can divert the full ~2000 SOL.
4. The graduation sweep and Squads transfer are not atomically coupled — there is a window where the curve is Graduated but authority remains a single key.

The finding is Tier 2 (not Tier 1) because:
- The Graduated constraint is sound and properly verified.
- The Squads governance path is documented, scripted, and proven on devnet.
- Post-Squads transfer, exploitation requires 2-of-3 key compromise + 15-minute window bypass.
- The vulnerable window is narrow in time (between graduation and AMM seeding, which is a single coordinated admin operation).

---

## Recommendations

1. **Immediate (pre-launch)**: Execute the `transfer_bc_admin` to the Squads vault PDA as part of Stage 3 of the mainnet deployment checklist, BEFORE calling `prepare_transition` on either curve. This eliminates the single-key window entirely.

2. **Consider atomic coupling**: Modify the deployment runbook so that `transfer_bc_admin` to Squads and the graduation sweep are performed in close proximity, ideally in the same Squads vault transaction batch if the SDK allows. This minimizes the gap between Squads transfer and fund movement.

3. **Operational monitoring**: Add an indexer alert on the `SolWithdrawn` and `TokenVaultClosed` events. Any firing of these events should be cross-referenced with the expected graduation timeline.

4. **Documentation clarity**: The mainnet deploy checklist (`Docs/mainnet-deploy-checklist.md`) should explicitly require that `BcAdminConfig.authority == Squads vault PDA` is verified BEFORE `prepare_transition` is called, not after.

5. **Long-term (low priority)**: Consider adding a `withdraw_destination` field to the `BcAdminConfig` that is set at initialization and cannot be changed post-facto. This would ensure graduation SOL can only ever flow to the AMM seeding wallet, not the admin's personal wallet, regardless of which key holds admin authority at graduation time. This would also make the intent explicit at the contract level.
