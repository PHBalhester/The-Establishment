# H054: Stale carnage_target After Expiry

**Status:** CONFIRMED (Benign in Current Code)
**Severity:** LOW
**Category:** State Machine / Data Cleanup

## Summary

When Carnage expires (either via `expire_carnage` or auto-expire in `consume_randomness`), the `carnage_pending` and `carnage_action` fields are cleared but `carnage_target` (0=CRIME, 1=FRAUD) retains its stale value from the expired execution attempt. This leaves state inconsistency where `carnage_target` refers to a Carnage action that is no longer pending. While the current codebase mitigates this by always checking `carnage_pending` before reading `carnage_target`, the stale value violates state cleanup principles and could become exploitable if code paths change.

## Code Evidence

**File:** `programs/epoch-program/src/instructions/expire_carnage.rs:87-90`

```rust
// Save values for event before clearing
let expired_target = epoch_state.carnage_target;
let expired_action = epoch_state.carnage_action;
...
// Clear pending state (SOL stays in vault)
epoch_state.carnage_pending = false;
epoch_state.carnage_action = CarnageAction::None.to_u8();
// carnage_target is NOT cleared here
```

**File:** `programs/epoch-program/src/instructions/consume_randomness.rs:122-127`

```rust
let expired_target = epoch_state.carnage_target;
let expired_action = epoch_state.carnage_action;
...
epoch_state.carnage_pending = false;
epoch_state.carnage_action = CarnageAction::None.to_u8();
// carnage_target is NOT cleared here (auto-expire path)
```

**Expected cleanup (missing in both locations):**
```rust
epoch_state.carnage_target = 0; // Clear to CRIME as default
```

## Verdict

**Currently benign, but poor state hygiene.**

All consumers of `carnage_target` in the current codebase check `carnage_pending == true` before reading the value:
- `execute_carnage_core` (carnage_execution.rs:152-160) checks `pending` before accessing `target`
- No other instruction references `target` directly

**Hypothetical exploit scenario:**
If a future code path were added that reads `carnage_target` without checking `carnage_pending`, it would observe stale data from a previous (expired) Carnage attempt. Example:
```rust
// Dangerous pattern (not in current code):
if epoch_state.carnage_target == 1 {  // Assumes target is valid
    // Do something with FRAUD token
}
```
If this path existed and Carnage had expired, the stale `carnage_target=1` would trigger logic for expired Carnage, creating confusion.

**Recommendation:** For defensive programming and state machine correctness, clear `carnage_target` when clearing `carnage_pending`:

**In expire_carnage.rs (line 89):**
```rust
epoch_state.carnage_pending = false;
epoch_state.carnage_action = CarnageAction::None.to_u8();
epoch_state.carnage_target = 0; // Clear stale target
```

**In consume_randomness.rs (line 127):**
```rust
epoch_state.carnage_pending = false;
epoch_state.carnage_action = CarnageAction::None.to_u8();
epoch_state.carnage_target = 0; // Clear stale target
```

This is a **LOW severity** issue (no current vulnerability) but classified as a state machine hygiene improvement.
