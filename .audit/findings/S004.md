# S004: Program Keypair Extraction from Git History

**Status:** CONFIRMED
**Severity:** CRITICAL
**Category:** Supply Chain, Key Management
**Inspired by:** H012 (POTENTIAL)

## Summary

H012 identified that program keypairs with full 64-byte secret keys are committed to the git repository and classified this as POTENTIAL (build-machine access required for the supply-chain attack). This strategy investigates the git history dimension specifically: even after keypairs are regenerated between phases, the old keypairs remain readable from any commit that touched the `keypairs/` directory. The investigation confirms that multiple generations of devnet program keypairs are directly extractable from git history, the current devnet keypairs (Phase 102 fresh deploy) are committed at HEAD, and — critically — the mainnet keypairs are fully gitignored and have NEVER appeared in the repository's history.

The mainnet vanity keypairs being gitignored separates the devnet exposure (real but low-consequence) from the mainnet exposure (not present in history — protected by gitignore). The risk profile is therefore more nuanced than H012's summary suggested.

## Analysis

### What Is In Git History: Confirmed Findings

**Current devnet program keypairs (HEAD, Phase 102):**

The most recent commit (`55254dd feat(102): clean-room devnet redeploy`) committed fresh program keypairs:

```
keypairs/amm-keypair.json        ← 64-byte secret key, CONFIRMED readable
keypairs/staking-keypair.json    ← 64-byte secret key, CONFIRMED readable
keypairs/epoch-program.json      ← 64-byte secret key, CONFIRMED readable
keypairs/bonding-curve-keypair.json
keypairs/tax-program-keypair.json
keypairs/transfer-hook-keypair.json
keypairs/vault-keypair.json
```

Direct verification: `git show 55254dd:keypairs/amm-keypair.json` returns the full 64-byte JSON array (confirmed).

The `.burned` suffix files (`keypairs/amm-keypair.json.burned`, etc.) in the working tree indicate previous keypair generations were intentionally rotated, but the previous keypairs still exist in their respective commits' history.

**Multiple historical generations visible in history:**

The following commits all added or modified program keypairs, meaning ALL generations are readable:
- `1914d28` (Phase 30): First reconciliation of all keypairs
- `31b0b60` (Phase 51): 5 fresh program keypairs generated
- `41b1fe0` (Phase 95): Full clean devnet deploy, all new keypairs
- `55254dd` (Phase 102): Latest clean-room deploy

Staking keypair from commit `1914d28` was directly extracted and confirmed to contain a full 64-byte secret key. This is the Phase 30 generation — no longer deployed on devnet, but its private key is permanently accessible to any git reader.

**Full git history spans 1,922 commits** — any of these historical keypairs represent program addresses that have been deployed to Solana devnet in the past. The upgrade authorities for those historical deployments may be of lower consequence (old devnet programs that are no longer active), but the pattern demonstrates that the repository's approach to keypair management has never separated secret material from git tracking.

### What Is NOT In Git History: Mainnet Keypairs Are Fully Protected

Mainnet vanity keypairs are explicitly gitignored (`keypairs/mainnet-*`):

```
# Mainnet keypairs — NEVER commit these
keypairs/mainnet-*
```

Verification:
- `git ls-files --error-unmatch keypairs/mainnet-crime-mint.json` → error (file not tracked)
- `git log --all --full-history -- "keypairs/mainnet-*"` → no output (never appeared in history)
- `git check-ignore -v keypairs/mainnet-crime-mint.json` → correctly matched by rule

The mainnet vanity keypairs (`cRiMEhAxoDhcEuh3Yf7Z2QkXUXUMKbakhcVqmDsqPXc`, `FraUdp6YhtVJYPxC2w255yAbpTsPqd8Bfhy9rC56jau5`, `pRoFiTj36haRD5sG2Neqib9KoSrtdYMGrM7SEkZetfR`) have NEVER been committed. The gitignore protection is working correctly for mainnet.

**The mainnet program keypairs** (for the 7 programs themselves — `keypairs/mainnet-amm-program.json`, etc.) are also gitignored via `keypairs/mainnet-*` and confirmed absent from history.

### Risk Assessment: Devnet History Exposure

**Devnet keypairs (all generations): exposed to any git reader.**

The consequence depends on whether the exposed devnet keypairs are still deployed. For each generation:

| Generation | Status | Risk |
|------------|--------|------|
| Phase 30 keypairs (`1914d28`) | Replaced in Phase 51/95/102 | LOW — old devnet addresses, inactive programs |
| Phase 51 keypairs (`31b0b60`) | Replaced in Phase 95/102 | LOW — old devnet addresses |
| Phase 95 keypairs (`41b1fe0`) | Replaced in Phase 102 | LOW-MEDIUM — may still have residual devnet state |
| Phase 102 keypairs (`55254dd` = HEAD) | **CURRENTLY DEPLOYED ON DEVNET** | MEDIUM — live devnet programs |

For the **currently deployed devnet programs**, anyone with read access to the repository can:
1. Extract the Phase 102 keypairs from HEAD
2. Derive the program addresses (already known from `Anchor.toml` and constants)
3. Potentially sign as the upgrade authority for devnet programs

However, the Phase 97 work transferred devnet upgrade authorities to a Squads multisig (`4SMcPtixKvjgj3U5N7C4kcnHYcySudLZfFWc523NAvXJ`). If that transfer completed successfully, the keypairs in `keypairs/` no longer represent the upgrade authority — they are only the program's address identity keypairs. The project memory notes "All 7 upgrade authorities burned (bugs during 97-02)" on devnet, which suggests the devnet state is irregular regardless.

**For mainnet: zero risk from git history.** The gitignore protection is sound.

### The H012 Primary Risk: Commit-Time Key Exposure vs. Build-Time Injection

H012's build-pipeline supply-chain attack requires local filesystem access to inject malicious addresses. That risk rating is POTENTIAL.

This strategy's distinct focus — git history extraction — has a different risk profile:
- **Devnet**: CONFIRMED directly exploitable from git history for currently deployed devnet programs (Phase 102 keypairs at HEAD). Any repo reader can extract keys.
- **Mainnet**: NOT vulnerable from git history. Mainnet keypairs are gitignored and have never appeared in the repository.

The critical distinction for mainnet launch readiness: **the devnet key exposure does not compromise mainnet** because:
1. Mainnet uses completely separate keypairs (different program addresses via fresh deploys)
2. Mainnet keypairs are gitignored
3. Mainnet upgrade authorities will be transferred to Squads before launch (reducing keypair sensitivity post-transfer)

### Residual Risk After Squads Transfer

After mainnet upgrade authorities are transferred to Squads:
- The program keypairs in `keypairs/mainnet-*` no longer sign upgrades
- They serve only as the program's on-chain address identity
- Extracting them post-Squads transfer gives an attacker knowledge of program addresses (already public) but no signing capability

The mainnet keypairs become less sensitive post-Squads. However, they should still be kept gitignored because:
1. Pre-Squads transfer, they hold upgrade authority
2. The gitignore rule is a permanent safeguard, not a conditional one
3. Even post-transfer, leaking them provides reconnaissance value to attackers

### Devnet Key Hygiene Issue

The devnet program keypairs being committed is a code hygiene issue that should be addressed for the following reasons:
1. **Habit formation**: Committing devnet keypairs normalizes a practice that would be catastrophic for mainnet.
2. **Contributor risk**: Any future contributor given repo access inherits all historical devnet program keys.
3. **Phase rotation cost**: Each phase regenerates keypairs and commits new ones, permanently expanding the "leaked key" surface in history.

The `.burned` suffix files in the working tree indicate that old keypairs were intentionally rotated but not purged from history. This is a process gap.

## Verdict

**CONFIRMED — CRITICAL severity** (for devnet key exposure); **NOT VULNERABLE** (for mainnet key exposure from git history).

The severity of CRITICAL applies specifically to the currently-deployed devnet programs: their full secret keys are readable from `git show HEAD:keypairs/amm-keypair.json` et al. by any reader of the repository. This is a complete key exposure for all active devnet program accounts.

For **mainnet**, the git history exposure risk is mitigated by the gitignore rule — the mainnet keypairs have never appeared in git and are correctly protected. This is the more security-relevant question for launch, and the answer is favorable.

The finding is rated CRITICAL overall because the devnet key exposure normalizes a practice that, if accidentally extended to mainnet (e.g., a developer forgets to check gitignore before committing), would be catastrophically irreversible. The devnet exposure also risks confusing the project's key management posture, especially as new contributors are onboarded.

**Does git history exposure compromise mainnet? No.** The mainnet keypairs are different from devnet, gitignored, and have never appeared in the repository.

## Recommendations

### Immediately (Pre-Launch)

1. **Move devnet keypairs to gitignore** (`keypairs/*.json`, but NOT `keypairs/devnet-wallet.json` if needed for CI). The program keypairs should never have been committed. Add them to `.gitignore` now to prevent new additions:
   ```
   keypairs/amm-keypair.json
   keypairs/staking-keypair.json
   keypairs/epoch-program.json
   keypairs/bonding-curve-keypair.json
   keypairs/tax-program-keypair.json
   keypairs/transfer-hook-keypair.json
   keypairs/vault-keypair.json
   ```

2. **Do not purge git history before mainnet launch** — the devnet keys are already public to anyone with repo access, and history purge carries risks of repo corruption. Prioritize mainnet security (which is already protected) over devnet hygiene cleanup.

3. **Rotate devnet keypairs after gitignoring** — generate new Phase 103 devnet program keypairs and keep them off git. This reduces the residual risk of the currently-exposed Phase 102 keys.

### Before/During Squads Migration

4. **Confirm Squads transfer completes for ALL 7 mainnet upgrade authorities before launch**. Once complete, the mainnet keypairs in `keypairs/mainnet-*` are no longer the highest-value secrets (they become address-identity only). Document this state transition.

5. **Verify no mainnet keypair has accidentally been staged or committed** before every mainnet deploy:
   ```bash
   git status keypairs/mainnet-* # should show nothing or "untracked" (gitignored)
   ```

### Long-Term

6. **Use a secrets manager** (Solana CLI default keystore with OS-level permissions, or HashiCorp Vault) for all production keypairs. The filesystem-committed JSON pattern is the root cause of this entire class of risk.

7. **Add a pre-commit hook** that rejects any commit containing 64-element JSON arrays (the signature of a Solana keypair). This prevents accidental mainnet keypair commits even if gitignore rules are misconfigured.
