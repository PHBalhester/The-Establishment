# H027: Bonding Curve Sybil Attack (Wallet Cap Bypass)

**Status:** CONFIRMED — KNOWN DESIGN LIMITATION (Not a Vulnerability)
**Severity:** INFORMATIONAL (downgraded from Tier 2)
**Category:** Access Control, Economic Design
**Source:** SOS Audit Phase 2 (Stronghold of Security)
**File:** `programs/bonding_curve/src/instructions/purchase.rs:134-141`

---

## Summary

The `MAX_TOKENS_PER_WALLET` cap of 20M tokens per wallet is enforced only at the individual wallet address level — it does not prevent a single actor from using multiple wallets (Sybil attack) to accumulate more than 20M tokens across the bonding curve. This is a per-wallet technical control, not a per-identity economic control. However, after full analysis, this is the expected and intentional design of a public, permissionless bonding curve: there is no on-chain mechanism to link identities, and the cap's purpose is to prevent a single transaction from draining disproportionate supply at low curve prices, not to enforce hard token distribution quotas.

A secondary finding is that the audit architecture documents (`.audit/ARCHITECTURE.md:393` and `.audit/INDEX.md:556, 572, 660, 664`) incorrectly describe the cap as "50 SOL cumulative" — the actual enforced limit is a **token balance cap of 20,000,000 tokens (20_000_000_000_000 base units)**, denominated in tokens, not SOL. This is a documentation error in the audit knowledge base, not in the production code.

---

## Analysis

### What the cap actually enforces

The cap is checked at two points in `purchase.rs`:

1. **Step 5 (pre-partial-fill):** `user_ata_balance + tokens_out <= MAX_TOKENS_PER_WALLET` — checked against the full calculated `tokens_out` before supply cap partial fill.
2. **Step 8 (post-partial-fill):** Re-checked against `actual_tokens` in case partial fill reduced the purchase.

The check reads the user's ATA balance at the time of the transaction and adds the proposed purchase to it. It does not track cumulative SOL spent, cumulative tokens ever held, or tokens across multiple wallets controlled by the same actor.

### MAX_TOKENS_PER_WALLET in context

- `MAX_TOKENS_PER_WALLET = 20_000_000_000_000` (20M tokens with 6 decimals)
- `TARGET_TOKENS = 460_000_000_000_000` (460M tokens)
- Cap fraction: 20M / 460M = **4.35% of total supply per wallet**

The cap limits a single wallet to 4.35% of the entire bonding curve supply. To acquire the full 20M tokens at the curve start (cheapest price, `P_START=450 lamports/human token`), a buyer would spend approximately:

`calculate_sol_for_tokens(0, 20_000_000_000_000)` — from the curve parameters this is approximately **~4.5 SOL** at the start of the curve (cheapest point). The "50 SOL cumulative" description in the audit docs appears to have been derived from a different, undocumented spec version and does not match the deployed constant.

### Can the cap be bypassed via Sybil?

Yes — trivially. An attacker with N wallets can each buy up to 20M tokens, accumulating N × 20M tokens total. At 23 wallets, an attacker could theoretically control 100% of supply (23 × 20M = 460M). The cost to buy the full curve is ~500 SOL; controlling a majority at the cheapest prices requires buying early across many wallets.

However, this is true of every permissionless token distribution mechanism on Solana. Bonding curves, IDOs, and fair launches all share this property. There is no on-chain identity layer that could prevent it without a KYC/whitelist system, which would contradict the protocol's permissionless design philosophy.

### Does the Sybil attack matter economically?

The economic model analysis in `.audit/context/09-economic-model.md:297-303` already notes this:

> "However, there IS a per-wallet cap of 20M tokens (MAX_TOKENS_PER_WALLET). Attackers could Sybil attack with multiple wallets to accumulate more than 20M tokens. The economic impact depends on whether graduation mechanics account for this."

After examining graduation mechanics:

1. **Post-graduation, tokens flow to the Tax Program AMM.** A whale accumulating via Sybil can dump those tokens on the AMM post-graduation, suppressing the initial price. However, the AMM buy/sell taxes (3-14%) and Transfer Hook whitelist provide structural resistance.

2. **The bonding curve fill is a positive outcome regardless.** Whether 23 whale wallets or 460 regular participants fill the curve, the protocol receives ~500 SOL in the SOL vault, distributes tokens, and graduates. The protocol's economic health at graduation is not materially affected by how many entities hold the supply.

3. **The wallet cap prevents the worst single-wallet concentration.** Without it, a single transaction from a single wallet could buy the entire 460M supply in one TX. The cap forces at least 23 separate transactions, adding friction and cost.

4. **There is a sell-back path.** Participants can sell tokens back to the curve (with 15% tax) while it is Active. A Sybil attacker that sells from wallet A could then re-buy from wallet B — but the 15% tax makes round-trip sell-and-rebuy economically unprofitable.

### The cap resetting on sell — a minor bypass amplifier

The wallet cap check reads the current ATA balance (`user_ata_balance = ctx.accounts.user_token_account.amount`) at transaction time. If an attacker sells their 20M tokens back to the curve, their ATA balance drops to 0, and they can buy 20M tokens again from the same wallet. This is noted in the code comments at `purchase.rs:284-288`:

```rust
// Note: If a user sells all tokens and buys again, the ATA balance is 0
// again and participant_count increments again. Per spec, this is acceptable
// -- participant_count is a convenience stat, not security-critical.
```

So a single wallet can bypass the lifetime cap by: buy 20M → sell all → buy 20M → sell all → repeat. This costs 15% tax on each sell leg. The maximum net position they can build this way is still 20M at any given time — the cap is a spot balance cap, not a lifetime accumulation cap.

### Documentation error: "50 SOL" cap description

Multiple audit index entries incorrectly state the cap as "50 SOL cumulative":

- `.audit/ARCHITECTURE.md:393`: "user wallet cap <= 50 SOL cumulative"
- `.audit/INDEX.md:556`: "per-wallet caps (50 SOL)"
- `.audit/INDEX.md:572`: "per-wallet cap (50 SOL)"
- `.audit/INDEX.md:660`: "per-wallet cap (50 SOL cumulative)"
- `.audit/INDEX.md:664`: "Per-wallet cap is cumulative (once 50 SOL invested, user cannot buy more)"

The actual code enforces a **token balance cap** (not a SOL cap). There is no SOL tracking per user in any on-chain state account (`CurveState` has no per-user SOL field; there is no `UserPurchaseRecord` account). This is a stale or misread description in the audit knowledge base. It does not represent a security issue in the production code, but it is a factual inaccuracy in the audit documentation.

## Code Evidence

```rust
// purchase.rs:134-141 — Step 5 wallet cap check
let user_ata_balance = ctx.accounts.user_token_account.amount;
require!(
    user_ata_balance
        .checked_add(tokens_out)
        .ok_or(CurveError::Overflow)?
        <= MAX_TOKENS_PER_WALLET,
    CurveError::WalletCapExceeded
);
```

```rust
// constants.rs:75-77
/// Maximum tokens any single wallet can hold per curve: 20M with 6 decimals.
/// 20,000,000 * 10^6 = 20,000,000,000,000.
pub const MAX_TOKENS_PER_WALLET: u64 = 20_000_000_000_000;
```

The `CurveState` struct (`state.rs`) contains no per-user SOL tracking field. There is no `UserPurchaseRecord` or `UserCap` PDA in the bonding curve program.

---

## Verdict

**NOT A VULNERABILITY.** The per-wallet cap is a technical anti-concentration measure that functions exactly as designed. Sybil bypass is inherent to all permissionless on-chain systems without identity layers. The intended protection — preventing a single-transaction sweep of the entire supply — is correctly enforced.

The finding is downgraded to **INFORMATIONAL** for two reasons:

1. The Sybil attack requires meaningful capital (~4.5 SOL per wallet to hit the cap at curve start, and the 15% sell tax deters the single-wallet roundtrip bypass).
2. The protocol's graduation mechanics function correctly regardless of how concentrated the distribution is. The downstream AMM taxes provide continued friction for concentrated holders.

The only actionable item from this investigation is the **documentation inaccuracy** regarding "50 SOL cumulative" in the audit knowledge base, which should not propagate to any final report or production documentation.

---

## Recommendations

1. **No production code changes required.** The cap implementation is correct for its stated purpose.

2. **Fix audit documentation inaccuracy.** The following files describe the cap as "50 SOL cumulative" — this should be corrected to "20M tokens (4.35% of supply) per wallet, current spot balance only":
   - `.audit/ARCHITECTURE.md:393`
   - `.audit/INDEX.md:556, 572, 660, 664`

3. **Clarify cap semantics in production documentation.** If the Docs or protocol specifications describe the cap, they should accurately state:
   - Cap is on **current token balance**, not lifetime SOL spent.
   - Cap resets to 0 if a user sells all tokens back to the curve.
   - Cap applies per wallet address, not per actor identity.

4. **(Optional, future consideration)** If the team later determines that stronger whale prevention is desirable for protocol health, options include: (a) tracking cumulative tokens ever received in a per-user PDA (higher cost, closes the sell-rebuy bypass), or (b) increasing the `MAX_TOKENS_PER_WALLET` documentation to clarify it is a soft anti-concentration measure rather than a hard whale prevention mechanism.
