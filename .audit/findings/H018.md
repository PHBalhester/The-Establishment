# H018: Carnage held_token Raw u8 Matching — Wrong Token Burn

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** State Machine, Token/Economic
**Origin:** Novel

---

## Summary

The hypothesis that `held_token` could be set to a value outside {0, 1, 2} — causing the `_` branch in burn/sell logic to silently skip burning and strand tokens forever — does not hold. The encoding path is provably closed. Every write site either uses the `Token` enum (which `from_u8()` validates before use) or writes a literal 0. The raw u8 matching is therefore an internal implementation detail with no exploitable gap, though it represents a minor code-quality concern (no type narrowing at read site).

---

## Analysis

### The Hypothesis's Core Fear

`burn_held_tokens` matches on `carnage_state.held_token` with:
```rust
let is_crime = match carnage_state.held_token {
    1 => true,
    2 => false,
    _ => return Ok(0),  // ← silent no-op if unexpected value
};
```

If `held_token` could ever be set to any value other than 0, 1, or 2, the `_` arm would silently return without burning, allowing the fund to accumulate holdings that can never be disposed of. This is the correct threat model. The question is whether that precondition can be satisfied.

### Encoding Path — Provably Closed

There is exactly one write site that sets a non-zero `held_token`:

```rust
// carnage_execution.rs:359
carnage_state.held_token = target.to_u8() + 1;
```

`target` is of type `Token` (an `#[repr(u8)]` enum with variants Crime=0 and Fraud=1). It is obtained immediately before this line at:

```rust
// carnage_execution.rs:147-148
let target = Token::from_u8(epoch_state.carnage_target)
    .ok_or(EpochError::InvalidCarnageTargetPool)?;
```

`Token::from_u8` is explicitly validated:
```rust
// enums.rs:29-35
pub fn from_u8(value: u8) -> Option<Token> {
    match value {
        0 => Some(Token::Crime),
        1 => Some(Token::Fraud),
        _ => None,
    }
}
```

The `.ok_or(...)` call propagates `InvalidCarnageTargetPool` and aborts the transaction if `carnage_target` holds any invalid byte. Therefore `target.to_u8()` is guaranteed to be in {0, 1}, making `target.to_u8() + 1` always in {1, 2}. Overflow is impossible because `to_u8()` returns 0 or 1 and u8 arithmetic `1 + 1 = 2` is not saturating — but the values are so small that u8 wrapping cannot be triggered.

### All Write Sites for held_token

| Location | Value Written | Reachable Via |
|---|---|---|
| `initialize_carnage_fund.rs:42` | `0` (None) | Init — once at deploy |
| `carnage_execution.rs:238` | `0` (None) | CarnageAction::Sell path after sell completes |
| `carnage_execution.rs:537` | `0` (None) | `burn_held_tokens()` after burn completes |
| `carnage_execution.rs:359` | `target.to_u8() + 1` → {1, 2} only | Execute path after successful buy |

There are no other write sites. The field is never written by external callers (it lives inside a PDA with seeds `["carnage_fund"]` that no external program holds a signer for) and is never deserialized from user-supplied data.

### The `_` Branch — Correct Sentinel Handling

The `_` arm in `burn_held_tokens` (and its mirror in `execute_sell_swap`) handles the `held_token == 0` (None) case. This is the correct and expected behavior: if `held_amount > 0` guard passes but `held_token` is somehow 0, returning `Ok(0)` silently is appropriate because the holdings are inconsistent and burning nothing is preferable to burning the wrong vault. In practice this state cannot arise because the only time `held_amount > 0` is after a successful buy, which always writes `held_token` to 1 or 2 atomically at line 359-360.

### Consistency Across All Use Sites

All functions that consume `held_token` use the same {1 → CRIME, 2 → FRAUD, _ → early return/fallback} pattern:

- `burn_held_tokens` (line 477): `match { 1 => true, 2 => false, _ => return Ok(0) }`
- `execute_sell_swap` (line 620): `match { 1 => true, 2 => false, _ => return Ok(()) }`
- Burn mint selection in `execute_carnage_core` (line 183): `if held_token == 1 { crime_mint } else { fraud_mint }` — note: this path is only reached inside `if held_amount > 0` within `CarnageAction::Burn`, so `held_token` must be 1 or 2 here; the `else` correctly catches FRAUD.
- `approve_delegate` (line 868): `if held_token == 1 { crime_vault } else { fraud_vault }` — same structural note as above; no guard needed because caller ensures `held_amount > 0`.

The only minor inconsistency is that `approve_delegate` uses `if/else` instead of a match with an explicit `_` guard, meaning if `held_token == 0` somehow reached it, it would silently approve the FRAUD vault. However, `approve_delegate` is only called from `CarnageAction::Sell` behind `if held_amount > 0`, and `held_amount > 0` with `held_token == 0` is an impossible state given the write-site analysis above.

**Primary write site** (`carnage_execution.rs:357-360`):

```rust
carnage_state.held_amount = tokens_bought;
```

**Validation gate before write** (`carnage_execution.rs:147-148`):



## Verdict

**NOT VULNERABLE.** The raw u8 matching in `burn_held_tokens` and `execute_sell_swap` is safe because the only non-clearing write site (`target.to_u8() + 1`) is gated by `Token::from_u8(...).ok_or(...)`, which enforces that `target` is always a valid `Token` variant before the addition. The resulting value is always in {1, 2}. The `_` branch is defensive dead code that handles a structurally impossible state.

There is one minor code-quality observation: the `approve_delegate` function uses `if held_token == 1 { ... } else { ... }` without an explicit guard for `held_token == 0`, unlike `burn_held_tokens` and `execute_sell_swap`. This is not exploitable given caller preconditions but creates asymmetric defensive depth across the three consumers.

---

## Recommendations

**[INFORMATIONAL — No Action Required for Security]**

1. **Consider type-narrowing at read time**: In `execute_carnage_core`, the local `held_token: u8` variable at line 166 could be replaced with `HeldToken::from_u8(carnage_state.held_token)` at the point of reading, propagating the validated enum to all downstream consumers. This would make the impossibility of invalid values explicit and discoverable rather than requiring the reader to trace every write site.

2. **Align `approve_delegate` guard style**: Add an explicit `held_token` guard matching the pattern in `burn_held_tokens` for structural consistency:
   ```rust
   let vault = match held_token {
       1 => accounts.crime_vault.to_account_info(),
       2 => accounts.fraud_vault.to_account_info(),
       _ => return Ok(()), // defensive — caller precondition ensures {1,2}
   };
   ```
   This makes the three consumers uniformly defensive and removes the implicit FRAUD fallback for `held_token == 0`.

3. **Invariant assertion (optional, devnet-only)**: A `debug_assert!(held_token == 1 || held_token == 2)` at the top of `burn_held_tokens` would surface any future regression immediately during testing without runtime cost in release builds.
