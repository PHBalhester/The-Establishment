# H015: Admin Authority Transfer to Wrong Address (Irreversible)

**Status:** CONFIRMED
**Severity:** HIGH
**Category:** Access Control, Key Management
**Origin:** KB (EP-068)

---

## Summary

All three authority transfer patterns in the protocol — AMM `transfer_admin`, Bonding Curve `transfer_bc_admin`, and Transfer Hook `transfer_authority` — are single-step: the new authority takes effect immediately upon transaction confirmation with no confirmation required from the new key holder. There is no two-step propose-and-accept pattern anywhere in the codebase. The sole guard in all three cases is rejection of `Pubkey::default()` to prevent accidental burns via this path. Any other mistyped or inaccessible address is accepted immediately and irreversibly.

---

## Analysis

### Pattern Present Across All Three Targets

Each transfer instruction follows the same structure:

1. Verify caller is the current admin (via `has_one` constraint or explicit `require!`)
2. Reject `Pubkey::default()` with `InvalidAuthority` / `Unauthorized` error
3. Overwrite the stored authority with the new value — immediately, atomically
4. Emit a log message
5. Return `Ok(())`

There is no intermediate pending state, no escrow of the new authority, no second transaction from the new key holder, and no time-delay window during which the transfer could be cancelled.

### Guard Analysis: What Is and Is Not Protected

| Guard | What it prevents | What it does NOT prevent |
|-------|-----------------|--------------------------|
| `new_admin != Pubkey::default()` | Accidental burn via wrong instruction | Typo to any other 32-byte address |
| `has_one = admin` / `require!(auth == Some(signer))` | Unauthorized caller (non-admin) | Admin calling with wrong destination |
| None | — | Paste of wrong multisig vault address |
| None | — | Clipboard hijack / address substitution |
| None | — | Fat-finger (one character off) |
| None | — | Sending to a key for which the private key is lost |

### Transfer Hook Structural Difference

The AMM and BC patterns use Anchor's `has_one` constraint at the accounts struct level to enforce that the signer matches the stored admin. The Transfer Hook's `TransferAuthority` accounts struct does **not** include a `has_one` or `constraint` on `whitelist_authority` — the authority match is checked inside the handler with an explicit `require!` at lines 30-33. This is functionally equivalent but provides no compile-time guarantee and is easier to silently drop during a refactor.

### Recovery Possibilities: None Exist On-Chain

Once `admin_config.admin` (AMM), `admin_config.authority` (BC), or `whitelist_authority.authority` (Hook) is set to an inaccessible pubkey:

- **AMM**: `transfer_admin` and `burn_admin` both require the stored `admin` as signer via `has_one = admin`. No one can sign as the new key → both instructions permanently fail. Pool creation via `initialize_pool` requires admin signer. The AMM pools already initialized continue to function for swaps (not admin-gated), but no new pools can ever be created again.
- **BondingCurve**: `transfer_bc_admin`, `burn_bc_admin`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, and `close_token_vault` all require the current authority. Loss of BC admin means: bonding curve cannot be started, transitions cannot be prepared, graduated SOL cannot be withdrawn by the team, and token vault rent cannot be recovered. This permanently strands SOL in graduated curve vaults.
- **Transfer Hook**: `transfer_authority`, `burn_authority`, and `add_whitelist_entry` all require the current authority. Loss of whitelist authority means the whitelist is permanently frozen in its current state — no new accounts can ever be whitelisted. This prevents adding new AMM pools, adding new liquidity venues, or modifying whitelist state in any way. It is functionally equivalent to a `burn_authority` but without the clean `None` terminal state — the authority is `Some(inaccessible_key)` which means `burn_authority`'s idempotency branch (lines 26-29) will NOT short-circuit, and the explicit authority check will fail.

### Upgrade Authority as Backstop: Partial

A program upgrade can change the instruction logic. However:
- The upgrade authority cannot directly modify on-chain PDA state without a new instruction
- A new `emergency_reset_admin` instruction could be added, but it would need to be carefully designed to not introduce new trust assumptions
- Upgrade authority transfer to Squads multisig (the plan per project memory) does not eliminate this risk — Squads can upgrade the program, but the social coordination overhead for an emergency upgrade is significant
- During the window between admin loss discovery and successful emergency upgrade, admin functions remain permanently blocked

## Code Evidence

### AMM — `programs/amm/src/instructions/transfer_admin.rs`

```rust
pub fn handler(ctx: Context<TransferAdmin>, new_admin: Pubkey) -> Result<()> {
    // Prevent accidental burn -- burn_admin exists for that purpose
    require!(
        new_admin != Pubkey::default(),
        AmmError::InvalidAuthority
    );

    let admin_config = &mut ctx.accounts.admin_config;
    let old_admin = admin_config.admin;

    admin_config.admin = new_admin;   // ← immediate, no confirmation

    msg!("Admin transferred from {} to {}", old_admin, new_admin);
    Ok(())
}

#[derive(Accounts)]
pub struct TransferAdmin<'info> {
    pub admin: Signer<'info>,

    #[account(
        mut,
        seeds = [ADMIN_SEED],
        bump = admin_config.bump,
        has_one = admin @ AmmError::Unauthorized,  // ← only protection
    )]
    pub admin_config: Account<'info, AdminConfig>,
}
```

### Transfer Hook — `programs/transfer-hook/src/instructions/transfer_authority.rs`

```rust
pub fn handler(ctx: Context<TransferAuthority>, new_authority: Pubkey) -> Result<()> {
    require!(
        new_authority != Pubkey::default(),
        TransferHookError::Unauthorized
    );

    let auth = &mut ctx.accounts.whitelist_authority;

    require!(
        auth.authority == Some(ctx.accounts.authority.key()),
        TransferHookError::Unauthorized
    );

    let old_authority = auth.authority.unwrap();
    auth.authority = Some(new_authority);   // ← immediate, no confirmation

    msg!("Authority transferred from {} to {}", old_authority, new_authority);
    Ok(())
}
```

## Operational Risk Assessment

### Risk Scenarios

**Scenario 1 — Manual Squads migration (planned)**
The project memory documents that all authorities are to be transferred to a Squads 2-of-3 multisig vault. The Squads vault PDA (`4SMcPtixKvjgj3U5N7C4kcnHYcySudLZfFWc523NAvXJ` on devnet) is derived and must be copy-pasted into the transfer transaction. A single character error or stale clipboard produces an irreversible loss of authority.

**Scenario 2 — Clipboard hijack / address substitution malware**
Address substitution malware is common in crypto contexts. It silently replaces the target pubkey in the clipboard. The signer (legitimate admin) confirms the transaction without noticing the substituted address. All three transfer instructions accept arbitrary 32-byte values.

**Scenario 3 — Compressed/truncated address display**
UI tools and CLI output frequently truncate public keys (e.g., `4SMcP...XTab`). If the full address is not verified before signing, a similar-prefix address could be used. Base58 visual similarity attacks (e.g., `O` vs `0`, `l` vs `1`) are not applicable in Base58 but similar-looking legitimate addresses exist.

**Scenario 4 — Lost key after successful transfer**
Transfer to the correct Squads vault address followed by loss of the multisig hardware wallet or key shard. The PDA stored in AdminConfig points to the Squads vault, which can no longer sign.

### Impact by Program

| Program | Authority Lost | Operational Impact | Financial Impact |
|---------|---------------|-------------------|-----------------|
| AMM | AdminConfig.admin | Cannot create new pools | No direct fund loss; existing pools unaffected |
| BondingCurve | BcAdminConfig.authority | Cannot start/complete curves; graduated SOL permanently stranded | HIGH — graduated SOL in vault becomes inaccessible |
| Transfer Hook | WhitelistAuthority | Whitelist frozen; no new accounts can be whitelisted | MEDIUM-HIGH — prevents protocol expansion; whitelist stuck at current state |

### Exploitation Difficulty

- **Attacker effort**: Negligible for clipboard attack (passive malware). Moderate for address substitution attack (needs to intercept transaction signing).
- **Trigger**: Must race with or cause the legitimate admin to execute a transfer. One-time events (Squads migration) are highest risk windows.
- **Detection**: Only detectable after the transaction is confirmed on-chain. No pre-commit validation in the current instruction set.

## Verdict

**CONFIRMED HIGH.** All three authority transfer patterns are single-step and irreversible for any `new_address != Pubkey::default()`. The `Pubkey::default()` rejection is the only guard. The planned Squads governance migration (documented in project memory as a near-term operation) creates a concrete, high-stakes opportunity window where this vulnerability could be triggered by clipboard malware, address substitution, or simple human error. The Bonding Curve case carries the highest financial impact because graduated SOL becomes permanently stranded. The Transfer Hook case creates permanent whitelist immutability without the clean `None` terminal state.

The finding is not theoretical: the project is actively preparing for authority transfers as part of the mainnet deployment sequence. This is a real operational risk that coincides with a known upcoming event.

---

## Recommendations

### Primary: Implement Two-Step Transfer Pattern

Replace the immediate transfer with a propose-and-accept pattern:

1. **propose_admin_transfer(new_admin: Pubkey)**: Current admin calls. Stores `pending_admin: Option<Pubkey>` in the config PDA. Does NOT change the active admin.
2. **accept_admin_transfer()**: New admin (the proposed address) calls and signs. Only then does `admin = pending_admin.take()`.

This ensures the new address is provably valid (signing key exists and is accessible) before the old admin loses control. The pending state can optionally expire after N slots to prevent stale proposals.

**State change required**: Add `pending_admin: Option<Pubkey>` to `AdminConfig`, `BcAdminConfig`, and add `pending_authority: Option<Pubkey>` to `WhitelistAuthority`.

### Secondary: Address Verification CLI Tool

Before the Squads migration (the highest-risk upcoming event), provide a read-only verification script that:
- Fetches the current on-chain admin state from all three PDAs
- Displays the full (non-truncated) pubkeys
- Verifies the intended new address (Squads vault) by fetching and displaying that account too

This does not fix the on-chain vulnerability but reduces human-error risk for the specific migration event.

### Tertiary: Upgrade Authority Backstop Instruction

Add a `recover_admin(new_admin: Pubkey)` instruction to each program gated exclusively by the BPF upgrade authority (via `program_data.upgrade_authority_address` check), bypassing the `has_one` constraint. This provides a recovery path without requiring a full program upgrade in the event of admin loss. Once Squads holds the upgrade authority, this recovery path goes through the timelock — acceptable for non-emergency recovery.

### Immediate Mitigation (Pre-Fix)

For the upcoming Squads authority transfer:
1. Verify the Squads vault PDA on-chain before constructing any transfer transaction
2. Construct and simulate the transfer transaction before signing
3. Use a dedicated signing environment (air-gapped or hardware wallet with full address display)
4. Transfer authorities one-at-a-time with on-chain verification between each step
5. Do NOT copy the destination address from documentation — re-derive it from the Squads program and multisig account address
