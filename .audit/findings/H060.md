# H060: Bonding Curve Refund Ordering Advantage

**Status:** POTENTIAL
**Severity:** LOW
**Category:** Arithmetic / Token Economics

## Summary

The bonding curve's `claim_refund` instruction computes proportional refunds using floor division: `refund_amount = floor(user_balance * refund_pool / total_outstanding)`. As each refund claim decreases both the refund pool (via SOL transfer) and the denominator (via token burn and `tokens_sold` decrement), the floor rounding dust from each claim's division can remain in the vault. This creates a small advantage for later claimers due to cumulative rounding dust accumulation.

## Code Evidence

- **File:** `programs/bonding_curve/src/instructions/claim_refund.rs:154-164`

```rust
// Step 6: Proportional refund with floor rounding (protocol-favored)
//
// floor(user_balance * refund_pool / total_outstanding)
// u128 intermediates prevent overflow.
let refund_amount_u128 = (user_balance as u128)
    .checked_mul(refund_pool as u128)
    .ok_or(CurveError::Overflow)?
    / (total_outstanding as u128);  // <-- Floor division (no checked_div equivalent for floor)
let refund_amount = u64::try_from(refund_amount_u128)
    .map_err(|_| error!(CurveError::Overflow))?;
```

And then:
```rust
// Step 8: Transfer refund SOL
**ctx.accounts.sol_vault.try_borrow_mut_lamports()? -= refund_amount;
**ctx.accounts.user.try_borrow_mut_lamports()? += refund_amount;

// Step 9: Update tokens_sold (denominator shrinks for next claimer)
curve.tokens_sold = curve
    .tokens_sold
    .checked_sub(user_balance)
    .ok_or(CurveError::Overflow)?;
```

- **Related:** `bonding_curve/src/instructions/claim_refund.rs:196-201` shows tokens_sold decrement

## Analysis

**The rounding scenario:**

Consider a failed curve with:
- Initial tokens_sold (denominator): 100M tokens
- Initial refund_pool (numerator): 500 SOL = 500,000,000,000 lamports
- First claimer: 30M tokens

**First claim:**
- refund_amount = floor(30M * 500B / 100M) = floor(150B) = 150B lamports
- vault transfer: 500B - 150B = 350B lamports remain in vault
- tokens_sold: 100M - 30M = 70M remain

**Second claimer:** 40M tokens (different size than first)
- refund_amount = floor(40M * 350B / 70M) = floor(200B) = 200B lamports (✓ fair)
- vault transfer: 350B - 200B = 150B remain
- tokens_sold: 70M - 40M = 30M remain

**Third claimer:** 30M tokens (last claimer, all remaining)
- refund_amount = floor(30M * 150B / 30M) = floor(150B) = 150B lamports
- vault transfer: 150B - 150B = 0 remain (exactly empty by design)
- tokens_sold: 30M - 30M = 0 (sentinel: no more claims possible)

**The edge case — three small claimers creating dust:**

Suppose instead:
- 33M, 33M, 34M token claimers from the same 100M / 500B pool

**Claim 1:** 33M
- refund = floor(33M * 500B / 100M) = floor(165B) = 165B
- Rounding dust: 0.3333B - 0B = 0.333B lamports lost to floor
- Vault: 335B, Denominator: 67M

**Claim 2:** 33M
- refund = floor(33M * 335B / 67M) = floor(165.223B) = 165B
- Dust: 0.223B
- Vault: 170B, Denominator: 34M

**Claim 3:** 34M (last)
- refund = floor(34M * 170B / 34M) = floor(170B) = 170B
- Dust: 0B
- Vault: 0B, Denominator: 0M

**Accumulated dust in vault:** 0 lamports (because last claimer gets exactly 170B)

**However, if claim sequence were different (claim 3 first as 34M):**
- refund = floor(34M * 500B / 100M) = floor(170B) = 170B
- Dust: 0B
- Vault: 330B, Denominator: 66M

**Then claim 1:** 33M
- refund = floor(33M * 330B / 66M) = floor(165B) = 165B
- Dust: 0B
- Vault: 165B, Denominator: 33M

**Then claim 2:** 33M
- refund = floor(33M * 165B / 33M) = floor(165B) = 165B
- Dust: 0B
- Vault: 0B

**Key insight:** The cumulative dust depends on the claim sequence. The scenario where dust accumulates in the vault is when many small claimers with high-fractional refund amounts go before a large final claimer. The large final claimer would potentially claim slightly more than their fair share.

**Worst case quantification (from context file H051 analysis):**

With TOTAL_FOR_SALE = 460M tokens and vault ~500 SOL = 500B lamports:
- Average price per token: 500B / 460M ≈ 1087 lamports
- Worst case dust per claim: 1086 lamports (one token's worth of rounding)
- With 1000 claims: ~1M lamports (~0.001 SOL) cumulative dust possible
- On mainnet, this is negligible in absolute terms

**The design choice:**

The bonding curve uses floor rounding on refunds, which is protocol-favorable. The spec (Bonding_Curve_Spec.md Section 8.8) explicitly acknowledges this: "pro-rata refund is computed using floor division to ensure protocol does not overpay." The dust accumulation is an accepted side effect.

## Verdict

**Potential, very low severity.** The refund ordering advantage exists mathematically but is bounded by the following factors:

1. **Absolute scale:** Max dust per claim ≤ TOTAL_FOR_SALE ≈ 1 lamport (negligible)
2. **Aggregate scale:** Max cumulative dust ≤ 1000 claims × 1 lamport ≈ 0.001 SOL (negligible)
3. **Design intent:** Floor rounding is intentional per spec; protocol-favorable dust is acceptable
4. **Sequence-dependent:** Advantage only manifests if small claimers precede a large final claimer; random ordering minimizes bias
5. **Incentive misaligned:** An attacker cannot orchestrate claim order (users claim independently); no profitable exploit exists

**Related finding:** H051 (from arithmetic agent) likely flagged the same pattern with the same conclusion.

**Mitigation:** Not required. The dust is negligible and the pattern is intentional. If future requirements demand exact proportional refunds (zero dust), a two-step mechanism could be implemented (request refund in TX1, claim approved amount in TX2), but this adds complexity for no material security benefit.
