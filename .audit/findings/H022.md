# H022: CPI Depth at 4/4 Hard Limit

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** CPI, Resource/DoS
**Origin:** RECHECK (H058)

## Summary

The Carnage swap execution chain reaches exactly Solana's 4-level CPI hard limit. The chain is correctly structured and functional today — there is no immediate exploitability. However, the architecture creates a maintenance fragility: any future CPI call added anywhere in the 5-program chain (Epoch, Tax, AMM, Token-2022, Transfer Hook) will cause all Carnage execution to fail at runtime with no compile-time warning. The finding from H058 is confirmed and refined with direct source inspection. No hidden CPI calls exist in the path today (logging via `emit!` is a log syscall, not a CPI; `msg!` is a pure syscall).

## Analysis

### Exact CPI Chain — Verified from Source

```
Depth 0: Crank TX → Epoch::execute_carnage_atomic / execute_carnage (entry point)
           [Pre-swap depth 0: burn_held_tokens (Token-2022 burn, no hook CPI)
                              wrap_sol_to_wsol (System transfer + sync_native)
                              approve (Token-2022 approve)]
  Depth 1: Epoch → Tax::swap_exempt          (invoke_signed, carnage_signer PDA)
    Depth 2: Tax → AMM::swap_sol_pool        (invoke_signed, swap_authority PDA)
      Depth 3: AMM → Token-2022::transfer_checked  (invoke_signed, pool PDA)
        Depth 4: Token-2022 → TransferHook::execute  (runtime CPI) ← SOLANA HARD LIMIT
```

Depth sourced from:
- `carnage_execution.rs:11-16` — documented comment block
- `swap_exempt.rs:10-16` — mirrored documented comment block
- `execute_swap_exempt_cpi()` at line 720 — single `invoke_signed` call at line 806
- `swap_exempt.rs:150-154` — single `invoke_signed` to AMM
- `amm/src/helpers/transfers.rs:119` — `invoke_signed` to Token-2022
- Token-2022 runtime → Transfer Hook (Solana runtime, not controlled by this codebase)

### Hidden CPI Audit — `emit!` and `msg!`

Both `emit!` and `msg!` appear inside the critical path programs. These are **not CPIs**:

- `msg!` — expands to `sol_log()` syscall. No CPI. Does not consume CPI depth.
- `emit!` (Anchor 0.32.1) — expands to `sol_log_data()` syscall. This is the standard Anchor event macro. It emits a program log entry with base64-encoded event data. **No CPI is generated.** This is confirmed by the absence of any `emit_cpi!` usage across the entire codebase (zero matches in all 7 programs).

`emit_cpi!` is the Anchor variant that generates a CPI to a dedicated event authority program — that macro is not used anywhere in this protocol.

**Confirmed `emit!` locations in the critical path:**
- `carnage_execution.rs:387` — `emit!(CarnageExecuted {...})` — after the swap CPI completes, at depth 0
- `swap_exempt.rs:158` — `emit!(ExemptSwap {...})` — after `invoke_signed` to AMM, still at depth 1
- `amm/src/instructions/swap_sol_pool.rs:338` — `emit!(SwapEvent {...})` — after token transfers, at depth 2

All `emit!` calls are placed **after** the invoke chain completes at each level, so they execute at their respective depth level as log syscalls, not nested CPIs. They have zero impact on CPI depth.

### Pre-Swap Operations at Depth 0

The following operations execute before the swap CPI chain begins, at depth 0:

| Operation | Function | CPI Target | Depth |
|-----------|----------|------------|-------|
| Token-2022 burn | `burn_held_tokens()` | Token-2022 (burn, no hook) | 0 |
| SOL→WSOL wrap | `wrap_sol_to_wsol()` | System::transfer + Token-2022::sync_native | 0 |
| Token-2022 approve | (inline in execute_carnage_atomic) | Token-2022::approve | 0 |

The `burn_held_tokens` function explicitly notes this: "Token-2022 burns do NOT trigger transfer hooks, so this doesn't consume CPI depth for the hook call." (carnage_execution.rs:455-458). The burn instruction uses discriminator `8u8` (Burn), which is correct — Token-2022 burn does not invoke the transfer hook extension.

### Transfer Hook — Zero Outgoing CPI

The Transfer Hook handler at `transfer-hook/src/instructions/transfer_hook.rs` contains:
- Zero `invoke` calls
- Zero `invoke_signed` calls
- Zero `CpiContext` usages
- Zero `emit_cpi!` calls

Its handler performs only: `require!` guards, `check_mint_owner()` (pure computation via `owner == TOKEN_2022_PROGRAM_ID`), `check_is_transferring()` (pure account data read), and `is_whitelisted()` (PDA derivation via `Pubkey::find_program_address`). All operations are pure computation. This is the architectural requirement — any CPI here would push the chain to depth 5 and break all Carnage execution.

### AMM — Only Token Transfers in Swap Path

The AMM's `swap_sol_pool.rs` uses `transfers.rs` helpers. The only CPI calls in the swap handler are:
1. `transfer_with_hook()` — `invoke_signed` to Token-2022 (depth 3 in Carnage path)
2. `transfer_sol()` — `CpiContext` to SPL Token for WSOL transfers (WSOL = SPL Token, no hook, so depth stops at 3)

No additional CPIs exist in the swap path. `emit!(SwapEvent {...})` at line 338 is a log syscall.

### Sequential vs. Nested: Why Two Transfers Are Safe

The AMM performs two transfer_checked calls per swap (input transfer + output transfer). For SOL pools:
- Input side (Carnage buy): WSOL → SPL Token `transfer` → depth 3 (no hook)
- Output side (Carnage buy): Token-2022 token → `invoke_signed` → depth 3 → Transfer Hook at depth 4

These are **sequential**, not nested. The depth stack unwinds after each call completes before the next begins. Both independently touch depth 4 but they do not compound.

### Fragility Surface

Any of these changes would cause all Carnage execution to fail at runtime:

| Change | Where | Effect |
|--------|-------|--------|
| Add any CPI inside Transfer Hook handler | `transfer_hook.rs` | Depth 5 — runtime failure |
| Add any CPI inside AMM `swap_sol_pool` beyond existing transfers | `swap_sol_pool.rs` | Depth 5 (if nested) or safe (if sequential to token transfers) |
| Add any CPI inside Tax `swap_exempt` beyond AMM | `swap_exempt.rs` | Depth 5 if called within swap chain |
| Wrap Epoch→Tax CPI in an intermediary program | Any refactor | Depth 5 at entry |
| Change `emit!` to `emit_cpi!` in any critical path file | Any of 3 files | Depth 5 |

### Enforcement Gap

There is **no compile-time enforcement** that the CPI depth remains within limits:
- No Rust attribute prevents adding CPI calls to the critical path files
- No linker check validates cross-program call depth
- Depth warning comments exist in `carnage_execution.rs:11-16` and `swap_exempt.rs:10-16` but NOT in `amm/src/instructions/swap_sol_pool.rs` or `amm/src/helpers/transfers.rs`
- No CI lint enforces this constraint

Integration tests exercise the full 4-deep chain and would catch depth violations at test time, but only if the test suite is run against the modified binary.

## Verdict

**CONFIRMED — MEDIUM severity.** The 4-level CPI chain is correctly architected and functioning today. No hidden CPI calls exist in the swap path (`emit!` is a log syscall, not a CPI; `emit_cpi!` is used nowhere). Pre-swap operations (burn, wrap, approve) correctly execute at depth 0 before the chain. The Transfer Hook is pure computation with zero outgoing CPIs — the architecturally correct implementation for this constraint.

The risk is entirely forward-looking: the protocol is one careless CPI addition away from silently breaking all Carnage execution. This creates a maintenance trap where developers making seemingly unrelated changes to any of 4 programs could break the Carnage path. The current comment-based warnings in 2 of 4 critical files are the sole enforcement mechanism. This is rated MEDIUM because (1) it is not currently exploitable, (2) the breakage would be self-inflicted during development, and (3) the existing test suite would catch it before deployment.

## Recommendations

1. **Add depth warning to AMM's `swap_sol_pool.rs`** — the existing warnings in `carnage_execution.rs` and `swap_exempt.rs` are absent from the AMM and Transfer Hook. A developer modifying `swap_sol_pool.rs` has no in-file warning that their change could break the Carnage path.

2. **Add depth warning to `amm/src/helpers/transfers.rs`** — this is where the actual `invoke_signed` to Token-2022 lives (depth 3 in Carnage path). A note here would complete the warning coverage.

3. **Add a CI lint rule** that flags any new `invoke`, `invoke_signed`, or `CpiContext` addition in the four critical-path files (`carnage_execution.rs`, `swap_exempt.rs`, `swap_sol_pool.rs`, `transfer_hook.rs`) and requires manual review acknowledgment.

4. **Guard against `emit_cpi!` migration** — if Anchor's event system is ever upgraded and `emit!` is replaced with `emit_cpi!` in the critical path files, the chain exceeds depth 4. The absence of `emit_cpi!` today is correct; document this explicitly.

5. **When Agave 3.0 ships** with 8-level CPI depth limit, update the "SOLANA LIMIT" comments to reflect the new runtime limit. At that point, the Carnage path will have 4 levels of headroom, removing this finding's forward risk.

6. **Integration test tripwire** — add a test that explicitly asserts the Carnage CPI chain completes (exercising depth 4), with a companion negative test that demonstrates a depth-5 chain fails. This creates a regression test that would fail before deployment if the chain is inadvertently deepened.
