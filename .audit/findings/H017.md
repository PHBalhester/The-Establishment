# H017: Duplicate Mutable Accounts in Carnage Execution Paths

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** Account Validation
**Origin:** KB (EP-009)

---

## Summary

The hypothesis posited that a caller could pass the same account for two different mutable positions in the 23-account Carnage instruction structs (e.g., `crime_pool == fraud_pool`), causing the second write to overwrite the first and corrupt pool state or break the k-invariant. After tracing every mutable account through the struct constraints, initialization logic, and the Solana runtime's duplicate-account detection, the protocol is **not vulnerable** to this class of attack. Three independent layers each individually prevent exploitation.

---

## Analysis

### Layer 1 — Anchor Constraint Binding via Stored Addresses

The two most dangerous potential aliases are `crime_vault`/`fraud_vault` (mutable Token-2022 accounts) and `crime_pool`/`fraud_pool` (mutable AMM pool accounts). Both are protected by binding constraints at the point of use.

**Vaults**: In both `ExecuteCarnageAtomic` and `ExecuteCarnage`, the vault accounts are constrained against the *stored* addresses in `CarnageFundState`:

```
constraint = crime_vault.key() == carnage_state.crime_vault @ EpochError::InvalidCarnageTargetPool
constraint = fraud_vault.key() == carnage_state.fraud_vault @ EpochError::InvalidCarnageTargetPool
```

These stored addresses are written once at initialization (`initialize_carnage_fund.rs:40-41`) from PDA-seeded accounts using distinct seeds (`CARNAGE_CRIME_VAULT_SEED` vs `CARNAGE_FRAUD_VAULT_SEED`). Since PDA seeds are collision-resistant, `carnage_state.crime_vault != carnage_state.fraud_vault` is guaranteed by construction. Any caller passing `crime_vault == fraud_vault` would fail the second constraint immediately.

**Pools**: The `crime_pool` and `fraud_pool` `AccountInfo` slots carry `owner = amm_program_id()` constraints and are CPI passthroughs to `Tax::swap_exempt`. At the AMM level (`swap_sol_pool.rs:388-394`), the pool is constrained with full PDA seeds:

```
seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
bump = pool.bump,
```

The CRIME/SOL pool and FRAUD/SOL pool have different `mint_b` values (CRIME vs FRAUD), so their PDA addresses are structurally distinct. A caller passing the same account for both `crime_pool` and `fraud_pool` would cause both swaps to execute against the CRIME pool — but the vault constraints at the AMM level (`vault_a.key() == pool.vault_a`, `vault_b.key() == pool.vault_b`) would cause the second swap to fail because the AMM's stored `vault_b` for the CRIME pool is the CRIME vault, not the FRAUD vault.

### Layer 2 — Solana Runtime Duplicate Mutable Account Detection

The Solana runtime explicitly forbids passing the same account in two writable positions within the same transaction's account list. Since Agave 1.14 this is a hard error at the runtime level, not a hint: a transaction containing `key_at_position_i == key_at_position_j` where both are marked writable will be rejected before any instruction executes. This is a protocol-level invariant documented in the Solana runtime source and applies regardless of what the program checks.

This layer alone defeats the duplicate-mutable-pool attack described in the hypothesis: even if all Anchor constraints were somehow bypassed, the transaction would be rejected by the runtime.

### Layer 3 — Semantic Separation in `execute_carnage_core`

Even if Layers 1 and 2 somehow failed, the execution logic in `carnage_execution.rs` reads pool reserves and vaults by semantic role (crime vs. fraud, held vs. target), not by position. The `execute_buy_swap` and `execute_sell_swap` functions each independently select the correct pool from the named `crime_pool`/`fraud_pool` fields based on `is_crime`. There is no position-based indexing that a duplicate account could exploit. The worst-case result of submitting `crime_pool == fraud_pool` is that both swaps touch the same pool, but this would be caught by the AMM's `VaultMismatch` constraint on the second CPI before any reserve corruption could occur.

### Remaining Observation (Not a Finding)

The `crime_pool_vault_a`, `crime_pool_vault_b`, `fraud_pool_vault_a`, `fraud_pool_vault_b` accounts have no explicit uniqueness constraints at the Epoch Program level beyond `#[account(mut)]` — they are documented as CPI passthroughs validated by Tax/AMM. This is an intentional design decision (noted in the struct comments) and is safe because:
1. The AMM's PDA-based pool constraint (`seeds = [POOL_SEED, mint_a, mint_b]`) ensures the pool owns specific vaults.
2. The AMM's `vault_a.key() == pool.vault_a` and `vault_b.key() == pool.vault_b` constraints prevent vault substitution.
3. Solana's runtime duplicate-mutable check applies to these accounts as well.

This is a documentation gap but not an exploitable vulnerability.

## Verdict

**NOT VULNERABLE.** The duplicate-mutable-account attack is defeated by three independent layers:

1. Anchor constraints bind `crime_vault` and `fraud_vault` to stored addresses that are structurally distinct by PDA seed construction.
2. The Solana runtime rejects transactions with duplicate writable accounts at the protocol level, before program logic executes.
3. The AMM enforces pool-vault integrity via PDA seeds and `has_one`-style vault constraints, which would catch any cross-pool alias that slipped past Layer 1 and Layer 2.

The k-invariant, vault balances, and pool state cannot be corrupted through this attack vector.

---

## Recommendations

No code changes required. The existing defense-in-depth is sufficient and correctly layered.

**Optional hardening (low priority):** Add an explicit uniqueness constraint at the Epoch level asserting `crime_pool.key() != fraud_pool.key()` to make the intent explicit and guard against future refactors that might remove the AMM-level protection. This would be a single `constraint = crime_pool.key() != fraud_pool.key()` in each instruction struct. Given the three existing layers, this is cosmetic rather than security-essential.
