# H052: Conversion Vault Truncation Loss

**Status:** CONFIRMED
**Severity:** LOW
**Category:** Token Economics / Arithmetic

## Summary

The conversion vault applies a fixed 100:1 conversion rate via integer division that loses up to 99 base units per conversion. A `compute_output` function at line 103 divides `amount_in / CONVERSION_RATE` with no remainder tracking. The `OutputTooSmall` guard prevents dust conversions but does not recover lost amounts, meaning fractional remainders are silently lost to the protocol.

## Code Evidence

**File:** `programs/conversion-vault/src/instructions/convert.rs:101-105`

```rust
if (*input_mint == *crime || *input_mint == *fraud) && *output_mint == *profit {
    // CRIME/FRAUD -> PROFIT: divide by 100
    let out = amount_in / CONVERSION_RATE;
    require!(out > 0, VaultError::OutputTooSmall);
    Ok(out)
}
```

**Constants Definition:**
- CONVERSION_RATE = 100 (hardcoded)
- No remainder variable

**Guard Location:** Line 104 checks `out > 0` but does not validate amount_in is divisible by 100.

## Verdict

The truncation loss is expected and documented behavior. The `OutputTooSmall` guard prevents conversions of amounts less than 100 base units, effectively requiring conversions to be in multiples of 100 (or accepting the loss for amounts 1-99 mod 100).

**Calculation of loss:**
- For a conversion of 150 units: output = 150 / 100 = 1 PROFIT (loss of 50 units)
- For a conversion of 199 units: output = 199 / 100 = 1 PROFIT (loss of 99 units)
- For a conversion of 100 units: output = 100 / 100 = 1 PROFIT (no loss)

This is a standard economic trade-off: the protocol collects dust from non-aligned conversions. The `OutputTooSmall` check ensures users do not attempt to convert trivially small amounts (< 100), but users who convert amounts like 150 or 199 lose the remainder.

**Is this exploitable?** No. The remainder stays in the user's account (not transferred), so the user simply receives fewer output tokens than if they had aligned their input to a multiple of 100. This is economically transparent.

**Recommendation:** Considered **LOW** severity and non-critical. Users should be aware via UI/docs that non-aligned conversions will lose fractional amounts. Current code is safe and intentional.
