# H044: Epoch Init Fix Verification

**Status:** CONFIRMED FIXED
**Severity:** N/A (RECHECK -- previously LOW)
**Category:** Access Control / Upgrade Admin
**Origin:** RECHECK (H021)

## Summary

The `initialize_epoch_state` instruction now enforces ProgramData upgrade-authority verification. The instruction struct contains `program` and `program_data` accounts with constraints requiring the payer to be the upgrade authority, aligning with the standard Anchor pattern and addressing the prior H021 defense-in-depth gap.

## Code Evidence

**File:** `programs/epoch-program/src/instructions/initialize_epoch_state.rs:100-133`

```rust
/// Accounts for initialize_epoch_state instruction.
#[derive(Accounts)]
pub struct InitializeEpochState<'info> {
    /// Payer for account creation rent.
    /// Typically the protocol deployer.
    #[account(mut)]
    pub payer: Signer<'info>,

    /// Global epoch state PDA.
    /// seeds = ["epoch_state"]
    #[account(
        init,
        payer = payer,
        space = EpochState::LEN,
        seeds = [EPOCH_STATE_SEED],
        bump,
    )]
    pub epoch_state: Account<'info, EpochState>,

    /// The Epoch program — used to look up its ProgramData address.
    #[account(
        constraint = program.programdata_address()? == Some(program_data.key())
    )]
    pub program: Program<'info, crate::program::EpochProgram>,

    /// ProgramData account — upgrade_authority must match payer.
    #[account(
        constraint = program_data.upgrade_authority_address == Some(payer.key())
    )]
    pub program_data: Account<'info, ProgramData>,

    /// System program for account creation.
    pub system_program: Program<'info, System>,
}
```

**Key lines:**
- Line 119-122: Validation that program.programdata_address matches program_data.key
- Line 125-128: Validation that program_data.upgrade_authority_address == Some(payer.key())

## Verdict

**CONFIRMED FIXED.** The initialization instruction now enforces ProgramData upgrade-authority verification, consistent with the pattern used in other protocol programs (AMM, Bonding Curve, Staking, Transfer Hook, Conversion Vault).

While the prior H021 finding correctly noted that `initialize_epoch_state` had no authority check and could be front-run, the practical impact was minimal because all initialization parameters are hardcoded constants or runtime-derived values -- the front-runner gains zero lasting privileges. The new fix provides proper defense-in-depth and aligns with the principle of least privilege.

This gating prevents:
1. Arbitrary parties from paying rent and "owning" the initialization (eliminated opportunity for privilege capture)
2. Defensive front-running attacks during deployment
3. Genesis slot manipulation to malicious values (though the prior risk was already bounded)
