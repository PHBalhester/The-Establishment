# H003: Combined Hook+BC Deployment Attack Fix

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** Access Control, Multi-Component
**Origin:** RECHECK (S006)

## Summary

Both fixes have been correctly applied. `initialize_authority` (Transfer Hook) and `initialize_bc_admin` (Bonding Curve) now independently require ProgramData upgrade authority verification, and all downstream BC admin instructions gate on `has_one = authority` against the stored `BcAdminConfig`. No shared state exists between the two programs' init paths, and there are no interaction effects that could be exploited between them.

## Previous Finding

Audit #2 (S006) rated this CRITICAL/HIGH confidence. The core attack was:

1. Attacker front-runs `initialize_authority` on Transfer Hook — any signer was accepted, making attacker permanent whitelist authority.
2. Attacker calls `prepare_transition` + `withdraw_graduated_sol` on Bonding Curve — any signer was accepted, draining ~2000 SOL from graduated curves.
3. Combined: permanent token freeze (hook authority) + SOL theft + inability to seed AMM pools = protocol kill shot.

Both vulnerabilities stemmed from bare `Signer<'info>` with no on-chain identity verification.

## Current Analysis

### Transfer Hook: `initialize_authority.rs`

The account struct now requires two additional accounts beyond the signer:

```rust
#[account(
    constraint = program.programdata_address()? == Some(program_data.key())
)]
pub program: Program<'info, crate::program::TransferHook>,

#[account(
    constraint = program_data.upgrade_authority_address == Some(signer.key())
)]
pub program_data: Account<'info, ProgramData>,
```

This is the standard ProgramData gate pattern. A transaction that omits `program` or `program_data` will fail Anchor's account resolution. A transaction that passes incorrect accounts will fail the `programdata_address()` cross-check. A transaction signed by anyone other than the current upgrade authority will fail the `upgrade_authority_address == Some(signer.key())` constraint. The `init` constraint on `whitelist_authority` ensures this can only succeed once.

### Bonding Curve: `initialize_bc_admin.rs`

Identical ProgramData gate pattern:

```rust
#[account(
    constraint = program.programdata_address()? == Some(program_data.key())
)]
pub program: Program<'info, crate::program::BondingCurve>,

#[account(
    constraint = program_data.upgrade_authority_address == Some(authority.key())
)]
pub program_data: Account<'info, ProgramData>,
```

Downstream admin instructions are chained through `BcAdminConfig` with `has_one = authority @ CurveError::Unauthorized`. Verified in `initialize_curve.rs:25` and `withdraw_graduated_sol.rs:34`. The attack path of calling `withdraw_graduated_sol` without holding the stored `authority` pubkey is blocked at the Anchor constraint level before any instruction logic executes.

### Interaction Effects Between the Two Programs

The two programs share no on-chain state. `WhitelistAuthority` is owned exclusively by the Transfer Hook program; `BcAdminConfig` is owned exclusively by the Bonding Curve program. Neither init reads or writes to the other's PDA. Deployment ordering (Hook init before BC init, or vice versa) has no effect on either program's security: each init is a fully independent, self-contained operation gated solely by the deployer's upgrade authority over that specific program.

The combined attack from S006 required both programs to be independently vulnerable. With both gates now enforced, the combined attack path is broken at both entry points simultaneously.

## Code Evidence

- `programs/transfer-hook/src/instructions/initialize_authority.rs:46-55` — ProgramData upgrade authority constraint
- `programs/bonding_curve/src/instructions/initialize_bc_admin.rs:44-56` — ProgramData upgrade authority constraint
- `programs/bonding_curve/src/instructions/initialize_curve.rs:22-27` — `has_one = authority` on BcAdminConfig
- `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs:31-36` — `has_one = authority` on BcAdminConfig

## Verdict

Both root causes identified in S006 are fixed. The combined attack is no longer possible because:

1. `initialize_authority` requires the Transfer Hook's upgrade authority — an attacker cannot front-run it without controlling the deployer key.
2. `initialize_bc_admin` requires the Bonding Curve's upgrade authority — same protection.
3. All downstream BC admin instructions require `has_one = authority` match against the stored `BcAdminConfig.authority` — even if `BcAdminConfig` were somehow initialized with an attacker's key, no downstream privileged instruction (including `withdraw_graduated_sol`) would accept a different signer.
4. No shared state between the two programs means there are no interaction effects to exploit.

The two programs are independently hardened and their combination introduces no new attack surface.

## Recommendations

None required for the vulnerability itself. One operational note: both programs now require upgrade authority to initialize, so if the upgrade authority is transferred to a Squads multisig before initialization, initialization will require a multisig proposal. The deployment pipeline (`scripts/deploy/deploy-all.sh`) should initialize both programs before transferring upgrade authority. This is consistent with existing deployment documentation.
