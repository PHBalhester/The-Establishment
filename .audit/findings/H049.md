# H049: get_current_price Silent Saturation on Division Failure

**Status:** POTENTIAL (Very Low Blast Radius)
**Severity:** LOW
**Category:** Arithmetic, Error Handling

## Summary

The `get_current_price()` function in `bonding_curve/src/math.rs:222-225` uses `.unwrap_or(0)` on division without checking if the division operand (TOTAL_FOR_SALE) could be zero. If `TOTAL_FOR_SALE` were somehow zero at runtime, the division `checked_mul(PRECISION).unwrap_or(0) / TOTAL_FOR_SALE` would panic with division-by-zero. More broadly, the function silently returns 0 if the multiplication overflows, masking the error. Since the function is **used only for events and display** (not financial calculations), an incorrect price of 0 would not cause token loss but would confuse off-chain observers.

## Code Evidence

**File:** `programs/bonding_curve/src/math.rs:217-226`
```rust
pub fn get_current_price(tokens_sold: u64) -> u64 {
    let price_range = P_END - P_START; // 1275
    // Scale by PRECISION to avoid losing precision in the division
    let progress = (tokens_sold as u128)
        .checked_mul(PRECISION)
        .unwrap_or(0)                    // <-- H049: Silent zero on overflow
        / TOTAL_FOR_SALE;                // <-- H049: Division by constant TOTAL_FOR_SALE
    let price = P_START + price_range * progress / PRECISION;
    u64::try_from(price).unwrap_or(u64::MAX)
}
```

**Constants:** `programs/bonding_curve/src/constants.rs` (inferred from context)
```rust
const TOTAL_FOR_SALE: u128 = 460_000_000_000_000;  // 460M tokens with 12 decimals
const PRECISION: u128 = 1_000_000_000_000;         // 1e12
```

**Usage Locations:**
1. `bonding_curve/src/instructions/sell.rs:334` — Emit event
2. `bonding_curve/src/instructions/purchase.rs:10` — Emit event

## Vulnerability Analysis

### Division-by-Zero Risk

**Question:** Can TOTAL_FOR_SALE be zero at runtime?

**Answer:** NO (provably safe)

TOTAL_FOR_SALE is a hardcoded constant defined in `constants.rs` as 460e12. It is never written to state, never parameterized, and cannot be changed without a program upgrade. Runtime division-by-zero is not possible.

### Silent Zero on Multiplication Overflow

**Question:** Can `tokens_sold * PRECISION` overflow u128?

**Conditions for overflow:**
- `tokens_sold`: max = TOTAL_FOR_SALE = 460e12 (at curve completion)
- `PRECISION`: 1e12
- Product: 460e12 * 1e12 = 460e24

**u128 capacity:** u128::MAX ≈ 3.4e38

**Result:** 460e24 << 3.4e38, so **no overflow is possible**.

The `.unwrap_or(0)` will never execute under normal operation.

### What Happens if `unwrap_or(0)` Executes?

If the multiplication somehow overflowed (which it cannot), the function would return:
```
price = P_START + price_range * 0 / PRECISION = 450
```

So `get_current_price()` would return 450 lamports (the minimum price at tokens_sold=0). This is semantically incorrect (should return current price), but affects only event emission, not protocol state.

### Blast Radius

If `get_current_price` returned 0 (or any wrong value):
- ✅ No tokens lost or incorrect transfers (financial logic unaffected)
- ✅ No state corruption (event does not modify blockchain state)
- ❌ Off-chain systems would see misleading event data
- ❌ UIs/indexers would display wrong bonding curve price

**Severity: INFORMATIONAL** — The defect would confuse observers but not cause protocol failure.

## How This Could Happen in Practice

Under what operational scenario would `get_current_price` return an incorrect value?

1. **Multiplication overflow:** BLOCKED by domain constraint (460e12 * 1e12 < u128::MAX)
2. **Division by zero:** BLOCKED by constant definition
3. **Code change during upgrade:** If TOTAL_FOR_SALE were changed and PRECISION were not adjusted proportionally, the function could return wrong values. But this would require a program upgrade, which is admin-gated and auditable.

**Conclusion:** This is not an exploitable vulnerability. It's a code quality issue (poor error handling) with a very low blast radius.

## Comparison to Proper Error Handling

The `claim_refund` handler uses the same proportional math **but with proper guarding:**

**File:** `bonding_curve/src/instructions/claim_refund.rs:159-164`
```rust
let refund_amount_u128 = (user_balance as u128)
    .checked_mul(refund_pool as u128)
    .ok_or(CurveError::Overflow)?
    / (total_outstanding as u128);
let refund_amount = u64::try_from(refund_amount_u128)
    .map_err(|_| error!(CurveError::Overflow))?;
```

This function:
1. Returns a Result, not silently masking errors
2. Validates the division denominator (total_outstanding is checked in line 144: `require!(total_outstanding > 0, ...)`)
3. Uses proper `.ok_or()` pattern instead of `.unwrap_or()`

`get_current_price` should follow the same pattern.

## Verdict

**POTENTIAL — Very Low Risk**

The error masking is present (using `.unwrap_or(0)` on division), but:
1. Under normal operation, the fallback never executes (overflow is impossible by domain constraint)
2. Even if the fallback did execute, the impact is informational only (incorrect event price)
3. Financial calculations are unaffected (uses separate functions with proper error handling)

**This is a code quality / consistency issue**, not a security vulnerability. The finding is valid as a reminder to use proper error handling patterns throughout the codebase.

## Recommendation

Refactor `get_current_price` to return Result and handle errors explicitly:

```rust
pub fn get_current_price(tokens_sold: u64) -> Result<u64, ArithmeticError> {
    let price_range = P_END - P_START;
    let progress = (tokens_sold as u128)
        .checked_mul(PRECISION)
        .ok_or(ArithmeticError::Overflow)?
        / TOTAL_FOR_SALE;
    let price = P_START + (price_range as u128) * progress / PRECISION;
    u64::try_from(price)
        .map_err(|_| ArithmeticError::CastOverflow)
}
```

Update event emission sites to handle the Result:
```rust
// In sell.rs:334
let current_price = get_current_price(ctx.accounts.curve_state.tokens_sold)
    .unwrap_or(0);  // Now explicit: "if price calc fails, default to 0"
emit!(SellEvent {
    tokens_sold: ctx.accounts.curve_state.tokens_sold,
    current_price,
});
```

Or better, propagate the error and let the instruction fail cleanly rather than emitting an event with invalid data.
