# H059: Token-2022 Transfer Hook Reentrancy Check

**Status:** CONFIRMED
**Severity:** LOW
**Category:** External Dependencies / Security

## Summary

The transfer hook program correctly implements the `check_is_transferring()` validation to verify it is being invoked from a legitimate Token-2022 `transfer_checked` context. The check reads the `transferring` flag extension on the source token account and rejects direct hook invocation attempts. This prevents a reentrancy attack where an attacker could call the hook instruction directly without going through Token-2022.

## Code Evidence

- **File:** `programs/transfer-hook/src/instructions/transfer_hook.rs:88-90, 132-152`

```rust
// Validation 3: Transferring flag check (security)
// SECU-01: Prevents direct hook invocation attack
check_is_transferring(&ctx.accounts.source_token.to_account_info())?;

/// Check that we're being called from a legitimate Token-2022 transfer.
///
/// The transferring flag is set by Token-2022 before calling the hook and
/// unset after the hook returns. If this flag is not set, someone is trying
/// to invoke the hook directly (attack vector).
fn check_is_transferring(source_token: &AccountInfo) -> Result<()> {
    let account_data = source_token.try_borrow_data()?;
    let account = PodStateWithExtensions::<PodAccount>::unpack(&account_data)
        .map_err(|_| TransferHookError::DirectInvocationNotAllowed)?;

    let extension = account.get_extension::<TransferHookAccount>()
        .map_err(|_| TransferHookError::DirectInvocationNotAllowed)?;

    if !bool::from(extension.transferring) {
        return err!(TransferHookError::DirectInvocationNotAllowed);
    }

    Ok(())
}
```

- **Validation order:** Lines 78-109 show the four-step defense-in-depth sequence.

## Analysis

**How the check works:**

1. **Flag lifecycle:** Token-2022's `transfer_checked` sets `source_token.transferring = true` before invoking the hook CPI
2. **Hook execution:** The hook program reads the flag via `PodStateWithExtensions::unpack()` and `get_extension::<TransferHookAccount>()`
3. **Flag unset:** After the hook returns, Token-2022 unsets `source_token.transferring = false`
4. **No overlap:** The flag is true only during the hook's execution window

**Defense-in-depth layers:**

1. **Transferring flag check** (H059): Rejects direct instruction invocation
2. **Mint owner check** (line 86): Verifies mint is owned by Token-2022 (defense-in-depth redundancy)
3. **Zero amount check** (line 80): Rejects zero-amount transfers (cheapest fail-fast)
4. **Whitelist check** (lines 96-109): Enforces access control (business rule)

**Reentrancy scenario analysis:**

**Direct hook invocation (prevented):**
- Attacker calls `transfer_hook` instruction directly
- source_token.transferring flag is never set by Token-2022
- `check_is_transferring()` reads flag as false
- Instruction rejected with `DirectInvocationNotAllowed`

**Cross-program reentrancy via transfer_checked (safe):**
- User calls `transfer_checked` on Token-2022
- Token-2022 sets transferring=true, invokes hook
- Hook can call other programs (staking, AMM, etc.)
- Those programs cannot call transfer_checked on the same source_token in the same instruction
  - Even if they could, Token-2022 would see transferring=true during nested transfer_checked call
  - Token-2022's own logic would reject the nested transfer (by design)
- After nested operations, control returns to hook
- Hook returns to Token-2022
- Token-2022 unsets transferring=false, completes transfer_checked

**Same-instruction callback reentrancy (safe):**
- Hook program is invoked mid-transfer with transferring=true
- Hook calls back to itself directly? Impossible (Solana runtime prevents self-calls at exact same instruction level)
- Hook calls another program that calls transfer_hook? That program would see transferring=true but cannot call transfer_hook again because transfer_hook is already on the stack

**Cross-program direct call to hook (prevented):**
- Another program calls `transfer_hook` instruction via CPI
- source_token.transferring is false (never set to true outside Token-2022)
- `check_is_transferring()` rejects

## Verdict

**Confirmed, no vulnerability.** The transfer hook correctly implements the Token-2022 security contract. The `transferring` flag is a runtime-enforced mechanism provided by Token-2022; the hook's check of this flag is the correct and only documented way to prevent direct invocation.

The implementation matches the SPL Transfer Hook specification (Section 7.4 / security requirements). No reentrancy issues exist because:

1. Direct invocation is blocked by the flag check
2. Cross-program reentrancy is prevented by Token-2022's own constraints
3. Same-instruction recursion is prevented by Solana's call stack model

**Code quality:** The check correctly handles error cases (account unpacking failure, extension missing) and fails closed with appropriate error codes. The check is placed early in validation order (step 3 of 4), optimizing for fail-fast evaluation.

**Minor observation:** The `check_is_transferring` function uses `try_borrow_data()` and unpacks the Token-2022 account layout directly. This is correct per spec and matches the reference implementation. No simplification is possible without removing security depth.
