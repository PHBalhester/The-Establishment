# H050: Bonding Curve Sell Tax u64 Overflow Window

**Status:** NOT VULNERABLE
**Severity:** LOW (Documentation Only)
**Category:** Arithmetic

## Summary

The bonding curve sell tax calculation at `bonding_curve/src/instructions/sell.rs:192-197` uses `sol_gross.checked_mul(SELL_TAX_BPS)` with u64 arithmetic (not u128 intermediates). The multiplication could theoretically overflow u64 if `sol_gross` and `SELL_TAX_BPS` were large enough. However, analysis shows the calculation is safe by domain constraints: max `sol_gross` is ~500 SOL (500e9 lamports), max `SELL_TAX_BPS` is 1500 (hardcoded), giving max product 7.5e14, which is well within u64::MAX (≈1.8e19). The parameters are hardcoded and not runtime-configurable, so overflow is provably impossible.

## Code Evidence

**File:** `programs/bonding_curve/src/instructions/sell.rs:192-197`
```rust
let tax = sol_gross
    .checked_mul(SELL_TAX_BPS)
    .ok_or(CurveError::Overflow)?
    .checked_add(BPS_DENOMINATOR - 1)
    .ok_or(CurveError::Overflow)?
    / BPS_DENOMINATOR;
```

**Constants:** (inferred from architecture and previous audits)
- `SELL_TAX_BPS` = 1500 (hardcoded, non-configurable)
- `BPS_DENOMINATOR` = 10000
- Target SOL capacity of bonding curve ≈ 500 SOL

## Vulnerability Analysis

### Maximum Product Calculation

**Bounds on `sol_gross`:**

The bonding curve is designed to distribute 460M tokens (TOTAL_FOR_SALE) for approximately 500 SOL (TARGET_SOL). The sell instruction can only sell back to the curve what has been purchased from it.

**Maximum `sol_gross` value:**
- At curve completion: full 460M tokens have been sold
- Reverse integral to recover SOL: approximately 500 SOL
- Per instruction limit: unknown (likely per-transaction limit or per-wallet limit), but conservatively at most 500 SOL
- **Absolute maximum `sol_gross`:** 500 SOL = 500e9 lamports = 5e11 lamports

**Bounds on `SELL_TAX_BPS`:**
- Value: 1500 (fixed constant in bonding_curve/src/constants.rs)
- Never changes at runtime (not a parameter)

**Maximum Product:**
```
sol_gross * SELL_TAX_BPS
= 5e11 * 1500
= 7.5e14 lamports
```

**u64 capacity:**
```
u64::MAX = 2^64 - 1 ≈ 1.844e19
```

**Comparison:**
```
7.5e14 << 1.844e19  (product is ~24,000x smaller than max)
```

### Conclusion

The `checked_mul` will **never overflow** under normal operation. The safe margin is enormous (24,000x).

### Parametrization Risk Analysis

**Question:** Could SELL_TAX_BPS or TARGET_SOL be parameterized in a future version?

**Current state (v1.3+):**
- `SELL_TAX_BPS` = hardcoded constant in bonding_curve/src/constants.rs, no admin configuration
- Target SOL is implicit in the curve parameters (TOTAL_FOR_SALE, P_START, P_END), not configurable

**If SELL_TAX_BPS were ever increased:**
- To cause overflow: `sol_gross * new_tax_bps > u64::MAX`
- At max 500 SOL input: `5e11 * new_tax_bps > 1.844e19`
- Required `new_tax_bps > 36,880`
- This would be a ~25x increase (1500 → 36,880+)

**Mitigation:** If SELL_TAX_BPS is ever made configurable (via admin parameter), it should:
1. Use u128 intermediates for the multiplication
2. Add validation: `tax_bps <= REASONABLE_MAX` (e.g., 5000 = 50%)

**Current code does not require this change** because the parameter is hardcoded.

## Defense-in-depth Check

The code already uses `.checked_mul(...).ok_or(CurveError::Overflow)?`, which is the correct pattern for protecting against overflow. Even though overflow is not possible here, the code is already defensively designed.

The next operation also uses `.checked_add()`, which is correct (ceil rounding requires adding `BPS_DENOMINATOR - 1` before division, and this addition could theoretically overflow if `checked_mul` returned near u64::MAX, but it doesn't).

## Comparison to Best Practices

**Current approach:** u64 checked arithmetic with domain constraint justification
**Best practice:** Use u128 intermediates for financial arithmetic

While the current code is **safe by bounds**, the SEI standard and industry best practices recommend:
```rust
let tax = ((sol_gross as u128)
    .checked_mul(SELL_TAX_BPS as u128)
    .ok_or(CurveError::Overflow)?
    .checked_add(BPS_DENOMINATOR as u128 - 1)
    .ok_or(CurveError::Overflow)?
    / BPS_DENOMINATOR as u128) as u64;
```

This eliminates the need to reason about bounds and makes future parameter changes safer.

## Verdict

**NOT VULNERABLE — Safe by Bounds**

The calculation is provably safe under current parameters (max 7.5e14 << u64::MAX of 1.8e19). The `checked_mul` guards against the theoretical overflow, and the guard will never trigger. No code change is required for security.

**Recommended enhancement:** Use u128 intermediates for consistency with other financial math in the codebase (tax_math.rs, amm/math.rs both use u128), but this is a hygiene improvement, not a bug fix.
