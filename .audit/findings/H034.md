## Summary

A user can include multiple Tax Program swap instructions (e.g., `swap_sol_buy` then `swap_sol_sell`) in the same Solana transaction. Because the AMM's reentrancy guard (`pool.locked`) is **set and cleared within each instruction**, it is `false` again when the next instruction in the same transaction begins. Each instruction therefore sees pool state — reserves, tax rates, locked flag — **as modified by all preceding instructions** in that transaction. This creates a potential vector for atomic intra-protocol arbitrage. However, due to the protocol's 50% minimum output floor and the constant-product reserve mechanics, the practical profit from such composability is highly constrained. The behavior is standard Solana instruction composability, not a reentrancy bug.

---

## Analysis

### How Solana Handles Multi-Instruction Transactions

On Solana, all instructions in a transaction share the same account state within a transaction's execution context. Account data written by instruction N is immediately visible to instruction N+1 in the same transaction. This is by design — Solana processes instructions sequentially within a transaction, and the runtime commits account state only after the entire transaction succeeds or rolls the full transaction back on failure.

This is distinct from reentrancy: reentrancy would require an instruction to re-enter itself (or a callee to call back into the caller) within a single instruction's execution frame. The AMM's `pool.locked` guard correctly prevents that. But **cross-instruction composability** is a separate property entirely; `pool.locked` provides no protection here because it is cleared at line 334 before the instruction returns, leaving it `false` by the time the Solana runtime hands control to the next instruction.

### The Reentrancy Guard Does Not Protect Against Multi-Instruction Composability

From `programs/amm/src/instructions/swap_sol_pool.rs`:

```rust
// Line 84: Set reentrancy guard
ctx.accounts.pool.locked = true;

// ... full swap execution ...

// Line 334: Clear reentrancy guard
ctx.accounts.pool.locked = false;
```

The Anchor constraint on the pool account is:
```rust
// Line 393:
constraint = !pool.locked @ AmmError::PoolLocked,
```

Anchor evaluates constraints at **account deserialization time** — before the handler body runs. For instruction N+1, the pool account is re-deserialized from on-chain state, which by the time instruction N+1 executes has `locked = false` (cleared by instruction N). The guard therefore enforces nothing across instruction boundaries; it only prevents CPI-level reentrancy within a single instruction invocation.

### What a Multi-Swap Transaction Could Achieve

A user could construct a transaction with:
- **IX1:** `swap_sol_buy` (SOL → CRIME, pool reserves updated: SOL reserve ↑, CRIME reserve ↓)
- **IX2:** `swap_sol_sell` (CRIME → SOL, pool now sees IX1's updated reserves)

Between IX1 and IX2, the pool has:
- More SOL in reserve (reserve_a ↑ after IX1)
- Fewer CRIME tokens (reserve_b ↓ after IX1)

IX2 (selling CRIME back for SOL) now executes against a pool that has moved against the user: the CRIME price in SOL has increased (fewer CRIME available relative to more SOL). A constant-product AMM price always moves adversely for a round-trip, so a naive buy-then-sell in the same pool would guarantee a loss (before even considering the buy tax on IX1 and sell tax on IX2).

The scenario where multi-swap composability could be profitable requires **different conditions between the two instructions**. The most plausible scenario within this protocol is:

1. **Cross-pool composability:** Buy CRIME via `swap_sol_buy(is_crime=true)` then immediately sell FRAUD via `swap_sol_sell(is_crime=false)`. The two pools are independent (`CRIME/SOL` vs `FRAUD/SOL`), so there is no shared reserve update between the two instructions. However, this is simply executing two independent swaps atomically — not an exploit.

2. **Cross-epoch rate arbitrage (theoretical):** If a new epoch's tax rates come into effect between IX1 and IX2 within the same transaction — this is not possible because `consume_randomness` is a separate transaction and cannot be interleaved within another transaction's instruction sequence. Tax rates are fixed for the duration of any transaction.

3. **Wash-trade to front-run a known price move:** A user with foreknowledge of a large pending trade could buy in the same transaction, let the external large trade execute (in a subsequent transaction), then profit on the price impact. This is a standard MEV pattern external to this protocol and is not unique to the multi-swap composability here.

### The 50% Output Floor as a Mitigant

The Tax Program enforces `MINIMUM_OUTPUT_FLOOR_BPS = 5000` (50%) on all user-facing swaps:

```rust
// swap_sol_buy.rs:106-111
let output_floor = calculate_output_floor(sol_reserve, token_reserve, sol_to_swap, MINIMUM_OUTPUT_FLOOR_BPS)
    .ok_or(error!(TaxError::TaxOverflow))?;
require!(
    minimum_output >= output_floor,
    TaxError::MinimumOutputFloorViolation
);
```

This floor is computed from the **current pool reserves at the time of each instruction**. If IX1 significantly moves the reserves, IX2's floor will be recalculated against the post-IX1 state. The floor does not prevent composability but does ensure that neither instruction can be used to construct a zero-slippage extraction.

### Tax Cost Analysis

Any buy-then-sell round-trip in the same transaction incurs:
- **Buy tax:** 3–14% on SOL input (deducted before AMM CPI)
- **Sell tax:** 3–14% on SOL output (deducted after AMM CPI)
- **Two × 1% AMM LP fee** (one per instruction)
- **Adverse price impact** from the round-trip itself (constant-product mechanics guarantee worse rates on the return leg)

At minimum tax rates (both buy and sell at 3%), the combined tax cost is approximately 6% on the round-trip, plus two LP fees (2%), plus price impact. This means round-trip profitability requires external price movement of at least ~8%+ to overcome costs. Such movement cannot be self-created within a single transaction (the user can't move their own price 8% in their favor by sandwiching their own transaction).

### Verdict on Exploitability

Multi-swap same-transaction composability is **standard Solana behavior** and does not constitute a protocol vulnerability. The combination of:
1. Constant-product AMM mechanics (round-trips are always net-negative)
2. Buy + sell taxes (6–28% combined cost floor)
3. AMM LP fees (2% combined)
4. 50% minimum output floor (prevents zero-slippage manipulation)

...makes it economically irrational to abuse intra-protocol composability for profit. The reentrancy guard serves its intended purpose (preventing re-entry within a single instruction via CPI callbacks) and functions correctly. Its absence of cross-instruction protection is not a deficiency — it would need to persist across instruction boundaries (at the transaction level) to protect against composability, which would be a different and much more restrictive design.

## Verdict

**Not a vulnerability.** Multi-instruction composability in a single Solana transaction is expected and by-design behavior of the runtime. The reentrancy guard (`pool.locked`) correctly prevents same-instruction CPI re-entry and is not deficient for not spanning instruction boundaries — that is not its purpose or scope. The economic costs (combined taxes + LP fees + adverse price impact) make same-pool round-trip exploitation unprofitable. No code change is required.

---

## Recommendations

### R1 — No Action Required (Informational)

The existing behavior is correct and expected for a Solana program. The comment on the reentrancy guard in `pool.rs` accurately describes its scope:

> "Solana's runtime borrow rules already prevent same-account reentry via CPI, and CEI ordering handles reserve consistency, but this provides an explicit belt-and-suspenders guard."

No change needed.

### R2 — Documentation (Optional, Low Priority)

If desired, add a comment to `swap_sol_pool.rs` near the guard clear (line 334) noting that the guard's scope is intra-instruction (CPI re-entry), not inter-instruction:

```rust
// Clear reentrancy guard.
// NOTE: This guard prevents CPI-level re-entry within a single instruction.
// Cross-instruction composability (multiple swap IXs in one TX) is standard
// Solana behavior and is handled by CEI ordering + the 50% output floor.
ctx.accounts.pool.locked = false;
```

This sets correct expectations for future maintainers auditing the guard's design intent.

### R3 — Monitor for Complex Composability Patterns (Operational)

If the protocol adds new instruction types in the future (e.g., flash loans, cross-pool arbitrage routes, or atomic liquidity operations), revisit this analysis. The current design is safe for the existing instruction set. A future instruction that can profitably intermediate between two swap instructions in the same transaction (e.g., by reading post-IX1 price and routing to a more favorable destination) could change the risk profile.
