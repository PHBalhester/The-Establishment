# H039: Carnage Fallback Executor — No Bounty Liveness Risk

| Field | Value |
|-------|-------|
| **ID** | H039 |
| **Status** | Confirmed |
| **Severity** | Low |
| **Category** | Liveness / Incentive Design |
| **Component** | `epoch-program/src/instructions/execute_carnage.rs`, `expire_carnage.rs` |
| **Reported Tier** | Tier 2 |

---

## Summary

The fallback Carnage execution path (`execute_carnage`) is permissionless but offers no economic bounty to the caller. The epoch trigger (`trigger_epoch_transition`) pays a 0.001 SOL bounty from the Carnage vault, but no equivalent payment exists for fallback Carnage execution. If the protocol's own crank service is offline, no external actor has a financial incentive to execute fallback Carnage. Pending Carnage expires after 300 slots (~2 minutes at ~400ms/slot), after which the SOL remains in the vault and Carnage is skipped for that epoch.

---

## Analysis

### Bounty Architecture: Trigger vs. Fallback Asymmetry

The epoch trigger instruction (`trigger_epoch_transition`) pays a hard-coded 0.001 SOL (`TRIGGER_BOUNTY_LAMPORTS = 1_000_000`) from the Carnage SOL vault to whichever caller submits the transition. This incentive ensures epoch liveness even when the protocol crank is down.

The `execute_carnage` fallback instruction accepts any `caller: Signer` but transfers nothing to them. The instruction comments confirm this design: the caller account is present only for transaction signing, not for reward delivery. There is no lamport transfer of any kind in the `execute_carnage` handler or in `execute_carnage_core` that flows to the caller.

### Expiry Behaviour: SOL Is Retained

On expiry (either via explicit `expire_carnage` or via auto-expire inside `consume_randomness`), the SOL vault balance is preserved. The `expire_carnage` handler explicitly comments: *"SOL is retained in vault for next trigger."* The `carnage_state` account is not `mut` in that instruction, confirming no state redistribution occurs. The expired Carnage is simply discarded:

```rust
// Clear pending state (SOL stays in vault)
epoch_state.carnage_pending = false;
epoch_state.carnage_action = CarnageAction::None.to_u8();
// Don't update last_carnage_epoch - Carnage didn't actually execute
```

This means an expiry is a pure no-op for value: accumulated vault SOL sits idle until the next VRF-triggered epoch (~4.3% probability per epoch).

### Fallback Window Is Short

The execution window for fallback Carnage is tight:
- Slots 0–50 (`CARNAGE_LOCK_SLOTS`): atomic-only, fallback rejected.
- Slots 51–300 (`CARNAGE_DEADLINE_SLOTS`): fallback window (~100 seconds of real time).

At mainnet throughput (~2,500 TPS), 300 slots pass in roughly 2 minutes. Even with a moderate crank outage, this window is easily missed.

### Severity Rationale

This is a **liveness** concern, not a **safety** concern:
- No funds are lost on expiry; SOL remains in the vault.
- No invariants are violated; the state machine clears cleanly.
- The protocol's own crank (running on Railway) is the intended executor, and the fallback path exists only as a safety valve for crank failures.
- The impact is a single skipped Carnage buy for the affected epoch: delayed buy pressure on the target token and one epoch's Carnage vault SOL remaining idle.
- The economic impact is bounded: at most one missed Carnage buy (max 1,000 SOL cap) per outage episode.

However, the **asymmetry** between the trigger bounty (paid) and the fallback bounty (absent) is a deliberate but undocumented design tension. An extended crank outage would cause repeated Carnage expirations, allowing the vault to accumulate SOL without deploying it — creating larger MEV targets when Carnage does eventually execute.

## Verdict

**Confirmed — Low severity.** The absence of a fallback execution bounty is a conscious liveness trade-off, not a security vulnerability. Safety properties (fund retention, clean state clearing) are maintained on expiry. The practical risk is limited to missed buy pressure during crank outages, with any accumulated vault SOL recovering and deploying in the next triggered Carnage.

The design tension worth noting: the epoch trigger bounty creates external crank competition, but the fallback bounty gap means the fallback path is fully dependent on the protocol's own infrastructure. This creates a single point of failure for the fallback mechanism specifically.

---

## Recommendations

1. **Add a small fallback execution bounty** (e.g., 0.0005 SOL from the Carnage vault). This would mirror the trigger bounty pattern and incentivize external actors to ensure Carnage executes even during crank outages. The cost is trivial relative to the swap sizes involved.

2. **Document the no-bounty-fallback design decision** explicitly in the protocol spec, noting that crank uptime on Railway is the assumed fallback executor. This makes the architectural dependency explicit for future maintainers.

3. **Alert on consecutive Carnage expirations** in off-chain monitoring. Two or more consecutive expirations signal a systemic crank failure and should trigger an operator page (the `CarnageFailed` event already emits the necessary data).

4. **Consider shortening the lock window** slightly if fallback bounty is added, to give more time for competitive external execution within the existing 300-slot deadline.
