# H028: remaining_accounts Forwarding Without Length Validation

**Status:** Confirmed — Partial (nuanced: two distinct failure modes, different severity)
**Severity:** Low-Medium (Tier 2 as assigned; nuanced below)
**Category:** Input Validation / CPI Account Handling
**Programs Affected:** Tax Program (`swap_sol_buy`, `swap_sol_sell`, `swap_exempt`), Epoch Program (`carnage_execution.rs` / `partition_hook_accounts`)

---

## Summary

Tax Program (`swap_sol_buy.rs:242-248`, `swap_sol_sell.rs:193-203`) and Epoch Program (`carnage_execution.rs:422-452`) forward `remaining_accounts` to downstream CPI calls without asserting exact length. The Bonding Curve Program, by contrast, performs an explicit `require!(ctx.remaining_accounts.len() == 4, CurveError::InvalidHookAccounts)` before any transfer. This inconsistency means:

1. **Zero accounts case (CRIME/FRAUD tokens):** The Token-2022 runtime will return an error when `transfer_checked` executes on a mint that has a transfer hook configured, because the required hook accounts (ExtraAccountMetaList PDA, whitelist source PDA, whitelist dest PDA, hook program) are absent from the instruction. The swap fails — but with a cryptic Token-2022 error rather than a clear application-level error.
2. **Extra accounts case:** Spurious accounts appended beyond the required 4 are forwarded into the CPI instruction's account list. Token-2022 ignores accounts beyond what it expects for hook invocation; they are silently ignored by the runtime. No security impact.

---

## Analysis

### The Forwarding Chain

The path from user instruction through to Token-2022 is:

```
User TX -> Tax Program (swap_sol_buy / swap_sol_sell)
             |-- forwards remaining_accounts via account_metas loop (no length check)
             v
         AMM (swap_sol_pool)
             |-- forwards ctx.remaining_accounts to transfer_t22_checked()
             v
         amm/helpers/transfers.rs::transfer_t22_checked()
             |-- appends hook_accounts to ix.accounts and account_infos
             v
         Token-2022::transfer_checked (CPI)
             |-- internally CPIs to Transfer Hook program using appended accounts
             v
         Transfer Hook::execute
```

For the Carnage path:
```
User TX -> Epoch Program (execute_carnage / execute_carnage_atomic)
             |-- partition_hook_accounts() partitions remaining_accounts into sell/buy slices
             v
         Tax Program (swap_exempt)
             |-- forwards ctx.remaining_accounts (no length check)
             v
         [same AMM/T22/Hook chain as above]
```

### Failure Mode 1: Zero remaining_accounts with T22 Token Mints

CRIME and FRAUD are Token-2022 mints with a transfer hook configured (the whitelist enforcement hook). When `transfer_checked` is called on Token-2022 with a hook-enabled mint, the runtime expects the extra accounts in the instruction. If none are provided:

- The `transfer_t22_checked` helper in `amm/src/helpers/transfers.rs` appends `hook_accounts` (which would be an empty slice `&[]`) to the instruction, leaving it with only the 4 standard transfer accounts.
- Token-2022 will attempt to invoke the Transfer Hook program. It needs the `ExtraAccountMetaList` PDA to look up the required hook accounts. Without this PDA in the account list, Token-2022 returns `AccountNotEnoughKeys` (error 3005) or a similar hook-resolution failure.
- The `invoke_signed` call propagates the error via `?` and the entire swap transaction fails.

**Impact:** A user or crank that calls `swap_sol_buy`, `swap_sol_sell`, or Carnage execution with 0 `remaining_accounts` for a CRIME/FRAUD pool will receive an opaque Token-2022 error rather than a clear `InvalidHookAccounts` error. The transaction fails either way — this is not a fund-loss vulnerability. The difference is purely in developer experience and debuggability.

**Note for Carnage path:** The `partition_hook_accounts` function in Epoch Program does gracefully degrade on insufficient accounts. When `remaining_accounts.len() < HOOK_ACCOUNTS_PER_MINT * 2` in the atomic case, or `< HOOK_ACCOUNTS_PER_MINT * 2` in the fallback Sell case, the function falls through to the final `else` branch which assigns `(&remaining_accounts[..0], remaining_accounts)` — passing an empty slice for sell hooks and all available accounts for buy hooks. This means for a `Sell` action with 0 accounts, both hooks get empty slices, which then causes Token-2022 to fail as above. The logic does not panic.

### Failure Mode 2: Extra remaining_accounts

If a caller passes more than 4 hook accounts (e.g., 8 or 12 for a dual-mint PROFIT pool scenario), all extra accounts are appended to the instruction's account list and forwarded.

- The AMM's `swap_sol_pool` forwards `ctx.remaining_accounts` directly to `transfer_t22_checked` for both T22 transfers. The inline comment at lines 191-201 explicitly documents this: for SOL pools, only one side is T22 so extra accounts appended for the SPL side are ignored. For PROFIT pools (dual T22), the Epoch Program pre-partitions before CPI so the AMM receives correctly-sized slices per-transfer.
- Token-2022 reads the `ExtraAccountMetaList` PDA to determine how many extra accounts to consume for the hook invocation. It reads exactly what the PDA specifies and ignores the rest.
- **Security impact: None.** Extra accounts cannot redirect hook logic or bypass whitelist checks. The accounts are inert from T22's perspective beyond the declared extra metas.

### Why Bonding Curve Validates but Tax/Epoch Don't

The Bonding Curve validates `len == 4` because it directly constructs and invokes the Token-2022 `transfer_checked` instruction itself (not via AMM intermediary). It knows the mint's hook always requires exactly 4 accounts (ExtraAccountMetaList + 2 whitelist PDAs + hook program). A mismatch at the BC level would be an undetected caller error with no downstream program to catch it.

The Tax and Epoch programs pass `remaining_accounts` through to the AMM, which passes them through to `transfer_t22_checked`. The length check at the Tax/Epoch layer would provide an earlier, clearer error message, but the failure is still caught by Token-2022 at the bottom of the chain. The missing check is a depth-of-defense gap rather than a vulnerability.

## Code Evidence

**Tax Program — no length check (buy path):**
```rust
// swap_sol_buy.rs:240-248
// 6c. Forward remaining_accounts for transfer hook support
//     The AMM passes these to Token-2022 transfer_checked calls
for account in ctx.remaining_accounts.iter() {
    if account.is_writable {
        account_metas.push(AccountMeta::new(account.key(), account.is_signer));
    } else {
        account_metas.push(AccountMeta::new_readonly(account.key(), account.is_signer));
    }
}
```

**Tax Program — no length check (sell path):**
```rust
// swap_sol_sell.rs:192-203
let remaining: Vec<AccountMeta> = ctx
    .remaining_accounts
    .iter()
    .map(|acc| {
        if acc.is_writable {
            AccountMeta::new(*acc.key, acc.is_signer)
        } else {
            AccountMeta::new_readonly(*acc.key, acc.is_signer)
```

**Bonding Curve — length check present:**
```rust

require!(
    ctx.remaining_accounts.len() == 4,
    CurveError::InvalidHookAccounts
);
```

**Epoch Program — conditional partitioning with silent fallback:**
```rust
// carnage_execution.rs:422-452
if atomic && remaining_accounts.len() >= HOOK_ACCOUNTS_PER_MINT * 2 {
    // ... slice correctly

    && remaining_accounts.len() >= HOOK_ACCOUNTS_PER_MINT * 2
{
    // ... slice correctly
} else {
    // Silent fallback: all remaining go to buy, empty slice for sell

}
```

**AMM `transfer_t22_checked` — no length validation, delegates to T22:**
```rust

}
// ...

    account_infos.push(account_info.clone());

invoke_signed(&ix, &account_infos, signer_seeds)?;
```

## Verdict

**Confirmed — Low-to-Medium severity, no fund-loss path exists.**

The finding as described is accurate: Tax and Epoch do not validate `remaining_accounts` length, while Bonding Curve does. However, the security consequence is bounded:

- **Zero accounts:** Transaction fails at Token-2022 with an opaque error. Funds are not lost. The failure is correct behavior — it prevents the hook bypass. The only harm is a poor developer/user experience.
- **Extra accounts:** Silently ignored by Token-2022. No security impact whatsoever.
- **Carnage path:** `partition_hook_accounts` degrades gracefully rather than panicking, which is the correct behavior. The swap will fail at Token-2022 level if hooks aren't satisfied, not silently corrupt state.

The Epoch Program's `partition_hook_accounts` conditional logic is actually the correct pattern for a function that must support multiple action types (BuyOnly vs Sell+Buy) with varying account counts. A strict `len == N` check would be wrong here because the required count varies by action.

The Tax Program (`swap_sol_buy` / `swap_sol_sell`) could benefit from a `len == 4` check equivalent to the Bonding Curve pattern, strictly for clearer error messages. This is a code quality / defensive programming improvement, not a security fix.

## Recommendations

### R1 — Add length guard in Tax Program buy/sell paths (Low priority — code quality)

Add the same guard that Bonding Curve uses, immediately before the `remaining_accounts` forwarding loop:

```rust
// In swap_sol_buy.rs and swap_sol_sell.rs, before forwarding remaining_accounts:
require!(
    ctx.remaining_accounts.len() == 4,
    TaxError::InvalidHookAccounts
);
```

This provides a clear application-level error instead of delegating to Token-2022's internal error. Matches the Bonding Curve's established pattern.

**Caveat:** If PROFIT pool swaps (dual T22) via Tax Program ever require 8 hook accounts (4 per mint), this guard would need to be `len == 4 || len == 8`. Confirm the expected count for each pool type before hardcoding.

### R2 — Document the Epoch Program's `partition_hook_accounts` contract explicitly (Informational)

The function's silent fallback on insufficient accounts is not obviously intentional. Add an inline comment or structured assertion documenting the minimum expected counts per action type, so future maintainers do not accidentally break the partitioning logic.

```rust
// Minimum accounts required per action:
// BuyOnly: >= 4 (buy hooks only)
// Burn:    >= 4 (buy hooks only)
// Sell:    >= 8 atomic (buy+sell); >= 8 fallback (sell+buy)
```

### R3 — No action required for extra-accounts case

The extra accounts scenario carries no exploitable surface. Token-2022 is the authoritative validator of hook account requirements and will ignore surplus accounts. No change needed.
