# H023: stake_pool Unconstrained at Epoch Program Level

**Status:** POTENTIAL
**Severity:** LOW
**Category:** Access Control, CPI
**Origin:** KB (EP-007)

## Summary

`stake_pool` in `ConsumeRandomness` is declared as `AccountInfo<'info>` with only `#[account(mut)]` — no owner check, no PDA seeds constraint. The account is forwarded directly into a manual `invoke_signed` CPI call to the Staking Program. A caller could pass any writable account as `stake_pool`. The Staking Program's own Anchor constraints (`seeds = [STAKE_POOL_SEED]`, `bump = stake_pool.bump`) will reject any non-canonical account, but this defence lives entirely downstream.

## Analysis

### What the constraint does (and does not) check

```rust
// consume_randomness.rs:63-66
/// Staking Program's pool state (mutable for update_cumulative).
/// CHECK: Validated by Staking Program during CPI.
#[account(mut)]
pub stake_pool: AccountInfo<'info>,
```

The `#[account(mut)]` guard only asserts that the account is marked writable in the transaction. It does not check:
- Owner (which program owns the account data)
- PDA derivation (seeds `[STAKE_POOL_SEED]` and canonical bump)
- Discriminator (Anchor type tag)
- Initialisation flag (`stake_pool.initialized`)

All four of those checks are enforced by `UpdateCumulative` inside the Staking Program:

```rust
// staking/src/instructions/update_cumulative.rs:44-51
#[account(
    mut,
    seeds = [STAKE_POOL_SEED],
    bump = stake_pool.bump,
    constraint = stake_pool.initialized @ StakingError::NotInitialized,
)]
pub stake_pool: Account<'info, StakePool>,
```

### CPI failure path for a fake account

When an attacker passes a fake `stake_pool`:

1. `consume_randomness` passes — `#[account(mut)]` is satisfied by any writable account.
2. `invoke_signed` fires the CPI with the fake account in position `[1]`.
3. The Staking Program attempts to deserialise the account as `Account<'info, StakePool>`.
4. Anchor checks: owner == staking_program_id → **FAILS** (custom program cannot own a fake account unless the attacker is the owner).
5. Even if ownership were spoofable, Anchor checks seeds derivation: `seeds = [STAKE_POOL_SEED]` with the bump stored inside the account itself → **FAILS** for any non-canonical address.

The CPI **will always revert** with an Anchor constraint error. The entire `consume_randomness` transaction is then rolled back, so no state mutation occurs.

### Is there any viable attack?

Two scenarios to consider:

**Scenario A — Denial-of-service griefing**: An attacker calls `consume_randomness` with a fake `stake_pool` knowing the CPI will revert. This wastes the honest crank's effort (the VRF has already been consumed; its slot must still be valid for `get_value`). However, the Switchboard randomness account is not mutated by `consume_randomness` at the point of failure — the honest crank can simply retry with the correct `stake_pool`. Net impact: at most one wasted crank submission with a small SOL fee.

**Scenario B — Staking Program exploitation through Epoch**: For a fake account to reach a `SUCCESS` path inside the Staking Program, the attacker would need to craft an account whose owner *is* the Staking Program and whose contents satisfy the seeds/bump/discriminator/initialized checks. That is equivalent to having a valid `StakePool` PDA, which is the singleton canonical account. There is no way to construct a second valid `StakePool` PDA from the same seed without controlling the Staking Program itself. This attack path does not exist.

### Defence-in-depth gap

The Epoch Program currently provides zero first-line validation for `stake_pool`. Industry practice (and Anchor best practice) is that each program independently constrains every account it mutates or passes to a downstream CPI. The comment `/// CHECK: Validated by Staking Program during CPI.` is the Anchor pattern for `AccountInfo` but is insufficient when the program controls the CPI — it relies entirely on the downstream program to catch bad input. If the Staking Program's constraint were ever relaxed (e.g., during a future upgrade to support multiple pools), Epoch's lack of constraint could become an exploitable gap.

## Code Evidence

**Unconstrained declaration** (`programs/epoch-program/src/instructions/consume_randomness.rs:63-66`):
```rust

/// CHECK: Validated by Staking Program during CPI.

```

**CPI construction passes account without independent verification** (`consume_randomness.rs:246-263`):

    program_id: ctx.accounts.staking_program.key(),

## Verdict

**POTENTIAL / LOW severity.** The vulnerability is real as a defence-in-depth gap but is not currently exploitable: any fake `stake_pool` account causes the CPI to revert, rolling back all state changes, with no funds at risk. The issue is architectural: Epoch Program trusts Staking Program to validate an account that Epoch itself constructs and forwards. This is a single point of failure should the Staking Program's constraints ever be weakened.

The finding is rated **LOW** (not MEDIUM) because:
1. Exploitation requires the Staking Program to independently regress or be upgraded in a weakening way.
2. The current CPI failure mode is a clean revert with no partial-state harm.
3. Even a successful CPI with a fake account would only execute `update_cumulative` logic on a state the attacker controls, not on real protocol funds.

The finding is rated **POTENTIAL** (not CONFIRMED) because no direct harm can be demonstrated against the current deployed code.

## Recommendations

1. **Add an explicit owner + seeds constraint at the Epoch Program level.** Replace the bare `#[account(mut)]` with a constraint that pins the account to the known Staking Program singleton PDA:

```rust
/// Staking Program's pool state (mutable for update_cumulative).
/// CHECK: Singleton PDA owned by Staking Program with canonical seeds.
#[account(
    mut,
    owner = staking_program_id() @ EpochError::InvalidStakingProgram,
)]
pub stake_pool: AccountInfo<'info>,
```

   Or, if the staking crate is available as a dependency, use a typed account:

```rust
#[account(
    mut,
    seeds = [STAKE_POOL_SEED],
    bump,
    seeds::program = staking_program_id(),
)]
pub stake_pool: Account<'info, staking::state::StakePool>,
```

2. **Add an address constraint** (most minimal fix): derive the expected `StakePool` PDA address from `STAKE_POOL_SEED` and assert `stake_pool.key() == expected_pda`. This can be done without a crate dependency and adds zero CPI overhead.

3. **Document the dependency explicitly.** If the constraint is intentionally delegated downstream for compile-dependency reasons, the code comment should be strengthened to name the specific constraint in `update_cumulative.rs` that provides the protection, so future maintainers understand that weakening `update_cumulative`'s `stake_pool` constraint would require a corresponding change here.
