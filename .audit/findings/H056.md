# H056: Staking Cooldown Timestamp Manipulation

**Status:** CONFIRMED (Negligible Risk on Mainnet, Trivial on Devnet)
**Severity:** LOW
**Category:** Timing & Ordering / Access Control

## Summary

The staking program enforces a 12-hour cooldown between claim and unstake using `clock.unix_timestamp`, which validators can drift by up to ~30 seconds (EP-089). On mainnet with COOLDOWN_SECONDS=43200 (12 hours), the 30-second drift represents a 0.07% deviation — economically negligible. On devnet with COOLDOWN_SECONDS=2, the 30-second drift is trivially exploitable but devnet is not security-relevant. The vulnerability exists in principle but is not practically exploitable on mainnet.

## Code Evidence

**File:** `programs/staking/src/instructions/claim.rs:129`

```rust
// Start cooldown timer -- unstake blocked for COOLDOWN_SECONDS
user.last_claim_ts = clock.unix_timestamp;
```

**File:** `programs/staking/src/instructions/unstake.rs:126-133`

```rust
// Cooldown gate: must wait COOLDOWN_SECONDS after last claim
// Skip check if user has never claimed (last_claim_ts == 0)
if ctx.accounts.user_stake.last_claim_ts > 0 {
    let elapsed = clock
        .unix_timestamp
        .checked_sub(ctx.accounts.user_stake.last_claim_ts)
        .unwrap_or(0); // clock weirdness -> treat as cooldown active
    require!(
        elapsed >= COOLDOWN_SECONDS,
        StakingError::CooldownActive
    );
}
```

**Constants:** `programs/staking/src/constants.rs`
```rust
pub const COOLDOWN_SECONDS: i64 = if cfg!(feature = "devnet") {
    2_i64      // Devnet: 2 seconds
} else {
    43200_i64  // Mainnet: 12 hours = 43200 seconds
};
```

**Trust boundary:** EP-089 states validators can drift unix_timestamp by ±30 seconds from real time.

## Verdict

**CONFIRMED: Timestamp manipulation is theoretically possible but practically negligible on mainnet.**

**Mainnet impact:**
- COOLDOWN_SECONDS = 43200 (12 hours = 43,200 seconds)
- Max validator drift = ±30 seconds
- Deviation = 30 / 43200 = **0.069% (less than 0.1%)**
- Economic impact: Negligible

A user wanting to unstake 12 hours and 30 seconds after claiming could unstake in 12 hours if a validator drifts backward, but the time saved is immaterial for yield farming.

**Devnet impact:**
- COOLDOWN_SECONDS = 2 seconds
- Max validator drift = ±30 seconds
- Deviation = 30 / 2 = **1500% (!)**
- Economic impact: TRIVIAL — devnet has no real assets

A devnet user can trivially bypass the 2-second cooldown by waiting for validator drift or setting their local time ahead.

**Design rationale:** The use of `unix_timestamp` is appropriate for staking cooldowns because:
1. The purpose is to prevent flash-loan reward extraction via rapid claim/unstake cycles
2. Sub-second precision is not needed for a 12-hour cooldown
3. Slot-based timing (alternative) would be more complex without better security properties

**The `unwrap_or(0)` pattern** at line 129 is also correct:
- If clock arithmetic underflows (extremely unlikely), default to cooldown active (safe)
- Prevents the rare case where system clock goes backward

**Recommendation:** **No code change required.** The cooldown is secure on mainnet. If desired for maximum pedantry, document that the 0.07% drift tolerance is intentional. Classified as **LOW** severity because:
1. Not practically exploitable on mainnet
2. No financial loss possible (only time savings)
3. Devnet is unrelated to production security

This is a good example of an attack surface that exists in principle but is mitigated by economics and constraints.
