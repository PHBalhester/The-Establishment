# H010: Carnage Fallback MEV Sandwich Extraction

**Status:** CONFIRMED
**Severity:** HIGH
**Category:** MEV, Token/Economic
**Origin:** KB (EP-060, EP-112)

## Summary

After the 50-slot atomic lock window expires, the Carnage fallback path is permissionless with `carnage_target` publicly visible on-chain and a 75% slippage floor (`CARNAGE_SLIPPAGE_BPS_FALLBACK = 7500`). A MEV actor can front-run the Carnage buy by pumping the target pool, then back-run after execution, extracting up to 25% of the Carnage swap amount — theoretically up to 250 SOL against the 1000 SOL cap.

## Analysis

### Attack Preconditions

All preconditions are observable on-chain with zero trusted information:

1. `epoch_state.carnage_pending == true` — readable from EpochState PDA
2. `epoch_state.carnage_target` — `u8` (0=CRIME, 1=FRAUD) readable from EpochState at on-chain byte offset 84
3. `epoch_state.carnage_lock_slot` — readable from EpochState at on-chain byte offset 94; when `clock.slot > carnage_lock_slot`, the fallback window is open
4. `epoch_state.carnage_deadline_slot` — readable at byte offset 86; execution expires after this slot

The EpochState is a known singleton PDA (`seeds = ["epoch_state"]`), so all fields are trivially polled by any observer.

### Attack Flow

1. Observe `carnage_pending = true` and `carnage_target = 0` (CRIME) via on-chain state monitoring.
2. Wait for slot > `carnage_lock_slot` (lock window expires after 50 slots, ~20 seconds).
3. Front-run: Buy a large amount of CRIME on the AMM to inflate the price (raise `reserve_sol`, lower `reserve_token`).
4. Submit `execute_carnage`. The program reads pre-swap reserves via `read_pool_reserves` and computes the expected output using the constant-product formula. The inflated state means the expected output is computed against the manipulated reserves:
   ```
   expected = (reserve_token * total_buy_amount) / (reserve_sol + total_buy_amount)
   min_output = expected * CARNAGE_SLIPPAGE_BPS_FALLBACK / 10_000  // 75%
   ```
   Because the attacker already bought CRIME, `reserve_token` is lower and `reserve_sol` is higher, so the constant-product formula yields a smaller `expected`. The 75% floor is computed from this already-depressed `expected`, meaning the effective protection floor is 75% of a manipulated baseline — not 75% of fair value.
5. Carnage executes at the inflated (sandwiched) price, receiving fewer tokens than at fair market price.
6. Back-run: The attacker sells their CRIME position for profit, benefiting from Carnage's buy pressure having restored the price.

### Slippage Floor Mechanics — Why It Does Not Prevent This

The slippage check at `carnage_execution.rs:331-349` is:

```rust
let expected = (reserve_token as u128)
    .checked_mul(total_buy_amount as u128)
    .and_then(|n| n.checked_div(
        (reserve_sol as u128).checked_add(total_buy_amount as u128)?
    ))
    ...;
let min_output = expected * CARNAGE_SLIPPAGE_BPS_FALLBACK / 10_000;  // 75% of expected
require!(bought >= min_output, EpochError::CarnageSlippageExceeded);
```

The critical flaw is that `reserve_sol` and `reserve_token` are read from the pool state **at the time of execution** (after the front-run has already occurred), not from a trusted price oracle or pre-committed snapshot. The `expected` value is therefore calculated from the already-manipulated reserve state. The 75% floor means the attacker can extract up to 25% relative to the manipulated baseline — and the manipulated baseline itself is already worse than fair value.

### AMM Layer Has Zero Slippage Protection

`swap_exempt.rs:111` hardcodes `MINIMUM_OUTPUT: u64 = 0`:

```rust
const MINIMUM_OUTPUT: u64 = 0; // Carnage accepts market execution
```

This means the AMM itself provides no backstop. If the upstream Epoch Program's slippage check were somehow bypassed or loosened, the AMM would execute at any price. The Epoch Program's 75% check is the **sole** protection.

### Exposure Window

- Lock window: 50 slots (~20s at 400ms/slot). During this period only atomic execution is possible.
- Fallback window: slots 50–300 (~100 seconds). The attacker has a ~100-second window to execute the sandwich.
- This is not a flash loan scenario — the attacker holds the position across multiple slots (not atomic), making this accessible to any well-funded actor, not just flash loan users. However, a flash loan could be used to amplify the front-run position if the attacker wants to minimize capital commitment.

## Quantitative Assessment

### Theoretical Maximum Extraction

- Carnage swap cap: `MAX_CARNAGE_SWAP_LAMPORTS = 1,000 SOL`
- Fallback slippage tolerance: 25% (100% - 75%)
- Maximum extractable per fallback event: `1000 SOL × 0.25 = 250 SOL`
- At SOL = $150 USD: ~$37,500 per fallback event

### Realistic Extraction

The 75% floor is applied to the **manipulated** expected output, not fair value. In practice:

- If the attacker pumps the pool such that Carnage receives 85% of fair-value tokens, the floor check passes (85% of manipulated expected ≥ 75%).
- The attacker captures the spread between their front-run cost and back-run proceeds.
- Realistic extraction with moderate pool depth: 10–20% of Carnage swap value.
- Against a large Carnage swap (500–1000 SOL), this is 50–200 SOL.

### Frequency

- Carnage trigger probability: ~4.3% per epoch (~30 min mainnet)
- Expected frequency: ~2–3 triggers/day on mainnet
- Fallback path activates only when atomic path fails — estimated <5% of Carnage triggers
- Expected fallback events: ~0.1–0.15/day
- Expected MEV extraction: ~5–30 SOL/day in adversarial conditions (moderate pool depth)

### Pool Depth Dependency

The attack profitability scales with pool depth. Thin pools are easier to manipulate but yield less Carnage buy — sandwich profit is bounded by pool size. Deeper pools require more capital to move price 25% but allow larger absolute extraction.

## Mitigating Factors

1. **Atomic path is primary**: The atomic path (`execute_carnage_atomic`) bundles VRF reveal + execution in one TX. Target is unknown until the TX executes, making atomic-path sandwiching require precognition. The fallback only activates if the atomic path fails (e.g., compute budget exceeded on Sell+Buy paths).

2. **VRF unpredictability**: The target token (`carnage_target`) is set by VRF randomness and is only revealed in `consume_randomness`. Between trigger and reveal, the attacker cannot know which pool to pre-position in.

3. **75% floor rejects extreme manipulation**: An attacker who pumps the pool so aggressively that Carnage would receive <75% of the (already-manipulated) expected output will cause the transaction to fail. This prevents the most extreme sandwich scenarios.

4. **250-slot deadline**: The attacker only has a 250-slot window (slots 50–300). If the protocol's own crank service executes the fallback promptly after slot 50, the attacker's window is narrow.

5. **No bounty reduces competition**: There is no economic bounty for fallback Carnage execution. The protocol crank (Railway) is the expected executor. This limits the number of actors monitoring for the opportunity.

6. **Pool access control**: The AMM is only accessible via Tax Program's `swap_authority` PDA for protocol-internal operations. However, the attacker's front-run and back-run are standard user swaps through `swap_sol_buy`/`swap_sol_sell` (Tax Program), which are permissionless. The 50% minimum output floor on user swaps limits how much the attacker can dump the pool in a single TX, but does not prevent multi-TX accumulation.

## Verdict

**CONFIRMED — HIGH severity.**

The attack is technically feasible without on-chain program modification, requires only standard swap access, and the observable window (50–300 slots after Carnage trigger) gives an attacker substantial time to act. The slippage floor is computed against live (manipulable) reserves rather than a time-weighted or pre-committed price, which is the core weakness. The maximum extractable value (250 SOL) is protocol-design-level significant, not dust.

Severity is rated HIGH rather than CRITICAL because:
- The fallback path is the exception, not the norm (atomic path succeeds in most cases)
- The 75% floor does prevent the worst-case scenarios (attacker cannot drain Carnage for nothing)
- The attack requires multi-slot positioning (not purely atomic/flash-loan), adding capital risk for the attacker
- The frequency is bounded by the ~4.3% Carnage trigger probability and the atomic path success rate

## Recommendations

1. **Minimize fallback window**: After the atomic lock expires at slot 50, trigger the fallback from the protocol's own crank service immediately (at slot 51). This collapses the 250-slot window to near-zero for external actors, as the crank executes before any MEV bot can react.

2. **Add a protocol-priority execution path**: Consider allowing `consume_randomness` itself to trigger a fallback execution in the same TX (rather than just atomic), with a configurable retry budget. This removes the need for a separate fallback window entirely in most cases.

3. **TWAP-based slippage for fallback**: For the fallback path specifically, derive the expected output from an on-chain TWAP or a pre-committed reserve snapshot (e.g., stored in EpochState at the time `consume_randomness` fires) rather than live reserves. This prevents the attacker's front-run from shifting the slippage baseline.

4. **Increase fallback slippage floor**: Raising `CARNAGE_SLIPPAGE_BPS_FALLBACK` from 7500 to 8500 (matching atomic) would reduce the tolerance window from 25% to 15%. This increases the risk of legitimate execution failures in high-volatility conditions but reduces MEV extraction space.

5. **Private mempool for fallback**: Execute the fallback Carnage TX through Jito bundles or another private relay to prevent front-running at the transaction ordering level.

Note: Recommendations 1 and 5 are operational mitigations (no program changes required). Recommendations 2, 3, and 4 require program upgrades.
