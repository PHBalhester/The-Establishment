# H029: Hardcoded CPI Discriminators Fragility

## Status

**Confirmed — Mitigated (tests exist, but runtime safety net is absent).**

All three discriminator constants are currently correct. Two of the three are
covered by unit tests that recompute the hash and assert equality. One stale
test in `staking/src/constants.rs` would fail if run, signalling a maintenance
drift problem rather than an active discriminator mismatch.

---

## Severity

**Tier 2 — Medium**

- No active exploitability: the discriminators are correct today.
- Silent failure mode: a mismatch causes CPI calls to fail with an opaque error
  (wrong instruction discriminator), not a crash with a meaningful message.
- Upgrade-coupling risk: renaming any of the three target instructions without
  simultaneously updating and redeploying all callers will silently break
  core protocol flows (tax revenue deposit, epoch reward accounting,
  Carnage swaps).
- Scope of impact: all three paths carry real value — deposit_rewards
  accounts for 71% of swap tax revenue; update_cumulative gates staking
  reward accrual; swap_exempt is the Carnage Fund execution path.

---

## Summary

Three programs construct CPI instructions manually using precomputed Anchor
discriminator bytes (sha256("global:<name>")[0..8]) stored as named constants:

| `UPDATE_CUMULATIVE_DISCRIMINATOR` | `epoch-program/src/constants.rs:117` | `[0x93,0x84,0xdb,0x65,0xa5,0x17,0x3d,0x71]` | `staking::update_cumulative` |

All three values are confirmed correct against their current instruction names.
Two of the three have self-verifying unit tests. However, no on-chain or
compile-time mechanism ties a discriminator constant to the instruction name in
the target program binary. An upgrade that renames an instruction — even as a
refactor — will silently invalidate callers.

Additionally, the staking program's `test_tax_program_id` test asserts
`"43fZ..."` (the mock-tax-program ID) but `tax_program_id()` returns
`"FGgi..."` (the real tax-program ID). This test would fail if run, and
indicates the stale-test maintenance risk predicted by this finding category is
already materialising.

## Analysis

### How Anchor Discriminators Work

Anchor uses the first 8 bytes of `sha256("global:<instruction_name>")` as a
routing discriminator in instruction data. When a client or another program
constructs a CPI manually (rather than using the generated Anchor CPI crate),
it must embed these bytes itself. There is no runtime type-safety between the
caller's hardcoded bytes and the target program's actual discriminator table;
a mismatch results in Anchor returning `InvalidInstructionData` (error 100),
which surfaces at the `invoke_signed` call site as a generic program error.

### Three Independent Discriminator Constants

**1. `DEPOSIT_REWARDS_DISCRIMINATOR` (`tax-program/src/constants.rs:156`)**

Used in `swap_sol_buy.rs` and `swap_sol_sell.rs` when building the manual CPI
to staking's `deposit_rewards`. This is called on every successful swap to
credit 71% of tax revenue to the staking escrow reward counter. A discriminator
mismatch here would silently prevent all staking reward deposits — stakers
would see zero yield while swap fees continue to arrive and get stranded.

The constant `[52, 249, 112, 72, 206, 161, 196, 1]` was independently verified:

```
sha256("global:deposit_rewards")[0..8] = [52, 249, 112, 72, 206, 161, 196, 1]  ✓
```

A matching unit test exists in both `tax-program/src/constants.rs:213` and
`staking/src/constants.rs:190`, recomputing the hash at test time.

**2. `UPDATE_CUMULATIVE_DISCRIMINATOR` (`epoch-program/src/constants.rs:117`)**

Used in `consume_randomness.rs:243` when building the manual CPI to staking's
`update_cumulative`. This is called at each epoch transition to advance the
reward-per-token accumulator. A mismatch here would cause every
`consume_randomness` call to fail after distributing VRF randomness, halting
epoch progression entirely.

The constant `[0x93, 0x84, 0xdb, 0x65, 0xa5, 0x17, 0x3d, 0x71]` was
independently verified:

```
sha256("global:update_cumulative")[0..8] = [0x93, 0x84, 0xdb, 0x65, 0xa5, 0x17, 0x3d, 0x71]  ✓
```

A unit test exists at `epoch-program/src/constants.rs:187`.

**3. `SWAP_EXEMPT_DISCRIMINATOR` (`epoch-program/src/constants.rs:177`)**

Used in `carnage_execution.rs:770` when building the manual CPI to tax-program's
`swap_exempt`. This is called for every Carnage Fund swap execution. A mismatch
here would cause all Carnage swaps to fail silently — the Carnage Fund would
accumulate SOL but never deploy it.

The constant `[0xf4, 0x5f, 0x5a, 0x24, 0x99, 0xa0, 0x37, 0x0c]` was
independently verified:

```
sha256("global:swap_exempt")[0..8] = [0xf4, 0x5f, 0x5a, 0x24, 0x99, 0xa0, 0x37, 0x0c]  ✓
```

A unit test exists at `epoch-program/src/constants.rs:250`.

### Stale Test Already Detected

`staking/src/constants.rs:133-140` has:

```rust
fn test_tax_program_id() {
    let id = tax_program_id();
    assert_eq!(id.to_string(), "43fZGRtmEsP7ExnJE1dbTbNjaP1ncvVmMPusSeksWGEj");
}
```

But `tax_program_id()` at line 97-100 returns `"FGgidfhNLwxhGHpyH7SoZdxAkAyQNXjA5o8ndV3LkG4W"`.
`43fZ...` is the `mock-tax-program`'s `declare_id!`. This test will fail on
execution, confirming that cross-program constant tests can fall out of sync
during redeploys. The discriminator tests are of the same character — correct
today, but not automatically enforced when the underlying instruction names or
program IDs change.

### Why Manual CPI?

The reason these three CPI calls are built manually rather than using Anchor's
generated CPI crate is documented in the codebase context: Anchor's CPI
framework does not forward `remaining_accounts` through the Token-2022 ->
Transfer Hook nested CPI chain. For consistency and to avoid introducing a
generated-crate dependency cycle, the same manual-invoke pattern is applied
across all internal CPI sites. This is architecturally sound but removes the
compile-time safety net that generated CPI crates provide.

### Upgrade Scenario

If `staking::deposit_rewards` were renamed to `deposit_tax_rewards` during an
upgrade (for clarity), its on-chain discriminator would change. The deployed
tax-program binary would still use the old bytes. Every subsequent swap would
revert with a cryptic error code during the staking CPI, causing revenue
distribution to fail. No compile-time or deploy-time check would catch this.
The only detection mechanism is the unit test — which must be run and must use
the *updated* expected value.

## Verdict

**Discriminator values are correct today.** All three constants match the
sha256 of their respective instruction names. Two of the three have unit tests
that recompute and assert the hash. The risk is entirely forward-looking:
there is no structural coupling between the constant and the instruction it
represents, meaning any instruction rename during a future upgrade will silently
break callers with no compile-time warning.

The stale `test_tax_program_id` test in `staking/src/constants.rs` demonstrates
that this drift pattern is already present in the codebase — a previous redeploy
updated `tax_program_id()` but not its test assertion.

**Severity rationale: Medium (not High)** because:
- All constants are correct at time of audit.
- Unit tests exist for all three discriminators and would catch a name change
  if the tests are run.
- The risk is upgrade-time operational hygiene, not an active vulnerability.
- The stale program ID test, while not a security risk itself, confirms the
  maintenance exposure class is real.

## Recommendations

### R1 — Fix the Stale `test_tax_program_id` in `staking/src/constants.rs` [Immediate]

Update line 138 from `"43fZGRtmEsP7ExnJE1dbTbNjaP1ncvVmMPusSeksWGEj"` to
`"FGgidfhNLwxhGHpyH7SoZdxAkAyQNXjA5o8ndV3LkG4W"` (or parameterise it through
`tax_program_id().to_string()`). A test that will always fail provides no safety
net and erodes confidence in the test suite.

### R2 — Add a Cross-Program Discriminator Consistency Test [High Priority]

Add an integration test (in `tests/cross-crate/` or a dedicated
`tests/discriminator-consistency/`) that:

1. Computes expected discriminators from the instruction names in the target
   program's IDL (or via `sha256("global:<name>")`).
2. Asserts they match the caller program's constants.
3. Is run as part of `anchor test` CI, not just per-program unit tests.

This creates a single failure point that fires whenever an instruction is
renamed in any of the three target programs, giving the developer an
actionable error before deployment.

### R3 — Document Upgrade Gate in Deployment Checklist [Medium Priority]

Add a step to `scripts/deploy/deployment-report.md` and
`Docs/mainnet-deploy-checklist.md`:

> Before upgrading staking-program: verify `DEPOSIT_REWARDS_DISCRIMINATOR`
> in tax-program and `UPDATE_CUMULATIVE_DISCRIMINATOR` in epoch-program
> match the new instruction names. Run discriminator consistency tests.
>
> Before upgrading tax-program: verify `SWAP_EXEMPT_DISCRIMINATOR` in
> epoch-program matches the new instruction name.

### R4 — Consider Anchor CPI Crate for Non-Hook Paths [Low Priority / Future]

The manual-invoke pattern was adopted to handle the Token-2022 hook forwarding
limitation. However, the three discriminator-sensitive CPIs (deposit_rewards,
update_cumulative, swap_exempt) do not themselves involve Token-2022 hook
forwarding. These three specific calls could potentially be replaced with
generated Anchor CPI crates, giving compile-time discriminator safety. This is
a refactor rather than a bug fix, and should be evaluated against the
complexity cost.
