# H012: Build-Pipeline Supply Chain — Cross-Program ID Injection

**Status:** POTENTIAL
**Severity:** HIGH
**Category:** Upgrade/Admin, Supply Chain
**Origin:** KB (EP-095)

---

## Summary

The build pipeline (`sync-program-ids.ts`, `patch-mint-addresses.ts`) reads secret-key material from the filesystem and patches cross-program IDs and mint addresses directly into Rust source files before compilation. An attacker who compromises the keypairs directory or the build scripts can inject arbitrary program IDs or mint addresses into the compiled binary, routing all CPI calls to attacker-controlled programs. The threat is real but constrained: exploitation requires prior local filesystem access (the build machine), which is not achievable remotely today, but the design eliminates all on-chain defenses against this class of tampering.

---

## Analysis

### How the pipeline works

`build.sh` runs two pre-build patchers before `anchor build`:

1. **`sync-program-ids.ts` (Step 0/4):** Reads 7 program keypairs from `keypairs/*.json`, derives their public keys, and writes those addresses into `declare_id!()` macros in each `lib.rs`, into `Anchor.toml`, and into 9 cross-program `constants.rs` entries (e.g., `amm_program_id()` in Epoch, `TAX_PROGRAM_ID` in AMM).

2. **`patch-mint-addresses.ts` (Step 0b/4):** Reads 3 mint keypairs from `scripts/deploy/mint-keypairs/*.json`, derives their public keys, and patches `crime_mint()`, `fraud_mint()`, `profit_mint()`, `treasury_pubkey()` etc. into 3 `constants.rs` files.

The compiled binary embeds these addresses as constants. At runtime, every CPI call validates the target program via `address = amm_program_id()` or `seeds::program = tax_program_id()`. If those constants were tampered with before compilation, on-chain validation passes for the attacker's program — because the binary was told the attacker's address IS the legitimate one.

### Attack scenario

1. Attacker gains filesystem access to the build machine (malware, compromised CI environment, SSH compromise, or supply-chain attack on a developer's laptop).
2. Attacker replaces `keypairs/amm-keypair.json` with a keypair they control, or directly edits `programs/amm/src/constants.rs` (which the script will not overwrite since it only patches specific patterns).
3. `sync-program-ids.ts` runs, patches `declare_id!` in AMM's `lib.rs` and `TAX_PROGRAM_ID` in AMM's `constants.rs` to the attacker's program address.
4. `anchor build` compiles a binary where every `seeds::program = TAX_PROGRAM_ID` constraint in AMM accepts the attacker's program as the legitimate swap authority.
5. The attacker deploys their own program at that address.
6. All deployed AMM CPIs from Tax Program route through the attacker's program; all `swap_authority` PDA derivations inside AMM accept calls from the attacker's program.

Alternatively: attacker edits `patch-mint-addresses.ts` or `sync-program-ids.ts` directly (both are world-readable, `rw-r--r--`) to insert a hardcoded attacker address before the normal keypair-derived one.

### Scope of compromise if exploited

Because ALL 7 programs' cross-program trust chains flow through these patched constants, a single keypair swap can break the entire protocol:

| Constant patched | Effect |
|-----------------|--------|
| `TAX_PROGRAM_ID` in AMM | Attacker's program becomes the only valid AMM swap caller |
| `amm_program_id()` in Epoch | Epoch's Carnage CPI targets attacker's AMM |
| `tax_program_id()` in Epoch | Epoch's `carnage_signer` PDA seeds target attacker's Tax |
| `crime_mint()` / `fraud_mint()` in Vault / BondingCurve | All token validation accepts attacker's mints |
| `epoch_program_id()` in Staking | Staking accepts `update_cumulative` from attacker's Epoch |
| `treasury_pubkey()` in Tax | All tax revenue is redirected to attacker's wallet |

## Code Evidence

### Keypair files are git-tracked with world-readable permissions

```
-rw-r--r-- keypairs/amm-keypair.json          (world-readable; contains 64-byte secret key)
-rw-r--r-- keypairs/tax-program-keypair.json
-rw-r--r-- keypairs/staking-keypair.json
-rw-r--r-- keypairs/epoch-program.json
-rw-r--r-- keypairs/vault-keypair.json
-rw-r--r-- keypairs/bonding-curve-keypair.json
-rw-r--r-- keypairs/transfer-hook-keypair.json
```

Git history confirms full 64-byte secret keys are committed and have been for multiple deploy cycles (5+ commits visible). Any person with read access to the git repository can extract the private keys for all 7 production programs.

### Build scripts are world-readable and run without integrity checking

```
-rwxr-xr-x scripts/deploy/build.sh
-rw-r--r-- scripts/deploy/sync-program-ids.ts
-rw-r--r-- scripts/deploy/patch-mint-addresses.ts
```

Neither `build.sh` nor any calling wrapper verifies checksums of these scripts before executing them. `build.sh` simply does:

```bash
npx tsx scripts/deploy/sync-program-ids.ts   # step 0
npx tsx scripts/deploy/patch-mint-addresses.ts  # step 0b
```

### sync-program-ids.ts reads keypairs and patches source unconditionally

```typescript
// scripts/deploy/sync-program-ids.ts:208-215
function loadKeypair(relativePath: string): Keypair {
  const resolved = path.resolve(PROJECT_ROOT, relativePath);
  if (!fs.existsSync(resolved)) {
    throw new Error(`Keypair not found: ${resolved}`);
  }
  const secretKey = JSON.parse(fs.readFileSync(resolved, "utf8"));
  return Keypair.fromSecretKey(new Uint8Array(secretKey));
}
```

No validation that the loaded keypair matches any expected address. A replaced keypair is accepted silently and its derived address is patched everywhere.

### patch-mint-addresses.ts reads treasury from environment variable with no validation

```typescript
// scripts/deploy/patch-mint-addresses.ts:168
const treasuryAddress = process.env.TREASURY_PUBKEY || "8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4";
```

If `TREASURY_PUBKEY` is set in the CI environment or a compromised `.env` file, the treasury address in the compiled binary becomes whatever the attacker sets it to. The tax revenue destination is entirely runtime-env-driven with no on-chain anchor.

### Post-build verification does NOT check program addresses against an expected manifest

`build.sh` Step 3 calls `scripts/verify-program-ids.ts`, which verifies that `declare_id!` in lib.rs matches the keypair-derived address, and that Anchor.toml matches. This is a consistency check — not an integrity check. If both the keypair and the `declare_id!` have been patched to an attacker's address, `verify-program-ids.ts` passes because they are mutually consistent.

### Architecture acknowledges the risk explicitly

`ARCHITECTURE.md` lists the build pipeline in the "IMPLICIT TRUST (UNVALIDATED)" zone:

> "Build pipeline (sync-program-ids.ts, patch-mint-addresses.ts)" — no on-chain validation

The upgrade-admin context (`07-upgrade-admin.md:96`) names this directly:

> "An attacker with access to the keypairs directory or the build scripts could inject arbitrary program IDs or mint addresses."

## Risk Assessment

**Likelihood:** LOW-MEDIUM in current state
- Requires local filesystem access to the build machine
- No remote exploit path identified
- Private keys are committed to git — anyone with repo access (contributors, CI systems with git clone access) can extract all 7 program keypairs
- If the repo is or ever becomes public, all program upgrade authorities are immediately exposed

**Impact:** CRITICAL if triggered
- Complete protocol compromise: all CPI routing subverted, all tax revenue redirected
- Attacker gains ability to sign as any PDA authority because they hold the private key for the program the PDA is derived from
- Irreversible without redeploying all 7 programs to new addresses
- On-chain there is no defense: the compiled constants ARE the security boundary

**Exploitability path ranking:**
1. Git repo access → extract committed private keys → deploy attacker programs at those addresses → no build tampering needed (keys ARE already known)
2. CI environment compromise → set `TREASURY_PUBKEY` env var → reroute all tax revenue
3. Build machine compromise → replace keypair JSON files → next build patches all program IDs

---

## Verdict

**POTENTIAL — HIGH severity.** The attack vector requires local build-machine access to subvert the compiled binary, which places this below CONFIRMED. However, a materially simpler related attack is already available: the 7 production program keypairs (including their private keys) are committed in git with world-readable permissions. Any person who has cloned the repository holds the private key for every production program's upgrade authority. This is an independent, directly exploitable credential exposure that enables program upgrades without any build-pipeline manipulation at all.

The build-pipeline supply-chain attack is therefore the secondary concern. The primary finding is that committed program secret keys give an adversary direct upgrade authority over all 7 programs.

---

## Recommendations

### R1 — CRITICAL: Remove committed secret keys from git history (Immediate)
All 7 production program keypairs committed to git contain full 64-byte secret keys. These must be rotated immediately (transfer upgrade authorities to new keypairs, then invalidate the old ones). The git history must be purged (git filter-repo or BFG Repo Cleaner) as the keys are already in all clones.

For devnet this is lower urgency; for mainnet this is a critical pre-launch requirement.

### R2 — HIGH: Restrict keypairs/ file permissions and access
Production keypair files should be `chmod 600` (owner read-write only), not `644`. They should never be in a git-tracked path. Consider storing them in a secrets manager (HashiCorp Vault, AWS Secrets Manager, or Solana CLI's default keystore with OS-level filesystem permissions).

### R3 — HIGH: Validate expected addresses in sync-program-ids.ts
Before patching, `sync-program-ids.ts` should compare each derived address against an expected value stored in a separate manifest file (e.g., `deployments/expected-ids.json`). If the derived address does not match the expected address, the script should abort with an error. This makes a keypair substitution attack loud instead of silent.

```typescript
// Proposed addition to sync-program-ids.ts
const EXPECTED = JSON.parse(fs.readFileSync("deployments/expected-ids.json", "utf8"));
for (const prog of PROGRAMS) {
  const derived = addressMap.get(prog.anchorName);
  if (EXPECTED[prog.anchorName] && derived !== EXPECTED[prog.anchorName]) {
    console.error(`SECURITY: ${prog.anchorName} keypair mismatch! Expected ${EXPECTED[prog.anchorName]}, got ${derived}`);
    process.exit(1);
  }
}
```

### R4 — MEDIUM: Validate TREASURY_PUBKEY against expected value
In `patch-mint-addresses.ts`, the treasury address must be validated against the known mainnet treasury pubkey (`3ihhwLnEJ2duwPSLYxhLbFrdhhxXLcvcrV9rAHqMgzCv`) before being patched into the binary. An unexpected value from the environment should abort the build.

### R5 — MEDIUM: Add a post-deploy on-chain verification step
After deployment, a verification script should fetch each program's `declare_id`-derived address from the chain and cross-check it against the `deployments/mainnet.json` manifest. This does not prevent the attack but creates an audit trail and catches mismatches before the protocol is used.

### R6 — LOW: Consider separating deploy keypairs from signing keypairs
The program keypairs in `keypairs/` serve as the program's address identity on Solana. The upgrade authority for a program is a separate, mutable on-chain field. Once all upgrade authorities are transferred to the Squads multisig (Phase 97 plan), the program keypairs in `keypairs/` become less sensitive (they identify programs but no longer sign upgrades). Document this distinction clearly so operators understand which keys are still sensitive post-Squads.
