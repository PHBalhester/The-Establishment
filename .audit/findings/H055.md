# H055: Epoch Skip Reward Forfeiture Quantification

**Status:** CONFIRMED (No Forfeiture; Finding Clarifies Prior Finding H041)
**Severity:** INFORMATIONAL
**Category:** Token Economics / Staking

## Summary

When epoch transitions skip (e.g., jumping from epoch 100 to epoch 105), a prior finding (H041) raised concern that staking rewards for epochs 101-104 are forfeited. Investigation confirms the opposite: **rewards are NOT forfeited**. The Synthetix cumulative reward-per-token model in this protocol accumulates rewards continuously regardless of epoch skips. However, the documentation comment in `trigger_epoch_transition.rs` (lines 145-148) is misleading and should be clarified.

## Code Evidence

**File:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:143-149`

```rust
// NOTE: Epoch skipping is acceptable. If the crank is delayed, expected_epoch
// may be > current_epoch + 1 (e.g., jumping from epoch 100 to epoch 105).
// This is harmless because:
// - No rewards accrue during gaps (staking tracks per-epoch)
// - Tax rates persist at last-set values (safe fallback)
// - Carnage only triggers from VRF (no implicit triggers from skips)
// The new epoch number is set directly to expected_epoch, not incremented.
```

The comment at line 146 states **"No rewards accrue during gaps"** — this is imprecise and could be misinterpreted.

**File:** `programs/staking/src/instructions/update_cumulative.rs` (reward accumulation)

The staking program uses the **Synthetix cumulative reward-per-token pattern**:
1. Tax Program calls `deposit_rewards(amount)` each swap → adds to `pending_rewards`
2. Epoch Program calls `update_cumulative()` at each epoch transition → converts pending to `rewards_per_token_stored`
3. User's earned reward = `user.staked_balance * (rewards_per_token_stored - user.checkpoint_rpt)`

**Key insight:** `rewards_per_token_stored` is cumulative across epochs. It never resets and only increases. Even if epochs 101-104 are skipped:
- All tax deposits during epochs 101-104 accumulate in `pending_rewards`
- When `update_cumulative` is called in epoch 105, the accumulated amount is converted to rpt_stored
- Users' rewards include all skipped epochs

**Verification via code logic:**
```rust
// From staking/src/helpers/math.rs (add_to_cumulative)
// Each time update_cumulative runs:
let delta = pool.pending_rewards / pool.total_staked;
pool.rewards_per_token_stored += delta;
pool.pending_rewards = 0;
```

The `pending_rewards` accumulates across all swaps, whether epochs skip or not. Once `update_cumulative` processes `pending_rewards`, it adds to `rpt_stored`. No rewards are lost; the update simply happens later.

## Verdict

**CONFIRMED: Rewards are NOT forfeited during epoch skips.**

The protocol's architecture is sound. The Synthetix reward model is designed to be epoch-agnostic: rewards accumulate based on total taxes collected, not on the number of epoch transitions. Epoch skipping merely delays when `pending_rewards` is finalized into `rpt_stored`, but does not lose the accumulated value.

**What the documentation should clarify:**
- Current: "No rewards accrue during gaps" (imprecise)
- Better: "Rewards continue to accrue from taxes during gaps. When epochs skip, update_cumulative processes all accumulated pending_rewards in a single update, ensuring no loss."

**Secondary observation (from H041):** H041 may have been concerned about the timing of `update_cumulative` calls relative to epoch skipping. This investigation confirms that the staking program's design **intentionally tolerates** variable-length epochs and makes no assumptions about one update per epoch. The design is robust.

**Recommendation:** Update the documentation comment in `trigger_epoch_transition.rs:146` to clarify that rewards are accumulated but delayed, not forfeited. This is **INFORMATIONAL** — no code change needed, but documentation clarity improves.
