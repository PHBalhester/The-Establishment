# H058: WSOL Intermediary Rent Assumption in Sell Path

**Status:** CONFIRMED
**Severity:** LOW
**Category:** Token Economics / Operational

## Summary

The sell-side tax flow creates and closes a WSOL intermediary account in the same instruction. The sequence is: (1) close intermediary, unwrapping WSOL to native SOL to swap_authority, (2) distribute SOL from swap_authority to three destinations, (3) re-create and re-initialize the intermediary. The design assumes swap_authority retains rent-exempt lamports (5000 lamports) for intermediary re-initialization, but provides no explicit guard to verify this assumption is met.

## Code Evidence

- **File:** `programs/tax-program/src/instructions/swap_sol_sell.rs:380-471`

Key operations:
```rust
// Step 10: Close intermediary (close_account CPI)
// Unwraps WSOL to native SOL, sends to swap_authority
// Line 380-420: close_wtoken CPI with invoke_signed

// Step 11: Distribute native SOL from swap_authority
// Line 423-471: Three System Program transfers (staking, carnage, treasury)

// Step 12: Re-create intermediary (create_account CPI)
// Line 472+: Requires rent-exempt SOL (~5000 lamports)
```

- **Related files:** `tax-program/src/helpers/tax_math.rs:105-107` (tax distribution split)

## Analysis

**The rent retention flow:**

1. **Close intermediary:** `close_wtoken()` CPI calls `spl_token::instruction::close_account()`, which:
   - Closes the WSOL token account (data discarded)
   - Transfers all lamports (rent + any remaining SOL) to the **closer** (swap_authority PDA)

2. **Distribute tax:** Three System Program transfers deduct from swap_authority:
   - staking_portion to staking_escrow
   - carnage_portion to Carnage sol_vault
   - treasury_portion to treasury_pubkey

3. **Re-create intermediary:** `initialize_wtoken()` CPI:
   - Calls `system_program::create_account()` with `WSOL_INTERMEDIARY_SIZE = 165` bytes
   - Requires rent_exempt amount (~5000 lamports on mainnet)
   - Calls `spl_token::initialize_account()` on the new account

**The assumption:**

The code assumes swap_authority has at least 5000 lamports **after** the three distributions. If the sum of the three distributions (staking + carnage + treasury) exceeds swap_authority's lamport balance minus 5000, the re-create instruction fails with insufficient funds.

**Risk quantification:**

Given the tax split (71/24/5 = 100%), the total tax distributed equals the gross tax collected. The close_account transfer provides:
- WSOL intermediary's data rent (~5000 lamports)
- Any SOL previously trapped in swap_authority from prior operations

For the re-create to fail, swap_authority would need to start with **zero additional SOL** and the intermediary would need to have been closed before without re-creation (impossible by instruction order). This is not a realistic failure mode because:

1. The WSOL intermediary is created fresh on deploy
2. Every sell closes and recreates it within the same instruction
3. The close_account transfer includes the rent-exempt amount (~5000 lamports) which is reserved for re-creation

**Potential edge case:**

If `close_account` were called twice in succession (via reentrancy or instruction batching), the second close would find zero lamports and fail. However, Token-2022 close_account checks the account is a valid token account; trying to close it twice in one slot would fail at the token program level (account no longer exists as a token account).

## Verdict

**Confirmed but low risk.** The assumption is implicit rather than explicit: the code relies on the close→distribute→recreate sequence executing atomically and rent-exempt lamports being preserved. No explicit rent-exempt guard exists, but the Solana runtime prevents lamport underflow (each account is owned by a program and its lamports can only decrease through that program's explicit transfers). The worst case is a failed sell instruction, not a fund loss or rent-exempt violation. The design is operationally sound: close_account's transfer to swap_authority includes the rent-exempt amount, which is then used for re-creation.

**Recommendation:** For defensive clarity, add a post-distribution check:
```rust
require!(
    ctx.accounts.swap_authority.lamports() >= WSOL_RENT_EXEMPT,
    TaxError::InsufficientRentForIntermediaryRecreation
);
```

This makes the assumption explicit and fails fast if the distribution logic ever changes. The check is zero-cost (one read), but provides operational confidence.
