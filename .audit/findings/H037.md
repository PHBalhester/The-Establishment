## Status

**CONFIRMED — REAL FINDING with a secondary root cause deeper than originally described.**

The original finding description is accurate: `tax-program/src/constants.rs:100` holds the mainnet AMM program ID (`5JsS...`) while `amm/src/lib.rs:12` currently has the devnet ID (`J7Jx...`). However, the root cause is more specific and more serious than a simple "sync resolves this" mitigation: **`sync-program-ids.ts` does NOT cover the Tax → AMM cross-reference at all**. The script's `CROSS_REFS` registry omits this entry entirely. Additionally, even if an entry were added, the patcher would silently fail because Tax Program uses `Pubkey::from_str("...").unwrap()` whereas the patcher only handles the `pubkey!("...")` macro pattern.

---

## Summary

`programs/tax-program/src/constants.rs` declares `amm_program_id()` returning `5JsSAL3kJDUWD4ZveYXYZmgm1eVqueesTZVdAvtZg8cR` (the Phase 95 mainnet/clean-devnet AMM ID). The AMM's own `declare_id!` in `programs/amm/src/lib.rs:12` currently holds `J7JxmNkzi3it6Q4TNYjTD6mKdgMaD1pxrstn1RnL3bR5` (an earlier devnet ID). These two values do not agree.

There are two compounding reasons the automation does not close this gap:

1. **Omission from CROSS_REFS registry**: `sync-program-ids.ts` explicitly documents which cross-program references it patches (lines 134–201). The registry covers Epoch → AMM, Staking → Tax, AMM → Tax, and test assertions. There is no entry for **Tax → AMM**. The header comment on line 11 (`epoch → tax_program_id(), amm_program_id(), staking_program_id()`) lists only Epoch-side references. Tax Program's `amm_program_id()` is never updated by the sync script.

2. **Incompatible pattern — `Pubkey::from_str` vs `pubkey!` macro**: Even if a Tax → AMM entry were added to `CROSS_REFS`, the patcher function `patchPubkeyMacroFn` targets the regex pattern `pubkey!("...")`. Tax Program's function body uses `Pubkey::from_str("...").unwrap()` — a different runtime-parse form. The regex would not match, and the patch would silently skip with "already correct" even though the value is wrong. Epoch Program's equivalent functions correctly use `pubkey!("...")` and are successfully patched.

A developer running `anchor build` directly (bypassing `build.sh`) will compile a Tax Program binary that validates AMM ownership against `5JsS...` while the actually deployed AMM has a different ID. This causes:
- The `owner = amm_program_id()` check in `pool_reader.rs` to accept pools owned by the mainnet AMM on devnet, or reject the live devnet AMM pool as unauthorized.
- `address = amm_program_id()` constraints on the `amm_program` account in swap instructions to fail, rejecting all swaps at account validation time.

## Analysis

### The Mismatch in Detail

| Location | Value | Cluster |
|---|---|---|
| `tax-program/src/constants.rs:100` | `5JsSAL3kJDUWD4ZveYXYZmgm1eVqueesTZVdAvtZg8cR` | Phase 95 mainnet/devnet-clean AMM |
| `amm/src/lib.rs:12` | `J7JxmNkzi3it6Q4TNYjTD6mKdgMaD1pxrstn1RnL3bR5` | Earlier devnet AMM |
| `epoch-program/src/constants.rs:26` | `J7JxmNkzi3it6Q4TNYjTD6mKdgMaD1pxrstn1RnL3bR5` | Earlier devnet AMM |
| `amm/src/constants.rs:10` (TAX_PROGRAM_ID) | `FGgidfhNLwxhGHpyH7SoZdxAkAyQNXjA5o8ndV3LkG4W` | devnet Tax |

Tax Program holds the Phase 95 AMM ID. Every other program (including AMM itself, Epoch, and AMM's own reference to Tax) holds a different, earlier devnet ID. The source is in a mixed-cluster state consistent with Tax Program being patched from a mainnet build cycle without a corresponding patch to its own `amm_program_id()`.

### Why sync-program-ids.ts Does Not Fix This

The script's `CROSS_REFS` array (lines 134–201) defines exactly seven cross-reference entries:

- Staking → Tax Program (identifier: `tax_program_id`)
- Staking → Epoch Program (identifier: `epoch_program_id`)
- Epoch → Tax Program (identifier: `tax_program_id`)
- **Epoch → AMM Program** (identifier: `amm_program_id`)
- Epoch → Staking Program (identifier: `staking_program_id`)
- AMM → Tax Program (identifier: `TAX_PROGRAM_ID`, pattern: `pubkey_macro_const`)
- Three test assertion entries for Epoch

There is no entry for **Tax → AMM** (`tax-program/src/constants.rs`, identifier `amm_program_id`). This cross-reference is simply absent from the registry.

### Why Adding the Entry Would Also Fail

`patchPubkeyMacroFn` constructs a regex that matches:

```
pub fn NAME() -> Pubkey { ... pubkey!("ADDRESS") }
```

Tax Program's `amm_program_id()` reads:

```rust
pub fn amm_program_id() -> Pubkey {
    Pubkey::from_str("5JsSAL3kJDUWD4ZveYXYZmgm1eVqueesTZVdAvtZg8cR").unwrap()
}
```

There is no `pubkey!` macro here. The regex match returns `None`, `patchPubkeyMacroFn` returns `{ patched: false }`, and the script logs "SKIP: already correct" — even though the value is wrong. The fix requires either: (a) converting Tax Program's functions to use `pubkey!` like Epoch does, or (b) adding a new patcher variant for the `Pubkey::from_str("...").unwrap()` form.

Tax Program's `epoch_program_id()` and `staking_program_id()` are also written with `Pubkey::from_str`. Cross-checking their values against current deployed state:

- `epoch_program_id()` → `E1u6fM9Pr3Pgbcz1NGq9KQzFbwD8F1uFkT3c9x1juA5h` (not in CROSS_REFS, never patched)
- `staking_program_id()` → `DrFg87bRjNZUmE6FZw5oPL9zGsbpdrVHrxPHSibfZv1H` (same as Epoch's value — consistent by accident)

The Staking and Epoch IDs in Tax constants happen to match current values and so the silent skip is benign today. The AMM ID does not match, making it an active mismatch.

### Runtime Impact

The `amm_program_id()` function in Tax Program is used in two places:

1. **`pool_reader.rs`** — validates that the pool account is owned by the AMM program before doing raw byte reads of reserves. A wrong ID either passes a rogue pool (if the ID happens to match something else) or rejects the real pool as not-owned-by-AMM.

2. **Swap instruction account constraints** — `address = amm_program_id()` on the `amm_program` `AccountInfo` in the buy/sell instruction structs. A mismatch causes Anchor to reject the transaction at deserialization time with a constraint error before any business logic runs.

Impact of the current mismatch on a naive `anchor build` (without `build.sh`):
- Tax Program compiled from source would expect AMM at `5JsS...`
- The live devnet AMM is at `J7Jx...`
- **Every swap (buy and sell) would fail** with an Anchor address constraint error
- The protocol would be completely non-functional

Impact during a correct build through `build.sh`:
- `sync-program-ids.ts` runs but does NOT patch Tax's `amm_program_id()`
- The mismatch survives into the compiled binary unchanged
- Whether this is benign depends on whether the keypair at `keypairs/amm-keypair.json` currently produces `5JsS...` or `J7Jx...`
- If the AMM keypair was regenerated (Phase 95 clean deploy), the compiled Tax Program still references the old ID

### Historical Context

The MEMORY.md entry for `sync-program-ids.ts` states: "New build.sh step [0/4] auto-syncs all program IDs from keypairs/ to declare_id!, Anchor.toml, cross-program refs, and target/deploy/." This claim is inaccurate with respect to Tax Program's cross-program references — they are not covered by the sync step. The Phase 95 deploy pitfalls memory documents 10 critical bugs found during mainnet rehearsal, and the tax-program `Pubkey::from_str` pattern is the likely cause of any Tax→AMM constraint failures that would have appeared during that rehearsal.

## Verdict

**Real, confirmed finding.** The severity is assessed as **Medium** rather than High because:

- The `build.sh` script must be followed for any deployment (it is the established pipeline, documented in CLAUDE.md and MEMORY.md).
- A naive `anchor build` producing a broken binary must still be deployed — which requires `deploy.sh` / `deploy-all.sh`, both of which call `build.sh` or equivalent steps.
- The defect would cause 100% swap failure if a broken binary were deployed, making it immediately visible in smoke testing (not a subtle data-corruption bug).

However, two factors prevent downgrading to Low:

1. **The automation gap is real**: The sync script's stated purpose is to make cross-program ID management a "one-command operation" and eliminate manual maintenance. It does not fulfill that promise for Tax Program's cross-references, creating a maintenance trap on every future keypair regeneration.

2. **Silent skip amplifies the gap**: The `Pubkey::from_str` incompatibility means the script not only omits the entry but would also silently fail to patch it even if the entry were added. A developer who adds the entry to `CROSS_REFS` but does not fix the patcher will believe the problem is solved while it remains broken.

## Recommendations

### R1 — Fix Tax Program's cross-program ID functions to use `pubkey!` macro (Preferred)

Convert all three cross-program ID functions in `programs/tax-program/src/constants.rs` from `Pubkey::from_str("...").unwrap()` to the `pubkey!("...")` macro form, consistent with Epoch Program and AMM:

```rust
// Before:
pub fn amm_program_id() -> Pubkey {
    Pubkey::from_str("5JsSAL3kJDUWD4ZveYXYZmgm1eVqueesTZVdAvtZg8cR").unwrap()
}

// After:
pub fn amm_program_id() -> Pubkey {
    pubkey!("5JsSAL3kJDUWD4ZveYXYZmgm1eVqueesTZVdAvtZg8cR")
}
```

Apply this change to `amm_program_id()`, `epoch_program_id()`, and `staking_program_id()` in `tax-program/src/constants.rs`. This makes the functions patchable by the existing `patchPubkeyMacroFn` logic.

The `pubkey!` macro is a compile-time constant evaluation (no runtime `unwrap()` panic risk) and is already used throughout Epoch and AMM constants for the same purpose.

### R2 — Add Tax → AMM (and Tax → Epoch, Tax → Staking) entries to CROSS_REFS

After applying R1, extend `scripts/deploy/sync-program-ids.ts` CROSS_REFS to cover Tax Program's outbound cross-references:

```typescript
// tax/constants.rs references amm + epoch + staking
{
  label: "Tax → AMM Program",
  file: "programs/tax-program/src/constants.rs",
  referencesProgram: "amm",
  pattern: "pubkey_macro_fn",
  identifier: "amm_program_id",
},
{
  label: "Tax → Epoch Program",
  file: "programs/tax-program/src/constants.rs",
  referencesProgram: "epoch_program",
  pattern: "pubkey_macro_fn",
  identifier: "epoch_program_id",
},
{
  label: "Tax → Staking Program",
  file: "programs/tax-program/src/constants.rs",
  referencesProgram: "staking",
  pattern: "pubkey_macro_fn",
  identifier: "staking_program_id",
},
```

Also add test assertion entries for Tax Program's `test_amm_program_id`, `test_staking_program_id` (if they exist), so the self-tests stay synchronized.

### R3 — Update the header comment in sync-program-ids.ts

The comment on line 9–13 currently claims:
```
 *   3. Cross-program references in constants.rs files:
 *      - staking → tax_program_id(), epoch_program_id()
 *      - epoch → tax_program_id(), amm_program_id(), staking_program_id()
 *      - amm → TAX_PROGRAM_ID const
```

Update it to accurately reflect the full set of covered references after R2 is applied, including Tax Program's outbound references.

### R4 — Resolve the current value disagreement before next build

Before the next `anchor build` or deploy, confirm which AMM ID is the intended target (devnet `J7Jx...` vs clean-deploy `5JsS...`) and manually correct `tax-program/src/constants.rs:100` to match. After R1 and R2 are applied, future builds will keep this synchronized automatically.

### R5 — Consider adding a pre-build cross-reference consistency check

`verify.ts` checks deployed on-chain addresses against `deployments/{cluster}.json`. Add a pre-build check (or extend `sync-program-ids.ts` output) that compares every cross-reference entry against the authoritative keypair-derived address and exits non-zero if any mismatch exists after patching. This would surface the silent-skip failure mode as an explicit error.
