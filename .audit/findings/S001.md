# S001: Carnage Suppression + Tax Arbitrage Combined Attack

**Status:** CONFIRMED
**Severity:** HIGH
**Category:** Economic, MEV, Access Control
**Inspired by:** H009 (CONFIRMED), H019 (CONFIRMED)

## Summary

H009 confirms that Carnage can be suppressed indefinitely by omitting `carnage_state` in `consume_randomness`. H019 confirms that VRF bytes are publicly readable 2-5 slots before `consume_randomness` lands, enabling forward-looking tax rate trades. This strategy investigates whether an attacker can exploit **both vulnerabilities simultaneously in the same epoch transition** — suppressing the Carnage buy while also front-running the tax rate change — and whether the combined profit exceeds either attack alone.

The answer is yes: the two attacks are fully composable with zero additional preconditions, and the combined profit is strictly super-additive for a sufficiently positioned attacker.

## Analysis

### Attack Composition Opportunity

Both vulnerabilities exploit the same ~2-5 slot window that opens when the Switchboard oracle reveals its VRF output (slot N+1 or N+2 after `trigger_epoch_transition` at slot N). During this window:

- **H009 vector**: VRF bytes 5-7 are known, so the attacker can calculate whether Carnage would trigger. Whether or not it would trigger, the attacker submits `consume_randomness` omitting `carnage_state` to permanently suppress it.
- **H019 vector**: VRF bytes 0-4 are known, so the attacker can compute the new tax rates (`derive_taxes()`). If the flip changes cheap/expensive sides, they know exactly which token's sell tax is about to increase.

**Both attacks require the same triggering condition** (Switchboard oracle reveal), fit into the same 2-5 slot window, and can be executed with the same funded wallet. There is no interaction penalty between them — they are mechanically independent but temporally aligned.

### Step-by-Step Combined Attack

```
Slot N:   trigger_epoch_transition fires
          - vrf_pending = true, old tax rates still active
          - Switchboard oracle queued with committed randomness account

Slot N+1: Switchboard oracle posts revealIx on-chain
          - RandomnessAccountData.get_value() returns 32 VRF bytes
          - Attacker reads bytes 0-7 off-chain in <100ms:
            bytes 0-4 → derive_taxes() → knows EXACT new rates
            bytes 5-7 → is_carnage_triggered() → knows if Carnage would fire

          DECISION TREE:
          A) Carnage would NOT trigger (96% probability):
             - Still submit consume_randomness WITHOUT carnage_state
               (No cost, ensures Carnage cannot fire even if code changes)
             - Execute tax rate front-run if flip is profitable

          B) Carnage WOULD trigger (4% probability):
             - CRITICAL to suppress: front-run crank with consume_randomness
               omitting carnage_state
             - Simultaneously execute tax rate front-run

Slot N+2: Honest crank's consume_randomness TX rejected (NoVrfPending)
          Tax rates updated. Carnage suppressed.

Slot N+3+: Attacker profits from:
           1) Prevention of Carnage buy pressure
           2) Having traded at favorable old tax rate
```

### Profit Calculation: Combined vs. Individual

**Individual attack profits (H009 alone, no position):**
Profit is indirect — preventing buy pressure benefits holders of the expensive token. Direct wallet profit from suppression alone = 0 unless you hold a position. Cost = 0.000005 SOL per epoch.

**Individual attack profits (H019 alone):**
For a 1000 SOL position in the about-to-become-expensive token:
- Old sell tax: 100-400 bps (cheap sell)
- New sell tax after flip: 1100-1400 bps (expensive sell)
- Alpha captured: (1400-100) bps max × 1000 SOL = up to **130 SOL** on a single transition

**Combined attack:**
The attacker in H019 necessarily holds a large position in one token. An attacker running H019 (tax front-run) has a **direct additional incentive** to suppress Carnage because:

1. If Carnage fires into the "cheap" token (the token the attacker just sold FROM), it will push the cheap token's price UP, partially reversing the attacker's exit position.
2. If Carnage fires into the "cheap" token (which is now the token the attacker did NOT exit), it will increase buy pressure on their remaining holdings — which sounds good, but it was THEIR sell pressure that created the divergence in the first place. Carnage undoes their advantage.
3. **Most importantly**: if the attacker runs the combined attack as a market actor, suppressing Carnage on EVERY triggering epoch prevents the cumulative vault balance from ever discharging, which amplifies the long-term price divergence the attacker exploits.

**Quantified uplift from combination:**

| Scenario | H019 alone | H009 alone | Combined |
|----------|-----------|-----------|----------|
| Epoch with 4.3% Carnage trigger, 75% flip | ~80 SOL alpha (1000 SOL position) | 0 direct | ~80 SOL alpha + prevents ~43 SOL buy pressure |
| Epoch with no Carnage trigger, 75% flip | ~80 SOL alpha | 0 direct | Same + zero cost suppression |
| Epoch with Carnage trigger, no flip | 0 alpha | 0 direct | 0 alpha + prevents Carnage buy pressure on held side |
| Long-run sustained (48 epochs/day) | Recurring MEV | Vault traps SOL | Both effects compound |

The 4.3% Carnage trigger probability means the two attacks overlap in roughly 4 epochs per 100. In those 4 epochs, the combined attacker:
- Prevents Carnage-induced price equalization
- Captures the tax rate alpha before equalization begins
- Has net benefit = tax alpha + prevented Carnage value

**Critical super-additivity condition:** An attacker who suppresses Carnage every epoch gradually traps the vault's 24% tax allocation, reducing systemic buy pressure. This means over time, the price divergence the H019 attacker exploits becomes *larger* and *more persistent*, increasing the alpha available from H019. The two attacks reinforce each other over multi-epoch windows.

### Defender's Dilemma

If only H009 is fixed (making `carnage_state` mandatory), H019 remains. If only H019 is fixed (requiring `taxes_confirmed` before swapping), H009 remains. The two fixes are independent but both necessary. A protocol that fixes only one leaves the other attack surface open — and because the attacks share a common observation point, a sophisticated attacker can still run whichever remains viable.

## Verdict

**CONFIRMED — HIGH severity.** The combined attack is feasible, rational, and strictly more profitable than either individual attack for any actor with a pre-positioned token holding. The super-additivity arises because:

1. Vault suppression amplifies long-term price divergence (benefits H019 attacker)
2. Tax front-running is most valuable when the token is about-to-lose-cheap-status, which often coincides with high Carnage probability (both effects peak at the same moment)
3. Both attacks have the same precondition (Switchboard oracle reveal) and can be executed with the same wallet in the same ~2-slot window

The combined attack does not introduce new vulnerabilities beyond H009 and H019 — it demonstrates they are co-exploitable and mutually reinforcing.

## Recommendations

All recommendations from H009 and H019 apply independently. Additionally:

1. **Fix H009 first** (mandatory `carnage_state`) as it is the lower-friction fix and also removes the combined attack surface for the Carnage-suppression component.
2. **Fix H019 second** (add `taxes_confirmed` guard in swap instructions) — after H009 is fixed, the combined attack reduces to the pure H019 case.
3. **Monitoring for co-occurrence**: Implement off-chain monitoring that flags any epoch where: (a) `consume_randomness` succeeds without emitting `CarnagePending` or `CarnageNotTriggered` AND (b) a large-size sell TX lands within 3 slots of oracle reveal. This pattern is the fingerprint of a combined attacker.
4. **Do not treat the fixes as optional sequentially** — if only one fix ships before mainnet, the other attack surface remains, and a combined attacker can simply pivot to the still-open path.
