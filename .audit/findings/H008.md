# H008: Mainnet Placeholder Fix Verification

**Status:** POTENTIAL
**Severity:** HIGH
**Category:** Upgrade/Admin, Configuration
**Origin:** RECHECK (H018)

## Summary

The original H018 finding (Pubkey::default() placeholders) is partially resolved: the treasury address is correctly set to `3ihhw...` and `Pubkey::default()` is no longer present in mainnet-path treasury code. However, a new configuration gap was introduced: the mainnet branches for all six mint-address functions across Bonding Curve and Conversion Vault are hardcoded to the stale Phase 69/95 devnet addresses, not the vanity mainnet mints (`cRiME...`, `FraUd...`, `pRoFiT...`). No `compile_error!` guards exist anywhere in any program.

## Previous Finding

H018 (Audit #2) identified 7 mainnet-path `Pubkey::default()` placeholders across Tax Program (`treasury_pubkey()`), Conversion Vault (`crime_mint()`, `fraud_mint()`, `profit_mint()`), and Bonding Curve (`crime_mint()`, `fraud_mint()`, `epoch_program_id()`). The root risk was that a mainnet build would silently use zero-address pubkeys, causing fund loss (treasury) or non-functional programs (mints). H018 recommended adding `compile_error!` guards to make accidental deployment impossible.

## Current Analysis

### Issue 1: Treasury pubkey — FIXED

`tax-program/src/constants.rs:146-149` now correctly returns `3ihhwLnEJ2duwPSLYxhLbFrdhhxXLcvcrV9rAHqMgzCv` in the non-devnet/non-localnet branch. This is the dedicated mainnet treasury wallet confirmed in project documentation. The `Pubkey::default()` that triggered H018 is gone. This part of H018 is fully resolved.

### Issue 2: Bonding Curve mainnet mint addresses — STALE (NEW GAP)

`bonding_curve/src/constants.rs:176-195` has the mainnet branches of `crime_mint()` and `fraud_mint()` returning devnet addresses:

- `crime_mint()` mainnet branch: `DtbDMB2dU8veALKTB12fi2HYBKMEVoKxYTbLp9VAvAxR` (Phase 69/95 devnet mint)
- `fraud_mint()` mainnet branch: `78EhS3i2wNM8RQMd8U3xX4eCYm5Xytr2aDcCUH4BzNtx` (Phase 69/95 devnet mint)

The mainnet vanity mints are:
- CRIME: `cRiMEhAxoDhcEuh3Yf7Z2QkXUXUMKbakhcVqmDsqPXc`
- FRAUD: `FraUdp6YhtVJYPxC2w255yAbpTsPqd8Bfhy9rC56jau5`

These are absent from the entire `programs/` directory tree. The bonding curve's `initialize_curve` constraint checks `token_mint.key() == crime_mint() || token_mint.key() == fraud_mint()`. If deployed to mainnet with these constants, the constraint would reject the real mainnet mints and accept the devnet mints (which do not exist on mainnet). All bonding curves would be non-functional, but the failure mode is closed (constraint rejection, no fund loss).

The `epoch_program_id()` in bonding curve (`E1u6fM9...`) matches the current git `declare_id!` in `epoch-program/src/lib.rs`. This is consistent — the build pipeline (`sync-program-ids.ts`) patches IDs from keypairs at build time, so the current source reflects the last deployed devnet IDs. The mainnet branch of this function carries the same ID, which will be patched to the correct mainnet ID at mainnet build time. This is expected behavior per the documented build pipeline, not a gap.

### Issue 3: Conversion Vault mainnet mint addresses — STALE (NEW GAP)

`conversion-vault/src/constants.rs:35-68` has all three mainnet branches returning devnet addresses:

- `crime_mint()` mainnet: `DtbDMB2dU8veALKTB12fi2HYBKMEVoKxYTbLp9VAvAxR`
- `fraud_mint()` mainnet: `78EhS3i2wNM8RQMd8U3xX4eCYm5Xytr2aDcCUH4BzNtx`
- `profit_mint()` mainnet: `Eaipvk74Cw7CYUJNafsai9jxQ913V76MF9EfdQ3nNp2a` (Phase 69/95 devnet PROFIT)

The mainnet vanity PROFIT mint is `pRoFiTj36haRD5sG2Neqib9KoSrtdYMGrM7SEkZetfR`. The conversion vault's `initialize` instruction validates all three mints against these constants. With devnet addresses in the mainnet build, the vault would reject real mainnet mints and only accept mints that do not exist on mainnet. Vault is non-functional, failure is closed (no fund loss).

### Issue 4: compile_error! guards absent — CONFIRMED NOT FIXED

H018 explicitly recommended adding `compile_error!` macros to all mainnet placeholder functions. A search across all program source files finds zero instances of `compile_error!`. The H018 recommendation was not implemented. However, the nature of the remaining gap has changed: instead of `Pubkey::default()` (which would compile and silently fail at runtime), the current code has real-but-wrong addresses (which would also compile and silently fail at runtime, but by rejecting real mainnet mints rather than by sending funds to the wrong place).

The absence of `compile_error!` guards means:
- A mainnet build with stale devnet mint addresses compiles without any warning
- The only safeguard is the `patch-mint-addresses.ts` build script, which must be run correctly
- If `anchor build` is run directly without `build.sh`, the stale addresses ship to mainnet

### Severity Assessment

The original H018 worst case (treasury to `Pubkey::default()` = irrecoverable fund loss) is resolved. The remaining gap is a deployment configuration risk:

- Programs will fail closed (reject valid mints rather than accept them), not open
- No user funds are at risk from a misconfigured-mint deployment — operations just brick
- The risk is operational disruption and redeploy cost, not financial loss
- `patch-mint-addresses.ts` is documented as a mandatory build step

This moves the severity from CRITICAL (potential fund loss) to HIGH (guaranteed non-functional mainnet deploy if build pipeline is skipped or broken).

## Verdict

**POTENTIAL** — The critical H018 failure mode (treasury funds to `Pubkey::default()`) is resolved. A new but lower-severity gap exists: the mainnet branches of 5 mint-address functions across Bonding Curve and Conversion Vault contain stale devnet addresses instead of the finalized mainnet vanity mints. These programs would be non-functional if deployed with these constants, but would fail closed with no fund loss. The `patch-mint-addresses.ts` build script is the operational control preventing this, but without `compile_error!` guards there is no compile-time safety net if that script is bypassed. The risk is a broken mainnet launch requiring a redeploy, not a financial exploit.

## Recommendations

1. **Add `compile_error!` guards (H018 recommendation, still open)**: Add these to every mainnet mint-address function that currently holds a stale or placeholder address. This converts a potential silent runtime failure into a guaranteed compile-time failure, making it impossible to accidentally deploy without running the full build pipeline.

```rust
#[cfg(not(any(feature = "devnet", feature = "localnet")))]
pub fn crime_mint() -> Pubkey {
    compile_error!("MAINNET: crime_mint() not patched. Run scripts/deploy/build.sh before building for mainnet.");
}
```

2. **Pre-flight check in build.sh**: After `patch-mint-addresses.ts` runs, add a grep-based assertion that `cRiME`, `FraUd`, and `pRoFiT` appear in the patched files before proceeding to `anchor build`. Catch address injection or script failure early.

3. **Mainnet deploy checklist gate**: Add an explicit pre-deploy step to verify that none of the mainnet-branch constant functions return known devnet addresses (`DtbD...`, `78EhS...`, `Eaipvk...`) after patching.

4. **Consider explicit `feature = "mainnet"` flag**: The current pattern uses `not(any(devnet, localnet))` as a silent mainnet fallback. An explicit `feature = "mainnet"` combined with `compile_error!` on the `not(any(...))` branch would require developers to consciously choose a target cluster, eliminating accidental defaults.
