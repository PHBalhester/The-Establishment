# H020: No Emergency Pause Mechanism

**Status:** CONFIRMED
**Severity:** HIGH
**Category:** Upgrade/Admin, Risk Management
**Origin:** RECHECK (S005)

## Summary

No program in the Dr. Fraudsworth protocol implements any form of pause, freeze, or emergency halt functionality. If a critical vulnerability is discovered post-launch, the sole available response is a full program upgrade executed through the Squads 2-of-3 multisig. During the mandatory timelock delay — and the additional time required to write, build, and stage the fix — the vulnerable code remains live and exploitable. Three independent prior audit documents (S005, context/07-upgrade-admin, context/03-state-machine) all independently confirmed this gap.

## Analysis

### Absence of Pause Mechanism (Complete Search)

A full regex search across all seven production programs for `is_paused`, `paused`, `pause`, `emergency`, `freeze`, `halt`, and `circuit_break` returned zero matches in any instruction handler, state struct, or helper. The only occurrences of `freeze` in the codebase are test-fixture comments setting `freeze_authority: None` on synthetic mints. This is consistent with the explicit statement in `context/07-upgrade-admin.md`:

> "No pause/emergency mechanism exists — There is zero pause functionality across all 7 programs. No `is_paused`, no `frozen`, no emergency stop."

And confirmed independently in `context/03-state-machine.md`:

> "No pause/freeze mechanism exists for any program: No `is_paused`, `frozen`, or emergency stop pattern was found in any production program."

### Exploit Window Quantification

When a critical bug is found post-launch, the remediation path is:

| Step | Activity | Time Estimate |
|------|-----------|---------------|
| 1 | Detect and triage the bug | 5–60 min |
| 2 | Write and review the patch | 15–120 min |
| 3 | `anchor build` (single program, feature-flagged) | 1–3 min |
| 4 | Create Squads proposal, get 2-of-3 signatures | 5–30 min |
| 5 | **Wait out the mandatory timelock** | see below |
| 6 | Execute upgrade on-chain, verify | 1–5 min |

**Timelock values documented across project sources (conflict noted):**
- `mainnet-governance.md` Section 3 / `SQUADS_TIMELOCK_SECONDS=900`: **15 minutes** at launch
- `mainnet-deploy-checklist.md` Stage 7 / timelock progression table: **300 seconds (5 minutes)** at launch

There is a factual discrepancy between these two governing documents. Regardless of which value is authoritative:

| Timelock Phase | Total Minimum Exploit Window |
|---------------|------------------------------|
| Launch (Day 0) | ~25 min (300s lock) to ~45 min (900s lock) + fix time |
| +48-72 hours | ~1 hr + fix time (3600s) |
| +1 month | ~24 hr + fix time (86400s) |
| +3 months | ~7 days + fix time (604800s) |

The word "minimum" is key: the clock does not start until the vulnerability is discovered. In a scenario where a sophisticated attacker finds the bug before the team, the exploit window begins at attacker awareness, not at team awareness.

### Programs Where Pause Would Be Most Effective

All user funds flow through exactly two program entry points:

1. **Tax Program** (`swap_sol_buy`, `swap_sol_sell`) — all token swap value flows through here. A single `paused: bool` field on the state account, checked at instruction entry, would freeze all swaps without touching any other program.
2. **Epoch Program** (`trigger_epoch_transition`, `consume_randomness`, `execute_carnage_*`) — controls epoch transitions and Carnage fund deployment. Pausing this halts new epoch starts and Carnage execution while leaving in-progress state intact.

The Bonding Curve, Staking, and Conversion Vault have independent state and could be paused separately or share a protocol-wide pause flag.

### Why the Risk is Elevated for This Protocol

Several protocol-specific factors amplify the impact of the missing pause mechanism:

1. **4-level CPI chain at the hard limit**: The Carnage execution path (Epoch → Tax → AMM → Token-2022 → Hook) operates at Solana's 4-deep CPI limit. A bug here is both hard to detect before exploitation and hard to patch quickly (potential cascade across multiple programs).

2. **Cross-program raw byte reads (offsets 137/145)**: If a future AMM upgrade corrupts the PoolState byte layout, silent wrong-value reads occur in both Tax and Epoch programs. There is no on-chain version check to catch this. A pause on Tax/Epoch would be the only way to stop ongoing silent corruption while the cascade rebuild is prepared.

3. **Fallback Carnage path minimum_output=0**: `execute_carnage` (fallback) passes `minimum_output=0` to the AMM. In a scenario where the AMM reserves are manipulated, a paused fallback path would prevent value extraction during the fix window.

4. **No on-chain parameter updates**: All economic parameters are hardcoded constants. There is no `set_fee` or `update_config` instruction. If a tax rate, slippage floor, or bounty value causes an issue, only a full upgrade can change it. A pause buys time without requiring a full upgrade.

### Cross-Document Discrepancy: Timelock Duration at Launch

`mainnet-governance.md` (Section 3, `SQUADS_TIMELOCK_SECONDS=900`) specifies **15 minutes** as the initial timelock. `mainnet-deploy-checklist.md` (Stage 7.4, timelock progression table) specifies **300 seconds (5 minutes)**. These documents disagree on a security-critical parameter. This needs resolution before mainnet deployment.

## Verdict

**CONFIRMED — HIGH severity.**

The protocol has zero circuit-breaker capability at the application layer. The Squads timelock is the only delay between a discovered exploit and a patch reaching mainnet, but the timelock clock does not start until the fix is staged and 2-of-3 signers approve — meaning the true exploit window is `detection_delay + fix_time + signing_coordination + timelock_delay`. At the most permissive planned timelock (5 min), this is still likely 30–90 minutes minimum. At the progressively longer timelocks (1 hr, 24 hr, 7 days), it is operationally impractical to rely on an upgrade as an emergency response.

This is a known, accepted risk documented across multiple prior audit files. It has not been mitigated since S005 was filed. The severity is HIGH (not CRITICAL) because: (1) it requires an independently exploitable vulnerability to be present — it is a response-capability gap, not a direct exploit; (2) the Squads upgrade authority is retained and functional, providing the remediation path; and (3) the 5–15 minute initial timelock does provide a narrow response window for truly trivial patches.

The timelock discrepancy (300s vs 900s) between governance documents is a separate but related issue that must be resolved before mainnet deployment, as it affects the planned hotfix response window.

## Recommendations

1. **Add a protocol-wide pause flag to Tax Program** (highest priority). A `paused: bool` field on a new or existing admin config account, checked at the top of `swap_sol_buy` and `swap_sol_sell`, can be toggled by the Squads vault via a single-instruction admin call — no timelock required for a boolean flip if designed as a separate admin-only instruction distinct from a program upgrade. This is the minimum viable circuit breaker.

2. **Add a pause check to Epoch Program** on `trigger_epoch_transition` and `execute_carnage_*`. This prevents new epochs from starting and new Carnage from being dispatched while a fix is staged, without disrupting already-confirmed state.

3. **Design the pause toggle as a non-upgrade admin instruction**, not a program change. This means it can be executed by the Squads vault immediately (no timelock) because it is a transaction calling an existing instruction, not a BPF loader upgrade. The timelock in Squads applies to program upgrades; standard CPI/instruction transactions can have zero delay if the multisig threshold (2-of-3) is met.

4. **Resolve the timelock discrepancy** between `Docs/mainnet-governance.md` (900s / 15 min) and `Docs/mainnet-deploy-checklist.md` (300s / 5 min) before mainnet deployment. Pick one value and update both documents.

5. **Document an emergency response runbook**: Exact steps, expected timing, who contacts whom, and how to coordinate 2-of-3 signers in an emergency (one may be a Ledger requiring physical access). S005 recommended this; it remains absent.

6. **Pre-stage patches for known high-risk paths**: The Carnage fallback MEV path (minimum_output=0) and the cross-program byte-offset read are already identified risks. Pre-writing, compiling, and staging — but not executing — these patches means the build step is eliminated from the emergency timeline.
