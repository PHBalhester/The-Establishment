# H013: VRF Freshness Underflow via Future-Dated Seed Slot

**Status:** POTENTIAL
**Severity:** LOW
**Category:** Oracle, Timing
**Origin:** KB (EP-022)

---

## Summary

The VRF freshness check in `trigger_epoch_transition.rs` and `retry_epoch_vrf.rs` uses `saturating_sub` to compute the age of a Switchboard randomness account. If `seed_slot > clock.slot` (future-dated), `saturating_sub` returns 0 instead of wrapping, and 0 passes the `<= 1` freshness gate. The vulnerability is real at the code level, but practical exploitability is effectively nil under the current Switchboard On-Demand architecture because:

1. `seed_slot` is written by the Switchboard program itself, not by the caller.
2. The `get_value()` gate in `consume_randomness` independently requires `clock.slot == reveal_slot`, which cannot be satisfied with a pre-staged future-dated account in the same transaction slot.
3. Exploiting this requires compromising or impersonating the Switchboard program at address level.

The finding is classified **POTENTIAL** (not NOT VULNERABLE) because the on-chain code contains no explicit defensive `require!(seed_slot <= clock.slot)` check, leaving residual trust placed in Switchboard oracle correctness that is not verified on-chain.

---

## Analysis

### The Vulnerability Mechanism

At `trigger_epoch_transition.rs:174`:

```rust
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
// ...
require!(slot_diff <= 1, EpochError::RandomnessExpired);
```

Intended: reject any randomness account whose `seed_slot` is more than 1 slot old.
Actual: if `seed_slot > clock.slot`, `saturating_sub` returns `0`. `0 <= 1` is `true` — the check passes silently.

The identical pattern appears at `retry_epoch_vrf.rs:83`:

```rust
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
// ...
require!(slot_diff <= 1, EpochError::RandomnessExpired);
```

### What `seed_slot` Is and Who Controls It

The Switchboard On-Demand Rust crate (version 0.11.3, confirmed in the project's Cargo registry) defines `RandomnessAccountData` as a `#[repr(C)]` struct. The `seed_slot` field is written by the Switchboard on-chain program when a randomness account is created. The struct is:

```rust
pub struct RandomnessAccountData {
    pub authority: Pubkey,
    pub queue: Pubkey,
    pub seed_slothash: [u8; 32],
    pub seed_slot: u64,      // ← Written by Switchboard program, NOT caller
    pub oracle: Pubkey,
    pub reveal_slot: u64,
    pub value: [u8; 32],
    ...
}
```

The protocol validates account ownership via `owner = SWITCHBOARD_PROGRAM_ID` at the Anchor account constraint level:

```rust
#[account(owner = SWITCHBOARD_PROGRAM_ID @ EpochError::InvalidRandomnessOwner)]
pub randomness_account: AccountInfo<'info>,
```

This means any account passed as `randomness_account` must be owned by the Switchboard program. To produce a future-dated `seed_slot`, an attacker would need to either:
- Compromise the Switchboard oracle network to create accounts with falsified `seed_slot` values, OR
- Control the Switchboard program itself (i.e., its upgrade authority)

Neither scenario is within the threat model of a non-state-level attacker targeting this protocol.

### The `get_value()` Secondary Gate

Even if the freshness check were bypassed via a future-dated `seed_slot`, the attack would still need to pass the "not-yet-revealed" check at `trigger_epoch_transition.rs:185`:

```rust
if randomness_data.get_value(clock.slot).is_ok() {
    return Err(EpochError::RandomnessAlreadyRevealed.into());
}
```

And then at `consume_randomness.rs:169`, the actual VRF bytes can only be read if:

```rust
pub fn get_value(&self, clock_slot: u64) -> Result<[u8; 32], OnDemandError> {
    if clock_slot != self.reveal_slot {  // ← STRICT EQUALITY
        return Err(OnDemandError::SwitchboardRandomnessTooOld);
    }
    Ok(self.value)
}
```

`get_value` requires `clock.slot == reveal_slot`. This is a strict equality check. A pre-staged account with a future-dated `seed_slot` would have its `reveal_slot` populated only after the oracle actually reveals (which is a subsequent transaction). The attacker cannot control which slot `consume_randomness` is called in to force `clock.slot == reveal_slot` for an account whose reveal timing is oracle-controlled.

### Hypothetical Attack Scenario (Oracle Compromise)

If Switchboard's oracle network were compromised to produce a randomness account where:
- `seed_slot = clock.slot + N` (N > 0, future-dated)
- `reveal_slot = T` (some pre-determined future slot)
- `value = [attacker_controlled_bytes...]`

Then:
1. At TX2 (`trigger_epoch_transition` at slot T-1 or earlier): `saturating_sub` returns 0, passes freshness. Commit succeeds. Account bound.
2. At TX3 (`consume_randomness` at exactly slot T): `get_value(T)` returns `Ok(attacker_bytes)`. VRF consumed with attacker-controlled bytes.

Impact: The attacker controls tax rates for one epoch and determines whether/how Carnage triggers. Given tax rates are bounded to {100, 200, 300, 400, 1100, 1200, 1300, 1400} BPS and Carnage is deterministic, the attacker could pick the most advantageous epoch outcome (e.g., force Carnage trigger on a specific token, or force favorable tax asymmetry).

This scenario requires sustained Switchboard oracle network compromise, which is far beyond this protocol's individual attack surface.

## Oracle Trust Model Assessment

The protocol's trust model for Switchboard is documented accurately in `.audit/ARCHITECTURE.md`:

> "Switchboard Oracle | SEMI-TRUSTED | Provide VRF randomness | External — validated by program owner check"

and

> "Switchboard oracle provides honest seed_slot (UNVALIDATED on-chain; saturating_sub masks future dates)"

**What IS validated on-chain:**
1. Account owner == SWITCHBOARD_PROGRAM_ID (Anchor constraint, both instructions)
2. Account data structure parses correctly via `RandomnessAccountData::parse()` (discriminator check)
3. Anti-reroll: same account bound at commit must be used at reveal (pubkey equality)
4. Reveal status: `get_value()` returns `Ok` only when `clock.slot == reveal_slot`

**What is NOT validated on-chain:**
1. `seed_slot <= clock.slot` — the sign of the freshness difference is unchecked
2. The integrity of the `seed_slothash` content (trusted as correctly set by Switchboard)
3. The `oracle` field (which oracle provided the value)

The `owner = SWITCHBOARD_PROGRAM_ID` constraint is the primary defense. Any account passing this constraint was created and is managed by the Switchboard program, which is an audited, production protocol. The probability of Switchboard producing a future-dated `seed_slot` in normal operation is effectively zero — this would be a Switchboard bug or an active compromise of the Switchboard oracle network itself.

**Threat level categorization:**
- **External attacker (no Switchboard access):** Cannot exploit. Cannot forge an account passing the owner check.
- **Malicious crank operator:** Cannot exploit. Crank creates the randomness account client-side, but the `seed_slot` is written by Switchboard's on-chain program, not the crank.
- **Compromised Switchboard oracle node (single):** May influence `reveal_slot` timing and `value`, but `seed_slot` is set by the Switchboard program at account creation, not oracle-node-controlled.
- **Switchboard program upgrade authority compromise:** Could theoretically deploy a modified Switchboard program that writes future-dated `seed_slot`. This represents a supply-chain attack at the infrastructure level, not a protocol-level vulnerability.

## Verdict

**POTENTIAL — Low Severity**

The `saturating_sub` pattern creates a real code-level gap where future-dated `seed_slot` values bypass the freshness check. However, the condition required to produce a future-dated `seed_slot` (Switchboard program or oracle compromise) is outside the realistic threat model for this protocol, and a second independent gate (`get_value`'s strict `clock.slot == reveal_slot` equality) would need to also be satisfied in coordination with the bypass.

The vulnerability is **not NOT VULNERABLE** because:
- The on-chain code contains no explicit defensive check against future-dated values
- The protocol implicitly trusts Switchboard oracle correctness for this property without asserting it
- Defense-in-depth best practice would add an explicit assertion

The severity is **LOW** (not MEDIUM or HIGH) because:
- Exploiting this requires infrastructure-level compromise of Switchboard
- The `get_value` gate provides independent correctness assurance
- Even if exploited, the outcome is bounded economic manipulation within protocol-defined ranges, not fund theft or permanent state corruption

---

## Recommendations

**R1 — Add explicit future-date rejection (defensive hardening, low effort):**

In both `trigger_epoch_transition.rs` and `retry_epoch_vrf.rs`, add a guard immediately before the `saturating_sub` computation:

```rust
// Explicitly reject future-dated seed_slot before computing difference.
// saturating_sub would silently return 0 for future dates, which passes the <= 1 check.
require!(
    randomness_data.seed_slot <= clock.slot,
    EpochError::RandomnessFutureDated
);
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
require!(slot_diff <= 1, EpochError::RandomnessExpired);
```

Add `RandomnessFutureDated` to `errors.rs`. This eliminates the implicit trust on `seed_slot` honesty and makes the invariant explicit and auditable.

**R2 — Document the trust assumption explicitly in code:**

If R1 is deferred, add an inline comment at the `saturating_sub` line explicitly documenting the trust boundary:

```rust
// TRUST ASSUMPTION: seed_slot is honest (not future-dated).
// If seed_slot > clock.slot, saturating_sub returns 0, which passes this check.
// Defense: randomness_account.owner == SWITCHBOARD_PROGRAM_ID is enforced above;
// a future-dated seed_slot would require Switchboard oracle compromise.
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
```

**R3 — Priority:** This is a hardening recommendation only. It should not block mainnet deployment but should be included in the next program upgrade cycle.
