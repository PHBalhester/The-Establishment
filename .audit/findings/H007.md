# H007: Cross-Program Layout Test Adequacy

**Status:** POTENTIAL
**Severity:** LOW
**Category:** CPI, Arithmetic
**Origin:** RECHECK (S007/H011)

## Summary

The cross-crate layout test crate (`tests/cross-crate/`) was implemented in response to S007/H011 and substantially addresses the previously reported gaps. The primary `epoch_to_tax_round_trip` test is thorough and covers all 23 fields of EpochState, and `byte_length_parity` provides a byte-for-byte serialization comparison. Two minor issues remain: (1) the `tax_to_epoch_round_trip` test only asserts 8 of 23 fields, making it a partial coverage test that would not catch layout drift in the reverse direction for 15 fields; (2) the `epoch_state.rs` doc comment contains a duplicate listing of `initialized` and `bump` (lines 41-45 list them twice, once before and once after `reserved`), which is a maintenance accuracy concern but does not affect runtime behavior.

## Previous Finding

**S007** (Audit #2): No cross-program layout tests existed anywhere in the test suite. Only a static size assertion existed in the Epoch Program (`DATA_LEN == 100`), and the Tax Program mirror had no compile-time assertion at all. The finding recommended creating a workspace-level test crate with round-trip tests for all three layout couplings (EpochState, AMM PoolState byte offsets, Staking discriminator).

**H011** (Audit #2): Confirmed S007's finding with a detailed field-by-field comparison showing the layouts matched at the time. Also flagged that the Tax mirror lacked a `DATA_LEN` compile-time assertion, that no cross-crate compile-time link existed, and recommended adding a `reserved: [u8; 64]` padding field to avoid future reallocation.

## Current Analysis

### What Was Fixed

All three major recommendations from S007/H011 have been implemented:

1. **Cross-crate test crate created**: `tests/cross-crate/src/lib.rs` exists with a `Cargo.toml` that correctly imports both `epoch-program` and `tax-program` as dev-dependencies using `no-entrypoint` and `localnet` features.

2. **Reserved padding added**: Both `epoch_state.rs` and `epoch_state_reader.rs` now include `pub reserved: [u8; 64]`, implementing the DEF-03 recommendation. This eliminates the need for account reallocation when adding fields in the future.

3. **DATA_LEN compile-time assertion added to the Tax mirror**: `epoch_state_reader.rs:64` now has `const _: () = assert!(EpochState::DATA_LEN == 164);`, matching the equivalent assertion in `epoch_state.rs:187`.

4. **DATA_LEN values agree**: Both programs compute `DATA_LEN = 8+4+8+1+2+2+2+2+2+2+8+1+1+32+1+1+1+8+8+4+64+1+1 = 164`. Independently verified: the sum is correct.

5. **EpochState field count**: Both structs now have 23 fields (the previous H011 count of 22 reflected the pre-`reserved` layout; `reserved` has since been added to both).

### Remaining Issue 1: Partial Coverage in `tax_to_epoch_round_trip`

The `epoch_to_tax_round_trip` test (lines 74-116) is the primary test and asserts all 23 fields â€” it is complete.

The `tax_to_epoch_round_trip` test (lines 119-144) asserts only **8 of 23 fields**:
- Asserted: `genesis_slot`, `current_epoch`, `bump`, `crime_buy_tax_bps`, `fraud_sell_tax_bps`, `pending_randomness_account`, `last_carnage_epoch`, `reserved`
- Not asserted (15 fields): `epoch_start_slot`, `cheap_side`, `low_tax_bps`, `high_tax_bps`, `crime_sell_tax_bps`, `fraud_buy_tax_bps`, `vrf_request_slot`, `vrf_pending`, `taxes_confirmed`, `carnage_pending`, `carnage_target`, `carnage_action`, `carnage_deadline_slot`, `carnage_lock_slot`, `initialized`

However, this test's limitation is **largely mitigated** by two other tests in the same file:
- `byte_length_parity` (lines 147-166) serializes both structs independently and asserts `epoch_buf == tax_buf` â€” a byte-for-byte comparison that is stronger than any field-level assertion. If any field has a different value or occupies a different position, this test will catch it.
- The `epoch_to_tax_round_trip` test already validates the primary cross-program read direction (Tax Program reading Epoch Program data).

The `byte_length_parity` test effectively provides complete coverage for the `tax_to_epoch_round_trip` direction. The missing field assertions in `tax_to_epoch_round_trip` are a documentation/readability gap rather than a security gap.

### Remaining Issue 2: Doc Comment Duplicate Listing in `epoch_state.rs`

Lines 38-45 of `epoch_state.rs` list `initialized` and `bump` twice in the doc comment: once before `reserved` (lines 41-42) and once after (lines 44-45). The `reserved: 64 bytes` line appears to have been inserted between the two listings when padding was added, leaving the original `initialized`/`bump` entries stranded above it. The actual struct layout is correct (reserved appears before initialized/bump in the field declarations), but the doc comment is misleading about field order.

This is a documentation accuracy issue, not a correctness issue, but it could confuse a future developer updating the layout.

### AMM PoolState Byte Offsets (S007 Item 2)

The previous S007 finding also recommended tests for AMM PoolState byte offsets at [137..145] and [145..153]. No dedicated test for these offsets appears in the cross-crate test crate. This was a known recommendation from S007 ("Test 2: AMM PoolState Byte Offset Stability"). However, the byte offsets were verified as correct in H011 and the AMM PoolState struct is unchanged. The absence of a test for these offsets is a residual gap from S007.

### Staking Discriminator (S007 Item 3)

S007 noted that a test for `DEPOSIT_REWARDS_DISCRIMINATOR` already existed in Tax Program unit tests. This remains present and is not in scope for the cross-crate tests.

## Verdict

**POTENTIAL (LOW severity)** â€” The core vulnerability identified in S007/H011 is RESOLVED. The `byte_length_parity` test provides a stronger guarantee than field-level assertions because it catches any byte-level difference regardless of which field. The `epoch_to_tax_round_trip` test covers all 23 fields explicitly.

The remaining issues are:
1. The `tax_to_epoch_round_trip` test is incomplete at the field-assertion level (8/23 fields), but this is functionally covered by `byte_length_parity`.
2. The AMM PoolState byte offset test (S007 Item 2) was never implemented, leaving that coupling untested by the cross-crate suite. This residual gap from S007 remains.
3. A misleading doc comment in `epoch_state.rs` lists `initialized`/`bump` twice.

None of these represent an active vulnerability. The risk profile is LOW â€” a future AMM PoolState layout change would still cause silent data corruption undetected by any automated test, but EpochState (the primary concern) is now well-protected.

## Recommendations

1. **Complete `tax_to_epoch_round_trip` field assertions** (cosmetic, low priority): Add the 15 missing `assert_eq!` calls to match the coverage of `epoch_to_tax_round_trip`. While `byte_length_parity` provides equivalent safety, explicit field assertions make failures self-documenting.

2. **Add AMM PoolState offset test** (medium priority â€” residual from S007): Add a test to the cross-crate crate verifying that `u64::from_le_bytes(pool_bytes[137..145])` and `[145..153]` correctly recover `reserve_a` and `reserve_b` from a constructed `amm::state::PoolState`. This would close the last open item from S007.

3. **Fix doc comment in `epoch_state.rs`** (trivial): Remove the duplicate `initialized`/`bump` entries at lines 41-42 (above `reserved`), leaving only the correct listing at lines 44-45.
