## Summary

The hypothesis is that a crafted proxy contract or alternative transfer mechanism could bypass the Token-2022 transfer hook, allowing un-whitelisted accounts to receive CRIME or FRAUD tokens without enforcement of the whitelist check. This finding evaluates every SPL/Token-2022 instruction type against the hook invocation matrix to determine the actual bypass surface and its exploitability given the protocol's constraints.

**Verdict:** The bypass surface is accurately understood and intentionally structured. Three instruction types — `burn`, `mint_to`, and `close_account` — do not trigger the transfer hook by Token-2022 design. The protocol correctly exploits the burn exemption for the Carnage Fund (to save CPI depth), and `mint_to` tokens always land in the deployer's ATA (which is already whitelisted). A legacy `transfer` (instruction 3 / `spl_token::transfer`) on a Token-2022 account IS documented to bypass hooks, but the AMM's `transfer_t22_checked` helper explicitly documents and enforces this constraint. The remaining question — whether an external actor can construct a proxy contract to issue a legacy transfer — is answered below.

---

## Background: Token-2022 Hook Invocation Matrix

Token-2022 enforces the transfer hook only on the **`transfer_checked`** (instruction 12) and **`transfer_checked_with_fee`** (instruction 26) code paths. The following instruction types do **not** invoke the hook:

| Instruction | Code | Hook Triggered? | Notes |
|-------------|------|-----------------|-------|
| `transfer` (legacy) | 3 | **NO** | Deprecated; Token-2022 accepts it but does NOT call hook |
| `transfer_checked` | 12 | YES | Standard enforced path |
| `transfer_checked_with_fee` | 26 | YES | Fee extension variant |
| `mint_to` | 7 | **NO** | Tokens minted directly to account; no source party |
| `mint_to_checked` | 14 | **NO** | Same |
| `burn` | 8 | **NO** | Tokens destroyed; no destination party |
| `burn_checked` | 15 | **NO** | Same |
| `close_account` | 9 | **NO** | Sends rent lamports to owner; no token transfer |

This matrix is a Token-2022 runtime property, not a protocol choice. The hook's design intent (SPL Transfer Hook Interface spec) is to intercept **transfers between token accounts** — not creation, destruction, or housekeeping operations.

## Analysis

### 1. Can a proxy contract use legacy `transfer` to bypass the hook?

**Finding: NO — not exploitable in this protocol.**

Legacy `transfer` (SPL instruction discriminator `3`) on a Token-2022 mint does NOT trigger the transfer hook. The AMM `transfer_t22_checked` helper explicitly documents this risk:

```
// Security
// - Always uses `transfer_checked` (never plain `transfer`). Plain `transfer` silently
//   bypasses T22 hooks, which would skip whitelist enforcement.
```
(`programs/amm/src/helpers/transfers.rs:50-51`)

However, an external attacker calling legacy `transfer` directly on a CRIME/FRAUD Token-2022 account faces a separate hard constraint: **Token-2022 requires the signer to be the token account's owner (or a delegate).** Since CRIME and FRAUD accounts are held by whitelisted PDAs (AMM pool, Carnage vaults, Bonding Curve, user wallets), an attacker cannot sign on behalf of those accounts.

The attack model for a "proxy contract" would be:
1. Attacker deploys Program X.
2. Program X CPIs into Token-2022 with legacy `transfer`, using attacker-controlled accounts as source.
3. Attacker's token account holds CRIME or FRAUD acquired through the whitelist (e.g., via the Tax Program's buy path).
4. Program X issues legacy `transfer` to an un-whitelisted destination.

This attack requires the attacker to already possess CRIME/FRAUD tokens. Acquiring them requires going through the Tax Program's buy path, which delivers tokens to the user's token account via `transfer_checked`. Once the user holds tokens in their own account, they ARE the token account owner and CAN issue a legacy `transfer` from their own wallet or via a proxy contract they control.

**Impact if exploited:** The attacker could move tokens to an un-whitelisted address (another wallet, a DEX pool, etc.) without triggering the whitelist check. This would allow CRIME/FRAUD tokens to escape the protocol's whitelist perimeter.

**Mitigating factors:**
- The whitelist model in this protocol is "source OR destination must be whitelisted" (`transfer_hook.rs:94-108`). This is an **open whitelist** model, not a closed circuit: the design intent is to ensure that protocol-controlled addresses (AMM pools, Carnage vaults, staking escrow) are on one side of every transfer. Users holding tokens in their own wallets represent the "unknown destination" side that is legitimately not whitelisted.
- Under this model, a user moving tokens to another user wallet via legacy `transfer` has the same effect as a normal user-to-user transfer where neither party is whitelisted — except the hook is not called. Given the hook only requires source OR destination to be whitelisted, a user who received tokens from a whitelisted source (e.g., bought from the AMM pool) can transfer to any destination without the hook blocking them.
- The whitelist exists primarily to ensure that **swaps route through the Tax Program** (i.e., that value flows are taxed). User-to-user OTC transfers are not the threat model. An attacker bypassing the hook to move tokens to a new wallet gains no protocol advantage — they cannot exit to SOL without going back through the Tax Program (the only SOL ↔ token path), at which point the hook is called again via `transfer_checked`.

**Residual risk:** If the hook model were ever tightened to enforce a "closed circuit" (both parties must be whitelisted), the legacy `transfer` bypass would become significant. Under the current open model, the economic impact is low.

### 2. Does `burn` bypass the hook? Is it exploited in this protocol?

**Finding: YES, burns bypass hooks. The protocol deliberately exploits this.**

The `burn_held_tokens` function in `carnage_execution.rs` explicitly documents:

```
/// Token-2022 burns do NOT trigger transfer hooks, so this doesn't
/// consume CPI depth for the hook call.
```
(`programs/epoch-program/src/helpers/carnage_execution.rs:457-458`)

The Carnage Fund's Burn action sends discriminator `8u8` directly to Token-2022, which is the raw `burn` instruction. No hook accounts are passed, and none are needed. This is a deliberate design choice to conserve CPI depth — the Carnage execution path is already at Solana's hard limit of 4 CPI levels (Epoch → Tax → AMM → T22 → Hook). Adding hook invocation to the burn path would exceed the limit.

**Is this a security problem?** No. Burns move tokens to supply=0 (they are destroyed). There is no "destination" party and no economic bypass. An attacker cannot use a burn to "steal" tokens; they can only permanently destroy their own tokens. The only actor with `carnage_state` PDA signing authority is the Epoch Program itself.

An external attacker could also burn their own CRIME/FRAUD tokens directly (they are the account owner). This is harmless — they lose tokens with no compensating benefit.

### 3. Does `mint_to` bypass the hook? Is it a risk?

**Finding: YES, `mint_to` bypasses hooks. Not a risk in this protocol.**

The CRIME and FRAUD mints have their `mint_authority` assigned to the deployer during `initialize.ts`, then minting is completed and the supply is fixed. If the mint authority is ever present, tokens can be minted directly to any account without hook invocation. Under the current design, the deployer holds the mint authority during the initialization phase only.

**Mitigating factors:**
- All whitelisted accounts (AMM pools, Bonding Curve vault, etc.) receive their token balances via `mint_to` during initialization. This is the intended flow.
- After initialization, if the mint authority is burned or transferred to a multisig, no future `mint_to` is possible without multisig approval.
- The current protocol state (pre-mainnet) retains mint authority for administrative flexibility. Post-graduation or post-Squads migration, this is controlled.

**Residual risk:** If the mint authority is compromised before it is transferred to Squads, an attacker could mint tokens to arbitrary un-whitelisted accounts. This is a privileged operation risk, not a hook bypass risk per se.

### 4. Can a proxy contract route around the hook via any other mechanism?

**Finding: No viable proxy attack path exists given AMM and Tax Program constraints.**

For a proxy contract to meaningfully bypass the whitelist:
- It would need to route tokens from a whitelisted source (e.g., AMM pool) to an un-whitelisted destination.
- The AMM pool accounts are owned by the AMM program. Only the AMM can sign on their behalf.
- The AMM calls `transfer_t22_checked` (enforced via Token-2022 program ID check) for all CRIME/FRAUD transfers.
- A proxy contract cannot CPI into the AMM's `swap_sol_pool` without a valid `swap_authority` PDA signed by the Tax Program (enforced via `seeds::program = TAX_PROGRAM_ID` constraint, tested by `test_cpi_access_control.rs`).

The layered architecture (Tax Program gates AMM; AMM uses `transfer_checked`; hook enforces whitelist) creates a closed-loop for all AMM-sourced transfers. The only escape is legacy `transfer` from user-owned accounts, addressed in point 1 above.

## Verdict

**NOT EXPLOITABLE under the current threat model.** The three non-hook instruction types (burn, mint_to, legacy transfer) each have mitigating factors that prevent protocol damage:

- **Legacy `transfer` bypass**: Requires the attacker to own CRIME/FRAUD tokens and move them to another unwhitelisted wallet. Under the open "source OR destination" whitelist model, this produces no economic advantage — the attacker cannot convert to SOL without re-entering the Tax Program, which calls `transfer_checked`. No funds are stolen from the protocol.
- **Burn bypass**: Intentional design choice to preserve CPI depth. Only the Epoch Program (via its PDA signer) can trigger protocol burns. User self-burns lose tokens with no compensating benefit.
- **Mint_to bypass**: Only relevant during the privileged initialization window. Post-Squads migration, controlled by multisig.

The hook implementation is correct for the intended threat model: preventing un-whitelisted programs from routing token flows outside the Tax Program's tax-collection surface. The `check_is_transferring` flag prevents direct hook invocation attacks.

**Severity: LOW** — The residual risk is that a user could move tokens to a new wallet via legacy `transfer`, bypassing the hook's whitelist check entirely. Under the current open whitelist design, this has no economic impact. If the whitelist model is tightened in a future upgrade to require both parties to be whitelisted (closed circuit), this would become a HIGH finding and would require disabling or deprecating the Token-2022 `transfer` instruction path for these mints.

---

## Recommendations

1. **[LOW] Document the open vs. closed whitelist design decision explicitly.** The hook's "source OR destination" model is an intentional choice that determines the severity of legacy `transfer` bypass. If the intent is ever to enforce that CRIME/FRAUD can only move between protocol-controlled accounts, the whitelist model must switch to "both must be whitelisted" AND the legacy `transfer` instruction path must be accounted for (Token-2022 does not block it at the program level).

2. **[LOW] Consider using `permanent_delegate` extension as defense-in-depth.** Token-2022's `permanent_delegate` extension can grant a program unconditional transfer authority over all accounts for a given mint. A protocol-owned PDA with `permanent_delegate` could intercept or reject transfers that bypass the hook. This is an optional hardening measure, not a required fix.

3. **[INFORMATIONAL] The `burn` hook exemption is correctly documented and justified.** No action needed. The CPI depth constraint is a hard Solana runtime limit; the burn path correctly exploits the exemption to stay within the 4-level CPI limit.

4. **[LOW] Verify post-Squads migration: mint authority must not remain on a hot wallet.** The `mint_to` bypass is only dangerous while the mint authority is held by an individual deployer key. Squads migration (planned for v1.4) resolves this by moving mint authority behind the 2-of-3 timelock.
