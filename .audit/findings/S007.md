# S007: Cross-Program Discriminator Mismatch After Instruction Rename

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** CPI, Upgrade Coupling
**Inspired by:** H011, H029

## Summary

Three hardcoded Anchor discriminator constants (sha256-derived 8-byte prefixes) in Tax Program and Epoch Program must precisely match the current instruction names in their CPI target programs. If any target instruction is renamed during an upgrade — even as a trivial refactor — the discriminator changes silently and all CPI calls to that instruction fail with an opaque error. H029 confirmed the three constants are currently correct. This finding investigates the forward-looking fragility: the absence of any structural coupling between the constant and the instruction it represents means a rename silently breaks core protocol flows with no compile-time warning.

## Analysis

### Confirmed Discriminator Locations

H029 identified and independently verified all three constants:

| Constant | File/Line | Target | Current Value (Verified) |
|---|---|---|---|
| `DEPOSIT_REWARDS_DISCRIMINATOR` | `tax-program/src/constants.rs:156` | `staking::deposit_rewards` | `[52, 249, 112, 72, 206, 161, 196, 1]` |
| `UPDATE_CUMULATIVE_DISCRIMINATOR` | `epoch-program/src/constants.rs:117` | `staking::update_cumulative` | `[0x93, 0x84, 0xdb, 0x65, 0xa5, 0x17, 0x3d, 0x71]` |
| `SWAP_EXEMPT_DISCRIMINATOR` | `epoch-program/src/constants.rs:177` | `tax-program::swap_exempt` | `[0xf4, 0x5f, 0x5a, 0x24, 0x99, 0xa0, 0x37, 0x0c]` |

All three are confirmed correct at the time of this audit.

### Why a Rename Breaks Silently

Anchor discriminators are computed as `sha256("global:<instruction_name>")[0..8]`. The name is a Rust identifier embedded in the `#[instruction]` macro expansion. When a program is upgraded with a renamed instruction:

1. The target program's on-chain binary rejects any instruction whose first 8 bytes do not match the new discriminator.
2. The calling program's constant still holds the old bytes.
3. The CPI call reaches the target, the discriminator check fails, and Anchor returns `InvalidInstructionData` (error 100).
4. This bubbles up as an opaque CPI error — `InstructionFallbackNotFound` or a raw program error code — with no message indicating the discriminator mismatch.

There is **no compile-time, deploy-time, or test-time mechanism** that automatically catches this unless the discriminator consistency test (H029 R2, not yet implemented) is run. The existing unit tests in each program recompute the hash from the instruction name string — but these tests live in the *caller* programs. They verify that the constant matches the hash of a hardcoded string, not that the string matches the actual current instruction name in the target binary.

### The Stale Test as Proof of Concept

H029 already documented a concrete instance of this drift pattern:

`staking/src/constants.rs:133-140`:
```rust
fn test_tax_program_id() {
    let id = tax_program_id();
    assert_eq!(id.to_string(), "43fZGRtmEsP7ExnJE1dbTbNjaP1ncvVmMPusSeksWGEj");
    //  ^^^ mock-tax-program ID, not real tax-program. Will fail on run.
}
```

This test was left stale when `tax_program_id()` was updated during a redeploy. The discriminator tests are identical in character: if `deposit_rewards` were renamed to `deposit_tax_rewards` and the test string updated in the staking constants but not in the tax-program constants test, the mismatch would go undetected until runtime CPI failure.

### Upgrade Scenarios That Trigger the Break

Three plausible rename scenarios exist based on the project roadmap:

**Scenario A — Staking program refactor (affects DEPOSIT_REWARDS + UPDATE_CUMULATIVE):**
The project memory notes USDC pool pairs as a planned future feature. A Staking Program upgrade supporting dual-pool rewards might rename `deposit_rewards` to `deposit_sol_rewards` for clarity. Both the Tax Program's `DEPOSIT_REWARDS_DISCRIMINATOR` and the unit test string would need simultaneous updates — which is a manual, undocumented process.

**Scenario B — Tax program refactor (affects SWAP_EXEMPT_DISCRIMINATOR):**
`swap_exempt` is described as the Carnage execution path. A rename to `carnage_swap` or `swap_exempt_carnage` during a security refactor would silently break all Epoch Program Carnage swaps. The Epoch Program's `SWAP_EXEMPT_DISCRIMINATOR` would still hold the old bytes.

**Scenario C — Squads multisig upgrade path:**
Post Phase 97, program upgrades require a 2-of-3 Squads approval with a 300s timelock. A developer who approves a refactor upgrade in the staking program may not be aware that Tax Program and Epoch Program hold discriminator constants pointing at the old instruction name. The upgrade succeeds, the callers remain on old binaries, and all revenue distribution or Carnage execution silently fails until the other programs are also upgraded.

### Protocol Impact by Constant

**`DEPOSIT_REWARDS_DISCRIMINATOR` breaks:** Every successful swap that generates tax revenue calls `deposit_rewards` CPI. A mismatch means 71% of all swap taxes are silently not deposited to the staking escrow. The staking program accepts no deposits, reward-per-token stored stagnates, and all PROFIT stakers earn zero yield. This is HIGH severity if triggered.

**`UPDATE_CUMULATIVE_DISCRIMINATOR` breaks:** Every `consume_randomness` call attempts `update_cumulative` CPI. A mismatch causes `consume_randomness` to fail entirely (CPI error propagates up), which halts epoch progression. No new epochs, no new tax rates, no Carnage. Protocol liveness failure. This is HIGH severity if triggered.

**`SWAP_EXEMPT_DISCRIMINATOR` breaks:** Every Carnage execution attempt (atomic and fallback) calls `swap_exempt` via CPI. A mismatch causes all Carnage swaps to fail silently. The Carnage Fund accumulates SOL indefinitely without deployment. HIGH severity if triggered (same impact category as H009, but from a different root cause).

### Compounding with H011 (Layout Drift Pattern)

H011 documented the same structural pattern for PoolState byte offsets: two independent programs hardcode layout assumptions about a third program's state, with no compile-time enforcement. This finding (S007) is the discriminator analogue: two independent programs hardcode instruction identity assumptions about other programs, with no compile-time enforcement. Both represent the same architectural debt: the absence of a versioned interface contract between caller and callee programs.

### Tests Exist But Are Insufficient

Two of three discriminators have unit tests that compute `sha256("global:<name>")[0..8]` and assert equality with the constant. These tests are:
- Correct for the current codebase.
- Unable to detect a rename in the target program, because they test the constant against a hardcoded string — not against the target program's actual IDL or source.
- Only run within the caller program's test suite, not as cross-program integration tests.

The missing test category is a cross-program consistency test that ingests the target program's IDL and verifies the caller's constant matches.

## Verdict

**CONFIRMED — MEDIUM severity.**

The discriminators are correct today. The vulnerability is entirely forward-looking: any instruction rename in a target program, without simultaneous update of all caller constants, silently breaks core protocol flows. The severity at time-of-trigger would be HIGH for all three constants. The MEDIUM classification reflects:

- No active mismatch exists at audit time.
- Unit tests exist and would catch an inconsistency *if run against updated strings*.
- The risk materialises only on an upgrade event — which now requires Squads 2-of-3 approval, adding a governance checkpoint (though the checkpoint does not inspect discriminator constants).

The stale `test_tax_program_id` test in staking/src/constants.rs is live proof that this exact drift pattern occurs naturally in this codebase during redeploys.

## Recommendations

### R1 — Add Cross-Program Discriminator Consistency Test (High Priority)

Create `tests/discriminator-consistency/src/lib.rs` (or extend `tests/cross-crate/`) with a test that:

1. Reads the target program's IDL JSON (`programs/staking/target/idl/staking.json`, etc.)
2. Extracts instruction names from the IDL
3. Computes `sha256("global:<name>")[0..8]` for each relevant instruction
4. Asserts equality with the caller program's constant

This test fails at build time if any instruction is renamed without updating the constant. It is the minimal-cost structural fix.

### R2 — Add Upgrade Gate to Deployment Checklist (Medium Priority)

Extend `Docs/mainnet-deploy-checklist.md`:

> **Before upgrading staking-program:** Verify `DEPOSIT_REWARDS_DISCRIMINATOR` in `tax-program/src/constants.rs:156` and `UPDATE_CUMULATIVE_DISCRIMINATOR` in `epoch-program/src/constants.rs:117` match the new instruction names. Run cross-program discriminator consistency tests.
>
> **Before upgrading tax-program:** Verify `SWAP_EXEMPT_DISCRIMINATOR` in `epoch-program/src/constants.rs:177` matches the new `swap_exempt` instruction name.

### R3 — Fix the Stale `test_tax_program_id` Test (Immediate)

As documented in H029 R1: update `staking/src/constants.rs:138` to use the correct tax program ID. A permanently-failing test provides no safety net.

### R4 — Consider Anchor CPI Crates for Non-Hook Paths (Low Priority / Future)

The three discriminator-sensitive CPIs (`deposit_rewards`, `update_cumulative`, `swap_exempt`) do not themselves involve Token-2022 hook forwarding (that is the reason for manual CPI in the first place — the hook accounts). These three could potentially be replaced with generated Anchor CPI crates, which compute the discriminator at compile time from the target program's IDL. This eliminates the runtime mismatch risk entirely.
