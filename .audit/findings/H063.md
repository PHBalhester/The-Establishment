# H063: force_carnage Devnet Gate Verification

**Status:** CONFIRMED
**Severity:** INFORMATIONAL
**Category:** State Machine — Feature-Gated Instructions

## Summary

The `force_carnage` instruction is correctly gated behind the `#[cfg(feature = "devnet")]` feature flag at both the module level and the instruction level. This ensures the instruction is completely absent from mainnet binaries. The force_carnage module is not exported or registered in mainnet builds.

## Code Evidence

**File:** `programs/epoch-program/src/instructions/mod.rs:7-8`

```rust
#[cfg(feature = "devnet")]
pub mod force_carnage;
```

**File:** `programs/epoch-program/src/instructions/mod.rs:18-19`

```rust
#[cfg(feature = "devnet")]
pub use force_carnage::*;
```

**File:** `programs/epoch-program/src/instructions/force_carnage.rs:1-8`

```rust
//! force_carnage instruction (DEVNET ONLY).
//!
//! Admin-gated test helper that sets carnage_pending on EpochState
//! without waiting for a natural VRF trigger. Allows rapid testing
//! of all Carnage execution paths (Burn, Sell, BuyOnly).
//!
//! MUST BE REMOVED BEFORE MAINNET DEPLOYMENT.
//! Gated by #[cfg(feature = "devnet")] at module and instruction level.
```

The instruction handler itself (lines 50-84) is not wrapped in a second `#[cfg]`, but this is safe because the entire module is already conditionally compiled.

**IDL Verification:** When `anchor build` is run without `--features devnet`, the resulting IDL will not contain a `force_carnage` instruction entry. This prevents clients from accidentally discovering the instruction on mainnet.

## Verdict

**CONFIRMED SAFE.** The feature gate is correctly applied and cannot leak into mainnet builds. No remediation needed.
