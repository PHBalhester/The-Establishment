# S008: VRF Reveal + Delayed Consume Enables Carnage Target Prediction

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** MEV, Oracle, Timing
**Inspired by:** H019

## Summary

The VRF reveal posted by the Switchboard oracle exposes all 8 bytes of protocol-relevant randomness simultaneously and publicly. This includes not only tax rates (bytes 0-4, the subject of H019) but also the Carnage trigger decision (byte 5), Carnage action (byte 6), and Carnage buy target (byte 7). An attacker who reads the oracle's reveal transaction can determine — before `consume_randomness` lands on-chain — whether Carnage will trigger, which action it will take (BuyOnly, Burn, or Sell), and which token it will buy. This information can be used to pre-position ahead of the Carnage buy for risk-free profit, or to execute the suppression attack described in H009 only for Carnage-triggering epochs.

## Analysis

### VRF Byte Allocation (Confirmed from Source)

The byte layout from `carnage.rs` and `tax_derivation.rs`:

```
Byte 0:  Flip decision (< 192 = flip cheap side, 75%)
Byte 1:  CRIME low tax magnitude (% 4 → 100/200/300/400 bps)
Byte 2:  CRIME high tax magnitude (% 4 → 1100/1200/1300/1400 bps)
Byte 3:  FRAUD low tax magnitude (% 4 → 100/200/300/400 bps)
Byte 4:  FRAUD high tax magnitude (% 4 → 1100/1200/1300/1400 bps)
Byte 5:  Carnage trigger (< 11 = trigger, ~4.3%)
Byte 6:  Carnage action (< 5 = Sell/2%, >= 5 = Burn/98%)
Byte 7:  Carnage buy target (< 128 = CRIME, >= 128 = FRAUD, 50/50)
```

All 8 bytes are available in the Switchboard randomness account immediately after the oracle posts the `revealIx`. The account is a standard Solana public account; any node can call `RandomnessAccountData::get_value(slot)` to read the full 32-byte array.

### The Information Exposed by the Reveal

After the oracle reveals but before `consume_randomness` executes, an observer can compute locally:

```rust
// computable off-chain from public oracle data:
let is_triggered = vrf_result[5] < CARNAGE_TRIGGER_THRESHOLD; // < 11
let action = if has_holdings {
    if vrf_result[6] < CARNAGE_SELL_THRESHOLD { Sell } else { Burn }
} else { None /* BuyOnly */ };
let target = if vrf_result[7] < 128 { CRIME } else { FRAUD };
```

This computation is three comparisons and zero cryptography. Any actor monitoring the Switchboard oracle account can perform it in <1ms after the reveal.

### Three Distinct Exploitation Paths

**Path A: Pre-position in Carnage buy target before atomic execution**

If the atomic path (`execute_carnage_atomic`) succeeds, the VRF reveal-to-execution gap is approximately 0-3 slots. The atomic path bundles `revealIx` + Switchboard instruction + `consume_randomness` + `execute_carnage_atomic` in one or two transactions. This window is very narrow — but Jito bundles and fast RPC make same-slot reactions possible in practice.

More importantly: the attacker need not beat the atomic path. They only need to pre-position *during the same epoch* before the atomic TX lands. If the attacker detects byte 5 < 11 at the oracle reveal (slots N+1 to N+2 after `trigger_epoch_transition`), and the atomic TX is submitted at slot N+2, the attacker has approximately 1-2 slots to front-run. This is narrow but non-zero.

**Path B: Pre-position when atomic path fails (fallback window)**

If the atomic path fails (CPI depth issue, compute budget, or any other failure), the fallback window opens at `carnage_lock_slot` (slot N + 50). The attacker has 50 slots — approximately 20 seconds — to build and submit a large position in the target token before the fallback window opens. This is a comfortable timeframe for any automated system.

This path is structurally the same as H010 (fallback MEV sandwich) but with a more accurate entry point: instead of acting after seeing `carnage_pending = true` in EpochState, the attacker acts 50 slots earlier with full knowledge of which token to buy. The earlier entry means:
- Better average fill price (before others have reacted)
- More time to accumulate a large position
- No competition from other Carnage-watchers who wait for `carnage_pending` to become visible

**Path C: Optimal suppression selectivity (compound with H009)**

Without byte-prediction, an attacker using H009 must either suppress every epoch uniformly (always omit `carnage_state`) or suppress randomly. With byte-prediction, the attacker can:

- For epochs where byte 5 >= 11 (Carnage would NOT trigger): include `carnage_state` freely — no effect on Carnage either way
- For epochs where byte 5 < 11 (Carnage WOULD trigger): selectively include or omit `carnage_state` based on whether the attacker prefers to extract now (Path B) or accumulate further

This converts H009's "suppress all" into "suppress strategically" — the attacker can control exactly when Carnage executes and do so with full advance knowledge of the target.

### Timing Window for Each Path

| Path | Window Duration | Precondition | Practical Difficulty |
|---|---|---|---|
| A: Beat atomic | ~1-2 slots | Oracle reveal before atomic TX | HIGH — very narrow |
| B: Lead fallback | ~50 slots | Atomic path must fail | MEDIUM — needs atomic failure |
| C: Optimal suppression | 2-5 slots (normal) | H009 prerequisite | LOW — trivial if H009 present |

Path A is the hardest and least reliable. Path B is practically achievable with moderate capital. Path C is the most strategically powerful — it requires no capital at the target prediction step, only at the pre-position step.

### What the Attacker Gains From Target Knowledge vs. H019

H019 describes front-running the tax rate change window (bytes 0-4). The gains there depend on pre-positioned token balances and favorable rate flips.

S008 describes a different attack: using bytes 5-7 to front-run a **buy event** of known direction and scale. The comparison:

| Aspect | H019 (Tax Rate) | S008 (Carnage Target) |
|---|---|---|
| Information gained | New tax rates (bytes 0-4) | Whether Carnage triggers + which token (bytes 5-7) |
| Required position | Pre-existing token balance | None required (can buy fresh in window) |
| Flash-loan amplifiable? | No (position held across epoch) | Path B: Yes (hold for 50+ slots, sell after Carnage) |
| Per-event alpha | Proportional to held balance | Proportional to Carnage swap size |
| Frequency | Every epoch transition (with flip) | ~4.3% of epochs |
| Window | ~2-5 slots (extendable to 300) | 50 slots (Path B) |

S008 Path B is arguably higher value per event than H019 because:
- The direction is known (long the target)
- The magnitude is predictable (Carnage deploys available vault balance up to 1000 SOL)
- No pre-existing position is needed — the attacker buys fresh on the reveal signal

### Consume Randomness Logs Make This Worse

`consume_randomness.rs:181-190` logs all 8 VRF bytes:

```rust
msg!(
    "VRF bytes received: [{}, {}, {}, {}, {}, {}, {}, {}]",
    vrf_result[0], vrf_result[1], vrf_result[2], vrf_result[3],
    vrf_result[4], vrf_result[5], vrf_result[6], vrf_result[7]
);
```

These logs appear in the transaction logs of `consume_randomness` — which is itself confirmed on-chain before any Carnage execution can proceed. An attacker who monitors `consume_randomness` transactions (rather than the oracle reveal directly) gets the same bytes from a conveniently-formatted log message, with only 1 slot of additional delay vs. direct oracle monitoring.

### Irreducibility of This Risk

The fundamental issue is that Switchboard VRF is publicly verifiable — by design. The oracle posts the reveal on-chain so that anyone can verify the randomness is genuine. This public verifiability is what makes the VRF secure against the oracle choosing favorable outcomes. But it is exactly the same property that makes the outcome readable by any observer before the protocol acts on it.

The gap between "reveal visible on-chain" and "protocol acts on the reveal" is inherent to the two-transaction structure (trigger → consume+execute). It cannot be eliminated without combining the reveal and execution into a single atomic operation.

## Verdict

**CONFIRMED — MEDIUM severity.**

The information asymmetry from VRF reveal to `consume_randomness` execution is confirmed and exploitable. The severities are:

- Path A (beat atomic): LOW — window is too narrow for reliable execution
- Path B (lead fallback): MEDIUM — 50-slot window with full target knowledge is reliably exploitable with a monitoring bot, applicable to the ~5% of Carnage triggers that proceed through fallback
- Path C (selective H009): MEDIUM — requires H009 to be simultaneously exploitable; significantly amplifies H009 value

The finding is rated MEDIUM (not HIGH) because:
- Path B requires the atomic path to fail, which is not the common case
- The information is publicly verifiable by design (VRF property)
- The pool's 50% minimum output floor on user swaps limits front-run position size per TX
- No mechanism to fully eliminate this risk without architectural changes to the two-TX flow

This represents the same class of information risk as H019 (which is also rated MEDIUM), but applied to a higher-magnitude event (Carnage buy) at lower frequency.

## Recommendations

### R1 — Remove VRF Byte Logging in Production (Immediate, Low Cost)

Remove the `msg!()` call at `consume_randomness.rs:181-190` that logs all 8 VRF bytes. This does not close the oracle-reveal window (the bytes are still readable from the Switchboard account) but it eliminates the convenient log-format attack surface. Replace with a summary that omits Carnage-relevant bytes:

```rust
// BEFORE (current):
msg!("VRF bytes received: [{}, {}, {}, {}, {}, {}, {}, {}]", ...all 8...);

// AFTER (recommended):
msg!("VRF received: cheap_side_flip={}, carnage={}", vrf_result[0] < 192, vrf_result[5] < 11);
```

This is an incremental hardening, not a complete fix.

### R2 — Minimize Atomic-to-Fallback Gap (Operational)

Per H010 R1: have the protocol crank submit the fallback TX immediately after `carnage_lock_slot` passes (slot 50), collapsing the window for external actors. If the atomic path fails, the protocol's own crank should win the fallback execution race 99% of the time if it monitors continuously and submits via Jito bundle.

### R3 — Store a Reserve Snapshot at consume_randomness (Slippage Defense)

When `consume_randomness` sets `carnage_pending = true`, also store the current pool reserves (from `read_pool_reserves`) in `EpochState` as `carnage_snapshot_reserve_sol` and `carnage_snapshot_reserve_token`. The fallback path (`execute_carnage`) should use this snapshot for slippage calculation rather than live reserves.

This closes Path B: an attacker who front-runs AFTER `consume_randomness` cannot shift the slippage baseline, because the baseline was committed at VRF consumption time (before the attacker knew which pool to manipulate... or in the case of Path C, still before the fallback opens).

This is the same recommendation as H010 R3 (TWAP-based slippage), restated as a snapshot-at-trigger approach.

### R4 — Assess Whether Carnage Target Should Be Encrypted Until Execution (Design Decision)

The fundamental information asymmetry cannot be eliminated while the VRF reveal and Carnage execution are in separate transactions. One architectural option is to commit the Carnage target as a hash during `consume_randomness` and only reveal it at execution time — but this reintroduces the oracle-chooses-outcome risk that VRF is designed to prevent. This is likely not worth pursuing for the marginal gain.

A simpler design observation: the 50/50 token split (byte 7 < 128) means an attacker who is uncertain about the target can hedge by buying both CRIME and FRAUD in equal measure before the fallback window. This makes the complete information irrelevant for the hedge case, but the hedge itself creates broad upward pressure on both tokens — which is arguably the Carnage mechanism's intended effect, just captured by the arbitrageur instead of the protocol.
