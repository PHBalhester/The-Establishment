# H047: Unchecked as u64 Cast in get_current_price

**Status:** CONFIRMED
**Severity:** LOW
**Category:** Arithmetic, Display/Events

## Summary

The `get_current_price()` function in `bonding_curve/src/math.rs:222-225` uses `.unwrap_or(0)` on a division and `.unwrap_or(u64::MAX)` on a u128-to-u64 cast, silently masking errors rather than returning a Result. If intermediate calculations overflow, the price is returned as 0 or u64::MAX. However, the function is used **only for events and display purposes** (line 334 in `sell.rs`, line 10 in `purchase.rs`), not for financial calculations, limiting the blast radius to informational inaccuracy rather than protocol vulnerability.

## Code Evidence

**File:** `programs/bonding_curve/src/math.rs:217-226`
```rust
pub fn get_current_price(tokens_sold: u64) -> u64 {
    let price_range = P_END - P_START; // 1275
    // Scale by PRECISION to avoid losing precision in the division
    let progress = (tokens_sold as u128)
        .checked_mul(PRECISION)
        .unwrap_or(0)           // <-- H047: Silent zero on overflow
        / TOTAL_FOR_SALE;
    let price = P_START + price_range * progress / PRECISION;
    u64::try_from(price).unwrap_or(u64::MAX)  // <-- H047: Saturation on cast overflow
}
```

**Usage 1:** `bonding_curve/src/instructions/sell.rs:334`
```rust
emit!(SellEvent {
    tokens_sold: ctx.accounts.curve_state.tokens_sold,
    current_price: get_current_price(ctx.accounts.curve_state.tokens_sold),
});
```

**Usage 2:** `bonding_curve/src/instructions/purchase.rs:10` (event emission)

## Vulnerability Analysis

### Potential Overflow Scenario

Under what conditions can `tokens_sold as u128 * PRECISION` overflow?

- `tokens_sold`: max = TOTAL_FOR_SALE = 460e12 (u64)
- `PRECISION`: 1e12
- Product: 460e12 * 1e12 = 460e24 = 4.6e25

Since u128::MAX ≈ 3.4e38, the product is **safely within u128**. The `.unwrap_or(0)` will not trigger under normal operation.

### Potential Cast Overflow Scenario

Under what conditions can the `price` u128 value overflow u64?

- `price = P_START + price_range * progress / PRECISION`
- P_START = 450, P_END = 1725, price_range = 1275, PRECISION = 1e12
- progress ranges [0, 1e12] (as progress = tokens_sold * 1e12 / TOTAL_FOR_SALE)
- price_range * progress = 1275 * [0..1e12] = [0..1.275e15]
- price = 450 + [0..1.275e15] = [450..1.275e15]

The max price is 1.275e15, which **exceeds u64::MAX (≈1.8e19)**... wait, 1.275e15 << 1.8e19, so this **fits safely in u64**. The `.unwrap_or(u64::MAX)` will not trigger under normal operation either.

### Conclusion on Overflow Risk

**Both overflow paths are prevented by domain constraints:**
- The multiplication overflow is prevented because 460e12 * 1e12 < u128::MAX
- The cast overflow is prevented because max price = 1275 < u64::MAX

**However,** the function is still **an anti-pattern** because it silently masks errors instead of returning `Result<u64, ArithmeticError>`. If either assumption were violated (e.g., TOTAL_FOR_SALE changed to 1e20, or PRECISION adjusted), the error would be masked silently.

## Financial Impact Assessment

Since `get_current_price` is **used only for event emission and display** (not for calculating token amounts or refunds), an incorrect price value would:
1. Show wrong price in transaction events/logs
2. Mislead off-chain indexers or UIs displaying bonding curve pricing
3. **NOT** affect on-chain financial calculations or token transfers

The actual pricing logic for buys/sells uses `calculate_tokens_out()` and `calculate_sol_for_tokens()`, which use proper error handling (`ok_or(CurveError::Overflow)?`).

## Defense-in-Depth Context

The `claim_refund.rs` handler **also** uses the same proportional math but **with proper guarding** at line 163:
```rust
let refund_amount = u64::try_from(refund_amount_u128)
    .map_err(|_| error!(CurveError::Overflow))?;
```

This shows the codebase is capable of proper error handling; `get_current_price` is simply not following the established pattern.

## Verdict

**CONFIRMED but LOW severity** — The error masking is present and is a code smell (anti-pattern), but domain constraints prevent actual overflow under normal operation. The restricted use case (events only, not financial) means even if the function returned 0 or u64::MAX, it would not cause protocol-level token loss or incorrect state changes. The finding is valid as a consistency/robustness issue, not an exploitable vulnerability.

## Recommendation

Replace unwrap_or fallbacks with proper error return:

```rust
pub fn get_current_price(tokens_sold: u64) -> Result<u64, ArithmeticError> {
    let price_range = P_END - P_START;
    let progress = (tokens_sold as u128)
        .checked_mul(PRECISION)
        .ok_or(ArithmeticError::Overflow)?
        / TOTAL_FOR_SALE;
    let price = P_START + (price_range as u128) * progress / PRECISION;
    u64::try_from(price)
        .map_err(|_| ArithmeticError::CastOverflow)
}
```

Then update event emission sites to handle the Result (which in practice will never error, but documents intent correctly).
