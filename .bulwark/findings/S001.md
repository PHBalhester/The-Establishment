## Combined Attack Path

A single act — obtaining a copy of the repository — simultaneously delivers two independent high-value weapons. This is not a sequential dependency (where step 1 unlocks step 2); it is a parallel unlock where one breach surface yields both assets.

**Step 1 — Repository access.**

Attacker obtains a clone of `https://github.com/MetalLegBob/dr-fraudsworth.git`. This requires one of:
- Leaked collaborator credentials
- Accidental public exposure of the (currently private) repository
- Future public release for open-source launch (explicitly mentioned in H001 as a forward risk)
- Any CI/CD system that has ever cloned the repo

**Step 2A — Devnet wallet drain (from H001).**

```bash
git show 53ca01b:.mcp.json | grep SOLANA_PRIVATE_KEY
# → 2zJgKnGrwgkcDLks6uJYcMWThWYHUCp1RuShDGajeutHvp7uWALoAwtSJEnXb2sx2GeizxrApX4QZcXoRwUC2FSS
```

Import into CLI, drain `8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4` (~59.5 SOL), transfer all SPL tokens. Because this wallet is also the devnet upgrade authority, attacker also has program upgrade capability over all deployed devnet programs.

**Step 2B — Webhook management access (from H004), executing in parallel.**

```bash
grep HELIUS_API_KEY .env.devnet
# → [REDACTED-DEVNET-HELIUS-KEY]
```

With this key, enumerate all registered webhooks:
```
GET https://api.helius.xyz/v0/webhooks?api-key=[REDACTED-DEVNET-HELIUS-KEY]
```

Then choose from:
- **Surveillance:** Register a shadow webhook mirroring all protocol events to attacker infrastructure
- **Disruption:** Delete the production webhook, severing the data pipeline
- **Escalation to S003:** Webhook URL discovered via this enumeration becomes the target for payload injection

**The key combination effect:** Both attacks execute within minutes of repository acquisition, independently, with no further prerequisites. The attacker does not choose between them — they execute both.

## Combined Impact

| Dimension | H001 Alone | H004 Alone | Combined |
|-----------|-----------|-----------|---------|
| Financial loss | ~59.5 SOL drained | None direct | Same as H001 |
| Program integrity | Upgrade authority compromised | Not affected | Upgrade authority compromised |
| Data pipeline | Not affected | Disrupted or surveilled | Disrupted AND upgrade authority compromised simultaneously |
| Attacker capability | Wallet + programs | Webhook CRUD | Wallet + programs + webhook CRUD |
| Detection likelihood | Low (key import is silent) | Low (Helius shows no alerts) | Very low — two attack surfaces, zero correlation |
| Recovery complexity | High (rotate key + redeploy) | Medium (rotate Helius key) | Very high — must remediate both simultaneously |

**The combined risk exceeds the sum of parts in one specific way:** an attacker holding both the devnet wallet private key AND the Helius API key can conduct a coordinated attack that is operationally difficult to interrupt. While the owner is rotating the Helius key (Step 2B remediation), the attacker has already extracted funds (Step 2A) and vice versa. A single point of breach — the repository — triggers a two-front incident response.

**Forward risk — mainnet transition:** H001 explicitly flags that the devnet wallet key may be reused near mainnet. H004 flags that the single Helius account/key may span both clusters. If either assumption holds during mainnet launch, the combined blast radius escalates to CRITICAL: mainnet fund theft AND mainnet webhook compromise via one repository exposure event.

**Upgrade authority + disruption compound:** With devnet upgrade authority, the attacker can deploy a modified program that accepts the attacker's own webhook-injected data as authoritative (or that routes funds differently). The webhook disruption then covers the evidence trail while the modified program operates. This compound scenario — malicious upgrade + data pipeline manipulation — is only possible because both keys come from the same source.

## Recommendation

The combination finding does not require different remediations from H001 and H004 individually, but it materially changes the **urgency sequencing**:

1. **Treat both as a single incident.** Rotate the Solana private key (H001) and the Helius API key (H004) in the same session, not sequentially over days. Staggered rotation leaves a window where one weapon is still active.

2. **Audit for coordination before rotating.** Before rotating either key, run `GET /v0/webhooks` to snapshot existing webhooks (check for attacker-created shadows), and check Solana account activity on `8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4` for unauthorized transactions.

3. **Purge git history as a single operation (not per-secret).** BFG Repo-Cleaner should remove all committed secrets in one pass: the `.mcp.json` private key, `.env.devnet` Helius key, and `shared/programs.ts` hardcoded URL. Running partial purges leaves residual cross-references.

4. **Establish single-secret-per-source discipline.** The root cause of S001 is that a single file (the repository) became the authoritative store for independent high-value secrets. Going forward: private keys in hardware or HSM-backed stores; API keys in environment-variable-only stores with no git representation, even in devnet files.

5. **Confirm mainnet key separation before launch.** Verify that mainnet Helius account is on a different API key from the devnet key in git, and that the mainnet deployer/crank wallets share no keys with the devnet wallet.
