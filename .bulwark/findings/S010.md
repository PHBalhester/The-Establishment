## Combined Attack Path

H001 and H012 are independently severe. Their combination closes all three possible defence gaps simultaneously: the wallet can drain funds, the wallet is also the upgrade authority for all programs, and the Squads keys neutralise the planned governance remediation before it is even executed. A single repository clone is sufficient to achieve complete devnet protocol control.

### Prerequisites

Repository read access only. No special permissions, no on-chain transactions prior to exploitation. The attack requires only a `git clone`.

### Step 0 — Repository clone (30 seconds)

```bash
git clone https://github.com/MetalLegBob/dr-fraudsworth.git
cd dr-fraudsworth
```

The repository is listed as private (HTTP 404 unauthenticated). However:
- Any current or past collaborator already has a clone.
- A collaborator token, CI/CD service account, or any leaked credential grants access.
- If the repository is ever made public (open-source launch, accidental visibility change), the entire attack chain executes from that moment.
- GitHub's 90-day cached view window means even a briefly-public repo exposes all content.

### Step 1 — Fund extraction (< 2 minutes)

1. Extract the devnet wallet private key from `.mcp.json` line 8:
   ```
   SOLANA_PRIVATE_KEY: 2zJgKnGrwgkcDLks6uJYcMWThWYHUCp1RuShDGajeutHvp7uWALoAwtSJEnXb2sx2GeizxrApX4QZcXoRwUC2FSS
   ```
   Alternatively extract from `keypairs/devnet-wallet.json` (64-byte JSON array — same key).

2. Import into any Solana wallet. Transfer all SOL and SPL tokens from `8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4` to attacker address:
   ```bash
   solana transfer <attacker-address> ALL --keypair .mcp.json --url devnet
   ```
   Result: ~59.5 devnet SOL drained.

3. Transfer any SPL token accounts (CRIME, FRAUD, PROFIT, WSOL) held by the wallet using the same key.

**Effect of Step 1 alone:** Wallet drained. But the attacker has not yet touched the protocol programs.

### Step 2 — Program upgrade — all 6 programs (< 10 minutes)

H012 confirms that upgrade authority for all 6 devnet programs equals the devnet wallet (`8kPzh...`). The devnet wallet key is now in attacker hands from Step 1.

4. Compile or prepare a malicious program binary. This can be:
   - A modified version of the legitimate program (e.g., AMM that drains swap input tokens to attacker address).
   - A minimal drainer stub that accepts any instruction and transfers lamports/tokens from any account that signs.
   - The original binary with a single modified constant (e.g., fee recipient changed to attacker address).

5. Deploy the malicious binary to each program ID in sequence:
   ```bash
   solana program deploy malicious-amm.so \
     --program-id keypairs/amm-keypair.json \
     --keypair keypairs/devnet-wallet.json \
     --url devnet
   ```
   Repeat for: `bonding-curve-keypair.json`, `epoch-program.json`, `staking-keypair.json`, `tax-program-keypair.json`, `transfer-hook-keypair.json`.

   The program address keypairs (H012, finding §8 note) are not required for upgrade — the upgrade authority key (`devnet-wallet.json`) is sufficient. The program address keypairs are available in git but are not the critical secret here.

6. All 6 devnet programs are now replaced. Every frontend interaction (swap, stake, epoch, bonding curve, tax, transfer hook) executes attacker-controlled logic.

**Effect of Step 2:** Complete devnet protocol replacement. Any user interacting with the devnet frontend is now interacting with attacker programs at the same addresses.

### Step 3 — Squads multisig pre-positioning (future-proofed control)

H012 documents that `squadsVault: null` — the Squads authority transfer has not yet occurred. Steps 1 and 2 already give the attacker full control. But the attacker who reads the project documentation knows that `transfer-authority.ts` is the planned remediation step. This step future-proofs control against that remediation.

7. Extract all four Squads keypairs from git:
   - `keypairs/squads-create-key.json` — Squads multisig creation key
   - `keypairs/squads-signer-1.json`, `squads-signer-2.json`, `squads-signer-3.json` — all three 2-of-3 signers

8. **If the operator runs `transfer-authority.ts` before discovering the compromise**, upgrade authority transfers to the Squads vault PDA. The attacker, holding any two of the three committed signer keys, immediately satisfies the 2-of-3 threshold.

9. Attacker proposes and approves (using signer-1 and signer-2, both in hand) a Squads transaction to:
   - Transfer upgrade authority from the Squads vault to an attacker-controlled address, OR
   - Execute an immediate program upgrade through Squads directly.

10. After the Squads timelock (300 seconds) elapses, the transaction executes. Programs are upgraded again.

**Effect of Step 3:** Even the intended governance remediation is compromised. Running `transfer-authority.ts` does not improve the security posture; it only changes which compromised key the attacker needs to use.

```

  │    ├─ drain 59.5 SOL + all SPL tokens
  │    └─ has upgrade authority for all 6 programs
```

The entire chain executes from a single `git clone` with no on-chain cost beyond devnet transaction fees (which the attacker just funded from the drained wallet).

### Secondary Attack Vector: AI Tool Exfiltration

H001 notes that `.mcp.json` is read by Claude Code and similar AI assistants at session start. Any AI tool that:
- Logs file content to telemetry,
- Sends project context to a third-party LLM API,
- Or stores conversation history on external servers

has already received this private key through a non-obvious channel that traditional secret scanners would miss. The attacker in this case is not a repository-level adversary but any external party with access to AI tool telemetry or conversation storage.

## Combined Impact

**Severity: CRITICAL (devnet scope)**

| Dimension | Assessment |
|-----------|-----------|
| Fund loss | ~59.5 devnet SOL + all protocol SPL tokens drained immediately |
| Protocol integrity | All 6 programs replaceable with malicious bytecode; every user interaction compromised |
| Governance | Squads 2-of-3 multisig fully controlled by attacker with 2 committed signer keys; planned authority transfer worsens posture |
| Persistence | Attacker control survives key rotation if Squads keys are not rotated simultaneously |
| Detection | Zero on-chain indicators until users report incorrect protocol behaviour |
| Mainnet scope | Currently NOT impacted — mainnet keypairs correctly gitignored, separate deployer wallet. Risk of mainnet regression: LOW but non-zero if procedural controls fail. |
| Devnet assurance | All devnet testing provides false security assurance post-compromise |

The combination is specifically more severe than H001 + H012 individually because:

1. H001 alone (wallet drain) does not upgrade programs — an attacker who only reads `.mcp.json` could drain funds but might not know to use it for program upgrades.
2. H012 alone (program keypairs) notes that program address keypairs do not grant upgrade authority — an attacker who only reads `keypairs/amm-keypair.json` etc. cannot upgrade programs.
3. Combined: the devnet wallet (`8kPzh...`) is both (a) the funded wallet exposed in `.mcp.json` AND (b) the upgrade authority for all 6 programs exposed in H012. The two findings point to the same key from two different attack angles. A single key extraction achieves both fund theft and full protocol replacement.

Furthermore, the Squads signer keys (H012 escalation) specifically neutralise the planned remediation path, making the combined finding worse than the sum of its parts.

## Recommendation

Remediating H001 and H012 individually remedies this combination. The recommendations below are ordered by urgency and cover both constituent issues.

### Immediate (within hours — before any further devnet use)

1. **Accept that existing devnet keys are permanently compromised.** They are in git history. No `git rm` addresses this — the keys remain in `git show 53ca01b:.mcp.json` and every `keypairs/*.json` historical commit. History must be rewritten.

2. **Generate a completely fresh devnet wallet keypair** outside the repository directory:
   ```bash
   solana-keygen new -o ~/fresh-devnet-wallet.json --no-bip39-passphrase
   ```
   Never move this file into the repository.

3. **Transfer any remaining devnet SOL** from `8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4` to the new wallet immediately, before any attacker acts.

4. **Generate fresh program keypairs** for all 6 programs (also outside the repo). A fresh deploy with new program IDs is required because the old program IDs' upgrade authority is the compromised key.

5. **Generate fresh Squads signer keypairs** (3 signers + create key). Store them in a password-protected keystore or hardware wallet, never in the repository.

6. **Replace `.mcp.json` private key with a placeholder**, add `.mcp.json` to `.gitignore`, commit the change. This stops the live key from being read by AI tools in future sessions.

### Short-term (within days — before devnet governance rehearsal)

7. **Add `keypairs/*.json` to `.gitignore`** (broad rule, belt-and-suspenders with existing `keypairs/mainnet-*` rule). The devnet keys that were in git cannot be un-exposed, but new keys generated from this point will not be tracked.

8. **Purge git history** using BFG Repo-Cleaner or `git filter-repo` to remove all keypair JSON files and `.mcp.json` from all historical commits:
   ```bash
   bfg --delete-files 'keypairs/*.json'
   bfg --replace-text secrets.txt  # contains the .mcp.json base58 key
   git reflog expire --expire=now --all && git gc --prune=now --aggressive
   git push --force --all && git push --force --tags
   ```
   All collaborators must re-clone after the force push.

9. **Do not run `transfer-authority.ts` until Squads signer keys are rotated and not in git.** The planned governance flow provides zero security benefit while any Squads signer key is accessible from the repository. Running it would transfer upgrade authority to a multisig that the attacker can already pass (2-of-3 threshold met with committed keys).

### Before Mainnet (CRITICAL GATE)

10. **Confirm mainnet deploy uses ONLY non-committed keypairs.** Verify that `deploy-all.sh` and `initialize.ts` never reference `keypairs/devnet-wallet.json` when given a mainnet cluster URL.

11. **Confirm the mainnet deployer (`23g7x...`) and crank wallet (`F84X...`) have no cryptographic relationship** to the compromised devnet wallet (`8kPzh...`). Confirm neither was derived from the same seed phrase or BIP44 derivation path.

### Long-term

12. **Add a pre-commit hook** (e.g., `gitleaks`, `truffleHog`, or `detect-secrets`) that rejects commits containing:
    - JSON arrays of 64 integers (keypair format)
    - Base58 strings of 87–88 characters (Solana private key format)
    - Any filename matching `*keypair*.json`, `*wallet*.json`, `*signer*.json`

13. **Use hardware wallets or HSMs for upgrade authority** on all environments, including devnet. The private key never exists as a file on disk, making accidental committing structurally impossible.
