# H005: Webhook Secret Compromise → Protocol State Injection

## Verdict: CONFIRMED (HIGH)

---

## Evidence

### Attack Surface

**File:** `app/app/api/webhooks/helius/route.ts` (lines 525–633)
**File:** `app/lib/protocol-store.ts` (lines 53–65)

### Authentication Chain (Correct)

The webhook authentication is correctly implemented:
- `timingSafeEqual()` constant-time comparison (lines 293–300)
- Fail-closed in production: `NODE_ENV=production` + missing secret → HTTP 500 (lines 273–284)
- Rate-limited at 120 req/min per IP (line 258)

The gate is sound. The vulnerability lies entirely downstream of it — **what happens after the gate is passed with a valid secret**.

### No Post-Decode Validation

After Anchor decodes `rawAccountData` into an account struct, **no bounds or plausibility validation is applied** before the data enters `protocolStore`:

```ts
// route.ts:590–606
const decoded = program.coder.accounts.decode(decodeInfo.accountType, rawBuffer);
const normalized = anchorToJson(decoded, ...);
protocolStore.setAccountState(pubkey, {
  ...normalized,
  updatedAt: Date.now(),   // freshness timestamp overwritten unconditionally
});
```

Anchor's `BorshCoder.decode()` performs structural deserialization only — it validates field types (u64 fits in u64), not semantic ranges. An attacker supplying crafted base64 `rawAccountData` can produce:
- `crimeBuyTaxBps: 9999` (99.99% tax)
- `reserveA: 1`, `reserveB: 999_999_999_999` (wildly skewed reserves)
- `cheapSide: 99` (invalid enum value, may not even throw)
- `currentEpoch: 99999` (impossible epoch number)

All of these pass Anchor deserialization without error and are stored verbatim.

### Alternative Injection Path (No Anchor Decode Required)

For the `CarnageSolVault` (SystemAccount), there is **no Anchor decode at all**:

```ts
// route.ts:553–560
if (decodeInfo === null) {
  const lamports = item.accountData?.[0]?.nativeBalanceChange;
  protocolStore.setAccountState(pubkey, {
    lamports: typeof lamports === "number" ? lamports : 0,
    updatedAt: Date.now(),
  });
```

Attacker sets `nativeBalanceChange` to any arbitrary integer — no validation required, no decode step.

### Decode Failure Also Injects (Fallback Path)

If the attacker provides malformed base64 that passes structural decode but triggers a downstream error, the catch block still calls `setAccountState` with the raw payload:

```ts
// route.ts:613–619
protocolStore.setAccountState(pubkey, {
  label,
  accountData: item.accountData,    // attacker-controlled
  rawAccountData: item.rawAccountData,
  updatedAt: Date.now(),
  decodeError: err.message,
});
```

This means **even a decode failure is an injection**, as the raw `accountData` object is stored and broadcast.

### No Dedup Protection Against Crafted Data

`protocolStore.setAccountState` deduplicates against the last broadcast:

```ts
// protocol-store.ts:58
if (serialized === this.lastSerialized.get(pubkey)) return;
```

This only blocks re-broadcast of **identical** data. Different crafted data (different reserves, different tax rates) bypasses dedup entirely and is immediately broadcast to all SSE clients.

### `updatedAt` Timestamp Forgery

The field `updatedAt: Date.now()` is injected by the server, not the attacker. However this means attacker-injected data **looks freshly sourced** — consumers cannot distinguish it from legitimate on-chain data by timestamp alone.

### SSE Broadcast: Zero Filtering

`sseManager.broadcast()` sends data to all subscribers unconditionally:

```ts
// sse-manager.ts:60–69
broadcast(event: string, data: unknown): void {
  const payload = `event: ${event}\ndata: ${JSON.stringify(data, bigintReplacer)}\n\n`;
  for (const callback of this.subscribers) {
    callback(payload);
  }
}
```

`/api/sse/protocol/route.ts` forwards any `protocol-update` event unchanged (line 84–88). No validation, no schema enforcement.

### Browser Consumption: Zero Validation

`useProtocolState` accepts and stores the SSE payload as-is:

```ts
// useProtocolState.ts:269–273
const { account, data } = JSON.parse(event.data, bigintReviver);
setAccounts((prev) => ({ ...prev, [account]: data }));
```

`useEpochState` extracts `crimeBuyTaxBps` directly with a type cast:

```ts
// useEpochState.ts:55
crimeBuyTaxBps: sseData.crimeBuyTaxBps as number,
```

`usePoolPrices` extracts `reserveA`/`reserveB` similarly with no range check:

```ts
// usePoolPrices.ts:73–75
reserveA: data.reserveA as number,
reserveB: data.reserveB as number,
```

### Direct Impact on `minimumOutput`

`useSwap` feeds SSE-derived reserves and tax rates directly into the quote engine, which produces `minimumOutput` — the on-chain slippage guard the user signs:

```ts
// useSwap.ts:408–418
const reserveWsol = poolData.reserveA;     // from SSE, no validation
const reserveToken = poolData.reserveB;    // from SSE, no validation
const buyTaxBps = epochState!.crimeBuyTaxBps;  // from SSE, no validation
const result = quoteSolBuy(BigInt(baseUnits), BigInt(reserveWsol), BigInt(reserveToken), BigInt(buyTaxBps), SOL_POOL_FEE_BPS_BI);
const minimumOutput = Math.floor(outputTokens * (10_000 - slippageBps) / 10_000);
```

If `reserveWsol` is injected as extremely large (e.g., `10^18`), `quoteSolBuy` computes a near-zero output, `minimumOutput` is near zero, and the TX is signed with `minimumOutput ≈ 0` — the on-chain slippage backstop is nullified. The actual on-chain swap executes at real reserves and steals any extractable value via MEV (or the attacker's own front-run).

### Scope of Accounts That Can Be Injected

The `KNOWN_PROTOCOL_ACCOUNTS` lookup (lines 538–548) does restrict injection to the **8 pre-registered protocol PDAs** (EpochState, CarnageFundState, CarnageSolVault, CRIME_SOL Pool, FRAUD_SOL Pool, StakePool, CurveState:CRIME, CurveState:FRAUD). An attacker cannot inject arbitrary pubkeys. However, all 8 accounts are high-value targets that directly influence displayed prices, tax rates, staking rewards, and bonding curve states.

## Attack Path

**Prerequisite:** Attacker has obtained `HELIUS_WEBHOOK_SECRET`.

**Acquisition vectors (ranked by likelihood):**
1. Railway dashboard breach (single point of compromise for all env vars)
2. Git history scan — prior audit found devnet key in `.env.devnet`; pattern suggests mainnet key may have been in a similar file
3. Server log exposure — the secret appears in request headers and could leak via error logging
4. Social engineering / phishing of Railway account
5. Supply chain compromise of npm dep on Railway build (crank builds with `npm install`)

**Exploitation sequence:**

1. Attacker crafts an enhanced webhook payload targeting the CRIME_SOL PoolState account:
   ```json
   [{
     "account": "<CRIME_SOL_POOL_PDA>",
     "accountData": [{"account": "<CRIME_SOL_POOL_PDA>", "nativeBalanceChange": 0}],
     "rawAccountData": {
       "data": "<base64-encoded crafted PoolState with reserveA=10^18, reserveB=1>",
       "encoding": "base64"
     }
   }]
   ```

2. POST to `https://<deployment>/api/webhooks/helius` with `Authorization: <stolen_secret>`.

3. Rate limit: 120 req/min per IP. Attacker can sustain continuous injection indefinitely at this rate.

4. `handleAccountChanges` decodes the crafted buffer. Anchor deserializes `reserveA=10^18` as a valid `u64`. No semantic check fails.

5. `protocolStore.setAccountState(poolPDA, { reserveA: 10^18, reserveB: 1, ... })` called. Dedup: data differs from current, broadcast fires.

6. All connected SSE clients receive `protocol-update` for the pool PDA. `useProtocolState` stores it. `usePoolPrices` extracts `reserveA=10^18`.

7. Any user opening a Buy order sees: displayed output = near-zero tokens (the formula `reserveToken * input / (reserveIn + input)` with `reserveIn=10^18` produces effectively 0). User is shown a bad quote.

8. More critically: `minimumOutput` computed from these fake reserves is near zero. User signs TX with `minimumOutput ≈ 0`. On-chain, actual reserves are legitimate, so the actual output is real — but the attacker (running a front-run MEV bot watching the mempool) can extract up to the full swap value because `minimumOutput=0` imposes no guard.

**Alternative — tax rate manipulation:**

Inject `EpochState` with `crimeBuyTaxBps=9999`. Users see "tax: 99.99%" in UI. Rational users avoid the pool, suppressing volume. Attacker takes opposing position at real rates. Or: inject `crimeBuyTaxBps=0`, users see "0% tax", are incentivized to buy, attacker sells into the volume.

**Alternative — bonding curve manipulation:**

Inject `CurveState:CRIME` with inflated `totalSold` or `currentPrice`. Affects the BuySellPanel display. Users make buy/sell decisions based on false curve state.

**Persistence:** The protocolStore is in-memory with no TTL. Injected state persists until the legitimate ws-subscriber overwrites it (next real account change from Helius, or the 30s/60s polling cycles).

## Impact

| Dimension | Assessment |
|-----------|-----------|
| Confidentiality | Low — SSE data is already public |
| Integrity | **CRITICAL** — all connected browsers display attacker-controlled prices, rates, and reserves |
| Availability | Medium — can inject zero-reserve state to make swaps appear impossible |
| Financial | **HIGH** — manipulated `minimumOutput` nullifies on-chain slippage guard; every affected user's TX is sandwichable with 0 protection |
| Scope | All simultaneously connected users (not per-user) |
| Duration | Persists until ws-subscriber next polls (up to 60s) or next legitimate webhook delivery |
| Detection | Difficult — injected data looks freshly sourced (`updatedAt: Date.now()`) |

The on-chain `minimumOutput` enforcement is the primary safety net for users. This attack neutralizes it without touching any on-chain program. Users sign transactions with `minimumOutput` computed from attacker-controlled reserves, exposing their full swap value to MEV.

## Recommendation

### Required (pre-mainnet)

**1. Semantic validation layer in `handleAccountChanges`**

After Anchor decode, validate field ranges before storing:

```ts
function validateDecodedAccount(label: string, decoded: Record<string, unknown>): void {
  if (label.startsWith("PoolState:")) {
    const reserveA = Number(decoded.reserveA);
    const reserveB = Number(decoded.reserveB);
    if (reserveA <= 0 || reserveB <= 0) throw new Error(`Invalid reserves: ${reserveA}, ${reserveB}`);
    // Plausibility: reserves should not exceed known program caps (e.g., 10B SOL = 10^19 lamports)
    if (reserveA > 1e19 || reserveB > 1e19) throw new Error(`Implausible reserves`);
  }
  if (label === "EpochState") {
    const crimeBuyTaxBps = Number(decoded.crimeBuyTaxBps);
    if (crimeBuyTaxBps < 0 || crimeBuyTaxBps > 10_000) throw new Error(`Invalid taxBps: ${crimeBuyTaxBps}`);
    // same for all 4 tax fields
  }
}
```

Throw on validation failure — **do not fall through to `setAccountState`**. Log and return without updating the store.

**2. Cross-verify against ws-subscriber (defense in depth)**

The ws-subscriber independently reads the same PDAs from Helius RPC. If webhook-injected data differs materially from the ws-subscriber's last known good state (e.g., reserve ratio changes >50% in one webhook), flag and reject the webhook update. This makes injection detectable even with a valid secret.

**3. Add replay/sequence protection to enhanced webhooks**

The current code has no sequence number or freshness timestamp on enhanced webhooks (the `AUTH-01` auditor noted this). Even a simple `X-Webhook-Timestamp` header check would narrow the injection window.

**4. Harden secret storage**

Rotate `HELIUS_WEBHOOK_SECRET` immediately after any suspected Railway breach. Document a rotation procedure (currently no rotation support exists). Consider HMAC-body signing if Helius adds support.

### Advisory (medium-term)

**5. Browser-side sanity floor on `minimumOutput`**

In `useSwap`, add a floor: if the computed `minimumOutput` is less than some percentage of the spot-rate quote (e.g., <50% of naive output), warn the user and require explicit confirmation. This is a second line of defense that protects users even if the server store is corrupted.

## Evolution

**Classification:** Novel — not present in Audit #1. This is a new attack path arising from the Audit #2 delta's introduction of the `handleAccountChanges` / `protocolStore` / SSE pipeline (the DBS refactor), which did not exist during Audit #1.

**Relationship to previous findings:**
- H001 (Webhook Auth Bypass, FIXED) — H005 is the *next stage* after H001. H001's fix correctly gates the entry point; H005 shows the gate alone is insufficient if the secret is compromised.
- H049 (Webhook No Replay Protection, FIXED for raw TX path) — H005 is the enhanced-path analogue. Enhanced webhooks still lack replay/freshness protection.
- ASM-6 (Helius delivers correct Solana data) — H005 is the off-chain equivalent: if the webhook channel is compromised, the assumption fails identically.
- INV-14 (Protocol store dedup fragile) — confirmed; dedup provides zero protection against distinct crafted data.

**Severity rationale:** Downgraded from the hypothesis's CRITICAL to HIGH because:
- Requires prior acquisition of `HELIUS_WEBHOOK_SECRET` (not a zero-day, requires a separate breach)
- The on-chain `minimumOutput` enforcement remains the ultimate backstop for funds; the attack degrades UX/slippage protection but cannot directly steal funds without a coordinated MEV front-run
- Effect is transient (clears on next legitimate webhook or ws-subscriber poll, ≤60s)

Remains HIGH (not MEDIUM) because: the financial impact via MEV nullification is direct and scalable, affects all concurrent users simultaneously, and is silent (no observable error to users).
