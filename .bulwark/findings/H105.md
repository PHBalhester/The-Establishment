# H105: Log Injection via Webhook Pool Type

**Tier**: 3
**Component**: `app/app/api/webhooks/helius/route.ts`
**Lines**: 407–409
**Status**: NOT VULNERABLE

## Summary

Webhook handler logs unknown pool types without explicit sanitization:

```typescript
// route.ts:407–409
console.warn(
  `[webhook] Unknown pool type: ${swap.poolType} in tx ${signature}`
);
```

Initial concern: `swap.poolType` is interpolated into a log message without validation. If `poolType` came from untrusted input, newlines or special characters could inject fake log entries.

## Analysis

### Source of `poolType`

The `poolType` field originates from on-chain Anchor events parsed by `event-parser.ts`:

```typescript
// event-parser.ts:258, 278
poolType: enumVariant(data.pool_type),
```

The `enumVariant()` function (lines 142–152) transforms Anchor deserialized enum values to camelCase strings:

```typescript
function enumVariant(value: unknown): string {
  if (typeof value === "number") {
    return String(value);
  }
  if (typeof value === "object" && value !== null) {
    const key = Object.keys(value)[0] ?? "unknown";
    // Lowercase first char: SolCrime -> solCrime, Buy -> buy
    return key.charAt(0).toLowerCase() + key.slice(1);
  }
  return String(value);
}
```

### Control Flow

1. **Deserialization**: Anchor's `BorshCoder` deserializes the enum from on-chain bytes according to the IDL schema.
2. **String extraction**: `enumVariant()` extracts the variant name from a trusted object key.
3. **Format guarantee**: The function always returns a camelCase string (e.g., `"solCrime"`, `"solFraud"`, `"buy"`, `"sell"`).

### Known Values

`POOL_TYPE_TO_ADDRESS` (lines 119–122) defines the only valid pool types:

```typescript
const POOL_TYPE_TO_ADDRESS: Record<string, string> = {
  solCrime: DEVNET_POOLS.CRIME_SOL.pool.toBase58(),
  solFraud: DEVNET_POOLS.FRAUD_SOL.pool.toBase58(),
};
```

Unknown types (e.g., historical `"crimeProfit"` from Phase 4) are logged as warnings and skipped.

## Vulnerability Assessment

**NOT VULNERABLE**

While the log statement lacks explicit sanitization, the `poolType` value is guaranteed to be a clean camelCase string derived from Anchor's IDL enum schema, not user-controllable input. The on-chain program controls the enum serialization; Helius delivers it unchanged.

Practical impact: An attacker would need to:
1. Compromise the on-chain Anchor program to emit a malformed enum variant, **OR**
2. Compromise Helius webhooks to inject arbitrary event data.

Both are out-of-scope for a log injection vulnerability (requires chain-level compromise).

## Recommendation

**No change required.** The `poolType` source is sufficiently controlled. If hardening log output is desired for defense-in-depth, consider:

```typescript
console.warn(`[webhook] Unknown pool type: ${String(swap.poolType).slice(0, 64)} in tx ${signature}`);
```

This limits log line length without affecting functionality (enum names are ~16 chars).
