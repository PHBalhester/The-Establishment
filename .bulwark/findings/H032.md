## Summary

The zero-dependency Sentry integration (`app/lib/sentry.ts`) is structurally intact and the custom implementation is correctly in place. However, the devnet environment has **no DSN configured** — `NEXT_PUBLIC_SENTRY_DSN` and `SENTRY_DSN` are absent from `app/.env.local` — meaning `captureException` silently no-ops on every call in the development and current Railway devnet deployment. Additionally, one file (`app/hooks/useProtocolState.ts`) imports `captureException` but never calls it — error paths swallow exceptions with only `console.warn`, producing dead import noise. The mainnet template has placeholder values (`CHANGE_ME_MAINNET`), which will also silently no-op if Railway is deployed without substituting real DSN values.

**Verdict: PARTIAL_REGRESSION** — the fix from H045 is architecturally present (zero-dep implementation, no `@sentry/*` packages, Turbopack-safe), but operational coverage is incomplete due to missing DSN configuration and one dead import.

## Findings

### 1. Sentry DSN Not Configured in Devnet Environment (Operational Gap)

**File:** `app/.env.local`

`app/.env.local` contains no `NEXT_PUBLIC_SENTRY_DSN` or `SENTRY_DSN` entry. The `captureException` function gates on DSN presence at line 146:

```typescript
if (!sentryDsn) return;
```

Since `typeof process !== "undefined"` is true in Node.js and the env var is unset, every call to `captureException` in the devnet deployment returns immediately without sending anything to Sentry. All 14 call sites across 8 files are silently inert.

**Impact:** Zero error visibility in devnet. Webhook decode failures, RPC proxy exhaustion, candle query errors, health check failures — none are reported to Sentry. If the devnet deployment is used for pre-mainnet staging validation, errors are invisible until Railway logs are manually inspected.

**Call sites currently producing no telemetry:**
- `app/app/api/webhooks/helius/route.ts` — 4 call sites (missing secret, candle upsert error, per-TX error, fatal error)
- `app/app/api/health/route.ts` — 2 call sites (Postgres check failure, Solana RPC check failure)
- `app/app/api/rpc/route.ts` — 1 call site (all RPC endpoints failed)
- `app/app/api/candles/route.ts` — 1 call site (query error)
- `app/app/api/carnage-events/route.ts` — 1 call site (query error)
- `app/app/api/sol-price/route.ts` — 1 call site (all price providers unavailable)
- `app/app/global-error.tsx` — 1 call site (React error boundary)
- `app/instrumentation-client.ts` — 2 call sites (window.error, unhandledrejection)

### 2. Mainnet Template Has Placeholder DSN Values (Pre-Mainnet Risk)

**File:** `app/.env.mainnet` lines 40, 52

```
SENTRY_DSN=CHANGE_ME_MAINNET
NEXT_PUBLIC_SENTRY_DSN=CHANGE_ME_MAINNET
```

`parseDsn()` in `sentry.ts` calls `new URL(dsn)` on this string. `new URL("CHANGE_ME_MAINNET")` throws a `TypeError` (invalid URL), which is caught by the outer `try/catch` at line 232 and silently swallowed. The net effect is identical to no DSN configured: no errors are reported, no exception surfaces.

**Impact:** If Railway's mainnet environment is deployed with these placeholder values intact (not substituted), the production deployment runs with zero Sentry coverage. The silent failure mode means the operator receives no signal that error reporting is broken — it appears configured but is not functional.

---

### 3. Dead Import in `useProtocolState.ts` (Code Quality / Minor Coverage Gap)

**File:** `app/hooks/useProtocolState.ts` line 40

```typescript
import { captureException } from "@/lib/sentry";
```

`captureException` is imported but never called anywhere in the file's 365 lines. SSE error paths (`es.onerror`) log nothing and swallow exceptions silently (lines 260, 278). The exponential-backoff reconnection loop at lines 285–299 has no error telemetry. RPC polling failures at line 214 emit only `console.warn`.

**Impact:** SSE disconnect events and RPC polling failures in the hook are invisible to Sentry even when a DSN is configured. This is the client-side data pipeline hook — sustained SSE unavailability would go unreported.

## What H045 Fixed (Still Intact)

The core architectural fix from H045 is **confirmed intact**:

- No `@sentry/*` npm packages installed (`node_modules/@sentry` does not exist)
- `next.config.ts` line 120 explicitly documents the removal: `// @sentry/nextjs REMOVED`
- `app/lib/sentry.ts` is a 236-line zero-dependency implementation using raw `fetch()` to Sentry's envelope API
- Turbopack-safe: no webpack monkey-patching, no module-level browser globals
- Edge runtime guard in `instrumentation.ts` lines 14–15 prevents crashes in edge runtime
- CSP in `next.config.ts` line 43 includes `https://*.ingest.sentry.io https://*.ingest.us.sentry.io` (US region support)
- `instrumentation-client.ts` correctly registers `window.error` and `unhandledrejection` handlers
- `global-error.tsx` correctly calls `captureException` in React error boundary `useEffect`

The implementation is sound. The regression is operational, not structural.

## Recommended Remediation

1. **Add DSN to devnet env**: Set `NEXT_PUBLIC_SENTRY_DSN` and `SENTRY_DSN` in `app/.env.local` (devnet project or a separate devnet Sentry project).

2. **Mainnet deploy gate**: Add a startup assertion in `instrumentation.ts` that validates the DSN is set and parseable when `NODE_ENV === "production"`. Log a clear error if missing rather than silently no-opping.

3. **Wire `captureException` in `useProtocolState.ts`**: Replace the dead import with actual calls in `es.onerror` (after N consecutive reconnect failures) and in the RPC polling catch block.

4. **Railway secret substitution verification**: Add a Railway pre-deploy check or CI step that asserts `NEXT_PUBLIC_SENTRY_DSN` does not equal the literal string `CHANGE_ME_MAINNET`.
