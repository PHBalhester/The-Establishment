## Revised Chain Assessment

The originally proposed chain for S003 was:

> Helius API key → webhook URL discovery → webhook secret brute-force → injection

H064 collapses the brute-force step entirely. The chain as it actually exists in the codebase is:

> Helius API key (H004) + webhook secret in git (H064) → webhook URL enumeration → authenticated payload injection (H005)

**There is no brute-force step.** The secret is directly readable from commit `0e8ff92` in the tracked file `.planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md`. The three findings combine into a complete end-to-end attack chain requiring only repository read access — the same single prerequisite as S001.

## Combined Attack Path

**Step 1 — Repository access.**

Same as S001. Attacker obtains a clone of the repository through any of the vectors described in H001 (leaked collaborator access, accidental public exposure, future open-source release).

**Step 2 — Extract Helius API key.**

```bash
grep HELIUS_API_KEY .env.devnet
# → [REDACTED-DEVNET-HELIUS-KEY]
```

Also available in `shared/programs.ts:24` as a bundled constant, directly in the browser JavaScript bundle of the live devnet deployment.

For mainnet, the primary API key `[REDACTED-MAINNET-HELIUS-KEY]` is in the SUMMARY.md commit (H064 incidental discovery).

**Step 3 — Enumerate production webhook URL.**

```bash
curl "https://api.helius.xyz/v0/webhooks?api-key=[REDACTED-DEVNET-HELIUS-KEY]"
```

Response includes the full `webhookURL` of all registered webhooks (e.g., `https://dr-fraudsworth-production.up.railway.app/api/webhooks/helius`), webhook IDs, monitored account addresses, and webhook type. The production webhook URL is now known. No guesswork.

**Step 4 — Extract the webhook secret (no brute-force required).**

```bash
git show 0e8ff92:.planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md | grep HELIUS_WEBHOOK_SECRET -A2
```

Output:
```
| HELIUS_WEBHOOK_SECRET | [REDACTED-WEBHOOK-SECRET] |
```

The mainnet webhook secret is in permanent git history. The SUMMARY.md document's own prose states the secret was "stored only in Railway env vars (not in any repo file)" — this statement is false, as the secret appears four lines earlier in the same document.

**Step 5 — Construct authenticated injection payload.**

Attacker now holds: the production webhook URL (Step 3), the webhook secret (Step 4). They have complete bypass of the `timingSafeEqual` authentication gate in `webhooks/helius/route.ts`.

Crafted payload targeting CRIME_SOL PoolState:
```json
POST https://dr-fraudsworth-production.up.railway.app/api/webhooks/helius
Authorization: [REDACTED-WEBHOOK-SECRET]
Content-Type: application/json

[{
  "account": "<CRIME_SOL_POOL_PDA>",
  "rawAccountData": {
    "data": "<base64-encoded crafted PoolState: reserveA=10^18, reserveB=1>",
    "encoding": "base64"
  },
  "accountData": [{"account": "<CRIME_SOL_POOL_PDA>", "nativeBalanceChange": 0}]
}]
```

**Step 6 — Protocol state injection (H005 impact).**

The authenticated payload passes `timingSafeEqual`. Anchor decodes `reserveA=10^18` as a valid `u64` — no semantic check fails. `protocolStore.setAccountState` stores the injected values. All SSE clients receive `protocol-update`. Every connected user's `useSwap` computes `minimumOutput ≈ 0`.

**Step 7 — Parallel candle corruption.**

H005's attack path also covers `TaxedSwap` event injection. With the secret known, the attacker can also inject crafted `TaxedSwap` events with extreme prices. The candle upsert uses `GREATEST`/`LEAST` for high/low — a single crafted event permanently corrupts the high/low for that candle bucket, persisting across server restarts. Chart data is durably corrupted.

**Step 8 (optional) — Redirect or delete webhook.**

The attacker can simultaneously, using the API key from Step 2:
```bash
DELETE https://api.helius.xyz/v0/webhooks/{id}?api-key=[REDACTED-DEVNET-KEY]...
```
This deletes the production webhook entirely, preventing Helius from delivering legitimate events. The injected state in protocolStore persists until the next ws-subscriber poll (30–60s), after which the store goes stale with no fresh data. This amplifies S002.

## Why H064 Changes the Severity Classification

The originally hypothesized S003 chain included a brute-force step, which would have significantly limited the attack's practicality (brute-forcing a 32-byte hex secret is computationally infeasible). H064 removes this limiting factor entirely.

| Chain Component | Original Hypothesis | Actual State |
|-----------------|--------------------|----|
| API key acquisition | Git clone | Git clone (confirmed) |
| Webhook URL discovery | Helius API enumeration | Helius API enumeration (confirmed) |
| Secret acquisition | Brute-force (infeasible) | Git history (trivial — one command) |
| Injection execution | Speculative | Fully verified (H005) |
| Total complexity | High | **Trivial — single attacker, one repo clone** |

The revised chain is **not a combination of three independent high-severity findings** — it is a **single fully-connected attack chain** where each link is already confirmed and the connective tissue (the secrets are co-located in the same repository) is also confirmed.

**Severity: CRITICAL.**

The combination is CRITICAL (not the HIGH of each constituent) because:
1. The secret acquisition step has zero cost — it is already readable from git history
2. The authentication gate (`timingSafeEqual`) is completely bypassed — this was the primary mitigating factor that justified H005's HIGH rather than CRITICAL rating
3. Candle corruption is durable — injected `TaxedSwap` events with extreme prices corrupt the OHLCV database permanently for those time buckets, surviving server restarts and protocol upgrades
4. The chain requires no specialized skill, no active network attack, and no timing — it is executable offline from a git clone

## Recommendation

**Fix 1 (CRITICAL — Immediate): Rotate `HELIUS_WEBHOOK_SECRET`.**

Generate a new secret:
```bash
openssl rand -hex 32
```
Update Railway env vars for the production deployment simultaneously with updating the Helius webhook `authHeader`. The rotation must be atomic — a window between updating Helius and updating Railway will either reject legitimate webhooks or accept old-secret payloads.

The old secret `[REDACTED-WEBHOOK-SECRET]` is permanently in git history. Rotation does not remove it from history; it only makes the committed value invalid going forward.

**Fix 2 (CRITICAL — Immediate): Purge the secret from git history.**

```bash
# Using git-filter-repo (preferred over BFG for this case):
git filter-repo --path .planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md --invert-paths
# OR: redact only the secret value while preserving the file:
git filter-repo --replace-text <(echo '[REDACTED-WEBHOOK-SECRET]==>REDACTED')
```

Force-push all branches. All collaborators must re-clone. If the repository is ever made public, the window between now and the force-push is a critical exposure period.

**Fix 3 (CRITICAL): Add semantic validation to `handleAccountChanges` (H005 fix).**

Even after secret rotation, the injection surface at the webhook endpoint remains. A future secret leak would re-enable the full chain. Semantic validation (reserve bounds, tax BPS ranges) provides defense-in-depth that limits H005 impact independently of secret hygiene.

**Fix 4 (HIGH): Rotate the Helius API key (H004 fix).**

The API key `[REDACTED-DEVNET-KEY]...` is the enabling tool for webhook URL discovery (Step 3). Without it, an attacker must guess the production webhook URL. Rotating the API key raises the bar for Step 3, though the webhook URL is still semi-discoverable via public domain conventions.

**Fix 5 (HIGH): Establish a documentation secret policy.**

The root cause of H064 — and therefore the enablement of S003 — is that a SUMMARY.md file documenting the deployment phase included live secret values in a markdown table. The document explicitly stated the secret was not in any repo file, contradicting reality.

Going forward: SUMMARY.md and planning files must never contain production secret values. Use placeholder markers like `<ROTATED — SEE RAILWAY>` in any committed documentation. Consider a pre-commit hook that scans for patterns matching `openssl rand -hex 32` output (64-character hex strings) in markdown files.

**Fix 6 (HIGH): Remove `shared/programs.ts` Helius API key from client bundle.**

The devnet API key bundled into the client JavaScript (and callable from the browser console against a CSP-whitelisted `api.helius.xyz`) enables Step 2 without any repository access — the key is in the live site's JS bundle. This is an independent acquisition vector that survives a git history purge.

**Fix 7 (MEDIUM): Implement webhook health monitoring.**

A server-side scheduled check (every 5 minutes) should verify the production webhook exists and is pointing to the correct URL. Alert on any change to `webhookURL` or `authHeader`. This would detect Step 8 (webhook deletion or redirect) within 5 minutes rather than at the next human review.
