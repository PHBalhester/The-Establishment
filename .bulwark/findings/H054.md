# H054: Crank Carnage Recovery Skips Atomic Bundling

## Verdict: CONFIRMED (Partial — Liveness Failure, Not Classic MEV Sandwich)

---

## Hypothesis Restatement

The VRF recovery path (stale VRF or oracle timeout) returns `carnageExecutedAtomically: false`.
If Carnage is triggered during recovery, the `carnage_pending` flag is visible on-chain, and MEV
bots can sandwich the subsequent `execute_carnage` call.

---

## Code Investigation

### Primary target: `scripts/vrf/lib/vrf-flow.ts`

Three distinct code paths can return `carnageExecutedAtomically: false` when `carnageTriggered: true`:

**Path A — TOCTOU early return (line 469-486):**
```
sendRevealAndConsume → VrfNotPending / "already" error
→ reads on-chain state
→ returns carnageTriggered: stateAfter.lastCarnageEpoch === stateAfter.currentEpoch
         carnageExecutedAtomically: false   ← hardcoded
```
This path fires when a *second* crank instance consumed the randomness before the current one.
The crank that hit TOCTOU never submitted TX3 at all.

**Path B — Stale VRF recovery, direct reveal success (lines 622-644):**
```
staleRevealSucceeded = true
→ sendRevealAndConsume(... accounts ...) called with full accounts object
→ returns carnageExecutedAtomically: false   ← hardcoded at line 644
```
`sendRevealAndConsume` checks `if (accounts.carnageAccounts && accounts.alt)` and DOES build the
bundled v0 transaction (reveal + consumeRandomness + executeCarnageAtomic) when those accounts are
present. The `carnageExecutedAtomically: false` on line 644 is therefore an **incorrect flag** —
Carnage may have executed atomically, but the return value reports it did not.

**Path C — Oracle timeout recovery (lines 824-832):**
```
retryRevealResult obtained
→ sendRevealAndConsume(... accounts ...) called
→ falls through to normal final-state return (lines 855-921)
→ carnageExecutedAtomically = !!(carnageAccounts && !stateAfter.carnagePending && stateAfter.lastCarnageEpoch === stateAfter.currentEpoch)
```
This path computes the flag correctly. No false negative here.

### On-chain Carnage timing constraints: `programs/epoch-program/src/`

From `constants.rs`:
- `CARNAGE_LOCK_SLOTS = 50` — fallback `execute_carnage` is blocked for the first 50 slots
- `CARNAGE_DEADLINE_SLOTS = 300` — after 300 slots, `expire_carnage` can clear the state

From `execute_carnage.rs` (handler):
```rust
require!(clock.slot <= epoch_state.carnage_deadline_slot, EpochError::CarnageDeadlineExpired);
require!(clock.slot > epoch_state.carnage_lock_slot, EpochError::CarnageLockActive);
```

From `execute_carnage_atomic.rs`: no `NoCarnagePending` constraint — it is a no-op when
`carnage_pending = false`. The lock constraint is absent — atomic path is callable during the lock
window (it is always bundled in TX3).

From `consume_randomness.rs` (lines 289-314): when Carnage triggers, `carnage_pending = true`,
`carnage_lock_slot` and `carnage_deadline_slot` are set, and a `CarnagePending` event is emitted
on-chain — publicly observable to any Solana listener.

### Crank-runner: no `execute_carnage` follow-up call

`scripts/crank/crank-runner.ts` uses `carnageExecutedAtomically` only for logging (line 519) and
spend accounting (line 499). There is **no branch** that calls the fallback `execute_carnage`
instruction when `carnageExecutedAtomically: false`. The crank does not attempt to salvage a
non-atomic Carnage.

## Confirmed Behaviors

### 1. Path B — False Negative Flag (Incorrect Logging, Not a Security Gap)

When the stale VRF recovery path succeeds via `sendRevealAndConsume` with full `carnageAccounts`,
Carnage executes atomically in the bundled v0 TX, but the function returns
`carnageExecutedAtomically: false`. This is a **reporting bug**: the flag is wrong, but the
execution was correct. No MEV window opens.

### 2. Path A — TOCTOU Genuine Non-Atomic Case

When another crank instance consumes randomness first (TOCTOU), the current crank never submitted
TX3. If the consuming instance also had `carnageAccounts` available, it likely executed Carnage
atomically — but the current crank cannot know this, and correctly returns `carnageExecutedAtomically: false`.

If the consuming instance lacked carnage accounts (legacy path, no ALT), Carnage would not have
been bundled, `carnage_pending` would remain `true` on-chain, and the fallback window opens after
50 slots. Neither crank instance calls `execute_carnage`. The Carnage execution is abandoned to
whoever (if anyone) calls the permissionless fallback instruction.

### 3. Permissionless Fallback Race — The Real MEV Surface

When `carnage_pending = true` persists on-chain after the lock expires (slot +50 to +300):
- `execute_carnage` is callable by anyone
- A bot observing the on-chain `CarnagePending` event (emitted during `consume_randomness`) can
  prepare and submit `execute_carnage` immediately after slot +50
- The bot chooses favorable timing but cannot alter the pre-committed `carnage_action` or
  `carnage_target` (set during `consume_randomness`)
- The fallback path uses `CARNAGE_SLIPPAGE_BPS_FALLBACK` (75% floor vs 85% atomic floor) —
  more lenient slippage

**This is not a classic sandwich.** A sandwich requires a victim transaction the attacker can
bracket. Here, there is no victim: the crank does not call `execute_carnage`. The bot calling it
first is simply racing the permissionless instruction. The bot is not extracting value from the
crank; it is doing the protocol's work (completing Carnage) with a slightly worse slippage floor.

The actual risk is a **liveness failure**: if no one calls `execute_carnage` before deadline slot
+300, `expire_carnage` clears the pending state and the Carnage swap never executes. SOL stays in
the vault. This epoch's buyback-and-burn event is skipped entirely.

## Impact Assessment

**Tier 2 — HIGH (as assigned)**

- **Liveness**: Carnage skipped when TOCTOU recovery occurs and no bot completes the fallback.
  ~4.3% epoch probability × VRF recovery probability (low, but non-zero) → infrequent but
  material. Each skipped Carnage means one epoch of buyback-and-burn value is not delivered to
  stakers.

- **False flag on reporting bug (Path B)**: `carnageExecutedAtomically: false` in logs when
  Carnage actually did execute atomically. Low operational risk but creates incorrect monitoring
  signals — if operators build alerting on `carnageAtomic: false`, they would fire false alarms
  during stale VRF recoveries that actually succeeded.

- **Permissionless race**: Any bot that calls `execute_carnage` in the fallback window is using
  the more lenient 75% slippage floor, potentially costing the protocol ~10% more slippage than
  the 85% atomic floor. Financial impact: bounded by `MAX_CARNAGE_SWAP_LAMPORTS` (1000 SOL cap)
  × 10% slippage difference = up to 100 SOL incremental cost per event. In practice, far less.

- **No user fund risk**: `execute_carnage` operates solely on protocol-owned vault SOL. User
  deposits in the AMM are not at risk.

## Reproduction Path

1. Set up two concurrent crank instances (or simulate TOCTOU by having one instance advance
   through TX1+TX2 then stall before TX3, while a second instance completes the flow).
2. Confirm first crank hits TOCTOU branch and returns `carnageExecutedAtomically: false` with
   `carnageTriggered: true`.
3. On-chain: `epoch_state.carnage_pending = true`, `carnage_lock_slot = S+50`, `carnage_deadline_slot = S+300`.
4. Confirm neither crank calls `execute_carnage`.
5. At slot S+51, submit `execute_carnage` from a separate wallet — succeeds.
6. Observe `CARNAGE_SLIPPAGE_BPS_FALLBACK` (75%) applied rather than atomic (85%).

## Recommendations

**R1 — Crank: add `execute_carnage` follow-up in recovery path**

After a recovery path returns `carnageTriggered: true` AND `carnageExecutedAtomically: false`,
the crank should wait past the lock window and submit the fallback `execute_carnage` instruction.
This prevents the liveness failure and ensures the crank completes the buyback-and-burn even when
the atomic path was missed.

Pseudocode in `crank-runner.ts` main loop:
```typescript
if (result.carnageTriggered && !result.carnageExecutedAtomically) {
  console.warn("[crank] Carnage pending but not atomic — scheduling fallback execute_carnage");
  // Wait past CARNAGE_LOCK_SLOTS (50 slots ≈ 25s devnet / 20s mainnet)
  await waitForSlotAdvance(connection, CARNAGE_LOCK_SLOTS + 5);
  await executeCarnageFallback(provider, epochProgram, vrfAccounts);
}
```

**R2 — Fix reporting bug in Path B**

After `staleRevealSucceeded = true` calls `sendRevealAndConsume`, the return value should
read `carnage_pending` from post-TX state to determine whether Carnage executed, instead of
hardcoding `false`. The same logic used on lines 855-859 (normal path) should be applied.

**R3 — No change needed for lock window**

The 50-slot `CARNAGE_LOCK_SLOTS` window is correctly designed. It gives the crank (or any bundled
TX3) priority over the fallback path. No on-chain changes are recommended.
