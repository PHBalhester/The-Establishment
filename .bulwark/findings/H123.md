# H123: No Emergency Pause

**Finding ID:** H123
**Risk Tier:** Tier 3 (Accepted Risk)
**Status:** VERIFIED (Accepted)
**Last Verified:** 2026-03-21
**Audit Cycle:** v1.3 → Ongoing

## Summary

The Dr. Fraudsworth protocol has **no emergency pause mechanism** — no `is_paused` flag, no kill switch, no governance ability to halt on-chain operations. This is a **deliberate architectural choice**, not an oversight.

## Verification Results

### Code Audit

Searched all on-chain programs for pause-related functionality:

- **EpochState** (`programs/epoch-program/src/state/epoch_state.rs`): 164 bytes, contains 13 fields (timing, tax config, VRF state, carnage state, reserved padding). **No pause flag.**
- **PoolState** (`programs/amm/src/state/pool.rs`): 216 bytes, contains 11 fields (pool type, mints, vaults, reserves, fees, reentrant guard, bumps, token programs). **No pause flag.**
- **All programs**: Grep search for `pause|emergency|kill_switch|disabled|halted` returned **zero matches**.

### Design Rationale

Per `Docs/DECISIONS/architecture.md`:

> **An emergency pause is a centralisation vector that contradicts the decentralized protocol ethos.** The protocol is relatively simple by DeFi standards, and the tiered timelock provides a bug-fix window pre-burn. Post-burn, the response to an exploit is for users to exit — the protocol cannot be changed.

This aligns with the project vision documented in `Docs/project-overview.md`:

> There is no emergency pause, no governance, no admin intervention post-burn. The protocol either thrives on its deployed code or it doesn't.

### Mitigation Strategy

Instead of a pause mechanism, the protocol implements:

1. **Retained upgrade authority** (pre-burn) — allows critical patches during the timelocked period (2–24 hours post-launch, then burned).
2. **User agency** — users can monitor for pending upgrades and exit if they disagree with the direction.
3. **Tiered timelock** — provides a window to discover and patch vulnerabilities before authority burn.
4. **Simplicity** — the protocol is intentionally minimal, reducing the surface area for exploits.

## Conclusion

**ACCEPTED_RISK** — No pause mechanism is present, and none has been added. This aligns with the protocol's decentralization-first design philosophy.

**Next Check:** Re-verify if new state fields are added to any program. If state schemas evolve, confirm that no pause flag is introduced.
