# H020: Webhook-to-SSE Data Injection Chain

## Verdict: CONFIRMED (CONDITIONAL) — HIGH

The injection chain is structurally real and fully traceable in source. However, it is gated on a single prerequisite: obtaining `HELIUS_WEBHOOK_SECRET`. The chain itself is then unobstructed — no secondary validation layer exists at any step between webhook receipt and browser display. The severity is HIGH (not CRITICAL) because the on-chain program enforces `minimumOutput` on every swap, meaning fund theft is blocked even if all displayed protocol state is corrupted. The attack degrades to display manipulation, MEV enablement, and user deception rather than direct fund extraction.

---

## Evidence

### Step 1 — Webhook Authentication Gate

`app/app/api/webhooks/helius/route.ts:286-301`

Authentication is a single shared-secret comparison via `timingSafeEqual`. Once an attacker possesses the secret, the gate offers no further challenge. The secret is a plain environment variable with no rotation mechanism (cross-referenced: API-04 context, SEC-02 findings).

### Step 2 — Payload Type Discrimination (No Validation on Enhanced Path)

`route.ts:339-342`

```ts
const firstItem = payload[0] as Record<string, unknown> | undefined;
if (firstItem && "accountData" in firstItem) {
  return handleAccountChanges(payload as HeliusAccountChange[]);
}
```

Any payload whose first element contains an `accountData` key routes to the enhanced handler. There is no schema validation, no field type enforcement, and no age/slot check. The raw transaction path has a `MAX_TX_AGE_SECONDS=300` replay guard (`route.ts:361`); the enhanced path has none.

### Step 3 — KNOWN_PROTOCOL_ACCOUNTS Allowlist (Effective but Narrow)

`route.ts:217-226, 538-548`

```ts
const KNOWN_PROTOCOL_ACCOUNTS: Record<string, string> = {
  [DEVNET_PDAS.EpochState.toBase58()]: "EpochState",
  [DEVNET_PDAS.CarnageFund.toBase58()]: "CarnageFundState",
  ...
};

const label = KNOWN_PROTOCOL_ACCOUNTS[pubkey];
if (!label) { console.warn(...); continue; }
```

The lookup is keyed by `item.account` (the claimed pubkey from the webhook body). This IS an effective allowlist — an attacker cannot inject state for arbitrary pubkeys. However, it does not prevent injecting crafted data for the *real* protocol PDA addresses. An attacker who knows the EpochState PDA (which is deterministic and public) can submit a payload that passes the allowlist check while carrying fabricated `rawAccountData`.

### Step 4 — Anchor Decode of Attacker-Supplied Bytes

`route.ts:577-606`

```ts
const rawBuffer = Buffer.from(item.rawAccountData.data, "base64");
const decoded = program.coder.accounts.decode(
  decodeInfo.accountType,
  rawBuffer,
);
const normalized = anchorToJson(decoded, ...);
protocolStore.setAccountState(pubkey, {
  ...normalized,
  updatedAt: Date.now(),
});
```

The `rawAccountData.data` field is attacker-controlled base64. The Anchor coder decodes it against the IDL struct definition. An attacker who knows the on-chain account layout (IDL is public in `app/idl/`) can craft a valid base64 blob that decodes successfully with extreme values: `crimeBuyTaxBps=0`, `reserveA=1`, `cheapSide=1`, etc. Anchor's `decode()` performs structural deserialization only — it does not validate field ranges or business logic constraints.

### Step 5 — Protocol Store: No Validation, Immediate Broadcast

`app/lib/protocol-store.ts:53-65`

```ts
setAccountState(pubkey: string, data: AccountState): void {
  const serialized = JSON.stringify(data, bigintReplacer);
  this.accounts.set(pubkey, data);

  if (serialized === this.lastSerialized.get(pubkey)) return;
  this.lastSerialized.set(pubkey, serialized);

  sseManager.broadcast("protocol-update", { account: pubkey, data });
}
```

`AccountState` is typed as `Record<string, unknown>`. There is no field validation, no range checking, no business logic constraints. The only guard is a dedup check based on serialized string equality. Any non-identical payload immediately fans out to all connected SSE clients.

### Step 6 — SSE Route: Pass-Through to All Browsers

`app/app/api/sse/protocol/route.ts:82-92`

```ts
const unsubscribe = sseManager.subscribe((payload: string) => {
  if (!payload.startsWith("event: protocol-update\n")) return;
  controller.enqueue(encoder.encode(payload));
});
```

Zero transformation, zero filtering, zero validation. Every `protocol-update` event is forwarded verbatim to every connected `EventSource`.

### Step 7 — Browser Hooks: Trust SSE Data Unconditionally

`app/hooks/useEpochState.ts:52-59`

```ts
return {
  currentEpoch: sseData.currentEpoch as number,
  crimeBuyTaxBps: sseData.crimeBuyTaxBps as number,
  reserveA: sseData.reserveA as number,
  ...
};
```

`useEpochState`, `usePoolPrices`, `useCurveState`, and `useCarnageData` all cast SSE fields directly with `as number` / `as string`. There is no bounds checking, no cross-verification against an independent RPC call, and no anomaly detection.

## Attack Path

**Prerequisites:** Attacker obtains `HELIUS_WEBHOOK_SECRET`. Acquisition vectors:
- Railway env var compromise (ASM-2, ASM-3 unvalidated assumptions)
- Committed `.env.devnet` in git history (SEC-02 confirmed finding, devnet key)
- Traffic interception if `x-forwarded-for` is spoofable and rate-limiter bypassed
- Any compromised Railway deployment pipeline access

**Execution (once secret is obtained):**

1. Attacker identifies target PDA addresses (deterministic, derived from public program IDs and seeds documented in the codebase).

2. Attacker constructs `HeliusAccountChange` payload:
   ```json
   [{
     "account": "<EpochState PDA>",
     "accountData": [{ "account": "<EpochState PDA>", "nativeBalanceChange": 0 }],
     "rawAccountData": {
       "data": "<base64-encoded EpochState with crimeBuyTaxBps=0, fraudBuyTaxBps=0>",
       "encoding": "base64"
     }
   }]
   ```

3. POST to `/api/webhooks/helius` with `Authorization: <secret>`.

4. Payload passes `KNOWN_PROTOCOL_ACCOUNTS` check (real PDA address).

5. Anchor decodes crafted bytes → produces `{ crimeBuyTaxBps: 0, fraudBuyTaxBps: 0, ... }`.

6. `protocolStore.setAccountState()` stores and immediately broadcasts via SSE.

7. All connected browser clients receive `protocol-update` event within milliseconds.

8. `useEpochState` hook updates React state → UI displays 0% tax rates.

9. Attacker (or coordinated buyers) execute swaps before victims detect the false display. The on-chain tax rates are unchanged — victims see 0% displayed, pay actual tax, receive less output than expected.

**Same vector for pool reserves:**
- Inject `PoolState` with `reserveA=1, reserveB=1_000_000_000_000` → quote engine computes enormous expected output → users submit swaps with grossly underestimated `minimumOutput` → on-chain slippage check fails OR users accept manipulated quotes.

**Replay variant (no secret needed if captured):**
- Capture a valid historical enhanced webhook payload (no replay protection on enhanced path).
- Replay at epoch boundary to reset displayed tax state to stale epoch values.
- More limited but requires only traffic observation, not secret theft.

## Impact

| Consequence | Blocked by On-Chain? | Notes |
|---|---|---|
| Direct fund theft | YES | `minimumOutput` always enforced on-chain |
| Quote manipulation → failed TXs | PARTIAL | Users lose priority fees; tx fails but no funds lost |
| Tax rate spoofing → user deception | NO | Pure display manipulation; user makes uninformed decisions |
| Pool reserve spoofing → sandwich setup | NO | Off-chain quote uses injected reserves; attacker can set up front-run |
| Carnage fund balance spoofing | NO | Display only, no on-chain enforcement |
| SSE amplification DoS | NO | Dedup bypass if field values differ; fans out to all 5000 connected clients |
| Permanent protocol store corruption | NO | Persists until next real webhook or ws-subscriber poll cycle (up to 60s) |

The most impactful vector is **MEV enablement via reserve manipulation**: attacker injects low `reserveA=1` for a pool, causing the quote engine to compute near-infinite expected output. Users set `minimumOutput` based on this phantom quote. Their transactions fail on-chain (wasting fees), but in the window between SSE broadcast and on-chain settlement, the attacker can exploit the artificially widened price spread.

## Recommendation

### R1 — Field Validation After Anchor Decode (High Priority)

After decoding, validate that numeric fields fall within protocol-defined bounds before writing to the store. Define an `AccountValidator` per account type:

```ts
function validateEpochState(decoded: Record<string, unknown>): boolean {
  const taxFields = ["crimeBuyTaxBps", "crimeSellTaxBps", "fraudBuyTaxBps", "fraudSellTaxBps"];
  return taxFields.every(f => typeof decoded[f] === "number" && decoded[f] >= 0 && decoded[f] <= 10000);
}
function validatePoolState(decoded: Record<string, unknown>): boolean {
  return typeof decoded.reserveA === "number" && decoded.reserveA >= 0 &&
         typeof decoded.reserveB === "number" && decoded.reserveB >= 0;
}
```

Reject and log (Sentry alert) any payload that fails validation. Do not write to store.

### R2 — Slot-Based Age Check on Enhanced Webhook Path (Medium Priority)

Helius enhanced webhooks should include a slot number. Reject account change payloads whose slot is older than the current protocol store slot by more than `MAX_SLOT_DRIFT` (e.g., 300 slots ≈ 2 minutes). This mitigates the replay variant.

### R3 — Cross-Verify Critical Fields Against ws-subscriber (Medium Priority)

The ws-subscriber independently polls protocol state from Helius RPC. When an enhanced webhook delivers EpochState with tax rates that differ from the ws-subscriber's last known values by more than a threshold (e.g., >50% change in a single update), suppress the update pending re-verification from RPC. This adds a secondary validation layer that the hypothesis identifies as absent.

### R4 — Secret Rotation Support (Low Priority, Defense-in-Depth)

Implement multi-secret support (primary + rotated) with atomic swap. This reduces the blast radius of a secret compromise to the rotation window.

### R5 — SSE Anomaly Detection in Browser Hooks (Low Priority)

In `useEpochState` and `usePoolPrices`, add sanity checks before updating React state:
- Tax BPS values must be in [0, 10000]
- Pool reserves must be positive and not deviate >90% from previous value in a single update
- Log anomalies to Sentry for alerting

## Evolution

**Extends H005** (enhanced webhook replay) by completing the full injection chain analysis. H005 identified the absence of replay protection on the enhanced path. H020 confirms that this gap, combined with the lack of field validation in the store and the pass-through SSE route, creates a complete injection chain from authenticated POST to all browser clients with zero secondary validation checkpoints.

**Relationship to other findings:**
- **H005**: Replay protection missing (enhanced path) — H020 is the full-chain elaboration
- **H001**: Auth chain is correctly fail-closed — does NOT prevent this attack if secret is leaked
- **H008**: SSE connection caps are enforced — does NOT limit broadcast to existing connections
- **SEC-02** (committed devnet key): If the devnet webhook secret matches production (key reuse), this attack is immediately executable

**Verdict refinement from hypothesis:** The hypothesis stated the attack requires "no secondary validation layer." This is confirmed. However, the hypothesis framed it as CRITICAL. The actual severity is HIGH because the on-chain `minimumOutput` constraint blocks direct fund theft. The attack surface is real and exploitable for display manipulation, user deception, and MEV amplification, but the protocol's on-chain safety net prevents the most severe outcome.
