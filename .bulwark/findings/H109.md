# H109: COMMITMENT Env Var No Validation

**Severity:** Tier 3 (Low)
**Category:** Configuration Validation
**Target:** `scripts/crank/crank-provider.ts:36–37`
**Status:** ACCEPTED_RISK

---

## Finding

Line 36–37 in `crank-provider.ts`:

```typescript
const commitment =
  (process.env.COMMITMENT as anchor.web3.Commitment) || "confirmed";
```

The `COMMITMENT` environment variable is cast directly to the `Commitment` type without runtime validation. Valid values are:

```
'processed' | 'confirmed' | 'finalized' | 'recent' | 'single' | 'singleGossip' | 'root' | 'max'
```

If an invalid value is passed (e.g., `COMMITMENT=invalid`), TypeScript's type assertion `as` allows it through. The Connection constructor will then silently receive an invalid commitment level, potentially causing unexpected RPC behavior.

---

## Impact

- **Invalid commitment values** may be silently ignored by the RPC client or cause opaque errors downstream.
- **No early detection** — the error surface only at RPC call time, not at provider initialization.
- **Operational confusion** — developers may set `COMMITMENT=finalized_typo` and wonder why behavior differs from expectations.

---

## Recommended Fix

Add runtime validation:

```typescript
const commitmentRaw = process.env.COMMITMENT || "confirmed";
const validCommitments = [
  "processed",
  "confirmed",
  "finalized",
  "recent",
  "single",
  "singleGossip",
  "root",
  "max",
];

if (!validCommitments.includes(commitmentRaw)) {
  throw new Error(
    `Invalid COMMITMENT env var: "${commitmentRaw}". ` +
      `Valid values: ${validCommitments.join(", ")}`
  );
}

const commitment = commitmentRaw as anchor.web3.Commitment;
```

---

## Verdict: ACCEPTED_RISK

**Rationale:**

1. **COMMITMENT is internal infrastructure config**, not user-facing. Only DevOps/developers set this env var.
2. **Default is safe** — defaults to `"confirmed"`, the standard commitment level.
3. **Railway/devnet usage** — On production deployment, the commitment level is tightly controlled and explicitly set.
4. **Low likelihood** — typos in env vars are caught quickly during operational tests.

**Mitigation already in place:**
- Developers running the crank locally will see TypeScript type errors if they pass a string literal that isn't a valid commitment.
- Railway environment is managed by the team and commits are code-reviewed.

**Accept but document:** Add a comment flagging that COMMITMENT should only be set to valid values. Consider adding validation in a future hardening phase if operational issues arise.
