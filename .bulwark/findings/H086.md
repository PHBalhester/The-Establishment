## Verdict: CONFIRMED (same as H034)

The concurrent `executeSwap` / `executeRoute` double-click vulnerability identified in H034 remains **UNRESOLVED**. Both function bodies lack in-function status guards and rely entirely on the UI layer (BigRedButton click handler) for double-submit prevention. H034 was marked as PARTIALLY_FIXED because BuyForm and SellForm were hardened, but the core AMM swap hook remains unguarded.

---

## Evidence

### 1. executeSwap (lines 690-698)

```ts
const executeSwap = useCallback(async () => {
  if (!wallet.publicKey || !wallet.connected || !quote) {
    return;
  }

  const connection = getConnection();
  const poolConfig = resolvePool(inputToken, outputToken);

  try {
    setStatus("building");
```

**Finding**: No `if (status !== "idle") return` guard. A second call while the first is in-flight will proceed past the wallet/quote checks and immediately set `setStatus("building")`, initiating a second transaction build and sign.

---

### 2. executeRoute (lines 839-860)

```ts
const executeRoute = useCallback(async () => {
  // If smart routing is OFF or no route selected, fallback to direct swap
  if (!smartRouting || !routesResult.selectedRoute) {
    return executeSwap();
  }

  const route = routesResult.selectedRoute;

  // Single-hop non-split route: use existing direct swap path
  if (route.hops === 1 && !route.isSplit) {
    return executeSwap();
  }

  // Multi-hop or split: atomic v0 transaction (single signature, all-or-nothing)
  if (!wallet.publicKey || !wallet.connected) return;

  const connection = getConnection();

  try {
    setStatus("building");
```

**Finding**: Same absence. Line 858 (`setStatus("building")`) is unguarded. Double-click or programmatic dual invocation will trigger two transaction builds.

---

### 3. Attack Path: Double-Click via BigRedButton

While the BigRedButton's `handleClick` (lines 67-73 in BigRedButton.tsx) does protect with `if (status === 'idle')`, React concurrent rendering or timing skew can create stale closures. Additionally:

- The `executeRoute` reference is passed as a prop to child components (SwapStation, MultiHopStatus).
- MultiHopStatus's `onRetry` handler (SwapStation.tsx:281) uses `setTimeout(() => swap.executeRoute(), 0)`, creating a 0ms race window after `resetForm()`.
- Any external consumer holding `useSwap`'s returned `executeSwap` or `executeRoute` can invoke directly without the BigRedButton gate.

---

### 4. H034's Partial Fix Did Not Address This

H034 correctly identified the vulnerability and fixed:
- ✅ BuyForm and SellForm (form-level `submitting` boolean)
- ✅ BigRedButton click handler guard
- ❌ **`executeSwap` in-function guard** (not added)
- ❌ **`executeRoute` in-function guard** (not added)

The functions are still **self-unprotected**. H086 confirms this re-finding.

---

## Root Cause

The original H034 fix addressed only the UI layer. The hook's execution functions (which are the actual entry points to transaction building) were not hardened for idempotence. They assume all callers are well-behaved; concurrent or programmatic invocations bypass the UI guards entirely.

---

## Fix Required

Add status guards to both functions (insert before the `try` block):

**`app/hooks/useSwap.ts` — `executeSwap`** (after wallet/quote guard, line 693):
```ts
if (status !== "idle") return;
```

**`app/hooks/useSwap.ts` — `executeRoute`** (after the multi-hop fallbacks, before multi-hop setup at line 853):
```ts
if (status !== "idle") return;
```

**Dependency array**: `status` must be added to the `useCallback` dependency array for both functions.

**SwapStation.tsx retry pattern** (line 281):
```ts
onRetry={() => {
  swap.resetForm();  // synchronous, sets status='idle'
  swap.executeRoute();  // direct call, no setTimeout gap
}}
```
