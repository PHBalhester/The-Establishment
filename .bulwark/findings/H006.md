# H006: NEXT_PUBLIC_RPC_URL Mainnet API Key in Client Bundle

## Verdict: POTENTIAL

---

## Evidence

### Primary Target Files Examined

**`app/lib/connection.ts` (lines 30-46) — Browser path completely bypasses NEXT_PUBLIC_RPC_URL:**
```ts
function resolveRpcUrl(override?: string): string {
  if (override) return override;

  // Browser: all RPC goes through the proxy to protect the API key.
  if (typeof window !== "undefined") {
    return `${window.location.origin}/api/rpc`;   // line 36 — never reads env var
  }

  // Server: use HELIUS_RPC_URL (preferred) or NEXT_PUBLIC_RPC_URL (fallback).
  const serverUrl = process.env.HELIUS_RPC_URL ?? process.env.NEXT_PUBLIC_RPC_URL;
  ...
}
```
The browser branch returns the proxy URL directly at line 36. `NEXT_PUBLIC_RPC_URL` is only reached on the non-browser (server-side) branch at line 41.

**`app/app/api/rpc/route.ts` (lines 128-132) — Server-only API route, not client-bundled:**
```ts
const endpoints = [
  process.env.HELIUS_RPC_URL,
  process.env.HELIUS_RPC_URL_FALLBACK,
  process.env.NEXT_PUBLIC_RPC_URL,          // line 131 — third fallback
].filter(Boolean) as string[];
```
This file is an API route (`export async function POST`). Next.js API routes execute on the server only and are never included in the client JS bundle.

**`app/providers/providers.tsx` (lines 37-39) — Client component, does NOT use NEXT_PUBLIC_RPC_URL:**
```ts
const endpoint = typeof window !== "undefined"
  ? `${window.location.origin}/api/rpc`
  : "http://localhost:3000/api/rpc";
```
The `providers.tsx` file is a "use client" component and directly constructs the proxy URL. It contains no reference to `process.env.NEXT_PUBLIC_RPC_URL`.

**`app/.env.mainnet` (line 49) — Template still includes NEXT_PUBLIC_RPC_URL with API key placeholder:**
```
# RPC URL for browser-side Solana connections (legacy, now proxied through /api/rpc).
# Still referenced by some wallet adapter config. Use mainnet Helius URL.
NEXT_PUBLIC_RPC_URL=https://mainnet.helius-rpc.com/?api-key=CHANGE_ME_MAINNET
```
The template comment says "legacy, now proxied through /api/rpc" yet still instructs setting the value with the mainnet Helius API key.

**`app/.env.local` (line 1) — Current devnet env file confirms pattern:**
```
NEXT_PUBLIC_RPC_URL=https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]
```
The devnet environment follows the template exactly — Helius key is present in `NEXT_PUBLIC_RPC_URL`.

### Next.js Bundle Inlining Mechanics

Next.js inlines `NEXT_PUBLIC_*` environment variables at build time via static replacement of `process.env.NEXT_PUBLIC_*` expressions. **This inlining only occurs in files that are compiled into the client bundle** (client components, pages, shared modules imported by client code). API route files (`app/app/api/**`) are server-only modules — they are not bundled into client-side JavaScript.

A search across all `app/` source files confirms that `process.env.NEXT_PUBLIC_RPC_URL` appears in exactly two places:
1. `app/lib/connection.ts:41` — inside the `else` branch that only executes when `typeof window === "undefined"` (server context)
2. `app/app/api/rpc/route.ts:131` — inside an API route POST handler (server-only)

Neither location causes the value to be inlined into client-side JavaScript bundles under the current codebase.

### Why the Verdict is POTENTIAL, Not NOT_VULNERABLE

Three factors prevent a clean NOT_VULNERABLE verdict:

1. **The `.env.mainnet` template actively instructs setting `NEXT_PUBLIC_RPC_URL`** with the mainnet Helius API key and describes it as used for "wallet adapter config." This comment is misleading — the wallet adapter (`providers.tsx`) does NOT use this variable. A developer following the template will set a `NEXT_PUBLIC_*` env var containing the live API key, even though no current code path exposes it client-side.

2. **Next.js inlines ALL `NEXT_PUBLIC_*` variables that appear in client-bundled code.** If any future developer adds a single reference to `process.env.NEXT_PUBLIC_RPC_URL` in a client component (for example, as a debugging aid or during a wallet adapter upgrade), the API key immediately becomes visible in the production bundle. The variable's presence in the build environment creates a latent exposure risk.

3. **The variable IS already set in the build environment** (Railway reads env vars at build time). Even though no current client code reads it, the value may still be visible in the `__NEXT_DATA__` JSON blob embedded in the HTML or in build artifacts depending on Next.js version and configuration. This has not been confirmed but cannot be fully excluded without a live production build inspection.

4. **`connection.ts` comment says "NEXT_PUBLIC_RPC_URL is no longer used"** (line 12-13) yet the mainnet template still instructs setting it. This documentation contradiction increases the probability a future operator re-introduces a client-side use.

## Attack Path

**If a developer sets `NEXT_PUBLIC_RPC_URL` in Railway AND any client-side code ever references it:**

1. Next.js build compiles client bundle with the value of `NEXT_PUBLIC_RPC_URL` inlined as a string literal
2. Attacker loads the production URL and views JavaScript source in browser DevTools
3. Search for `helius-rpc.com` or `api-key=` in the bundle
4. Extract the Helius API key from the inlined string
5. Call Helius RPC API directly: `curl -X POST https://mainnet.helius-rpc.com/?api-key=<extracted_key> -d '{"jsonrpc":"2.0","method":"getSlot","id":1}'`
6. This bypasses the `/api/rpc` proxy entirely — no method allowlist, no rate limiting, no credit tracking
7. Use the key for: credit exhaustion via `getProgramAccounts` spam, arbitrary `sendTransaction` relay (botnet TX spam via project's paid key), webhook management (register shadow webhooks, delete production webhook)

**Current state (no client-side reference):** The attack path above is blocked by the fact that no client component currently imports `process.env.NEXT_PUBLIC_RPC_URL`. The API key in `NEXT_PUBLIC_RPC_URL` is not inlined into the client bundle.

---

## Impact

**If exploited (upon regression or future change):** HIGH

- Helius credit exhaustion (paid plan credits burned by attacker)
- Arbitrary Solana transaction relay through project's endpoint (botnet use)
- Complete bypass of the RPC method allowlist (17-method restriction)
- Complete bypass of the per-IP rate limiter (300 req/min)
- Webhook management: register passive surveillance webhooks or delete production webhook to blind the data pipeline
- Key rotation requires Railway redeploy (downtime)

**Current state:** LOW — the risk is latent, not active. No current code path exposes the key to the browser.

## Recommendation

**Fix the root cause: remove `NEXT_PUBLIC_RPC_URL` from the mainnet template entirely.**

The variable serves no purpose in the current architecture. The browser uses `/api/rpc` (hardcoded in `providers.tsx` and `connection.ts`). The server uses `HELIUS_RPC_URL`. There is no legitimate reason for `NEXT_PUBLIC_RPC_URL` to hold a Helius API key.

1. **Remove line 49 from `app/.env.mainnet`** (the `NEXT_PUBLIC_RPC_URL=...` entry)
2. **Remove the fallback at `rpc/route.ts:131`**: `process.env.NEXT_PUBLIC_RPC_URL` as the third endpoint in the proxy's failover chain. Use `HELIUS_RPC_URL_FALLBACK` only. There is no scenario where the browser-public variable should be used as a server-side RPC endpoint.
3. **Remove the fallback at `connection.ts:41`**: The `?? process.env.NEXT_PUBLIC_RPC_URL` on the server branch should be removed. The server should fail fast on missing `HELIUS_RPC_URL` rather than fall back to a variable that may contain an API key destined for the public env.
4. **Update the comment on `connection.ts:39`** to remove the reference to `NEXT_PUBLIC_RPC_URL` entirely.

These changes eliminate the variable from the codebase, preventing any future regression.

**Secondary fix:** Update `.env.local` (devnet) to use `HELIUS_RPC_URL` instead of `NEXT_PUBLIC_RPC_URL` so local development follows the same pattern as production. This file is not git-tracked but the pattern matters for developer habit formation.

## Evolution

This finding is a RECHECK of H002 (Helius API Key in Client Bundle, FIXED in Audit #1) and H009 (Devnet Fallback in Production, FIXED in Audit #1).

**H002 fix status (Audit #1):** The original H002 fix removed the direct Helius URL from `shared/constants.ts` and introduced the `/api/rpc` proxy pattern. The fix is **intact** — no client code references `process.env.NEXT_PUBLIC_RPC_URL`. The proxy correctly prevents key exposure in the current build.

**H009 fix status (Audit #1):** The cross-cluster devnet fallback was removed from `connection.ts`. The fix is **intact** — server-side RPC requires `HELIUS_RPC_URL` and no cross-cluster default exists.

**New surface introduced since Audit #1:** The `.env.mainnet` template (`app/.env.mainnet`) was added as part of the Railway deployment preparation. Its line 49 creates a latent risk by instructing operators to set a `NEXT_PUBLIC_*` variable containing the live API key. The `rpc/route.ts` also added `NEXT_PUBLIC_RPC_URL` as a third failover endpoint (likely as a development convenience) — this server-side-only use is safe but reinforces the false belief that this variable is needed.

**Audit #2 delta context:** `app/lib/connection.ts` was modified in this audit cycle (RECHECK flag). The current version shows the migration note at line 12 ("NEXT_PUBLIC_RPC_URL is no longer used") and the safe browser branch, confirming the fix from H002 has survived the refactor. However, the mainnet template and the two server-side fallback references remain as technical debt that creates deployment risk.
