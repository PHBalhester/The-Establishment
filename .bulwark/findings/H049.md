# H049: Polling Fallback Produces Incompatible Data Shape

**Tier:** 2
**Origin:** Novel (confirmed by CHAIN-04 context auditor finding #7)
**Verdict:** CONFIRMED

---

## Hypothesis

When SSE has been down for >30 seconds, `useProtocolState` activates a polling fallback
via `pollViaRpc()`. The hypothesis is that this fallback stores raw account metadata
(`{lamports, owner, dataLength}`) instead of Anchor-decoded fields, causing all downstream
hooks that type-guard on decoded field names to receive null or zero state.

---

## Evidence

### The Vulnerable Code — `pollViaRpc` (lines 196-212)

```typescript
setAccounts((prev) => {
  const next = { ...prev };
  for (let i = 0; i < ALL_MONITORED_PUBKEYS.length; i++) {
    const info = infos[i];
    if (info) {
      const pubkey = ALL_MONITORED_PUBKEYS[i].toBase58();
      next[pubkey] = {
        label: `rpc-poll`,
        lamports: info.lamports,
        owner: info.owner.toBase58(),
        dataLength: info.data.length,
        updatedAt: Date.now(),
      };
    }
  }
  return next;
});
```

`getMultipleAccountsInfo` returns raw `AccountInfo<Buffer>`. The code stores only the
metadata envelope (`lamports`, `owner`, `dataLength`) and discards `info.data` entirely.
No Anchor decoding is performed. The data buffer that contains the actual account state
(reserves, epoch fields, staking balances, etc.) is thrown away.

Compare to the SSE path, which delivers Anchor-decoded fields like `currentEpoch`,
`reserveA`, `totalStaked`, `totalTriggers`, etc., as plain numbers and bigints via
`anchorToJson()` + `bigintReviver`.

---

### Downstream Hook Impact — Confirmed Type Guard Failures

**`useEpochState` (`useEpochState.ts:50`)**
```typescript
if (!sseData || typeof sseData.currentEpoch !== "number") return null;
```
Polling data has no `currentEpoch` field. Guard fails. Returns `null`. Hook reports
`loading: true` indefinitely. Tax rates show as unavailable. Swap quote engine cannot
compute fees.

**`usePoolPrices` (`usePoolPrices.ts:71`)**
```typescript
if (data && typeof data.reserveA === "number") {
  // decoded path
} else {
  // fallback: reserveA: 0, reserveB: 0
}
```
Polling data has no `reserveA`. Falls to the else branch. Both pools return
`reserveA: 0, reserveB: 0`. Price calculations that divide by reserves produce
division-by-zero or return `Infinity`/`NaN`. Swap quotes display as zero or break.

**`useStaking` (`useStaking.ts:244`)**
```typescript
if (!stakePoolSse || typeof stakePoolSse.totalStaked !== "number") return null;
```
Polling data has no `totalStaked`. `stakePoolRaw` becomes `null`. All reward calculations
return zero. Users cannot see their staking positions or pending rewards.

**`useCarnageData` (`useCarnageData.ts:44`)**
```typescript
if (!carnageFund || typeof carnageFund.totalTriggers !== "number") return null;
```
Polling data has no `totalTriggers`. Returns `null` for the entire Carnage panel.
**Partial exception**: `carnageSolVault.lamports` is incidentally correct because
`pollViaRpc` does store `lamports` — so vault balance would display correctly if
`carnageFund` guard passed (which it does not, so the whole object is null anyway).

---

## Failure Mode Classification

The bugs are **silent wrong values or indefinite loading**, not crashes. Downstream
hooks do not throw; they return `null` or zeroes through type guards. The result is:

- During SSE outage >30s, the UI enters a degraded state showing empty/loading spinners
  for epoch, pool prices, and staking data.
- The polling fallback, intended to keep data "somewhat fresh", instead actively makes
  the state _worse_ than simply leaving the last-known SSE values in place, because it
  overwrites valid stale SSE data with structurally incompatible polling data.
- A user attempting to swap during an SSE outage would see zero pool reserves and
  potentially compute a swap with `minimumOutput = 0`. The on-chain program's
  `minimumOutput` enforcement is the only backstop preventing financial loss in this path.

---

## Root Cause

`pollViaRpc` was implemented to update `lamports` for the `carnageSolVault` SOL balance,
which is the one field that _is_ available without decoding. The function was generalised
to cover all `ALL_MONITORED_PUBKEYS` without performing the Anchor decode step that the
SSE path relies on. The correct fix is to Anchor-decode `info.data` for each account
(using the appropriate IDL coder per account type) before storing into the accounts map,
or to restrict polling to only the accounts whose useful state can be derived from raw
metadata alone (i.e. `carnageSolVault` lamports only) and leave others unchanged.

## Severity Rationale

**Tier 2 (High).**  Financial functionality (swap quotes, staking display) is silently
broken during SSE outages lasting >30 seconds. The polling fallback is intended as a
reliability improvement but degrades UX below the baseline of simply displaying stale
data. The on-chain `minimumOutput` guard prevents direct fund loss, but a user seeing
zero reserves and zero fees has no indication the data is bad and may interpret the UI
state incorrectly. Outage scenarios (Railway restart, SSE route crash, network partition)
are realistic production conditions.
