# H028 — Rate Limiting Regression (RECHECK H024)

**Tier:** 2 — HIGH
**Origin:** RECHECK of H024
**Status:** CONFIRMED (partial regression — 5 of 8 routes remain unprotected)
**Verdict:** NOT_FIXED (H024 closed the wrong scope; the original finding H024 covered rpc/webhook/sol-price only. Three high-traffic DB-querying endpoints and the health endpoint remain unprotected. No new routes added, but the gaps are confirmed present in current source.)

---

## Summary

H024 applied `checkRateLimit` to three API routes: `/api/rpc`, `/api/webhooks/helius`, and `/api/sol-price`. This was the H024 fix scope. However, 5 of the 8 total API routes have never had rate limiting applied and remain unprotected in the current codebase. The gap is not a regression in the sense of something that was working and broke — it is an incomplete fix from H024 that was never widened to cover all routes. The three most dangerous unprotected routes are `/api/candles` (unbounded gap-fill amplification), `/api/carnage-events` (DB pool exhaustion), and `/api/health` (live Postgres + RPC calls per request plus internal telemetry disclosure). The two SSE routes (`/api/sse/protocol`, `/api/sse/candles`) use connection-level caps instead of request-level rate limits, which is appropriate for their long-lived streaming nature and is not a gap.

## Evidence — Unprotected Routes (Source)

### `/api/candles` (`app/app/api/candles/route.ts`)

The GET handler at line 186 has zero rate limiting. It accepts user-controlled `from`, `to`, `resolution`, and `gapfill` parameters. The gap-fill function (lines 94–167) iterates from `alignedStart` to `rangeEnd` in steps of the resolution bucket size. With `resolution=1m` and a one-year range, this generates ~525,600 synthetic candle objects in memory — entirely outside the 2000-row Drizzle `.limit()` cap, which only bounds the DB result set. The gap-fill loop operates on the URL-supplied `from`/`to` values, not the DB row count.

**No import of `checkRateLimit` in `app/app/api/candles/route.ts`.** Confirmed at line 30–34 (imports are only `NextRequest`, `NextResponse`, `captureException`, `db`, `candles`, `and`, `eq`, `gte`, `lte`, `asc`).

### `/api/carnage-events` (`app/app/api/carnage-events/route.ts`)

The GET handler at line 32 executes a Postgres query on every call with no rate limiting. The query is hardcoded `LIMIT 5` (safe from amplification), but with no request cap an attacker can saturate the 10-connection Postgres pool. `carnage-events/route.ts` imports only `NextResponse`, `captureException`, `db`, `carnageEvents`, `desc`. No `checkRateLimit`.

### `/api/health` (`app/app/api/health/route.ts`)

The GET handler at line 32 executes `SELECT 1` on Postgres on every call. If the protocol store has no cached slot, it additionally calls `connection.getSlot()` (an RPC call). The response body includes `wsSubscriber` internal status (initialized, wsConnected, latestSlot, lastSlotReceivedAt, fallbackActive) and per-method RPC credit counts from `creditCounter.getStats()`. No `checkRateLimit` import or call present.

## Attack Path

### Vector 1: Gap-fill Candle Memory Exhaustion (HIGH)

```
Attacker sends:
  GET /api/candles?pool=<any-valid-pool>&resolution=1m&from=0&to=<current-unix>

Server processing:
  1. DB query executes, returns ≤2000 rows (bounded)
  2. gapFillCandles(rows, "1m", "0", "<now>") called
  3. Loop: from epoch (0) to now (~1.7B seconds / 60 = ~29M iterations)
     Even with a 1-year range: 525,600 iterations
  4. Each iteration allocates a ~100-byte object
  5. JSON.stringify on the array before sending (duplicates allocation)

Impact at 10 concurrent requests with 1-year range:
  ~525,600 objects × 100 bytes × 2 (alloc + serialization) × 10 requests
  = ~1 GB memory allocation
  Next.js process OOM-kills or stalls, Railway restarts container
  All in-memory state lost: rate limiter, SSE subscribers, protocol store
```

No authentication or rate limiting prevents this. Any browser can issue the request. The DB row limit does not protect the gap-fill path.

### Vector 2: DB Pool Exhaustion via Unprotected Endpoints (MEDIUM)

```
Attacker opens 300 concurrent connections and sends:
  GET /api/carnage-events (or /api/candles or /api/health)

Postgres pool is limited to max: 10 (app/db/connection.ts)
No connectionTimeoutMillis configured (queries can block forever)

With 300 concurrent requests hitting carnage-events:
  - 10 grab connections from pool
  - 290 block waiting for a connection
  - Webhook handler cannot acquire DB connection → fails to insert swap events
  - Candle upserts fail → chart data stale
  - Health endpoint itself becomes the weapon and the victim simultaneously
```

### Vector 3: Health Endpoint as Infrastructure Oracle + DoS Weapon (MEDIUM)

```
Attacker queries /api/health continuously at high rate:
  1. Each request runs SELECT 1 on Postgres (pool connection consumed)
  2. If __slot not in protocol store, also calls getSlot() via /api/rpc
     (falls back to direct RPC connection, burning Helius credits)
  3. Response leaks: wsConnected, latestSlot, lastSlotReceivedAt, fallbackActive,
     totalRpcCalls, per-method call counts
  4. Attacker uses latestSlot timing to detect when ws-subscriber is degraded
     (fallbackActive=true) → optimal window for webhook replay or SSE poisoning
```

### Vector 4: IP Spoofing Amplifies All Vectors (compounding)

All three rate-limited routes trust `x-forwarded-for` first hop without Railway proxy validation. An attacker who sets a random `x-forwarded-for` IP per request bypasses the per-IP limits on the 3 protected routes AND gets fresh buckets on every request. Combined with the 5 unprotected routes, this makes the overall rate limiting posture effectively unenforced for a determined attacker.

## Recommendation

### R1: Add rate limiting to `/api/candles` (CRITICAL path — gap-fill amplification)

```typescript
// In app/app/api/candles/route.ts GET handler, before parameter extraction:
import { checkRateLimit, getClientIp } from "@/lib/rate-limit";

const CANDLES_RATE_LIMIT: RateLimitConfig = { windowMs: 60_000, maxRequests: 60 };

export async function GET(req: NextRequest): Promise<NextResponse> {
  const clientIp = getClientIp(req);
  const rateCheck = checkRateLimit(clientIp, CANDLES_RATE_LIMIT, "candles");
  if (!rateCheck.allowed) {
    return NextResponse.json({ error: "Too Many Requests" }, {
      status: 429,
      headers: { "Retry-After": String(rateCheck.retryAfter) },
    });
  }
  // ... existing parameter extraction
}
```

Additionally, cap the gap-fill output independently of `from`/`to` range:
```typescript
// In gapFillCandles, after computing alignedStart and rangeEnd:
const MAX_GAP_FILL_SLOTS = 10_000;
const totalSlots = Math.ceil((rangeEnd - alignedStart) / step);
if (totalSlots > MAX_GAP_FILL_SLOTS) {
  // Trim rangeEnd to prevent amplification
  rangeEnd = alignedStart + MAX_GAP_FILL_SLOTS * step;
}
```

### R2: Add rate limiting to `/api/carnage-events`

```typescript
// In app/app/api/carnage-events/route.ts:
import { NextRequest } from 'next/server';
import { checkRateLimit, getClientIp } from '@/lib/rate-limit';

const CARNAGE_EVENTS_RATE_LIMIT: RateLimitConfig = { windowMs: 60_000, maxRequests: 60 };

export async function GET(req: NextRequest): Promise<NextResponse> {
  const clientIp = getClientIp(req);
  const rateCheck = checkRateLimit(clientIp, CARNAGE_EVENTS_RATE_LIMIT, "carnage-events");
  if (!rateCheck.allowed) {
    return NextResponse.json({ error: "Too Many Requests" }, {
      status: 429,
      headers: { "Retry-After": String(rateCheck.retryAfter) },
    });
  }
  // ... existing DB query
}
```

Note: The function signature must change from `GET(): Promise<NextResponse>` to `GET(req: NextRequest): Promise<NextResponse>` to access the request object.

### R3: Add rate limiting to `/api/health` and strip telemetry from public response

```typescript
// In app/app/api/health/route.ts:
const HEALTH_RATE_LIMIT: RateLimitConfig = { windowMs: 60_000, maxRequests: 30 };

export async function GET(req: NextRequest) {
  const clientIp = getClientIp(req);
  const rateCheck = checkRateLimit(clientIp, HEALTH_RATE_LIMIT, "health");
  if (!rateCheck.allowed) {
    return new Response("Too Many Requests", { status: 429 });
  }
  // ...
  // Strip internal telemetry from public response:
  return NextResponse.json({
    status,
    checks: { postgres, solanaRpc },
    timestamp: new Date().toISOString(),
    // Remove: wsSubscriber, credits (internal operational data)
  });
}
```

### R4: Add gap-fill range validation at parameter-parse time

Reject `from`/`to` combinations that would produce more than `MAX_GAP_FILL_SLOTS` iterations before executing the DB query:
```typescript
if (fromParam && toParam && gapfill) {
  const rangeMs = parseTimestamp(toParam) - parseTimestamp(fromParam);
  const stepMs = RESOLUTION_SECONDS[resolution]! * 1000;
  if (rangeMs / stepMs > MAX_GAP_FILL_SLOTS) {
    return NextResponse.json({ error: "Range too large for gap-fill at this resolution" }, { status: 400 });
  }
}
```
