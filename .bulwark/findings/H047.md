# Finding: H047 — No Rate Limits on DB Endpoints (Pool Exhaustion)

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

Three API routes — `/api/candles`, `/api/carnage-events`, and `/api/health` — execute PostgreSQL queries on every request with zero rate limiting. The database connection pool is capped at 10 connections (`app/db/connection.ts:57`). Because postgres.js holds a connection for the full duration of a query and has no `connectionTimeoutMillis` configured, an attacker sending concurrent requests can exhaust all 10 connections in a single HTTP volley. Once the pool is saturated, every other DB-dependent subsystem (webhook handler, candle upserts, event storage) blocks indefinitely waiting for a free connection, producing a full data-pipeline DoS from a trivially cheap attack.

This finding extends H028 (health endpoint information disclosure, Audit #1) by quantifying the pool exhaustion vector that was flagged as an incidental discovery there, and by confirming the same gap exists across all three unprotected DB routes.

---

## Hypothesis Investigated

`/api/candles`, `/api/carnage-events`, and `/api/health` have no `checkRateLimit` call. Rapid concurrent requests exhaust the PostgreSQL connection pool (max 10), blocking all DB-dependent routes.

### Step 2: Confirm pool configuration and absence of timeout

**`app/db/connection.ts:57`**

```typescript
const client =
  globalForDb.pgClient ?? postgres(connectionString, { max: 10, ...sslConfig });
```

Pool ceiling: **10 connections**. The postgres.js client is initialised with only `max` and optional `ssl`. The following safety parameters are absent:

| Parameter | Status | Consequence |
|-----------|--------|-------------|
| `connectionTimeoutMillis` | Not set | Acquiring a new connection blocks forever when pool is full |
| `idleTimeoutMillis` | Not set | Idle connections are never recycled |
| `max_lifetime` / `maxUses` | Not set | Connections never rotated |
| Statement-level timeout | Not set | Slow queries hold connections indefinitely |

This is corroborated by the ERR-03 context analysis (L-ERR03-10) and matches what was observed in Audit #1.

### Step 3: Pool exhaustion rate calculation

Each of the three routes holds a connection for the duration of its DB query:

- `/api/carnage-events`: `SELECT ... LIMIT 5 ORDER BY timestamp DESC` — indexed timestamp column, typically **2–10 ms** at low load.
- `/api/health`: `SELECT 1` — trivial, typically **1–3 ms**.
- `/api/candles`: `SELECT ... WHERE pool=? AND resolution=? ... LIMIT 2000` — depends on index selectivity, typically **5–50 ms** at low load. Can climb to 100–500 ms under concurrent load.

With a pool ceiling of 10 and no `connectionTimeoutMillis`, the exhaustion condition is:

```
concurrent_requests_needed = pool_size / (query_duration_ms / request_processing_ms)
```

Simplified: if 10 requests each holding a connection arrive before the first one finishes, the pool is full. For the health endpoint at ~2 ms per query, an attacker needs **10 concurrent requests arriving within the same ~2 ms window** to guarantee exhaustion. At 100 concurrent requests per second (trivially achievable with any HTTP load tool), steady-state pool saturation is effectively guaranteed: new requests queue behind the pool, and with no `connectionTimeoutMillis`, those queued requests sit indefinitely, consuming memory and Node.js worker capacity.

**Worst-case attack: candles endpoint with slow query**

A crafted candle request with a wide `from`/`to` range forces a large scan even with the 2000-row `LIMIT`, because the WHERE clause must scan the index range before applying LIMIT. Under sustained concurrent load, each query holds its connection for longer, making it easier to saturate all 10 slots before any connection is released.

**Combined attack across all three routes:**

An attacker does not need to hammer a single endpoint. Distributing 10+ concurrent requests across `/api/candles` (6), `/api/carnage-events` (2), and `/api/health` (2) achieves the same result while reducing the per-endpoint request rate — making simple rate limits on a single endpoint insufficient as the sole mitigation.

---

### Step 4: Impact on other subsystems once pool is exhausted

The postgres.js connection pool is a globalThis singleton shared across the entire Next.js process. All DB consumers compete for the same 10 connections:

| Subsystem | DB Usage | Impact when pool full |
|-----------|----------|-----------------------|
| Webhook handler | `swapEvents` insert, `candles` upsert | Inserts queue indefinitely — swap events lost or delayed |
| Candle API | SELECT | Hangs, user sees no chart data |
| Carnage events API | SELECT | Hangs |
| Health endpoint | `SELECT 1` | Hangs, Railway liveness check gets no response (→ container restart if timeout exceeded) |

The Railway liveness check consequence is notable: if `/api/health` cannot complete a `SELECT 1` because the pool is exhausted, Railway's health check will eventually time out. Depending on Railway's restart policy, this could trigger a container restart, which resets all in-memory state (rate limiter, SSE subscriber map, protocol store) — creating additional disruption.

---

### Step 5: Cross-reference with H028

H028 (Audit #1) identified `/api/health` as having no rate limiting and noted as an incidental discovery: "Each health check call triggers a Postgres query... without rate limiting, an attacker could poll the health endpoint at high frequency." The current health route has expanded since Audit #1 — it now also exposes `wsSubscriber` internal status and `credits` per-method RPC breakdown (lines 63–64, `health/route.ts`), making it more resource-intensive (these involve Map reads, not DB calls) and providing richer reconnaissance data. The rate limiting gap has not been addressed.

## Recommended Fix

**Primary: Add rate limiting to all three routes.** The existing `checkRateLimit` infrastructure in `app/lib/rate-limit.ts` supports custom profiles. Add a `DB_READ_RATE_LIMIT` profile (e.g., 60 req/min per IP) and call `checkRateLimit` at the top of each handler, before any DB call:

```typescript
// app/app/api/carnage-events/route.ts
import { checkRateLimit, DB_READ_RATE_LIMIT } from '@/lib/rate-limit';
import { getClientIp } from '@/lib/rate-limit';

export async function GET(req: NextRequest): Promise<NextResponse> {
  const ip = getClientIp(req);
  const rl = checkRateLimit(ip, 'carnage-events', DB_READ_RATE_LIMIT);
  if (!rl.allowed) {
    return NextResponse.json({ error: 'Too many requests' }, {
      status: 429,
      headers: { 'Retry-After': String(rl.retryAfter) },
    });
  }
  // ... existing DB query ...
}
```

Apply the same pattern to `/api/candles` and `/api/health`.

**Secondary: Configure postgres.js connection timeout.** Add `connectionTimeoutMillis: 5000` to the postgres.js client options in `app/db/connection.ts:57`. This causes queued connection acquirers to reject with an error rather than blocking indefinitely, keeping the request cycle bounded:

```typescript
const client = globalForDb.pgClient ?? postgres(connectionString, {
  max: 10,
  connection_timeout: 5,   // postgres.js uses seconds for this option
  idle_timeout: 20,
  ...sslConfig,
});
```

**Tertiary (defence-in-depth):** Consider a global concurrent-connection guard (semaphore) at the DB layer with a max of 8 (leaving 2 for internal use), so that even if rate limiting is bypassed via IP spoofing, the DB cannot be fully exhausted.
