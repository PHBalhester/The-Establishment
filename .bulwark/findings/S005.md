# S005: Batch Amplification + IP Spoofing → Unlimited Helius Credit Burn

## Verdict: CONFIRMED

---

## Constituent Findings

| Finding | Verdict | Summary |
|---------|---------|---------|
| H008 | CONFIRMED | RPC proxy batch endpoint accepts unlimited-size JSON-RPC batch arrays; rate limiter counts HTTP requests, not RPC method calls; 500 calls per request × 300 requests/min = 150,000 Helius credits/min per IP |
| H015 | CONFIRMED | `getClientIp()` takes `x-forwarded-for[0]` verbatim; attacker cycles any IP string per request to obtain a fresh rate-limit bucket; all three protected endpoints affected |

---

## Combined Attack Path

H008 establishes the batch amplification multiplier: one HTTP request can carry an unbounded number of RPC method calls, each consuming Helius credits, while the rate limiter charges only one token against the per-IP bucket. H015 establishes that the per-IP bucket itself is trivially bypassed by injecting a fresh IP value in `X-Forwarded-For` on each request.

These two vulnerabilities compose multiplicatively, not additively. H008 without H015 is bounded by 300 HTTP requests/min/IP (still destructive at 150,000 credits/min, but rate-limited per real IP). H015 without H008 is an availability degradation bounded by the RPC method cost per request (not an amplifier by itself). Together, both limits are simultaneously eliminated:

```
1. Attacker opens a loop with no IP constraint:
   for i in $(seq 1 $UNLIMITED):
     ip = random_ipv4()                # any string passes getClientIp() verbatim
     batch = [{"jsonrpc":"2.0","id":i,"method":"getAccountInfo","params":["<pubkey>",{"encoding":"base64"}]}] * 500

2. POST /api/rpc with:
   Header: X-Forwarded-For: <ip>       # fresh IP, fresh rate-limit bucket
   Body: <batch of 500 getAccountInfo calls>

3. Server-side flow:
   a. getClientIp() returns the injected IP string.
   b. checkRateLimit(injectedIp, RPC_RATE_LIMIT, "rpc") → allowed (brand new bucket).
   c. isBatch = true → requests array = 500 items.
   d. All 500 pass ALLOWED_METHODS validation loop.
   e. JSON.stringify(body) → forwarded to Helius as single batch POST.
   f. Helius processes 500 RPC method calls, consuming ~500 credits.
   g. creditCounter.recordCall() fires 500 times (observational only, no gate).

4. Rate: Each iteration = 1 HTTP request + 500 Helius credits.
   With no IP constraint: limited only by attacker's outbound bandwidth and Helius
   upstream concurrency.

   Conservative estimate (10ms per round trip, no parallelism):
   → 100 iterations/second × 500 credits = 50,000 credits/second
   → 3,000,000 credits/minute
   → Helius 1M-credit plan exhausted in ~20 seconds.
   → Helius Free 500K-credit/month plan exhausted in ~10 seconds.

5. Helius returns HTTP 429 for all subsequent calls from the API key.
   ALL consumers of HELIUS_RPC_URL are immediately blinded:
   - Frontend RPC proxy: all swaps, balance queries, TX submission → fail
   - ws-subscriber: SSE data pipeline → stale/dead
   - Crank runner (if sharing the same key): epoch transitions, VRF, Carnage → halted
```

**Sustained attack:** An attacker can trivially automate this loop to re-exhaust credits the instant they reset (monthly), keeping the protocol permanently degraded for the cost of a few HTTP requests per billing cycle.

**Peak timing amplification:** Credit exhaustion during epoch boundary windows or Carnage events maximally disrupts the protocol — users cannot submit swaps, the crank cannot execute transitions, and VRF cannot be committed, extending the epoch indefinitely.

## Combined Impact

| Dimension | H008 Alone | H015 Alone | Combined |
|-----------|-----------|-----------|----------|
| Credits consumed/min | 150,000 (300 req/min × 500/batch) | Bounded by 1 call/request × req/min limit | **Unlimited** (no IP constraint, no batch limit) |
| Attacker IP requirement | Real IP, rate-limited | Any IP string bypasses limit | Any IP string, unlimited request rate |
| Mitigation needed to stop | Batch size cap only | IP validation only | Both fixes required simultaneously |
| Time to exhaust 1M credits | ~7 minutes | Hours (no batch amplification) | **~20 seconds** |
| Protocol services affected | Frontend + crank + ws-subscriber | Same | Same, but at wire speed |

**Severity of combination: HIGH** (availability — no funds at direct risk, but complete protocol RPC denial is achievable by any external party with a single command).

The combination is CONFIRMED rather than POTENTIAL because both constituent mechanisms are confirmed in source code and their interaction is deterministic: `getClientIp()` returning attacker-supplied strings and the batch array being forwarded without size validation are both verified code paths with no conditional bypass.

---

## Recommendation

**Both fixes are required to close the combination. Either fix alone leaves a residual attack surface.**

**Fix 1 — Batch size cap (addresses H008; required):**

In `app/app/api/rpc/route.ts`, before the batch is forwarded to Helius, add:

```typescript
const MAX_BATCH_SIZE = 10; // or 1 if frontend never uses batch
if (requests.length > MAX_BATCH_SIZE) {
  return NextResponse.json(
    { jsonrpc: "2.0", error: { code: -32600, message: "Batch too large" }, id: null },
    { status: 400 }
  );
}
```

If the frontend never sends batch requests, disabling batch support entirely (`Array.isArray(body)` → 400) is simpler and safer.

**Fix 2 — Correct IP extraction (addresses H015; required):**

In `app/lib/rate-limit.ts`, replace `x-forwarded-for[0]` with the rightmost IP (the one Railway's proxy appended, not the attacker-supplied prefix):

```typescript
export function getClientIp(request: Request): string {
  const forwarded = request.headers.get("x-forwarded-for");
  if (forwarded) {
    const ips = forwarded.split(",").map((s) => s.trim()).filter(Boolean);
    if (ips.length > 0) return ips[ips.length - 1]!; // rightmost = Railway's view
  }
  const realIp = request.headers.get("x-real-ip");
  if (realIp) return realIp;
  return "unknown";
}
```

**Fix 3 — Rate-limit on method calls, not HTTP requests (defense in depth):**

Even with Fix 1, a determined attacker sending 10-item batches from fresh IPs still gets 10× amplification. Charging the rate-limit bucket once per RPC method call (not per HTTP request) eliminates the residual amplification:

```typescript
for (let i = 0; i < requests.length; i++) {
  const rateCheck = checkRateLimit(clientIp, RPC_RATE_LIMIT, "rpc");
  if (!rateCheck.allowed) {
    return new Response("Too Many Requests", { status: 429 });
  }
}
```

**Fix 4 — Separate Helius API keys for crank and frontend:**

If the crank and frontend share the same `HELIUS_API_KEY`, credit exhaustion on the frontend endpoint cascades to the crank. Provisioning separate Helius projects (each with their own API key) isolates the blast radius so a frontend DoS attack cannot halt epoch transitions.
