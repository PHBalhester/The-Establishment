# H030: Enhanced Webhook Replay Protection — Regression Recheck (RECHECK of H049/H011)

## Verdict: NOT REGRESSED — CONFIRMED PERSISTENT GAP (HIGH, no new regression introduced)

---

## Summary

This is a recheck investigating whether replay protection may have regressed on the raw TX path (the original H049 fix), and whether the enhanced account-change path has acquired any replay protection since H011 was confirmed. The findings are:

1. **Raw TX path (H049 fix):** The `MAX_TX_AGE_SECONDS = 300` check is intact at lines 380-388. No regression. The fix is still present and correctly implemented.
2. **Enhanced path (H011 gap):** The `handleAccountChanges()` function still has zero replay protection. No slot check, no timestamp check, no monotonicity guard in `protocolStore`. This gap was documented in H011 and remains completely unaddressed.
3. **New observation (type-confusion escape hatch):** The type discrimination gate at line 340 (`"accountData" in firstItem`) can be used to route a crafted payload through the enhanced path, bypassing the raw TX replay guard entirely. This was noted in context API-04 as a "novel attack surface" and remains present.

## Evidence

### 1. Raw TX Replay Guard — Still Intact (route.ts:359-388)

```typescript
// H049: Maximum age for raw transactions (5 minutes). Stale
// transactions are skipped to prevent webhook replay attacks.
// Enhanced account change webhooks skip this (handled separately above).
const MAX_TX_AGE_SECONDS = 300;

for (const tx of transactions) {
  // ...
  if (blockTime != null) {
    const age = Math.floor(Date.now() / 1000) - blockTime;
    if (age > MAX_TX_AGE_SECONDS) {
      console.warn(
        `[webhook] Skipping stale transaction (blockTime ${blockTime}, age ${age}s): ${signature}`,
      );
      continue;
    }
  }
}
```

The guard is intact. The code comment itself documents the enhanced-path exclusion ("Enhanced account change webhooks skip this, handled separately above"). No regression on the H049 fix.

### 2. Enhanced Path Routing — No Temporal Check (route.ts:336-342)

```typescript
const firstItem = payload[0] as Record<string, unknown> | undefined;
if (firstItem && "accountData" in firstItem) {
  return handleAccountChanges(payload as HeliusAccountChange[]);
}
```

Any payload whose first element contains an `accountData` key routes to `handleAccountChanges()` and returns immediately. The raw TX replay guard is never reached.

### 3. HeliusAccountChange Interface — No Temporal Fields (route.ts:191-208)

```typescript
interface HeliusAccountChange {
  account: string;
  accountData: {
    account: string;
    nativeBalanceChange?: number;
    tokenBalanceChanges?: unknown[];
  }[];
  rawAccountData?: {
    data: string;
    encoding: string;
  };
}
```

The interface defines no `slot`, no `blockTime`, no sequence number. Since H011 was written this interface has not been extended. Helius does provide a `slot` field on enhanced webhook payloads in practice, but the code does not declare, extract, or use it.

### 4. handleAccountChanges — No Replay Protection (route.ts:525-633)

The full function (108 lines) contains:
- A loop over `accounts` array
- A `KNOWN_PROTOCOL_ACCOUNTS` label lookup
- Anchor decode or lamports extraction
- `protocolStore.setAccountState(pubkey, { ...normalized, updatedAt: Date.now() })`

There is no timestamp check, no slot comparison, no age check, no dedup-by-payload-hash. The `updatedAt` field stamped on the stored object reflects the server's wall clock at arrival time — not the account's on-chain slot. A payload captured a week ago arrives today and gets `updatedAt = now`, making it indistinguishable from a fresh delivery.

### 5. ProtocolStore — No Monotonicity Guard (protocol-store.ts:53-65)

```typescript
setAccountState(pubkey: string, data: AccountState): void {
  const serialized = JSON.stringify(data, bigintReplacer);
  this.accounts.set(pubkey, data);          // unconditional overwrite

  if (serialized === this.lastSerialized.get(pubkey)) return;
  this.lastSerialized.set(pubkey, serialized);
  sseManager.broadcast("protocol-update", { account: pubkey, data });
}
```

The store has two fields: `accounts` (Map) and `lastSerialized` (Map). There is no `lastSlot` map, no `lastUpdatedAt` map. The dedup guard on line 58 compares serialized content — it is a content-equality check, not a recency check. A replayed payload containing stale data will differ from current state (different epoch number, different reserves) and therefore passes the dedup check, triggering an SSE broadcast to all connected clients.

### 6. Type-Confusion Bypass of Raw TX Guard

Because type discrimination uses only `"accountData" in firstItem`, an attacker can craft a hybrid payload that is structurally an enhanced webhook but contains swap-event log data. The server routes it to `handleAccountChanges()`, where it is rejected by the `KNOWN_PROTOCOL_ACCOUNTS` check (unknown pubkey) but still bypasses the raw TX replay guard. This does not directly inject state, but confirms the enhanced path is the "low-security lane" of the two paths.

## Attack Path

**Prerequisites:** Attacker possesses the `HELIUS_WEBHOOK_SECRET`. This is obtainable from:
- `.env.devnet` committed to git (SEC-01/SEC-02 findings — confirmed via git history)
- The committed Helius API key enables webhook delivery log access via the Helius management API
- Devnet non-production environments where `NODE_ENV != "production"` skip auth entirely

```
Step 1 — Obtain a valid enhanced webhook payload
  Source A: Helius dashboard delivery logs (API key is in git history)
  Source B: Network capture of a prior delivery
  Source C: Craft a plausible payload — the schema is simple (account + rawAccountData)

Step 2 — Wait for protocol state to advance past the captured payload
  Example: Capture EpochState at epoch=5, pool reserves at block 280000000.
  Wait until epoch=7 and reserves have moved significantly.

Step 3 — Replay the stale payload
  POST /api/webhooks/helius
  Authorization: <webhook-secret>
  Content-Type: application/json
  Body: [{ "account": "<EpochState PDA>", "accountData": [...], "rawAccountData": {"data": "<base64-of-epoch-5>", "encoding": "base64"} }]

  Handler execution:
  a. Auth passes (correct secret)
  b. "accountData" in firstItem → routes to handleAccountChanges()
  c. No timestamp check, no slot check
  d. pubkey matches KNOWN_PROTOCOL_ACCOUNTS → proceed to decode
  e. Anchor-decodes stale rawAccountData bytes (epoch=5)
  f. protocolStore.setAccountState(pubkey, { epoch: 5, ..., updatedAt: Date.now() })
  g. Dedup: stale data (epoch=5) != current data (epoch=7) → PASSES dedup → SSE broadcast fires
  h. All connected browser clients receive protocol-update with epoch=5 state

Step 4 — Sustain the injection
  Replay at < 120 req/min (rate limit).
  ws-subscriber polls (30s supply, 10s staleness) will overwrite EpochState with
  fresh data from on-chain RPC, potentially recovering the store.
  Replay rate > 2 req/s for EpochState can stay ahead of the staleness check.
  For CarnageSolVault (nativeBalanceChange path — no Anchor decode), the lamports
  value can be set to 0, displaying a drained Carnage fund to all users.
```

## Impact

**Severity: HIGH** (unchanged from H011 — no regression means no escalation, no downgrade)

**Data pipeline integrity compromise.** The protocol store is the single authoritative source for all SSE-fed UI state. Successful replay injection affects every connected browser client simultaneously:

- **EpochState**: Stale epoch number, wrong tax rates (buy/sell BPS). Quote engine uses SSE-delivered tax rates → displayed quotes diverge from on-chain reality. On-chain `minimumOutput` enforcement is the backstop but displayed price is wrong.
- **PoolState (CRIME_SOL, FRAUD_SOL)**: Stale reserves → wrong price display, wrong quote computation, wrong chart prices.
- **CurveState (CRIME, FRAUD)**: Stale bonding curve supply → wrong current price, wrong progression display.
- **StakePool**: Wrong total staked, wrong staker count displayed.
- **CarnageFundState**: Wrong fund balance.
- **CarnageSolVault**: Attacker-supplied `nativeBalanceChange` (the only field used) — can display zero or any value as the vault balance.

**No direct fund loss** — on-chain slippage enforcement (minimumOutput) is intact. The attack corrupts display and quote computation, not execution.

**MEV amplification** — an attacker operating an SSE listener (unauthenticated endpoint) sees real state from ws-subscriber poll; all other users see the attacker-injected stale state. This creates a structural information asymmetry favoring the attacker.

---

## Recommendation

The recommendations from H011 remain valid and unimplemented. Summarized:

**Primary (slot-based monotonicity in protocolStore):**
1. Extend `HeliusAccountChange` interface: add `slot?: number`
2. Add `private lastSlot = new Map<string, number>()` to `ProtocolStore`
3. Thread slot through `handleAccountChanges → setAccountState`
4. In `setAccountState`: `if (slot <= prevSlot) return` before overwriting

**Secondary (payload-level request dedup):**
- Maintain a bounded LRU set (e.g., 1000 entries, 10-minute TTL) of SHA-256 hashes of accepted request bodies
- Reject exact-duplicate payloads within the TTL window

**Tertiary (CarnageSolVault lamports validation):**
- The `nativeBalanceChange` field is already noted as ASM-7 (likely incorrect — delta vs. absolute). Replay of a crafted payload with `nativeBalanceChange: 0` displays a drained vault to all users. The slot guard above would mitigate this for replays; the ASM-7 data correctness issue is separate.
