# H059: SSE Candle Route Leaks Protocol State

**Tier:** 2
**Origin:** Novel
**Verdict:** CONFIRMED

---

## Hypothesis

`/api/sse/candles` subscribers receive ALL `sseManager` events — including `protocol-update` events — because the subscriber callback has no event-type filter.

---

## Target

`app/app/api/sse/candles/route.ts` lines 71–76

---

## Evidence

### The vulnerable subscriber (candles/route.ts:71–78)

```typescript
const unsubscribe = sseManager.subscribe((payload: string) => {
  try {
    controller.enqueue(encoder.encode(payload));
  } catch {
    // Client disconnected -- ReadableStream controller is closed
    unsubscribe();
  }
});
```

The callback receives the raw SSE-formatted string directly from `sseManager.broadcast()` and enqueues it unconditionally. There is no check on event type.

### The correct pattern already exists — in the protocol route (protocol/route.ts:82–92)

```typescript
const unsubscribe = sseManager.subscribe((payload: string) => {
  // Only forward protocol-update events (not candle-update, etc.)
  if (!payload.startsWith("event: protocol-update\n")) return;

  try {
    controller.enqueue(encoder.encode(payload));
  } catch {
    unsubscribe();
  }
});
```

The protocol route explicitly guards with `payload.startsWith("event: protocol-update\n")`. The candles route has no equivalent guard.

### The sseManager broadcasts multiple event types to one shared subscriber set

`sseManager` is a single in-memory singleton (`app/lib/sse-manager.ts`). All subscribers are stored in one `Set<SSECallback>`. `broadcast(event, data)` formats the payload as:

```
event: <eventName>\n
data: <JSON>\n\n
```

and calls every subscriber without event-type discrimination. Two distinct event types are broadcast to this shared bus:

- `candle-update` — issued by the webhook handler after candle upsert
- `protocol-update` — issued by `protocolStore.setAccountState()` for every decoded Anchor account change (EpochState, PoolState, StakePool, CarnageFundState, CurveState, plus synthetic keys like `__supply:CRIME`, `__slot`)

### What leaks

Every `protocol-update` event is forwarded verbatim to every client connected to `/api/sse/candles`. The payload shape is:

```json
{
  "account": "<pubkey or __synthetic_key>",
  "data": { ...decoded protocol account fields... }
}
```

This includes full decoded account state for:
- `EpochState` — current epoch number, vote tallies, tax BPS rates, faction pool totals
- `PoolState` — reserve amounts for all four pools (CRIME/SOL, FRAUD/SOL, CRIME/PROFIT, FRAUD/PROFIT)
- `StakePool` — staked balances, reward accumulators
- `CarnageFundState` — carnage fund lamport balance
- `CurveState` — bonding curve reserves
- Synthetic keys: `__supply:CRIME`, `__supply:FRAUD`, `__supply:PROFIT`, `__slot`

Chart-only clients that opened an `EventSource` on `/api/sse/candles` — expecting only price candle data — receive the full protocol account state continuously.

---

## Impact

**Bandwidth / resource waste (primary):** Every protocol account change generates an extra fan-out write to all candle-only subscribers. In production with 5,000 global SSE connections split across both endpoints, every webhook delivery could trigger double the expected outbound SSE writes.

**Unintended data exposure (secondary):** Candle clients are not the intended consumers of protocol state. Although the protocol account data is on-chain public data (no private information), the data is being delivered to a channel whose documentation explicitly scopes it to price candles only. This breaks the trust boundary between the two SSE endpoints and means any future addition of sensitive synthetic keys to the protocol store would leak to candle clients automatically, without any code change required.

**Client-side processing confusion:** Clients that parse all received SSE events generically (e.g., explorers or third-party integrations connecting to `/api/sse/candles`) will receive and may act on `protocol-update` events they are not designed to handle.

---

## Root Cause

The candles route was implemented without the event-type filter that was later added to the protocol route. The `sseManager` has no concept of channels or topics — it is a flat pub/sub bus. Correct filtering is entirely the responsibility of each subscriber callback, and the candles route subscriber omits it.

---

## Fix

Add a `startsWith` guard to the candles route subscriber callback, mirroring the protocol route:

```typescript
const unsubscribe = sseManager.subscribe((payload: string) => {
  // Only forward candle-update events (not protocol-update, etc.)
  if (!payload.startsWith("event: candle-update\n")) return;

  try {
    controller.enqueue(encoder.encode(payload));
  } catch {
    unsubscribe();
  }
});
```

One line added. No other changes required.
