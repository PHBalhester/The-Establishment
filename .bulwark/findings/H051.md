## Verdict: CONFIRMED

`batchSeed` contains four bare `await` calls (lines 115, 168, 187, 195) with no
surrounding try/catch. Any one of them throwing propagates the exception out of
`batchSeed`, out of `init()`, and is caught by `instrumentation.ts`'s outer
try/catch at line 23. As a result:

1. `state.initialized` at line 474 is **never set to `true`**.
2. All four ongoing pipelines (`startSlotSubscription`, `startSupplyPoll`,
   `startStakerPoll`, `startStalenessMonitor`) are **never started**.
3. The double-init guard at line 456 (`if (state.initialized)`) permanently
   blocks any subsequent retry because it tests a flag that is still `false` —
   meaning the guard that is supposed to prevent double-init *also* prevents
   recovery-init.

The process stays alive (instrumentation's outer catch prevents a crash), but the
subscriber is permanently dead for the lifetime of that server instance. Every SSE
client that connects afterward receives only the partial store snapshot written
before the failure point, with no live updates ever arriving.

---

## Hypothesis

If `getTokenSupply` / `getSlot` / `getProgramAccounts` fails during `batchSeed`,
all previously-seeded account data is retained but `state.initialized` is never
set to `true`, and supply/slot/staker data is empty until the next poll cycle.

The actual impact is broader than the hypothesis stated: the next poll cycle never
arrives because the poll timers are never started when `batchSeed` throws.

### Failure scenarios and their outcome

| Failing call | Store state at throw | state.initialized | Timers started |
|---|---|---|---|
| `getMultipleAccountsInfo` (line 115) | All 8 PDAs absent | false | None |
| `getTokenSupply` (line 168) | 8 PDAs seeded, supply absent | false | None |
| `getSlot` (line 187) | 8 PDAs + supply seeded, slot absent | false | None |
| `getProgramAccounts` (line 195) | 8 PDAs + supply + slot seeded, staker stats absent | false | None |

In every case the poll timers that would refresh the missing data are never
registered, so the partial state is frozen for the process lifetime.

### Double-init guard blocks recovery

After `instrumentation.ts` catches the error and continues, `state.initialized`
remains `false`. However, Next.js does not call `register()` again after the
initial boot. There is no retry loop, no exponential backoff, and no external
trigger that would re-invoke `init()`. A future request that reaches the SSE
endpoint does not attempt re-initialization:

```ts
// app/app/api/sse/protocol/route.ts — no call to init()
// protocolStore.getAllAccountStates() returns whatever was seeded before the throw
```

Even if something did call `init()` again, the double-init guard at line 456 only
fires when `state.initialized` is `true`. Since it is `false`, a second call would
re-enter `batchSeed` — which is actually the desired recovery path but is not
wired up anywhere.

### Partial-seed window visible to SSE clients

When `batchSeed` throws mid-way (e.g. at the `getSlot` call), the protocol store
contains decoded PDA data and token supply, but `__slot` is absent. New SSE clients
that connect receive `getAllAccountStates()` as their initial snapshot. They will:

- Render pool prices and epoch state (PDA data present) — OK
- Show correct CRIME/FRAUD supply — OK
- Receive `undefined` for the current slot — UI slot counter shows nothing
- Never receive staker count or locked/unlocked PROFIT values if gPA was the
  failing call
- Never receive a live update for any of these values because the timers were
  not started

The user-facing effect depends on which call failed, but the subscriber is silently
broken with no visible server-side alarm beyond the single `console.error` in
`instrumentation.ts`.

## Attack / Failure Scenario

**Most likely trigger — Helius RPC overload at server boot:**

1. Server process starts (Railway restart after deploy or container recycle).
2. `batchSeed` issues `getMultipleAccountsInfo` (8 accounts), two
   `getTokenSupply` calls, `getSlot`, and `getProgramAccounts` — five RPC
   operations within seconds of each other against a single Helius endpoint.
3. Under brief RPC rate-limiting or a Helius maintenance window, any one of
   these throws (e.g. HTTP 429, network timeout, or JSON parse error on a
   malformed response).
4. The exception propagates; `instrumentation.ts` logs it and continues.
5. `state.initialized = false`; no timers started.
6. The Next.js server stays up and serves requests, but every SSE client gets a
   stale or empty snapshot for the rest of that process lifetime.
7. The condition self-heals only on the next Railway restart — a non-deterministic
   event.

**Secondary trigger — Helius gPA throttle on mainnet:**

`getProgramAccounts` is the last call in `batchSeed`. On mainnet with hundreds of
stakers, this is the largest and slowest call. If Helius throttles it (common on
shared-endpoint plans), it throws, the staker count is never written to the store,
and the staker poll timer is never started. The UI shows zero/stale staker
statistics indefinitely.

## Root Cause

`batchSeed` is structured as a sequential async function with four bare `await`
calls that can throw network/RPC errors. `init()` calls `batchSeed` without a
try/catch, and sets `state.initialized = true` only at the very end — after all
four calls succeed. There is no partial-success handling, no retry loop, and no
mechanism to resume from the last successful step.

The instrumentation-level catch suppresses the error gracefully at the process
level (correct), but provides no recovery path (the gap).

---

## Recommended Fix

### Option A — Per-section try/catch inside batchSeed (minimal diff)

Wrap the three sequential sections (supply, slot, gPA) in independent try/catch
blocks so a failure in one section does not abort the others, and still allows
`init()` to reach `state.initialized = true` and start the timers:

```ts
async function batchSeed(connection: Connection): Promise<void> {
  // Section 1: account batch — already has per-item try/catch
  try {
    const accountInfos = await connection.getMultipleAccountsInfo(pubkeys);
    // ... decode loop (existing try/catch per item)
  } catch (err) {
    console.error("[ws-subscriber] batchSeed: getMultipleAccountsInfo failed:", err);
    // Continue — poll timers will refresh missing data
  }

  // Section 2: token supply
  try {
    const [crimeSupply, fraudSupply] = await Promise.all([
      connection.getTokenSupply(MINTS.CRIME),
      connection.getTokenSupply(MINTS.FRAUD),
    ]);
    protocolStore.setAccountStateQuiet("__supply:CRIME", { ... });
    protocolStore.setAccountStateQuiet("__supply:FRAUD", { ... });
  } catch (err) {
    console.error("[ws-subscriber] batchSeed: getTokenSupply failed:", err);
  }

  // Section 3: current slot
  try {
    const slot = await connection.getSlot();
    protocolStore.setAccountStateQuiet("__slot", { slot });
    state.latestSlot = slot;
    state.lastSlotReceivedAt = Date.now();
    state.lastSlotBroadcastAt = Date.now();
  } catch (err) {
    console.error("[ws-subscriber] batchSeed: getSlot failed:", err);
  }

  // Section 4: staker gPA
  try {
    const stakerGpaAccounts = await connection.getProgramAccounts(
      PROGRAM_IDS.STAKING, { filters: [...] }
    );
    // ... decode + store (existing logic)
  } catch (err) {
    console.error("[ws-subscriber] batchSeed: getProgramAccounts failed:", err);
  }
}
```

With this change, `batchSeed` no longer throws. `init()` always reaches
`state.initialized = true` and starts all four timers. Sections that failed during
seed are refreshed on the next poll cycle (within 30–60 s).

### Option B — Retry loop in instrumentation.ts (defence in depth)

Add a bounded retry in `instrumentation.ts` so a transient RPC failure at boot
is retried before giving up:

```ts
// instrumentation.ts
const MAX_RETRIES = 3;
const RETRY_DELAY_MS = 5_000;

for (let attempt = 1; attempt <= MAX_RETRIES; attempt++) {
  try {
    const { init } = await import("@/lib/ws-subscriber");
    await init();
    break; // success
  } catch (err) {
    console.error(`[instrumentation] ws-subscriber init failed (attempt ${attempt}/${MAX_RETRIES}):`, err);
    if (attempt < MAX_RETRIES) {
      await new Promise(res => setTimeout(res, RETRY_DELAY_MS));
    }
  }
}
```

Option A is preferred as it handles partial failure gracefully. Option B alone
does not handle mid-batchSeed partial state — a retry from the top re-runs all
four calls, potentially re-seeding sections that already succeeded. Both options
combined provide the strongest guarantee.
