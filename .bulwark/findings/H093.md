# H093: DB TLS Only in Production

**Severity**: Tier 3 (Medium)
**Category**: Data/Encryption (DATA-05)
**Status**: CONFIRMED
**Target**: `app/db/connection.ts:51-52`, `app/db/migrate.ts:42`

## Summary

Database connections enforce TLS **only in production** (`NODE_ENV === "production"`). Preview/staging environments and development may transmit database credentials and queries in plaintext over the network if connected to a remote host.

## Details

### connection.ts Analysis (Lines 51-52)

```typescript
const isProductionDb = process.env.NODE_ENV === "production";
const sslConfig = isProductionDb ? { ssl: "require" as const } : {};
```

**Finding**: TLS is gated on production environment check. Non-production environments connect without `ssl: "require"` option.

**Existing Mitigations**:
- Lines 59-74 include a warning (`VH-L002`) if a non-production connection targets a remote host without TLS
- Warning triggers if host is not localhost/127.0.0.1/::1
- This is advisory only; connection still proceeds unencrypted

### migrate.ts Analysis (Line 42)

```typescript
const client = postgres(databaseUrl, { max: 1 });
```

**Finding**: Migration runner does NOT apply TLS options at all. Runs unconditionally without `ssl: "require"`, even in production.

**Risk**: If Railway's preDeployCommand (which runs migrations) is triggered from a preview/staging environment with DATABASE_URL pointing to a production database, credentials transmit unencrypted.

## Risk Assessment

**Probability**: Medium
- Preview/staging environments commonly exist in modern deployments
- NODE_ENV=production check is insufficient if DATABASE_URL points to shared infrastructure
- Railway's environment handling may not guarantee NODE_ENV in all contexts

**Impact**: High
- Plaintext database credentials on the wire
- Potential sniffing attacks if attacker controls network segment
- Postgres credentials can be captured and replayed

## Remediation

### Option 1: CONFIRMED (Require TLS Always)

Apply TLS requirement unconditionally in both connection.ts and migrate.ts:

```typescript
// connection.ts line 52
const sslConfig = { ssl: "require" as const };
```

```typescript
// migrate.ts line 42
const client = postgres(databaseUrl, { max: 1, ssl: "require" });
```

**Rationale**: Production Postgres (Railway) supports TLS. No legitimate use case for unencrypted connections to production or preview databases.

### Option 2: ACCEPTED_RISK (Document & Warn)

Keep current behavior but enhance migrate.ts warning:

```typescript
// migrate.ts line 42
console.log("[migrate] WARNING: TLS not enforced in migrations (H093). Ensure DATABASE_URL connects to trusted network.");
const client = postgres(databaseUrl, { max: 1 });
```

**Rationale**: If migrations only run in Railway's internal network, TLS is less critical. But this is not documented clearly.

## Recommendation

**RECOMMENDED: Option 1 (Require TLS Always)**

Reason: Rail way Postgres supports TLS by default. Always-on TLS eliminates entire class of network-layer attacks and adds zero operational burden. The current NODE_ENV check creates false confidence — it doesn't prevent staging from connecting to production databases unencrypted.
