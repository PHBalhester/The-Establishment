# H050 — No Process-Level Error Handlers

| Field | Value |
|-------|-------|
| **ID** | H050 |
| **Tier** | 2 |
| **Verdict** | CONFIRMED |
| **Origin** | KB (OC-268, OC-269) |
| **Target** | Project-wide (Next.js app + crank runner) |
| **Files** | `app/instrumentation.ts`, `app/instrumentation-client.ts`, `scripts/crank/crank-runner.ts`, `app/lib/ws-subscriber.ts` |

---

## Hypothesis

No `process.on("unhandledRejection")` handler exists in the Next.js server process. An unhandled async error crashes the container silently, and Railway's auto-restart may obscure the failure with no diagnostic trace.

---

## Evidence

### 1. Grep Results — Confirmed Absence in Source Code

A project-wide search for `process.on("unhandledRejection")` and `process.on("uncaughtException")` found **zero matches in any application source file**. The only match in the project tree is the compiled `.next/build/chunks/[root-of-the-server]__5d0724af._.js` (line 342):

```js
process.on('uncaughtException', (err) => {
    IPC.sendError(err);
});
```

This is a **Turbopack internal handler** — it sends the error to the Turbopack build IPC channel (`createIpc(parseInt(PORT, 10))`), not to Sentry, not to Railway logs in any structured form. It covers only Turbopack's worker subprocess context and provides no application-level recovery or alerting.

### 2. instrumentation.ts — No Global Handlers Registered

`app/instrumentation.ts` is the canonical location to register server-side global handlers in Next.js (it runs before any request is served). It contains only a `try/catch` around `ws-subscriber` init — no `process.on` calls of any kind.

### 3. instrumentation-client.ts — Browser Only

`app/instrumentation-client.ts` does register `window.addEventListener("error")` and `window.addEventListener("unhandledrejection")` for the **browser** context (guarded by `typeof window !== "undefined"`). This correctly covers the client side. There is no corresponding server-side equivalent.

### 4. crank-runner.ts — SIGINT/SIGTERM Only

`scripts/crank/crank-runner.ts` has `process.on("SIGINT")` and `process.on("SIGTERM")` for graceful shutdown (lines 196, 202), but no `unhandledRejection` or `uncaughtException` handlers. The crank uses multiple async paths that are outside the main try/catch:
- Dynamic imports in the VRF flow
- Switchboard SDK internal promise chains
- The circuit breaker loop itself (if it throws outside the caught interval)

### 5. ws-subscriber.ts — setInterval Async Callbacks

`app/lib/ws-subscriber.ts` registers four `setInterval` timers with async callbacks (lines 282, 304, 360, 430). Each is individually try/caught. However, if any future code change introduces an async throw before the catch, or if the catch block itself throws, the promise rejection is unhandled. There is no safety net at the process level to capture this.

### 6. Context File Confirmation

`27-ERR-01-error-handling.md` line 376 explicitly documents:

> "The Next.js application has **NO** `process.on("unhandledRejection")` or `process.on("uncaughtException")` handlers."

The only listed process handlers are SIGINT/SIGTERM in crank-runner and overnight-runner (signal handlers, not error handlers).

### 7. Railway Restart Behavior

Architecture notes confirm Railway is a single-process deployment (ASM-5 VALIDATED). Railway does auto-restart crashed containers, but:
- Restart latency means a window of complete downtime (SSE dead, webhooks unprocessed, crank halted)
- The crash reason is logged only to Railway's stdout — not to Sentry, not to any structured alert
- The Sentry integration (`app/lib/sentry.ts`) uses `fetch()` to POST error envelopes, but it is only called from explicit `captureException()` call sites. An uncaught crash bypasses all call sites entirely, so nothing reaches Sentry

---

## Attack / Failure Scenario

1. A new feature adds an `async` function called from a `setInterval` without a surrounding `try/catch` (or a `catch` block that itself throws).
2. The resulting rejected promise has no handler. Node.js emits `unhandledRejection`.
3. In Node.js 15+, unhandled rejections **terminate the process** by default (previously emitted a warning). The Next.js server exits.
4. Railway detects the crash and restarts the container. Restart takes ~10–30 seconds.
5. During restart: all SSE connections drop, in-flight webhooks from Helius time out (Helius retries, but with up to 60s gap), the protocol store is wiped and must re-seed from RPC.
6. No Sentry alert fires. The only evidence is Railway's crash log, which users don't monitor in real time.

The crank is a second distinct surface: a silent crash stops epoch advancement entirely, triggering the zombie-crank scenario documented in H031.

---

## Severity Assessment

**Tier 2 (MEDIUM)** — Not directly exploitable by an adversary, but:
- Availability impact is real: any unhandled rejection kills the server process with no alerting
- The gap grows over time as new async code is added without guaranteed catch coverage
- Combined with the "no Sentry on crash" gap (H050 x DATA-04 intersection), the team has no visibility into crash frequency

Mitigated partially by:
- Railway auto-restart limits downtime duration
- The individual `try/catch` blocks on the four `setInterval` callbacks in `ws-subscriber.ts` reduce the immediate surface
- Next.js App Router handles per-request async errors with its own error boundary (returns 500, does not crash the process)

The unmitigated surface is: any async code path **not** inside a request handler or an explicitly caught interval callback.

---

## Recommended Fix

Add global handlers in `app/instrumentation.ts` inside the Node.js runtime guard:

```typescript
export async function register() {
  if (typeof window !== "undefined") return;
  if (process.env.NEXT_RUNTIME === "edge") return;

  // Global crash handlers — must be first, before any async init
  process.on("unhandledRejection", (reason) => {
    console.error("[process] unhandledRejection:", reason);
    // Import is async — use synchronous captureException path if available
    // or write directly to stderr so Railway captures it
  });
  process.on("uncaughtException", (err) => {
    console.error("[process] uncaughtException:", err);
    process.exit(1); // Required — process state is undefined after uncaughtException
  });

  try {
    const { init } = await import("@/lib/ws-subscriber");
    await init();
  } catch (err) {
    console.error("[instrumentation] ws-subscriber init failed:", err);
  }
}
```

For the crank runner, add equivalent handlers at the top of `scripts/crank/crank-runner.ts` after the imports.

Note: `captureException` cannot be called inside `process.on("uncaughtException")` if it uses `fetch()` (async) — the process must exit immediately. A synchronous write to stderr is the only safe option there. `unhandledRejection` can safely call `captureException` since the event loop is still running.
