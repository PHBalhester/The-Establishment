# Finding: H046 - SSE Connection Limit Bypass via Race

## Status: CONFIRMED — NOT VULNERABLE (POTENTIAL under architectural change)
## Confidence Score: 8

## Executive Summary

`acquireConnection()` in `app/lib/sse-connections.ts` performs a non-atomic read-check-write sequence: it reads `ipCount`, compares against `MAX_PER_IP`, then writes back `ipCount + 1`. Two simultaneous HTTP requests from the same IP could theoretically both read `ipCount = 9`, both pass the `< MAX_PER_IP` check, and both write `10` — resulting in the counter underreporting the true connection count of 11.

However, this is **not exploitable in the current deployment architecture**. Node.js runs on a single-threaded event loop: JavaScript execution within a synchronous tick cannot be interleaved. `acquireConnection()` contains zero `await` points and performs only synchronous Map reads and writes. It executes atomically within a single tick. No two concurrent HTTP handlers can interleave inside it.

The finding is rated **POTENTIAL** because it becomes a real vulnerability if the deployment ever adopts Node.js Worker Threads sharing the `sseConnState` globalThis object, or if the code is ported to a multi-threaded runtime (e.g., Deno or Bun with parallel request handling). The current implementation's correctness is an implicit architectural assumption, not an explicit guarantee.

## Hypothesis Investigated

Two simultaneous SSE connections from the same IP can both pass `ipCount < MAX_PER_IP`, exceeding the per-IP limit of 10.

## Investigation Path

### Step 1: Read the acquireConnection implementation
**Location:** `app/lib/sse-connections.ts:49-57`

```typescript
export function acquireConnection(ip: string): boolean {
  if (state.globalCount >= MAX_GLOBAL) return false;
  const ipCount = state.connections.get(ip) ?? 0;
  if (ipCount >= MAX_PER_IP) return false;

  state.connections.set(ip, ipCount + 1);
  state.globalCount++;
  return true;
}
```

The three-step sequence (read `ipCount` → compare → write `ipCount + 1`) is the classic check-then-act race window. In a multi-threaded or truly concurrent system this would allow concurrent callers to both see `ipCount = 9`, both pass `< 10`, and both write `10` (leaving the actual slot count at 11 while the counter shows 10).

### Step 2: Assess Node.js concurrency model
**Context:** `28-ERR-02-race-concurrency.md` (Trust Boundaries section)

> "Within a single synchronous tick, execution is serialized -- no true data races are possible. The race conditions identified here are all async interleaving races: multiple `await` points create windows where other async callbacks can interleave."

`acquireConnection()` is entirely synchronous: no `await`, no I/O, no callbacks. It completes within a single event loop tick. Two incoming HTTP requests — even if they arrive microseconds apart — are handled sequentially by Node.js. The second request's handler cannot begin executing JavaScript until the first handler yields (calls `await` or returns). By that point, `acquireConnection()` has already returned and the counter is already updated.

### Step 3: Verify no async boundary exists within acquireConnection
**Location:** `app/lib/sse-connections.ts:49-57`

Confirmed: the function body contains only:
- `Map.get()` — synchronous
- Numeric comparison — synchronous
- `Map.set()` — synchronous
- Integer increment — synchronous
- Boolean return — synchronous

Zero `await`, `Promise`, `setTimeout`, or I/O. The invariant holds under Node.js.

### Step 4: Examine SSE route — gap between acquire and abort handler registration
**Location:** `app/app/api/sse/protocol/route.ts:44-119`

There is a related (but distinct) risk noted in the context doc: if a client disconnects during the window between `acquireConnection()` on line 44 and the `req.signal.addEventListener("abort", ...)` registration on line 110, the slot is acquired but the abort handler never fires. The `cancel()` callback on line 121 (stream cancel) does call `release()`, but the abort handler is registered deep inside `ReadableStream.start()`, which runs synchronously at `new ReadableStream(...)` construction time on line 66. This gap is narrow (synchronous construction) but does exist between the connection check at line 44 and the Response being returned.

The 30-minute `scheduleAutoRelease` (line 52) is the only safety net for this scenario, not for the race itself.

### Step 5: Assess exploitability under current and future architectures
- **Current (single Railway process, Node.js, no Worker Threads):** NOT VULNERABLE. Event loop serializes all JavaScript execution within a tick. `acquireConnection` is fully synchronous. Two requests cannot interleave inside it.
- **Future (Worker Threads sharing globalThis):** VULNERABLE. Worker Threads run in separate V8 isolates but can share memory via SharedArrayBuffer. The `sseConnState` globalThis is a plain JS Map — not a SharedArrayBuffer — so it would not be shared across Worker Threads automatically. Safe here too.
- **Future (multiple Railway instances / horizontal scaling):** VULNERABLE. Each process has its own in-memory `sseConnState`. Per-IP limit is completely ineffective across processes. This is a separate (known) concern from the race condition itself; ARCHITECTURE.md notes "Single Railway process (no horizontal scaling)" as a validated assumption (ASM-5).
- **Future (Bun/Deno multi-threaded runtime):** POTENTIALLY VULNERABLE. Both runtimes have multi-threaded HTTP handling options. Porting this code without adding atomics would make the race real.

## Blocking Factors (why it doesn't fire)
- Node.js single-threaded event loop guarantees sequential execution within a synchronous call stack.
- `acquireConnection()` is entirely synchronous — no yield points.
- SSE connection state is a plain JS Map, not shared memory.

## Enabling Factors (why it could fire in future)
- No mutex, lock, or atomic operation protects the read-check-write.
- The code's correctness is an implicit Node.js assumption, not documented as a requirement.
- No test exists that would catch this under a hypothetical concurrent executor.

## Severity Assessment
- **Current Severity:** INFORMATIONAL (not exploitable under current runtime)
- **Potential Severity:** MEDIUM (if runtime assumptions change)
- **Likelihood (current):** N/A — not triggerable
- **Impact if triggered:** Per-IP limit can be exceeded by ~1-2 connections per race window. Weakens H008 DoS protection but does not eliminate it (global cap still applies synchronously). An attacker would need many simultaneous connections to cause meaningful resource pressure.

## Recommended Fix

Add a comment to `acquireConnection()` documenting the Node.js single-threaded invariant explicitly, so future maintainers understand the safety assumption:

```typescript
export function acquireConnection(ip: string): boolean {
  // Note: this read-check-write is non-atomic in the general case.
  // It is safe under Node.js because acquireConnection() is synchronous
  // (no await points), so the event loop cannot interleave two callers
  // within this function. If this code is ever moved to a multi-threaded
  // runtime or run across multiple processes, an atomic counter (e.g.,
  // Atomics.add on a SharedArrayBuffer, or a Redis INCR) is required.
  if (state.globalCount >= MAX_GLOBAL) return false;
  const ipCount = state.connections.get(ip) ?? 0;
  if (ipCount >= MAX_PER_IP) return false;

  state.connections.set(ip, ipCount + 1);
  state.globalCount++;
  return true;
}
```

If horizontal scaling is ever enabled (ASM-5 violated), replace with a Redis-backed atomic counter using `INCR` / `DECRBY` with TTL.
