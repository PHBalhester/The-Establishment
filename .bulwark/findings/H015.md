# H015: IP Spoofing Bypasses All Per-IP Rate Limits

## Verdict: CONFIRMED — MEDIUM

The rate limiter in `app/lib/rate-limit.ts` unconditionally trusts the first entry of the `x-forwarded-for` header as the client IP. There is no validation that this header was set by a trusted proxy, no allowlist of known proxy addresses, and no stripping of attacker-injected values. An attacker can set `X-Forwarded-For: <random-ip>` on each request to obtain a fresh rate-limit bucket, bypassing all per-IP limits.

---

## Evidence

### IP extraction — `app/lib/rate-limit.ts:129-151`

```typescript
export function getClientIp(request: Request): string {
  const forwarded = request.headers.get("x-forwarded-for");
  if (forwarded) {
    // Take the first IP (original client)
    const firstIp = forwarded.split(",")[0]?.trim();
    if (firstIp) return firstIp;            // ← returned verbatim, no validation
  }

  const realIp = request.headers.get("x-real-ip");
  if (realIp) return realIp;

  // No proxy headers found — all requests share a single bucket.
  if (process.env.NODE_ENV === "production") {
    console.warn("[rate-limit] WARNING: No proxy headers ...");
  }

  return "unknown";
}
```

The function takes whatever the client places in `x-forwarded-for` at position `[0]` and returns it as the canonical IP. There is no check against a proxy IP allowlist, no comparison of the header length (i.e., no validation that Railway appended rather than the client prepended), and no use of a framework-level `trustProxy` configuration.

### Rate limit key construction — `app/lib/rate-limit.ts:86`

```typescript
const key = `${ip}:${endpoint}`;
```

Every unique string returned by `getClientIp()` creates a distinct bucket in the `entries` Map. Since the function returns whatever is in `x-forwarded-for[0]`, generating a fresh IP string per request is sufficient to obtain a fresh bucket.

### All three protected routes call the same function

- `app/app/api/rpc/route.ts:82` — `getClientIp(request)` → `checkRateLimit(..., RPC_RATE_LIMIT, "rpc")` (300/min)
- `app/app/api/webhooks/helius/route.ts:257` — `getClientIp(req)` → `checkRateLimit(..., WEBHOOK_RATE_LIMIT, "webhook")` (120/min)
- `app/app/api/sol-price/route.ts:84` — `getClientIp(request)` → `checkRateLimit(..., SOL_PRICE_RATE_LIMIT, "sol-price")` (30/min)

All three endpoints share the same vulnerable IP-extraction path, so all three per-IP limits are bypassable with the same technique.

### No Next.js `trustProxy` or proxy IP validation in `app/next.config.ts`

The `next.config.ts` file configures CSP headers, image optimization, and Turbopack aliases. There is no `trustProxy` setting, no IP allowlist, and no framework-level configuration that would cause Next.js to overwrite or validate `x-forwarded-for` before it reaches the route handler.

### `credit-counter.ts` is not affected

`app/lib/credit-counter.ts` tracks RPC method usage for monitoring only; it does not use IP-based keying and is not a rate-limiting mechanism. No vulnerability here.

### ARCHITECTURE.md confirms the assumption is unvalidated

`ARCHITECTURE.md` (line 163, ASM-3): "x-forwarded-for accurately reflects client IP — **UNVALIDATED** — All rate limits bypassable."

## Attack Path

**Precondition:** Railway's reverse proxy appends the real client IP to `x-forwarded-for` rather than replacing the entire header. This is the standard RFC 7239 proxy behavior: each proxy appends its view of the upstream IP. The existing header value (set by the client) is therefore preserved as `x-forwarded-for[0]`.

**Standard proxy behavior is the vulnerability.** If Railway appended instead of overwrote, `x-forwarded-for[0]` is always the client-controlled value.

**Exploitation steps:**

1. Attacker sends `POST /api/rpc` with header `X-Forwarded-For: 203.0.113.1`.
2. `getClientIp()` returns `"203.0.113.1"`.
3. `checkRateLimit("203.0.113.1", RPC_RATE_LIMIT, "rpc")` → allowed (fresh bucket).
4. Next request: `X-Forwarded-For: 203.0.113.2`.
5. `getClientIp()` returns `"203.0.113.2"` → fresh bucket again.
6. Repeat with any IP string. The rate limiter is bypassed for every request.

**Combined amplification:** Spoofed IP bypass works across all five unprotected endpoints (`/api/candles`, `/api/carnage-events`, `/api/health`, plus the three protected ones). The attacker can simultaneously spam all endpoints at unbounded rate while also bypassing the limits on the three protected ones.

**Webhook spoofing consideration:** The webhook endpoint (`/api/webhooks/helius`) has a second authentication layer (timingSafeEqual shared secret, fail-closed in production). The rate limit bypass on the webhook route is therefore exploitable only if the attacker also knows `HELIUS_WEBHOOK_SECRET`. For the RPC proxy and sol-price routes there is no second layer.

## Impact

**Severity: MEDIUM** (downgraded from the hypothesized CRITICAL for the following reasons):

- **Direct financial / data theft impact: none.** Rate limit bypass does not grant access to any privileged operation. No funds are at risk on the off-chain layer from rate limit bypass alone.
- **Primary impact is availability (DoS amplifier).** The bypass makes the in-memory rate limiter ineffective against a determined attacker, removing the primary defense for RPC proxy and sol-price routes. Combined with the confirmed gaps (no rate limiting on `/api/candles`, `/api/carnage-events`, `/api/health`) the overall DoS posture is significantly weakened.
- **RPC credit drain.** The RPC proxy rate limit (300/min per IP) protects against runaway Helius credit burn. With IP spoofing, an attacker can drive unlimited RPC calls through the proxy, exhausting the project's Helius credit allocation, which degrades the entire protocol UI for all users.
- **Platform dependency.** If Railway's reverse proxy does overwrite `x-forwarded-for` (prepend behavior disabled, replacement mode enabled), the vulnerability does not exist in practice. This is not documented or validated anywhere in the codebase, making the assumption unverifiable without live testing.

---

## Recommendation

**Option A — Trust only the rightmost IP (most robust):**

When Railway (or any well-configured proxy) appends the real client IP, the real IP is the **last** entry in `x-forwarded-for`. Taking the rightmost IP prevents header injection because the proxy's appended value is authoritative.

```typescript
export function getClientIp(request: Request): string {
  const forwarded = request.headers.get("x-forwarded-for");
  if (forwarded) {
    const ips = forwarded.split(",").map((s) => s.trim()).filter(Boolean);
    // Rightmost IP is the one the trusted proxy observed — not attacker-controllable
    if (ips.length > 0) return ips[ips.length - 1]!;
  }
  const realIp = request.headers.get("x-real-ip");
  if (realIp) return realIp;
  return "unknown";
}
```

**Option B — Proxy count configuration:**

If the number of trusted proxy hops is known (e.g., always exactly 1 Railway hop), take `x-forwarded-for[length - trustedHops]`. This is the `express` `trust proxy = N` approach.

**Option C — Validate Railway proxy behavior (minimum viable):**

Confirm in Railway's documentation or support that their proxy replaces (not appends to) `x-forwarded-for`. Document this as a validated invariant. If confirmed, the current behavior is safe; if not, apply Option A.

**Additional hardening regardless of option chosen:**

- Add rate limiting to `/api/candles`, `/api/carnage-events`, and `/api/health` — these routes are currently unprotected and represent a larger immediate risk.
- Add a batch-size cap to the RPC proxy (e.g., max 10 items per batch array) to limit amplification even if per-IP limits are bypassed.
