### 8. Program keypairs themselves do NOT grant upgrade authority (architectural note)

Important nuance for accuracy: on Solana, the program keypair (the address keypair) is only needed at INITIAL deployment time to sign as the program ID being registered. After deployment, the upgrade authority recorded in the ProgramData account is what controls upgrades — not possession of the program address keypair itself. The critical key for upgrade authority is `devnet-wallet.json` (the deployer), not the program-address keypairs.

However, program address keypairs remain sensitive for a different reason: knowing a program's private key could enable replacing a program at that same address on a fresh deploy, and for the bonding-curve program this has additional significance (anyone running their own devnet instance with this keypair creates a colliding address).

## Attack Path

### Path A: Devnet Program Replacement (Primary)

1. Attacker clones the public repository (GitHub: `MetalLegBob/dr-fraudsworth`)
2. Extracts `keypairs/devnet-wallet.json` (64-byte JSON array, no encryption)
3. Constructs `anchor.Wallet(Keypair.fromSecretKey(...))` or `solana program deploy` using this keypair
4. Deploys a malicious program binary to any of the 6 live devnet program IDs using `BPFLoaderUpgradeable` upgrade instruction, signing with devnet-wallet.json
5. Malicious program executes arbitrary logic — e.g., drainer that transfers all tokens from interacting wallets, forged epoch transitions, manipulated AMM reserves

**Exploit command:**
```bash
solana program deploy malicious.so \
  --program-id keypairs/amm-keypair.json \
  --keypair keypairs/devnet-wallet.json \
  --url devnet
```

This is a one-step, no-exploit exploit. The private key is in plaintext in git.

### Path B: Squads Multisig Compromise (Future / Post-Transfer)

1. Attacker obtains `squads-signer-1.json` + `squads-signer-2.json` from git
2. Waits for `transfer-authority.ts` to be run (authority transferred to Squads vault PDA)
3. Uses any two of the three committed Squads signer keys to reach 2-of-3 threshold
4. Proposes and approves a transaction to transfer upgrade authority to attacker-controlled key or execute arbitrary program upgrade directly through Squads
5. Upgrades programs with malicious bytecode

**Effect:** Squads governance that was intended as the security upgrade over direct key control is rendered equally compromised, because all three signer private keys are publicly readable from git.

### Path C: Social Engineering / Devnet Discredit

1. Replace devnet programs with modified versions that emit misleading on-chain events
2. Frontend displays manipulated data (prices, epoch state, staking rewards)
3. Used to discredit the project with screenshots or to confuse devnet testers

## Impact

**Devnet scope (confirmed):** All 6 deployed devnet programs can be upgraded by anyone with repo access. All devnet protocol state (pools, staking, epochs) can be corrupted or replaced.

**Mainnet scope (NOT yet impacted):** Mainnet uses separate keypairs (`keypairs/mainnet-*`) that ARE gitignored. Mainnet deployer wallet (`23g7x...`) is separate and not in this repo. As long as the mainnet deploy uses fresh, never-committed keypairs, mainnet is safe.

**Risk amplifier — Squads keys committed:** The Phase 97 governance work added Squads signer keypairs to git, meaning the intended security upgrade (devnet governance rehearsal) is itself compromised on devnet. Any devnet Squads governance rehearsal provides false assurance, as an attacker could replicate every "successful" governance action.

**Risk amplifier — Git history permanence:** Even if all keypairs are removed from HEAD today, they remain extractable from git history. Every historical commit that touched `keypairs/` contains the full private key bytes. `git log --all` is sufficient to recover them. BFG Repo-Cleaner or `git filter-repo` with a complete force-push is required to actually remove them.

**Severity assessment:** HIGH for devnet (direct, immediate). LOW for mainnet (correct gitignore pattern in place, separate keys). The overall finding severity is HIGH because the pattern demonstrates insufficient key hygiene that could regress to mainnet if procedural controls fail.

## Recommendation

### Immediate

1. Accept that existing devnet keys are permanently exposed in git history. Rotate all committed devnet keypairs — generate fresh ones and redeploy. Do NOT simply git-rm; the old keys in history are still valid.

2. Add a broad gitignore rule for all keypairs:
   ```gitignore
   keypairs/*.json
   ```
   Keep the existing `keypairs/mainnet-*` rule as belt-and-suspenders.

3. Purge git history using `git filter-repo` or BFG Repo-Cleaner to remove all keypair JSON files from all historical commits. Force-push. Notify any collaborators to re-clone.

4. For Squads devnet rehearsal: generate new Squads signer keypairs that are NOT committed. Store them locally only, outside the repo, or use password-protected keystores.

### Before Mainnet (CRITICAL GATE — already planned, verify execution)

5. Verify mainnet deploy uses ONLY non-committed keypairs. The `keypairs/mainnet-*` gitignore is correct but procedural — confirm the mainnet deploy scripts never fall back to committed devnet keypairs when given a mainnet cluster URL.

6. Run `transfer-authority.ts` with a fresh, non-committed Squads multisig setup so that the devnet governance rehearsal provides meaningful security assurance.

### Long-term

7. Consider using hardware wallets or HSMs for upgrade authority keys even on devnet — this prevents accidental committing because the private key never exists as a file.

8. Add a pre-commit hook that scans for keypair JSON files (arrays of 64 integers) using a pattern like `git-secrets` or `gitleaks`, blocking commits that contain them.
