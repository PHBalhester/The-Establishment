## Summary

`skipPreflight: true` is present on all bonding curve buy and sell transactions
(`BuyForm.tsx:191`, `SellForm.tsx:200`). The flag is unchanged since the original
implementation (commit `eb38997`) and was not addressed by the H039-period changes
to `BuySellPanel.tsx`. No code comment provides a justification for this choice in
the bonding curve path.

## Verdict: CONFIRMED

The flag persists without justification. This is distinct from the multi-hop case
and warrants a separate fix.

---

## Evidence

### BuyForm.tsx:188–192

```typescript
const signature = await sendTransaction(tx, connection, {
  skipPreflight: true,
});
```

### SellForm.tsx:197–201

```typescript
const signature = await sendTransaction(tx, connection, {
  skipPreflight: true,
});
```

### Transaction type: legacy (not v0)

Both BC transactions are built as standard legacy `Transaction` objects:

```typescript
const tx = new Transaction();
tx.add(instruction);
tx.feePayer = publicKey;
tx.recentBlockhash = blockhash;
```

This is a single-instruction legacy TX — not a VersionedTransaction (v0). The
documented justification for `skipPreflight:true` elsewhere in the codebase
**explicitly applies only to v0 transactions**:

> `multi-hop-builder.ts:20–21` — *"v0 TX + skipPreflight: Devnet simulation
> rejects v0 TX with stale blockhash. We use skipPreflight:true and check
> confirmation.value.err instead."*

Legacy transactions do not suffer from the "Blockhash not found" simulation
rejection that affects v0 TXs on devnet. There is therefore **no technical
reason** why BC buy/sell TXs require `skipPreflight:true`.

### Git history confirms no fix attempted

```
de83928  feat(94): fix frontend display bugs + add curve state polling + refund UX
```

The only changes to `BuyForm.tsx` and `SellForm.tsx` in commit `de83928` were:
- Adding an `onTxConfirmed` callback prop
- Calling `onTxConfirmed?.()` after successful TX

The `skipPreflight` option was not touched. The H039 "BuySellPanel.tsx modified"
note referred to unrelated panel-level changes.

## Risk Assessment

**What simulation catches that on-chain execution does not catch cheaply:**
- Anchor account constraint violations (wrong PDA, wrong owner) before paying tx fee
- Transfer Hook account errors (missing/wrong extra_account_meta_list)
- Bonding curve cap overflow (wallet token cap, curve completion check)
- Insufficient balance for SOL vault transfer

Without preflight simulation, a malformed transaction (e.g., after a program
upgrade that changes account layout, or a stale PDA derivation) burns the user's
tx fee and only reports failure via `pollTransactionConfirmation.result.err`.
The user experience is: "Transaction failed" with a parsed curve error code —
worse than catching the error pre-submission.

**On-chain safety net:** `minimumOutput` is enforced on-chain, so fund loss from
slippage manipulation is prevented regardless of skipPreflight. The risk is
wasted tx fees and degraded error UX, not loss of funds.

**Severity justification (Tier 2 / HIGH):**
The bonding curve handles direct SOL transfers from users. Bypassing simulation
means a bug in the BC program upgrade path or a stale PDA would silently drain
tx fees and produce opaque errors. Combined with the absence of ComputeBudget
instructions (separate finding), congestion can compound the impact. Under normal
conditions the risk is low; under program changes or network stress the degraded
UX becomes a real user harm.

## Comparison: BC vs Multi-Hop skipPreflight

| Property                     | Bonding Curve (BuyForm/SellForm) | Multi-Hop (executeAtomicRoute) |
|------------------------------|----------------------------------|-------------------------------|
| TX type                      | Legacy Transaction               | VersionedTransaction (v0)     |
| Justification in code        | None                             | Documented (devnet v0 issue)  |
| Simulation rejection on devnet? | No (legacy TX simulates fine)  | Yes ("Blockhash not found")   |
| `confirmation.err` checked?  | Yes (`result.err`)               | Yes (`confirmation.err`)      |
| Should skipPreflight = true? | **No**                           | Arguable (devnet workaround)  |

## Recommended Fix

Remove `skipPreflight: true` from both BC transaction submissions, allowing the
RPC to simulate the transaction before broadcasting. Simulation failures should
surface to the user via `parseCurveError` in the catch block.

```typescript
// BuyForm.tsx:190 — BEFORE
const signature = await sendTransaction(tx, connection, {
  skipPreflight: true,
});

// BuyForm.tsx:190 — AFTER
const signature = await sendTransaction(tx, connection, {
  skipPreflight: false,
  preflightCommitment: 'confirmed',
});
```

Apply the identical change to `SellForm.tsx:199`.

If simulation rejections are observed on devnet for legacy BC TXs (unexpected),
document the specific error and apply a targeted workaround rather than a blanket
`skipPreflight:true`.
