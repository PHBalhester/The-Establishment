# H003: npm Supply Chain Attack via Crank Build — Partial Fix (npm install vs npm ci)

## Verdict: POTENTIAL

---

## Evolution

**Audit #1 status:** CONFIRMED (CRITICAL) — root cause was two-part: (1) `package-lock.json` gitignored, defeating all version pinning; (2) `railway-crank.toml` uses `npm install` not `npm ci`.

**Fix claimed in Audit #1 (commit 807ba9e, 2026-03-09):** Added `.npmrc` with `ignore-scripts=true` and committed `package-lock.json` to git. Described as "H003 fix" in commit message.

**Audit #2 finding:** The fix is **incomplete**. `railway-crank.toml` still reads `buildCommand = "npm install"` as of HEAD (`dc063ec`). The commit message claimed H003 fixed but the primary attack vector — `npm install` instead of `npm ci` — was never patched in `railway-crank.toml`.

---

## Evidence

### What the fix DID address

1. **`package-lock.json` is now committed** — `git ls-files package-lock.json` returns `package-lock.json`. The lockfile is tracked, not gitignored. Audit #1 showed it was at `.gitignore:9`; that line has been removed.

2. **`.npmrc` has `ignore-scripts=true`** — Added in commit 807ba9e. Blocks all 13 packages that declare lifecycle scripts (`protobufjs`, `sharp`, `esbuild`, `bigint-buffer`, `bufferutil`, `keccak`, `secp256k1`, `utf-8-validate`, `fsevents`, `unrs-resolver`). This applies to both `npm ci` and `npm install`.

### What the fix DID NOT address

**`railway-crank.toml:3` still reads `buildCommand = "npm install"`:**

```toml
[build]
builder = "NIXPACKS"
buildCommand = "npm install"   # ← UNCHANGED since initial commit be95eba
```

Git history confirms: `railway-crank.toml` has exactly ONE commit — `be95eba` (Feb 22, 2026, initial creation). The H003 fix commit `807ba9e` touched `.npmrc`, `app/db/connection.ts`, `app/next.config.ts`, and `scripts/e2e/lib/swap-flow.ts` — but not `railway-crank.toml`.

### Behavioral difference: `npm install` vs `npm ci` with a committed lockfile

With npm v7+ (confirmed: npm 11.3.0 in local environment):

| Behavior | `npm ci` | `npm install` |
|----------|----------|---------------|
| Enforces exact lockfile match | **Yes** — errors and aborts if `package-lock.json` would change | **No** — updates lockfile if resolution differs |
| Installs different version if lockfile and `package.json` disagree | **No** — fails hard | **Yes** — installs what `package.json` range allows |
| Protects against lockfile tampering on build host | **Yes** | **No** |
| Deletes and reinstalls `node_modules` | **Yes** | **No** |

Concretely: if an attacker who has write access to the Railway service (e.g., via compromised Railway account, Railway team member, or Railway platform compromise) modifies a Railway environment variable or triggers a fresh build after modifying a dependency version, `npm install` will resolve freely from the caret-ranged deps in `package.json` (e.g., `"@coral-xyz/anchor": "^0.32.1"`, `"@switchboard-xyz/on-demand": "^3.7.3"`). `npm ci` would reject any such resolution that deviates from the committed lockfile.

Additionally, on the crank build specifically: `npm install` does NOT do a clean install. If Railway's Nixpacks build layer caches `node_modules` between builds, a poisoned `node_modules` from a previous compromised build could persist. `npm ci` always deletes and reinstalls `node_modules` from scratch.

## Attack Path

The original CRITICAL path (postinstall script execution) is now **blocked** by `ignore-scripts=true`. However, two residual attack paths remain:

### Path A: Lockfile Drift / Version Bump (LOWER likelihood, same impact)

```
1. Attacker publishes malicious version of a caret-ranged dep
   e.g., @switchboard-xyz/on-demand@3.7.4 with malicious module-body code
   (NOT postinstall — ignore-scripts blocks that)

2. The committed package-lock.json still pins 3.7.3 initially.
   BUT — if any legitimate developer runs `npm install` locally and
   the lockfile updates to 3.7.4, then commits the updated lockfile,
   the malicious version is now pinned.

   OR: `npm install` on the Railway build host could theoretically
   update the lockfile in the build container (changes don't persist,
   but the resolution for that build uses the new version).

   NOTE: In practice, npm install respects the lockfile when
   package-lock.json is present and the dep is within range — so
   active lockfile drift requires the lockfile to be modified first.
   This path has lower likelihood than Audit #1 assessed.
```

### Path B: Module-Body Code Injection (NOT blocked by ignore-scripts)

```
1. Attacker publishes malicious patch of any crank dependency with
   malicious code in the module body (not in install scripts)
   e.g., evil code in index.js that runs when the module is imported.

2. `npm install` does NOT guarantee the pinned lockfile is used if
   `package.json` range allows a newer version AND the build container
   does not have a clean node_modules.

3. Malicious module-body code executes at import time when
   crank-runner.ts starts: `import * as anchor from "@coral-xyz/anchor"`

4. Code reads process.env.WALLET_KEYPAIR (full 64-byte JSON array) and
   exfiltrates via outbound HTTP/HTTPS (no Railway egress filtering).

5. Attacker signs arbitrary Solana transactions with the crank wallet,
   draining SOL and compromising epoch/carnage operations.
```

**Key distinction from Audit #1:** `ignore-scripts=true` blocks **install-time** code execution (postinstall hooks). It does NOT block **runtime** code execution in module bodies. If a dependency's published source code itself is malicious, `ignore-scripts` provides zero protection. `npm ci` vs `npm install` is the only build-time defense for ensuring the exact pinned version is what gets imported.

---

## Impact

**If crank lockfile drifts to malicious version (Path A or B):**
- `WALLET_KEYPAIR` env var (full 64-byte Solana keypair) exfiltrated
- Crank wallet drained (any SOL balance, including protocol vault top-up funds)
- Epoch transitions can be manipulated (crank is the only practical caller)
- Carnage buyback-and-burn operations can be manipulated or blocked
- All other Railway env vars exposed: `DATABASE_URL`, `HELIUS_API_KEY`, `HELIUS_WEBHOOK_SECRET`

**Severity reduction from Audit #1:** Downgraded from CONFIRMED CRITICAL to POTENTIAL because:
1. Committed `package-lock.json` significantly raises the bar — a drift attack now requires either lockfile modification or a caret-range dep having a malicious version published WITHIN the pinned range (npm will still use the lockfile for exact-match deps, and will error on strict mismatches in npm v7+)
2. `ignore-scripts=true` eliminates the most trivially exploitable vector (postinstall scripts)
3. The residual risk requires a module-body supply chain attack (harder to execute without detection than a postinstall script)

The vulnerability is not eliminated — it is substantially mitigated by the partial fix, but the single-line change from `npm install` to `npm ci` in `railway-crank.toml` was the primary recommendation in Audit #1 and was not applied.

## Recommendation

### Immediate (one-line fix)

Change `railway-crank.toml:3`:

```toml
# Before (current state — Audit #1 fix was incomplete)
[build]
buildCommand = "npm install"

# After (completes the Audit #1 fix)
[build]
buildCommand = "npm ci"
```

`npm ci` with `ignore-scripts=true` in `.npmrc` provides defense-in-depth:
- `npm ci` enforces the committed `package-lock.json` exactly (fails if lockfile would change)
- `ignore-scripts=true` blocks postinstall execution
- Together: guarantees the exact versions from the committed lockfile are installed, with no script execution

### Secondary hardening

1. **Verify main app Railway build also uses `npm ci`**: `railway.toml` does not specify an install command — it relies on Nixpacks auto-detection. Nixpacks uses `npm ci` when a `package-lock.json` is present, but this is not guaranteed across Nixpacks versions. Add explicit: `buildCommand = "npm ci && npm run --workspace app build"`.

2. **Add `npm audit --audit-level=high` to CI**: 30 known CVEs (9 high) are invisible to CI. Even one CI gate would surface critical findings before deploy.

3. **Pin exact versions in `package.json`**: Remove `^` from `@coral-xyz/anchor`, `@switchboard-xyz/on-demand`, `@solana/web3.js` in root `package.json`. With a committed lockfile AND exact pinning, supply chain drift requires lockfile modification, which is detectable in code review.
