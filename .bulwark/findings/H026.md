# H026: Devnet Fallback in Production (RECHECK H009)

## Verdict: CONFIRMED — Partially Mitigated, Residual Risk Remains

---

## Evidence

### connection.ts — Devnet fallback ELIMINATED

`app/lib/connection.ts` no longer contains any devnet fallback. The `resolveRpcUrl()` function (lines 30–46) has two paths:

1. **Browser**: returns `${window.location.origin}/api/rpc` — proxied, cluster-agnostic.
2. **Server**: reads `process.env.HELIUS_RPC_URL ?? process.env.NEXT_PUBLIC_RPC_URL`. If neither is set, it **throws** (`"HELIUS_RPC_URL or NEXT_PUBLIC_RPC_URL env var is required for server-side RPC"`) — a hard fail-closed stance. No devnet URL is hardcoded anywhere in the file.

H009's core fix (removal of a hardcoded devnet fallback in `connection.ts`) is confirmed intact. `connection.ts` is CLEAN.

---

### protocol-config.ts — Silent devnet fallback PERSISTS

`app/lib/protocol-config.ts:25`:
```typescript
const rawCluster = process.env.NEXT_PUBLIC_CLUSTER || "devnet";
```

If `NEXT_PUBLIC_CLUSTER` is absent or empty at build time, the entire address layer — mints, pools, PDAs, program IDs, treasury, ALT — silently resolves to devnet values. There is no throw, no warning, and no startup assertion. The fallback is unconditional.

---

### next.config.ts — Silent devnet fallback PERSISTS

`app/next.config.ts:10`:
```typescript
const cluster = process.env.NEXT_PUBLIC_CLUSTER || "devnet";
```

The same pattern controls which Helius RPC domain is injected into the Content Security Policy (`connect-src`). If `NEXT_PUBLIC_CLUSTER` is missing on a mainnet build, the CSP whitelists `devnet.helius-rpc.com` but not `mainnet.helius-rpc.com`. All user RPC calls through the proxy succeed (the proxy uses `HELIUS_RPC_URL`, which is a separate server-only var and does throw on missing), but address resolution and CSP both point at devnet.

---

### ClusterConfigProvider.tsx — Silent devnet fallback PERSISTS

`app/providers/ClusterConfigProvider.tsx:27`:
```typescript
const raw = process.env.NEXT_PUBLIC_CLUSTER || "devnet";
```

The React provider that supplies cluster addresses to all client-side hooks carries the same pattern. On any mainnet deploy where the env var is accidentally omitted, every swap, stake, and bonding-curve transaction targets devnet accounts because the provider supplies devnet `MINTS`, `DEVNET_POOLS`, `DEVNET_PDAS`, and `PROGRAM_IDS` to every hook.

---

### getClusterConfig() — Fallback-on-unknown PERSISTS

`shared/constants.ts:565–568`:
```typescript
export function getClusterConfig(cluster: string): ClusterConfig {
  if (cluster === "mainnet-beta") return CLUSTER_CONFIG["mainnet-beta"];
  return CLUSTER_CONFIG.devnet;
}
```

Any cluster string that is not exactly `"mainnet-beta"` silently returns the devnet config. There is no `throw` for unrecognized cluster names. If `NEXT_PUBLIC_CLUSTER` were set to a typo such as `"mainnet-beta "` (trailing space), `"Mainnet-Beta"` (wrong case), or `"mainnet"` (handled by the callers via a remapping step in `protocol-config.ts` and `ClusterConfigProvider.tsx`, but not by `getClusterConfig` itself), the result is silent devnet.

The Railway MEMORY note marks `NEXT_PUBLIC_CLUSTER=mainnet on Railway is CRITICAL` — this is operator knowledge only, not enforced in code.

## Attack Path

**Scenario A — Accidental misconfiguration**

1. Railway mainnet service is redeployed after an env var audit clears/resets `NEXT_PUBLIC_CLUSTER`.
2. Next.js bakes the build-time fallback `"devnet"` into all NEXT_PUBLIC_ inlined values.
3. `protocol-config.ts`, `ClusterConfigProvider`, and `next.config.ts` all resolve devnet addresses.
4. All user swap and staking transactions are constructed with devnet mint/pool addresses.
5. Transactions submitted to mainnet Solana via `/api/rpc` (which correctly points at mainnet Helius RPC via `HELIUS_RPC_URL`) fail with "account not found" or target incorrect on-chain state.
6. Worse: if devnet mints happen to exist on mainnet as random accounts, swap instructions could silently land in unexpected program state.
7. Users lose gas fees; protocol appears non-functional; no server-side error or alarm fires.

**Scenario B — Build cache poisoning**

1. Previous Railway build artifact has devnet inlined (legitimate devnet test run).
2. Railway cache is not invalidated when `NEXT_PUBLIC_CLUSTER=mainnet` is set.
3. The cached bundle with devnet addresses is deployed to mainnet.
4. Same outcome as Scenario A, but more insidious — env var is set correctly in Railway config but the running code does not reflect it.

**Scenario C — Typo cluster value**

1. `NEXT_PUBLIC_CLUSTER` is set to `"Mainnet"` or `"mainnet-beta"` (with wrong remapping path in `protocol-config.ts`: `rawCluster === "mainnet" ? "mainnet-beta" : rawCluster` — the `rawCluster === "mainnet-beta"` case is NOT remapped, so it passes `"mainnet-beta"` directly, which does work correctly; but `"Mainnet"` with capital M would fall through to `getClusterConfig("Mainnet")`, which silently returns devnet config).
2. All three fallback layers silently return devnet config; no error is logged.

## Impact

**Severity: MEDIUM-HIGH** (CONFIRMED, partially mitigated)

- **Primary impact**: All client-constructed transactions target devnet program IDs and token mints on a mainnet deployment. Transactions fail or, in pathological cases, hit unexpected accounts.
- **Financial impact**: Gas SOL lost per failed transaction. No protocol funds at direct risk because the on-chain programs are not themselves affected — the risk is entirely in client address resolution.
- **UX impact**: Entire frontend appears broken. Support burden high. Reputation risk on mainnet launch.
- **Detection difficulty**: No server-side logging fires. The app starts cleanly. The only signals are transaction failures visible to users.
- **Blast radius**: Every hook consuming `useClusterConfig()`, `protocol-config.ts` PROGRAM_IDS, MINTS, or DEVNET_POOLS is affected simultaneously — this is the entire swap, staking, and bonding-curve surface.

The RPC connection layer itself (`connection.ts`) is **not** affected — it correctly throws if `HELIUS_RPC_URL` is absent, providing a hard stop for the data pipeline. The remaining vulnerability is in the **address resolution** layer, which silently falls back.

---

## Recommendation

**R1 — Add startup assertion for NEXT_PUBLIC_CLUSTER (HIGH)**

In `app/lib/protocol-config.ts`, replace the silent fallback with a validated resolution:

```typescript
const rawCluster = process.env.NEXT_PUBLIC_CLUSTER;
if (!rawCluster && process.env.NODE_ENV === "production") {
  throw new Error("NEXT_PUBLIC_CLUSTER must be set in production");
}
const resolvedCluster = rawCluster || "devnet";
```

This aligns with the fail-closed pattern already established in `connection.ts`.

**R2 — Harden getClusterConfig() against unknown values (MEDIUM)**

In `shared/constants.ts`, throw on unrecognized cluster names rather than silently returning devnet:

```typescript
export function getClusterConfig(cluster: string): ClusterConfig {
  if (cluster === "mainnet-beta") return CLUSTER_CONFIG["mainnet-beta"];
  if (cluster === "devnet") return CLUSTER_CONFIG.devnet;
  throw new Error(`Unknown cluster: "${cluster}". Expected "devnet" or "mainnet-beta".`);
}
```

**R3 — Apply same pattern to ClusterConfigProvider and next.config.ts (MEDIUM)**

Mirror the production assertion in `ClusterConfigProvider.tsx:27` and log a warning (or throw) in `next.config.ts:10` when `NEXT_PUBLIC_CLUSTER` is unset during a production build.

**R4 — Add Railway pre-deploy check (LOW)**

Add a `scripts/check-env.ts` that validates required env vars are present and have expected values. Run as a Railway pre-deploy step (`preDeployCommand` in `railway.toml`). Include `NEXT_PUBLIC_CLUSTER`, `HELIUS_RPC_URL`, `HELIUS_WEBHOOK_SECRET`, and `DATABASE_URL`.
