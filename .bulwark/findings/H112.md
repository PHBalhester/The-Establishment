# H112: autoConnect with Empty Adapter List

**Severity:** Tier 3 (Accepted Risk)
**Component:** `app/providers/providers.tsx:43-47`
**Category:** Frontend Wallet Integration
**Type:** Malicious Extension Replacement Attack

---

## Vulnerability Description

The `WalletProvider` is configured with:
- An empty wallets array (`const wallets = useMemo(() => [], [])`)
- The `autoConnect` flag enabled (`<WalletProvider wallets={wallets} autoConnect>`)

When `autoConnect=true`, Solana's wallet-adapter library attempts to reconnect to the previously-connected wallet by reading its name from `localStorage`. If a user switches browser sessions or reinstalls a malicious browser extension with the same name as a previously-trusted wallet (e.g., a compromised "Phantom" extension), the `autoConnect` flow could silently reconnect the user to the malicious extension.

---

## Attack Scenario

1. User installs legitimate Phantom wallet and connects to Dr. Fraudsworth
   - `localStorage` stores: `{ walletName: "Phantom" }`

2. Between sessions:
   - User's system is compromised
   - Attacker installs fake "Phantom" extension with matching adapter name
   - Legitimate extension is removed or disabled

3. User returns to Dr. Fraudsworth on the compromised system
   - `autoConnect=true` reads localStorage
   - WalletProvider calls `select("Phantom")` automatically
   - Fake extension is reconnected without user consent
   - User unknowingly connects to attacker-controlled keys

---

## Risk Assessment

**Why This Is Mitigated:**

1. **Wallet-Standard Protocol**: The codebase uses wallet-standard auto-discovery (empty wallets array forces this). The wallet-adapter library filters detected wallets via the `getCapabilities()` API, which verifies cryptographic capability checks before allowing connection.

2. **User Agency**: On desktop, no wallets are pre-loaded. Users must explicitly:
   - Install a wallet extension
   - See the ConnectModal, which shows only "Installed" or "Loadable" wallets
   - Click their wallet name to initiate connection

   Even with `autoConnect`, the reconnection happens only if the wallet was genuinely installed before.

3. **Mobile Deep-Links**: On mobile, wallet-standard auto-detect only works inside the wallet's own in-app browser (not during app-to-browser navigation). Deep links are explicit user actions.

4. **No Implicit Approval**: Dr. Fraudsworth doesn't send automatic transactions. Connection alone doesn't grant transaction signing authority. All swap/stake operations require explicit user confirmation via the wallet UI.

## Technical Details

From `app/providers/providers.tsx`:
```tsx
// Line 40-43: Empty wallets array forces wallet-standard auto-detection
const wallets = useMemo(() => [], []);

// Line 47: autoConnect enabled
<WalletProvider wallets={wallets} autoConnect>
```

From `app/components/wallet/ConnectModal.tsx`:
```tsx
// Line 57-59: Only shows detected (installed) wallets
const detectedWallets = wallets.filter(
  (w) => w.readyState === "Installed" || w.readyState === "Loadable",
);
```

**Why autoConnect is acceptable here:**
- `autoConnect` reconnects only to wallets that were previously installed AND still present
- The wallet-adapter library validates that the reconnected adapter implements the required Solana Wallet Standard interface
- A malicious extension would need to:
  1. Match the exact adapter name of a previously-trusted wallet
  2. Implement the wallet-standard capability interface
  3. Be installable on the user's system without the user noticing two conflicting extensions

---

## Verdict

**ACCEPTED_RISK**

The `autoConnect` feature is safely implemented because:
1. Auto-reconnection only occurs to wallets that are currently installed and registered via wallet-standard
2. No implicit transaction signing occurs after auto-connection
3. User still controls all transaction approvals through the wallet UI
4. The risk of malicious extension replacement is mitigated by the requirement that the fake extension must implement the full wallet-standard interface
5. If a user is compromised enough to install a malicious extension with a matching adapter name, other attack vectors (system-level key logging, malware) are already present

**Mitigation in Place:** Wallet-standard protocol validation + explicit user confirmation per transaction.
