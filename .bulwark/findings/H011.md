# H011: Enhanced Webhook Replay — Stale State Injection via Unprotected Account-Change Path

## Verdict: CONFIRMED (HIGH) → **FIXED** (2026-03-25)

### Fix Summary
Slot-monotonic freshness check added to `handleAccountChanges()` in `route.ts`.
Helius enhanced webhook payloads include `slot: number` — the handler now rejects
any payload with `slot < lastAcceptedSlot[account]`. Watermarks are seeded during
`batchSeed()` with the current on-chain slot so replays of pre-startup data are
also rejected. 16 unit tests cover all edge cases (same-slot, undefined slot,
replay after seed, multi-account independence).

**Files changed:** `protocol-store.ts` (lastSlots map + getter/setter),
`route.ts` (interface + freshness check + watermark update),
`ws-subscriber.ts` (batchSeed watermark seeding).

## Evidence

### 1. Type Discrimination Gate (route.ts:336-342)

```typescript
const firstItem = payload[0] as Record<string, unknown> | undefined;
if (firstItem && "accountData" in firstItem) {
  return handleAccountChanges(payload as HeliusAccountChange[]);
}
```

Any payload whose first element has an `accountData` key is routed to `handleAccountChanges` and returns immediately — it never reaches the raw-TX path.

### 2. Raw TX Path Has Replay Protection; Enhanced Path Does Not (route.ts:359-388)

```typescript
// H049: Maximum age for raw transactions (5 minutes).
// Enhanced account change webhooks skip this (handled separately above).
const MAX_TX_AGE_SECONDS = 300;
for (const tx of transactions) {
  if (blockTime != null) {
    const age = Math.floor(Date.now() / 1000) - blockTime;
    if (age > MAX_TX_AGE_SECONDS) { continue; }
  }
}
```

The comment explicitly documents the gap: the 5-minute blockTime guard only applies to the raw TX path. The enhanced path (`handleAccountChanges`) has already returned before this loop executes.

### 3. HeliusAccountChange Type Has No Temporal Fields (route.ts:191-208)

```typescript
interface HeliusAccountChange {
  account: string;
  accountData: { account: string; nativeBalanceChange?: number; ... }[];
  rawAccountData?: { data: string; encoding: string; };
}
```

No `slot`, no `blockTime`, no sequence number. The interface carries zero information that would allow a freshness check inside `handleAccountChanges`.

### 4. protocolStore.setAccountState Has No Monotonicity Guard (protocol-store.ts:53-65)

```typescript
setAccountState(pubkey: string, data: AccountState): void {
  const serialized = JSON.stringify(data, bigintReplacer);
  this.accounts.set(pubkey, data);  // unconditional overwrite

  if (serialized === this.lastSerialized.get(pubkey)) return;
  this.lastSerialized.set(pubkey, serialized);
  sseManager.broadcast("protocol-update", { account: pubkey, data });
}
```

The dedup guard (line 58) compares serialized content for equality, not recency. If the replayed payload differs from current state (which it will — it is intentionally stale), it **passes the dedup check** and triggers an SSE broadcast. There is no slot comparison, no `updatedAt` comparison against any stored timestamp, and no monotonic sequence enforcement.

### 5. `updatedAt` Is Server Arrival Time, Not Account Slot (route.ts:603-606)

```typescript
protocolStore.setAccountState(pubkey, {
  ...normalized,
  updatedAt: Date.now(),   // wall-clock of server receipt
});
```

`updatedAt` is set at the moment the server processes the payload. It reflects when the server received the webhook, not when the account was last written on-chain. A replayed payload from 24 hours ago arrives today and gets `updatedAt = now`. The store has no prior-state timestamp to compare against for "is this older than what I already have?".

### 6. Evolution from Audit #1 H049

Audit #1 (VERIFY-H049, FIXED) closed the raw-TX replay vector by adding `MAX_TX_AGE_SECONDS`. The enhanced webhook path was noted as handled "separately above" — but the separate handler was never given equivalent protection. The fix was incomplete: it closed one door and left the other open.

## Attack Path

**Prerequisites:** Attacker has the `HELIUS_WEBHOOK_SECRET` (it is committed to `.env.devnet` per SEC-01/SEC-02 findings, and the value is stored in git history) OR is attacking during a window when `NODE_ENV != "production"` (auth skipped when secret unset in non-production).

```
Step 1 — Capture payload
  Attacker captures one legitimate Enhanced Account Change delivery.
  Source: Helius dashboard (webhook delivery logs), network interception,
          or the committed .env.devnet Helius API key (which allows listing
          webhook deliveries via the Helius management API).
  Result: A valid JSON array with accountData elements for one or more PDAs.
  The payload contains the Anchor-decoded account bytes for that point in time.

Step 2 — Wait for state to advance
  Let the protocol operate: swaps happen, an epoch transition occurs, taxes
  update, pool reserves change. The captured payload is now materially stale.
  Example: epoch N=5 data captured; protocol is now at epoch N=7.

Step 3 — Replay the stale payload indefinitely
  POST /api/webhooks/helius
  Authorization: <captured/committed secret>
  Content-Type: application/json
  Body: <captured enhanced payload from Step 1>

  The handler:
  a. Auth passes (correct secret or non-prod)
  b. "accountData" in firstItem → routes to handleAccountChanges()
  c. No timestamp check, no slot check
  d. Anchor-decodes the old account bytes
  e. protocolStore.setAccountState() called with stale data
  f. Dedup check: stale data != current data → PASSES → SSE broadcast fires
  g. All connected browser clients receive old epoch/pool/curve state

Step 4 — State injection confirmed
  protocolStore.accounts now holds stale data for affected PDAs.
  New SSE clients receive stale initial-state snapshot.
  Existing clients receive stale protocol-update event.

Step 5 — Sustain the injection
  Replay continuously (1 req/s to stay under 120/min rate limit).
  Each ws-subscriber poll (30s for supply, 10s staleness check) may
  overwrite with fresh data — but replaying faster than the poll interval
  creates a persistent stale-state window.
```

---

## Impact

**Data pipeline integrity compromise.** The protocol store is the single source of truth for all SSE-fed UI state. Successful injection affects:

- **EpochState**: Users see wrong epoch number, wrong tax rates (buy/sell BPS). Quote engine uses SSE-delivered tax rates → mispriced quotes. On-chain `minimumOutput` enforcement is the backstop, but the displayed quote diverges from reality.
- **PoolState (CRIME_SOL / FRAUD_SOL)**: Users see wrong reserve ratios → wrong price display, wrong quote, wrong chart. Stale reserves cause `minimumOutput` to be computed from an old price, potentially giving users worse execution than expected.
- **CurveState (CRIME / FRAUD)**: Stale bonding curve state (wrong current price, wrong supply remaining) misleads bonding curve buyers.
- **StakePool**: Wrong staked supply totals in the UI.
- **CarnageFundState**: Wrong Carnage fund balance displayed.

**Severity calibration:**
- No direct fund loss: on-chain enforcement (minimumOutput) is intact.
- MEV amplification: Attacker with SSE access (unauthenticated) sees the real state from ws-subscriber while all users see stale state. Structural information asymmetry.
- Trust erosion: Persistent display of incorrect prices and epoch state undermines user confidence.
- Not a DoS: The replayed data is accepted and stored; the server does not crash.

**Severity: HIGH** — Persistent, authenticated data injection into the protocol's primary state pipeline. Impacts all connected users' displayed state, amplifies MEV asymmetry, and requires only a captured webhook payload to execute.

## Recommendation

### Primary Fix — Slot-Based Monotonicity Guard in protocolStore

Enhanced Account Change payloads from Helius include the slot at which the account was observed. Add `slot` to the `HeliusAccountChange` interface and thread it through to the store:

```typescript
// 1. Extend the interface
interface HeliusAccountChange {
  account: string;
  slot?: number;          // add this — Helius includes it on enhanced payloads
  accountData: { ... }[];
  rawAccountData?: { ... };
}

// 2. In protocolStore, track last-seen slot per key
private lastSlot = new Map<string, number>();

setAccountState(pubkey: string, data: AccountState, slot?: number): void {
  if (slot !== undefined) {
    const prevSlot = this.lastSlot.get(pubkey) ?? 0;
    if (slot <= prevSlot) return;  // reject stale-or-equal slot
    this.lastSlot.set(pubkey, slot);
  }
  // ... existing dedup + broadcast logic
}

// 3. In handleAccountChanges, pass slot
protocolStore.setAccountState(pubkey, { ...normalized, updatedAt: Date.now() }, item.slot);
```

This makes the store monotonically advancing: replaying an old payload (lower slot) is silently rejected.

### Secondary Fix — Payload-Level Timestamp Check

If Helius does not reliably include `slot` on enhanced payloads, add a server-side wall-clock guard using `updatedAt`:

```typescript
// In protocolStore
private lastUpdatedAt = new Map<string, number>();

setAccountState(pubkey: string, data: AccountState): void {
  const now = Date.now();
  const prev = this.lastUpdatedAt.get(pubkey) ?? 0;
  // Reject if arrived more than 5 minutes before the current store entry
  // (allows for normal Helius delivery latency)
  if (now < prev) return;  // clock skew guard only — not sufficient alone
  this.lastUpdatedAt.set(pubkey, now);
  // ... existing logic
}
```

Note: wall-clock alone is insufficient because a replayed payload arrives *now* and gets `updatedAt = now`. This must be combined with the slot check.

### Tertiary Fix — Nonce / Request Dedup

Maintain a bounded LRU set of recently processed payload hashes. Reject exact-duplicate payloads within a TTL window (e.g., 10 minutes). This is a defense-in-depth layer that blocks byte-for-byte replays even without slot data.
