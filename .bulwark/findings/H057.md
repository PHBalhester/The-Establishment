# H057: migrate.ts Missing TLS

## Verdict: CONFIRMED

---

## Hypothesis Restatement

The DB migration script (`app/db/migrate.ts`) creates a `postgres` client without any SSL
configuration. Railway's `preDeployCommand` runs this script in production before each deployment,
potentially transmitting the DATABASE_URL credentials and every migration SQL statement in plaintext
over the network.

---

## Code Investigation

### Primary target: `app/db/migrate.ts:42`

```typescript
const client = postgres(databaseUrl, { max: 1 });
```

The only option passed is `max: 1` (connection count). There is no `ssl` option, no
`ssl: "require"`, and no conditional TLS enforcement based on `NODE_ENV` or any other signal.
The `postgres.js` driver defaults to `ssl: false` when no `ssl` option is provided and the URL
does not contain `?sslmode=require`. Credentials in the connection string (username, password,
database name) and all SQL in the migration files are therefore sent unencrypted unless the
DATABASE_URL itself happens to include `?sslmode=require`.

### Comparison: `app/db/connection.ts:48-57`

The main application connection intentionally adds TLS in production (comment tag: H011):

```typescript
// H011: Enforce TLS for database connections in production.
const isProductionDb = process.env.NODE_ENV === "production";
const sslConfig = isProductionDb ? { ssl: "require" as const } : {};

const client =
  globalForDb.pgClient ?? postgres(connectionString, { max: 10, ...sslConfig });
```

`connection.ts` also logs a non-fatal warning if a non-production environment connects to a
remote host without TLS (lines 61-73). Neither safeguard is present in `migrate.ts`.

### Railway DATABASE_URL format

Railway injects `DATABASE_URL` as a connection string of the form:
```
postgresql://user:password@host.railway.internal:5432/railway
```

The internal Railway network (`*.railway.internal`) is a private VPC. Railway does not document
whether the internal network enforces encryption at the transport layer. The `postgres.js` driver
requires explicit `ssl: "require"` to negotiate TLS with the server; without it, TLS is not
used even if the server supports it.

The `preDeployCommand` is not a local command ‚Äî it runs inside the Railway build container, which
may be on a different physical host from the Postgres service. The connection traverses Railway's
internal network infrastructure.

### What transmits in plaintext without TLS

When `ssl` is absent and the URL lacks `sslmode=require`:
- The full PostgreSQL handshake, including `user` and `password` from DATABASE_URL
- All SQL statements in the migration files (CREATE TABLE, ALTER TABLE, index creation)
- Row data in any data migrations (none currently, but a future migration could include seed data)

Schema DDL is not sensitive in the same way as row data, but the credentials transmitted during
the handshake are identical to those used by the main application pool. Exposure of those
credentials via a network intercept would grant full database access.

---

## Confirmed Behaviors

1. **`migrate.ts` creates a `postgres` client with zero SSL options** ‚Äî `app/db/migrate.ts:42`
   is `postgres(databaseUrl, { max: 1 })`. There is no `ssl` key anywhere in the file.

2. **`connection.ts` explicitly adds `ssl: "require"` in production** ‚Äî the fix for H011 was
   applied to the main pool but was not propagated to the migration runner.

3. **The gap is structural, not incidental** ‚Äî `migrate.ts` is a standalone script (not a Next.js
   module). It does not import `connection.ts` and has no shared TLS logic. Any future TLS change
   to `connection.ts` would not automatically apply here.

4. **`postgres.js` does not enable TLS by default** ‚Äî without `ssl: "require"`, the driver
   completes the connection without requesting a TLS upgrade, even if the server offers it.

---

## Impact Assessment

**Tier 2 ‚Äî HIGH (as assigned)**

- **Credentials in transit**: DATABASE_URL (username + password) traverses the network in plaintext
  on every Railway deployment. If Railway's internal network can be tapped (e.g., a compromised
  container on the same virtual network, a network monitoring appliance, or a future infrastructure
  misconfiguration), the full database credential is exposed.

- **Scope of credential exposure**: The same DATABASE_URL credential governs the main application
  pool. Credential theft during a migration run grants persistent read/write access to all
  application tables: `swap_events`, `candles`, `epoch_events`, `carnage_events` ‚Äî the full
  persistent audit trail of protocol activity.

- **Frequency**: Every Railway deployment triggers `preDeployCommand`. The migration runner is
  invoked on every deploy, not just when schema changes exist (Drizzle's `migrate()` is a no-op
  if no pending migrations, but the TCP connection and handshake ‚Äî including credential
  transmission ‚Äî still occur).

- **No mitigation in the migration path**: Unlike `connection.ts`, `migrate.ts` has no
  `NODE_ENV` check, no `ssl` option, and no warning log. The credential travels in plaintext with
  no operational visibility.

## Recommendations

**R1 ‚Äî Mirror the `connection.ts` TLS pattern in `migrate.ts`**

Apply `ssl: "require"` unconditionally in the migration runner (migrations always run in
production contexts ‚Äî there is no development use case where a remote DB is reached without TLS).
If local development requires running migrations against localhost, `ssl: "require"` against
`127.0.0.1` either connects successfully (if the local Postgres has TLS) or falls back gracefully:

```typescript
// At app/db/migrate.ts:42 ‚Äî replace:
const client = postgres(databaseUrl, { max: 1 });

// With:
const isProduction = process.env.NODE_ENV === "production";
const client = postgres(databaseUrl, {
  max: 1,
  ...(isProduction ? { ssl: "require" } : {}),
});
```

Alternatively, since migrations are exclusively a production concern, enforce TLS unconditionally:

```typescript
const client = postgres(databaseUrl, { max: 1, ssl: "require" });
```

**R2 ‚Äî Add `?sslmode=require` to the DATABASE_URL template**

If Railway's `DATABASE_URL` for the production Postgres service is set in the Railway dashboard,
append `?sslmode=require` to the URL itself. This provides defence-in-depth: any code that
creates a `postgres.js` client from this URL without an explicit `ssl` option will still request
TLS because the postgres.js driver respects the `sslmode` URL parameter.

**R3 ‚Äî Apply the same fix to any other standalone scripts that create raw `postgres()` clients**

Audit all files under `scripts/` and `app/db/` for additional `postgres(databaseUrl, ...)` calls
that lack `ssl` options. The migration runner is the highest-risk instance (it runs in production),
but other scripts that connect to the production database share the same exposure.

## Evolution

**Stacked finding** ‚Äî This is a gap left by the H011 fix applied to `connection.ts`. H011
addressed TLS enforcement for the main application pool. The migration runner was not in scope for
that fix and was not updated. The two files share the same `postgres.js` driver and the same
DATABASE_URL, but diverge in TLS handling.

**Relationship to prior findings:**
- Directly caused by incomplete scope of H011 fix (`connection.ts` only, not `migrate.ts`)
- Related to ASM-2 (`Railway sets NODE_ENV=production` ‚Äî UNVALIDATED in ARCHITECTURE.md): if
  `NODE_ENV` is absent on Railway, even the `connection.ts` TLS guard would silently fall back to
  plaintext. The `migrate.ts` gap exists regardless of `NODE_ENV`.
- Noted as finding #4 in `19-DATA-01-database-query.md` risk observations
