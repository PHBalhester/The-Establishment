# H104: Log Injection via RPC Method Field

**Location**: `app/app/api/rpc/route.ts:117`
**Verdict**: NOT VULNERABLE
**Severity**: N/A (mitigated by allowlist)

## Issue Description

Line 117 interpolates user-controlled `req.method` into a console.warn log message:

```typescript
console.warn(`[rpc-proxy] Blocked disallowed RPC method: ${req.method}`);
```

This pattern could enable log injection attacks if a malicious method name containing newlines or ANSI escape codes could reach the logger.

## Analysis

The vulnerability path is **blocked by the method allowlist check** at lines 116-122:

```typescript
if (!ALLOWED_METHODS.has(req.method)) {
  console.warn(`[rpc-proxy] Blocked disallowed RPC method: ${req.method}`);
  return NextResponse.json(...);
}
```

### Why It's Safe

1. **Allowlist Gate** (lines 31-59): Only 15 hardcoded methods are allowed:
   - getAccountInfo, getBalance, getMultipleAccounts, etc.
   - Every rejected method is already known to be "disallowed"
   - The log message only fires for rejected requests

2. **Logging Intent**: The console.warn is intentional defensive logging:
   - Documents which invalid methods were attempted (important for security monitoring)
   - Helps detect attack patterns or frontend bugs
   - Not a user-facing message

3. **Console Limitations**: Even if a method name contained newlines or escape sequences:
   - Modern Node.js/browsers filter or escape such sequences in console output
   - Log injection primarily affects **file-based loggers** that parse structured formats (JSON, syslog)
   - Server-side console.warn writes to stdout/stderr, not a parsed log file

4. **No Log Aggregation Risk Visible**: The code doesn't feed console.warn output into:
   - Structured logging libraries (winston, pino, bunyan)
   - Log aggregation systems (Datadog, Splunk, ELK)
   - Syslog or machine-parseable formats

## Conclusion

**NOT VULNERABLE**. The method name is already rejected by the allowlist, so only known-invalid strings reach the log. Log injection risk is negligible for console.warn in a Node.js/browser environment. The logging serves a legitimate security purpose (documenting blocked attempts).

## Recommendation

No fix required. If future log aggregation is added, ensure it sanitizes or escapes user-controlled fields. Current state is secure.
