# H042 — Minimum Sell Amount Still Missing

**Tier:** 2
**Status:** CONFIRMED
**Previous finding:** H069 (NOT_FIXED — launch components modified)
**Recheck date:** 2026-03-21

---

## Summary

No minimum sell amount validation exists on either sell path (bonding curve or AMM). The buy path has an enforced floor (`MIN_PURCHASE_SOL`: 0.05 SOL mainnet / 0.001 SOL devnet), but the equivalent for sells was never added. A user can sell 1 base unit (0.000001 tokens) with no UI or on-chain rejection.

---

## Affected Files

- `app/components/launch/SellForm.tsx` — bonding curve sell UI
- `app/components/swap/SwapForm.tsx` + `app/hooks/useSwap.ts` — AMM sell UI (post-graduation)
- `app/components/station/SwapStation.tsx` — `swapDisabled` guard
- `programs/bonding_curve/src/instructions/sell.rs` — on-chain sell handler
- `programs/bonding_curve/src/constants.rs` — no `MIN_SELL_*` constant defined

---

## Bonding Curve Sell Path (SellForm.tsx)

The debounced validation block (lines 114–155) applies these checks:

1. `isNaN(parsed) || parsed <= 0` → reject
2. `parsed > userBalance` → "Insufficient balance"
3. `baseUnits > curve.tokensSold` → "Cannot sell more than curve has sold"

There is no minimum token amount or minimum SOL return floor. The `canSubmit` guard (line 251–257) only requires `grossSol !== null && grossSol > 0n`, which is satisfied by selling 1 base unit.

The `curve-constants.ts` file defines `MIN_PURCHASE_SOL` (line 85) for buys but has no corresponding sell constant.

---

## On-Chain sell.rs

Step 3 (line 137) only checks `tokens_to_sell > 0`:

```rust
require!(tokens_to_sell > 0, CurveError::ZeroAmount);
```

No `MIN_SELL_TOKENS` or `MIN_SOL_OUT` floor exists in `constants.rs`. The program accepts any non-zero sell amount.

---

## AMM Sell Path (SwapForm / useSwap)

The `swapSolSell` quote path in `useSwap.ts` (lines 434–461) only guards `baseUnits <= 0`. The `swapDisabled` computation in `SwapStation.tsx` (lines 160–164) only requires `parseFloat(inputAmount) > 0`. No minimum token input or minimum SOL return threshold is enforced before transaction construction.

---

## Impact

**Dust sell griefing / economic nuisance:** A bot or user can submit thousands of 1-base-unit sells. Each sell:
- Executes a Token-2022 `transfer_checked` CPI (Transfer Hook invocation included)
- Triggers on-chain math and state writes to `CurveState`
- Returns ~0 lamports net after the 15% tax rounds up via ceiling division

On-chain, the `SOLVENCY_BUFFER_LAMPORTS` (10 lamports) and rent-exempt guard provide some protection against vault drain, but do not prevent the spam itself.

**State pollution:** Each dust sell decrements `tokens_sold`, increments `tokens_returned` / `sol_returned` / `tax_collected`, and emits two events (`TokensSold`, `TaxCollected`). This pollutes the curve's accounting history and chart data.

**Tax escrow dust:** Ceiling-rounded tax on 1 base unit yields 1 lamport of tax routed to the escrow. Repeated micro-sells accumulate dust in the escrow that never reaches a meaningful distribution threshold.

**No financial loss to the protocol** (net SOL is near-zero per tx), but the attack vector is cheap to execute given low Solana tx fees.

---

## Contrast with Buy Path

`BuyForm.tsx` lines 118–121 explicitly block dust buys:

```typescript
if (lamports < MIN_PURCHASE_SOL) {
  setValidationError('Minimum 0.05 SOL');
  setTokensOut(null);
  return;
}
```

This guard exists in both the UI (`BuyForm.tsx`) and mirrored in `constants.rs` as `MIN_PURCHASE_SOL`. The sell path has neither.

---

## Recommended Fix

**Client-side (SellForm.tsx):** Add a `MIN_SELL_TOKENS` constant to `curve-constants.ts` (e.g. `1_000_000n` base units = 1 full token on mainnet, or a SOL-floor equivalent). Validate before computing the quote:

```typescript
if (baseUnits < MIN_SELL_TOKENS) {
  setValidationError('Minimum sell: 1 token');
  setGrossSol(null);
  return;
}
```

**Client-side (useSwap.ts):** Add a minimum token input check before building the `swapSolSell` transaction, mirroring the AMM buy path's structure.

**On-chain (sell.rs / constants.rs):** Add a `MIN_SELL_TOKENS` constant and a corresponding `require!` check in the sell handler after Step 3 for defense-in-depth. Note: changing on-chain constants requires a program upgrade.

**Discussion required:** The exact minimum value (token units vs. SOL-return floor) and whether to enforce on-chain (requiring a redeploy) should be agreed with the project owner before implementation.
