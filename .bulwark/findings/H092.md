# H092 -- WebSocket Reconnection Loses Events

**Status:** CONFIRMED
**Severity:** Tier 3 (Non-Critical)
**Target:** `app/lib/ws-subscriber.ts:299-323` (startStalenessMonitor)

## Hypothesis

During WebSocket reconnection, slot updates may be lost for up to 25 seconds due to detection latency and fallback activation delays.

## Analysis

### Staleness Detection Timing

The staleness monitor (lines 299-323) uses three time constants:

```typescript
const STALENESS_CHECK_INTERVAL = 10_000;      // Check every 10s
const STALENESS_THRESHOLD = 15_000;           // Activate fallback if 15s+ stale
const FALLBACK_POLL_INTERVAL = 5_000;         // HTTP poll every 5s
```

### Data Loss Window Breakdown

1. **WS Death to Detection:** If the WebSocket connection silently fails (TCP keep-alive may mask this), the last `lastSlotReceivedAt` is stale but unnoticed until the next staleness check.

2. **Check Interval Lag:** The staleness monitor runs every 10s (line 304). In worst case:
   - WS dies at T=0
   - Previous staleness check ran at T=0
   - Next check at T=10s → elapsed ≤ 10s, no action
   - Check at T=20s → elapsed = 20s, fallback activates (exceeds 15s threshold)
   - **Latency: 20 seconds before fallback starts**

3. **First Fallback Poll:** Once fallback activates (line 312), the HTTP poll is scheduled but doesn't execute immediately—it waits for the next poll interval. If fallback activated at T=20s and poll interval is 5s, the first poll executes around T=20-25s.

4. **Total Data Loss:** Maximum ~20-25 seconds of slot updates are missed while the system detects WS failure and switches to HTTP polling.

### Why This Occurs

- **WS onSlotChange** updates `lastSlotReceivedAt` only when events arrive (line 258)
- **Staleness check runs every 10s**, not continuously (line 304)
- **Threshold is 15s** — safe margin to avoid false positives on temporary latency, but combined with check interval creates ~20s worst-case lag

### Practical Impact

- Frontend UI consuming `__slot` via SSE may display stale slot data for up to 25s
- Downstream calculations (slippage, timeout windows) using slot data become inaccurate
- Recovery is automatic once HTTP fallback kicks in (no user action needed)

### Code Evidence

**Lines 299-323:**
```typescript
function startStalenessMonitor(connection: Connection): void {
  const STALENESS_CHECK_INTERVAL = 10_000;
  const STALENESS_THRESHOLD = 15_000;
  const FALLBACK_POLL_INTERVAL = 5_000;

  state.stalenessTimer = setInterval(() => {
    const elapsed = Date.now() - state.lastSlotReceivedAt;

    if (elapsed > STALENESS_THRESHOLD && !state.slotFallbackTimer) {
      // Fallback activates here (up to 20s after WS died)
      console.warn(...);
      state.wsConnected = false;
      startSlotFallbackPoll(connection, FALLBACK_POLL_INTERVAL);
    }
    // ...
  }, STALENESS_CHECK_INTERVAL);  // Runs every 10s
}
```

**Lines 278-293 (Fallback Poll):**
```typescript

  connection: Connection,
  interval: number,

  state.slotFallbackTimer = setInterval(async () => {
    const slot = await connection.getSlot();
    // First execution happens after next interval boundary
```

## Verdict

**CONFIRMED** — Data loss is real but expected behavior.

The 20-25s lag is inherent to the detection mechanism. The system correctly:
1. Detects WS failure (via staleness threshold)
2. Activates HTTP fallback (via polling)
3. Recovers automatically when WS reconnects (line 315-321)

This is acceptable for non-critical slot data (used for UI updates, not for transaction safety). Transactions continue to be confirmed by on-chain finality, not by fronted slot tracking.

## Recommendations

If frontend accuracy during WS outages is critical, consider:

1. **Reduce check interval** (currently 10s) — trade off CPU for faster detection. Try 3-5s.
2. **Add exponential backoff** — Fallback poll could reduce from 5s to 2-3s initially, then back off.
3. **Notify client of fallback mode** — Frontend could show "Using fallback data" badge during WS outage.
4. **Subscribe directly to HTTP polling** — For critical features, bypass WS and use deterministic HTTP polling.

Currently, no fix is required. This is **accepted risk** for a dev environment.
