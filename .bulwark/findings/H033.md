## Summary

H047 (original finding: RPC proxy lacked multi-endpoint failover, making the entire frontend
dependent on a single Helius endpoint) was previously marked FIXED. This recheck verifies
whether the fix is still intact and looks for regressions.

**Verdict: RESOLVED — no regression. The failover chain is intact and correctly implemented.**

---

## Evidence: What Was Checked

### 1. Failover Chain (`app/app/api/rpc/route.ts:128-137`)

The endpoint list is built at request-time from environment variables, ordered by preference:

```
HELIUS_RPC_URL  →  HELIUS_RPC_URL_FALLBACK  →  NEXT_PUBLIC_RPC_URL
```

Empty/unset values are filtered out with `.filter(Boolean)`, so the list automatically
contracts to however many endpoints are actually configured. No hardcoded URLs are present,
which is correct — each env var is cluster-specific on Railway.

### 2. Sticky Routing Logic (`app/app/api/rpc/route.ts:134-137`)

```typescript
const orderedEndpoints = lastSuccessfulEndpoint && endpoints.includes(lastSuccessfulEndpoint)
  ? [lastSuccessfulEndpoint, ...endpoints.filter((e) => e !== lastSuccessfulEndpoint)]
  : endpoints;
```

Sticky routing is implemented correctly:
- `lastSuccessfulEndpoint` is promoted to position 0 only when it is still present in the
  current endpoint list (the `endpoints.includes()` guard prevents a stale value from being
  tried if env vars have changed since last request).
- If `lastSuccessfulEndpoint` is null (first request) or no longer in the list, the default
  order is used unchanged.
- `lastSuccessfulEndpoint` is updated on every successful non-5xx response (line 160).

### 3. Multi-Endpoint Rotation (`app/app/api/rpc/route.ts:142-179`)

The failover loop iterates all endpoints in `orderedEndpoints`:

- **HTTP 5xx from upstream**: continues to next endpoint (line 152-156)
- **Network error (DNS, timeout, connection refused)**: caught by `catch`, continues to next
  endpoint (line 172-178)
- **HTTP 2xx/4xx from upstream**: treated as a successful response from the infrastructure
  layer; the RPC-level error (if any) is returned as-is to the browser (line 159-171). This
  is the correct behaviour — an RPC error like "account not found" is a valid Helius response,
  not an infrastructure failure.
- **All endpoints exhausted**: returns HTTP 502 and calls `captureException` (line 182-187).

### 4. Endpoint Masking (`app/app/api/rpc/route.ts:72-78`)

The `maskEndpoint()` helper strips URLs to hostname-only before logging. API keys embedded
in URL paths or query parameters are not leaked to logs. This is a security hygiene measure
present in the fix and still intact.

### 5. `connection.ts` — Single-Endpoint Singleton (by design)

`app/lib/connection.ts` does NOT implement failover — it is a singleton that resolves to one
URL per execution context:
- Browser: `/api/rpc` (the proxy, which does the failover)
- Server: `HELIUS_RPC_URL` env var directly

This is correct. The browser never needs multi-endpoint logic; all failover is centralised in
the proxy. Server-side code (ws-subscriber, crank) uses the primary Helius URL directly and
has its own retry patterns at the call site.

## Residual Observations (Not Regressions of H047)

The following weaknesses exist in the failover implementation but are pre-existing issues
unrelated to the H047 fix correctness:

1. **No per-request fetch timeout** (pre-existing, tracked separately): `fetch(endpoint, ...)`
   on line 144 has no `signal: AbortSignal.timeout(N)`. A hung upstream endpoint blocks the
   worker thread for the duration of Node's default socket timeout (~2 min) before the catch
   clause fires and tries the next endpoint. Under this condition, failover works eventually
   but is extremely slow. This is listed in ARCHITECTURE.md Tier 2 risk #5.

2. **`lastSuccessfulEndpoint` is process-level state** (pre-existing, noted in CHAIN-02
   context `04-CHAIN-02-rpc-node-trust.md` under "Cross-Focus Handoffs"): If Railway scales
   to multiple instances, each process maintains an independent sticky-routing state. Not a
   regression; Railway currently runs a single instance (ASM-5 validated).

3. **No health-probe before adding endpoint back** (by design): Once all endpoints fail, the
   next request starts over from `orderedEndpoints[0]`. There is no circuit-breaker or
   cooldown per endpoint. Endpoints that are flapping (frequently returning 5xx) will be
   retried on every request. Acceptable for the current traffic volume.

---

## Conclusion

The H047 fix is fully intact. The three-endpoint failover chain, sticky routing with
re-validation guard, HTTP 5xx detection, and network-error catch are all present and
functioning as designed. No regression has been introduced between H047's fix and the
current codebase state.

The residual concerns above (fetch timeout, process-level state, no per-endpoint circuit
breaker) were known at the time of the H047 fix and are tracked as separate findings.
H033 closes as **RESOLVED**.
