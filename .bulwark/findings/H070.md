# H070: Cluster Config Poisoning via Build Cache

**Severity:** MEDIUM (Tier 3)
**Target:** `app/lib/protocol-config.ts:25`, `app/next.config.ts:10`, Railway Nixpacks build system
**Status:** CONFIRMED — Baked Build-Time Values Without Invalidation

---

## Vulnerability Summary

`NEXT_PUBLIC_CLUSTER` is read **at build time** by Next.js and embedded as a constant in the production bundle. When the env var is changed on Railway without triggering a full rebuild, the old cluster value persists in the compiled bundle, causing all subsequent requests to target the wrong on-chain addresses (devnet addresses on mainnet RPC, or vice versa).

This is a **build artifact poisoning** issue: the build cache contains hardcoded cluster values that do not update when the environment variable changes.

---

## Root Cause Analysis

### 1. Build-Time Reading (CRITICAL)

**File:** `app/lib/protocol-config.ts:25`
```typescript
const rawCluster = process.env.NEXT_PUBLIC_CLUSTER || "devnet";
const clusterName = rawCluster === "mainnet" ? "mainnet-beta" : rawCluster;
const config = getClusterConfig(clusterName);

export const PROGRAM_IDS = config.programIds;
export const MINTS = config.mints;
// ... all re-exports are CONSTANTS
```

At module evaluation time (during build), Node.js reads `NEXT_PUBLIC_CLUSTER`, resolves the cluster config, and exports the addresses as **static constants**. These are baked into the compiled `.next/` bundle as literal values.

**File:** `app/next.config.ts:10`
```typescript
const cluster = process.env.NEXT_PUBLIC_CLUSTER || "devnet";
const isMainnet = cluster === "mainnet";
const heliusRpcDomain = isMainnet ? "mainnet.helius-rpc.com" : "devnet.helius-rpc.com";
```

The CSP header is also built at compile time, embedding the cluster-specific Helius domain.

### 2. Next.js Build Pipeline

**File:** `app/package.json:8-9`
```json
"prebuild": "node scripts/sync-idl.mjs",
"build": "next build",
```

The `npm run build` command:
1. Runs `sync-idl.mjs` (pre-build hook)
2. Invokes `next build` (Turbopack compilation)
3. Turbopack bundles `protocol-config.ts` with `NEXT_PUBLIC_CLUSTER` already resolved
4. Artifacts written to `.next/`

**File:** `railway.toml:3`
```toml
buildCommand = "npm run --workspace app build"
```

Railway runs the build command once. If the env var changes after the build, the cached `.next/` bundle is not invalidated.

### 3. Default Fallback (Silent Failure Vector)

Line 25 of `protocol-config.ts` and line 10 of `next.config.ts` both have:
```typescript
const rawCluster = process.env.NEXT_PUBLIC_CLUSTER || "devnet";
```

If `NEXT_PUBLIC_CLUSTER` is **unset**, the code silently defaults to "devnet". This means:
- **Fresh Railway deploy without env var set** → builds with devnet addresses
- **Mainnet promotion without explicitly setting NEXT_PUBLIC_CLUSTER=mainnet** → deploys mainnet binary but all requests target devnet

## Attack Scenarios

### Scenario A: Mainnet Deploy Without Env Var (REALISTIC)

1. **Initial state:** Railway has `NEXT_PUBLIC_CLUSTER=devnet`
2. **Action:** Operator changes NEXT_PUBLIC_CLUSTER=mainnet in Railway console
3. **Problem:** Railway does **not** auto-rebuild on env var changes
4. **Result:** Old devnet-compiled bundle still running, all TXs sent to devnet addresses on mainnet RPC
5. **User impact:** WORMHOLE SWAP — user sends mainnet SOL, gets "Program not found" error or zero output (TXs fail on-chain, silent if skipPreflight=true)

### Scenario B: Build Cache Poisoning During Stage 5 (CRITICAL FOR LAUNCH)

1. **Phase 100 state:** Devnet fully deployed and tested, mainnet binaries staged on Railway
2. **Action:** Operator re-runs `npm run build` locally with wrong `NEXT_PUBLIC_CLUSTER` (e.g., left as "devnet" by mistake)
3. **Artifact:** `.next/` contains devnet program IDs embedded as constants
4. **Deployment:** Commits poisoned bundle to git, or uploads to Railway
5. **Launch failure:** Protocol appears "broken" on mainnet because all on-chain calls target devnet addresses

### Scenario C: Hotfix Redeploy (MEV WINDOW)

1. **Status:** Mainnet running with correct `NEXT_PUBLIC_CLUSTER=mainnet`
2. **Incident:** Need to deploy a hotfix for a client-side bug
3. **Action:** Operator runs `npm run build` but forgets to set env var (defaults to "devnet")
4. **Window:** 5-30 minutes before operator notices TXs failing
5. **Damage:** All users hit devnet addresses; MEV bots can intercept failed TXs

## Detection & Proof

### Static Analysis

Both files read `NEXT_PUBLIC_CLUSTER` at module-initialization time:

1. **`protocol-config.ts:25`** — reads env var, immediately resolves config, exports constants
2. **`next.config.ts:10`** — reads env var, builds CSP header with hardcoded domain
3. **`ClusterConfigProvider.tsx`** — similar pattern (reads at runtime, but only for display)

No runtime lookup. No cache invalidation on env var change.

### Reproduction Steps

1. Set `NEXT_PUBLIC_CLUSTER=devnet` on Railway
2. Run `npm run build` in the app directory
3. Inspect `.next/server/chunks/[root-of-the-server]*.js` (look for hardcoded `8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4` or mainnet program IDs)
4. Change `NEXT_PUBLIC_CLUSTER=mainnet` on Railway
5. **WITHOUT rebuilding**, restart the server
6. Fetch `/api/health` or any page that imports from `protocol-config.ts`
7. **Observe:** Old cluster value persists (check PROGRAM_IDS exported or CSP header domain)

## Root Cause

1. **Build-time constant baking:** Next.js embeds `NEXT_PUBLIC_CLUSTER` values at compile time
2. **No runtime override:** All exports from `protocol-config.ts` are static constants, not runtime-evaluated
3. **No env var validation:** Silent fallback to "devnet" if unset — no warning or error
4. **No build cache invalidation:** Railway Nixpacks does not auto-rebuild on env var changes
5. **Single-process deployment:** Railway runs one container; no staged/blue-green deployment to catch misconfigurations

## Remediation

### IMMEDIATE (Pre-Launch Gate)

**1. Explicit Env Var Validation**

Add to `app/lib/protocol-config.ts` before any exports:

```typescript
// Fail-closed cluster validation (build-time)
const rawCluster = process.env.NEXT_PUBLIC_CLUSTER;
if (!rawCluster) {
  throw new Error(
    "NEXT_PUBLIC_CLUSTER environment variable is not set. " +
    "This must be explicitly defined at build time. " +
    "Set NEXT_PUBLIC_CLUSTER=devnet or NEXT_PUBLIC_CLUSTER=mainnet"
  );
}
```

Also add to `app/next.config.ts:10`:

```typescript
if (!process.env.NEXT_PUBLIC_CLUSTER) {
  throw new Error(
    "NEXT_PUBLIC_CLUSTER not set at build time. " +
    "Next.js config cannot determine cluster. Set NEXT_PUBLIC_CLUSTER=devnet or =mainnet"
  );
}
```

**Benefit:** Build fails loudly if env var is missing, preventing silent "default to devnet" poisoning.

**2. Runtime Assertion (Defense in Depth)**

Add a runtime safety check in the main server entrypoint (`app/app/layout.tsx` or `app/instrumentation.ts`):

```typescript
// Runtime cluster validation (catches bundle mismatch)
const buildTimeCluster = process.env.NEXT_PUBLIC_CLUSTER;
const runtimeCluster = process.env.RUNTIME_CLUSTER_OVERRIDE;
if (runtimeCluster && buildTimeCluster !== runtimeCluster) {
  console.error(
    `CLUSTER MISMATCH: Built for ${buildTimeCluster}, ` +
    `but runtime is ${runtimeCluster}. This indicates bundle poisoning. ` +
    `DO NOT START SERVER. Rebuild with correct env var.`
  );
  process.exit(1);
}
```

**Benefit:** Catches accidental redeploy of wrong bundle to production.

**3. Railway Build Trigger Configuration**

Add `.railway.toml` or use Railway UI to enforce:

```toml
[build]
# Force rebuild when NEXT_PUBLIC_CLUSTER env var changes
# by invalidating cache (workaround: use Railway Deploy Trigger)
builder = "NIXPACKS"
buildCommand = "npm run --workspace app build"

# Add health check that verifies cluster matches expected
[deploy]
healthcheckPath = "/api/health"
# Add custom validation endpoint that checks PROGRAM_IDS against expected set
```

Or create a deploy pre-flight script:

```bash
#!/bin/bash
# scripts/deploy/validate-cluster.sh
set -e

CLUSTER=${NEXT_PUBLIC_CLUSTER:-unknown}
if [[ "$CLUSTER" != "devnet" && "$CLUSTER" != "mainnet" ]]; then
  echo "ERROR: NEXT_PUBLIC_CLUSTER=$CLUSTER is invalid"
  exit 1
fi

if [[ "$CLUSTER" == "mainnet" && -z "$MAINNET_WALLET_ADDRESS" ]]; then
  echo "ERROR: Mainnet deployment requires MAINNET_WALLET_ADDRESS"
  exit 1
fi

npm run --workspace app build
```

**4. Deployment Checklist (Manual Gate)**

Before any deploy, verify in Railway UI:
- [ ] `NEXT_PUBLIC_CLUSTER` is set to the correct value
- [ ] Run `/api/health` endpoint and verify `cluster` field matches expectation
- [ ] Check PROGRAM_IDS returned match the on-chain deployed addresses
- [ ] Run a test swap on staging and confirm TX succeeds on correct cluster

### SHORT-TERM (Post-Launch Monitoring)

**5. Telemetry & Alerting**

Add cluster mismatch detection to `/api/health`:

```typescript
// app/app/api/health/route.ts
export async function GET() {
  const cluster = process.env.NEXT_PUBLIC_CLUSTER || "unknown";
  const expectedCluster = process.env.EXPECTED_CLUSTER || cluster;

  if (cluster !== expectedCluster) {
    // Log alert to Sentry
    captureException(
      new Error(`Cluster mismatch: built=${cluster}, expected=${expectedCluster}`)
    );
  }

  return Response.json({
    status: "ok",
    cluster,
    expectedCluster,
    mismatch: cluster !== expectedCluster,
    timestamp: Date.now(),
  });
}
```

**6. CI/CD Pre-Deployment Checks**

Before Railway deploy (GitHub Actions / pre-commit hook):

```bash
#!/bin/bash
# Verify NEXT_PUBLIC_CLUSTER is set and valid
if [[ -z "$NEXT_PUBLIC_CLUSTER" ]]; then
  echo "FATAL: NEXT_PUBLIC_CLUSTER not set"
  exit 1
fi

if [[ "$NEXT_PUBLIC_CLUSTER" == "mainnet" ]]; then
  # Additional checks for mainnet (e.g., require branch protection)
  if [[ "$GITHUB_REF" != "refs/heads/main" ]]; then
    echo "FATAL: Mainnet deploys only allowed from main branch"
    exit 1
  fi
fi
```

### LONG-TERM (Architecture)

**7. Runtime Cluster Resolution (Future Refactor)**

Instead of baking addresses at build time, resolve from a runtime config endpoint:

```typescript
// Future: app/lib/cluster-config.ts (runtime-evaluated)
export async function getClusterConfig() {
  // Fetch from /.well-known/cluster-config.json (served by Next.js)
  const response = await fetch("/.well-known/cluster-config.json");
  return response.json();
}
```

This requires moving away from static exports, which is a larger refactor. **Not recommended for pre-launch, only for v1.5+.**

**8. Build Artifact Signature**

Sign the `.next/` bundle with the cluster value, verify on startup:

```typescript
// Verify build artifact cluster match
const bundleCluster = fs.readFileSync(".next/.cluster-signature", "utf8");
if (bundleCluster !== process.env.NEXT_PUBLIC_CLUSTER) {
  throw new Error(
    `Build artifact cluster ${bundleCluster} does not match ` +
    `NEXT_PUBLIC_CLUSTER=${process.env.NEXT_PUBLIC_CLUSTER}`
  );
}
```
