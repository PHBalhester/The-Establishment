# H069: SSE Zombie Connection via NAT/CGNAT

**Tier:** 3 (Medium)
**Severity:** Medium
**Category:** Resource Exhaustion / Rate Limiting
**Target:** `app/lib/sse-connections.ts:85-104` + integration in `app/app/api/sse/*/route.ts`
**Status:** CONFIRMED

---

## Vulnerability Summary

The SSE connection tracker implements an auto-release timeout of **30 minutes** (`MAX_CONNECTION_MS = 30 * 60_000`) to reclaim "zombie" connections from clients that crash without properly closing the SSE connection. However, this timeout creates a **legitimate user blocking window** in CGNAT/NAT scenarios where:

1. **Client crashes or network resets** → SSE stream disconnects uncleanly
2. **Connection slot remains held** for 30 minutes (zombie state)
3. **All users behind the same NAT IP are blocked** → Cannot acquire new SSE connections
4. **Impact:** Up to 10 connections from the IP can be held; after 10 zombies, all legitimate users behind that NAT IP receive HTTP 429 "Too Many Connections"

---

## Root Cause Analysis

### The Zombie Problem (Legitimate)
When a browser crashes or network abruptly resets without sending a clean `FIN`, the SSE connection remains open from the server's perspective. Without cleanup, these would accumulate indefinitely, exhausting resources.

**Current Implementation (sse-connections.ts:88-93):**
```typescript
const timeout = setTimeout(() => {
  if (!released) {
    released = true;
    releaseConnection(ip);
  }
}, MAX_CONNECTION_MS);  // 30 minutes = 1,800,000 ms
```

### The NAT Blocking Problem (Legitimate Trade-off)
- **MAX_PER_IP = 10** (by design: 5 tabs × 2 SSE routes)
- **x-forwarded-for extraction** (rate-limit.ts:129-150) trusts the first IP in the header
- **All users behind the same CGNAT** share one public IP

**Attack Scenario:**
1. One user behind 203.0.113.42 (CGNAT) crashes with an active SSE connection
2. Browser tab closes without sending `close` event
3. Server's `scheduleAutoRelease()` triggers at 30-minute mark
4. Meanwhile, other users behind same CGNAT try to open SSE
5. They face 4 simultaneous failures (each waiting for the 30-min timeout to clear one of the 10 slots)

### Why This Is a DoS Risk
- **Zombie count accumulates:** Each crashed connection holds a slot
- **Legitimate users affected:** Not the attacker's IP; downstream NAT peers
- **Timeout is long:** 30 minutes is an operational pain for interactive apps
- **No manual release mechanism:** Ops cannot force cleanup without restarting Railway
- **No monitoring:** No endpoint exposes `getGlobalCount()` or `getIpCount()` to detect zombies

## Proof of Concept

**Scenario: Corporate CGNAT (100 employees, 1 public IP)**

1. Employee behind `203.0.113.42` opens trading dashboard
2. Browser tab crashes (or WiFi dropout) while connected to `/api/sse/protocol`
3. Server holds 1 slot (zombie)
4. 10 more employees crash their browsers over the next hour → 10 zombie slots consumed
5. New employee tries to open dashboard:
   - Attempts to acquire connection for IP `203.0.113.42`
   - `state.connections.get("203.0.113.42")` returns `10`
   - Check: `if (ipCount >= MAX_PER_IP) return false` → **REJECTED**
   - Employee receives HTTP 429, cannot trade

**Time to recovery:** 30 minutes from the **last** zombie creation (whichever slot was filled most recently)

## Severity Justification: MEDIUM (NOT HIGH)

### Why Not Critical/High
1. **Affects only users behind the same NAT** — not protocol-wide
2. **Timeout is absolute:** After 30 minutes, connection is reclaimed (not permanent)
3. **MaxPerIP cap is reasonable** — 10 slots per IP covers legitimate multi-tab scenarios
4. **Recovery is automatic** — no manual intervention needed (just wait)
5. **Requires crash/reset to trigger** — not triggered by normal disconnect

### Why Not Low
1. **Real-world impact:** CGNAT is standard in corporate networks, mobile carriers
2. **User-facing outage:** Legitimate users cannot trade (business impact)
3. **Detection gap:** No alerting or monitoring; ops are unaware of zombie state
4. **No emergency pressure valve:** Can't force-release a zombie slot

## Affected Code Paths

### 1. **Connection Acquisition**
`app/app/api/sse/candles/route.ts:40-48` + `app/app/api/sse/protocol/route.ts:40-48`

```typescript
if (!acquireConnection(clientIp)) {
  return new Response("Too Many Connections", {
    status: 429,
    headers: { "Retry-After": "30" },
  });
}
```

Both SSE routes check the same per-IP cap. A crash on one route counts against the other.

### 2. **Auto-Release Schedule**
`app/lib/sse-connections.ts:85-104`

30-minute timeout with no configuration option. The timeout is hard-coded.

### 3. **IP Extraction**
`app/lib/rate-limit.ts:129-150`

Trusts `x-forwarded-for` header from Railway's reverse proxy. If Railway's proxy is misconfigured or if an attacker controls a proxy layer, IPs can be spoofed, worsening the problem.

## Attack Tree

```
┌─ SSE Zombie via NAT Blocking (H069)
│
├─ Trigger zombie accumulation (passive)
│  ├─ 10 different users behind CGNAT crash browsers
│  └─ Each holds 1 slot for 30 minutes
│
└─ Escalate to 11th user
   ├─ Try to open SSE connection
   └─ acquireConnection fails → HTTP 429
      └─ Cannot access trading UI (no real-time quotes)
```

**Note:** An attacker **cannot directly trigger this on their own IP** because:
- They need to simulate 10 crashes from distinct users (impossible from one browser)
- They can only affect IPs they control or share with legitimate users

However, **an attacker at a CGNAT** (e.g., shared office, mobile network) could:
1. Intentionally crash 10 browser tabs on their user ID
2. Block all other users on the same CGNAT from trading (collateral damage)

## Current Mitigations

### ✓ Idempotent Release
`sse-connections.ts:99-103` has a `released` flag preventing double-release when both normal disconnect + timeout fire.

### ✓ Math Protection
`sse-connections.ts:72` uses `Math.max(0, state.globalCount - 1)` to prevent global count underflow.

### ✓ Connection Cap
`MAX_PER_IP = 10` is reasonable for 5-tab multi-route scenarios.

### ✗ No Monitoring Export
`getGlobalCount()` and `getIpCount()` are defined but **not exported to any monitoring/debug endpoint**. Ops cannot detect zombie state.

### ✗ No Configuration
30-minute timeout is hard-coded. Cannot be adjusted at runtime or per-deployment.

### ✗ No IP Spoofing Protection
If `x-forwarded-for` is spoofed or RFC 7239 `Forwarded` header is trusted without validation, an attacker can aggressively fill slots on target IPs.

## Recommendations

### 1. **Reduce Timeout (Immediate)**
Change `MAX_CONNECTION_MS` from 30 minutes to **5 minutes** or **10 minutes**:

```typescript
const MAX_CONNECTION_MS = 5 * 60_000; // 5 minutes instead of 30
```

**Rationale:**
- SSE is a real-time protocol; if the browser hasn't sent a message in 5 minutes, it's likely crashed
- 5 minutes is a reasonable balance: catches most crashes quickly without being too aggressive
- Corporate CGNAT users can reconnect in a shorter window

**Impact:** Reduces zombie blocking window from 30 min → 5 min

### 2. **Export Connection Stats to Debug Endpoint (Medium)**
Create a new internal debug endpoint (not advertised, but available for monitoring):

```typescript
// app/app/api/internal/sse-stats.ts
export async function GET(req: NextRequest) {
  // Only allow from localhost or admin IP
  const clientIp = getClientIp(req);
  if (clientIp !== "127.0.0.1" && process.env.ADMIN_IP !== clientIp) {
    return new Response("Forbidden", { status: 403 });
  }

  return new Response(JSON.stringify({
    globalCount: getGlobalCount(),
    perIp: Array.from(sseConnState.connections.entries()).map(([ip, count]) => ({
      ip,
      connections: count,
    })),
  }));
}
```

**Rationale:** Allows ops to see zombie state and decide if manual restart is needed

### 3. **Implement Heartbeat + Explicit Close (Future)**
Add a per-route heartbeat or explicit close signal to detect when browsers are truly gone:

```typescript
// In sse route:
encoder.encode(`data: {"type":"heartbeat"}\n\n`);
// Browser can send CLOSE event instead of FIN
```

**Rationale:** Distinguishes between "crashed" and "just silent" connections

### 4. **Monitor and Alert (Future)**
Track zombie connection ratio and alert when:
- Global count approaches MAX_GLOBAL (5000)
- Any IP approaches MAX_PER_IP with >50% zombies

## Testing Recommendations

1. **Unit Test: Timeout fires**
   - Create connection, don't release, wait MAX_CONNECTION_MS
   - Verify `releaseConnection()` is called automatically

2. **Unit Test: CGNAT blocking**
   - Acquire 10 connections for same IP
   - Verify 11th acquisition returns false

3. **Integration Test: Normal disconnect cancels timeout**
   - Acquire connection, immediately release, verify timeout is cancelled

4. **Load Test: Zombie accumulation**
   - Simulate 100 connections, release half normally, let half timeout
   - Verify no memory leak, global count eventually decays

## Verdict

**CONFIRMED** ✓

The SSE zombie connection timeout creates a real blocking window for legitimate users behind shared NAT IPs. While not a critical vulnerability (timeout is finite, requires crash to trigger), it represents a **legitimate DoS risk in real-world deployment scenarios** (corporate CGNAT, mobile networks). The 30-minute timeout is unnecessarily long and lacks monitoring visibility.

**Recommended Priority:** Fix immediately before mainnet launch (reduce timeout, add monitoring).
