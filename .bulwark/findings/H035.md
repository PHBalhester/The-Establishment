# Finding: H035 - SSE Single-Process Regression (RECHECK H092)

## Status: RESOLVED
## Tier: 2
## Confidence Score: 10

## Recheck Scope

H092 from Audit #1 was originally confirmed as LOW: in-memory SSE pub/sub
(`sse-manager.ts`) would silently break if horizontal scaling was added. A
separate VERIFY-H092 (Round 3, 2026-03-12) closed the connection-exhaustion
vector (H008) as FIXED after `sse-connections.ts` was added. This recheck
verifies:

1. globalThis singleton pattern is intact and correct (no Turbopack regression)
2. No horizontal scaling assumptions have been introduced since Audit #1
3. No process-level state leaks between the two singleton stores

---

## Finding Detail

### 1. globalThis Singleton Pattern — INTACT

**`app/lib/sse-manager.ts` (lines 86–92)**

```typescript
const globalForSSE = globalThis as unknown as {
  sseManager: SSEManager | undefined;
};

export const sseManager = globalForSSE.sseManager ?? new SSEManager();
globalForSSE.sseManager = sseManager;
```

**`app/lib/protocol-store.ts` (lines 118–125)**

```typescript
const globalForStore = globalThis as unknown as {
  protocolStore: ProtocolStore | undefined;
};

export const protocolStore =
  globalForStore.protocolStore ?? new ProtocolStore();
globalForStore.protocolStore = protocolStore;
```

**`app/lib/sse-connections.ts` (lines 33–37)**

```typescript
const globalForSSEConn = globalThis as unknown as {
  sseConnState: { connections: Map<string, number>; globalCount: number } | undefined;
};
const state = globalForSSEConn.sseConnState ?? { connections: new Map<string, number>(), globalCount: 0 };
globalForSSEConn.sseConnState = state;
```

All three singletons follow the correct unconditional pattern: `const x = globalThis.x ?? new X(); globalThis.x = x;`. The CRITICAL Turbopack anti-pattern (`if (NODE_ENV !== "production") { globalThis.x = x; }`) is absent in all three files. No `NODE_ENV` guard was found wrapping any globalThis assignment.

The `sse-connections.ts` singleton was added after Audit #1 and also follows the correct pattern.

### 2. Horizontal Scaling Assumptions — NONE INTRODUCED

**`railway.toml`** contains no replica count, no scaling policy, and no multi-instance configuration. `restartPolicyType = "ON_FAILURE"` with `maxRetries = 3` describes single-process crash recovery, not horizontal scaling.

**`app/lib/sse-manager.ts` comment (lines 7–11)** explicitly acknowledges the single-process constraint and the Redis upgrade path — unchanged since Audit #1. No code has been added that would be correct only under multi-process assumptions.

**ARCHITECTURE.md ASM-5** (`Single Railway process — VALIDATED`) remains the documented deployment model. The audit synthesis confirms this assumption is actively tracked.

### 3. Process-Level State Leaks — NONE

The three singleton stores are independent with clean separation:

- `sseManager` (SSEManager): subscriber `Set<SSECallback>` — write-only from `protocolStore`, read via `broadcast()`
- `protocolStore` (ProtocolStore): `Map<string, AccountState>` + `Map<string, string>` dedup cache — written by webhook handler, read by SSE route for initial snapshot
- `sseConnState` (plain object): `Map<string, number>` per-IP counters + `globalCount` — written/read only by `sse-connections.ts` public API

No reference cycles between the three stores. `protocolStore` imports `sseManager` (unidirectional dependency). `sse-connections.ts` is fully independent. All three hang off separate `globalThis` keys (`sseManager`, `protocolStore`, `sseConnState`) — no key collision possible.

The `app/instrumentation.ts` `register()` guard correctly limits initialization to the Node.js runtime only:

```typescript
if (typeof window !== "undefined") return;
if (process.env.NEXT_RUNTIME === "edge") return;
```

This prevents the ws-subscriber (and its polling timers) from running in the Edge runtime, which could otherwise create a second set of state-writing timers against the same globalThis stores.

### 4. H092 Original Finding — Status Confirmed Accurate

The original H092 finding (Audit #1) identified the in-memory SSE as a known, documented limitation conditional on single-process deployment. That condition remains true. The VERIFY-H092 fix (connection caps via `sse-connections.ts`) addressed the DoS exhaustion angle, which was the exploitable component. The architectural limitation (Redis needed for horizontal scale) is unchanged, but is not a security finding — it is a documented operational constraint with a clear, identified upgrade path.
