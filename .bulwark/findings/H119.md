# H119: Decode Failure Broadcasts Raw Data via SSE

## Verdict: CONFIRMED (HIGH) — Reference H005/H020 Fallback Path

---

## Evidence

### Location

**File:** `app/app/api/webhooks/helius/route.ts`
**Lines:** 607–619
**Context:** `handleAccountChanges()` function, error handling for Anchor decode failures

### Code Under Analysis

```ts
try {
  // Lazy-init programs on first decodable account
  if (!programs) {
    programs = {
      amm: getAmmProgram(),
      bondingCurve: getBondingCurveProgram(),
      epoch: getEpochProgram(),
      staking: getStakingProgram(),
    };
  }

  const program = programs[decodeInfo.programKey];
  const rawBuffer = Buffer.from(item.rawAccountData.data, "base64");
  const decoded = program.coder.accounts.decode(
    decodeInfo.accountType,
    rawBuffer,
  );
  const bigintFields = ... // BigInt normalization
  const normalized = anchorToJson(decoded, ...);

  protocolStore.setAccountState(pubkey, {
    ...normalized,
    updatedAt: Date.now(),
  });
} catch (err) {
  // ─── H119 ISSUE: Decode failure stores raw attacker data ───
  console.error(
    `[webhook] Failed to decode ${label} at ${pubkey}:`,
    err,
  );
  protocolStore.setAccountState(pubkey, {
    label,
    accountData: item.accountData,           // ← attacker-controlled
    rawAccountData: item.rawAccountData,    // ← attacker-controlled
    updatedAt: Date.now(),
    decodeError: err instanceof Error ? err.message : String(err),
  });
}
```

### The Vulnerability

When Anchor's `coder.accounts.decode()` throws an error (malformed base64, corrupt layout, truncated buffer), the exception handler calls `protocolStore.setAccountState()` with the raw, unvalidated webhook payload:

1. **`accountData`** — attacker-supplied object from the webhook body
2. **`rawAccountData`** — attacker-supplied base64-encoded bytes with `{ data: "...", encoding: "base64" }`
3. **No semantic validation** — the error handler stores exactly what the attacker sent
4. **Immediate SSE broadcast** — `setAccountState()` immediately calls `sseManager.broadcast("protocol-update", ...)` (protocol-store.ts:61)

This means **a decode error is not a protective boundary — it is itself an injection mechanism**.

### Interaction with H005/H020

**H005** (lines 61–76) identified this exact path:

> If the attacker provides malformed base64 that passes structural decode but triggers a downstream error, the catch block still calls `setAccountState` with the raw payload... This means **even a decode failure is an injection**, as the raw `accountData` object is stored and broadcast.

**H020** (implicit in Step 4–5) confirmed the injection chain:

> The `rawAccountData.data` field is attacker-controlled base64. Anchor decodes it... An attacker who knows the on-chain account layout (IDL is public) can craft a valid base64 blob. **But what happens if decode throws?** The catch block still stores and broadcasts the raw payload.

### Attack Mechanics

**Scenario 1: Intentional malformation to trigger catch block**

1. Attacker crafts an enhanced webhook with:
   - `account`: real protocol PDA address (passes KNOWN_PROTOCOL_ACCOUNTS check)
   - `rawAccountData.data`: garbage base64 that decodes but produces an intermediate error (e.g., truncated layout)
   - `accountData`: crafted object with malicious numeric fields

2. Anchor decode throws → catch block fires

3. Instead of silently failing, the catch block stores:
   ```ts
   {
     label: "EpochState",
     accountData: {
       // attacker-controlled Helius response data
       "account": "...",
       "nativeBalanceChange": 999_999_999_999,  // fake lamports
       "tokenBalanceChanges": [/* attacker payload */]
     },
     rawAccountData: {
       data: "<garbage base64>",
       encoding: "base64"
     },
     decodeError: "Cannot read property X of undefined",
     updatedAt: Date.now()
   }
   ```

4. `protocolStore.setAccountState()` broadcasts this via SSE to all connected clients

5. Frontend `useProtocolState` hook stores it: `setAccounts(prev => ({ ...prev, [account]: data }))`

6. Components consume `data.accountData` or `data.rawAccountData` directly (if future code does)

**Scenario 2: Crafting base64 that triggers a specific intermediate error**

An attacker can:
- Study the Anchor struct layout (public in IDL files)
- Craft base64 that decodes partially (field 1, 2 are valid; field 3 causes an error during normalization or BigInt extraction)
- The error handler catches the exception and stores the original bytes + error message

The attacker's injected `accountData` object from Helius (which is opaque Helius metadata) is broadcast unvalidated.

### Why This Bypasses the Intended Safety Net

The try-catch was intended as a **safety boundary**: if decoding fails, don't store data. However, it became an **injection funnel** because:

1. **Catch block doesn't fail-closed** — it stores data anyway, just a different shape
2. **Error message is exposed** — the `decodeError` field confirms the attack succeeded ("decode failed means my payload made it through")
3. **Raw payload preserved** — future code (or SSE consumers) might attempt to parse `accountData` or `rawAccountData` again, re-triggering the vulnerability

### Scope of Damage

| Target | Damage | Example |
|--------|--------|---------|
| Frontend state | `useProtocolState` stores arbitrary object | UI re-renders based on attacker data |
| SSE subscribers | Any number of concurrent browsers | Broadcast reaches all 5000+ connected clients simultaneously |
| Browser cache | Object persists in React state until next legitimate webhook | Up to 60s corruption window |
| Log visibility | Error message reveals protocol PDA targeted | Attacker can enumerate which accounts were injected successfully |

## Attack Path

**Prerequisite:** Attacker has `HELIUS_WEBHOOK_SECRET` (same as H005/H020).

1. POST malformed enhanced webhook to `/api/webhooks/helius`:
   ```json
   [{
     "account": "<EpochState_PDA>",
     "accountData": [{
       "account": "<EpochState_PDA>",
       "nativeBalanceChange": 99999999,
       "tokenBalanceChanges": []
     }],
     "rawAccountData": {
       "data": "aW52YWxpZA==",  // invalid base64 or truncated buffer
       "encoding": "base64"
     }
   }]
   ```

2. `handleAccountChanges()` reaches line 590 → tries `program.coder.accounts.decode()`

3. Decode fails (truncated, malformed, or layout mismatch) → exception thrown

4. Catch block (line 607–619) executes

5. Stores raw `accountData` (attacker-controlled) in protocol store:
   ```ts
   protocolStore.setAccountState(EpochState_PDA, {
     label: "EpochState",
     accountData: { ... },  // malicious
     rawAccountData: { ... },
     decodeError: "Buffer index out of bounds",
     updatedAt: Date.now()
   })
   ```

6. `setAccountState()` broadcasts via SSE → all connected browsers receive the injected object

7. React hooks consume the raw object → display is corrupted

|-----------|-----------|

| Scope | All concurrent SSE subscribers |

| Detectability | Low — error message confirms attack, but injection persists silently |

The severity is **HIGH** (not CRITICAL) because:
- Requires prior compromise of `HELIUS_WEBHOOK_SECRET` (same gate as H005/H020)
- Damage is contained to frontend display/state (on-chain `minimumOutput` enforces slippage guard)
- Effect is temporary (clears on next legitimate data arrival)

It is NOT a sub-class of H005/H020 because the injection mechanism differs: H005/H020 inject data by crafting **valid Anchor-decodable bytes**. H119 injects by **triggering a decode failure and storing the raw webhook payload in the catch block**, bypassing Anchor validation entirely.

## Recommendation

### Required

**1. Fail-closed on decode error (Critical)**

Do not call `setAccountState()` in the catch block. Instead:

```ts
} catch (err) {
  console.error(
    `[webhook] Failed to decode ${label} at ${pubkey}: `,
    err,
  );
  // CHANGE: log and return, do NOT store raw payload
  captureException(new Error(`[webhook] Decode failed for ${label}: ${err}`));
  continue;  // Skip to next account, do not update store
}
```

**Rationale:** A decode failure means the attacker-supplied raw bytes are unvalidated. Storing them (even with an error flag) is a liability. The legitimate state will be supplied by the next `ws-subscriber` polling cycle (≤60s).

**2. Add pre-decode payload schema validation (High)**

Before passing `item.rawAccountData.data` to Anchor, validate:
- Base64 decodes successfully
- Decoded buffer length matches expected struct size (±tolerance)
- First N bytes match expected discriminator (if accounts have discriminators)

```ts
if (!item.rawAccountData?.data) {
  console.warn(`[webhook] Missing rawAccountData for ${label}`);
  continue;  // skip
}

try {
  const rawBuffer = Buffer.from(item.rawAccountData.data, "base64");
  // Check minimum length: prevents truncated buffers that fail mid-decode
  const expectedMinLength = 100; // adjust per account type
  if (rawBuffer.length < expectedMinLength) {
    throw new Error(`Buffer too short: ${rawBuffer.length} < ${expectedMinLength}`);
  }
} catch (bufferError) {
  console.error(`[webhook] Invalid base64 or buffer for ${label}:`, bufferError);
  continue;  // skip, do NOT attempt decode
}

// Safe to call decode now
const decoded = program.coder.accounts.decode(...);
```

### Advisory

**3. Browser-side filtering (Medium)**

In `useProtocolState` hook, filter out objects with `decodeError` field (they are malformed):

```ts
useEffect(() => {
  const handleMessage = (event: MessageEvent) => {
    const { account, data } = JSON.parse(event.data, bigintReviver);

    // Reject any account state with a decode error
    if (data.decodeError) {
      console.warn(`[SSE] Ignoring malformed account ${account} with error: ${data.decodeError}`);
      return;
    }

    setAccounts((prev) => ({ ...prev, [account]: data }));
  };
  // ...
}, []);
```

**4. Sentry alerting for decode failures (Medium)**

When a decode failure occurs, alert on-call (it may indicate an attack or a schema mismatch that needs investigation).
