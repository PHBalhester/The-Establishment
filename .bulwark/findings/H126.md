# H126: EpochState Layout Coupling

**Severity:** Tier 3 (Low)
**Category:** Implementation Risk
**Status:** VERIFY
**Scope:** On-chain (epoch-program, tax-program deserialization)
**Related:** H104 (PoolState byte offset reads)

---

## Finding Summary

The Tax Program and Epoch Program rely on a precise EpochState struct layout for cross-program deserialization. If the struct is modified without updating both the source and mirror definitions, byte alignment drift could cause deserialization failures.

---

## Technical Details

### EpochState Definition (Source of Truth)

**File:** `programs/epoch-program/src/state/epoch_state.rs`

The Epoch Program defines EpochState with:
- `#[account]` and `#[repr(C)]` for binary layout determinism
- Compile-time assertion: `assert!(EpochState::DATA_LEN == 164)` (line 187)
- Total size: 8 (discriminator) + 164 (data) = **172 bytes**

Layout breakdown:
```
Timing (20 bytes):
  - genesis_slot: u64 (8)
  - current_epoch: u32 (4)
  - epoch_start_slot: u64 (8)

Tax Configuration (7 bytes):
  - cheap_side: u8 (1)
  - low_tax_bps: u16 (2)
  - high_tax_bps: u16 (2)
  [Derived taxes cached: 8 bytes]

VRF State (42 bytes):
  - vrf_request_slot: u64 (8)
  - vrf_pending: bool (1)
  - taxes_confirmed: bool (1)
  - pending_randomness_account: Pubkey (32)

Carnage State (23 bytes):
  - carnage_pending: bool (1)
  - carnage_target: u8 (1)
  - carnage_action: u8 (1)
  - carnage_deadline_slot: u64 (8)
  - carnage_lock_slot: u64 (8)
  - last_carnage_epoch: u32 (4)

Reserved Padding (64 bytes):
  - reserved: [u8; 64] -- DEF-03, absorbs future schema changes

Protocol (2 bytes):
  - initialized: bool (1)
  - bump: u8 (1)
```

**Ref:** Phase 47 (carnage_lock_slot +8), Phase 80 (reserved padding +64)

### Cross-Program Mirror

**File:** `programs/tax-program/src/state/epoch_state_reader.rs`

Tax Program imports a read-only EpochState mirror:
- Must match epoch-program's layout **exactly**
- Uses `#[account]` + `#[repr(C)]` + compile-time assertion (line 64)
- Assertion: `assert!(EpochState::DATA_LEN == 164)`
- Deserialized via Anchor's `AccountDeserialize` derive (implicit)

**Critical requirement:** Struct name must be exactly `"EpochState"` because Anchor's discriminator is derived from `sha256("account:EpochState")`. A mismatch (e.g., `"EpochStateReader"`) causes discriminator divergence → deserialization failure.

### Off-Chain Usage (Safe)

**File:** `app/lib/ws-subscriber.ts` + `app/hooks/useEpochState.ts`

Off-chain code:
- Uses Anchor's IDL-based deserialization (no raw byte offsets)
- IDL types auto-generated from on-chain struct via `anchor idl fetch`
- No coupling to struct byte layout

**Files:**
- `app/idl/types/epoch_program.ts` (auto-generated)
- `app/idl/types/tax_program.ts` (auto-generated)

## Risk Assessment

### Why This Is On-Chain Only

1. **On-chain deserialization:** Both Epoch and Tax Programs use Anchor's automatic `AccountDeserialize` via `#[account]` macro. Any struct layout change requires recompilation. The compile-time assertions catch mismatches at build time.

2. **Off-chain safety:** The off-chain code relies on **IDL-based deserialization**, not hard-coded byte offsets. IDL types are auto-generated from the Anchor build output, so they always reflect the current on-chain layout.

3. **No raw byte reads on EpochState:** Unlike PoolState (which uses raw byte offset reads in `pool_reader.rs` for cross-crate decoupling), EpochState is never read by raw offsets. Tax Program always deserializes the full account through Anchor.

### Failure Mode

If EpochState is modified **without updating the mirror**:
- Build succeeds locally (epoch-program compiles first)
- Tax Program compilation uses stale mirror struct
- On-chain, Tax Program's deserialization discriminator differs from Epoch Program's stored discriminator
- Cross-program account read fails: `AccountDiscriminatorMismatch` error
- Swap instructions revert with "Account not found" (misleading error)

**Severity:** Tier 3 because:
- Build-time assertions detect the mismatch immediately
- Devnet testing catches it before production
- No silent data corruption (deserialization fully fails)

✓ **Field order:** Identical in both files

## Verdict

**ACCEPTED_RISK**

### Rationale

1. **Compile-time enforcement:** The `#[account]` macro + `#[repr(C)]` + compile-time assertions (`const _: () = assert!(...)`) make struct drift impossible without explicit code changes and recompilation.

2. **No runtime coupling:** All deserialization happens through Anchor, which verifies the discriminator before attempting to decode. A mismatch is caught at the account-read level, not hidden in the data.

3. **Safe off-chain:** IDL-based deserialization decouples off-chain code from struct byte layout. The IDL is regenerated from the latest on-chain build, so it always matches.

4. **Defensive programming:** The two compile-time assertions (one in each file) serve as a gate. Any future developer modifying EpochState must update both or face a build failure.

### Mitigation (Developer Guidance)

- **Before modifying EpochState fields:**
  1. Update the source struct in `programs/epoch-program/src/state/epoch_state.rs`
  2. Update the mirror struct in `programs/tax-program/src/state/epoch_state_reader.rs` **immediately after**
  3. Verify DATA_LEN calculation matches in both files
  4. Run `cargo build` for both programs to confirm compile-time assertions pass
  5. Redeploy both programs together (they must have matching discriminators on-chain)

- **Reserved padding (DEF-03):** The 64-byte reserved field absorbs future schema changes without triggering discriminator changes. Add new fields by consuming reserved bytes, in order, to preserve byte alignment.

---

## Conclusion

This is an implementation risk that is **well-mitigated by compile-time assertions** and **out of scope for off-chain audit**. The coupling exists only in on-chain code, where Rust's type system and Anchor's macros enforce correctness. No runtime vulnerability exists.

**Recommendation:** Document this guidance in a CONTRIBUTING.md or similar developer guide, and carry the testing gap to v1.4 if automated struct alignment tests are desired.
