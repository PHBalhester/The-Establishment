# H058: Webhook Type Confusion

## Verdict: CONFIRMED

## Evidence

**The type discrimination at lines 339-342 is a single-field presence check with no priority or ambiguity guard. A payload element containing both `accountData` and `signature` is unconditionally routed to the enhanced (weaker) handler.**

1. **The discrimination logic (lines 339-342):**
   ```typescript
   const firstItem = payload[0] as Record<string, unknown> | undefined;
   if (firstItem && "accountData" in firstItem) {
     return handleAccountChanges(payload as HeliusAccountChange[]);
   }
   // falls through to raw TX path
   ```
   The check is `"accountData" in firstItem` — a single key presence test. If `firstItem` has *both* `accountData` AND `signature`, the enhanced branch fires and the raw TX path is never reached. There is no ambiguity detection, no dual-field check, and no fallback to the stronger handler.

2. **The two handlers have materially different validation strength:**

   | Property | Raw TX path | Enhanced path (`handleAccountChanges`) |
   |---|---|---|
   | Replay protection (blockTime age check) | YES — MAX_TX_AGE_SECONDS=300, lines 360-385 | NONE |
   | Signature validation | YES — `tx.signature` required, line 367 | NONE |
   | Error/log guard | YES — `tx.meta?.err` check, line 373 | NONE |
   | Array element limit | Inherits loop, skips on missing fields | No per-element guard |

   A payload that bypasses the raw TX path avoids the 5-minute replay window enforced by H049.

3. **The crafted payload structure that triggers the confusion:**
   ```json
   [
     {
       "signature": "REAL_OR_REPLAYED_TX_SIG",
       "blockTime": 1700000000,
       "meta": { "logMessages": ["Program log: ..."], "err": null },
       "transaction": { "signatures": ["REAL_OR_REPLAYED_TX_SIG"] },
       "accountData": [{ "nativeBalanceChange": 0 }]
     }
   ]
   ```
   The presence of `accountData` sends this to `handleAccountChanges`. The enhanced handler then looks up `item.account` (undefined here) in `KNOWN_PROTOCOL_ACCOUNTS`, gets `undefined` → logs a warning and skips — so this specific shape results in a no-op rather than data injection. However, see attack paths below.

4. **Two concrete exploitation shapes:**

   **Shape A — Stale enhanced payload replay (higher-impact):**
   An attacker replays a legitimately-captured enhanced webhook payload (which contains `accountData` natively). The raw TX path would have rejected it via the 5-minute blockTime check. The enhanced path has no such guard. The replayed decoded state (stale pool reserves, epoch counters, carnage fund balance) is written to `protocolStore` and broadcast to all SSE subscribers. This is the no-auth path to corrupt the live UI state.

   **Shape B — Hybrid payload injection (requires webhook secret):**
   An attacker holding the webhook secret crafts a payload with both `accountData` and a valid `account` PDA address (e.g., a CRIME_SOL pool PDA), along with `rawAccountData.data` containing attacker-controlled base64. This is routed to the enhanced handler's Anchor decode path (lines 577-621). If the base64 decodes successfully as a `PoolState`, the decoded struct overwrites the protocol store entry and broadcasts fake reserve values to all connected browsers — manipulating every quote displayed in the UI.

5. **Shape A requires no special privileges.** Any past Helius webhook delivery is replayable because the shared secret is static and the auth check at lines 266-302 only validates the secret, not the payload's age or content hash. The type confusion is what allows the replay to succeed — the enhanced path is taken, sidestepping the H049 blockTime check that would have blocked it on the raw TX path.

6. **The comment in the code at line 379 explicitly acknowledges the gap:**
   ```typescript
   // H049: Skip stale transactions (blockTime older than 5 minutes).
   // Prevents replay attacks where old webhook payloads are re-sent.
   // Enhanced account change webhooks skip this (handled separately above).
   ```
   The phrase "handled separately above" implies the enhanced path was *intended* to have its own protection, but none was implemented.

## Attack Path (Shape A — Replay via Type Confusion)

1. Attacker observes a legitimate Helius enhanced webhook POST (e.g., captured via a MITM proxy during a dev session, or obtained from logs)
2. The captured payload element natively contains `accountData` (it is a real enhanced payload)
3. Attacker replays the payload to `POST /api/webhooks/helius` with the same `Authorization: <secret>` header (secret is static; no rotation mechanism exists)
4. Line 340: `"accountData" in firstItem` → true → `handleAccountChanges` is called
5. The H049 blockTime check on lines 380-385 is **never reached** — it lives exclusively in the raw TX path
6. `handleAccountChanges` looks up `item.account` in `KNOWN_PROTOCOL_ACCOUNTS` — if the account is a recognized protocol PDA, it Anchor-decodes the (stale) `rawAccountData.data` and calls `protocolStore.setAccountState`
7. `protocolStore` overwrites its live state with the stale values, dedup check passes if state differs from current (which it likely does — pool reserves have moved since capture)
8. `sseManager` broadcasts stale pool reserves, stale epoch state, or stale carnage fund balance to all connected SSE clients
9. Browser UIs re-render with stale data — swap quotes are miscalculated, epoch timers are wrong, carnage fund balance is misreported

## Impact

**Tier 2 — HIGH.**

- **Replay attacks on SSE state pipeline**: Stale protocol state is injected into the in-memory store and broadcast to all connected browser clients. Quote accuracy depends on pool reserves from SSE; stale reserves cause users to see incorrect prices and potentially submit TXs with out-of-date minimumOutput parameters. The on-chain slippage check is the last line of defence, but incorrect UI state is a meaningful UX and trust failure.
- **No authentication bypass is required for Shape A**: The webhook secret is static with no rotation mechanism. Any party who has ever observed a webhook delivery (Helius employee, intercepted dev traffic, logged request) can replay it indefinitely.
- **No time window closes this**: Because there is no blockTime check on the enhanced path and no nonce/sequence counter, a valid captured payload remains exploitable forever.
- **Blast radius**: All SSE subscribers see the corrupted broadcast simultaneously. On mainnet with many active users, this affects every connected browser.

## Recommended Fix

Apply the same age check to the enhanced path, using the `slot` field present on enhanced payloads (or a server-received timestamp as a conservative fallback):

```typescript
// In handleAccountChanges, before processing each item:
const MAX_ENHANCED_AGE_MS = 5 * 60 * 1000; // 5 minutes
const receivedAt = Date.now();

for (const item of accounts) {
  // Reject if payload slot implies age > 5 minutes
  // (or use a top-level timestamp if Helius provides one)
  if (item.timestamp && (receivedAt - item.timestamp * 1000) > MAX_ENHANCED_AGE_MS) {
    console.warn(`[webhook] Skipping stale enhanced payload for ${item.account}`);
    continue;
  }
  // ... existing processing
}
```

Additionally, harden the type discrimination to treat ambiguous payloads as errors rather than silently routing to the weaker path:

```typescript
const hasAccountData = firstItem && "accountData" in firstItem;
const hasSignature = firstItem && ("signature" in firstItem || "transaction" in firstItem);

if (hasAccountData && hasSignature) {
  // Ambiguous payload — reject rather than silently prefer one path
  return NextResponse.json({ error: "Ambiguous payload type" }, { status: 400 });
}
if (hasAccountData) {
  return handleAccountChanges(payload as HeliusAccountChange[]);
}
// ... raw TX path
```
