## Hypothesis

If an attacker controls the Helius webhook payload, a field with value
`{ __bigint: "0" }` bypasses normal type expectations after JSON round-trip,
converting an object to BigInt via the custom `bigintReviver` function. This
causes type confusion in downstream React components that consume SSE data.

## Verdict: CONFIRMED (prerequisite required)

The injection chain is structurally real. Exploitation requires bypassing the
webhook HMAC authentication gate. Given that gate is enforced via
`timingSafeEqual` (fail-closed in production), practical likelihood is low —
but the underlying code path is undefended once the gate is passed.

---

## Attack Chain

### Step 1 — Webhook auth bypass (prerequisite)

`app/app/api/webhooks/helius/route.ts:286-301` performs HMAC verification using
`timingSafeEqual` and is fail-closed in production (rejects all requests if
`HELIUS_WEBHOOK_SECRET` is unset). An attacker must either:

- Obtain the `HELIUS_WEBHOOK_SECRET` env var (e.g., via secret disclosure), OR
- Exploit the dev-mode bypass where `NODE_ENV !== "production"` skips auth.

### Step 2 — Crafted payload triggers fallback path

The fallback path at `app/app/api/webhooks/helius/route.ts:564-619` activates
when either:
- `decodeInfo` is missing for a known pubkey, OR
- `item.rawAccountData?.data` is absent.

In this path the handler stores webhook-controlled objects directly:

```typescript
// route.ts:566-572 (missing decode info or data path)
protocolStore.setAccountState(pubkey, {
  label,
  accountData: item.accountData,      // attacker-controlled
  rawAccountData: item.rawAccountData, // attacker-controlled
  updatedAt: Date.now(),
  decodeError: "missing_decode_info_or_data",
});
```

A crafted payload such as:

```json
[{
  "account": "<any Base58 pubkey>",
  "accountData": [{ "__bigint": "0" }],
  "rawAccountData": null
}]
```

places `[{ "__bigint": "0" }]` as the value of `accountData` in the protocol
store without any sanitisation.

### Step 3 — SSE serialization preserves the tag

`app/lib/protocol-store.ts:53-65` calls `sseManager.broadcast("protocol-update", { account, data })`.

`app/lib/sse-manager.ts:61` serializes with `JSON.stringify(data, bigintReplacer)`.
`bigintReplacer` only activates for `typeof value === "bigint"` — plain objects
are passed through unchanged. The `{ "__bigint": "0" }` tag survives intact in
the SSE wire payload.

### Step 4 — Client reviver fires on crafted tag

`app/hooks/useProtocolState.ts:269` parses the SSE event:

```typescript
const { account, data } = JSON.parse(event.data, bigintReviver) as { ... };
```

`bigintReviver` at `app/lib/bigint-json.ts:46-51` checks
`isBigIntTag(value)` — which matches any object where:
- `typeof v === "object"`
- `v !== null`
- `"__bigint" in v`
- `typeof v.__bigint === "string"`

The crafted `{ "__bigint": "0" }` passes all four checks. `BigInt("0")` is
returned, converting the expected array `[{ "__bigint": "0" }]` element to
`BigInt(0)`.

The test at `app/lib/__tests__/bigint-json.test.ts:48-56` explicitly
demonstrates and acknowledges this behaviour as a "known trade-off".

---

## Impact Analysis

### Numeric string injection — silent type substitution

`BigInt("0")` succeeds silently. Any field value that was an object is
now the BigInt `0n`. Downstream consumers that perform arithmetic on it
(without type guards) would produce incorrect results.

### Non-numeric string injection — controlled parse error

`BigInt("malicious")` throws a `SyntaxError` inside `JSON.parse`. The
`try/catch` in `useProtocolState.ts:252` and `:268` catches this and
silently discards the update. The UI falls back to stale/null state. This
is a denial-of-display, not a financial error.

### Downstream guard effectiveness

`useCurveState.ts:162-170` and `useStaking.ts:248-251` both guard BigInt
fields with runtime `typeof data.X === "bigint"` checks and fall back to
`BigInt(0)` if the check fails. An injected `BigInt(0)` would pass the
`typeof === "bigint"` check and appear as a zero value — suppressing
real data (zeroed-out balance/staking displays).

A targeted attack crafting `{ "__bigint": "999999999999999999" }` into a
field that normally holds a small number (e.g. `solRaised` during early
bonding curve phase) would display a falsely inflated figure in the UI,
potentially inducing panic selling or manipulation of user decisions. The
on-chain truth is unaffected — but the display layer shows corrupted data.

## Root Cause

The `bigintReviver` has no mechanism to distinguish:
- Tags produced by the trusted `bigintReplacer` (server-generated, from
  Anchor-decoded account data), versus
- Tags that arrive from untrusted webhook payload contents that were stored
  verbatim in the fallback path.

The fallback path stores `item.accountData` and `item.rawAccountData` as
opaque values without scanning for or stripping `__bigint` sentinel
patterns.

| SSE client connected | Required — browser must be listening when broadcast fires |

## Recommended Mitigations

### Mitigation 1 — Strip `__bigint` tags from fallback-path data (primary fix)

Before storing webhook-derived data in the fallback path, strip any
`__bigint` keys from nested objects:

```typescript
function stripBigIntTags(obj: unknown): unknown {
  if (Array.isArray(obj)) return obj.map(stripBigIntTags);
  if (obj && typeof obj === "object") {
    const result: Record<string, unknown> = {};
    for (const [k, v] of Object.entries(obj as Record<string, unknown>)) {
      if (k === "__bigint") continue; // strip sentinel
      result[k] = stripBigIntTags(v);
    }
    return result;
  }
  return obj;
}
```

Apply to `item.accountData` and `item.rawAccountData` before passing them
to `protocolStore.setAccountState`.

### Mitigation 2 — Use a less-guessable sentinel (defence in depth)

Replace `__bigint` with a randomly-generated, build-time constant such as
`__bf_<random_hex>`. This prevents guessing the sentinel from reading the
open-source code. Update both `isBigIntTag`, `bigintReplacer`, and all
tests.

### Mitigation 3 — Narrow the reviver to known account key paths

Instead of a global reviver that fires on every value in the parsed tree,
apply BigInt conversion only to fields in `CURVE_BIGINT_FIELDS` and
`STAKING_BIGINT_FIELDS` by key name. The reviver receives the key as its
first argument:

```typescript
const BIGINT_FIELD_SET = new Set([
  ...CURVE_BIGINT_FIELDS, ...STAKING_BIGINT_FIELDS,
]);

export function bigintReviver(key: string, value: unknown): unknown {
  if (BIGINT_FIELD_SET.has(key) && isBigIntTag(value)) {
    return BigInt(value.__bigint);
  }
  return value;
}
```

This limits tag conversion to known schema fields, preventing injection via
arbitrary nested objects.

## Severity Assessment

| Dimension | Assessment |
|---|---|
| Confidentiality | No impact |
| Integrity | Medium — display layer data corruption (UI shows zeroed or inflated values) |
| Availability | Low — non-numeric injection causes stale/null display state |
| Exploitability | Low — requires HMAC bypass + known pubkey + fallback path trigger |
| On-chain safety | Unaffected — on-chain programs enforce correct values |

**Overall: MEDIUM** — The finding is confirmed as a real code path with
demonstrable type confusion behaviour. The on-chain enforcement backstop
prevents financial loss, but a UI-level display attack is achievable given
the prerequisite. The fix is straightforward (Mitigation 1) and should be
applied as part of any pre-mainnet hardening pass.
