# H108: Crank Health Endpoint Binds 0.0.0.0

**Severity**: Tier 3 (Low)
**Category**: Infrastructure / Network Exposure
**Target**: `scripts/crank/crank-runner.ts:185`
**Status**: CONFIRMED

## Vulnerability Description

The health check HTTP server in the crank runner binds to `0.0.0.0` on port 8080 (configurable via `HEALTH_PORT` env var), making it accessible from any network interface.

**Code Location** (`scripts/crank/crank-runner.ts:185`):
```typescript
server.listen(HEALTH_PORT, "0.0.0.0", () => {
  console.log(`[health] Listening on :${HEALTH_PORT}/health`);
});
```

## Exposed Data

The `/health` endpoint returns internal state (lines 166-176):
```json
{
  "status": "running|halted",
  "consecutiveErrors": <number>,
  "circuitBreakerThreshold": <number>,
  "hourlySpendLamports": <number>,
  "maxHourlySpendLamports": <number>,
  "uptime": <number>,
  "lastSuccessAt": <ISO8601 timestamp>
}
```

## Risk Analysis

### Current Deployment Context (Railway)
- **No public domain**: Railway assigns `internal` CNAME; traffic routes through Railway's private network only
- **No port forwarding**: Port 8080 not exposed to public internet by default
- **Attack surface**: Internal Railway network only

### Hypothetical Risk
If Railway configuration changes or traffic is routed through a public gateway:
- **Information disclosure**: Spending patterns, error frequency, uptime, last successful execution time could be leaked
- **Operational reconnaissance**: Attacker gains visibility into crank health/stability for timing attacks
- **Non-critical**: Endpoint exposes no secrets, wallet addresses, transaction data, or RPC credentials

## Verdict

**CONFIRMED** — Binds to `0.0.0.0` as documented in comment (line 156: "Binds to 0.0.0.0 so Railway's internal probes can reach it")

However, **risk is MITIGATED** by deployment architecture:
- Railway's internal health checks require private network binding (`0.0.0.0`)
- Public internet has no route to port 8080 (no public domain, no port exposure)
- Exposed data contains no secrets or critical operational info

## Recommendation

**ACCEPT RISK** — Current configuration is intentional and safe for Railway deployment.

If migrating to a different hosting provider, verify:
1. Health endpoint remains on private/internal network only
2. Consider binding to `127.0.0.1` + Unix socket for local-only access (if provider supports it)
3. Document the binding choice in deployment runbooks

**No code change required for current Railway deployment.**
