## Summary

Wallet adapter icons are rendered as `<img src={wallet.adapter.icon}>` in `ConnectModal.tsx:125-132` and `WalletStation.tsx:64-69`. A malicious wallet extension could provide a tracking pixel URL instead of a legitimate icon, causing the browser to make an HTTP(S) request to an attacker-controlled domain, leaking that the user visited this dApp.

**Verdict: NOT_VULNERABLE** — CSP `img-src` directive and protocol implementation details prevent this risk.

---

## Root Cause

In both `ConnectModal.tsx` and `WalletStation.tsx`, wallet adapter icons are rendered without validation:

```tsx
{wallet.adapter.icon && (
  <img
    src={wallet.adapter.icon}
    alt={wallet.adapter.name}
    width={28}
    height={28}
    className="rounded-md"
  />
)}
```

The `wallet.adapter.icon` property is provided by the wallet extension itself (as part of the wallet-standard provider interface). A malicious extension could set this to a tracking pixel URL like:
- `https://attacker.com/track?user=<some-identifier>`
- `https://analytics.evil.com/log-dapp-visit`

When rendered, the browser would fetch the image, exposing the user's IP address and visit to the attacker.

## Risk Assessment

### Why This Looks Like a Risk

1. **External URL in `src` attribute**: The `img` tag can load from external domains.
2. **No URL validation**: The code does not check if the URL is a data URI, local path, or external URL.
3. **Tracking pixel pattern**: An attacker can encode analytics or surveillance in a 1x1 pixel URL.

### Why It's NOT a Risk (NOT_VULNERABLE)

#### 1. CSP `img-src` Restriction

The application has a strict Content Security Policy (CSP) defined in `app/next.config.ts:35`:

```typescript
img-src 'self' data: blob:;
```

This CSP directive **only allows**:
- `'self'` — same-origin images (served from `fraudsworth.fun` or `dr-fraudsworth-production.up.railway.app`)
- `data:` — data URIs (e.g., `data:image/svg+xml;base64,...`)
- `blob:` — blob URLs created by JavaScript

**External URLs like `https://attacker.com/track` are blocked by CSP.** The browser will refuse to load the image and will log a CSP violation. No HTTP request to the attacker's server occurs.

#### 2. Wallet Adapter Icons Are Data URIs

In practice, wallet-standard implementations provide icons as **data URIs**, not external URLs. For example:

```javascript
// Typical wallet-standard icon
wallet.adapter.icon = "data:image/svg+xml;base64,PHN2ZyB..."
```

Data URIs are **processed locally** — no external network request. They satisfy the CSP `data:` allowance.

#### 3. Local Fallback for Mobile Wallets

For mobile wallets in `ConnectModal.tsx:141-167` and `WalletStation.tsx:79-102`, the code explicitly uses local static files:

```typescript
// From app/lib/mobile-wallets.ts
const MOBILE_WALLETS = [
  {
    name: "Phantom",
    icon: "/wallets/phantom.png",  // Local static file
    deepLink: (url: string) => ...
  },
  // ...
];
```

These local paths (`/wallets/phantom.png`, etc.) are served from the origin and satisfy CSP `'self'`. They bypass any attacker control.

## Attack Scenario (Blocked)

**Hypothetical attack:**

1. User installs a malicious browser extension posing as "Phantom Wallet."
2. The extension registers with wallet-standard as a wallet provider.
3. It sets `adapter.icon = "https://attacker.com/track?visit_time=2026-03-21"`.
4. User clicks "Connect Wallet" on the dApp.
5. `ConnectModal` renders the icon: `<img src="https://attacker.com/track?..." />`.
6. **CSP blocks the request.** Browser console logs: `Content Security Policy: image-from-blocked-source`.
7. The icon fails to load (user sees a broken image icon), but no data is leaked.

Without CSP, the tracking request would succeed.

## CSP Context

From `app/next.config.ts:31-47`:

```typescript
const cspHeader = `
  default-src 'self';
  script-src 'self' 'unsafe-inline';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: blob:;           // <-- BLOCKS tracking pixels
  font-src 'self';
  object-src 'none';
  base-uri 'self';
  form-action 'self';
  frame-ancestors 'none';
  // ... other directives
`;
```

The `img-src 'self' data: blob:;` directive is the critical defense:
- **No external domains** (not `https://*` or unsafe-domain lists)
- **No `unsafe-url`** (which would allow any URL)
- **Only local and embedded images** are loaded

---

## Recommendation

**No code changes required.** The CSP policy is sufficient to prevent tracking pixel attacks. However, maintainers should:

1. **Monitor CSP violations** via error reporting (Sentry logs CSP violations by default)
2. **Document this defense** in security runbook (already noted in this finding)
3. **Test CSP enforcement** during security regression testing to ensure it's not accidentally relaxed

The design is **defense-in-depth**:
- CSP blocks external URLs
- Wallet-standard typically provides data URIs
- Mobile wallets use local static files
- Result: tracking pixels are not possible even with malicious wallet
