# H079: ATA Creation TOCTOU Race (Single-Hop)

**Severity**: Tier 3 (Low) | **Category**: Race Condition / Concurrency
**Target**: `app/lib/swap/swap-builders.ts:229-240, 463-474`
**Status**: CONFIRMED — Potential race condition, low impact

---

## Vulnerability Description

The swap builders check for ATA existence, then create if missing using a check-then-act pattern:

```typescript
// Line 228-240 (SOL buy path)
const userTokenB = await getToken2022Ata(userPublicKey, tokenMint);
const userTokenBInfo = await connection.getAccountInfo(userTokenB);
if (!userTokenBInfo) {
  tx.add(
    createAssociatedTokenAccountInstruction(
      userPublicKey, // payer
      userTokenB,    // ATA address
      userPublicKey, // owner
      tokenMint,     // mint
      TOKEN_2022_PROGRAM_ID,
    ),
  );
}
```

This pattern occurs in **3 locations**:
1. `buildSwapSolBuyTx()` line 229-240 (userTokenB for output)
2. `buildSwapSolSellTx()` line ~341 (userTokenB for input)
3. `buildVaultConvertTx()` line 463-474 (userOutputAccount for PROFIT)

**Race Condition**: Between `getAccountInfo()` (line 229) and transaction submission, another transaction could create the same ATA. If both TXs are confirmed:
- First TX: Creates ATA, succeeds
- Second TX: Attempts `createAssociatedTokenAccountInstruction()` on already-existing ATA → **Account Already In Use** error

---

## Root Cause

The RPC query (getAccountInfo) is **not atomic** with transaction submission. This is a classic Time-Of-Check-Time-Of-Use (TOCTOU) race:

```
Timeline:
T0: Frontend reads RPC → ATA doesn't exist
T1: Another user/client creates same ATA → account now exists on-chain
T2: First client submits TX with CreateATA instruction → fails with "Account Already In Use"
```

---

## Impact Assessment

**Actual Risk**: **Low**
- **Why it's unlikely**: ATAs are deterministically derived from `(owner, mint)`. Only the **same user signing with the same keypair** will generate the same ATA address.
- **Multi-device scenario**: If the same user opens the app in 2 browser tabs simultaneously and initiates swaps:
  - Both check ATA at T0 (both see it doesn't exist)
  - Both create CreateATA instruction
  - First TX succeeds, second TX fails
  - **User sees 1:1 TX failure rate** if they spam swaps in parallel

**Occurrence Likelihood**: Very low in production (single-user, single-device typical usage). Higher in:
- Bots/scripts that batch-send TXs
- Multi-tab concurrent swaps (same user, same wallet)
- Testing/stress scenarios

## Recommended Fix

**Use idempotent ATA creation**. Solana `@solana/spl-token` provides:

```typescript
import { createAssociatedTokenAccountIdempotentInstruction } from "@solana/spl-token";

// Replace 3 occurrences of:
// createAssociatedTokenAccountInstruction(...)
// with:
// createAssociatedTokenAccountIdempotentInstruction(...)
```

**Idempotent behavior**: If the ATA already exists, the instruction succeeds silently (consumes compute but does not fail). If it doesn't exist, it creates it.

### Changed Lines

**app/lib/swap/swap-builders.ts line 232** (SOL buy):
```diff
- createAssociatedTokenAccountInstruction(
+ createAssociatedTokenAccountIdempotentInstruction(
```

**app/lib/swap/swap-builders.ts line ~341** (SOL sell):
```diff
- createAssociatedTokenAccountInstruction(
+ createAssociatedTokenAccountIdempotentInstruction(
```

**app/lib/swap/swap-builders.ts line 466** (vault convert):
```diff
- createAssociatedTokenAccountInstruction(
+ createAssociatedTokenAccountIdempotentInstruction(
```

### Alternative: Skip the Check Entirely

Since idempotent creation is available, the RPC check is **redundant**:

```typescript
// Remove lines 229-230 and 463-464 entirely
// Always add the instruction — idempotency handles it
tx.add(createAssociatedTokenAccountIdempotentInstruction(...));
```

This **eliminates RPC latency**, reduces calls, and fully eliminates the race condition.
