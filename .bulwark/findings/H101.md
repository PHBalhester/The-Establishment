# H101: iframe sandbox allow-same-origin + allow-scripts

**Severity**: Tier 3 (High)
**Status**: CONFIRMED
**Files**:
- `app/components/launch/DocsModal.tsx:105`
- `app/components/station/DocsStation.tsx:66`

## Vulnerability

Both iframe elements are sandboxed with **`allow-scripts allow-same-origin allow-popups`**. This combination is **dangerous**:

### The Problem

When `allow-same-origin` is set alongside `allow-scripts`, a compromised docs site can:
1. **Access parent window via `window.parent`** (same-origin origin check passes)
2. **Call parent JavaScript** and access parent's DOM
3. **Steal wallet connections** (Phantom or other adapters in parent window)
4. **Modify UI** to trick users into signing malicious transactions
5. **Read localStorage/sessionStorage** if docs site shares origin

### Attack Scenario

1. Attacker compromises the docs site (malicious redirect, DNS hijack, supply chain attack)
2. Docs iframe executes attacker's JavaScript with `allow-scripts`
3. Script calls `window.parent.postMessage()` or direct DOM access
4. Parent app is running Solana wallet adapter → attacker can intercept `signTransaction()` calls
5. User unknowingly signs a malicious transaction (drain wallet, authorize token transfer, etc.)

## Evidence

**DocsModal.tsx:105**
```jsx
<iframe
  src={DOCS_URL}
  title="Dr. Fraudsworth Documentation"
  className="w-full h-[calc(100%-48px)] border-0"
  sandbox="allow-scripts allow-same-origin allow-popups"
/>
```

**DocsStation.tsx:66**
```jsx
<iframe
  src={DOCS_URL}
  title="Dr. Fraudsworth Documentation"
  className="w-full border-0 rounded"
  style={{ height: '60vh', minHeight: '280px' }}
  sandbox="allow-scripts allow-same-origin allow-popups"
  onLoad={handleLoad}
/>
```

Both use identical sandbox attributes.

## Risk Assessment

- **Docs URL**: Falls back to `/docs` (same-origin) or `NEXT_PUBLIC_DOCS_URL` env var
- **Wallet exposure**: Parent app has Phantom/Solana adapters running → direct access to signing functions
- **User impact**: Financial loss (stolen SOL, tokens, NFTs)
- **Probability**: Medium (requires docs site compromise)

## Recommended Fix

**Remove `allow-same-origin` OR remove `allow-scripts`.**

### Option A: Maximum Security (Recommended)
```jsx
sandbox="allow-popups"
```
- Docs site can open links in new windows (`allow-popups`)
- Cannot execute scripts in isolated origin
- Cannot access parent window
- **Tradeoff**: May break Nextra's client-side features (search, navigation)

### Option B: Moderate Security
```jsx
sandbox="allow-scripts allow-popups"
```
- Docs scripts can run (Nextra works fully)
- Cannot access parent window (different origin isolation)
- **Requires**: Docs site must be on **different domain** (e.g., `docs.example.com` NOT `example.com`)

### Option C: If same-origin is unavoidable
Use `postMessage()` API with strict origin validation instead of same-origin iframe:
```jsx
// Parent (app)
const iframe = useRef<HTMLIFrameElement>(null);

const handleMessage = (e: MessageEvent) => {
  // CRITICAL: Validate origin
  if (e.origin !== allowedDocsOrigin) return;
  // Handle message safely
};

useEffect(() => {
  window.addEventListener('message', handleMessage);
  return () => window.removeEventListener('message', handleMessage);
}, []);

// Child (docs iframe)
// window.parent.postMessage({ action: 'openWallet' }, targetOrigin);
```

## Implementation Notes

1. **Nextra compatibility**: Test Option A locally first. If navigation breaks, use Option B with separate docs domain.
2. **Env var**: `NEXT_PUBLIC_DOCS_URL` is already in use (good practice). Ensure it points to a different origin on production.
3. **Both files need fix**: Changes must be made to both `DocsModal.tsx` and `DocsStation.tsx`.
