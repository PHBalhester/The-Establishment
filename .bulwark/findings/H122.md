# H122: Staking Escrow Monitoring Absent

**Finding ID:** H122
**Risk Tier:** Tier 3 (Informational/Accepted Risk)
**Status:** CONFIRMED
**Last Verified:** 2026-03-21
**Audit Cycle:** v1.3 → Ongoing
**Related:** H017 (original observation in BOT-01 context audit)

---

## Summary

The crank runner (`scripts/crank/crank-runner.ts`) does **not monitor the staking escrow (StakePool PDA) rent balance**. The StakePool is the central PDA that holds all staked tokens and state. If its rent balance were to be depleted, it could become non-rent-exempt and risk being garbage-collected by the Solana runtime.

**Current monitoring scope** (in `crank-runner.ts`):
- ✅ Carnage SOL vault balance (lines 414-448)
- ✅ Crank wallet SOL balance (lines 403-410)
- ✅ Hourly spending cap (lines 139-150)
- ❌ **StakePool PDA rent balance** (missing)

---

## Investigation Details

### 1. Crank Runner Monitoring Architecture

**File:** `scripts/crank/crank-runner.ts:390-553` (main loop)

The crank's main loop executes every epoch cycle:

```typescript
while (!shutdownRequested) {
  // 1. Read current state
  epochState = await readEpochState(...)

  // 2. Check wallet balance (log warning if low)
  const balance = await provider.connection.getBalance(provider.wallet.publicKey)
  if (balance < LOW_BALANCE_SOL * LAMPORTS_PER_SOL) {
    console.log(`[crank] WARNING: Wallet balance low: ...`)
  }

  // 2b. Check carnage vault balance — top up if needed
  const vaultBalance = await provider.connection.getBalance(vaultPubkey)
  if (vaultBalance < MIN_VAULT_BALANCE) {
    // Top up with capped amount (5M lamports, max 0.1 SOL)
  }

  // 3-4. Wait for epoch boundary and advance epoch
  // ... (no other balance checks)
}
```

**Gap identified:** The StakePool PDA is not queried for rent balance.

### 2. StakePool as Critical State Container

**File:** `programs/staking/src/state/stake_pool.rs`

The StakePool PDA is a data account (not a system account) that holds:
- **Staked CRIME and FRAUD tokens** (in token vaults)
- **Epoch reward state** (last reward, cumulative reward per stake)
- **Staker count and total stake**
- **Admin configuration**

The struct size is approximately 300+ bytes, requiring:
- **Rent-exempt minimum:** ~2.8–3.2 SOL (0-data accounts need ~0.9 SOL; 300-byte account needs additional rent)

If the StakePool account falls below rent-exempt minimum, the Solana runtime can garbage-collect it (though this is unlikely in practice because the account accumulates rent credits over time and is never directly debited).

### 3. Rent Risk Scenarios

**Scenario A: Legitimate Rent Depletion (Very Low Risk)**
- Solana rent model: accounts accumulate rent credits when they're above rent-exempt minimum
- A rent-exempt StakePool remains safe indefinitely unless the cost of rent-exemption increases (which would require a network upgrade)
- **Likelihood:** ~0% (Solana has not increased rent-exempt costs since mainnet launch)

**Scenario B: Account State Corruption (Unlikely but Unmonitored)**
- If the StakePool account's lamports were somehow directly debited (via erroneous transfer instruction or exploit), it could fall below rent-exempt minimum
- Currently, **no on-chain instruction drains the StakePool account's own SOL balance** — the staking program only transfers tokens, not the account's native SOL
- **Likelihood:** ~0% (not a code path in the staking program)

**Scenario C: Operational Awareness Gap (Actual Risk)**
- The crank has no visibility into the StakePool's rent status
- If a future protocol upgrade or unexpected on-chain behavior depletes the account, the crank would not alert
- **Likelihood:** ~5% (informational; operational awareness is valuable even if depletion is unlikely)

## Assessment

### Why This Matters (Context from H017)

H017 was originally documented in the BOT-01 context audit (`.bulwark/context/09-BOT-01-keeper-crank.md:19`) as:

> | H017 (Staking escrow rent) | NOT_FIXED | Crank doesn't monitor staking escrow |

This observation was marked as a **gap in operational observability**, not a critical vulnerability. The distinction is important:

- **Critical vulnerability:** Code could directly cause rent depletion ❌ (does not exist)
- **Operational gap:** No monitoring/alerting for a low-probability but high-impact event ✅ (confirmed present)

### Why It's Low Risk in Practice

1. **Rent-exempt status is stable:** Once above rent-exempt minimum, accounts remain safe indefinitely without intervention
2. **No active depletion mechanism:** The staking program never drains the StakePool's own SOL balance
3. **Network guarantees:** Solana's economics assume rent-exempt accounts stay safe; protocols don't typically monitor this
4. **Historical precedent:** No major Solana protocol monitors individual PDA rent balances in production

### Why Monitoring Would Be Valuable

1. **Early warning:** If an unexpected exploit or on-chain behavior drains the account, alert immediately
2. **Operational completeness:** The crank already monitors 2 other critical balances (wallet, carnage vault); StakePool would be a third data point
3. **Mainnet readiness:** Pre-mainnet, ensuring complete observability reduces post-launch surprises

## Recommendation

**Tier 3 — Informational / Accepted Risk**

Three options, in priority order:

### Option 1: Document and Accept (PREFERRED)
- Accept this as a low-risk operational gap
- Document it in the operational runbook so the crank operator is aware that rent-exempt status is expected to remain stable
- No code change required
- **Rationale:** The risk is negligible; Solana's rent model guarantees stability

### Option 2: Add Periodic Monitoring (NICE-TO-HAVE)
- Add a health check in the crank main loop (every N epochs, or on startup):
  ```typescript
  const stakePoolBalance = await provider.connection.getBalance(vrfAccounts.stakePoolPda);
  const stakePoolRentExempt = await provider.connection.getMinimumBalanceForRentExemption(300); // approx
  if (stakePoolBalance < stakePoolRentExempt) {
    console.error('[crank] CRITICAL: StakePool below rent-exempt minimum!');
    // Optional: top up from treasury / admin
  } else {
    console.log(`[crank] StakePool rent-safe: ${stakePoolBalance} lamports (min: ${stakePoolRentExempt})`);
  }
  ```
- Cost: ~1 RPC call per epoch cycle (negligible)
- Benefit: Complete observability of all three critical balances

### Option 3: Top-Up Strategy (OVER-ENGINEERING)
- Similar to vault top-up, periodically re-fund the StakePool if it falls below a threshold
- **Not recommended:** StakePool will never deplete in practice; this would be defensive programming against a non-existent threat
- Cost: Unnecessary SOL spending
- Benefit: Maximum safety margin (not justified)
