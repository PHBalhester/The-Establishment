# H008: RPC Proxy Batch Amplification → Helius Credit Exhaustion

## Verdict: CONFIRMED

## Evidence

### Batch Passthrough — No Size Limit

`app/app/api/rpc/route.ts:102-106` explicitly detects and supports JSON-RPC batch arrays:

```typescript
// Support single requests and batch requests
const isBatch = Array.isArray(body);
const requests: Array<...> = isBatch ? (body as Array<...>) : [body as Record<string, unknown>];
```

The batch array is accepted unconditionally. There is no upper bound on `requests.length`. A `body` of `[...×500]` becomes a `requests` array of 500 items, all forwarded to Helius as a single upstream HTTP POST (`bodyStr = JSON.stringify(body)`, line 139).

### Rate Limiter Counts HTTP Requests, Not RPC Method Calls

`app/lib/rate-limit.ts:81-112` — `checkRateLimit` is called once per HTTP request (line 83 of `route.ts`), before the batch is unpacked. The sliding window increments by 1 for the entire HTTP call regardless of how many JSON-RPC methods are contained in the body:

```typescript
entry.timestamps.push(now);  // one timestamp per HTTP request
return { allowed: true };
```

The profile is `RPC_RATE_LIMIT = { windowMs: 60_000, maxRequests: 300 }`. An attacker may make 300 HTTP requests per minute. Each HTTP request may carry an arbitrary-length batch array. The rate limiter and the upstream credit cost are completely decoupled.

### Credit Counter Records After Forwarding, Not Before

Lines 163-165 of `route.ts` iterate over `requests` and call `creditCounter.recordCall(req.method)` **after** the upstream response is returned. The credit counter is observational only — it does not gate, throttle, or reject based on accumulated credit use.

### Method Allowlist Does Not Reduce Blast Radius

The method allowlist at lines 31-59 (`ALLOWED_METHODS`, 17 methods) validates each element of the batch individually. A batch of 500 identical `getAccountInfo` calls passes all allowlist checks and reaches Helius intact.

## Attack Path

1. Attacker issues a single HTTP POST to `/api/rpc`:
   ```json
   [
     {"jsonrpc":"2.0","id":1,"method":"getAccountInfo","params":["<any-pubkey>",{"encoding":"base64"}]},
     ... (×500 copies)
   ]
   ```
2. `checkRateLimit` fires once. Rate limit bucket increments by 1 (of 300 allowed per minute).
3. All 500 items pass the `ALLOWED_METHODS` check in the validation loop.
4. `JSON.stringify(body)` serialises all 500 requests and forwards them to Helius as a single batch POST.
5. Helius processes 500 RPC calls, consuming ~500 credits.
6. Rate limiter allows 300 such HTTP requests per minute from the same IP.
7. Sustained throughput: 300 HTTP requests × 500 RPC methods = **150,000 Helius credits consumed per minute per IP**.

With IP spoofing (x-forwarded-for, confirmed spoofable per ASM-3 / M-ERR03-05 in the ERR-03 context), this is unlimited:
- Each HTTP request from a new spoofed IP gets a fresh 300-req/min budget.
- At 500 calls per batch: a single attacker cycling IPs can exhaust a 1M-credit Helius plan in under 7 minutes.

Helius Free plan caps: 500k credits/month. Paid plans range from 1M–100M credits/month. Credit exhaustion causes all RPC calls from the Helius API key to return HTTP 429, taking down:
- Frontend RPC proxy (all swaps, balance queries, TX submission)
- ws-subscriber server-side WebSocket connection (SSE data pipeline)
- Crank runner (epoch transitions, VRF, Carnage) — if it shares the same Helius key

## Impact

**Severity: HIGH**

| Dimension | Assessment |
|-----------|-----------|
| Availability | Full frontend + crank RPC denial until credits reset or plan upgraded |
| Financial | Forces emergency plan upgrade (Helius paid plans: ~$49–$499/mo) or service outage |
| Reversibility | Credits reset monthly; attacker can repeat each billing cycle |
| Requires auth | No — endpoint is fully public |
| Complexity | Trivial — a single `curl` with a JSON array body |
| IP spoofing needed | No for high impact; yes for unlimited scale |

The crank shares or may share the same Helius API key (`HELIUS_RPC_URL` env var). If credit exhaustion kills the Helius key, the crank cannot advance epochs or execute Carnage, directly impacting protocol liveness.

## Recommendation

**Primary fix — enforce a batch element count limit before forwarding:**

```typescript
const MAX_BATCH_SIZE = 10; // or 1 if batch is not a needed feature

const isBatch = Array.isArray(body);
const requests = isBatch
  ? (body as Array<Record<string, unknown>>)
  : [body as Record<string, unknown>];

if (requests.length > MAX_BATCH_SIZE) {
  return NextResponse.json(
    { jsonrpc: "2.0", error: { code: -32600, message: "Batch too large" }, id: null },
    { status: 400 },
  );
}
```

If the frontend never sends batch requests, disable batch support entirely: reject any `Array.isArray(body)` as a `400 Bad Request`.

**Secondary fix — rate limit on RPC method calls, not HTTP requests:**

Charge the rate limit bucket once per method call in the batch, not once per HTTP request:

```typescript
// Check rate limit once per method call in the batch
for (let i = 0; i < requests.length; i++) {
  const rateCheck = checkRateLimit(clientIp, RPC_RATE_LIMIT, "rpc");
  if (!rateCheck.allowed) {
    return new Response("Too Many Requests", { status: 429, ... });
  }
}
```

**Tertiary fix — add a request body size limit:**

Enforce a maximum payload size (e.g., 64KB) on the RPC proxy before parsing JSON. A 500-element batch of `getAccountInfo` requests is typically 30–50KB.

**Also address ASM-3 / IP spoofing:** Validate that `x-forwarded-for` contains only IPs added by trusted Railway proxies (e.g., only accept the last IP in the chain, or configure Railway to strip the header before appending its own).
