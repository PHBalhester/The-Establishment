# H014: Webhook Body Size Guard Bypassable via Chunked Transfer Encoding

## Verdict: CONFIRMED — MEDIUM (Requires Valid Auth Secret)

---

## Evidence

**Primary location:** `app/app/api/webhooks/helius/route.ts:309-315`

```typescript
const MAX_BODY_BYTES = 1_048_576; // 1MB
const contentLength = req.headers.get("content-length");
if (contentLength && parseInt(contentLength, 10) > MAX_BODY_BYTES) {
  return NextResponse.json(
    { error: "Payload too large" },
    { status: 413 },
  );
}
```

The guard is logically structured as `if (contentLength && ...)`. When the `content-length` header is absent — which is the case for any HTTP/1.1 request using chunked transfer encoding (`Transfer-Encoding: chunked`) — the condition evaluates to falsy (`null && ...`) and the entire size check is silently skipped. Execution falls through directly to `await req.json()` at line 321.

**Framework confirmation — no fallback limit exists:**
- The route exports `export const runtime = "nodejs"` (line 105), confirming it runs in Next.js's Node.js runtime, not Edge.
- `app/next.config.ts` contains no `experimental.proxyClientMaxBodySize`, no `bodyParser` config, and no other body size restriction. The `proxyClientMaxBodySize` feature (Next.js 16, default 10MB) only activates when the `proxy` middleware is used — it does not apply to App Router route handlers.
- The Pages Router `bodyParser.sizeLimit` config (`export const config = { api: { bodyParser: { sizeLimit } } }`) does not exist in App Router. No equivalent mechanism is configured.
- Railway explicitly has no platform-level request body size limit (confirmed via Railway support documentation: "There is no limit, but there is a maximum timeout").
- Conclusion: `req.json()` in this handler will buffer the entire incoming body stream regardless of size, with no framework or platform safety net.

**Auth order — production exploitability is gated:**

The execution order is:
1. Rate limit check (line 258)
2. Auth check — `timingSafeEqual` on `Authorization` header vs `HELIUS_WEBHOOK_SECRET` (lines 286-302)
3. Body size check (lines 309-315) ← bypass point
4. `await req.json()` (line 321)

In production, an unauthenticated attacker is rejected at step 2 with a 401 before the body is ever read. The vulnerability is therefore **post-auth**: exploitation requires knowledge of `HELIUS_WEBHOOK_SECRET`.

**The guard's logical flaw applies even with standard Content-Length:**
The check also fails silently if `parseInt(contentLength, 10)` returns `NaN` (e.g., malformed header value like `content-length: abc`). `NaN > MAX_BODY_BYTES` is `false`, so the check is bypassed with a non-numeric content-length header as well.

## Attack Path

**Precondition:** Attacker has obtained `HELIUS_WEBHOOK_SECRET` (e.g., via the committed `.env.devnet` key in git history per SEC-02 findings, via a Railway environment compromise, or through an insider threat).

1. Attacker sends POST to `/api/webhooks/helius` with:
   - `Authorization: <valid HELIUS_WEBHOOK_SECRET>`
   - `Transfer-Encoding: chunked` (omits `Content-Length`)
   - Body: a deeply-nested or very large JSON array (e.g., 50–100MB)
2. Rate limiter passes (120 req/min per IP; bypass via x-forwarded-for spoofing per H028/ASM-3 if needed)
3. Auth check passes (valid secret provided)
4. Body size check: `contentLength` is `null` → `if (null && ...)` is false → check skipped
5. `await req.json()` buffers the entire body into memory
6. Node.js heap grows by the size of the JSON body
7. With a single 100MB payload, a 512MB Railway container is under significant pressure. With concurrent requests (each at 120/min = 2 req/s per IP, multiple IPs via spoofing), OOM becomes achievable.

**Secondary path (non-production / dev environment):**
If `NODE_ENV !== "production"` and `HELIUS_WEBHOOK_SECRET` is unset, the auth check is completely skipped (lines 273-284, 286-302). In this case, any unauthenticated attacker on a dev/staging deployment can directly exploit the bypass without credentials.

---

## Impact

- **Availability — Production:** Memory exhaustion of the Railway container process. A sufficiently large or concurrent burst of oversized payloads can trigger OOM, causing a process crash and brief service outage. Recovery is automatic (Railway restarts) but could be sustained by an attacker with the secret.
- **Availability — Non-production:** Any unauthenticated attacker can exhaust memory on dev/staging deployments where auth is not enforced.
- **Blast radius:** The Next.js server process is single-process on Railway (INV-5 / ASM-5). An OOM crash affects all routes: the webhook pipeline, SSE broadcasts, the RPC proxy, the frontend, and the crank's HTTP interactions — all go down simultaneously until Railway restarts the container.
- **Severity modifier (down):** Exploitation requires the webhook secret in production, which is a significant precondition. This is not zero-click.

---

## Recommendation

Replace the header-based conditional check with a streaming read that enforces a hard byte limit before JSON parsing:

```typescript
// Read body as stream with hard byte cap — bypasses Content-Length bypass
const MAX_BODY_BYTES = 1_048_576; // 1MB

const reader = req.body?.getReader();
if (!reader) {
  return NextResponse.json({ error: "No body" }, { status: 400 });
}

let totalBytes = 0;
const chunks: Uint8Array[] = [];
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  totalBytes += value.byteLength;
  if (totalBytes > MAX_BODY_BYTES) {
    reader.cancel();
    return NextResponse.json({ error: "Payload too large" }, { status: 413 });
  }
  chunks.push(value);
}

const bodyText = new TextDecoder().decode(
  chunks.reduce((acc, c) => {
    const merged = new Uint8Array(acc.length + c.length);
    merged.set(acc); merged.set(c, acc.length);
    return merged;
  }, new Uint8Array(0))
);

let payload: unknown[];
try {
  payload = JSON.parse(bodyText);
} catch {
  return NextResponse.json({ error: "Malformed JSON payload" }, { status: 400 });
}
```

This approach enforces the byte limit at the stream level regardless of what headers are present, and rejects the request before the full body is ever materialized in memory.

Additionally, fix the NaN edge case by using `Number.isFinite(size) && size > MAX_BODY_BYTES` if reverting to a header-based check as a secondary layer.

## Evolution

**Audit #1 (H050):** Finding was raised — Content-Length bypass on body size check. A fix was applied: the size guard code block at lines 305-315 was added, labeled with `// ── H050: Reject oversized payloads ──`. The fix addressed the case where Content-Length is present and oversized.

**Audit #2 (this finding, H014):** The fix is confirmed incomplete. The `if (contentLength && ...)` conditional means the entire guard is skipped when `content-length` is absent, which is standard behavior for HTTP chunked transfer encoding. The root bug from H050 remains exploitable via a straightforward HTTP framing technique. The vulnerability is now classified MEDIUM rather than the originally hypothesized CRITICAL, because the auth gate in production means a valid secret is required — narrowing the attack surface to credential-theft scenarios and non-production environments.

**Status of INV-15** (`app/.bulwark/ARCHITECTURE.md`): Correctly marked **PARTIALLY enforced** — "Content-Length only, bypassable".
