## Code Review

**File:** `app/hooks/useSwap.ts`
**Lines:** 301–309

```typescript
const toBaseUnits = useCallback(
  (amount: string, token: TokenSymbol): number => {
    const parsed = parseFloat(amount);
    if (isNaN(parsed) || parsed <= 0) return 0;
    const decimals = getDecimals(token);
    return Math.floor(parsed * 10 ** decimals);
  },
  [getDecimals],
);
```

**Pattern:** `parseFloat(amount) * 10^decimals` in Number (IEEE 754 double precision) space.

## Precision Analysis

### Safe Integer Boundary
JavaScript's `Number` type has `MAX_SAFE_INTEGER = 2^53 - 1 = 9,007,199,254,740,991`.

For exact integer multiplication:
- **SOL (9 decimals):** Safe up to `2^53 / 10^9 ≈ 9,007,199 SOL`
- **Tokens (6 decimals):** Safe up to `2^53 / 10^6 ≈ 9,007,199,254 tokens`

### Protocol Scale Reality
- **Max user SOL input:** ~100 SOL (practical balance limit)
- **Max SOL pool reserve:** ~500 SOL (Liquidity Slippage Analysis)
- **Max token supply:** CRIME/FRAUD = 1B each, PROFIT = 20M
- **Max token pool reserve:** ~290M tokens (Phase 52 Canonical Mint Ordering)
- **Max user token input:** ≤20M tokens (token supply cap)

**Headroom calculation:**
- SOL: `9,007,199 / 100 ≈ 90,072x`
- Tokens: `9,007,199,254 / 20,000,000 ≈ 450x`

### Sub-Unit Rounding
For highly decimal inputs (e.g., `"0.123456789"` tokens), IEEE 754 rounding during `* 10^6` multiplication may lose sub-unit precision, rounding down by 1 base unit. This is bounded to:
- **Tokens:** ±1 millionth of 1 token (negligible)
- **SOL:** ±1 nanoSOL (negligible)

This does not pose a practical risk since:
1. Users input at most 6 decimal places (display precision).
2. The result is floored, making rounding errors one-directional (conservative).
3. The loss is sub-cent for any realistic price.
