# H007: Dependency Confusion via Unclaimed @dr-fraudsworth npm Scope

## Verdict: POTENTIAL

---

## Evidence

### The Vulnerable Reference

`app/package.json:18` declares:
```json
"@dr-fraudsworth/shared": "0.0.1"
```

This is a **bare version string**, not the npm workspace protocol (`workspace:*`).

### How Workspace Resolution Works (and Fails)

npm workspaces resolve `@dr-fraudsworth/shared` locally because:
1. The root `package.json` declares `"workspaces": ["shared", "app"]`
2. The lockfile records `"node_modules/@dr-fraudsworth/shared": { "resolved": "shared", "link": true }`
3. The installed symlink is `node_modules/@dr-fraudsworth/shared -> ../../shared`

Under **normal** operation — cloning from root, running `npm ci` or `npm install` from the workspace root — the package resolves to the local `shared/` directory and never touches the public npm registry.

### The Scope is Unclaimed

The DEP-01 auditor verified via `npm info @dr-fraudsworth/shared` that the scope returns **E404** — the `@dr-fraudsworth` scope is not registered on npm. An attacker can claim this scope and publish `@dr-fraudsworth/shared@0.0.1` at any time.

### Why `workspace:*` Matters

The bare version `"0.0.1"` causes npm to use workspace resolution **as a first preference**, but falls back to the public registry if:
- The install is run from a directory that is NOT the workspace root (e.g., `cd app && npm install`)
- The `package.json` workspaces field is absent or the `shared/` directory is missing (partial clone, submodule misconfiguration, repository restructure)
- A future CI job accidentally runs `npm install` inside `app/` rather than the root

With `workspace:*`, npm would **hard-fail** rather than fall back to the registry. The bare version silently succeeds through fallback.

### Confirmed Exploit-Adjacent Path: railway-crank.toml

`railway-crank.toml:3` uses:
```
buildCommand = "npm install"
```

This uses `npm install` (not `npm ci`) which allows lockfile modification. While the current lockfile pins `@dr-fraudsworth/shared` as `link: true`, if an attacker could corrupt the lockfile entry (e.g., via a compromised PR, a merge conflict, or any lockfile regeneration with a tainted npm environment), the bare version `"0.0.1"` would resolve to the public registry.

### Audit #1 Clearance (H066) — Re-evaluated

Audit #1 cleared H066 with: *"Workspace protocol (`workspace:*`) prevents npm registry resolution."*

This clearance was based on an incorrect premise — the project does NOT use `workspace:*`. It uses bare `"0.0.1"`. Audit #1's clearance reasoning describes the mitigation that would exist if `workspace:*` were used, but that mitigation is not actually in place.

The correct description of the current state is: workspace resolution is the default behavior for bare version strings **when the workspace root is the install context**, which is how the project is currently set up. The scope being unclaimed makes fallback exploitable if workspace context is ever lost.

## Attack Path

### Prerequisites
1. Attacker registers the `@dr-fraudsworth` npm org/scope on npmjs.com (free, immediate)
2. Attacker publishes `@dr-fraudsworth/shared@0.0.1` with a malicious payload

### Exploitation Scenario A — CI Misconfiguration (Realistic)

```
1. Attacker claims npm scope → publishes malicious @dr-fraudsworth/shared@0.0.1
2. Developer or CI runs `npm install` from `app/` subdirectory (e.g., `cd app && npm install`)
3. npm does not detect workspaces (no package.json workspaces field in app/)
4. npm resolves @dr-fraudsworth/shared@0.0.1 from the public registry
5. Malicious package installs → executes during postinstall (but ignore-scripts=true blocks this)
   → Alternative: malicious code in module body runs at require() time during build/test
6. Build output (JS bundle) includes attacker-controlled code
```

**Note:** `ignore-scripts=true` in `.npmrc` blocks `postinstall` hooks but does NOT prevent code in the package's module body from running when the package is `require()`d/`import`ed. The malicious package would execute the moment Next.js imports from `@dr-fraudsworth/shared` during `next build`.

### Exploitation Scenario B — Railway Crank (Critical Impact)

```
1. Attacker claims npm scope → publishes malicious @dr-fraudsworth/shared@0.0.1
2. Crank lockfile is regenerated (any npm install that touches lockfile)
3. railway-crank.toml uses `npm install` — if lockfile is corrupted/regenerated, resolves from registry
4. Malicious code runs in crank process at import time
5. Crank process has access to WALLET_KEYPAIR env var (funded wallet)
6. Attacker exfiltrates private key → drains crank wallet
```

### Exploitation Scenario C — New Developer Clone (Realistic)

```
1. New developer clones the repo
2. Runs `npm install` inside `app/` (not unusual — Next.js devs often work in the app dir)
3. npm resolves @dr-fraudsworth/shared from registry (attacker-controlled)
4. Developer runs `next dev` → malicious code executes in their local environment
5. Any secrets in local .env files or dev credentials may be exfiltrated
```

### Exploitation Scenario D — Repository Restructure (Future Risk)

If the monorepo is restructured and `shared/` is moved, renamed, or the root `package.json` workspaces config is changed, the `app/package.json` reference would silently resolve from the registry.

## Factors That Reduce Exploitability

1. **Current normal operation is safe**: `npm ci` from the workspace root always resolves locally via the lockfile's `link: true` entry. The main CI pipeline and local development follow this pattern.

2. **`ignore-scripts=true` blocks postinstall**: The `.npmrc` prevents install hooks. A malicious package would need to embed its payload in the module body (trivially easy, but requires the attacker to know this).

3. **Two conditions must align**: The scope must be claimed AND a non-root install context must occur. Currently neither condition is met. But the scope being unclaimed is a passive, always-available attack surface — an attacker can claim it now and wait.

## Recommendation

### Fix 1 — Use workspace protocol (PRIMARY FIX)

In `app/package.json:18`, change:
```json
"@dr-fraudsworth/shared": "0.0.1"
```
to:
```json
"@dr-fraudsworth/shared": "workspace:*"
```

This causes npm to hard-fail with a clear error if the workspace is not found, instead of silently falling back to the public registry. This is a one-line change.

**Note**: `workspace:*` requires npm v7+ (workspace support was added in npm v7). The project already uses npm workspaces, confirming npm v7+ is in use.

### Fix 2 — Register the npm scope (DEFENSE IN DEPTH)

Register `@dr-fraudsworth` on npm.com, even as a private/empty scope. This costs nothing and permanently closes the namespace-squatting window. An empty `@dr-fraudsworth/shared@0.0.1` with a `"description": "Internal package — do not use"` can be published as a placeholder.

### Fix 3 — Fix railway-crank.toml (RELATED — HIGH)

In `railway-crank.toml:3`, change:
```
buildCommand = "npm install"
```
to:
```
buildCommand = "npm ci"
```

This prevents lockfile modification during crank builds and is a prerequisite for full supply chain integrity.

## Evolution

**Audit #1 Finding:** H066 — Dependency Confusion @dr-fraudsworth/shared

**Audit #1 Verdict:** NOT_VULNERABLE

**Audit #1 Clearance Reasoning:** *"Workspace protocol (`workspace:*`) prevents npm registry resolution."*

**Audit #2 Re-evaluation:** The Audit #1 clearance was based on a factual error — it assumed `workspace:*` was in use, but the code uses bare `"0.0.1"`. The actual protection comes from npm's workspace-root resolution behavior (not the `workspace:*` protocol). This is weaker: it works when the install context is the workspace root, but silently falls back to the registry in other contexts.

The unclaimed npm scope confirmed by DEP-01 (E404) means the precondition for exploitation is one free npm registration away. The `railway-crank.toml` `npm install` usage (a separate confirmed finding) compounds this risk by allowing lockfile modification in the crank's production build.

This finding is **POTENTIAL** rather than CONFIRMED because: exploitation requires either (a) a workspace-context error (non-root install) or (b) lockfile corruption in the crank, neither of which is confirmed to occur in practice. However, the unclaimed scope is a persistent, low-cost attack surface that remains open until either `workspace:*` is adopted or the scope is registered.

**Recommended Audit #2 Tier:** HIGH — Pre-mainnet fix required. The one-line `workspace:*` change is trivial and eliminates the primary attack vector. The npm scope registration is strongly advised as defense-in-depth.
