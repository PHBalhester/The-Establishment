## New Attack Surface from Modified `shared/constants.ts`

The two most recent commits to `shared/constants.ts` are address regenerations (`dc063ec` and `55254dd`) from a clean-room devnet redeploy. The last diff shows only a null ALT address being patched to a real address (`FwAetEADes6Q19naJQ5eXBet9M5uVstAhjtvwnHRbMFL`).

The webhook handler at `route.ts:217-225` builds `KNOWN_PROTOCOL_ACCOUNTS` at module-load time by calling `.toBase58()` on constants imported from `shared/constants.ts`:

```ts
const KNOWN_PROTOCOL_ACCOUNTS: Record<string, string> = {
  [DEVNET_PDAS.EpochState.toBase58()]:      "EpochState",
  [DEVNET_PDAS.CarnageSolVault.toBase58()]: "CarnageSolVault",
  [DEVNET_POOLS.CRIME_SOL.pool.toBase58()]: "PoolState:CRIME_SOL",
  ...
};
```

This means **the webhook allowlist of known protocol accounts is derived directly from the auto-generated constants file**. The constants file carries the comment `// AUTO-GENERATED from deployments/devnet.json -- DO NOT EDIT MANUALLY`.

**Supply-chain intersection with constants:** If an attacker could corrupt `shared/constants.ts` (e.g., via a dependency confusion attack that reaches the build step, or via a compromised `generate-constants.ts` script, or a malicious PR), the `KNOWN_PROTOCOL_ACCOUNTS` map in the webhook handler would be silently populated with attacker-controlled addresses. Any enhanced webhook payload for those addresses would be decoded and injected into `protocolStore`, broadcasting corrupt price/epoch data to all SSE-connected frontend clients. This is the "data pipeline takeover" leg of the original S001 chain ÔÇö and it now flows through the auto-generated constants file as an intermediary.

**Assessment:** This is a **new indirect attack surface** introduced by the DBS refactor binding protocol addresses into the webhook dispatch table via imported constants. The attack path requires either a supply chain compromise (H003/H007 precondition) OR a malicious commit to the auto-generated file (protected by normal git review). On its own this does not worsen the severity ÔÇö supply chain compromise was already the prerequisite for the original S001 chain ÔÇö but it adds a non-obvious pivot point.

## Webhook Attack Chain ÔÇö Current Status

### Enhanced Webhook Path (No Replay Protection ÔÇö Unchanged)

The enhanced account change path (`handleAccountChanges`, `route.ts:525-620`) still has **no replay protection** for stale enhanced webhook payloads. The raw TX path has the H049 `blockTime` age gate. The enhanced path does not:

```ts
async function handleAccountChanges(accounts: HeliusAccountChange[]): Promise<NextResponse> {
  // No timestamp check, no nonce, no sequence number
  for (const item of accounts) {
    ...
    protocolStore.setAccountState(pubkey, { ...decoded, updatedAt: Date.now() });
  }
}
```

An attacker who can replay an authenticated enhanced webhook (i.e., who possesses `HELIUS_WEBHOOK_SECRET`) can inject arbitrarily stale account state into `protocolStore`, which SSE-broadcasts to all frontend clients immediately. This can cause the swap UI to display stale reserves, leading users to sign transactions with stale `minimumOutput` calculations (mitigated on-chain by the actual slippage check, but the UI experience is degraded and MEV-exploitable).

**Status:** This was a known gap (ARCHITECTURE.md `ASM-7`, Critical Intersection #3). Not a regression ÔÇö it remains unfixed.

## Combined Chain Assessment

The original S001 combined supply chain (H003) + webhook manipulation into a single attack chain:

```
1. Supply chain compromise Ôćĺ crank WALLET_KEYPAIR exfiltration
2. With HELIUS_WEBHOOK_SECRET also exposed from Railway env vars:
   Ôćĺ Inject false enhanced webhook payloads Ôćĺ corrupt SSE state Ôćĺ users see manipulated prices
3. With manipulated price state, craft and execute on-chain swaps before UI corrects
```

**Step 1 (supply chain):** PARTIALLY_FIXED ÔÇö `ignore-scripts=true` and committed lockfile close the most trivial vector, but `npm install` in `railway-crank.toml` keeps the door open for lockfile-drift or module-body injection attacks. H003 is POTENTIAL, not CONFIRMED.

**Step 2 (webhook injection):** UNCHANGED ÔÇö enhanced path still has no replay protection. Requires possession of `HELIUS_WEBHOOK_SECRET`.

**Step 3 (on-chain impact):** MITIGATED ÔÇö on-chain `minimumOutput` enforcement limits financial loss even with manipulated UI state.

**Overall chain viability:** Reduced from CONFIRMED CRITICAL to POTENTIAL HIGH. The primary reduction comes from `ignore-scripts=true` and committed lockfile (Step 1 harder), not from any webhook-side fix. The chain is still theoretically completable if H003's residual `npm install` vector is exploited AND Railway env vars are exfiltrated.
