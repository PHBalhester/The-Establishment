# H088: Protocol Store Unbounded Growth

**Tier**: 3
**Status**: NOT VULNERABLE
**Target**: `protocol-store.ts:37-38`
**CWE**: CWE-770 (Allocation of Resources Without Limits or Throttling)

## Summary

H088 suspected that the in-memory Map in `protocol-store.ts` could grow unboundedly if an attacker sends account change notifications for many different pubkeys, causing memory exhaustion.

**Verdict: NOT VULNERABLE** — The protocol store only accepts keys from a hardcoded whitelist (`KNOWN_PROTOCOL_ACCOUNTS`). Unknown accounts are rejected at the webhook handler level before reaching the store.

## Analysis

### 1. Protocol Store Structure

File: `app/lib/protocol-store.ts:37-38`

```typescript
private accounts = new Map<string, AccountState>();
private lastSerialized = new Map<string, string>();
```

The two Maps store:
- `accounts`: Latest state for each protocol account
- `lastSerialized`: Last broadcasted JSON (for dedup)

**No explicit size limit** on either Map in the store itself.

### 2. Validation at Webhook Handler

File: `app/app/api/webhooks/helius/route.ts:217-226`

```typescript
const KNOWN_PROTOCOL_ACCOUNTS: Record<string, string> = {
  [DEVNET_PDAS.EpochState.toBase58()]: "EpochState",
  [DEVNET_PDAS.CarnageFund.toBase58()]: "CarnageFundState",
  [DEVNET_PDAS.CarnageSolVault.toBase58()]: "CarnageSolVault",
  [DEVNET_POOLS.CRIME_SOL.pool.toBase58()]: "PoolState:CRIME_SOL",
  [DEVNET_POOLS.FRAUD_SOL.pool.toBase58()]: "PoolState:FRAUD_SOL",
  [DEVNET_PDAS_EXTENDED.StakePool.toBase58()]: "StakePool",
  [DEVNET_CURVE_PDAS.crime.curveState.toBase58()]: "CurveState:CRIME",
  [DEVNET_CURVE_PDAS.fraud.curveState.toBase58()]: "CurveState:FRAUD",

```

**8 known accounts** — a fixed, compile-time constant.

### 3. Rejection of Unknown Accounts

File: `app/app/api/webhooks/helius/route.ts:538-548`

```typescript
for (const item of accounts) {
  const pubkey = item.account;
  const label = KNOWN_PROTOCOL_ACCOUNTS[pubkey];

  if (!label) {
    console.warn(
      `[webhook] Received account change for unknown account: ${pubkey}. ` +
        "Check webhook accountAddresses configuration.",
    );
    continue;  // <-- SKIP UNKNOWN ACCOUNTS
  }
  // ... proceed only if label exists
}
```

**Unknown pubkeys are silently rejected** (logged as warning, skipped, never passed to `protocolStore.setAccountState()`).

### 4. WebSocket Subscriber Validation

File: `app/lib/ws-subscriber.ts:67-84`

```typescript
const BATCH_ACCOUNTS: Array<{
  pubkey: PublicKey;
  accountType: string | null;
  program: "amm" | "bondingCurve" | "epoch" | "staking" | "system";
}> = [
  { pubkey: DEVNET_PDAS.EpochState, accountType: "epochState", program: "epoch" },
  { pubkey: DEVNET_PDAS.CarnageFund, accountType: "carnageFundState", program: "epoch" },
  // ... 6 more known accounts
];
```

The WS subscriber's batch-seed only fetches **8 known accounts** at startup and polls **synthetic keys** (`__supply:CRIME`, `__supply:FRAUD`, `__slot`, `__staking:globalStats`), not arbitrary user inputs.

## Bounds

| Boundary | Value | Notes |
|----------|-------|-------|
| Real protocol accounts | 8 | Fixed at compile-time in `KNOWN_PROTOCOL_ACCOUNTS` |
| Synthetic keys (ws-subscriber) | 4 | `__supply:CRIME`, `__supply:FRAUD`, `__slot`, `__staking:globalStats` |
| **Total hard cap** | **12** | Guaranteed max entries across both Maps |

## Why Not Vulnerable

1. **Webhook handler whitelist**: Only 8 known pubkeys can be stored via Helius webhooks. Attacker cannot add new pubkeys.
2. **WS subscriber isolation**: Synthetic keys are hardcoded in the server-side code, not derived from user input.
3. **No client control**: Frontend clients cannot inject new keys via SSE (they only read broadcasts).
4. **No accumulation path**: There is no endpoint that accepts arbitrary pubkey strings and stores them.

## Minor Observations

- `setAccountState()` has no input validation (trusts caller), but that's not an issue because the only callers are the webhook handler and WS subscriber, both of which validate upstream.
- Each Map entry is small (~100-1000 bytes per account), so even 12 entries at max is negligible (~12 KB total).

## Conclusion

**NOT VULNERABLE.** The protocol store's unbounded Maps are protected by a hardcoded whitelist (`KNOWN_PROTOCOL_ACCOUNTS`) that allows only 8 accounts, plus 4 synthetic keys. An attacker cannot forge account changes for arbitrary pubkeys. Memory growth is bounded at ~12 KB worst-case.
