# H071: Missing CORS Headers on SSE Responses

**Severity:** Tier 3 — Medium
**Category:** Web Security (WEB-02)
**Target:** `app/app/api/sse/protocol/route.ts:127-135`
**Status:** CONFIRMED / POTENTIAL

---

## Vulnerability Summary

The SSE response handler for `/api/sse/protocol` lacks explicit `Access-Control-Allow-Origin` headers. While EventSource enforces same-origin policy by default, the absence of explicit CORS denial headers creates ambiguity about intended cross-origin behavior and could allow unintended access if deployment environment adds permissive headers globally.

---

## Technical Details

### Current Response Headers (lines 127-135)

```typescript
return new Response(stream, {
  headers: {
    "Content-Type": "text/event-stream",
    "Cache-Control": "no-cache, no-transform",
    Connection: "keep-alive",
    "X-Accel-Buffering": "no",
  },
});
```

**Missing:** No `Access-Control-Allow-Origin` or `Access-Control-*` headers.

### EventSource CORS Behavior

1. **EventSource supports CORS** (standard in Firefox 11+, Chrome, Safari, all modern browsers since ~2014)
2. **Cross-origin EventSource requires** `Access-Control-Allow-Origin` header in response
3. **Same-origin EventSource** works regardless of CORS headers (browser enforces origin check internally)
4. **Missing headers = implicit deny** for cross-origin, but not explicitly stated

### Risk Vector

If a reverse proxy (nginx, Cloudflare, Railway) is misconfigured to add `Access-Control-Allow-Origin: *` globally, the SSE endpoint would become accessible cross-origin despite no explicit opt-in in the route handler. This violates the principle of explicit intent for sensitive data flows.

**Architecture context:** The SSE endpoint broadcasts real-time protocol state (account balances, pool reserves, tax rates) to authenticated browser clients via same-origin EventSource connections. Cross-origin access could:
- Expose protocol state to hostile origins (timing attacks, pre-trade sniffing)
- Bypass expected same-origin isolation in the threat model

---

## Evidence

1. **No CORS headers in route.ts (lines 127-135):** Confirmed by code inspection
2. **next.config.ts does not add CORS headers:** No global middleware adding `Access-Control-*` (confirmed by inspection of async headers() section)
3. **EventSource requires Access-Control-Allow-Origin for cross-origin:** Confirmed by MDN, StackOverflow consensus, multiple SSE framework examples
4. **Sibling SSE route (candles) also lacks headers:** `/api/sse/candles/route.ts` identical pattern (lines 113-123)

---

## Verdict

**CONFIRMED** — SSE routes lack explicit CORS headers. While same-origin restriction is implicitly enforced by EventSource, explicit headers should be added to:
1. Clarify intent (same-origin only)
2. Defend against misconfigured reverse proxies
3. Align with explicit security-by-declaration principle

**Severity reduced to Tier 3 because:**
- EventSource same-origin default is robust
- No auth bypass possible (no credentials exposed)
- Data exfiltration requires hosting on same origin or explicit CORS opt-in upstream
- Off-chain data replication (no financial impact)

---

## Remediation

### Recommended Fix

Add explicit CORS denial headers to both SSE routes:

```typescript
return new Response(stream, {
  headers: {
    "Content-Type": "text/event-stream",
    "Cache-Control": "no-cache, no-transform",
    Connection: "keep-alive",
    "X-Accel-Buffering": "no",
    // H071: Explicitly deny cross-origin EventSource access.
    // EventSource enforces same-origin by default, but explicit headers
    // prevent downstream misconfiguration (e.g., overly permissive nginx).
    "Access-Control-Allow-Origin": "null",
    "Access-Control-Allow-Credentials": "false",
  },
});
```

**Why `null` + `false`:**
- `null` explicitly signals "no cross-origin allowed"
- `false` on credentials prevents any auth context leakage
- Same-origin requests from actual origin will NOT receive `Access-Control-*` headers but WILL succeed (EventSource doesn't check headers for same-origin)
