# H053: Split Route Amplifies Sandwich Surface

## Verdict: POTENTIAL

Confidence: 6/10

---

## Summary

Split routes (SOL→PROFIT via both CRIME and FRAUD pools, or PROFIT→SOL in reverse) pack four swap instructions into a single atomic VersionedTransaction (v0). The hypothesis that this "doubles" the sandwich surface is **structurally incorrect**: atomicity prevents inter-instruction sandwiching. However, a weaker form of the risk survives — the split TX presents a **wider visible slippage window in aggregate** than a single-hop route, and the two AMM legs are observable in the mempool simultaneously. Whether this translates to any net additional extraction depends on bot capability and epoch tax rates. At current tax rates (H016: ≥6% round-trip cost vs ≤5% extraction) sandwich attacks on any individual leg remain unprofitable. The finding is POTENTIAL because: (1) atomicity neutralises the "double sandwich" scenario, and (2) the tax economics block profitable sandwiching under all epoch configurations.

## Evidence

### 1. Split routes produce four swap instructions in one v0 TX

`app/hooks/useRoutes.ts:153-155` — A SOL→PROFIT split builds four steps:

```
Leg A: SOL → CRIME/SOL pool → CRIME → CRIME/Vault → PROFIT
Leg B: SOL → FRAUD/SOL pool → FRAUD → FRAUD/Vault → PROFIT
```

`app/lib/swap/multi-hop-builder.ts:1-8` confirms all steps are packed into **a single VersionedTransaction**:

> "Combines all swap steps (1-hop, 2-hop, or 4-step split) into a SINGLE VersionedTransaction (v0)."

`processInstructionsForAtomic()` at line 177 strips per-step compute budgets and merges all instructions into one flat list. The v0 TX carries both AMM swap instructions — for CRIME/SOL and FRAUD/SOL — alongside the two deterministic vault conversion instructions.

### 2. Atomicity prevents mid-route sandwiching

Because all four instructions execute atomically within a single block slot, no external transaction can be inserted between Leg A and Leg B. The "doubling" scenario from the hypothesis — a bot sandwiching each leg independently — **requires sequential transactions**, which atomicity prevents. The code comment at line 2-3 of `multi-hop-builder.ts` makes this explicit:

> "Atomic execution — all hops succeed or all revert (no partial state)."

This is enforced at the Solana runtime level: all instructions in a transaction execute in one atomic unit within a block. No other transaction touches those pools between Leg A's AMM swap and Leg B's AMM swap.

### 3. The aggregate slippage window is wider than a single-hop TX

`app/hooks/useRoutes.ts:170-172` — `minimumOutput` for the split route is computed from `splitOutput` (total PROFIT out), using the same `slippageBps` that a single-hop swap would use:

```typescript
const minimumOutput = Math.floor(
  splitOutput * (10_000 - slippageBps) / 10_000,
);
```

With `slippageBps=500` (H016 default), the bot's observable extraction window is 5% of the **total output**, which spans two pools. For a large enough swap where a split is recommended (above `SPLIT_THRESHOLD_BPS=50`), the aggregate nominal SOL at risk is the full swap amount — the same as a single-hop route of the same size. There is no net amplification of the slippage window.

### 4. Both AMM legs are visible in the mempool simultaneously

`executeAtomicRoute()` at line 381 uses `skipPreflight: true` and submits via standard Helius RPC:

```typescript
signature = await wallet.sendTransaction(
  build.transaction,
  connection,
  { skipPreflight: true, maxRetries: 3 },
);
```

A bot monitoring the Dr. Fraudsworth program ID in the Helius mempool (or via the unauthenticated SSE endpoint, per ARCHITECTURE.md Novel Attack Surface #4) sees the full v0 TX before it lands. The bot can decode both the CRIME/SOL and FRAUD/SOL swap instructions — pool addresses, swap direction, and the single `minimumOutput` constraint that covers both legs — in one observation. This is marginally more information than a single-hop TX discloses, but does not enable a different class of attack.

### 5. Tax economics block profitable sandwiching on both legs

H016 establishes the on-chain tax defense: every SOL pool swap incurs a 3–14% tax, making sandwich attacks unprofitable. For a split-route sandwich attempt:

- Leg A front-run: bot pays CRIME buy tax (3–14%)
- Leg A back-run: bot pays CRIME sell tax (3–14%)
- Leg B front-run: bot pays FRAUD buy tax (3–14%)
- Leg B back-run: bot pays FRAUD sell tax (3–14%)

A bot attempting to sandwich both pools in the same block would pay **four tax deductions** (two per pool) against a total profit bounded by the user's 5% slippage across both legs. The economics are even more unfavourable than a single-leg sandwich. This defense is on-chain and unavoidable — `programs/tax-program` wraps every AMM swap atomically.

### 6. Vault conversion steps (CRIME→PROFIT, FRAUD→PROFIT) provide no sandwich surface

`split-router.ts` and `useRoutes.ts:211-217` — The CRIME/Vault and FRAUD/Vault steps use a fixed 100:1 conversion rate with no AMM reserves. There is no pool state a bot can manipulate. These two of the four instructions carry zero MEV surface.

## Attack Path Analysis

### Scenario A: Classic double-sandwich (BLOCKED by atomicity)

```
Hypothesis: bot sandwiches Leg A, then independently sandwiches Leg B.
Reality: both legs execute in the same TX. No bot instruction can be
inserted between them at the protocol level. Attack is structurally impossible.
```

### Scenario B: Aggregate-position sandwich (BLOCKED by tax economics)

```
1. Bot observes split v0 TX in mempool: CRIME/SOL swap + FRAUD/SOL swap.
2. Bot front-runs both pools (two buy TXs at higher priority fees).
3. User's TX lands: both AMM legs complete at degraded price, total output hits 5% slippage floor.
4. Bot back-runs both pools (two sell TXs).

Tax cost per pool: 2 × tax_rate × bot_position.
At minimum epoch tax (3%): 4 × 3% = 12% of bot position size.
Maximum sandwich profit: 5% of user's split input (split to two pools).
Attack is deeply unprofitable under all epoch configurations.
```

### Scenario C: Single-pool targeted sandwich (PARTIALLY BLOCKED)

```
1. Bot ignores Leg B, sandwiches only the CRIME/SOL leg.
2. User's TX degrades only Leg A; Leg B completes at quoted price.
3. minimumOutput is set on aggregate output — on-chain enforcement
   cannot distinguish which leg produced less.

If Leg A output is reduced enough that aggregate output falls below
minimumOutput, the entire TX reverts (atomic). Bot front-run was wasted.
If Leg A output degrades within the 5% slippage window but stays above
the per-leg minimumOutput threshold, the TX succeeds with degraded total output.

Tax cost: 2 × tax_rate × bot_position (one pool only).
Same economics as single-hop H016 Scenario A — still unprofitable.
```

### Scenario D: Stale-reserve quote degradation (LIVE — not MEV, but user harm)

```
useRoutes.ts:47 — auto-refresh is every 30 seconds (REFRESH_INTERVAL_SECONDS).
split-router.ts:76-146 — computeOptimalSplit uses quoted reserves at computation time.
By the time the user signs and the v0 TX lands (5–15s), reserves in both pools
may have moved.

A split route has two pools drifting independently. In the worst case, both pools
move adversely: Leg A degrades 3%, Leg B degrades 3%. Combined degradation of ~6%
against a 5% slippage tolerance would revert the TX. But if each pool drifts
< 5% individually, the aggregate output may still pass the single minimumOutput check
while the user has absorbed a compounded adverse move across two pools.

This is ordinary market risk, not MEV. Same class of issue as H016 Scenario B,
marginally amplified by the two-pool exposure.
```

---

## Impact

**Scenario A (double-sandwich):** Not exploitable. Atomicity is a hard block.

**Scenarios B and C (profitable sandwich):** Not exploitable under current epoch configurations. On-chain tax makes sandwiching any SOL pool swap unprofitable regardless of route complexity.

**Scenario D (stale-reserve compounded drift):** Low-additional. Split routes expose users to reserve drift in two pools instead of one, but the per-pool drift is independent and bounded by the same on-chain `minimumOutput` check applied to the aggregate. The incremental risk over H016 Scenario B is the potential for two independently adverse reserve movements to compound. With a 5% aggregate slippage tolerance and two pools each potentially drifting, a scenario where both legs individually degrade within tolerance but the combined degradation causes a silent underpayment becomes slightly more likely.

**Quantified additional risk vs single-hop:**
- Additional sandwich extraction: ~$0 (tax blocks it at the pool level)
- Additional market-drift exposure: marginal; two pools drifting vs one, same slippage budget applied to aggregate output

## Recommendation

### No code change required for MEV protection (tax economics are the defense)

The on-chain tax structure (H016) is the primary defense. Reducing the default slippage (H016 Fix 1: 500→100 BPS) addresses Scenario D for all route types, including split routes, and is the correct lever.

### Strengthen per-leg minimumOutput enforcement (optional, low priority)

`buildAtomicRoute()` at line 305-309 derives a single `slippageBps` and applies it per-step:

```typescript
const minimumOutput = Math.floor(
  step.outputAmount * (10_000 - slippageBps) / 10_000,
);
```

Each step already has its own `minimumOutput` passed to the on-chain instruction. This means on-chain enforcement already validates each leg independently, not just the aggregate. Scenario C (single-leg sandwich causing the aggregate to fail) is therefore handled — if either leg degrades beyond its per-step `minimumOutput`, the on-chain program rejects that step and the whole TX reverts atomically. No change needed.

### Document the atomic execution guarantee in user-facing UI (optional)

The split route label (`SOL -> CRIME/FRAUD -> PROFIT (Split X/Y)`) could include a tooltip explaining that both legs execute atomically. This is a UX transparency improvement, not a security fix.

## Evolution

**Status: NEW — POTENTIAL (not CONFIRMED)**

The "doubling" framing in the hypothesis does not hold: atomicity prevents inter-leg sandwiching, and the tax economics block profitable single-leg sandwiching. The finding is not dismissed entirely because: (1) Scenario D (compounded reserve drift) is marginally worse for split routes than single-hop routes, and (2) future epoch configurations with very low tax rates could change the economics — though the tax floor (3%) still makes classic sandwiching unprofitable at any realistic slippage setting. The correct remediation for the remaining risk is the default slippage reduction from H016, which already has a concrete fix recommendation.
