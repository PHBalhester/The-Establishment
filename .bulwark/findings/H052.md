## Hypothesis

The route engine computes `priceImpactBps` for every route but never uses it to block execution. A user can submit a swap with 50%+ price impact and the frontend will proceed without any blocking guard. Only a UI color change is shown.

## Verification

### 1. Price impact is computed but never checked against a threshold

`route-engine.ts:351-354` accumulates `totalPriceImpactBps` by summing per-step values:

```ts
const totalPriceImpactBps = steps.reduce(
  (sum, s) => sum + s.priceImpactBps,
  0,
);
```

The value is returned on the `Route` object and propagated to the quote state in `useSwap.ts:827`, but `computeRoutes()` contains no guard that filters or rejects routes by price impact level. High-impact routes are returned and ranked the same as low-impact routes — by `outputAmount` descending only (`route-engine.ts:442`).

### 2. UI shows color warning only — no execution block

`RouteCard.tsx:111-116` changes the impact display to `text-factory-error` when impact exceeds 500 BPS (5%), and `text-factory-warning` above 100 BPS (1%). `FeeBreakdown.tsx:138-146` applies the same color thresholds. Neither component sets `disabled` on the swap submit button nor prevents `executeSwap()` / `executeRoute()` from being called. The button's only `disabled` guard is `isTransacting` (`SwapForm.tsx:301`).

### 3. Direct swap path in useSwap has the same gap

`useSwap.ts:415-431` (buy path) and `useSwap.ts:443-459` (sell path) capture `priceImpactBps` from the quote engine result and place it in quote state, but there is no conditional that sets quote to null or prevents form submission when `priceImpactBps` exceeds any threshold.

### 4. On-chain backstop: 50% floor — what it does and does not cover

`programs/tax-program/src/constants.rs:40`:
```rust
pub const MINIMUM_OUTPUT_FLOOR_BPS: u64 = 5000;
```

`swap_sol_buy.rs:106-111` and `swap_sol_sell.rs:113-118` enforce:
```rust
let output_floor = calculate_output_floor(sol_reserve, token_reserve, sol_to_swap, MINIMUM_OUTPUT_FLOOR_BPS)...;
require!(minimum_output >= output_floor, TaxError::MinimumOutputFloorViolation);
```

This checks that the **user-supplied `minimum_output` parameter** is at least 50% of the constant-product expected output. It does **not** reject swaps with high price impact directly. The distinction matters:

- A user with 60% price impact on a large trade sets `minimumOutput` using their chosen `slippageBps` (default 100 BPS = 1%). The frontend computes `minimumOutput = expectedOutput * (10000 - 100) / 10000 = 99% of expectedOutput`. Since `99% of expectedOutput > 50% of expectedOutput`, the on-chain check passes. The swap executes and the user absorbs the full 60% price impact.
- The 50% floor only blocks transactions where `minimumOutput` is set below 50% of expected output (i.e., zero-slippage bots or sandwich exploits setting `min_out=0`). It does not protect a normal user from executing a trade with catastrophic price impact.

## Impact

A user (especially a retail user unfamiliar with AMM mechanics) can unknowingly submit a large swap that moves the pool price by 50%+ or even 90%+. The frontend shows a red "Impact: X%" label but does not block submission. The on-chain 50% floor is not triggered because the user's `minimumOutput` (derived from their small `slippageBps`, e.g. 1%) still clears the floor calculation. The user receives correct on-chain execution but at a highly unfavorable price they were not meaningfully warned about beyond color.

This is primarily a user-protection / UX-safety gap rather than a direct fund-theft vector against the protocol, but it creates material financial harm for individual users and reputational risk for the protocol.

## Verdict

**CONFIRMED**

The hypothesis is accurate in full:
- `route-engine.ts` computes price impact but applies zero rejection threshold anywhere in the routing or execution path.
- UI shows color coding only (red at >500 BPS, yellow at >100 BPS) with no submit block.
- The on-chain 50% floor (`MINIMUM_OUTPUT_FLOOR_BPS=5000`) does not protect against high price impact — it only blocks `minimum_output` values set below 50% of expected output, which normal UI-derived `minimumOutput` values always exceed.

## Recommended Fix

Add a client-side rejection threshold (configurable, suggested 1500–2000 BPS) in `computeRoutes()` or at the execute callsite. When `totalPriceImpactBps` exceeds the threshold, either filter the route from results or disable the swap submit button and display an explicit blocking message (not just a color change). A secondary "I understand the risk" acknowledgement modal for very high impact (e.g., >1000 BPS) would be standard DeFi UX practice.
