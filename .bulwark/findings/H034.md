## Verdict: PARTIALLY_FIXED

The primary click path (BigRedButton) is now correctly protected. BuyForm and SellForm are correctly protected. However, `executeSwap` and `executeRoute` themselves contain no in-function status guard â€” the entire double-submit prevention for the AMM swap path relies on the UI layer, creating an unguarded programmatic execution surface. Additionally, a narrow race window exists in the MultiHopStatus retry path.

---

## Evidence

### 1. BigRedButton â€” Primary path is correctly guarded

`app/components/station/BigRedButton.tsx:67-73`

```ts
const handleClick = () => {
  if (status === 'confirmed' || status === 'failed') {
    onReset();
  } else if (status === 'idle' && !disabled && connected) {
    onSwap();
  }
};
```

`onSwap()` fires only when `status === 'idle'`. The HTML `disabled` attribute is also set when `isTransacting` (line 105: `disabled={(disabled && status === 'idle') || isTransacting}`). For a user clicking the button, double-submit is prevented.

### 2. executeSwap â€” No in-function guard

`app/hooks/useSwap.ts:690-698`

```ts
const executeSwap = useCallback(async () => {
  if (!wallet.publicKey || !wallet.connected || !quote) {
    return;
  }

  const connection = getConnection();
  const poolConfig = resolvePool(inputToken, outputToken);

  try {
    setStatus("building");
```

There is no `if (status !== 'idle') return` guard. Any caller that invokes `executeSwap` while the hook's `status` is `building`, `signing`, or `confirming` will proceed past the guard and attempt to build and sign a second transaction. The function immediately sets `setStatus("building")` without first checking the current status.

### 3. executeRoute â€” No in-function guard

`app/hooks/useSwap.ts:839-860`

```ts
const executeRoute = useCallback(async () => {
  if (!smartRouting || !routesResult.selectedRoute) {
    return executeSwap();
  }

  const route = routesResult.selectedRoute;

  if (route.hops === 1 && !route.isSplit) {
    return executeSwap();
  }

  if (!wallet.publicKey || !wallet.connected) return;

  const connection = getConnection();

  try {
    setStatus("building");
```

Same absence: no status check before proceeding. Both the single-hop fallback (delegates to `executeSwap`) and the multi-hop branch (`setStatus("building")` at line ~858) are unguarded.

### 4. MultiHopStatus retry â€” Race window

`app/components/station/SwapStation.tsx:281-284`

```ts
onRetry={() => {
  swap.resetForm();
  setTimeout(() => swap.executeRoute(), 0);
}}
```

`resetForm()` sets status to `"idle"` synchronously. `executeRoute()` is then called via `setTimeout(..., 0)`, deferring to the next event loop tick. In that tick gap, the BigRedButton becomes enabled again (`status === 'idle'`). If the user clicks the button before the `setTimeout` fires (extremely tight window, ~0ms), two executions could be dispatched. This is theoretical but the pattern is unsound â€” `executeRoute()` should be called directly or the button should be kept disabled for a guard interval.

---

### 5. BuyForm and SellForm â€” Correctly fixed

Both components protect their submit handlers with a local `submitting` boolean:

`app/components/launch/BuyForm.tsx:239-245`
```ts
const canSubmit =
  connected &&
  solInput.trim() !== '' &&
  !validationError &&
  tokensOut !== null &&
  tokensOut > 0n &&
  !submitting;
```

`handleSubmit` also early-returns at line 159 if `!tokensOut || tokensOut === 0n`. The button's `disabled={!canSubmit}` enforces this in the DOM. The `finally { setSubmitting(false) }` block at lines 218-220 ensures cleanup. SellForm mirrors this pattern identically. These are **RESOLVED**.

## Root Cause

The original H034 fix hardened the UI layer (BigRedButton click handler, `disabled` attribute, form-level `submitting` booleans) but did not add idempotent status guards inside `executeSwap` and `executeRoute` themselves. The function bodies are not self-protecting â€” they rely entirely on callers to gate them.

This matters because:
- `executeRoute` is passed as a prop (`executeRoute={swap.executeRoute}`) and exposed in the public return object of `useSwap`. Any consumer that holds a reference to these callbacks can invoke them unconditionally.
- The `onRetry` pattern in MultiHopStatus calls `executeRoute` programmatically, not via the button click handler, bypassing the `handleClick` guard.
- React concurrent features and fast event replays can produce stale-closure captures that fire callbacks with out-of-date `status` values.

---

## Fix Required

Add an early-return guard at the top of both functions:

**`app/hooks/useSwap.ts` â€” `executeSwap`** (insert after existing wallet/quote guard):
```ts
if (status !== "idle") return;
```

**`app/hooks/useSwap.ts` â€” `executeRoute`** (insert before the `smartRouting` check, or after the wallet check for the multi-hop branch):
```ts
if (status !== "idle") return;
```

**`app/components/station/SwapStation.tsx` â€” `onRetry`** (replace setTimeout pattern):
```ts
onRetry={() => {
  swap.resetForm();
  swap.executeRoute();   // call directly; resetForm is synchronous, status is 'idle' before this line
}}
```

Note: `status` must be included in the `useCallback` dependency array for `executeSwap` and `executeRoute` after this change.
