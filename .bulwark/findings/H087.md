# H087: Cross-Epoch Tax Rate Sniping via SSE

**Tier:** 3 (Low)
**Category:** Edge case / Race condition
**Status:** NOT VULNERABLE
**Verdict:** Mitigated by on-chain validation and transaction finality

## Summary

The vulnerability proposes that a sophisticated client could:
1. Monitor EpochState changes via SSE (server-sent events)
2. Detect when new tax rates are broadcast
3. Construct a swap instruction using the **new (favorable) tax rates** before UI renders
4. Submit the swap to gain advantage over honest users

This is a **timing + information asymmetry attack** targeting the SSE pipeline that streams EpochState updates to connected browsers.

## Investigation

### 1. SSE Pipeline Architecture

**Data flow:**
```
Epoch Program (consume_randomness)
  → EpochState account written onchain
  → Helius Enhanced Webhook fires
  → POST /api/webhooks/helius
  → protocolStore.setAccountState("epoch_state_pubkey", {...})
  → sseManager.broadcast("protocol-update", {account: pubkey, data: {...}})
  → All connected browsers receive "protocol-update" event via EventSource
  → useProtocolState hook updates local state
  → useEpochState extracts tax fields
  → Route calculation recalculates quotes with new tax rates
```

**Key files:**
- `app/lib/protocol-store.ts`: Stores account changes in-memory
- `app/lib/sse-manager.ts`: Broadcasts via SSE (no ordering guarantees)
- `app/hooks/useProtocolState.ts`: Browser-side subscription to SSE
- `app/hooks/useEpochState.ts`: Extracts tax rates from EpochState

### 2. Timeline Analysis

When a new epoch's `consume_randomness` transaction lands on-chain:

**On-chain (T=0, in same block):**
- Epoch Program writes new EpochState with new tax rates
- Transaction confirms in block
- Block propagates to Helius / RPC nodes

**Helius Webhook (T=~100-500ms):**
- Helius detects account change from webhook listener
- Parses EpochState and decodes via Anchor
- POSTs to deployment's `/api/webhooks/helius`
- SSE broadcasts to all connected clients

**Browser (T=~500-2000ms after block):**
- EventSource receives "protocol-update" event
- JavaScript parses event data
- `useEpochState` hook extracts new tax rates
- SwapForm component re-renders with fresh quotes

### 3. Attack Scenario (Theoretical)

An attacker with special access could:

1. **Run a full Solana validator/RPC node** to receive blocks 100-500ms before public Helius
2. **Parse consume_randomness on block arrival** (no waiting for webhook)
3. **Construct SwapSolBuy IX with new favorable tax rates** (e.g., 1% instead of 8%)
4. **Bundle with reveal IX in same block** or submit immediately after
5. **Settle swap on-chain with old tax rates if block order favors attacker**, or new rates if new block already includes epoch transition

**Problem: The on-chain tax rate is *deterministic* and *enforced by the program*.**

### 4. On-Chain Enforcement (Critical)

When `swap_sol_buy` or `swap_sol_sell` executes in Tax Program:

**File:** `programs/tax-program/src/instructions/swap_sol_buy.rs` (and swap_sol_sell)

```rust
// The instruction reads EpochState at execution time
pub fn handler(ctx: Context<SwapSolBuy>, ...) -> Result<()> {
    let epoch_state = &ctx.accounts.epoch_state;

    // Calculate tax using CURRENT epoch's rates (from account)
    let tax_bps = if is_crime {
        epoch_state.crime_buy_tax_bps
    } else {
        epoch_state.fraud_buy_tax_bps
    };

    let tax_amount = calculate_tax(amount, tax_bps);
    // ... apply tax to user's output
}
```

**The critical fact:**
- The instruction reads `epoch_state` **at execution time** from the account
- It derives the tax rate from the **current on-chain value**
- The client passes `tax_bps` as an *informational* parameter (for UI display)
- But the **actual tax is re-derived from the account** in the program

This means:
- If a user's TX lands **after** a `consume_randomness` TX, the new tax rates apply
- If a user's TX lands **before** a `consume_randomness` TX, the old tax rates apply
- **No amount of SSE sniping changes what the program enforces**

### 5. Instruction Details (Verification)

Checking swap instruction structure in `app/lib/swap/swap-builders.ts`:

```typescript
const ix = program.instruction.swapSolBuy(
  // Client can pass minimumOutput based on anticipated tax
  new BN(minimumOutput),
  isCrime,
)
.accountsStrict({
  user: userPublicKey,
  epochState: DEVNET_PDAS_EXTENDED.EpochState,  // <-- Program reads this
  swapAuthority,
  taxAuthority,
  pool,
  // ... other accounts
});
```

The client **cannot** override the epoch_state account. The program **always** reads from the canonical EpochState PDA and derives tax in-band.

### 6. Slippage as Defense

Even if a sophisticated attacker somehow forced a lower tax rate into their swap:

- **Route engine validates `minimumOutput`** in route-engine.ts
- User sets slippage tolerance (e.g., 1%)
- **If actual output < minimumOutput, transaction fails**
- Program returns error before any state change

So the attack would need to:
1. SSE-snipe new tax rates
2. Calculate quote with new rates
3. Set minimumOutput
4. **Get lucky** that no other transactions reorder in mempool
5. Land before the program's tax derivation shifts

And even then, **the program's on-chain tax is authoritative**. If rates changed between quote and execution, the slippage check fails.

### 7. Concrete Example

**Honest user flow:**
1. User in browser sees current tax rate: 8% (old epoch)
2. Epoch transition happens on-chain
3. Helius webhooks new tax: 2%
4. SSE client receives update
5. Route engine recalculates: 0.5 SOL input with 2% tax = ~0.49 SOL output (vs ~0.46 before)
6. User sees improved quote and clicks "Swap"
7. Submits SwapSolBuy with minimumOutput=0.485 SOL
8. **Program reads EpochState, sees 2% tax**
9. Applies 2% tax, outputs ~0.49 SOL
10. Output >= 0.485, transaction succeeds ✓

**Attacker scenario (fails):**
1. Attacker runs own validator, sees consume_randomness lands
2. Attacker sees new tax: 2% (before Helius webhook)
3. Attacker constructs quote with 2% tax: expects 0.49 SOL output
4. Sets minimumOutput=0.485 SOL
5. Submits SwapSolBuy **but epoch hasn't transitioned yet** (was in different block)
6. **Program reads EpochState, still sees 8% tax**
7. Applies 8% tax, outputs ~0.46 SOL
8. Output < 0.485 SOL minimum → **Transaction fails** ✗

Even if attacker gets block ordering right, they can't exploit beyond normal MEV (sandwich attacks), and **slippage protection handles that**.

## Verdict: NOT VULNERABLE

### Why This Is Not Exploitable

1. **On-chain tax rate is authoritative**
   - Program reads EpochState account at instruction time
   - Client cannot override or pre-specify the tax rate
   - New rates are always applied on the block they're confirmed

2. **Transaction finality creates clean boundaries**
   - If user's swap lands after `consume_randomness`, new rates apply
   - If user's swap lands before, old rates apply
   - **No in-between state** where SSE creates advantage

3. **Slippage protection catches misquotes**
   - If attacker quotes with new rate but program applies old rate (or vice versa)
   - minimumOutput validation fails transaction
   - User's output is protected

4. **SSE only provides information, not advantage**
   - Attacker learning about new rates earlier than public Helius webhook doesn't help
   - The on-chain program **always re-derives the tax** from current EpochState
   - There's no "window" to use an old rate if it's already changed on-chain

5. **Normal MEV still exists, but isn't tax-specific**
   - General sandwich attacks (front-running + back-running) are possible
   - But these affect **slippage on pool reserves**, not tax rates
   - Tax rates are **state-based, not order-dependent**

## Impact if Vulnerable

- User could theoretically lock in a favorable tax rate by seeing it via SSE before official block confirmation
- Worth ~100-400 bps (1-4%) per trade for large volumes
- Medium severity *if* on-chain enforcement were missing

## Impact Assessment (Actual)

**0 impact** — The on-chain Tax Program enforces whatever tax is in EpochState at execution time. SSE provides no advantage because the program always re-reads the state.

## Related Finding

- **H018 (Stale Quote Rounding)**: Addresses quote staleness for non-tax reasons (pool reserve changes). Not applicable to tax rates since they're epoch-based, not block-by-block.

## Recommendation

**No fix required.** The on-chain validation is sufficient. Document in comments that:
- Tax rates are enforced by the program, not the client
- SSE is informational only (for UX, to show users upcoming rates)
- Slippage protection is the user's defense against any quote/execution mismatch

The current architecture is **secure by design**.
