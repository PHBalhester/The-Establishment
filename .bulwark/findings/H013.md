# H013: CarnageSolVault Balance Desync via nativeBalanceChange Delta Misuse

## Verdict: CONFIRMED — HIGH

---

## Evidence

### The Bug (route.ts:552-558)

```typescript
// SystemAccount (CarnageSolVault) — no Anchor decode, just lamports
if (decodeInfo === null) {
  const lamports = item.accountData?.[0]?.nativeBalanceChange;
  protocolStore.setAccountState(pubkey, {
    lamports: typeof lamports === "number" ? lamports : 0,
    updatedAt: Date.now(),
  });
```

`nativeBalanceChange` is **a signed delta** (lamport change relative to the pre-transaction state), not the post-transaction absolute balance.

### Helius Documentation Confirmation

The Helius webhooks documentation shows the canonical enhanced transaction payload:

```json
"accountData": [
  {
    "account": "CKs1E69a2e9TmH4mKKLrXFF8kD3ZnwKjoEuXa6sz9WqX",
    "nativeBalanceChange": -72938049280,
    "tokenBalanceChanges": []
  }
]
```

A value of `-72938049280` is impossible as an absolute balance (negative lamports cannot exist). The field name itself — "Change" — and its signed integer semantics are unambiguous: this is a delta, not an absolute.

### Initial Seed Is Correct, But Overwritten

`ws-subscriber.ts:batchSeed()` (lines 133-138) correctly uses `getMultipleAccountsInfo` on boot and stores `info.lamports` — the genuine absolute balance:

```typescript
if (accountType === null) {
  // SystemAccount — store lamports only
  protocolStore.setAccountStateQuiet(pubkey.toBase58(), {
    lamports: info.lamports,   // <-- correct: absolute from RPC
    updatedAt: Date.now(),
  });
}
```

The correct absolute value from `batchSeed` is then **unconditionally overwritten** by the first incoming enhanced webhook that touches the CarnageSolVault, replacing it with the delta of that single transaction.

### Downstream Consumption

`useCarnageData.ts:46-48` reads the lamports directly and exposes them as `vaultBalanceLamports`:

```typescript
const vaultLamports = carnageSolVault
  ? (carnageSolVault.lamports as number) ?? 0
  : 0;
```

This value drives the displayed Carnage fund size in the UI. After the first post-boot vault transaction, users see a nonsensical number (positive deltas like `+240000` lamports from a small tax deposit, or even negative values on withdrawal).

### RPC Polling Fallback Masks the Bug Sometimes

`useProtocolState.pollViaRpc()` (lines 196-212) does correctly store `info.lamports` (absolute) during the SSE fallback polling path. However, this path only activates after 30 seconds of SSE downtime and runs every 60 seconds — it does not correct the webhook-induced desync under normal operating conditions.

---

## Attack Path

This is a **natural operation** bug, not a deliberate attack:

1. Server boots. `batchSeed()` runs `getMultipleAccountsInfo` and stores the correct absolute lamport balance for CarnageSolVault (e.g., `2_500_000_000` = 2.5 SOL).
2. A user executes any swap. The Tax Program routes 24% of the tax to CarnageSolVault (e.g., `+240_000` lamports).
3. Helius delivers an enhanced account change webhook for CarnageSolVault. The `nativeBalanceChange` field = `+240_000` (the delta for that transaction).
4. `handleAccountChanges()` overwrites the protocolStore with `lamports: 240_000`.
5. All SSE clients receive `protocol-update` and immediately display the Carnage vault balance as `240_000 lamports` (~0.000240 SOL) instead of the true ~2.5 SOL.
6. Every subsequent vault transaction sets the displayed balance to that transaction's delta (e.g., a Carnage buyback that spends 0.1 SOL would set `lamports: -100_000_000`, displaying as a negative balance).

---

## Impact

**Severity: HIGH** (display-layer only — no on-chain funds at risk)

- **Incorrect Carnage fund display**: Every user sees a wildly wrong vault balance after the first vault transaction post-boot. The displayed value swings between small positive deltas and potentially negative numbers with each transaction.
- **User manipulation risk**: The Carnage fund size is a key protocol signal — a large vault suggests an imminent high-value burn event, which influences buy/sell decisions. Displaying `0.00024 SOL` instead of `2.5 SOL` depresses perceived protocol value and suppresses buy pressure. Displaying a negative value erodes trust.
- **Frequency**: This fires on every swap (tax routed to vault each time) and every Carnage buyback execution. High-frequency desync under normal usage.
- **No funds at risk**: The actual on-chain vault balance is unaffected. Only the displayed value is wrong.
- **Partial mitigation exists**: Server restart resets the correct seed via `batchSeed`. The RPC polling fallback in `useProtocolState` (active only during SSE outage) would also correct it temporarily, but the next webhook delivery overwrites it again.

---

## Recommendation

Replace `nativeBalanceChange` with the post-transaction absolute balance. The correct source is `rawAccountData` (not used for SystemAccounts) or — since Helius enhanced account webhooks for SystemAccounts may not provide raw bytes — query the actual balance from `item.accountData[0]` if Helius provides a `lamports` field, or maintain a running total in the protocol store.

**Preferred fix**: Use `rawAccountData` or switch to deriving the absolute balance by keeping the last known absolute value and applying the delta:

```typescript
if (decodeInfo === null) {
  const delta = item.accountData?.[0]?.nativeBalanceChange ?? 0;
  // Option A: read previous value and apply delta
  const prev = protocolStore.getAccountState(pubkey);
  const prevLamports = typeof prev?.lamports === "number" ? prev.lamports : 0;
  protocolStore.setAccountState(pubkey, {
    lamports: prevLamports + delta,
    updatedAt: Date.now(),
  });
}
```

However, this approach accumulates errors if any webhook delivery is missed (no gap detection). The **robust fix** is to use Helius's `rawAccountData` field for the SystemAccount (it contains 0 data bytes but the lamport balance is available via the standard `getAccountInfo` response), or — simplest and most reliable — issue a targeted `getAccountInfo` RPC call for CarnageSolVault on each webhook receipt:

```typescript
if (decodeInfo === null) {
  // Fetch true absolute balance on each vault change notification
  const info = await connection.getAccountInfo(new PublicKey(pubkey));
  protocolStore.setAccountState(pubkey, {
    lamports: info?.lamports ?? 0,
    updatedAt: Date.now(),
  });
}
```

This adds one RPC call per vault transaction but guarantees correctness and resilience to missed webhooks.
