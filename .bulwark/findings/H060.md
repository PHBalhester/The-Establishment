## Hypothesis

`sendTransaction` in the RPC method allowlist means any unauthenticated client can use `/api/rpc` as a free Solana transaction relay, burning the project's Helius credits and potentially causing API key bans or service degradation.

## Verdict: CONFIRMED

The vulnerability is structurally present and confirmed by direct code inspection. `sendTransaction` is explicitly listed in `ALLOWED_METHODS` at line 44 of `rpc/route.ts`. The proxy applies only a per-IP rate limit of 300 requests/minute — no per-method rate limit exists, and there is no authentication gate whatsoever. Any actor on the internet can submit arbitrary pre-signed Solana transactions through this endpoint at will.

## Evidence

**`app/app/api/rpc/route.ts:31-59` — Allowlist includes `sendTransaction`:**

```ts
const ALLOWED_METHODS = new Set([
  // Account & balance queries
  "getAccountInfo",
  "getBalance",
  "getMultipleAccounts",
  "getTokenAccountsByOwner",
  "getTokenAccountBalance",
  "getProgramAccounts",

  // Transaction lifecycle
  "getLatestBlockhash",
  "sendTransaction",       // ← line 44: open relay
  "simulateTransaction",
  "getSignatureStatuses",
  "confirmTransaction",
  "getBlockHeight",
  ...
]);
```

**`app/app/api/rpc/route.ts:80-89` — Rate limit is per-IP, not per-method:**

```ts
export async function POST(request: NextRequest) {
  const clientIp = getClientIp(request);
  const rateCheck = checkRateLimit(clientIp, RPC_RATE_LIMIT, "rpc");
  if (!rateCheck.allowed) { ... }
  // ← no authentication, no method-specific throttle
```

**`app/lib/rate-limit.ts:44-48` — RPC limit configuration:**

```ts
export const RPC_RATE_LIMIT: RateLimitConfig = {
  windowMs: 60_000,
  maxRequests: 300,   // 300 sendTransaction calls/min per IP
};
```

**`app/lib/rate-limit.ts:129-135` — IP extraction trusts `x-forwarded-for` first hop:**

```ts
const forwarded = request.headers.get("x-forwarded-for");
if (forwarded) {
  const firstIp = forwarded.split(",")[0]?.trim();
  if (firstIp) return firstIp;
}
```

The rate limiter is also bypassable (ASM-3 / H050 regression): if an attacker spoofs distinct `x-forwarded-for` values, each spoofed IP gets its own 300 req/min bucket, giving unbounded relay throughput from a single actual machine.

**`app/app/api/rpc/route.ts:102-106` — Batch requests compound the issue:**

```ts
const isBatch = Array.isArray(body);
const requests = isBatch ? (body as Array<...>) : [body as ...];
// ← batch of N sendTransaction calls = N Helius credits burned per HTTP request
```

A single HTTP POST with a batch of `sendTransaction` items burns N credits simultaneously. There is no cap on batch size for any method, including `sendTransaction`.

## Attack Scenarios

### Scenario A: Helius Credit Exhaustion (Low sophistication)
An attacker scripts a loop sending 300 `sendTransaction` requests per minute per spoofed IP. Each request burns at least 1 Helius credit. With IP spoofing across N source IPs, this scales linearly. Helius free/starter tiers have monthly credit caps; exhaustion silently kills upstream RPC service for all legitimate users.

### Scenario B: Batched Credit Burn (Single request)
A single POST with a 1,000-item batch of `sendTransaction` calls burns 1,000 credits in one round trip, before the rate limiter can react (rate limit counts HTTP requests, not RPC calls within a batch). This is the same batch amplification vector as H038 (RPC batch amplification) but applied to the highest-credit-cost method.

### Scenario C: IP-Based Upstream Ban
All transactions relayed through `/api/rpc` originate from the Railway server IP. A sustained flood of invalid or spam transactions could cause Helius to flag the API key or the server IP for abuse, severing RPC access for all users.

### Scenario D: Transaction Spam Funded by Others
An attacker does not need their own Solana wallet or SOL to spam the Solana network. They can submit pre-signed garbage transactions (invalid signatures intentionally) through `/api/rpc`. These will be rejected by the validator, but each submission still costs the project a Helius credit and adds validator load.

This finding was explicitly anticipated by multiple audit auditors. It is not a speculation — it is a structurally confirmed open relay.

## No Per-Method Rate Limit Exists

There is no evidence of any additional `sendTransaction`-specific throttle anywhere in the codebase. Searching confirms there is only one rate limit check in the RPC POST handler (`checkRateLimit(clientIp, RPC_RATE_LIMIT, "rpc")`), applied uniformly to all methods. The `creditCounter` records calls for observability but imposes no limits.

## Recommended Fixes

**Fix 1 (Immediate): Add per-method rate limit for `sendTransaction`**

Apply a tighter, separate rate limit bucket when the method is `sendTransaction`. Example: 10 `sendTransaction` calls per minute per IP, independently of the general 300 req/min RPC budget.

```ts
const SEND_TX_RATE_LIMIT: RateLimitConfig = { windowMs: 60_000, maxRequests: 10 };

// After allowlist check, before forwarding:
if (requests.some(r => r.method === "sendTransaction")) {
  const txCheck = checkRateLimit(clientIp, SEND_TX_RATE_LIMIT, "rpc:sendTransaction");
  if (!txCheck.allowed) return new Response("Too Many Requests", { status: 429, ... });
}
```

**Fix 2 (Batch hardening): Cap batch size globally and per `sendTransaction`**

Reject batches exceeding a max size (e.g., 10 requests). Reject batches containing more than 1 `sendTransaction` call.

**Fix 3 (Defence-in-depth): Validate `sendTransaction` payload structure**

The proxy could inspect the transaction bytes for minimum plausibility (valid base58/base64 encoding, expected length range) before forwarding, rejecting obviously malformed payloads without burning a credit.

**Fix 4 (Long-term): Remove `sendTransaction` from the proxy**

The sign-then-send pattern (`signTransaction()` + `connection.sendRawTransaction()`) already routes through `/api/rpc`. Evaluate whether removing `sendTransaction` from the allowlist and routing all submissions through a dedicated endpoint (with stricter auth or CAPTCHA) is feasible without breaking the wallet adapter flow.
