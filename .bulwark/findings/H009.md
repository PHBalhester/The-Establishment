# H009: Next.js HTTP Request Smuggling (GHSA-ggv3) — CVE Present, Prerequisite Absent

## Verdict: NOT VULNERABLE

---

## Evidence

### Version confirmed vulnerable
`app/package.json:29` pins `"next": "16.1.6"`. GHSA-ggv3 affects Next.js versions prior to 16.2.1 and is confirmed present in the `npm audit` output documented in `.bulwark/context/30-DEP-01-package-deps.md` (R3, line 323–327). The fix is available at `next@16.2.1`.

### Prerequisite — `rewrites()` — is absent
GHSA-ggv3 is specifically a request smuggling vulnerability triggered by Next.js's internal reverse-proxy behavior when `rewrites()` is configured. The vulnerability manifests when Next.js proxies a matched rewrite rule to an upstream origin, allowing a crafted request to smuggle a second request past the upstream's parsing logic.

**Exhaustive search result**: `app/next.config.ts` contains no `rewrites()` function. The only async configuration function present is `headers()` (lines 82–117), which applies six static security headers to the `/(.*)`  pattern. No `redirects()`, no `rewrites()`, no `basePath` proxy, and no `experimental.proxyTimeout` setting exist in the file.

A project-wide grep for `rewrites`, `async rewrites`, and `redirects` across all TypeScript, JavaScript, and JSON files in `app/` returned zero matches.

### Configuration that is present (and irrelevant to this CVE)
The `next.config.ts` uses:
- `headers()` — static header injection only, no proxying
- `turbopack.resolveAlias` — module aliasing for browser builds
- `images` — Next.js image optimization config

None of these engage the internal proxy path that GHSA-ggv3 exploits.

### Webhook auth is independent of HTTP routing
Even if smuggling were theoretically possible, the Helius webhook endpoint (`app/app/api/webhooks/helius/route.ts:286–301`) uses application-level authentication: a constant-time `timingSafeEqual` comparison of `HELIUS_WEBHOOK_SECRET` against the `Authorization` header. This check runs at the handler level inside the Next.js request handler — after routing, after any rewrite — meaning it cannot be bypassed by transport-layer smuggling alone. Smuggling would need to forge a request that both (a) bypasses the routing layer and (b) carries a valid HMAC secret. The HMAC secret is only known to Helius and Railway.

### Railway reverse proxy
Railway deploys behind an nginx-based reverse proxy. Nginx's HTTP/1.1 chunked-encoding and Content-Length normalisation provides an additional layer of smuggling mitigation upstream of Next.js, though this is not explicitly verified in the codebase. Even without this, the absent `rewrites()` is the decisive factor.

---

## Attack Path

For GHSA-ggv3 to be exploitable the following chain would be required:

1. `next.config.ts` must define `rewrites()` targeting an upstream server — **ABSENT**
2. Attacker crafts an ambiguous Content-Length / Transfer-Encoding request pair
3. Next.js's rewrite proxy forwards the outer request to the upstream, which parses the smuggled inner request as a new HTTP request
4. Inner request lands on an unauthenticated or differently-authenticated route (e.g., an admin-only internal path)

Step 1 is not satisfied. The attack path is broken at the first prerequisite.

---

## Impact

None in current configuration. The CVE is present in the installed package but is inert because the triggering feature (`rewrites`) is not configured.

Residual risk: if a future developer adds `rewrites()` to `next.config.ts` without upgrading to `next@16.2.1`, the vulnerability becomes active. This is a latent risk tied to the unpatched package version, not an immediate exploitable condition.

---

## Recommendation

**Primary**: Upgrade `next` from `16.1.6` to `16.2.1` in `app/package.json`. This closes GHSA-ggv3 along with four other CVEs (null-origin CSRF bypass GHSA-mq59, disk cache exhaustion, postponed buffering DoS, HMR CSRF bypass). The version is pinned exactly (`"next": "16.1.6"`) so a direct bump to `"next": "16.2.1"` is a one-line change. Verify `eslint-config-next` matches (`"eslint-config-next": "16.2.1"`).

**Secondary**: Add `npx audit-ci --moderate` as a CI step in `.github/workflows/ci.yml` after `npm ci`. This gates future CVEs before deployment. Tracked under the broader DEP-01 recommendation (H009 is not a new separate finding — the upgrade resolves it alongside other CVEs in the same package).

**Do not add `rewrites()`** to `next.config.ts` without first upgrading Next.js. Adding rewrites to the current version would immediately activate this vulnerability.
