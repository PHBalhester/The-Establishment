# H056: No External Alerting on Circuit Breaker Trip

## Verdict: CONFIRMED

**Tier:** 2 (High)
**Origin:** KB (OC-251) + VERIFY (H004 Audit #1)
**Status:** NOT_FIXED

---

## Executive Summary

When the crank's circuit breaker trips — after 5 consecutive epoch failures — it logs a CRITICAL message to `console.error` and exits the main loop via `break`. No external notification is sent. No webhook, no Sentry event, no email, no Slack/Discord message. The process then falls through to the shutdown block and exits. Railway's `ON_FAILURE` restart policy applies only to the Next.js app service defined in `railway.toml`, not the crank process. Even if the crank process exits, Railway may restart it — meaning the same underlying error condition repeats, trips the breaker again, and again emits only console output that nobody reads overnight.

The operational consequence at mainnet: epochs stop advancing. Stakers stop earning yield. Carnage executions are missed. The only human-visible signal is Railway log output, which requires someone to manually open the dashboard and scroll. There is no push notification.

## Evidence

### 1. Circuit Breaker — Console Only

**Location:** `scripts/crank/crank-runner.ts:535-539`

```typescript
if (consecutiveErrors >= CIRCUIT_BREAKER_THRESHOLD) {
  console.error(
    `[crank] CRITICAL: Circuit breaker tripped after ${CIRCUIT_BREAKER_THRESHOLD} consecutive errors. Halting.`
  );
  break;
}
```

This is the totality of the circuit breaker notification logic. There is no call to any external alerting service before or after this block. The `break` exits the `while (!shutdownRequested)` loop and falls through to the shutdown banner at lines 555-562, which also emits only console output.

### 2. Spending Cap Halt — Console Only

**Location:** `scripts/crank/crank-runner.ts:141-145`

```typescript
if (currentSpend + lamports >= MAX_HOURLY_SPEND_LAMPORTS) {
  console.error(
    `[crank] CRITICAL: Hourly spending cap reached! ` +
    `Current: ${currentSpend} + ${lamports} >= ${MAX_HOURLY_SPEND_LAMPORTS} lamports. Halting.`
  );
  return false;
}
```

The `recordSpend()` function returns `false`, which causes the caller to `break` the main loop. Same pattern: console only, no external push.

### 3. Low Balance Warning — Console Only

**Location:** `scripts/crank/crank-runner.ts:406-410`

```typescript
if (balance < LOW_BALANCE_SOL * LAMPORTS_PER_SOL) {

```

Low balance is a pre-failure warning state. It is also console-only. An operator would need to be watching Railway logs in real time to catch this before it becomes a hard failure.

There is no import for Sentry, a Discord/Slack SDK, PagerDuty, OpsGenie, Telegram, or any HTTP fetch call to a webhook URL. No `ALERT_WEBHOOK_URL`, `DISCORD_WEBHOOK_URL`, `SENTRY_DSN`, or equivalent env var is referenced anywhere in the file (confirmed by full-file grep: zero matches).

### 5. Crank Health Endpoint Does Not Push Alerts

**Location:** `scripts/crank/crank-runner.ts:163-189`

The crank exposes a `/health` endpoint on port 8080 that reports `status: "halted"` when the circuit breaker has tripped:

```typescript
status: consecutiveErrors >= CIRCUIT_BREAKER_THRESHOLD ? "halted" : "running",
```

However, this is a passive pull endpoint — it answers HTTP GET requests. It does not push notifications to any external system. Railway's internal health check polls the Next.js app (`/api/health` per `railway.toml:8`), not the crank's port 8080. Even if Railway did poll the crank health endpoint, the Railway healthcheck is a liveness check for container restarts, not an alerting mechanism that sends notifications to operators.

### 6. Railway Restart Policy Does Not Alert — It Restarts

**Location:** `railway.toml:10-11`

```toml
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 3
```

This restart policy applies to the service defined in `railway.toml`, which is the Next.js app (`startCommand = "npm run --workspace app start"`). The crank runner is a separate Railway service with its own `railway.toml` (not present in the repository — the crank is deployed separately). Regardless: Railway's restart-on-failure policy silently restarts the container; it does not send a push notification to the operator. After `maxRetries: 3` is exhausted, Railway marks the service as crashed — again with no push notification unless Railway's built-in alerting is manually configured in the Railway dashboard (no evidence this has been done).

### 7. No Sentry Integration in the Crank Process

**Location:** `app/lib/sentry.ts` (Next.js app only)

The zero-dependency Sentry reporter exists in the Next.js app (`app/lib/sentry.ts`). It is not imported or used by the crank runner. The crank process has no Sentry DSN configured, no error envelope posting, and no breadcrumb capture. Circuit breaker trips, spending cap halts, and VRF errors are all invisible to Sentry.

## Attack / Failure Scenario

```
Night 1 (02:00 UTC):
  RPC node degrades → VRF commit TX starts timing out
  Crank logs: "[crank] ERROR cycle 47 (1/5 consecutive): ..."
  Crank logs: "[crank] ERROR cycle 48 (2/5 consecutive): ..."
  ...
  Crank logs: "[crank] CRITICAL: Circuit breaker tripped after 5 consecutive errors. Halting."
  Crank process exits.

Night 1 (02:02 UTC):
  Railway detects process exit → restarts container (ON_FAILURE).
  Same RPC node is still degraded → same error sequence.
  Circuit breaker trips again after 5 cycles.
  Restart #2 → same result. Restart #3 → same result.
  Railway marks service as crashed after maxRetries = 3.

Night 1 (02:15 UTC) through Night 2 (09:00 UTC):
  Epochs stop advancing (on-chain epoch boundary has been passed for ~7 hours).
  Stakers stop earning yield for the entire window.
  Multiple Carnage executions are missed, reducing PROFIT token burn rate.
  Frontend shows stale epoch data via SSE (ws-subscriber continues running, but
  no new epoch transitions are recorded).

Morning of Night 2 (09:00 UTC):
  Operator opens Railway dashboard, notices service is in crashed state.
  Investigates logs, finds circuit breaker message from 7 hours ago.
  Manually restarts service.
```

No on-call notification, no incident ticket, no wake-up. The failure window is bounded only by how often the operator happens to check the Railway dashboard.

## Recommendation

### Tier 1 — Immediate (pre-mainnet required)

Add a `sendAlert(message: string): Promise<void>` helper to the crank that fires a webhook POST on any CRITICAL condition. The cheapest implementation is a Discord or Slack incoming webhook (free, no dependencies, single `fetch` call):

```typescript
// crank-runner.ts — add near top
async function sendAlert(message: string): Promise<void> {
  const webhookUrl = process.env.ALERT_WEBHOOK_URL;
  if (!webhookUrl) return; // No-op if not configured — never throw from alerting
  try {
    await fetch(webhookUrl, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ content: `[Dr. Fraudsworth Crank] ${message}` }),
    });
  } catch {
    // Alerting must never crash the crank
  }
}

// Then at circuit breaker trip (line 535):
if (consecutiveErrors >= CIRCUIT_BREAKER_THRESHOLD) {
  await sendAlert(
    `CRITICAL: Circuit breaker tripped after ${CIRCUIT_BREAKER_THRESHOLD} consecutive errors. Epochs halted. Manual restart required.`
  );
  console.error(`[crank] CRITICAL: Circuit breaker tripped...`);
  break;
}

// At spending cap halt (recordSpend function):
await sendAlert(
  `CRITICAL: Hourly spending cap reached (${currentSpend + lamports} >= ${MAX_HOURLY_SPEND_LAMPORTS} lamports). Crank halted.`
);

// At low balance warning (line 406):
await sendAlert(
  `WARNING: Crank wallet balance low: ${(balance / LAMPORTS_PER_SOL).toFixed(3)} SOL (threshold: ${LOW_BALANCE_SOL} SOL)`
);
```

Set `ALERT_WEBHOOK_URL` in Railway env to a Discord/Slack incoming webhook URL. Cost: zero dollars, ~20 lines of code.

### Tier 2 — Short-term

- Port the existing `app/lib/sentry.ts` zero-dependency reporter to a shared utility usable by the crank. Add Sentry error capture at the circuit breaker trip point so the full error chain (all 5 consecutive errors) is captured in a Sentry issue with context.

### Tier 3 — Long-term / mainnet production

- Configure Railway's built-in deployment notifications (Settings → Notifications) to alert on service crashes. This handles the Railway restart exhaustion case.
- Consider PagerDuty or OpsGenie for on-call escalation on CRITICAL crank events.
