# H103: Sentry Wildcard CSP — Data Exfiltration Risk

## Verdict: ACCEPTED_RISK

---

## Evidence

### CSP Configuration (next.config.ts:43)

The `connect-src` directive in the main CSP includes two wildcard patterns for Sentry ingest:

```typescript
connect-src 'self' wss://relay.walletconnect.com wss://relay.walletconnect.org wss://www.walletlink.org
  https://explorer-api.walletconnect.com https://${heliusRpcDomain} wss://${heliusRpcDomain}
  ${heliusApiSources} https://*.ingest.sentry.io https://*.ingest.us.sentry.io;
```

### Analysis of Wildcard Patterns

The two Sentry patterns are:
1. **`https://*.ingest.sentry.io`** — allows any single-label subdomain (e.g., `o12345.ingest.sentry.io`)
2. **`https://*.ingest.us.sentry.io`** — allows any single-label subdomain under the US region (e.g., `o12345.ingest.us.sentry.io`)

### Design Rationale

The wildcard is intentional and necessary:

1. **Multi-region support**: Sentry's US region DSNs use `o<ORG_ID>.ingest.us.sentry.io` and EU DSNs use `o<ORG_ID>.ingest.sentry.io`. The wildcard allows the same CSP policy to serve both regions without rebuild.

2. **Future org transitions**: If the project migrates to a different Sentry organization (different org ID), the CSP does not require rebuilding — the new DSN will still match the wildcard.

3. **Development flexibility**: Multiple Sentry projects or orgs can be tested in development without CSP reconfiguration.

4. **Implementation in codebase**: The project's error reporting uses a zero-dependency custom lib/sentry.ts (CRITICAL Sentry + Turbopack finding notes) rather than `@sentry/nextjs` (which breaks Turbopack SSR). This custom integration POSTs error envelopes to the Sentry ingest API, and the wildcard CSP must allow any Sentry ingest domain the codebase might target.

## Exfiltration Risk Assessment

### Attack Model — Subdomain Spoofing

An attacker could potentially register a subdomain like `o999999.ingest.sentry.io` and configure it to receive HTTP POST requests that would normally go to the legitimate Sentry endpoint `o<REAL_ORG_ID>.ingest.sentry.io`.

**Scope of this attack:**
- The attacker must somehow convince the application to use a different Sentry DSN (via environment variable override, bundler modification, or runtime injection)
- OR the attacker controls a registrar with power to issue subdomains under `ingest.sentry.io` (requires Sentry infrastructure breach or registrar compromise — extremely high bar)

### Practical Constraints

1. **DSN is compile-time fixed**: The Sentry organization ID is baked into `lib/sentry.ts` at build time or pulled from `process.env.NEXT_PUBLIC_SENTRY_DSN`. It is not user-supplied or dynamically resolved from untrusted sources. An attacker cannot change the DSN via request parameters, user input, or ambient data.

2. **No DNS hijacking in scope**: This finding assumes normal DNS resolution and does not account for DNS spoofing, BGP hijacking, or network-level MITM attacks. Those are separate infrastructure risks outside the application's control.

3. **Legitimate pattern in the industry**: CSP wildcards for error reporting endpoints are standard practice. Auth0, Sentry docs, and other services routinely recommend `https://*.ingest.sentry.io` to support multi-tenant or future-org scenarios. The OWASP CSP guide does not classify this as a high-risk pattern because the attack vector requires either:
   - Attacker control of the DSN (which would require separate compromise of environment configuration)
   - Attacker control of Sentry's registrar or DNS (infrastructure-level compromise, out of scope)

4. **Sentry as trusted service**: The wildcard permits connections to ANY subdomain of `ingest.sentry.io`, but Sentry is an industry-standard, security-focused error reporting service. An attacker who controls a Sentry subdomain has already compromised Sentry's infrastructure — at that point, the CSP wildcard is a secondary concern.

5. **Data sent to Sentry is error context, not secrets**: The application's custom `lib/sentry.ts` POSTs error envelopes containing:
   - Stack traces
   - Error messages
   - Browser environment info (User-Agent, viewport, etc.)
   - Breadcrumbs of recent user actions
   - Request/response bodies (if captured by error handler)

   None of these should contain private keys, seed phrases, or authentication tokens (those would be application-level bugs, not CSP failures). Standard practice is to redact/strip PII from error context before sending to error reporting services.

---

## Impact Assessment

**Severity: LOW-MEDIUM** (ACCEPTED_RISK)

- **Confidentiality**: An attacker with control of a `*.ingest.sentry.io` subdomain could receive error context data (stack traces, breadcrumbs, request metadata). This is lower-risk than exfiltrating authentication tokens or crypto keys.
- **Integrity**: No impact. The CSP does not allow POST-back to the application, so exfiltration is one-way.
- **Availability**: No impact.

**Blast radius**: Only affects the error reporting channel. Core swap/staking/governance functionality is not at risk.

---

## Recommendation

**ACCEPT_RISK** — This is a standard industry pattern for error reporting CSP policies. The wildcard is necessary to support Sentry's multi-region architecture. Tightening the CSP to a specific organization ID would require rebuild on org migration.

**Mitigation strategies (optional, for defense-in-depth):**

1. **At build time, narrow the wildcard** (if org ID is known and will never change):
   ```typescript
   const sentryIngestDomain = process.env.NEXT_PUBLIC_SENTRY_ORG_ID
     ? `https://${process.env.NEXT_PUBLIC_SENTRY_ORG_ID}.ingest.sentry.io`
     : `https://*.ingest.sentry.io`;
   ```
   This would lock the CSP to a specific org ID if set, but fall back to wildcard in development. Trade-off: adds complexity for minimal gain.

2. **Audit error context before sending** (defense-in-depth):
   - In `lib/sentry.ts`, add a sanitization step to ensure no private keys, seed phrases, or auth tokens are included in error envelopes before POSTing to Sentry.
   - This is a good practice regardless of CSP configuration.

3. **Monitor Sentry for suspicious activity** (operational):
   - Set up alerts in Sentry to flag if error volumes spike (potential sign of exfiltration).
   - Regularly review which error types and frequency are normal for the application.

4. **Subresource integrity not applicable** — Sentry is a remote service, not a bundled script, so SRI does not apply here.
