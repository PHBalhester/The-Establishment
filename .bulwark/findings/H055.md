## Hypothesis

Two simultaneous crank instances can run concurrently — causing duplicate VRF TX1 randomness account creation (double SOL rent spend), double vault top-ups, and a race on epoch state reads and commits. There is no distributed lock, file lock, PID file, Redis semaphore, or any other mechanism preventing a second crank from starting.

## Verification

### 1. No lock mechanism in crank-runner.ts

A full read of `scripts/crank/crank-runner.ts` (569 LOC) confirms no lock of any kind:

- No file-based PID lock (`/var/run/crank.pid` or equivalent)
- No Redis/database distributed lock
- No process-level singleton guard
- No Railway instance count check
- The only coordination mechanism is on-chain state (VrfAlreadyPending) which prevents functional duplicate epoch advances but does NOT prevent wasted SOL

The crank directory contains exactly two files: `crank-runner.ts` and `crank-provider.ts`. No lock helper exists.

This is the Next.js app service config, not the crank service. No `numReplicas` or horizontal scaling field appears in the toml. Railway defaults to a single instance per service and does NOT auto-scale to multiple replicas unless explicitly configured. However:

- Railway's rolling deployment strategy (new container starts before old one stops) creates a window during each deploy where **two crank processes run simultaneously**
- Manual double-deploy or accidental configuration error can produce the same result
- No Railway-level "max instances = 1" constraint is programmatically enforced in the project files

### 4. What happens when two instances run simultaneously

**TX1 — Randomness account creation (both succeed):**
Both instances call `sb.Randomness.create()`, each paying rent for a new randomness keypair (~1.3M lamports / ~0.0013 SOL per account). Both accounts are created on-chain. The losing instance's randomness account is orphaned and its rent is unrecoverable.

**TX2 — Commit + trigger_epoch_transition (only one succeeds):**
On-chain VrfAlreadyPending gate rejects the second TX2. The losing crank receives an error, increments `consecutiveErrors`, and will retry next cycle. No functional damage to the protocol.

**Vault top-up (both may fire):**
Both instances independently read vault balance and independently decide to top up. The top-up is a `SystemProgram.transfer`, not idempotent. Two instances can each transfer `VAULT_TOP_UP_LAMPORTS` (5M lamports / 0.005 SOL) before either's state update is visible to the other. The `MAX_TOPUP_LAMPORTS` cap (0.1 SOL) applies per-operation, not globally — both operations individually pass the check.

**Epoch state read race:**
Both instances read EpochState at the start of each cycle. The winning instance advances the epoch; the losing instance enters error retry with a stale state snapshot. On the retry, it re-reads correctly. No functional protocol harm.

### 5. Blast radius analysis

**Devnet (current):** Wasted rent ~0.0013 SOL per duplicate creation. At one epoch per ~5 min (devnet: 750 slots × 0.4s), worst-case overlap duration during Railway rolling deploy is ~60s → at most 1-2 duplicate TX1s per deploy. Negligible cost.

**Mainnet (pre-launch):** One epoch per ~30 min (mainnet: 4500 slots × 0.4s). Rolling deploy overlap is still ~60s. At most 1-2 duplicate TX1s per deploy, but at mainnet SOL price (~$200/SOL), 0.0013 SOL × 2 ≈ $0.52 per deploy. Acceptable.

**Worst case — persistent double-run (bug or manual error):** Both instances run for a full hour. Spending cap (0.5 SOL/hour) is in-memory per process, not shared. Two instances could each spend 0.5 SOL/hour = 1 SOL/hour total = $200/hour at mainnet prices. This is the real risk ceiling.

**On-chain protocol correctness:** NOT affected. On-chain VrfAlreadyPending and epoch boundary checks prevent duplicate epoch advances. Users see correct state. The risk is purely financial (crank wallet SOL drain) and operational (circuit breaker trips on the losing instance from repeated TX2 failures).

## Verdict

**CONFIRMED — ACCEPTED_RISK**

The vulnerability is real and structurally present: no distributed lock exists, and two simultaneous crank instances would double-spend on VRF TX1 rent and vault top-ups. However, the ACCEPTED_RISK designation from Audit #1 stands for the following reasons:

1. **On-chain correctness is preserved.** Protocol users and stakers are unaffected. VrfAlreadyPending prevents duplicate epoch advances. The 50% output floor remains enforced.

2. **Railway default is single-instance.** Without explicit horizontal scaling configuration (none is present), Railway runs one crank instance. The overlap window only occurs during rolling deploys (~60s).

3. **Spending cap limits the damage.** In the worst case (persistent dual-run), each process is capped at 0.5 SOL/hour. Total drain ceiling is 1 SOL/hour, tolerable for the deploy overlap scenario.

4. **Adding a distributed lock requires external infrastructure.** A proper solution needs Redis or a database-backed advisory lock — infra not currently in the Railway deployment. A file-based PID lock would protect only within a single container, not across Railway rolling deploy containers (separate filesystems). This complexity is out of scope for the current phase.

5. **Mainnet blast radius is bounded.** At worst, a rolling deploy wastes ~$0.52 in duplicate randomness account rent. Vault double-top-up adds 0.005 SOL max. Both are acceptable operational costs.

**Carry-forward note for mainnet:** If Railway's deploy configuration ever changes to explicitly set `numReplicas > 1` or if the crank is scaled out, this finding MUST be re-evaluated and escalated to MUST_FIX. The accepted risk is predicated on single-instance operation by platform default.
