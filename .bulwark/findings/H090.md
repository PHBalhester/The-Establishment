# H090: Slot Estimation Drift

**Severity**: Tier 3 (Low Impact)
**Status**: CONFIRMED - DISPLAY ONLY (ACCEPTED RISK)
**Target**: `app/hooks/useCurrentSlot.ts:53-61`

---

## Finding Summary

Linear extrapolation in `useCurrentSlot.ts` can drift 10+ slots during network congestion when Solana's actual slot time exceeds the hardcoded 400ms assumption.

---

## Technical Analysis

### Extrapolation Logic

```typescript
// app/hooks/useCurrentSlot.ts:53-61
useEffect(() => {
  const interval = setInterval(() => {
    if (baseSlotRef.current === null) return;
    const elapsedMs = Date.now() - baseTimestampRef.current;
    const elapsedSlots = Math.floor(elapsedMs / MS_PER_SLOT);
    setDisplaySlot(baseSlotRef.current + elapsedSlots);
  }, ESTIMATION_INTERVAL_MS);
  return () => clearInterval(interval);
}, []);
```

Where `MS_PER_SLOT = 400` (from `shared/constants.ts:137`).

### Drift Mechanism

1. **Base assumption**: 400ms per slot = 2.5 slots/second
2. **Reality during congestion**: Slot times can exceed 600ms (measured on-chain during peak load)
3. **Estimation error**:
   - Expected: 1 slot every 400ms
   - Actual: 1 slot every 600ms during congestion
   - Drift after 10s: 25 SSE updates × 400ms = +10 slots overestimated
4. **No recalibration**: SSE updates arrive ~5s apart (ESTIMATION_INTERVAL_MS). Between updates, the linear extrapolation compounds the error.

### Drift During Example Scenario

- SSE sends slot N at t=0ms
- At t=5000ms (5s later):
  - Expected estimate: N + floor(5000/400) = N + 12 slots
  - If actual slot time is 500ms: actual = N + floor(5000/500) = N + 10 slots
  - **Drift = +2 slots**
- Cumulative over 10 consecutive updates: **+20 slots possible**

## Impact Assessment

### CountdownTimer Display Behavior

The countdown shows remaining time as:
```typescript
const remainingSlots = deadlineSlot - currentSlot;
const remainingMs = remainingSlots * MS_PER_SLOT;
const timeStr = hours > 0 ? `~${hours}h ${minutes}m` : `~${minutes}m`;
```

Note: `~` prefix indicates approximate time.

**Drift effects**:
- Overestimated slot count → countdown shows MORE time remaining than true
- User sees "~5h 30m remaining" when actual is "~5h 20m"
- Worst case (20-slot drift): 8-second error in displayed time
- **Negligible UX impact** given:
  - `~` prefix explicitly signals approximation
  - 8s error over hours-long countdown is imperceptible
  - No functional consequence (user can still trade or refund)

### Non-Impact on Security/Functionality

1. **Refund deadline logic**: Uses on-chain `deadlineSlot`, not browser estimate
2. **Swap execution**: Uses on-chain blockhash, not slot estimate
3. **Transaction success**: Depends on on-chain state, not display slot
4. **VRF timeout**: Uses hardcoded `VRF_TIMEOUT_SLOTS = 300` on-chain

---

## Verdict

**CONFIRMED** — Slot estimation can drift 10+ slots during congestion.
**ACCEPTED_RISK** — Impact limited to UI countdown display (non-functional).

### Rationale

1. ✅ Zero impact on transaction construction or on-chain behavior
2. ✅ Approximation is explicitly disclosed in UI (`~` prefix)
3. ✅ Error magnitude (8s max) is negligible relative to hour-long countdown durations
4. ✅ No user action depends on countdown precision
5. ✅ Fixing would require:
   - Either frequent RPC `getSlot()` calls (kills the devnet savings of this optimization)
   - Or full re-architecture with VRF oracle slot feed
   - Cost/benefit strongly unfavorable for display-only feature

---

## Mitigation (Future, Optional)

If countdown accuracy becomes critical post-launch:

1. **Recalibration interval**: Every 3-5 SSE updates, re-baseline from latest SSE slot
   - Bounds drift to ~12 slots (1 missed update cycle)
   - Negligible additional complexity

2. **Actual slot provider**: Integrate Switchboard VRF oracle timestamp instead of local estimation
   - Eliminates drift entirely
   - Requires VRF callback plumbing
   - Overkill for countdown display
