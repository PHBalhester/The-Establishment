# H074: Error Truncation Loses Anchor Logs

**Tier:** 3 (Medium)
**Status:** CONFIRMED, NOT_VULNERABLE
**Target:** `scripts/crank/crank-runner.ts:529`
**Date Verified:** 2026-03-21

---

## Finding

Line 529 in `crank-runner.ts` truncates error messages to 300 characters:

```typescript
const errStr = String(err).slice(0, 300);
console.error(
  `[crank] ERROR cycle ${cycleCount} (${consecutiveErrors}/${CIRCUIT_BREAKER_THRESHOLD} consecutive): ${errStr}`
);
```

Anchor framework errors include a `.logs` property (an array of program execution logs from failed transactions). Converting the error to string via `String(err)` does not include this `.logs` array, and the 300-character truncation further discards any context about the actual failure.

---

## Root Cause

1. **No `.logs` extraction**: The code uses `String(err)` which converts the error to its string representation. Anchor errors contain `.logs` as a separate property (array of strings) that provides on-chain program execution context.
2. **Truncation removes context**: The `.slice(0, 300)` limit discards detailed error messages that might be needed to diagnose issues.

---

## Evidence

Across the codebase, other scripts properly extract `.logs` from Anchor errors:

- **scripts/deploy/initialize.ts:2111-2115**
  ```typescript
  if (err.logs) {
    console.error("Program logs:");
    for (const log of err.logs) {
      console.error("  ", log);
    }
  }
  ```

- **scripts/deploy/transfer-authority.ts:688-690**
  ```typescript
  if (err.logs) {
    for (const log of err.logs) {
      console.error("  ", log);
    }
  }
  ```

- **scripts/e2e/smoke-test.ts:74-75**
  ```typescript
  if (err.logs) {
    for (const log of err.logs) {

  ```

This pattern is consistent across 9+ deployment and test scripts. The crank-runner.ts is the **only** main loop process that fails to extract `.logs`.

---

## Risk Assessment

**Severity: MEDIUM**

### Why NOT Vulnerable (to exploitation)
- The crank runner's error handling is fail-closed: it logs the error and either retries or triggers the circuit breaker.
- Truncating logs does not change the security behavior — both the code and on-chain state are unaffected.
- The crank operator (whoever runs it on Railway) loses debugging context, but external attackers cannot exploit this loss.

### Why It's Problematic (for operability)
- **Debugging burden**: When the crank fails, the operator gets a garbled 300-char substring with no on-chain context.
  - Example: A program instruction returns error code 0x1780 (oracle timeout in VRF flow). The `.logs` would explain this; the string truncation loses it.
  - Example: An account balance check fails. The `.logs` would show "CarnageSolVault balance: 1000 lamports, required: 2000". Truncation hides it.
- **Operational risk**: The crank operator cannot quickly diagnose why the process halted, delaying recovery.
- **Inconsistency**: All other error paths in the project extract `.logs`; this one doesn't.

---

## Verification

**Target inspection:** Line 529 of `scripts/crank/crank-runner.ts`

✅ **CONFIRMED**: The error truncation exists and discards `.logs`.
✅ **NOT_VULNERABLE**: No security boundary is violated; the error log loss is an operability issue, not an exploit path.
✅ **ACCEPTED_RISK**: The crank's fail-closed design (circuit breaker + retry) is sound. However, for a production system, `.logs` extraction is strongly recommended for faster incident response.

---

## Recommendation

Extract Anchor error `.logs` before truncation:

```typescript
} catch (err) {
  consecutiveErrors++;
  const errMsg = String(err);
  const errStr = errMsg.slice(0, 300);

  let logOutput = `[crank] ERROR cycle ${cycleCount} (${consecutiveErrors}/${CIRCUIT_BREAKER_THRESHOLD} consecutive): ${errStr}`;
  if ((err as any).logs && Array.isArray((err as any).logs)) {
    logOutput += `\n  Program logs:\n    ${(err as any).logs.join("\n    ")}`;
  }

  console.error(logOutput);
  // ... rest of error handling
}
```

This maintains the 300-char cap on the main error message (preventing log spam) while preserving on-chain diagnostics in `.logs`.
