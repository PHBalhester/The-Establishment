# H048 ‚Äî Stale Quote-to-Execution TOCTOU

**Tier:** 2 (High)
**Status:** CONFIRMED
**Origin:** KB (OC-122, OC-271)
**Auditor:** Dinh's Bulwark Audit #2

---

## Hypothesis

Quote computed with 300ms-debounced SSE data. User think time + signing + RPC round trip = 5‚Äì15 s gap. `minimumOutput` computed from stale data is the only slippage protection.

---

## Targets Investigated

- `app/hooks/useSwap.ts:690‚Äì808` ‚Äî `executeSwap` / `executeRoute`
- `app/hooks/useRoutes.ts:582‚Äì592` ‚Äî 30 s auto-refresh countdown
- `app/lib/swap/multi-hop-builder.ts:298‚Äì327` ‚Äî `buildAtomicRoute`
- `app/lib/swap/route-engine.ts:367‚Äì384` ‚Äî `minimumOutput` at quote time

## Code Evidence

### 1. Quote is frozen at click time ‚Äî never recomputed before signing

`executeSwap` (line 690) reads `quote` directly from React state ‚Äî the value set by the last `computeQuote` debounce call. No freshness check, no re-quote, no staleness guard:

```typescript
// useSwap.ts:691
if (!wallet.publicKey || !wallet.connected || !quote) { return; }
// ...
minimumOutput: quote.minimumOutput,   // lines 735, 747 ‚Äî passed unchanged to builder
```

`executeRoute` (line 839‚Äì900) reads `routesResult.selectedRoute` from React state ‚Äî the object produced by the last `computeAllRoutes` call. It passes the route directly to `buildAtomicRoute` with no re-quote:

```typescript
const route = routesResult.selectedRoute;        // line 845 ‚Äî stale React state
const build = await buildAtomicRoute(route, connection, wallet.publicKey, ...);
```

`buildAtomicRoute` (multi-hop-builder.ts:306‚Äì308) reconstructs per-step `minimumOutput` values from `route.minimumOutput / route.outputAmount` ‚Äî but both of these numbers came from the frozen route object, so the derived slippage BPS is numerically correct relative to the stale quote, **not** to current pool state:

```typescript
const slippageBps =
  route.outputAmount > 0
    ? Math.floor((1 - route.minimumOutput / route.outputAmount) * 10_000)
    : 100;
```

### 2. `minimumOutput` is set from stale SSE reserves at debounce time

For direct swaps (`useSwap.ts:418`):
```typescript
const minimumOutput = Math.floor(outputTokens * (10_000 - slippageBps) / 10_000);
```
For multi-hop routes (`route-engine.ts:369`):
```typescript
const minimumOutputBI = finalOutputBI * BigInt(10_000 - slippageBps) / 10_000n;
```

In both cases the output amount ‚Äî and therefore the `minimumOutput` floor ‚Äî is computed from whatever pool reserves were last pushed via SSE at the time the debounce fired. These reserves can be up to **30 s stale** between auto-refreshes, plus an additional 5‚Äì60 s propagation latency for the SSE pipeline itself (noted in ARCHITECTURE.md ASM-8).

### 3. Auto-refresh cadence vs. real staleness window

`useRoutes.ts:50`: `REFRESH_INTERVAL_SECONDS = 30`

The auto-refresh fires `computeAllRoutes()` every 30 s. However:
- The refresh uses the same SSE-fed `pools` and `epochState` props ‚Äî it does **not** perform a fresh on-chain RPC read.
- If the SSE pipeline is stale (ws-subscriber backpressure, Helius delay), the auto-refresh recomputes from the same stale data.
- The 30 s timer resets on every input change (`useRoutes.ts:521‚Äì568`), so a user who frequently edits the amount can push the last actual auto-refresh far into the past.

### 4. Total staleness window measurement

| Phase | Latency |
|---|---|
| SSE debounce | 300 ms |
| User think time (review quote, open wallet) | 2‚Äì10 s |
| Wallet popup + signing | 1‚Äì3 s |
| `buildAtomicRoute` async work (ALT fetch + blockhash) | 0.5‚Äì1.5 s |
| RPC round trip + mempool wait | 0.5‚Äì3 s |
| **Worst-case total** | **~18 s** |
| SSE pipeline latency at quote computation time | 0‚Äì60 s |
| **Combined worst-case** | **up to ~78 s** |

In the worst case a user executes a swap against a quote that reflects pool state from over a minute ago.

### 5. Impact amplified by 500 BPS default slippage

`SettingsProvider.tsx:170` sets `slippageBps = 500` by default (H015, NOT_FIXED). This means `minimumOutput` allows up to 5% adverse movement from the already-stale quoted price. A realistic 1‚Äì3% price move during the staleness window is fully absorbed by the slippage tolerance without triggering an on-chain revert, meaning the user loses real value without any indication that the execution deviated from the displayed quote.

---

## Verdict: CONFIRMED

The staleness window is real, measurable, and unmitigated in code. The combination of:
1. No re-quote or freshness check between the user clicking "Swap" and signing, and
2. `minimumOutput` baked from stale SSE reserves at debounce time, and
3. 500 BPS default slippage absorbing realistic pool drift silently

means a user can sign and execute a transaction whose `minimumOutput` protection is anchored to price data that is up to 30‚Äì78 seconds old. On a liquid pair this is low impact; on a low-liquidity memecoin pair with high volatility (the exact use case here) this is a meaningful financial risk.

**The on-chain `minimum_output` check does enforce the floor** ‚Äî the transaction will revert if the actual output falls below `minimumOutput`. This is the safety net. But `minimumOutput` is computed from a stale quote, so the floor itself may be set lower than it would be if computed from current reserves. The vulnerability is not fund loss via reversion bypass; it is the user silently accepting a worse price than they think they agreed to.

---

## Recommended Remediation

1. **Re-quote at execution time**: Immediately before building the transaction, call `computeAllRoutes()` (or `computeQuote()`) synchronously using the current `pools` and `epochState` props. Compare the freshly computed `minimumOutput` to the displayed quote ‚Äî if they differ by more than a configured threshold (e.g., 50 BPS), warn the user or abort.

2. **Staleness guard on `executeSwap` / `executeRoute`**: Record a `quotedAtMs` timestamp when `minimumOutput` is computed. In the execute path, check `Date.now() - quotedAtMs > QUOTE_MAX_AGE_MS` (e.g., 10 s) and re-quote or warn if exceeded.

3. **Reduce SSE data age**: The auto-refresh should perform a live `getMultipleAccountsInfo` RPC call for pool reserves rather than recomputing from cached SSE data, so the 30 s refresh actually reflects on-chain state.

4. **Separate concerns**: Items 1 and 2 are off-chain UX improvements that do not require program changes. Item 3 improves data freshness across all quote consumers.
