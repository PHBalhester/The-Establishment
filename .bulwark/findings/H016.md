# H016: Default 5% Slippage — Sandwich Attack Surface Partially Mitigated by On-Chain Tax Structure

## Verdict: PARTIALLY_MITIGATED

Confidence: 8/10

---

## Evidence

### 1. Default slippage is confirmed at 500 BPS (5%)

`app/providers/SettingsProvider.tsx:170` — `getDefaults()` returns `slippageBps: 500`. This has not changed since Audit #1 (H015) or any subsequent verification round.

```typescript
return {
  slippageBps: 500,   // 5% — unchanged since Audit #1
  priorityFeePreset: 'medium',
  ...
};
```

The `loadSettings()` function at line 194-199 validates the range as `[0, 10_000]` — meaning a value injected via localStorage can reach 100% slippage. The UI caps custom input at 50% (`SlippageConfig.tsx:133`) but `SettingsProvider` does not mirror that cap.

### 2. No Jito integration anywhere in the off-chain codebase

Grep of the entire `app/` directory for `jito`, `private.*mempool`, `MEV.*protect`, `bundle.*jito` returns zero matches in application code. The only MEV-related string (`"Typically bundled in the same transaction for MEV protection"`) appears only in the on-chain IDL metadata for epoch operations, not in any swap submission path.

### 3. Transaction submission is standard RPC — no MEV protection layer

`app/hooks/useProtocolWallet.ts:111` — All user transactions go through:

```
signTransaction()  →  connection.sendRawTransaction()  →  /api/rpc proxy  →  Helius RPC  →  Solana validators
```

No MEV-shielded endpoint (Helius staked connections, Jito block engine) is used at any layer. The sign-then-send pattern also explicitly bypasses wallet-level MEV protection (Phantom's `signAndSendTransaction` endpoint, which sends through Phantom's own RPC).

### 4. minimumOutput is computed from 5% slippage default and passed on-chain

`app/hooks/useSwap.ts:418` (buy path) and `app/hooks/useSwap.ts:447` (sell path):

```typescript
const minimumOutput = Math.floor(outputTokens * (10_000 - slippageBps) / 10_000);
```

With `slippageBps=500`, this sets `minimumOutput = floor(output * 0.95)`, giving a 5% extraction window between the quoted price and the on-chain enforcement floor.

### 5. SlippageConfig presets are lower than the default

`app/components/swap/SlippageConfig.tsx:43-47` offers preset buttons at 0.5% (50 BPS), 1% (100 BPS), and 2% (200 BPS). The default loaded from `SettingsProvider` is 500 BPS. Users who never open slippage settings execute at 5% even though the highest preset is 2%. This mismatch was noted in Audit #1 and remains unfixed.

### 6. On-chain tax structure provides a structural MEV defense (KEY MITIGATING FACTOR)

`Docs/tax-mev-defense.md` (added Phase 90, documented at VERIFY-H015 round 3) formalises the defence:

- Every SOL pool swap attracts a buy or sell tax of 3–14% per leg, set per epoch via Switchboard VRF.
- A sandwich attack requires a front-run buy AND a back-run sell. Each leg is taxed.
- Minimum round-trip tax cost: 2 × 3% = **6% of bot position size**.
- Maximum typical round-trip tax cost: 3% + 14% = **17% of bot position size**.
- Maximum sandwich profit against a user with 5% slippage: **~4.9% of user's trade size** (not bot position, which is typically larger).

For a 50 SOL bot position attacking a 10 SOL user trade:
- Bot tax cost: ≥ 3 SOL (minimum epoch), typically 10.5 SOL (typical epoch)
- Bot maximum profit from slippage: ~0.25 SOL (5% of 5 SOL median slippage)
- **Net: deeply unprofitable under every epoch configuration**

This is on-chain and unavoidable — the Tax Program wraps every AMM swap atomically. A bot cannot route around it.

### 7. On-chain 50% output floor provides hard backstop independent of slippage setting

Per VERIFY-H015 and ARCHITECTURE.md INV-5: `MINIMUM_OUTPUT_FLOOR_BPS = 5000` in `programs/tax-program/src/constants.rs:40`. Even if a user sets slippage to 100%, the on-chain program rejects any swap where output falls below 50% of expected. This is the absolute floor, independent of the client-supplied `minimumOutput`.

### 8. Remaining live exposure: high slippage hurts users, not just bots

The tax defense makes sandwich attacks economically irrational for SOL-pool swaps. However, the 5% default slippage still creates user-adverse outcomes in a different scenario: **legitimate price movement during the quote-to-execution window**. If pool reserves shift 4% between when the quote is computed (debounced 300ms) and when the TX lands (5–15s later due to signing + RPC round-trip), the user's output degrades silently to the 95% floor without any on-chain rejection. This is not MEV — it is ordinary market risk being absorbed by the user because the tolerance is set too wide.

## Attack Path

### Scenario A: Classic Sandwich (BLOCKED by tax structure)

```
1. MEV bot monitors standard Helius RPC mempool for Dr Fraudsworth program ID TXs
2. Bot decodes instruction: inputAmount, minimumOutput=95% of expected, pool accounts
3. Bot front-runs with a buy at higher priority fee (e.g., turbo = 1M microLamports)
4. Bot's buy TX lands → pool price moves up
5. User's TX lands → user receives between 95% and 100% of expected output (within slippage)
6. Bot back-runs with a sell TX

--- BLOCKED by tax structure ---
Bot pays buy tax (3-14%) on front-run + sell tax (3-14%) on back-run.
Minimum cost: 6% of bot position >> maximum profit: ~5% of user trade amount.
Attack is unprofitable under all epoch configurations.
```

### Scenario B: Price-movement degradation (LIVE — not an attack, but user harm)

```
1. User enters 10 SOL swap, receives quote for 950 CRIME at current reserves
2. 300ms debounce → quote computed; user spends 5s reviewing, 3s on wallet popup
3. In that 8+ second window, organic trades move pool reserves by 4%
4. User's TX lands → on-chain execution yields 912 CRIME (≥ 950 * 0.95 = 902.5 floor)
5. TX succeeds; user received 912 CRIME vs 950 quoted — a 4% degradation
6. No error, no rejection, no warning shown post-hoc
```

This is exacerbated by SSE staleness (reserves may be 5–60s stale before the user even starts the quote flow, per ASM-8 in ARCHITECTURE.md).

### Scenario C: localStorage slippage injection (CONDITIONAL — requires extension/XSS)

```
1. Malicious browser extension or XSS sets localStorage key:
   dr-fraudsworth-settings = {"slippageBps": 10000, ...}
2. On next page load, SettingsProvider.loadSettings() reads slippageBps=10000 (valid, ≤10_000)
3. minimumOutput = 0 for all subsequent swaps
4. On-chain 50% floor is the only protection (still active)
5. All user swap outputs can be degraded to 50% of expected without rejection
```

This requires a compromised browser environment. The on-chain 50% floor still applies.

---

## Impact

**Sandwich attack (Scenario A):** Blocked by on-chain tax structure. Real-world impact is effectively zero for SOL-pool swaps — the economics make it unprofitable for any rational bot. This is a genuine protocol-level defence and should be credited as such.

**Price-movement degradation (Scenario B):** Medium. The 5% tolerance is 2.5–10× wider than industry-standard defaults (Jupiter: 0.5%, Raydium: 0.5–1%). Users silently absorb up to 4.9% of additional price movement that would have caused a failed TX under a tighter default, allowing them to retry. The on-chain enforcement works correctly — the issue is user experience and value leakage, not a security exploit.

**LocalStorage injection (Scenario C):** Low-conditional. Requires external compromise (extension, XSS). On-chain 50% floor limits worst-case to 50% degradation.

**Quantified exposure (mainnet scale):**
- Scenario A: ~$0 (tax structure renders attack irrational)
- Scenario B: Up to 4.9% of swap value in additional price slippage per trade vs industry-standard 0.5% default
- Scenario C: Up to 50% degradation per swap (on-chain floor; requires prior browser compromise)

## Recommendation

### Fix 1 — Reduce default slippage to 100 BPS (1%) [IMMEDIATE, one-line change]

`app/providers/SettingsProvider.tsx:170`:

```typescript
// Current
slippageBps: 500,

// Recommended
slippageBps: 100,   // 1% — matches UI preset, aligns with SwapForm.tsx:455 intent
```

This reduces Scenario B exposure by 5× with zero on-chain change. The on-chain 50% floor is unaffected. Users with existing localStorage keep their saved preference; only first-visit defaults change.

### Fix 2 — Mirror the 50% UI cap in SettingsProvider validation [IMMEDIATE, one-line change]

`app/providers/SettingsProvider.tsx:197`:

```typescript
// Current
parsed.slippageBps <= 10_000

// Recommended
parsed.slippageBps <= 5_000   // cap to 50%, matching SlippageConfig.tsx UI cap
```

This closes Scenario C's localStorage injection path. The on-chain 50% floor provides defence-in-depth, but defence-in-depth means both layers should enforce the same bound.

### Fix 3 — Revisit sign-then-send for mainnet [PRE-MAINNET]

The sign-then-send pattern was adopted to work around a devnet-specific Phantom RPC bug (MEMORY.md). On mainnet, evaluate switching back to `wallet.sendTransaction()` (the wallet-adapter default) to allow Phantom's built-in MEV protection features to activate. This is a wallet-layer improvement; the on-chain tax defence operates regardless.
