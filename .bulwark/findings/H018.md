# H018: No MEV-Protected Submission for Any User Swap

## Verdict: CONFIRMED

---

## Evidence

### 1. Transaction Submission Pipeline — No MEV Protection at Any Layer

`app/hooks/useProtocolWallet.ts:111` is the single choke point through which all user-facing swap transactions flow. The code is unambiguous:

```ts
// 3. Submit via OUR connection (Helius RPC) -- we control the endpoint
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: sendOptions.skipPreflight,
  ...
});
```

The comment itself explains *why* this pattern was chosen (devnet Phantom RPC dropping TXs), but the consequence is that every signed transaction is submitted directly to the standard Helius JSON-RPC endpoint — a fully public mempool.

### 2. Single-Hop Swaps (`useSwap.ts:763-766`)

```ts
const signature = await wallet.sendTransaction(tx, connection, {
  skipPreflight: false,
  maxRetries: 2,
});
```

Calls `wrappedSendTransaction` → `connection.sendRawTransaction`. No Jito tip account instruction prepended. No private-transaction relay.

### 3. Multi-Hop Swaps (`multi-hop-builder.ts:378-381`)

```ts
signature = await wallet.sendTransaction(
  build.transaction,
  connection,
  { skipPreflight: true, maxRetries: 3 },
);
```

Same path. `skipPreflight: true` is documented as a devnet workaround for v0 TX simulation rejections (`MEMORY.md`) — it is not a Jito skipPreflight flag.

### 4. Connection Layer (`app/lib/connection.ts`)

All browser RPC resolves to `/api/rpc` (the Helius proxy). The server-side connection uses `HELIUS_RPC_URL` directly. Neither routing path is a MEV-protected RPC (Jito block engine, Nozomi, bloXroute, etc.).

### 5. Full Codebase Search — Zero MEV Infrastructure

A project-wide grep for `jito`, `nozomi`, `private_transaction`, `private_mempool`, `privateMempool`, and MEV-protected RPC patterns across all `.ts` and `.tsx` files returned **zero matches** outside of:

- **IDL comment** (`epoch_program.ts:469`): `"Typically bundled in the same transaction for MEV protection."` — this refers to instruction-level atomic bundling (reveal + consume + executeCarnageAtomic in one v0 TX), not Jito bundle submission.
- **Crank comments** (`crank-runner.ts:16`, `vrf-flow.ts:106,284`): All describe the CARN-002 fix where three *instructions* are packed into one *transaction* to prevent front-running of Carnage execution. Completely unrelated to MEV-protected RPC or Jito.
- **Context doc** (`.bulwark/context/07-CHAIN-05-mev-ordering.md`): The auditor explicitly flags this as unfixed.

### 6. Sign-Then-Send Bypasses Phantom's Own MEV Protection

Phantom has introduced MEV-protected transaction submission through its own RPC. By using `signTransaction` + `sendRawTransaction` instead of `signAndSendTransaction`, the application explicitly bypasses any wallet-level MEV protection. This is documented in the code as an intentional devnet workaround but ships unchanged to mainnet.

### 7. Default Slippage Amplifies the Attack Surface

`app/providers/SettingsProvider.tsx:170` sets `slippageBps` default to `500` (5%). The CHAIN-05 auditor confirmed this is the effective default — most users never open settings. With 5% slippage, every swap exposes up to 5% of the swap value to sandwich extraction. For a 10 SOL swap, a sandwich bot can extract up to ~0.5 SOL while remaining on-chain valid.

---

## Attack Path

**Sandwich Attack on Every User Swap**

1. User initiates a swap (SOL → CRIME, SOL → FRAUD, or any multi-hop route).
2. App builds TX with `minimumOutput = floor(output × 0.95)` (5% default slippage).
3. TX is signed by user wallet, then submitted to Helius public RPC via `connection.sendRawTransaction`.
4. TX is immediately visible in the Solana public mempool.
5. MEV bot observes the pending TX, reads pool addresses and `minimumOutput` from the instruction data.
6. Bot calculates maximum extractable value: up to the full 5% slippage tolerance.
7. Bot submits a front-run TX (pushes price unfavorably for user) + back-run TX (restores price, capturing spread), both with higher priority fees to guarantee ordering around the user TX.
8. User TX executes at worst-case output (≥ `minimumOutput`). Bot collects the spread.

**Amplified variant — Split Route**

For SOL → PROFIT split routes (SOL → PROFIT via both CRIME and FRAUD pools simultaneously), two AMM swap instructions are embedded in one TX. A sophisticated bot can sandwich both legs independently, doubling the extraction surface versus a single-hop route.

**Passive intelligence advantage — SSE oracle**

The unauthenticated SSE endpoint (`/api/sse/protocol`) delivers pre-parsed pool reserve state faster than raw RPC polling. MEV bots can subscribe to SSE to receive optimally-timed quotes and track which swaps are likely pending in the mempool based on protocol state changes.

---

## Impact

**Severity: HIGH**

- **Systematic user value extraction**: Every swap is sandwichable up to the configured slippage tolerance. With a 5% default, this is a structural tax on all users proportional to swap size.
- **No user opt-out exists**: There is no Jito or private mempool option anywhere in the application. Users cannot protect themselves regardless of settings chosen.
- **Mainnet materiality**: On memecoin pairs with active trading, sandwich bots are highly incentivized. The protocol's own SSE feed provides bots with a structural timing advantage.
- **Partially mitigated by on-chain floor**: The on-chain 50% output floor (Tax Program) prevents catastrophic loss, but the 5%–50% range between the off-chain slippage guard and the on-chain floor represents fully extractable value that on-chain constraints cannot prevent.
- **Note**: This is a user-funds impact, not a protocol-funds impact. Protocol security (on-chain programs, admin authorities) is unaffected. The risk is borne entirely by swap users.

**Quantified example**: At 1000 SOL daily swap volume with average 5% slippage tolerance and 50% sandwich success rate, bots extract ~25 SOL/day (~$2,500-$5,000 at typical SOL prices) from users.

---

## Recommendation

### Immediate (Pre-Mainnet)

1. **Reduce default slippage**: Change `SettingsProvider.tsx:170` from `500` (5%) to `100` (1%) or `50` (0.5%). This directly shrinks the sandwich profit ceiling without any infrastructure changes. The CHAIN-05 auditor noted the UI presets already go as low as 0.5% — the default should match the lowest preset.

2. **Add high-price-impact warning gate**: If `priceImpact > 200 BPS` (2%), display a blocking confirmation dialog and refuse to submit without explicit user acknowledgment. Currently price impact is computed but never gates execution.

### Short-Term (Post-Launch)

3. **Jito bundle submission (optional tip)**: Add an optional Jito integration. When the user enables "MEV protection" in settings, prepend a `SystemProgram.transfer` tip instruction (e.g., 0.001 SOL) to the tip account and submit via the Jito block engine endpoint instead of Helius. This is a well-understood pattern with open-source reference implementations.

4. **Restore `signAndSendTransaction` on mainnet**: The sign-then-send pattern was adopted to fix a devnet-specific Phantom RPC issue. On mainnet, `signAndSendTransaction` would allow Phantom's own MEV protection (if enabled by the user) and restore the wallet simulation preview (Blowfish). Gate the pattern via `NEXT_PUBLIC_CLUSTER`.

### Design Note

The on-chain 50% output floor is a strong safety net against catastrophic loss but was designed for the Carnage sell path, not as a general MEV mitigation. The gap between user-configured slippage (5%) and the on-chain floor (50%) is the exploitable zone. Shrinking the off-chain slippage default is the highest-leverage single fix available.
