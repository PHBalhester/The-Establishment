# H022: Next.js GHSA-mq59 Server Action CSRF — Not Applicable to This Codebase

## Verdict: NOT_VULNERABLE

---

## Evidence

### 1. Exact Next.js version confirmed

`app/package.json:29`:
```json
"next": "16.1.6"
```

This version falls within the affected range (`>=16.0.1`, patched in `16.1.7`) for GHSA-mq59-m269-xvcx (CVE-2026-27978).

### 2. What GHSA-mq59 actually covers

The advisory is specifically and exclusively a **Server Actions CSRF bypass**:

> `origin: null` was treated as a "missing" origin during Server Action CSRF validation. As a result, requests from opaque contexts (such as sandboxed iframes) could bypass origin verification instead of being validated as cross-origin requests.

The mechanism: a victim is lured to a page containing a sandboxed iframe (or other opaque context). That iframe submits a Server Action POST to the target Next.js app. Because the browser sends `Origin: null` for sandboxed contexts, and Next.js 16.1.6 treated `null` as a missing/absent origin rather than an explicit cross-origin value, the CSRF check was silently skipped.

This is not a general API route CSRF bypass. It is narrowly scoped to the **Server Actions** mechanism — functions marked with the `"use server"` directive that Next.js wires up to a special `/_next/action` endpoint with its own CSRF validation layer.

### 3. Zero Server Actions in this codebase

Exhaustive grep of all application source directories (`app/app/`, `app/lib/`, `app/components/`, `app/db/`, `app/providers/`) for the `"use server"` directive returns **zero results**.

No `.ts` or `.tsx` source file in this project contains a Server Action declaration. The codebase uses only:
- Standard Next.js App Router **API Route Handlers** (`export async function POST(req: NextRequest)`)
- Client Components with wallet-adapter hooks

### 4. The webhook route is a plain API Route Handler, not a Server Action

`app/app/api/webhooks/helius/route.ts:255`:
```typescript
export async function POST(req: NextRequest): Promise<NextResponse> {
```

This is a standard Next.js Route Handler. It is completely separate from the Server Actions subsystem. Next.js's CSRF validation for Server Actions does not apply to Route Handlers — Route Handlers receive raw HTTP requests and are responsible for their own authentication, which this route implements via `HELIUS_WEBHOOK_SECRET` (lines 270–302).

### 5. Webhook auth is independent of any CSRF mechanism

The webhook endpoint authenticates exclusively via HMAC-style shared secret comparison:

```typescript
const secretBuf = Buffer.from(webhookSecret, "utf-8");
const headerBuf = Buffer.from(authHeader, "utf-8");
const lengthMatch = secretBuf.length === headerBuf.length;
const compareBuf = lengthMatch ? headerBuf : secretBuf;
if (!lengthMatch || !timingSafeEqual(secretBuf, compareBuf)) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

This is a bearer-token pattern (Authorization header). It operates entirely outside the Next.js CSRF machinery. A CSRF attack cannot inject a fabricated `Authorization` header from a cross-origin browser context — the browser's CORS policy prevents JavaScript from setting `Authorization` on cross-origin requests, and Helius itself is a server-to-server caller with no browser session.

### 6. No `serverActions.allowedOrigins` misconfiguration

`app/next.config.ts` contains no `serverActions` configuration block. The workaround vectors specified in the advisory (allowlisting `'null'` in `allowedOrigins`) are not present.

---

## Attack Path

The hypothesised attack chain does not exist:

```
Attacker creates sandboxed iframe
  → iframe submits POST to /api/webhooks/helius with Origin: null
  → Next.js CSRF check skipped (GHSA-mq59 bypass)
  → Webhook processes attacker-controlled payload
```

This chain fails at every step:

1. **GHSA-mq59 only guards Server Actions**, not Route Handlers. The webhook endpoint is a Route Handler. The Next.js CSRF validation layer for Server Actions is never invoked.

2. **Even if the CSRF bypass applied**, the attacker has no `HELIUS_WEBHOOK_SECRET`. Without a valid `Authorization` header containing the pre-shared secret, the request is rejected `401` before any payload processing.

3. **Even if both guards were absent**, the CORS restriction prevents a browser-originated cross-origin request from setting an `Authorization` header. A sandboxed iframe using `fetch()` with `Authorization` would be blocked by the browser's CORS preflight (no `Access-Control-Allow-Origin` is returned by this endpoint — the WEB-02 audit confirmed no CORS middleware exists, which is the secure default here).

---

## Impact

None. The vulnerability class (Server Action CSRF via null origin) has no attack surface in this application because the application does not use Server Actions.

---

## Recommendation

1. **Upgrade next to >=16.1.7** regardless. Even though this specific CVE is not exploitable here, the same release also patches GHSA-ggv3 (HTTP request smuggling via rewrites) and other moderate CVEs, all of which are present in 16.1.6. Upgrading is low-risk and eliminates multiple advisories at once. This is already captured in the broader DEP-01 finding.

2. **No Server Action CSRF mitigations are required** because Server Actions are not used. If Server Actions are introduced in future, review them in the context of this now-patched CSRF advisory.

3. **Do not introduce `"use server"` functions** for state-changing operations without evaluating the CSRF posture of the version of Next.js in use at that time.

---

## Evolution

**NEW → NOT_VULNERABLE**

The hypothesis was well-formed but required code-level verification to resolve. The CVE is real and the version is within the affected range, but the attack surface (Server Actions) does not exist in this codebase. The webhook endpoint uses bearer-token authentication that is structurally independent of Next.js CSRF machinery. Upgrade to next@16.1.7+ is still recommended via DEP-01 to close this and adjacent CVEs.

**Cross-reference**: DEP-01 (30 unpatched CVEs including next@16.1.6 GHSA-mq59, GHSA-ggv3) — the upgrade recommendation lives there.
