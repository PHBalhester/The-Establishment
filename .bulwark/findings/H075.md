---

## Hypothesis

The crank runner process lacks global error handlers for `unhandledRejection` and `uncaughtException`. This is the crank-specific instance of H050 (project-wide missing handlers). An unhandled async error in the VRF flow, Switchboard SDK internals, or circuit breaker logic silently crashes the crank process with no logging or recovery, halting epoch advancement and triggering the zombie-crank scenario.

---

## Evidence

### 1. Grep Results — No process.on("unhandledRejection") in crank-runner.ts

```
scripts/crank/crank-runner.ts:196:process.on("SIGINT", () => {
scripts/crank/crank-runner.ts:202:process.on("SIGTERM", () => {
```

**Missing:**
- `process.on("unhandledRejection", ...)`
- `process.on("uncaughtException", ...)`

Only graceful shutdown signal handlers are present (SIGINT/SIGTERM at lines 196 and 202). No error handlers exist.

### 2. Async Surfaces in crank-runner.ts

The crank has multiple async paths outside the main try/catch block (lines 394–552):

1. **VRF Flow (`advanceEpochWithVRF`)** — lines 477–481
   - Delegates to `scripts/vrf/lib/vrf-flow.ts` which uses Switchboard SDK
   - Switchboard internals may emit unhandled rejections if callbacks fail

2. **Provider async calls** — lines 371–411, 453–454
   - `readEpochState()` — line 371, 396
   - `provider.connection.getBalance()` — lines 403, 415
   - `provider.connection.getSlot()` — line 453
   - None of these are directly in the main try/catch if they fail during init (lines 310–374)

3. **waitForSlotAdvance** — line 471
   - Waits for slot advancement; if it throws outside the try/catch at an unexpected time, it crashes silently

4. **Health server** — lines 385, 199, 205
   - HTTP server started at line 385
   - If the server crashes or the listen callback throws, no recovery handler catches it

5. **Circuit Breaker Loop** — lines 390–553
   - The while loop itself is inside try/catch, but any synchronous error thrown during setup before the loop (lines 300–389) crashes with no handler
   - Example: `loadCrankProvider()` at line 310 can throw; if it does, there's no global handler

### 3. H050 Cross-Reference

H050 established project-wide absence of process-level error handlers. This finding confirms **the crank-specific instance**: while `app/instrumentation.ts` lacks handlers and `app/lib/ws-subscriber.ts` has only per-interval try/catch, the crank runner has the additional critical gap because it is a **long-lived process with no try/catch wrapping the initialization phase** and the main loop has async Switchboard SDK code that may throw outside the main catch block.

### 4. Silent Crash Scenario (H031 Link)

From H050 evidence section (line 74–79):
> "In Node.js 15+, unhandled rejections **terminate the process** by default. The Next.js server exits... the crank is a second distinct surface: a silent crash stops epoch advancement entirely, triggering the zombie-crank scenario documented in H031."

The crank runner **is running on Node.js 20+** (no explicit version pin, but Anchor/Solana stack targets LTS). Unhandled rejections **will terminate the process**.

---

## Attack / Failure Scenario

1. During normal epoch advancement cycle, `advanceEpochWithVRF()` internally calls Switchboard SDK functions (e.g., random number fetch, timeout handling).
2. A Switchboard network error or SDK bug causes a promise rejection inside the SDK.
3. The SDK callback has no try/catch wrapping the internal rejection.
4. Node.js emits `unhandledRejection` → process terminates.
5. The crank process exits silently. No error log, no stderr message with details.
6. Railway detects the crash ~10–30s later and restarts the container.
7. During the gap: no epoch is advanced, epoch boundary passes uncycled, the zombie-crank scenario begins.
8. No Sentry alert (process crash bypasses captureException), no structured log to the team — only Railway's opaque "restart detected" event.

---

## Severity Assessment

**Tier 3** — Availability impact on a critical background job (epoch advancement). Mitigated partially by:
- Railway auto-restart limits gap to ~10–30s
- The main try/catch around the epoch cycle (lines 394–552) catches most errors
- However: errors during init (lines 300–389) crash with no handler
- Switchboard SDK internal errors may not be caught by the try/catch if they occur in async callbacks

Elevated because:
- The crank is the **only** mechanism for epoch advancement on devnet and mainnet
- A crash that goes unnoticed means the team loses time responding to zombie-crank alerts downstream
- The fix is trivial (2–3 lines) and the cost of not fixing is recurring operational friction

---

## Recommended Fix

Add global error handlers at the top of the main function, immediately after configuration loading and before the main loop:

```typescript
// In main() function, after line 388, before the while loop:

// ---- Global Error Handlers ----
process.on("unhandledRejection", (reason) => {
  console.error(
    "[crank] FATAL unhandledRejection:",
    reason instanceof Error ? reason.message : String(reason)
  );
  // Optionally: send to Sentry here if available
  // For now, log to stderr so Railway captures it
  process.exit(1);
});

process.on("uncaughtException", (err) => {
  console.error("[crank] FATAL uncaughtException:", err.message);
  console.error(err.stack);
  process.exit(1);
});

console.log("[crank] Starting epoch advancement loop...");
console.log();

while (!shutdownRequested) {
  // ... existing loop
}
```

This ensures:
- Any unhandled rejection or exception terminates the process **with a clear log message**
- Railway sees the error in logs before restarting
- The team can monitor for these crashes in Railway's log UI
- Future unhandled async errors are visible, not silent
