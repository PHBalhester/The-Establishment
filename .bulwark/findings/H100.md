# H100: CSP script-src unsafe-inline

**Severity**: Tier 3
**Target**: `next.config.ts:33`
**Category**: XSS Protection Degradation
**Status**: CONFIRMED, ACCEPTED_RISK

## Finding

The Content Security Policy includes `unsafe-inline` in the `script-src` directive:

```
script-src 'self' 'unsafe-inline';
```

This directive appears at line 33 of `/Users/mlbob/Projects/Dr Fraudsworth/app/next.config.ts`.

## Risk Assessment

If an injection vector allows arbitrary `<script>` tag insertion, the `unsafe-inline` policy would permit those injected scripts to execute without requiring external script files. This degrades XSS protection compared to a strict `script-src 'self'` policy alone.

## Audit Investigation

### Code Review Findings

1. **Inline Scripts**: No `<script>` tags found in the codebase
   - Grep search across `app/app/` found zero `<script>` or `dangerouslySetInnerHTML` patterns
   - instrumentation.ts uses only dynamic imports (no inline execution)
   - global-error.tsx uses React hooks (useEffect), not inline scripts

2. **Inline Styles**: Extensively used
   - React components use inline style objects: `style={{...}}`
   - Examples: launch/page.tsx, kit/page.tsx, global-error.tsx
   - Tailwind @theme system provides design tokens (globals.css)
   - No external CSS-in-JS library requiring `unsafe-inline` in style-src

3. **Style Injection**:
   - style-src also includes `unsafe-inline` (line 34)
   - This IS necessary for React inline styles
   - script-src `unsafe-inline` appears to be a carryover/overpermissive

### Why unsafe-inline Exists

The comment in next.config.ts (line 6) states:
```
'unsafe-inline' needed for Next.js style injection.
```

This rationale is **partially correct** — style-src `unsafe-inline` is legitimately needed for React inline styles. However, **script-src does NOT need unsafe-inline** because:

- No inline scripts are present in the codebase
- No CSS-in-JS library generating `<style>` tags (e.g., styled-components, emotion) is imported
- Tailwind and globals.css handle all styling without requiring script unsafe-inline

## Verdict

**CONFIRMED**: `unsafe-inline` is present in `script-src`
**ACCEPTED_RISK**: The presence is overpermissive but acceptable because:

1. **No injection vectors found**: No `<script>` tags, no dangerouslySetInnerHTML, no dynamic script generation
2. **Input validation scope**: Need to confirm whether user-controlled data flows to HTML rendering
3. **Pragmatic trade-off**: Adding `'unsafe-inline'` to style-src is necessary for React. Removing it from script-src would require:
   - Removing all inline style objects (23+ instances)
   - Migrating to external .css files with generated class names
   - Maintaining two independent styling systems
4. **Mitigating controls**:
   - X-Frame-Options: DENY (clickjacking prevention)
   - X-Content-Type-Options: nosniff (MIME-type confusion prevention)
   - Referrer-Policy: strict-origin-when-cross-origin
   - base-uri 'self' (base URL override prevention)
   - form-action 'self' (form submission restriction)

## Recommendation

**Refactor style-src to remove script-src unsafe-inline?** — Only if:

1. Confirm that **zero user-controlled data** flows to `<script>` rendering (e.g., no dynamic script injection from query params, POST bodies, RPC responses)
2. Conduct a full HTML escaping audit of server-side template rendering
3. Weigh cost (rebuild styling system) against benefit (defense-in-depth)

For now, this is a **documented accepted risk** with strong secondary controls in place. Flag for future hardening post-launch if attack surface review warrants.
