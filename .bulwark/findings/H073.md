# H073: Candle Close Price Ordering (Last-Write-Wins, Not Chronological)

**Severity:** Tier 3 | **Status:** VERIFIED (ACCEPTED_RISK)
**Target:** `app/db/candle-aggregator.ts:101-130`, schema: `app/db/schema.ts:59-82`
**Investigator:** Dinh's Bulwark (Lightweight)
**Date:** 2026-03-21

---

## Finding

The candle aggregator uses a **last-write-wins** model for the `close` price field. When concurrent webhook deliveries race to update the same candle (pool, resolution, openTime), the final `close` price reflects whichever trade commits last to the database, **not** the chronologically latest trade.

### Root Cause

In `candle-aggregator.ts` at line 123, the `close` price is unconditionally overwritten on every upsert:

```typescript
set: {
  high: sql`GREATEST(${candles.high}, ${update.price})`,
  low: sql`LEAST(${candles.low}, ${update.price})`,
  close: sql`${update.price}`,  // ← Always overwrites to latest input
  volume: sql`${candles.volume} + ${update.volume}`,
  tradeCount: sql`${candles.tradeCount} + 1`,
}
```

The database schema (`schema.ts:69`) stores only the `close` price as a single `real` column with no associated timestamp:

```typescript
close: real("close").notNull(),
```

There is no `closeTime` or `closeTimestamp` column to track when the close price was set, and no trigger or comparison logic to update only if the incoming trade is chronologically later.

### Vulnerability Conditions

1. **Multiple swaps in the same candle period** (e.g., two trades within a 1-minute bucket)
2. **Concurrent webhook deliveries** from Helius RPC (especially during high activity)
3. **Network jitter or TX ordering variance** causing out-of-order webhook arrival
4. **Result:** The displayed close price on the frontend may be an earlier trade's price, not the final trade of the period

Example:
- Trade A: 14:35:10, price 50.0 SOL/token
- Trade B: 14:35:55, price 52.0 SOL/token
- Webhook for B arrives first → close = 52.0
- Webhook for A arrives second → close = 50.0 (incorrect)

### Current Safeguards

✓ **Swap event deduplication:** `/api/webhooks/helius` uses `onConflictDoNothing` on the `swap_events` table with unique key `(txSignature, swapIndex)` (line 680 in `route.ts`), preventing double-counting of the same swap. This means a given trade will only upsert candles once.

✓ **High/Low tracking is robust:** Uses SQL `GREATEST` and `LEAST` functions, immune to ordering.

✓ **Volume and trade count are additive:** Both accumulate correctly regardless of arrival order.

✗ **Close price lacks timestamp guard:** No mechanism to prefer the chronologically later trade.

## Impact Assessment

### UX Consequences
- Chart candles may display incorrect closing prices for volatile periods
- Historical candle data (via `/api/candles` endpoint) is forever corrupted for periods with concurrent trades
- Frontend consumers (ChartComponent, RouteEngine trading logic) cannot distinguish stale close prices

### Trading Logic Risk
- **Quote engine:** Uses pool reserves (from SSE/webhook), not candle data → **NOT AFFECTED** (minimumOutput backstop)
- **Route display:** Crank/UI may reference candle close prices for visual indication → Minor (UX only)
- **Arbitrage detection:** Any off-chain logic relying on candle close times will be wrong

### Data Pipeline Risk
- Candle DB is permanent; no correction mechanism
- Reprocessing swap events will re-corrupt historical candles (same last-write-wins logic)

### Likelihood
- **High** if trading volume is sustained (multiple swaps/candle period is common)
- **Medium** if webhook delivery is asynchronous (standard for event-driven systems)

## Verdict

**CONFIRMED VULNERABLE** — The behavior is exactly as stated: last-write-wins, no chronological ordering of close prices.

**ASSESSMENT: ACCEPTED_RISK**

### Reasoning

1. **Swap events are deduplicated** before upsert, so a given trade cannot double-count. The race is only between different trades in the same candle period.

2. **On-chain enforcement is the security boundary:** The protocol's fundamental safety (slippage, MEV, fund protection) depends on on-chain constraints (minimumOutput, PDA checks, arithmetic), not off-chain candle accuracy.

3. **Candle data is display-only:** Historical charts, visual price tracking, and analytics use candles. The quote engine reads real-time pool reserves from the SSE-fed protocolStore, not from candle_close.

4. **Frontend UI is untrusted:** Per the architecture, on-chain code is the enforcement point. Stale or misordered candles are a UX bug, not a security bug.

5. **Historical correction is infeasible:** Candles cannot be recalculated retroactively without access to tick-by-tick trade history (not stored). Retroactive fixes would require epoch-level snapshots or reprocessing the entire blockchain state.

6. **Priority tier:** This is a Tier 3 (MEDIUM) issue alongside other rate-limit and resource exhaustion concerns. No Tier 1 (CRITICAL) or Tier 2 (HIGH) impact.

## Remediation (Optional, Post-Launch)

If strict historical accuracy is desired later, options include:

| Option | Effort | Impact |
|--------|--------|--------|
| **Add `closeTimestamp` column** | Low | Schema migration required; backfill strategy TBD |
| **Update only if incoming trade is later** | Medium | Requires `closeTimestamp` + SQL CASE logic |
| **Accept last-write-wins semantics** | None | Document as expected behavior in candle spec |
| **Use tick-database (ClickHouse)** | High | Separate fast OLAP engine for granular tick history |

### Recommended Approach (Phase 104+)
Add an optional `closeTimestamp` column. During upsert, use:
```sql
close = CASE
  WHEN closeTimestamp IS NULL OR ${swapTimestamp} > closeTimestamp
  THEN ${newPrice}
  ELSE close
END,
closeTimestamp = CASE
  WHEN closeTimestamp IS NULL OR ${swapTimestamp} > closeTimestamp
  THEN ${swapTimestamp}
  ELSE closeTimestamp
END
```
This preserves historical behavior (NULL timestamps treated as earliest) while enabling future strictness.
