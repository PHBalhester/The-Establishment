## Hypothesis

Check-then-set gap on `state.initialized` allows concurrent callers to both pass
the guard before either sets the flag, creating duplicate WS subscriptions and
poll timers. This doubles RPC credit consumption and generates duplicate SSE
broadcasts.

## Code Under Review

```typescript
// app/lib/ws-subscriber.ts:456-474
export async function init(): Promise<void> {
  if (process.env.WS_SUBSCRIBER_ENABLED !== "true") {
    return;
  }

  if (state.initialized) {           // ← CHECK (line 456)
    console.log("[ws-subscriber] Already initialized, skipping");
    return;
  }

  console.log("[ws-subscriber] Initializing...");
  const connection = getConnection();

  await batchSeed(connection);       // ← async gap (line 466)

  startSlotSubscription(connection); // 4 subscriptions/timers started
  startSupplyPoll(connection);
  startStakerPoll(connection);
  startStalenessMonitor(connection);

  state.initialized = true;          // ← SET (line 474)
}
```

The guard is a non-atomic check-then-set separated by an `await` at line 466.
Any concurrent caller that enters `init()` before line 474 resolves will pass
the check at line 456 and proceed to start a second full set of subscriptions
and timers.

## Verification

### 1. Guard is non-atomic

Confirmed. The pattern is:

```
if (state.initialized) return;    // CHECK — line 456
...
await batchSeed(connection);       // async suspension point
...
startSlotSubscription(connection); // side effects
startSupplyPoll(connection);
startStakerPoll(connection);
startStalenessMonitor(connection);
state.initialized = true;          // SET — line 474
```

`batchSeed()` calls `connection.getMultipleAccountsInfo()`, a network round-trip
of roughly 50–500ms on Helius devnet/mainnet. During that window the event loop
is free to execute other microtasks and macrotasks. Any call to `init()` that
arrives during this window sees `state.initialized === false` and proceeds.

### 2. Can instrumentation.ts call init() twice?

**Yes — confirmed in production via the Turbopack context-isolation bug.**

`instrumentation.ts` uses the Prisma-style HMR singleton pattern:

```typescript
// app/lib/ws-subscriber.ts:94-107
const state: WsSubscriberState = globalForWsSub.wsSubscriber ?? {
  initialized: false,
  ...
};
globalForWsSub.wsSubscriber = state;
```

In development this is safe — HMR re-runs the module but globalThis retains the
instance. However, as documented in `.planning/debug/ws-subscriber-context-isolation.md`,
**all 5 singleton modules contain a production guard that skips the globalThis
assignment**:

```typescript
// The Prisma-style pattern present in ws-subscriber and peer singletons
if (process.env.NODE_ENV !== "production") {
  globalForWsSub.wsSubscriber = state;
}
```

Under Turbopack (Next.js 16), the instrumentation entry point and API route
entry points compile into separate chunk graphs. Turbopack can assign different
numeric module IDs to the same source file across those graphs. When module IDs
diverge, Node.js evaluates the module factory twice, producing two independent
`state` objects — each starting with `initialized: false`.

Concrete crash path:

1. Railway starts the server. Turbopack loads the instrumentation chunk graph.
   `ws-subscriber.ts` evaluates → `state_A = { initialized: false }`.
   `globalForWsSub.wsSubscriber` is NOT assigned (production guard skips it).
2. `register()` in `instrumentation.ts` calls `await import("@/lib/ws-subscriber")`.
   `init()` runs: reads `state_A.initialized === false` (passes guard), begins
   `await batchSeed()`.
3. An API route (e.g. `/api/health`) is hit, loading the route chunk graph.
   `ws-subscriber.ts` evaluates AGAIN → `state_B = { initialized: false }`.
   `globalForWsSub.wsSubscriber` is still NOT assigned. `state_B` is invisible
   to `state_A`.
4. `getStatus()` called from the health route reads `state_B.initialized` which
   is `false` — confirming the two instances diverge.
5. If `register()` is called a second time (Next.js hot-restart on Railway crash
   loop, or a deploy without process replacement), a new `state_A` is created
   from the `?? { initialized: false }` default, and `init()` runs again on the
   same process, starting a second set of timers alongside the first.

### 3. Does module-level state prevent double-init?

No. The module-level `state` object is the attempted guard, but as shown above:

- In production the `globalForWsSub.wsSubscriber = state` assignment is gated
  behind `NODE_ENV !== "production"`, which evaluates to `false` on Railway.
- Without globalThis persistence, each module evaluation produces a fresh
  `{ initialized: false }` state, making the guard structurally ineffective
  across process restart / chunk-graph divergence.
- Even within a single chunk-graph instance, the TOCTOU gap during `batchSeed()`
  would allow two concurrent `init()` calls from different async callers to both
  pass the check before either reaches the set.

The current single call site (`instrumentation.ts`) makes concurrent invocation
unlikely in steady state, but it is not enforced — nothing prevents a future API
route or test helper from calling `init()` directly.

## Impact

**Doubled RPC credit consumption (operational cost)**

Each duplicate `init()` starts:
- 1× `onSlotChange` WS subscription (~constant; Helius does not charge per event)
- 1× `setInterval` supply poll (60s cadence, `getTokenSupply` × 2 mints = 2 calls/min doubled to 4)
- 1× `setInterval` staker poll (30s cadence, `getProgramAccounts` doubled)
- 1× `setInterval` staleness monitor (5s HTTP slot poll when fallback active)

Under Helius's credit model, `getProgramAccounts` costs 100 credits. A doubled
staker poll timer costs an extra 200 credits/min (~288,000 credits/day).

**Duplicate SSE broadcasts**

Both timer instances write to `protocolStore`. The dedup check in
`protocol-store.ts:58` is also subject to a serialization-then-compare race (see
H046), so duplicate writes can produce duplicate SSE events broadcast to all
connected clients.

**No financial loss**

Double-init does not affect on-chain state. No private keys, PDAs, or transaction
logic is reachable from this code path. The on-chain enforcement layer (slippage,
balance checks) is unaffected.

## Severity Assessment

- **Confidentiality**: None
- **Integrity**: Low (duplicate/stale SSE events to clients; display-only)
- **Availability**: Medium (doubled RPC credits; prolonged crash-loop could
  accumulate N×timers, eventually causing Helius rate limiting or process OOM
  from uncollected interval handles)
- **Exploitability**: Low externally. The crash-loop scenario requires either a
  repeated server restart or an attacker triggering repeated 5xx to induce
  Railway auto-restarts. No external API surface directly calls `init()`.

**Tier 2 (Medium)** — Confirmed operational impact, no financial loss.

## Recommended Fix

**Fix 1: Atomic initialization promise (addresses the TOCTOU gap)**

Replace the boolean flag with a promise that is set synchronously before the
first `await`. Any concurrent caller awaits the same promise rather than
re-entering the function body.

```typescript
let initPromise: Promise<void> | null = null;

export function init(): Promise<void> {
  if (!initPromise) {
    initPromise = _doInit();
  }
  return initPromise;
}

async function _doInit(): Promise<void> {
  if (process.env.WS_SUBSCRIBER_ENABLED !== "true") return;
  const connection = getConnection();
  await batchSeed(connection);
  startSlotSubscription(connection);
  startSupplyPoll(connection);
  startStakerPoll(connection);
  startStalenessMonitor(connection);
  state.initialized = true;
}
```

`initPromise` is assigned synchronously before the first `await`, so concurrent
callers racing through the same module instance will always find a non-null
promise and skip `_doInit`.

**Fix 2: Remove the production guard from globalThis assignment (addresses chunk-graph divergence)**

In `ws-subscriber.ts` (and all peer singleton modules), change:

```typescript
// BEFORE — skips globalThis in production
if (process.env.NODE_ENV !== "production") {
  globalForWsSub.wsSubscriber = state;
}
```

to:

```typescript
// AFTER — always populate globalThis
globalForWsSub.wsSubscriber = state;
```

This ensures that even when Turbopack evaluates the module twice across chunk
graphs, the second evaluation picks up the live `state` object from the first via
`globalForWsSub.wsSubscriber ?? { initialized: false }`, and the initialized flag
is visible to all consumers.

This fix is documented as the root-cause resolution in
`.planning/debug/ws-subscriber-context-isolation.md` and is confirmed necessary
for the DBS (data broadcast system) to function correctly on Railway.

Both fixes should be applied together: Fix 2 prevents phantom re-initialization
from chunk-graph divergence; Fix 1 hardens the in-process init path against any
future concurrent callers.
