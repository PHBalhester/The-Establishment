## Summary

H081 targets mixed-denomination fee percentage calculations in the legacy (non-smart-routing) sell path of `useSwap.ts`. H038's investigation confirmed this defect exists but was deemed display-only after the Phase 52.1 BPS fix was applied to the route engine and RouteCard. This investigation confirms whether H038's recommendation has been implemented.

## Current State

**CONFIRMED UNFIXED.** Lines 448-449 and 521-522 still use mixed-denomination arithmetic:

### swapSolSell forward quote (lines 448-449):
```typescript
const totalFeePct = grossSolOutput > 0
  ? ((taxAmount / grossSolOutput) * 100 + (lpFee / baseUnits) * 100).toFixed(1) + "%"
  : "0%";
```

- `taxAmount / grossSolOutput`: SOL-denominated fee (tax relative to gross SOL output)
- `lpFee / baseUnits`: Token-denominated fee (LP fee relative to token input)
- **The sum has no meaningful interpretation** — it adds percentages computed over different-denomination totals

### swapSolSell reverse quote (lines 521-522):
```typescript
const totalFeePct = desiredOutput > 0
  ? ((taxAmount / (desiredOutput + taxAmount)) * 100 + (lpFee / inputTokensNeeded) * 100).toFixed(1) + "%"
  : "0%";
```

- First term: SOL tax as percentage of gross SOL output
- Second term: token LP fee as percentage of token input needed
- **Same denomination-mixing defect**

## Why This Matters

**Display-only impact.** The `totalFeePct` shown in `FeeBreakdown.tsx` inflates or deflates the displayed total fee on sell swaps when smart routing is disabled. However:

1. **No on-chain effect**: `minimumOutput` slippage guard is computed independently via `Math.floor(outputSol * (10_000 - slippageBps) / 10_000)` using BigInt-derived values
2. **Smart routing (default)** bypasses these paths — the `route-engine.ts` correctly uses BPS summation (`steps.reduce((sum, s) => sum + s.lpFeeBps + s.taxBps, 0)`)
3. **Zero-denominator safety** is present — both lines guard against zero denominators, preventing NaN or division-by-zero errors

## Recommendation (from H038)

Apply BPS-based fee percentage computation to both affected lines, matching the route-engine pattern:

```typescript
// swapSolSell forward quote — replace lines 448-450:
const totalFeeBps = sellTaxBps + SOL_POOL_FEE_BPS;
const totalFeePct = `${(totalFeeBps / 100).toFixed(1)}%`;

// swapSolSell reverse quote — replace lines 521-523:
const totalFeeBps = sellTaxBps + SOL_POOL_FEE_BPS;
const totalFeePct = `${(totalFeeBps / 100).toFixed(1)}%`;
```

Both `sellTaxBps` and `SOL_POOL_FEE_BPS` are already available in scope (line 439 and line 54 respectively).

## Impact of Leaving Unfixed

- **Severity**: Low (display artifact only, execution unaffected)
- **User experience**: Confusing fee percentage on sell quotes when smart routing is disabled
- **Mainnet risk**: Negligible (smart routing is default and correct)

## Verdict

**CONFIRMED** — defect exists, H038 recommendation not yet applied. Display-only, safe to defer if smart routing remains default, but should be fixed for code consistency and disabling smart routing edge case.
