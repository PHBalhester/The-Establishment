# H115: Webhook Response Leaks Processing Counts

## Vulnerability Summary

The Helius webhook handler returns a 200 success response that includes processing statistics: transaction count, swap count, epoch count, and carnage count.

```typescript
return NextResponse.json({
  ok: true,
  processed: {
    transactions: transactions.length,
    swaps: swapCount,
    epochs: epochCount,
    carnages: carnageCount,
  },
});
```

## Risk Assessment

**Severity: LOW**

### Information Value

The counts reveal:
1. **Activity volume per request**: How many transactions were processed in a single webhook batch
2. **Event type distribution**: Ratio of swaps to epochs to carnages
3. **Temporal patterns**: By observing counts over time, an attacker can infer protocol activity cycles

### Threat Model

An attacker with knowledge of the webhook secret (required for auth at lines 286-301) can:
- Monitor the webhook response counts over time
- Build a profile of trading activity patterns
- Detect trading peaks/valleys
- Estimate protocol health based on event volume
- Potentially time attacks during low-activity windows

**Impact**: Reconnaissance data for timing-based attacks (e.g., sandwich attacks during low-liquidity periods).

### Severity Justification

- **Data is not sensitive**: Counts are summaries of on-chain events, not private key material or credentials
- **Attacker already has webhook secret**: By definition of being able to call the webhook (auth gate at lines 286-301)
- **Data is derivable**: An attacker can already observe the same swaps/epochs/carnages by reading the blockchain directly via RPC
- **Low operational impact**: These counts are for debugging only and don't affect protocol functionality

The risk is **reconnaissance**, not **confidentiality breach** or **system compromise**.

## Detection Methodology

1. Read webhook response structure at lines 490-498
2. Identify what information is exposed: `transactions.length`, `swapCount`, `epochCount`, `carnageCount`
3. Assess whether each count is: (a) sensitive, (b) derived from public data, (c) useful for attacks
4. Check if response auth gate is in place (lines 286-301: `timingSafeEqual` check with `HELIUS_WEBHOOK_SECRET`)

## Code Context

**File**: `app/app/api/webhooks/helius/route.ts`

**Lines 286-301 (Auth Gate)**:
```typescript
// Verify webhook secret
if (!HELIUS_WEBHOOK_SECRET) {
  if (process.env.NODE_ENV === "production") {
    return NextResponse.json({ error: "Unauthorized" }, { status: 500 });
  }
} else {
  const providedSecret = req.headers.get("Authorization") ?? "";
  if (!timingSafeEqual(providedSecret, HELIUS_WEBHOOK_SECRET)) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 403 });
  }
}
```

**Lines 490-498 (Leaking Response)**:
```typescript
return NextResponse.json({
  ok: true,
  processed: {
    transactions: transactions.length,
    swaps: swapCount,
    epochs: epochCount,
    carnages: carnageCount,
  },
});
```

## Comparison to Similar Data

**CoinGecko price endpoint** (`app/app/api/sol-price/route.ts`):
- Returns current SOL price: **public data**, acceptable to expose
- No authentication required

**SSE protocol stream** (`app/app/api/sse/protocol/route.ts`):
- Broadcasts pool reserves, epoch state, staker stats: **on-chain public data**, acceptable (H007)
- No authentication required, but data is on-chain anyway

**Health endpoint** (`app/app/api/health/route.ts`):
- Returns infrastructure status (WS connection, RPC method counts): **operational data**, H028 (MEDIUM risk)
- No authentication required, higher risk than H115 because it reveals server-side state

**Webhook response counts** (H115):
- Returns event counts from authenticated requests: **derived from on-chain data**, lower risk than health endpoint
- Authentication required (webhook secret), further reduces risk

## Why This Is Accepted Risk

1. **Helius enforces auth**: Only callers with the webhook secret can observe these counts
2. **Counts are not secrets**: Swaps, epochs, and carnages are all on-chain events readable by anyone
3. **Reconnaissance, not compromise**: The counts enable timing-based attacks, not code execution or fund theft
4. **Operational utility**: The response helps Helius and the team verify webhook delivery success
5. **No sensitive data**: No PII, private keys, credentials, or strategy information in the counts

## Mitigation Recommendation

**NOT REQUIRED** for this release, but consider for future hardening:

1. **Reduce response verbosity in production**:
   ```typescript
   if (process.env.NODE_ENV === "production") {
     return NextResponse.json({ ok: true });
   } else {
     return NextResponse.json({ ok: true, processed: { ... } });
   }
   ```

2. **Or, randomize counts** (adds noise):
   ```typescript
   processed: {
     transactions: Math.max(1, transactions.length + Math.random() - 0.5),
     swaps: swapCount,
     epochs: epochCount,
     carnages: carnageCount,
   }
   ```

3. **Or, remove counts entirely** (simplest):
   ```typescript
   return NextResponse.json({ ok: true, deliveredAt: new Date().toISOString() });
   ```

- **→ SEC-01 (Access Control)**: Auth gate is in place (webhook secret). No elevation of privilege risk.
- **→ API-04 (Webhook Callback)**: This is a webhook response pattern question. Helius expects a 200 response. The counts are optional debugging info.

## Verdict

**CONFIRMED (LOW) — ACCEPTED_RISK**

The vulnerability is real but poses low practical risk. The counts are:
- Authenticated access only (webhook secret required)
- Derived from public on-chain data
- Useful for reconnaissance but not for immediate compromise
- Operationally useful for debugging webhook delivery

The cost of removing them (lose debugging visibility) exceeds the benefit (prevent minimal reconnaissance data). Acceptance is rational.
