# H082: Additive Price Impact (H072 Refinement)

**Severity**: Tier 3 (Minor Accuracy Issue)
**Status**: CONFIRMED
**Category**: Swap Routing / Price Calculation
**Target**: `app/lib/swap/route-engine.ts:351-354`

---

## Summary

Multi-hop swap routes accumulate price impact **additively** instead of **multiplicatively**. This under-reports true slippage to users and makes multi-hop routes appear worse than they actually are, potentially discouraging optimal routing choices.

---

## Technical Details

### The Vulnerability

In `route-engine.ts`, the `quoteRoute` function computes total price impact as a simple sum:

```typescript
// Line 348-354
const totalPriceImpactBps = steps.reduce(
  (sum, s) => sum + s.priceImpactBps,
  0,
);
```

For a multi-hop route (e.g., SOL → CRIME → PROFIT), if:
- Step 1 (SOL → CRIME) has 50 bps impact (0.5%)
- Step 2 (CRIME → PROFIT) has 50 bps impact (0.5%)

**Current (incorrect) calculation**:
- `totalPriceImpactBps = 50 + 50 = 100 bps (1.0%)`

**Correct (multiplicative) calculation**:
- Impact compounds: `(1 - 0.005) × (1 - 0.005) = 0.995 × 0.995 = 0.990025`
- Total impact: `1 - 0.990025 = 0.009975 ≈ 99.75 bps (0.9975%)`

The additive approach **overstates** impact by ~0.25 bps, making multi-hop routes appear less attractive.

### Root Cause

The code comments acknowledge that "impact compounds" (lines 349-350) but the implementation uses arithmetic addition rather than geometric multiplication. This is a logic-to-code mismatch.

### Impact Assessment

**User-facing effect**:
- Multi-hop routes display higher price impact than they actually have
- Users may select single-hop routes with worse actual outcomes over superior multi-hop routes
- Example: If route A (1-hop) shows 150 bps impact vs. route B (2-hop) shows 100 bps impact (additive), but route B's true impact is ~99.75 bps, the UI incorrectly suggests route A is better

**Severity justification** (Tier 3 not Tier 2):
- Impact is **cumulative but small** (at typical 40-60 bps per-step rates, 2-hop error ≈ 1-2 bps)
- Only affects **routing display logic**, not actual swap execution (slippage protection still works correctly)
- User funds are **not at direct risk** — slippage tolerance is applied to actual output amounts, not the impact BPS

## Proof of Concept

Test case for 2-hop route with equal impacts:

```typescript
// Mock 2-hop route: each step 100 bps impact
const steps = [
  { priceImpactBps: 100 },
  { priceImpactBps: 100 },
];

// Current (wrong): 100 + 100 = 200 bps
const additive = steps.reduce((sum, s) => sum + s.priceImpactBps, 0);
console.log("Additive impact:", additive); // 200 bps

// Correct: (1 - 0.01) * (1 - 0.01) - 1 = 0.0199 = 199 bps
const multiplicative = steps.reduce(
  (product, s) => Math.floor(product * (10_000 - s.priceImpactBps) / 10_000),
  10_000,
) - 10_000;
console.log("Multiplicative impact:", multiplicative); // 199 bps
```

---

## Remediation

Update `quoteRoute` to compute impact multiplicatively:

```typescript
// Aggregate price impact: compound across hops using geometric mean
const totalPriceImpactBps = steps.length === 1
  ? steps[0].priceImpactBps
  : steps.reduce((accumulated, step) => {
      // impact compounds: (1 - i_accumulated) * (1 - i_step) - 1
      // Equivalent: accumulated * (10_000 - step) / 10_000
      return Math.floor(accumulated * (10_000 - step.priceImpactBps) / 10_000);
    }, 10_000) - 10_000;
```

**Rationale**:
- For N steps with impacts i₁, i₂, ..., iₙ:
  - Total multiplier: `(1 - i₁/10_000) × (1 - i₂/10_000) × ... × (1 - iₙ/10_000)`
  - Total impact: `1 - (multiplier) = 1 - product` (in basis points)
- Single-hop routes correctly use step[0].priceImpactBps unchanged
- Multi-hop routes now reflect true compounding
