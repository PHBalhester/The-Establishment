## Verdict: PARTIALLY_FIXED with REGRESSION

H012 was marked FIXED — meaning the quote-engine and route-engine BigInt paths were verified clean. That core fix holds. However, two new hooks introduced since Audit #1 (`useStaking.ts`) and the unchanged `useSwap.ts` `toBaseUnits` helper contain the same `parseFloat * 10^n` Number-space multiplication pattern. Critically, in `useStaking.ts` this pattern feeds directly into on-chain transaction amounts, not just display values. This constitutes a regression relative to H012's fix intent.

---

## Finding 1 — REGRESSION: useStaking.ts stake/unstake amounts via Number multiplication

**File:** `app/hooks/useStaking.ts`
**Lines:** 454, 516, 540

In the `minimumStakeWarning` useMemo (line 454) and in both `case "stake"` (line 516) and `case "unstake"` (line 540) of the `execute` callback, the amount entered by the user is converted to base units using the pattern:

```typescript
const amountBaseUnits = Math.floor(parsedAmount * 10 ** PROFIT_DECIMALS);
```

where `PROFIT_DECIMALS = 6`.

This is the same `parseFloat * 10^n` Number-space multiplication that H012 flagged in `useSwap.ts`. In `useStaking.ts` the result is **not** used for display — it is passed directly as `amount` to `buildStakeTransaction` and `buildUnstakeTransaction`, which construct on-chain instructions.

**Precision boundary:** `parseFloat(amount) * 1_000_000` produces exact results for up to ~9,007,199 PROFIT tokens (2^53 / 10^6 ≈ 9.007 × 10^9 human tokens). The PROFIT supply is 20M tokens, so the maximum user holding is 20M * 10^6 = 2×10^13 base units. `20_000_000 * 1_000_000 = 2e13`, which is well within `Number.MAX_SAFE_INTEGER` (9.007×10^15). However:

- The multiplication `Math.floor(parsedAmount * 10 ** PROFIT_DECIMALS)` is performed in IEEE 754 double precision. For inputs with many decimal places (e.g. `"0.123456"`) the result of `0.123456 * 1_000_000` may not be exactly `123456`. Specifically, `0.1 + 0.2 !== 0.3` class errors apply here.
- Example: `0.100001 * 1_000_000` in JavaScript yields `100001.00000000001`, which `Math.floor` corrects to `100001`. But `0.123456789 * 1_000_000` can yield `123456.789000000001` — floored to `123456`, losing the sub-unit fraction. Since PROFIT has only 6 decimals, sub-unit fractions are expected to be zero (the user types at most 6 decimal places), so this is bounded.
- The true risk is staking precision diverging from the on-chain expected amount if a user enters a value that cannot be represented exactly in IEEE 754 at the `* 10^6` scale. The on-chain program receives the floored Number-space value, which may be 1 base unit fewer than intended.

**Comparison to H012 fix:** The quote-engine (`app/lib/swap/quote-engine.ts`) and the slippage computations were fixed in H012 to use BigInt throughout. The staking hook was written after H012 and did not carry forward that BigInt discipline for its `parseFloat → base units` conversion.

**Severity impact:** The financial risk is bounded to 1 base unit (1 millionth of 1 PROFIT token ≈ $0.000001 at typical prices). This is not directly exploitable for profit. However, it violates the codebase's stated invariant (`INV-8`: "All financial quote math uses BigInt") and sets a pattern that could be copied to higher-precision contexts.

**Correct pattern (from BuyForm.tsx:114 — already implemented correctly):**
```typescript
const lamports = BigInt(Math.floor(parsed * LAMPORTS_PER_SOL));
```
Or preferably, for tokens:
```typescript
const amountBaseUnits = BigInt(Math.round(parsed * 10 ** PROFIT_DECIMALS));
// Better: parse integer and fractional parts separately to avoid IEEE 754
```

## Finding 2 — UNCHANGED (not fixed, not regressed): useSwap.ts toBaseUnits

**File:** `app/hooks/useSwap.ts`
**Lines:** 301–307

```typescript
const toBaseUnits = useCallback(
  (amount: string, token: TokenSymbol): number => {
    const parsed = parseFloat(amount);
    if (isNaN(parsed) || parsed <= 0) return 0;
    const decimals = getDecimals(token);
    return Math.floor(parsed * 10 ** decimals);
  },
  [getDecimals],
);
```

This is unchanged from the state audited in H012. The return type is `number` (not `bigint`). The result is used:
- In `computeQuote` to feed `BigInt(baseUnits)` into quote-engine functions — the BigInt wrapping is safe since `baseUnits` is a Number that passed through `Math.floor` and is within safe integer range.
- In `executeSwap` (lines 713, 730, 742) as `amountInLamports` / `amountInBaseUnits` passed to transaction builders. These builders receive a `number` type.

The precision boundary for SOL is `2^53 / 10^9 ≈ 9,007,199 SOL` — far beyond any realistic user input. For tokens (6 decimals) it is `2^53 / 10^6 ≈ 9,007,199,254 tokens`. At mainnet pool scale (290M tokens max reserve), this gives >30x headroom. **No change from H012 assessment: LOW risk, not exploitable at current scale.**

H012's actual fix addressed the quote-engine and minimumOutput paths (BigInt throughout). The `toBaseUnits` Number multiplication was assessed as LOW/acceptable in H012 and remains so. It is documented as a known deviation from the BigInt invariant, not a regression.

## Finding 3 — DISPLAY-ONLY (unchanged): useSwap.ts fee percentage mixing

**File:** `app/hooks/useSwap.ts`
**Lines:** 448–449, 521–522

The sell-path `totalFeePct` display string mixes denominations:

```typescript
// Line 448-449 (forward sell quote)
const totalFeePct = grossSolOutput > 0
  ? ((taxAmount / grossSolOutput) * 100 + (lpFee / baseUnits) * 100).toFixed(1) + "%"
  : "0%";

// Line 521-522 (reverse sell quote)
const totalFeePct = desiredOutput > 0
  ? ((taxAmount / (desiredOutput + taxAmount)) * 100 + (lpFee / inputTokensNeeded) * 100).toFixed(1) + "%"
  : "0%";
```

The first term is SOL-denominated (taxAmount / SOL gross). The second is token-denominated (lpFee / token input). Adding these produces a meaningless percentage. This was noted in H012 as a LOW display-only issue. It has not been fixed and has not worsened. The route-engine correctly uses BPS summation for its fee computation. **Status: unchanged LOW, display-only, no financial impact.**

## Finding 4 — SAFE (bounded): BuyForm.tsx and SellForm.tsx

**File:** `app/components/launch/BuyForm.tsx` lines 87–88, 114
**File:** `app/components/launch/SellForm.tsx` lines 96, 122

Both forms use `Math.floor(parseFloat * factor)` patterns. Analysis:

- `BuyForm.tsx:114`: `BigInt(Math.floor(parsed * LAMPORTS_PER_SOL))` — max user SOL input is bounded by balance (practical max ~100 SOL). `100 * 1e9 = 1e11`, well within safe range. Safe.
- `BuyForm.tsx:87`: `BigInt(Math.floor(crimeBalance * Number(TOKEN_DECIMAL_FACTOR)))` — `crimeBalance` is already a display float from `useTokenBalances`. Max 20M tokens, `20e6 * 1e6 = 2e13 < 2^53`. Marginally safe.
- `SellForm.tsx:96`: Same as BuyForm:87 pattern. Safe.
- `SellForm.tsx:122`: `BigInt(Math.floor(parsed * Number(TOKEN_DECIMAL_FACTOR)))` — max token input is user's balance (≤20M tokens). `20e6 * 1e6 = 2e13 < 2^53`. Safe, though the intermediate Number multiplication has the same sub-unit rounding exposure as Finding 1. However, since `TOKEN_DECIMAL_FACTOR = 1_000_000n` and users type at most 6 decimal places, the fractional rounding is bounded to ±1 base unit. The bonding curve's `buildSellInstruction` receives this as a `bigint`, so the conversion happens one step later. No regression from H012.

## Root Cause

H012's fix was scoped to the quote-engine and route-engine BigInt paths. The staking hook was introduced (or significantly modified) after H012, and the developer reused the pre-H012 `parseFloat * 10^n` Number pattern for the on-chain amount, rather than following the BigInt discipline established in the fix.

---

## Recommended Fix

In `useStaking.ts`, replace the three instances of `Math.floor(parsedAmount * 10 ** PROFIT_DECIMALS)` with a BigInt-safe parse:

```typescript
// Replace:
const amountBaseUnits = Math.floor(parsedAmount * 10 ** PROFIT_DECIMALS);

// With:
const amountBaseUnits = Number(BigInt(Math.round(parsedAmount * 10 ** PROFIT_DECIMALS)));
// Or, for full correctness, split integer/fractional parts:
const [intPart, fracPart = ""] = amount.split(".");
const paddedFrac = fracPart.padEnd(PROFIT_DECIMALS, "0").slice(0, PROFIT_DECIMALS);
const amountBaseUnits = Number(BigInt(intPart) * 10n ** BigInt(PROFIT_DECIMALS) + BigInt(paddedFrac));
```

The second form (string split) is immune to IEEE 754 rounding at any decimal scale. A shared `parseAmountToBaseUnits(amount: string, decimals: number): bigint` utility should be extracted and used by `useSwap.ts`, `useStaking.ts`, `BuyForm.tsx`, and `SellForm.tsx` to prevent future recurrence.
