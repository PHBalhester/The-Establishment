# H111: deploy logger Writes to Arbitrary Path

**Severity**: Tier 3 (Medium)
**Component**: `scripts/deploy/lib/logger.ts:64-79`
**Type**: Path Traversal / Arbitrary File Write

## Summary

The `createLogger()` function accepts an optional `logFilePath` parameter with **no validation or sanitization**. A malicious actor with the ability to supply this parameter could write log files to arbitrary locations on the filesystem, potentially overwriting sensitive files or directories outside the intended deploy directory.

## Vulnerable Code

```typescript
export function createLogger(logFilePath?: string): Logger {
  // ...
  const logDir = path.resolve(__dirname, "..");
  const logPath = logFilePath || path.join(logDir, `deploy-log-${timestamp}.txt`);

  // Create/truncate the log file with a header line
  fs.mkdirSync(path.dirname(logPath), { recursive: true });
  fs.writeFileSync(logPath, `${new Date().toISOString()} | Deploy log started\n`);
  // ...
}
```

**Lines 72–79**: The function uses `logFilePath` directly in `fs.writeFileSync()` and `fs.mkdirSync()` without any bounds checking, path normalization validation, or verification that the path stays within an allowed directory.

## Attack Vector

If an attacker can control the `logFilePath` parameter (e.g., via environment variable, command-line argument, or compromised configuration file), they could:

1. **Write to parent directories**: `../../../etc/passwd` (on Unix-like systems)
2. **Overwrite critical files**: `../../package.json`, `../../.env`, `../../tsconfig.json`
3. **Create symlinks**: Point to sensitive files and truncate them
4. **Directory traversal**: Escape the `scripts/deploy/` directory entirely

### Example Exploit

```bash
DEPLOY_LOG_PATH="../../../tmp/evil.txt" npx tsx scripts/deploy/initialize.ts
```

This would write to `/tmp/evil.txt` instead of the intended `scripts/deploy/` directory.

## Current Usage

Both callsites (`initialize.ts:184` and `verify.ts:202`) invoke `createLogger()` **without arguments**:

```typescript
const log = createLogger();
```

This means the `logFilePath` parameter is currently unused in production code. However, the vulnerability exists in the function signature and could be exploited if:

- A future refactor passes user input to this parameter
- A script or tool wraps this function and accepts external configuration
- An attacker gains write access to environment variables or config files

## Verdict

**CONFIRMED** (Low exploitability in current usage, but the vulnerability pattern is present)

The code does not validate the `logFilePath` parameter, allowing path traversal via `../` sequences or absolute paths. The `fs.mkdirSync(..., { recursive: true })` call on line 78 makes the attack more dangerous by creating intermediate directories.

## Recommendation

1. **Validate the log path** to ensure it stays within the intended directory:

```typescript
export function createLogger(logFilePath?: string): Logger {
  const timestamp = new Date()
    .toISOString()
    .replace(/[-:]/g, "")
    .replace(/\.\d+Z$/, "Z");

  const logDir = path.resolve(__dirname, "..");
  const resolvedPath = logFilePath ? path.resolve(logFilePath) : path.join(logDir, `deploy-log-${timestamp}.txt`);

  // Ensure the resolved path is within logDir
  if (!resolvedPath.startsWith(logDir + path.sep) && resolvedPath !== logDir) {
    throw new Error(`Log path must be within ${logDir}, got ${resolvedPath}`);
  }

  fs.mkdirSync(path.dirname(resolvedPath), { recursive: true });
  fs.writeFileSync(resolvedPath, `${new Date().toISOString()} | Deploy log started\n`);

  return {
    // ... rest of the implementation
  };
}
```

2. **Restrict `mkdirSync` recursive flag** or validate parent directories explicitly.
3. **Document that `logFilePath` must be a filename only** (no path separators).
