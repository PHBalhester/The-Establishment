# H066

## Verdict
**CONFIRMED — Tier 3 Information Disclosure (Unauthenticated)**

Credit consumption stats (totalCalls, per-method breakdown, elapsed time since process start) are exposed without authentication via `/api/health`. An attacker can estimate Helius rate limit approach and time-window attacks when limits are near.

---

## Evidence

### Location 1: `credit-counter.ts:41-47`
```typescript
getStats(): CreditStats {
  return {
    totalCalls: this.totalCalls,
    methodCounts: { ...this.methodCounts },
    startedAt: this.startedAt,
  };
}
```
The `getStats()` method returns a full snapshot of credit consumption with no filtering or rate limiting. It exposes:
- `totalCalls` — running sum of all RPC calls issued by the server
- `methodCounts` — object literal mapping method name → call count (e.g., `{ getAccountInfo: 234, getBalance: 12 }`)
- `startedAt` — ISO timestamp when the counter was initialized

### Location 2: `health/route.ts:64, 70`
```typescript
const credits = creditCounter.getStats();
// ...
return NextResponse.json({
  status,
  checks: { postgres, solanaRpc },
  wsSubscriber,
  credits,  // <-- EXPOSED
  timestamp: new Date().toISOString(),
});
```

The `/api/health` endpoint calls `getStats()` and returns the result unmodified in the HTTP response. No authentication, no rate limiting, no filtering.

---

## Attack Path

1. **Monitor Credit Consumption**: Attacker polls `/api/health` at 1–10 Hz intervals to build a timeline of credit burn rate.
2. **Estimate Timelock**: Helius resets daily. With known burn rate and `startedAt` timestamp, attacker estimates remaining credits before daily reset.
3. **Window Attacks**:
   - When credits approach limit, RPC responses degrade. Attacker times UI delays to infer that limits are near.
   - Launch MEV + spam during low-credit window when server cannot respond to legitimate RPC calls.
   - Prioritize high-impact queries when server is rate-limited (e.g., frontrun staker queries to stale data).

**Example**: If credits burn at 100/min and the endpoint reports 4,500 calls since server start (9 mins ago), attacker knows 45 minutes remain until daily reset. Launch DoS attack at T-10min, when server is at 90% quota and user TXs will fail.

---

## Impact

- **Severity**: Medium (Tier 3)
- **CVSS Estimate**: 5.3 (Access Vector: Network, Privileged: No, User Interaction: No, Scope: Unchanged, CIA: Partial)
- **Actual Risk**: Enables timing attacks on availability and helps adversaries coordinate MEV/spam campaigns.
- **Business Impact**: Users experience degraded service when attacker times RPC exhaustion. Protocol is not stolen or drained, but UX suffers.

---

## Recommendation

### Required Fix
Remove credit stats from `/api/health` response or gate behind authentication:

**Option A: Remove stats entirely** (Recommended for public endpoint)
```typescript
// health/route.ts
const wsSubscriber = getWsSubscriberStatus();
// DO NOT expose creditCounter.getStats()

return NextResponse.json({
  status,
  checks: { postgres, solanaRpc },
  // wsSubscriber optional, but already contains internal state
  timestamp: new Date().toISOString(),
});
```

**Option B: Gated endpoint** (If monitoring dashboard needs stats)
- Create a separate `/api/admin/health-internal` endpoint that requires Bearer token
- Require `X-Admin-Token` header matching ADMIN_HEALTH_TOKEN env var
- Move `wsSubscriber` and `credits` to that endpoint
- Keep `/api/health` lean for Railway healthcheck

**Option C: Aggregate and limit precision** (Middle ground)
- Report only `totalCalls` (not per-method breakdown)
- Round to nearest 100 to obscure exact state
- But this is security-by-obscurity and doesn't fully mitigate the risk

### Priority
- **Pre-Launch**: Required for mainnet (public endpoint exposes operational state)
- **Effort**: 5 minutes
- **Test**: Verify `/api/health` returns only `status`, `checks`, `timestamp` (no `credits`)
