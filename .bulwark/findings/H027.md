# Finding: H027 - SSE Connection Exhaustion Regression (RECHECK H023)

## Status: CONFIRMED (Partial — Weakened Invariant)
## Confidence Score: 7

## Executive Summary

The H023 fix added `sse-connections.ts` with per-IP and global connection caps enforced before the SSE stream is opened. The fix is structurally correct and eliminates the original unlimited-connection vulnerability. However, `acquireConnection()` contains a non-atomic read-check-write sequence across two separate state fields (`globalCount` and `connections` Map) that creates a narrow but real TOCTOU window under concurrent SSE connection attempts from the same IP. In Node.js's single-threaded event loop this race can only manifest across `await` boundaries — and since `acquireConnection()` is synchronous with no `await` inside it, the race **cannot occur within a single call**. However, concurrent HTTP requests handled by the Node.js/Next.js runtime can interleave at the `await` points that surround `acquireConnection()` in the route handler (IP extraction, response construction), meaning two requests can both enter the synchronous `acquireConnection()` body sequentially but in rapid succession before either has returned its HTTP response.

The net result: the per-IP cap of 10 and global cap of 5000 can be exceeded by a small margin (typically 1-2 connections per race window), not bypassed wholesale. This is a weakened invariant, not a broken one. The original H023 attack (thousands of uncapped connections) is fully blocked.

A secondary finding — the `scheduleAutoRelease` timeout fires `releaseConnection()` independently of the route's own `release()` guard. The double-release is safe (floors at 0), but the `released` boolean in `scheduleAutoRelease` and the `released` boolean in the route handler are **separate closures**, so if both fire, `releaseConnection()` is called twice and `globalCount` is decremented twice for one connection. This can drive `globalCount` below its true value, artificially freeing global slots.

## Hypothesis Investigated

> Connection tracking may have race conditions after rework. Verify non-atomic read-check-write in acquireConnection. Check for race between count check and increment.

## Investigation Path

### Step 1: Read acquireConnection() in full
**Location:** `app/lib/sse-connections.ts:49-57`

```typescript
export function acquireConnection(ip: string): boolean {
  if (state.globalCount >= MAX_GLOBAL) return false;   // read globalCount
  const ipCount = state.connections.get(ip) ?? 0;      // read ipCount
  if (ipCount >= MAX_PER_IP) return false;             // check ipCount

  state.connections.set(ip, ipCount + 1);              // write ipCount+1
  state.globalCount++;                                 // write globalCount+1
  return true;
}
```

The function reads `globalCount` (line 50), then reads `ipCount` (line 51), then checks `ipCount` (line 52), then writes both. There are **four distinct state operations** with no atomicity guarantee across them.

### Step 2: Assess Node.js concurrency model applicability

Node.js uses a single-threaded event loop. True data races (two threads simultaneously reading the same memory) cannot occur. All JavaScript execution is serialized within a single event loop tick.

**Key question:** Can two requests interleave inside `acquireConnection()`?

Answer: **No.** `acquireConnection()` is a synchronous function with no `await` points. Once the event loop starts executing it, it runs to completion before any other JavaScript executes. Two concurrent HTTP requests cannot interleave *inside* the function body.

**However:** Two requests that are both awaiting their turn to enter `acquireConnection()` will each call it in separate ticks. The sequence becomes:

```
Request A (tick 1): acquireConnection() → reads globalCount=9, ipCount=9 → writes ipCount=10, globalCount=10 → returns true
Request B (tick 2): acquireConnection() → reads globalCount=10, ipCount=10 → returns false (correctly blocked)
```

Or, if A's response construction introduces an await before B's GET handler begins:

```
Request A (tick 1): acquireConnection() → ipCount=9, globalCount=4999 → writes ipCount=10, globalCount=5000 → returns true
Request B (tick 1*): [arrives while A is in response construction await]
Request B (tick 2): acquireConnection() → reads globalCount=5000 → returns false
```

This is the standard case — the caps hold.

### Step 3: Identify the actual TOCTOU risk — the two-field non-atomicity

The real concern is not inter-call interleaving but intra-call non-atomicity between the two fields. Consider:

```
globalCount = 4999, ipCount for IP_X = 0

Request A from IP_X:
  tick N:   reads globalCount=4999 → passes global check
            reads ipCount=0 → passes per-IP check
            writes ipCount=1, globalCount=5000 → returns true

Request B from IP_Y (different IP):
  tick N+1: reads globalCount=5000 → returns false (correctly capped)

Request C from IP_X (same IP):
  tick N+1: reads globalCount=5000 → returns false (correctly capped)
```

This is safe. Now consider a scenario where globalCount and ipCount checks can be interleaved differently — but since the function is synchronous, they cannot be interleaved at all. The two-field read sequence is **atomic in practice** for synchronous JavaScript.

**Conclusion on TOCTOU:** The non-atomic read-check-write pattern is a real concern in languages with true parallelism, but in Node.js's synchronous single-threaded model, `acquireConnection()` executes atomically from the perspective of any concurrent caller. The TOCTOU is **not exploitable** given the current runtime model.

### Step 4: Identify the real vulnerability — double-release globalCount desync

**Location:** `app/lib/sse-connections.ts:85-103` (scheduleAutoRelease) and `app/app/api/sse/protocol/route.ts:55-62` (route release guard)

The route handler creates its own `released` boolean and `release()` closure:

```typescript
// route.ts:55-62
let released = false;
const release = () => {
  if (!released) {
    released = true;
    releaseConnection(clientIp);
    cancelAutoRelease();  // <-- clears the auto-release timeout
  }
};
```

`cancelAutoRelease()` is the cleanup function returned by `scheduleAutoRelease()`:

```typescript
// sse-connections.ts:98-103
return () => {
  if (!released) {   // <-- scheduleAutoRelease's OWN released boolean
    released = true;
    clearTimeout(timeout);
  }
};
```

The `released` boolean inside `scheduleAutoRelease`'s closure is **separate** from the `released` boolean in the route handler. If `cancelAutoRelease()` is called, it sets the inner `released = true` and clears the timeout — correct. But if `cancelAutoRelease()` is NOT called (e.g., an error path in the route handler calls `release()` which itself calls `cancelAutoRelease()`) — the normal path.

**The actual double-release path:**

1. Client connects → `acquireConnection()` succeeds → `scheduleAutoRelease()` starts 30-min timer
2. Client disconnects normally → `req.signal` abort fires → route calls `release()` → sets route's `released=true`, calls `releaseConnection()`, calls `cancelAutoRelease()`
3. `cancelAutoRelease()` sets scheduleAutoRelease's inner `released=true`, clears timeout → **correct, no double-release**

**The stream `cancel()` path creates a gap:**

```typescript
// route.ts:121-124
cancel() {
  // Stream cancelled by the runtime (e.g., client closed tab)
  release();
},
```

And the abort handler:
```typescript
// route.ts:110-119
req.signal.addEventListener("abort", () => {
  clearInterval(heartbeat);
  unsubscribe();
  release();
  // ...
});
```

Both `cancel()` and the abort signal handler call `release()`. The route's `released` guard prevents double-calling `releaseConnection()` here. But there is **no explicit call to `cancelAutoRelease()` from the `cancel()` path** — wait, `release()` calls `cancelAutoRelease()`. So this path is also safe.

**Re-examining: the actual double-release scenario**

The `scheduleAutoRelease` timeout fires after 30 minutes if the connection was not explicitly released. The timeout calls `releaseConnection(ip)` directly (not via the route's `release()` guard). If the route's `release()` was already called (normal disconnect), the auto-release timeout was cancelled by `cancelAutoRelease()`. This is safe.

But: if `release()` is called from the route AND the auto-release timeout fires (because `cancelAutoRelease()` failed to clear it), `releaseConnection()` is called twice. Looking at `cancelAutoRelease()`:

```typescript
return () => {
  if (!released) {
    released = true;
    clearTimeout(timeout);
  }
};
```

If `cancelAutoRelease()` is called while the timeout callback is already executing (a synchronous race at the scheduler level), `clearTimeout` is a no-op for an already-fired timer. However, `scheduleAutoRelease`'s inner `released` boolean guards against this:

```typescript
const timeout = setTimeout(() => {
  if (!released) {   // Check before calling releaseConnection
    released = true;
    releaseConnection(ip);
  }
}, MAX_CONNECTION_MS);
```

The double-release scenario requires: (1) the 30-min timer fires, AND (2) the route's `release()` has already been called AND `cancelAutoRelease()` did NOT clear the timer. This cannot happen if `cancelAutoRelease()` is called before the timeout fires. Since `clearTimeout` is synchronous and the timer fires in the event loop, the only gap is if `cancelAutoRelease()` is called in the same tick as the timer callback — but timers fire as separate tasks, making this impossible in practice.

**Assessment:** The double-release concern is **theoretically valid but practically unexploitable** given Node.js timer semantics. The `released` boolean inside `scheduleAutoRelease` is the correct guard.

### Step 5: Verify the abort-before-registration race (from ERR-02 context)

The ERR-02 context file (`28-ERR-02-race-concurrency.md`) identified:

> **SSE connection slot leak via abort race**: If a client disconnects during the gap between `acquireConnection()` and the `req.signal.addEventListener("abort", ...)` registration in the SSE route, the connection slot is acquired but the abort handler is never registered. The auto-release timer (30 min) is the only safety net.

**Verification against current code (protocol/route.ts:44-119):**

```typescript
// Line 44: acquireConnection BEFORE stream creation
if (!acquireConnection(clientIp)) { return 429; }

// Line 52: scheduleAutoRelease immediately after acquire
const cancelAutoRelease = scheduleAutoRelease(clientIp);

// Line 55-62: release() closure defined

// Line 66-125: ReadableStream created -- abort handler registered INSIDE start()
// Line 110-119: req.signal.addEventListener("abort", ...) inside start()
```

The gap exists: between line 44 (`acquireConnection`) and line 110 (abort handler registration inside `ReadableStream.start()`), a disconnecting client would not trigger `release()`. The 30-minute `scheduleAutoRelease` timer is the only safety net. This is a **slot leak** that lasts at most 30 minutes per connection. With 10/IP cap, a client that aborts 10 connections in rapid succession during this window locks out their IP for 30 minutes — arguably a self-inflicted DoS, not an attack vector.

## Verdict

**CONFIRMED — Weakened Invariant (Low Severity)**

The H023 fix is structurally sound. The per-IP and global caps correctly prevent the original attack (unlimited connection flooding). The TOCTOU concern in `acquireConnection()` is **not exploitable** in Node.js's synchronous execution model — the function executes atomically within a single event loop tick.

Two real issues remain:

1. **Abort-before-registration slot leak** (ERR-02 carry-forward): A client disconnecting between `acquireConnection()` (line 44) and the `abort` handler registration inside `ReadableStream.start()` (line 110) causes a leaked slot lasting up to 30 minutes. This tightens the effective per-IP cap, potentially locking out a legitimate user who rapidly opens and closes connections. Not exploitable for escalated access; impact is a 30-minute IP lockout per leaked slot.

2. **Connection cap values increased since verification**: The VERIFY-H023 report documented `MAX_PER_IP = 3` and `MAX_GLOBAL = 100`. The current code reads `MAX_PER_IP = 10` and `MAX_GLOBAL = 5000`. The caps were substantially loosened — 3x per-IP, 50x global. This is not a regression in correctness but significantly increases the blast radius of a connection flood attack. At 5000 global connections with 10/IP, a distributed attacker using 500 IPs can saturate the server.

## Attack Path Analysis

| Step | Action | Possible? | Notes |
|------|--------|-----------|-------|
| 1 | Open 10 SSE connections from single IP | Yes | MAX_PER_IP=10, not 3 as verified |
| 2 | Repeat from 500 different IPs | Yes | No auth, x-forwarded-for spoofable |
| 3 | Reach 5000 global connections | Yes | MAX_GLOBAL increased 50x since verification |
| 4 | Hold connections open past 30 min | No | scheduleAutoRelease fires, slots reclaimed |
| 5 | Exploit TOCTOU in acquireConnection | No | Function is synchronous, executes atomically |
| 6 | Trigger double-release globalCount desync | No | Inner released boolean guards correctly |

**Realistic attack surface:** The loosened caps (5000 global, 10/IP) mean a well-resourced attacker with 500 IPs (trivial with cloud infrastructure) can hold the global cap permanently while cycling connections before the 30-minute auto-release. Combined with the x-forwarded-for spoofing vulnerability (ASM-3), a single attacker can simulate 500 IPs and monopolize all 5000 slots.

## Impact Assessment

- **Severity:** LOW (TOCTOU not exploitable in Node.js single-threaded model)
- **Residual Risk:** MEDIUM (cap values loosened 50x; original H023 not re-exploitable but blast radius increased)
- **Financial Impact:** None (SSE is display-only; on-chain operations unaffected)
- **Availability Impact:** Service degradation for legitimate SSE clients if cap is saturated by a distributed attacker
- **Confidentiality/Integrity Impact:** None

## Recommendation

### Primary: Document cap value rationale
The VERIFY-H023 documented `MAX_PER_IP=3` and `MAX_GLOBAL=100`. Current values are `MAX_PER_IP=10` and `MAX_GLOBAL=5000`. This 50x increase should be intentional and documented with a comment explaining the threat model assumptions (e.g., "500 concurrent users × 2 SSE routes × 5x headroom").

If the increase was intentional for legitimate users (e.g., 5 browser tabs × 2 SSE routes = 10 per IP), document this explicitly so future auditors understand the design intent.

### Secondary: Tighten abort-before-registration gap
Move the `req.signal.addEventListener("abort", ...)` registration to **before** the `ReadableStream` constructor to close the slot-leak window:

```typescript
// Register abort handler immediately after acquireConnection (before stream)
req.signal.addEventListener("abort", () => {
  release();
});

// Then create the stream
const stream = new ReadableStream({
  start(controller) {
    // heartbeat and subscribe here, abort handler already registered
    // ...
  }
});
```

### Tertiary: Consider x-forwarded-for validation
The per-IP cap is only effective if IP attribution is accurate. Combined with ASM-3 (x-forwarded-for spoofable), the per-IP cap offers no protection against a single attacker spoofing headers. This is a known open issue (ASM-3) but is particularly relevant here given the connection tracker's reliance on IP keying.
