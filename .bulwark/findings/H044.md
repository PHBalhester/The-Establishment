## Verdict: CONFIRMED

Both poll loops (`pollSupply` at line 360 and `pollStakers` at line 430) use bare
`setInterval` with no overlap-prevention guard. When the RPC responds slowly the
previous invocation is still awaiting network I/O when the next tick fires, creating
unbounded concurrency.

---

## Hypothesis

WS subscriber setInterval polls (supply 60 s, staker gPA 30 s) lack overlap
protection. A slow or degraded RPC endpoint causes concurrent poll invocations to
pile up, creating unbounded parallel RPC consumption that can exhaust Helius credit
quotas and amplify recovery time.

---

## Evidence

### Supply poll — `startSupplyPoll` (lines 329–362)

```ts
// app/lib/ws-subscriber.ts:335-357
const pollSupply = async () => {
  try {
    const [crimeSupply, fraudSupply] = await Promise.all([
      connection.getTokenSupply(MINTS.CRIME),
      connection.getTokenSupply(MINTS.FRAUD),
    ]);
    // ... store results
  } catch (err) { ... }
};

state.supplyPollTimer = setInterval(pollSupply, interval);  // line 360
```

There is no `isRunning` flag, no mutex, and no `clearInterval`-then-reschedule
pattern. If either `getTokenSupply` call takes longer than `interval`
(default 60 000 ms), the next tick fires and a second `pollSupply` call enters the
`Promise.all` concurrently. Under a degraded RPC:

- Tick 1 at t=0 s: calls `getTokenSupply` × 2, stalls at 55 s.
- Tick 2 at t=60 s fires before tick 1 resolves: calls `getTokenSupply` × 2 again.
- Tick 3 at t=120 s: same.
- N overlapping calls accumulate until the RPC either recovers or the process OOMs.

### Staker poll — `startStakerPoll` (lines 368–432)

```ts
// app/lib/ws-subscriber.ts:374-427
const pollStakers = async () => {
  try {
    const accounts = await connection.getProgramAccounts(
      PROGRAM_IDS.STAKING, { filters: [...] }
    );
    // decode + store
  } catch (err) { ... }
};

state.stakerPollTimer = setInterval(pollStakers, interval);  // line 430
```

`getProgramAccounts` is the heaviest RPC call in the codebase — it fetches full
account data for every `UserStake` account without a `dataSlice`. On mainnet with
thousands of stakers this response is large. The default interval is 30 000 ms.
If Helius gPA latency exceeds 30 s (common under load-shedding), each subsequent
tick queues another full gPA request, multiplying both credit consumption and
in-flight connection count.

### No guard present

A search of the file confirms neither function contains:
- An `isRunning` / `isPollActive` boolean checked at function entry.
- A `clearInterval` + setTimeout reschedule pattern (safe interval).
- A semaphore or promise-chain serialisation (`if (pendingRef.current) return`).

The only protection is the outer `try/catch`, which handles thrown errors but does
not prevent overlap when calls succeed slowly.

### Slot fallback poll — secondary concern

`startSlotFallbackPoll` (lines 278–293) has the same pattern but with a shorter
interval (5 000 ms) and a lighter call (`getSlot`). Impact is lower but the
structural vulnerability is identical.

---

## Attack / Failure Scenario

1. **Degraded RPC** (Helius maintenance, rate-limiting, or network partition):
   - gPA calls begin taking >30 s to respond.
   - Every 30 s a new `pollStakers` invocation is queued.
   - After 5 min: 10 concurrent gPA calls in-flight.
   - Each gPA call at mainnet scale costs ~10–50 CUs of Helius credits.
   - Credit quota exhausted → subsequent calls fail → staker count display stalls.

2. **Recovery amplification** — When the RPC recovers, all N stacked calls resolve
   nearly simultaneously, triggering N rapid `setAccountState` + `sseManager.broadcast`
   calls. With 5 000 SSE clients, this produces N × 5 000 write operations in a
   tight burst, spiking Node.js CPU and potentially triggering Railway's process
   restart threshold.

3. **Memory growth** — Each in-flight `pollStakers` holds the decoded
   `stakerGpaAccounts` array in memory until the promise resolves. Under storm
   conditions, peak memory grows with O(concurrent_calls × accounts × account_size).

## Root Cause

`setInterval` fires on a fixed wall-clock cadence regardless of whether the
previous invocation has completed. The async poll bodies contain no entry guard to
detect and skip an already-running invocation.

---

## Recommended Fix

Replace bare `setInterval` with a **self-rescheduling async loop** (safe interval
pattern) that waits for the previous call to complete before scheduling the next:

```ts
// Safe interval pattern — fires next tick only AFTER current completes
async function safeInterval(fn: () => Promise<void>, ms: number): Promise<() => void> {
  let active = true;
  const run = async () => {
    while (active) {
      await fn();
      if (active) await new Promise(res => setTimeout(res, ms));
    }
  };
  run(); // fire-and-forget the loop
  return () => { active = false; };
}
```

Apply to both `startSupplyPoll` and `startStakerPoll`. Store the returned canceller
in `state` in place of the `ReturnType<typeof setInterval>` timers.

Alternatively, add an `isRunning` flag guard at minimal diff cost:

```ts
let isSupplyPollRunning = false;
const pollSupply = async () => {
  if (isSupplyPollRunning) return;
  isSupplyPollRunning = true;
  try {
    // ... existing body
  } finally {
    isSupplyPollRunning = false;
  }
};
```

The `finally` block ensures the guard resets even on exception, preventing
permanent lockout.
