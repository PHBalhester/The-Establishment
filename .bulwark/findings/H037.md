## Finding

**Verdict: NOT_VULNERABLE — all BN.toNumber() call sites in the three modified targets produce values well within Number.MAX_SAFE_INTEGER (9.007 × 10^15). No new BN.toNumber() call sites were introduced. The H096 ACCEPTED_RISK rationale is validated by the actual token supply constants.**

---

## Investigation

### Target 1: `usePoolPrices.ts:29-31` — `reserveA: number`, `reserveB: number`

`usePoolPrices.ts` does not contain any direct `BN.toNumber()` call. The hook reads `data.reserveA` and `data.reserveB` from `useProtocolState`, which delivers values that were already converted upstream by `anchorToJson()` in `ws-subscriber.ts`. `anchorToJson` calls `.toNumber()` on BN objects for fields not in `bigintFields`. PoolState is not passed a `bigintFields` list, so `reserveA` and `reserveB` arrive as JavaScript `number`.

**On-chain type:** `u64` (IDL confirmed at `app/idl/types/amm.ts:923,928`).

**Maximum realistic value:**
- CRIME/SOL pool seed: 290M CRIME tokens × 10^6 base units = **2.9 × 10^14**
- FRAUD/SOL pool seed: same order of magnitude
- SOL side: initial seed ~2.5 SOL = 2.5 × 10^9 lamports (negligible)

**Safety margin:** 2.9 × 10^14 is ~31× below MAX_SAFE_INTEGER (9.007 × 10^15). Even if protocol reserves grew 10× from swaps, the value stays safely within range. Overflow would require approximately 9 trillion CRIME/FRAUD tokens worth of liquidity — impossible given the fixed 1B token supply (10^15 base units maximum, still within MAX_SAFE_INTEGER).

**Financial consequence of display overflow:** None. Pool reserves are consumed by the quote engine only after being passed through BigInt conversion at `quote-engine.ts` (which takes `number` inputs and immediately converts via `BigInt(reserveA)`). Even a slightly corrupted display reserve would not affect on-chain execution — `minimumOutput` guards remain correct.

**Conclusion:** No new calls added. Existing implicit conversion is safe within protocol supply constraints.

### Target 2: `useCarnageData.ts:51-55` — `totalCrimeBurned`, `totalFraudBurned`, `totalSolSpent`

`useCarnageData.ts` contains no direct `BN.toNumber()` calls (confirmed by grep: only line 11-12 in a comment). Fields arrive pre-converted via `anchorToJson()` in `ws-subscriber.ts` batchSeed/webhook path.

**On-chain types** (IDL confirmed at `app/idl/types/epoch_program.ts:2122-2141`):
- `totalSolSpent`: `u64` (SOL lamports spent on buybacks)
- `totalCrimeBurned`: `u64` (CRIME base units burned lifetime)
- `totalFraudBurned`: `u64` (FRAUD base units burned lifetime)

**Maximum realistic values:**
- `totalCrimeBurned` / `totalFraudBurned`: bounded by CRIME/FRAUD total supply = 1B × 10^6 = **10^15**. MAX_SAFE_INTEGER = 9.007 × 10^15. These fields accumulate across every Carnage event for the full protocol lifetime. Even if 100% of all CRIME tokens were burned, the value (10^15) remains within safe integer range.
- `totalSolSpent`: a protocol lifetime SOL spend counter. At hypothetical extreme (9M SOL spent = 9 × 10^9 lamports × 10^6 = meaningless; actual is 9 × 10^15 lamports ≈ 9M SOL), this approaches MAX_SAFE_INTEGER. However, 9M SOL ≈ $1.5B at $165/SOL — a scale far beyond any realistic Carnage Fund accumulation. No guard is present, but the exposure is theoretical only.
- `totalTriggers`: `u32` — trivially safe.
- `lastTriggerEpoch`: `u32` — trivially safe.
- `vaultBalanceLamports`: native SOL lamports from SystemAccount — bounded by practical SOL amounts, safe.

**Financial consequence of overflow:** These fields feed display-only UI components (`CarnageStation.tsx`). No on-chain transaction is constructed from these values. Overflow would produce a cosmetically wrong display (negative or modulo-wrapped burn count) but would not affect protocol operation, fund safety, or transaction construction.

**Conclusion:** No new BN.toNumber() calls. Existing implicit conversions are safe for all fields at realistic protocol scale. The `totalSolSpent` theoretical ceiling is noted but practically unreachable.

### Target 3: `ws-subscriber.ts:219` and `:401` — `stakedBalance.toNumber()`, `lastClaimTs.toNumber()`

Two `BN.toNumber()` calls exist in `ws-subscriber.ts`, both in staker gPA processing loops (batchSeed at line 219 and the periodic staker poll at line 401).

**`decoded.stakedBalance.toNumber()` (lines 219, 401):**
- On-chain type: `u64` (`app/idl/types/staking.ts:1509-1514`)
- Maximum value: individual user's staked PROFIT balance. PROFIT total supply = 20M × 10^6 = **2 × 10^13** base units.
- Safety margin: 2 × 10^13 is ~450× below MAX_SAFE_INTEGER. Cannot overflow regardless of staking pattern.

**`decoded.lastClaimTs.toNumber()` (lines 222, 405):**
- On-chain type: `i64` (`app/idl/types/staking.ts:1563`). Unix timestamp in seconds.
- Current value: ~1.74 × 10^9 (year 2026). Year 2038 problem threshold: 2.14 × 10^9. MAX_SAFE_INTEGER threshold: 9 × 10^15 seconds = year 285,428,751.
- Safety margin: essentially infinite for a Unix timestamp.

**Aggregated accumulator `unlockedProfit += balance` (lines 226, 410-412):**
- This sums `stakedBalance.toNumber()` across all staker accounts. Maximum possible sum = entire PROFIT supply = 2 × 10^13 base units. This is the worst-case aggregate across all users, still ~450× below MAX_SAFE_INTEGER.
- The accumulator `unlockedProfit` and `lockedProfit` are stored in `__staking:globalStats` and flow to the staking dashboard display only. No transaction uses these values.

**Conclusion:** Both explicit `BN.toNumber()` calls are safe. Values are bounded by PROFIT total supply (2 × 10^13), which provides a 450× safety margin. No new calls were added since H096.

## Comparison with H096 ACCEPTED_RISK

H096 was accepted as risk because the values were expected to be safe but lacked a runtime guard. This recheck validates the safety claim with concrete supply constants:

- PROFIT supply = 20M tokens (2 × 10^13 base units) — confirmed in `scripts/deploy/initialize.ts:115`
- CRIME/FRAUD supply = 1B tokens each (10^15 base units) — confirmed in `initialize.ts:113-114`
- All values remain well within Number.MAX_SAFE_INTEGER at any achievable protocol state

No new BN.toNumber() call sites were introduced in the modified hooks. The `anchorToJson` conversion path in `ws-subscriber.ts` (line 149) is the upstream source for `usePoolPrices` and `useCarnageData` — it correctly applies `bigintFields` protection for `rewardsPerTokenStored` (u128, which would overflow) but does not need to for the fields examined here.

## Residual Note (Carry-Forward from Context)

The `totalCrimeBurned` and `totalFraudBurned` fields have a theoretical ceiling of 10^15 (entire token supply), which is within MAX_SAFE_INTEGER by only a 9× margin. If the protocol ever implements token inflation (currently not planned), the safety assumption would need re-evaluation. This is not a current vulnerability but is noted for future-proofing.

## Verdict

**NOT_VULNERABLE.** All BN.toNumber() conversions at the three target locations produce values bounded well below Number.MAX_SAFE_INTEGER given the protocol's fixed token supply constants. No new call sites were introduced. The H096 ACCEPTED_RISK status is promoted to NOT_VULNERABLE upon verification of supply constraints.
