# H010: RPC Proxy fetch() Has No Timeout — Worker Exhaustion DoS

## Verdict: CONFIRMED

## Evidence

The upstream `fetch()` call in `app/app/api/rpc/route.ts` at lines 144–148 carries no `signal` option:

```ts
const upstream = await fetch(endpoint, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: bodyStr,
});
```

No `AbortSignal.timeout()`, no `AbortController`, no `setTimeout`-driven cancel is present anywhere in the file. The comment on the `catch` block at line 173 mentions "timeout" as a possible error source (`// Network error (DNS failure, timeout, etc.)`) but this refers only to OS-level TCP timeouts — there is no application-level deadline.

This is confirmed by a project-wide grep: the only two `AbortSignal.timeout` usages in the entire `app/` directory are in `app/app/api/sol-price/route.ts` (lines 48 and 66), which correctly applies a 5,000 ms deadline to its CoinGecko and Binance fetches. The RPC proxy was not given the same treatment.

The rate limit for `/api/rpc` is 300 requests per minute per IP (`app/lib/rate-limit.ts:45–47`). The rate limiter counts requests at arrival time, not at completion time. There is no concurrent request cap.

The deployment model is a single Node.js process on Railway with no horizontal scaling (`app/lib/sse-manager.ts:8–11` note, confirmed by ARCHITECTURE.md). Next.js API routes share the same libuv thread pool and event loop. Node.js does not pre-empt hung async I/O — a large number of pending `await fetch(...)` calls each hold a microtask continuation and an underlying TCP socket, consuming file-descriptor budget and event loop queue slots.

## Attack Path

1. Attacker identifies that Helius is slow or sends crafted requests that cause Helius to delay responses (e.g., `getProgramAccounts` with a broad filter on a high-traffic validator — a known slow call).
2. Attacker sends 300 POST requests per minute to `/api/rpc` from a single IP (within the rate limit). With 3 IPs the attacker can place 900 requests/min before hitting any wall.
3. Each request enters the `for (const endpoint of orderedEndpoints)` loop and awaits `fetch()` indefinitely. Sticky routing means all requests go to the same endpoint first (the last successful one), so all 300 concurrent fetches pile up against the same Helius connection.
4. Because `await fetch()` yields the current async frame but does not release the in-flight HTTP socket, file descriptors accumulate. Node.js defaults to ~1024 open file descriptors per process (Railway does not expand this by default).
5. Once all workers are blocked on pending fetches, new incoming requests queue in the Next.js HTTP server backlog. The SSE heartbeat timer (15 s) cannot fire because the event loop is saturated. Health checks return 200 only if they land in the queue before workers are fully exhausted — once the backlog fills, the Railway health check times out and the container is marked unhealthy.
6. All API routes fail: SSE connections drop (no heartbeat), webhook processing stops, all browser interactions fail. Recovery requires either Helius responding or a Railway container restart.

The `failover` logic at lines 142–178 provides no defense: it only moves to the next endpoint after a completed response with HTTP 5xx or a caught network exception. A hanging connection never throws — it just waits. The failover loop is also sequential (`for...of`), so if endpoint 0 hangs, endpoints 1 and 2 are never tried at all.

## Impact

- **Availability**: Complete service outage achievable by a single attacker from 1–3 IPs within the existing rate limit. No special access, no compromised credentials required.
- **Recovery**: Automatic only if Helius eventually responds. If Helius is truly hanging (e.g., a maintenance window, deliberate slow-loris at the Helius layer, or a network partition), Railway's health check will eventually restart the container — but after a delay of minutes, not seconds.
- **Secondary blast radius**: SSE connections drop (browser clients lose real-time data), crank runner unaffected (uses its own Connection object), but browser-initiated transactions cannot be submitted or confirmed through the proxy.
- **No authentication required**: The endpoint is public. The rate limit is the only gate, and it is bypassed by the attack design (attacker stays under 300/min/IP).

This is a HIGH-severity availability vulnerability. It requires no chain interaction, no on-chain funds, and no insider knowledge beyond the public `/api/rpc` endpoint.

## Recommendation

Apply `AbortSignal.timeout()` to the upstream fetch, consistent with the pattern already used in `sol-price/route.ts`. A 10-second deadline is appropriate for RPC calls (Helius p99 latency is well under 2 s under normal conditions; 10 s gives ample headroom for transient slowness while bounding the hang window):

```ts
const upstream = await fetch(endpoint, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: bodyStr,
  signal: AbortSignal.timeout(10_000),  // 10 s hard deadline
});
```

When `AbortSignal.timeout` fires it throws a `DOMException` with `name === "TimeoutError"`, which is caught by the existing `catch (err)` block (line 172) and causes the failover loop to try the next endpoint. No other code changes are required — the fix slots cleanly into the existing error handling structure.

Additionally, add a concurrent-request cap (e.g., reject with 503 if more than N requests are already in-flight per IP) to provide a second line of defence independent of per-fetch timeouts. This is a defense-in-depth measure; the timeout fix alone closes the primary vector.
