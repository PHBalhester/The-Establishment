# H094: Bigint mode:"number" Truncation Risk

**Category**: Data Type Safety
**Severity**: Tier 3 (Low)
**Status**: ACCEPTED_RISK
**Target**: `/app/db/schema.ts:36-40, 70, 99-102, 121-127`

## Summary

The Drizzle ORM schema uses `mode: "number"` for all bigint columns, storing on-chain u64 values in JavaScript Number type. This truncates values above 2^53 (9,007,199,254,740,992), but does not represent a practical risk given the protocol's realistic operational scale.

### Root Cause

JavaScript's native `Number` type uses IEEE 754 double precision, losing precision for integers > 2^53 − 1. Drizzle ORM's `mode: "number"` bypasses string/BigInt handling, allowing silent truncation.

### Maximum Safe Value

```
2^53 = 9,007,199,254,740,992 lamports ≈ 9,007 SOL
Solana Max Supply = 500M SOL = 5 × 10^17 lamports >> 2^53
```

### Value Range Assessment

**Devnet/Mainnet Realistic Scenarios**:
- Single swap: ≤ 1,000 SOL = 10^12 lamports ✓ (safe)
- Daily protocol volume: ≤ 10,000 SOL = 10^13 lamports ✓ (safe)
- Epoch staking reward: ≤ 1 SOL = 10^9 lamports ✓ (safe)
- Carnage fund balance: ≤ 500 SOL = 5 × 10^11 lamports ✓ (safe)
- Token mints (CRIME/FRAUD): ≤ 10^9 base units ✓ (safe)

**Extreme Scenario** (multi-decade protocol at scale):
- 10-year total volume: 10 billion SOL = 10^19 lamports ✗ (exceeds 2^53)

## Verdict

**CONFIRMED** — Drizzle uses Number for stored bigints, precision loss is possible above 2^53.

**ACCEPTED_RISK** — All realistic protocol values are orders of magnitude below the safety threshold. Even extreme multi-year scenarios require global-scale Solana adoption (10B SOL annual volume) to trigger truncation.
