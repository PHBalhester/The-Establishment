## Combined Attack Path

H005 and H020 describe the same underlying chain from two angles — H005 documents the absence of semantic validation after Anchor decode, and H020 traces the full propagation from webhook receipt to browser display. Together they confirm a complete, unobstructed path from a single authenticated POST to manipulated `minimumOutput` values in every connected user's pending swap transaction.

The MEV outcome is the terminal consequence of that path:

### On-Chain Backstop Analysis

The critical question is: does the on-chain `minimumOutput` enforcement fully protect users?

**For Outcome A (minimumOutput ≈ 0):** The on-chain check `output_amount >= minimum_output_amount` passes trivially (anything ≥ 0). The swap executes. The user receives real tokens but has voluntarily waived all slippage protection by signing a transaction where `minimum_output_amount = 0`. The on-chain program is not at fault — it correctly enforces what the user signed. The MEV extraction opportunity is real and the on-chain program provides no backstop against it.

**The on-chain program is the last honest layer but it cannot protect a user who signed away their slippage floor.** This is the crux: the attack does not break the on-chain program, it manipulates the off-chain layer that produces the parameters the user signs.

## Recommendation

**The on-chain program is not the fix** — it correctly enforces `minimumOutput`. The fix must be in the off-chain layers that produce `minimumOutput`.

**Fix 1 — Semantic validation after Anchor decode (blocks Phase 2-3; required):**

In `app/app/api/webhooks/helius/route.ts`, after `program.coder.accounts.decode()`, validate field ranges before calling `protocolStore.setAccountState()`:

```typescript
function validateDecodedPoolState(decoded: Record<string, unknown>): void {
  const reserveA = Number(decoded.reserveA);
  const reserveB = Number(decoded.reserveB);
  // Reserves must be positive and within plausible protocol bounds
  // (10^19 lamports = ~43M SOL — far beyond any realistic pool)
  if (reserveA <= 0 || reserveB <= 0) throw new Error(`Non-positive reserves`);
  if (reserveA > 1e19 || reserveB > 1e19) throw new Error(`Implausible reserves`);
}

function validateDecodedEpochState(decoded: Record<string, unknown>): void {
  const taxFields = ["crimeBuyTaxBps", "crimeSellTaxBps", "fraudBuyTaxBps", "fraudSellTaxBps"];
  for (const f of taxFields) {
    const v = Number(decoded[f]);
    if (v < 0 || v > 10_000) throw new Error(`Invalid ${f}: ${v}`);
  }
}
```

On validation failure: log to Sentry, return without updating the store.

**Fix 2 — Browser-side minimumOutput floor (blocks Phase 4-5; required as defense in depth):**

In `useSwap.ts`, add a sanity check that the computed `minimumOutput` is at least a configured floor percentage of the naive spot-rate output:

```typescript
const naiveOutput = estimateNaiveOutput(inputAmount, SPOT_RESERVES); // from last ws-subscriber read
const minimumOutputFloor = Math.floor(naiveOutput * MIN_OUTPUT_FLOOR_RATIO); // e.g., 0.5

if (minimumOutput < minimumOutputFloor) {
  // Warn user: reserves appear anomalous
  showWarning("Price data may be stale or incorrect. Swap paused for safety.");
  return;
}
```

This provides protection even if the server store is poisoned.

**Fix 3 — Cross-verify webhook data against ws-subscriber (defense in depth):**

The ws-subscriber independently polls the same PDAs from Helius RPC. When enhanced webhook data for a pool differs from the ws-subscriber's last known state by more than a configured threshold (e.g., reserve ratio changes >50% in one update), suppress the webhook update and re-fetch from RPC before broadcasting. This makes the injection detectable and self-healing.

**Fix 4 — Secret rotation procedure:**

Document and implement a `HELIUS_WEBHOOK_SECRET` rotation procedure. The current implementation has no multi-secret support and no rotation path. Rotation capability limits the blast radius of any secret compromise to the rotation window.
