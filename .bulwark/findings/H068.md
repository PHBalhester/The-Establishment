# H068: Protocol Store Dedup Bypass via Key Ordering

**Tier:** 3 (Medium)
**Category:** Data Integrity / Deduplication Logic
**Target:** `app/lib/protocol-store.ts:54-58`
**Status:** CONFIRMED
**Severity:** POTENTIAL (practical exploitation requires webhook interception)

---

## Summary

The protocol store's deduplication mechanism at line 54-58 compares serialized account state using `JSON.stringify()` string equality. However, JSON serialization preserves JavaScript object insertion order — not content equivalence. Two identical account state objects with different key ordering will produce different JSON strings, bypassing the dedup check and triggering redundant SSE broadcasts.

---

## Vulnerability Details

### Root Cause

Lines 54-59 in `protocol-store.ts`:
```typescript
setAccountState(pubkey: string, data: AccountState): void {
  const serialized = JSON.stringify(data, bigintReplacer);
  this.accounts.set(pubkey, data);

  // Dedup: skip broadcast if data hasn't changed
  if (serialized === this.lastSerialized.get(pubkey)) return;
  this.lastSerialized.set(pubkey, serialized);
  sseManager.broadcast("protocol-update", { account: pubkey, data });
}
```

### Why It Fails

JavaScript's `JSON.stringify()` **preserves insertion order** — it does NOT perform content-based comparison:

```javascript
const data1 = { a: 1, b: 2, c: 3, updatedAt: 1234 };
const data2 = { c: 3, b: 2, a: 1, updatedAt: 1234 };

JSON.stringify(data1) === JSON.stringify(data2) // FALSE
// Output differs:
// '{"a":1,"b":2,"c":3,"updatedAt":1234}'
// '{"c":3,"b":2,"a":1,"updatedAt":1234}'
```

### Data Flow Scenarios That Could Cause Key Reordering

1. **Anchor Decoder Source (Standard)**
   - Webhook → `program.coder.accounts.decode()` → `anchorToJson()` iterates `Object.entries(decoded)` (preserves IDL field order)
   - Spread into object: `{ ...normalized, updatedAt: Date.now() }`
   - Result: Key order = [IDL fields] + ["updatedAt"]

2. **Raw Error Fallback (Line 613-619)**
   - Decode fails → `protocolStore.setAccountState(pubkey, { label, accountData, rawAccountData, updatedAt, decodeError })`
   - Different key order than Anchor path
   - If Helius later delivers the same account successfully decoded, the Anchor path will have different key ordering → dedup fails

3. **Webhook from Alternate Gateway**
   - Helius enhanced webhooks could be routed through different API gateways
   - Each gateway might parse JSON and reconstruct objects with different key ordering
   - Same account data arrives twice with different object serialization

4. **Man-in-the-Middle Reordering**
   - Attacker intercepts webhook → re-orders JSON keys → re-signs with forged webhook secret
   - Identical data bypasses dedup → duplicate broadcast

### Impact

- **SSE Broadcast Storm**: Identical state changes broadcast multiple times, wasting bandwidth
- **Client-Side Side Effects**: Downstream consumers (e.g., `useProtocolState` hook) may trigger unnecessary re-renders or other side effects
- **Cache Coherency**: In-memory store has correct state, but SSE clients see duplicate updates
- **Amplification Vector**: Attacker can force redundant broadcasts without modifying actual account state

## Recommendation

**Replace string equality dedup with content-based equality:**

```typescript
setAccountState(pubkey: string, data: AccountState): void {
  this.accounts.set(pubkey, data);

  // Content-based dedup: compare sorted JSON (key-order independent)
  const lastSerializedSorted = this.lastSerialized.get(pubkey);
  const currentSerialized = JSON.stringify(data, bigintReplacer); // current insertion order

  // Normalize both to canonical form (sort keys) for comparison
  const currentCanonical = JSON.stringify(
    JSON.parse(currentSerialized),
    Object.keys(JSON.parse(currentSerialized)).sort(),
    bigintReplacer
  );
  const lastCanonical = lastSerializedSorted
    ? JSON.parse(lastSerializedSorted) // re-parse and normalize
    : null;

  // Skip broadcast if canonical forms match
  if (lastCanonical && JSON.stringify(lastCanonical) === currentCanonical) {
    return;
  }

  this.lastSerialized.set(pubkey, currentCanonical);
  sseManager.broadcast("protocol-update", { account: pubkey, data });
}
```

**Better Alternative:** Use deep object comparison library or structured comparison:

```typescript
import { deepEqual } from "@/lib/deep-equal"; // or use lodash.isequal

setAccountState(pubkey: string, data: AccountState): void {
  const lastData = this.accounts.get(pubkey);
  this.accounts.set(pubkey, data);

  // Skip broadcast if data content is identical
  if (lastData && deepEqual(lastData, data)) return;

  sseManager.broadcast("protocol-update", { account: pubkey, data });
}
```

This eliminates JSON serialization entirely and compares actual object content.
