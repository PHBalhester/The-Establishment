# H085: Token Supply Uses INITIAL_SUPPLY Fallback

**Tier:** 3 (Display-Only, Low Impact)
**Severity:** Low
**Status:** CONFIRMED, ACCEPTED_RISK
**Target:** `app/hooks/useTokenSupply.ts:19-22`

---

## Finding

The `useTokenSupply` hook uses a hardcoded `INITIAL_SUPPLY` constant as a fallback while SSE (Server-Sent Events) token supply data is in-flight from the backend. Before the first `__supply:CRIME` or `__supply:FRAUD` message arrives from `ws-subscriber`, the UI displays 1B tokens as the circulating supply.

```typescript
// Lines 19-22 in useTokenSupply.ts
const INITIAL_SUPPLY: Record<string, number> = {
  CRIME: 1_000_000_000,
  FRAUD: 1_000_000_000,
};
```

The fallback is used in two places:
- **SwapStatsBar.tsx:188** — Computes MCAP as `price * (supply[key] ?? 1_000_000_000)`
- **SwapStation.tsx:133** — Computes chart y-axis MCAP as `tokenSupply[tokenName] ?? 1_000_000_000`

---

## Impact Assessment

### Scope
- **Display-only** — affects MCAP calculations on the frontend only
- **Temporary** — fallback is used only during the brief window before SSE data arrives (typically <1-2s on page load)
- **No on-chain effect** — user balances, swap logic, and token minting/burning are unaffected

### Risk Level
**ACCEPTED_RISK** — The fallback value (1B) happens to equal TOTAL_SUPPLY for CRIME and FRAUD, so the display is correct once SSE data arrives. However:

1. **If actual supply drops below 1B** (e.g., Carnage burns tokens), the fallback will overstate MCAP during the SSE load window
2. **If token burns scale significantly** (e.g., Carnage accumulates more holdings), the discrepancy grows
3. **Load time depends on network conditions** — slow connections may display stale MCAP for 3-5s

### Current State
- CRIME supply = ~1B (no burns yet on v1.3 mainnet)
- FRAUD supply = ~1B (no burns yet on v1.3 mainnet)
- SSE updates arrive within 1-2s on typical connections
- The `loading` flag correctly indicates when supply data is in-flight (line 49)

---

## Verdict

**Status:** CONFIRMED, ACCEPTED_RISK

The hardcoded fallback is a known and acceptable trade-off:

✅ **Confirmed Present:**
- Fallback is explicitly defined and used as intended
- Code correctly indicates loading state

✅ **Accepted Risk:**
- Display inaccuracy is temporary and brief (< few seconds)
- Initial value happens to match actual supply (for now)
- No financial impact (display-only, not on-chain)
- MCAP is corrected automatically when SSE data arrives

---

## Recommendation

No action required at this phase. However:

1. **For future scaling** (when Carnage burns >10% of supply), consider:
   - Querying actual supply from `useProtocolState` on mount before rendering MCAP
   - Caching supply from the previous session in localStorage as a warm start

2. **Add comment** explaining the fallback strategy and its TTL (already done at line 18).

3. **Monitor in production** — if SSE consistently takes >2s to deliver first supply update, implement retry logic or fallback refresh.
