# H096: Anchor Decode No Bounds Check on Account Fields

## Verdict: CONFIRMED (HIGH)

---

## Evidence

### Location

**File:** `app/app/api/webhooks/helius/route.ts` (lines 589–593)

```ts
const program = programs[decodeInfo.programKey];
const rawBuffer = Buffer.from(item.rawAccountData.data, "base64");
const decoded = program.coder.accounts.decode(
  decodeInfo.accountType,
  rawBuffer,
);
```

### The Issue

After Anchor's `coder.accounts.decode()` deserializes the raw buffer into an account struct, **no semantic bounds validation is applied to numeric fields before they are stored in `protocolStore`**.

The `decode()` method performs only **structural deserialization** — it validates that field types fit their declared Borsh type (e.g., a `u64` field can hold the decoded bytes). It does NOT enforce business-logic constraints:

- Tax BPS fields (`crimeBuyTaxBps`, etc.) should be in `[0, 10000]`, but any `u64` value is accepted
- Pool reserves (`reserveA`, `reserveB`) should be positive, but zero or negative interpretations may bypass validation
- Enum variants like `cheapSide` should be `0` (CRIME) or `1` (FRAUD), but any `u8` value decodes without error
- Epoch numbers should be within protocol-defined ranges, but any `u32` is accepted

Once decoded, these unconstrained values are stored directly:

```ts
const normalized = anchorToJson(decoded, ...);
protocolStore.setAccountState(pubkey, {
  ...normalized,
  updatedAt: Date.now(),
});
```

No validation layer exists between decode and store.

### Relationship to Prior Findings

**H005 (Webhook Secret Compromise → Protocol State Injection)** and **H020 (Webhook-to-SSE Data Injection Chain)** both identify and exploit this exact gap:

- **H005, lines 37–43:** Demonstrates that an attacker-supplied `rawAccountData` with `crimeBuyTaxBps: 9999` decodes successfully without error and is stored verbatim.

- **H020, lines 48–64:** Confirms that Anchor's `decode()` is purely structural; an attacker who knows the on-chain layout can craft a valid base64 blob with extreme values that passes decode.

H096 is the **root cause** that both H005 and H020 exploit. Anchor decode itself has no semantic validation; the obligation to validate falls on the application layer.

---

## Impact

Without field bounds checks after decode, any of the following attack sequences become feasible (assuming attacker has `HELIUS_WEBHOOK_SECRET`):

| Field | Exploit | Consequence |
|-------|---------|-------------|
| `crimeBuyTaxBps` | Set to 0 or 10000 | Users see false tax rates; decision-making corrupted |
| `reserveA`, `reserveB` | Set to extreme values (e.g., `10^18`) | Quote engine computes phantom reserves; `minimumOutput` nullified |
| `cheapSide` | Set to invalid enum (e.g., `99`) | Enum decode may fail or produce undefined behavior |
| `currentEpoch` | Set to impossible value (e.g., `999999`) | Epoch display corrupted; bonding curve state invalid |

In combination with H005/H020's authentication bypass (if the webhook secret leaks), these unchecked values propagate directly to all connected SSE browsers with zero intermediate validation.

---

## Recommendation

### Required (pre-mainnet)

Add a validation layer immediately after Anchor decode, before writing to `protocolStore`. Define account-specific validators:

```ts
function validateDecodedAccount(label: string, decoded: Record<string, unknown>): boolean {
  switch (label) {
    case "EpochState":
      // All tax BPS fields must be in [0, 10000]
      const taxFields = ["crimeBuyTaxBps", "crimeSellTaxBps", "fraudBuyTaxBps", "fraudSellTaxBps"];
      return taxFields.every(
        (f) => typeof decoded[f] === "number" && decoded[f] >= 0 && decoded[f] <= 10_000
      );

    case "CarnageFundState":
      // All balance/count fields must be non-negative
      const numFields = ["solBalance", "crimeBalance", "fraudBalance", "executionCount"];
      return numFields.every(
        (f) => typeof decoded[f] === "number" && decoded[f] >= 0
      );

    case "StakePool":
      // All reserve/stake fields must be positive
      return (
        typeof decoded.totalStaked === "number" && decoded.totalStaked >= 0 &&
        typeof decoded.rewardPerShare === "bigint" && decoded.rewardPerShare >= 0n
      );

    case /^PoolState:/.test(label):
      // Both reserves must be positive
      return (
        typeof decoded.reserveA === "number" && decoded.reserveA > 0 &&
        typeof decoded.reserveB === "number" && decoded.reserveB > 0 &&
        typeof decoded.fee === "number" && decoded.fee >= 0 && decoded.fee <= 10_000
      );

    case /^CurveState:/.test(label):
      // Price, total_sold must be non-negative
      return (
        typeof decoded.currentPrice === "bigint" && decoded.currentPrice >= 0n &&
        typeof decoded.totalSold === "number" && decoded.totalSold >= 0
      );

    default:
      return true; // SystemAccount (CarnageSolVault) has no validation
  }
}
```

**Apply this check immediately after decode, before `setAccountState`:**

```ts
const decoded = program.coder.accounts.decode(decodeInfo.accountType, rawBuffer);

// NEW: Validate field ranges before storing
if (!validateDecodedAccount(label, decoded)) {
  console.error(`[webhook] Validation failed for ${label} at ${pubkey}. Invalid field ranges.`);
  // Log to Sentry for alerting
  // Do NOT write to protocolStore
  continue; // Skip this account entirely
}

const normalized = anchorToJson(decoded, ...);
protocolStore.setAccountState(pubkey, {...normalized, updatedAt: Date.now()});
```

This ensures that **no unconstrained data ever reaches the store**, regardless of whether H005/H020's webhook secret compromise occurs.

## Evolution

**Classification:** Root cause of H005 and H020.

H096 is the underlying technical vulnerability that makes the injection attacks in H005 and H020 feasible. While H005 and H020 focus on the authentication/injection path (what happens if the secret leaks), H096 identifies that even a valid secret should not permit arbitrary field values to reach the application state.

**Why H096 is separate from H005/H020:** H005 and H020 assume an attacker with the secret and focus on exploitation pathways. H096 is the **application design gap** that the attack exploits. Fixing H096 (bounds validation) is orthogonal to and independent of fixing H005 (authentication hardening); both must be done.

**Severity:** Downgraded from CRITICAL to HIGH because:
- Requires prerequisite secret compromise (H005/H020)
- On-chain `minimumOutput` enforcement provides some protection against the most severe outcomes
- Detection is possible via cross-verification against ws-subscriber state

Remains HIGH (not MEDIUM) because the validation gap is fundamental: any new account types added to the webhook handler inherit the same vulnerability until explicit validators are written.
