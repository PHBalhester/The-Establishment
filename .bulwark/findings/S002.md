# S002: State Injection + RPC Exhaustion → Coordinated MEV Amplification

## Verdict: CONFIRMED

---

## Constituent Findings

| Finding | Title | Severity | Status |
|---------|-------|----------|--------|
| H005 | Webhook Secret Compromise → Protocol State Injection | HIGH | CONFIRMED |
| H008 | RPC Proxy Batch Amplification → Helius Credit Exhaustion | HIGH | CONFIRMED |

---

## Combined Attack Path

H005 and H008 operate on different layers of the same user interaction: H005 corrupts the *input* to the swap quote engine (reserves and tax rates shown in the UI), while H008 destroys the *output channel* (the RPC proxy the user's TX travels through for submission and confirmation). Combined, they create a trap: the user sees a manipulated quote, signs a transaction with a nullified `minimumOutput`, and then submits through an RPC path the attacker controls the timing of.

**Prerequisites:**

- H005 requires: `HELIUS_WEBHOOK_SECRET` (obtained via H064 — mainnet secret committed in git; see S003)
- H008 requires: nothing — the RPC proxy batch endpoint is fully unauthenticated

Both prerequisites are independently satisfiable. H008 requires zero credentials.

**Phase 1 — State Poisoning (H005, active)**

Attacker POSTs a crafted webhook payload to `/api/webhooks/helius` with the stolen webhook secret. The payload injects a `PoolState` for the CRIME_SOL pool with `reserveA=10^18` (near-infinite SOL reserves), `reserveB=1`:

```json
{
  "rawAccountData": {
    "data": "<base64 PoolState: reserveA=10^18, reserveB=1>",
    "encoding": "base64"
  }
}
```

This propagates through `protocolStore` → SSE → all connected browser clients within milliseconds. Every user with an open swap panel now sees a near-zero output quote. Critically, `minimumOutput` computed in `useSwap` from these fake reserves is approximately 0. The user's transaction will be signed with `minimumOutput ≈ 0` — the on-chain slippage guard is nullified.

The rate limit of 120 req/min per IP gives the attacker enough throughput to sustain this injection continuously. The ws-subscriber overwrites injected state within 30–60s from the next legitimate Helius event; the attacker re-injects before that window closes.

**Phase 2 — RPC Exhaustion (H008, timed)**

Simultaneously, the attacker begins credit exhaustion against the RPC proxy:

```bash
# Sustained: 300 HTTP requests/min × 500 getAccountInfo each = 150,000 credits/min
for i in {1..300}; do
  curl -s -X POST https://<deployment>/api/rpc \
    -H "Content-Type: application/json" \
    -d "[$(printf '{"jsonrpc":"2.0","id":%d,"method":"getAccountInfo","params":["11111111111111111111111111111111",{"encoding":"base64"}]},' {1..500})]" &
done
```

The batch size limit does not exist (H008 confirmed). The rate limiter counts HTTP requests, not RPC method calls. Within minutes, Helius credits are exhausted for the API key that serves both the frontend proxy and the crank/ws-subscriber.

**Phase 3 — MEV Window**

When RPC credits are exhausted:
- The RPC proxy returns HTTP 429 for all user TX submissions
- The ws-subscriber loses its Helius WebSocket connection — protocolStore stops receiving legitimate updates
- Injected state from Phase 1 **persists indefinitely** in protocolStore (no TTL, no ws-subscriber correction)
- The crank cannot advance epochs or trigger Carnage

Users who have already computed a swap with `minimumOutput ≈ 0` from Phase 1, and whose browser has queued a TX, now submit via fallback RPC (if any) or retry once proxy recovers. When the proxy briefly recovers (or user switches to another RPC), their TX with `minimumOutput=0` lands on-chain. The attacker's MEV bot — monitoring the mempool on a separate, unexhausted RPC connection — sandwiches the transaction, extracting up to 100% of swap value because no slippage floor exists.

**Why the combination amplifies individual finding impact:**

H005 alone: the ws-subscriber overwrites injected state within 30–60s, limiting the injection window. Users may notice the anomalous quote and abort. The attacker must maintain continuous injection at 120 req/min.

H008 alone: credit exhaustion disrupts service, but user funds are not at risk — on-chain slippage guards remain intact.

H005 + H008: exhausting the RPC kills the ws-subscriber's correction mechanism. Injected state becomes *permanent* for the outage duration (minutes to hours until credits reset or plan upgraded). Every user who connected during the Phase 1 injection window has a poisoned quote for the entire outage. The exploitation window expands from 30–60s to the full outage duration.

## Combined Impact

| Dimension | H005 Alone | H008 Alone | Combined |
|-----------|-----------|-----------|---------|
| minimumOutput nullification | Yes, 30–60s window | No | Yes, duration of RPC outage (minutes to hours) |
| ws-subscriber correction | Clears injection within 60s | ws-subscriber loses connection | ws-subscriber cannot correct; injection persists |
| Crank liveness | Not affected | Disrupted | Disrupted (compound) |
| Affected user scope | All concurrent users | All users | All users, extended window |
| Financial impact | MEV extraction per TX during window | Forced plan upgrade | Scaled MEV extraction over outage duration |
| Detection difficulty | Difficult | Medium (Helius credit alerts) | Very difficult — two events appear unrelated |

**Severity assessment: CRITICAL in combination.**

Individual findings are both rated HIGH. The combination earns CRITICAL because:
1. The exploitation window for fund-impacting MEV extends from seconds to the full outage duration
2. The ws-subscriber — the primary mitigation for H005 — is disabled by H008
3. The combined attack requires coordination of two unauthenticated/low-barrier primitives (webhook injection + unauthenticated batch amplification)
4. The two attack channels produce correlated symptoms that appear unrelated in monitoring: "webhook data looks weird" and "RPC is slow" are not typically correlated by incident responders

**Protocol liveness impact:** With the crank unable to advance epochs (H008 kills its Helius connection), epoch tax rates freeze. If the attacker times the attack around an epoch boundary, tax rates that should change remain stale. This creates systematic mispricing of all swaps for the outage duration, independent of the MEV attack.

## Recommendation

The combination finding requires coordinated fixes at both layers. Fixing only one leaves the amplification path open.

**Priority 1 — Break the ws-subscriber dependency on the shared Helius key.**

The ws-subscriber uses `HELIUS_RPC_URL` (same API key as the frontend proxy). Credit exhaustion on the proxy key should not kill the ws-subscriber. Use a separate Helius API key for the server-side ws-subscriber and crank, isolated from the user-facing proxy key. If the proxy key is exhausted, the ws-subscriber continues correcting injected state.

**Priority 2 — Enforce batch size limit in the RPC proxy (H008 fix).**

```typescript
const MAX_BATCH_SIZE = 10;
if (requests.length > MAX_BATCH_SIZE) {
  return NextResponse.json({ error: "Batch too large" }, { status: 400 });
}
```

This alone closes the amplification vector. The attacker is limited to 300 × 10 = 3,000 RPC calls/min — below the threshold for rapid credit exhaustion.

**Priority 3 — Add semantic validation to the webhook decoder (H005 fix).**

Per H005's recommendation: validate `reserveA`, `reserveB`, and `*TaxBps` fields after Anchor decode. A reserve value of `10^18` lamports (1 billion SOL) is physically impossible given Solana's total supply. Reject and log without storing.

**Priority 4 — Add a TTL to protocolStore entries.**

If a protocol account has not received a legitimate ws-subscriber update for >90s, mark it stale and exclude it from the swap quote engine. This limits injection persistence to the ws-subscriber polling interval regardless of RPC status.

**Priority 5 — Separate rate-limit accounting for batch requests.**

Even without fixing batch sizes, charging the rate limit bucket once per RPC method call (not once per HTTP request) makes the 300 req/min limit meaningful against batch amplification.
