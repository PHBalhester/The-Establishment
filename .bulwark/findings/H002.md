# H002: Mainnet Crank Wallet Key in Working Tree

## Verdict: POTENTIAL

---

## Evidence

### Primary: `.env.mainnet` contains live mainnet private key — but is gitignored

**File:** `.env.mainnet` (working tree only, never committed)
**Line 88:**
```
WALLET_KEYPAIR=[REDACTED — mainnet crank wallet private key]
```

This 64-byte JSON array is the complete secret key for the mainnet crank wallet `F84XUxo5VM8FJZeGvC3CrHYwLzFod3ep57CULjZ4ZXc1` (confirmed via public key derivation and cross-reference with project memory). The key is present in plaintext in the working tree.

**Gitignore status confirmed:**
- `.gitignore` line 1-2: `.env` and `.env.mainnet` are explicitly gitignored
- `git ls-files .env.mainnet` returns 0 entries — file is NOT tracked
- `git log --all --full-history -- '.env.mainnet'` returns no output — file has NEVER been committed

**Git history search for keypair bytes:**
- `git log --all -S '[REDACTED-CRANK-KEY-PREFIX]'` — returns zero commits
- The specific keypair byte sequence has never appeared in any git commit

### Secondary: Co-located secrets in `.env.mainnet`

The same file also contains (all untracked, confirmed gitignored):
- `HELIUS_WEBHOOK_SECRET=[REDACTED-WEBHOOK-SECRET]`
- `HELIUS_API_KEY=[REDACTED-MAINNET-HELIUS-KEY]` (mainnet key)
- `CLUSTER_URL` containing the mainnet Helius API key in the query string
- `PDA_MANIFEST` JSON containing the Helius RPC URL with API key embedded

### Partial: Mainnet Helius API key HAS leaked into git history

While `.env.mainnet` itself is gitignored, the mainnet Helius API key (`[REDACTED-MAINNET-HELIUS-KEY]`) appears in **3 committed files**:

| Commit | File | Context |
|--------|------|---------|
| `40aa0d9` | `scripts/deploy/deployment-report.md` | Committed deployment report shows `Cluster: mainnet (https://mainnet.helius-rpc.com/?api-key=[REDACTED-MAINNET-KEY]-...)` |
| `0e8ff92` | `.planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md` | Planning doc lists Helius key in infrastructure table |
| `1f552da` | `.dbs/` audit files | DBS audit analysis mentions the key |

**The WALLET_KEYPAIR bytes themselves are NOT in git history.** The Helius API key IS.

### File permissions gap

`.env.mainnet` has world-readable permissions (`-rw-r--r--`, mode 0644), meaning any process running as a different user on the same machine can read the full private key:
```
Permissions: -rw-r--r-- Owner: mlbob Group: staff
```

## Attack Path

**Path A: Filesystem access (local machine)**
1. Attacker gains read access to `/Users/mlbob/Projects/Dr Fraudsworth/` (e.g., via malware, shared machine, compromised process running as different user)
2. Read `.env.mainnet` — no git required, file is world-readable (`0644`)
3. Extract `WALLET_KEYPAIR` byte array
4. Reconstruct keypair: `Keypair.fromSecretKey(new Uint8Array([REDACTED]))`
5. Sign arbitrary transactions as `F84XUxo5VM8FJZeGvC3CrHYwLzFod3ep57CULjZ4ZXc1`
6. Drain crank SOL balance

**Path B: Repository access (any git clone)**
- Does NOT expose WALLET_KEYPAIR — file was never committed. Path B is **not viable** for the private key.
- Does expose mainnet Helius API key via `scripts/deploy/deployment-report.md` and planning docs in git history.

**Path C: Helius API key from git history**
1. Clone repo (public or with read access)
2. `git log --all -S '[REDACTED-MAINNET-KEY]' --format='%H %s'` finds 3 commits
3. `git show 40aa0d9 -- scripts/deploy/deployment-report.md` reveals the mainnet Helius key
4. Use Helius API to: register shadow webhooks, delete production webhook, or exhaust RPC credits

**Path D: AI tool context (novel vector)**
- `.env.mainnet` is in the working directory. AI tools (Claude Code, GitHub Copilot) that scan working tree files could inadvertently include the key in context, logs, or telemetry.
- This is a structural risk that exists as long as the key is in the working tree in plaintext.

---

## Impact

**Crank wallet compromise (`F84XUxo5VM8FJZeGvC3CrHYwLzFod3ep57CULjZ4ZXc1`):**
- Direct SOL theft (crank wallet is funded, confirmed by project memory: "Crank wallet (mainnet): funded")
- Submit malicious epoch transitions (epoch transitions are permissionless — any signer can call `trigger_epoch_transition`, but controlling the crank timing gives MEV advantage and disrupts VRF flow)
- Note: epoch transitions are permissionless on-chain, so crank compromise does NOT give protocol-level control (confirmed in ARCHITECTURE.md: "Crank wallet compromise ≠ protocol compromise (permissionless)")

**Helius API key compromise (leaked in git):**
- Register shadow webhooks to receive all protocol event data
- Delete production webhook to blind the SSE/chart data pipeline
- Exhaust mainnet RPC credits

**Why POTENTIAL, not CONFIRMED:**
- The working tree is on a developer's local machine — not a public server
- `.env.mainnet` is properly gitignored and has NEVER been committed
- Exploitation requires local filesystem access (physical/remote machine compromise) or discovery of the file through an indirect channel (AI tool logs, backup systems, etc.)
- The threat is real but requires a pre-condition (local access) rather than being exploitable from the internet alone

---

## Recommendation

**Immediate (before mainnet launch):**

1. **Rotate the crank wallet** — generate a new keypair, transfer funds to it, and update Railway's `WALLET_KEYPAIR` env var. The current key at `.env.mainnet:88` should be considered compromised from this point forward given its exposure to working-tree risks.

2. **Restrict `.env.mainnet` file permissions** — change from `0644` to `0600`:
   ```bash
   chmod 600 .env.mainnet
   ```
   This prevents other local users/processes from reading it.

3. **Rotate mainnet Helius API key** — `[REDACTED-MAINNET-HELIUS-KEY]` is in git history in 3 committed files. Any repo clone exposes it. Rotate via Helius dashboard and update Railway env vars. Consider purging from git history with BFG Repo-Cleaner if the repo is or will be public.

4. **Remove WALLET_KEYPAIR from `.env.mainnet`** — once confirmed set on Railway, delete the key from the local `.env.mainnet` file. The env var belongs in Railway's secret store only, not on disk.

5. **Rotate `HELIUS_WEBHOOK_SECRET`** — `[REDACTED-WEBHOOK-SECRET]...` was committed in planning docs (`0e8ff92`). Although the planning doc only shows partial key in the commit message, the full value is in the committed SUMMARY file.

**Structural (medium-term):**

6. **Add `.env.mainnet` to `.gitignore` with comment** — already done for `env.mainnet` (confirmed), but add a comment documenting the policy:
   ```
   # Mainnet secrets — NEVER commit (contains WALLET_KEYPAIR and Helius keys)
   .env.mainnet
   ```

7. **Scrub API keys from deployed docs** — `scripts/deploy/deployment-report.md` is committed and contains the Helius RPC URL with API key. This is generated by deploy scripts — update the generator to redact keys from committed reports.

8. **Consider KMS for crank wallet** — long-term, migrate from raw WALLET_KEYPAIR env var to a KMS-backed signing service (AWS KMS, GCP Cloud HSM, or Turnkey). This removes the key from plaintext env vars entirely.
