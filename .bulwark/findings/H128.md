## Verification Details

### 1. Constant Definition
- Line 82: `MAX_TOPUP_LAMPORTS = 100_000_000` (0.1 SOL)
- Line 76: `VAULT_TOP_UP_LAMPORTS = 5_000_000` (0.005 SOL, typical top-up)
- Since 5_000_000 < 100_000_000, typical top-ups pass through unchanged

### 2. Cap Application Logic
**Lines 418-426:**
```typescript
if (vaultBalance < MIN_VAULT_BALANCE) {
  const requestedTopUp = VAULT_TOP_UP_LAMPORTS;
  const cappedTopUp = Math.min(requestedTopUp, MAX_TOPUP_LAMPORTS);
  if (requestedTopUp > MAX_TOPUP_LAMPORTS) {
    console.log(`WARNING: Top-up amount ${requestedTopUp} exceeds ceiling...`);
  }
  // ...
}
```

- Condition: Only cap if vault balance < ~2M lamports
- `Math.min()` ensures capped amount never exceeds 0.1 SOL
- Warning logged if requested > ceiling (defensive logging for off-by-one budget errors)

### 3. Cap Enforcement in Transaction
**Lines 437-443:**
```typescript

  SystemProgram.transfer({
    fromPubkey: provider.wallet.publicKey,
    toPubkey: vaultPubkey,
    lamports: cappedTopUp,  // ← CAPPED VALUE USED

```

- The transaction uses `cappedTopUp`, not raw `VAULT_TOP_UP_LAMPORTS`
- No bypass path exists

### 4. Spending Cap Integration
**Line 429:**
```typescript

  break; // Spending cap reached — halt

```

- The capped value is recorded against the hourly spending limit (H019)
- Dual-layer protection: per-operation cap + rolling-hour cap
