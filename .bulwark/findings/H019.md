# H019: gapFillCandles Memory Amplification — Unbounded Synthetic Candle Generation
## Verdict: CONFIRMED

---

## Evidence

### Source: `app/app/api/candles/route.ts`

**No rate limiting on the endpoint (line 186):**
```typescript
export async function GET(req: NextRequest): Promise<NextResponse> {
  // No checkRateLimit() call anywhere in this handler
```

Confirmed by grep: zero occurrences of `checkRateLimit` in `app/app/api/candles/`.

**Gap-fill loop is bounded only by `from`/`to` parameters — not by the DB row limit (lines 139–156):**
```typescript
for (let t = alignedStart; t <= rangeEnd; t += step) {
  if (rowIdx < rows.length && rows[rowIdx].openTime.getTime() === t) {
    filled.push(formatCandle(rows[rowIdx]));
    lastPrice = rows[rowIdx].close;
    rowIdx++;
  } else {
    // Gap: carry forward last known price
    filled.push({ time: ..., open: lastPrice, ... volume: 0, tradeCount: 0 });
  }
}
```

- `alignedStart` = `Math.floor(fromParam / stepSec) * stepSec` — user-controlled
- `rangeEnd` = `parseTimestamp(toParam)` — user-controlled
- `step` = `RESOLUTION_SECONDS[resolution] * 1000` — minimum is 60,000 ms (1m)
- The DB `.limit(limit)` cap (max 2000 rows, line 237) applies only to the rows fetched from Postgres; `gapFillCandles` is called afterward with the original `fromParam`/`toParam` values, so the DB row cap provides zero protection against the loop running millions of times.

**DB query is correctly limited — but gap-fill ignores that limit (lines 232–244):**
```typescript
const rows = await db
  .select()
  .from(candles)
  .where(and(...conditions))
  .orderBy(asc(candles.openTime))
  .limit(limit);          // max 2000 rows from DB

const result = gapfill
  ? gapFillCandles(rows, resolution, fromParam, toParam)  // ← fromParam/toParam drive loop, not row count
  : rows.map(formatCandle);
```

**`parseTimestamp` accepts Unix seconds including `0` (lines 172–180):**
```typescript
function parseTimestamp(value: string): number {
  const asNum = Number(value);
  if (!isNaN(asNum)) {
    return asNum * 1000;  // 0 * 1000 = 0 (Unix epoch, 1970-01-01)
  }
  return new Date(value).getTime();
}
```

**Middleware provides no protection (app/middleware.ts):**
The only middleware is a site-mode routing toggle. API routes explicitly bypass it (`!pathname.startsWith('/api')`). No rate limiting, no body validation.

**No Next.js framework-level body size limit relevant to GET:** The `next.config.ts` configures only CSP headers and image optimization. No per-route size caps or request throttling.

### Worst-Case Calculation (verified with Node.js)

| Scenario | Iterations | Heap (array objects ~200B ea.) | JSON serialization | Total per request |
|----------|------------|--------------------------------|--------------------|-------------------|
| `from=0&to=now` (1m, epoch to 2026-03-21) | ~29,567,521 | ~5,640 MB | ~2,820 MB | **~8,459 MB** |
| `from=<1yr ago>&to=now` (1m, 1 year) | ~525,601 | ~100 MB | ~50 MB | **~150 MB** |
| 10 concurrent × 1-year range | — | ~1,003 MB | ~501 MB | **~1.5 GB** |
| 10 concurrent × epoch range | — | ~56,396 MB | ~28,198 MB | **~84 GB** (OOM) |

The 1-year range case (~150 MB per request, ~1.5 GB for 10 concurrent) is the practical attack: it requires no special tooling and runs entirely within a browser-reachable GET request.

## Attack Path

```
Attacker crafts request:
  GET /api/candles?pool=<any_valid_pool>&resolution=1m&from=1704067200&to=1735689600
    (from = 2024-01-01, to = 2025-01-01, exactly 1 year)

Server execution:
  1. No rate limit check — request accepted unconditionally
  2. DB query: returns up to 2000 rows (legitimately capped)
  3. gapFillCandles() called with fromParam="1704067200", toParam="1735689600"
  4. step = 60,000 ms (1 minute)
  5. alignedStart = 1704067200 * 1000 (already aligned to 1m boundary)
  6. Loop runs 525,601 iterations, pushing one ~200-byte object per iteration
  7. filled[] array = ~100 MB on V8 heap
  8. NextResponse.json(result) serializes filled[] → ~50 MB JSON string
  9. Total heap spike: ~150 MB from this single request

At 10 concurrent requests from distinct IPs (or spoofed x-forwarded-for):
  - ~1.5 GB heap allocation
  - Railway single-process Node.js likely OOM-killed or swap-thrashed
  - All API routes become unavailable (single process, no recovery)

Maximum severity: from=0 (Unix epoch) pushes one request to ~8.4 GB — guaranteed OOM
on any cloud container (Railway free tier: 512 MB RAM; paid: typically 4-8 GB).
```

**Prerequisites:**
- Knowledge of one valid pool address (trivially obtained from `/api/sse/protocol` SSE stream, which is unauthenticated and broadcasts all pool PDAs)
- HTTP client capable of sending concurrent GET requests (curl, ab, wrk, browser DevTools)
- No authentication, no token, no account required

## Impact

**Availability: CRITICAL**

| Vector | Score |
|--------|-------|
| Attack complexity | LOW — single GET request, no auth |
| Attacker capability | ANY — public endpoint, no account required |
| Blast radius | FULL SERVICE — single Railway process, OOM kills all routes |
| Persistence | Until process restart — Railway auto-restarts but OOM loop is sustainable |
| Detectability | LOW — no logs distinguish this request from normal chart loads |

**Secondary effects:**
- DB connection pool exhausted by concurrent requests (10-connection max, no query timeout)
- Crank API calls affected if Next.js process is killed (crank runs separately, but webhook ingestion stops)
- SSE broadcast pipeline killed (all connected browsers lose real-time data)
- Repeated OOM cycling causes Railway restarts → downtime visible to all users

**Financial impact:** Repeated OOM restarts during active trading periods cause users to see stale prices and potentially submit transactions with outdated quotes. On-chain slippage enforcement is the backstop, but degraded UX during high-volatility epochs is directly harmful.

## Recommendation

**Fix 1 — Add a max range guard in gapFillCandles (immediate, low-risk):**
```typescript
const MAX_GAP_FILL_CANDLES = 10_000; // ~7 days at 1m resolution

function gapFillCandles(...) {
  // ADDED: reject before loop
  const totalSlots = Math.floor((rangeEnd - alignedStart) / step) + 1;
  if (totalSlots > MAX_GAP_FILL_CANDLES) {
    // Trim rangeEnd to fit within cap, or throw and let handler return 400
    throw new Error(`Gap-fill range too large: ${totalSlots} slots (max ${MAX_GAP_FILL_CANDLES})`);
  }
  // ... existing loop
}
```

**Fix 2 — Add rate limiting to the candles endpoint (immediate):**
```typescript
// In GET handler, before DB query:
const ip = getClientIp(req);
const rl = checkRateLimit(ip, CANDLES_RATE_LIMIT, "candles");
if (!rl.allowed) {
  return NextResponse.json({ error: "Too many requests" }, {
    status: 429,
    headers: { "Retry-After": String(rl.retryAfter) },
  });
}

// Add profile to rate-limit.ts:
export const CANDLES_RATE_LIMIT: RateLimitConfig = {
  windowMs: 60_000,
  maxRequests: 60, // 1/s per IP is generous for chart loading
};
```

**Fix 3 — Reject oversized from/to ranges at the parameter validation layer:**
```typescript
// In GET handler, after parsing fromParam/toParam:
if (fromParam && toParam) {
  const rangeMs = parseTimestamp(toParam) - parseTimestamp(fromParam);
  const maxRangeMs = MAX_GAP_FILL_CANDLES * RESOLUTION_SECONDS[resolution]! * 1000;
  if (rangeMs > maxRangeMs) {
    return NextResponse.json(
      { error: `Time range too large for resolution ${resolution}. Max: ${MAX_GAP_FILL_CANDLES} candles.` },
      { status: 400 }
    );
  }
}
```

**Fix 4 — Apply Node.js `--max-old-space-size` cap on Railway** to bound OOM blast radius and trigger controlled restart instead of full memory exhaustion.

Apply all four fixes. Fix 1 + Fix 3 stop the amplification. Fix 2 limits request rate. Fix 4 is a backstop. Priority order: Fix 3 (fastest, zero behavioral change for normal users) → Fix 2 → Fix 1 → Fix 4.

**Root cause:** Gap-fill logic was designed for TradingView chart UX (continuous time series) without considering adversarial input. The DB `limit` guard was added to the query but not propagated to the gap-fill range computation. Rate limiting was added reactively to high-risk endpoints (webhook, RPC, sol-price) but candles was treated as a low-cost read endpoint without accounting for the O(range/resolution) gap-fill complexity.
