# H089: SSE Initial-State No Freshness Indicator

**Severity:** Tier 3 (Informational)
**Component:** SSE Protocol Stream
**Affected Code:** `app/app/api/sse/protocol/route.ts:71-76`
**Status:** CONFIRMED

## Summary

New SSE clients receive an initial state snapshot containing all cached protocol account states, but the initial state payload includes no timestamp or slot indicator to help clients determine **how stale the snapshot is**. This makes it difficult for consumers to assess data freshness, particularly during SSE outages or network delays.

## Technical Details

### Initial State Send (Confirmed)
In `/app/app/api/sse/protocol/route.ts` at lines 71-76:

```typescript
const initialState = protocolStore.getAllAccountStates();
controller.enqueue(
  encoder.encode(
    `event: initial-state\ndata: ${JSON.stringify(initialState, bigintReplacer)}\n\n`,
  ),
);
```

The `initialState` is a raw snapshot from `protocolStore.getAllAccountStates()`, which returns a plain `Record<string, AccountState>` with no metadata wrapper.

### Data Flow
1. **Protocol Store** (`lib/protocol-store.ts:53-65`): When accounts update via webhook, `setAccountState()` adds an `updatedAt: Date.now()` timestamp **to individual account data**.
2. **Initial State** (route.ts:71): Retrieves all accounts as-is, preserving per-account `updatedAt` fields.
3. **SSE Stream**: Sends this map directly to the client without wrapping metadata.
4. **Client Consumption** (hooks/useProtocolState.ts:250-262): Receives the `initial-state` event and applies the raw account map to state.

### Issue: Missing Snapshot-Level Freshness

While **individual accounts** include `updatedAt` timestamps (set in webhook handler at `/app/app/api/webhooks/helius/route.ts:557, 605`), the initial state snapshot itself carries **no indicator** of:
- **When the snapshot was taken** (server time)
- **Current slot** (important for Solana state semantics)
- **Staleness relative to RPC head** (for critical use cases like arbitrage)

This is problematic because:
1. **SSE Outage Recovery**: If SSE is down for 10+ minutes then reconnects, the client receives a stale initial state with no way to know the snapshot age relative to current block time.
2. **Debugging**: Hard to reason about whether data is from 1 second ago or 5 minutes ago when inspecting logs.
3. **Account Staleness Assumptions**: If an account lacks an `updatedAt` (decode error or synthetic keys like `__slot`), there's no fallback timestamp.

## Risk Assessment

| Aspect | Risk | Details |
|--------|------|---------|
| **Data Correctness** | Low | Accounts have per-item timestamps; consumers can detect staleness if they check. |
| **Client Confusion** | Medium | Without snapshot-level metadata, debugging SSE outages is harder. UI may display old data. |
| **Arbitrage / Timing-Sensitive Logic** | Medium | Strategies that depend on knowing "how fresh is this state?" have no snapshot-level anchor. |
| **Observability** | Medium | Logs don't show when initial state was captured, complicating post-mortems. |

## Verdict

**CONFIRMED, POTENTIAL**

The issue is **confirmed** to exist: initial-state payloads include no snapshot timestamp or slot. The risk is **potential** rather than **critical** because:
1. Individual accounts already carry `updatedAt`, so savvy consumers can infer freshness.
2. For non-timing-sensitive UI updates (most use cases), this is informational.
3. However, it represents a **gap in observability** and could lead to subtle bugs in future time-aware features.

## Recommended Mitigation

### Option 1: Lightweight (Minimal Change)
Wrap the initial-state payload with metadata:

```typescript
// app/app/api/sse/protocol/route.ts:68-76
const now = Date.now();
const initialState = protocolStore.getAllAccountStates();
const wrappedState = {
  snapshotAt: now,
  accounts: initialState,
};
controller.enqueue(
  encoder.encode(
    `event: initial-state\ndata: ${JSON.stringify(wrappedState, bigintReplacer)}\n\n`,
  ),
);
```

Update client to unpack:
```typescript
// hooks/useProtocolState.ts:250-254
es.addEventListener("initial-state", (event) => {
  const payload = JSON.parse(event.data, bigintReviver);
  const data = payload.accounts ?? payload; // Backward compat
  setAccounts(data);
  // ...
});
```

### Option 2: Include RPC Slot (More Context)
Fetch current slot from RPC before sending:

```typescript
const connection = getConnection();
const slot = await connection.getSlot();
const wrappedState = {
  snapshotAt: Date.now(),
  snapshotSlot: slot,
  accounts: initialState,
};
```

**Trade-off:** Adds 1 RPC call per client connection (small overhead, easy to cache).

### Option 3: Server-Side Logging Only
Add server-side observability without changing client protocol:

```typescript
console.log(`[sse] Initial state sent to client (${Object.keys(initialState).length} accounts, age=${maxAccountAge}ms)`);
```

Useful for debugging but doesn't surface staleness to client.
