## Summary

`shared/constants.ts` is auto-generated from `deployments/devnet.json` via `generate-constants.ts` (called by `stage-4-infra.sh` → `deploy-all.sh`). ### PDAs in devnet.json Not Exported in constants.ts — By Design

The following PDAs exist in devnet.json but are absent from constants.ts:
`adminConfig`, `whitelistAuthority`, `carnageCrimeVault`, `carnageFraudVault`, `carnageSigner`, `stakingAuthority`, `vaultConfig`, `vaultCrime`, `vaultFraud`, `vaultProfit`.

These are **not frontend-consumed pre-computed constants**. Investigation confirms they are derived on-the-fly at call sites using `PublicKey.findProgramAddressSync` in `app/lib/swap/swap-builders.ts` (vaultConfig, vaultCrime, vaultFraud, vaultProfit) and resolved by Anchor's IDL machinery (carnageSigner, whitelistAuthority, adminConfig, stakingAuthority). Their absence from constants.ts is intentional and correct — they do not need to be pre-computed.

### Mainnet Addresses — Also Verified

`CLUSTER_CONFIG["mainnet-beta"]` in constants.ts was cross-checked against `deployments/mainnet.json` (generatedAt: `2026-03-15T20:24:09.638Z`). All 7 program IDs, 3 mints, 2 pool configs (with vaultA/vaultB), 3 core PDAs, 6 extended PDAs, 4 curve PDAs, treasury, and ALT match exactly between the two files.

### Sync Pipeline

The sync mechanism is fully automated:

1. `scripts/deploy/deploy-all.sh` Stage 4 calls `scripts/deploy/stage-4-infra.sh`
2. `stage-4-infra.sh` (line 97) calls: `npx tsx scripts/deploy/generate-constants.ts "$CLUSTER"`
3. `generate-constants.ts` reads `deployments/{cluster}.json` and overwrites `shared/constants.ts` entirely
4. The AUTO-GENERATED header in constants.ts (`// AUTO-GENERATED from deployments/devnet.json -- DO NOT EDIT MANUALLY`) prevents accidental manual edits
5. `sync-program-ids.ts` (invoked by `build.sh` at step [0a]) handles the inverse direction: keypair → Rust `declare_id!` + Anchor.toml + cross-program references. It does NOT touch constants.ts (different scope).

The two scripts have non-overlapping responsibilities and no circular dependency.

## Root Cause of H084 NOT_FIXED

H084 was marked NOT_FIXED because `shared/constants.ts` had been modified. That modification was the generate-constants.ts script itself writing the AUTO-GENERATED file with the correct Phase 102 devnet addresses after the clean redeploy. The modification was the fix, not further drift.

## Verdict

**RESOLVED.**

The constants drift vulnerability is remediated. `shared/constants.ts` is auto-generated from `deployments/devnet.json` and contains the correct addresses for the current deployment. The sync pipeline is part of `deploy-all.sh` and runs automatically on every deploy. No manual address management is required or possible (the AUTO-GENERATED header enforces this).

**Residual note (INV-16 from ARCHITECTURE.md):** The architecture document flags INV-16 as NOT ENFORCED: "Pre-computed PDA addresses match on-chain deployment — relies on manual `generate-constants.ts` run." This is now incorrect for post-Phase-95 deploys where `deploy-all.sh` is used. However, it would still apply if someone ran `anchor deploy` directly without going through `deploy-all.sh`. This is a process risk, not a code vulnerability, and is out of scope for this finding. The AUTO-GENERATED header and the fact that constants.ts is checked into git provides a detectable signal if drift occurs.
