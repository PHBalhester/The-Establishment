# H065: Health Endpoint as DoS Timing Oracle

## Verdict: CONFIRMED

## Executive Summary

The `/api/health` endpoint exposes real-time WebSocket and credit-counter state without authentication or rate limiting. An attacker can poll this endpoint to detect operational degradation windows (when `wsSubscriber.wsConnected = false` or `wsSubscriber.fallbackActive = true`), then time a DDoS attack to coincide with the monitoring window when the server is least able to respond. Additionally, the endpoint leaks server restart timestamps (`credits.startedAt`) and per-method RPC call volumes (`credits.methodCounts`), enabling infrastructure fingerprinting and activity-pattern inference.

**Severity: Tier 3 (MEDIUM) — Reconnaissance oracle for DoS timing, information disclosure, activity inference**

## Evidence

### 1. Endpoint Characteristics (lines 32–73)

```typescript
// app/app/api/health/route.ts:32-73
export async function GET() {
  let postgres = false;
  let solanaRpc = false;

  // Postgres check (can timeout or fail)
  try {
    await db.execute(sql`SELECT 1`);
    postgres = true;
  } catch (err) { ... }

  // Solana RPC check (can timeout or fail)
  try {
    const cachedSlot = protocolStore.getAccountState("__slot");
    if (cachedSlot && typeof cachedSlot.slot === "number") {
      solanaRpc = true;
    } else {
      const connection = getConnection();
      await connection.getSlot();
      solanaRpc = true;
    }
  } catch (err) { ... }

  const status = postgres && solanaRpc ? "ok" : "degraded";

  const wsSubscriber = getWsSubscriberStatus();   // Lines 63–64: NEW EXPOSURE
  const credits = creditCounter.getStats();

  return NextResponse.json({
    status,
    checks: { postgres, solanaRpc },
    wsSubscriber,    // ← Reveals WS health
    credits,         // ← Reveals activity metrics
    timestamp: new Date().toISOString(),
  });
}
```

**No authentication.** No rate limiting. Always returns HTTP 200.

### 4. No Rate Limiting

The endpoint is routed through Next.js's default handler with:
- No explicit rate limiting (verified against `app/lib/rate-limiter.ts`)
- No per-IP caps (unlike SSE which has 10/IP, 5000 global)
- No API key or Bearer token requirement

Polling `/api/health` at 1 req/sec incurs zero client-side cost and minimal server cost (Postgres `SELECT 1` is fast; RPC falls back to cached `__slot`).

## Attack Scenarios

### Scenario 1: Timed DDoS During WS Degradation (Primary Attack)

```
Attacker's bot:
1. Poll /api/health every 5 seconds
2. Detect: wsConnected = false && fallbackActive = true
3. Inference: "Server's monitoring pipeline has degraded to HTTP polling"
4. Action: Launch volumetric DDoS attack on /api/webhooks/helius
5. Expected outcome:
   - Server's ws-subscriber is already on HTTP fallback (no real-time WS data)
   - RPC polling loop is under network pressure
   - Crank's own RPC connection (separate) is unaffected
   - Frontend SSE clients receive stale state longer than normal
   - Attacker can submit toxic orders that exploit staleness

Detectability: LOW. A single /api/health poll request is indistinguishable from legitimate health monitoring.
```

### Scenario 2: Deployment Window Snipe (Secondary Attack)

```
Attacker's bot:
1. Poll /api/health once per minute
2. Detect: credits.startedAt changes to a recent timestamp
3. Inference: "Server just restarted. protocolStore is re-seeding."
4. Check: wsSubscriber.initialized = false
5. Action: Rapid-fire swap requests exploiting stale state
6. Expected outcome:
   - MinimumOutput is computed from stale reserves
   - User accepts the stale quote, submits TX
   - On-chain slippage floor (50%) catches catastrophic losses, but 5% MEV extraction succeeds

Detectability: LOW. Looks like a normal trading bot.
```

### Scenario 3: Activity-Pattern Inference

```
Attacker:
1. Poll /api/health every 10 seconds, log credits.methodCounts
2. Observe patterns:
   - getTokenSupply spikes → supply change (likely staking event)
   - getProgramAccounts spike → staker list changed
   - getAccountInfo volume spike → swap event volume
3. Inference: Free "market heat" oracle without spending Helius credits
4. Use: Decide whether to deploy capital, or front-run a known event

Detectability: VERY LOW. Passive monitoring only.
```

### Scenario 4: Infrastructure Fingerprinting

```
Attacker's reconnaissance:
1. Note exact server start time (credits.startedAt)
2. Combined with external metadata (Sentry release notes, GitHub commits), identify deployed version
3. Cross-reference with known CVEs in that version
4. Plan targeted exploit

Example: If startedAt = 2026-03-21T08:00:00Z and a GitHub Actions deploy ran at ~8:15 UTC, attacker knows this is a recent deploy with commit hash within that window.
```

## Recommendation

### Preferred Fix: Tiered Health Response

Split liveness probe (for Railway) from diagnostic health (for ops team):

```typescript
export async function GET(req: Request) {
  const authHeader = req.headers.get("authorization");
  const isMonitor = process.env.HEALTH_CHECK_SECRET &&
    authHeader === `Bearer ${process.env.HEALTH_CHECK_SECRET}`;

  // Liveness only — always returned
  const liveness = { status: "ok", timestamp: new Date().toISOString() };

  if (!isMonitor) {
    return NextResponse.json(liveness);
  }

  // Full diagnostics — authenticated callers only
  const [postgres, solanaRpc] = await runHealthChecks();
  return NextResponse.json({
    ...liveness,
    status: postgres && solanaRpc ? "ok" : "degraded",
    checks: { postgres, solanaRpc },
    wsSubscriber: getWsSubscriberStatus(),
    credits: creditCounter.getStats(),
  });
}
```

**Railway healthcheck URL:** `GET /api/health` (no auth needed, gets liveness-only response)
**Ops dashboard URL:** `GET /api/health` with `Authorization: Bearer <HEALTH_CHECK_SECRET>` (gets full diagnostics)

### Minimum Viable Fix

Remove `wsSubscriber` and `credits` from the public response immediately:

```typescript
return NextResponse.json({
  status,
  checks: { postgres, solanaRpc },
  timestamp: new Date().toISOString(),
});
```

This reduces the reconnaissance surface. The `checks` object alone is lower-value (H028 — already documented).

### Additional: Rate Limiting

Regardless of response body changes, add a rate limit to `/api/health`:

```typescript
// app/lib/rate-limiter.ts — add new limiter
const healthLimiter = new Ratelimit({
  redis: Redis.fromEnv(),
  analytics: true,
  prefix: "ratelimit:health",
  limiter: Ratelimit.slidingWindow(60, "1 m"), // 60 requests per minute per IP
});

// app/app/api/health/route.ts
export async function GET(req: Request) {
  const ip = req.headers.get("x-forwarded-for") || "unknown";
  const { success } = await healthLimiter.limit(ip);
  if (!success) {
    return NextResponse.json({ status: "rate-limited" }, { status: 429 });
  }
  // ... rest of handler
}
```
