# H102: Cross-Program Upgrade Cascade

**Tier:** 3 (Medium)
**Status:** ACCEPTED_RISK (Verified)
**Category:** Build Pipeline / Deployment
**Audit:** Bulwark #2 (2026-03-21)

---

## Finding

When a Solana program is upgraded (new keypair, new program ID), all cross-program references pointing to that program must be updated in dependent programs' source code. Failing to update all references causes:

1. **Silent runtime failures:** Transactions succeed on-chain but reference the old program ID
2. **State mismatches:** On-chain calls use wrong program constraints
3. **Multi-phase deployment cascades:** Updating program A requires redeploying programs B, C, D that reference it
4. **Human error risk:** Manual patching of 20+ locations is error-prone

This is especially critical because:
- Anchor's `declare_id!()` macro must match the deployed keypair
- Anchor.toml must match deployed IDs for RPC client generation
- Cross-program constants (e.g., `TAX_PROGRAM_ID` in AMM) must match live program IDs
- Test assertions hardcode program IDs and will fail on new deployments

     - `pubkey!()` macro functions (e.g., `tax_program_id()`)
     - `pubkey!()` macro consts (e.g., `TAX_PROGRAM_ID`)
     - Test assertion strings (e.g., `assert_eq!(id.to_string(), "OLD_ADDRESS")`)
     - (lines 428–477)
   - **Step 4:** Copy keypairs to target/deploy/ (lines 480–505)

3. **Cross-Program Registry (CROSS_REFS):**
   - Staking references: tax_program, epoch_program
   - Epoch references: tax_program, amm_program, staking_program
   - AMM references: tax_program
   - Test fixtures properly excluded from sync (lines 107–112)

4. **Integrated Into Build Pipeline:**
   - Called by `scripts/deploy/build.sh` as **step [0a/3]** before mint patching
   - Ensures all IDs are synchronized before compilation
   - Eliminates manual patching entirely
