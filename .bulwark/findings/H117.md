# Finding: H117 - SSE Connection Double-Release Race

## Status: NOT VULNERABLE
## Confidence Score: 9

## Executive Summary

H117 investigates a potential double-release race in the SSE protocol route handler where both the abort signal handler and the stream's `cancel()` method could fire, calling `release()` twice. The `released` boolean guard (lines 55-62) prevents both from executing `releaseConnection()` concurrently. The guard is correctly implemented and the double-release is **not exploitable**.

---

## Hypothesis Investigated

> Both abort handler and cancel() can fire, running release() twice.
> Verify the released boolean guard exists and is correctly scoped.

---

## Investigation Path

### Step 1: Inspect route handler cleanup architecture

**Location:** `app/app/api/sse/protocol/route.ts:50-125`

The route creates:
1. `released` boolean closure at line 55 (initialized to `false`)
2. `release()` function at lines 56-62 that:
   - Checks `if (!released)` before executing cleanup
   - Sets `released = true` (guards against re-entry)
   - Calls `releaseConnection(clientIp)` to decrement connection cap
   - Calls `cancelAutoRelease()` to clear the 30-minute auto-release timeout

Two code paths can call `release()`:
- **Path A (Abort handler)**: Lines 110-119 — registers `req.signal.addEventListener("abort", () => { release(); })`
- **Path B (Stream cancel)**: Lines 121-124 — stream's `cancel()` callback calls `release()`

### Step 2: Trace abort-to-release calling sequence

```typescript
// Line 110-119
req.signal.addEventListener("abort", () => {
  clearInterval(heartbeat);
  unsubscribe();
  release();  // <-- calls release()
  try {
    controller.close();
  } catch {
    // Controller already closed
  }
});
```

The abort handler is registered inside `ReadableStream.start()`. If the client disconnect fires an abort signal:
1. JavaScript event loop calls the abort listener
2. `release()` executes (line 113)
3. Sets `released = true` inside the closure
4. Calls `releaseConnection(clientIp)` once
5. Calls `cancelAutoRelease()` to clear the 30-min timer

### Step 3: Trace cancel-to-release calling sequence

```typescript
// Line 121-124
cancel() {
  // Stream cancelled by the runtime (e.g., client closed tab)
  release();
}
```

The `cancel()` callback is part of the `ReadableStream` constructor's `UnderlyingSource`. It fires when the stream is cancelled by the runtime (client disconnects, browser tab closes, fetch is aborted).

If `cancel()` fires:
1. JavaScript event loop calls `cancel()`
2. `release()` executes (line 123)
3. Sets `released = true` inside the closure
4. Calls `releaseConnection(clientIp)` once
5. Calls `cancelAutoRelease()` to clear the 30-min timer

### Step 4: Identify double-release race window

Can both Path A (abort handler) and Path B (cancel callback) fire before one completes?

**Node.js execution model:** Both event listeners are synchronous callbacks without internal `await` points. The abort signal handler and the stream cancel callback are both registered on the same stream object. When a client disconnects:

- **Scenario 1 (Abort fires first):**
  ```
  Tick N: abort event fires → executes abort listener → calls release()
          → released becomes true → releaseConnection() called ONCE
          → cancelAutoRelease() clears timer
  Tick N+ε: cancel() callback fires → calls release()
           → checks if (!released) → true? NO (already set to true)
           → skips releaseConnection() call (guarded by released check)
  ```

- **Scenario 2 (Cancel fires first):**
  ```
  Tick N: cancel() callback fires → calls release()
          → released becomes true → releaseConnection() called ONCE
          → cancelAutoRelease() clears timer
  Tick N+ε: abort event fires → calls release()
           → checks if (!released) → true? NO (already set to true)
           → skips releaseConnection() call (guarded by released check)
  ```

- **Scenario 3 (True simultaneous fire):**
  Both listeners fire in the same event loop tick. JavaScript executes one after the other (event listeners are serialized per event loop semantics). The first one sets `released = true`, the second checks and finds `released === true`, exits early.

### Step 5: Verify guard is correctly scoped

The `released` boolean is declared once at line 55:
```typescript
let released = false;
```

Both `release()` (lines 56-62) and the abort handler (line 113) and the cancel callback (line 123) share the **same closure** over this variable. There is **no separate `released` boolean** in the cancel callback or abort handler — they all read/write the same shared variable from the outer function scope.

Contrast with H027's finding about `scheduleAutoRelease()`, which has its **own separate `released` closure** inside the timeout. That is a different issue. Here, the route handler's guard is correct because all three code paths (release() itself, abort handler, cancel callback) share one `released` variable.

### Step 6: Verify releaseConnection() is idempotent for out-of-order calls

Even if `releaseConnection()` were called twice (which the guard prevents), would it be safe?

**Location:** `app/lib/sse-connections.ts:64-75` (inferred from H027)

From H027's investigation, `releaseConnection()` appears to:
1. Decrement `state.connections.get(ip)`
2. Decrement `state.globalCount`
3. Floor at 0 if negative

So even without the guard, double-release would floor both counters, not go negative. However, the guard exists and is correct, so this is moot.

### Step 7: Verify no interleaving between abort/cancel and release guards

The `released` variable is a boolean primitive. Reading and writing it is atomic in JavaScript (no multi-threaded race). Even if abort and cancel fire in the same tick:

```javascript
// Tick N: both abort and cancel execute (serialized)
// Execution order determined by event loop, not concurrent:
release();  // executes first (example)
  released = true;  // atomic write
// Execution resumes:
release();  // executes second (example)
  if (!released) {  // atomic read — sees true from first call
    // skipped
  }
```

There is no race because:
1. Boolean read/write is atomic in JavaScript
2. Event listeners are serialized in the event loop
3. The guard is checked before any state mutation (`releaseConnection()`, `cancelAutoRelease()`)

---

## Verdict

**NOT VULNERABLE**

The `released` boolean guard at lines 55-62 is correctly implemented, correctly scoped to all three code paths (direct call, abort handler, cancel callback), and correctly prevents double-release. Both Path A (abort) and Path B (cancel) can fire, but the guard ensures only the first one executes `releaseConnection()` and `cancelAutoRelease()`. The second call exits at the guard check.

## Attack Path Analysis

| Step | Action | Possible? | Notes |
|------|--------|-----------|-------|
| 1 | Client disconnects, abort event fires | Yes | Normal SSE client behavior |
| 2 | Abort handler calls release() | Yes | Code path exists (line 113) |
| 3 | release() sets released = true | Yes | First execution succeeds |
| 4 | Stream cancel callback fires | Yes | Can occur after abort or before it |
| 5 | cancel() calls release() again | Yes | Code path exists (line 123) |
| 6 | Second release() checks `if (!released)` | Yes | Guard check executes |
| 7 | Guard prevents releaseConnection() call | Yes | Guard is true, skips body |
| 8 | Double-release of global cap | No | Guard prevents second call |
| 9 | globalCount underflows | No | Even without guard, releaseConnection() floors at 0 |

**Conclusion:** No exploitable double-release exists. The guard is sufficient.
