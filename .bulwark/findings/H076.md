## Summary

The atomic multi-hop route builder uses `skipPreflight: true` when sending v0 VersionedTransactions
to the network. **Unlike the bonding curve case (H043), this usage IS documented.** However, the
documentation explicitly ties the workaround to a devnet-specific RPC issue ("Blockhash not found"
simulation rejection on v0 TXs). If this issue does not manifest on mainnet, the workaround should
be disabled to restore the safety net of preflight simulation.

## Verdict: ACCEPTED_RISK

The implementation is:
1. ✅ **Well-documented**: Lines 20–21, 374–375, and 395–396 all explain the rationale
2. ✅ **Correctly mitigated**: Lines 395–403 check `confirmation.err` to catch on-chain failures
3. ✅ **Specific to v0 TXs**: The issue documented in comments applies only to VersionedTransaction, not legacy TXs
4. ✅ **Devnet-scoped**: The blockhash simulation error is a known devnet RPC behavior

**Risk on mainnet**: If v0 TX blockhash simulation rejects do NOT occur on mainnet (likely), then
`skipPreflight: true` removes a useful safety net. Malformed instructions from a program upgrade
or stale PDA derivation would broadcast and burn tx fees rather than failing pre-submission.

## Risk Assessment

### What is lost by skipping simulation?

Preflight simulation (on-chain) catches before broadcasting:
- Anchor account constraint violations (wrong PDA, wrong owner)
- Transfer Hook account errors (missing extra_account_meta_list)
- Insufficient SOL balance for atomic swaps
- ATA creation failures (if ALT becomes stale)
- ComputeBudget instruction errors

**Without simulation**, a bug in any of these manifests as:
- TX fee burned (no refund on failure)
- `confirmation.err` shows the failure later (worse UX than preflight rejection)
- Network spam from failed TXs

### Why this is ACCEPTED_RISK

1. **Devnet-specific root cause**: The "Blockhash not found" error is a known Solana devnet RPC quirk
   affecting v0 TXs. Mainnet RPC does not exhibit this issue.

2. **Explicit mitigation in place**: `confirmation.err` is checked immediately after confirmation polling.
   On-chain failures are caught and surfaced to the user.

3. **Safe for Phase 102 (devnet-only)**: Devnet users tolerate slower error reporting and wasted fees
   as a tradeoff for atomicity.

4. **Mainnet action item (NOT urgent)**: Before mainnet, test whether v0 TX simulation on mainnet RPC
   (e.g., Triton, Helius mainnet endpoint) successfully simulates v0 TXs with blockhashes. If yes,
   change `skipPreflight: false` and remove the workaround comment.

### Severity (Tier 3 / LOW)

- **Devnet**: Acceptable tradeoff; atomicity is more valuable than early simulation feedback
- **Mainnet**: Unknown until tested; should be re-evaluated before launch

If mainnet RPC rejects v0 TX simulation (unlikely), this finding becomes REVISIT_MAINNET.

## Recommended Action

### Phase 102 (Current)
- **Status**: Keep `skipPreflight: true` for atomicity on devnet

### Phase 100+ (Mainnet preparation)
- **Test**: Verify whether mainnet RPC (Triton/Helius) simulates v0 TXs successfully
  ```typescript
  // Pseudocode: test v0 TX simulation on mainnet
  const messageV0 = new TransactionMessage({...}).compileToV0Message([alt]);
  const vTx = new VersionedTransaction(messageV0);
  try {
    const sim = await connection.simulateTransaction(vTx, [keypair]);
    console.log("v0 TX simulation succeeded on mainnet");
  } catch (err) {
    console.error("v0 TX simulation failed on mainnet:", err);
    // If this fails, blockhash bug exists on mainnet too; keep skipPreflight
  }
  ```
- **If simulation succeeds**: Change to `{ skipPreflight: false, preflightCommitment: 'confirmed' }`
- **If simulation fails**: Add a cluster-aware conditional:
  ```typescript
  const skipPreflight = connection.rpcEndpoint.includes('devnet') ||
                        connection.rpcEndpoint.includes('testnet');
  await wallet.sendTransaction(build.transaction, connection, { skipPreflight });
  ```
