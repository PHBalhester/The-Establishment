# H116: Protocol Store Accepts Arbitrary Keys

**Tier**: 3
**Status**: NOT VULNERABLE
**Target**: `app/lib/protocol-store.ts:53-65`
**CWE**: CWE-99 (Improper Control of Resource Identifiers) + CWE-639 (Authorization Bypass Through User-Controlled Key)

## Summary

H116 suspected that `setAccountState()` in `protocol-store.ts` accepts any string as a key with no validation, allowing an attacker to arbitrarily populate the in-memory cache via webhook manipulation.

**Verdict: NOT VULNERABLE** — The store itself accepts any key, but only the webhook handler (`handleAccountChanges`) and WebSocket subscriber (`ws-subscriber.ts`) call `setAccountState()`. Both enforce a whitelist (`KNOWN_PROTOCOL_ACCOUNTS`) that limits writes to exactly **8 real protocol PDAs + 4 synthetic keys = 12 total entries**.

## Analysis

### 1. Store Method Signature

File: `app/lib/protocol-store.ts:53-65`

```typescript
setAccountState(pubkey: string, data: AccountState): void {
  const serialized = JSON.stringify(data, bigintReplacer);
  this.accounts.set(pubkey, data);

  if (serialized === this.lastSerialized.get(pubkey)) return;
  this.lastSerialized.set(pubkey, serialized);

  sseManager.broadcast("protocol-update", {
    account: pubkey,
    data,
  });
}
```

**No validation of `pubkey` parameter.** The store trusts its callers and accepts any string.

### 3. WebSocket Subscriber Validation (SECONDARY GATE)

File: `app/lib/ws-subscriber.ts:67-84`


Plus hardcoded synthetic keys in initialization:
- `__supply:CRIME`
- `__supply:FRAUD`
- `__slot`
- `__staking:globalStats`

**Both lists are server-side constants,** not derived from user input. WS subscriber stores 8 + 4 = 12 keys total.

### 4. Frontend SSE (NO WRITE PATH)

The frontend receives SSE broadcasts via `useProtocolState` hook but has no mechanism to inject new keys. Reads only.

## Attack Vectors Considered & Dismissed

| Vector | Status | Reason |
|--------|--------|--------|
| Webhook forges unknown pubkey | Blocked | `KNOWN_PROTOCOL_ACCOUNTS` whitelist at line 540 |
| Webhook sender sends custom JSON key | Blocked | Helius envelope is signed; `item.account` field is Helius-controlled |
| Frontend calls setAccountState directly | Blocked | Method is server-only (Node.js runtime) |
| Frontend injects synthetic key via SSE | Blocked | SSE is read-only broadcast; no request path to store |
| WebSocket subscriber injects new key | Blocked | BATCH_ACCOUNTS is hardcoded server-side constant |
| Attacker hits /api/webhooks/helius with fake pubkey | Blocked | Line 540 rejection gate + line 542 `continue` statement |

## Bounds

| Boundary | Value | Source | Notes |
|----------|-------|--------|-------|
| Real protocol accounts | 8 | `KNOWN_PROTOCOL_ACCOUNTS` (webhook) | Fixed at compile-time |
| Synthetic keys | 4 | WS subscriber hardcoded | `__supply:*`, `__slot`, `__staking:globalStats` |
| **Total hard cap** | **12** | Sum of above | Zero accumulation path |
| Memory per entry | ~100–1000 bytes | Typical account state object | 12 entries = ~12 KB max |

## Design Quality

**Store method follows principle of least privilege:**
- `setAccountState()` has no input validation (trusts caller)
- Only 2 trusted callers with validation upstream:
  1. Webhook handler (validates against `KNOWN_PROTOCOL_ACCOUNTS`)
  2. WS subscriber (hardcoded keys)
- No public endpoint accepts arbitrary pubkeys

**This is correct separation-of-concerns:** validation at the input boundary (webhook), not at the storage layer.

## Conclusion

**NOT VULNERABLE.** The protocol store's parameterless `setAccountState()` is protected by hardcoded whitelists at both entry points (webhook + WS subscriber). An attacker cannot inject arbitrary keys. Memory growth is bounded at 12 entries (~12 KB total).
