# H099: Crank Spending Cap Uses Estimated Cost

**Severity**: Tier 3 (Acceptable Risk)
**Category**: Operational Risk (H-Risk)
**Status**: CONFIRMED, ACCEPTED_RISK
**Target**: `scripts/crank/crank-runner.ts:110, 500`

## Finding

The crank's hourly spending cap (`recordSpend()`) uses an estimated per-transaction cost (`ESTIMATED_TX_COST_LAMPORTS = 10_000`) rather than actual transaction fees from confirmed transactions.

### Evidence

**Line 110** (constant definition):
```typescript
const ESTIMATED_TX_COST_LAMPORTS = 10_000;
```
Comment indicates this is "Conservative per-transaction cost estimate: base fee (5000 lamports) + priority fee headroom. Imprecise but safe — the cap has 50x headroom."

**Line 500** (recording spend):
```typescript
const txCount = result.carnageTriggered ? 4 : 3;
if (!recordSpend(ESTIMATED_TX_COST_LAMPORTS * txCount)) {
  break; // Spending cap reached — halt
}
```

The crank sends multiple transactions (3–4 per cycle) via:
- `provider.sendAndConfirm()` (lines 444, 546, 358)
- `connection.sendRawTransaction()` (line 522)
- `sendV0Transaction()` (line 340 in vrf-flow.ts)

**None of these record actual transaction fees.** No call to `getTransaction()` or inspection of `TransactionResponse.meta.fee` is performed. The spending cap is based entirely on the fixed estimate.

### Risk Assessment

**ACCEPTED_RISK** — The design mitigates the concern:

1. **50x Safety Margin**: MAX_HOURLY_SPEND = 500M lamports (0.5 SOL). Normal usage is ~10K lamports/min (one TX every ~5 min at estimated 10K).
   - Estimate can undershoot actual fees by ~2–4x on mainnet (priority fees), but worst-case scenarios would still stay well within the 0.5 SOL budget.
   - Even if actual fees average 30K lamports/TX, a 0.5 SOL cap supports ~16 TXs/hour. The crank sends ~12 TXs/hour (2 per cycle × 6 cycles).

2. **Defensive Capture**: If actual fees drift significantly higher (e.g., mainnet congestion pushes fees to 100K/TX), the circuit breaker (CIRCUIT_BREAKER_THRESHOLD = 5 consecutive errors) will halt the crank before spending cap is reached, protecting the wallet from drain.

3. **No Incentive to Underestimate**: The estimate is intentionally conservative ("safe"), not adversarially low. Actual devnet fees are typically 5K–10K lamports, so the 10K estimate is realistic.

4. **Operational Observability**: The health endpoint (line 170) exposes `hourlySpendLamports`, allowing external monitoring of actual spend. A separate audit process can compare recorded vs. reported spend and flag drift.

## Why Not CONFIRMED?

Using estimates instead of actuals is a **design choice, not a vulnerability**. The budget has sufficient headroom that inaccuracy doesn't create exploitable conditions. The estimate is (1) intentionally conservative, (2) backed by a large safety margin, and (3) monitored via health checks.

Fixing this would require fetching actual fees post-TX (via `getTransaction(sig)`), which adds:
- 1 additional RPC call per TX (cost: ~100 credits/TX on Helius)
- Latency (sequential vs. bundled fee lookups)

The 50x headroom makes this optimization unnecessary.

## Recommendation

**No action required.** The spending cap's primary goal — prevent crank wallet drain due to software bugs — is achieved through:
- Conservative estimate
- Large safety margin (50x)
- Circuit breaker (5 consecutive errors)
- Health endpoint (external monitoring)

If mainnet fees exceed 200K lamports during sustained congestion, increase MAX_HOURLY_SPEND_LAMPORTS or lower MIN_TOPUP_LAMPORTS.

---

**Verdict**: CONFIRMED (estimated, not actual), ACCEPTED_RISK (50x headroom + circuit breaker)
