# H078: ALT Cache Never Invalidated

**Severity**: Tier 3 (Potential Risk)
**Target**: `app/lib/swap/multi-hop-builder.ts:261`
**Module**: Multi-Hop Swap Builder
**Status**: CONFIRMED

---

## Summary

The Address Lookup Table (ALT) is cached at module-level with no invalidation logic. If the protocol's ALT is extended with new addresses on-chain, the client-side cached copy becomes stale, causing transaction construction failures or use of outdated account references.

---

## Vulnerability Details

### Root Cause

**File**: `/Users/mlbob/Projects/Dr Fraudsworth/app/lib/swap/multi-hop-builder.ts`

```typescript
// Line 260-261: Module-level cache persists for the lifetime of the process
let cachedALT: AddressLookupTableAccount | null = null;

async function fetchProtocolALT(
  connection: Connection,
): Promise<AddressLookupTableAccount> {
  if (cachedALT) return cachedALT;  // <-- Returns stale cache

  const altAccount = await connection.getAddressLookupTable(PROTOCOL_ALT);
  if (!altAccount.value) {
    throw new Error(
      `Protocol ALT ${PROTOCOL_ALT.toBase58()} not found on-chain`,
    );
  }

  cachedALT = altAccount.value;  // <-- Set once, never invalidated
  return cachedALT;
}
```

The `cachedALT` variable is set on first call (line 275) and reused for all subsequent swaps without checking:
- Whether the on-chain ALT has been extended with new addresses
- Whether the cached version matches the current on-chain state
- Any TTL or slot-based expiration

### Impact Scenario

1. **Initial state**: Protocol ALT has 46 addresses. Client fetches and caches it.
2. **Protocol updates**: On-chain ALT extended to 50 addresses (e.g., new pool added).
3. **User attempts swap**: Client uses cached 46-address version, omits the 2 new accounts.
4. **Transaction fails**: "AccountNotEnoughKeys" or missing account error when instruction references account #47.

### Chain of Events

- **On-chain ALT extension** (via upgrade authority) adds new pool mints or PDAs
- **Multiple web clients** have cached older ALT state
- **Swaps referencing new pools** fail with AccountNotFound or InvalidAccountData
- **User experience degradation**: Swaps error until client redeployment + cache clear

## Evidence

**Codebase Analysis**:

1. **No slot-based expiration**: Cache is never cleared based on slot height or time
2. **No version tracking**: No way to detect if on-chain ALT state changed
3. **No invalidation API**: No public function to clear cache (e.g., `clearALTCache()`)
4. **Singleton pattern**: Single `cachedALT` variable applies to ALL swap requests

**Configuration Source** (`shared/constants.ts:421`):
```typescript
// Source: deployments/devnet.json alt
export const PROTOCOL_ALT = new PublicKey("FwAetEADes6Q19naJQ5eXBet9M5uVstAhjtvwnHRbMFL");
```

The ALT address is static, but its on-chain account state (address list) can change.

## Verdict

**CONFIRMED** — The vulnerability is real and present:

- ✅ Module-level `cachedALT` variable confirmed at line 261
- ✅ No invalidation logic exists in the codebase
- ✅ No TTL, slot-based expiry, or version tracking
- ✅ Cache persists for the entire process lifetime (Next.js runtime or browser session)

## Recommended Fixes

### Option 1: Slot-Based Expiry (Recommended)

Add slot height tracking and periodically refresh:

```typescript
interface ALTCacheEntry {
  alt: AddressLookupTableAccount;
  slotFetched: number;
}

let cachedALT: ALTCacheEntry | null = null;
const CACHE_EXPIRY_SLOTS = 300; // ~2 min at 2 slots/sec

async function fetchProtocolALT(
  connection: Connection,
): Promise<AddressLookupTableAccount> {
  const currentSlot = await connection.getSlot();

  if (cachedALT && currentSlot - cachedALT.slotFetched < CACHE_EXPIRY_SLOTS) {
    return cachedALT.alt;
  }

  const altAccount = await connection.getAddressLookupTable(PROTOCOL_ALT);
  if (!altAccount.value) {
    throw new Error(`Protocol ALT ${PROTOCOL_ALT.toBase58()} not found on-chain`);
  }

  cachedALT = { alt: altAccount.value, slotFetched: currentSlot };
  return altAccount.value;
}
```

### Option 2: Hash-Based Verification

Store and verify the hash of address list:

```typescript
interface ALTCacheEntry {
  alt: AddressLookupTableAccount;
  addressHash: string;
}

// Compute SHA256 of address list on fetch
// On subsequent calls, fetch fresh and compare hash
// If hash differs, replace cache
```

### Option 3: Always Fetch on Protocol Update

Provide invalidation hook for deployment/admin actions:

```typescript
export function invalidateALTCache() {
  cachedALT = null;
}

// Call on-chain after any ALT extension
```
