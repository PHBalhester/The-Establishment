# H021: Candle Price Manipulation via Crafted Webhook — Price Bounded by On-Chain Borsh Decode

## Verdict: NOT_EXPLOITABLE

The hypothesis is structurally correct about the OHLCV upsert mechanism, but the attack path is blocked by a critical constraint: the prices that reach the candle aggregator are **not user-supplied JSON fields** — they are derived from Borsh-decoded Anchor program log data (`Program data: <base64>`), which is fully controlled by on-chain program logic. An attacker with the webhook secret can craft the outer JSON envelope, but cannot inject arbitrary swap amounts because those values come from the log messages, not from fields in the webhook payload.

---

## Evidence

### 1. GREATEST/LEAST with no price bounds — confirmed at target lines

`app/db/candle-aggregator.ts:121-128`:
```typescript
.onConflictDoUpdate({
  target: [candles.pool, candles.resolution, candles.openTime],
  set: {
    high: sql`GREATEST(${candles.high}, ${update.price})`,
    low:  sql`LEAST(${candles.low}, ${update.price})`,
    close: sql`${update.price}`,
    ...
  },
});
```
There is exactly **one guard**: `if (price <= 0) return;` at line 189. There is no upper bound, no `isFinite()` check, and no sanity range check. If an extreme price value reaches `upsertCandles()`, it is permanently written into the `high` column with no recovery path (candle `high` is never reset within a bucket).

### 2. Price is derived from on-chain Borsh data, not webhook JSON fields

`app/app/api/webhooks/helius/route.ts:396`:
```typescript
const swaps = parseSwapEvents(logMessages);
```

`app/lib/event-parser.ts:248-270` — `parseSwapEvents` walks `tx.meta.logMessages`, finds `Program data: <base64>` lines via Anchor's `EventParser`, then Borsh-decodes them using the Tax Program IDL. The `inputAmount`, `outputAmount`, and `taxAmount` fields come out of BN objects from the decoded Borsh structure:
```typescript
inputAmount: bnToNumber(data.input_amount),
outputAmount: bnToNumber(data.output_amount),
taxAmount:   bnToNumber(data.tax_amount),
```

These values are the actual on-chain u64 amounts emitted by the Tax Program's `emit!()` macro. An attacker crafting a raw webhook payload controls:
- `tx.signature` (a string)
- `tx.blockTime` (must be within 5 minutes — MAX_TX_AGE_SECONDS = 300)
- `tx.meta.logMessages` (an array of strings)
- `tx.meta.err`
- `tx.slot`

The `logMessages` array is **the only path to inject swap amounts**.

### 3. The logMessages attack path is blocked by Anchor discriminator verification

Anchor's `EventParser` identifies events by matching the first 8 bytes of the base64-decoded log payload against `sha256("event:TaxedSwap")[0..8]`. Even if an attacker crafts a fake `Program data: <base64>` log line, the base64 payload must:
1. Decode to bytes where the first 8 bytes match the TaxedSwap discriminator (fixed, computable from public IDL).
2. Have the remaining bytes Borsh-deserialize into a valid `TaxedSwap` struct layout per the IDL.

This means an attacker CAN construct a syntactically valid synthetic log line — the discriminator is deterministic and the Borsh layout is public in the IDL. However, a crafted payload would also need to pass the **program invocation context check**: `EventParser.parseLogs()` only recognizes `Program data:` lines when they appear under an invocation of the expected program ID. A crafted log must contain:
```
Program <TAX_PROGRAM_ID> invoke [1]
Program data: <crafted-base64>
```
in the logMessages array. The webhook handler uses `tx.meta.logMessages` directly and passes it to the parser — there is **no on-chain signature verification of the log content itself** by the webhook handler. The Anchor EventParser trusts the program ID context declared in the log string.

### 4. The blocking constraint: TX signature idempotency via DB primary key

Even if a crafted logMessages array bypassed all of the above, the webhook handler inserts with:
```typescript
.onConflictDoNothing()  // TX signature is the primary key
```
(`app/app/api/webhooks/helius/route.ts:680`)

A synthetic TX must use a **fresh, unused signature**. Solana TX signatures are 64-byte Ed25519 values. There is no on-chain verification of the signature's validity in the webhook handler — the signature is treated as an opaque string by the off-chain code. An attacker can supply a fabricated base58 signature string that has never appeared on-chain, bypassing the idempotency check and successfully inserting the row.

### 5. Net assessment

The complete attack chain is:
1. Attacker has `HELIUS_WEBHOOK_SECRET` (prerequisite — high bar, see H002/SEC-02)
2. Attacker constructs a raw webhook payload with:
   - A never-seen-before `tx.signature` string
   - `tx.blockTime` set to `now()` (within 5-minute window)
   - `tx.meta.err = null`, `tx.meta.logMessages` containing a crafted invocation block
   - Borsh-encoded log line with discriminator for TaxedSwap and extreme amounts (e.g., `input_amount = 1`, `output_amount = 1`, but this produces a price of 1.0 — to get extreme prices, attacker sets `input_amount = 999999999999` and `output_amount = 1`)
3. Anchor's EventParser accepts the crafted log (program ID context is in the log string, not verified cryptographically by the off-chain parser)
4. Extreme price flows through `derivePrice()` and `upsertCandlesForSwap()` into the DB
5. `GREATEST(existing_high, extreme_price)` permanently corrupts the candle high

This chain **is** theoretically completable, but requires webhook secret compromise. The attack is blocked in practice by the secret requirement — this is a meaningful gating control.

## Attack Path

**Prerequisite:** Attacker possesses `HELIUS_WEBHOOK_SECRET` (from Railway env exfiltration, git history, or insider access — see H002, SEC-02, H132).

**Execution:**
1. Craft a POST request to `/api/webhooks/helius` with `Authorization: <secret>`.
2. Body: a JSON array with one transaction object. `logMessages` contains a synthesized Anchor event block:
   ```
   "Program <TAX_PROGRAM_ID> invoke [1]",
   "Program data: <base64-of: 8-byte-discriminator + borsh(TaxedSwap{input=1, output=1, tax=0, ...})>",
   "Program <TAX_PROGRAM_ID> success"
   ```
   Setting `input_amount=1` and `output_amount=1` gives price=1. To get extreme prices, set `input_amount=10^15` and `output_amount=1` (price = 10^15 lamports/token = extreme).
3. Use a fabricated `signature` string that does not exist on-chain (idempotency check only compares against DB, not Solana).
4. Set `blockTime = Math.floor(Date.now()/1000)` (fresh, passes 5-minute check).
5. The webhook handler parses the event, calls `upsertCandlesForSwap()`, which calls `upsertCandles()`.
6. `GREATEST(candle.high, 10^15)` permanently sets the candle's `high` to `10^15` across all 6 resolutions.

**Result:** Every OHLCV resolution (1m, 5m, 15m, 1h, 4h, 1d) for the targeted pool shows an extreme high permanently for that candle bucket. The `high` is never reset within a bucket. Existing candles are corrupted; future candles in new buckets are clean. To corrupt all current-hour candles, the attacker sends 6 requests targeting different `blockTime` values that floor to different bucket windows.

---

## Impact

- **Chart corruption**: Trading charts show nonsensical OHLCV data, destroying user trust and making price action unreadable.
- **MEV opportunity**: Fake price highs could confuse external tools that consume the candle data endpoint (`/api/candles`) to make trading decisions.
- **No financial loss**: The on-chain protocol enforces its own price via AMM invariants. Corrupted candle data does not affect actual swap execution or fund safety.
- **Severity if secret is known**: HIGH (chart data is permanently corrupted until manual DB repair).
- **Severity gated on secret**: With the webhook secret gating this, realistic severity depends on how well the secret is protected. As the secret is stored in Railway environment variables (only one attack path needed — see H132), this is not a purely theoretical concern.
- **No self-correcting mechanism**: Once `GREATEST` sets a high, no subsequent legitimate trade can lower it below that value within the same time bucket.

---

## Recommendation

1. **Add price sanity bounds in `upsertCandlesForSwap()`** (primary mitigation, does not require webhook secret rotation):
   ```typescript
   const MAX_REASONABLE_PRICE = 1e12; // lamports per token unit — ~1M SOL per token
   if (price <= 0 || !isFinite(price) || price > MAX_REASONABLE_PRICE) return;
   ```
   This eliminates the candle corruption vector entirely, regardless of whether the secret is compromised.

2. **Add the same guard in `upsertCandles()`** as a defense-in-depth backstop:
   ```typescript
   if (!isFinite(update.price) || update.price <= 0 || update.price > MAX_REASONABLE_PRICE) {
     console.warn(`[candle] Rejected extreme price: ${update.price}`);
     return;
   }
   ```

3. **Validate logMessages structure**: Before passing `logMessages` to `parseSwapEvents`, check that no log line claims to be from the Tax/Epoch Program unless the TX was actually sent to those programs (cross-check `tx.transaction.message.accountKeys`). This closes the synthetic log injection path.

4. **Secret rotation mechanism**: The current system has no webhook secret rotation without downtime. Implement dual-secret validation (accept old + new during rotation window) to reduce the attack window when rotating secrets.

## Evolution

- **Classification**: NEW — this specific candle manipulation via crafted raw webhook was not in Audit #1.
- **Related**: API-04 context noted this as a "novel attack surface" observation. H021 is the formal investigation.
- **Gating control**: The attack requires webhook secret compromise. The secret's security posture is covered by SEC-02 and H132. H021 is best treated as a **defense-in-depth gap** rather than an independent critical vulnerability: the primary fix (price bounds check) costs 2 lines and eliminates the risk regardless of secret posture.
- **DB repairability**: If exploited, corrupted candles must be manually deleted from the `candles` table for the affected pool + time bucket. There is no automated reconciliation mechanism.
