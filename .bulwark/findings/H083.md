# H083: Patch-Mint Trust Amplifier (RECHECK H021)

**Severity:** Tier 3 (RECHECK)
**Category:** Build & Deployment Integrity
**Target:** `generate-constants.ts`
**Status:** CONFIRMED, ACCEPTED_RISK

---

## Summary

H083 investigates whether `sync-program-ids.ts` (which patches program IDs from raw keypair JSON) creates a trust amplification vulnerability that undermines the integrity of `generate-constants.ts`. The concern: if a keypair is compromised or manipulated, does the patch propagate undetected into compiled constants?

**Verdict:** CONFIRMED vulnerability exists, but ACCEPTED_RISK due to control architecture.

---

## Details

### Root Cause: Multi-Source Program ID Chain

The program ID trust model has **three independent sources** that must stay synchronized:

1. **Keypair Files** (`keypairs/*.json`) — canonical source of truth
   - `sync-program-ids.ts` reads these and patches source code
   - Also copied to `target/deploy/` for `anchor build`

2. **Anchor.toml** (`[programs.devnet]`, `[programs.localnet]`)
   - Patched by `sync-program-ids.ts` step 2
   - Also read by `generate-deployment-json.ts` as **primary source for deployment.json**

3. **Rust declare_id! macros** (each program's `lib.rs`)
   - Patched by `sync-program-ids.ts` step 1
   - Embedded in on-chain bytecode during `anchor build`

4. **generate-constants.ts**
   - Reads program IDs from `deployments/{cluster}.json`
   - Which gets program IDs from `Anchor.toml` (step 2 of sync-program-ids)
   - Final constants exported to frontend

### Trust Amplification Path

```
keypairs/*.json (compromise source)
    ↓ (sync-program-ids reads + patches)
    → Anchor.toml [programs.devnet]
    → target/deploy/ (used by anchor build)
    → declare_id! embedded in bytecode

ALSO:
    → Anchor.toml [programs.devnet]
    → generate-deployment-json reads Anchor.toml
    → deployments/{cluster}.json
    → generate-constants.ts reads deployment.json
    → shared/constants.ts (frontend)
```

### Vulnerability: Undetected Patch Propagation

If a keypair JSON file is **silently modified** (e.g., via supply chain attack, storage compromise, or build system hijacking):

1. `sync-program-ids.ts` derives the incorrect public key
2. Patches **all 20+ locations** in the codebase (Anchor.toml, declare_id!, cross-refs, target/deploy/)
3. `anchor build` compiles with the wrong program ID in bytecode
4. `generate-deployment-json.ts` reads the poisoned Anchor.toml
5. `generate-constants.ts` pulls poisoned program IDs from deployment.json
6. Frontend receives wrong program ID and **silently sends transactions to attacker-controlled program**

**No integrity check, no warning, no hash verification.**

## Attack Scenarios

### Scenario 1: Devnet Compromise (Medium Risk)
- Attacker modifies `keypairs/tax-program-keypair.json` locally
- Re-runs `build.sh` and `initialize.ts`
- Entire devnet stack now calls attacker's tax program
- Testnet credentials, balances, and state sent to attacker
- **Detection:** Manual keypair inspection (brittle)

### Scenario 2: CI/CD Supply Chain (High Risk)
- Build pipeline runs `sync-program-ids.ts` on untrusted agent
- Agent fetches keypairs from Git LFS, artifact cache, or stolen secrets
- Attacker injects modified keypair before patch step
- Mainnet deployment binaries and constants contain wrong program IDs
- **Detection:** Post-deploy on-chain verification (too late)

### Scenario 3: Mint Keypairs (Covered by H021)
- H021 (referenced in title) identified similar issue with mint keypairs
- `initialize.ts` persists mint keypairs to `scripts/deploy/mint-keypairs/`
- `generate-deployment-json.ts` reads mint keypairs (lines 86-90)
- Same trust gap: compromised mint keypair → poisoned mints in constants
- **H083 = generalized H021 to program IDs**

## Current Safeguards (Why ACCEPTED_RISK)

### 1. **Human Review at Key Points**
- Keypairs live in `keypairs/*.json` (git-ignored)
- Typically generated once, used for entire cluster lifetime
- Deployer can manually inspect before committing
- Anchor.toml is reviewed before merge (visible in diffs)

### 2. **Local Development Workflow**
- Developers build locally with `./scripts/deploy/build.sh`
- Full control over their machine and keypair sources
- Easy to verify: `solana address -k keypairs/amm-keypair.json`

### 3. **Deterministic Program IDs (Intra-Deploy Consistency)**
- Program IDs are derived from keypairs, not random
- Same keypair always produces same program ID
- On-chain verification can cross-check: `solana program show <ID>`

### 4. **Mainnet Vanity Addresses** (Phase 95+)
- Mainnet program keypairs checked for vanity properties
- Example: `AMM=5JsS*`, `Hook=CiQP*`, `Tax=43fZ*`
- Visual mismatch would catch obvious bit-flips
- **But:** Doesn't defend against complete keypair replacement

### 5. **Deployment Pipeline Steps** (Sequential Guard)
- `build.sh` [0] = `sync-program-ids.ts` (patches source + Anchor.toml)
- `build.sh` [1] = `anchor build` (compiles with patched IDs)
- `deploy.sh` [2] = Deploys programs (uses target/deploy/ keypairs)
- `initialize.ts` [3] = Generates deployment.json from Anchor.toml

Sequential pipeline means attacker must compromise **both**:
- Keypair files (for sync step)
- **AND** Anchor.toml read by generate-deployment-json (for constants step)

Modifying only keypair doesn't poison constants; modifying only Anchor.toml breaks declare_id! macro.

## Technical Assessment

### Code Review: sync-program-ids.ts (lines 206–527)

**Positive Controls:**
- Keypairs loaded via `Keypair.fromSecretKey(new Uint8Array(secretKey))` — standard Solana SDK
- Base58 address derivation (`kp.publicKey.toBase58()`) — deterministic
- Regex patterns for patching are precise and scoped
- File cache prevents double-reads
- Changes only written if content differs (line 513)

**Missing Controls:**
- ✗ No hash verification of keypair files before reading
- ✗ No checksum of program IDs against known-good values
- ✗ No warning if keypair file modification time is recent
- ✗ No comparison with previous deployment state (e.g., last-known program IDs)
- ✗ No out-of-band verification source (e.g., on-chain query of deployed program)

### Code Review: generate-constants.ts (lines 1–71)

**Positive Controls:**
- Reads from `deployments/{cluster}.json` (generated after sync step)
- Validates schema at deployment-schema.ts line 176–260
- Validates base58 addresses (lines 156–163)
- Clear header: "AUTO-GENERATED" warns not to edit manually

**Missing Controls:**
- ✗ No verification that program IDs in deployment.json match on-chain state
- ✗ No checksum against Anchor.toml to detect drift
- ✗ No warning if deployment.json is stale (e.g., >1 hour old)
- ✗ No fallback consistency check (e.g., derived PDA matches expected)

## Remediation Options (Not Implemented)

### Option A: Keypair Hash Verification (Medium Effort)
- Store SHA256 hash of each keypair file in Git (checked-in, immutable)
- `sync-program-ids.ts` verifies hash before deriving address
- Detects silent modification; prevents accidental corruption
- **Cost:** +1 file to maintain; requires hash update on keypair regen

### Option B: Known-Good Program ID Registry (Low Effort)
- Add `scripts/deploy/PROGRAM_ID_REGISTRY.ts` with expected addresses per cluster
- `sync-program-ids.ts` and `generate-deployment-json.ts` assert program IDs match
- Catches typos and obvious corruption
- **Cost:** Manual registry update; doesn't catch sophisticated attacks

### Option C: On-Chain Verification (High Effort)
- After deployment, query each program on-chain
- Compare deployed program's declare_id with generated constants
- Add to `verify.ts` verification suite
- **Cost:** +50 lines; requires live RPC connection

### Option D: Out-of-Band Checksum (High Effort, Best)
- Before building, compute and sign checksum of all keypairs
- Signature stored in deployment.json or Squads multisig
- Build pipeline rejects unsigned keypair changes
- **Cost:** Requires Squads integration; +2–3 commands to build.sh

## Why ACCEPTED_RISK

1. **Compromise Vector is Narrow**
   - Attacker must modify keypair files *and* be present at build time
   - Local development mode limits exposure to individual developer machines
   - Mainnet deployments go through GitHub Actions → Railway pipeline (centralized)

2. **Detection is Feasible Post-Deploy**
   - Program queries: `solana program show <ID>` returns on-chain bytecode hash
   - Admin authority: only deployer wallet has upgrade authority initially
   - Pool queries: `getPoolState(CRIME_SOL)` fails if program ID is wrong
   - Smoke tests in E2E suite (Phase 96) catch wrong program IDs at runtime

3. **Mainnet Mitigations** (Phase 100+)
   - Authority transferred to Squads 2-of-3 multisig (timelock + 2 cosigners)
   - Regressions require multisig approval, not just deployer action
   - Each multisig action is logged and auditable

4. **Business Risk is Managed**
   - Devnet: Low cost to redeploy if compromised (regenerate keypairs, rebuild, re-initialize)
   - Mainnet: Only deployer wallet has upgrade authority until multisig transfer
   - If mainnet program IDs are poisoned, Squads blocks upgrade attempts

---

## Verdict

**Status:** CONFIRMED, ACCEPTED_RISK

The vulnerability is real: keypair compromise propagates undetected through the build pipeline into on-chain and client code. However:
- Local development + human review provides **practical defense** against accident
- Multisig authority + on-chain verification provides **runtime defense** against attack
- No **current** remediations are urgent; can be deferred to Phase 104 (build hardening)
