# S009: NEXT_PUBLIC_RPC_URL Template Regression → Credit Exhaustion

## Verdict: POTENTIAL

**Tier 3 — Latent / Regression Risk**

---

## Constituent Findings

| Finding | Verdict | Summary |
|---------|---------|---------|
| H006 | POTENTIAL | `NEXT_PUBLIC_RPC_URL` in `.env.mainnet` template still instructs setting a Helius API key in a `NEXT_PUBLIC_*` variable. No current code path exposes it client-side, but the variable is present in the Railway build environment. |
| H008 | CONFIRMED | RPC proxy batch amplification: rate limiter counts HTTP requests (not RPC method calls), allowing 300 HTTP req/min × unlimited batch size = up to 150,000 Helius credits/minute/IP. |

## Combined Attack Path

H006 and H008 are independent vulnerabilities in the current codebase, but they compose into a single high-severity attack path **if a regression causes `NEXT_PUBLIC_RPC_URL` to be referenced in any client-bundled code**. H006 documents exactly why that regression is likely. This finding assesses the combined probability and impact.

### The Regression Gate

H006 identifies four concrete regression triggers — any one of which activates the full combined attack chain without any further attacker input:

**Trigger A — Wallet adapter upgrade:** `providers.tsx` currently hardcodes `/api/rpc` as the endpoint. A future wallet adapter upgrade (e.g., migrating to a newer `@solana/wallet-adapter-react` version or switching to Privy) could re-introduce a `process.env.NEXT_PUBLIC_RPC_URL` reference in a `"use client"` component, causing Next.js to inline the value at build time.

**Trigger B — Developer follows the template verbatim:** `.env.mainnet:49` instructs:
```
NEXT_PUBLIC_RPC_URL=https://mainnet.helius-rpc.com/?api-key=CHANGE_ME_MAINNET
```
with comment: "Still referenced by some wallet adapter config." A developer onboarding to the project reads this comment, sets the variable with a real key in Railway, and adds a single debug log or component reference to verify the connection — the key is now baked into the production bundle.

**Trigger C — `connection.ts` server/client confusion:** `connection.ts` line 41 reads `process.env.NEXT_PUBLIC_RPC_URL` in an `else` branch guarded by `typeof window === "undefined"`. If this file is ever imported from a client component (or if a refactor moves the `resolveRpcUrl` call to a shared module without the guard), the `NEXT_PUBLIC_` prefix causes Next.js to inline the key regardless of the runtime branch taken.

**Trigger D — `__NEXT_DATA__` exposure:** H006 notes (unconfirmed, cannot exclude without live build inspection) that `NEXT_PUBLIC_*` variables present in the Railway build environment may appear in the `__NEXT_DATA__` JSON blob embedded in the HTML regardless of whether any JS code reads them. If this is the case on the current Next.js version, the key is already exposed.

### Post-Regression Attack Chain

Once `NEXT_PUBLIC_RPC_URL` is inlined into the client bundle (any trigger above):

**Step 1 — Key extraction (< 5 minutes)**

1. Attacker loads the production URL in a browser.
2. Opens DevTools → Sources → Search all scripts for `helius-rpc.com` or `api-key=`.
3. Extracts the mainnet Helius API key from the bundle string literal.

**Step 2 — Bypass the protected proxy entirely**

The extracted key grants direct access to `https://mainnet.helius-rpc.com/?api-key=<key>`. This bypasses:
- The 17-method allowlist in `/api/rpc` (attacker can now call any Helius RPC method)
- The per-IP rate limiter (300 req/min — attacker calls Helius directly, no proxy involved)
- The credit counter (observational only even at the proxy, irrelevant here)

**Step 3 — H008 amplification without proxy rate limiting**

H008 is an attack through the RPC proxy. With the key extracted directly from the bundle, the attacker has no rate limiter at all — not even the HTTP-level one at `/api/rpc`. The batch amplification math from H008 applies here with no ceiling:

- H008 (through proxy): 300 HTTP req/min × 500 batch = 150,000 credits/min
- Post-regression (direct): Helius account-level rate limits only — typically 100 req/s or higher on paid plans = **potentially millions of credits/min**

Helius Free plan: 500k credits/month exhausted in seconds. Paid plans ($49/mo = 1M credits) exhausted in under a minute.

**Step 4 — Secondary: webhook management**

With the mainnet Helius API key (not the devnet key from H004, but the mainnet key), the attacker also controls the mainnet webhook. This combines with the S008 attack pattern at mainnet scope — shadow webhook registration plus production webhook deletion on the mainnet data pipeline.

### Attack Timeline Summary

```
Day 0:    Developer sets NEXT_PUBLIC_RPC_URL in Railway (following the template)
Day 0:    Next.js build inlines the mainnet API key into _next/static/chunks/*.js
Day 0:    Bundle is deployed to Railway, key is now public
Day 1-N:  Key sits unnoticed; attacker discovers it at any point via bundle search

          Emergency plan upgrade required or full service outage until credits reset
```

## Likelihood Assessment

**Regression likelihood: MEDIUM-HIGH**

The H006 finding attributes POTENTIAL status to the absence of a current client-side reference. However, four independent regression triggers exist (A–D above), and the project explicitly plans the following future work that increases trigger probability:

- Jupiter integration (Phase 101 deferred): likely requires RPC URL changes in client components.
- Privy wallet adapter migration (noted in project memory): direct rewrite of `providers.tsx` which currently hardcodes `/api/rpc`.
- Ongoing onboarding of collaborators who will read `.env.mainnet` as the authoritative deployment guide.

The combination finding is rated POTENTIAL (not CONFIRMED) because it requires the regression to have occurred. But the probability of the regression occurring before or after mainnet launch is not negligible given the template actively invites it.

**Combined impact if regression occurs: CRITICAL**

Credit exhaustion takes down the entire protocol's off-chain layer simultaneously: frontend RPC proxy, crank runner (if key is shared), and the webhook data pipeline — all at once and at mainnet scale.

## Recommendation

The combined risk is eliminated by fixing the root cause identified in H006: removing `NEXT_PUBLIC_RPC_URL` from the codebase entirely.

### Immediate (closes both H006 and the combined regression risk)

1. **Remove `NEXT_PUBLIC_RPC_URL` from `app/.env.mainnet`** (line 49). Delete the line entirely — do not replace with a placeholder, as placeholders invite re-population.

2. **Remove the `NEXT_PUBLIC_RPC_URL` fallback from `rpc/route.ts:131`**. The proxy's failover chain should use only `HELIUS_RPC_URL` and `HELIUS_RPC_URL_FALLBACK`. A `NEXT_PUBLIC_*` variable should never appear in a server-side failover list — it creates a false signal that the variable is needed.

3. **Remove the `?? process.env.NEXT_PUBLIC_RPC_URL` fallback from `connection.ts:41`** and update the comment to state the variable is no longer used in any code path.

### Secondary (addresses H008 batch amplification at the proxy layer, independent of regression)

4. **Enforce batch element count limit** in `rpc/route.ts` before forwarding. Set `MAX_BATCH_SIZE = 10` (or 1 if batch is not used by any frontend code). This bounds the proxy-level amplification regardless of whether the key is also exposed via bundle.

5. **Rate limit on RPC method calls, not HTTP requests.** Charge the sliding window once per method in the batch, not once per HTTP POST.

### Long-term

6. **If `NEXT_PUBLIC_RPC_URL` is ever re-introduced** (e.g., for a wallet adapter requirement), it must contain the proxy URL (`/api/rpc`) not a Helius key. A pre-commit lint rule or CI check that rejects `NEXT_PUBLIC_*` values matching `helius-rpc.com` patterns would enforce this.

7. **Separate Helius accounts for devnet/mainnet** (also recommended in H004/S008) limits blast radius so a devnet key regression never exposes mainnet credits or webhooks.
