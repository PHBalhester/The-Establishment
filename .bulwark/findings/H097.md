# H097: anchorToJson Shallow Conversion — Tier 3

**Status:** CONFIRMED
**Target:** `app/lib/bigint-json.ts:93-117`
**Severity:** Medium (Low impact, potential future risk)

## Vulnerability Description

The `anchorToJson()` function performs only a shallow conversion of Anchor-decoded account objects. It iterates over top-level keys (lines 102-114) and converts:
- BN instances to `number` (via `.toNumber()`) or `{ __bigint: "..." }` tags
- PublicKey instances to base58 strings
- Primitive values pass through unchanged

**Nested objects containing BN or PublicKey values are NOT recursively converted.**

```typescript
// Line 102-114 — shallow loop
for (const [key, val] of Object.entries(decoded)) {
  if (val && typeof val === "object" && "toNumber" in val) {
    // Convert BN
  } else if (val && typeof val === "object" && "toBase58" in val) {
    // Convert PublicKey
  } else {
    result[key] = val;  // Pass through, including nested objects
  }
}
```

## Current Risk Assessment

**NOT CURRENTLY EXPLOITABLE** because:

1. All protocol account types (PoolState, StakePool, CurveState, EpochState, CarnageFund, etc.) have flat field structures. Complex types like `pool_type: PoolType` are enums, not nested structs.

2. Verified field structure across all IDL types:
   - PoolState: All fields are primitives (pubkey, u64, u16, u8, bool) or enums
   - StakePool: All fields are primitives (u64, u128, u32, bool, u8)
   - CurveState: All fields are primitives (enum, pubkey, u64, u32, bool, u8) or enums
   - CarnageFund: (similar flat structure)

3. No on-chain account currently has nested struct types with BN/PublicKey fields.

## Future Risk (Potential Vulnerability)

If a new account type is added with a nested struct containing BN or PublicKey fields, those nested values will NOT be converted:

```typescript
// Hypothetical future account type (NOT current):
type FutureAccountState = {
  owner: PublicKey,           // ✓ Converted by top-level check
  metadata: {                 // ✗ NOT converted (nested object)
    created: PublicKey,       // ✗ Passed as PublicKey instance
    totalBurned: BN,          // ✗ Passed as BN instance
  }
}

// After anchorToJson():
{
  owner: "base58string",
  metadata: {                 // Still an object
    created: PublicKey {},    // BN serializes as {"_bn": "..."} on JSON.stringify
    totalBurned: BN {},       // Same
  }
}
```

This would cause:
1. JSON.stringify failures or invalid JSON when these objects are sent over SSE
2. Silent truncation if `.toNumber()` is applied downstream without checking type
3. Inconsistent data representation compared to top-level fields

## Affected Code Paths

**Current usage (safe):**
- `ws-subscriber.ts:149` — Decodes PoolState, CurveState, StakePool, EpochState (all flat)
- `helius/route.ts:598` — Decodes same account types

**Function definition:**
- `bigint-json.ts:93-117` — No recursive traversal of nested objects
- Documentation (line 84): "performs a **shallow** conversion" — correctly documents behavior but doesn't warn about implications

## Recommendation

**No immediate action required** because current account structures are flat. However, when adding new account types to the protocol:

1. **Before adding a nested struct type**, consider flattening it to match existing patterns (all current accounts are flat for exactly this reason).

2. **If nesting is unavoidable**, either:
   - Extend `anchorToJson()` to recursively convert nested objects, OR
   - Apply conversions at the field-reading layer (downstream) instead of at decode time

3. **Add a runtime assertion** to catch unconverted BN/PublicKey instances:
   ```typescript
   // After anchorToJson() call
   if (typeof result === "object" && result !== null) {
     for (const v of Object.values(result)) {
       if (typeof v === "object" && ("toNumber" in v || "toBase58" in v)) {
         console.warn(`BN/PublicKey not converted: ${String(v)}`);
       }
     }
   }
   ```

## Verdict

**CONFIRMED** — Shallow conversion is real and documented. Risk is **POTENTIAL** (not current), triggered only if new nested account types are added. Add a warning to the function docstring and implement the assertion above.
