## Finding

The crank runner logs the wallet balance every cycle when it falls below a configurable threshold:

```typescript
// scripts/crank/crank-runner.ts:403-410
const balance = await provider.connection.getBalance(
  provider.wallet.publicKey
);
if (balance < LOW_BALANCE_SOL * LAMPORTS_PER_SOL) {
  console.log(
    `[crank] WARNING: Wallet balance low: ${(balance / LAMPORTS_PER_SOL).toFixed(3)} SOL (threshold: ${LOW_BALANCE_SOL} SOL)`
  );
}
```

Additionally, the wallet public key is logged at startup (line 364):
```typescript
console.log(`  Wallet: ${provider.wallet.publicKey.toBase58()}`);
```

All output goes to stdout, which is **captured automatically by Railway** (as documented in the crank runner header at line 13).

---

## Analysis

### Information Disclosed
1. **Wallet public key** (startup, line 364) — already public on-chain, but concentrated in application logs
2. **Wallet SOL balance** (per-cycle, lines 407-409) — public on-chain but only when below threshold
3. **Time of low-balance events** — timestamp correlation with balance < threshold

### Risk Assessment

**Public Data:** Both the wallet public key and SOL balance are publicly queryable on Solana via any RPC endpoint. Neither is secret or derived from private keys.

**Concentrated Monitoring Surface:** Logging creates a time-indexed record of balance state in application logs (Railway), which:
- Could be used for reconnaissance (e.g., "crank wallet was running out of SOL on [date]")
- Reveals operational heartbeat and spending patterns
- Correlates wallet depletion with protocol activity

**Practical Risk:** LOW. An attacker already monitoring the on-chain wallet would gather the same information. The logging adds convenience but not new exploitability (unlike, e.g., logging private keys or API secrets).

**Railway Logs:** Railway logs are not public by default, but they are accessible to project members and potentially Railway support. This is consistent with running a custodial service.

---

## Verdict

### Status: CONFIRMED — ACCEPTED_RISK

**Confirmation:** The balance logging code is present and unmodified from Audit #1.

**Justification for Acceptance:**
1. Wallet balance is public on-chain data (no information leakage)
2. Logging is necessary for operational monitoring (early warning of funding issues)
3. Railway logs are already protected (not public-facing)
4. Alternative (removing logging) would blind operators to critical wallet depletion

**Recommendation (informational, not required fix):**
- If in a hostile environment, consider hashing or masking balance values in logs
- For mainnet, prefer structured logging with role-based access control
- Monitor Railway log access permissions
