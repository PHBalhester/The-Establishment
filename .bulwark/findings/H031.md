## Summary

The H050 fix is **present and intact**. The body size guard added in H050 has not been removed or weakened. The code at the target lines is unchanged from its fixed state: a 1 MB `MAX_BODY_BYTES` constant is checked against the `Content-Length` header before `req.json()` is called. No regression has occurred.

The Content-Length-only limitation (chunked encoding bypass) is an independent weakness documented as H014. This recheck does not re-raise that issue — it only confirms the H050 fix code survives.

## Verdict

## Evidence

### Source: `app/app/api/webhooks/helius/route.ts:305-315`

```typescript
// ── H050: Reject oversized payloads ──────────────────────────────
// Prevents memory exhaustion from maliciously large webhook bodies.
// 1MB is generous for Helius payloads (typical batch ~50-100KB).
const MAX_BODY_BYTES = 1_048_576; // 1MB
const contentLength = req.headers.get("content-length");
if (contentLength && parseInt(contentLength, 10) > MAX_BODY_BYTES) {
  return NextResponse.json(
    { error: "Payload too large" },
    { status: 413 },
  );
}
```

- The comment explicitly names H050 as the origin of this guard.
- The 1 MB constant (`1_048_576`) is unchanged.
- The conditional correctly short-circuits only when `Content-Length` is present **and** exceeds the limit.
- `req.json()` follows immediately at line 321 with no secondary stream-length guard between them.

### Observation: No secondary guard added

The block from line 315 to `req.json()` at line 321 contains no additional body-stream size enforcement (e.g., reading the `ReadableStream` byte-by-byte with a running counter). This means the Content-Length-only bypass described in H014 still applies, but that is a pre-existing, separately-tracked issue — not a regression introduced since H050 was fixed.
