# Finding: H025 - SSE Amplification DoS Regression (RECHECK H008)

## Status: NOT VULNERABLE (limits expanded but structurally sound; one latent issue noted)
## Confidence Score: 9

---

## Executive Summary

The H008 SSE amplification DoS fix has **not regressed**. Both SSE routes (`/api/sse/protocol` and `/api/sse/candles`) correctly call `acquireConnection(clientIp)` before any stream creation, returning HTTP 429 if the cap is exceeded. The protection structure is identical to Audit #1 — the gate is in the right place and cannot be bypassed.

However, the connection limit values were significantly relaxed between Audit #1 and Audit #2: `MAX_PER_IP` grew from 3 to 10, and `MAX_GLOBAL` grew from 100 to 5000. This increases the ceiling for legitimate fan-out amplification. The limits remain rational for the stated headroom goals (500 users × 2 routes × 5×), but the amplification math has changed materially.

One latent structural issue is confirmed: `sseManager.subscribers` (the internal `Set<SSECallback>`) has **no size cap of its own**. It relies entirely on `sse-connections.ts` to enforce the ceiling at the HTTP layer. If the two counts can ever desync (e.g., due to a bug in cleanup paths), subscribers could exceed the intended bound. This desync risk is low but non-zero and was flagged in the H008 recommended fix (which was not implemented).

---

## Hypothesis Investigated

The H008 SSE connection-cap fix may have regressed during the DBS rework. Specifically: whether `acquireConnection()` is still called before stream creation on all SSE routes, whether the limits are the same, and whether any bypass path exists.

## Investigation Path

### Step 1: Locate all SSE route handlers
**Files found:** exactly two
- `/app/app/api/sse/protocol/route.ts`
- `/app/app/api/sse/candles/route.ts`

No new SSE routes were introduced during the DBS rework. The surface is unchanged.

### Step 2: Verify acquireConnection() call order in both routes

**`/api/sse/protocol/route.ts` (lines 42–49):**
```typescript
const clientIp = getClientIp(req);
if (!acquireConnection(clientIp)) {
  return new Response("Too Many Connections", {
    status: 429,
    headers: { "Retry-After": "30" },
  });
}
```
`acquireConnection` is called at line 44, before the `ReadableStream` is constructed at line 66. The gate cannot be bypassed.

**`/api/sse/candles/route.ts` (lines 42–49):**
```typescript
const clientIp = getClientIp(req);
if (!acquireConnection(clientIp)) {
  return new Response("Too Many Connections", {
    status: 429,
    headers: { "Retry-After": "30" },
  });
}
```
Identical pattern. Gate is in place before stream construction at line 65. Cannot be bypassed.

**Verdict on call order: INTACT — acquireConnection() fires before ReadableStream in both routes.**

### Step 3: Check current limit values vs Audit #1 baseline

**`/app/lib/sse-connections.ts` (lines 23–24):**
```typescript
const MAX_PER_IP = 10;
const MAX_GLOBAL = 5000;
```

**Audit #1 values (from VERIFY-H008.md):**
```
MAX_PER_IP = 3
MAX_GLOBAL = 100
```

The limits were expanded by 3.3× per-IP and 50× globally. The architecture comment explains the reasoning: "5 tabs × 2 SSE routes per user" for MAX_PER_IP, and "500 users × 2 SSE routes × 5× headroom" for MAX_GLOBAL. These are rational justifications for a single-Railway-process deployment expecting legitimate concurrent users.

**Verdict on limit values: CHANGED (relaxed) but intentionally and with documented rationale. Not a regression in the structural sense.**

### Step 4: Check for bypass paths — release lifecycle

Both routes implement the same double-release guard:
```typescript
let released = false;
const release = () => {
  if (!released) {
    released = true;
    releaseConnection(clientIp);
    cancelAutoRelease();
  }
};
```
`release()` is called in: `req.signal` abort handler (line 113 / 99) and `cancel()` callback (line 122 / 107). `unsubscribe()` is called in both abort and heartbeat catch. This is correct. The `released` flag prevents double-decrements in `sse-connections.ts`.

**Verdict on release lifecycle: CORRECT — no bypass path via double-release.**

### Step 5: Verify sseManager.subscribers has no independent size cap

**`/app/lib/sse-manager.ts` (lines 40–45):**
```typescript
subscribe(callback: SSECallback): () => void {
  this.subscribers.add(callback);
  return () => {
    this.subscribers.delete(callback);
  };
}
```

No size guard. `this.subscribers.size` is exposed as a getter but is not checked before `add()`. The recommended fix from H008 (a `MAX_SUBSCRIBERS` check inside `subscribe()`) was **not implemented**. The H008 fix instead enforced the cap at the HTTP routing layer (`acquireConnection`), which is architecturally sound as long as the two counters remain in sync.

**The desync risk:** If a client's HTTP connection is accepted by the route (acquireConnection returns true) but the ReadableStream's `start(controller)` callback fails before `sseManager.subscribe()` is called, the HTTP slot is consumed but no subscriber is added. The inverse — subscriber added without slot consumed — cannot happen because `acquireConnection` is called before `ReadableStream` construction. The one-way desync (slot consumed, no subscriber) is safe (it wastes a slot, reducing capacity, but does not cause amplification). The dangerous direction (subscriber added, no slot counted) is architecturally prevented.

**Verdict: The single layer of enforcement at HTTP routing is sufficient as long as code ordering is maintained. Currently it is.**

### Step 6: Check for candles route event-type leak (API-03 flagged issue)

The candles SSE route subscriber does **not filter by event type**:
```typescript
// /api/sse/candles/route.ts:71-77
const unsubscribe = sseManager.subscribe((payload: string) => {
  try {
    controller.enqueue(encoder.encode(payload));  // ALL events forwarded
  } catch {
    unsubscribe();
  }
});
```

The protocol route filters correctly:
```typescript
// /api/sse/protocol/route.ts:82-84
if (!payload.startsWith("event: protocol-update\n")) return;
```

This means candle SSE clients also receive `protocol-update` events (account states). This is a separate finding (already captured in API-03 context), not a regression of H008. It does slightly increase per-connection bandwidth but does not bypass the connection cap.

## Impact Assessment

**No regression of H008 fix:** The structural protection (acquireConnection before stream creation) is intact and cannot be bypassed through normal HTTP interaction.

**Expanded limits impact:** With 5000 global connections vs 100 at Audit #1, a saturated server (all 5000 slots filled by an attacker) produces 50× more fan-out per webhook broadcast. A single webhook event delivering one swap triggers 6 resolution broadcasts × 5000 subscribers = 30,000 `controller.enqueue()` calls. This is bounded but significant. At Audit #1 levels it was 600 calls; now it is 30,000. However, reaching 5000 connections from many IPs is a considerably harder attack than the original 100-connection scenario.

**Zombie risk is unchanged:** 30-minute zombie timeout was present at Audit #1 and remains. With 5000 slots, an attacker could squatter-fill the global cap with 5000 connections from 500 unique IPs (10 per IP), wait 30 minutes per rotation, and continuously block legitimate users from connecting. This requires sustained effort with IP diversity.

**Severity of current state: LOW** (defense-in-depth gap in `sseManager.subscribe()`) with an **informational note** on the relaxed limits.

## Recommendation

### R1 — Add defense-in-depth cap to sseManager.subscribe() (LOW priority)
The H008 original recommended fix was not implemented. While not exploitable given correct HTTP layer enforcement, adding a size guard to `subscribe()` would prevent amplification if the two counters ever desync:

```typescript
// In sse-manager.ts
private static readonly MAX_SUBSCRIBERS = 5100; // slightly above MAX_GLOBAL

subscribe(callback: SSECallback): (() => void) | null {
  if (this.subscribers.size >= SSEManager.MAX_SUBSCRIBERS) {
    return null;
  }
  this.subscribers.add(callback);
  return () => { this.subscribers.delete(callback); };
}
```
Both SSE routes would need to handle a `null` return from `subscribe()` by closing the stream and calling `release()`.

### R2 — Fix event-type filter leak in candles SSE route (MEDIUM — separate issue)
`/api/sse/candles/route.ts` forwards all sseManager events, including `protocol-update`. Add the same filter present in the protocol route:
```typescript
if (!payload.startsWith("event: candle-update\n")) return;
```
(Already captured in API-03 context findings.)
