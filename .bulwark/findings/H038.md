## Summary

The Phase 52.1 BPS-based fee display fix is **intact in the route engine and RouteCard** — both correctly use BPS summation (`steps.reduce((sum, s) => sum + s.lpFeeBps + s.taxBps, 0)`) for all multi-hop and split routes. This is the codepath exercised by smart routing, which is on by default.

However, the **legacy direct-quote path in `useSwap.ts`** (non-smart-routing, single-hop AMM swaps) was not fully converted. Two fee percentage calculations still use mixed-denomination arithmetic rather than BPS:

1. **`swapSolSell` forward quote** (`useSwap.ts:448-449`): fee is `(taxAmount / grossSolOutput) * 100 + (lpFee / baseUnits) * 100`. The two terms are individually homogeneous (SOL/SOL and token/token), but their sum is not a meaningful fee percentage — it adds percentages computed over different-denomination totals.

2. **`swapSolSell` reverse quote** (`useSwap.ts:521-522`): fee is `(taxAmount / (desiredOutput + taxAmount)) * 100 + (lpFee / inputTokensNeeded) * 100`. Same denominator-mixing pattern.

For dust-level inputs specifically, these two paths exhibit the original H119 symptom in a subtle form:

- When `grossSolOutput` floors to `0n` in BigInt (extreme dust), `Number(result.grossSolOutput)` is `0`. The guard `grossSolOutput > 0` (line 448) correctly short-circuits to `"0%"` — so no division-by-zero occurs.
- However, for inputs just above zero that produce a non-zero `grossSolOutput` but zero `taxAmount` (e.g., `taxBps=0`), the formula adds `(0 / grossSolOutput) * 100 + (lpFee / baseUnits) * 100`. The LP-fee term uses `baseUnits` (token input) as denominator. For token sells this is correct in isolation, but the sum with the SOL-denominated tax term remains a display defect.

The BPS-based `swapSolBuy` forward quote (line 419-421) is **also not using BPS**: it computes `(taxAmount + lpFee) / baseUnits * 100`. For a SOL buy, `taxAmount` and `lpFee` are both in lamports and `baseUnits` is also in lamports — this is actually denomination-consistent and produces a correct result for the buy path. The sell paths are the residual issue.

## Dust Minimum Enforcement

The zero-input guard is correctly implemented across all paths:

- `route-engine.ts:416`: `if (inputAmount <= 0) return []`
- `useSwap.ts:400`: `if (baseUnits <= 0) { setQuote(null); return; }`
- `useSwap.ts:347`: vault path `if (baseUnits <= 0) { setQuote(null); return; }`
- `quote-engine.ts:64`: `calculateSwapOutput` returns `0n` when denominator is zero
- `quote-engine.ts:261`: `quoteVaultConvert` returns `0n` when `inputAmount <= 0n`
- `route-engine.ts:334`: `if (!result || result.step.outputAmount <= 0) return null`

No path passes dust through to yield a misleadingly non-zero fee. Dust amounts correctly produce `setQuote(null)` or return no routes.

## Impact

**Display-only.** The `totalFeePct` value from the legacy direct-quote path is shown in `FeeBreakdown.tsx:95` as `"~X% total fees"`. It has no on-chain effect — `minimumOutput` is computed via BigInt arithmetic independently (`Math.floor(outputSol * (10_000 - slippageBps) / 10_000)`) and is correct. The defect inflates or deflates the displayed fee percentage on sell swaps when smart routing is disabled, but the execution price and slippage protection are unaffected.

Smart routing is enabled by default (`useSwap.ts:204`). When smart routing is on, the route engine's BPS-based fee computation is used and these legacy code paths in `computeQuote` are bypassed for the fee display — the `quote.totalFeePct` shown to the user comes from `selectedRoute.totalFeePct` instead, which is correctly BPS-computed.

## Verdict

**PARTIALLY_FIXED.** The Phase 52.1 fix was applied to the route engine and RouteCard (the primary smart-routing code path), but the legacy `computeQuote` function in `useSwap.ts` retains two mixed-denomination sell fee percentage calculations. The original dust zero-fee symptom (division by zero or silent `"0%"` for non-zero fees) is not present — both sell paths guard against zero denominators. The residual issue is an inaccurate fee percentage display on the sell path when smart routing is disabled, not a financial invariant violation.

## Recommendation

Apply BPS summation to the two remaining legacy paths in `useSwap.ts`:

```typescript
// swapSolSell forward quote — replace lines 448-450:
const totalFeeBps = sellTaxBps + SOL_POOL_FEE_BPS;
const totalFeePct = `${(totalFeeBps / 100).toFixed(1)}%`;

// swapSolSell reverse quote — replace lines 521-523:
const totalFeeBps = sellTaxBps + SOL_POOL_FEE_BPS;
const totalFeePct = `${(totalFeeBps / 100).toFixed(1)}%`;
```

This removes the denomination dependency entirely and makes all four direct-quote fee percentage computations consistent with the route engine pattern.
