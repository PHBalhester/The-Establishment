# H095: Connection Pool No Idle Timeout

**Severity**: Tier 3 (Potential)
**Status**: CONFIRMED
**Target**: `/app/db/connection.ts:56-57`

## Finding

The postgres.js connection pool is configured with `max: 10` but **does not set an `idle_timeout`** parameter. Without idle timeout, leaked/abandoned connections remain in the pool indefinitely, eventually exhausting the 10-connection limit and causing new queries to hang or fail.

### Code

```typescript
// Line 56-57: postgres.js client created with max pool size but no idle_timeout
const client =
  globalForDb.pgClient ?? postgres(connectionString, { max: 10, ...sslConfig });
```

## Technical Details

The postgres.js library (v3.x) supports an `idle_timeout` option (in seconds) that automatically closes idle connections. The library's documented behavior:

- **Default**: No idle timeout (connections persist indefinitely)
- **Standard practice**: 30 seconds (30,000ms equivalent in other drivers)
- **AWS Lambda**: 1-2 seconds (context: Lambdas freeze after request; long-lived pools leak)
- **Next.js dev**: 2-5 seconds (rapid HMR cycles create many connections without cleanup)

The current code explicitly mentions this concern in comments:

```typescript
// Lines 25-27: "Without this, each HMR cycle creates a new connection pool,
// eventually exhausting Postgres connection limits."
```

But the mitigation (globalThis singleton) only prevents *creating* new pools per HMR cycle. It does **not** prevent individual connections within the pool from leaking.

### Attack Vector

1. **Gradual pool exhaustion**: A long-running server or a client that leaks connections (connection is obtained but not properly returned) will fill the pool over hours/days.
2. **Cascading failure**: Once all 10 slots are occupied by idle/leaked connections, new requests queue indefinitely or timeout.
3. **Railway free tier impact**: Railway Postgres (free plan) has a 10-connection hard limit. Exceeding this causes connection rejections at the database level.

## Risk Factors

- **Railway deployment**: Production uses Railway Postgres with a 10-connection limit. No buffer.
- **Long-running processes**: Express/Next.js API routes may hold connections longer than expected.
- **Error paths**: Connections not explicitly returned on errors/exceptions.
- **Monitoring gap**: No observability for idle connection count.

## Verdict

**CONFIRMED** — The pool lacks idle timeout configuration. This is a configuration oversight, not a code bug. The risk is medium-to-high in production (Railway) and high in development (frequent HMR).

## Recommendation

Add `idle_timeout` to the postgres.js client configuration:

```typescript
const client =
  globalForDb.pgClient ?? postgres(connectionString, {
    max: 10,
    idle_timeout: 30,  // Close idle connections after 30 seconds
    ...sslConfig,
  });
```

**Rationale**:
- 30 seconds is industry standard (used by node-postgres, pg-pool, Npgsql).
- Balances connection reuse (warm pool) vs. cleanup (prevents leaks).
- For Railway free tier, connections are cheap to re-establish (~20-30ms). Closing idle ones is safer than risking pool exhaustion.

### Alternative: Monitor Pool Utilization

If 30s is too aggressive, add logging to detect pool pressure:

```typescript
// After creating client
if (client.pool) {
  setInterval(() => {
    console.log(`[db] Pool: ${client.pool.size} active, ${client.pool.idle} idle`);
  }, 10000);
}
```

However, this only monitors; it doesn't prevent leaks. `idle_timeout` is the primary fix.
