# H120: No Secret Rotation Mechanism

**Severity:** Tier 3 (Medium)
**Category:** API Security (FE-02)
**Status:** CONFIRMED

---

## Summary

The webhook authentication handler accepts **only a single secret** (`HELIUS_WEBHOOK_SECRET`). There is no support for dual-key acceptance, allowing zero-downtime secret rotation.

---

## Root Cause

1. **Single-source design:** `HELIUS_WEBHOOK_SECRET` is the only valid secret; no secondary is checked.
2. **No dual-key environment variable:** No `HELIUS_WEBHOOK_SECRET_NEXT` or similar pattern to hold the new secret during rotation.
3. **Atomic secret swap required:** Changing the env var invalidates all in-flight requests until Helius propagates the new secret to all edge nodes (typically 30–120 seconds of downtime).

---

## Operational Impact

**Scenario: Secret rotation (e.g., daily rotation, leaked key recovery)**

1. Generate new secret S2
2. Deploy app with S2 in `HELIUS_WEBHOOK_SECRET`
3. Helius still sends requests with S1 (old secret) until propagation (~60–120 seconds)
4. All incoming webhook requests fail (401 Unauthorized)
5. Event data is lost; database events, SSE broadcasts, and candles are not updated
6. **Downtime: ~60–120 seconds of data loss**

If rotation happens during a volatile market (high trading volume), missed swap events may inflate/deflate price calculations and cause chart data corruption.

---

## Remediation

### Option 1: Dual-Key Acceptance (Recommended)

**File:** `app/app/api/webhooks/helius/route.ts`

Add support for both current and next secret:

```typescript
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
const webhookSecretNext = process.env.HELIUS_WEBHOOK_SECRET_NEXT;
const isProduction = process.env.NODE_ENV === "production";

if (!webhookSecret && isProduction) {
  console.error("[webhook] CRITICAL: HELIUS_WEBHOOK_SECRET not set in production.");
  captureException(new Error("[webhook] HELIUS_WEBHOOK_SECRET not set in production"));
  return NextResponse.json(
    { error: "Webhook secret not configured" },
    { status: 500 },
  );
}

if (webhookSecret) {
  const authHeader = req.headers.get("authorization") ?? "";
  const secretBuf = Buffer.from(webhookSecret, "utf-8");
  const nextSecretBuf = webhookSecretNext ? Buffer.from(webhookSecretNext, "utf-8") : null;
  const headerBuf = Buffer.from(authHeader, "utf-8");

  // Check current secret
  const currentMatch =
    secretBuf.length === headerBuf.length &&
    timingSafeEqual(secretBuf, headerBuf);

  // Check next secret (if set)
  const nextMatch =
    nextSecretBuf &&
    nextSecretBuf.length === headerBuf.length &&
    timingSafeEqual(nextSecretBuf, headerBuf);

  if (!currentMatch && !nextMatch) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }

  // Log which secret was accepted (for audit trail)
  if (nextMatch && !currentMatch) {
    console.log("[webhook] Request authenticated with next secret (rotation in progress)");
  }
}
```

**Rotation workflow:**
1. Generate S2, set `HELIUS_WEBHOOK_SECRET_NEXT=S2` in Railway env vars
2. Deploy app (now accepts both S1 and S2)
3. Update Helius webhook to use S2 via `webhook-manage.ts update`
4. Wait 2–5 minutes for Helius to propagate
5. Set `HELIUS_WEBHOOK_SECRET=S2`, unset `HELIUS_WEBHOOK_SECRET_NEXT`
6. Deploy app (S1 no longer accepted)
7. **Zero downtime**

---

### Option 2: Staged Secret Array

Support multiple secrets via comma-separated or JSON env var:

```typescript
// HELIUS_WEBHOOK_SECRETS="secret1,secret2,secret3"
const validSecrets = (process.env.HELIUS_WEBHOOK_SECRETS || "").split(",").map(s => s.trim());

for (const secret of validSecrets) {
  const secretBuf = Buffer.from(secret, "utf-8");
  if (secretBuf.length === headerBuf.length && timingSafeEqual(secretBuf, headerBuf)) {
    // Authenticated
    break;
  }
}
```

**Pros:** Supports N secrets (useful for key rotation pipelines)
**Cons:** More overhead; need to update env var parsing logic

## Verdict

**CONFIRMED — No secret rotation mechanism exists.**

The webhook authentication uses a single, fixed secret. Rotation requires deploying new code or briefly disrupting service. Recommend implementing **Option 1 (Dual-Key Acceptance)** for zero-downtime rotation.

**Effort:** Low (10–15 lines of code, ~30 minutes)
**Priority:** Medium (rotation is infrequent, but data loss on rotation is severe)
