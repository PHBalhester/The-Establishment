## Verdict

**NOT_VULNERABLE.** The HSTS header introduced as the H026 fix is fully intact. No regression occurred. All six security headers are present and correctly configured in `app/next.config.ts`.

---

## Hypothesis

H026 was previously FIXED by adding a `Strict-Transport-Security` header to `next.config.ts`. The concern was that subsequent rework of `next.config.ts` (CSP cluster-awareness additions, Turbopack config additions, Sentry removal comment) may have inadvertently removed or weakened the HSTS directive.

---

## Evidence

**`app/next.config.ts`, lines 107–113 (current state):**

```typescript
{
  // H026: HSTS -- force HTTPS for 2 years, include subdomains, and
  // opt into the browser preload list. This prevents SSL-stripping
  // attacks where an attacker downgrades HTTPS to HTTP.
  key: "Strict-Transport-Security",
  value: "max-age=63072000; includeSubDomains; preload",
},
```

The HSTS header is:
- **Present**: Yes, in the `headers()` array at lines 107–113.
- **max-age**: 63072000 seconds = 730 days = 2 years. Meets the minimum for HSTS preload list submission (requires ≥ 1 year; 2 years is the recommended value).
- **includeSubDomains**: Present. Ensures all subdomains are also covered by HSTS policy.
- **preload**: Present. Opts the domain into browser HSTS preload lists, preventing the first-visit downgrade window entirely.

The header is applied to all routes via `source: "/(.*)"` at line 85, which matches every path on the server.

**Complete security header bundle confirmed intact (lines 86–114):**

| Header | Value | Status |
|--------|-------|--------|
| Content-Security-Policy | Full CSP policy (cluster-aware, 13 directives) | PRESENT |
| X-Frame-Options | DENY | PRESENT |
| X-Content-Type-Options | nosniff | PRESENT |
| Referrer-Policy | strict-origin-when-cross-origin | PRESENT |
| Permissions-Policy | camera=(), microphone=(), geolocation=() | PRESENT |
| Strict-Transport-Security | max-age=63072000; includeSubDomains; preload | PRESENT |

The H026 comment is preserved verbatim in the source, indicating the fix was intentional and was not removed during subsequent edits.

## What Changed in the Rework (Non-Regressing)

The following changes were made to `next.config.ts` after H026 was fixed, none of which affected the HSTS or other security headers:

1. **Lines 1–11**: Build-time cluster detection (`const cluster`, `const isMainnet`) added for CSP cluster-awareness.
2. **Lines 13–29**: Cluster-specific domain variables (`heliusRpcDomain`, `heliusApiSources`, `docsFrameSources`) added.
3. **Lines 31–47**: CSP header string extended with cluster-aware values (Helius domains, docs frame sources) and additional directives (`child-src`, `frame-src`, `worker-src`, `manifest-src`, Sentry ingest wildcards).
4. **Lines 65–77**: `turbopack` config block added (resolveAlias stubs for `fs`/`net`/`tls`).
5. **Line 120–121**: Comment documenting @sentry/nextjs removal.

The `async headers()` block at lines 82–117 is structurally unchanged from the H026 fix. The HSTS entry is the sixth and final header in the array and was not displaced by any of the rework.

## Attack Path (Hypothetical — Not Applicable)

Had the HSTS header been removed:

1. Attacker positions as network MITM (e.g., rogue Wi-Fi, ISP-level interception).
2. User's first-ever visit to `fraudsworth.fun` is intercepted and served over HTTP.
3. Attacker strips or modifies TLS, injecting malicious JavaScript into the trading UI.
4. Injected script captures wallet interaction, manipulates swap parameters (amounts, slippage), or extracts in-page secrets.
5. Because no HSTS is stored from a prior visit, the browser accepts the HTTP downgrade silently.

With `preload`, this attack is blocked even for first-time visitors whose browsers include the preloaded list — the browser refuses HTTP connections before the first request is even sent.

---

## Impact

None — vulnerability is not present.

Hypothetical impact if HSTS were absent: **HIGH**. SSL-stripping attacks on a DeFi trading UI are among the most damaging MITM vectors — they allow full page content manipulation, enabling transaction parameter hijacking, fake approval dialogs, and private key theft if any key material were ever present in the page. The `preload` attribute makes this protection strongest-class.

---

## Recommendation

No action required for this finding.

**Optional hardening (LOW priority, post-launch):**
- Verify the domain is submitted to the HSTS preload list at https://hstspreload.org if not already done. The `preload` directive in the header alone is not sufficient — the domain must be actively submitted to the list. This is a one-time operational step, not a code change.
- Add `Cross-Origin-Opener-Policy: same-origin` for Spectre-class side-channel protection (noted as LOW in the WEB-02 context audit). Does not interact with HSTS.
