# H118: Rate Limiter Memory Growth — Cleanup Latency

**Severity**: Medium | **Status**: POTENTIAL | **CWE**: CWE-400 (Uncontrolled Resource Consumption)

---

## Summary

The in-memory rate limiter in `app/lib/rate-limit.ts` uses a Map to track request timestamps per `ip:endpoint` key. While cleanup is implemented, stale entries are only removed every 60 seconds, and only if all timestamps exceed a 5-minute threshold. During high-traffic periods with diverse IP sources (botnets, crawlers, scanners), the Map can temporarily accumulate thousands of dormant entries before the next cleanup cycle, causing memory to grow faster than release.

---

## Vulnerable Code

**File**: `app/lib/rate-limit.ts`
**Lines**: 66, 160–181

```typescript
// Line 66: Global state, unbounded until cleanup
const entries = new Map<string, RateLimitEntry>();

// Lines 160–181: Cleanup runs every 60s, removes entries >5min old
const CLEANUP_INTERVAL_MS = 60_000;
const STALE_THRESHOLD_MS = 5 * 60_000;

if (!(globalThis as Record<symbol, unknown>)[CLEANUP_KEY]) {
  const interval = setInterval(() => {
    const cutoff = Date.now() - STALE_THRESHOLD_MS;
    for (const [ip, entry] of entries) {
      if (entry.timestamps.length === 0 || entry.timestamps.every((t) => t < cutoff)) {
        entries.delete(ip);
      }
    }
  }, CLEANUP_INTERVAL_MS);
  if (interval.unref) interval.unref();
  (globalThis as Record<symbol, unknown>)[CLEANUP_KEY] = true;
}
```

---

## Root Cause

1. **Lazy Entry Creation** (line 92–93): Entries are created on first request and only deleted by the cleanup interval.
2. **Cleanup Latency**: An IP making a single request then going dormant will occupy a Map entry for:
   - **Worst case**: 60s (until next cleanup runs) + 5m (until timestamps become stale) = **360 seconds**
3. **No Per-Entry TTL**: Individual entries don't have creation timestamps; cleanup only considers request timestamps within each entry.
4. **No Eviction Backpressure**: If cleanup can't keep up (e.g., 10k unique IPs per minute), Map size grows linearly.

---

## Attack Scenario

**Botnet IP Rotation**: An attacker sends HTTP requests from 10,000 unique IP addresses, each making 1–2 requests to `/api/rpc` over a 2-minute window. Expected behavior:

- Each IP creates one Map entry
- ~10k entries accumulate during minutes 0–2
- Cleanup at minute 1 removes nothing (timestamps <5min old)
- Cleanup at minute 6 removes all (~360s after last request per IP)

**Memory spike**: Map holding 10k entries × ~40 bytes per entry (key string + array overhead) = ~400 KB temporary overhead. Scales with attacker IP diversity.

---

## Impact

- **Memory**: Temporary unbounded growth during attack or high legitimate traffic diversity
- **GC Pressure**: Frequent allocations/deallocations of stale entries stall the event loop
- **Denial of Service**: No backpressure; if cleanup is slow, event loop threads could be blocked by GC

## Proof of Concept

```typescript
// Simulate 1000 unique IPs, each making 1 request
for (let i = 0; i < 1000; i++) {
  checkRateLimit(`192.168.1.${i % 256}`, RPC_RATE_LIMIT, "/api/rpc");
}

// Cleanup hasn't run yet; Map still has 1000 entries
console.log(entries.size); // 1000

// Wait 60s: cleanup runs, but timestamps are <5min old
// Wait 300s: all entries stale, deleted on next cleanup at 360s total
```

---

## Recommendation

**Priority**: Implement aggressive stale entry eviction with bounded cleanup cost:

1. **Per-Entry Creation Time** (Tier 1 fix):
   ```typescript
   interface RateLimitEntry {
     timestamps: number[];
     createdAt: number; // Add this
   }

   // Then in cleanup, delete if:
   // - timestamps array is empty, OR
   // - (now - createdAt) > 5 minutes (not just timestamp age)
   ```

2. **Incremental Cleanup** (Tier 2 fix):
   ```typescript
   // Limit cleanup work per cycle to prevent GC stalls
   const entriesToCheck = 100; // Check oldest 100 per interval
   let cleanupIndex = 0;

   const interval = setInterval(() => {
     const entriesArray = Array.from(entries.entries());
     for (let i = 0; i < entriesToCheck && i < entriesArray.length; i++) {
       const [key, entry] = entriesArray[(cleanupIndex + i) % entriesArray.length];
       if (entry.timestamps.every((t) => t < cutoff)) {
         entries.delete(key);
       }
     }
     cleanupIndex = (cleanupIndex + entiesToCheck) % entriesArray.length;
   }, CLEANUP_INTERVAL_MS);
   ```

3. **Monitoring** (Tier 3):
   ```typescript
   if (entries.size > 1000) {
     console.warn(`[rate-limit] Map size: ${entries.size} (possible attack)`);
   }
   ```

---

## Verdict

**POTENTIAL** — Cleanup is implemented and will eventually free memory, but latency between entry creation and deletion (up to 360s per dormant IP) allows temporary unbounded growth during diverse-IP attack or legitimate traffic spike. No infinite leak, but GC pressure and event loop stalls are possible at scale.
