# H017: Sign-Then-Send Bypasses Wallet MEV Protection

## Verdict: POTENTIAL
## Confidence Score: 6

## Executive Summary

The sign-then-send pattern in `useProtocolWallet.ts` (lines 87-121) is **confirmed present** — `signTransaction()` + `connection.sendRawTransaction()` is the live implementation, used across all 6 swap/staking/bonding-curve signing paths. This pattern was introduced deliberately to route transactions through the app's own Helius RPC rather than Phantom's internal RPC, which silently drops devnet transactions.

The hypothesis that this "explicitly bypasses wallet-level MEV protection" is **partially correct**. The pattern does bypass Phantom's MEV-protected RPC submission path. However, the practical MEV impact is rated POTENTIAL rather than CONFIRMED because: (1) Phantom's Blowfish simulation still fires at `signTransaction()` time regardless of submission path, (2) the real MEV extraction surface is the 5% default slippage (H015, NOT_FIXED) not the submission routing, and (3) Solana's public mempool means even wallet-submitted TXs are visible to validators — wallet MEV protection is probabilistic, not absolute.

The finding **escalates the ACCEPTED_RISK status from Audit #1 H048** specifically for the mainnet dimension: the devnet rationale no longer applies on mainnet, yet the pattern ships unchanged.

## Evidence

### 1. Sign-Then-Send Pattern Confirmed in Source

**File:** `app/hooks/useProtocolWallet.ts:87-121`

The pattern is live and documented as intentional:

```typescript
// 1. Wallet signs the transaction (single popup, Blowfish-compatible)
const signed = await signTransaction(tx);

// 2. Serialize the signed transaction.
const serialized = signed.serialize();

// 3. Submit via OUR connection (Helius RPC) -- we control the endpoint
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: sendOptions.skipPreflight,
  // ...
});
```

The rationale comment at lines 77-86 explicitly states: "For wallet-standard wallets (Phantom, Solflare, Backpack), sendTransaction calls signAndSendTransaction which sends the TX through the wallet's own RPC endpoint. On devnet, Phantom's RPC silently drops transactions."

### 2. No Jito or MEV-Protected Submission Anywhere

A grep across all TypeScript files for `jito`, `mev.protect`, `private.mempool`, `blockengine`, `tip.account`, `bundle` found zero matches in any app or scripts code. The only hit is a comment string in `app/idl/types/epoch_program.ts:469` ("Typically bundled in the same transaction for MEV protection") which is IDL documentation, not an implementation.

### 3. All 6 Signing Paths Funnel Through This Abstraction

Every user-facing signing operation passes through `useProtocolWallet.sendTransaction()`:
- `useSwap.ts:763` — direct AMM swap
- `useSwap.ts:875` — multi-hop route execution
- `useStaking.ts:580` — staking operations
- `BuyForm.tsx:190` — bonding curve buy
- `SellForm.tsx:199` — bonding curve sell
- `RefundPanel.tsx:207` — bonding curve refund

All 6 paths bypass wallet-native submission and send through the Helius RPC proxy at `/api/rpc`.

### 4. Submission Chain: Helius Standard RPC, Not MEV-Protected

**File:** `app/lib/connection.ts:36`

Browser RPC resolves to `${window.location.origin}/api/rpc`, which proxies to `HELIUS_RPC_URL`. Helius is a standard Solana RPC provider — not a private mempool (Jito), not a MEV-protected relay (Flashbots-equivalent). Transactions enter the standard public validator mempool.

### 5. MEMORY.md Deferred Decision Still Open

MEMORY.md (Wallet adapter section): "Phantom's `signAndSendTransaction` sends TX via Phantom's own RPC — on devnet it silently drops TXs. Fix: use `signTransaction()` + `connection.sendRawTransaction()` (sign-then-send). **Revisit for mainnet.**"

This "revisit for mainnet" note has not been acted on. The codebase is in pre-mainnet state and ships the same pattern.

### 6. Audit #1 H048 Context

Audit #1 H048 assessed the sign-then-send specifically for the **transaction simulation preview** angle (POTENTIAL, ACCEPTED_RISK). It concluded Blowfish still fires at `signTransaction()` time. H017 Audit #2 focuses on the **MEV submission routing** angle, which H048 noted ("bypasses any MEV protection Phantom's RPC might offer") but did not assess as a primary finding.

## Attack Path

The attack path is structural, not adversarial code injection:

```
User initiates swap on Dr. Fraudsworth frontend
  -> useProtocolWallet.sendTransaction() called
  -> wallet.signTransaction(tx) — wallet popup, user approves
  -> connection.sendRawTransaction(signed) via /api/rpc
  -> Helius standard RPC endpoint
  -> Solana public validator gossip network
  -> TX is visible to all MEV searchers monitoring the mempool

MEV Bot Path (concurrent):
  -> Bot monitors Helius/public RPC mempool
  -> Detects pending swap TX (SOL -> CRIME/FRAUD, slippage 500 BPS)
  -> Front-run: submit buy TX with higher priority fee (same pair)
  -> User's TX lands, slippage extracted up to ~5% of swap value
  -> Back-run: sell to capture price difference
```

What wallet-native `sendTransaction` *might* add:
- Some wallets route through MEV-protected RPC (Phantom has been building this)
- Private mempool submission bypasses public gossip visibility window
- Probabilistic protection: attacker can't observe TX until inclusion, not before broadcast

What this app loses by bypassing wallet submission:
- If Phantom/Solflare have MEV-protected RPC on mainnet, every Dr. Fraudsworth swap skips it
- Users who chose a wallet specifically for its MEV protection get none of it

**Critical nuance**: Solana's mempool is not perfectly MEV-protected by any path. Even wallet-native TXs eventually reach validators. The bypass reduces the probability of protection, it does not guarantee extraction.

## Impact

**Severity:** MEDIUM (elevated from Audit #1's ACCEPTED_RISK framing due to mainnet proximity)
**Likelihood:** HIGH (every swap TX takes this path; sandwich bots are active on Solana mainnet)
**Extractable value:** Up to 5% of every swap amount (bounded by H015 slippage default — the primary amplifier)

**User Impact:** Users who expect their wallet's MEV protection to apply (e.g., Phantom users who have seen MEV protection marketing) receive zero of that protection. The app silently routes around it.

**Protocol Impact:** This is not a fund-theft vulnerability — the on-chain 50% output floor is the absolute backstop. The loss is value extraction by MEV bots from user swaps, not loss to the protocol itself.

**Comparison to H015 (default 5% slippage):** The slippage setting is the primary MEV attack surface. Sign-then-send is a contributing factor that determines *how easily* that surface is exploited. H015 is the larger risk; H017 amplifies it.

## Recommendation

**For mainnet launch (pre-launch action):**

**Option A — Switch to `wallet.sendTransaction()` (preferred if Phantom mainnet RPC is reliable):**
The original reason for sign-then-send was Phantom devnet TX dropping. Test whether Phantom's mainnet RPC (`signAndSendTransaction`) reliably lands transactions. If yes, revert to the wallet-native path and let wallets that offer MEV protection use it.

```typescript
// In useProtocolWallet.ts — replace wrappedSendTransaction with:
const wrappedSendTransaction = useCallback(
  async (tx, connection, opts) => {
    // Use wallet-adapter's sendTransaction — routes through wallet's own RPC
    // which may include MEV protection on mainnet
    return await sendTransaction(tx, connection, opts);
  },
  [sendTransaction]
);
```

**Option B — Jito bundle submission for swap transactions:**
Integrate Jito tip accounts for swap TXs. This provides MEV protection (private mempool) at the cost of a small tip (0.001-0.01 SOL). Would require adding Jito's block engine endpoint.

**Option C — Document and accept (if A and B are deferred):**
If the decision is made to ship sign-then-send on mainnet, document it explicitly in user-facing copy so users understand MEV protection from their wallet does not apply. Also reduce default slippage (H015 fix) to minimize extractable value.

**Priority ordering:** Fix H015 (5% slippage default) first — that is the primary attack surface amplifier and reduces extraction regardless of submission path. Then evaluate A or B for mainnet launch.
