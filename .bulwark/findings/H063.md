## Verdict: PARTIALLY_FIXED

## Tier: 2

---

## Executive Summary

H063 is the Audit #2 recheck of strategy S004 ("Mainnet Launch Day Attack Bundle"), updated to reflect the Audit #2 constituent findings. The original S004 bundled on-chain gaps (H010 bonding-curve authority, H016 hook init front-run, H005 keypairs in git, H009 devnet fallback) and was marked FIXED in VERIFY-S004 round 3. H063 re-examines the launch-day risk surface using the Audit #2 finding set, which surfaces a **different and more severe combination**: a private key in git history (H001), a committed API key enabling webhook management abuse (H004), protocol-state injection via compromised webhook secret (H005), RPC credit exhaustion via batch amplification (H008), and rate-limit bypass via IP spoofing (H015).

Three of the five constituent findings remain **entirely unmitigated** (H001, H004, H005). Two have documented partial defenses but no structural fixes (H008, H015). The combination creates a multi-vector launch-day scenario where an attacker with read access to the git repository can:
1. Extract the devnet wallet private key (H001) and study the full deployment setup
2. Use the committed Helius API key (H004) to delete the production webhook, blinding the data pipeline
3. Exhaust Helius RPC credits via batch amplification (H008 + H015) before the attack is detected
4. If the webhook secret is separately obtained, inject crafted protocol state to nullify user slippage protection (H005)

The original S004 on-chain components are all FIXED or ACCEPTED_RISK per VERIFY-S004. This finding concerns the off-chain attack surface only.

### H005 — Webhook Secret Compromise → Protocol State Injection
**Current status: NOT FIXED**

Evidence (verified at HEAD, 2026-03-21):
- `app/app/api/webhooks/helius/route.ts:590–606`: Anchor decode output is stored in `protocolStore` without any semantic range validation
- `app/lib/protocol-store.ts:58`: Dedup only blocks identical repeated data, not distinct crafted data
- `app/hooks/useSwap.ts:408–418`: `reserveA`, `reserveB`, and `crimeBuyTaxBps` are read directly from SSE-derived state with no client-side range validation
- No semantic validation layer has been added to `handleAccountChanges`
- No cross-verification against ws-subscriber state has been implemented
- No replay/sequence protection on enhanced webhooks

**Role in H063:** Requires prior acquisition of `HELIUS_WEBHOOK_SECRET`. At mainnet launch, if the webhook secret is obtained (via Railway dashboard breach, social engineering, or supply chain attack), an attacker can inject crafted `PoolState` with `reserveA=10^18` into the protocolStore, causing all connected users' `minimumOutput` computations to produce near-zero values. Users sign transactions with effectively no slippage protection, exposing their full swap value to MEV front-running. Effect persists up to 60 seconds until ws-subscriber next polls.

**Partial note:** The authentication gate (H001-fix: timingSafeEqual, fail-closed) is intact and correctly prevents unauthenticated injection. The vulnerability requires a valid secret, which elevates the prerequisite bar above zero-knowledge exploitation. This is the primary reason H063 is PARTIALLY_FIXED rather than CONFIRMED.

### H008 — RPC Proxy Batch Amplification → Credit Exhaustion
**Current status: NOT FIXED (no batch cap added)**

Evidence (verified at HEAD, 2026-03-21):
- `app/app/api/rpc/route.ts:102–105`: Batch arrays are accepted unconditionally with no size limit
- `checkRateLimit` is called once per HTTP request (line 83), not per method call in batch
- No `MAX_BATCH_SIZE` constant, no `requests.length` guard, no per-element rate counting exists in the current code
- Credit counter at lines 163–165 is observational only, does not gate or throttle

**Role in H063:** Combined with H015 (IP spoofing), a single attacker can exhaust the project's entire monthly Helius credit allocation in under 7 minutes (300 HTTP requests/min × 500 methods/batch × spoofed IPs = unbounded credit burn). At mainnet launch, this kills the RPC proxy (all swaps fail), the ws-subscriber (SSE data pipeline goes dark), and potentially the crank (epoch transitions fail) if they share the same Helius key. Railway's restart policy (`restartPolicyMaxRetries: 3`) means a sustained attack causes permanent downtime until manual intervention.

**Partial note:** The rate limiter is a structural improvement over no rate limiting (the Audit #1 state), but the batch amplification gap means per-IP limits are ineffective against even a single unsophisticated attacker.

### H015 — IP Spoofing Bypasses Per-IP Rate Limits
**Current status: NOT FIXED**

Evidence (verified at HEAD, 2026-03-21):
- `app/lib/rate-limit.ts:129–151`: `getClientIp()` takes `x-forwarded-for[0]` verbatim with no proxy validation
- No rightmost-IP logic, no trusted-proxy allowlist, no `trustProxy` configuration in `next.config.ts`
- A warning is logged in production when no proxy headers are detected (`rate-limit.ts:142–148`), but the fix for the opposite case (headers present but unvalidated) is not implemented
- No Railway proxy documentation has been referenced to validate whether `x-forwarded-for` is overwritten vs. appended

**Role in H063:** IP spoofing is the force multiplier for H008. Without it, a single IP is capped at 300 RPC calls/min (still 150,000 Helius credits/min with 500-item batches). With IP spoofing, rate limiting is fully bypassed and the credit exhaustion becomes unbounded. The same bypass also removes per-IP rate limits from the webhook route (120/min) and sol-price route (30/min), enabling unlimited webhook flooding and further amplification.

## Why This Is PARTIALLY_FIXED (Not CONFIRMED)

The combination is viable but **not at maximum severity** because:

1. **H001 (key exposure)** — The exposed key is devnet only. Mainnet deployer and crank wallet are separate addresses (`23g7xmrtXA6LSWopQcAUgiptGUArSLEMakBKcY1S59YR` and `F84XUxo5VM8FJZeGvC3CrHYwLzFod3ep57CULjZ4ZXc1`). Direct fund theft on mainnet requires a separate key compromise. The devnet key exposure does provide intelligence for timing attacks.

2. **H004 (API key)** — The key is labeled devnet. Whether it spans the mainnet Helius account depends on Helius's account model (one account per key vs. separate devnet/mainnet accounts). This is unverified from source. If separate accounts exist, the devnet key cannot delete the mainnet webhook — significantly limiting Phase 1.

3. **H005 (state injection)** — Requires prior acquisition of `HELIUS_WEBHOOK_SECRET`. The authentication gate is fail-closed and uses `timingSafeEqual`. This is a meaningful prerequisite that prevents zero-knowledge exploitation.

4. **On-chain backstop** — Even with `minimumOutput ≈ 0` from injected reserves, the on-chain `MINIMUM_OUTPUT_FLOOR_BPS = 5000` (50%) in `programs/tax-program/src/constants.rs:40` provides a hard floor. User funds cannot be fully drained by minimumOutput manipulation alone — MEV extraction is capped at the 50% floor.

5. **The S004 on-chain components are genuinely fixed** — H010 (bonding curve authority), H016 (hook init front-running), H009 (devnet RPC fallback) are all fixed per VERIFY-S004 round 3. The on-chain launch-day theft scenario is closed.

## Recommended Remediation Priority

1. **H004 (Helius API key rotation) — URGENT:** Rotate the key before mainnet launch. One day of delay costs nothing; launching with the key live in the repo risks the entire webhook data pipeline.

2. **H001 (Remove .mcp.json key and add to .gitignore) — URGENT:** Replace the private key with a placeholder and add `.mcp.json` to `.gitignore`. Purge git history with BFG before any public repo access.

3. **H008 (Batch size cap) — HIGH, one-line fix:**
   ```typescript
   const MAX_BATCH_SIZE = 10;
   if (requests.length > MAX_BATCH_SIZE) {
     return NextResponse.json({ error: "Batch too large" }, { status: 400 });
   }
   ```

4. **H015 (Rightmost IP extraction) — HIGH:**
   Switch `getClientIp()` to use the rightmost `x-forwarded-for` entry (Railway's appended value), not the first (client-controlled).

5. **H005 (Semantic validation) — HIGH, pre-mainnet required:**
   Add range validation after Anchor decode in `handleAccountChanges`. Reject payloads with implausible reserve values, invalid tax BPS, or impossible epoch numbers before storing to protocolStore.
