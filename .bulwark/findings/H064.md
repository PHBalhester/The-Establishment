# Finding: H064 - Browser-Console Webhook Hijack (RECHECK S008)

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

S008 was marked FIXED on the grounds that the webhook route was modified (auth fail-closed). The recheck verdict is **CONFIRMED** because S008's root attack chain — Helius API key enables webhook management via `api.helius.xyz` — remains fully exploitable. Three independent sub-issues survive:

1. **Webhook URL is not in the client bundle** (partial fix holds for this narrow claim).
2. **H004-confirmed devnet Helius API key (`[REDACTED-DEVNET-KEY]-...`) still embedded in `shared/programs.ts:24`** and is accessible via the browser-bundled package. The key grants full webhook CRUD on `api.helius.xyz`. The CSP `connect-src` for devnet still explicitly whitelists `https://api.helius.xyz`, making the management API callable directly from the browser console.
3. **Mainnet `HELIUS_WEBHOOK_SECRET` (`[REDACTED-WEBHOOK-SECRET]`) is committed verbatim in the tracked git file `.planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md`** — the document notes it was "stored only in Railway env vars (not in any repo file)" but then places it in a markdown table in the same file that was committed to git.

The combination means: an attacker with git read access has the mainnet webhook secret AND the Helius API key needed to enumerate, delete, and redirect webhooks. The original S008 attack chain is fully reconstructable.

---

## Hypothesis Investigated

**Primary:** Webhook URL is not discoverable from client-side code.
**Secondary (H004 link):** Helius API key (H004 CONFIRMED) enables webhook URL discovery via Helius management API.
**Tertiary:** Webhook secret is not extractable from any client-side source.

## Investigation Path

### Check 1: Webhook URL in Client Bundle

**Target:** Any `NEXT_PUBLIC_` variable, hardcoded URL, or import that exposes `/api/webhooks/helius` in client-side code.

**Finding:** The path `/api/webhooks/helius` appears only in server-side files and comments — never as a runtime value in client-bundled code:
- `app/app/api/webhooks/helius/route.ts` — server route handler (never bundled to browser)
- `app/lib/rate-limit.ts:56` — comment only, server-side
- `app/lib/protocol-store.ts:9` — comment only, server-side
- `app/hooks/useProtocolState.ts:26-27` — comment only (the hook itself only opens `/api/sse/protocol`, not the webhook route)

The webhook URL path is NOT in the client bundle. This narrow claim from S008's fix description holds.

**However**, the full webhook URL (`https://dr-fraudsworth-production.up.railway.app/api/webhooks/helius`) was present in the original S008 finding sourced from `scripts/webhook-manage.ts`. That Railway subdomain is publicly known from Railway deployment conventions and the `fraudsworth.fun` domain. The path `/api/webhooks/helius` is a convention visible in any public Next.js project following Helius documentation, making URL discovery trivial regardless of whether it appears in the bundle.

**Verdict on this check: Webhook URL not in bundle (true), but still trivially discoverable via public domain + standard path.**

---

### Check 2: Helius API Key Enables Webhook URL Discovery via API

**Target:** `shared/programs.ts:24`, `app/next.config.ts:19-23`, Helius webhook management API capabilities.

**Finding — Key still present in source:**
```typescript
// shared/programs.ts:24 (current working tree, git-tracked)
export const DEVNET_RPC_URL =
  "https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]";
```

The key `[REDACTED-DEVNET-HELIUS-KEY]` is exported from the `@dr-fraudsworth/shared` package. This package is imported by client-side modules (`app/providers/ClusterConfigProvider.tsx`, `app/hooks/useSwap.ts`, etc.), causing `shared/programs.ts` to be bundled. The key is visible in the browser JS bundle (DevTools → Sources).

**Finding — CSP still allows `api.helius.xyz` for devnet:**
```typescript
// app/next.config.ts:19-23
const heliusApiSources = isMainnet
  ? "https://api-mainnet.helius-rpc.com"
  : "https://api.helius.xyz https://api-devnet.helius-rpc.com";
// ...
// connect-src includes heliusApiSources
```

For the devnet build, `https://api.helius.xyz` is whitelisted in `connect-src`. An attacker can call the Helius webhook management API directly from the browser console on the live devnet site.

**Finding — API key grants full webhook CRUD:**
```bash
# Executable from browser console on the devnet site:
fetch("https://api.helius.xyz/v0/webhooks?api-key=[REDACTED-DEVNET-HELIUS-KEY]")
  .then(r => r.json()).then(d => console.log(JSON.stringify(d, null, 2)))
# Returns: all registered webhooks including webhookURL, webhookID, webhookType, accountAddresses
```

The Helius management API returns the full `webhookURL` of registered webhooks. An attacker does not need to guess the URL — they can enumerate it directly. This makes the "URL not in client bundle" fix meaningless: the webhook URL is one API call away using the still-bundled API key.

**Verdict on this check: CONFIRMED. H004 API key enables full webhook URL enumeration and CRUD.**

### Check 3: Webhook Secret Extractable from Any Source

**Target:** `HELIUS_WEBHOOK_SECRET` in env files, git history, tracked files, and client-side code.

**Finding — Secret committed to tracked git file:**

The file `.planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md` is tracked by git (`git ls-files` confirms) and contains the full mainnet webhook secret in a markdown table:

```markdown
### Webhook Security

| Item | Value |
|------|-------|
| HELIUS_WEBHOOK_SECRET | [REDACTED-WEBHOOK-SECRET] |
| Location | Railway mainnet-web env vars |
```

**Commit:** `0e8ff92` (`docs(98.1-02): complete Cloudflare DNS + Railway mainnet + Helius verification`, 2026-03-15)

This commit is in the permanent git history. The document's own "Decisions Made" section states: *"HELIUS_WEBHOOK_SECRET generated via `openssl rand -hex 32` and stored only in Railway env vars (not in any repo file)"* — this statement is false; the secret appears in the same document four lines earlier.

The secret also appears in the root `.env.mainnet` file (`HELIUS_WEBHOOK_SECRET=[REDACTED-WEBHOOK-SECRET]`), but `.env.mainnet` is correctly gitignored (confirmed: `git ls-files .env.mainnet` returns no match). The SUMMARY.md file is NOT gitignored.

**Finding — Secret not in client bundle (this vector is not present):**
`HELIUS_WEBHOOK_SECRET` has no `NEXT_PUBLIC_` prefix and does not appear in any client-side file. The webhook handler reads it via `process.env.HELIUS_WEBHOOK_SECRET` (server-only). This specific vector does not exist.

**Verdict on this check: CONFIRMED. Webhook secret is in git history via SUMMARY.md commit `0e8ff92`.**

## Enabling Factors

- **`shared/programs.ts:24`**: Devnet Helius API key bundled into client JS. Not removed since Audit #1.
- **`app/next.config.ts:23`**: `connect-src` for devnet whitelists `https://api.helius.xyz`. Enables management API calls from browser console.
- **`.planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md` (commit `0e8ff92`)**: Mainnet `HELIUS_WEBHOOK_SECRET` committed verbatim to a tracked planning file. The documenting author believed it was not in any repo file, but the act of documenting it in a committed SUMMARY made it so.
- **No webhook health monitoring**: No automated check that the production webhook exists, points to the correct URL, or is delivering data. Silent pipeline failure on deletion.

---

## Blocking Factors

- **Mainnet CSP**: `heliusApiSources` for mainnet is `https://api-mainnet.helius-rpc.com`, NOT `https://api.helius.xyz`. The management API endpoint is not whitelisted for mainnet builds. An attacker must call the management API externally (not from the browser console) for the mainnet variant of this attack — still possible, just not from a browser on the site.
- **Devnet API key ≠ Mainnet API key**: The bundled key `[REDACTED-DEVNET-KEY]-...` is devnet-only. The mainnet management API key `[REDACTED-MAINNET-KEY]-...` is in the untracked `.env.mainnet`. However, `[REDACTED-MAINNET-KEY]-...` is also in the tracked SUMMARY.md at line 122-124.

## Recommended Fixes

**Fix 1 (CRITICAL — Immediate): Rotate the mainnet HELIUS_WEBHOOK_SECRET.**
The secret `[REDACTED-WEBHOOK-SECRET]` is in permanent git history (commit `0e8ff92`, file `.planning/phases/98.1-production-infrastructure-staging/98.1-02-SUMMARY.md`). Any git clone or GitHub repository access exposes it. Generate a new secret, update Railway env vars, and update the Helius webhook `authHeader` configuration simultaneously.

**Fix 2 (CRITICAL — Immediate): Redact or remove the secret from the tracked SUMMARY.md.**
Redact the table value in the current file and consider a git-filter-repo purge of the commit if the repo is or will be public. At minimum, update the file to remove the plaintext value.

**Fix 3 (HIGH): Remove `[REDACTED-DEVNET-KEY]-...` from `shared/programs.ts`.**
The devnet Helius API key provides webhook management access (list, delete, re-register). Move it to an environment variable not bundled into client code, or create a separate read-only Helius key for client-side RPC only.

**Fix 4 (HIGH): Remove `https://api.helius.xyz` from devnet CSP `connect-src`.**
The frontend has no legitimate need to call the Helius management API. Removing it prevents browser-console execution of the attack. The RPC endpoint (`devnet.helius-rpc.com`) and the management API (`api.helius.xyz`) are different hosts.

**Fix 5 (MEDIUM): Add webhook health monitoring.**
A server-side scheduled check (e.g., every 5 minutes) should verify the webhook exists, points to the correct URL, and has been delivering data recently. Alert (Sentry or Railway log) on failure. This limits the dwell time of a deletion attack.
