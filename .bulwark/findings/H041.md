## Information Value to Attacker â€” Assessment

### 1. Server restart / deployment window detection (`credits.startedAt`)

`credits.startedAt` is an ISO timestamp that resets every time the process restarts (class constructor at `credit-counter.ts:31`). An attacker who polls `/api/health` even once per minute can detect:
- Exact deployment times (new `startedAt` = new process = deployment or crash)
- Crash recovery windows (process restarted â†’ pipeline still re-seeding â†’ `wsSubscriber.initialized = false`)

**Attack use case**: Time a sandwich attack or rapid-fire swap during the `initialized = false` window when protocolStore is empty and SSE clients are disconnected.

### 2. WS subscriber degradation window (`wsSubscriber.wsConnected`, `wsSubscriber.fallbackActive`)

When `wsConnected = false` and `fallbackActive = true`, the server has lost its real-time Helius WebSocket and is falling back to HTTP polling. During this window:
- Protocol state updates are less frequent
- The SSE broadcast pipeline is running on slower, possibly stale data
- The crank is not impaired (it uses its own RPC connection), but the frontend state diverges

**Attack use case**: An automated trader monitoring this endpoint gains advance notice of a degraded-monitoring window when on-chain state changes will propagate to the UI more slowly â€” potentially enabling front-running or toxic order flow.

### 3. Trading activity inference (`credits.methodCounts`)

The per-method breakdown reveals:
- `getTokenSupply` call count / time elapsed â†’ supply poll frequency
- `getProgramAccounts` call count â†’ staker gPA poll frequency; a sudden spike could indicate a large staking event was processed
- `getAccountInfo` count â†’ rough proxy for swap/webhook processing volume since last restart

None of these are precise enough to reconstruct individual user transactions, but the aggregate gives an attacker a real-time "market activity heat" signal without spending any Helius credits.

### 4. Current Solana slot (`wsSubscriber.latestSlot`)

The latest confirmed slot is publicly available via any Solana RPC anyway. Its presence here is low incremental value, but combined with `lastSlotReceivedAt`, an attacker can compute pipeline latency: `Date.now() - lastSlotReceivedAt` = staleness of the server's state. A large staleness value confirms the server's SSE data is stale before the frontend has noticed.

### 5. DoS amplification (unchanged from H028)

Each unauthenticated call to `/api/health` may trigger a live Postgres `SELECT 1` and/or Solana `getSlot` RPC call (the Solana check falls back to live RPC when `protocolStore.__slot` is not cached). No rate limiting is present on this endpoint. This remains unchanged from H028.

## Recommended Fix

**Preferred (tiered response):** Split liveness from diagnostics.

```typescript
export async function GET(req: Request) {
  const authHeader = req.headers.get("authorization");
  const isMonitor = process.env.HEALTH_CHECK_SECRET &&
    authHeader === `Bearer ${process.env.HEALTH_CHECK_SECRET}`;

  // Liveness only â€” always returned, for Railway and public callers
  const liveness = { status: "ok", timestamp: new Date().toISOString() };

  if (!isMonitor) {
    return NextResponse.json(liveness);
  }

  // Full diagnostics â€” authenticated callers only (monitoring dashboard, ops team)
  const [postgres, solanaRpc] = await runChecks();
  return NextResponse.json({
    ...liveness,
    status: postgres && solanaRpc ? "ok" : "degraded",
    checks: { postgres, solanaRpc },
    wsSubscriber: getWsSubscriberStatus(),
    credits: creditCounter.getStats(),
  });
}
```

**Minimum viable fix (no new env var):** Remove `wsSubscriber` and `credits` from the public response. The `checks` object (H028) is a lower-value disclosure â€” retain it or remove it as preferred.

**Rate limiting:** Regardless of content change, add rate limiting to `/api/health` (e.g., 60 req/min/IP) to prevent DoS amplification via the Postgres `SELECT 1` + RPC fallback path.
