## 3. Failed Verification: INV-STAKE-008

### Description
**Invariant:** Silent u64 truncation boundary characterization in `update_rewards`
**Tool:** Kani
**Status:** FAILED (harness bug)

### Root Cause

The Kani harness at line 240 contains an assertion that computes `claim_u128 * PRECISION` using unchecked u128 multiplication. When the harness explores the domain where `claim_u128 > u64::MAX`, this intermediate multiplication overflows u128, triggering a Kani assertion failure at "check 3".

```rust
// HARNESS BUG (line 240):
let verification_product = claim_u128 * PRECISION;  // Overflows u128!

// FIX:
let verification_product = claim_u128.checked_mul(PRECISION);
```

### Impact Assessment

**Severity: LOW (harness bug, not production code)**

The production code at `math.rs:50` uses `as u64` truncation, which is the documented known behavior this harness characterizes. All named invariant assertions (checks 4, 5, 6, 7) passed successfully:
- Check 4: When `claim_u128 <= u64::MAX`, `as u64` cast is lossless
- Check 5: When `claim_u128 > u64::MAX`, `as u64` truncates (silent data loss)
- Check 6: Truncated value is always less than the true value
- Check 7: The truncation boundary is exactly `u64::MAX`

### Exploit Scenario (if the *underlying* truncation in production code were exploitable)

The `as u64` cast at `math.rs:50` silently truncates when `balance * reward_delta / PRECISION > u64::MAX`. With current PROFIT supply (50M = 5e13 base units), this requires `reward_delta > 3.688e23` (approximately 368,800 PRECISION units per token). This threshold is unreachable under normal operation but could be reached after ~31 epochs of dead-stake-only operation with maximum reward deposits.

### Recommended Fix

Replace the unchecked `as u64` cast with `try_from`:

```rust
// BEFORE (math.rs:50):
let pending = (balance_u128
    .checked_mul(reward_delta)
    .ok_or(StakingError::Overflow)?
    .checked_div(PRECISION)
    .ok_or(StakingError::DivisionByZero)?) as u64;

// AFTER:
let pending = u64::try_from(
    balance_u128
        .checked_mul(reward_delta)
        .ok_or(StakingError::Overflow)?
        .checked_div(PRECISION)
        .ok_or(StakingError::DivisionByZero)?
).map_err(|_| StakingError::Overflow)?;
```

**Risk: LOW** -- Currently unreachable with PROFIT's supply cap. The `try_from` change converts a silent data loss into an explicit error, which is safer but would also block the affected user from claiming. Consider whether a cap on `reward_delta` is preferable.

## 8. Known Findings from Invariant Analysis (Not Verification Failures)

These are architectural observations discovered during invariant specification, not failures in the verification run.

### FINDING-A: Sell Path Has No Slippage Floor (INV-CARN-014)
- **Severity:** LOW (Accepted Risk)
- **Description:** The Carnage sell step (selling held tokens for WSOL) has no slippage floor. Only the buy step is checked.
- **Mitigation:** Analysis shows the failure mode is denial-of-service (Carnage rejection), not fund theft. The buy-side floor uses TARGET pool reserves, so manipulation of the held-token pool causes the buy to fail rather than execute at a bad price.
- **Recommendation:** Document as accepted risk. A sell-side floor could be added for defense-in-depth but would increase complexity.

### FINDING-B: Three Independent Pool Reserve Readers (INV-CARN-006)
- **Severity:** LOW (Maintainability)
- **Description:** Three copies of the raw byte pool reader exist in `execute_carnage_atomic.rs`, `execute_carnage.rs`, and `pool_reader.rs`. If AMM PoolState layout changes, all three must be updated.
- **Recommendation:** Extract to a shared helper or add a cross-program layout test.

### FINDING-C: `calculate_output_floor` Missing `floor_bps` Validation (INV-TAX-016)
- **Severity:** LOW (Defense-in-depth)
- **Description:** Unlike `calculate_tax` which rejects `bps > 10000`, `calculate_output_floor` has no guard on `floor_bps`. Currently all callers pass valid constants, but the function itself is unguarded.
- **Recommendation:** Add `if floor_bps > 10000 { return None; }` guard.
