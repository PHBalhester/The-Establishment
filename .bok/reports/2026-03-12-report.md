## 2. Findings

### Finding 1: Vault Solvency — Rounding Dust (MEDIUM)

**Source:** Proptest `inv_bc_007_vault_solvency`, LiteSVM `inv_bc_014b_sequential_sells`
**Program:** `bonding-curve`

**Description:** Accumulated rounding dust from many micro-purchases/sells can erode the bonding curve vault below rent-exempt minimum. LiteSVM counterexample: vault=890,875 vs rent=890,880 (off by 5 lamports after sequential sells).

**Root cause:** Each `calculate_sol_for_tokens()` floors the result. Individually safe, but over hundreds of operations the cumulative shortfall can breach rent-exempt minimum.

**Exploit scenario:** An attacker repeatedly buys 1 token at a time (bypassing MIN_PURCHASE_SOL if possible), accumulating rounding in their favor. After enough operations, the vault account drops below rent-exempt minimum and could be garbage-collected by the Solana runtime, destroying all remaining funds.

**Practical risk:** LOW — `MIN_PURCHASE_SOL` prevents 1-token buys on-chain, limiting the accumulation rate. But the guard is on purchase size, not on vault health.

**Suggested fix:**
```rust
// BEFORE (programs/bonding-curve/src/instructions/sell_tokens.rs):
let payout = calculate_sol_for_tokens(tokens_sold, curve_position)?;
// Transfer payout to user...

// AFTER:
let payout = calculate_sol_for_tokens(tokens_sold, curve_position)?;
let vault_after = vault_balance.checked_sub(payout)
    .ok_or(BondingCurveError::InsufficientVaultBalance)?;
require!(
    vault_after >= rent_exempt_minimum,
    BondingCurveError::VaultBelowRentExempt
);
// Transfer payout to user...
```

### Finding 2: u128 Overflow at Extreme Staking Values (LOW)

**Source:** Proptest `inv_stake_005_precision_overflow`
**Program:** `staking`

**Description:** `balance * delta` overflows u128 when both values are extremely large (balance=456T, delta_scale=745B).

**Root cause:** Missing `checked_mul` — the code uses raw `*` operator for u128 multiplication.

**Exploit scenario:** Not exploitable in practice. Protocol bounds are 20M PROFIT max supply and ~1000 SOL/epoch max rewards, keeping both operands well within u128 range. However, defense-in-depth demands explicit overflow protection.

**Suggested fix:**
```rust
// BEFORE (programs/staking/src/state/rewards.rs):
let user_reward = balance * reward_delta;

// AFTER:
let user_reward = balance.checked_mul(reward_delta)
    .ok_or(StakingError::ArithmeticOverflow)?;
```

**v1.3 coverage:** Phase 79 (Financial Safety)

### Finding 3: Price Accuracy Threshold Exceeded (INFORMATIONAL)

**Source:** Proptest `inv_bc_011_price_accuracy`
**Program:** `bonding-curve`

**Description:** Integer approximation exceeds 0.01% threshold by 0.0001% at curve position 94,010,200,000,000. The error is 1 part in 140,000.

**Root cause:** Inherent limitation of integer arithmetic at specific curve positions. The PRECISION constant doesn't fully compensate for truncation at this range.

**Exploit scenario:** None. 0.0101% error means a user paying 10 SOL gets 0.00101 SOL less/more than the theoretical continuous price. Rounding always favors the protocol (verified by `inv_stake_007_truncation_favors_protocol`).

**Suggested fix:**
```rust
// Option A: Relax threshold to 0.02% (acceptable for DeFi)
// No code change — update documentation only

// Option B: Increase PRECISION constant for this range
// Requires analysis of impact on other curve positions
```

**v1.3 coverage:** Phase 80 (Defense-in-Depth)

| Property | Program | Proptest Status |

| Output bounded by reserve | AMM | PASSED (10K) |

| Round-trip value | Bonding Curve | PASSED (10K) |
| Input monotonicity | Bonding Curve | PASSED (10K) |
| Sell tax ceil ≥ floor | Bonding Curve | PASSED (10K) |

All 6 are validated by proptest. Inconclusive means "not formally proven for ALL inputs" — not that the properties are false.

**Future improvement options:**
1. Tighter variable bounds (restrict to realistic protocol ranges)
2. Decomposed proofs (prove sub-steps independently)
3. Alternative solver (Bitwuzla/Z3 may handle u128 bit-vectors better)
4. Reduced precision proofs (u16 exhaustive for complex formulas)
