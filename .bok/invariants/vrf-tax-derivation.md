### INV-TD-001: Flip Probability Is Exactly 75%

**Function:** `derive_taxes` at `tax_derivation.rs:86`
**Pattern:** VP-096 (Modulo Bias) / VP-097 (Distribution Fairness)
**Tool:** Proptest
**Confidence:** high

**Plain English:** The cheap-side flip happens when byte 0 is strictly less than 192. Since byte 0 is uniform over [0, 255], exactly 192 of the 256 possible values trigger a flip, giving a probability of 192/256 = 75.0% with zero modulo bias.

**Why It Matters:** If the threshold were off by even 1 (e.g., `<= 192` giving 193/256 = 75.39%), the flip frequency would deviate from spec. Over thousands of epochs, this biases one side as "cheap" more often, creating a persistent arbitrage asymmetry that informed traders could exploit for risk-free profit against less-informed users.

**Formal Property:**
```
For all b in [0, 255]:
  flip(b) = (b < 192)

P(flip) = |{b : b < 192}| / 256 = 192 / 256 = 0.75

Boundary: flip(191) = true, flip(192) = false
```

**Verification Approach:**
Proptest with `any::<u8>()` for byte 0. Exhaustive check: iterate all 256 values, count how many satisfy `b < 192`, assert count == 192. Additionally, boundary test: assert `derive_taxes` with byte0=191 flips, byte0=192 does not. The existing tests at lines 147-167 cover boundaries but not exhaustive counting.

### INV-TD-002: Low Rate Table Is Exactly {100, 200, 300, 400} BPS

**Function:** `derive_taxes` at `tax_derivation.rs:23, 89, 91`
**Pattern:** VP-096 (Modulo Bias)
**Tool:** Proptest
**Confidence:** high

**Plain English:** The LOW_RATES lookup table contains exactly four values: 100, 200, 300, 400 basis points. Since the index is computed as `byte % 4`, and `byte % 4` produces values in {0, 1, 2, 3} for any u8 input, the output is guaranteed to be one of these four values. No other tax rate is possible.

**Why It Matters:** If a code change added a 5th element to LOW_RATES but the modulo stayed at 4, the extra element would be unreachable. Conversely, if the modulo changed to 5 with only 4 elements, an out-of-bounds panic would crash the program on-chain (causing transaction failure, not exploit, but still a liveness issue). The invariant ensures the table size and modulo are synchronized.

**Formal Property:**
```
LOW_RATES = [100, 200, 300, 400]
|LOW_RATES| = 4

For all b in [0, 255]:
  LOW_RATES[b % 4] in {100, 200, 300, 400}

Exhaustive: { LOW_RATES[b % 4] | b in [0, 255] } = {100, 200, 300, 400}
```

**Verification Approach:**
Proptest with `any::<u8>()` for bytes 1 and 3. For each generated byte, compute `byte % 4`, index into LOW_RATES, and assert the result is in {100, 200, 300, 400}. Exhaustive variant: iterate all 256 byte values, collect unique results, assert the set equals {100, 200, 300, 400} exactly. Also assert `LOW_RATES.len() == 4` as a compile-time-compatible check.

**Plain English:** The HIGH_RATES lookup table contains exactly four values: 1100, 1200, 1300, 1400 basis points. The same `byte % 4` indexing guarantees only these four values are ever produced.

```
For all b in [0, 255]:

```

**Verification Approach:**
Same as INV-TD-002 but for HIGH_RATES. Iterate all 256 byte values, confirm each maps to one of {1100, 1200, 1300, 1400}. Assert `HIGH_RATES.len() == 4`.

### INV-TD-004: Modulo-4 Bias Is Exactly Zero

**Function:** `derive_taxes` at `tax_derivation.rs:89-92`
**Pattern:** VP-096 (Modulo Bias)
**Tool:** Proptest (exhaustive)
**Confidence:** high

**Plain English:** When computing `byte % 4` for a uniformly distributed byte in [0, 255], each residue class {0, 1, 2, 3} occurs exactly 64 times out of 256. This means each of the four rates in LOW_RATES and HIGH_RATES is selected with exactly 25% probability, with zero modulo bias. This is because 4 divides 256 evenly (256 / 4 = 64).

**Why It Matters:** Modulo bias occurs when `n` does not divide `2^k`. Here n=4 divides 256=2^8 evenly, so there is no bias. If the rate table ever expanded to, say, 5 entries (which does NOT divide 256), the distribution would be uneven: residues 0-3 would each have 52/256 probability while residue 4 would have 48/256, a ~8% relative bias toward the first entry's rate.

**Formal Property:**
```
For k in {0, 1, 2, 3}:
  |{b in [0, 255] : b % 4 == k}| = 64

256 % 4 == 0  (no remainder, perfect division)

P(rate = LOW_RATES[k]) = 64/256 = 25.0% for each k
```

**Verification Approach:**
Exhaustive count: for each residue class k in {0,1,2,3}, count how many of the 256 byte values produce `b % 4 == k`. Assert each count is exactly 64. This is a mathematical property of 4|256 but should be verified as a regression guard against future changes to table size.

### INV-TD-005: CRIME and FRAUD Use Independent VRF Bytes

**Function:** `derive_taxes` at `tax_derivation.rs:89-92`
**Pattern:** VP-098 (Seed Derivation Determinism)
**Tool:** Proptest
**Confidence:** high

**Plain English:** CRIME magnitude is derived from bytes 1 (low) and 2 (high). FRAUD magnitude is derived from bytes 3 (low) and 4 (high). These are four distinct byte positions. No byte is shared between CRIME and FRAUD derivations, ensuring the two tokens' tax rates are statistically independent.

**Why It Matters:** If CRIME and FRAUD inadvertently shared a VRF byte (e.g., both reading byte 1 for their low rate), their magnitudes would be perfectly correlated. This would make the system predictable: knowing CRIME's rate would reveal FRAUD's rate, halving the entropy and enabling more efficient MEV extraction by traders who detect the regime first.

**Formal Property:**
```
crime_low_byte_index  = 1
crime_high_byte_index = 2
fraud_low_byte_index  = 3
fraud_high_byte_index = 4

{1, 2} intersection {3, 4} = empty set

Additionally: flip uses byte 0, which is not in {1, 2, 3, 4}.
All 5 decisions use distinct bytes: {0, 1, 2, 3, 4}.
```

**Verification Approach:**
Proptest: generate random 32-byte arrays, call `derive_taxes` with two arrays that differ ONLY in byte 1 -- assert CRIME rates change but FRAUD rates do not. Repeat for bytes 2, 3, 4 in isolation. This proves each byte controls only its designated output. Alternatively, source-code audit confirms indices are hardcoded literals {0, 1, 2, 3, 4} at lines 86, 89, 90, 91, 92 with no computed indices.

### INV-TD-006: Cheap Side Gets Low Buy Tax, High Sell Tax

**Function:** `derive_taxes` at `tax_derivation.rs:106-117`
**Pattern:** VP-097 (Distribution Fairness -- game-theoretic correctness)
**Tool:** Proptest
**Confidence:** high

**Plain English:** When a token is designated as the "cheap side," its buy tax comes from that token's own LOW_RATES roll (1-4% range), and its sell tax comes from that token's own HIGH_RATES roll (11-14% range). The expensive side gets the inverse: high buy tax, low sell tax.

**Why It Matters:** If the assignment were inverted (cheap side gets HIGH buy tax), it would destroy the core game mechanic: the cheap side is supposed to be attractive to buy. Inverting this would make the "cheap" token actually more expensive to purchase, removing all trading incentive from regime flips and collapsing protocol volume to near zero.

**Formal Property:**
```
For all vrf_result, current_cheap:
  let config = derive_taxes(vrf_result, current_cheap)

  If config.cheap_side == Crime:
    config.crime_buy_tax_bps in {100, 200, 300, 400}     -- low range
    config.crime_sell_tax_bps in {1100, 1200, 1300, 1400} -- high range
    config.fraud_buy_tax_bps in {1100, 1200, 1300, 1400}  -- high range (expensive)
    config.fraud_sell_tax_bps in {100, 200, 300, 400}      -- low range (expensive)

  If config.cheap_side == Fraud:
    config.fraud_buy_tax_bps in {100, 200, 300, 400}      -- low range
    config.fraud_sell_tax_bps in {1100, 1200, 1300, 1400}  -- high range
    config.crime_buy_tax_bps in {1100, 1200, 1300, 1400}   -- high range (expensive)
    config.crime_sell_tax_bps in {100, 200, 300, 400}       -- low range (expensive)
```

**Verification Approach:**
Proptest: generate arbitrary 32-byte VRF arrays and both Token variants as current_cheap. Call `derive_taxes`, then check that the cheap token's buy tax is in the low set and sell tax is in the high set, and vice versa for the expensive token. Run with 10,000+ cases to exercise all byte combinations.

### INV-TD-007: No Tax Rate Outside Defined Ranges

**Function:** `derive_taxes` at `tax_derivation.rs:84-128`
**Pattern:** VP-097 (Distribution Fairness)
**Tool:** Proptest
**Confidence:** high

**Plain English:** For any possible 32-byte VRF input and any current_cheap value, every per-token tax rate in the output TaxConfig is a member of either LOW_RATES or HIGH_RATES. No intermediate, zero, or out-of-range value is possible.

**Why It Matters:** A tax rate of 0 bps would mean zero-fee trading, allowing MEV bots to extract value from pools without contributing to the protocol. A rate exceeding 1400 bps (~14%) would be punitive beyond spec and could trap user funds if the round-trip tax exceeded 100%.

**Formal Property:**
```
VALID_RATES = {100, 200, 300, 400, 1100, 1200, 1300, 1400}

For all vrf in [u8; 32], current_cheap in {Crime, Fraud}:
  let c = derive_taxes(vrf, current_cheap)
  c.crime_buy_tax_bps  in VALID_RATES
  c.crime_sell_tax_bps in VALID_RATES
  c.fraud_buy_tax_bps  in VALID_RATES
  c.fraud_sell_tax_bps in VALID_RATES
  c.low_tax_bps  == 0  (legacy field)
  c.high_tax_bps == 0  (legacy field)
```

**Verification Approach:**
Proptest: generate arbitrary `[u8; 32]` and `Token` (via `prop_oneof![Just(Crime), Just(Fraud)]`). Call `derive_taxes`, assert all four rate fields are in VALID_RATES and both legacy fields are 0. This is the most important "catch-all" invariant -- any future code change that introduces an unexpected rate will be caught.

### INV-TD-008: Flip Is a Toggle (Double Flip = Identity)

**Function:** `derive_taxes` at `tax_derivation.rs:97-101`, `Token::opposite` at `enums.rs:20-24`
**Pattern:** VP-098 (Seed Derivation Determinism)
**Tool:** Proptest
**Confidence:** high

**Plain English:** The flip operation uses `Token::opposite()`, which is an involution: `opposite(opposite(x)) == x`. If two consecutive epochs both flip, the cheap side returns to its original value. This ensures the system oscillates rather than drifting.

**Why It Matters:** If `opposite()` were not an involution (e.g., due to a third token variant added carelessly), flipping could become one-directional, permanently locking one token as the cheap side. This would eliminate the regime-change trading dynamic entirely.

**Formal Property:**
```
For all t in {Crime, Fraud}:
  t.opposite().opposite() == t

For all t in {Crime, Fraud}:
  t.opposite() != t
```

**Verification Approach:**
This is already tested at `enums.rs:130-134` (`test_token_double_opposite`). Proptest adds coverage by checking that for any VRF bytes where byte0 < 192, the resulting cheap_side is `current_cheap.opposite()`, and where byte0 >= 192, it equals `current_cheap` unchanged.

### INV-TD-009: Deterministic Output (Same Input = Same Output)

**Function:** `derive_taxes` at `tax_derivation.rs:84-128`
**Pattern:** VP-098 (Seed Derivation Determinism)
**Tool:** Proptest
**Confidence:** high

**Plain English:** `derive_taxes` is a pure function with no side effects, no timestamps, no global state reads. Given the same 32-byte VRF input and the same `current_cheap` value, it always returns the identical `TaxConfig`.

**Why It Matters:** Non-determinism in tax derivation would mean different validators could disagree on the epoch's tax rates, causing transaction failures or (worse) different users experiencing different tax regimes within the same epoch. On Solana, non-determinism in BPF programs typically comes from uninitialized memory or pointer comparisons -- neither applies here since the function uses only array indexing and integer comparisons.

**Formal Property:**
```
For all vrf in [u8; 32], current_cheap in {Crime, Fraud}:
  derive_taxes(vrf, current_cheap) == derive_taxes(vrf, current_cheap)
```

**Verification Approach:**
Proptest: generate arbitrary inputs, call `derive_taxes` twice with the same arguments, assert `TaxConfig` equality via `PartialEq`. The struct derives `Eq` (line 37), confirming the comparison is structural. Run with 10,000+ cases.

### INV-TD-010: Cheap Side Buy Tax Is Always Strictly Less Than Expensive Side Buy Tax

**Function:** `derive_taxes` at `tax_derivation.rs:106-117`
**Pattern:** VP-097 (Distribution Fairness -- game-theoretic correctness)
**Tool:** Proptest
**Confidence:** high

**Plain English:** The cheap side's buy tax is drawn from LOW_RATES (max 400 bps), while the expensive side's buy tax is drawn from HIGH_RATES (min 1100 bps). Since max(LOW_RATES) = 400 < 1100 = min(HIGH_RATES), the cheap side's buy tax is always strictly less than the expensive side's buy tax. The gap is at least 700 bps (7%).

**Why It Matters:** If the ranges overlapped, there could be epochs where both tokens have the same buy tax, removing the incentive to preferentially buy the "cheap" token. The 700+ bps gap ensures the regime designation is always economically meaningful.

**Formal Property:**
```
For all vrf, current_cheap:
  let c = derive_taxes(vrf, current_cheap)

  If c.cheap_side == Crime:
    c.crime_buy_tax_bps < c.fraud_buy_tax_bps
    c.fraud_buy_tax_bps - c.crime_buy_tax_bps >= 700

  If c.cheap_side == Fraud:
    c.fraud_buy_tax_bps < c.crime_buy_tax_bps
    c.crime_buy_tax_bps - c.fraud_buy_tax_bps >= 700
```

**Verification Approach:**
Proptest: for each generated input, compute `derive_taxes`, identify the cheap side, and assert the cheap buy tax is strictly less than the expensive buy tax with a minimum gap of 700 bps. This also serves as a regression test for the rate table values themselves.

### INV-TD-011: No Byte Index Collision Across Tax and Carnage Subsystems

**Function:** `derive_taxes` at `tax_derivation.rs:86-92`, `is_carnage_triggered` at `carnage.rs:26`, `get_carnage_action` at `carnage.rs:41`, `get_carnage_target` at `carnage.rs:59`
**Pattern:** VP-098 (Seed Derivation Determinism -- isolation)
**Tool:** Source audit (static)
**Confidence:** high

**Plain English:** Tax derivation uses VRF bytes {0, 1, 2, 3, 4}. Carnage uses VRF bytes {5, 6, 7}. These sets are disjoint. No byte is read by both subsystems.

**Why It Matters:** If tax and carnage shared a byte, the carnage trigger probability would be correlated with the tax regime. For example, if byte 0 were reused for carnage triggering, carnage would occur more often in non-flip epochs (byte >= 192), creating a predictable pattern that traders could front-run.

**Formal Property:**
```
TAX_BYTES = {0, 1, 2, 3, 4}
CARNAGE_BYTES = {5, 6, 7}

TAX_BYTES intersection CARNAGE_BYTES = empty set
TAX_BYTES union CARNAGE_BYTES = {0, 1, 2, 3, 4, 5, 6, 7}
```

**Verification Approach:**
Static source audit only. The byte indices are hardcoded integer literals in both files. A grep-based CI check could verify that `tax_derivation.rs` only references `vrf_result[0]` through `vrf_result[4]` and `carnage.rs` only references `vrf_result[5]` through `vrf_result[7]`. Any future refactor that introduces a shared index would be caught.
