# Milestone v0.5: Epoch/VRF Program

**Status:** SHIPPED 2026-02-06
**Phases:** 22-25
**Total Plans:** 16

## Overview

Build the Epoch/VRF Program — the randomness engine that determines tax rates and executes Carnage rebalancing operations with cryptographic unpredictability.

Four phases build from foundation to full execution: Phase 22 establishes EpochState core with slot-based timing and Tax Program integration. Phase 23 adds Switchboard VRF for manipulation-proof randomness with anti-reroll protection. Phase 24 connects staking yield finalization. Phase 25 delivers Carnage Fund execution at CPI depth 4 with atomic guarantees.

## Phases

### Phase 22: EpochState Core + Tax Integration

**Goal:** Establish EpochState account structure with slot-based epoch timing, and update Tax Program to read dynamic tax rates instead of hardcoded values.

**Depends on:** Phase 21 (v0.4 Tax Program complete)

**Requirements:** EPO-01, EPO-02, EPO-03, EPO-04, EPO-05, EPO-06, TAX-01, TAX-02, TAX-03

**Success Criteria:**
1. EpochState account exists with genesis epoch initialized (CRIME cheap side, 300/1400 bps rates)
2. Tax Program reads tax rates from EpochState instead of hardcoded 400 bps
3. Epoch boundaries are calculated from slots (SLOTS_PER_EPOCH = 4,500)
4. Tax Program validates carnage_signer PDA with seeds::program = EPOCH_PROGRAM_ID
5. swap_exempt instruction accepts Epoch Program's carnage_signer as authority

**Plans:** 4 plans

- [x] 22-01-PLAN.md — Epoch Program foundation with EpochState account and initialize_epoch_state instruction
- [x] 22-02-PLAN.md — Tax Program EpochState reader module for cross-program deserialization
- [x] 22-03-PLAN.md — Update swap_sol_buy to read dynamic tax rates from EpochState
- [x] 22-04-PLAN.md — Update swap_sol_sell to read dynamic tax rates from EpochState

### Phase 23: VRF Integration + Anti-Manipulation

**Goal:** Integrate Switchboard On-Demand VRF with anti-manipulation protection (anti-reroll, freshness validation, timeout recovery) to determine tax rates with cryptographic unpredictability.

**Depends on:** Phase 22

**Requirements:** VRF-01, VRF-02, VRF-03, VRF-04, VRF-05, VRF-06, VRF-07, VRF-08, VRF-09, VRF-10, EPO-07, EPO-08

**Success Criteria:**
1. Epoch transitions can be triggered permissionlessly when epoch boundary reached (anyone can call, 0.01 SOL bounty paid)
2. VRF randomness determines new tax rates via "cheap side" regime flip (75% probability) and discrete tax bands (100-400 bps or 1100-1400 bps)
3. Anti-reroll protection prevents randomness substitution (pending_randomness_account bound at commit, verified at consume)
4. Stale randomness rejected (seed_slot freshness check within 1 slot, not-yet-revealed validation)
5. Protocol cannot deadlock from VRF failure (300-slot timeout with permissionless retry_epoch_vrf)

**Plans:** 4 plans

- [x] 23-01-PLAN.md — VRF foundation: switchboard-on-demand dependency, tax derivation helper, VRF events
- [x] 23-02-PLAN.md — trigger_epoch_transition instruction with randomness validation and anti-reroll binding
- [x] 23-03-PLAN.md — consume_randomness instruction with anti-reroll verification and tax derivation
- [x] 23-04-PLAN.md — retry_epoch_vrf instruction and TypeScript devnet test script

### Phase 25: Carnage Fund Execution

**Goal:** Implement Carnage Fund rebalancing with ~4.3% trigger probability per epoch, VRF-determined action/target, and atomic execution at CPI depth 4.

**Depends on:** Phase 24

**Requirements:** CRN-01, CRN-02, CRN-03, CRN-04, CRN-05, CRN-06, CRN-07, CRN-08, CRN-09, CRN-10, CRN-11, CRN-12

**Success Criteria:**
1. CarnageFundState account and three vault PDAs (SOL/CRIME/FRAUD) exist and are initialized
2. Carnage triggers ~4.3% of epochs (VRF byte 3 < 11) with action (98% burn / 2% sell) and target (50/50 CRIME/FRAUD) determined by VRF bytes 4-5
3. Carnage executes atomically within consume_randomness using Tax::swap_exempt (0% tax, carnage_signer PDA validated)
4. Full CPI chain works at depth 4 (Epoch -> Tax -> AMM -> Token-2022 -> Hook)
5. Fallback mechanisms exist: execute_carnage (100-slot deadline) and expire_carnage (clears pending state, retains SOL)

## Milestone Summary

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| EpochState size 100 bytes (8 + 92) | Spec had 93-byte error; verified field-by-field | ✓ Good |
| Switchboard On-Demand v0.11.3 exact pin | On-chain code frozen at deploy | ✓ Good |
| 75% flip threshold at byte < 192 | 192/256 = 75% probability per spec | ✓ Good |
| VRF_TIMEOUT_SLOTS = 300 (~2 min) | Balance between retry speed and oracle time | ✓ Good |
| Bounty payment deferred | Treasury infrastructure not in scope | ⚠ Tech debt |
| Token-2022 burn via manual instruction | Avoid spl_token_2022 crate compatibility issues | ✓ Good |
| Carnage state read before mutable borrow | Rust borrow checker satisfaction pattern | ✓ Good |

**Issues Resolved:**

- Cross-program EpochState deserialization with owner validation
- Anti-reroll protection via pending_randomness_account binding
- CPI depth 4 limit respected for Carnage execution
- Fallback execute_carnage gap closed (25-06)

**Technical Debt:**

- Tax Program epoch_program_id() placeholder needs update post-deployment
- VRF-05 bounty payment emits 0 (treasury integration deferred)
- Carnage event return values are approximations
