## Overview

v0.7 wired the 5 on-chain programs together and proved the full protocol works on devnet with real transactions. The journey started with fixing program ID placeholders that blocked cross-program calls, built local integration test infrastructure to validate every CPI path before touching devnet, automated the ~32-transaction deployment sequence into idempotent scripts, deployed to devnet, validated Switchboard VRF with real oracles, and culminated in end-to-end user flow testing across multiple epochs. No new program code was written -- this milestone was purely about integration, deployment, and validation.

## Milestone Summary

### Key Decisions

1. **Program ID Strategy**: Used keypair files for all 5 programs (AMM, Hook, Tax, Epoch, Staking), with centralized constants and automated verification to prevent cross-program mismatches.

2. **Integration Test Approach**: Built unified test-integration suite loading all 5 programs into single local validator, enabling full CPI chain validation before devnet deployment.

3. **Deployment Pipeline**: Created idempotent scripts with PDA manifest generation and post-deployment verification, enabling reliable redeploys and upgrades.

4. **VRF Configuration**: Used SLOTS_PER_EPOCH=750 on devnet (50-60 minutes per epoch) for practical testing, with retry-on-default-gateway-only pattern after gateway rotation failures.

5. **Carnage Bug Fix**: Implemented dual-pool Carnage (REG-001 burn mint + 50% slippage floor), protocol-wide ALT for v0 transactions, and combined sell proceeds to prevent stranded SOL.

6. **Independent Tax Rolls**: Changed from 1-byte VRF magnitude to 4-byte independent rolls (bytes 1-4), shifting Carnage bytes to 5/6/7 for better tax distribution randomness.

7. **Continuous Runner**: Built overnight E2E runner for sustained validation (swaps every 10 epochs, natural Carnage triggers, staking yield verification).

### Issues Resolved

1. **Cross-Program References**: Fixed placeholder IDs blocking CPI calls (epoch_program_id(), amm_program_id()).

2. **BPF Stack Overflow**: Root-caused and fixed by Boxing large InterfaceAccount structs and reducing remaining_accounts passthrough.

3. **Switchboard VRF Gateway Rotation**: Discovered per-oracle signature validation prevents gateway rotation; implemented default-gateway-only retry pattern.

4. **VRF Devnet Stability**: Tuned retry logic (20→10 attempts), added timeout recovery (300 slots), and implemented fresh randomness regeneration.

5. **Carnage WSOL Stranding**: Created CarnageSigner-owned WSOL account during protocol init, fixed sell proceeds accumulation, prevented SOL loss.

6. **Security Audit P0-P2**: Implemented tax destination PDA validation, CPI program target validation, VRF owner checks, fallback Carnage access control, LP fee cap, deposit_rewards reconciliation.

7. **Hook Accounts Per Mint**: Fixed HOOK_ACCOUNTS_PER_MINT from 3 to 4 (meta_list, wl_source, wl_dest, hook_program) to match Token-2022 createTransferCheckedWithTransferHookInstruction.

8. **ALT for Large Transactions**: Built protocol-wide Address Lookup Table (46 addresses) to support Carnage sell path (23 named + 8 remaining accounts).

### Issues Deferred

1. **Mainnet VRF Stability**: Devnet VRF works but occasional oracle timeouts require retry logic. Will monitor in mainnet beta with user-facing timeout warnings.

2. **Frontend Integration**: Protocol fully functional on devnet but no web UI yet. Deferred to next milestone (v0.8 Frontend).

3. **LP Fee Cap Enforcement**: Implemented in AMM but not stress-tested across all pool states. Will monitor in production.

4. **Carnage Slippage Floor**: 50% floor prevents worst-case losses but may cause transaction failures on extreme pool imbalances. Will tune based on mainnet data.

5. **Continuous Runner SOL Budget**: Averages 1.5-3 SOL/day on devnet with 10-epoch swap frequency. Mainnet costs unknown; will implement dynamic budget monitoring.

### Tech Debt

1. **Test Isolation**: Integration tests run in single validator; separate test files (staking.ts, token-flow.ts, security.ts) still require separate validators for singleton PDA conflicts.

2. **Error Handling**: E2E scripts use basic error logging; no structured retry/recovery for transient RPC failures beyond VRF.

3. **Documentation**: Deployment scripts have inline comments but no formal runbook for disaster recovery or state reconstruction.

4. **Monitoring**: Overnight runner logs to JSONL but no alerting for anomalies (e.g., Carnage failure, yield calculation errors).

5. **Code Duplication**: VRF flow helpers duplicated across vrf-flow.ts, overnight-runner.ts, and test files. Should consolidate into shared lib.

6. **Hard-coded Constants**: Swap amounts, epoch counts, and retry attempts scattered across scripts. Should centralize in config file.
