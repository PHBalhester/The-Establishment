Create `programs/bonding_curve/tests/dual_curve_test.rs` following the exact patterns from `refund_clock_test.rs` (LiteSVM setup, type bridge, PDA derivation, instruction builders).

**Shared helper:** Create a `setup_dual_curves()` function that:
- Deploys the bonding curve .so binary into LiteSVM
- Creates two Token-2022 mints (CRIME and FRAUD)
- Initializes BcAdminConfig PDA
- Initializes two CurveState PDAs (one per mint) with `partner_mint` pointing at each other
- Starts both curves
- Returns struct with all accounts needed for subsequent test operations
- Accepts optional config for pre-filling one or both curves to specific levels

**Test 1 - TEST-01: One curve fills, other fails, prepare_transition rejects**
- Use `setup_dual_curves()` with default config
- Purchase CRIME curve up to TARGET_TOKENS (fully filled)
- Leave FRAUD curve unfilled (or partially filled, well below target)
- Call `prepare_transition` on the filled CRIME curve
- Assert it fails because the partner (FRAUD) curve has not reached target
- Verify specific error code (the constraint that checks partner curve's `total_sold >= TARGET_TOKENS`)

**Test 2 - TEST-02: Purchase during grace period returns DeadlinePassed**
- Use `setup_dual_curves()`
- Warp clock past DEADLINE_SLOTS (use `svm.set_sysvar()` for Clock, same pattern as refund_clock_test.rs)
- But stay within DEADLINE_SLOTS + FAILURE_GRACE_SLOTS (grace period)
- Attempt a purchase on either curve
- Assert it returns `DeadlinePassed` error
- Also test at exact DEADLINE_SLOTS boundary (should fail with DeadlinePassed since purchase checks `clock.slot > start_slot + DEADLINE_SLOTS` — verify the exact comparison operator in purchase.rs)

**Test 3 - TEST-03: Multiple refund claimants complete lifecycle**
- Use `setup_dual_curves()`
- Have 3 different users purchase different amounts on one curve
- Warp clock past deadline + grace period
- Call `mark_failed` to transition curve to Failed state
- Call `consolidate_for_refund` to close token vault and calculate refund pool
- Each of the 3 users calls `claim_refund`
- Assert each gets proportional SOL back (their_tokens / total_tokens * refund_pool)
- Assert the curve's refund_claimed counter tracks correctly
- Assert a 4th claim from an already-claimed user fails

**Test 4 - TEST-04: Vault solvency breach triggers VaultInsolvency**
- Use `setup_dual_curves()` with one curve partially filled
- Artificially drain the curve's SOL vault using LiteSVM account manipulation (`svm.set_account()` to reduce lamports below what a sell would require)
- Attempt a sell operation
- Assert VaultInsolvency error is returned
- This tests the on-chain guard, not whether natural ops can cause insolvency (proptest already proves they cannot)

**Cargo.toml:** Ensure `litesvm`, `solana-account`, `solana-address`, `solana-keypair`, `solana-signer`, `solana-instruction`, `solana-message`, `solana-transaction`, `sha2` are in `[dev-dependencies]` (they should already be from refund_clock_test.rs — verify, don't duplicate).

**IMPORTANT:**
