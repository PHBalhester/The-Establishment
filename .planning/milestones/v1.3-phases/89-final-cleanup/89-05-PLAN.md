    **Subsection: Vault Solvency Invariant**

    1. **State the invariant:** At all times, `vault_sol_balance >= integral(P(x), 0, tokens_sold)` where P(x) is the linear price function.

    2. **Derive the integral:** For linear curve P(x) = P_START + (P_END - P_START) * x / TOTAL_TOKENS:
    - Integral from 0 to T: `P_START * T + (P_END - P_START) * T^2 / (2 * TOTAL_TOKENS)`
    - This represents the minimum SOL the vault must hold to cover a full walk-back (all buyers sell back to 0)

    3. **Prove buy preserves invariant:**
    - `calculate_sol_for_tokens(tokens)` uses **ceil** rounding
    - Buyer pays >= integral cost (ceil >= true value)
    - Vault receives >= integral obligation
    - Therefore: vault balance after buy >= integral(P, 0, new_tokens_sold)

    4. **Prove sell preserves invariant:**
    - `calculate_tokens_out(sol_amount)` uses **floor** rounding
    - Seller receives <= integral refund value (floor <= true value)
    - Tax escrow further reduces payout (15% retained)
    - Vault gives back < integral obligation
    - Therefore: vault balance after sell > integral(P, 0, new_tokens_sold)

    5. **Show the gap:** ceil(buy) - floor(sell) >= 1 lamport per transaction. Over N transactions, the vault accumulates at least N lamports of surplus plus 15% tax escrow. Provide a concrete numerical example with real protocol constants.

    **Subsection: Rounding Asymmetry**

    1. **Document the asymmetry:**
    - `calculate_sol_for_tokens`: uses ceil (`(numerator + denominator - 1) / denominator`)
    - `calculate_tokens_out`: uses floor (integer division truncates)
    - Direction: protocol-favored (buyer overpays by up to 1 unit, seller under-receives by up to 1 unit)

    2. **Bound the error:** For a single human-scale token (1e9 base units), the rounding error is at most 1 lamport (1e-9 SOL). Show the calculation.

    Add a new section titled "## Dual-Curve State Machine" to Bonding_Curve_Spec.md (after the math proofs section).

    **Subsection: State Definitions**

    Document each CurveState status:
    - **Funding**: Curve is accepting purchases and sell-backs
    - **Filled**: Curve has sold all tokens, awaiting transition
    - **Graduated**: Curve completed, SOL moved to pool seeding
    - **Failed**: 48hr timeout expired without both curves filling, refunds enabled

    **Subsection: Transition Table**

    Create an exhaustive markdown table:

    ```
    | Current State | Event/Action | Preconditions | Next State | Side Effects |
    |---------------|-------------|---------------|------------|-------------|
    | Funding | purchase() | tokens_remaining > 0, not expired | Funding | vault += sol, tokens_sold += amount |
    | Funding | purchase() fills curve | tokens_remaining = 0 after purchase | Filled | vault += sol, status = Filled |
    | Funding | sell() | tokens_sold > 0, not expired | Funding | vault -= sol_refund, tokens_sold -= amount |
    | Funding | 48hr timeout | clock >= deadline | Failed | (implicit, checked on next interaction) |
    | Funding | prepare_transition() | - | ERROR: CurveNotFilled | - |
    | Filled | purchase() | - | ERROR: CurveAlreadyFilled | - |
    | Filled | sell() | - | ERROR: SellsDisabledWhenFilled | - |
    | Filled | prepare_transition() | BOTH curves Filled | Graduated | SOL transferred to pool seeder |
    | Filled | prepare_transition() | other curve NOT Filled | ERROR: PartnerCurveNotFilled | - |
    | Filled | 48hr timeout (other curve still Funding) | - | Failed | (checked lazily) |
    | Graduated | any user action | - | ERROR: CurveGraduated | - |
    | Failed | claim_refund() | user has tokens to burn | Failed | burn tokens, return pro-rata SOL |
    | Failed | claim_refund() | user has no tokens | ERROR: NothingToRefund | - |
    ```

    **Subsection: Edge Cases**

    Document each edge case explicitly:

    1. **One-sided fill:** CRIME fills, FRAUD stays in Funding. After 48hr, BOTH become Failed. CRIME holders can claim refunds even though their curve was full.

    2. **Partial fill timeout:** Neither curve fills. After 48hr, both become Failed. Refund pro-rata based on vault_sol / tokens_sold ratio.

    3. **Exactly simultaneous fill:** Both curves fill in the same slot. `prepare_transition()` can be called immediately.

    4. **Race condition on transition:** Two callers submit `prepare_transition()` simultaneously. First succeeds (Filled -> Graduated), second fails (CurveAlreadyGraduated). On-chain atomicity prevents double-transition.

    5. **Refund math:** When Failed, each user burns their tokens and receives `vault_sol_balance * user_tokens / total_tokens_sold`. Tax escrow (15%) is retained in vault, so refund < original purchase. Document this as a known property (sell-tax is non-refundable).

    6. **Grace period purchase:** A purchase transaction submitted before deadline but landing after deadline. On-chain checks `clock.unix_timestamp >= deadline` — the transaction fails with DeadlinePassed. No partial execution.

    7. **Sell during Filled:** Sells are disabled once curve reaches Filled status. This prevents the curve from un-filling and re-entering Funding.

    Read the actual instruction source files (purchase.rs, sell.rs, prepare_transition.rs, claim_refund.rs) to get exact error names and conditions. Do NOT invent error codes — use the real ones.
