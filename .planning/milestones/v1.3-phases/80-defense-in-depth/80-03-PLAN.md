    **Both structs must change in lockstep. Any mismatch = wrong byte reads at runtime.**

    1. **In `epoch_state.rs`:**
       - Add `#[repr(C)]` attribute to `EpochState` struct (before or after `#[account]` — both are compatible)
       - Add `pub reserved: [u8; 64]` field BEFORE `initialized` and `bump` fields (maintaining their position as last two fields)
       - Update `DATA_LEN` constant: add 64 to current value (e.g., if currently 100, new value is 164)
       - Update `LEN` constant: `8 + DATA_LEN` (discriminator + data)
       - Add compile-time assertion:
         ```rust
         const _: () = assert!(EpochState::DATA_LEN == NEW_VALUE);
         ```
         Where NEW_VALUE is the computed data length including the 64-byte padding.

    2. **In `initialize_epoch_state.rs`:**
       - Add `reserved: [0u8; 64]` to the EpochState initialization block where fields are set
       - Verify `space` allocation uses `EpochState::LEN` (Anchor `init` space param)

    3. **In `epoch_state_reader.rs` (Tax Program mirror):**
       - Add `#[repr(C)]` attribute to the mirror struct
       - Add `pub reserved: [u8; 64]` field in the EXACT same position as EpochState (before initialized/bump)
       - Update any size constants to match
       - Add matching compile-time assertion:
         ```rust
         const _: () = assert!(MIRROR_DATA_LEN == SAME_NEW_VALUE);
         ```
       - If the mirror struct uses raw byte offsets to read fields, update ALL offsets that come after the reserved field insertion point

    4. **Add a unit test** in epoch-program tests that asserts both structs have the same DATA_LEN. If cross-crate testing isn't straightforward, add a comment documenting the expected matched value.

    **CRITICAL: Read both struct definitions carefully before making changes.** Count every field and its size. The compile-time assertion must use the ACTUAL sum of field sizes, not a guessed value.
