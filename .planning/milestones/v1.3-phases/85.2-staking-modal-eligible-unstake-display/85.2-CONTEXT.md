# Phase 85.2: Staking Modal — Eligible Unstake Display - Context

**Gathered:** 2026-03-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Add individual unstake eligibility indicator and protocol-wide locked/unlocked PROFIT stats to the staking modal. No on-chain changes — all computation is off-chain. Personal cooldown badge replaces the existing UnstakeTab cooldown banner (single source of truth in StakingStats).

</domain>

<decisions>
## Implementation Decisions

### Individual Eligibility Badge
- Green status badge ("Eligible to Unstake") displayed as a standalone row ABOVE the StakingStats 2-column grid
- Visible on ALL tabs (Stake/Unstake/Claim) — not scoped to UnstakeTab
- During cooldown: amber badge with countdown ("Cooldown: Xh Ym")
- Only renders when user has an active stake (userStakeData exists). No badge = no stake (implicit).
- Replaces the existing cooldown banner in UnstakeTab entirely — remove that banner from UnstakeTab.tsx
- Unstake action button remains disabled during cooldown (current behavior preserved)

### Global Eligible Balance
- Two new stat items added to the StakingStats 2-column grid: "Unlocked PROFIT" and "Locked PROFIT"
- Grid grows from 6 items (3 rows) to 8 items (4 rows)
- "Unlocked PROFIT": sum of staked_balance across all UserStake accounts where last_claim_ts == 0 OR (now - last_claim_ts) >= COOLDOWN_SECONDS
- "Locked PROFIT": sum of staked_balance across all UserStake accounts where (now - last_claim_ts) < COOLDOWN_SECONDS AND last_claim_ts > 0
- Off-chain computation only — no on-chain state changes needed (DBS interview decision PC-3, decision #5)
- Data source: getProgramAccounts on the staking program, filtering for UserStake accounts

### Claude's Discretion
- Performance strategy: poll-on-modal-open vs server-side API route with TTL cache — pick based on expected staker count and existing patterns
- Badge visual design (pill shape, colors, icon usage)
- Grid item ordering (where Locked/Unlocked sit relative to existing 6 items)
- Loading/skeleton state for global stats (may load slower than per-user data)

</decisions>

<specifics>
## Specific Ideas

- DBS interview (PC-3) explicitly decided off-chain computation: "Frontend computes total PROFIT eligible for unstake by reading all UserStake accounts and checking last_claim_ts against current time. On-chain tracking has unsolvable staleness (no Solana cron)."
- UserStake accounts are 105 bytes, PDA seeds = ["user_stake", user_pubkey], have `last_claim_ts: i64` (0 = never claimed, no cooldown applies)
- COOLDOWN_SECONDS is imported from `@dr-fraudsworth/shared`
- StakingStats is a props-only component — new data flows through useStaking or a new hook, consumed by StakingForm

</specifics>
