**1. Create `rust-toolchain.toml` at project root.**

The existing `Anchor.toml` has an empty `[toolchain]` section. Create a proper `rust-toolchain.toml` file (separate from Anchor.toml) to pin the Rust version:

```toml
[toolchain]
channel = "1.93.0"
components = ["rustfmt", "clippy"]
```

Why: Ensures CI uses the exact same Rust version as local dev (1.93.0). Without this, `ubuntu-latest` runners use whatever Rust ships with the image, causing mysterious build failures.

**2. Create `.github/workflows/ci.yml` with 3 parallel jobs.**

The workflow has these characteristics:
- **Trigger**: `push` to `main` only (no PRs, no manual dispatch)
- **3 parallel jobs**: rust-tests, litesvm-tests, ts-tests
- **Aggressive caching**: Cargo registry, target dir, Solana CLI, Anchor CLI, node_modules

**Shared setup across all 3 jobs** (each job needs its own setup since GH Actions jobs run on separate VMs):
- Runner: `ubuntu-latest`
- Install Rust via `dtolnay/rust-toolchain@stable` with `toolchain: 1.93.0` (reads rust-toolchain.toml)
- Install Solana CLI v3.0.13: Download from `https://github.com/anza-xyz/agave/releases/download/v3.0.13/solana-release-x86_64-unknown-linux-gnu.tar.bz2`, extract to `~/.local/share/solana/install/active_release`, add to PATH
- Cache Solana CLI with key `solana-3.0.13`
- Install Anchor CLI v0.32.1 via `cargo install --git https://github.com/coral-xyz/anchor avm --force` then `avm install 0.32.1 && avm use 0.32.1`. Cache `~/.cargo/bin/anchor` and `~/.avm/` with key `anchor-0.32.1`
- Actually, a simpler approach for Anchor: use `cargo install --git https://github.com/coral-xyz/anchor --tag v0.32.1 anchor-cli` to install directly without AVM
- Install Node.js 22 via `actions/setup-node@v4` with `node-version: 22` and `cache: 'npm'`
- Run `npm ci` for node_modules
- Cache cargo registry and target: key on `Cargo.lock` hash

**IMPORTANT for Anchor/Solana on Linux CI:**
- Anchor build requires the Solana BPF SDK. The Solana CLI tarball includes `sdk/sbf/` under the install directory.
- Set `PATH` to include both `~/.local/share/solana/install/active_release/bin` and `~/.cargo/bin`

**Job 1: `rust-tests` (cargo test unit + proptest)**
```
- anchor build
- anchor build -p epoch_program -- --features devnet
- anchor build -p tax_program -- --features devnet
- anchor build -p conversion_vault -- --features devnet
- anchor build -p bonding_curve -- --features devnet
- cargo test --workspace --lib --features devnet
```
Why separate `--lib`: This runs only unit tests (lib.rs, inline #[cfg(test)] modules), NOT integration tests in `tests/` directories. The LiteSVM integration tests get their own job.
Why `--features devnet`: bonding_curve has `compile_error!` without devnet feature, per Phase 86 decisions.
Proptest uses default 256 iterations (controlled by `PROPTEST_CASES` env var or default config).
Timeout: 30 minutes.

**Job 2: `litesvm-tests` (Rust integration tests using LiteSVM)**
```
- anchor build
- anchor build -p epoch_program -- --features devnet
- anchor build -p tax_program -- --features devnet
- anchor build -p conversion_vault -- --features devnet
- anchor build -p bonding_curve -- --features devnet
- cargo test --workspace --test '*' --features devnet
```
Why `--test '*'`: This runs only integration test files (programs/*/tests/*.rs), not lib tests. These are the LiteSVM tests that load .so binaries.
Why separate job: LiteSVM tests are heavier (load program binaries, simulate transactions) and the StakePool PDA singleton means some test files need sequential execution.
Timeout: 45 minutes.

**IMPORTANT consideration:** Both rust-tests and litesvm-tests need the same `anchor build` step. To avoid building twice, consider combining them into a single job:
```
- anchor build (all programs)
- anchor build -p {4 programs} -- --features devnet
- cargo test --workspace --features devnet
```
This runs ALL Rust tests (unit + integration) in one job. This is actually simpler and the CONTEXT.md says "3 parallel jobs" but the real constraint from the user discussion was "3 test suites". If combining Rust tests into one job is cleaner (avoids duplicate 15-min build), do that instead. Use 2 jobs: (1) rust-all-tests, (2) ts-tests. The Rust job handles both unit and LiteSVM since they share the build artifact.

**Revised approach - 2 jobs for efficiency:**

**Job 1: `rust-tests`** (all Rust tests: unit + proptest + LiteSVM integration)
- Build: `anchor build` then 4x `anchor build -p {program} -- --features devnet`
- Test: `cargo test --workspace --features devnet` (runs everything)
- Timeout: 45 minutes

**Job 2: `ts-tests`** (TypeScript integration tests via anchor test)
- Build: same anchor build steps (needed for .so binaries and IDL)
- Test: `anchor test --skip-build` (skips rebuild, just starts validator + runs ts-mocha)
- But check: `anchor test` uses the `[scripts] test` from Anchor.toml which runs `npx ts-mocha ... tests/staking.ts tests/cross-program-integration.ts`
- For full TS coverage, the CI should run ALL test scripts:
  - `anchor test --skip-build` (default test script)
  - `anchor test --skip-build --run 'test-integration'` (integration tests)
  - `anchor test --skip-build --run 'test-account-validation'` (security tests)
  - `anchor test --skip-build --run 'test-security'` (security tests)
- Actually, `anchor test --run` isn't a flag. The approach is: `anchor test --skip-build` runs the default `[scripts] test` entry. To run other scripts, you'd need to run them as separate commands while the validator is up.
- **Simplest approach**: Override the test command in CI to run ALL .ts test files:
  ```
  npx ts-mocha -p ./tsconfig.json -t 1000000 'tests/**/*.ts'
  ```
  But this requires a running validator. Use `anchor test --skip-build` but set the Anchor.toml `[scripts] test` to the broader glob. Actually, DON'T modify Anchor.toml - that changes local behavior.
- **Better approach**: Use `solana-test-validator` directly in the background, then run ts-mocha manually:
  ```
  solana-test-validator --bpf-program <id> target/deploy/<prog>.so ... &
  sleep 5
  npx ts-mocha -p ./tsconfig.json -t 1000000 'tests/**/*.ts'
  ```
  But this requires listing every program and account fixture.
- **Simplest correct approach**: Just use `anchor test --skip-build`. It spins up a local validator with all programs from Anchor.toml, runs the default test script. The default test script covers `tests/staking.ts` and `tests/cross-program-integration.ts`. The integration tests under `tests/integration/` can be run separately. For CI simplicity, run `anchor test --skip-build` which handles validator lifecycle.
- To run all TS test files, create a small CI-specific override: set env var `ANCHOR_TEST_COMMAND` or use `anchor test --skip-build -- --test 'npx ts-mocha ...'`. Actually `anchor test` accepts the test command after `--`:
  ```
  anchor test --skip-build -- 'npx ts-mocha -p ./tsconfig.json -t 1000000 tests/**/*.ts'
  ```
  Check if this syntax works. If not, the cleanest approach is to use `anchor test --skip-build` for the default test suite only. The integration tests can be a follow-up.

**Final decision for TS tests**: Use `anchor test --skip-build` which handles validator lifecycle automatically. This runs the Anchor.toml `[scripts] test` entry (staking + cross-program tests). It's the established pattern. If additional TS test suites need CI coverage, that's a future enhancement. The CONTEXT.md says "ts-mocha TypeScript tests" without specifying every file.

**Workflow structure outline:**
```yaml
name: CI
on:
  push:
    branches: [main]

env:
  SOLANA_VERSION: "3.0.13"
  ANCHOR_VERSION: "0.32.1"
  PROPTEST_CASES: "256"

jobs:
  rust-tests:
    runs-on: ubuntu-latest
    timeout-minutes: 45
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@1.93.0
      - uses: actions/cache@v4 (cargo registry + target)
      - Install Solana CLI (with cache)
      - Install Anchor CLI (with cache)
      - uses: actions/setup-node@v4 (node 22, npm cache)
      - npm ci
      - anchor build
      - anchor build -p epoch_program -- --features devnet
      - anchor build -p tax_program -- --features devnet
      - anchor build -p conversion_vault -- --features devnet
      - anchor build -p bonding_curve -- --features devnet
      - cargo test --workspace --features devnet

  ts-tests:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - (same setup as rust-tests)
      - anchor build + devnet rebuilds
      - anchor test --skip-build
```

Write the actual YAML with all steps fully specified. Use `if: steps.cache-*.outputs.cache-hit != 'true'` guards on install steps so cached binaries skip re-installation.

For installing Anchor CLI without AVM:

This takes ~5 minutes uncached. With cargo bin cache, it's instant.

Actually, reconsider the job structure. Both jobs need the full `anchor build` which takes 10-15 minutes. That's duplicated work across 2 parallel jobs. Alternative: use a build job that uploads artifacts, then 2 test jobs download them. But GH Actions artifact upload/download for `target/` (which is huge) may be slower than just building twice in parallel.

**Keep 2 parallel jobs, each builds independently.** The parallelism gains (both run simultaneously) outweigh the duplicate build cost. Total wall clock: max(45min, 30min) = 45min instead of sequential 75min.
