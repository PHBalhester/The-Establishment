1. **CarnageAccounts struct** with raw AccountInfo parameters for the ~14 shared accounts. Use mixed typing as decided in CONTEXT.md:

3. **All 7 helper functions** extracted from the handlers. These currently exist as private fns duplicated in both files:
   - `burn_held_tokens` -- already context-free, move as-is
   - `wrap_sol_to_wsol` -- change ctx param from `&Context<..., ExecuteCarnage/Atomic>` to take individual AccountInfos via CarnageAccounts
   - `execute_sell_swap` -- same refactor
   - `execute_buy_swap` -- same refactor
   - `execute_swap_exempt_cpi` -- same refactor (this is the big one, ~100 lines)
   - `read_pool_reserves` -- already context-free, move as-is
   - `approve_delegate` -- same refactor

4. **execute_carnage_core()** function signature:
   ```rust
   pub fn execute_carnage_core<'info>(
       accounts: &CarnageAccounts<'_, 'info>,
       epoch_state: &mut Account<'info, EpochState>,
       carnage_state: &mut Account<'info, CarnageFundState>,
       remaining_accounts: &'info [AccountInfo<'info>],
       carnage_signer_bump: u8,
       sol_vault_bump: u8,
       slippage_bps: u64,
       atomic: bool,
   ) -> Result<()>
   ```
   This function implements the full dispose->buy->update flow (Steps 1, 1.5, 2, 3 from the existing handlers). The 3 parameterized differences are:
   - `slippage_bps`: CARNAGE_SLIPPAGE_BPS_FALLBACK (7500) vs CARNAGE_SLIPPAGE_BPS_ATOMIC (8500)
   - `atomic`: true/false for the CarnageExecuted event
   - Entry guards are NOT in core -- each handler does its own guard before calling core

5. **HOOK_ACCOUNTS_PER_MINT** const: define in this module (value = 4)

6. **Promote SWAP_EXEMPT_DISCRIMINATOR** to `constants.rs`:
   - Add `pub const SWAP_EXEMPT_DISCRIMINATOR: [u8; 8] = [0xf4, 0x5f, 0x5a, 0x24, 0x99, 0xa0, 0x37, 0x0c];` to constants.rs
   - Import from constants in carnage_execution.rs
   - This sets up Phase 83's VRF-10 validation test cleanly

7. **Update helpers/mod.rs**: Add `pub mod carnage_execution;` and `pub use carnage_execution::*;`

IMPORTANT: Preserve all existing comments and doc strings. The comments explain WHY (CPI depth chain, stack budget, etc.) and are critical for future maintenance.

DO NOT change the account validation structs (`ExecuteCarnage` and `ExecuteCarnageAtomic`) -- those stay in their respective instruction files. The Anchor `#[derive(Accounts)]` structs must remain in-place for Anchor's code generation.

DO NOT change any behavioral logic. This is a pure extraction refactor. The slippage math, CPI construction, burn logic, wrap logic must be byte-for-byte identical in behavior.
