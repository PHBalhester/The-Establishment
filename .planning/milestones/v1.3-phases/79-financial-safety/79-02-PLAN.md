**purchase.rs (FIN-04) -- Add partial fill overcharge assertion:**

After step 7 (line ~162, after `actual_sol` is computed), add:
```rust
// Step 7b: Defense-in-depth -- actual_sol must never exceed user's input
require!(actual_sol <= sol_amount, CurveError::PartialFillOvercharge);
```

This is a simple `require!` per CONTEXT.md -- bugs should surface as errors, not be hidden.

**sell.rs (FIN-04) -- Audit and add defensive assertion:**

After step 7 (sol_gross computed via reverse integral), add a defensive check that sol_gross does not exceed available vault balance (minus rent-exempt):
```rust
// Defense-in-depth: gross payout must not exceed available vault balance
let vault_lamports = ctx.accounts.sol_vault.lamports();
let rent = Rent::get()?;
let rent_exempt = rent.minimum_balance(0);
let available = vault_lamports.saturating_sub(rent_exempt);
require!(sol_gross <= available, CurveError::VaultInsolvency);
```

This catches any math bug where the reverse integral computes more SOL than the vault actually holds. Use the existing `VaultInsolvency` error since it's the same class of invariant violation.

Note: The existing solvency assertion at step 16 is a post-state check. This new check is a pre-transfer guard -- belt AND suspenders.

**claim_refund.rs (FIN-05) -- Add partner curve identity validation:**

After step 1 (refund eligibility check), add:
```rust
// Step 1b: Validate partner curve identity
// partner_curve_state must be the actual counterpart, not an arbitrary curve
require!(
    ctx.accounts.partner_curve_state.token_mint == curve.partner_mint,
    CurveError::InvalidPartnerCurve
);
```

**consolidate_for_refund.rs (FIN-05) -- Add partner curve identity validation:**

After step 1 (refund eligibility check), add the same validation:
```rust
// Step 1b: Validate partner curve identity
require!(
    ctx.accounts.partner_curve_state.token_mint == curve.partner_mint,
    CurveError::InvalidPartnerCurve
);
```

Both claim_refund and consolidate_for_refund already have `partner_curve_state` as an account. The `partner_curve_state.key() != curve_state.key()` constraint prevents self-referencing but does NOT prevent passing an arbitrary third curve. The new `partner_mint` check closes this gap.
