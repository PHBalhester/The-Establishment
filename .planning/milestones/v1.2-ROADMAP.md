# Roadmap: v1.2 Bonding Curves & Launch Page

## Overview

Build the 7th on-chain program -- dual linear bonding curves for CRIME and FRAUD token distribution with buy/sell mechanics, 15% sell tax escrow, coupled graduation, and proportional refunds on failure. Then build the frontend launch page. The spec update comes first, security testing is integrated into each program phase (not deferred), and the deploy pipeline is wired before the frontend so it can target a live devnet program.

### Phase 72: Sell-Back + Tax Escrow
**Goal**: Users can sell tokens back to the curve and receive SOL minus a 15% tax, with the tax routed to a separate escrow PDA. Selling is disabled once a curve reaches Filled status. The SOL vault solvency invariant holds across all buy/sell sequences.
**Depends on**: Phase 71
**Requirements**: CURVE-03, CURVE-04, SAFE-02
**Success Criteria** (what must be TRUE):
  1. A user can sell tokens back to either curve, receiving SOL computed via the reverse integral minus 15% tax, with minimum_sol_out slippage protection
  2. The 15% sell tax is deducted and transferred to a separate tax escrow PDA (not mixed with curve SOL reserves)
  3. Sells are rejected with a clear error once a curve reaches Filled status (all 460M tokens sold)
  4. SOL vault solvency invariant verified at every state transition: vault_balance >= expected_balance_from_integral - rent_exempt_minimum
  5. Property tests (10K+ iterations) prove: buy/sell round-trip returns <= spent SOL after tax, no vault insolvency across random buy/sell sequences

### Phase 73: Graduation + Refund
**Goal**: The bonding curve state machine handles success (both curves fill within 48 hours) and failure (deadline expires without both graduating) with permissionless triggers, correct asset staging, and proportional refunds.
**Depends on**: Phase 72
**Requirements**: CURVE-05, CURVE-06, CURVE-07, CURVE-08
**Success Criteria** (what must be TRUE):
  1. When one curve reaches Filled, it locks and waits for its partner curve; both must fill within 48 hours of the first curve starting
  2. Anyone can call the graduation trigger (permissionless) when both curves are Filled, and anyone can call the failure trigger after the 48-hour deadline if both have not graduated
  3. On success: prepare_transition locks curves, SOL and token assets are staged for pool seeding (available for withdrawal by graduation orchestrator), tax escrow is marked for carnage fund routing
  4. On failure: each user can claim a proportional refund of (SOL vault + tax escrow) based on their token holdings relative to total outstanding tokens; users keep their tokens
  5. State transitions are irreversible: Filled cannot revert to Active, Graduated/Failed are terminal states
