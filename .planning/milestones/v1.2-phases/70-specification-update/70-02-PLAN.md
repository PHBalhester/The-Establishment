**ADD new Section 8.6: sell instruction -- GAP-03:**
This is a NEW instruction. Document it in full detail matching the style of existing instructions:

Instruction name: `sell`

Accounts:
- `user` (signer, mut) -- the seller
- `curve_state` (mut) -- PDA for the curve being sold on
- `user_token_account` (mut) -- user's ATA for this curve's token
- `token_vault` (mut) -- program's token vault PDA (receives tokens back)
- `sol_vault` (mut) -- program's SOL vault PDA (sends SOL to user)
- `tax_escrow` (mut) -- tax escrow PDA (receives 15% tax)
- `token_mint` (mut) -- the token mint (if burning) OR just for validation
- `token_program` -- Token-2022 program
- `system_program` -- for SOL transfers

Args:
- `tokens_to_sell: u64` -- number of tokens to sell back
- `minimum_sol_out: u64` -- minimum SOL (net, after tax) user will accept (slippage protection)

Validation:
1. `curve_state.status == CurveStatus::Active` (NOT Filled, NOT Failed, NOT Graduated) -- error: `CurveNotActiveForSell`
2. `user_token_account.amount >= tokens_to_sell` -- error: `InsufficientTokenBalance`
3. `tokens_to_sell > 0` -- error: `ZeroAmount`
4. `clock.slot <= curve_state.deadline_slot` -- error: `DeadlinePassed` (cannot sell after deadline even if still Active)

Logic (numbered steps -- matches Section 4.5 tax ordering):
1. Read `x1 = curve_state.tokens_sold` (current position on curve)
2. Compute `x2 = x1 - tokens_to_sell` (new position after sell)
3. Compute `SOL_gross = integral from x2 to x1` using the reverse integral formula from Section 4.5
4. Compute `tax = SOL_gross * 15 / 100` (integer division, rounds down -- user-favorable)
5. Compute `SOL_net = SOL_gross - tax`
6. `require!(SOL_net >= minimum_sol_out, CurveError::SlippageExceeded)` -- slippage check
7. Transfer `tokens_to_sell` tokens from `user_token_account` to `token_vault` (tokens return to vault, NOT burned)
8. Transfer `SOL_net` lamports from `sol_vault` to `user` via system program
9. Transfer `tax` lamports from `sol_vault` to `tax_escrow` via system program
10. Update `curve_state`: `tokens_sold -= tokens_to_sell`, `sol_raised -= SOL_gross`, `tokens_returned += tokens_to_sell`, `sol_returned += SOL_gross`, `tax_collected += tax`

Events: Emit `TokensSold` event (see Section 10)

Notes:
- Tokens go BACK to the vault (not burned) -- they can be re-purchased by another buyer
- `tokens_sold` decreasing means the price drops for the next buyer (curve walks backward)
- Tax rounding down (integer division) means the protocol may collect slightly less than exactly 15% -- acceptable, benefits user
- Per-wallet cap: selling frees cap space. After selling, user's ATA balance decreases, allowing future buys up to the 20M cap

**Section 8.7 (claim_refund) -- GAP-04, GAP-10: REPLACE ENTIRELY with burn-and-claim.**

Instruction name: `claim_refund` (same name, completely different implementation)

Accounts:
- `user` (signer, mut) -- the refund claimer
- `curve_state` (mut) -- PDA (status must be Failed, OR Filled with partner Failed)
- `user_token_account` (mut) -- user's ATA for this curve's token
- `sol_vault` (mut) -- SOL vault PDA (contains sol_raised + consolidated escrow)
- `token_mint` (mut) -- for burning tokens
- `token_program` -- Token-2022 program
- `system_program` -- for SOL transfer

Args: None (burns entire ATA balance)

Validation:
1. Curve is refund-eligible: `curve_state.status == Failed` OR (`curve_state.status == Filled` AND partner curve is Failed) -- error: `CurveNotRefundable`
2. `user_token_account.amount > 0` -- error: `NothingToBurn`
3. Escrow has been consolidated: either verify via a flag, or check that tax_escrow PDA has been drained -- error: `EscrowNotConsolidated`

Logic:
1. Read `user_balance = user_token_account.amount`
2. Read `total_outstanding = curve_state.tokens_sold` (current outstanding supply)
3. Read `refund_pool = sol_vault.lamports - rent_exempt_minimum` (available SOL for refunds)
4. Compute `refund_amount = (user_balance as u128 * refund_pool as u128 / total_outstanding as u128) as u64`
5. Burn `user_balance` tokens from `user_token_account` (permanent destruction via token_program burn)
6. Transfer `refund_amount` lamports from `sol_vault` to `user`
7. Update `curve_state.tokens_sold -= user_balance` (denominator shrinks for next claimer)

Events: Emit `RefundClaimed` event with tokens_burned, refund_amount, remaining_tokens_sold

Include WORKED EXAMPLE (from RESEARCH.md Pitfall 2):
- 3 users: Alice (100M tokens), Bob (200M tokens), Carol (160M tokens), total = 460M
- Sol vault after consolidation: 800 SOL (example numbers)
- Alice claims first: 100M/460M * 800 = 173.9 SOL. Remaining: 360M tokens, 626.1 SOL
- Bob claims second: 200M/360M * 626.1 = 347.8 SOL. Remaining: 160M tokens, 278.3 SOL
- Carol claims last: 160M/160M * 278.3 = 278.3 SOL. Remaining: 0 tokens, 0 SOL
- Total refunded: 173.9 + 347.8 + 278.3 = 800 SOL (exactly the pool)
- This proves the formula is order-independent and fully solvent
