  <name>Curve Math Module (TDD)</name>
  <files>programs/bonding_curve/src/math.rs</files>
  <behavior>
    All three functions implement the linear curve P(x) = P_START + (P_END - P_START) * x / TOTAL_FOR_SALE using u128 integer arithmetic with PRECISION=1e12 scaling.

    **Function 1: calculate_tokens_out(sol_lamports: u64, current_sold: u64) -> Result<u64>**
    - Implements the quadratic formula from spec Section 4.3-4.4
    - For linear curve P(x) = a + bx, given SOL input S and current position x1:
      dx = (-(a + b*x1) + sqrt((a + b*x1)^2 + 2*b*S)) / b
    - Uses `u128::isqrt()` (Rust stdlib, available on SBF since platform-tools v1.51)
    - Returns floor(dx) as u64 (protocol-favored: user gets slightly fewer tokens)
    - Critical identity: calculate_tokens_out(1_000_000_000_000, 0) == 460_000_000_000_000 (within 1 token)

    **Function 2: calculate_sol_for_tokens(current_sold: u64, tokens: u64) -> Result<u64>**
    - Computes SOL cost for buying `tokens` starting at `current_sold` position
    - Uses the linear integral: SOL = a*N + b*(x1 + x1 + N)*N/2
    - Returns ceil(sol) as u64 (protocol-favored: user pays slightly more SOL)
    - Critical identity: calculate_sol_for_tokens(0, 460_000_000_000_000) == 1_000_000_000_000 (within 1 lamport)
    - This is also the reverse integral needed for sells (Phase 72)

    **Function 3: get_current_price(tokens_sold: u64) -> u64**
    - P(x) = P_START + (P_END - P_START) * x / TOTAL_FOR_SALE
    - Returns price in the same scaling as P_START/P_END

    Test cases (RED phase):
    - Integral identity: full curve cost == TARGET_SOL
    - Reverse identity: full curve tokens == TARGET_TOKENS
    - Boundary: price at 0 == P_START, price at TARGET_TOKENS == P_END
    - Dust: 1 lamport -> some tokens (or graceful error)
    - Max: TARGET_SOL at position 0 -> TARGET_TOKENS
    - Partial: 500 SOL at position 0 -> roughly half the tokens
    - Near-full: buying when only 1 token remains
    - Proptest 500K: no overflow, tokens_out <= remaining, vault solvency
    - Proptest: monotonic pricing (more sold = higher price)
    - Proptest: round-trip consistency (tokens_out -> sol_for_tokens >= original SOL)
  </behavior>
  <implementation>
    Follow the code pattern from 71-RESEARCH.md "Core Curve Math" section. Key implementation details:

    1. All three functions are pure (no Anchor accounts, no Solana runtime) -- only `use crate::constants::*` and `use crate::error::CurveError`
    2. Use checked arithmetic throughout (`checked_mul`, `checked_add`, `checked_sub`, `checked_div`) returning `CurveError::Overflow` on failure
    3. PRECISION = 1e12 scaling applied consistently: multiply inputs by PRECISION early, divide results by PRECISION late
    4. `u128::isqrt()` for the discriminant square root -- no hand-rolled Newton's method
    5. For `calculate_sol_for_tokens`, the formula is: SOL = tokens * (2*a + b*(2*x1 + tokens)) / 2, all in PRECISION-scaled u128
    6. Rounding: `calculate_tokens_out` floors (truncation is natural in integer division), `calculate_sol_for_tokens` ceils (add denominator-1 before dividing)
    7. `#[cfg(test)]` block with proptest at bottom of file

    WARNING: The spec pseudocode (Section 4.4) has conceptual formulas. The actual PRECISION scaling must be carefully ordered to avoid overflow while maintaining precision. Document the maximum intermediate value at each step (research notes: max discriminant ~1e31, well within u128 range of ~3.4e38).

    Proptest strategies: Use percentage-based derivation (not prop_assume!) for current_sold to avoid >50% rejection. Pattern from staking/math.rs.
  </implementation>
