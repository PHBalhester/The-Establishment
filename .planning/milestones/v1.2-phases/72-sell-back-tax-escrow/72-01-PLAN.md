Mirror the research's recommended struct from 72-RESEARCH.md "Sell Accounts Struct" section. Key differences from Purchase:

2. NO `associated_token_program` -- not needed since we don't create the ATA.

**Handler signature** (matches purchase pattern for remaining_accounts):

```rust
pub fn handler<'info>(
    ctx: Context<'_, '_, 'info, 'info, Sell<'info>>,
    tokens_to_sell: u64,
    minimum_sol_out: u64,

```

**Handler logic -- follow spec Section 8.6, implementing all steps:**

**VALIDATION (Steps 1-4):**
1. **Status check:** `require!(curve.status == CurveStatus::Active, CurveError::CurveNotActiveForSell)` -- NOTE: This is ALSO enforced as an Anchor constraint on the curve_state account, but we double-check in the handler for clarity.
2. **Balance check:** `require!(ctx.accounts.user_token_account.amount >= tokens_to_sell, CurveError::InsufficientTokenBalance)`
3. **Non-zero check:** `require!(tokens_to_sell > 0, CurveError::ZeroAmount)`
4. **Deadline check:** `require!(Clock::get()?.slot <= curve.deadline_slot, CurveError::DeadlinePassed)`

**COMPUTATION (Steps 5-10):**
5. **Current position:** `let x1 = curve.tokens_sold;`
6. **New position after sell:** `let x2 = x1.checked_sub(tokens_to_sell).ok_or(CurveError::Overflow)?;`
7. **Reverse integral - gross SOL:** `let sol_gross = calculate_sol_for_tokens(x2, tokens_to_sell)?;` -- NOTE: x2 is the START position, not x1. The function computes integral(x2, x2+tokens_to_sell) which equals the area the user is selling back.
8. **Tax (ceil rounding, protocol-favored):**
   ```rust
   let tax = sol_gross
       .checked_mul(SELL_TAX_BPS)
       .ok_or(CurveError::Overflow)?
       .checked_add(BPS_DENOMINATOR - 1)
       .ok_or(CurveError::Overflow)?
       / BPS_DENOMINATOR;
   ```
   IMPORTANT: CONTEXT.md explicitly decides ceil rounding, overriding the spec's floor/truncation pseudocode. Use BPS ceil.
9. **Net payout:** `let sol_net = sol_gross.checked_sub(tax).ok_or(CurveError::Overflow)?;`
10. **Slippage check:** `require!(sol_net >= minimum_sol_out, CurveError::SlippageExceeded)`

**TRANSFERS (Steps 11-13):**
11. **Token transfer: user -> vault (Token-2022 with Transfer Hook):**
    Use manual `invoke` (NOT `invoke_signed` -- user is a real signer, not a PDA) following the purchase.rs pattern for consistency. Build the transfer_checked instruction, append remaining_accounts for Transfer Hook, then invoke.
    ```rust
    let mut ix = spl_token_2022::instruction::transfer_checked(
        ctx.accounts.token_program.key,
        &ctx.accounts.user_token_account.key(),    // from: user's ATA
        &ctx.accounts.token_mint.key(),             // mint
        &ctx.accounts.token_vault.key(),            // to: curve's vault
        &ctx.accounts.user.key(),                   // authority: user (real signer)
        &[],                                        // no multisig
        tokens_to_sell,
        TOKEN_DECIMALS,
    )?;
    // Append Transfer Hook accounts from remaining_accounts
    for account_info in ctx.remaining_accounts {
        ix.accounts.push(AccountMeta {
            pubkey: *account_info.key,
            is_signer: account_info.is_signer,
            is_writable: account_info.is_writable,
        });
    }
    let mut account_infos = vec![
        ctx.accounts.user_token_account.to_account_info(),
        ctx.accounts.token_mint.to_account_info(),
        ctx.accounts.token_vault.to_account_info(),
        ctx.accounts.user.to_account_info(),
    ];
    for account_info in ctx.remaining_accounts {
        account_infos.push(account_info.clone());
    }
    anchor_lang::solana_program::program::invoke(&ix, &account_infos)?;
    ```
    Note the use of `invoke` (not `invoke_signed`) since the user is a real signer.

12. **SOL transfer: sol_vault -> user (direct lamport manipulation):**
    The sol_vault is owned by the bonding curve program, NOT the system program. You MUST use direct lamport manipulation. `system_program::transfer` CPI DOES NOT WORK for program-owned accounts.
    ```rust
    **ctx.accounts.sol_vault.try_borrow_mut_lamports()? -= sol_net;
    **ctx.accounts.user.try_borrow_mut_lamports()? += sol_net;
    ```

13. **Tax transfer: sol_vault -> tax_escrow (direct lamport manipulation):**
    ```rust
    **ctx.accounts.sol_vault.try_borrow_mut_lamports()? -= tax;
    **ctx.accounts.tax_escrow.try_borrow_mut_lamports()? += tax;
    ```

**STATE UPDATE (Step 14-15):**
14. **Update tokens_sold:** `curve.tokens_sold = x2;` (x2 was already computed as x1 - tokens_to_sell)
15. **Update cumulative counters:**
    ```rust
    curve.sol_returned = curve.sol_returned.checked_add(sol_gross).ok_or(CurveError::Overflow)?;
    curve.tokens_returned = curve.tokens_returned.checked_add(tokens_to_sell).ok_or(CurveError::Overflow)?;
    curve.tax_collected = curve.tax_collected.checked_add(tax).ok_or(CurveError::Overflow)?;
    ```
    NOTE: `sol_returned` tracks gross SOL (before tax) per Phase 70-02 decision. This preserves the identity: `vault_balance = sol_raised - sol_returned`.

**SOLVENCY ASSERTION (Step 16):**
16. **Post-state defense-in-depth check:**
    ```rust
    let rent = Rent::get()?;
    let rent_exempt_min = rent.minimum_balance(0);
    let expected_from_integral = calculate_sol_for_tokens(0, curve.tokens_sold)?;
    let vault_balance = ctx.accounts.sol_vault.lamports();
    require!(
        vault_balance + rent_exempt_min as u64 >= expected_from_integral,
        CurveError::VaultInsolvency
    );
    ```
    This follows spec Invariant 9. The vault starts with rent-exempt lamports (from initialization), so we ADD rent_exempt to the vault side rather than subtracting from integral side (avoids underflow for edge cases where integral is very small).

    Wait -- re-read the spec formula: `vault_balance >= integral(0, tokens_sold) - rent_exempt_minimum`. This means `vault_balance + rent_exempt >= integral`. Both formulations are equivalent. Use whichever avoids potential underflow. Since `expected_from_integral` could be 0 when tokens_sold is 0 (and rent_exempt_min is ~890K), the spec formula would have negative right side, which is fine for unsigned (saturating). Use the spec's formula for clarity:

    ```rust
    require!(
        vault_balance >= expected_from_integral.saturating_sub(rent_exempt_min as u64),

    ```
