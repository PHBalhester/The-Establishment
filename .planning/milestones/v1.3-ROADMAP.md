### v1.3 Protocol Hardening & Polish (In Progress)

**Milestone Goal:** Fix all audit findings, harden every program, polish frontend, complete documentation — leave the protocol fully finished and ready for final devnet lifecycle test (v1.4) and mainnet deployment.

#### Phase 78: Authority Hardening
**Goal**: Every program initialization is gated by ProgramData upgrade authority, preventing front-running and unauthorized setup
**Depends on**: Nothing (first phase of v1.3)
**Requirements**: AUTH-01, AUTH-02, AUTH-03, AUTH-04, AUTH-05, AUTH-06, AUTH-07
**Success Criteria** (what must be TRUE):
  1. Bonding Curve program has BcAdminConfig PDA and all 6 admin instructions require `has_one = authority` — non-authority signers are rejected
  2. All 5 non-BC programs (Hook, Staking, Epoch, Epoch Carnage, Vault) reject initialization unless caller matches ProgramData upgrade authority
  3. Authority map and gradual burn timeline are documented in PROJECT.md decisions table
  4. Existing devnet deployment still functions after authority hardening (no regression)
**Plans:** 3/3 plans complete
Plans:
- [x] 78-01-PLAN.md — BcAdminConfig PDA and admin instruction gating (AUTH-01)
- [x] 78-02-PLAN.md — ProgramData init gating for 5 programs (AUTH-02 to AUTH-06)
- [x] 78-03-PLAN.md — Authority map documentation and build verification (AUTH-07)

#### Phase 79: Financial Safety
**Goal**: Every SOL transfer instruction protects against rent-exempt drain, zero-minimum CPI, and partial fill edge cases
**Depends on**: Phase 78
**Requirements**: FIN-01, FIN-02, FIN-03, FIN-04, FIN-05
**Success Criteria** (what must be TRUE):
  1. Staking claim instruction cannot drain escrow PDA below rent-exempt minimum — attempting to claim rewards that would breach rent threshold returns an error
  2. Tax Program sell path propagates computed gross minimum to AMM CPI instead of passing `minimum_amount_out = 0`
  3. Epoch trigger bounty payment only executes when vault balance covers bounty + rent-exempt minimum
  4. Bonding Curve partial fill has explicit assertion that `actual_sol <= sol_amount` and refund claims validate partner curve identity
**Plans:** 2/2 plans complete
Plans:
- [x] 79-01-PLAN.md — Rent-exempt guards for staking/epoch + AMM floor for tax sell (FIN-01, FIN-02, FIN-03)
- [x] 79-02-PLAN.md — Partial fill assertion + partner curve identity validation (FIN-04, FIN-05)

#### Phase 80: Defense-in-Depth
**Goal**: All cross-program byte reads are ownership-verified, all u128-to-u64 casts are checked, and struct layout stability is enforced
**Depends on**: Phase 79
**Requirements**: DEF-01, DEF-02, DEF-03, DEF-04, DEF-05, DEF-06, DEF-07, DEF-08
**Success Criteria** (what must be TRUE):
  1. `read_pool_reserves()` rejects accounts not owned by AMM program — a spoofed account with valid byte layout but wrong owner fails
  2. All `u128 as u64` casts across staking, tax, and AMM are replaced with `try_from()` — overflow returns a descriptive error instead of silent truncation
  3. EpochState has 64-byte reserved padding, `#[repr(C)]` on both EpochState and Tax mirror struct, with a unit test asserting layout match
  4. Carnage pool owner constraint, bonding curve remaining_accounts validation, and Token::from_u8 checked conversion are all enforced
**Plans**: 3 plans
Plans:
- [ ] 80-01-PLAN.md — Pool reader ownership verification + is_reversed detection (DEF-01, DEF-02, DEF-06)
- [ ] 80-02-PLAN.md — Checked u128-to-u64 casts + Token::from_u8 safety (DEF-04, DEF-07)
- [x] 80-03-PLAN.md — EpochState reserved padding + layout assertions + remaining_accounts validation (DEF-03, DEF-05, DEF-08)
