### 1.3 Rationale for Separate `app/package.json`

**Recommendation: Use a separate package.json for the frontend.**

Why:
- The root `package.json` targets CommonJS (`"module": "commonjs"`) for Anchor tests. Next.js uses ESM internally. Conflicting module systems cause build errors.
- The root `tsconfig.json` uses `"target": "es2020"` and `"module": "commonjs"`. Next.js needs its own TypeScript configuration with JSX support, `"moduleResolution": "bundler"`, and React-specific settings.
- Frontend dependencies (React, Next.js, Tailwind, wallet-adapter, Privy, TradingView, Drizzle ORM) are large and unrelated to Anchor tests. Keeping them separate avoids bloating `node_modules` for the Rust build pipeline.
- Railway deployment only needs `app/` -- a separate package.json makes the build context clean.

**Do NOT use npm/pnpm workspaces** unless complexity demands it later. The `shared/` directory uses relative imports (no package resolution needed). Keep it simple.

### 1.4 Shared Code Strategy

The current codebase has three locations with overlapping constants:

| Location | What | Used By |
|----------|------|---------|
| `tests/integration/helpers/constants.ts` | PDA seeds, fee BPS, derivation helpers | Integration tests |
| `scripts/deploy/lib/pda-manifest.ts` | PDA derivation + manifest generation | Deploy scripts |
| `scripts/e2e/lib/swap-flow.ts` | Hook resolution, swap building | E2E tests |

**Problem:** These import from each other in fragile relative chains (e.g., `pda-manifest.ts` imports from `../../../tests/integration/helpers/constants`).

**Solution: Extract shared constants to `shared/`.**

```
shared/
  constants.ts    # Extract from tests/integration/helpers/constants.ts
                  # All PDA seeds, token decimals, fee BPS
                  # PDA derivation functions (derivePoolPDA, deriveVaultPDAs, etc.)
  program-ids.ts  # Extract from Anchor.toml + pda-manifest.json
                  # Export by cluster: { localnet, devnet, mainnet }
  types.ts        # EpochStateSnapshot, BalanceSnapshot, PDAManifest
                  # Extracted from scripts/vrf/lib/epoch-reader.ts, etc.
  manifest.ts     # PDA manifest reader (reads pda-manifest.json)
```

Then update imports:
- `tests/integration/helpers/constants.ts` -> re-exports from `../../shared/constants`
- `scripts/deploy/lib/pda-manifest.ts` -> imports from `../../../shared/constants`
- `app/src/lib/pda.ts` -> imports from `../../shared/constants`

**This is a refactor step that should happen BEFORE building frontend components.** Without it, the frontend will either duplicate constants (drift risk) or create deeply nested relative imports that break.

## 2. Technology Decisions

### 2.1 Next.js App Router (Recommended)

**Use Next.js App Router, NOT Pages Router.**

| Criterion | App Router | Pages Router |
|-----------|-----------|--------------|
| Server Components | Yes (default) | No |
| API Routes | Route Handlers (app/api/) | Pages API (/pages/api/) |
| Streaming/Suspense | Native | Hack |
| Layout nesting | Built-in | Manual |
| Status | Recommended for new projects | Legacy |
| Data fetching | Server Components + React Query | getServerSideProps/getStaticProps |

For this project:
- Server Components are valuable for the chart data API (SSR the initial candle data, hydrate on client)
- Route Handlers handle Helius webhooks cleanly
- Layout nesting maps naturally to the app structure (dashboard layout wrapping swap/stake/chart)

**Confidence: MEDIUM** -- App Router is the recommended path for new Next.js projects as of my training data (early 2025). Verify this is still the case with Next.js 15.x.

### 2.2 State Management

**Use Zustand for client state + React Query (TanStack Query) for server/RPC state.**

| State Type | Tool | Why |
|------------|------|-----|
| On-chain data (pool balances, epoch state) | TanStack Query | Auto-refresh, cache invalidation, loading/error states |
| UI state (modal open, selected token) | Zustand | Lightweight, no boilerplate |
| Wallet state | wallet-adapter context | Already provided by library |
| Transaction state | Custom hook + Zustand | TX pending/confirmed/error |

Why not Redux: Too much boilerplate for this use case. Zustand is ~1KB and sufficient.

**Confidence: HIGH** -- These are well-established patterns for Solana dApps.

### 2.3 Styling

**Use Tailwind CSS.**

Rationale: Fast iteration, consistent styling, works perfectly with Next.js App Router. No runtime overhead (static CSS generation). The DeFi space overwhelmingly uses Tailwind.

**Component library: Consider shadcn/ui for base components (buttons, modals, inputs, tabs).** It provides unstyled, accessible primitives built on Radix UI, styled with Tailwind. Not a dependency -- copies components into your project, so you own the code.

**Confidence: HIGH** for Tailwind. **MEDIUM** for shadcn/ui -- verify latest compatibility with Next.js App Router.

### 2.4 Database

**Use Postgres (via Railway) with Drizzle ORM.**

| Criterion | Drizzle | Prisma |
|-----------|---------|--------|
| Bundle size | ~50KB | ~5MB (query engine) |
| Type safety | Excellent (schema-first) | Excellent (generated) |
| Serverless | Native (no engine) | Requires Accelerate or query engine |
| SQL control | Full (write SQL if needed) | Abstracted |
| Migrations | Built-in | Built-in |

For this project:
- OHLCV candle storage needs efficient time-range queries. Drizzle lets you write raw SQL when needed.
- Railway Postgres is a managed service -- Drizzle connects directly via connection string.
- Serverless-friendly matters because Next.js API routes are ephemeral.

**Confidence: MEDIUM** -- Drizzle is the recommended choice based on training data. Verify current version compatibility.

### 2.6 Charting

**Use TradingView Lightweight Charts (open source) via `lightweight-charts` npm package.**

NOT the TradingView Charting Library (which requires a commercial license). The Lightweight Charts library is Apache-2.0 licensed, provides OHLCV candlestick charts, and is commonly used in DeFi frontends.

The data flow:
1. Helius webhook fires on swap transactions
2. API route parses swap data, computes price, inserts into `swap_events`
3. A background job (or on-insert trigger) aggregates `swap_events` into `candles` at multiple resolutions
4. Frontend fetches candles via API route, renders with Lightweight Charts
5. New swaps update the chart via polling (or WebSocket if warranted later)

**Confidence: MEDIUM** -- Lightweight Charts is the standard open-source choice. Verify current React wrapper situation (there was an official React wrapper in development).

### 3.2 Real-Time Data: Polling vs WebSocket

**Recommendation: Start with polling via React Query. Add WebSocket later if needed.**

| Data Type | Update Frequency | Method | Interval |
|-----------|-----------------|--------|----------|
| Pool prices (reserves) | On every swap (~seconds to minutes) | RPC poll via React Query | 5-10 seconds |
| Epoch state (tax rates, cheap side) | Every ~30 minutes | RPC poll via React Query | 30 seconds |
| Carnage fund balance | On trigger (~2x/day) | RPC poll via React Query | 30 seconds |
| Staking rewards (user) | On every epoch | RPC poll via React Query | 30 seconds |
| Chart candles | On every swap | API poll (latest candle) | 10 seconds |

**Why NOT WebSocket for initial build:**

1. **Solana's `onAccountChange` WebSocket** is available but unreliable:
   - Public RPC WebSocket connections drop frequently
   - Helius free tier has WebSocket limits
   - Connection management adds complexity
   - React Query polling is simpler and equally effective for 5-10s intervals

2. **Server-Sent Events (SSE)** is an option for chart updates but adds backend complexity

3. **The protocol's update frequency doesn't demand sub-second latency:**
   - Epochs change every 30 minutes
   - Carnage triggers ~2x/day
   - Pool prices move on swaps but 5-10s delay is acceptable for a UI

**Upgrade path:** If the swap volume grows and users demand real-time charts, add a WebSocket layer:
- Backend: Postgres LISTEN/NOTIFY on `swap_events` insert -> SSE to connected clients
- Frontend: EventSource connection upgrades React Query cache

**Confidence: HIGH** -- This is the standard pattern. WebSocket adds complexity without proportional benefit at launch volume.

### 3.3 Pool Price Reading

Pool prices are derived from on-chain PoolState accounts. The existing codebase reads pool reserves directly from account bytes:

From `carnage-flow.ts` context and the `read_pool_reserves()` pattern:
- PoolState data is at fixed byte offsets (offset 137 for reserve_a, offset 145 for reserve_b)
- Price = reserve_a / reserve_b (or inverse depending on direction)

**Frontend approach:**

```typescript
// hooks/usePoolPrices.ts
import { useQuery } from '@tanstack/react-query';
import { Connection, PublicKey } from '@solana/web3.js';

const POOL_STATE_RESERVE_A_OFFSET = 137;
const POOL_STATE_RESERVE_B_OFFSET = 145;

function readReserves(data: Buffer): { reserveA: bigint; reserveB: bigint } {
  const reserveA = data.readBigUInt64LE(POOL_STATE_RESERVE_A_OFFSET);
  const reserveB = data.readBigUInt64LE(POOL_STATE_RESERVE_B_OFFSET);
  return { reserveA, reserveB };
}

export function usePoolPrice(poolAddress: string) {
  const { connection } = useSolanaConnection();

  return useQuery({
    queryKey: ['pool-price', poolAddress],
    queryFn: async () => {
      const accountInfo = await connection.getAccountInfo(
        new PublicKey(poolAddress)
      );
      if (!accountInfo?.data) throw new Error('Pool not found');
      const { reserveA, reserveB } = readReserves(Buffer.from(accountInfo.data));
      return {
        reserveA: Number(reserveA),
        reserveB: Number(reserveB),
        price: Number(reserveA) / Number(reserveB),
      };
    },
    refetchInterval: 5000, // 5 seconds
  });
}
```

**Critical note:** The byte offsets (137, 145) are derived from the Anchor account discriminator (8 bytes) + preceding fields. These MUST be verified against the current PoolState struct. Alternatively, use Anchor's `program.account.poolState.fetch()` which deserializes properly but requires the full Anchor Program setup in the browser.

**Recommendation: Use Anchor deserialization in the frontend, not raw byte offsets.** Raw byte offsets are fragile and break if the struct changes. The Anchor IDL provides safe deserialization. The performance difference is negligible.

### 3.5 Staking Data

```typescript
// hooks/useStaking.ts
export function useStaking(userPublicKey: PublicKey | null) {
  const { stakingProgram, stakePoolPda } = useProtocol();

  // Global pool state
  const poolQuery = useQuery({
    queryKey: ['stake-pool'],
    queryFn: () => stakingProgram.account.stakePool.fetch(stakePoolPda),
    refetchInterval: 30000,
  });

  // User stake position
  const userStakeQuery = useQuery({
    queryKey: ['user-stake', userPublicKey?.toBase58()],
    queryFn: async () => {
      if (!userPublicKey) return null;
      const [userStakePda] = PublicKey.findProgramAddressSync(
        [Buffer.from('user_stake'), userPublicKey.toBuffer()],
        stakingProgram.programId
      );
      try {
        return await stakingProgram.account.userStake.fetch(userStakePda);
      } catch {
        return null; // User has not staked
      }
    },
    enabled: !!userPublicKey,
    refetchInterval: 30000,
  });

  return { poolQuery, userStakeQuery };
}
```

### 4.3 ALT Loading in Frontend

The ALT is protocol infrastructure -- one per network, cached at `scripts/deploy/alt-address.json`.

**Frontend approach:**

```typescript
// app/src/lib/transactions.ts

// ALT address is a constant per deployment (stored in config)
const ALT_ADDRESS = {
  devnet: 'EyUncwUhSwCVyTnbeoKe7Ug33sUDGAQDLDBQ5fVP9Vuf',
  mainnet: '', // TBD at mainnet launch
};

let cachedALT: AddressLookupTableAccount | null = null;

export async function getProtocolALT(
  connection: Connection,
  cluster: 'devnet' | 'mainnet'
): Promise<AddressLookupTableAccount> {
  if (cachedALT) return cachedALT;

  const altAddress = new PublicKey(ALT_ADDRESS[cluster]);
  const altAccount = await connection.getAddressLookupTable(altAddress);

  if (!altAccount.value) {
    throw new Error(`Protocol ALT not found at ${altAddress.toBase58()}`);
  }

  cachedALT = altAccount.value;
  return cachedALT;
}
```

### 4.4 Hook Account Resolution in Browser

The existing `resolveHookAccounts()` in `scripts/e2e/lib/swap-flow.ts` uses `createTransferCheckedWithTransferHookInstruction` from `@solana/spl-token`. This works in the browser -- the function queries the on-chain ExtraAccountMetaList PDA to resolve the hook accounts.

**Key requirement:** The frontend must call `resolveHookAccounts()` before building any swap transaction. This is an RPC call (reads ExtraAccountMetaList), so it adds latency. Cache the result -- hook accounts are static (the ExtraAccountMetaList is set at protocol initialization and never changes).

```typescript
// Cache hook accounts per mint -- they never change
const hookAccountCache = new Map<string, AccountMeta[]>();

export async function getCachedHookAccounts(
  connection: Connection,
  source: PublicKey,
  mint: PublicKey,
  dest: PublicKey,
  authority: PublicKey
): Promise<AccountMeta[]> {
  const cacheKey = `${mint.toBase58()}-${source.toBase58()}-${dest.toBase58()}`;
  if (hookAccountCache.has(cacheKey)) {
    return hookAccountCache.get(cacheKey)!;
  }

  const accounts = await resolveHookAccounts(
    connection, source, mint, dest, authority, BigInt(0)
  );
  hookAccountCache.set(cacheKey, accounts);
  return accounts;
}
```

### 4.5 Transaction Signing: Wallet-Adapter vs Privy

**Both must work.** The transaction building code must be wallet-agnostic.

```typescript
// Unified transaction sender
export async function sendProtocolTransaction(
  connection: Connection,
  instructions: TransactionInstruction[],
  feePayer: PublicKey,
  signTransaction: (tx: VersionedTransaction) => Promise<VersionedTransaction>,
  alt: AddressLookupTableAccount
): Promise<string> {
  const { blockhash, lastValidBlockHeight } =
    await connection.getLatestBlockhash('confirmed');

  const messageV0 = new TransactionMessage({
    payerKey: feePayer,
    recentBlockhash: blockhash,
    instructions,
  }).compileToV0Message([alt]);

  const vtx = new VersionedTransaction(messageV0);
  const signedVtx = await signTransaction(vtx); // wallet-agnostic!

  const txSig = await connection.sendTransaction(signedVtx, {
    skipPreflight: true, // Required for v0 on devnet
    maxRetries: 3,
  });

  const confirmation = await connection.confirmTransaction(
    { signature: txSig, blockhash, lastValidBlockHeight },
    'confirmed'
  );

  // CRITICAL: v0 + skipPreflight means failed TXs are still "confirmed"
  if (confirmation.value.err) {
    throw new Error(
      `Transaction failed on-chain: ${JSON.stringify(confirmation.value.err)}`
    );
  }

  return txSig;
}
```

The `signTransaction` callback comes from either:
- **wallet-adapter:** `wallet.signTransaction(vtx)` from `useWallet()` hook
- **Privy:** `privyWallet.signTransaction(vtx)` from Privy's Solana wallet interface

This abstraction means all transaction logic is wallet-agnostic.

## 5. Wallet Architecture: Privy + Wallet-Adapter Coexistence

### 5.1 The Problem

The project needs two wallet systems:
1. **@solana/wallet-adapter** for external wallets (Phantom, Solflare, Backpack)
2. **Privy** for embedded wallets (social login, email, phone)

These provide different APIs but must produce the same result: a `publicKey` and a `signTransaction()` function.

### 5.2 Recommended Pattern

**Use a unified wallet provider that wraps both systems.**

```typescript
// providers/WalletProvider.tsx (conceptual)

interface UnifiedWallet {
  publicKey: PublicKey | null;
  connected: boolean;
  signTransaction: (tx: VersionedTransaction) => Promise<VersionedTransaction>;
  signAllTransactions: (txs: VersionedTransaction[]) => Promise<VersionedTransaction[]>;
  disconnect: () => Promise<void>;
  walletType: 'external' | 'embedded' | null;
}

function useUnifiedWallet(): UnifiedWallet {
  const walletAdapter = useWallet(); // from @solana/wallet-adapter-react
  const { authenticated, user } = usePrivy();
  const privySolanaWallet = useSolanaWallets(); // from @privy-io/react-auth

  // Priority: If Privy is authenticated, use Privy wallet
  // Otherwise, use wallet-adapter
  if (authenticated && privySolanaWallet) {
    return {
      publicKey: new PublicKey(privySolanaWallet.address),
      connected: true,
      signTransaction: privySolanaWallet.signTransaction,
      signAllTransactions: privySolanaWallet.signAllTransactions,
      disconnect: () => privyLogout(),
      walletType: 'embedded',
    };
  }

  if (walletAdapter.connected) {
    return {
      publicKey: walletAdapter.publicKey,
      connected: true,
      signTransaction: walletAdapter.signTransaction,
      signAllTransactions: walletAdapter.signAllTransactions,
      disconnect: walletAdapter.disconnect,
      walletType: 'external',
    };
  }

  return {
    publicKey: null,
    connected: false,
    signTransaction: async () => { throw new Error('No wallet connected'); },
    signAllTransactions: async () => { throw new Error('No wallet connected'); },
    disconnect: async () => {},
    walletType: null,
  };
}
```

### 6.1 Architecture

### 6.2 Helius Webhook Configuration

Set up webhooks for:
1. **Swap transactions** -- Monitor the Tax Program ID for `swapSolBuy` and `swapSolSell` instructions
2. **Epoch transitions** -- Monitor the Epoch Program ID for `consumeRandomness` instruction
3. **Carnage events** -- Monitor the Epoch Program ID for `executeCarnageAtomic` instruction
4. **Staking events** -- Monitor the Staking Program ID for `stake`, `unstake`, `claimRewards`

**Webhook type: "Enhanced Transaction"** -- Helius parses the raw transaction and provides structured data including instruction names, account labels, and token transfers. This is much easier to work with than raw transaction data.

### 6.3 Webhook Receiver

```typescript
// app/src/app/api/webhooks/helius/route.ts
import { NextRequest, NextResponse } from 'next/server';

export async function POST(req: NextRequest) {
  // Verify webhook authenticity (Helius provides a webhook secret)
  const authHeader = req.headers.get('authorization');
  if (authHeader !== `Bearer ${process.env.HELIUS_WEBHOOK_SECRET}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  const body = await req.json();

  // Helius sends an array of enhanced transactions
  for (const tx of body) {
    await processTransaction(tx);
  }

  return NextResponse.json({ ok: true });
}

async function processTransaction(tx: HeliusEnhancedTransaction) {
  // Identify instruction type from the first program instruction
  const instruction = tx.instructions?.[0];
  if (!instruction) return;

  const programId = instruction.programId;

  if (programId === PROGRAM_IDS.TaxProgram) {
    await processSwapEvent(tx);
  } else if (programId === PROGRAM_IDS.EpochProgram) {
    if (instruction.name === 'consumeRandomness') {
      await processEpochTransition(tx);
    } else if (instruction.name === 'executeCarnageAtomic') {
      await processCarnageEvent(tx);
    }
  }
}
```

### 6.4 Price Calculation from Swap Events

Price is calculated from the swap amounts:

```typescript
function calculatePrice(
  amountIn: number,  // SOL lamports
  amountOut: number, // Token base units
  direction: 'buy' | 'sell'
): number {
  // For buy: price = SOL_in / tokens_out
  // For sell: price = SOL_out / tokens_in
  const solAmount = direction === 'buy' ? amountIn : amountOut;
  const tokenAmount = direction === 'buy' ? amountOut : amountIn;

  // Convert to human-readable (both use 6 decimals for SOL/token)
  return (solAmount / 1e9) / (tokenAmount / 1e6);
}
```

### 6.5 Candle Aggregation

Aggregate `swap_events` into `candles`:

```sql
-- Example: Aggregate 1-minute candles
INSERT INTO candles (pool, resolution, open_time, open, high, low, close, volume, tx_count)
SELECT
  pool,
  '1m',
  date_trunc('minute', block_time) as open_time,
  (ARRAY_AGG(price ORDER BY block_time ASC))[1] as open,
  MAX(price) as high,
  MIN(price) as low,
  (ARRAY_AGG(price ORDER BY block_time DESC))[1] as close,
  SUM(CASE WHEN direction = 'buy' THEN amount_in ELSE amount_out END) as volume,
  COUNT(*) as tx_count
FROM swap_events
WHERE pool = $1 AND block_time >= $2 AND block_time < $3
GROUP BY pool, date_trunc('minute', block_time)
ON CONFLICT (pool, resolution, open_time)
DO UPDATE SET
  high = GREATEST(candles.high, EXCLUDED.high),
  low = LEAST(candles.low, EXCLUDED.low),
  close = EXCLUDED.close,
  volume = candles.volume + EXCLUDED.volume,
  tx_count = candles.tx_count + EXCLUDED.tx_count;
```

Run this aggregation:
- **On each webhook insert** for 1m candles (low latency)
- **Via periodic cron** (Railway cron or pg_cron) for 5m, 15m, 1h, 4h, 1d (batch efficiency)

```typescript
import taxProgramIdl from '../idl/tax_program.json';

// Note: Anchor's Program constructor in the browser needs a provider
// but for READ-ONLY operations, a "read-only provider" without a wallet works.
// For WRITE operations, the user's wallet must be the provider's wallet.

export function createReadOnlyPrograms(connection: Connection) {
  // Anchor requires a wallet for Provider, but for reads we use a dummy
  const readOnlyProvider = new AnchorProvider(
    connection,
    { publicKey: PublicKey.default, signTransaction: async (tx) => tx, signAllTransactions: async (txs) => txs },
    { commitment: 'confirmed' }

    amm: new Program(ammIdl as any, readOnlyProvider),
    taxProgram: new Program(taxProgramIdl as any, readOnlyProvider),
    epochProgram: new Program(epochProgramIdl as any, readOnlyProvider),
    staking: new Program(stakingIdl as any, readOnlyProvider),
  };
}

export function createWritePrograms(provider: AnchorProvider) {
  return {
    amm: new Program(ammIdl as any, provider),
    taxProgram: new Program(taxProgramIdl as any, provider),
    epochProgram: new Program(epochProgramIdl as any, provider),
    staking: new Program(stakingIdl as any, provider),
  };
}
```

## 12. Critical Pitfalls Specific to This Architecture

### 12.1 v0 Transaction + skipPreflight Error Handling

The existing codebase documents this extensively: when using `skipPreflight: true` with v0 transactions, **failed transactions are still "confirmed" on Solana**. The frontend MUST check `confirmation.value.err` after confirmation. Forgetting this means the UI shows "success" when the transaction actually failed on-chain.

This is already implemented correctly in `scripts/e2e/lib/alt-helper.ts` lines 390-409. Reuse this pattern exactly.

### 12.2 Anchor IDL Enum Handling

The existing `epoch-reader.ts` handles two different enum serialization formats:
```typescript
// cheapSide can be either:
// - A number (0 = CRIME, 1 = FRAUD) -- Anchor numeric enum
// - An object ({ crime: {} } or { fraud: {} }) -- Anchor object enum
```

This dual handling is required because Anchor's serialization format for fieldless Rust enums varies depending on the Anchor version and whether the account is fetched via `fetch()` or deserialized manually. The frontend must replicate this.

### 12.3 Buffer Polyfill in Next.js

PDA derivation uses `Buffer.from("seed_string")`. The `Buffer` global is not available in the browser by default. Next.js with webpack 5 does not polyfill Node.js globals.

Solutions:
- Install `buffer` package and add to webpack config
- Or use `TextEncoder` instead of `Buffer.from` for seed bytes
- Or use Uint8Array directly

The `@solana/web3.js` package typically handles this via its own polyfills, but verify.

### 12.4 BigInt Serialization

Anchor account fields like `held_amount`, `total_sol_spent`, etc. are `u64` which deserialize as `BN` (big number) objects. React components cannot render `BN` directly. Convert to `string` or `number` at the data boundary (in the query function, not the component).

### 12.5 Helius Webhook Reliability

Webhooks can be missed (Helius does retry, but there is no guarantee). The chart data pipeline must be able to backfill:
- On startup, query recent transactions via Helius `getSignaturesForAddress` API
- Compare with stored `swap_events` and fill gaps
- This is a background job, not blocking for chart display

### 12.6 RPC Rate Limiting

The project uses Helius for RPC. Free tier has rate limits. With React Query polling multiple accounts every 5-30 seconds:
- 4 pool accounts at 5s = ~48 calls/min
- 1 epoch state at 30s = 2 calls/min
- 1 stake pool at 30s = 2 calls/min
- 1 user stake at 30s = 2 calls/min
- Total: ~54 calls/min base load

This should be within Helius free tier limits but monitor and adjust intervals if needed. Consider `getMultipleAccountsInfo` to batch pool reads into a single call.
