2. **Add accounts to ConsumeRandomness struct:**
   ```rust
   /// Staking authority PDA - Epoch Program signs CPIs to Staking.
   /// CHECK: PDA derived from this program's seeds
   #[account(
       seeds = [STAKING_AUTHORITY_SEED],
       bump,
   )]
   pub staking_authority: AccountInfo<'info>,

   /// Staking Program's pool state (mutable for update_cumulative).
   /// CHECK: Validated by Staking Program during CPI
   #[account(mut)]
   pub stake_pool: AccountInfo<'info>,

   /// Staking Program
   /// CHECK: Program ID validated in CPI
   pub staking_program: AccountInfo<'info>,
   ```

3. **Add CPI logic in handler** after step 7 (Clear VRF pending state), before step 8 (Emit event):

   Insert as new step 7.5: "CPI TO STAKING: FINALIZE EPOCH YIELD"

   ```rust
   // === 7.5. CPI TO STAKING: FINALIZE EPOCH YIELD ===
   // Notify staking that the epoch has advanced and yield is finalized.
   // This happens AFTER tax derivation, BEFORE Carnage check (per CONTEXT.md).
   let staking_authority_bump = ctx.bumps.staking_authority;
   let staking_authority_seeds: &[&[u8]] = &[
       STAKING_AUTHORITY_SEED,
       &[staking_authority_bump],
   ];

   // Build instruction data: discriminator (8) + epoch (4)
   let mut ix_data = Vec::with_capacity(12);
   ix_data.extend_from_slice(&UPDATE_CUMULATIVE_DISCRIMINATOR);
   ix_data.extend_from_slice(&epoch_state.current_epoch.to_le_bytes());

   let update_cumulative_ix = Instruction {
       program_id: ctx.accounts.staking_program.key(),
       accounts: vec![
           AccountMeta::new_readonly(ctx.accounts.staking_authority.key(), true),
           AccountMeta::new(ctx.accounts.stake_pool.key(), false),
       ],
       data: ix_data,
   };

   invoke_signed(
       &update_cumulative_ix,
       &[
           ctx.accounts.staking_authority.to_account_info(),
           ctx.accounts.stake_pool.to_account_info(),
           ctx.accounts.staking_program.to_account_info(),
       ],
       &[staking_authority_seeds],
   )?;

   msg!("Staking cumulative updated for epoch {}", epoch_state.current_epoch);
   ```

4. **Error handling:** The CPI uses `?` so if staking fails, consume_randomness reverts atomically. Per CONTEXT.md, this is the desired behavior - staking update is critical.

5. **Update errors.rs** (optional but good practice):
   Add error code for staking CPI failure if you want custom error handling:
   ```rust
   /// Staking CPI failed
   #[msg("Staking CPI failed")]
   StakingCpiFailed,
   ```
   However, the `?` propagation will return the underlying Staking error, which may be more informative. Adding this error is optional.

TIMING NOTE: The CPI must happen AFTER tax derivation (steps 5-6), AFTER vrf_pending cleared (step 7), but BEFORE the TaxesUpdated event (step 8) and BEFORE any Carnage check (Phase 25). This ordering is per CONTEXT.md: "validate randomness -> derive new rates -> finalize old epoch yield -> check Carnage".

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_update_cumulative_discriminator() {
        // Verify discriminator matches sha256("global:update_cumulative")[0..8]
        use sha2::{Sha256, Digest};
        let mut hasher = Sha256::new();
        hasher.update(b"global:update_cumulative");
        let result = hasher.finalize();
        let expected: [u8; 8] = result[0..8].try_into().unwrap();
        assert_eq!(UPDATE_CUMULATIVE_DISCRIMINATOR, expected);
    }

}
