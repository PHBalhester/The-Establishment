Purpose: The desktop iris-open animation and sized modals don't work on mobile. Mobile needs fullscreen slide-up panels (like iOS sheets) with a left-aligned back arrow close button (iOS convention). These CSS and component changes are the foundation that makes all existing modal content mobile-ready without any changes to station components.

<task type="auto">
  <name>Task 1: Add mobile modal CSS overrides and slide-up/slide-down keyframes to globals.css</name>
  <files>app/app/globals.css</files>
  <action>
Add a new CSS section at the end of the modal-related CSS (after the existing Privy top-layer section, before the station content section, or at a logical point after all desktop modal CSS). The section should be clearly commented as "Mobile Modal Overrides -- Phase 58".

Add these two keyframe definitions (outside the media query, since keyframes are global):

```css
@keyframes mobile-slide-up {
  from { transform: translateY(100%); opacity: 0; }
  to   { transform: translateY(0); opacity: 1; }
}

@keyframes mobile-slide-down {
  from { transform: translateY(0); opacity: 1; }
  to   { transform: translateY(100%); opacity: 0; }
}
```

Then add the `@media (width < 64rem)` block with these overrides:

1. `dialog.modal-shell` -- fullscreen: `width: 100%; max-width: none !important; max-height: none; height: 100dvh; margin: 0; border-radius: 0; clip-path: none !important;`
   - `!important` on max-width is necessary to override the inline `style={{ maxWidth }}` in ModalShell.tsx
   - `!important` on clip-path overrides the JS-set inline clip-path from iris animation
   - `100dvh` instead of `100vh` for iOS Safari address bar safety (Pitfall 1 from research)
   - `max-height: none` overrides browser UA default `max-height` on modal dialogs (Pitfall 2)

2. `.modal-chrome` -- fullscreen inner: `border-radius: 0; max-height: none; height: 100%;`

3. `.modal-bolt` -- hidden on mobile: `display: none;` (corner bolts too small on mobile, waste of DOM)

4. `dialog.modal-shell.iris-opening` -- replace iris with slide-up: `animation: mobile-slide-up 300ms cubic-bezier(0.22, 1, 0.36, 1) forwards;`

5. `dialog.modal-shell.closing` -- replace shrink with slide-down: `animation: mobile-slide-down 200ms ease-in forwards;`

6. `dialog.modal-shell[open]::backdrop` -- simpler backdrop on mobile (no blur, saves GPU): `animation: backdrop-fade-in 200ms ease-out forwards;` (same animation but shorter duration)

7. `.modal-close-btn` -- hide desktop close button: `display: none;` (mobile uses MobileBackButton instead)

8. `.mobile-back-btn` -- style the mobile back button: positioned absolutely at left side of header, `display: flex; align-items: center; justify-content: center; width: 48px; height: 48px; min-width: 48px; min-height: 48px;` (48px minimum tap target per success criteria), background transparent, border none, cursor pointer. On hover: slight brightness increase.

9. `.modal-header` -- adjust padding for mobile: remove the extra right padding (was for close button), add left padding to accommodate back button. Use `padding: 0.75rem 1rem 0.75rem 3.5rem;` to leave space for the absolutely-positioned back button on the left.

Also add a rule OUTSIDE the media query for `.mobile-back-btn` base styles for desktop -- on desktop it should have `display: none`. Then inside the media query, override to `display: flex`.

Structure the CSS so that:
- `.mobile-back-btn { display: none; }` (desktop default -- hidden)
- Inside `@media (width < 64rem)`: `.mobile-back-btn { display: flex; ... }` and `.modal-close-btn { display: none; }` (swap which is visible)

```tsx
function MobileBackButton({ onClick }: { onClick: () => void }) {
  return (

      type="button"

      onClick={onClick}
      aria-label="Close"
    >
      <svg
        width="20"
        height="20"

          strokeWidth="2.5"
          strokeLinecap="round"
          strokeLinejoin="round"
        />
      </svg>
    </button>
  );
}
```

In the ModalShell component's render section, add `MobileBackButton` to the header. Place it BEFORE the `<h2>` element so the DOM order is: back button -> title -> close button. The CSS media query handles which button is visible at which viewport:

```tsx
<header className="modal-header">
  <MobileBackButton onClick={closeModal} />
  <h2>{title}</h2>
  <ModalCloseButton onClick={closeModal} />
</header>
```

The responsive visibility is handled entirely by CSS in globals.css (Task 1):
- Desktop (>=1024px): `.mobile-back-btn { display: none }` and `.modal-close-btn { display: flex }`
- Mobile (<1024px): `.mobile-back-btn { display: flex }` and `.modal-close-btn { display: none }`

This approach avoids JS viewport detection (no hydration mismatches) and follows the existing CSS-first pattern.

The MobileBackButton's `position: absolute; left: 0.75rem; top: 50%; transform: translateY(-50%);` positioning is defined in the CSS (Task 1), mirroring how `.modal-close-btn` is positioned absolutely on the right.
