**Test pattern for each test:**
```typescript
it("rejects fake {account} in {instruction}", async () => {
  const fakeAccount = Keypair.generate();
  // Fund fake account so it doesn't fail on rent/lamports before reaching our constraint
  const sig = await connection.requestAirdrop(fakeAccount.publicKey, LAMPORTS_PER_SOL);
  await connection.confirmTransaction(sig);

  try {
    await taxProgram.methods
      .swapSolBuy(new BN(swapAmount), new BN(0), true)
      .accounts({
        ...correctAccounts,
        stakingEscrow: fakeAccount.publicKey, // SUBSTITUTED
      })
      .remainingAccounts(hookAccounts)
      .rpc();
    expect.fail("Should have rejected fake account");
  } catch (err) {
    // Verify it's the RIGHT error, not a random unrelated failure
    const errorCode = err?.error?.errorCode?.code;
    expect(errorCode).to.be.oneOf([
      "InvalidStakingEscrow", // Custom error if @ annotation works
      "ConstraintSeeds",       // Anchor default if seeds constraint fires first
    ]);
  }
});
```

**SEC-03: VRF Randomness Owner (3 tests):**
Tests 16-18: Fake randomness_account in trigger_epoch_transition, consume_randomness, retry_epoch_vrf

Attack: Create a regular system account (funded with SOL so it's rent-exempt) and pass it as randomness_account. The `owner = SWITCHBOARD_PROGRAM_ID` constraint checks the runtime owner field of the account. A system account has owner = System Program, which != SWITCHBOARD_PROGRAM_ID, so the constraint fires.

Pattern:
```typescript
it("rejects non-Switchboard randomness in trigger_epoch_transition", async () => {
  const fakeRandomness = Keypair.generate();
  await connection.requestAirdrop(fakeRandomness.publicKey, LAMPORTS_PER_SOL);
  // ... await confirmation ...

  try {
    await epochProgram.methods
      .triggerEpochTransition()
      .accounts({
        ...correctTriggerAccounts,
        randomnessAccount: fakeRandomness.publicKey,
      })
      .rpc();
    expect.fail("Should reject non-Switchboard randomness");
  } catch (err) {
    const errorCode = err?.error?.errorCode?.code;
    expect(errorCode).to.be.oneOf([
      "InvalidRandomnessOwner",
      "ConstraintOwner",
    ]);
  }
});
```

For trigger_epoch_transition: Needs epoch_state.initialized and epoch boundary reached. The owner constraint fires BEFORE the handler checks epoch boundary, so we don't need to wait for an actual epoch boundary. Just initialize epoch state (done in protocol-init).

For consume_randomness and retry_epoch_vrf: Same pattern. The owner constraint fires at deserialization time.

**SEC-07: Carnage WSOL Ownership (2 tests):**
Tests 19-20: Wrong-owner carnage_wsol in execute_carnage_atomic and execute_carnage

Attack: Create a WSOL token account owned by a DIFFERENT authority (not the carnage_signer PDA). The `constraint = carnage_wsol.owner == carnage_signer.key()` checks the TOKEN owner field (.owner is the token account authority, not the program owner).

Pattern:
```typescript
it("rejects wrong-owner carnage_wsol in execute_carnage_atomic", async () => {
  // Create a real WSOL token account but owned by a random authority
  const wrongOwner = Keypair.generate();
  await airdrop(wrongOwner.publicKey);
  const fakeWsol = await createWrappedNativeAccount(
    connection,
    payer,
    wrongOwner.publicKey, // wrong owner
    LAMPORTS_PER_SOL,
  );

  try {
    await epochProgram.methods
      .executeCarnageAtomic(/* params */)
      .accounts({
        ...correctCarnageAccounts,
        carnageWsol: fakeWsol, // wrong owner
      })
      .rpc();
    expect.fail("Should reject wrong-owner carnage_wsol");
  } catch (err) {
    const errorCode = err?.error?.errorCode?.code;
    expect(errorCode).to.be.oneOf([
      "InvalidCarnageWsolOwner",
      "ConstraintRaw",      // constraint = ... uses ConstraintRaw
      "NoCarnagePending",   // May reach handler if constraint doesn't fire first
    ]);
  }
});
```

NOTE on SEC-07 tests: The `constraint = carnage_wsol.owner == carnage_signer.key()` uses Anchor's `constraint` attribute which fires at deserialization. HOWEVER, `carnage_wsol` is typed as `Box<InterfaceAccount<'info, TokenAccount>>` which means Anchor will try to deserialize the account as a TokenAccount first. If the fake WSOL account IS a valid TokenAccount (just with wrong owner), the constraint will fire correctly.

Important: `createWrappedNativeAccount` from `@solana/spl-token` creates a valid WSOL token account under the SPL Token program (TOKEN_PROGRAM_ID). The carnage_wsol constraint checks the token account's `.owner` field (who can transfer from it), not the program owner. So a WSOL account created with a different authority will be valid structurally but fail the ownership constraint.

If execute_carnage_atomic has too many accounts to construct without carnage state being initialized, the constraint error should still fire before NoCarnagePending since Anchor validates accounts at deserialization time. If it doesn't and we get NoCarnagePending instead, that's still a PASS for security (the fake account didn't enable the attack). Accept either error code.
