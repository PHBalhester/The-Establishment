```typescript
import {

```

**PdaManifest interface:**
```typescript
interface PdaManifest {
  generatedAt: string;
  clusterUrl: string;
  programs: Record<string, string>;  // name -> programId pubkey string
  mints: Record<string, string>;     // name -> mint pubkey string
  pdas: Record<string, string>;      // descriptive name -> derived address string
  pools: Record<string, { pool: string; vaultA: string; vaultB: string }>;
}
```

**`generateManifest` function:**
- Parameters: `programIds: { amm, transferHook, taxProgram, epochProgram, staking }` (all PublicKey), `mints: { crime, fraud, profit }` (all PublicKey), `clusterUrl: string`
- Derive ALL PDAs:
  - **Transfer Hook:** WhitelistAuthority (["authority"]), ExtraAccountMetaList for each mint (["extra-account-metas", mint])
  - **AMM:** AdminConfig (["admin"]), SwapAuthority (["swap_authority"]), 4 pools + their vault pairs (using `derivePoolPDA`, `deriveVaultPDAs`, `canonicalOrder`)
  - **Tax:** TaxAuthority (["tax_authority"])
  - **Epoch:** EpochState (["epoch_state"]), CarnageFund (["carnage_fund"]), CarnageSolVault (["carnage_sol_vault"]), CarnageCrimeVault (["carnage_crime_vault"]), CarnageFraudVault (["carnage_fraud_vault"]), CarnageSigner (["carnage_signer"]), StakingAuthority (["staking_authority"])
  - **Staking:** StakePool (["stake_pool"]), EscrowVault (["escrow_vault"]), StakeVault (["stake_vault"])
- Include a `canonicalOrder` helper (copy from protocol-init.ts) -- sort two PublicKeys lexicographically for pool PDA derivation
- For pool entries in the manifest, use descriptive keys: "CRIME/SOL", "FRAUD/SOL", "CRIME/PROFIT", "FRAUD/PROFIT"
- Return the complete PdaManifest object

**`writeManifest` function:**
- Takes PdaManifest and writes two files:
  - `scripts/deploy/pda-manifest.json` -- JSON.stringify with 2-space indent
  - `scripts/deploy/pda-manifest.md` -- Markdown table with sections for Programs, Mints, PDAs, Pools

The markdown format should look like:
```markdown
# PDA Manifest
Generated: {timestamp}
Cluster: {url}

## Programs
| Program | Address |
|---------|---------|
| AMM | zFW9mo... |
...

## Mints
| Token | Address |
|-------|---------|
| CRIME | ... |
...

## Protocol PDAs
| Name | Address | Program |
|------|---------|---------|
| WhitelistAuthority | ... | Transfer Hook |
...

## Pool Addresses
| Pool | Pool PDA | Vault A | Vault B |
|------|----------|---------|---------|
| CRIME/SOL | ... | ... | ... |
...
```

<task type="auto">
  <name>Task 2: Create idempotent protocol initialization script</name>
  <files>scripts/deploy/initialize.ts</files>
  <action>
Create `scripts/deploy/initialize.ts` -- the main initialization script that executes ~32 transactions in strict dependency order with idempotent check-before-init for every step.

**Structure:** A single `main()` async function that runs all steps sequentially. No retry logic -- fail fast on any error (per CONTEXT.md decision).

**Environment:** Uses lib/connection.ts for provider, lib/logger.ts for progress, lib/account-check.ts for idempotency, lib/pda-manifest.ts for manifest output.

**Mint keypair persistence (critical -- see 33-RESEARCH.md Pitfall 2):**
- On first run, generate 3 mint Keypairs and save to `scripts/deploy/mint-keypairs/` (crime-mint.json, fraud-mint.json, profit-mint.json)
- On subsequent runs, load existing keypairs from that directory
- The `mint-keypairs/` directory should be gitignored (add to .gitignore)
- Helper function: `loadOrCreateMintKeypair(name: string): Keypair`

**Initialization sequence -- model on protocol-init.ts 17-step sequence but with these differences:**
1. No airdrops (handled by deploy.sh)
2. No test wallets (Step 17 is test-only)
3. Idempotent checks before every step
4. Save/load mint keypairs
5. Real error handling (not test assertions)
6. Logger for progress output

**Steps (numbered to match protocol-init.ts where applicable):**

Step 1: Create 3 Token-2022 mints with TransferHook extension (CRIME, FRAUD, PROFIT)
- For each mint: check if mint keypair already on disk -> load or generate
- For each mint: check `mintExists(connection, mintKeypair.publicKey)` -> skip if exists
- If creating: build transaction with `getMintLen([ExtensionType.TransferHook])`, create account, `createInitializeTransferHookInstruction`, `createInitializeMintInstruction`
- Set transfer hook program to the Transfer Hook program ID
- Set mint authority and freeze authority to the wallet pubkey
- Mint initial supply to admin wallet: 1,000,000,000 tokens (1B at 6 decimals = 1_000_000_000_000_000 base units) per mint
- Pattern reference: protocol-init.ts Step 2 (lines 246-298)

Step 2: Initialize Transfer Hook WhitelistAuthority
- Check `accountExists(connection, whitelistAuthorityPDA)` -> skip if exists
- Call `transferHook.methods.initializeAuthority().accountsStrict({...}).rpc()`
- Pattern reference: protocol-init.ts Step 3

Step 3: Initialize ExtraAccountMetaList for each mint (3 transactions)
- For each mint, derive ExtraAccountMetaList PDA
- Check existence -> skip if exists
- Call `transferHook.methods.initExtraAccountMetaList().accountsStrict({...}).rpc()`
- Pattern reference: protocol-init.ts Step 4

Step 4: Initialize AMM AdminConfig
- Derive AdminConfig PDA and ProgramData address (for upgrade authority check)
- Check `accountExists(connection, adminConfigPDA)` -> skip if exists
- Call `amm.methods.initializeAdmin().accountsStrict({...}).rpc()`
- Pattern reference: protocol-init.ts Step 5

Step 5: Create admin token accounts for seed liquidity
- Create Token-2022 accounts for CRIME, FRAUD, PROFIT (admin wallet)
- Create WSOL account (SPL Token, not Token-2022)
- Mint tokens to admin accounts (seed liquidity amounts from constants.ts)
- Skip if accounts already have balance
- Pattern reference: protocol-init.ts Step 6

Step 6: Whitelist admin Token-2022 accounts
- For each admin T22 account, derive whitelist entry PDA
- Check existence -> skip if exists
- Call `transferHook.methods.addWhitelistEntry().accountsStrict({...}).rpc()`
- Pattern reference: protocol-init.ts Step 6b

Steps 7-10: Initialize 4 AMM pools
- Step 7: CRIME/SOL pool (MixedPool)
- Step 8: FRAUD/SOL pool (MixedPool)
- Step 9: CRIME/PROFIT pool (PureT22Pool)
- Step 10: FRAUD/PROFIT pool (PureT22Pool)
- For each: derive pool PDA using canonicalOrder, check existence -> skip
- For each: call `amm.methods.initializePool(feeBps, poolType, seedAmountA, seedAmountB).accountsStrict({...}).rpc()`
- For T22 transfers during pool init: build remaining_accounts for Transfer Hook (extraAccountMetaList, whitelistSource, whitelistDest, hookProgramId)
- Pattern reference: protocol-init.ts Steps 7-10

Step 11: Whitelist all pool vault addresses
- For each of the 4 pools: derive vaultA and vaultB PDAs
- Only whitelist T22 vaults (not WSOL vaults, but CONTEXT.md says whitelist them all for simplicity -- match protocol-init.ts behavior)
- Check existence of each whitelist entry -> skip if exists
- Pattern reference: protocol-init.ts Step 11

Step 12: Initialize EpochState
- Derive EpochState PDA
- Check existence -> skip if exists
- Call `epochProgram.methods.initializeEpochState().accountsStrict({...}).rpc()`
- Pattern reference: protocol-init.ts Step 12

Step 13: Initialize StakePool (with dead stake)
- Derive StakePool, StakeVault, EscrowVault PDAs
- Check existence -> skip if exists
- Mint MINIMUM_STAKE (1,000,000 base units) of PROFIT to admin if not already done
- Build Transfer Hook remaining_accounts for the dead stake transfer
- Call `staking.methods.initializePool().accountsStrict({...}).remainingAccounts([...hookAccounts]).rpc()`
- Pattern reference: protocol-init.ts Step 13 (including Step 13b hook accounts)

Step 14: Whitelist StakeVault
- Derive whitelist entry PDA for StakeVault
- Check existence -> skip if exists
- Pattern reference: protocol-init.ts Step 14

Step 15: Initialize Carnage Fund
- Derive CarnageFund, CarnageSolVault, CarnageCrimeVault, CarnageFraudVault PDAs
- Check existence -> skip if exists
- Call `epochProgram.methods.initializeCarnageFund().accountsStrict({...}).rpc()`
- Note: Carnage token vaults are T22 accounts created by the program
- Pattern reference: protocol-init.ts Step 15

Step 16: Whitelist Carnage token vaults
- Derive whitelist entries for CarnageCrimeVault and CarnageFraudVault
- Check existence -> skip if exists
- Pattern reference: protocol-init.ts Step 16

Step 17: Fund Carnage SOL vault (with rent-exempt minimum)
- Check if CarnageSolVault has > 0 lamports -> skip if already funded
- Transfer rent-exempt minimum (from Solana runtime) to CarnageSolVault via SystemProgram.transfer
- This prevents tax deposit failures when the vault has 0 lamports
- Pattern reference: protocol-init.ts (Carnage SOL vault funding, discovered in Phase 32)

Step 18: Generate PDA manifest
- Call `generateManifest(...)` with all program IDs and mint addresses
- Call `writeManifest(manifest)` to write JSON and markdown files
- Print manifest file paths

**After all steps:**
- Print summary: total steps, completed, skipped
- Print PDA manifest file path
- Print tx log file path
- Exit 0

**Important implementation notes:**
- Use `canonicalOrder(mintA, mintB)` for ALL pool PDA derivations -- the on-chain program sorts mints lexicographically
- For pool type: use `{ mixedPool: {} }` for SOL pools, `{ pureT22Pool: {} }` for PROFIT pools (Anchor IDL enum format)
- For MixedPool SOL-side: WSOL (NATIVE_MINT) uses TOKEN_PROGRAM_ID, not TOKEN_2022_PROGRAM_ID
- Token-2022 transfers during init need remaining_accounts: [extraAccountMetaList, whitelistSource, whitelistDest, hookProgramId] -- see protocol-init.ts Step 13b for the exact pattern
- Seed liquidity amounts: use constants from tests/integration/helpers/constants.ts (SOL_POOL_SEED_SOL, SOL_POOL_SEED_TOKEN, PROFIT_POOL_SEED_A, PROFIT_POOL_SEED_B)
- All PublicKey derivations use `PublicKey.findProgramAddressSync` with the seeds from constants.ts
