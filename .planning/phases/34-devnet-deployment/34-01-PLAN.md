  <name>Task 1: Create .env with Helius devnet RPC and update connection.ts commitment</name>
  <files>.env, scripts/deploy/lib/connection.ts</files>
  <action>
  Two changes:

  **1. Create .env with Helius devnet config:**

  Add these lines to the existing `.env` file (which already has `SUPERMEMORY_CC_API_KEY`):

  ```
  HELIUS_API_KEY=[REDACTED-DEVNET-HELIUS-KEY]
  CLUSTER_URL=https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]
  COMMITMENT=finalized
  ```

  Do NOT remove the existing `SUPERMEMORY_CC_API_KEY` line.

  **2. Update connection.ts to read COMMITMENT env var:**

  In `loadProvider()` (scripts/deploy/lib/connection.ts), the commitment level is hardcoded to `"confirmed"`. Update it to read from `process.env.COMMITMENT`, defaulting to `"confirmed"` if not set. This preserves backward compatibility for localnet/test usage while allowing devnet to use `finalized`.

  Specifically, change the connection and provider creation to:
  ```typescript
  const commitment = (process.env.COMMITMENT as anchor.web3.Commitment) || "confirmed";
  const connection = new Connection(url, commitment);
  const provider = new anchor.AnchorProvider(connection, wallet, {
    preflightCommitment: commitment,
    commitment: commitment,
  });
  ```

  Update the JSDoc comment to mention the new COMMITMENT env var.

  **Why finalized?** On devnet, `confirmed` can occasionally report success for transactions that later get dropped. `finalized` guarantees the transaction is permanently recorded. The extra latency (~6s vs ~0.5s per tx) is acceptable for a one-time deployment of ~32 transactions.
  </action>

  3. `grep COMMITMENT .env` shows `finalized`
  4. `grep "process.env.COMMITMENT" scripts/deploy/lib/connection.ts` finds the env var read
  5. `grep '"confirmed"' scripts/deploy/lib/connection.ts` finds it only as a default fallback, not hardcoded
  </verify>
  <done>

  - The `_OVERRIDE` suffix makes it clear these are non-default values

  ```
  PROFIT_POOL_SEED_A_OVERRIDE=250000000000000
  PROFIT_POOL_SEED_B_OVERRIDE=25000000000000
  ```

  **Important:** Verify that `Number()` can handle these values without precision loss. JavaScript's `Number` is safe up to 2^53 (~9 quadrillion). The largest value is 290_000_000_000_000 (~290 trillion), well within safe integer range.

<task type="auto">
  <name>Task 3: Clear localnet mint keypairs and update deploy-all.sh to source .env</name>
  <files>scripts/deploy/mint-keypairs/, scripts/deploy/deploy-all.sh</files>
  <action>
  Two changes:

  **1. Delete existing localnet mint keypairs:**

  Remove the `scripts/deploy/mint-keypairs/` directory entirely. This was created during the Phase 33 localnet dry run and contains keypairs for localnet-specific mint addresses. Devnet needs its own fresh mint keypairs (initialize.ts will generate them on first run).

  ```bash
  rm -rf scripts/deploy/mint-keypairs/
  ```

  **Why fresh keypairs?** Localnet and devnet are separate clusters. Reusing localnet mint addresses on devnet could create confusion when comparing PDA manifests. Fresh keypairs give devnet its own clean identity.

  **2. Update deploy-all.sh to source .env if present:**

  Add a `.env` file loader at the top of deploy-all.sh (after the toolchain sourcing, before Configuration section). This ensures all env vars (CLUSTER_URL, COMMITMENT, seed liquidity overrides) are available to all child processes.

  Add this block after the `cd "$(dirname "$0")/../.."` line and before the Configuration section:

  ```bash
  # ---------------------------------------------------------------------------
  # Load .env file if present
  #
  # Why source .env? Configuration like CLUSTER_URL, COMMITMENT, HELIUS_API_KEY,
  # and seed liquidity overrides are stored in .env (gitignored). Sourcing it
  # here makes all values available to child processes (initialize.ts, verify.ts).
  # ---------------------------------------------------------------------------
  if [ -f .env ]; then
    set -a  # auto-export all variables
    source .env
    set +a
    echo "Loaded .env configuration"
  fi
  ```

  The `set -a` / `set +a` pattern ensures all variables from .env are exported to child processes. Without `set -a`, variables sourced from .env would only be available in the current shell, not in `npx tsx` child processes.

  **Why not just `export $(cat .env)`?** That pattern breaks with values containing special characters (like the `?` and `=` in Helius URLs). `source .env` with `set -a` handles all value formats correctly.
  </action>
