  <action>
    Create `scripts/deploy/test-upgrade.ts` that performs TWO complete upgrade cycles on the Conversion Vault (guinea pig — simplest program, fewest dependencies):

    **Cycle 1: Upgrade to modified version**
    1. Make a trivial source code change: add `msg!("Squads upgrade test v2");` to the `convert` instruction in `programs/conversion_vault/src/instructions/convert.rs` (or similar safe location). Save original file content for revert.
    2. Rebuild ONLY conversion_vault: `anchor build -p conversion_vault` (do NOT use build.sh which rebuilds everything)
    3. Write the new binary to a buffer account:
       `solana program write-buffer target/deploy/conversion_vault.so --url devnet --keypair keypairs/devnet-wallet.json`
       Parse the buffer address from stdout.
    4. Set buffer authority to the Squads vault PDA:
       `solana program set-buffer-authority <BUFFER> --new-buffer-authority <VAULT_PDA> --skip-new-upgrade-authority-signer-check --keypair keypairs/devnet-wallet.json --url devnet`
    5. Get the next transaction index from the multisig account.
    6. Create a vault transaction containing the BPFLoaderUpgradeable::Upgrade instruction:
       - programData PDA (derived from program ID)
       - program account (conversion vault program ID)
       - buffer account
       - spill account (vault PDA — receives buffer rent refund)
       - rent sysvar
       - clock sysvar
       - upgrade authority (vault PDA)
       Use `multisig.rpc.vaultTransactionCreate()` with the serialized instruction.
    7. Create a proposal: `multisig.rpc.proposalCreate()`
    8. Approve with signer 1: `multisig.rpc.proposalApprove()` — load keypairs/squads-signer-1.json
    9. Approve with signer 2: `multisig.rpc.proposalApprove()` — load keypairs/squads-signer-2.json
       (Threshold is 2, so proposal is now Approved)
    10. Wait for timelock: Poll every 10 seconds. Read the proposal account, check if enough time has elapsed since approval. Timelock is 300 seconds (5 min) on devnet.
        Log countdown: "Timelock: {remaining}s remaining..."
    11. Execute: `multisig.rpc.vaultTransactionExecute()` with sendOptions: { skipPreflight: true }
    12. Verify: Read the program's executable data hash (or simply check that the ProgramData account's `last_deploy_slot` changed). Log success.

    **Cycle 2: Revert to original version**
    13. Restore the original source file (remove the test msg!).
    14. Rebuild conversion_vault.
    15. Repeat steps 3-12 with the original binary (write buffer, set authority, propose, approve x2, wait timelock, execute, verify).

    **Script design notes:**
    - Load signer keypairs from keypairs/squads-signer-{1,2,3}.json
    - Load multisig PDA and vault PDA from deployments/devnet.json (squadsMultisig, squadsVault fields)
    - Use existing lib/connection.ts patterns
    - Handle errors gracefully — if any step fails, log which step and the error
    - The BPFLoaderUpgradeable::Upgrade instruction has discriminator = 3 (little-endian u32). Accounts order: [programData(w), program(w), buffer(w), spill(w), rent(r), clock(r), authority(s)]. The vault PDA is the authority but does NOT sign directly — Squads program handles the PDA signing.
    - For the TransactionMessage payerKey, use the vault PDA (it's the "signer" of the inner instruction)
    - IMPORTANT: The project path has a space. Use proper quoting for all CLI commands.
    - IMPORTANT: Buffer write can be expensive on devnet (~0.5-1 SOL). Check wallet balance first.

    - 3 wallets: which devices/locations
    - One signer should be a Ledger hardware wallet (mainnet requirement from CONTEXT.md)
    - How to add Ledger signer to Squads (if supported by SDK, otherwise via Squads web UI)

    **3. Initial Setup Procedure**
    - Step-by-step: run setup-squads.ts with mainnet env
    - Verification: confirm multisig on Squads app, check vault PDA on explorer
    - Exact env vars needed: SQUADS_TIMELOCK_SECONDS=900

    **4. Authority Transfer Procedure**
    - Step-by-step: run transfer-authority.ts with mainnet env
    - Order: all 7 upgrade authorities first, then 3 admin PDAs
    - Verification after EACH transfer (not batch)
    - Run verify-authority.ts to confirm all 10
    - WARNING: This is one-way for upgrade authorities. Triple-check before executing.

    **5. Performing a Program Upgrade**
    - When: only for critical bug fixes or auditor-recommended changes
    - Steps: modify code, build, write buffer, set buffer authority, create proposal via test-upgrade.ts pattern (or Squads app), approve with 2 of 3, wait timelock, execute
    - Verification: confirm program bytecode changed, run verify.ts
    - Rollback: keep original .so binary, can revert using same flow

    **6. Timelock Progression Schedule**
    - Launch: 15 minutes (SQUADS_TIMELOCK_SECONDS=900)
    - After 48-72hr stable operation: extend to 24 hours (86400)
    - After external audit: consider authority burn
    - How to change timelock: create a config change proposal through Squads (requires 2-of-3 + current timelock wait)

    **7. Emergency Procedures**
    - Lost signer key: 2-of-3 still works with remaining 2. Add replacement signer via config change proposal.
    - All signers compromised: impossible to prevent upgrade (that's the tradeoff). Timelock gives community notice.
    - Protocol bug: hotfix through standard upgrade flow. 15-min timelock at launch allows fast response.

    **8. Authority Burn Sequence** (post-audit only)
    - Order: WhitelistAuthority (burn_authority) -> AMM AdminConfig (burn_admin) -> BcAdminConfig (burn_bc_admin) -> upgrade authorities (set to None via Squads proposal)
    - Each burn is IRREVERSIBLE. Document the "last things you can do" before each burn.
    - Upgrade authority burn = program becomes immutable forever.

    **9. Verification Commands**
    - List every verification command with expected output
    - `npx tsx scripts/deploy/verify-authority.ts` — all 10 PASS
    - `solana program show <PROGRAM_ID>` — shows vault PDA as authority
    - Squads app: https://app.squads.so/ — shows pending/executed proposals
