# Phase 91: Deploy Config Foundation - Research

**Researched:** 2026-03-12
**Domain:** Deployment infrastructure, TypeScript code generation, bash pipeline orchestration
**Confidence:** HIGH

## Summary

This phase restructures existing deployment infrastructure rather than introducing new libraries. The core work is: (1) evolving `pda-manifest.json` into a richer `deployments/{cluster}.json` config, (2) building a TypeScript code generator that produces `shared/constants.ts` from that config, (3) extending `deploy-all.sh` from 4 phases to 7 with mainnet safety gates, and (4) upgrading `verify.ts` with deep on-chain validation.

All building blocks already exist in the codebase. `pda-manifest.ts` generates the manifest schema, `shared/constants.ts` is the target file (currently hand-maintained, ~594 lines), and `deploy-all.sh` already orchestrates the 4-phase pipeline. The work is restructuring and automating what's currently manual.

**Primary recommendation:** Build incrementally -- config schema first, then code generator, then pipeline extension, then verification. Each piece is independently testable by comparing its output against the existing hand-maintained equivalents.

**When to use:** Any time code needs a protocol address -- read from deployment.json (scripts) or from auto-generated constants.ts (app/frontend).

### Pattern 2: Code Generator with Full Overwrite
**What:** `generate-constants.ts` reads deployment.json and writes the complete `shared/constants.ts` file. No partial updates, no merge logic.
**When to use:** After deployment.json is created/updated (pipeline Phase 4).
**Key aspects:**
- Template is a TypeScript string built programmatically
- Static values (seeds, decimals, fee constants) are hardcoded in the generator template
- Dynamic values (addresses) are read from deployment.json
- Pre-computed PDAs emitted as `new PublicKey("...")` constants (no async derivation at runtime)
- `// AUTO-GENERATED` header prevents accidental manual edits
- Generator must produce the EXACT same exports and types as current constants.ts to avoid breaking 41+ consumer files

### Pattern 3: Pipeline Extension with Explicit Cluster Argument
**What:** `deploy-all.sh` requires `devnet` or `mainnet` as first argument. No auto-detection from Solana CLI config.
**When to use:** Every deploy invocation.
**Key aspects:**
- Sources `.env.devnet` or `.env.mainnet` based on argument
- Cross-validates: env file's CLUSTER_URL must match the cluster argument
- Mainnet path adds confirmation prompts (SOL balance, cost estimate)
- Phase numbering: 0-6 (mint keypairs, build, deploy, initialize, generate-constants, ALT, verify)

### Anti-Patterns to Avoid
- **Partial config updates:** Never patch individual fields in deployment.json. Always regenerate the full file from on-chain state + derived PDAs. Partial updates lead to stale/inconsistent data.
- **Manual constants.ts edits:** The generator must be the ONLY writer. Any manual edit will be overwritten on next pipeline run. The `// AUTO-GENERATED` header + git diff will catch violations.
- **Cluster auto-detection from Solana CLI config:** The Solana CLI config (`solana config get`) can silently point to the wrong cluster. Explicit argument prevents deploying mainnet programs to devnet or vice versa.
- **Hardcoded addresses in multiple places:** Every address should trace back to deployment.json. If an address appears in code that isn't in deployment.json or derived from it, that's a bug.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| PDA derivation | Manual seed computation | Existing `generateManifest()` in `lib/pda-manifest.ts` | Already correct, tested across 90 phases, handles canonical mint ordering |
| Canonical mint ordering | Manual byte comparison | Existing `canonicalOrder()` in `lib/pda-manifest.ts` | NATIVE_MINT edge case already handled |
| .so binary inspection | Custom ELF parser | `grep -c` on compiled .so files for base58 address strings | Solana addresses are plaintext strings in compiled BPF bytecode, no binary parsing needed |
| JSON schema validation | Hand-written validators | TypeScript interface + type assertion | deployment.json is generated by our own code, not user input |
| Two-pass build detection | Manual feature flag tracking | Read existing `patch-mint-addresses.ts` pattern | Already knows which programs need mint addresses compiled in |

**Key insight:** This phase is 90% restructuring existing code and 10% new code. The PDA derivation, manifest generation, and verification logic are already battle-tested. The new work is wiring them together differently.

## Common Pitfalls

### Pitfall 1: Breaking 41 Consumer Files
**What goes wrong:** Changing `shared/constants.ts` export names or types breaks app/, scripts/, tests/ imports.
**Why it happens:** The generated file must be API-compatible with the current hand-maintained version.
**How to avoid:** Before writing the generator, catalog every export from current `constants.ts` and every import across the 41 consumer files. The generator's output must export the same names with the same types.
**Warning signs:** TypeScript compilation errors after running the generator.

### Pitfall 2: pda-manifest.json Consumers Break During Migration
**What goes wrong:** Scripts that read `scripts/deploy/pda-manifest.json` break when it disappears.
**Why it happens:** Multiple scripts import from pda-manifest.json directly (verify.ts, create-alt.ts, crank-runner.ts, e2e scripts).
**How to avoid:** During migration, keep `pda-manifest.json` as a symlink or thin wrapper that reads from `deployments/{cluster}.json`. Remove after all consumers are updated. Or update all consumers in the same plan.
**Warning signs:** `ENOENT: no such file or directory` errors in scripts.

### Pitfall 3: Missing Fields in deployment.json vs pda-manifest.json
**What goes wrong:** The new deployment.json schema is missing fields that pda-manifest.json had (or vice versa).
**Why it happens:** pda-manifest.json has been incrementally extended over 90 phases. Easy to miss a field.
**How to avoid:** Diff the existing pda-manifest.json schema against the new deployment.json schema. Every field in pda-manifest.json must appear in deployment.json.
**Warning signs:** Undefined values when code reads deployment.json fields.

### Pitfall 4: Env Var Guard Not Catching All Required Vars
**What goes wrong:** initialize.ts checks for some env vars but misses others, leading to partial initialization with wrong defaults.
**Why it happens:** The Phase 69 bug -- pool seed env vars were missing, test defaults (10 SOL / 10K tokens) were used instead of production values (2.5 SOL / 290M tokens). Cost 50 SOL.
**How to avoid:** Collect ALL required env vars in a single check at the top of initialize.ts. List: `SOL_POOL_SEED_SOL_OVERRIDE`, `SOL_POOL_SEED_TOKEN_OVERRIDE`, `HELIUS_API_KEY`, `CLUSTER_URL`, plus any metadata URIs for non-localhost.
**Warning signs:** Test defaults appearing in deployment output.

### Pitfall 5: .so Binary Grep False Positives
**What goes wrong:** Grepping for devnet addresses in mainnet .so files produces false positives (partial matches or coincidental byte patterns).
**Why it happens:** Base58 addresses are 32-44 characters. Short substrings could match other data.
**How to avoid:** Grep for the FULL base58 string of each devnet address. Use `grep -c "FULL_ADDRESS"` not substring matching. Only check the 3 feature-flagged programs (vault, tax, epoch, bonding_curve) since those are the only ones with compiled-in addresses.
**Warning signs:** False positive aborts on clean mainnet builds.

### Pitfall 6: Cluster Argument Parsing in Bash
**What goes wrong:** `./deploy-all.sh` with no argument silently proceeds with wrong cluster.
**Why it happens:** Current script defaults to localhost if no argument provided.
**How to avoid:** Check `$1` at the top of the script. If empty or not "devnet"/"mainnet", print usage and exit 1.
**Warning signs:** Deploying to wrong cluster.

## Code Examples

### Example 1: generate-constants.ts Core Structure
```typescript
// Reads deployment.json, writes shared/constants.ts
import * as fs from "fs";
import * as path from "path";

interface DeploymentConfig {
  schemaVersion: number;
  cluster: string;
  programs: Record<string, string>;
  mints: Record<string, string>;
  pdas: Record<string, string>;
  pools: Record<string, { pool: string; vaultA: string; vaultB: string }>;
  curvePdas: Record<string, { curveState: string; tokenVault: string; solVault: string; taxEscrow: string }>;
  alt: string;
  treasury: string;
}

function generateConstants(config: DeploymentConfig): string {
  // Build the full constants.ts file as a string
  // Include AUTO-GENERATED header
  // Emit all exports matching current constants.ts API
  let out = `// AUTO-GENERATED from deployments/${config.cluster}.json -- DO NOT EDIT MANUALLY\n`;
  out += `// Generated: ${new Date().toISOString()}\n\n`;
  // ... emit imports, PROGRAM_IDS, MINTS, SEEDS (static), DEVNET_PDAS, etc.
  return out;
}
```

### Example 2: Env Var Guard Pattern
```typescript
// Top of initialize.ts -- fail fast on missing env vars
const REQUIRED_VARS_NON_LOCALHOST = [
  "SOL_POOL_SEED_SOL_OVERRIDE",
  "SOL_POOL_SEED_TOKEN_OVERRIDE",
  "HELIUS_API_KEY",
  "CLUSTER_URL",
];

function validateEnvVars(clusterUrl: string): void {
  if (clusterUrl === "http://localhost:8899") return; // localhost is test-only

  const missing = REQUIRED_VARS_NON_LOCALHOST.filter(v => !process.env[v]);
  if (missing.length > 0) {
    console.error("\n!!! MISSING REQUIRED ENVIRONMENT VARIABLES !!!\n");
    for (const v of missing) {
      console.error(`  - ${v}`);
    }
    console.error("\nSet these in .env.devnet or .env.mainnet before running.\n");
    process.exit(1);
  }
}
```

### Example 3: Binary Address Verification
```bash
# In deploy-all.sh Phase 2 (mainnet guard)
if [ "$CLUSTER" = "mainnet" ]; then
  echo "Checking for devnet addresses in mainnet binaries..."
  DEVNET_CONFIG="deployments/devnet.json"
  if [ -f "$DEVNET_CONFIG" ]; then
    FEATURE_PROGRAMS=("conversion_vault" "tax_program" "epoch_program" "bonding_curve")
    FOUND_DEVNET=false
    for prog in "${FEATURE_PROGRAMS[@]}"; do
      SO_FILE="target/deploy/${prog}.so"
      # Extract all address values from devnet config
      while IFS= read -r addr; do
        if grep -q "$addr" "$SO_FILE" 2>/dev/null; then
          echo "  ABORT: Found devnet address $addr in $SO_FILE"
          FOUND_DEVNET=true
        fi
      done < <(jq -r '.. | strings | select(length >= 32 and length <= 44)' "$DEVNET_CONFIG" | sort -u)
    done
    if [ "$FOUND_DEVNET" = true ]; then
      echo ""
      echo "ERROR: Devnet addresses found in mainnet binaries."
      echo "Did you rebuild with mainnet mint keypairs?"
      exit 1
    fi
  fi
fi
```

### Example 4: Cluster-Aware .env Sourcing
```bash
# deploy-all.sh top section
CLUSTER="${1:-}"
if [ -z "$CLUSTER" ] || { [ "$CLUSTER" != "devnet" ] && [ "$CLUSTER" != "mainnet" ]; }; then
  echo "Usage: ./deploy-all.sh <devnet|mainnet>"
  echo ""
  echo "  Cluster argument is REQUIRED. No auto-detection."
  exit 1
fi

ENV_FILE=".env.${CLUSTER}"
if [ ! -f "$ENV_FILE" ]; then
  echo "ERROR: ${ENV_FILE} not found. Create it from .env.example."
  exit 1
fi

set -a
source "$ENV_FILE"
set +a

# Cross-validate cluster URL matches argument
if [ "$CLUSTER" = "devnet" ] && ! echo "$CLUSTER_URL" | grep -qi "devnet"; then
  echo "ERROR: CLUSTER_URL in $ENV_FILE doesn't look like devnet: $CLUSTER_URL"
  exit 1
fi
if [ "$CLUSTER" = "mainnet" ] && echo "$CLUSTER_URL" | grep -qi "devnet"; then
  echo "ERROR: CLUSTER_URL in $ENV_FILE looks like devnet but target is mainnet: $CLUSTER_URL"
  exit 1
fi
```

## Open Questions

1. **PROFIT Pool Addresses**
   - What we know: Current pda-manifest.json only has CRIME/SOL and FRAUD/SOL pools. But constants.ts references no PROFIT pools in DEVNET_POOL_CONFIGS.
   - What's unclear: Does deployment.json need CRIME/PROFIT and FRAUD/PROFIT pool entries? These are vault conversions, not AMM pools.
   - Recommendation: Vault conversions don't have pools -- they use VaultConfig/VaultCrime/VaultFraud/VaultProfit PDAs which are already in the pdas section. No pool entries needed for PROFIT conversions.

2. **Existing .env Migration**
   - What we know: There's currently a single `.env` file. INFRA-13 requires `.env.devnet` and `.env.mainnet`.
   - What's unclear: Can we rename existing `.env` to `.env.devnet` without breaking other workflows (Railway, local dev)?
   - Recommendation: Create `.env.devnet` and `.env.mainnet` as the deploy-pipeline files. Keep `.env` as the "active" file for local dev/Railway (unchanged). deploy-all.sh sources `.env.{cluster}` explicitly.

3. **Anchor.toml Sync**
   - What we know: Anchor.toml `[programs.devnet]` has program IDs that must match deployment.json.
   - What's unclear: Should generate-constants.ts also update Anchor.toml? Or is that a separate concern?
   - Recommendation: Keep Anchor.toml updates separate -- `verify-program-ids.ts` already checks consistency. deployment.json is the source of truth; Anchor.toml is an Anchor framework requirement. Don't auto-modify Anchor.toml.
