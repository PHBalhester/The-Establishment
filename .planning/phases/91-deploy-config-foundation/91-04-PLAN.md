<task type="auto">
  <name>Task 1: Upgrade verify.ts to read deployment.json and perform deep on-chain checks</name>
  <files>
    scripts/deploy/verify.ts
  </files>
  <action>
  Extend the existing verify.ts (currently ~663 lines) to read from deployment.json instead of (or in addition to) pda-manifest.json:

  1. **Config loading**: Read `deployments/{cluster}.json` as the primary source. Import `DeploymentConfig` type from `./lib/deployment-schema`. Determine cluster from CLUSTER_URL env var (existing pattern). Fall back to pda-manifest.json if deployment.json doesn't exist yet (backward compat during migration).

  2. **Deep verification checks** (extend existing checks):
     - **Programs**: For each of the 7 programs in deployment.json, verify:
       - Account exists on-chain
       - Account is executable (`accountInfo.executable === true`)
       - Owner is BPF Loader (`BPFLoaderUpgradeable`)
       - Upgrade authority matches deployment.json `authority.deployer` (or `authority.squadsVault` if transferred)
     - **Mints**: For each of the 3 mints, verify:
       - Mint account exists
       - Mint authority is correct (deployer or expected PDA)
       - Decimals match expected (6)
       - Supply is non-zero (tokens have been minted)
     - **PDAs**: For each PDA in deployment.json, verify:
       - Account exists on-chain
       - Owner is the expected program
     - **Pools**: For each pool, verify:
       - Pool account exists and is initialized
       - Vault accounts exist with non-zero token balance (if seeded)
       - Pool mint ordering matches expected (canonical order)
     - **ALT**: If alt address is non-null, verify:
       - ALT account exists
       - ALT contains expected addresses (spot-check a few key addresses)
     - **Authority**: Verify deployer wallet matches deployment.json `authority.deployer`

  3. **Report output**: Produce a tabular report (similar to existing format) with PASS/FAIL per check. At the end, print summary: "X/Y checks passed". Exit with code 1 if any FAIL.

  4. **Keep backward compat**: If pda-manifest.json exists, still read it and cross-check against deployment.json (warn on mismatches). This eases the migration period.
  </action>
  <verify>
  - `npx tsx scripts/deploy/verify.ts` reads from deployments/devnet.json (check log output)
  - Verify report shows checks for all 7 programs, 3 mints, PDAs, pools, and ALT
  - Exit code is 0 if all checks pass on current devnet deployment
  - `npx tsc --noEmit scripts/deploy/verify.ts` compiles without errors
  </verify>
  <done>verify.ts reads deployment.json and performs deep on-chain verification of all protocol addresses including program executability, mint properties, PDA ownership, pool initialization, and ALT contents.</done>
</task>

<task type="auto">
  <name>Task 2: Add devnet-address-in-mainnet-binary detection to deploy-all.sh</name>
  <files>
    scripts/deploy/deploy-all.sh
  </files>
  <action>
  Add a binary address verification step to deploy-all.sh that runs ONLY on mainnet builds, AFTER Phase 1 (Build) and BEFORE Phase 2 (Deploy):

  1. **Guard condition**: Only run when `CLUSTER="mainnet"` and `deployments/devnet.json` exists (need devnet addresses to check against).

  2. **Check logic**:
     - Read all address strings from `deployments/devnet.json` using `jq` (extract all string values that look like base58 addresses: 32-44 chars, alphanumeric)
     - Only check the 4 feature-flagged programs that compile-in mint addresses: `conversion_vault.so`, `tax_program.so`, `epoch_program.so`, `bonding_curve.so`
     - For each .so file, grep for each devnet address (FULL string match, not substring)
     - If ANY devnet address is found in ANY mainnet .so file:
       - Print clear error: "ABORT: Found devnet address {ADDR} in {PROGRAM}.so"
       - Print hint: "Did you rebuild with mainnet mint keypairs? Run Phase 0 with mainnet keypairs first."
       - `exit 1`

  3. **Label**: Add this as "Phase 1.5: Binary Address Verification" between Build and Deploy phases in the output.

  4. **Skip gracefully**: If `deployments/devnet.json` doesn't exist (first-ever deploy), skip with a warning: "No devnet.json found, skipping binary address cross-check".

  5. **Performance**: The grep is fast (< 1 second per .so file). No need for optimization.

  NOTE: This task modifies deploy-all.sh which is also modified by Plan 03 Task 1. Since both plans are Wave 2, the executor should apply Plan 03's changes first (it does the major rewrite), then Plan 04 adds this specific section. If the executor runs Plan 04 first, this task should be aware of the original deploy-all.sh structure and the changes will be integrated when Plan 03 runs. The key insertion point is between Build and Deploy phases regardless of the overall structure.
  </action>
  <verify>
  - deploy-all.sh contains a "Binary Address Verification" section
  - The check only runs when CLUSTER="mainnet"
  - The check greps for full base58 addresses from devnet.json in .so files
  - The check covers conversion_vault.so, tax_program.so, epoch_program.so, bonding_curve.so
  - If a devnet address is found, the script aborts with exit 1
  </verify>
  <done>Mainnet builds abort if any devnet address from deployment.json is found baked into the compiled .so binaries. Clear error message identifies which address in which program.</done>
</task>
