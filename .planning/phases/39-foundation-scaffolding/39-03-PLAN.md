  <action>
    1. Install Drizzle ORM as a dev dependency in app/:
       ```bash
       cd app && npm install -D drizzle-orm drizzle-kit
       ```
       Drizzle ORM is dev-only for now because we're defining schema without connecting to a database. When Phase 44 provisions Postgres, it will move to a regular dependency.

    2. Create `app/db/schema.ts` with all 4 tables. Follow the exact schema from 39-RESEARCH.md "Drizzle Schema for All 4 Event Tables" section, with these specifics per CONTEXT.md decisions:

    **swap_events** (rich data capture per CONTEXT.md):
    - `tx_signature` varchar(128) PRIMARY KEY -- Solana TX signature, natural idempotency key
    - `pool` varchar(64) NOT NULL -- pool PDA address string
    - `direction` varchar(4) NOT NULL -- "buy" or "sell"
    - `sol_amount` bigint NOT NULL -- lamports
    - `token_amount` bigint NOT NULL -- base units (6 decimals)
    - `price` real NOT NULL -- token price in SOL
    - `tax_amount` bigint NOT NULL -- lamports of tax paid
    - `lp_fee` bigint NOT NULL -- lamports of LP fee
    - `slippage` real -- actual slippage percentage (nullable, may not always be calculable)
    - `user_wallet` varchar(64) NOT NULL -- user's wallet address
    - `epoch_number` integer NOT NULL -- which epoch the swap occurred in
    - `timestamp` timestamp NOT NULL
    - Indexes: pool, epoch_number, timestamp, user_wallet

    **candles** (OHLCV for all 4 pools):
    - `id` integer PRIMARY KEY generated always as identity
    - `pool` varchar(64) NOT NULL -- pool PDA address
    - `resolution` varchar(4) NOT NULL -- "1m", "5m", "15m", "1h", "4h", "1d"
    - `open_time` timestamp NOT NULL -- start of the candle period
    - `open` real NOT NULL
    - `high` real NOT NULL
    - `low` real NOT NULL
    - `close` real NOT NULL
    - `volume` bigint NOT NULL -- in lamports
    - `trade_count` integer NOT NULL DEFAULT 0
    - Unique index: (pool, resolution, open_time) -- prevents duplicate candles
    - Indexes: (pool, resolution), open_time

    **epoch_events** (snapshot at each epoch transition):
    - `id` integer PRIMARY KEY generated always as identity
    - `epoch_number` integer NOT NULL
    - `tx_signature` varchar(128) NOT NULL
    - `cheap_side` varchar(8) NOT NULL -- "crime" or "fraud"
    - `crime_buy_tax` integer NOT NULL -- bps
    - `crime_sell_tax` integer NOT NULL -- bps
    - `fraud_buy_tax` integer NOT NULL -- bps
    - `fraud_sell_tax` integer NOT NULL -- bps
    - `staking_reward_deposited` bigint -- lamports deposited to staking this epoch (nullable for epochs with no deposit)
    - `carnage_fund_balance` bigint -- lamports in carnage fund at transition time
    - `timestamp` timestamp NOT NULL
    - Unique index: epoch_number
    - Index: timestamp

    **carnage_events** (full execution trace per Claude's discretion from CONTEXT.md):
    - `id` integer PRIMARY KEY generated always as identity
    - `epoch_number` integer NOT NULL
    - `tx_signature` varchar(128) NOT NULL
    - `crime_burned` bigint NOT NULL -- base units burned
    - `fraud_burned` bigint NOT NULL -- base units burned
    - `sol_used_for_buy` bigint NOT NULL -- lamports used in buy step
    - `crime_bought` bigint -- base units bought (nullable, depends on path)
    - `fraud_bought` bigint -- base units bought (nullable, depends on path)
    - `carnage_sol_before` bigint -- lamports in fund before execution
    - `carnage_sol_after` bigint -- lamports in fund after execution
    - `path` varchar(32) -- "BuyOnly", "Burn", "BurnAndSell"
    - `target_token` varchar(8) -- "CRIME" or "FRAUD" (which was the cheap side)
    - `timestamp` timestamp NOT NULL
    - Unique index: epoch_number (one carnage per epoch max)
    - Index: timestamp

    Use these Drizzle imports:
    ```typescript
    import {
      pgTable, varchar, integer, bigint, real,
      timestamp, boolean, text, index, uniqueIndex
    } from "drizzle-orm/pg-core";
    ```

    Use `bigint("col_name", { mode: "number" })` for lamport/token amounts. This gives JavaScript number type (safe up to 2^53 which covers all practical Solana amounts).

    Use `integer("col_name").primaryKey().generatedAlwaysAsIdentity()` for auto-incrementing IDs (not `serial` -- identity columns are the modern PostgreSQL best practice).

    Add a header comment explaining this is schema-only (no DB connection) and referencing Phase 44 for DB provisioning.
