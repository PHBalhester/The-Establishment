**BigRedButton.tsx** (`app/components/station/BigRedButton.tsx`):

Create the 3D physical swap execution button (INTERACT-05). The CSS classes (.big-red-button and all data-state variants) are already in globals.css from Plan 01.

1. `'use client'` directive, JSDoc header.
2. Props interface:
   ```tsx
   interface BigRedButtonProps {
     status: SwapStatusType; // from useSwap: 'idle' | 'building' | 'signing' | 'sending' | 'confirming' | 'confirmed' | 'failed'
     disabled: boolean;
     onSwap: () => void;
     onReset: () => void;
     txSignature: string | null;
     errorMessage: string | null;
     connected: boolean;
   }
   ```
3. Import `SwapStatusType` from `@/hooks/useSwap` (or inline the union type if not exported -- check the file).

4. **data-state attribute management** -- derive and auto-reset:
   - Map status to data-state value:
     - 'idle' or undefined -> no data-state (default red glow from CSS)
     - 'building' | 'signing' | 'sending' | 'confirming' -> data-state="loading"
     - 'confirmed' -> data-state="success"
     - 'failed' -> data-state="error"
   - Use `useState` to hold current `dataState: string | undefined`.
   - Use a `useEffect` watching `status` to set `dataState`:
     ```tsx
     useEffect(() => {
       if (['building', 'signing', 'sending', 'confirming'].includes(status)) {
         setDataState('loading');
       } else if (status === 'confirmed') {
         setDataState('success');
         // Auto-reset after success animation completes (1.5s matches CSS keyframe)
         const timer = setTimeout(() => setDataState(undefined), 1500);
         return () => clearTimeout(timer);
       } else if (status === 'failed') {
         setDataState('error');
         // Auto-reset after shake animation completes (0.4s matches CSS keyframe)
         const timer = setTimeout(() => setDataState(undefined), 400);
         return () => clearTimeout(timer);
       } else {
         setDataState(undefined);
       }
     }, [status]);
     ```

5. **Toast notifications** -- use `useToast()` from ToastProvider:
   - Use a separate `useEffect` watching `status` + `txSignature` for toast triggers:
     - On transition to 'confirmed': `showToast('success', 'Swap confirmed!', { label: 'View on Solscan', href: \`https://solscan.io/tx/${txSignature}?cluster=devnet\` })`
     - On transition to 'failed': `showToast('error', errorMessage || 'Swap failed')`
   - Use a `prevStatusRef` (useRef) to detect transitions (only fire toast when status CHANGES to confirmed/failed, not on re-renders):
     ```tsx
     const prevStatusRef = useRef(status);
     useEffect(() => {
       const prev = prevStatusRef.current;
       prevStatusRef.current = status;
       if (prev !== status) {
         if (status === 'confirmed') showToast('success', ...);
         if (status === 'failed') showToast('error', ...);
       }
     }, [status, txSignature, errorMessage, showToast]);
     ```

6. onClick handler:
   - If status is 'confirmed' or 'failed', call `onReset()`.
   - If status is 'idle' and not disabled and connected, call `onSwap()`.
   - Otherwise, no-op.

7. Render a `<button>` element:
   - `className="big-red-button"` (CSS is in globals.css from Plan 01)
   - `data-state={dataState}` (drives CSS visual state transitions)
   - `disabled={disabled && status === 'idle'}` (not disabled during loading/success/error)
   - `aria-label` describing current state (e.g., "Execute swap", "Swap in progress", "Swap confirmed", "Swap failed")

8. Content inside button:
   - idle + connected: "SWAP" text (font-heading, text-sm, uppercase, text-white)
   - idle + !connected: "Connect" text (dimmed, text-white/50)
   - loading: a small spinning element (use CSS `animate-spin` on a gear/circle icon)
   - confirmed: checkmark icon or "OK" text (brief, before auto-reset)
   - failed: "!" or warning text (brief, before auto-reset)

Create `app/components/station/SwapStation.tsx` -- the layout compositor for the swap modal.

**Layout decision:** BigRedButton renders INSIDE SwapForm via the `renderAction` prop. This is the correct approach because SwapForm's internal `useSwap()` hook owns the swap state (inputAmount, status, txSignature, etc.). Calling useSwap() a second time at SwapStation level would create a separate form state instance where BigRedButton's `onSwap` has no input amount. The renderAction pattern keeps BigRedButton connected to the correct swap state without duplication. Visually, the BigRedButton replaces SwapStatus in the exact same position (below the swap inputs) -- the 3D button appearance is the visual upgrade, not its position.

1. `'use client'` directive, JSDoc header explaining layout: stats bar (top), chart (middle), form with BigRedButton (bottom).
2. DEFAULT EXPORT (required by React.lazy in ModalContent.tsx).
3. Import: SwapStatsBar, BigRedButton, SwapForm, CandlestickChart, ChartControls, useChartData, DEVNET_POOLS from @dr-fraudsworth/shared.

**Chart data wiring:**

4. Set up chart state with pool selector:
   ```tsx
   import { useState } from 'react';
   import { DEVNET_POOLS } from '@dr-fraudsworth/shared';

   const DEFAULT_POOL = DEVNET_POOLS.CRIME_SOL.pool.toBase58();

   export default function SwapStation() {
     const [chartPool, setChartPool] = useState(DEFAULT_POOL);
     const {
       candles,
       loading: chartLoading,
       range,
       setRange,
       resolution,
       setResolution,
       connectionStatus,
     } = useChartData(chartPool);
   ```

5. Pass chart data to ChartControls and CandlestickChart:

   ```tsx
     <ChartControls
       pool={chartPool}
       onPoolChange={setChartPool}
       range={range}
       onRangeChange={setRange}

       onResolutionChange={setResolution}
       connectionStatus={connectionStatus}
     />
     <CandlestickChart
       candles={candles}

     />
   </div>
   ```

   - `chartPool` state is managed at SwapStation level (not shared with SwapForm's token pair selection -- the chart pool selector in ChartControls is independent).

**Layout structure:**
