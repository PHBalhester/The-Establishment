  <name>Task 1: Mathematical Invariant Deep-Dive (GAP-02)</name>
  <files>.planning/audit/GAPS.md</files>
  <action>
Systematically identify ALL mathematical invariants the protocol must maintain.

Expected invariants to verify are documented:
1. **AMM Constant Product**: reserve_a * reserve_b >= k_initial (per pool)
2. **Total Supply Conservation**: sum(all_balances) == total_supply (per token)
3. **No Negative Balances**: balance >= 0 always
4. **Tax Distribution**: yield_share + carnage_share + treasury_share == 100%
5. **Epoch Monotonicity**: epoch_n+1 > epoch_n always
6. **Yield Escrow Solvency**: escrow_balance >= sum(all_pending_claims)
7. **Cumulative Only Increases**: new_cumulative >= old_cumulative always

For each invariant, check if the specs document:
- Explicit statement of the invariant
- What happens if violated
- How to detect violation
- Recovery procedure (if any)

Log gaps for:
- Invariants not explicitly stated
- Invariants stated but no violation consequences documented
- Missing boundary conditions (min/max values)
- Missing overflow protection documentation

Continue sequential GAP-XXX numbering. Use severity:
- CRITICAL: Security/correctness invariant not explicitly stated
- HIGH: Invariant stated but consequences/detection missing
- MEDIUM: Boundary conditions not explicit

Systematically audit ALL state machines for complete transition documentation.

State machines to audit:
1. **Epoch State Machine** (Epoch_State_Machine_Spec.md)
   - States: ACTIVE, VRF_PENDING
   - Transitions: advance_epoch, vrf_callback, retry_vrf
   - CRITICAL: What happens DURING VRF_PENDING state? (partially documented - verify complete)

2. **Carnage State Machine** (Carnage_Fund_Spec.md)
   - States: IDLE, PENDING_EXECUTION, EXECUTED
   - Transitions: trigger, execute, expire
   - CRITICAL: What happens DURING PENDING_EXECUTION? Can other operations proceed?

3. **Bonding Curve State Machine** (Bonding_Curve_Spec.md)
   - States: NOT_STARTED, ACTIVE, FILLED, FAILED
   - Transitions: start, purchase, fill, expire, refund
   - Check: What happens during ACTIVE state edge cases?

For each state machine, verify documentation of:
- All possible states (complete enumeration)
- All transition triggers (what causes state change)
- All transition guards (what prevents invalid transitions)
- **"During wait" behaviors** (what other operations are allowed while in intermediate states)
- Timeout handling
- Failure modes and recovery

Focus especially on "during wait" behaviors per requirement GAP-03:
- Can users swap during VRF_PENDING? (Answer should be: Yes, old taxes apply)
- Can users swap during CARNAGE_PENDING? (What happens?)
- What transactions are blocked during intermediate states?

Log gaps with clear "State Machine:" prefix for easy Phase 5 filtering.

  <name>Task 3: CPI Depth and Compute Budget Deep-Dive (GAP-04)</name>
  <files>.planning/audit/GAPS.md</files>
  <action>
Map ALL Cross-Program Invocation chains and verify compute budget documentation.

CPI chains to trace:

1. **Swap Flow (Tax -> AMM -> Token Programs)**
```
Tax Program::swap
  └─> AMM Program::swap_internal (depth 1)
      └─> Token-2022::transfer_checked (depth 2)
          └─> Transfer Hook (depth 3)
      └─> SPL Token::transfer_checked (depth 2) [for WSOL]
```

2. **VRF Callback Flow (Switchboard -> Epoch -> Staking)**
```
Switchboard VRF::callback
  └─> Epoch Program::vrf_callback (depth 1)
      └─> Staking Program::update_cumulative (depth 2)
      └─> [If Carnage] inline execution
          └─> Tax Program::swap_exempt (depth 2)
              └─> AMM Program (depth 3)
                  └─> Token transfers (depth 4)
```

3. **Carnage Execution Flow**
```
Epoch Program::execute_carnage (or inline in vrf_callback)
  └─> Tax Program::swap_exempt (depth 1)
      └─> AMM Program (depth 2)
          └─> Token transfers (depth 3)
              └─> Transfer Hook (depth 4)
  └─> Token-2022::burn (depth 1) [no hook]
```

4. **Yield Claim Flow**
```
Staking Program::claim
  └─> System Program::transfer (depth 1) [SOL transfer]
```

For each CPI chain, verify documentation of:
- Maximum CPI depth (Solana limit is 4)
- Estimated compute units
- Whether compute budget documentation exists in specs
- Authority/signer requirements at each level

Log gaps for:
- CPI chains approaching depth 4 without explicit acknowledgment
- Missing compute budget estimates
- Missing authority patterns for CPI signing
- Unclear which program is the "top level" entry point

Use severity:
- CRITICAL: CPI depth > 4 possible OR security concern
- HIGH: Compute budget not estimated for complex flows
- MEDIUM: Authority patterns unclear
