# Phase 94.1: Launch Page Railway Hosting & Curve Target Recalibration - Research

**Researched:** 2026-03-13
**Domain:** Next.js middleware/proxy, Solana bonding curve constants, graduation scripting
**Confidence:** HIGH

## Summary

This phase has two distinct deliverables: (1) a Next.js proxy file that locks the site to the `/launch` route during the bonding curve period, and (2) recalibrating the mainnet bonding curve target from 1000 SOL to 500 SOL.

The Next.js middleware approach is straightforward but has a **critical version-specific change**: the project uses Next.js 16.1.6, which has **renamed `middleware.ts` to `proxy.ts`** and the exported function from `middleware` to `proxy`. The old `middleware.ts` convention is deprecated in v16. The proxy file goes at `app/proxy.ts` (same level as the `app/` directory) and uses the `NEXT_PUBLIC_SITE_MODE` env var to decide between launch-only and full-site modes.

The curve recalibration is a constants cascade: change P_START from 900 to 450, P_END from 3450 to 1725, and TARGET_SOL from 1000 SOL to 500 SOL across on-chain Rust, client-side TypeScript, tests, graduation script, and documentation. The math is verified: (450+1725)/2 * 460M / 1e6 = 500.25 SOL (same ~0.05% surplus as the 1000 SOL curve). The 3.83x price ratio is preserved exactly.

**Primary recommendation:** Use `proxy.ts` (not `middleware.ts`) for the site mode toggle, and update constants in dependency order: Rust first, then client TS, then tests, then docs.

## Architecture Patterns

### Pattern 1: Next.js 16 Proxy File (NOT middleware.ts)

**What:** Next.js 16.1.6 renamed `middleware.ts` to `proxy.ts` and the exported function from `middleware` to `proxy`. The old convention is deprecated.

**When to use:** For the site mode toggle.

**CRITICAL:** The file must be named `proxy.ts`, placed at `app/proxy.ts` (same level as the `app/` directory inside the `app/` workspace package), and export a function named `proxy`.

**Example:**
```typescript
// Source: https://nextjs.org/docs/app/api-reference/file-conventions/proxy (verified 2026-03-13)
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function proxy(request: NextRequest) {
  const siteMode = process.env.NEXT_PUBLIC_SITE_MODE || 'launch';
  const { pathname } = request.nextUrl;

  if (siteMode === 'launch') {
    // In launch mode, redirect everything to /launch
    // Except /launch itself, API routes, static files, images
    if (pathname !== '/launch' && !pathname.startsWith('/api') &&
        !pathname.startsWith('/_next')) {
      return NextResponse.redirect(new URL('/launch', request.url));
    }
  }

  // In live mode, /launch stays accessible (shows completed state)
  return NextResponse.next();
}

export const config = {
  matcher: [
    // Match all routes except static assets
    '/((?!_next/static|_next/image|favicon.ico|icon.png|apple-icon.png|public).*)',
  ],
};
```

**What:** On-chain constants use `#[cfg(feature = "devnet")]` compile-time flags. Client-side mirrors use `isDevnet` runtime check. Both must stay in sync.

**Update order (dependency chain):**

8. Documentation files

### Pattern 3: Graduation Script Dynamic Pool Seeding

**What:** Instead of a hardcoded `GRADUATION_POOL_SEED_SOL` constant, read the admin wallet's SOL balance before and after `withdrawGraduatedSol` and use the delta as the pool seed amount.

**Current bug:** `GRADUATION_POOL_SEED_SOL` defaults to `1_000 * LAMPORTS_PER_SOL` (1000 SOL). After recalibration, mainnet curves raise ~500 SOL, but the env override or default would be wrong.

**Fix pattern:**
```typescript
// Before withdrawal
const balanceBefore = await connection.getBalance(authority.publicKey);

// Execute withdrawal
await programs.bondingCurve.methods.withdrawGraduatedSol()...

// After withdrawal -- the delta IS the pool seed amount
const balanceAfter = await connection.getBalance(authority.publicKey);
const actualWithdrawn = balanceAfter - balanceBefore;
// Use actualWithdrawn as GRADUATION_POOL_SEED_SOL for this curve's pool
```

Note: Transaction fees (~5000 lamports) are negligible vs 500 SOL. The minor fee deduction is acceptable.

### Anti-Patterns to Avoid
- **Using middleware.ts instead of proxy.ts:** Next.js 16 deprecated middleware.ts. It may still work but generates deprecation warnings and may break in future versions.
- **Manually editing shared/constants.ts:** This file is auto-generated by generate-constants.ts. Edits will be overwritten.
- **Changing devnet constants:** CONTEXT.md decision locks devnet at 5 SOL / P_START=5 / P_END=17. Only localnet and mainnet (default) change.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Route redirection | Custom layout-level redirect logic | Next.js proxy.ts | Built-in, runs before rendering, no flash of wrong content |
| Env var toggle | Cookie-based or database-backed feature flag | NEXT_PUBLIC_SITE_MODE env var | Simple, Railway dashboard toggle, no code deploy needed |
| Vault balance reading | Parsing on-chain CurveState.sol_raised field | Read wallet balance delta around withdrawal TX | sol_raised is cumulative (includes sells), vault balance is actual withdrawable amount |

## Common Pitfalls

### Pitfall 1: middleware.ts vs proxy.ts (Next.js 16)
**What goes wrong:** Creating `middleware.ts` instead of `proxy.ts` -- deprecated in Next.js 16.1.6
**Why it happens:** Most tutorials and training data reference the old `middleware.ts` convention
**How to avoid:** File must be `app/proxy.ts` with exported function named `proxy`
**Warning signs:** Deprecation warning in console, or proxy not executing at all

### Pitfall 2: Proxy File Location
**What goes wrong:** Placing proxy.ts in wrong directory
**Why it happens:** The app uses a workspace structure with `app/` as the Next.js package root
**How to avoid:** proxy.ts goes at `app/proxy.ts` (same level as `app/app/`, `app/next.config.ts`)
**Warning signs:** Proxy function never executes, no redirect happens

### Pitfall 3: NEXT_PUBLIC_ Prefix in Proxy
**What goes wrong:** `NEXT_PUBLIC_SITE_MODE` is accessible in proxy because proxy runs server-side AND the NEXT_PUBLIC_ prefix makes it available in client bundles. However, in proxy.ts, you could also use a non-NEXT_PUBLIC env var since proxy runs server-side only.
**Why it happens:** Confusion about where proxy runs
**How to avoid:** Using `NEXT_PUBLIC_SITE_MODE` is fine -- it works in proxy AND is useful for client-side conditional rendering of the "graduated" banner on `/launch`
**Warning signs:** None, but be aware the value is exposed to the client bundle

### Pitfall 4: Hardcoded 1000 SOL in math.rs Tests
**What goes wrong:** Changing TARGET_SOL constant but leaving hardcoded `1_000_000_000_000` values in math.rs proptests and unit tests
**Why it happens:** Some test assertions use literal values instead of the TARGET_SOL constant
**How to avoid:** Search for `1_000_000_000_000` in math.rs -- there are at least 6 hardcoded occurrences in refund proptests (lines 1278, 1289, 1350, 1456, 1509, 1559) plus comments
**Warning signs:** Proptests generate values up to 1000 SOL instead of 500 SOL, making vault_sol ranges unrealistic

### Pitfall 5: generate-constants.ts Hardcoded Value
**What goes wrong:** The generate-constants.ts script has `CURVE_TARGET_SOL = 1_000_000_000_000` hardcoded at line 261
**Why it happens:** The script doesn't read from on-chain constants, it has its own hardcoded value
**How to avoid:** Update generate-constants.ts AND re-run it to regenerate shared/constants.ts
**Warning signs:** shared/constants.ts still shows 1000 SOL after on-chain constants are updated

### Pitfall 6: Graduation Script Default SOL Amount
**What goes wrong:** `GRADUATION_POOL_SEED_SOL` defaults to `1_000 * LAMPORTS_PER_SOL` if no env override
**Why it happens:** Hardcoded fallback from Phase 69 lesson (preventing test defaults)
**How to avoid:** Replace with dynamic vault balance reading (the CONTEXT.md decision)
**Warning signs:** Pool seeded with wrong SOL amount, creating incorrect initial price

### Pitfall 7: Redirect Loop on /launch
**What goes wrong:** Proxy redirects `/launch` to `/launch` infinitely
**Why it happens:** Matcher catches `/launch` path and redirects it to itself
**How to avoid:** Exclude `/launch` from redirect logic when in launch mode
**Warning signs:** ERR_TOO_MANY_REDIRECTS in browser

### Pitfall 8: Static Assets Blocked by Proxy
**What goes wrong:** Images, CSS, JS bundles blocked by the redirect
**Why it happens:** Proxy matcher is too broad
**How to avoid:** Use negative lookahead in matcher to exclude `_next/static`, `_next/image`, `favicon.ico`, `icon.png`, `apple-icon.png`, and image/scene files in `public/`
**Warning signs:** Launch page renders without styles, images, or JavaScript

import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function proxy(request: NextRequest) {
  const siteMode = process.env.NEXT_PUBLIC_SITE_MODE || 'launch';
