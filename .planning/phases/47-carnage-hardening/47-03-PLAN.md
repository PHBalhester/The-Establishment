<objective>
Bundle consume_randomness + executeCarnageAtomic in the same Solana transaction when Carnage triggers, closing the 85-slot MEV window (FIX-02/CARN-002). Update carnage-flow.ts to work with the new atomic bundling.

Purpose: Currently, vrf-flow.ts sends reveal+consume as TX3, then carnage-flow.ts sends executeCarnageAtomic as a separate TX (~85 slots later on devnet). This gap allows MEV bots to see the CarnagePending event and sandwich the swap. By bundling all three instructions (reveal, consume_randomness, executeCarnageAtomic) in one VersionedTransaction with ALT compression, the swap executes atomically -- no instructions from other transactions can interleave.

Output: Modified vrf-flow.ts with Carnage-aware TX bundling, updated carnage-flow.ts exports, and a shared instruction builder.
</objective>

    1. Send reveal+consume as before (TX3)
    2. After TX3 confirms, read EpochState to check carnage_pending
    3. If Carnage triggered: immediately build and send executeCarnageAtomic as TX4 (still within the 50-slot lock window)

    WHY NOT bundle all in one TX: The client doesn't know the VRF result before submission. If we always include executeCarnageAtomic and Carnage doesn't trigger, the entire TX fails (executeCarnageAtomic checks carnage_pending=true). Option C is safer -- the 50-slot lock window prevents MEV front-running because no one else can call the fallback path during that window. The atomic path (executeCarnageAtomic) IS permissionless, but speed + lock window provide protection.
