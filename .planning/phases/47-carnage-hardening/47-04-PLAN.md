<objective>
Replace the two-sequential-TX approach (Option C) with true single-TX atomic bundling. The on-chain change makes executeCarnageAtomic gracefully no-op when Carnage doesn't trigger, allowing it to be safely included in every reveal+consume TX. The client change bundles all three instructions (reveal, consume_randomness, executeCarnageAtomic) in one VersionedTransaction v0 with ALT compression.

Why: Option C exposes a MEV window between TX3 (consume) and TX4 (execute). True bundling eliminates this entirely -- no CarnagePending event is visible on-chain until the swap has already executed.
</objective>

    2. **Add a no-op early return** at the top of the handler function (line ~202-203), before any state reads:

       ```rust
       // No-op guard: If Carnage wasn't triggered this epoch, return immediately.
       // This allows executeCarnageAtomic to be safely bundled in every
       // reveal+consume TX without knowing the VRF result beforehand.
       // When Carnage doesn't trigger, this is a harmless no-op.
       // When Carnage triggers, execution proceeds normally.

           return Ok(());
       }
       ```

    WHY this is safe:
    - Account validation still runs (PDA seeds, bumps, initialized checks)
    - No state mutation occurs on the no-op path
    - No events emitted on the no-op path
    - Compute cost is minimal (~100 CU for the account validation + early return)
    - The instruction remains permissionless -- the only behavioral change is it returns success instead of error when not pending

    Update the doc comments:
    - Change "can be called by anyone when carnage_pending = true" to "can be called by anyone; no-ops when carnage_pending is false"
