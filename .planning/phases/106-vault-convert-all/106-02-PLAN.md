<objective>
Update all client-side swap builders to use `convert_v2` instead of `convert`. Multi-hop vault steps use convert-all mode (amount_in=0), direct vault converts use exact amounts. Add Vault error codes to the error map.

Purpose: This wires the frontend to the new on-chain instruction. Multi-hop vault steps pass amount_in=0 so the vault reads the user's actual on-chain balance (whatever the AMM deposited), eliminating intermediate token leakage. Direct vault converts pass exact amounts with tight slippage for user safety.

Output: All client swap paths use convert_v2, error map includes vault errors.
</objective>

    **multi-hop-builder.ts changes:**

    1. In `buildStepTransaction`, find the vault conversion section (around line 133, the `if (step.pool.includes("Vault"))` block). Currently it passes `step.inputAmount` as `amountInBaseUnits`. Change this:

       For multi-hop routes (route has multiple steps), vault steps should use convert-all mode:
       ```typescript
       if (step.pool.includes("Vault")) {
         const inputMint = step.inputToken === "PROFIT" ? MINTS.PROFIT : (isCrime ? MINTS.CRIME : MINTS.FRAUD);
         const outputMint = step.outputToken === "PROFIT" ? MINTS.PROFIT : (isCrime ? MINTS.CRIME : MINTS.FRAUD);

         return buildVaultConvertTransaction({
           connection,
           userPublicKey,
           amountInBaseUnits: step.inputAmount,  // <-- This line changes
           minimumOutput,
           inputMint,
           outputMint,
           priorityFeeMicroLamports,
         });
       }
       ```

       The `buildStepTransaction` function is called from `buildAtomicRoute`. The caller already computes `minimumOutput` per step. The key insight: we do NOT need to change `amountInBaseUnits` to 0 here because the multi-hop builder's `buildAtomicRoute` already passes step.inputAmount for all steps, and the vault's convert-all mode is triggered by amount_in=0.

       However, per the CONTEXT.md decision: "Multi-hop intermediate steps: `convert_v2(0, minimum_derived_from_amm_min)` -- convert-all mode". So we DO need to detect when a vault step is an intermediate step in a multi-hop route.

       The `buildStepTransaction` function doesn't currently know if it's multi-hop. We have two options:
       - Pass a flag to indicate multi-hop
       - Check at the `buildAtomicRoute` level

       Cleanest approach: Add an optional `isMultiHopStep` parameter to `buildStepTransaction`. Set it to `true` when `route.steps.length > 1` in `buildAtomicRoute`. In the vault section:
       ```typescript
       const effectiveAmountIn = isMultiHopStep ? 0 : step.inputAmount;
       ```

       This way:
       - 1-hop direct vault convert: passes exact amount (same as before, but through convertV2)
       - 2-hop or 4-step multi-hop vault steps: passes 0 (convert-all mode)

    2. In `buildAtomicRoute` (around line 360), where `buildStepTransaction` is called, pass the multi-hop flag:
       ```typescript
       const isMultiHop = route.steps.length > 1;
       const tx = await buildStepTransaction(
         effectiveStep,
         connection,
         userPublicKey,
         minimumOutput,
         priorityFeeMicroLamports,
         isMultiHop,  // new parameter
       );
       ```

    3. **Multi-hop minimumOutput derivation**: The current code in `buildAtomicRoute` already computes per-step minimumOutput as `Math.floor(step.outputAmount * (10_000 - slippageBps) / 10_000)`. For vault steps this is correct -- the vault rate is deterministic, so the quoted output IS the exact output (zero price impact). The slippage adjustment provides a safety margin for the AMM steps that precede it. No changes needed to the minimumOutput derivation.

    4. **Remove the "Known tradeoff" comment** in buildAtomicRoute (around line 323-324): "Known tradeoff: ~slippage% of intermediate tokens (CRIME/FRAUD) may remain unconverted..." This tradeoff is eliminated by convert-all mode.
