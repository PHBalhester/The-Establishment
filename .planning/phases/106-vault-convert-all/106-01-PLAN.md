<objective>
Add the `convert_v2` on-chain instruction to the Conversion Vault program with convert-all sentinel mode and on-chain slippage protection. Extend the BOK proptest suite to verify convert_v2 math properties.

Purpose: This is the core on-chain change that enables users to convert their full token balance in a single instruction without client-side amount prediction. The sentinel value (amount_in=0) reads the user's actual on-chain balance, eliminating intermediate token leakage in multi-hop swaps. The minimum_output parameter adds on-chain slippage protection. The existing `convert` instruction is NOT modified.

Output: Working `convert_v2` instruction that builds successfully, new error variants, and extended proptest suite.
</objective>

    1. **error.rs** -- APPEND two new variants at the END of the VaultError enum (after MathOverflow). Order matters for error code stability:
       - `SlippageExceeded` with msg "Output below minimum -- slippage protection" (will be error code 6006)
       - `InvalidOwner` with msg "Input account not owned by signer" (will be error code 6007)
       Do NOT insert these between existing variants -- that shifts error codes and breaks clients.

    2. **instructions/convert_v2.rs** -- Create new file. Import `Convert` struct and `compute_output`/`compute_output_with_mints` from `convert.rs`. The handler:
       - Owner check: `require!(ctx.accounts.user_input_account.owner == ctx.accounts.user.key(), VaultError::InvalidOwner)` -- CRITICAL for convert-all safety, prevents draining another user's account
       - Sentinel resolution: if `amount_in == 0`, read `ctx.accounts.user_input_account.amount` as effective_amount, require balance > 0 (VaultError::ZeroAmount). If `amount_in > 0`, use amount_in directly.
       - Compute output: same cfg-gated pattern as convert.rs (localnet uses compute_output_with_mints from VaultConfig state, production uses compute_output with hardcoded mints)
       - Slippage guard: `require!(amount_out >= minimum_output, VaultError::SlippageExceeded)`
       - Log: `msg!("convert_v2: effective_amount={}, output={}", effective_amount, amount_out)`
       - Transfers: identical to convert.rs -- split remaining_accounts at midpoint, user->vault (user-signed), vault->user (PDA-signed via vault_config bump + VAULT_CONFIG_SEED). Use helpers::hook_helper::transfer_t22_checked for both.

       See 106-RESEARCH.md "convert_v2 Handler (Complete)" for the full reference implementation.

    3. **instructions/mod.rs** -- Add `pub mod convert_v2;` and `pub use convert_v2::*;` after the existing convert entries.

    4. **lib.rs** -- Register `convert_v2` in the `#[program]` module. Same generic lifetime pattern as existing `convert`:
       ```rust
       pub fn convert_v2<'info>(
           ctx: Context<'_, '_, 'info, 'info, Convert<'info>>,
           amount_in: u64,
           minimum_output: u64,
       ) -> Result<()> {
           instructions::convert_v2::handler(ctx, amount_in, minimum_output)
       }
       ```
       Keep existing `convert` function EXACTLY as-is. Do not add a deprecation log or any modification.
