  <action>
    Run a targeted security audit of the convert_v2 diff using the SOS (Stronghold of Security) methodology. Scope is ONLY the changed files:

    **Files to audit:**
    - `programs/conversion-vault/src/instructions/convert_v2.rs` (new file -- full review)
    - `programs/conversion-vault/src/error.rs` (diff: two new variants appended)
    - `programs/conversion-vault/src/lib.rs` (diff: convert_v2 registration)
    - `programs/conversion-vault/src/instructions/mod.rs` (diff: new module export)

    **Cross-reference against:**
    - `programs/conversion-vault/src/instructions/convert.rs` (the proven, battle-tested handler)

    **Audit checklist (minimum):**

    1. **Owner check correctness**: Verify `user_input_account.owner == user.key()` prevents an attacker from passing another user's token account. Confirm this check happens BEFORE balance reading. Confirm that the existing `convert` instruction doesn't need this check (it passes exact amount, so the transfer itself enforces authority).

    2. **Sentinel safety**: Verify that `amount_in == 0` sentinel cannot be exploited. Check: what happens if user has 0 balance and passes amount_in=0? (Should get ZeroAmount error.) What if user has dust amount < 100? (Should get OutputTooSmall for faction->PROFIT, or succeed for PROFIT->faction.)

    3. **Transfer equivalence**: Line-by-line compare transfer logic in convert_v2 vs convert. Must be identical: same split_at(mid), same hook account ordering, same signer seeds, same token program, same TOKEN_DECIMALS. Any divergence is a finding.

    4. **Error code stability**: Verify SlippageExceeded and InvalidOwner are APPENDED after MathOverflow (not inserted). Confirm error codes 6000-6005 are unchanged.

    5. **Slippage guard correctness**: Verify `amount_out >= minimum_output`. Check: minimum_output=0 effectively disables the guard (this is by design per CONTEXT.md). Check: can minimum_output overflow? (No -- it's u64, compared to u64 output.)

    6. **cfg feature parity**: Verify convert_v2 has the same `#[cfg(feature = "localnet")]` / `#[cfg(not(feature = "localnet"))]` pattern as convert for compute_output calls.

    7. **No new account constraints on Convert struct**: Verify the shared `Convert<'info>` struct was NOT modified. Any new constraint there would change convert behavior too.

    8. **Instruction discriminator uniqueness**: Anchor generates different discriminators for `convert` vs `convert_v2`. No collision risk -- just document for completeness.

    **Output**: Write audit report to `.audit/findings/VAULT-CONVERT-V2.md` with:
    - Scope and methodology
    - Per-check findings (pass/fail/note)
    - Any issues found with severity and remediation
    - Final verdict: CLEARED or BLOCKED (with reason)
  </action>
