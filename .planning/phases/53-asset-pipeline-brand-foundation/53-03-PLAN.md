  <action>
    **Create `app/components/scene/FactoryBackground.tsx`:**

    A client component ('use client') that renders the factory scene background image with progressive loading:

    1. Imports `Image` from `next/image` and `SCENE_DATA` from `@/lib/image-data`
    2. Renders a full-viewport container (`relative w-full h-screen`) with `bg-factory-bg` as the immediate background color (prevents white flash)
    3. Uses `<Image>` with:
       - `src={SCENE_DATA.background.src}` (the 1920w WebP)
       - `alt="Dr. Fraudsworth's Finance Factory"` with `role="img"` on the container (for A11Y-04)
       - `fill` mode with `style={{ objectFit: 'cover' }}` for full-viewport coverage
       - `sizes="100vw"` to let Next.js select the right responsive size
       - `quality={80}` matching our next.config.ts qualities array
       - `placeholder="blur"` with `blurDataURL={SCENE_DATA.background.blurDataURL}`
       - `preload` (NOT `priority` -- deprecated in Next.js 16) since this is the LCP element
       - `className="z-background"` using the named z-index token
    4. The container div has `aria-label="Factory scene depicting Dr. Fraudsworth's steampunk finance factory"` and `role="img"` for accessibility
    5. Accepts an optional `children` prop (React.ReactNode) to allow overlays and other elements to be positioned within the scene container

    **Loading sequence happens automatically via Next.js Image:**
    - bg-factory-bg shows immediately (dark warm brown, no white)
    - blurDataURL appears as tiny blurred preview
    - Full image sharpens when loaded

    **Create `app/components/scene/FactoryOverlay.tsx`:**

    A client component that renders a single positioned overlay image:

    1. Props interface:
       ```typescript
       interface FactoryOverlayProps {
         overlayId: string;  // key into SCENE_DATA.overlays (e.g., 'carnage-cauldron')
         className?: string; // additional classes for hover effects, animations, etc.
         children?: React.ReactNode; // for tooltips, hover effects layered on top
       }
       ```

    2. Reads overlay metadata from `SCENE_DATA.overlays[overlayId]`
    3. If `overlay.available === false`, render nothing (return null) -- this handles the missing swap-station
    4. Renders an absolutely-positioned container using percentage-based coordinates from metadata:
       ```tsx
       <div
         className={`absolute z-overlays ${className ?? ''}`}
         style={{
           left: `${overlay.left}%`,
           top: `${overlay.top}%`,
           width: `${overlay.widthPct}%`,
           height: `${overlay.heightPct}%`,
         }}
       >
       ```
    5. Inside the container, renders `<Image>` with:
       - `src={overlay.src}`
       - `alt` derived from overlayId (convert kebab-case to Title Case, e.g., "Carnage Cauldron")
       - `fill` mode with `style={{ objectFit: 'contain' }}` (overlay must fit within its bounding box)
       - `quality={82}` matching our overlay compression quality
       - `placeholder="blur"` with `blurDataURL={overlay.blurDataURL}`
       - `loading="lazy"` (overlays are not LCP)
    6. Renders `{children}` inside the container after the Image (for future hover/tooltip layers)

    **Important implementation notes:**
    - Both components must be 'use client' (they use Next.js Image which needs client-side hydration for lazy loading)
    - Use the z-index tokens from @theme: `z-background` for background, `z-overlays` for overlays
    - Do NOT add hover effects or click handlers -- those come in Phase 55
    - Do NOT render all overlays in a loop here -- this is a single-overlay component. The scene assembly (Phase 55) will map over overlay IDs
    - The FactoryBackground component is a CONTAINER that other things go inside (overlays, future animations, etc.)
  </action>

  <action>
    Create `app/components/scene/LoadingSpinner.tsx` -- a CSS-only steampunk gear spinner:

    1. A client component ('use client') that renders a small animated gear/cog
    2. The gear is built entirely with CSS (no SVG files, no images, no external dependencies):
       - A circular base with `border-factory-border` border
       - 6-8 "teeth" created with CSS pseudo-elements or rotated child divs
       - `animate-gear-spin` class for continuous rotation (defined in @theme)
       - Styled with steampunk colors: `bg-factory-secondary`, `border-factory-accent`
       - Size: 40-48px diameter (small, unobtrusive)

    3. Props interface:
       ```typescript
       interface LoadingSpinnerProps {
         size?: 'sm' | 'md' | 'lg';  // 32px, 48px, 64px
         className?: string;
       }
       ```

    4. Uses `z-spinner` from the z-index system (highest layer -- should show above everything during load)

    5. An alternative simpler approach (if pure CSS gear is too complex): use a ring/circle with notches that rotates, resembling a gear. The visual quality matters less than it existing and rotating smoothly. A bordered circle with dashed outline that spins is acceptable if a detailed gear is too complex for CSS.

    6. The gear respects `prefers-reduced-motion` automatically via the media query in globals.css (animations disabled).

    **Important:** This is a reusable spinner, not a page-level loading state. It will be used:
    - During initial page load (Phase 55 scene assembly)
    - As fallback while images retry on error (Phase 55)
    - Potentially within modals for loading states (Phase 56)

    Keep it simple. A well-styled rotating element is better than an overengineered gear that is fragile.
