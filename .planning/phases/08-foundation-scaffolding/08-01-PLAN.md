  <name>Task 1: Fork, strip, and set up Anchor 0.32.1 workspace</name>
  <files>

  <action>
**Why this task exists:** The fork (arrayappy/solana-uniswap-v2) provides an Anchor project skeleton but uses Anchor 0.29.0, SPL Token only, and external math crates (`fixed`, `fixed-sqrt`). We need to extract the structural pattern, upgrade to 0.32.1, and configure for our T22/SPL mixed-token project. Starting from a fork saves boilerplate; stripping it prevents inheriting bugs (the fork has a k-invariant bug and no T22 support).

**Step-by-step:**

1. **Clone the fork** into a temporary directory:
   ```bash
   git clone https://github.com/arrayappy/solana-uniswap-v2.git /tmp/solana-uniswap-v2-fork
   ```

2. **Extract the skeleton** -- copy the Anchor directory structure pattern into our project root. We need:
   - The `programs/` directory structure (but NOT the source code inside)
   - The `Anchor.toml` pattern (but rewritten for our config)
   - The `Cargo.toml` workspace pattern (but rewritten for 0.32.1)
   - Note the fork's Apache-2.0 license for attribution

3. **Create workspace root `Cargo.toml`** with these exact contents:
   ```toml
   [workspace]
   members = ["programs/*"]
   resolver = "2"

   [profile.release]
   overflow-checks = true
   lto = "fat"
   codegen-units = 1

   [profile.release.build-override]
   opt-level = 3
   incremental = false
   ```
   **Why `overflow-checks = true`:** This is a safety net -- if any arithmetic accidentally uses `+` instead of `checked_add`, the program panics instead of silently wrapping. Defense in depth for financial math (MATH-04).
   **Why `resolver = "2"`:** Required for modern Cargo workspaces; ensures feature unification works correctly with anchor-spl's multiple features.

4. **Create `programs/amm/Cargo.toml`** with these exact contents:
   ```toml
   [package]
   name = "amm"
   version = "0.1.0"
   description = "Dr Fraudsworth AMM Program - Constant-product swap pools"
   edition = "2021"

   [lib]
   crate-type = ["cdylib", "lib"]

   [dependencies]
   anchor-lang = { version = "0.32.1", features = ["init-if-needed"] }
   anchor-spl = { version = "0.32.1", features = ["token", "token_2022", "associated_token"] }

   [dev-dependencies]
   proptest = "1.9"

   [features]
   default = []
   cpi = ["no-entrypoint"]
   no-entrypoint = []
   no-idl = []
   no-log-ix-name = []
   idl-build = ["anchor-lang/idl-build", "anchor-spl/idl-build"]
   ```
   **Why these anchor-spl features:** `token` for SPL Token (WSOL), `token_2022` for T22 tokens (CRIME/FRAUD/PROFIT), `associated_token` for ATA derivation. All three needed for mixed pool support (SCAF-02).
   **Why `init-if-needed`:** Required for pool initialization patterns in Phase 9.
   **Why `crate-type = ["cdylib", "lib"]`:** `cdylib` produces the .so for on-chain deployment; `lib` allows `cargo test` to run unit tests without the BPF target.
   **Why proptest in dev-dependencies only:** proptest is huge (~50 crates in dep tree). It must NEVER be in program dependencies -- only dev-dependencies. This keeps the on-chain binary small.

5. **Create `Anchor.toml`:**
   ```toml
   [toolchain]

   [features]
   resolution = true
   skip-lint = false

   [programs.localnet]
   amm = "AMM_PROGRAM_ID_PLACEHOLDER"

   [registry]
   url = "https://api.apr.dev"

   [provider]
   cluster = "Localnet"
   wallet = "keypairs/devnet-wallet.json"

   [scripts]
   test = "yarn run ts-mocha -p ./tsconfig.json -t 1000000 tests/**/*.ts"
   ```
   **Why `resolution = true`:** Anchor 0.32.1 feature -- resolves token program accounts automatically, reducing boilerplate in instruction accounts.
   **Note:** The program ID placeholder will be replaced after the first `anchor build` generates a keypair, OR we generate one with `solana-keygen grind`. The exact ID doesn't matter for localnet.

6. **After first `anchor build`**, update the program ID:
   - `anchor build` generates `target/deploy/amm-keypair.json`
   - Extract the public key: `solana address -k target/deploy/amm-keypair.json`
   - Update `declare_id!("...")` in `lib.rs` and `amm = "..."` in `Anchor.toml`
   - Run `anchor build` again to confirm the ID matches

7. **Create `tests/.gitkeep`** -- empty file to preserve the tests directory for future integration tests (Phases 9+).

8. **Clean up** -- remove the temporary fork clone: `rm -rf /tmp/solana-uniswap-v2-fork`

**What NOT to carry from the fork:**
- Do NOT copy any `.rs` source files (we write from scratch per our specs)
- Do NOT copy any test files (clean slate per CONTEXT.md)
- Do NOT copy `Xargo.toml` (deprecated, not needed for modern Anchor)
- Do NOT copy `fixed` or `fixed-sqrt` dependencies (we use u128 checked arithmetic)
- Do NOT copy the fork's `migrations/` directory

**Why this task exists:** The file structure must match exactly what the research doc specifies (08-RESEARCH.md Architecture Patterns). Each module has a specific role, and getting the module tree right now prevents restructuring later. Empty scaffolds let `anchor build` succeed while Phase 08-02 and Phases 9-13 fill in the logic.

**Step-by-step:**

1. **Create `programs/amm/src/lib.rs`** -- the program entrypoint:
   ```rust
   use anchor_lang::prelude::*;

   pub mod constants;
   pub mod errors;
   pub mod events;
   pub mod helpers;
   pub mod instructions;
   pub mod state;

   declare_id!("PROGRAM_ID_FROM_TASK_1");

   #[program]
   pub mod amm {
       use super::*;

       // Instructions will be added in Phases 9-13
   }
   ```
   **Why `pub mod` for all modules:** Later phases need to import types across modules (e.g., instructions need state::PoolState, helpers::math functions). Public visibility prevents visibility refactoring later.
   **Why the empty `#[program]` block:** Anchor requires at least the `#[program]` attribute to generate the IDL and entrypoint. An empty program module is valid and compiles.

2. **Create `programs/amm/src/constants.rs`:**
   ```rust
   /// LP fee for SOL pools (CRIME/SOL, FRAUD/SOL) in basis points.
   /// 100 bps = 1.0% fee per swap.
   /// Source: AMM_Implementation.md Section 6
   pub const SOL_POOL_FEE_BPS: u16 = 100;

   /// LP fee for PROFIT pools (CRIME/PROFIT, FRAUD/PROFIT) in basis points.
   /// 50 bps = 0.5% fee per swap.
   /// Source: AMM_Implementation.md Section 6
   pub const PROFIT_POOL_FEE_BPS: u16 = 50;

   /// Basis points denominator (10,000 = 100%).
   pub const BPS_DENOMINATOR: u128 = 10_000;

   // PDA seeds will be added in Phase 9 (pool initialization)
   ```
   **Why define fee constants now:** The math module (Phase 08-02) tests against these exact fee values. Having them in constants.rs means tests and future instruction code reference the same source of truth.
   **Why reference the spec section:** Every constant traces back to a specification section. This is audit-friendly and prevents "where did this number come from?" questions.

3. **Create `programs/amm/src/errors.rs`:**
   ```rust
   use anchor_lang::prelude::*;

   #[error_code]
   pub enum AmmError {
       /// Arithmetic overflow in swap calculation.
       /// The math module returns None on overflow; this error is the on-chain mapping.
       #[msg("Arithmetic overflow in swap calculation")]
       Overflow,

       /// The k-invariant (reserve_a * reserve_b) decreased after swap.
       /// This indicates a bug in swap logic -- k must never decrease.
       #[msg("K-invariant violation: k decreased after swap")]
       KInvariantViolation,

       // Additional errors will be added in Phases 9-13
       // (SlippageExceeded, InvalidPoolType, Unauthorized, etc.)
   }
   ```
   **Why Overflow and KInvariantViolation now:** These are the two error conditions the math module can produce (MATH-04). Instruction handlers map `None` from math functions to `AmmError::Overflow`, and `Some(false)` from `verify_k_invariant` to `AmmError::KInvariantViolation`. Defining them now documents the error contract.

4. **Create `programs/amm/src/events.rs`:**

   ```rust
   // Empty module for now to satisfy the mod declaration in lib.rs.
   ```

6. **Create `programs/amm/src/helpers/math.rs`:**

   ```rust
   //!
   //! Implementation will be added in Phase 08-02.
   ```
   **Why the doc comment is detailed:** This module is the security-critical core. The comment explains WHY it's pure (testability), WHAT it contains (3 functions), and WHERE it's specified (requirement IDs). Future contributors (or future Claude sessions) can understand the design intent without reading planning docs.
