Add `PARTIAL_DEPLOY` env var check at the top of initialize.ts. When `PARTIAL_DEPLOY=true`:
- Execute: Step 1 (CRIME mint creation), Step 2 (FRAUD mint creation) -- with Arweave metadata URIs from env vars
- Skip: Step 3 (PROFIT mint creation), Step 4-5 (any mint steps for PROFIT)

The simplest approach: wrap each step's execution in a check like:
```typescript
if (!isPartial || PARTIAL_STEPS.includes(stepNumber)) {
```

**Create scripts/test/ directory if it doesn't exist.**

**pathway1-test.ts:**

A self-contained test script that:

1. **Setup phase:**
   - Reads deployment.json for program IDs, mint addresses, curve PDAs
   - Loads devnet wallet keypair from WALLET env var
   - Generates 5 fresh keypairs for test wallets
   - Funds each test wallet with SOL from devnet wallet (0.5 SOL each -- enough for buys + TX fees)
   - Creates Token-2022 ATAs for CRIME and FRAUD on each test wallet (init_if_needed in purchase IX handles this, but pre-creating avoids issues)

2. **Buy phase:**
   - Wallet 1: Buy CRIME with 0.003 SOL (minimum buy test)
   - Wallet 2: Buy CRIME with 0.05 SOL (medium buy)
   - Wallet 3: Buy FRAUD with 0.1 SOL (larger buy)
   - Wallet 4: Buy CRIME with 0.02 SOL AND buy FRAUD with 0.02 SOL (both curves)
   - Wallet 5: Buy FRAUD with 0.05 SOL (for sell-then-refund test)
   - Log each buy: {wallet, curve (CRIME/FRAUD), action: "buy", solAmount, txSig, slot}

3. **Sell phase (partial sells to test tax escrow inclusion):**
   - Wallet 5: Sell ~50% of FRAUD tokens back (exercises sell tax escrow)
   - Log each sell: {wallet, curve, action: "sell", tokenAmount, solReceived, txSig, slot}

4. **Wait phase:**
   - Read deadline_slot from CurveState PDA
   - Poll current slot every 10 seconds
   - Print countdown: "Waiting for deadline... X slots remaining (~Ys)"
   - Wait until current_slot > deadline_slot + FAILURE_GRACE_SLOTS (150) + 10 (safety margin)

5. **Mark failed phase:**
   - Call mark_failed on CRIME curve (permissionless)
   - Call mark_failed on FRAUD curve (permissionless)
   - Verify both CurveState.status == Failed

6. **Consolidate phase:**
   - Call consolidate_for_refund on CRIME curve
   - Call consolidate_for_refund on FRAUD curve
   - Read and log refund_pool_lamports and total_outstanding_tokens from each CurveState after consolidation

7. **Snapshot phase (pre-claim):**
   - For each wallet: query SOL balance and token balance for CRIME and FRAUD
   - Log pre-claim state

8. **Claim phase:**
   - For each wallet that holds tokens: call claim_refund
   - Partner curve: CRIME's partner is FRAUD and vice versa (from CurveState.partner_mint)
   - For wallets holding tokens on both curves: claim from each curve separately
   - Log post-claim SOL balance for each wallet

9. **Output:**
   - Write structured JSON to scripts/test/pathway1-log.json with all actions, balances, and states

**Transaction building:** Use Anchor program client (Program<BondingCurve>) for mark_failed, consolidate_for_refund, claim_refund. For purchase and sell, use the existing `buildPurchaseInstruction` and `buildSellInstruction` from app/lib/curve/curve-tx-builder.ts if importable, or build raw Anchor instructions. The program IDL is at target/idl/bonding_curve.json.

**Transfer Hook accounts:** Purchase and sell transactions require Transfer Hook remaining_accounts. Use the pattern from existing test code -- derive ExtraAccountMetaList PDA, whitelist source/dest PDAs, and hook program ID as the 4 extra accounts per mint.

**Error handling:** Wrap each transaction in try/catch. Log failures but continue with other wallets. A wallet failing to buy shouldn't abort the whole test.

**verify-refunds.ts:**

A verification script that:

1. Reads scripts/test/pathway1-log.json
2. For each wallet, calculates expected refund per curve:
   - `expected_refund = floor(wallet_token_balance * refund_pool_lamports / total_outstanding_tokens)`
   - refund_pool_lamports and total_outstanding_tokens come from the post-consolidation CurveState snapshot in the log
3. Compares expected refund to actual SOL received (post_claim_balance - pre_claim_balance - rent_refund_if_any)
4. Generates Docs/pathway1-report.md with:
   - Header: Phase 94 Pathway 1 Test Report, date, deploy addresses
   - Table of all wallet actions (buys/sells with amounts)
   - Per-wallet refund verification table: wallet address, curve, token balance, expected refund (lamports), actual refund (lamports), delta, PASS/FAIL
   - Allow 1 lamport tolerance for rounding
   - Summary: X/Y wallets passed, total SOL refunded, test result
  </action>

3. Both scripts have proper imports and reference deployment.json for addresses
  </verify>
