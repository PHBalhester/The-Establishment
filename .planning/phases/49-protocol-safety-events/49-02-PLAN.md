    Both instructions follow the same pattern. The helpers (read_pool_reserves, calculate_output_floor) and constants (MINIMUM_OUTPUT_FLOOR_BPS) were created in Plan 01.

    **swap_profit_buy.rs changes (CRIME/FRAUD -> PROFIT):**

    1. Add imports at top:
       ```rust
       use crate::helpers::pool_reader::read_pool_reserves;
       use crate::helpers::tax_math::calculate_output_floor;
       use crate::constants::MINIMUM_OUTPUT_FLOOR_BPS;
       ```

    2. BEFORE the AMM CPI (after building swap_authority_seeds, before building instruction data), add floor enforcement:
       ```rust
       // =========================================================================
       // 1b. Enforce protocol minimum output floor (SEC-10)
       // =========================================================================
       // For buy (AtoB): reserve_in = reserve_a (CRIME/FRAUD), reserve_out = reserve_b (PROFIT).
       let (reserve_a, reserve_b) = read_pool_reserves(&ctx.accounts.pool)?;
       let output_floor = calculate_output_floor(reserve_a, reserve_b, amount_in, MINIMUM_OUTPUT_FLOOR_BPS)
           .ok_or(error!(TaxError::TaxOverflow))?;
       require!(
           minimum_output >= output_floor,
           TaxError::MinimumOutputFloorViolation
       );
       ```
       - WHY no tax adjustment: PROFIT pool swaps are untaxed. amount_in goes directly to the AMM. No adjustment needed (unlike swap_sol_buy where sol_to_swap is used).

    3. BEFORE the AMM CPI, snapshot output token balance:
       ```rust
       // =========================================================================
       // 1c. Snapshot output token balance for event (FIX-06)
       // =========================================================================
       let output_before = ctx.accounts.user_token_b.amount;
       ```

    4. AFTER the AMM CPI (invoke_signed), reload and compute output + LP fee:
       ```rust
       // =========================================================================
       // 3b. Compute actual output and LP fee (FIX-06)
       // =========================================================================
       ctx.accounts.user_token_b.reload()?;
       let actual_output = ctx.accounts.user_token_b.amount
           .checked_sub(output_before)
           .ok_or(error!(TaxError::TaxOverflow))?;

       // Read LP fee BPS from pool data at byte offset 153-155
       let pool_data = ctx.accounts.pool.try_borrow_data()?;
       let lp_fee_bps = u16::from_le_bytes(
           pool_data[153..155].try_into()
               .map_err(|_| error!(TaxError::TaxOverflow))?
       );
       drop(pool_data); // Release borrow before emit
       let lp_fee = (amount_in as u128)
           .checked_mul(lp_fee_bps as u128)
           .and_then(|n| n.checked_div(10_000))
           .ok_or(error!(TaxError::TaxOverflow))? as u64;
       ```
       - WHY read lp_fee_bps from pool bytes: The Tax Program doesn't import AMM types. Reading at offset 153-155 follows the same pattern as read_pool_reserves (known PoolState byte layout). LP fee is amount_in * lp_fee_bps / 10_000 (the AMM deducts fee from input before constant-product calc).

    5. In the emit! block, replace `output_amount: 0` with `output_amount: actual_output` and `lp_fee: 0` with `lp_fee`

    **swap_profit_sell.rs changes (PROFIT -> CRIME/FRAUD):**

    1. Add same imports (read_pool_reserves, calculate_output_floor, MINIMUM_OUTPUT_FLOOR_BPS)

       ```rust
           TaxError::MinimumOutputFloorViolation
       );
       ```
       - WHY reserve_b as reserve_in: Selling PROFIT (token B) for CRIME/FRAUD (token A). Input goes into reserve_b, output comes from reserve_a.

    3. Snapshot `ctx.accounts.user_token_a.amount` (output is token A for sell direction)

    4. After CPI: reload `user_token_a`, compute `actual_output = after - before`, read lp_fee_bps from pool data, compute lp_fee

    5. Replace emit! block zeroes with actual values

    CRITICAL: Both instructions use `user_token_b` as output for buy (AtoB) and `user_token_a` as output for sell (BtoA). Make sure to snapshot and reload the CORRECT account for each direction.

    **staking events.rs changes:**

    1. Add two new fields to the `RewardsDeposited` event struct:
       ```rust
       /// Emitted when Tax Program deposits rewards via CPI.
       ///
       /// This is the 75% yield portion from taxed swaps.
       #[event]
       pub struct RewardsDeposited {
           /// Amount of SOL deposited.
           pub amount: u64,
           /// New pending_rewards balance after deposit.
           pub new_pending: u64,
           /// Escrow vault pubkey (for monitoring/filtering).
           pub escrow_vault: Pubkey,
           /// Escrow vault SOL balance at time of deposit (for reconciliation monitoring).
           pub escrow_balance: u64,
           /// Slot when deposit occurred.
           pub slot: u64,
       }
       ```
       - WHY add escrow_vault: Off-chain monitoring needs the pubkey to filter events per vault and to verify the vault address matches expectations without additional RPC calls.
       - WHY add escrow_balance: Enables dashboard anomaly detection -- if escrow_balance < new_pending, something is wrong. The on-chain require! already catches this and reverts, but the event gives visibility into healthy deposits too.
       - NOTE: Adding fields to an event struct is backwards-compatible for event consumers -- new fields appear at the end, old parsers can ignore them.

    **deposit_rewards.rs changes:**

    2. Update the emit! block to include the new fields:
       ```rust

           escrow_balance: ctx.accounts.escrow_vault.lamports(),
           slot: clock.slot,
       });
       ```
       - The escrow_vault AccountInfo is already in the account struct (line 62-66). We just read its key and lamports -- zero new accounts, zero new constraints.

    IMPORTANT: Do NOT modify the reconciliation logic itself (the `require!` on line 99-102). It is correct as-is. This task is event enrichment only.
