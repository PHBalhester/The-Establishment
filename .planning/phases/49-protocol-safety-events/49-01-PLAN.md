    Create the foundational pieces needed by all 4 swap instructions for minimum output floor enforcement:

    1. **Create `programs/tax-program/src/helpers/pool_reader.rs`** -- a new helper module with `read_pool_reserves(pool_info: &AccountInfo) -> Result<(u64, u64)>`:
       - Borrow pool AccountInfo data
       - Validate data length >= 153 bytes (error: TaxError::InvalidPoolType)
       - Read reserve_a from bytes [137..145] as u64 little-endian
       - Read reserve_b from bytes [145..153] as u64 little-endian
       - Return (reserve_a, reserve_b)
       - This is the EXACT same pattern used in `epoch-program/src/instructions/execute_carnage_atomic.rs` -- copy the approach, not the code (different error types).
       - WHY raw byte reads: The Tax Program has no Cargo dependency on the AMM crate and doesn't import PoolState. Raw byte reads at known offsets avoid cross-crate coupling. This pattern is proven in Carnage code.

    2. **Update `programs/tax-program/src/helpers/mod.rs`** -- add `pub mod pool_reader;`

    3. **Add `calculate_output_floor` to `programs/tax-program/src/helpers/tax_math.rs`**:
       ```rust
       /// Calculate protocol-enforced minimum output floor.
       ///
       /// Formula: floor = (reserve_out * amount_in / (reserve_in + amount_in)) * floor_bps / 10_000
       /// Uses u128 intermediate math to prevent overflow (same pattern as Carnage slippage).
       ///
       /// Returns 0 if any input is 0 (empty pool edge case -- no floor enforceable).
       pub fn calculate_output_floor(
           reserve_in: u64,
           reserve_out: u64,
           amount_in: u64,
           floor_bps: u64,
       ) -> Option<u64>
       ```
       - If reserve_in == 0 || reserve_out == 0 || amount_in == 0, return Some(0)
       - Compute expected output using constant-product: `expected = reserve_out * amount_in / (reserve_in + amount_in)` with u128 intermediates
       - Compute floor: `floor = expected * floor_bps / 10_000` with u128 intermediates
       - Return Some(floor)
       - WHY no LP fee adjustment: At 50% floor, the ~1% LP fee is absorbed. Raw constant-product is simpler and the 50% threshold provides massive tolerance (see 49-RESEARCH.md Pitfall 4).
       - Add unit tests:
         - `floor_50pct_equal_reserves`: reserves (1000, 1000), amount_in=100, floor_bps=5000 -> expected ~91, floor ~45
         - `floor_zero_reserves`: (0, 1000) or (1000, 0) -> floor = 0
         - `floor_zero_amount`: reserves (1000, 1000), amount_in=0 -> floor = 0
         - `floor_100pct`: floor_bps=10000 -> floor equals expected
         - `floor_large_reserves`: test with u64::MAX-scale reserves to verify no overflow

    4. **Add constant to `programs/tax-program/src/constants.rs`**:
       ```rust
       /// Protocol-enforced minimum output floor (50% = 5000 bps).
       /// User swaps with minimum_amount_out below this floor are rejected.
       /// Consistent with Carnage's 50% slippage floor approach.
       /// Source: Phase 49 CONTEXT.md (SEC-10)
       pub const MINIMUM_OUTPUT_FLOOR_BPS: u64 = 5000;
       ```

    5. **Add error variant to `programs/tax-program/src/errors.rs`**:
       ```rust
       /// User's minimum_amount_out is below the protocol-enforced floor.
       /// The floor is 50% of constant-product expected output.
       /// Set minimum_amount_out to at least the floor value.
       #[msg("Minimum output below protocol floor (50% of expected)")]
       MinimumOutputFloorViolation,
       ```

    WHY hard reject (not silent upgrade): Per 49-RESEARCH.md recommendation -- silent upgrade masks broken frontends that send minimum_amount_out=0. The error message educates users/bots. Carnage uses hard reject too.
  </action>

    2. BEFORE the AMM CPI (after step 3 "Calculate SOL to swap"), add floor enforcement:
       ```rust
       // =========================================================================
       // 3b. Enforce protocol minimum output floor (SEC-10)
       // =========================================================================
       // Read pool reserves from raw AccountInfo bytes (no AMM crate dependency).
       // For buy (AtoB): reserve_in = reserve_a (SOL), reserve_out = reserve_b (token).
       let (reserve_a, reserve_b) = read_pool_reserves(&ctx.accounts.pool)?;
       let output_floor = calculate_output_floor(reserve_a, reserve_b, sol_to_swap, MINIMUM_OUTPUT_FLOOR_BPS)
           .ok_or(error!(TaxError::TaxOverflow))?;
       require!(
           minimum_output >= output_floor,
           TaxError::MinimumOutputFloorViolation
       );
       ```
       - WHY use sol_to_swap (not amount_in): Tax is deducted from input before the swap. The effective input to the AMM is sol_to_swap. Using amount_in would compute a higher expected output than achievable, making the floor too tight. (See 49-RESEARCH.md Pitfall 2)

    3. BEFORE the AMM CPI, snapshot user_token_b balance for balance-diff:
       ```rust
       // =========================================================================
       // 5b. Snapshot output token balance for event (FIX-06)
       // =========================================================================
       let token_b_before = ctx.accounts.user_token_b.amount;
       ```

    4. AFTER the AMM CPI (invoke_signed), reload and compute output:
       ```rust
       // =========================================================================
       // 6h. Compute actual output via balance-diff (FIX-06)
       // =========================================================================
       ctx.accounts.user_token_b.reload()?;
       let tokens_received = ctx.accounts.user_token_b.amount
           .checked_sub(token_b_before)
           .ok_or(error!(TaxError::TaxOverflow))?;
       ```
       - WHY reload: After CPI, Anchor's cached struct has stale values. `.reload()` re-reads from the runtime AccountInfo. (See 49-RESEARCH.md Pitfall 1)

    5. In the emit! block, replace `output_amount: 0` with `output_amount: tokens_received`
