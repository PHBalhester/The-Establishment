  **SolPoolAmm struct:**
  ```rust
  #[derive(Clone)]
  pub struct SolPoolAmm {
      key: Pubkey,          // Pool PDA address (CRIME/SOL or FRAUD/SOL)
      is_crime: bool,       // true = CRIME, false = FRAUD
      reserve_sol: u64,     // SOL reserve (from PoolState)
      reserve_token: u64,   // Token reserve (from PoolState)
      lp_fee_bps: u16,      // LP fee in BPS (100 = 1%)
      buy_tax_bps: u16,     // Current epoch buy tax
      sell_tax_bps: u16,    // Current epoch sell tax
  }
  ```

  **Amm trait implementation (10 methods):**

  1. `from_keyed_account(keyed_account, _context)`: Parse PoolState from `keyed_account.account.data` using `ParsedPoolState::from_bytes()`. Determine `is_crime` by checking if pool key matches CRIME_SOL_POOL. Initialize tax rates to 0 (will be populated by first `update()` call). Initialize reserves from parsed pool state.

  2. `label()`: Return `"Dr Fraudsworth"`.

  3. `program_id()`: Return `TAX_PROGRAM_ID` (the Tax Program is what Jupiter calls for swaps, not the AMM directly).

  4. `key()`: Return `self.key` (the pool PDA).

  5. `get_reserve_mints()`: Return `vec![NATIVE_MINT, CRIME_MINT]` or `vec![NATIVE_MINT, FRAUD_MINT]` based on `is_crime`.

  6. `get_accounts_to_update()`: Return `vec![self.key, EPOCH_STATE_PDA]`. Jupiter refreshes these accounts and passes them to `update()`.

  7. `update(account_map)`: Parse PoolState from `account_map[self.key]` using `ParsedPoolState::from_bytes()`. Update `reserve_sol`, `reserve_token`, `lp_fee_bps`. Parse EpochState from `account_map[EPOCH_STATE_PDA]` using `ParsedEpochState::from_bytes()`. Update `buy_tax_bps` and `sell_tax_bps` using `get_tax_bps(self.is_crime, is_buy)`.

  8. `quote(params)`: This is the critical method. Determine direction:
     - **Buy (SOL -> token)**: `params.input_mint == NATIVE_MINT`
       1. tax = calculate_tax(params.amount, buy_tax_bps)
       2. sol_to_swap = params.amount - tax (return error if underflows)
       3. effective_input = calculate_effective_input(sol_to_swap, lp_fee_bps)
       4. out_amount = calculate_swap_output(reserve_sol, reserve_token, effective_input)
       5. Return Quote { out_amount, fee_amount: tax + (sol_to_swap - effective_input as u64), fee_mint: NATIVE_MINT, fee_pct: Decimal, ... }
     - **Sell (token -> SOL)**: `params.input_mint != NATIVE_MINT`
       1. effective_input = calculate_effective_input(params.amount, lp_fee_bps)
       2. gross_sol = calculate_swap_output(reserve_token, reserve_sol, effective_input)
       3. tax = calculate_tax(gross_sol, sell_tax_bps)
       4. net_sol = gross_sol - tax
       5. lp_fee_amount = params.amount - effective_input as u64
       6. Return Quote { out_amount: net_sol, fee_amount: tax + lp_fee_amount_in_sol_terms, fee_mint: NATIVE_MINT, ... }
     - For fee_pct: Use `Decimal::from(total_fee_bps) / Decimal::from(10_000)`.
     - Set `min_in_amount: None`, `min_out_amount: None` (standard ExactIn).

  9. `get_swap_and_account_metas(params)`: Determine buy vs sell from `params.source_mint`:
     - Buy: call `build_buy_account_metas(user, source_token_account, destination_token_account, is_crime)` + append hook remaining_accounts for output token mint
     - Sell: call `build_sell_account_metas(user, source_token_account, destination_token_account, is_crime)` + append hook remaining_accounts (input token mint hooks, then output ignored for SOL)
     - Return `SwapAndAccountMetas { swap: Swap::TokenSwap, account_metas }` (placeholder variant -- Jupiter assigns real variant during integration).

  10. `supports_exact_out()`: Return `false`.

  **CRITICAL buy vs sell tax application point:**
  - Buy tax: deducted from SOL INPUT before AMM swap
  - Sell tax: deducted from SOL OUTPUT after AMM swap
  Getting this wrong produces quotes that are off by the entire tax amount.

  ```rust
  #[derive(Clone)]
  pub struct VaultAmm {
      key: Pubkey,          // Synthetic unique key for this instance
      input_mint: Pubkey,   // e.g., CRIME_MINT
      output_mint: Pubkey,  // e.g., PROFIT_MINT
      label_suffix: String, // e.g., "CRIME/PROFIT Vault"
  }
  ```

  **Instance keying strategy:** Since all vault instances share one VaultConfig PDA but Jupiter needs unique keys per Amm instance, derive synthetic keys: `Pubkey::find_program_address(&[b"jup_vault", input_mint.as_ref(), output_mint.as_ref()], &CONVERSION_VAULT_PROGRAM_ID).0`. These are deterministic, unique per pair, and don't need to be real on-chain accounts.

  **Supported vault pairs (4 total):**
  - CRIME -> PROFIT (divide by 100)
  - FRAUD -> PROFIT (divide by 100)
  - PROFIT -> CRIME (multiply by 100)
  - PROFIT -> FRAUD (multiply by 100)

  CRIME <-> FRAUD is NOT supported. The on-chain `compute_output_with_mints` returns `InvalidMintPair` for CRIME<->FRAUD. Jupiter can still route CRIME->PROFIT->FRAUD via multi-hop. This is a known limitation documented in REQUIREMENTS.md (Out of Scope).
