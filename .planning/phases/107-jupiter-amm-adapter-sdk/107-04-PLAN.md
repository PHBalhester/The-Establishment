  **README.md:** Create a comprehensive README.md for the SDK crate. This IS the integration documentation (JUP-07). Structure:

  **1. Overview section:**
  - What: Jupiter AMM adapter SDK for Dr. Fraudsworth DEX protocol
  - Why: Enables Jupiter's routing engine to route swaps through Dr. Fraudsworth pools
  - Key feature: Exact quote accuracy (proven by LiteSVM parity tests), zero network calls

  **2. Pool Types section (6 Amm instances):**

  | Instance | Type | Key | Reserves | Fees |
  |----------|------|-----|----------|------|
  | CRIME/SOL | SolPoolAmm | Pool PDA | Dynamic (AMM) | 1% LP + dynamic tax (4-14%) |
  | FRAUD/SOL | SolPoolAmm | Pool PDA | Dynamic (AMM) | 1% LP + dynamic tax (4-14%) |
  | CRIME->PROFIT | VaultAmm | Synthetic PDA | Fixed rate (100:1) | Zero |
  | FRAUD->PROFIT | VaultAmm | Synthetic PDA | Fixed rate (100:1) | Zero |
  | PROFIT->CRIME | VaultAmm | Synthetic PDA | Fixed rate (1:100) | Zero |
  | PROFIT->FRAUD | VaultAmm | Synthetic PDA | Fixed rate (1:100) | Zero |

  6 total Amm instances. 2 SOL pool instances are bidirectional (buy and sell), covering 4 directions. 4 vault instances are unidirectional, one per conversion direction.

  Note: CRIME<->FRAUD direct conversion is not supported on-chain. Jupiter routes CRIME->PROFIT->FRAUD via multi-hop.

  **3. Pool Discovery section:**
  - `known_instances() -> Vec<(Pubkey, VaultAmm)>`: Returns all 4 VaultAmm instances (fixed-pool protocol, no getProgramAccounts needed)
  - `known_sol_pool_keys() -> Vec<Pubkey>`: Returns 2 SOL pool keys for `from_keyed_account` initialization
  - `all_pool_keys() -> Vec<Pubkey>`: Convenience method returning all 6 instance keys
  - Jupiter integrators call these at startup to register all Dr. Fraudsworth pools

  **4. Fee Structure section:**
  - SOL pools: 1% LP fee (fixed, deducted from swap amount) + dynamic tax (4-14% based on epoch)
  - Buy tax: deducted from SOL input BEFORE swap
  - Sell tax: deducted from SOL output AFTER swap
  - Tax changes every epoch (~13 minutes). Jupiter's `update()` refreshes EpochState for current rates.
  - Vault conversions: Zero fees, fixed 100:1 rate

  **5. Epoch Dynamics section:**
  - Tax rates rotate every epoch between CRIME and FRAUD
  - Cheap side gets 4% buy / 14% sell, expensive side gets 14% buy / 4% sell
  - This creates arbitrage opportunities that Jupiter can route through
  - EpochState PDA is declared in `get_accounts_to_update()` so Jupiter refreshes it

  **6. Account Metas section:**
  - Explain the account structure for each instruction type
  - SwapSolBuy: 20 named + 4 transfer hook = 24 accounts
  - SwapSolSell: 21 named + 4 transfer hook = 25 accounts (extra: wsol_intermediary)
  - VaultConvert: 9 named + 8 transfer hook = 17 accounts
  - Transfer hook accounts are deterministic (hardcoded, no network calls)

  **7. Quick Start / Usage section:**
  ```rust
  use drfraudsworth_jupiter_adapter::{SolPoolAmm, VaultAmm, known_instances, known_sol_pool_keys};
  use jupiter_amm_interface::{Amm, KeyedAccount, QuoteParams, SwapMode};
  // ... brief usage example
  ```

  **8. Program IDs section:**
  List all program IDs that Jupiter needs to know about:
  - Tax Program (called for SOL pool swaps)
  - Conversion Vault (called for vault conversions)
  - AMM Program (called indirectly via Tax Program CPI)

  **9. IDL Locations section (JUP-08):**
  - IDLs available at: `target/idl/*.json` in the source repo
  - Public repo: `https://github.com/MetalLegBob/drfraudsworth`
  - List each IDL file and which program it describes

  **10. Jupiter Integration Notes section:**
  - Swap variant: placeholder `Swap::TokenSwap` used -- Jupiter assigns real variants during review
  - Vault instance keying: synthetic PDAs derived from mint pairs
  - supports_exact_out: false for all instances
  - No network calls in any method (required by Jupiter)

  **11. Mainnet Addresses section:**
