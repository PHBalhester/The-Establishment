  **LiteSVM test approach (REAL on-chain execution, not math tracing):**

  The test flow for each parity test is:
  1. Create a LiteSVM instance
  2. Deploy the actual program .so files (compiled from the workspace) to the LiteSVM validator
  3. Initialize all required state (mints, pools, epoch state, token accounts, whitelist entries)
  4. Set up known reserves and tax rates
  5. Execute a REAL swap transaction via the Tax Program instruction
  6. Read the user's output token account balance change = actual on-chain output
  7. Construct a SolPoolAmm with the SAME initial state
  8. Call SolPoolAmm::quote() with the same input amount
  9. Assert: sdk_quote.out_amount == on_chain_output (within 1 lamport tolerance)

  **Program deployment to LiteSVM:**
  Load .so files from `target/deploy/` directory. The programs must be built with `localnet` feature (which uses runtime mints from VaultConfig instead of hardcoded mainnet addresses). Use `litesvm.add_program(program_id, &program_bytes)` for each program.

  Programs to deploy:
  - AMM Program (swap execution)
  - Tax Program (tax calculation + swap orchestration)
  - Epoch Program (EpochState storage)
  - Staking Program (tax distribution target)
  - Transfer Hook Program (T22 hook validation)

  **Initialization helper functions:**
  Create test helpers for:
  - `deploy_all_programs(svm: &mut LiteSVM)` -- load all 5 .so files
  - `create_pool(svm, mint_a, mint_b, reserve_a, reserve_b, lp_fee_bps)` -- initialize AMM pool
  - `create_epoch_state(svm, crime_buy_bps, crime_sell_bps, fraud_buy_bps, fraud_sell_bps)` -- initialize EpochState
  - `create_token_accounts(svm, user, mints)` -- create user ATAs
  - `setup_whitelist(svm, accounts)` -- whitelist token accounts for transfer hook
  - `mint_tokens(svm, mint, dest, amount)` -- fund test accounts

  **NOTE on complexity:** This test setup IS complex -- it's initializing a multi-program stack. But it's the ONLY way to prove on-chain parity. The project's `tests/cross-crate/` already demonstrates this pattern exists in the codebase. Reference that for Anchor instruction building patterns.

  If deploying all 5 programs proves too complex in a single context (e.g., hook initialization requires too many steps), use a fallback approach: deploy ONLY the AMM program to LiteSVM, execute a raw AMM swap (bypassing Tax Program), and verify the AMM math portion. Then separately verify the tax math using the already-proven unit tests from Plan 01. Document in the test file which portions are verified by LiteSVM vs unit tests.

  **parity_sol_pool.rs -- Test structure:**

  ```rust
  // Helper: Set up LiteSVM with all programs and state
  fn setup_test_env() -> (LiteSVM, TestAccounts) { ... }

  // Helper: Execute on-chain swap and return output amount
  fn execute_on_chain_buy(svm: &mut LiteSVM, accounts: &TestAccounts, amount_in: u64, is_crime: bool) -> u64 { ... }
  fn execute_on_chain_sell(svm: &mut LiteSVM, accounts: &TestAccounts, amount_in: u64, is_crime: bool) -> u64 { ... }

  // Helper: Get SDK quote for the same swap
  fn get_sdk_buy_quote(reserve_sol: u64, reserve_token: u64, lp_fee_bps: u16, buy_tax_bps: u16, amount_in: u64) -> u64 { ... }
  fn get_sdk_sell_quote(reserve_sol: u64, reserve_token: u64, lp_fee_bps: u16, sell_tax_bps: u16, amount_in: u64) -> u64 { ... }

  #[test]
  fn parity_buy_crime_1sol() {
      let (mut svm, accounts) = setup_test_env();
      let amount_in = 1_000_000_000; // 1 SOL
      let on_chain_output = execute_on_chain_buy(&mut svm, &accounts, amount_in, true);
      let sdk_output = get_sdk_buy_quote(
          accounts.crime_pool_reserve_sol,
          accounts.crime_pool_reserve_token,
          100, // lp_fee_bps
          accounts.crime_buy_tax_bps,
          amount_in,
      );
      assert!(
          sdk_output.abs_diff(on_chain_output) <= 1,
          "SDK={}, on-chain={}, diff={}",
          sdk_output, on_chain_output, sdk_output.abs_diff(on_chain_output)
      );
  }
  ```

  **parity_vault.rs -- LiteSVM test for vault conversions:**

  Vault parity tests deploy the Conversion Vault program to LiteSVM and execute real convert_v2 transactions.

  **Setup:**
  1. Deploy Conversion Vault .so to LiteSVM (with localnet feature)
  2. Deploy Transfer Hook .so (needed for T22 transfer hooks)
  3. Initialize VaultConfig PDA
  4. Create T22 mints (CRIME, FRAUD, PROFIT) with transfer hook extension
  5. Create vault token accounts + user token accounts
  6. Mint tokens to user and vault as needed
  7. Initialize whitelist for all token accounts

  **Test flow per direction:**
  1. Record user's output token balance before
  2. Execute convert_v2 instruction on LiteSVM
  3. Record user's output token balance after
  4. on_chain_output = after - before
  5. Compute SDK VaultAmm::quote() with same input
  6. Assert: sdk_quote.out_amount == on_chain_output (exact match for vault -- no rounding variance)

  **Test scenarios (4 directions x 3 amounts = 12 tests):**

  1. **parity_crime_to_profit_exact** (10000 CRIME -> 100 PROFIT)
  2. **parity_crime_to_profit_with_remainder** (10099 CRIME -> 100 PROFIT, 99 lost to integer division)
  3. **parity_crime_to_profit_dust** (99 CRIME -> should fail on-chain with OutputTooSmall, SDK should return error)
  4. **parity_fraud_to_profit_exact** (100_000_000_000 FRAUD -> 1_000_000_000 PROFIT)
  5. **parity_fraud_to_profit_large** (500_000_000_000_000 FRAUD -> 5_000_000_000_000 PROFIT)
  6. **parity_fraud_to_profit_small** (100 FRAUD -> 1 PROFIT)
  7. **parity_profit_to_crime_exact** (100 PROFIT -> 10000 CRIME)
  8. **parity_profit_to_crime_large** (1_000_000_000 PROFIT -> 100_000_000_000 CRIME)
  9. **parity_profit_to_crime_overflow** (u64::MAX / 50 PROFIT -> should succeed with checked_mul)
  10. **parity_profit_to_fraud_exact** (1 PROFIT -> 100 FRAUD)
  11. **parity_profit_to_fraud_large** (10_000_000 PROFIT -> 1_000_000_000 FRAUD)
  12. **parity_profit_to_fraud_overflow_guard** (u64::MAX PROFIT -> should fail on-chain with MathOverflow, SDK should return error)
