<task type="auto">
  <name>Task 1: Create railway.toml, migration script, and health check endpoint</name>
  <files>
    railway.toml
    app/db/migrate.ts
    app/app/api/health/route.ts
    app/package.json
  </files>
  <action>
    **1. Create `railway.toml` at the REPO ROOT** (not inside app/):

    ```toml
    [build]
    builder = "RAILPACK"

    [deploy]
    startCommand = "npm run start"
    preDeployCommand = "npx tsx db/migrate.ts"
    healthcheckPath = "/api/health"
    healthcheckTimeout = 120
    restartPolicyType = "ON_FAILURE"
    restartPolicyMaxRetries = 3
    ```

    CRITICAL: The railway.toml lives at the repo root, but the Root Directory is set to `app/` in the Railway dashboard. The `preDeployCommand` runs inside the build context (app/ directory), so `db/migrate.ts` is relative to app/. The `startCommand` runs `npm run start` which is the app/package.json `start` script (`next start`).

    **2. Create `app/db/migrate.ts`** -- programmatic migration runner:

    Use `drizzle-orm/postgres-js/migrator` (NOT `drizzle-kit` which is a devDependency and may be pruned). Create a single-connection client (max: 1) for migrations, run all pending migrations from `./db/migrations`, then close the connection. Exit with code 1 on failure so Railway halts deployment. Log migration start/complete for deploy log visibility.

    Key imports: `drizzle` from `drizzle-orm/postgres-js`, `migrate` from `drizzle-orm/postgres-js/migrator`, `postgres` from `postgres`.

    The `DATABASE_URL` env var is required -- exit with error if unset. The `migrationsFolder` is `./db/migrations` (relative to app/ where preDeployCommand runs).

    **3. Create `app/app/api/health/route.ts`** -- health check endpoint:

    - Export `runtime = "nodejs"` and `dynamic = "force-dynamic"` (no caching).
    - Export async GET handler.
    - Check Postgres by executing `SELECT 1` via the existing `db` from `@/db/connection`.
    - Check Solana RPC by calling `getSlot()` on the connection from `@/lib/connection`.
    - Return JSON `{ status: "ok" | "degraded", checks: { postgres: boolean, solanaRpc: boolean }, timestamp: ISO string }`.
    - Return HTTP 200 if all checks pass, 503 if any fail.
    - Each check is wrapped in try/catch independently (one failure shouldn't prevent the other from being checked).
    - Import `sql` from `drizzle-orm` for the raw SQL query.

    IMPORTANT: The health check must NEVER throw an unhandled error. Wrap the entire handler in try/catch and return 503 with error details on any unexpected failure.

    **4. Add `tsx` to `app/package.json` dependencies:**

    Add `"tsx": "^4.21.0"` to the `dependencies` object (NOT devDependencies). This is needed because Railway's `preDeployCommand` runs in the built image where devDependencies may be pruned. tsx is used to execute the TypeScript migration script.

    Why not compile migrate.ts to JS? Because tsx is a simple, zero-config TypeScript runner that avoids build pipeline complexity. The ~2MB size increase is negligible.
  </action>
  <verify>

    - `cat app/app/api/health/route.ts` shows GET handler checking postgres + RPC
    - `grep tsx app/package.json` shows tsx in dependencies
  </verify>
  <done>
    railway.toml exists at repo root with correct config. migrate.ts exists and uses programmatic drizzle-orm migrate(). Health check endpoint returns structured JSON with postgres and solanaRpc status. tsx is in app dependencies.

<task type="auto">
  <name>Task 2: Generate initial Drizzle migration files from existing schema</name>
  <files>
    app/db/migrations/
  </files>
  <action>
    Generate the initial SQL migration files from the existing Drizzle schema (app/db/schema.ts) using drizzle-kit.

    Run (from the app/ directory):
    ```bash
    export PATH="/opt/homebrew/bin:$PATH"
    cd app
    npx drizzle-kit generate
    ```

    This reads `drizzle.config.ts` (schema: "./db/schema.ts", out: "./db/migrations", dialect: "postgresql") and generates SQL migration files in `app/db/migrations/`.

    NOTE: This command does NOT require DATABASE_URL -- it only generates SQL from the schema definition. The generated files should contain CREATE TABLE statements for: swap_events, candles, epoch_events, carnage_events (the 4 tables defined in schema.ts).

    After generation, verify the migration files exist and contain the expected table creation SQL. The migration folder should contain:
    - A `.sql` file with CREATE TABLE statements
    - A `meta/` directory with migration metadata (journal, snapshots)

    Do NOT attempt to apply migrations locally -- that requires a running Postgres instance. The migrations will be applied automatically by Railway's preDeployCommand on first deploy.
  </action>
  <verify>
    - `ls app/db/migrations/` shows at least one .sql file and a meta/ directory
    - The .sql file contains CREATE TABLE for swap_events, candles, epoch_events, carnage_events
  </verify>
