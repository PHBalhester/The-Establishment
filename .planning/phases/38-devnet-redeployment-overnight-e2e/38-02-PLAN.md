  <files>scripts/vrf/lib/vrf-flow.ts</files>
  <action>
  Modify the `tryReveal` function in `scripts/vrf/lib/vrf-flow.ts` to implement gateway rotation when the primary oracle's gateway fails. This resolves STATE.md pending todo #5.

  **Current behavior (tryReveal):**
  - Calls `randomness.revealIx()` up to `maxAttempts` times with 3s delays
  - Each attempt hits the SAME oracle/gateway
  - If the gateway is down, all attempts fail and we fall through to 300-slot timeout recovery (~2 min wasted)

  **New behavior (tryRevealWithGatewayRotation):**
  1. First, try the default `randomness.revealIx()` (uses the oracle's own gateway). If it works, return immediately.
  2. If the default fails, fetch all available gateways via `CrossbarClient.fetchGateways('devnet')`.
  3. For each alternative gateway URL, try `gateway.fetchRandomnessReveal()` manually with the same randomness account data.
  4. If a gateway responds, build the reveal instruction manually from the response and return it.
  5. If ALL gateways fail, return null (caller falls through to timeout recovery).

  **Implementation details:**
  - Import `CrossbarClient` from `@switchboard-xyz/common` or via the `sb` namespace (check `@switchboard-xyz/on-demand` re-exports).
  - The CrossbarClient needs a crossbar URL. Use `sb.CrossbarClient.default()` or construct with the default URL from the SDK. Check `node_modules/@switchboard-xyz/on-demand` for how the SDK creates its crossbar client.
  - `gateway.fetchRandomnessReveal()` needs: `randomnessAccount` (pubkey), `slothash` (base58), `slot` (number), `rpc` (endpoint URL).
  - The reveal instruction from gateway rotation has the same structure as `revealIx()` -- it needs to be constructed from the gateway response fields (`signature`, `recovery_id`, `value`).
  - Since building the instruction manually from SDK internals is complex and fragile, use a SIMPLER approach: create a new `sb.Gateway(gatewayUrl)` for each URL, then call the SDK's higher-level API. Check if `randomness.revealIx()` accepts an optional gateway parameter, or if the SDK has `randomness.reveal({gateway})`.
  - If the SDK does NOT support gateway override, use the manual approach from 38-RESEARCH.md Pattern 2. But first check the SDK source at `node_modules/@switchboard-xyz/on-demand/dist/esm/accounts/randomness.js` to see if there is a simpler way.

  **Key constraints:**
  - Do NOT change the function signature of `advanceEpochWithVRF` -- it's called by multiple scripts
  - The gateway rotation is internal to the reveal retry logic
  - Log which gateway succeeded/failed for diagnostic purposes
  - Keep the existing 300-slot timeout recovery as the final fallback if ALL gateways fail
  - Gateway list should be fetched ONCE per epoch (not per retry), cached for the attempt

  **Rename pattern:**
  - Keep `tryReveal` as a thin wrapper that calls the new `tryRevealWithGatewayRotation`
  - Or modify `tryReveal` in-place to add gateway rotation after the first round of retries

  **Testing approach:**
  - No unit tests needed (this is script code, not program code)
  - The overnight runner will exercise this in production against real devnet oracles
  - Console logging will show which gateways were tried and which succeeded

  **Part A: Create `scripts/e2e/lib/overnight-reporter.ts`**

  Build an `OvernightReporter` class that generates `Docs/Overnight_Report.md` from an array of epoch JSONL records.

  The EpochRecord interface (from 38-RESEARCH.md):
  ```typescript
  interface EpochRecord {
    timestamp: string;           // ISO 8601
    epochIndex: number;          // 0-based index in this run
    epochNumber: number;         // On-chain epoch number
    cheapSide: string;           // "CRIME" | "FRAUD"
    crimeBuyTaxBps: number;
    crimeSellTaxBps: number;
    fraudBuyTaxBps: number;
    fraudSellTaxBps: number;
    vrfBytes: number[];          // First 8 VRF bytes
    carnageTriggered: boolean;
    carnageExecuted: boolean;
    swapPerformed: boolean;
    swapPool: string;
    swapSig: string | null;
    taxDistribution: {
      staking: number;
      carnage: number;
      treasury: number;
    } | null;
    stakingYieldDelta: number;
    errors: string[];
    txSignatures: string[];
    vrfDurationMs: number;
    totalDurationMs: number;
    walletBalance: number;
    carnageVaultBalance: number;
  }
  ```

  The OvernightReporter.generate() method produces a Markdown file with:
  1. **Header**: Run date, duration, epoch count, cluster
  2. **Executive Summary**: Total epochs, Carnage triggers, error rate, total swaps
  3. **Tax Rate Distribution Table**: Per-epoch CRIME/FRAUD buy/sell rates, cheapSide
  4. **Carnage Events**: Table of any Carnage triggers with epoch numbers and VRF bytes
  5. **Staking Yield**: Total yield delta across all epochs
  6. **Error Summary**: Grouped errors with frequency counts
  7. **Epoch Detail Table**: Compact table with key metrics per epoch (epoch #, cheapSide, tax rates, carnage, swap sig, duration)
  8. **Footer**: Run parameters (target epochs, actual epochs, total duration)

  Use Solana Explorer links for TX signatures (same pattern as E2EReporter).

  **Part B: Create `scripts/e2e/overnight-runner.ts`**

  Build the main overnight runner as a single TypeScript long-lived process. Structure follows carnage-hunter.ts pattern but extended for 100 epochs.

  **Constants:**
  - `TARGET_EPOCHS = 100` (configurable via env `OVERNIGHT_EPOCHS`)
  - `AIRDROP_THRESHOLD = 5 * LAMPORTS_PER_SOL`
  - `AIRDROP_AMOUNT = 2 * LAMPORTS_PER_SOL`
  - `SLOT_WAIT_BETWEEN_EPOCHS = 760` (750 minimum + 10 buffer)
  - `WSOL_BUDGET = 15 * LAMPORTS_PER_SOL` (100 swaps at 0.1 SOL + buffer)
  - `SWAP_AMOUNT = 100_000_000` (0.1 SOL)
  - `LOG_PATH = path.resolve(__dirname, "overnight-run.jsonl")`

  **Main loop structure:**
  ```
  1. Load provider, programs, manifest
  2. Check wallet balance, log starting state
  3. Create E2E user with WSOL_BUDGET
  4. Stake 10 PROFIT (so staking yield accrues across all epochs)
  5. Build VRF accounts from manifest
  6. Set up graceful shutdown handlers (SIGINT, SIGTERM)
  7. FOR epoch 0..TARGET_EPOCHS:
     a. Check shutdownRequested, break if true
     b. checkAndAirdrop (if balance < threshold)
     c. If epoch > 0: waitForSlotAdvance(SLOT_WAIT_BETWEEN_EPOCHS)
     d. advanceEpochWithVRF -> result
     e. executeSolBuySwap (alternating CRIME/SOL and FRAUD/SOL)
     f. If result.carnageTriggered: attempt execute Carnage
     g. Read post-epoch state (escrow balance, carnage vault balance, wallet balance)
     h. Build EpochRecord, append to JSONL via appendFileSync
     i. Console log epoch summary line
  8. Claim staking yield
  9. Generate Docs/Overnight_Report.md via OvernightReporter
  10. Log final summary to console and JSONL
  ```

  **Error handling per epoch:**
  - Wrap each epoch in try/catch
  - On error: log the error to the EpochRecord.errors array and JSONL, continue to next epoch
  - VRF failures are expected (~33% per gateway on devnet, but gateway rotation should handle most)
  - Non-VRF failures (swap, balance read): log and continue

  **Graceful shutdown:**
  ```typescript
  let shutdownRequested = false;
  process.on('SIGINT', () => { shutdownRequested = true; console.log('\nShutdown requested...'); });
  process.on('SIGTERM', () => { shutdownRequested = true; });
  ```
  Check at the top of each epoch loop iteration. On shutdown, skip remaining epochs, claim staking yield, generate report.

  **Auto-airdrop:**
  ```typescript
  async function checkAndAirdrop(provider): Promise<void> {
    const balance = await provider.connection.getBalance(provider.wallet.publicKey);
    if (balance < AIRDROP_THRESHOLD) {
      const sig = await provider.connection.requestAirdrop(provider.wallet.publicKey, AIRDROP_AMOUNT);
      await provider.connection.confirmTransaction(sig, 'confirmed');
    }
  }
  ```

  **JSONL logging:**
  Use `fs.appendFileSync` for crash safety (same pattern as E2ELogger). Each EpochRecord is one JSON line.

  **Carnage execution on trigger:**
  When `result.carnageTriggered` is true, attempt to call `execute_carnage_atomic`. Import the Carnage execution logic from `carnage-flow.ts` or build inline. The key accounts are in the manifest. Log success/failure. If Carnage execution fails, log the error but continue -- the trigger was still captured.

  **Console output per epoch:**
  One line per epoch for monitoring:
  ```
  [42/100] Epoch 578 | CRIME cheap | 200/1300 bps | swap OK | 45.2s | no carnage
  ```

  **Run command (add to header comment):**
  ```bash
  set -a && source .env && set +a && npx tsx scripts/e2e/overnight-runner.ts
  ```

  **Important: Do NOT re-initialize Switchboard setup per epoch.**
  The `advanceEpochWithVRF` function currently calls `sb.getProgramId`, `Program.fetchIdl`, and `sb.getDefaultQueue` internally on every invocation. This wastes ~3-5 seconds per epoch. For the overnight runner, consider whether we can pass a pre-built sbProgram/queueAccount to avoid this overhead. If the function signature of advanceEpochWithVRF cannot be changed (it's used elsewhere), accept the overhead for now -- 100 epochs * 5s = 8 min total, acceptable for an 8-hour run.
