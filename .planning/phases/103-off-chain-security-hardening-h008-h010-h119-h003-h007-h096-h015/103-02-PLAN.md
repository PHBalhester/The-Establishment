<task type="auto">
  <name>Task 1: Create per-account-type bounds validators</name>
  <files>app/lib/webhook-validators.ts</files>
  <action>
  Create a new file `app/lib/webhook-validators.ts` that exports a single function `validateDecodedAccount(label: string, normalized: Record<string, unknown>): boolean`.

  The function dispatches to a per-account-type validator based on the `label` parameter (which comes from `KNOWN_PROTOCOL_ACCOUNTS` in the webhook handler -- values like "EpochState", "CarnageFundState", "PoolState:CRIME_SOL", etc.).

  **CRITICAL: Validators run AFTER `anchorToJson()` normalization.** This means:
  - BN fields are converted to JavaScript `number` (or `{ __bigint: "..." }` for bigintFields)
  - PublicKey fields are converted to base58 strings
  - Field names are camelCase (Anchor 0.32's `convertIdlToCamelCase()`)
  - For `{ __bigint }` tagged values, extract the numeric value for range checks

  Write a helper `numericValue(v: unknown): number | null` that:
  - Returns `v` directly if `typeof v === "number"`
  - Returns `Number(v.__bigint)` if `v` is a `{ __bigint: string }` tag (checking for precision loss is unnecessary since we're doing range checks, not exact comparisons)
  - Returns `null` otherwise

  **Per-account-type validators:**

  1. **EpochState** (labels: "EpochState"):
     - All BPS fields must be in [0, 10000]: crimeBuyTaxBps, crimeSellTaxBps, fraudBuyTaxBps, fraudSellTaxBps, lowTaxBps, highTaxBps
     - Use `numericValue()` for each field. If any field is missing from the object, skip it (don't reject -- some fields may be optional or renamed between versions)

  2. **PoolState** (labels starting with "PoolState:"):
     - reserves: reserveA >= 0, reserveB >= 0 (use >= 0 NOT > 0 -- pools can briefly have zero during init)
     - fee: feeNumerator >= 0, feeDenominator > 0 (denominator must be positive to avoid division by zero)

  3. **CarnageFundState** (label: "CarnageFundState"):
     - All numeric fields >= 0 (no field should be negative in a fund balance context)
     - Check fields: solBalance, tokenBalance (any present numeric field >= 0)

  4. **StakePool** (label: "StakePool"):
     - totalStaked >= 0, rewardsPerTokenStored >= 0
     - rewardsPerTokenStored may be a `{ __bigint }` tag (it's in STAKING_BIGINT_FIELDS)

  5. **CurveState** (labels starting with "CurveState:"):
     - currentPrice >= 0, tokensSold >= 0, solRaised >= 0
     - tokensSold, solRaised may be `{ __bigint }` tags (they're in CURVE_BIGINT_FIELDS)

  For unknown labels (like "CarnageSolVault"), return `true` (don't reject -- these are simple lamport stores with no Anchor decode).

  Add a JSDoc comment explaining the purpose: "Validates decoded protocol account data against known bounds. Runs AFTER anchorToJson normalization. Returns false if any field violates bounds (caller should reject storage and fire Sentry alert)."
  </action>
  <verify>
  Verify the file exists and exports `validateDecodedAccount`. Grep for each account type validator (EpochState, PoolState, CarnageFundState, StakePool, CurveState). Confirm the numericValue helper handles both plain numbers and __bigint tags.
  </verify>
  <done>
  webhook-validators.ts exports validateDecodedAccount with 5 account-type-specific validators and a numericValue helper for BN/bigint-tag handling.
  </done>

<task type="auto">
  <name>Task 2: Wire fail-closed decode + bounds validation into webhook handler</name>
  <files>app/app/api/webhooks/helius/route.ts</files>
  <action>
  Two changes to the `handleAccountChanges()` function in the webhook handler:

  **Change 1 -- Fail-closed decode error (H119):**
  Replace the catch block at lines 607-620. The current catch block calls `protocolStore.setAccountState()` with raw attacker-controlled data (accountData, rawAccountData). Remove that entirely. The new catch block should:
  - Log the error: `console.error(\`[webhook] Failed to decode ${label} at ${pubkey}:\`, err);`
  - Fire Sentry alert: `captureException(new Error(\`[webhook] Decode failed for ${label} at ${pubkey}: ${err}\`));`
  - `continue;` to skip to the next account
  - Do NOT call protocolStore.setAccountState() -- existing good data stays in store

  **Change 2 -- Bounds validation before storage (H096):**
  Add `import { validateDecodedAccount } from "@/lib/webhook-validators";` at the top of the file.
  After the `anchorToJson()` call (line 598-601) and BEFORE the `protocolStore.setAccountState()` call (line 603), add:
  ```
  if (!validateDecodedAccount(label, normalized)) {
    console.error(`[webhook] Bounds validation failed for ${label} at ${pubkey}`);
    captureException(new Error(`[webhook] Bounds validation rejected ${label} at ${pubkey}`));
    continue;
  }
  ```
  This ensures only validated data reaches protocolStore.

  **Also fix the "missing decode info" fallback at lines 564-575:**
  The current code at line 564-575 stores raw data with `decodeError: "missing_decode_info_or_data"` when decode info is absent or raw data is missing. This is a minor version of the H119 issue. Change it to:
  - Log: `console.warn(\`[webhook] No decode info or raw data for ${label} at ${pubkey}, skipping\`);`
  - `continue;` (do not store)
  - This case should be rare (only if ANCHOR_DECODE_MAP is misconfigured)

  Do NOT change: raw transaction handler, authorization logic, rate limiting, storage functions, candle/SSE broadcasting, or the CarnageSolVault SystemAccount path (that one is fine -- it only stores lamports from nativeBalanceChange, not raw Anchor data).
