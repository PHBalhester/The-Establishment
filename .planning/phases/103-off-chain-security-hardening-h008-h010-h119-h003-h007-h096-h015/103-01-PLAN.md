<objective>
Harden the RPC proxy against batch amplification (H008), upstream timeout exhaustion (H010), and IP spoofing rate limit bypass (H015). Also add rate limits to two unprotected database-hitting endpoints (/api/candles and /api/carnage-events) flagged by H015.

Purpose: Close 3 Bulwark audit findings (H008, H010, H015) that enable DoS vectors against the RPC proxy and database endpoints. The RPC proxy currently forwards batch requests to Helius (amplification), has no fetch timeout (worker exhaustion), and all rate limiting uses the leftmost x-forwarded-for IP (attacker-spoofable). Two database endpoints have zero rate limiting.

Output: Hardened RPC proxy with 4 defense layers, fixed IP extraction for all rate-limited endpoints, and rate limits on /api/candles and /api/carnage-events.
</objective>

<task type="auto">
  <name>Task 2: Harden RPC proxy with batch rejection, body limit, timeout, and concurrent cap</name>
  <files>app/app/api/rpc/route.ts</files>
  <action>
  Apply 4 defense layers to the POST handler in order. The existing handler structure at line 80 needs to be restructured with a try/finally wrapper for concurrent request tracking.

  1. **Concurrent request cap (module-level state + handler guard):**
     - Add module-level: `const inFlight = new Map<string, number>(); const MAX_CONCURRENT = 20;`
     - After the existing rate limit check (line 89), add the concurrent check: get current count for clientIp, reject with 503 "Service Unavailable" if >= MAX_CONCURRENT, otherwise increment.
     - Wrap the ENTIRE remaining handler body in `try { ... } finally { decrement }`. The finally block MUST decrement or delete the IP entry regardless of outcome (timeout, error, success). Use: `const count = inFlight.get(clientIp) ?? 1; if (count <= 1) inFlight.delete(clientIp); else inFlight.set(clientIp, count - 1);`

  2. **Body size limit (inside try block, before JSON parsing):**
     - Before the existing `request.json()` call, check `content-length` header. If present and > 65_536 (64KB), return 413 "Request Entity Too Large". This is defense-in-depth alongside batch rejection. A single RPC request is ~200 bytes; 64KB is generous.

  3. **Batch rejection (after JSON parsing, replace lines 102-105):**
     - Remove the current `isBatch` / `requests` array logic (lines 102-105). Instead, after successfully parsing `body`, check `Array.isArray(body)`. If true, return 400 with JSON-RPC error: `{ jsonrpc: "2.0", error: { code: -32600, message: "Batch requests not supported" }, id: null }`.
     - After the batch check, create: `const requests = [body as Record<string, unknown>];` (always single-element array so the method validation loop still works).

  4. **Fetch timeout (in the failover loop):**
     - Add `signal: AbortSignal.timeout(10_000)` to the `fetch()` call at line 144. This is the same pattern used in sol-price/route.ts. When the signal fires, it throws a DOMException caught by the existing catch block at line 172, which logs and tries the next endpoint. No additional error handling needed.

  Do NOT change: the method allowlist, failover logic, sticky routing, or credit counter. Only add the 4 new guards.
  </action>
  <verify>
  Grep for "Batch requests not supported" in rpc/route.ts. Grep for "AbortSignal.timeout" in rpc/route.ts. Grep for "MAX_CONCURRENT" in rpc/route.ts. Grep for "65_536" or "65536" in rpc/route.ts. Run `export PATH="/opt/homebrew/bin:$PATH" && cd app && npx next build 2>&1 | tail -20` to verify the build succeeds (no TypeScript errors).
