# Phase 57.1 Plan 03: useCarnageEvents + usePoolPrices Optimization Summary

**Migrated useCarnageEvents from client-side RPC parsing (900 credits/hr) to Postgres API route (0 credits/hr), batched usePoolPrices initial fetch from 4 calls to 1 via getMultipleAccountsInfo, both hooks now visibility-aware**

## Accomplishments

- useCarnageEvents now fetches from /api/carnage-events Postgres route (zero RPC credits)
- Removed ~170 lines of client-side RPC parsing (getSignaturesForAddress, getParsedTransaction, Anchor event discriminator decoding)
- CarnageEvent interface updated to match API response (epochNumber, txSignature, targetToken, path, crimeBurned, fraudBurned, etc.)
- carnageActionLabel updated to accept path strings ("BuyOnly", "Burn", "BurnAndSell")
- CarnageCard.tsx updated for all new field names and improved action display (Buy Only, Burn, Burn & Sell)
- usePoolPrices initial fetch batched to single getMultipleAccountsInfo call (1 credit vs 4)
- Both hooks use useVisibility for modal-scoped pausing with burst-refresh on tab return
- Full app compiles cleanly (npx tsc --noEmit passes)

## Decisions Made

- CarnageEvent interface matches Postgres API response exactly (no client-side field mapping layer needed)
- carnageActionLabel takes path strings directly rather than mapping numbers to strings (simpler, no enum translation)
- CarnageCard derives tokensBurned/tokensBought from crimeBurned/fraudBurned based on targetToken at render time (data model is richer than the old flat format)
- formatEventDate updated to accept ISO date strings from Postgres instead of Unix timestamps
- usePoolPrices manages WS subscriptions as a group (subscribe all 4 or unsubscribe all 4) based on visibility transitions
