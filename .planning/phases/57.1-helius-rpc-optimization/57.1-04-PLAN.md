<objective>
Add visibility-based pausing to the remaining 4 data-fetching hooks: useCarnageData (reduce poll to 30s + pause), useStaking (modal-scoped pause), useTokenBalances (tab visibility pause), and useSolPrice (tab visibility pause). These hooks keep their existing data-fetching patterns but gain intelligent pausing.

Purpose: These hooks don't need fundamental architecture changes (no WS conversion or API migration), but they're responsible for ~960 credits/hr of unnecessary fetching when the tab is hidden or the wrong modal is open. Adding visibility pausing cuts their consumption by 50-80%.

Output: Four modified hooks with visibility-aware polling/subscription management.
</objective>

<task type="auto">
  <name>Task 1: Add visibility pausing to useCarnageData</name>
  <files>app/hooks/useCarnageData.ts</files>
  <action>
Modify `useCarnageData.ts` to:

1. Import `useVisibility` and call `useVisibility("carnage")` -- carnage data is relevant in CarnageStation and DashboardGrid (no modal).

2. **Reduce polling interval from 10s to 30s:**
   Change `POLL_INTERVAL_MS = 10_000` to `POLL_INTERVAL_MS = 30_000`. The CarnageFundState PDA only changes when: (a) a Carnage event executes (~4% chance per epoch, epochs are ~5 min), or (b) tax collection deposits SOL to the vault. 30s is more than sufficient.

3. **Visibility-aware polling:**
   - Replace the single useEffect that sets up both WS + polling.
   - New lifecycle: when `isActive` is true, start the 30s polling interval AND subscribe to vault WS. When `isActive` becomes false, clear the interval AND unsubscribe vault WS.
   - Use a ref to track `isActive` transitions (same pattern as Plan 02).

4. **Visibility-aware WebSocket subscription for vault balance:**
   - The existing `connection.onAccountChange(DEVNET_PDAS.CarnageSolVault, ...)` subscription should only be active when `isActive` is true.
   - On pause: `connection.removeAccountChangeListener(wsSubIdRef.current)`, set to null.
   - On resume: burst-refresh (fetch balance via RPC), then re-subscribe WS.

5. Register burst-refresh via `onResume`:
   ```
   useEffect(() => {
     return onResume(() => {
       fetchCarnageState();
       // Also re-fetch vault balance
       connection.getBalance(DEVNET_PDAS.CarnageSolVault).then(balance => {
         vaultBalanceRef.current = balance;
         setCarnageData(prev => prev ? { ...prev, vaultBalanceLamports: balance } : prev);
       }).catch(() => {});
     });
   }, [onResume, fetchCarnageState]);
   ```

6. **Preserve exact same return type** -- consumers don't know about the pausing.

**Do NOT:**
- Remove the vault WS subscription entirely (it provides real-time vault balance updates).
- Change the CarnageData interface.
  </action>

Run `npx tsc --noEmit app/hooks/useCarnageData.ts`. Verify `POLL_INTERVAL_MS` is 30_000. Verify `useVisibility` is imported and called.
  </verify>
  <done>
useCarnageData polls every 30s (down from 10s) and pauses both polling and vault WS subscription when tab is hidden or carnage station is not active. Burst-refreshes on resume. Return type unchanged.
  </done>

**useStaking.ts:**

1. Import `useVisibility` and call `useVisibility("staking")` -- staking data is only needed in StakingStation modal.
   - Note: useStaking is a large hook (590 lines). Only modify the data-fetching lifecycle (the useEffect with the 30s polling interval around line 272-287). Do NOT modify the execute function, reward calculation, or any other logic.

2. Replace the polling useEffect:
   - When `isActive` is true: set loading, call fetchStakingData, start 30s interval.
   - When `isActive` becomes false: clear interval.
   - Use `isActive` as a dependency of the polling effect.

3. Register burst-refresh via `onResume`:
   ```
   useEffect(() => {
     return onResume(() => { fetchStakingData(); });
   }, [onResume, fetchStakingData]);
   ```

4. **IMPORTANT:** The re-fetch on `status === "confirmed"` effect (line 291-294) should NOT be gated by visibility. A successful transaction should always trigger a re-fetch, even if the tab is somehow hidden.

**useTokenBalances.ts:**

1. Import `useVisibility` and call `useVisibility()` -- NO station specified, because token balances are needed by both SwapStation and StakingStation. Tab visibility pausing is sufficient.

2. Replace the polling interval setup (lines 110-124):
   - When `isActive` is true AND publicKey exists: fetch + start 30s interval.
   - When `isActive` becomes false: clear interval.
   - Keep the existing publicKey null check (reset to zeros).

3. Register burst-refresh via `onResume`:
   ```
   useEffect(() => {
     return onResume(() => { fetchBalances(); });
   }, [onResume, fetchBalances]);
   ```

4. **Preserve the cross-instance sync** (BALANCE_REFRESH_EVENT). The CustomEvent pattern must still work -- when useSwap calls refresh(), all instances re-fetch regardless of visibility state. The refresh() function should NOT be gated by isActive (it's triggered by user actions, not polling).

**useSolPrice.ts:**

1. Import `useVisibility` and call `useVisibility()` -- NO station specified, SOL price is used everywhere.

2. Replace the polling effect:
   - When `isActive` is true: fetch + start 30s interval.
   - When `isActive` becomes false: clear interval.

3. Register burst-refresh via `onResume`.

**Common pattern for all three:**
```typescript
const { isActive, onResume } = useVisibility(/* station or nothing */);

useEffect(() => {
  if (!isActive) return;  // Don't start anything when paused

  mountedRef.current = true;
  fetchData();
  const interval = setInterval(fetchData, POLL_INTERVAL_MS);

  return () => {
    clearInterval(interval);
  };
}, [isActive, fetchData]);

useEffect(() => {
  return onResume(() => { fetchData(); });
}, [onResume, fetchData]);
```

**Do NOT:**
- Change any return types or exported interfaces.
- Modify useStaking's execute function, reward calculation, or form state management.
- Gate useTokenBalances' refresh() function behind visibility (it's user-action triggered).
