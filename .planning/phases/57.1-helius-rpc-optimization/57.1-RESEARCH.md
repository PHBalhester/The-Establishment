# Phase 57.1: Helius RPC/WebSocket Optimization - Research

**Researched:** 2026-02-24
**Domain:** Solana RPC credit optimization, WebSocket subscriptions, Page Visibility API, Postgres API routes
**Confidence:** HIGH (verified against Helius docs, Solana docs, and codebase inspection)

## Summary

This research covers the full technical landscape needed to plan Phase 57.1: reducing Helius RPC credit consumption from ~2,800 credits/hr per idle browser tab to <500 credits/hr.

The project currently uses a mix of polling (setInterval + RPC calls every 10-60s) and WebSocket subscriptions (`onAccountChange`). The codebase already has a solid foundation -- `usePoolPrices` uses WebSocket subscriptions, and the Helius webhook pipeline already writes carnage/epoch/swap events to Postgres via Drizzle ORM. The key wins are: (1) replacing 3 polling hooks with WebSocket subscriptions, (2) migrating `useCarnageEvents` from expensive client-side RPC parsing to a Postgres API route, (3) batching the 4 individual pool `.fetch()` calls into a single `getMultipleAccounts`, (4) pausing all data fetching when the tab is hidden or the relevant modal is closed.

**Primary recommendation:** Use standard Solana WebSocket subscriptions (free on all Helius plans) for `useEpochState` and `useCurrentSlot`, keep `usePoolPrices` as-is (already uses WebSocket), migrate `useCarnageEvents` to a new `/api/carnage-events` route, batch pool initial fetches with `getMultipleAccounts`, and add a `useVisibilityPause` hook that gates all data fetching.

### Pattern 1: WebSocket Subscription with Automatic Reconnection

**What:** Replace `setInterval` + RPC call with `connection.onAccountChange()` for account data that changes infrequently.

**When to use:** When monitoring a specific on-chain account (EpochState PDA, pool accounts) that changes at known intervals.

**Example:**
```typescript
// Source: @solana/web3.js Connection API, verified in existing usePoolPrices.ts
useEffect(() => {
  const connection = getConnection();

  // Initial fetch for immediate data
  fetchData();

  // WebSocket subscription replaces setInterval polling
  const subId = connection.onAccountChange(
    accountPubkey,
    (accountInfo: AccountInfo<Buffer>) => {
      // Decode and update state
      const decoded = program.coder.accounts.decode("epochState", accountInfo.data);
      setState(decoded);
    },
    "confirmed"
  );

  return () => {
    connection.removeAccountChangeListener(subId);
  };
}, []);
```

### Pattern 2: Slot Subscription

**What:** Replace `getSlot()` polling with `connection.onSlotChange()`.

**When to use:** For tracking current slot (used in epoch countdown timer).

**Example:**
```typescript
// Source: Solana RPC WebSocket docs - slotSubscribe
useEffect(() => {
  const connection = getConnection();

  // Initial fetch
  connection.getSlot().then(setCurrentSlot);

  // WebSocket subscription -- fires on every new slot (~400ms)
  const subId = connection.onSlotChange((slotInfo) => {
    setCurrentSlot(slotInfo.slot);
  });

  return () => {
    connection.removeSlotChangeListener(subId);
  };
}, []);
```

**Important note:** `onSlotChange` fires on EVERY slot (~2.5 per second). The hook should NOT re-render on every notification. Use a ref to store the slot value and only update React state periodically (e.g., every 5 seconds via a separate interval that reads from the ref). This prevents ~150 re-renders per minute.

### Pattern 3: Visibility-Aware Data Fetching

**What:** A custom hook that combines Page Visibility API with modal context to determine whether data fetching should be active.

**When to use:** Wrap all data-fetching hooks to pause when tab is hidden or when the parent modal is not active.

**Example:**
```typescript
// useVisibility.ts -- custom hook
function useVisibility(requiredStation?: StationId): {
  isActive: boolean;
  onResume: (callback: () => void) => void;
} {
  const [tabVisible, setTabVisible] = useState(!document.hidden);
  const { state } = useModal();

  useEffect(() => {
    const handler = () => setTabVisible(!document.hidden);
    document.addEventListener("visibilitychange", handler);
    return () => document.removeEventListener("visibilitychange", handler);
  }, []);

  // Active = tab visible AND (no station required OR matching station is open)
  const isActive = tabVisible && (
    !requiredStation || state.activeStation === requiredStation
  );

  return { isActive, onResume: ... };
}
```

### Pattern 4: getMultipleAccounts Batching

**What:** Fetch multiple account data in a single RPC call instead of N individual fetches.

**When to use:** When loading initial data for multiple known accounts (pool states, PDAs).

**Example:**


### Anti-Patterns to Avoid

- **Re-rendering on every slot notification:** `onSlotChange` fires ~2.5 times per second. Storing every update in React state causes 150 re-renders/min. Use a ref + periodic state flush.
- **Subscribing to WebSocket when tab is hidden:** Standard WS notifications are free on Helius, but they still consume browser memory/CPU. Always unsubscribe when paused.
- **Creating new Connection instances per hook:** The existing singleton pattern in `lib/connection.ts` is correct. All WS subscriptions share one connection. Do not break this.
- **Fetching carnage events from RPC when DB already has them:** The Helius webhook handler already writes parsed carnage events to Postgres. Fetching and re-parsing them client-side wastes ~1,260 credits/hr.

|---------|-------------|-------------|-----|

The webhook handler writes to Postgres, the connection singleton handles WS, and `usePoolPrices` already demonstrates the correct WebSocket pattern. This phase is mostly about extending existing patterns to the remaining hooks.

## Common Pitfalls

### Pitfall 1: slotSubscribe Flood

**What goes wrong:** `onSlotChange` fires on every new slot (~2.5/sec = ~9,000/hr). If each notification triggers a React `setState`, the component re-renders 150 times per minute, causing high CPU usage and janky UI.

**Why it happens:** Developers treat `onSlotChange` like `onAccountChange` (which fires only when data changes). Slots advance constantly.

**How to avoid:** Store slot value in a `useRef`, update React state only every N seconds via a throttling interval. The countdown timer only needs ~1 update/second for a smooth display.

**Warning signs:** React DevTools showing rapid re-renders; browser performance tab showing high scripting time.

### Pitfall 2: WebSocket Subscription Leak on Tab Return

**What goes wrong:** When the tab becomes visible again, a naive implementation creates NEW subscriptions without cleaning up the old ones. Over time, each tab-hide/tab-show cycle adds another subscription, causing duplicate state updates and credit waste.

**Why it happens:** The visibility change handler calls the subscription setup function without first tearing down existing subscriptions.

**How to avoid:** Track subscription IDs in refs. On pause: unsubscribe all. On resume: subscribe fresh. The cleanup-before-setup pattern prevents leaks.

**Warning signs:** Helius dashboard showing increasing WS connections; duplicate state updates in React DevTools.

### Pitfall 3: Stale Closures in Visibility Handler

**What goes wrong:** The `visibilitychange` event handler captures stale state from the initial render closure. When the tab returns and the handler tries to force-refresh, it uses outdated connection/program references.

**Why it happens:** Standard React closure gotcha. Event handlers registered in `useEffect` capture variables from the render when the effect ran.

**How to avoid:** Use `useRef` for values needed in event handlers. The existing codebase already follows this pattern (see `mountedRef` in all hooks).

**Warning signs:** Force-refresh on tab return silently fails or fetches from wrong RPC endpoint.

### Pitfall 4: Enhanced WebSocket Plan Requirements

**What goes wrong:** Developer implements Enhanced WebSocket features (`transactionSubscribe` with advanced filtering) and it works on development/free plan, but Enhanced WS features are only available on Business+ plans ($499/mo).

**Why it happens:** Helius docs mention Enhanced WS prominently. Easy to conflate standard and enhanced.

**How to avoid:** We are NOT using Enhanced WebSockets. We use standard Solana WebSocket methods (`accountSubscribe`, `slotSubscribe`) which are included on ALL plans for free (1 credit to open connection, streaming is free). This phase explicitly avoids Enhanced WS to keep costs down.

**Warning signs:** Import from Helius-specific SDK instead of using `@solana/web3.js` Connection methods.

### Pitfall 5: Race Condition Between Pause and Burst-Refresh

**What goes wrong:** Tab becomes visible, burst-refresh fires, but WebSocket subscriptions haven't reconnected yet. The burst-refresh reads stale cached data from the Connection.

**Why it happens:** WebSocket reconnection is async. The `visibilitychange` handler fires synchronously.

**How to avoid:** On tab return: (1) first do the burst-refresh via HTTP RPC (synchronous, guaranteed fresh), (2) then re-establish WS subscriptions for ongoing updates. The RPC calls provide immediate freshness; WS provides ongoing efficiency.

**Warning signs:** Tab return briefly shows stale data that then jumps to correct values.

### Pitfall 6: Modal-Aware Pausing vs DashboardGrid

**What goes wrong:** DashboardGrid (which lives outside the modal system) calls ALL 6 data hooks always. Even with modal-aware pausing in the hooks, DashboardGrid's instances keep running.

**Why it happens:** DashboardGrid is the orchestrator for the main dashboard view, rendered behind the modal. Its hooks are always mounted.

**How to avoid:** DashboardGrid hooks need to check if ANY modal is open. If a modal is open, DashboardGrid should reduce its polling (only epoch state at 30s). When no modal is open, DashboardGrid runs normally. The station-specific modals (CarnageStation, StakingStation, SwapStation) each call their own hooks which mount/unmount with the modal lifecycle (React.lazy).

**Warning signs:** Credit consumption doesn't drop when a modal is open because DashboardGrid keeps polling behind it.

```typescript
const slotRef = useRef<number | null>(null);
const [displaySlot, setDisplaySlot] = useState<number | null>(null);

  const connection = getConnection();

    connection.removeSlotChangeListener(subId);
    clearInterval(flushInterval);
  };
```

### Page Visibility Hook Pattern

```typescript
// useVisibility.ts
import { useState, useEffect, useRef, useCallback } from "react";
import type { StationId } from "@/components/modal/ModalProvider";
import { useModal } from "./useModal";

export function useVisibility(requiredStation?: StationId) {
  const [tabVisible, setTabVisible] = useState(
    typeof document !== "undefined" ? !document.hidden : true
  );
  const { state } = useModal();
  const resumeCallbacksRef = useRef<Array<() => void>>([]);

  // Track tab visibility
  useEffect(() => {
    const handler = () => {
      const visible = !document.hidden;
      setTabVisible(visible);
      if (visible) {
        // Fire burst-refresh callbacks
        for (const cb of resumeCallbacksRef.current) cb();
      }
    };
    document.addEventListener("visibilitychange", handler);
    return () => document.removeEventListener("visibilitychange", handler);
  }, []);

  // Active when tab visible AND either no station required or matching station open
  const isActive = tabVisible && (
    !requiredStation ||
    state.activeStation === requiredStation ||
    state.activeStation === null  // Dashboard view (no modal open)
  );

  // Register a callback to fire on resume (burst-refresh)
  const onResume = useCallback((cb: () => void) => {
    resumeCallbacksRef.current.push(cb);
    return () => {
      resumeCallbacksRef.current = resumeCallbacksRef.current.filter(c => c !== cb);
    };
  }, []);

  return { isActive, tabVisible, onResume };
}
```

### DashboardGrid Hook Architecture (CRITICAL)

`DashboardGrid` is the main orchestrator component that calls ALL 6 data hooks and passes results as props to card components. It lives OUTSIDE the modal system -- it's rendered on the main page behind modals.

Station modals (CarnageStation, StakingStation, SwapStation) mount/unmount via React.lazy when the modal opens/closes. They call their OWN hook instances which are scoped to the modal lifecycle.

**Implication for pausing:**
- When NO modal is open: DashboardGrid hooks run (reduced cadence: epoch 30s, others paused or very slow)
- When a modal IS open: DashboardGrid hooks should pause (user isn't looking at dashboard cards). The station's own hooks handle its data.
- `useVisibility` needs to understand both tab visibility AND modal state.

## Open Questions

### 1. Carnage Fund Balance: Bundle in API or Separate Call?

**What we know:** The `useCarnageData` hook currently fetches CarnageFundState (polling) AND CarnageSolVault balance (WS subscription). The CONTEXT.md decision says "Claude's discretion on whether to bundle fund balance in the API response."

**What's unclear:** Should the `/api/carnage-events` response include the current vault balance? This would require the API route to make an RPC call (`getBalance`), which is a server-side credit cost.

**Recommendation:** Keep vault balance as a separate client-side WS subscription. It changes on every tax collection (frequent), while carnage events are rare. Bundling would add latency to the API response and server-side credit costs. The existing WS subscription for vault balance is already efficient and real-time.

### 2. DashboardGrid Pausing Strategy

**What we know:** DashboardGrid calls all 6 hooks and is always mounted (behind modals). Station modals mount their own hook instances.

**What's unclear:** When a modal opens, should DashboardGrid's hooks fully pause (zero activity) or just reduce cadence? If they fully pause, the dashboard cards behind the modal will show stale data when the modal closes.

**Recommendation:** When a modal is open, DashboardGrid hooks should fully pause. When the modal closes, fire a burst-refresh to update all cards instantly. The 1-second refresh lag is imperceptible to users. This maximizes credit savings.

### 3. Sentry Error Reporting Threshold

**What we know:** CONTEXT.md says "Claude picks threshold for consecutive failures before Sentry reporting."

**Recommendation:** 3 consecutive failures of the same RPC call type. Why 3: First failure could be transient network blip. Second could be retry during recovery. Third suggests a persistent problem. Reset counter on any success.

### 4. WebSocket Fallback Strategy

**What we know:** CONTEXT.md says "Claude's discretion on polling fallback vs infinite WS retry."

**Recommendation:** After 5 failed WS reconnection attempts (with exponential backoff up to 30s), fall back to polling at the decided intervals (30s for epoch, 5s for pools). Continue WS reconnection attempts in the background. If WS reconnects successfully, disable polling fallback. This ensures data freshness even during prolonged WS outages.

### 5. Exponential Backoff Parameters

**Recommendation:** Base: 1 second. Multiplier: 2x. Max: 30 seconds. Jitter: +/- 20% random. Sequence: 1s, 2s, 4s, 8s, 16s, 30s, 30s, 30s... This matches the existing `useChartSSE` pattern (which uses the same 1s base, 2x multiplier, 30s max).
