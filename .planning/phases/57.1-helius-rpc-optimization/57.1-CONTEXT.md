# Phase 57.1: Helius RPC/WebSocket Optimization - Context

**Gathered:** 2026-02-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminate wasteful RPC credit consumption (~2,800 credits/hr per idle browser tab) by replacing polling with WebSocket subscriptions, batching account fetches, adding visibility-based pausing, and migrating client-side transaction parsing to existing Postgres queries. Target: <500 credits/hr per active tab. No new features -- pure infrastructure efficiency.

</domain>

<decisions>
## Implementation Decisions

### Data freshness intervals
- Pool prices: 5-second refresh (near real-time for active traders)
- Epoch state (current epoch, time remaining, carnage fund balance): 30-second refresh
- Staking data (staked balance, pending rewards, pool state): only fetch when Rewards Vat station modal is open
- All 4 pool accounts (CRIME/SOL, FRAUD/SOL, CRIME/PROFIT, FRAUD/PROFIT) batched in one `getMultipleAccounts` call every cycle, regardless of which pool is selected
- After successful swap transaction: immediate force-refresh of pool prices (don't wait for next 5s cycle)

### Carnage events migration
- Move from client-side `getSignaturesForAddress` + `getParsedTransaction` to Postgres API route (crank already writes events to DB)
- Show last 5 carnage events in the Cauldron station (compact list)
- No real-time push for new events -- next 30s poll cycle picks them up naturally
- Claude's discretion on API response format (raw vs formatted) and whether to bundle fund balance in the API response or keep as separate RPC call

### Connection resilience
- Silent reconnect on WebSocket disconnect -- no user-visible indicators, auto-reconnect with exponential backoff
- Claude's discretion on whether to fall back to polling after N failed WS reconnects
- RPC errors: silent retry for transient failures, report to Sentry after N consecutive failures of the same call (Claude picks threshold)

### Pausing behavior
- **Tab hidden:** Pause ALL data fetching immediately (zero RPC/WS activity). Maximum credit savings.
- **Tab return:** Immediate burst refresh -- fetch all stale data in one batch on visibility change. Fresh in <1s.
- **Modal-aware pausing:** Only the active station's hooks run. If Swap Station is open, staking/carnage hooks pause. If no modal is open, only epoch state polls (30s).
- **Pool prices are modal-scoped:** Dashboard cards and swap form both live inside Swap Station modal. Price fetching only active when Swap Station is the open modal.

### Claude's Discretion
- WebSocket fallback strategy (polling fallback vs infinite WS retry)
- Sentry error threshold (after how many consecutive failures to report)
- Carnage events API format (raw on-chain data vs display-ready)
- Whether carnage fund balance is bundled in events API or separate getBalance call
- Exact exponential backoff parameters for reconnection
- Which hooks use WebSocket subscriptions vs optimized polling (based on Helius capabilities research)

</decisions>
