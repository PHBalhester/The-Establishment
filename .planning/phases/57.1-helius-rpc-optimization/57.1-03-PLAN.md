  <name>Task 1: Migrate useCarnageEvents to Postgres API route</name>
  <files>app/hooks/useCarnageEvents.ts</files>
  <action>
Rewrite `useCarnageEvents.ts` to fetch from the `/api/carnage-events` Postgres route instead of doing client-side RPC transaction parsing. This is the single biggest credit savings in the entire phase (~900 credits/hr -> 0 credits/hr).

**What to remove:**
- All imports related to RPC parsing: `getConnection`, `DEVNET_PDAS`, `PROGRAM_IDS`
- The `SIGNATURES_TO_FETCH` constant
- The `getCarnageExecutedDiscriminator()` function and related vars
- The `parseCarnageExecutedEvent()` function
- The entire complex `fetchEvents` implementation (the 100+ line function with getSignaturesForAddress + getParsedTransaction + log parsing)

**What to keep:**
- The `CarnageEvent` interface (but update it to match the API response format)
- The `UseCarnageEventsResult` interface
- The `carnageActionLabel()` utility function (still used by CarnageCard)
- The `"use client"` directive
- The `MAX_EVENTS = 5` constant

**What to add:**
- `import { useVisibility } from "@/hooks/useVisibility"` for pausing

**New implementation:**

1. Update `CarnageEvent` interface to match the API response shape:
   ```typescript
   export interface CarnageEvent {
     epochNumber: number;
     txSignature: string;
     targetToken: string | null;   // "CRIME" or "FRAUD"
     path: string | null;          // "BuyOnly", "Burn", "BurnAndSell"
     crimeBurned: number;
     fraudBurned: number;
     solUsedForBuy: number;
     crimeBought: number | null;
     fraudBought: number | null;
     carnageSolBefore: number | null;
     carnageSolAfter: number | null;
     timestamp: string;            // ISO date string from Postgres
   }
   ```

2. Change poll interval to 30 seconds (CONTEXT.md says 30s for carnage data):
   ```
   const POLL_INTERVAL_MS = 30_000;
   ```

3. Call `useVisibility("carnage")` -- carnage events are only needed in CarnageStation modal and DashboardGrid (no modal open):
   ```
   const { isActive, onResume } = useVisibility("carnage");
   ```

4. New `fetchEvents` is trivially simple:
   ```typescript
   const fetchEvents = useCallback(async () => {
     try {
       const res = await fetch("/api/carnage-events");
       if (!res.ok) throw new Error(`HTTP ${res.status}`);
       const data: CarnageEvent[] = await res.json();
       if (!mountedRef.current) return;
       setEvents(data);
       setError(null);
     } catch (err) {
       if (!mountedRef.current) return;
       setError(err instanceof Error ? err.message : String(err));
     } finally {
       if (mountedRef.current) setLoading(false);
     }
   }, []);
   ```

5. **Visibility-aware polling:**
   - Use an effect that watches `isActive`.
   - When `isActive` is true: initial fetch + start interval.
   - When `isActive` becomes false: clear interval.
   - Register burst-refresh via `onResume`.

6. Update the JSDoc header to explain the migration: "Fetches carnage events from Postgres API route instead of client-side RPC parsing. This eliminated ~900 credits/hr of getSignaturesForAddress + getParsedTransaction calls."

**Update `carnageActionLabel` to work with path strings instead of action numbers:**
The old hook used numeric action codes (0=None, 1=Burn, 2=Sell). The new API returns path strings ("BuyOnly", "Burn", "BurnAndSell"). Update `carnageActionLabel` to accept a string path and return a readable label:
```typescript
export function carnageActionLabel(path: string | null): string {
  switch (path) {
    case "BuyOnly": return "Buy Only";
    case "Burn": return "Burn";
    case "BurnAndSell": return "Burn & Sell";
    default: return "Unknown";
  }
}
```

The CarnageCard component should be updated to pass `event.path` instead. Read CarnageCard.tsx first and update any references to old field names.

<task type="auto">
  <name>Task 2: Batch usePoolPrices initial fetch + add visibility pausing</name>
  <files>app/hooks/usePoolPrices.ts</files>
  <action>
Optimize `usePoolPrices.ts` to use `getMultipleAccountsInfo` for the initial batch fetch (1 RPC call instead of 4) and add visibility-based WebSocket subscription management.

**Changes to initial fetch:**

1. Replace the `for (const entry of POOL_ENTRIES)` loop that calls `program.account.poolState.fetch(poolPubkey)` individually for each pool.

2. Instead, use a single batch call:
   ```typescript
   const poolPubkeys = POOL_ENTRIES.map(e => e.pool);
   const accounts = await connection.getMultipleAccountsInfo(poolPubkeys, "confirmed");
   ```

3. Then iterate the results and decode each:
   ```typescript
   for (let i = 0; i < POOL_ENTRIES.length; i++) {
     const entry = POOL_ENTRIES[i];
     const accountInfo = accounts[i];
     if (!accountInfo) {
       // Pool account doesn't exist -- set error
       setPools(prev => ({ ...prev, [entry.label]: { ...prev[entry.label], loading: false, error: "Account not found" } }));
       continue;
     }
     const decoded = program.coder.accounts.decode("poolState", accountInfo.data);
     setPools(prev => ({
       ...prev,
       [entry.label]: {
         label: entry.label,
         address: entry.pool.toBase58(),
         reserveA: toNum(decoded.reserveA),
         reserveB: toNum(decoded.reserveB),
         mintA: decoded.mintA.toBase58(),
         mintB: decoded.mintB.toBase58(),
         loading: false,
         error: null,
       }
     }));
   }
   ```

**Changes for visibility pausing:**

4. Import `useVisibility` and call `useVisibility("swap")` -- pool prices are needed for SwapStation and DashboardGrid (no modal open).

5. **Visibility-aware WebSocket management:**
   - When `isActive` transitions to false: unsubscribe all 4 WS subscriptions (iterate `subIdsRef` and call `removeAccountChangeListener`).
   - When `isActive` transitions to true: do batch re-fetch via RPC first (burst-refresh), then re-subscribe all 4 WS subscriptions.
   - Track `isActive` transitions via a `prevIsActiveRef`.

6. Register burst-refresh callback via `onResume` to immediately batch-fetch on tab return.

**Preserve the exact same return type and exported interfaces.** The WebSocket callback for real-time updates stays exactly the same -- only the initial fetch changes from 4 calls to 1.

**Do NOT:**
- Change the WebSocket subscription pattern (it's already correct per the research).
- Remove the per-pool error handling (each pool can fail independently).
