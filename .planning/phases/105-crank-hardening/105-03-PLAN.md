  <action>
Create a new file `scripts/crank/lib/telegram.ts` — a zero-dependency Telegram alert module following the project's established pattern of raw fetch() for external services (same approach as lib/sentry.ts).

The module must implement:

**1. AlertContext interface:**
```typescript
export interface AlertContext {
  event: string;              // e.g. "CIRCUIT BREAKER TRIPPED"
  lastError: string;          // Last error message (will be truncated)
  epoch: number;              // Current epoch number
  walletBalanceSol: number;   // Wallet balance in SOL
  consecutiveErrors: number;  // Error count that triggered the alert
  uptimeSeconds: number;      // Process uptime via process.uptime()
}
```

**2. escapeHtml() helper:**
Escape `&`, `<`, `>` for safe HTML embedding of dynamic content (error messages may contain these). This is the minimal set needed for HTML parse_mode.

**3. buildAlertMessage() function:**
Build an HTML-formatted message with these fields:
- Bold header with event name
- Epoch number
- Wallet balance (3 decimal places)
- Consecutive error count
- Uptime in hours + minutes format
- Last error in `<code>` block, truncated to 200 characters
All dynamic content escaped with escapeHtml().

**4. Cooldown mechanism:**
- Module-level `let lastAlertTs = 0`
- `const ALERT_COOLDOWN_MS = 5 * 60 * 1000` (5 minutes)
- Check `Date.now() - lastAlertTs < ALERT_COOLDOWN_MS` before sending
- When cooldown active, log `[alert] Cooldown active, skipping Telegram alert` and return false
- Update `lastAlertTs` after successful send (even if API returned non-200)

**5. sendAlert() exported function:**
```typescript
export async function sendAlert(ctx: AlertContext): Promise<boolean>
```
- Check cooldown first (return false if active)
- Read `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` from `process.env`
- If either is missing, log `[alert] TELEGRAM_BOT_TOKEN or TELEGRAM_CHAT_ID not set, skipping alert` and return false (graceful disable)
- POST to `https://api.telegram.org/bot${botToken}/sendMessage` with JSON body: `{ chat_id, text: buildAlertMessage(ctx), parse_mode: "HTML" }`
- NEVER log the full URL (contains bot token -- Pitfall 3 from RESEARCH.md)
- On success (res.ok): log `[alert] Telegram alert sent successfully` and return true
- On non-200: log `[alert] WARNING: Telegram API returned ${res.status}` and return false
- On fetch error: log `[alert] WARNING: Telegram send failed: ${error_first_100_chars}` and return false
- NEVER throw. NEVER rethrow. The crank must not be affected by alerting failures.

**6. File header:**
Add JSDoc explaining the module purpose, the zero-dependency approach, and the best-effort guarantee.

Use HTML parse_mode (NOT MarkdownV2 -- Pitfall 2 from RESEARCH.md).

Integrate the Telegram alert module into crank-runner.ts:

**1. Add import at top of file (with other imports):**

```typescript
import { sendAlert } from "./lib/telegram";
```

**2. Track start time for uptime calculation:**
Add `const crankStartMs = Date.now();` near the top of `main()`, right after the banner is printed (around line 373). This gives precise uptime rather than process.uptime() which includes startup loading.

**3. Fire alert when circuit breaker trips:**
In the catch block (around line 628-634), AFTER the circuit breaker threshold check but BEFORE the `break` statement, add the alert call:

```typescript
if (consecutiveErrors >= CIRCUIT_BREAKER_THRESHOLD) {
  console.error(
    `[crank] CRITICAL: Circuit breaker tripped after ${CIRCUIT_BREAKER_THRESHOLD} consecutive errors. Halting.`
  );

  // Send Telegram alert (best-effort, non-blocking relative to halt)
  const balance = await provider.connection.getBalance(
    provider.wallet.publicKey
  ).catch(() => 0);

  await sendAlert({
    event: "CIRCUIT BREAKER TRIPPED",
    lastError: errStr,
    epoch: epochState?.currentEpoch ?? 0,
    walletBalanceSol: balance / LAMPORTS_PER_SOL,
    consecutiveErrors,
    uptimeSeconds: (Date.now() - crankStartMs) / 1000,
  });

  break;
}
```

Note: `epochState` may be stale (from the last successful read at the top of the cycle) or from before the error. That's fine -- the alert is informational.

The `getBalance` call is wrapped in `.catch(() => 0)` because the RPC itself might be the failure causing the circuit breaker trip.

**4. Add env var documentation in the file header:**
In the "Optional env vars" section of the JSDoc at the top of the file (around line 30-32), add:
```
 *   TELEGRAM_BOT_TOKEN      - Telegram bot token for crank alerts (optional)
 *   TELEGRAM_CHAT_ID        - Telegram chat ID for crank alerts (optional)
```

**5. Do NOT add alerts for any other event:**
Per CONTEXT.md: "Alert events: circuit breaker trip only". No low-balance alerts, no VRF recovery alerts, no epoch stall alerts.
