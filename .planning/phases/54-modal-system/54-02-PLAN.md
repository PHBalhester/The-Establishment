  <action>
Create the core `<dialog>`-based modal wrapper that syncs with ModalProvider state and handles all animation/interaction logic.

**Component interface:**
```typescript
interface ModalShellProps {
  /** Display name shown in the header (e.g., "Swap Machine", "Carnage Cauldron") */
  title: string;
  /** Maximum width for this station's modal (e.g., '1100px' for swap, '500px' for settings) */
  maxWidth?: string;
  /** Modal body content -- rendered inside the scrollable area */
  children: React.ReactNode;
}
```

**Implementation requirements:**

1. **Dialog ref + state sync:** Use `useRef<HTMLDialogElement>` to hold the dialog DOM reference. Use `useModal()` hook to get `state.activeStation`, `state.irisOrigin`, and `closeModal`. Use a `useEffect` to sync: when `activeStation` becomes non-null AND this ModalShell should be visible, call `dialogRef.current.showModal()`. When activeStation becomes null, trigger the close animation sequence.

   **Critical: ModalShell is a singleton.** There is ONE ModalShell rendered in the app (inside ModalProvider or near the top of the tree). It receives the content for whichever station is currently active. The content switching happens via a parent component or the provider pattern. ModalShell does not conditionally render -- it always exists in the DOM as a `<dialog>` element.

2. **Iris origin setup:** Before calling `showModal()`, set CSS custom properties `--iris-x` and `--iris-y` on the dialog element from `state.irisOrigin`. This positions the iris-open clip-path animation at the clicked scene object. If no irisOrigin (e.g., keyboard-triggered), default to center of viewport (50vw, 50vh).

3. **Close animation sequence (RESEARCH.md Pattern 3):**
   - When closeModal is called (from X button, Escape, or backdrop click), do NOT call `dialog.close()` immediately.
   - Add the `closing` CSS class to the dialog element.
   - Listen for `animationend` event (with `{ once: true }`).
   - In the callback: remove `closing` class, call `dialog.close()`, restore focus to `triggerRef.current` (from useModal), and remove `modal-open` class from body.

4. **Native Escape handling (RESEARCH.md Pitfall 3):**
   - Listen for the dialog's native `cancel` event (fired when user presses Escape).
   - Call `e.preventDefault()` to prevent the instant close (which would skip the animation).
   - Then call `closeModal()` which triggers the animated close sequence.
   - Do NOT add a separate `keydown` listener for Escape -- that would double-fire.

5. **Backdrop click detection (RESEARCH.md Pattern 4):**
   - Add `onClick` handler to the `<dialog>` element.
   - Check `if (e.target === e.currentTarget)` -- when clicking the backdrop, the event target IS the dialog element itself (not a child).
   - If true, call `closeModal()`.
   - The inner content div uses `onClick={e => e.stopPropagation()}` is NOT needed because the target check handles it.

6. **Station crossfade (RESEARCH.md Pitfall 7):**
   - When switching stations (activeStation changes from one StationId to another), do NOT close and reopen the dialog.
   - Instead, apply the `modal-content-exit` class to the outgoing content, wait for animation, then swap content and apply `modal-content-enter`.
   - Track the `previousStation` via useRef to detect station switches vs fresh opens.
   - On a fresh open (previous was null): use iris-open animation.
   - On a station switch (previous was different station): skip iris, use crossfade.
   - The dialog stays open throughout the switch.

7. **Layout structure (HTML):**
   ```
   <dialog class="modal-shell" style="--iris-x: Xpx; --iris-y: Ypx; max-width: {maxWidth}">
     <div class="modal-chrome">
       <header class="modal-header">
         <h2>{title}</h2>
         <ModalCloseButton onClick={closeModal} />
       </header>
       <div class="modal-body">
         {children}
       </div>
     </div>
   </dialog>
   ```

8. **Dialog base styles:** The `<dialog>` element itself should have:
   - `padding: 0; border: none; background: transparent;` -- strip browser defaults
   - `position: fixed; margin: auto;` -- center in viewport (browser default for modal dialogs)
   - `max-height: 85vh;` -- prevent full-viewport height
   - `width: 90vw;` -- responsive base, constrained by maxWidth prop
   - The dialog element should NOT have padding (per RESEARCH.md Pitfall 2 -- clip-path coordinates need direct viewport mapping)

9. **will-change optimization:** Add `will-change: clip-path` to the dialog element via JS just before calling `showModal()`, and remove it after the iris-open animation completes (listen for `animationend` with `{ once: true }`). This hints the browser to optimize the clip-path animation. Do NOT leave it permanently (wastes GPU memory).

**Why a singleton ModalShell:**
The single-dialog approach prevents the scene flash described in RESEARCH.md Pitfall 7. If each station had its own dialog, switching would require close+open with a gap. With one dialog, we swap content inside it.

**Note on title/maxWidth/children:** In this plan, the ModalShell component takes these as props. The actual wiring of which station renders which content happens in Phase 55/56. For now, the ModalShell is a reusable container.

Actually, ModalShell needs to know which content to render based on activeStation. But station content doesn't exist yet (Phase 56). So the architecture should be:

**Why CSS-only chrome (not image assets):**
- CONTEXT.md explicitly chose CSS-only first with architecture that supports future image-based frame swap
- Gradients + box-shadows render crisply at any DPI/viewport size
- No additional network requests for frame textures
- The decorative bolts (radial-gradient circles) add the Victorian mechanical feel

**Why decorative bolts as positioned elements:**
- CSS ::before/::after only gives 2 pseudo-elements per container
- 4 corner bolts need either 4 small spans or nested containers
- Small positioned spans with `pointer-events: none` are the simplest approach
