# Phase 52.1: Canonical Mint Ordering Fix

## Problem Statement

The AMM enforces canonical mint ordering (`mint_a < mint_b` by raw `[u8; 32]` byte comparison) when initializing pools. Our Tax Program and client code assume a fixed ordering:
- PROFIT pools: `mint_a = faction token (CRIME/FRAUD)`, `mint_b = PROFIT`
- SOL pools: `mint_a = WSOL (NATIVE_MINT)`, `mint_b = faction token`

The CRIME/PROFIT pool's actual canonical ordering is **reversed** (`mint_a = PROFIT, mint_b = CRIME`) because PROFIT's raw bytes sort before CRIME's. This causes `InvalidMint` errors (AMM error 6013) for all CRIME/PROFIT swaps.

### Root Cause

Phase 51 redeploy created new mint addresses. The old CRIME mint (`6PyHby...`) sorted before the old PROFIT mint (`J4CzJ5...`) in raw bytes. The new CRIME mint (`F65o4z...`, first byte `0xD1`) sorts AFTER the new PROFIT mint (`8y7Mat...`, first byte `0x76`).

### Current Pool Ordering (devnet)

| Pool | mint_a | mint_b | Code assumes | Status |
|------|--------|--------|---|---|
| CRIME/SOL | WSOL (0x06) | CRIME (0xD1) | A=WSOL | Works |
| FRAUD/SOL | WSOL (0x06) | FRAUD (0x68) | A=WSOL | Works |
| CRIME/PROFIT | PROFIT (0x76) | CRIME (0xD1) | A=CRIME | **BROKEN** |
| FRAUD/PROFIT | FRAUD (0x68) | PROFIT (0x76) | A=FRAUD | Works |

### Decision: Scope of Fix

- **PROFIT pools**: Fix code to be ordering-agnostic (MUST do -- one is broken, code is simpler)
- **SOL pools**: Use vanity addresses at mainnet (safer -- SOL pool code is complex with tax distribution, WSOL wrapping, staking/carnage integration, 299 tested paths)
- **Deploy-time safety**: Add assertion in initialize.ts that validates canonical ordering before pool creation

## Wave 1: Tax Program On-Chain Fix

### Task 1.1: Make `swap_profit_buy.rs` ordering-agnostic

**File**: `programs/tax-program/src/instructions/swap_profit_buy.rs`

**What changes** (handler logic only, NOT the account struct):

1. **Read `pool.mint_a` from pool data** (bytes [9..41]) to detect canonical ordering
2. **Determine if ordering is reversed**: compare `pool.mint_a` with `ctx.accounts.mint_a.key()`
3. **Conditionally set direction**:
   - Normal (pool.mint_a matches passed mint_a): `direction = 0` (AtoB, input=A=faction, output=B=PROFIT)
   - Reversed (pool.mint_a doesn't match): `direction = 1` (BtoA, input=B=faction, output=A=PROFIT)
4. **Conditionally interpret reserves for output floor**:
   - Normal: `calculate_output_floor(reserve_a, reserve_b, ...)` (in=A=faction, out=B=PROFIT)
   - Reversed: `calculate_output_floor(reserve_b, reserve_a, ...)` (in=B=faction, out=A=PROFIT)
5. **Conditionally snapshot output balance**:
   - Normal: `output_before = user_token_b.amount` (output=B=PROFIT)
   - Reversed: `output_before = user_token_a.amount` (output=A=PROFIT)
6. **Conditionally reload correct output account** for actual_output calculation

**Why the account struct doesn't change**: The struct has `mint_a`, `mint_b`, `pool_vault_a`, `pool_vault_b`, `user_token_a`, `user_token_b` — these are positional labels, not semantic. The client will pass them in canonical order (matching the pool), and the handler logic adapts.

### Task 1.2: Make `swap_profit_sell.rs` ordering-agnostic

**File**: `programs/tax-program/src/instructions/swap_profit_sell.rs`

Same approach as 1.1 but mirrored:
- Normal: `direction = 1` (BtoA, input=B=PROFIT, output=A=faction)
- Reversed: `direction = 0` (AtoB, input=A=PROFIT, output=B=faction)
- Reserve and snapshot interpretation flipped accordingly


Verify existing 74 tests still pass. The ordering detection is additive logic -- existing tests use FRAUD/PROFIT pool ordering (which is "normal"), so they exercise the non-reversed path.

## Wave 2: Client-Side Fixes

### Task 2.1: Fix `swap-builders.ts` for CRIME/PROFIT canonical ordering

**File**: `app/lib/swap/swap-builders.ts`

When `isCrime=true` for PROFIT pool operations, the client must pass accounts in canonical order (PROFIT as A, CRIME as B):

**`buildProfitBuyTransaction`** (lines 508-521):
- Currently: `mintA: tokenMint (CRIME)`, `mintB: MINTS.PROFIT`
- Fix: When `isCrime`, swap to `mintA: MINTS.PROFIT`, `mintB: tokenMint (CRIME)`
- Also swap: `poolVaultA/B`, `userTokenA/B`
- Hook accounts: Swap A/B hook ordering (input is now B, output is now A)

**`buildProfitSellTransaction`** (lines 620-633):
- Same swap logic for `isCrime`
- Hook accounts: Input is now A (PROFIT), output is now B (CRIME)

**Implementation approach**: Add a helper `getCrimePoolCanonicalAccounts()` that returns the correctly ordered accounts, or use a conditional swap at the top of each builder.

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Tax Program upgrade breaks existing swaps | Account structs unchanged; handler-only logic; run full test suite |
| Wrong direction sent to AMM | Detect ordering from on-chain pool data (source of truth), not hardcoded |
| Reserve interpretation wrong | Follows same conditional as direction -- tied to single `is_reversed` boolean |
| FRAUD/PROFIT pool regression | Currently "normal" ordering -- default path unchanged |
| Hook account ordering wrong for CRIME/PROFIT | Client swaps hook A/B order to match canonical vault positions |

## Estimated Cost

- Tax Program upgrade: ~0.001 SOL (transaction fee only)
- No new accounts, no new PDAs, no pool reinitialization
- Total devnet SOL cost: negligible
