# Phase 106.1: Cluster-aware Transaction Submission for Devnet Testing - Research

**Researched:** 2026-03-26
**Domain:** Solana frontend transaction submission (wallet-adapter, devnet RPC behavior)
**Confidence:** HIGH

## Summary

This phase addresses a well-understood and narrowly-scoped problem: devnet transaction submission fails silently because (a) Phantom's `signAndSendTransaction` routes through Phantom's own RPC which drops devnet transactions, and (b) Solana's devnet RPC rejects v0 (versioned) transaction simulation with "Blockhash not found". The project already solved both problems in `useProtocolWallet.ts` (sign-then-send on devnet) and `multi-hop-builder.ts` (skipPreflight on devnet), but the `skipPreflight: true` override is not centralized -- some callsites pass `skipPreflight: false` explicitly, which breaks them on devnet.

The fix is a single-line change in `useProtocolWallet.ts`: in the devnet branch of `wrappedSendTransaction`, force `skipPreflight: true` regardless of what callers pass. Then remove the redundant `isDevnet` logic in `multi-hop-builder.ts` since useProtocolWallet now handles it centrally. No mainnet code is touched.

**Primary recommendation:** Override `skipPreflight` to `true` in useProtocolWallet's devnet `sendRawTransaction` call (line 116), and remove the redundant `isDevnet` check in `multi-hop-builder.ts` (lines 426-432).

### Anti-Patterns to Avoid
- **Per-callsite devnet checks:** Adding `isDevnet ? skipPreflight: true : false` at each callsite is fragile and redundant. The whole point of useProtocolWallet is to absorb cluster differences.
- **Changing mainnet behavior:** The user explicitly decided: "Mainnet path untouched. Callers' opts flow through as-is." Do not add any logic to the mainnet branch.

## Common Pitfalls

### Pitfall 1: Forgetting that skipPreflight means "confirmed" does NOT mean "succeeded"
**What goes wrong:** When `skipPreflight: true`, Solana validators accept any well-formed transaction. The TX lands on-chain and gets "confirmed" even if it fails (e.g., insufficient funds, wrong PDA). The `confirmation.value.err` field must be checked.
**Why it happens:** Developers assume "confirmed" = "success" because that is true when preflight is enabled (preflight catches errors before submission).
**How to avoid:** All callsites already use `pollTransactionConfirmation` which returns `{ err }`. The callers (useSwap, useStaking, multi-hop-builder) all check `confirmation.err`. No additional work needed -- just verify this during code review.
**Warning signs:** Swap appears to succeed but tokens don't move.

### Pitfall 2: Breaking the mainnet Blowfish-safe path
**What goes wrong:** If someone accidentally applies the skipPreflight override to the mainnet branch, Phantom will still simulate the TX locally but it won't get server-side preflight validation. More importantly, if someone changes the mainnet branch to use sign-then-send, Phantom's Blowfish security check flags the dApp as potentially malicious.
**Why it happens:** Refactoring that doesn't respect the cluster branching.
**How to avoid:** The fix ONLY modifies line 116 in the devnet branch. The mainnet branch (line 100) remains a single passthrough: `return walletSendTransaction(tx, connection, opts)`.
**Warning signs:** Users on mainnet see "This dApp could be malicious" warning in Phantom.

### Pitfall 3: v0 transaction simulation rejects on devnet
**What goes wrong:** Solana devnet validators occasionally return "Blockhash not found" when simulating v0 (versioned) transactions during preflight, even with a fresh blockhash. This is a known devnet RPC issue.
**Why it happens:** Devnet validator infrastructure is less reliable than mainnet. The simulation endpoint sometimes rejects valid blockhashes.
**How to avoid:** `skipPreflight: true` bypasses simulation entirely. The transaction is submitted directly to the leader.
**Warning signs:** Transactions fail immediately with "Blockhash not found" before even reaching the chain.

### Pitfall 4: maxRetries inconsistency
**What goes wrong:** Different callsites use different `maxRetries` values (useSwap: 2, multi-hop: 3, BuyForm/SellForm: none). On devnet, the retry behavior might differ unexpectedly.
**How to avoid:** This is Claude's discretion per the CONTEXT.md. Recommendation: leave `maxRetries` as caller-controlled. The central override only forces `skipPreflight`. Standardizing maxRetries is a separate concern and not urgent.

## Code Examples

### The Core Fix (useProtocolWallet.ts devnet branch)

```typescript
// Current (BROKEN for callers passing skipPreflight: false):
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: sendOptions.skipPreflight,
  preflightCommitment: sendOptions.preflightCommitment ?? connection.commitment ?? undefined,
  maxRetries: sendOptions.maxRetries,
  minContextSlot: sendOptions.minContextSlot,
});

// Fixed (force skipPreflight on devnet):
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: true, // Forced on devnet: v0 simulation returns "Blockhash not found"
  preflightCommitment: sendOptions.preflightCommitment ?? connection.commitment ?? undefined,
  maxRetries: sendOptions.maxRetries,
  minContextSlot: sendOptions.minContextSlot,
});
```

Source: Existing `useProtocolWallet.ts` line 116 + CONTEXT.md decisions.

### The Cleanup (multi-hop-builder.ts executeAtomicRoute)

```typescript
// Current (redundant isDevnet check):
const isDevnet = process.env.NEXT_PUBLIC_CLUSTER !== "mainnet";
```

## Open Questions

1. **maxRetries default on devnet**
   - What we know: Callers currently pass different values (2 or 3). Solana RPC's default is unlimited retries.
   - What's unclear: Whether a single standardized value is better than caller-controlled.
   - Recommendation (Claude's discretion per CONTEXT.md): Leave caller-controlled. The central override only forces `skipPreflight`. If a caller wants `maxRetries: 2`, that's their business. This keeps the change minimal and avoids second-order effects. If devnet proves unreliable, a default `maxRetries` for devnet can be added later.
