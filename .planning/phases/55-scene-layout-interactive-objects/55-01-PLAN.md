**Create `app/components/scene/scene-data.ts`:**

This is the centralized metadata module that bridges SCENE_DATA overlay keys (kebab-case asset names from image-data.ts) to ModalProvider StationId values. It defines the tab order, display labels, and clip-path polygons for all 6 stations.

Define a `StationMeta` interface:
```typescript
interface StationMeta {
  /** Key into SCENE_DATA.overlays (e.g., 'carnage-cauldron') */
  overlayId: string;
  /** StationId for useModal().openModal() */
  stationId: StationId;
  /** Tooltip label text */
  label: string;
  /** CSS clip-path polygon() for restricting clickable area to visible pixels.
   *  Coordinates are percentages relative to the overlay bounding box. */
  clipPath: string;
}
```

Export a `STATIONS` array ordered by **tab priority** (this IS the tab order -- DOM render order determines keyboard navigation):
1. `connect-wallet` -> `wallet` -> "Connect Wallet"
2. `swap-station` -> `swap` -> "Swap Machine"
3. `carnage-cauldron` -> `carnage` -> "Carnage Cauldron"
4. `rewards-vat` -> `staking` -> "Rewards Vat"
5. `documentation-table` -> `docs` -> "Documentation Table"
6. `settings` -> `settings` -> "Settings"

**Clip-path polygons:** These are approximate outlines of the visible pixels in each overlay. The executor MUST examine each overlay image (load the WebP files from `public/scene/overlays/`) and trace ~8-15 key contour points as percentage coordinates. The polygon uses `%` values relative to the overlay's bounding box.

For the swap-station (no asset exists), use a simple rectangular clip-path: `polygon(10% 5%, 90% 5%, 90% 95%, 10% 95%)`.

For the other 5 overlays, open each image file, identify the approximate shape of the visible machine content, and express as a polygon. These don't need to be pixel-perfect -- just rough outlines that exclude large transparent regions. Example approach:
- `connect-wallet`: The lever/pipe machine occupying roughly the center-right of its bounding box
- `carnage-cauldron`: The cauldron shape (wider at top, narrow at bottom base)
- `rewards-vat`: Tall vertical vat shape
- `documentation-table`: Table/desk shape (wider at desk level)
- `settings`: Gear/mechanism shape

If the images are not easily viewable programmatically, use generous polygons (cover ~80% of the bounding box) that exclude the obvious transparent corners/edges. Better to be slightly too large (catching some transparent pixels) than too small (missing visible content).

**Why this file matters:** It decouples station configuration from component logic. When new stations are added or overlay assets change, only this file needs updating.

Import `StationId` type from `@/components/modal/ModalProvider`.

```tsx

interface StationTooltipProps {
  /** Unique ID for aria-describedby linking */

  /** Station name label */
  label: string;
}
```

Render a `<span>` with:
- `id={id}` and `role="tooltip"` for ARIA compliance
- Positioned absolutely below the button center: `absolute left-1/2 -translate-x-1/2` with a bottom offset (e.g., `top-[calc(100%+8px)]` to sit below the overlay)
- Styled with factory theme: `bg-factory-surface/90 text-factory-text text-sm font-heading px-3 py-1.5 rounded whitespace-nowrap`
- `pointer-events-none` so it doesn't interfere with the button's click area
- Hidden by default: `opacity-0`
- Visible on hover/focus: `group-hover:opacity-100 group-focus-visible:opacity-100`
- Smooth transition: `transition-opacity duration-200`
- Subtle brass border: `border border-factory-border`

**Why separate component:** Keeps SceneStation focused on interaction logic while tooltip rendering is isolated and testable.

Create `app/components/scene/SceneStation.tsx` -- the core interactive component of this phase. This is a `<button>` element that wraps a FactoryOverlay image (or SwapPlaceholder for unavailable assets) and provides all interactivity: hover glow, click feedback, tooltip, focus indicator, and modal opening.

**Component interface:**
```typescript
'use client';

import { useCallback } from 'react';
import Image from 'next/image';
import { SCENE_DATA } from '@/lib/image-data';
import { useModal } from '@/hooks/useModal';
import { StationTooltip } from './StationTooltip';
import { SwapPlaceholder } from './SwapPlaceholder';
import type { StationMeta } from './scene-data';
import type { StationId } from '@/components/modal/ModalProvider';

interface SceneStationProps {
  /** Station metadata from STATIONS array */
  station: StationMeta;
}
```

**Component structure (two-layer approach for clip-path + filter conflict):**

The RESEARCH.md identifies a critical conflict: `clip-path` clips `filter: drop-shadow()`, cutting off the glow halo. The recommended approach (RESEARCH.md "Simple approach") is to apply clip-path to the button and accept tight glow. This is the approach to use.

```
<button>         -- positioned absolute, HAS clip-path AND filter effects
                    clip-path clips the glow slightly but it looks intentional (tight halo)
  <Image />      -- the overlay image (or SwapPlaceholder for unavailable)
  <StationTooltip /> -- tooltip label (pointer-events-none, positioned outside clip via overflow-visible)
</button>
```

Wait -- the tooltip would also be clipped by clip-path on the button. This is a problem.

**Revised structure:**
```
<div>              -- positioned absolute (from SCENE_DATA %), pointer-events-none
  <button>         -- full-size within div, HAS clip-path, HAS filter effects
    <Image />      -- the overlay image (via FactoryOverlay pattern, but inline for button semantics)
  </button>
  <StationTooltip />  -- positioned relative to the outer div, NOT clipped
</div>
```

Actually, a cleaner approach: The tooltip can be positioned using a wrapper div that does NOT have clip-path. The button has clip-path for the hit area and filter for the glow. The tooltip sits as a sibling outside the button but inside a positioning wrapper.

**Final structure:**
```tsx
<div
  className="group absolute z-overlays"
  style={{
    left: `${overlay.left}%`,
    top: `${overlay.top}%`,
    width: `${overlay.widthPct}%`,
    height: `${overlay.heightPct}%`,
  }}
>
  <button
    type="button"
    onClick={handleClick}
    aria-label={station.label}
    aria-describedby={tooltipId}
    className="
      w-full h-full cursor-pointer
      transition-[filter,transform] duration-150 ease-out
      hover:drop-shadow-[0_0_16px_rgba(212,160,74,0.6)] hover:brightness-120
      focus-visible:drop-shadow-[0_0_16px_rgba(212,160,74,0.6)] focus-visible:brightness-120
      focus-visible:outline-none
      active:scale-95 active:duration-100
    "
    style={{ clipPath: station.clipPath }}
  >
    {/* Overlay image OR placeholder */}
    {overlay.available ? (
      <Image
        src={overlay.src}
        alt=""  {/* aria-label on button is the accessible name; image is decorative */}
        fill
        quality={82}
        placeholder="blur"
        blurDataURL={overlay.blurDataURL}
        loading="lazy"
        className="object-contain pointer-events-none"
      />
    ) : (
      <SwapPlaceholder />
    )}
  </button>
  <StationTooltip id={tooltipId} label={station.label} />
</div>
```

**Key implementation details:**

1. **Positioning:** The outer `<div>` uses SCENE_DATA percentage coordinates (left, top, widthPct, heightPct) from `SCENE_DATA.overlays[station.overlayId]`. Read the overlay data from SCENE_DATA.

2. **Click handler:** Use `e.clientX`/`e.clientY` (actual click point) instead of button center for the iris animation origin. This feels more natural -- the iris opens from exactly where the user clicked. Pass these coordinates to `openModal(station.stationId, { x: e.clientX, y: e.clientY })`. For keyboard activation (Enter/Space), fall back to button center via `getBoundingClientRect()` since there are no mouse coordinates.

3. **Hover glow:** Use Tailwind arbitrary value `drop-shadow-[0_0_16px_rgba(212,160,74,0.6)]` combined with `brightness-120` (or `brightness-[1.2]` if 120 is not a valid Tailwind value -- check Tailwind v4 docs, brightness utilities use percentage scale: `brightness-125` = 1.25). The glow uses the factory-accent color (#d4a04a) at 60% opacity. Tailwind v4 stacks multiple filter utilities into a single CSS `filter` property automatically.

4. **Active press:** `active:scale-95` with `active:duration-100` for the mechanical press feel. The transition on the base state is `duration-150` for smooth glow appear/disappear.

5. **Focus indicator:** `focus-visible:` matches the hover glow exactly (same drop-shadow + brightness). This satisfies CONTEXT.md decision that focus and hover states are visually identical. `focus-visible:outline-none` removes the browser default.

6. **Clip-path:** Applied via inline `style={{ clipPath: station.clipPath }}` from scene-data.ts. This restricts both the visual boundary and the pointer event hit area of the button.

7. **Group pattern for tooltip:** The outer `<div>` has `className="group"` so the StationTooltip can use `group-hover:opacity-100` and `group-focus-within:opacity-100` (use `group-focus-within` instead of `group-focus-visible` because focus-visible is on the button child, not the group div itself).

8. **Image rendering:** For available overlays, render the Image directly inside the button (do NOT use FactoryOverlay component -- it wraps in a div with absolute positioning that would conflict with the button structure). Copy the Image props from FactoryOverlay but mark `alt=""` since the button's aria-label is the accessible name. Set `pointer-events-none` on the image so clicks go to the button.

9. **Swap placeholder:** For unavailable overlays (swap-station), render `<SwapPlaceholder />` instead of Image.

10. **Tooltip ID:** Generate as `tooltip-${station.stationId}` for the `aria-describedby` link.

**Why NOT use FactoryOverlay component:** FactoryOverlay wraps the image in a positioned `<div>` with its own absolute positioning and z-index. SceneStation needs the image directly inside a `<button>` for proper interaction semantics. The FactoryOverlay component was designed as a standalone positioned element -- here we need the image as a child of the button. Copying the Image props is cleaner than fighting FactoryOverlay's wrapper structure.

**Important:** The component must be `'use client'` since it uses useModal hook and event handlers.
