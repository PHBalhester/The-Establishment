**Part A: Create `scripts/test/pathway2-fill.ts`**

Model after `scripts/test/pathway1-test.ts` but optimized for FILLING both curves to capacity with organic-looking traffic.

Key design decisions:
- **8-10 wallets** generated and funded from devnet wallet
- **Fund each wallet ~1.2 SOL** (need ~5.06 SOL per curve = ~10.12 total + gas)
- **Mixed operations**: buys AND sells interleaved across both CRIME and FRAUD curves
- **Buy amounts**: varied between 0.1-1.5 SOL per buy (randomized)
- **Sell amounts**: small (10-20% of held tokens) to create organic sell pressure without depleting
- **Randomized spacing**: 1-5 seconds between operations for organic feel
- **Alternating curves**: CRIME buy, FRAUD buy, CRIME sell, FRAUD buy, etc.
- **Target ~30 seconds** total fill time
- **Parallel execution**: multiple wallets buying simultaneously
- **Wallet cap awareness**: MAX_TOKENS_PER_WALLET is 20M tokens (20e12 base units). Each wallet stays well under by distributing buys across 8-10 wallets.
- **Progress logging**: Print running total of SOL deposited per curve, percentage filled

Script must:
1. Read addresses from `deployments/devnet.json` (program IDs, mints, curve PDAs)
2. Load devnet wallet from `keypairs/devnet-wallet.json`
3. Generate N wallets, fund them
4. Create ATAs for CRIME and FRAUD (Token-2022) for each wallet
5. Execute buy/sell operations in a pattern that fills both curves
6. After each operation, check curve state to see if target reached
7. Stop when both curves show `tokens_sold >= TOTAL_FOR_SALE` (460M tokens per curve)
8. Print summary: total SOL in, tokens distributed, time elapsed

Use the `getHookAccounts()` pattern from pathway1-test.ts (4 accounts: metaList, wlSource, wlDest, hookProgram).

Use Anchor program loading pattern from pathway1-test.ts.

Handle errors gracefully -- if a buy fails (e.g., curve full, wallet cap), log and continue with next operation.

Do NOT change WebSocket subscription logic -- only the fallback polling interval.
