**Add new section: "## ATA Derivation Differences"**

1. **Explanation paragraph**:
Explain that ATA derivation differs between token programs because the token_program_id is part of the PDA seeds. A WSOL ATA and an IPA ATA for the same wallet have DIFFERENT addresses.

2. **Derivation Formula**:
```
ATA PDA seeds: [wallet_address, token_program_id, mint_address]
Program: Associated Token Program (ATokenGPvbdGVxr1b2hvZbsiqW5xWH25efTNsLJA8knL)
```

3. **Practical Implications** table:

| Token | Token Program ID for Derivation | Notes |
|-------|--------------------------------|-------|
| IPA | TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb | Token-2022 |
| IPB | TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb | Token-2022 |
| OP4 | TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb | Token-2022 |
| WSOL | TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA | SPL Token |

4. **Code Example** (from research):
```rust
// For WSOL (SPL Token)
let wsol_ata = get_associated_token_address_with_program_id(
    &wallet,
    &NATIVE_MINT,
    &spl_token::id(),  // SPL Token program
);

// For IPA (Token-2022)
let ipa_ata = get_associated_token_address_with_program_id(
    &wallet,
    &IPA_MINT,
    &spl_token_2022::id(),  // Token-2022 program
);
```

5. **Common Pitfall Warning**:
> **Warning: ATA Address Mismatch**
>
> Using the wrong token_program_id in ATA derivation will derive a different (non-existent) address.
> - Always verify token program before ATA derivation
> - Use `get_associated_token_address_with_program_id`, not the simpler variant

**Add new section: "## Security Threat Model"**

**Introduction:**
This threat model covers security implications specific to token program choices. All threats relate to the T22/SPL mixed architecture.

**Threat Table format:**

| ID | Threat | Likelihood | Impact | Current Mitigation | Status |
|----|--------|------------|--------|-------------------|--------|

**Threats to document:**

1. **TM-01: Unauthorized WSOL vault extraction**
   - Likelihood: LOW
   - Impact: CRITICAL (loss of pool liquidity)
   - Mitigation: AMM vaults are PDAs owned by AMM program; require authorized CPI with Tax Program PDA signature
   - Status: Mitigated by design

2. **TM-02: Direct WSOL injection to vaults**
   - Likelihood: MEDIUM
   - Impact: LOW (benign - increases pool liquidity)
   - Mitigation: None needed - injections benefit liquidity providers
   - Status: Accepted risk

3. **TM-03: Bypass whitelist via WSOL path**
   - Likelihood: LOW
   - Impact: MEDIUM (could enable unauthorized trading)
   - Mitigation: AMM swap instructions require Tax Program PDA signature; users cannot call AMM directly
   - Status: Mitigated by design

4. **TM-04: ATA derivation confusion attack**
   - Likelihood: LOW
   - Impact: MEDIUM (failed transactions, UX issues)
   - Mitigation: Client libraries must use correct token_program_id; testing validates all paths
   - Status: Mitigated by implementation

5. **TM-05: Mixed CPI program ID errors**
   - Likelihood: MEDIUM (during development)
   - Impact: HIGH (transaction failures)
   - Mitigation: Pass separate token_program accounts for each token side; extensive testing
   - Status: Mitigated by design

6. **TM-06: Extension size parsing errors**
   - Likelihood: LOW
   - Impact: MEDIUM (account deserialization failures)
   - Mitigation: Use StateWithExtensions::<Account>::unpack, not Account::unpack
   - Status: Mitigated by implementation
