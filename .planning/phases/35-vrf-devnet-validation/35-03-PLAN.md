  <name>Task 1: Build security test and swap verification modules</name>
  <files>scripts/vrf/lib/security-tests.ts, scripts/vrf/lib/swap-verifier.ts</files>
  <action>
  Create two new modules that test the security properties of the VRF integration and verify tax rates propagate to swaps.

  **scripts/vrf/lib/security-tests.ts** -- VRF security edge case tests:

  Each test function returns `{ passed: boolean, details: string }`.

  1. `testAntiReroll(provider, epochProgram, accounts)`:
     - Prerequisite: An epoch transition is already in progress (VRF committed, pending consume).
     - Create a DIFFERENT randomness account (wrongRngKp).
     - Attempt `consume_randomness` with the WRONG randomness account.
     - Expect failure with `RandomnessAccountMismatch` error (error code 6004 or similar).
     - If the transaction succeeds, this is a SECURITY FAILURE -- return `{ passed: false, details: "Anti-reroll check did not reject wrong randomness" }`.
     - If it correctly rejects, return `{ passed: true, details: "RandomnessAccountMismatch correctly thrown" }`.
     - Then proceed with the CORRECT randomness to complete the transition (so the epoch is not stuck).

  2. `testDoubleCommit(provider, epochProgram, accounts)`:
     - Prerequisite: VRF is already pending (trigger_epoch_transition was called, waiting for reveal+consume).
     - Attempt another `trigger_epoch_transition` with a new randomness account.
     - Expect failure with `VrfAlreadyPending` error.
     - Return pass/fail result.

  3. `testStaleRandomness(provider, epochProgram, accounts)`:
     - This test is tricky on devnet because "stale" means the randomness account was created too long ago.
     - Create a randomness account, wait a few slots, then attempt to commit.
     - If the Switchboard SDK or program rejects stale randomness, document the behavior.
     - If it doesn't reject (because the freshness check is lenient on devnet), document this as expected behavior with notes.
     - Return result with explanation.

  4. `testTimeoutRecovery(provider, epochProgram, accounts)`:
     - This is the most important security test. It validates VRF-03.
     - Step 1: Start a normal VRF flow -- create randomness, commit + trigger.
     - Step 2: Do NOT send reveal. Simulate oracle failure.
     - Step 3: Wait for VRF_TIMEOUT_SLOTS (300 slots, ~2 minutes on devnet). Poll slot advancement with progress logging every 30 seconds.
     - Step 4: After 300+ slots, create a FRESH randomness account.
     - Step 5: Call `retry_epoch_vrf` with the fresh randomness account. This instruction validates that timeout has elapsed and replaces the pending randomness.
     - Step 6: Complete the VRF flow with the NEW randomness: commit (if needed -- check if retry_epoch_vrf already re-commits), reveal, consume.
     - Step 7: Verify EpochState advanced correctly.
     - Return result including: original commit slot, timeout wait duration, retry slot, final epoch state.

     Important: `retry_epoch_vrf` accounts struct from the codebase is: `{payer, epoch_state, randomness_account}`. The `randomness_account` is the NEW fresh one, not the stale one.

     After retry_epoch_vrf replaces the pending randomness, the client must:
     - Commit the NEW randomness (commitIx) -- because retry_epoch_vrf only replaces the stored account, the oracle still needs to process it.
     - Wait for oracle.
     - Reveal + consume as normal.

  **scripts/vrf/lib/swap-verifier.ts** -- Post-VRF swap tax rate verification (VRF-04):
  - Export `verifyTaxRateAppliedToSwap(provider, programs, manifest, epochState)`:
    - Read current EpochState to get the CRIME buy tax rate.
    - Execute a small CRIME buy swap (e.g., 0.01 SOL input) via Tax Program's `swap_sol_buy`.
    - The swap goes through Tax Program -> AMM CPI chain.
    - Compare the actual tax collected vs. expected (input * crimeBuyTaxBps / 10000).
    - This is a sanity check for VRF-04, not exhaustive swap testing (that's Phase 36).
    - Return `{ passed: boolean, expectedTaxBps: number, details: string, txSignature: string }`.
  - Account setup for `swap_sol_buy`:
    - Requires: payer, epochState, poolState (CRIME/SOL), vaultA, vaultB, user's SOL source, user's CRIME destination, mintA, mintB, tokenProgramA (TOKEN_PROGRAM_ID for WSOL side), tokenProgramB (TOKEN_2022_PROGRAM_ID for CRIME side), ammProgram, Transfer Hook remaining_accounts, plus tax distribution accounts (stakingEscrow, carnageSolVault, treasury).
    - Load all addresses from PDA manifest.
    - The user needs a CRIME token account (Token-2022) to receive output. Create one if needed.
    - Use a small amount (0.01 SOL = 10,000,000 lamports) to minimize impact on pool state.
  - If swap setup is too complex for a validation script (many accounts), simplify: read EpochState after VRF, verify rates are non-zero and in-band, and document that "Tax Program reads EpochState dynamically" as already proven by integration tests. Note this in the report as "verified by state read, full swap testing deferred to Phase 36."
  - The CONTEXT.md says "one swap after VRF updates taxes to verify the Tax Program reads the new rates correctly (quick sanity check)." If the full swap CPI chain is too complex to set up in a script (it requires ~15 accounts plus Transfer Hook remaining_accounts), it is acceptable to verify by reading the Tax Program's view of EpochState instead. Document the approach taken.

  **Orchestrator updates:**
  Add a `--security` flag (or always run security tests after epoch transitions). The execution flow becomes:

  1. Pre-flight checks (existing)
  2. 5 epoch transitions (existing from Plan 02)
  3. Security tests:
     a. Anti-reroll test (uses the 6th epoch transition as substrate)
     b. Double-commit test (during the anti-reroll transition while VRF is pending)
     c. Complete the 6th transition (clean up after security tests)
     d. Timeout recovery test (7th epoch transition with deliberate oracle skip + 300-slot wait + retry)
  4. Post-VRF swap verification (after taxes are confirmed from the latest epoch)
  5. Generate report using ValidationReporter
  6. Write report to `Docs/VRF_Devnet_Validation_Report.md`
  7. Print summary to console

  **Timing considerations:**
  - 5 epoch transitions: ~25-30 minutes
  - Security tests (anti-reroll + double-commit): ~2-3 minutes
  - Timeout recovery (300-slot wait): ~2-3 minutes
  - Post-VRF swap: ~30 seconds
  - Total expected: ~30-40 minutes

  **Error handling:**
  - If any epoch transition fails, log the error and continue with remaining transitions. Report partial results.
  - If a security test fails (expected rejection doesn't happen), this is a CRITICAL finding -- log it prominently.
  - If timeout recovery fails, log it as a blocker finding.
  - If post-VRF swap fails, log it but note that full swap testing is Phase 36.

  **Report generation:**
  Use the `ValidationReporter` from lib/reporter.ts. The final report should include:
  - Header: timestamp, cluster URL, wallet, SLOTS_PER_EPOCH=750
  - Summary: X/5 epoch transitions passed, Y/Z security tests passed, timeout recovery passed/failed, swap verification passed/failed
  - Detailed epoch transition table (per 35-RESEARCH.md format)
  - Security test results with error messages
  - Timeout recovery details with slot numbers
  - Swap verification details (if attempted)
  - Write to `Docs/VRF_Devnet_Validation_Report.md`

  **Execution:**
  Run the complete suite:
  ```bash
  set -a && source .env && set +a && npx tsx scripts/vrf/devnet-vrf-validation.ts
  ```

  Monitor console output. The script should take 30-40 minutes total.

  After the script completes, verify the report was generated and contains all required sections.
