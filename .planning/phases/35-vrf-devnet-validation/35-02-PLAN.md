<task type="auto">
  <name>Task 1: Build VRF flow helper and epoch state reader modules</name>
  <files>scripts/vrf/lib/vrf-flow.ts, scripts/vrf/lib/epoch-reader.ts, scripts/vrf/lib/reporter.ts</files>
  <action>
  Create three helper modules under `scripts/vrf/lib/`. These modules encapsulate reusable VRF logic extracted from the existing `tests/devnet-vrf.ts` pattern, enhanced with the full account list from 35-RESEARCH.md.

  **scripts/vrf/lib/vrf-flow.ts** -- Reusable VRF 3-TX flow:
  - Export `EpochTransitionResult` interface containing: epoch number, cheapSide, lowTaxBps, highTaxBps, crimeBuyTaxBps, crimeSellTaxBps, fraudBuyTaxBps, fraudSellTaxBps, flipped (boolean), vrfBytes (first 6 bytes from randomness), createSig, commitSig, consumeSig, durationMs.
  - Export `advanceEpochWithVRF(provider, epochProgram, accounts)` function implementing the complete 3-TX flow:
    - TX 1: Create randomness account. Use `sb.Randomness.create(sbProgram, rngKp, queueAccount.pubkey)`. Sign with wallet.payer + rngKp. Wait for "finalized" confirmation (CRITICAL -- commitIx reads account client-side).
    - TX 2: Commit + trigger_epoch_transition. Build commitIx from SDK, build triggerIx from epochProgram.methods. Bundle with 400,000 CU limit. Send with confirmed.
    - Wait for oracle: `waitForSlotAdvance(connection, 3)` then retry loop for revealIx (20 attempts, 3-second delay each -- accounts for the SDK's built-in 3-second delay per call).
    - TX 3: Reveal + consume_randomness. Build revealIx from SDK, build consumeIx from epochProgram.methods with FULL account list:
      ```
      caller, epochState, randomnessAccount, stakingAuthority, stakePool, stakingProgram, carnageState
      ```
      Bundle with 400,000 CU limit. Send with confirmed.
    - Read EpochState after consume, compute flipped, capture all tax rates.
    - Return `EpochTransitionResult`.
  - Export `waitForSlotAdvance(connection, targetSlots)` and `sleep(ms)` utilities.
  - The `accounts` parameter should be a struct containing: epochStatePda, treasuryPda, stakingAuthorityPda, stakePoolPda, stakingProgramId, carnageFundPda. Load these from `scripts/deploy/pda-manifest.json`.

  Anti-patterns to avoid (from 35-RESEARCH.md):
  - NEVER combine TX 1 and TX 2 (SDK reads account client-side after create).
  - NEVER use "confirmed" for TX 1 create -- always "finalized".
  - NEVER hardcode Switchboard addresses -- use `sb.getProgramId()` and `sb.getDefaultQueue()`.
  - Add 100-200ms delays between RPC calls to avoid Helius rate limiting.

  **scripts/vrf/lib/epoch-reader.ts** -- EpochState reading and verification:
  - Export `EpochStateSnapshot` interface: currentEpoch, cheapSide, vrfPending, taxesConfirmed, lowTaxBps, highTaxBps, crimeBuyTaxBps, crimeSellTaxBps, fraudBuyTaxBps, fraudSellTaxBps, genesisSlot, epochStartSlot, vrfRequestSlot, pendingRandomnessAccount.
  - Export `readEpochState(epochProgram, epochStatePda)` that fetches and maps EpochState to the snapshot interface.
  - Export `verifyTaxRates(snapshot)` that validates:
    - lowTaxBps is in [100, 400] range
    - highTaxBps is in [1100, 1400] range
    - If cheapSide === 0 (CRIME): crimeBuyTax === lowTax, crimeSellTax === highTax, fraudBuyTax === highTax, fraudSellTax === lowTax
    - If cheapSide === 1 (FRAUD): opposite assignment
    - Returns `{ valid: boolean, errors: string[] }`

  **scripts/vrf/lib/reporter.ts** -- Validation report generator:
  - Export `ValidationReporter` class that accumulates test results and generates a markdown report.
  - Methods: `addEpochTransition(result: EpochTransitionResult)`, `addSecurityTest(name, passed, details)`, `addTimeoutRecovery(result)`, `addSwapVerification(result)`, `generate(): string`, `writeToFile(path)`.
  - Report format per 35-RESEARCH.md's "Validation Report Format" section: summary table, epoch transition details (table with epoch/cheapSide/flipped/taxes/VRF bytes/tx sigs/duration), security test results, timeout recovery results, swap verification results.
  - Include cluster URL, wallet address, SLOTS_PER_EPOCH, timestamp in header.

  All modules import provider/program loading from `scripts/deploy/lib/connection.ts` (existing shared library). Use `@switchboard-xyz/on-demand` (already installed, v3.8.2).
  </action>

<task type="auto">
  <name>Task 2: Build main validation orchestrator and run 5 epoch transitions</name>
  <files>scripts/vrf/devnet-vrf-validation.ts</files>
  <action>
  Create `scripts/vrf/devnet-vrf-validation.ts` -- the main validation orchestrator that executes the complete VRF validation suite on devnet.

  **Structure:**
  1. Setup: Load provider, programs, and PDA addresses from `scripts/deploy/pda-manifest.json`.
  2. Pre-flight checks: Verify wallet balance >= 3 SOL. Verify EpochState is initialized and readable. Log current epoch state. Check if VRF is already pending (handle gracefully).
  3. Run 5 consecutive epoch transitions using `advanceEpochWithVRF()` from lib/vrf-flow.ts. Between each transition, wait for the next epoch boundary by polling slot advancement. For each transition:
     - Call `advanceEpochWithVRF()`
     - Call `readEpochState()` + `verifyTaxRates()` to validate results
     - Log progress: "Epoch N/5: epoch=X, cheapSide=Y, low=Z, high=W, flipped=true/false"
     - Add result to reporter via `addEpochTransition()`
     - If VRF byte 3 < 11 (carnage triggered), log it and note in report. This is nice-to-have per CONTEXT.md -- we observe it but don't need to execute Carnage.
     - Wait for next epoch boundary before starting next transition. Calculate wait time: `(epochStartSlot + 750 - currentSlot) * 400ms`. Add a small buffer (5 slots).
  4. After all 5 transitions, continue to security tests (Plan 03 will handle these, but the orchestrator should have a modular structure that allows Plan 03 to add security test functions).

  **Execution:**
  The script runs with: `set -a && source .env && set +a && npx tsx scripts/vrf/devnet-vrf-validation.ts`

  **Key implementation details:**
  - Load PDA manifest from `scripts/deploy/pda-manifest.json` for all addresses (epochStatePda, stakePoolPda, stakingAuthorityPda, carnageFundPda, etc.).
  - The treasuryPda for trigger_epoch_transition can be the wallet public key (bounty transfer is deferred, existing pattern from tests/devnet-vrf.ts).
  - Use `loadProvider()` and `loadPrograms()` from `scripts/deploy/lib/connection.ts` for consistency with deployment scripts.
  - Switchboard setup: `sb.getProgramId(connection)`, `Program.fetchIdl(sbProgramId, provider)`, `sb.getDefaultQueue(connection.rpcEndpoint)` -- all dynamic, no hardcoded addresses.
  - Add 200ms sleep between non-critical RPC calls to respect Helius rate limits.
  - If the script detects VRF is already pending at startup (from a previous failed run), handle it: either wait for timeout and retry, or attempt to reveal+consume the pending request.
  - The script should exit 0 on success, exit 1 on any failure. Failures include: any epoch transition failing, any tax rate out of band, wallet balance too low.

  After building the script, execute it on devnet and verify all 5 epoch transitions complete successfully. This will take ~25-30 minutes of real time due to epoch waits. Log all output to console for real-time monitoring.

  Important timing note: Each epoch transition takes ~30-90 seconds (create finalization + oracle wait + reveal retry). Plus ~5 minutes waiting for the next epoch boundary. Total: ~25-35 minutes for 5 transitions. Plan for long-running execution.
  </action>

  4. Each transition's cheap_side tax assignment is consistent (verified by `verifyTaxRates()`)
  5. At least one transition should show a different cheap_side than the previous (with 75% flip probability, P(no flip in 5 epochs) = 0.25^5 < 0.1%)
  </verify>
