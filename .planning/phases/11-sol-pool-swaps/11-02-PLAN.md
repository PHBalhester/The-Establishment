- `mint_spl_tokens()`, `mint_t22_tokens()` helpers

**New test infrastructure specific to swap testing:**

1. **`setup_initialized_pool()` helper function** that:
   - Creates a litesvm instance with the AMM program deployed
   - Creates an admin keypair and funds it
   - Calls `initialize_admin` instruction
   - Creates a T22 mint (for CRIME-like token, 9 decimals) and an SPL mint (for WSOL-like token, 9 decimals)
   - Ensures canonical mint ordering (mint_a < mint_b by pubkey bytes) -- this determines which side is T22 vs SPL
   - Calls `initialize_pool` with seed amounts (e.g., 1_000_000_000 each side = 1 token each at 9 decimals) and `lp_fee_bps = 100`
   - Creates user keypair, funds it, creates user token accounts for both mints
   - Mints tokens to user accounts (enough for multiple test swaps)
   - Returns a struct/tuple with: svm, admin, user, pool PDA, vault PDAs, mint keys, user token account keys, canonical ordering info (which position is T22 vs SPL)
   - **CRITICAL**: The helper must track which pool position (a/b) is T22 and which is SPL. Since canonical ordering is by pubkey bytes, the T22 mint may end up as either mint_a or mint_b. Tests need to know this to determine the correct SwapDirection for T22->SOL vs SOL->T22 swaps.

2. **`build_swap_instruction()` helper function** that:
   - Takes: pool, vaults, mints, user, user token accounts, token programs, amount_in, direction (as u8), minimum_amount_out
   - Builds the Anchor instruction data: discriminator("swap_sol_pool") + amount_in.serialize() + direction.serialize() + minimum_amount_out.serialize()
   - Builds the account meta list matching the SwapSolPool account struct ordering (pool, vault_a, vault_b, mint_a, mint_b, user_token_a, user_token_b, user, token_program_a, token_program_b)
   - Returns an `Instruction`

3. **`read_pool_state()` helper** that reads pool account data and deserializes reserves (reserve_a at byte offset after the known prefix fields, reserve_b after that). Use byte offsets from the PoolState struct layout:
   - Discriminator: 8 bytes
   - pool_type: 1 byte
   - mint_a: 32 bytes
   - mint_b: 32 bytes
   - vault_a: 32 bytes
   - vault_b: 32 bytes
   - reserve_a: 8 bytes (offset 137)
   - reserve_b: 8 bytes (offset 145)
   - lp_fee_bps: 2 bytes (offset 153)
   - initialized: 1 byte (offset 155)
   - locked: 1 byte (offset 156)
   - bump: 1 byte (offset 157)
   Returns at minimum: reserve_a, reserve_b, locked

4. **`get_token_balance()` helper** that reads a token account's balance from litesvm.

<task type="auto">
  <name>Task 2: Write swap integration tests for both directions, fees, slippage, events</name>
  <files>programs/amm/tests/test_swap_sol_pool.rs</files>
  <action>
Add integration tests to test_swap_sol_pool.rs. Each test uses `setup_initialized_pool()` for a fresh pool.

**Test 1: `test_swap_a_to_b_correct_output`** (SWAP-01, SWAP-03)
- Setup pool with known reserves (e.g., 1_000_000_000 each)
- Execute swap: AtoB direction, amount_in = 10_000_000 (0.01 token), minimum_amount_out = 0
- Read pool state after swap
- **Verify output amount**: Calculate expected output manually using the constant-product formula:
  - effective_input = 10_000_000 * 9900 / 10000 = 9_900_000
  - amount_out = 1_000_000_000 * 9_900_000 / (1_000_000_000 + 9_900_000) = 9_802_970 (integer truncation)
- Check user token balances: input side decreased by amount_in, output side increased by expected amount_out
- Check pool reserves: reserve_in increased by amount_in (pre-fee!), reserve_out decreased by amount_out

**Test 2: `test_swap_b_to_a_correct_output`** (SWAP-01, SWAP-03)
- Same pool setup
- Execute swap: BtoA direction, same amount_in
- Verify output amount matches formula in reverse direction
- Check balances and reserves are updated correctly

**Test 3: `test_swap_fee_compounds_into_reserves`** (SWAP-04)
- Execute a swap with known amount_in
- Read reserve_in after swap
- **Verify**: new_reserve_in == old_reserve_in + amount_in (NOT + effective_input)
- This proves the fee stays in the pool (the difference between amount_in and effective_input is LP revenue)
- Calculate lp_fee = amount_in - effective_input_as_u64, verify it's correct (amount_in * 100 / 10000 = amount_in / 100)

**Test 4: `test_swap_slippage_protection`** (SWAP-05, TEST-04)
- Setup pool, calculate expected output for a swap amount
- Set minimum_amount_out = expected_output + 1 (one unit above actual output)
- Execute swap -- MUST FAIL
- Verify the error contains "Slippage" or matches the SlippageExceeded error code
- Then execute same swap with minimum_amount_out = expected_output -- MUST SUCCEED

**Test 5: `test_swap_k_invariant_holds`** (SWAP-07)
- Execute a swap
- Read reserves before and after
- Calculate k_before = reserve_a_before * reserve_b_before
- Calculate k_after = reserve_a_after * reserve_b_after
- **Verify**: k_after >= k_before (k should increase slightly due to integer truncation and fees)

**Test 6: `test_swap_zero_amount_rejected`** (SWAP-08)
- Attempt swap with amount_in = 0
- MUST FAIL with ZeroAmount error

**Test 7: `test_swap_event_emitted`** (SWAP-09)
- Execute a swap
- Check transaction logs/metadata for SwapEvent emission
- Note: litesvm may not provide direct event parsing. Alternative approach: verify the swap succeeds (the emit! would fail compilation if struct mismatched) and check transaction logs contain "SwapEvent" or the event discriminator.
- If direct event verification is impractical in litesvm, document this limitation and verify the event exists in the IDL as a compile-time proof.

**Test 8: `test_consecutive_swaps_succeed`** (proves reentrancy guard clears)
- Execute swap 1 (AtoB) -- must succeed
- Read pool state, verify locked = false
- Execute swap 2 (BtoA) -- must succeed (proves pool.locked was cleared after first swap)
- Verify both swaps produced correct outputs

**Notes on test implementation:**
- For balance assertions, record balances BEFORE swap, execute swap, record AFTER, compare deltas
- Use the math formulas from `helpers/math.rs` to compute expected values in test code (replicate the pure math, don't import from program crate -- test boundary should be independent)
- Pool reserve byte offsets MUST account for the new `locked` field (1 byte after `initialized`)
- If canonical ordering puts SPL mint as mint_a and T22 as mint_b, then AtoB = SOL->CRIME and BtoA = CRIME->SOL. The test setup helper must make this mapping clear so tests use the correct direction.
