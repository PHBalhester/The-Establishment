  <name>Task 1: On-chain security verification (6 checks)</name>
  <files></files>
  <action>
Run or create on-chain tests for each of the 6 Phase 46-50 security fixes. Check if `scripts/e2e/devnet-e2e-validation.ts` already covers these. If not, write a targeted verification script or extend the existing one.

**The 6 verification checks:**

1. **Phase 46 SEC-01: Fake staking_escrow reverts**
   Send a swap_sol_buy transaction with a fabricated staking_escrow address (not the real PDA). Expect the transaction to fail with a constraint violation error.

2. **Phase 46 SEC-02: Fake amm_program reverts**
   Send a swap_sol_buy transaction with a fake amm_program address. Expect failure with an address mismatch error.

3. **Phase 46 SEC-03: Non-Switchboard randomness reverts**
   Send trigger_epoch_transition with a randomness account that is NOT owned by the Switchboard program. Expect failure with InvalidRandomnessOwner.

4. **Phase 48 FIX-01: Sell tax from WSOL output**
   Execute a sell swap. Verify: (a) user's native SOL balance is unchanged (or only changes by TX fee), (b) user receives net WSOL (gross minus tax), (c) tax arrives at staking_escrow/carnage_vault/treasury as native SOL.

5. **Phase 49 SEC-08: Minimum output floor**
   Send a swap with minimum_amount_out=0. Expect the program to reject with MinimumOutputFloorViolation (the protocol enforces a 50% constant-product floor).

6. **Phase 50 FIX-04: VRF bounty payment**
   Trigger an epoch transition. Check: (a) triggerer's SOL balance increases by the bounty amount (0.001 SOL / TRIGGER_BOUNTY_LAMPORTS), (b) carnage_sol_vault balance decreases correspondingly. If the vault is empty, verify graceful degradation (bounty_paid: 0 in event, no error).

For each check, log: PASS/FAIL, transaction signature, and relevant account state changes.

**Note:** Some checks (3, 6) depend on epoch state. If the epoch system isn't at the right state for a VRF trigger, the check may need to advance the epoch first (by swapping to accumulate tax and waiting for enough slots). Use the existing e2e infrastructure for this.

If any check FAILS, investigate immediately. A failure means a security fix is NOT live on-chain, which is a blocker.

This script:
1. Swaps to accumulate tax in carnage vaults
2. Triggers epoch transitions until Carnage triggers (VRF byte check)
3. Executes Carnage atomically (reveal+consume+executeCarnageAtomic in single v0 TX)
