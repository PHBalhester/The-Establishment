Add a new `describe("Edge Cases")` block to `tests/security.ts` (after the existing Security blocks from 29-02). Each test should have a comment explaining the edge case being validated and which requirement it maps to.

**Test 1: `it("zero total_staked is prevented by dead stake (MINIMUM_STAKE)")`**
- Verify pool.totalStaked >= MINIMUM_STAKE after initialization
- This prevents division-by-zero in reward math (add_to_cumulative divides by total_staked)
- Maps to: SEC-01, MATH-05

**Test 2: `it("mid-epoch stake earns no rewards from current epoch")`**
- Verify user.rewardsPerTokenPaid == pool.rewardsPerTokenStored after staking
- Calculate delta = pool.rewardsPerTokenStored - user.rewardsPerTokenPaid == 0
- Therefore pending = 0 * balance / PRECISION = 0
- Maps to: SEC-02

**Test 3: `it("partial unstake leaves correct remaining balance")`**
- Create fresh user, stake 1000 PROFIT
- Unstake 300 PROFIT
- Verify user.stakedBalance == 700 PROFIT
- Verify pool.totalStaked decreased by 300
- Verify vault balance decreased by 300
- assertEscrowSolvency after each operation
- Maps to: ERR-02

**Test 4: `it("claim with zero rewards fails with NothingToClaim (ERR-04)")`**
- Stake tokens (no rewards deposited yet, so rewards_earned = 0)
- Attempt claim, expect NothingToClaim error (error code 0x1773 or "NothingToClaim")
- Maps to: ERR-04

**Test 5: `it("unstake exceeding balance fails with InsufficientBalance (ERR-02)")`**
- Stake 500 PROFIT
- Try unstake 501 PROFIT
- Expect InsufficientBalance error
- Maps to: ERR-02

**Test 6: `it("auto-full-unstake when remaining < MINIMUM_STAKE")`**
- Create fresh user, stake MINIMUM_STAKE + 100 units
- Unstake 101 units (would leave MINIMUM_STAKE - 1)
- Verify user.stakedBalance == 0 (auto-full-unstake triggered)
- Maps to: dust prevention (Phase 26-04 decision)

**Test 7: `it("rapid stake/unstake cycling cannot capture rewards (economic manipulation)")`**
- Create fresh user with 5000 PROFIT
- Loop 10 times: stake 500, immediately unstake 500
- Verify totalClaimed == 0 after all cycles
- Verify rewardsEarned == 0
- assertEscrowSolvency after each operation
- Maps to: SEC-02 (checkpoint prevents manipulation)

Add two more describe blocks to `tests/security.ts`:

**Block 1: `describe("Multi-User Proportional Distribution")`**

```
/**
 * INVARIANT: Proportional Reward Distribution
 *
 * Rewards must be distributed proportionally to each user's stake.
 * If User A has 25% of total stake, they receive 25% of rewards.
 * Due to integer truncation (MATH-05), the sum of individual claims
 * may be slightly less than total deposited (protocol keeps dust).
 *
 * PROPERTY VALIDATED: user_reward / total_rewards ≈ user_stake / total_staked
 * PROPERTY VALIDATED: sum(all user rewards) <= total deposited (conservation)
 */
```

Tests:
- `it("10 stakers receive proportional rewards (BigInt math validation)")` -- Create 10 stakers with amounts [100, 200, 300, 400, 500, 600, 700, 800, 900, 1000] PROFIT (in base units with 6 decimals). Calculate total_staked. Simulate 10 SOL deposit: rewardPerToken = (10_SOL * PRECISION) / total_staked. For each staker, calculate their reward: (rewardPerToken * staker.amount) / PRECISION. Verify sum(all rewards) <= 10 SOL. Verify dust < 10 (less than number of stakers). Verify each user's share matches their stake proportion (within 1 lamport tolerance due to truncation).

- `it("late staker earns 0 from pre-existing rewards (checkpoint correctness)")` -- Pure BigInt math test. Simulate: epoch 1 has cumulative X. User stakes after epoch 1. Their checkpoint = X. Delta = 0. Rewards from epoch 1 = 0. This confirms late stakers don't dilute existing stakers' earnings.

- `it("equal stakers receive equal rewards")` -- 5 stakers each with 200 PROFIT. Simulate reward deposit. Each should get exactly 1/5 of rewards (with possible 1-lamport rounding difference). Use BigInt arithmetic.

**Block 2: `describe("Multi-User Stress Test")`**

```
/**
 * STRESS TEST: 100 Concurrent Stakers
 *
 * Validates the staking system under scale: 100 users stake
 * different amounts, perform operations, and the system maintains
 * all invariants throughout.
 *
 * Uses batch funding pattern (pre-fund treasury, distribute via transfer)
 * to avoid airdrop rate limiting (see 29-RESEARCH.md Pitfall 3).
 *
 * PROPERTY VALIDATED: System remains consistent under high concurrency
 * PROPERTY VALIDATED: Escrow solvency maintained with 100+ users
 */
```

First, create a `createBatchStakers(count: number)` helper function at the top of this describe block (or alongside the existing helpers from 29-02). Implementation steps:

1. **Create and fund treasury keypair:** Generate a new Keypair. Airdrop 200 SOL to this treasury (single airdrop, avoids rate limiting).
2. **Loop to create user keypairs:** For `i = 0..count`, generate a new Keypair for each user.
3. **Batch SOL distribution:** For each user keypair, send a `SystemProgram.transfer` instruction from treasury to user (0.5 SOL each). Use `skipPreflight: true` for speed. Batch confirmations: collect all tx signatures, then `confirmTransaction` in a second loop (or use `Promise.all` with a small concurrency limit of 10).
4. **Create token accounts and mint PROFIT for each user:** For each user, create an Associated Token Account for PROFIT mint (with Transfer Hook extension). Mint 2_000_000_000 PROFIT base units to each user's token account from the admin mint authority. Whitelist each user's token account in the Transfer Hook program.
5. **Return array of user contexts:** Return `Array<{ keypair: Keypair, tokenAccount: PublicKey }>` so tests can reference each staker.

The helper signature:
```typescript
async function createBatchStakers(
  count: number,
  connection: Connection,
  admin: Keypair,
  profitMint: PublicKey,
  transferHookProgram: Program,
): Promise<Array<{ keypair: Keypair; tokenAccount: PublicKey }>>
```

Tests:
- `it("100 stakers can all stake and pool tracks total correctly")` -- Call `createBatchStakers(100, ...)`. Each stakes a random amount between 100_000 and 1_000_000_000 PROFIT (base units). After all 100 stake, verify pool.totalStaked == MINIMUM_STAKE + sum(all stakes). Verify vault balance == pool.totalStaked. assertEscrowSolvency.

  **IMPORTANT PERFORMANCE NOTES:**
  - Use `skipPreflight: true` on transactions for speed
  - Batch confirmations where possible
  - The existing Anchor.toml timeout is 1000000ms (1000s) which should be sufficient
  - If 100 users is too slow, scale down to 50 and document the reason

- `it("multi-user solvency invariant holds after interleaved operations")` -- Using 10 of the 100 stakers from the previous test (or create 10 fresh ones if they are in a separate test scope). Perform 50 interleaved operations: randomly choose a user, randomly stake or unstake a random amount (within their balance). assertEscrowSolvency after EACH operation. After all operations, verify pool.totalStaked == vault balance.
