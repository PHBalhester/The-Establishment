**Step 3: Write the 4 attack simulation test blocks**

Each test MUST have a multi-line JSDoc comment block (attack narrative) explaining:
- ATTACK: Name of the attack
- SCENARIO: Step-by-step attack description
- MITIGATION: How the system prevents it
- PROPERTY VALIDATED: The security invariant being tested

**Test Block 1: `describe("Security: First-Depositor Attack (SEC-01)")`**

```
/**
 * ATTACK: First-Depositor Inflation Attack
 *
 * SCENARIO:
 * 1. Attacker is first staker with minimal stake (1 lamport)
 * 2. Large reward deposited to pool
 * 3. Attacker claims disproportionate rewards as sole staker
 * 4. Subsequent stakers receive diluted yield
 *
 * MITIGATION: MINIMUM_STAKE dead stake during initialization (SEC-01)
 * Protocol stakes 1 PROFIT (1,000,000 units) as irrecoverable "dead stake"
 * during initialize_stake_pool. This ensures:
 * - Pool always has meaningful denominator for reward math
 * - Attacker cannot be sole staker with dust amount
 *
 * PROPERTY VALIDATED: total_staked >= MINIMUM_STAKE always
 */
```

Tests:
- `it("pool starts with MINIMUM_STAKE dead stake")` -- Verify pool.totalStaked >= MINIMUM_STAKE after init, vault.amount >= MINIMUM_STAKE
- `it("attacker with 1 unit cannot capture majority of rewards")` -- Stake 1 unit as attacker. Calculate attacker_share = 1 / (MINIMUM_STAKE + 1). Assert attacker_share < 0.000001 (less than 0.0001%). Use BigInt math for precision.
- `it("dead stake is irrecoverable (no UserStake PDA for dead stake)")` -- Verify no UserStake account exists for the "dead stake" -- it's protocol-owned, not user-claimable. Try to derive a UserStake PDA for the pool authority and verify it either doesn't exist or has 0 balance.

**Test Block 2: `describe("Security: Flash Loan Attack (SEC-02)")`**

```
/**
 * ATTACK: Flash Loan Same-Epoch Exploitation
 *
 * SCENARIO:
 * 1. Attacker sees deposit_rewards tx in mempool
 * 2. Attacker flash-loans PROFIT tokens
 * 3. Attacker stakes PROFIT before epoch end
 * 4. update_cumulative finalizes rewards
 * 5. Attacker claims disproportionate rewards
 * 6. Attacker unstakes and repays flash loan
 *
 * MITIGATION: Checkpoint pattern (SEC-02)
 * On stake, user.rewards_per_token_paid = pool.rewards_per_token_stored.
 * User only earns from FUTURE cumulative increases.
 * update_cumulative is CPI-gated to Epoch Program (SEC-04).
 * Flash loans repay in same transaction; cannot span epochs.
 *
 * PROPERTY VALIDATED: same-epoch stake/unstake earns 0 rewards
 */
```

Tests:
- `it("stake captures current cumulative as checkpoint")` -- Stake tokens. Verify userStake.rewardsPerTokenPaid == pool.rewardsPerTokenStored.
- `it("same-epoch stake/unstake earns exactly 0 rewards")` -- Create fresh user. Stake, then immediately unstake. Verify totalClaimed == 0 and rewardsEarned == 0.
- `it("checkpoint math: stake after deposit earns 0 from that deposit")` -- Pure BigInt math test. If cumulative is X when user stakes, their checkpoint = X. If cumulative stays X (no update_cumulative called), delta = 0, rewards = 0.
- Call `assertEscrowSolvency()` after each test.

**Test Block 3: `describe("Security: CPI Forgery (SEC-03, SEC-04)")`**

```
/**
 * ATTACK: CPI Forgery (Unauthorized deposit_rewards/update_cumulative)
 *
 * SCENARIO:
 * 1. Attacker deploys fake program with same interface
 * 2. Fake program derives PDA with same seeds (TAX_AUTHORITY_SEED)
 * 3. Fake program CPIs to deposit_rewards with inflated amount
 * 4. Pool state corrupted with phantom rewards
 *
 * MITIGATION: seeds::program constraint (SEC-03, SEC-04)
 * Staking Program verifies PDA was derived from SPECIFIC program IDs:
 * - deposit_rewards: seeds::program = tax_program_id()
 * - update_cumulative: seeds::program = epoch_program_id()
 * A PDA from any other program ID will fail constraint validation.
 *
 * PROPERTY VALIDATED: Only Tax/Epoch Program can call CPI-gated instructions
 */
```

Tests:
- `it("rejects deposit_rewards from unauthorized keypair")` -- Create unauthorized keypair, attempt deposit_rewards, expect ConstraintSeeds error
- `it("rejects update_cumulative from unauthorized keypair")` -- Create unauthorized keypair, attempt update_cumulative, expect ConstraintSeeds error
- `it("rejects deposit_rewards with zero amount")` -- Even if authorized, zero amount should fail with ZeroAmount. Use a random keypair (it will fail on seeds first, which is fine -- the test documents the dual protection).

**Test Block 4: `describe("Security: Escrow Solvency (SEC-06)")`**

```
/**
 * INVARIANT: Escrow Solvency
 *
 * The escrow vault (native SOL) must ALWAYS hold enough to cover
 * all pending rewards. If violated, users cannot claim their earned yield.
 *
 * escrow_balance >= pool.pending_rewards
 *
 * This invariant is tested after EVERY state-modifying operation.
 * The 100+ operation stress test simulates adversarial ordering:
 * rapid stake/unstake cycling, varying amounts, multiple users.
 *
 * PROPERTY VALIDATED: escrow_balance >= pending_rewards after every operation
 */
```

Tests:
- `it("solvency holds after 100+ stake/unstake operations (single user)")` -- Loop 50+ times: stake random amount (100-500 PROFIT), assertEscrowSolvency, unstake random amount, assertEscrowSolvency. Use 50 iterations (each has 2 operations = 100+ ops total). Amounts can vary but must stay within user's balance.
- `it("solvency holds after multi-user concurrent operations (5 users)")` -- Create 5 users. Each stakes different amounts. Interleave stake/unstake across users (user A stakes, user B unstakes, user C stakes, etc.). assertEscrowSolvency after EACH operation. At least 20 operations total.
- `it("solvency holds with deposit_rewards simulation")` -- Fund escrow with SOL via SystemProgram.transfer (simulating Tax Program deposit). Call deposit_rewards directly (will fail due to CPI gating, but the math simulation validates solvency). Instead: manually verify that after funding escrow, the balance covers pending. Note: Since we can't call deposit_rewards without Tax Program, manually fund the escrow and check balance vs pool.pendingRewards (which should be 0 since no deposits occurred).

For the multi-user test, create a helper function `createStakerWithTokens()` following the pattern from cross-program-integration.ts `createStaker()` but adapted for the Transfer Hook setup (this validator has a Transfer Hook-enabled mint).
