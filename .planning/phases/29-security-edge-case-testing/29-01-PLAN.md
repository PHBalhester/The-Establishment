**Step 2: Add proptest tests to math.rs**

In `programs/staking/src/helpers/math.rs`, add a new `proptest!` block INSIDE the existing `#[cfg(test)] mod tests` section, after all existing tests. Add `use proptest::prelude::*;` at the top of the test module.

Write 4 property tests with `ProptestConfig::with_cases(10_000)`:

**Property 1: add_to_cumulative never panics**
- Inputs: `total_staked in 1u64..=u64::MAX` (never 0, enforced by MINIMUM_STAKE), `pending in 0u64..=u64::MAX`, `existing_cumulative in 0u128..=(u128::MAX / 2)` (headroom for addition)
- The function may return Ok or Err (overflow is acceptable for extreme values) but must NEVER panic
- On Ok: assert cumulative >= existing, pending_rewards == 0
- On Err: acceptable (overflow for extreme values)

**Property 2: Reward conservation (sum of claims <= deposited)**
- Inputs: `total_staked in 1u64..=1_000_000_000_000u64` (realistic range), `pending in 1u64..=1_000_000_000_000u64` (up to ~1000 SOL), `user_balance in 1u64..=total_staked` (use `prop_assume!(user_balance <= total_staked)`)
- After add_to_cumulative, calculate user's reward: `(user_balance as u128) * pool.rewards_per_token_stored / PRECISION`
- Assert: user reward as u64 <= pending (no individual user can claim more than was deposited)

**Property 3: update_rewards formula never panics**
- Test the FORMULA in isolation (not the function, which calls Clock::get())
- Inputs: `balance in 0u64..=u64::MAX`, `reward_delta in 0u128..=(u128::MAX / (u64::MAX as u128))` (bounded to prevent u128 overflow in multiply)
- Calculate: `(balance as u128).checked_mul(reward_delta).map(|v| v / PRECISION)`
- Assert: result is Some (no overflow within the bounded range)

**Property 4: Cumulative is monotonically non-decreasing**
- Inputs: `total_staked in 1u64..=1_000_000_000u64`, `pending1 in 0u64..=1_000_000_000u64`, `pending2 in 0u64..=1_000_000_000u64`
- Start pool at cumulative 0, apply add_to_cumulative with pending1, record cumulative
- Set pending_rewards to pending2, apply add_to_cumulative again
- Assert: second cumulative >= first cumulative

Each property test should have a doc comment with `///` explaining the security property being validated, following the attack narrative style from 29-CONTEXT.md.
