  <behavior>
    **Split Router (split-router.ts):**

    `computeOptimalSplit` finds the input split ratio across two parallel multi-hop paths that maximizes total output. It uses a 1% granularity grid search (100 iterations, <1ms).

    Interface:
    ```typescript
    export const SPLIT_THRESHOLD_BPS = 50; // 0.5% -- only recommend split if gain exceeds this

    export interface SplitResult {
      shouldSplit: boolean;        // true if split beats best single path by >= SPLIT_THRESHOLD_BPS
      splitRatioA: number;         // percentage routed through path A (0-100)
      splitRatioB: number;         // percentage routed through path B (0-100)
      splitOutput: number;         // total output from split
      bestSingleOutput: number;    // best single-path output for comparison
      improvementBps: number;      // basis points improvement of split over single
    }

    export function computeOptimalSplit(
      totalInput: number,           // base units
      pathAQuoter: (input: number) => number,  // returns output for given input through path A
      pathBQuoter: (input: number) => number,  // returns output for given input through path B
    ): SplitResult;
    ```

    The pathAQuoter/pathBQuoter are callback functions that the caller composes from quote-engine primitives. This keeps split-router generic (no knowledge of specific pool topology).

    Logic:
    1. Compute pathA-only output: pathAQuoter(totalInput)
    2. Compute pathB-only output: pathBQuoter(totalInput)
    3. bestSingleOutput = max(pathA-only, pathB-only)
    4. Grid search: for ratioA in [1..99] by 1%:
       - inputA = floor(totalInput * ratioA / 100)
       - inputB = totalInput - inputA
       - if inputA <= 0 or inputB <= 0, skip
       - total = pathAQuoter(inputA) + pathBQuoter(inputB)
       - track maximum
    5. Compare best split output vs bestSingleOutput
    6. shouldSplit = (splitOutput - bestSingleOutput) / bestSingleOutput * 10000 >= SPLIT_THRESHOLD_BPS

    Test cases:
    1. Equal pools, small input -> shouldSplit=false (no price impact benefit)
    2. Equal pools, LARGE input (50% of one pool's reserve) -> shouldSplit=true, ~50/50 split
    3. Asymmetric pools (one has 10x more liquidity) -> shouldSplit=true, skewed ratio (e.g., 20/80)
    4. Zero input -> shouldSplit=false, splitOutput=0
    5. One pool has zero reserves -> shouldSplit=false, all through the other pool

    **Multi-Hop Builder (multi-hop-builder.ts):**

    `buildMultiHopTransactions` creates an array of unsigned Transaction objects for a multi-hop route.
    `executeMultiHopRoute` signs all transactions, then sends them sequentially.

    Interface:
    ```typescript
    export interface MultiHopResult {
      status: "confirmed" | "partial" | "failed";
      signatures: string[];       // TX signatures in order
      failedAt?: number;          // index of the failed hop (0-based)
      error?: string;             // parsed error message for the failed hop
      intermediateToken?: string;  // token symbol the user holds after partial failure
    }

    export async function buildMultiHopTransactions(
      route: Route,
      connection: Connection,
      userPublicKey: PublicKey,
      priorityFeeMicroLamports: number,
    ): Promise<Transaction[]>;

    export async function executeMultiHopRoute(
      transactions: Transaction[],
      route: Route,
      wallet: ProtocolWallet,
      connection: Connection,
    ): Promise<MultiHopResult>;
    ```

    buildMultiHopTransactions logic:
    1. For each step in route.steps, determine the swap type from step.pool + step.inputToken + step.outputToken
    2. Call the correct existing swap-builder:
       - SOL buy step: buildSolBuyTransaction({ amountInLamports: step.inputAmount, minimumOutput: floor(step.outputAmount * (10000-slippageBps)/10000), ... })
       - SOL sell step: buildSolSellTransaction(...)
       - PROFIT buy step: buildProfitBuyTransaction(...)
       - PROFIT sell step: buildProfitSellTransaction(...)
    3. For hop 2+: use wider slippage (2x route slippage) to account for reserves changing after hop 1
    4. Set recentBlockhash and feePayer on each TX
    5. Return array of unsigned transactions

    executeMultiHopRoute logic:
    1. Check if wallet supports signAllTransactions (typeof wallet.signAllTransactions === 'function')
    2. If yes: const signedTxs = await wallet.signAllTransactions(transactions)
    3. If no (fallback): sign each TX individually with wallet.signTransaction
    4. Send TX[0], wait for confirmation
    5. If TX[0] confirmed, send TX[1], wait for confirmation
    6. If any TX fails: return { status: "partial", failedAt: i, signatures: [...completed], intermediateToken: route.steps[i-1].outputToken, error: parseSwapError(err) }
    7. If all confirmed: return { status: "confirmed", signatures: [...all] }

    **Note:** multi-hop-builder.ts imports real Connection/Transaction types and swap-builders, so it CANNOT be tested with pure unit tests (needs RPC). Write the module with clean separation so the split-router (pure math) gets TDD tests, while multi-hop-builder is tested during integration in Plan 05/06.
