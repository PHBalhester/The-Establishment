Discrepancies to register (from research):

DISC-01: Timing Model
- Spec: Slot-based (4,500 slots per epoch, ~30 minutes)
- V3: Timestamp-based (3,600 seconds per epoch, 1 hour)
- Analysis: Slot-based is more deterministic (no clock drift), timestamp-based is simpler (no slot rate assumptions). Spec argues slot-based creates predictable arbitrage windows.

DISC-02: Tax Model
- Spec: "Cheap side" regime with discrete bands (Low: 1-4%, High: 11-14%)
- V3: Independent continuous rates per pool (0.75%-14.75% linear interpolation)
- Analysis: Spec model creates clear arbitrage windows (traders know which token is cheap). V3 model is simpler and proven working but less game-theoretically structured.

DISC-03: VRF Byte Usage
- Spec: 6 bytes (flip decision, low/high magnitude, Carnage trigger/action/target)
- V3: 4 bytes (one per tax rate: pool A buy, pool A sell, pool B buy, pool B sell)
- Analysis: Spec integrates Carnage into VRF callback. V3 doesn't handle Carnage at all in VRF. This affects compute budget and instruction design.

DISC-04: VRF Integration Pattern
- Spec: CPI to Switchboard (old VRF v2 pattern)
- V3: Client-side commit-reveal (On-Demand pattern)
- Analysis: The spec describes an outdated integration pattern. The v3 client-side approach is the ONLY viable approach with current Switchboard SDKs. **This discrepancy should almost certainly be resolved in favor of v3** but is flagged for explicit confirmation.

DISC-05: Trigger Bounty
- Spec: Fixed 0.01 SOL (TRIGGER_BOUNTY_LAMPORTS = 10_000_000)
- V3: Dynamic 0.1% of treasury, capped at 0.05 SOL (ADVANCER_REWARD_BPS = 10, MAX_ADVANCER_REWARD = 50_000_000)
- Analysis: Dynamic bounty scales with protocol growth. Fixed bounty is simpler but may become too small or too large relative to treasury size.

DISC-06: Tax Rate Range
- Spec: Low band 1-4% (100-400 bps), High band 11-14% (1100-1400 bps), 4 discrete steps each
- V3: Continuous 0.75%-14.75% (75-1475 bps), linear interpolation from random byte
- Analysis: Discrete bands create predictable regimes for traders. Continuous range has more variety but less predictable arbitrage opportunities.

DISC-07: Carnage Integration
- Spec: Atomic within VRF callback (Carnage executes in same instruction as tax regime update)
- V3: Not implemented in VRF flow (Carnage was a separate, unintegrated concern)
- Analysis: Atomic Carnage has compute budget concerns (CPI depth 4, ~400k CU already for VRF). May need separate instruction or creative batching.

**Section 6: Open Questions**
- Switchboard SDK version stability (v0.11.3 vs future versions, solana-v2/v3 feature flags)
- Mainnet VRF cost per request (devnet testing showed it works, but exact SOL cost unknown)
- Compute budget headroom for combined VRF + Carnage execution
