**Section 2: Architecture Overview**
- Switchboard On-Demand model: client-side commit-reveal, NOT CPI callback
- Program side is simple (validate and read); client side is complex (SDK orchestration)
- Three-transaction flow diagram (ASCII art sequence):
  1. Create randomness account (separate tx, must finalize)
  2. Commit randomness + program lock (bundled)
  3. Reveal randomness + program consume (bundled, after slot advance)
- Why three transactions, not two: SDK's commitIx() reads account data client-side, so account must exist on-chain first

**Section 3: On-Chain Program (Rust)**

3.1 EpochState Account Structure:
- Copy the EpochState struct from research (epoch_number, epoch_start, duration_seconds, bump, randomness_commit_slot, randomness_pending, randomness_commit_time, pending_randomness_account)
- Document byte layout: 8 (disc) + 8 + 8 + 8 + 1 + 8 + 1 + 8 + 32 = 82 bytes
- Note: DEFAULT_DURATION was 3600 seconds in v3 (spec says 4,500 slots -- this is a known discrepancy)

3.2 commit_epoch_randomness Instruction:
- Parse RandomnessAccountData from the passed account (no CPI needed)
- Validate seed_slot freshness (current or previous slot)
- Verify randomness NOT yet revealed (commit phase check)
- Store pending_randomness_account pubkey for anti-reroll binding
- Set randomness_pending = true, record commit_slot and commit_time
- Include the Rust code examples from research (Pattern 2)

3.3 consume_randomness Instruction:
- Verify same randomness account as committed (Pattern 3 anti-reroll)
- Parse RandomnessAccountData and call get_value(clock.slot) to read revealed bytes
- Call derive_tax_rates() to convert 4 random bytes into 4 tax rates
- Apply rates to pools via remaining_accounts pattern (Pattern 5)
- Reset randomness_pending = false, increment epoch
- Include the Rust code examples from research

3.4 Tax Rate Derivation:
- derive_tax_rate(): bias-free linear mapping from u8 (0-255) to BPS range
- derive_tax_rates(): takes 4 bytes, returns [u16; 4] for (pool A buy, pool A sell, pool B buy, pool B sell)
- Include the Rust code from research
- Note: v3 used MIN_TAX_BPS=75, MAX_TAX_BPS=1475 (continuous). Spec uses discrete bands -- this is a known discrepancy.

3.5 Timeout Recovery:
- RANDOMNESS_TIMEOUT = 3600 seconds (1 hour)
- If randomness_pending = true for longer than timeout, a new commit is allowed
- This prevents permanent epoch stall if oracle is down
- Document the check: `clock.unix_timestamp - epoch.randomness_commit_time > RANDOMNESS_TIMEOUT`

**Section 4: Client-Side Orchestration (TypeScript)**

4.1 SDK Setup:
- Dynamic address resolution: sb.getProgramId(), sb.getDefaultQueue() (Pattern 4)

4.2 Transaction 1 -- Create Randomness Account:
- Generate fresh keypair for randomness account
- Randomness.create(sbProgram, rngKp, queue) returns [randomness, createIx]
- Send and wait for FINALIZATION (not just confirmation)
- Why finalization: commitIx reads account data, account must be fully materialized

4.3 Transaction 2 -- Commit:
- randomness.commitIx(queue) returns SDK commit instruction
- program.methods.commitEpochRandomness() returns program commit instruction
- Bundle with ComputeBudgetProgram.setComputeUnitLimit({ units: 400_000 })
- Send and confirm

4.4 Transaction 3 -- Reveal + Consume:
- Wait for slot advancement (~3 slots minimum)
- randomness.revealIx() returns reveal instruction (may need retries -- oracle processing time)
- program.methods.consumeRandomness() with .remainingAccounts([pool PDAs])
- Bundle with compute budget
- Include retry logic pattern: up to 10 retries with 2s delays

**Section 5: Security Model**

5.1 Anti-Reroll Protection:
- Program stores committed randomness account pubkey
- At consume, verifies exact same account is passed
- Prevents attacker from committing, seeing unfavorable result, and trying with different account

5.2 Timeout Recovery:
- 1-hour timeout prevents permanent stall
- After timeout, new commit allowed even if randomness_pending = true
- Tradeoff: 1 hour of stalled epochs is acceptable vs permanent stuck state

5.3 Stale Randomness Prevention:
- seed_slot freshness check at commit time (slot_diff <= 1)
- Already-revealed check at commit time (prevents reuse of old randomness)
