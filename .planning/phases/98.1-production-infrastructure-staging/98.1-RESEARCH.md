# Phase 98.1: Production Infrastructure Staging - Research

**Researched:** 2026-03-15
**Domain:** Production infrastructure (Railway, Helius, DNS, monitoring)
**Confidence:** HIGH

## Summary

This phase provisions all production infrastructure before mainnet launch day: Helius mainnet RPC + webhooks, Railway mainnet services (web + crank + Postgres), fraudsworth.fun domain binding, environment-driven CSP, and UptimeRobot monitoring. The project already has extensive devnet infrastructure code that is production-ready -- the work is primarily configuration, environment setup, and one code change (dynamic CSP).

The critical discovery is that **GoDaddy does not support CNAME flattening for apex domains**. To make `fraudsworth.fun` (not just `www.fraudsworth.fun`) work with Railway, the domain's nameservers must be moved to Cloudflare. This is a mandatory prerequisite and the longest lead-time item in the phase.

The second key finding is that `webhook-manage.ts` uses `https://api.helius.xyz/v0` as the Helius API base, but Helius's current official documentation shows `https://api-mainnet.helius-rpc.com/v0` as the mainnet endpoint. The script also hardcodes `rawDevnet` webhook type -- mainnet requires `raw` (not `rawDevnet`). Both need parameterization.

**Primary recommendation:** Start with Cloudflare nameserver migration (DNS propagation takes up to 72 hours), then Railway services + Helius in parallel, then domain binding + webhooks + monitoring last.

## Architecture Patterns

### Railway Service Layout

```
Railway Project: Dr Fraudsworth
├── devnet-web (existing)         # Devnet frontend -- stays running
├── devnet-crank (existing)       # Devnet crank -- stays running
├── devnet-postgres (existing)    # Devnet database -- stays running
├── mainnet-web (NEW)             # Mainnet frontend service
├── mainnet-crank (NEW)           # Mainnet crank service
└── mainnet-postgres (NEW)        # Mainnet database (Railway Postgres plugin)
```

**Key pattern:** Mainnet services are separate Railway services in the SAME project. This gives:
- Independent deployments and restarts (crank crash does not affect frontend)
- Separate env vars per service (no env var collision)
- Private networking between services within the project
- Separate Postgres instance = complete data isolation

### Environment Variable Strategy

Each Railway service gets its own env vars. The `.env.mainnet` files in the repo are REFERENCE TEMPLATES only -- Railway reads from its dashboard, not `.env` files.

**Mainnet-web service env vars:**
- `DATABASE_URL` (auto-injected from mainnet-postgres)
- `HELIUS_RPC_URL` (mainnet Helius with API key)
- `HELIUS_RPC_URL_FALLBACK` (second mainnet Helius key)
- `HELIUS_WEBHOOK_SECRET` (shared secret for webhook auth)
- `SENTRY_DSN` / `NEXT_PUBLIC_SENTRY_DSN` (same DSN as devnet)
- `NEXT_PUBLIC_CLUSTER=mainnet`
- `NEXT_PUBLIC_SITE_MODE=launch` (bonding curve phase default)
- `NEXT_PUBLIC_DOCS_URL=https://fraudsworth.fun/docs`
- All program IDs / mint addresses as `CHANGE_ME_MAINNET` placeholders

**Mainnet-crank service env vars:**
- `CLUSTER_URL` (mainnet Helius RPC)
- `WALLET_KEYPAIR` (JSON byte array -- NOT a file path on Railway)
- `PDA_MANIFEST` (full JSON string)
- `CARNAGE_WSOL_PUBKEY`
- `HEALTH_PORT=8080`
- All as `CHANGE_ME_MAINNET` placeholders initially

### DNS Architecture

```
fraudsworth.fun (apex)
  └── CNAME → [railway-provided-value].up.railway.app  (via Cloudflare CNAME flattening)

www.fraudsworth.fun
  └── CNAME → fraudsworth.fun  (redirect to apex via Cloudflare Page Rule or Railway)
```

**Cloudflare settings:**
- SSL/TLS mode: **Full** (NOT Full Strict -- Railway uses its own *.up.railway.app cert)
- Proxy status: **Proxied** (orange cloud) for apex CNAME flattening
- Page Rule: `www.fraudsworth.fun/*` -> 301 redirect to `https://fraudsworth.fun/$1`

### Pattern: Environment-Driven CSP

The CSP in `next.config.ts` is currently hardcoded with devnet URLs. For mainnet, it must dynamically include only cluster-appropriate URLs.

```typescript
// next.config.ts -- dynamic CSP based on NEXT_PUBLIC_CLUSTER
const cluster = process.env.NEXT_PUBLIC_CLUSTER || 'devnet';
const isMainnet = cluster === 'mainnet';

const heliusRpc = isMainnet
  ? 'https://mainnet.helius-rpc.com wss://mainnet.helius-rpc.com'
  : 'https://devnet.helius-rpc.com wss://devnet.helius-rpc.com';

// api.helius.xyz is NOT needed in CSP -- webhook management is server-side only
// api-devnet.helius-rpc.com only needed on devnet for legacy client-side calls
const heliusApi = isMainnet ? '' : 'https://api-devnet.helius-rpc.com';

const cspHeader = `
  ...
  connect-src 'self' wss://relay.walletconnect.com ... ${heliusRpc} ${heliusApi} https://*.ingest.sentry.io https://*.ingest.us.sentry.io;
  ...
`;
```

```typescript
// Webhook type differs

```

### Anti-Patterns to Avoid

- **Hardcoded devnet URLs in CSP:** Current state. Must be environment-driven.
- **Using `rawDevnet` webhook type for mainnet:** Will silently fail or return empty data.
- **Using `api.helius.xyz` for mainnet webhook API:** May work (legacy endpoint) but official mainnet endpoint is `api-mainnet.helius-rpc.com`.
- **Setting up DNS before Cloudflare migration:** GoDaddy apex CNAME will fail silently.
- **Sharing Postgres between devnet and mainnet:** Data contamination risk. Separate instances required.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Apex domain CNAME | A-record workarounds | Cloudflare CNAME flattening | RFC-compliant, automatic, free |
| SSL certificates | Manual cert management | Railway auto-provisioned Let's Encrypt | Automatic renewal every 90 days |
| Uptime monitoring | Custom health check cron | UptimeRobot free tier | 50 monitors, email alerts, zero maintenance |
| DNS propagation checks | Manual dig/nslookup polling | Cloudflare dashboard + Railway domain verification | Both show real-time status |

## Common Pitfalls

### Pitfall 1: GoDaddy Apex CNAME Limitation
**What goes wrong:** Adding a CNAME record for `@` (apex) in GoDaddy fails silently or creates an A record instead.
**Why it happens:** RFC 1034 prohibits CNAME records at zone apex. GoDaddy follows the RFC strictly with no CNAME flattening support.
**How to avoid:** Move nameservers to Cloudflare (free tier). Keep domain registration at GoDaddy -- only NS records change.
**Warning signs:** Railway domain verification hangs at "Waiting for DNS" for apex domain while www works fine.

### Pitfall 2: Helius Mainnet Webhook Type
**What goes wrong:** Webhooks registered with `rawDevnet` on mainnet deliver nothing.
**Why it happens:** `rawDevnet` explicitly targets Solana devnet. Mainnet requires `raw` type.
**How to avoid:** Parameterize `webhook-manage.ts` with cluster-aware webhook type selection.
**Warning signs:** Webhook registered successfully (200 OK) but no events delivered after mainnet programs deploy.

### Pitfall 3: Helius API Endpoint for Mainnet
**What goes wrong:** Webhook CRUD calls to `api.helius.xyz` may work but is the legacy/devnet endpoint.
**Why it happens:** Helius official docs now show `api-mainnet.helius-rpc.com` as the mainnet endpoint.
**How to avoid:** Use cluster-aware base URL in `webhook-manage.ts`.
**Warning signs:** API calls may work (backwards compatibility) but could be deprecated.

### Pitfall 4: Cloudflare SSL Mode
**What goes wrong:** Pages show SSL errors or redirect loops after Cloudflare setup.
**Why it happens:** Cloudflare default is "Flexible" SSL, which sends HTTP to Railway. Railway expects HTTPS.
**How to avoid:** Set Cloudflare SSL/TLS to **Full** mode (not Full Strict, not Flexible).
**Warning signs:** ERR_TOO_MANY_REDIRECTS in browser, or mixed content warnings.

### Pitfall 5: DNS Propagation Delay
**What goes wrong:** Domain doesn't resolve immediately after NS or CNAME changes.
**Why it happens:** DNS TTL propagation can take up to 72 hours globally (typically 1-24 hours).
**How to avoid:** Change nameservers to Cloudflare FIRST (longest lead time). Do Railway service setup in parallel.
**Warning signs:** `dig fraudsworth.fun` still shows GoDaddy NS records hours after change.

### Pitfall 6: Railway Custom Domain Plan Limit
**What goes wrong:** Can't add custom domain to Railway service.
**Why it happens:** Trial plan = 1 domain, Hobby = 2 per service, Pro = 20.
**How to avoid:** Ensure Railway plan supports needed domain count (apex + www = 2 domains minimum).
**Warning signs:** Railway UI shows "upgrade plan" when adding second domain.

### Pitfall 7: NEXT_PUBLIC_ Env Vars Are Build-Time
**What goes wrong:** Changing `NEXT_PUBLIC_CLUSTER` in Railway dashboard doesn't take effect.
**Why it happens:** Next.js inlines `NEXT_PUBLIC_*` variables at build time, not runtime.
**How to avoid:** After changing `NEXT_PUBLIC_*` vars, trigger a redeploy (Railway rebuilds automatically on env var change).
**Warning signs:** CSP still contains devnet URLs after changing cluster env var.

### Pitfall 8: Webhook URL Must Match Exactly
**What goes wrong:** Helius webhook delivers to wrong URL or fails TLS.
**Why it happens:** Webhook URL must be HTTPS, publicly reachable, and match the exact domain Railway serves.
**How to avoid:** Register webhook with `https://fraudsworth.fun/api/webhooks/helius` only AFTER domain is verified and SSL is active.
**Warning signs:** Helius webhook test returns connection timeout or TLS error.

## Open Questions

1. **Helius API endpoint backwards compatibility**
   - What we know: `api.helius.xyz` is the legacy endpoint; `api-mainnet.helius-rpc.com` is in current docs
   - What's unclear: Whether `api.helius.xyz` still works for mainnet webhook CRUD
   - Recommendation: Use `api-mainnet.helius-rpc.com` for mainnet (official), keep `api.helius.xyz` for devnet

2. **Crank health endpoint public URL**
   - What we know: Crank runs on port 8080 internally; Railway provides a public domain
   - What's unclear: Whether crank service should get a custom domain or use Railway's auto-generated URL
   - Recommendation: Use Railway auto-generated URL for crank health monitoring (no public traffic expected)

3. **Docs iframe domain in CSP**
   - What we know: CSP currently allows `docs-drfraudsworth.up.railway.app` for docs iframe
   - What's unclear: Whether docs will be served from `fraudsworth.fun/docs` (same origin) or remain on Railway subdomain
   - Recommendation: Keep both in CSP for now; `'self'` covers same-origin, Railway URL covers iframe fallback
