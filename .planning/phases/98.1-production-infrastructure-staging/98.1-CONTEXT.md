<decisions>
## Implementation Decisions

### Railway Prod Environment
- Same Railway project as devnet, with separate services (mainnet-web, mainnet-crank)
- Crank runner is a dedicated Railway service (same pattern as devnet) — independent restarts, own health check, crank crashes don't affect frontend
- Separate Postgres database (Railway auto-provisions) — complete data isolation from devnet, clean slate for mainnet candle/event data
- Services set up during this phase with `CHANGE_ME_MAINNET` placeholders for secrets/TBD values — real values filled during Phase 98 checklist execution
- Devnet services stay running alongside mainnet services

### Domain & DNS (fraudsworth.fun)
- Domain already owned, registered at GoDaddy
- CNAME record pointing to Railway mainnet-web service
- Docs at same domain `/docs` path (not a subdomain) — Nextra embedded via existing iframe pattern
- Both apex (fraudsworth.fun) and www.fraudsworth.fun configured — www redirects to apex
- Railway custom domain binding with automatic SSL

### Helius Mainnet RPC & Webhooks
- Paid Helius plan already active — generate mainnet API key during this phase
- Two separate Helius API keys: primary (`HELIUS_RPC_URL`) + fallback (`HELIUS_RPC_URL_FALLBACK`) — same dual-key failover pattern from Phase 84 (H047)
- Separate `HELIUS_API_KEY` for Helius REST API (webhook management, not RPC)
- Register mainnet webhooks during this phase pointing to `fraudsworth.fun/api/webhooks/helius` — validates URL + secret work end-to-end before deploy pressure. Webhooks idle until programs are deployed
- Environment-driven CSP in `next.config.ts` — dynamically includes only cluster-appropriate Helius URLs based on `NEXT_PUBLIC_CLUSTER` env var (tightest security, allows only what's needed)

### Sentry & Monitoring
- Same Sentry project as devnet, differentiated by `cluster=mainnet` tag — Phase 92 already set up cluster tagging via `NEXT_PUBLIC_CLUSTER` env var
- UptimeRobot free tier for uptime monitoring:
  - Monitor 1: `fraudsworth.fun` (frontend health) — 5-min interval
  - Monitor 2: `fraudsworth.fun/api/health` (API health) — 5-min interval
  - Monitor 3: Crank `/health` endpoint (crank liveness) — 5-min interval, alerts on "halted" status
- Email alerts on downtime from UptimeRobot

### Claude's Discretion
- Exact Railway service naming convention
- Order of operations for DNS propagation vs Railway domain binding
- Whether to test webhook delivery with a curl before registering with Helius
- UptimeRobot alert threshold configuration (how many failures before alerting)

</decisions>

### Reusable Assets
- `app/app/api/rpc/route.ts`: RPC proxy with dual-endpoint failover (H047) — mainnet just needs env vars changed

- `scripts/crank/crank-runner.ts`: Crank with health endpoint, circuit breaker, spending cap — production-ready
- `railway.toml`: Build/start/health config — needs mainnet service variant

- CSP headers in `app/next.config.ts` (currently hardcoded devnet URLs — will be made dynamic)

- `app/next.config.ts`: CSP `connect-src` directive needs environment-driven Helius URLs

- GoDaddy DNS → Railway CNAME for custom domain
