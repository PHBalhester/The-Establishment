    ```typescript
    import * as anchor from "@coral-xyz/anchor";
    import { BN, Program, AnchorProvider } from "@coral-xyz/anchor";

    import { E2EUser } from "./user-setup";
    import { executeSolBuySwap, resolveHookAccounts } from "./swap-flow";
    ```

    **stakePROFIT(provider, programs, manifest, user, amount, logger):**
    - Resolve Transfer Hook remaining_accounts for PROFIT transfer (user -> stake_vault):
      - Source: user.profitAccount
      - Mint: manifest.mints.PROFIT
      - Dest: manifest.pdas.StakeVault
      - Authority: user.keypair.publicKey
      - Amount: BigInt(amount)
    - Build `programs.staking.methods.stake(new BN(amount))` instruction with accounts:
      - user: user.keypair.publicKey
      - stakePool: manifest.pdas.StakePool
      - userStake: derive PDA with seeds ["user_stake", user.keypair.publicKey.toBuffer()] from Staking program
      - userTokenAccount: user.profitAccount
      - stakeVault: manifest.pdas.StakeVault
      - profitMint: manifest.mints.PROFIT
      - tokenProgram: TOKEN_2022_PROGRAM_ID
      - systemProgram: SystemProgram.programId
    - Add `.remainingAccounts(hookAccounts)` and `.signers([user.keypair])`
    - The `userStake` PDA may not exist yet -- Anchor's `init_if_needed` should create it. If the account struct requires `init_if_needed`, the instruction handles creation automatically.
    - Capture before/after PROFIT balance of user and StakeVault
    - Log TX signature and balance changes
    - Wait 200ms after TX for rate limiting
    - Amount: stake 10 PROFIT = 10_000_000 raw units (6 decimals). Must be >= MINIMUM_STAKE (1_000_000 raw = 1 PROFIT)

    **claimYield(provider, programs, manifest, user, logger):**
    - Derive userStake PDA: `["user_stake", user.keypair.publicKey.toBuffer()]`
    - Capture pre-claim SOL balance of user
    - Build `programs.staking.methods.claim()` instruction with accounts:
      - user: user.keypair.publicKey
      - stakePool: manifest.pdas.StakePool
      - userStake: derived PDA
      - escrowVault: manifest.pdas.EscrowVault
    - Sign with user keypair
    - Capture post-claim SOL balance
    - Calculate yield received = postBalance - preBalance + txFee (approximately)
    - Log result: yield amount, TX signature
    - Note: claim transfers native SOL from escrow to user -- no token transfer, no hook accounts needed

    **Export both functions** -- they are used by Task 1b's orchestrator.

    **Important patterns:**
    - Use `new anchor.BN()` for amounts (not raw numbers)
    - All manifest values need `new PublicKey()` wrapping
    - 200ms sleep between RPC calls for rate limiting
    - Wrap each operation in try/catch -- log failures and continue

  <action>
    Add the multi-epoch VRF cycling and staking orchestrator functions to `scripts/e2e/lib/staking-flow.ts` (same file as Task 1a).

    **runMultiEpochCycling(provider, programs, manifest, logger, epochCount):**
    - `epochCount` defaults to 3 (minimum for E2E-04)
    - `advanceEpochWithVRF` handles Switchboard setup INTERNALLY (it calls `sb.getProgramId`, `Program.fetchIdl`, `sb.getDefaultQueue` inside the function). You do NOT need to set up Switchboard externally.
    - The function signature is: `advanceEpochWithVRF(provider: AnchorProvider, epochProgram: Program, accounts: VRFAccounts): Promise<EpochTransitionResult>`
    - Construct `VRFAccounts` from the manifest:
      ```typescript
      const vrfAccounts: VRFAccounts = {
        epochStatePda: new PublicKey(manifest.pdas.EpochState),
        treasuryPda: provider.wallet.publicKey,  // Treasury placeholder
        stakingAuthorityPda: new PublicKey(manifest.pdas.StakingAuthority),
        stakePoolPda: new PublicKey(manifest.pdas.StakePool),
        stakingProgramId: new PublicKey(manifest.programs.Staking),
        carnageFundPda: new PublicKey(manifest.pdas.CarnageFund),
      };
      ```
    - Reference `scripts/vrf/devnet-vrf-validation.ts` lines 108-115 for the exact VRFAccounts construction pattern
    - For each epoch transition:
      1. Read EpochState snapshot BEFORE transition
      2. Call `advanceEpochWithVRF(provider, programs.epochProgram, vrfAccounts)`
         - This handles the full 3-TX VRF flow: create randomness -> commit+trigger -> reveal+consume
         - The consume_randomness instruction calls Staking's update_cumulative via CPI, so rewards are finalized automatically
      3. Read EpochState snapshot AFTER transition
      4. Log transition result: epoch number, old/new tax rates, cheapSide, flip status, TX signatures
      5. Verify tax rates are in spec bands (100-400 bps low, 1100-1400 bps high)
    - Between epochs, run a small swap to generate tax revenue for staking yield:
      - Call `executeSolBuySwap` (already imported from `./swap-flow`)
      - This generates SOL tax that flows to the staking escrow via `deposit_rewards`
      - The tax revenue becomes claimable after the next epoch's `update_cumulative`
    - Log each epoch's results incrementally
    - The `update_cumulative` is called automatically during `consume_randomness` CPI -- no separate call needed (see programs/epoch-program/src/instructions/consume_randomness.rs)

    **runStakingFlow(provider, programs, manifest, user, logger):**
    - Orchestrator for the full staking lifecycle:
      1. Stake 10 PROFIT (via stakePROFIT from Task 1a)
      2. Run a swap to generate tax revenue (call executeSolBuySwap from swap-flow.ts)
      3. Advance 3 epochs (via runMultiEpochCycling)
      4. Claim yield (via claimYield from Task 1a)
      5. Verify non-zero yield received
      6. Log overall staking flow result
    - Return boolean success

    **Export both functions** -- runStakingFlow is imported by the orchestrator in Task 2.
