    ```typescript
    import { executeSolBuySwap, resolveHookAccounts } from "./swap-flow";
    ```

    - `resolveHookAccounts` is used for Transfer Hook remaining_accounts on Carnage token transfers

    **Also import VRF helpers from Plan 02's dependencies:**
    ```typescript
    import { advanceEpochWithVRF, VRFAccounts, sleep } from "../../vrf/lib/vrf-flow";
    import { readEpochState } from "../../vrf/lib/epoch-reader";
    import { E2ELogger, LogEntry } from "./e2e-logger";
    ```

    1. **captureCarnageSnapshot(connection, programs, manifest)** helper:
       - Read and return:
         - CarnageFundState account data (carnage_state PDA): total collected, crime/fraud held amounts, last action
         - CarnageSolVault balance (SOL lamports)
         - CarnageCrimeVault token balance (if exists)
         - CarnageFraudVault token balance (if exists)
         - All 4 pool states (reserves for CRIME/SOL, FRAUD/SOL, CRIME/PROFIT, FRAUD/PROFIT)
       - Use `programs.epochProgram.account.carnageFundState.fetch(carnageFundPda)` for typed account
       - Use `connection.getBalance()` for SOL vault
       - Use `connection.getTokenAccountBalance()` for token vaults (Token-2022)
       - Wrap each in try/catch -- some accounts may not exist yet or may be empty
       - Return typed snapshot object

    2. **testForcedCarnage(provider, programs, manifest, user, logger)** function:
       - **This is EXPECTED TO FAIL due to known Carnage bugs.** The function should handle failure gracefully.
       - First, check if `carnage_pending` is true on EpochState:
         - Read EpochState via `readEpochState`
         - If carnage_pending is false, log "No Carnage pending -- skipping forced test (need VRF byte 3 < 11 first)"
         - This test can only run if a previous epoch transition set carnage_pending = true
       - If carnage_pending is true:
         a. Capture pre-Carnage snapshot
         b. Build `execute_carnage_atomic` instruction. Reference the ExecuteCarnageAtomic accounts struct:
            - caller: user.keypair.publicKey (or provider.wallet.publicKey -- permissionless)
            - epochState: manifest.pdas.EpochState
            - carnageState: manifest.pdas.CarnageFund
            - carnageSigner: manifest.pdas.CarnageSigner
            - carnageSolVault: manifest.pdas.CarnageSolVault
            - carnageCrimeVault: manifest.pdas.CarnageCrimeVault (token account)
            - carnageFraudVault: manifest.pdas.CarnageFraudVault (token account)
            - Plus: target pool, pool vaults, mints, token programs
            - Plus: Tax Program accounts for swap_exempt CPI
            - Plus: AMM Program accounts
            - Plus: Transfer Hook remaining_accounts
         c. This is complex (~25+ accounts). Build carefully referencing the Rust struct.
         d. Try to send the TX
         e. **Expected failures (log as "known_issue", NOT "fail"):**
            - held_amount stores SOL lamports not token count (audit H041, H042, H063, H089, H094)
            - Fallback discriminator mismatch (audit H018, H052, H058, H099)
            - Missing swap_authority in fallback (audit H019, H059)
         f. Capture post-Carnage snapshot (even on failure -- verify no state corruption)
         g. Log result with full error message, TX signature (if any), before/after snapshots
       - Return { tested: boolean, success: boolean, knownIssue: boolean, details: string }

    3. **testNaturalCarnage(provider, programs, manifest, user, logger, maxEpochs)** function:
       - Cycle epochs until VRF byte 3 < 11 triggers Carnage naturally
       - `maxEpochs` defaults to 50 (~4 hours at 5 min/epoch)
       - For each epoch:
         a. Read EpochState BEFORE transition
         b. Advance epoch with VRF (using advanceEpochWithVRF)
         c. Read EpochState AFTER transition
         d. Check `epoch_state.carnage_pending` -- if true, Carnage was triggered!
         e. Log epoch result: epoch number, VRF bytes, carnage_pending, tax rates
         f. If Carnage triggered, attempt `execute_carnage_atomic` (call testForcedCarnage)
         g. If Carnage executed (or failed with known issue), break the loop
         h. Run a small swap between epochs to keep generating tax revenue
       - If maxEpochs reached without Carnage trigger:
         - Log "Natural Carnage not triggered in {maxEpochs} epochs (probability ~4.3%/epoch, expected ~1 in 23 epochs)"
         - This is NOT a failure -- it's a probabilistic outcome. Log as "skip" status.
       - Return { triggered: boolean, epoch: number, details: string }

    4. **postCarnageHealthCheck(provider, programs, manifest, user, logger)** function:
       - After any Carnage attempt (success or failure), verify the protocol is still operational
       - Execute a simple SOL buy swap using `executeSolBuySwap` from swap-flow.ts
       - If swap succeeds: log "Post-Carnage health check PASSED -- protocol operational"
       - If swap fails: log "Post-Carnage health check FAILED" with error details
       - This proves the protocol survives Carnage events (or attempted Carnage)

    5. **runCarnageFlow(provider, programs, manifest, user, logger)** orchestrator:
       - Step 1: Test forced Carnage (if carnage_pending is true from a previous epoch)
       - Step 2: Test natural Carnage (run epochs until triggered, or hit max)
       - Step 3: Post-Carnage health check
       - Log overall Carnage flow result
       - Return boolean success

    **Wire into devnet-e2e-validation.ts:**
    - Import `runCarnageFlow` from `./lib/carnage-flow`
    - Add after staking flow section:
      ```typescript
      // ─── Phase 3: Carnage Trigger Testing (E2E-05) ──────────────
      logger.log({ ... phase: "carnage", message: "Starting Carnage trigger testing" });
      const carnageSuccess = await runCarnageFlow(provider, programs, manifest, user, logger);
      ```
    - After ALL flows complete, generate the final report and write to `Docs/E2E_Devnet_Test_Report.md`

    **Important notes:**
    - Carnage has KNOWN BUGS -- the forced test WILL likely fail. This is expected. Log as "known_issue" not "fail".
    - Natural Carnage has ~4.3% probability per epoch. May need up to 50 epochs. Script should be robust for long unattended runs.
    - The `execute_carnage_atomic` instruction is complex with ~25 accounts. Reference the Rust struct carefully.
    - After each Carnage-related operation, add 200ms sleep for rate limiting.
    - The user must have enough SOL for the extended run (VRF transitions cost ~0.01 SOL each). With 50 epochs, that's ~0.5 SOL plus swap costs.
    - If the wallet runs low on SOL during natural Carnage cycling, log the balance warning and stop early.
