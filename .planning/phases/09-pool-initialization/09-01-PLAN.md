Build both instruction handlers and wire them into the program module.

**1. Create `instructions/initialize_admin.rs`** (NEW file):

Accounts struct `InitializeAdmin`:
- `authority: Signer<'info>` (mut, payer)
- `admin_config: Account<'info, AdminConfig>` (init, payer = authority, space = 8 + AdminConfig::INIT_SPACE, seeds = [ADMIN_SEED], bump)
- `program: Program<'info, crate::program::Amm>` (constraint: `program.programdata_address()? == Some(program_data.key())`)
- `program_data: Account<'info, ProgramData>` (constraint: `program_data.upgrade_authority_address == Some(authority.key())`)
- `system_program: Program<'info, System>`

Handler `pub fn handler(ctx: Context<InitializeAdmin>, admin: Pubkey) -> Result<()>`:
1. Set `admin_config.admin = admin`
2. Set `admin_config.bump = ctx.bumps.admin_config`
3. Return Ok

**Why ProgramData pattern:** Per 09-CONTEXT.md, only the program upgrade authority can call initialize_admin. This allows setting a multisig as admin from day one. The ProgramData account stores the upgrade authority and Anchor provides `programdata_address()` to link it to the program.

**2. Create `instructions/initialize_pool.rs`** (NEW file):

This is the most complex file. Follow 09-RESEARCH.md Patterns 2-5 carefully.

Accounts struct `InitializePool`:
- `payer: Signer<'info>` (mut) -- pays for account creation rent
- `admin_config: Account<'info, AdminConfig>` (seeds = [ADMIN_SEED], bump = admin_config.bump, has_one = admin @ AmmError::Unauthorized)
- `admin: Signer<'info>` -- must match admin_config.admin
- `pool: Account<'info, PoolState>` (init, payer = payer, space = 8 + PoolState::INIT_SPACE, seeds = [POOL_SEED, mint_a.key().as_ref(), mint_b.key().as_ref()], bump)
- `vault_a: InterfaceAccount<'info, TokenAccount>` (init, payer = payer, token::mint = mint_a, token::authority = pool, token::token_program = token_program_a, seeds = [VAULT_SEED, pool.key().as_ref(), VAULT_A_SEED], bump)
- `vault_b: InterfaceAccount<'info, TokenAccount>` (init, payer = payer, token::mint = mint_b, token::authority = pool, token::token_program = token_program_b, seeds = [VAULT_SEED, pool.key().as_ref(), VAULT_B_SEED], bump)
- `mint_a: InterfaceAccount<'info, Mint>` -- first mint (must be < mint_b lexicographically)
- `mint_b: InterfaceAccount<'info, Mint>` -- second mint
- `source_a: InterfaceAccount<'info, TokenAccount>` (mut) -- deployer's source for token A
- `source_b: InterfaceAccount<'info, TokenAccount>` (mut) -- deployer's source for token B
- `token_program_a: Interface<'info, TokenInterface>` -- token program for side A
- `token_program_b: Interface<'info, TokenInterface>` -- token program for side B
- `system_program: Program<'info, System>`

**IMPORTANT constraints to add:**
- Validate mint ownership matches token program: `constraint = mint_a.to_account_info().owner == token_program_a.key @ AmmError::InvalidTokenProgram` (same for mint_b/token_program_b). This prevents passing the wrong token program for a mint (Pitfall 2 from 09-RESEARCH.md).

Handler `pub fn handler(ctx: Context<InitializePool>, lp_fee_bps: u16, amount_a: u64, amount_b: u64) -> Result<()>`:

1. **Validate canonical mint ordering**: `require!(ctx.accounts.mint_a.key() < ctx.accounts.mint_b.key(), AmmError::MintsNotCanonicallyOrdered)`. Per 09-RESEARCH.md Pattern 3 note: true auto-sort is not feasible with Anchor's PDA derivation (seeds are evaluated at deserialization from account fields). Caller must pass mints in order; we validate and give a clear error.

2. **Validate non-duplicate mints**: `require!(ctx.accounts.mint_a.key() != ctx.accounts.mint_b.key(), AmmError::DuplicateMints)`

3. **Validate non-zero seed amounts**: `require!(amount_a > 0 && amount_b > 0, AmmError::ZeroSeedAmount)`

4. **Infer pool type** from token programs (09-RESEARCH.md Pattern 4):
   - Both T22 -> `PoolType::PureT22Pool`
   - One T22 + one SPL -> `PoolType::MixedPool`
   - Neither valid -> `AmmError::InvalidTokenProgram`

5. **Transfer initial liquidity** into vaults using `token_interface::transfer_checked` (09-RESEARCH.md Pattern 5):
   - Transfer `amount_a` from `source_a` to `vault_a` via `token_program_a`, using `admin` as authority, with `mint_a.decimals`
   - Transfer `amount_b` from `source_b` to `vault_b` via `token_program_b`, using `admin` as authority, with `mint_b.decimals`

6. **Populate pool state**:
   - `pool_type` = inferred type
   - `mint_a`, `mint_b` = mint keys
   - `vault_a`, `vault_b` = vault keys
   - `reserve_a` = amount_a, `reserve_b` = amount_b
   - `lp_fee_bps` = passed parameter
   - `initialized` = true
   - `bump` = ctx.bumps.pool
   - `vault_a_bump` = ctx.bumps.vault_a
   - `vault_b_bump` = ctx.bumps.vault_b
   - `token_program_a`, `token_program_b` = program keys

7. **Emit PoolInitializedEvent** with all pool data. Map PoolType to u8 for the event.
