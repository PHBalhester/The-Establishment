**2. Create test file structure** in `programs/amm/tests/test_pool_initialization.rs`:

Build these reusable helpers at the top of the file:

- `setup_litesvm() -> LiteSVM`: Create a LiteSVM instance, deploy the AMM program from the compiled .so file. Load the program with upgrade authority set to a known keypair so we can test the ProgramData constraint.

- `create_spl_mint(svm, authority, decimals) -> Pubkey`: Create a standard SPL Token mint (for WSOL-like tokens). Use `spl_token::instruction::initialize_mint2`.

- `create_t22_mint(svm, authority, decimals) -> Pubkey`: Create a Token-2022 mint WITHOUT transfer hook extensions. Per 09-RESEARCH.md Pitfall 7: Phase 9 tests use T22 mints without hooks because transfer hook testing is Phase 10+. Use `spl_token_2022::instruction::initialize_mint2`.

- `create_token_account(svm, mint, owner, token_program) -> Pubkey`: Create a token account for the deployer's source accounts.

- `mint_tokens(svm, mint, dest, amount, authority, token_program)`: Mint tokens to the deployer's source accounts.

- `initialize_admin_ix(program_id, authority, admin) -> Instruction`: Build the initialize_admin instruction with correct accounts (authority, admin_config PDA, program, program_data, system_program).

- `initialize_pool_ix(program_id, payer, admin, mint_a, mint_b, source_a, source_b, token_program_a, token_program_b, lp_fee_bps, amount_a, amount_b) -> Instruction`: Build the initialize_pool instruction with all required accounts. Compute PDAs for pool, vault_a, vault_b using `Pubkey::find_program_address` with the correct seeds from constants.

- `setup_pool_test(pool_config) -> TestContext`: A higher-level helper that creates the SVM, deploys the program, creates an admin, creates the mints and source accounts based on pool configuration (mixed vs pure T22), and mints tokens. Returns a struct with all the keypairs and pubkeys needed for the test.

**Pool configurations to support:**
1. Mixed pool (T22 mint + SPL mint) -- represents CRIME/SOL or FRAUD/SOL
2. Pure T22 pool (T22 mint + T22 mint) -- represents CRIME/PROFIT or FRAUD/PROFIT

**Key detail for litesvm program deployment:** The compiled program .so file is at `target/deploy/amm.so` after `anchor build`. Load it with `svm.add_program_from_file(program_id, "target/deploy/amm.so")` or equivalent litesvm API. The program's upgrade authority must be set to a known keypair for the ProgramData constraint tests. Check litesvm docs/API for how to set upgrade authority -- this may require deploying via BPF loader upgradeable manually, or litesvm may provide a helper.

**CRITICAL:** If litesvm's API has changed or doesn't match expectations, adapt. The research flagged this as MEDIUM confidence for testing patterns. Check the actual litesvm crate API before writing. Use `cargo doc` or inspect the crate source if needed. Do NOT guess at API methods -- verify them.

**Do NOT:**
- Use solana-test-validator (too slow for integration tests)
- Create T22 mints WITH transfer hook extensions (Phase 10+ concern per 09-RESEARCH.md Pitfall 7)
- Hardcode PDA addresses -- always derive with `Pubkey::find_program_address`

Write the full test suite covering every Phase 9 requirement. Build on the helpers from Task 1.

**Happy path tests (POOL-01 through POOL-06, SWAP-10):**

1. **`test_initialize_admin_success`**: Call initialize_admin with program upgrade authority. Verify AdminConfig PDA exists and stores the correct admin pubkey and bump.

2. **`test_initialize_mixed_pool`** (CRIME/SOL pattern): Create a T22 mint and an SPL mint. Ensure T22 mint pubkey < SPL mint pubkey (swap if needed by generating new keypairs until canonical order is achieved). Call initialize_pool with T22 as mint_a, SPL as mint_b, token_program_a = Token-2022, token_program_b = SPL Token, lp_fee_bps = 100, amounts = (1_000_000, 1_000_000). Verify:
   - Pool state account exists at expected PDA
   - pool_type == MixedPool
   - mint_a, mint_b correct
   - vault_a, vault_b at expected PDA addresses
   - reserve_a == amount_a, reserve_b == amount_b
   - lp_fee_bps == 100
   - initialized == true
   - vault_a has pool PDA as authority (POOL-03)
   - vault_b has pool PDA as authority (POOL-03)
   - vault_a balance == amount_a
   - vault_b balance == amount_b
   - token_program_a == Token-2022 program ID
   - token_program_b == SPL Token program ID

3. **`test_initialize_pure_t22_pool`** (CRIME/PROFIT pattern): Two T22 mints, lp_fee_bps = 50. Same verifications as above but pool_type == PureT22Pool and both token programs are Token-2022.

4. **`test_initialize_mixed_pool_second_config`** (FRAUD/SOL pattern): Another mixed pool with different mints. Verifies the AMM correctly handles multiple pools.

5. **`test_initialize_pure_t22_pool_second_config`** (FRAUD/PROFIT pattern): Another pure T22 pool with different mints.

6. **`test_pool_initialized_event`** (SWAP-10): After initializing a pool, inspect the transaction logs/events to verify PoolInitializedEvent was emitted with correct data. Use litesvm's transaction metadata to find the event. If litesvm doesn't expose Anchor events easily, verify by checking the transaction logs contain the event discriminator and data. At minimum, verify the transaction succeeded (event emission happens inside the handler, so success implies emission).

**Negative tests (POOL-07, POOL-08, and validation):**

7. **`test_initialize_admin_non_upgrade_authority_rejected`** (POOL-07 prerequisite): Call initialize_admin with a signer that is NOT the program upgrade authority. Must fail with a constraint error.

8. **`test_initialize_pool_non_admin_rejected`** (POOL-07): Call initialize_pool with a signer that doesn't match admin_config.admin. Must fail with `AmmError::Unauthorized`.

9. **`test_duplicate_pool_rejected`** (POOL-08): Initialize a pool, then try to initialize another pool with the same mint pair. Must fail because the pool PDA already exists (Anchor `init` constraint rejects already-initialized accounts).

10. **`test_non_canonical_order_rejected`**: Pass mint_a > mint_b. Must fail with `AmmError::MintsNotCanonicallyOrdered`.

11. **`test_duplicate_mints_rejected`**: Pass the same mint as both mint_a and mint_b. Must fail with `AmmError::DuplicateMints`.

12. **`test_zero_seed_amount_rejected`**: Pass amount_a = 0 or amount_b = 0. Must fail with `AmmError::ZeroSeedAmount`.

13. **`test_token_program_mismatch_rejected`**: Pass Token-2022 program ID for an SPL Token mint (or vice versa). Must fail with `AmmError::InvalidTokenProgram`.

**Test implementation notes:**
- For each test, use the helpers from Task 1 to reduce boilerplate.
- For negative tests, assert that the transaction fails and (where possible) verify the specific error code.
- For PDA verification, compute expected PDAs with `Pubkey::find_program_address` and compare against pool state fields.
- For vault authority checks, read the vault token account data and verify the `owner` (authority) field matches the pool PDA.
- Canonical ordering trick: Generate keypairs in a loop until you get the ordering you need, OR simply check which pubkey is smaller and assign mint_a/mint_b accordingly.

**If any test infrastructure issue arises** (litesvm API incompatibility, program loading issues, T22 account creation issues): Document the issue clearly in comments, try alternative approaches (e.g., using `solana_program_test` instead of litesvm if litesvm doesn't work), and ensure at least the happy path tests pass. Flag any issues for the SUMMARY.

**Do NOT:**
- Skip negative tests -- they are as important as happy path for security
- Test transfer hook behavior (Phase 10+)
- Use `assert!` without messages -- always include descriptive failure messages
