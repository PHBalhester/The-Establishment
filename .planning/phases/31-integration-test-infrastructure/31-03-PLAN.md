**Test 1: SOL Buy Swap (Tax -> AMM -> Token-2022 -> Transfer Hook)**

This test proves the most complex CPI chain in the protocol works end-to-end.

Steps:
1. Get the trader wallet from protocol.wallets.trader
2. Determine which SOL pool to use (CRIME/SOL). Find which pool has WSOL as one side.
3. Build the swap accounts matching `SwapSolBuy` struct exactly (see swap_sol_buy.rs):
   - `user`: trader keypair (signer, mutable)
   - `epochState`: protocol.epochState (UncheckedAccount)
   - `swapAuthority`: PDA seeds = ["swap_authority"], program = taxProgram.programId
   - `taxAuthority`: PDA seeds = ["tax_authority"], program = taxProgram.programId
   - `pool`: the CRIME/SOL pool address
   - `poolVaultA` / `poolVaultB`: from protocol.poolVaults
   - `mintA` / `mintB`: WSOL and CRIME (in canonical order matching the pool)
   - `userTokenA`: trader's WSOL account
   - `userTokenB`: trader's CRIME token account
   - `stakePool`: protocol.stakePool
   - `stakingEscrow`: protocol.escrowVault
   - `carnageVault`: protocol.carnageSolVault (native SOL PDA from Epoch Program)
   - `treasury`: authority's pubkey (or any valid account to receive SOL)
   - `ammProgram`: ammProgram.programId
   - `tokenProgramA`: TOKEN_PROGRAM_ID (standard SPL Token for WSOL)
   - `tokenProgramB`: TOKEN_2022_PROGRAM_ID (for CRIME)
   - `systemProgram`: SystemProgram.programId
   - `stakingProgram`: stakingProgram.programId

4. Resolve Transfer Hook remaining_accounts for the Token-2022 side of the swap.
   Use `createTransferCheckedWithTransferHookInstruction` from `@solana/spl-token` to resolve ExtraAccountMetas for the CRIME token transfer (from pool_vault_b to user_token_b, since this is a buy = user receives CRIME).

   IMPORTANT: The AMM's swap transfers Token B OUT of the vault TO the user. So resolve hook accounts for: source = poolVaultB (CRIME vault), dest = userTokenB (trader's CRIME account), mint = crimeMint, authority = pool PDA (since pool is the vault authority).

   Extract remaining_accounts from the transfer instruction keys (skip first 4: source, mint, dest, authority).

5. Call `taxProgram.methods.swapSolBuy(swapAmount, minimumOutput, isCrime)`:
   - `swapAmount`: 0.1 SOL (100_000_000 lamports) -- small enough to not drain the pool
   - `minimumOutput`: 0 (no slippage protection needed for smoke test)
   - `isCrime`: true (using CRIME/SOL pool)
   - `.accountsStrict({...})` with all accounts from step 3
   - `.remainingAccounts(hookAccounts)` with resolved hook accounts
   - `.signers([trader.keypair])`

6. Verify post-transaction state:
   - Trader's CRIME token balance increased (was some amount, now higher)
   - Trader's SOL balance decreased (paid for swap + tax)
   - Staking escrow received 75% of tax (check lamport balance > 0)
   - Transaction succeeded (no error thrown)

**Test 2: Stake PROFIT Tokens (Staking Program CPI)**

This test proves the staking CPI path works in the shared multi-program environment.

Steps:
1. Get the staker wallet from protocol.wallets.staker
2. Resolve Transfer Hook remaining_accounts for PROFIT token transfer:
   - Source: staker's PROFIT account
   - Dest: protocol.stakeVault
   - Mint: protocol.profitMint
   - Use `createTransferCheckedWithTransferHookInstruction`
   - Extract remaining_accounts (keys after first 4)

3. Derive UserStake PDA: seeds = ["user_stake", staker.keypair.publicKey.toBuffer()], program = stakingProgram.programId

4. Call `stakingProgram.methods.stake(stakeAmount)`:
   - `stakeAmount`: 100 PROFIT (100_000_000 base units at 6 decimals)
   - `.accountsStrict({ user, stakePool, userStake, userTokenAccount, stakeVault, profitMint, tokenProgram: TOKEN_2022_PROGRAM_ID, systemProgram })`
   - `.remainingAccounts(hookAccounts)`
   - `.signers([staker.keypair])`

5. Verify post-transaction state:
   - UserStake account exists and has staked_amount = 100_000_000
   - StakeVault token balance increased by stakeAmount
   - Staker's PROFIT balance decreased by stakeAmount

IMPORTANT NOTES:
- Use `getAccount` from `@solana/spl-token` with TOKEN_2022_PROGRAM_ID to query Token-2022 balances
- Use `provider.connection.getBalance()` to check SOL balances
- The before() hook runs protocol initialization. If it fails, ALL tests fail. Add clear error messages.
- Use `try/catch` in test bodies to log transaction signatures on failure for debugging
- `anchor test` with `--run test-integration` will start a fresh validator, deploy all programs, and run this test file

CANONICAL ORDERING NOTE:
The pool is keyed by canonical (sorted) mint pair. When looking up a pool, you need to know which mint is A and which is B. For CRIME/SOL:
- WSOL mint = So11111111111111111111111111111111111111112
- Compare WSOL pubkey bytes vs CRIME mint pubkey bytes
- Smaller = mintA, Larger = mintB
- The pool vaults match this ordering
- UserTokenA = user's account for mintA, UserTokenB = user's account for mintB

The ProtocolState should store which mint is A and which is B for each pool, or the test should determine it dynamically from the canonical ordering.

Common issues to watch for (from RESEARCH.md pitfalls):

1. **AMM initialize_admin ProgramData not found**: Ensure ProgramData address is derived correctly using BPFLoaderUpgradeab1e as program. The correct derivation is `PublicKey.findProgramAddressSync([ammProgram.programId.toBuffer()], BPF_LOADER_UPGRADEABLE_ID)`.

2. **MintsNotCanonicallyOrdered**: Mint ordering wrong for a pool. Fix by comparing pubkey buffers.

3. **NoWhitelistedParty**: Missing whitelist entry for a vault. Add the missing entry.

4. **Transfer hook account resolution**: Hook accounts may need different resolution depending on transfer direction (who is source vs dest). For the swap, the AMM transfers tokens between pool vaults and user accounts.

5. **Account constraint violations**: Account names in `.accountsStrict()` must match the Anchor IDL exactly (camelCase from snake_case). Check generated IDL types in `target/types/`.

6. **InsufficientFunds**: Test wallet doesn't have enough SOL/tokens. Increase airdrop or mint amounts.

7. **StakePool already initialized**: If a previous test run left state. This shouldn't happen with `anchor test` (fresh validator each time).

For each failure:
- Read the error message and transaction logs
- Identify which step failed and why
- Fix the root cause in the appropriate helper file
- Re-run `anchor test --run test-integration`
- Iterate until 2/2 tests pass

If a fix requires changing protocol-init.ts or constants.ts, make the fix and document what was wrong and why the fix works.
