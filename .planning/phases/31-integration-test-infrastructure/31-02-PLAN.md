**`initializeProtocol(provider, programs)` function** -- implements this exact sequence:

**Phase 1: Mints & Hook Foundation (Steps 1-5)**

Step 1: Airdrop SOL to the provider wallet (the authority). Request 100 SOL via airdrop. This is the admin/deployer.

Step 2: Create 3 Token-2022 mints with TransferHook extension.
- For each mint (CRIME, FRAUD, PROFIT):
  - Generate a Keypair
  - Calculate mint space with `getMintLen([ExtensionType.TransferHook])`
  - Build transaction: createAccount (system) + initializeTransferHook (pointing to hook program) + initializeMint (6 decimals, authority as mint authority, no freeze authority)
  - Send and confirm
- Use helpers from `@solana/spl-token`: `getMintLen`, `createInitializeTransferHookInstruction`, `createInitializeMintInstruction`
- Token-2022 program ID: `TOKEN_2022_PROGRAM_ID`

Step 3: Initialize Transfer Hook WhitelistAuthority.
- Derive PDA: seeds = ["authority"], program = hookProgram.programId
- Call `hookProgram.methods.initializeAuthority()` with `.accountsStrict({ signer, whitelistAuthority, systemProgram })`

Step 4: Initialize ExtraAccountMetaList for each mint (CRIME, FRAUD, PROFIT).
- For each mint:
  - Derive PDA: seeds = ["extra-account-metas", mint.toBuffer()], program = hookProgram.programId
  - Call `hookProgram.methods.initializeExtraAccountMetaList()` with `.accountsStrict({ payer, whitelistAuthority, authority, extraAccountMetaList, mint, systemProgram })`

**Phase 2: AMM Infrastructure (Steps 5-11)**

Step 5: Initialize AMM AdminConfig.
- Derive PDA: seeds = ["admin"], program = ammProgram.programId
- Get ProgramData address: `PublicKey.findProgramAddressSync([ammProgram.programId.toBuffer()], new PublicKey("BPFLoaderUpgradeab1e11111111111111111111111"))`
- Call `ammProgram.methods.initializeAdmin(authority.publicKey)` with `.accountsStrict({ authority, adminConfig, program: ammProgram.programId, programData, systemProgram })`
- CRITICAL: This requires the AMM to be deployed as upgradeable (which `anchor test` does automatically). The authority must be the upgrade authority.

Step 6: Create admin token accounts for seed liquidity.
- Create admin's WSOL account using `createWrappedNativeAccount` from `@solana/spl-token` (standard SPL Token program, NOT Token-2022). Wrap enough SOL for all SOL pools (e.g., 50 SOL).
- Create admin's Token-2022 accounts for CRIME, FRAUD, PROFIT using `createAccount` with TOKEN_2022_PROGRAM_ID.
- Mint tokens to admin's accounts: 100,000 each of CRIME, FRAUD, PROFIT.

Step 7-10: Initialize 4 AMM pools with seed liquidity.
For each pool, follow this pattern:
- Determine canonical mint ordering: compare pubkeys, smaller = mint_a, larger = mint_b
- Derive pool PDA: seeds = ["pool", mintA.toBuffer(), mintB.toBuffer()], program = ammProgram.programId
- Determine the correct token programs (SPL Token for WSOL side, Token-2022 for T22 side)
- Determine fee bps (100 for SOL pools, 50 for PROFIT pools)
- Determine seed amounts from constants

The 4 pools:
- **CRIME/SOL pool**: MixedPool (WSOL via SPL Token + CRIME via Token-2022), 100 bps fee
  - One side is WSOL (native_mint = So11111111111111111111111111111111111111112), other is CRIME
  - Canonical ordering: compare WSOL pubkey vs CRIME pubkey
- **FRAUD/SOL pool**: MixedPool (WSOL via SPL Token + FRAUD via Token-2022), 100 bps fee
- **CRIME/PROFIT pool**: PureT22Pool (both Token-2022), 50 bps fee
- **FRAUD/PROFIT pool**: PureT22Pool (both Token-2022), 50 bps fee

For each pool call `ammProgram.methods.initializePool(feeBps, amountA, amountB)` with `.accountsStrict({ payer, adminConfig, admin, pool, vaultA, vaultB, mintA, mintB, sourceA, sourceB, tokenProgramA, tokenProgramB, systemProgram })`.

IMPORTANT for InitializePool:
- `payer` and `admin` can be the same keypair (the authority)
- `adminConfig` is the AdminConfig PDA from Step 5
- `sourceA` / `sourceB` are the admin's token accounts holding seed liquidity
- `vaultA` / `vaultB` PDAs: seeds = ["vault", pool.toBuffer(), "a"/"b"], program = ammProgram.programId
- The vaults are CREATED by this instruction (init constraint), do not create them separately
- The initial liquidity is TRANSFERRED by the instruction from sourceA/sourceB into the vaults

Step 11: Whitelist all pool vault addresses in Transfer Hook.
- After all 4 pools are initialized, the vault PDAs now exist on-chain
- For each pool, derive vaultA and vaultB PDAs
- For each vault, derive whitelist entry PDA: seeds = ["whitelist", vault.toBuffer()], program = hookProgram.programId
- Call `hookProgram.methods.addWhitelistEntry()` for each vault
- This is 8 whitelist entries (4 pools x 2 vaults)
- NOTE: Only the Token-2022 vaults strictly need whitelisting (the hook only runs on T22 transfers). But whitelisting WSOL vaults too is harmless and keeps it simple.

**Phase 3: Epoch + Staking (Steps 12-15)**

Step 12: Initialize EpochState.
- Derive PDA: seeds = ["epoch_state"], program = epochProgram.programId
- Call `epochProgram.methods.initializeEpochState()` with `.accountsStrict({ payer, epochState, systemProgram })`

Step 13: Initialize StakePool.
- Follow the pattern from `scripts/init-localnet.ts` exactly (it's proven to work)
- Derive PDAs: stakePool (["stake_pool"]), escrowVault (["escrow_vault"]), stakeVault (["stake_vault"])
- Create admin token account for PROFIT if not already created (should already exist from Step 6)
- Whitelist admin's PROFIT token account BEFORE init (needed for dead stake transfer)
- Build hook remaining_accounts manually (stakeVault doesn't exist yet during init, so can't use createTransferCheckedWithTransferHookInstruction):
  - extraAccountMetaList for PROFIT mint
  - whitelistEntry for admin's PROFIT token account (source)
  - whitelistEntry for stakeVault PDA (dest -- doesn't exist yet, but PDA address is deterministic)
  - hookProgram.programId
- Call `stakingProgram.methods.initializeStakePool()` with `.accountsStrict({ authority, stakePool, escrowVault, stakeVault, authorityTokenAccount, profitMint, tokenProgram: TOKEN_2022_PROGRAM_ID, systemProgram })` and `.remainingAccounts(hookAccounts)`

Step 14: Whitelist StakeVault in Transfer Hook.
- Derive whitelist entry PDA for stakeVault
- Call `hookProgram.methods.addWhitelistEntry()`

**Phase 4: Carnage Fund (Steps 15-16)**

Step 15: Initialize Carnage Fund.
- Derive carnageState PDA: seeds = ["carnage_fund"], program = epochProgram.programId
- Derive sol_vault PDA: seeds = ["carnage_sol_vault"], program = epochProgram.programId
- Derive crime_vault PDA: seeds = ["carnage_crime_vault"], program = epochProgram.programId
- Derive fraud_vault PDA: seeds = ["carnage_fraud_vault"], program = epochProgram.programId
- Call `epochProgram.methods.initializeCarnageFund()` with `.accountsStrict({ authority, carnageState, solVault, crimeVault, fraudVault, crimeMint, fraudMint, tokenProgram: TOKEN_2022_PROGRAM_ID, systemProgram })`

Step 16: Whitelist Carnage vaults in Transfer Hook.
- Whitelist the carnage CRIME vault and FRAUD vault (both are T22 token accounts)
- The SOL vault is a SystemAccount (native SOL), does not need whitelisting (no T22 transfer hook)

**Phase 5: Test Wallets (Step 17)**

Step 17: Create test wallets using `createTestWallets()` from test-wallets.ts.
- Pass the CRIME, FRAUD, PROFIT mints
- The admin Keypair is needed to mint tokens to test wallets

**Return the ProtocolState** with all addresses.

CRITICAL IMPLEMENTATION NOTES:
- Use `@solana/web3.js` NATIVE_MINT (`So11111111111111111111111111111111111111112`) for WSOL, imported from `@solana/spl-token`.
- For canonical mint ordering: `const [mintA, mintB] = mint1.toBuffer().compare(mint2.toBuffer()) < 0 ? [mint1, mint2] : [mint2, mint1]`
- The admin/authority is the provider wallet. Load from `provider.wallet` or use `Keypair.generate()` with airdrop -- but since `anchor test` uses the wallet from Anchor.toml, use the provider's wallet as the payer and create a separate admin keypair with airdrop for signing.
- Actually, `anchor test` sets the provider wallet as the Anchor.toml wallet (keypairs/devnet-wallet.json). This wallet IS the upgrade authority. So use `(provider.wallet as any).payer` for the Keypair, or simply use `provider.publicKey` as the authority and let the provider sign. For `initializeAdmin`, the `authority` must be the upgrade authority = the provider wallet.
- Add generous `console.log` messages for each step so failures are easy to diagnose.
- Wrap the whole function in try/catch with descriptive error messages showing which step failed.
