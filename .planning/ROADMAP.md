### v1.5 Post-Launch Hardening & Expansion (In Progress)

**Milestone Goal:** Harden crank operations for sustained mainnet reliability, fix the live multi-hop vault UX bug, and build the Jupiter AMM adapter SDK for aggregator routing.

Three phases ship in dependency order: crank hardening (off-chain, immediate wins), vault convert-all (on-chain upgrade via Squads), Jupiter SDK (off-chain Rust crate, external review dependency).

#### Phase 105: Crank Hardening
**Goal**: Crank runner operates reliably with immediate cleanup, informed retry logic, and proactive failure alerting
**Depends on**: Nothing (first phase, off-chain only)
**Requirements**: CRANK-01, CRANK-02, CRANK-03, CRANK-04, CRANK-05
**Success Criteria** (what must be TRUE):
  1. Recovery-path randomness accounts are closed within the same epoch they are consumed (not deferred to next startup)
  2. Stale randomness accounts from prior crank runs are discovered and closed during startup sweep
  3. Switchboard mainnet gateway behavior is documented with findings on oracle topology, failover options, and 503 error patterns
  4. Crank retry/timeout logic reflects gateway research findings (improved backoff, oracle-aware recovery)
  5. VRF failures and epoch stalls trigger alerts to Discord or Telegram within 60 seconds
**Plans**: 3 plans

Plans:
- [x] 105-01-PLAN.md -- Fix randomness account leak paths + periodic sweep safety net
- [x] 105-02-PLAN.md -- VRF instrumentation fields + exponential backoff tuning
- [x] 105-03-PLAN.md -- Telegram alerting module + circuit breaker integration

#### Phase 106: Vault Convert-All
**Goal**: Users can convert their full token balance in a single transaction without intermediate token leakage or wallet security warnings
**Depends on**: Phase 105 (crank drain mode enables safe program upgrade window)
**Requirements**: VAULT-01, VAULT-02, VAULT-03, VAULT-04, VAULT-05, VAULT-06
**Success Criteria** (what must be TRUE):
  1. User can pass amount_in=0 to convert_v2 and have their full on-chain balance converted automatically
  2. convert_v2 rejects transactions where output falls below the caller's minimum_output parameter
  3. Existing convert instruction works identically to pre-upgrade behavior (no breaking changes)
  4. Multi-hop SOL-to-PROFIT swaps (4-step split route) complete without leftover intermediate tokens in the user's wallet
  5. Large multi-hop swaps (40+ SOL) simulate cleanly in Phantom/Backpack wallet previews without Blowfish "malicious" warnings
**Plans**: 4 plans

Plans:
- [x] 106-01-PLAN.md -- On-chain convert_v2 instruction + error variants + BOK proptest extension
- [x] 106-02-PLAN.md -- Client swap builders + multi-hop convert-all mode + vault error map
- [x] 106-03-PLAN.md -- SOS diff-audit of convert_v2 on-chain changes
- [x] 106-04-PLAN.md -- Devnet in-place upgrade + wallet simulation test matrix
