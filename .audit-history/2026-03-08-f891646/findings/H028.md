# H028 -- Epoch Transition Timing (H064 Recheck)

**Auditor:** Hypothesis Investigator (sos-002)
**Category:** Timing
**Severity:** MEDIUM (maintained from H064)
**Status:** VERIFIED -- No exploitable edge cases found. Known bounty-rent issue remains (documented TODO).

---

## Hypothesis

Epoch transition timing logic may have edge cases around slot boundaries or concurrent transitions that could cause double transitions or skipped epochs.

## Target Code

- `programs/epoch-program/src/instructions/trigger_epoch_transition.rs` (primary)
- `programs/epoch-program/src/instructions/consume_randomness.rs` (VRF completion)
- `programs/epoch-program/src/instructions/retry_epoch_vrf.rs` (timeout recovery)
- `programs/epoch-program/src/constants.rs` (timing parameters)
- `programs/epoch-program/src/state/epoch_state.rs` (state structure)

## Investigation

### 1. Double-Transition Prevention

The handler at line 149 enforces:
```rust
require!(!epoch_state.vrf_pending, EpochError::VrfAlreadyPending);
```

This is checked AFTER the epoch boundary check (line 137: `expected_epoch > epoch_state.current_epoch`). The ordering is correct: first confirm we should transition, then confirm we are not already mid-transition. Since `vrf_pending` is set to `true` at line 181 and only cleared by `consume_randomness` (line 206), no concurrent or duplicate transition can occur. Two calls in the same slot would both pass the boundary check, but only the first would pass the `vrf_pending` gate; the second would fail with `VrfAlreadyPending`.

**Verdict:** SECURE. Double-transition is impossible.

### 2. Epoch-Skipping Behavior

The `current_epoch()` function (line 80-81) computes:
```rust
((slot.saturating_sub(genesis_slot)) / SLOTS_PER_EPOCH) as u32
```

If the crank is late and multiple epoch boundaries pass before `trigger_epoch_transition` is called, `expected_epoch` will be greater than `current_epoch + 1`. The handler at line 176 sets `epoch_state.current_epoch = expected_epoch`, jumping directly to the correct epoch. This is correct behavior -- it does not skip epochs in a harmful sense, it simply advances to the current one. Intermediate epochs are never instantiated, which means:
- No tax rates are derived for skipped epochs (old rates persist until VRF completes)
- No Carnage rolls occur for skipped epochs
- Staking `update_cumulative` is called once (in `consume_randomness`) with the new epoch number

This is by design and documented. The protocol gracefully handles late crank execution.

**Verdict:** SECURE. Epoch skipping is handled correctly.

### 3. Off-By-One in Slot Boundary

The boundary check uses strict greater-than (line 137):
```rust
expected_epoch > epoch_state.current_epoch
```

Since `current_epoch()` uses integer division `(slot - genesis) / SLOTS_PER_EPOCH`, the boundary is clean:
- Slot `genesis + SLOTS_PER_EPOCH - 1` -> epoch 0 (no trigger)
- Slot `genesis + SLOTS_PER_EPOCH` -> epoch 1 (triggers)

Unit tests at lines 257-388 verify these boundaries including zero genesis, non-zero genesis, and the `saturating_sub` underflow case. No off-by-one exists.

**Verdict:** SECURE.

### 4. VRF Freshness Window (slot_diff <= 1)

Line 159-166 enforces that the randomness account's `seed_slot` is within 1 slot of the current slot. This is tight but correct -- the client creates the randomness account and bundles `commitIx + trigger_epoch_transition` in the same transaction. Since both execute in the same slot, `slot_diff` should be 0 or 1. Network congestion could cause the randomness account creation TX to land in a prior slot (diff=1), which is accommodated. If the randomness account is older than 1 slot, it fails with `RandomnessExpired`.

The concern is operational, not security: under extreme congestion, the trigger may fail spuriously and need retrying. This does not create a vulnerability.

**Verdict:** SECURE. Tight but appropriate.

### 5. Anti-Reroll Protection

At line 186, the handler binds the randomness account pubkey:
```rust
epoch_state.pending_randomness_account = ctx.accounts.randomness_account.key();
```

`consume_randomness` at line 154-157 enforces:
```rust
require!(
    ctx.accounts.randomness_account.key() == epoch_state.pending_randomness_account,
    EpochError::RandomnessAccountMismatch
);
```

An attacker cannot substitute a different randomness account to get favorable VRF outcomes. The binding is set atomically with the `vrf_pending = true` flag.

**Verdict:** SECURE.

### 6. VRF Timeout and Retry

`retry_epoch_vrf` at line 71-73 uses strict greater-than:
```rust
require!(elapsed_slots > VRF_TIMEOUT_SLOTS, EpochError::VrfTimeoutNotElapsed);
```

This means retry is allowed at slot `vrf_request_slot + 301` (not 300), which is correct -- it ensures the full timeout window has passed. The retry overwrites `pending_randomness_account` and `vrf_request_slot` but does NOT re-advance the epoch or change `current_epoch`. The epoch number remains as set by the original trigger. This is correct: the epoch has already advanced, we are just re-doing the VRF commit.

**Verdict:** SECURE.

### 7. `as u32` Truncation in current_epoch()

Line 81: `((slot.saturating_sub(genesis_slot)) / SLOTS_PER_EPOCH) as u32`

With SLOTS_PER_EPOCH=4500, overflow to u32::MAX requires ~4.3 billion epochs, which at 30 minutes per epoch is approximately 245,000 years. Not a practical concern.

**Verdict:** INFORMATIONAL. No real-world risk.

### 8. epoch_start_slot() Overflow

Line 99: `genesis_slot + (epoch as u64 * SLOTS_PER_EPOCH)`

With the u32 epoch cap, maximum value is `genesis_slot + (4.3B * 4500)` which is approximately `1.9 * 10^13` -- well within u64 range. However, this uses unchecked arithmetic. Consistent with `start_curve.rs:66` pattern noted in the timing context.

**Verdict:** INFORMATIONAL. Practically safe, but violates checked-arithmetic convention.

### 9. Bounty Payment Rent-Exempt Gap (Known TODO)

Line 195: `vault_balance >= TRIGGER_BOUNTY_LAMPORTS` does not account for rent-exempt minimum. If the vault balance is exactly at or slightly above `TRIGGER_BOUNTY_LAMPORTS` (1M lamports = 0.001 SOL) but below `TRIGGER_BOUNTY_LAMPORTS + rent_exempt_minimum`, paying the bounty could drop the vault below rent-exempt, risking account garbage collection. The `else` branch (line 220-227) gracefully skips payment when balance is insufficient, but the threshold is wrong.

The crank auto-tops-up when below a danger zone (per project memory), so this is operationally mitigated.

**Verdict:** KNOWN ISSUE. Mitigated by crank, on-chain fix deferred.

## Conclusion

The epoch transition timing logic is well-designed and correctly handles:
- Concurrent transition attempts (blocked by `vrf_pending` gate)
- Late crank execution (epoch-skip to current)
- Slot boundary precision (integer division, no off-by-one)
- VRF freshness enforcement (1-slot window)
- Anti-reroll binding (pubkey stored at commit, checked at consume)
- Timeout recovery (strict > check, fresh randomness required)

No exploitable edge cases were found. The severity remains **MEDIUM** due to:
1. The bounty rent-exempt gap (known TODO, operationally mitigated)
2. Unchecked arithmetic in `epoch_start_slot()` (informational)
3. Tight VRF freshness window creating operational fragility under congestion (not exploitable)

**Overall verdict: MAINTAINED AT MEDIUM.** Consistent with Audit #1 finding H064.
