# H129 -- Epoch Program: Multiple VRF Requests

**Status:** SAFE
**Severity:** N/A
**Target:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs`

## Hypothesis

Can multiple VRF requests be pending simultaneously, potentially causing state confusion or double-spend of epoch transitions?

## Analysis

The `trigger_epoch_transition` handler enforces a strict single-pending-VRF invariant through the `vrf_pending` boolean flag on the singleton `EpochState` PDA.

### Guard (line 149)

```rust
require!(!epoch_state.vrf_pending, EpochError::VrfAlreadyPending);
```

This check runs **before** any state mutations. If `vrf_pending` is already `true`, the instruction fails immediately.

### State Lifecycle

1. **trigger_epoch_transition** (line 181): Sets `vrf_pending = true` and binds `pending_randomness_account` to a specific Switchboard account.
2. **consume_randomness** (line 206): Sets `vrf_pending = false` after successfully deriving tax rates from the revealed randomness. Also verifies the passed randomness account matches `pending_randomness_account` (anti-reroll, INV-6).
3. **retry_epoch_vrf** (line 62): Requires `vrf_pending == true` AND timeout elapsed (>300 slots). Overwrites the `pending_randomness_account` with a fresh one but keeps `vrf_pending = true`. Does NOT create a second pending request -- it replaces the stale one.

### Key Properties

- **Singleton state**: `EpochState` is a single PDA (`seeds = [b"epoch_state"]`). There is exactly one `vrf_pending` flag for the entire protocol.
- **No parallel paths**: The `vrf_pending` flag is a mutex. Between `trigger_epoch_transition` setting it `true` and `consume_randomness` setting it `false`, no second trigger can execute.
- **Retry replaces, not adds**: `retry_epoch_vrf` overwrites the pending randomness account rather than creating a second parallel request.
- **Anti-reroll binding**: `consume_randomness` validates `pending_randomness_account` matches the passed account, preventing use of a different (attacker-chosen) randomness source.

## Verdict

**No vulnerability.** The `vrf_pending` boolean flag acts as a single-slot mutex that prevents multiple simultaneous VRF requests. The lifecycle is clean: trigger -> (optional retry) -> consume. At no point can two VRF requests coexist.
