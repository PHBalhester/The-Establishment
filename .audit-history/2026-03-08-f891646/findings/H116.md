# H116 -- Bonding Curve: SOL Vault Rent Accounting in withdraw_graduated_sol

## Status: NOT A BUG -- Correctly Implemented

## Target
`programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs`

## Hypothesis
Does `withdraw_graduated_sol` account for rent-exempt minimum? Could it over-withdraw (draining rent, causing the PDA to be garbage-collected) or under-withdraw (leaving excess SOL beyond rent)?

## Analysis

The handler computes:
```rust
let rent = Rent::get()?.minimum_balance(0);
let withdrawable = vault_balance.checked_sub(rent).unwrap_or(0);
```

This correctly:
1. Computes the rent-exempt minimum for a 0-data-byte account (~0.00089 SOL).
2. Subtracts it from the vault's total lamports.
3. Uses `unwrap_or(0)` to handle the case where vault_balance < rent (returns 0, no withdrawal).
4. Returns early with `Ok(())` if withdrawable is 0 (idempotent).

The rent-exempt minimum is preserved in the vault, so the PDA survives and is not garbage-collected. No over-withdrawal is possible.

## Authority Issue (Cross-Reference)

While the rent math is correct, the instruction accepts ANY `Signer<'info>` as `authority` with zero on-chain verification (no `has_one`, no ProgramData check, no stored authority comparison). This is the CRITICAL authority gap documented in ARCHITECTURE.md Section 7.1 and flagged by 5 audit agents. Anyone can call this instruction and drain ~1000 SOL per graduated curve to their own wallet.

This is NOT a rent accounting bug -- it is an access control bug already tracked as a cross-cutting concern.

## Verdict

**Rent accounting: SECURE.** The rent-exempt minimum is correctly computed and preserved. No over-withdrawal or under-withdrawal of the rent portion.

**Access control: CRITICAL (pre-existing finding).** Not in scope for this hypothesis but noted for completeness.
