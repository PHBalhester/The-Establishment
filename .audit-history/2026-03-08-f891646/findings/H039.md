# H039 -- Admin Privilege Escalation

**Auditor:** Hypothesis Investigator (sos-002)
**Severity:** LOW (no viable escalation path found for properly gated admins)
**Sub-finding:** INFORMATIONAL (Bonding Curve authority gap enables lateral movement, but is a known critical from other findings)
**Status:** CONFIRMED SAFE (with caveats)
**Confidence:** HIGH

---

## Hypothesis

Can the AMM admin or any admin role escalate privileges to affect other programs? Specifically, can the AMM admin manipulate pool state to influence Tax Program behavior?

## Scope

Admin-gated instructions across all 7 programs:

| Program | Admin Role | Gated Instructions | Escalation Surface |
|---------|-----------|-------------------|-------------------|
| AMM | AdminConfig.admin (ProgramData-verified) | initialize_pool, burn_admin | Pool creation params |
| Bonding Curve | ANY SIGNER (no verification) | 6 instructions | N/A -- already fully open |
| Transfer Hook | First-caller stored authority | add_whitelist_entry, burn_authority | Whitelist mutation |
| Epoch Program | None post-init | force_carnage (devnet only) | Devnet-only |
| Staking | None post-init | N/A | None |
| Tax Program | None (no admin) | N/A | None |
| Conversion Vault | None post-init | N/A | None |

## Investigation

### Path 1: AMM Admin -> Tax Program (via pool state manipulation)

**Question:** Can the AMM admin create a malicious pool to influence Tax Program swap behavior?

**Analysis:**

1. **AMM admin capabilities are limited to pool creation.** The AMM has exactly 4 instructions: `initialize_admin`, `burn_admin`, `initialize_pool`, `swap_sol_pool`. There is no `update_pool`, `update_fee`, or any post-creation modification instruction. Once a pool is created, the admin cannot change its parameters (fee, reserves, mints).

2. **Pool creation parameters are bounded.** `lp_fee_bps` is capped at `MAX_LP_FEE_BPS = 500` (5%) at `programs/amm/src/instructions/initialize_pool.rs:134`. Seed amounts must be non-zero. Pool type is inferred from token programs, not caller-declared.

3. **Tax Program reads pool reserves for output floor only.** The Tax Program's `read_pool_reserves()` at `programs/tax-program/src/helpers/pool_reader.rs:39-58` reads raw bytes from the pool AccountInfo to compute a minimum output floor. This happens BEFORE the AMM CPI. However:
   - The pool AccountInfo passed to Tax is the SAME account forwarded to the AMM CPI.
   - The AMM CPI validates the pool via PDA seeds (`[POOL_SEED, pool.mint_a, pool.mint_b]`) at `programs/amm/src/instructions/swap_sol_pool.rs:376-383`.
   - If a fake pool were passed, the output floor calculation would be wrong, but the AMM CPI would reject it.
   - The output floor is a LOWER BOUND on the user's `minimum_output` parameter -- a manipulated floor could only make it more permissive (lowering the floor), not steal funds directly.

4. **The AMM admin cannot create duplicate pools for the same mint pair.** Pool PDAs are derived from `[POOL_SEED, mint_a, mint_b]` with canonical mint ordering enforced. One pool per mint pair.

5. **Initial seed amounts set the initial price.** The admin controls `amount_a` and `amount_b` at pool creation, which determines the initial price ratio. A malicious admin could set an extreme initial price to front-run early swaps. However, this is an initialization parameter, not an ongoing escalation -- it is a one-time setting that becomes public immediately.

**Verdict: NO ESCALATION PATH.** The AMM admin cannot modify pool state post-creation. Pool creation parameters are bounded. The Tax Program's pool reserve read has no owner check, but the AMM CPI validates the pool independently, so a fake pool cannot be used for actual swaps.

### Path 2: AMM Admin -> Staking/Epoch (indirect)

The AMM admin has zero direct interaction with Staking or Epoch programs. The only cross-program path is:
- Tax Program -> AMM (via swap_authority PDA, CPI)
- Tax Program -> Staking (via tax_authority PDA, CPI)

The AMM admin is not in this chain. The admin's only power is pool creation, which does not affect tax rates, staking rewards, or epoch state.

**Verdict: NO ESCALATION PATH.**

### Path 3: Transfer Hook Authority -> Other Programs

The Transfer Hook authority controls the whitelist. A malicious authority could:
- Refuse to whitelist necessary addresses (DoS, not escalation)
- Whitelist unauthorized addresses (but all transfers still go through Tax Program for swaps)

The whitelist only controls whether Token-2022 transfers succeed. It does not grant any privileges in other programs.

**Verdict: NO ESCALATION PATH** (DoS vector only, already covered by S005).

### Path 4: Bonding Curve "Authority" -> Other Programs

The Bonding Curve has no stored admin and accepts ANY signer. This is not privilege escalation -- it is a complete absence of privilege checking. The Bonding Curve does not CPI into other programs (except via `distribute_tax_escrow` which is permissionless and state-gated). An attacker exploiting the Bonding Curve authority gap does not gain privileges in AMM, Tax, Staking, or Epoch programs.

**Verdict: NOT ESCALATION** (already critical as standalone finding -- BC authority gap).

### Path 5: Program Upgrade Authority -> All Programs

The program upgrade authority can replace any program's code. This is the ultimate escalation vector but is inherent to Solana's upgradeable BPF loader model, not a protocol-specific vulnerability. Mitigated by: (a) revoking upgrade authority before mainnet, or (b) transferring to a governance program/multisig.

**Verdict: INHERENT PLATFORM RISK** (documented in Architecture, not a protocol bug).

## Key Observation: Pool AccountInfo Owner Check Gap

The Tax Program's `pool` account in `SwapSolBuy` (line 389-390) and `SwapSolSell` is declared as raw `AccountInfo` with the comment "CHECK: Validated in AMM CPI." The `read_pool_reserves()` function reads from this account BEFORE AMM CPI validation occurs. While this does not enable privilege escalation (the output floor is advisory, not authoritative), it is a defense-in-depth gap:

- An attacker could pass a carefully crafted fake pool AccountInfo with manipulated reserve bytes.
- The output floor calculation would use fake reserves, potentially allowing a lower `minimum_output` than the real pool warrants.
- The AMM CPI would then use the REAL pool (validated by PDA seeds), and the swap would proceed with the user's potentially-too-low `minimum_output`.
- This is already documented as a separate finding (Architecture 7.6, "Pool Reserve Read Without Owner Check").

This is NOT an admin escalation issue -- any user can pass a fake pool, not just an admin.

## Conclusion

**No admin privilege escalation paths exist in the protocol.** The admin roles are properly isolated:

1. **AMM admin** can only create pools (bounded params, one-time). Cannot modify existing pools or affect Tax/Staking/Epoch behavior.
2. **Transfer Hook authority** can only control the whitelist (DoS vector, not escalation).
3. **Bonding Curve has no admin** -- the authority gap is a standalone critical, not an escalation from another role.
4. **Epoch/Staking/Tax/Vault have no post-init admin** -- no admin actions possible.

The `seeds::program` CPI gating (4 trust boundaries) prevents any program from impersonating another. The admin roles do not grant CPI signing authority.

## Risk Rating

| Factor | Assessment |
|--------|-----------|
| Escalation from AMM admin to Tax | NOT POSSIBLE -- no pool modification, CPI validates independently |
| Escalation from AMM admin to Staking | NOT POSSIBLE -- no interaction path |
| Escalation from Hook authority to others | NOT POSSIBLE -- whitelist only affects transfers |
| Escalation from BC "authority" to others | NOT POSSIBLE -- BC has no CPI to other programs |
| Cross-program CPI impersonation | NOT POSSIBLE -- seeds::program prevents |

**Overall: LOW / INFORMATIONAL** -- No privilege escalation vectors found. The protocol's admin isolation is sound. The Bonding Curve authority gap is not an escalation issue but a standalone access control failure (covered by other findings).
