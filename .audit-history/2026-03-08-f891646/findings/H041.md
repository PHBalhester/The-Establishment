# H041 -- Staking Reward Precision Loss (H075 Recheck)

**Auditor:** Hypothesis Investigator (sos-002)
**Target:** `programs/staking/src/helpers/math.rs`
**Supporting files:** `programs/staking/src/constants.rs`, `programs/staking/src/instructions/update_cumulative.rs`, `programs/staking/src/instructions/claim.rs`, `programs/staking/src/instructions/unstake.rs`, `programs/staking/src/instructions/stake.rs`
**Hypothesis:** PRECISION=1e18 in staking may cause reward precision loss for small deposits or large reward pools; rounding direction may favor users over protocol.

---

## Verdict: RESOLVED -- No Exploitable Precision Loss

The staking reward math is sound. PRECISION=1e18 provides sufficient precision across all realistic parameter ranges. Rounding consistently favors the protocol (floor division). The `as u64` cast on the reward result is safe under current parameter bounds. One informational observation noted below.

---

## Analysis

### 1. Precision Model

The staking system uses the Synthetix cumulative reward-per-token pattern with two core formulas:

**Distribution (add_to_cumulative / update_cumulative handler):**
```
reward_per_token = pending_rewards * PRECISION / total_staked
```
Where PRECISION = 1e18 (u128).

**Claim calculation (update_rewards):**
```
pending = staked_balance * reward_delta / PRECISION
```
Where `reward_delta = rewards_per_token_stored - user.rewards_per_token_paid`.

### 2. Small Deposit Precision Analysis

**Worst case:** 1 token staked (1e6 base units with 6 decimals), 1 lamport reward distributed across maximum stakers.

- `reward_per_token = 1 * 1e18 / total_staked`
- With total_staked = 20M PROFIT (2e13 base units, full supply): `1e18 / 2e13 = 5e4`
- User claim: `1e6 * 5e4 / 1e18 = 5e10 / 1e18 = 0` (truncated to 0)

This means a user staking 1 PROFIT token loses a 1-lamport reward when 20M PROFIT is staked. This is correct behavior -- the dust stays in the escrow, favoring the protocol. The user would need to accumulate rewards across many epochs before their share rounds up to 1 lamport.

**Precision loss per epoch for a minimum staker:**
- With 1 SOL (1e9 lamports) distributed and 20M PROFIT staked:
  - `reward_per_token = 1e9 * 1e18 / 2e13 = 5e13`
  - User with 1 PROFIT (1e6 units): `1e6 * 5e13 / 1e18 = 5e19 / 1e18 = 50 lamports`
  - Exact: 50.0 lamports. No precision loss at this scale.

**Threshold for zero reward:** A user staking `B` base units gets 0 reward when `B * reward_per_token < PRECISION`. With 1 SOL rewards and 20M staked, `reward_per_token = 5e13`, so any balance > `1e18 / 5e13 = 20000 base units = 0.02 PROFIT` earns at least 1 lamport. This is well below MINIMUM_STAKE (1e6 = 1 PROFIT).

### 3. Large Reward Pool Overflow Analysis

**Distribution overflow check:** `pending_rewards * PRECISION` in u128.
- Max pending_rewards = u64::MAX = ~1.84e19
- 1.84e19 * 1e18 = 1.84e37, within u128 max (3.4e38)
- Headroom: ~18.5x before overflow. Checked_mul catches this.

**Cumulative accumulation overflow:** `rewards_per_token_stored` grows each epoch.
- Worst case per epoch: u64::MAX * 1e18 / MINIMUM_STAKE = 1.84e37 / 1e6 = 1.84e31
- u128::MAX / 1.84e31 = ~1.85e7 epochs before overflow
- At realistic 10-minute epochs, this is ~351 years of continuous operation at maximum reward rate. Safe.
- With realistic rewards (~1000 SOL/epoch = 1e12 lamports) and realistic stake (10K PROFIT = 1e10): 1e12 * 1e18 / 1e10 = 1e20 per epoch. ~3.4e18 epochs = ~6.5e13 years. Not a concern.

### 4. Rounding Direction Verification

Both division operations use integer floor division:

| Operation | Location | Rounding | Favors |
|-----------|----------|----------|--------|
| `pending_rewards * PRECISION / total_staked` | math.rs:107-111, update_cumulative.rs:89-93 | Floor | Protocol (less reward per token stored) |
| `balance * reward_delta / PRECISION` | math.rs:46-50 | Floor | Protocol (user gets fewer lamports) |

**Double-floor compounding:** The floor in distribution PLUS the floor in claim means users collectively receive strictly less than deposited. The difference (dust) remains permanently in the escrow vault. This is the correct, safe direction.

### 5. `as u64` Cast Safety (math.rs:50)

```rust
let pending = (user.staked_balance as u128)
    .checked_mul(reward_delta)
    .ok_or(StakingError::Overflow)?
    .checked_div(PRECISION)
    .ok_or(StakingError::DivisionByZero)? as u64;
```

The result of `(balance * reward_delta) / PRECISION` could theoretically exceed u64::MAX if `reward_delta` is astronomically large. However:

- `reward_delta` is bounded by cumulative accumulation (analyzed in section 3)
- For the result to exceed u64::MAX: `balance * reward_delta / PRECISION > 2^64 - 1`
- With max balance = 2e13 (20M PROFIT): `reward_delta > (2^64 * 1e18) / 2e13 = ~9.2e23`
- At 1e20 per epoch (realistic), this requires ~9200 epochs (~64 days at 10-min epochs) of rewards without any claims
- This is reachable in theory but would require: (a) a single user staking ALL 20M PROFIT, (b) no claims for 64 days, (c) 1000 SOL/epoch continuously

**Realistic assessment:** The `as u64` truncation risk is extremely low but not impossible under extreme accumulation. The `checked_div` returns a u128 that is then silently truncated. Using `u64::try_from(...).map_err(|_| StakingError::Overflow)?` would add defense-in-depth at zero runtime cost.

### 6. Dead Stake Protection

MINIMUM_STAKE = 1e6 (1 PROFIT) is staked during initialization as "dead stake." This prevents:
- Division by zero in `pending * PRECISION / total_staked`
- First-depositor inflation attack (attacker can't be sole staker to manipulate reward_per_token)
- Total_staked going to zero (which would cause rewards to accumulate in pending without distribution)

The dead stake protection is correctly implemented: `initialize_stake_pool` stakes MINIMUM_STAKE, and it can never be unstaked because it has no UserStake account.

### 7. Checkpoint Pattern Verification

`update_rewards` is called BEFORE balance changes in all three paths:
- `stake.rs:118` -- before `staked_balance += amount`
- `unstake.rs:156` -- before `staked_balance -= amount`
- `claim.rs:92` -- before `rewards_earned = 0`

This is correct. If update_rewards were called AFTER a balance change, a user could inflate their rewards by staking a large amount, triggering reward calculation on the inflated balance, then immediately unstaking.

### 8. Forfeiture Recycling

On unstake, accrued `rewards_earned` is added back to `pool.pending_rewards` (unstake.rs:163-166). This recycled amount enters the next epoch's distribution, benefiting remaining stakers. The proptest suite (Properties T21-T23) validates that forfeited amounts never exceed deposited amounts.

## Informational Observations

### INFO-1: `as u64` Cast Without `try_from` (math.rs:50)

**Severity:** Informational (defense-in-depth)
**Location:** `programs/staking/src/helpers/math.rs:50`

The `as u64` cast on the u128 division result will silently truncate if the value exceeds u64::MAX. While this is safe under current parameters (see section 5), replacing with `u64::try_from(...).map_err(|_| StakingError::Overflow)?` would match the defensive pattern used elsewhere in the codebase (e.g., `tax_math.rs:52`, `amm/math.rs:75`).

**Current code:**
```rust
.checked_div(PRECISION)
.ok_or(StakingError::DivisionByZero)? as u64;
```

**Suggested:**
```rust
.checked_div(PRECISION)
.ok_or(StakingError::DivisionByZero)?;
let pending = u64::try_from(pending_u128).map_err(|_| StakingError::Overflow)?;
```

This is a code quality observation, not a vulnerability. The existing proptest suite (80K+ iterations, Properties 1-4 and T21-T23) validates conservation under realistic parameter ranges.

---

## Conclusion

H075 from audit #1 is **RESOLVED**. The PRECISION=1e18 model with u128 intermediates provides adequate precision for all realistic staking scenarios. Floor rounding consistently favors the protocol. The dead stake mechanism prevents edge-case attacks. The only remaining observation is a cosmetic `as u64` cast that could be hardened with `try_from` for defense-in-depth.
