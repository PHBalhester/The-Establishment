# H034 -- Conversion Vault Truncation Loss

| Field         | Value                                                    |
|---------------|----------------------------------------------------------|
| Hypothesis    | 100:1 conversion with integer division loses value for non-round amounts. Convert 99 CRIME -> 0 PROFIT. |
| Severity      | Informational                                            |
| Status        | **Invalid** (mitigated by design)                        |
| File          | `programs/conversion-vault/src/instructions/convert.rs`  |
| Lines         | 101-105                                                  |

## Evidence

### The Code

```rust
// convert.rs:101-105
if (*input_mint == *crime || *input_mint == *fraud) && *output_mint == *profit {
    let out = amount_in / CONVERSION_RATE;   // integer division, remainder lost
    require!(out > 0, VaultError::OutputTooSmall);
    Ok(out)
}
```

`CONVERSION_RATE = 100` (constants.rs:7). `TOKEN_DECIMALS = 6` for all tokens.

### Zero-Output Case: Handled

The specific scenario in the hypothesis -- converting 99 raw units of CRIME yields `99 / 100 = 0` -- is explicitly caught by the `OutputTooSmall` guard at line 104. The transaction reverts with error code 6001. This is tested directly in `test_dust_amount_rejected` (test_vault.rs:115-129).

The minimum input that succeeds is 100 raw units (0.000100 CRIME), producing 1 raw unit (0.000001 PROFIT). This boundary is tested in `test_dust_boundary_accepted` (test_vault.rs:135-143).

### Remainder Dust Loss: Real but Negligible

For non-round inputs, up to 99 raw units (0.000099 tokens) are lost per conversion. Examples:

| Input (CRIME)     | Raw Units     | Output (PROFIT) | Raw Units  | Dust Lost       |
|-------------------|---------------|-----------------|------------|-----------------|
| 100.000000        | 100,000,000   | 1.000000        | 1,000,000  | 0               |
| 100.000099        | 100,000,099   | 1.000000        | 1,000,000  | 0.000099 CRIME  |
| 0.000199          | 199           | 0.000001        | 1          | 0.000099 CRIME  |

Maximum dust per conversion: 99 base units = 0.000099 tokens. At any realistic token price, this is economically zero.

### Where the Dust Goes

The remainder tokens are transferred into the vault's input token account (the full `amount_in` is sent at line 145-155), but only `amount_in / 100` is sent out. The extra tokens accumulate in the vault permanently. They cannot be extracted -- the vault has no admin, no withdrawal function, and no upgrade path (Trust Tier 6 per ARCHITECTURE.md). This is value locked, not value stolen.

### Reverse Direction Is Lossless

PROFIT -> CRIME/FRAUD uses `checked_mul(100)` (line 108-110), which is exact with no remainder. Only the divide direction has dust loss.

### Test Coverage

The conversion vault has 12 unit tests covering:
- T6: Dust rejection (99 raw -> OutputTooSmall)
- T7: Dust boundary (100 raw -> 1 raw)
- T1/T2: Round conversions at whole-token scale
- T10: Large amounts (20M PROFIT -> 2B CRIME, no overflow)

## Conclusion

The hypothesis is **invalid**. The zero-output case (99 CRIME -> 0 PROFIT) is caught by `OutputTooSmall` and reverts. Remainder dust loss of up to 0.000099 tokens per conversion exists by design, is documented in code comments ("integer division, remainder lost"), tested, and economically negligible. The dust accumulates in a vault with no extraction mechanism. No fix needed.
