# H095 -- AMM: Pool with Same Token Both Sides

**Status:** SAFE
**Severity:** N/A (properly mitigated)
**Component:** `programs/amm/src/instructions/initialize_pool.rs`

## Hypothesis

Can `initialize_pool` be called with `mint_a == mint_b`, creating a degenerate self-pairing pool?

## Evidence

Two independent guards prevent this:

### Guard 1: Canonical ordering (line 35-38)
```rust
require!(
    ctx.accounts.mint_a.key() < ctx.accounts.mint_b.key(),
    AmmError::MintsNotCanonicallyOrdered
);
```
Strict less-than (`<`) inherently rejects equal keys. If `mint_a == mint_b`, this fails.

### Guard 2: Explicit duplicate check (line 41-44)
```rust
require!(
    ctx.accounts.mint_a.key() != ctx.accounts.mint_b.key(),
    AmmError::DuplicateMints
);
```
Belt-and-suspenders explicit check for the same condition.

### Guard 3: PDA uniqueness (line 218-224)
The pool PDA is seeded with `[POOL_SEED, mint_a.key(), mint_b.key()]`. The canonical ordering requirement means there is exactly one PDA per unordered mint pair, preventing duplicate pools with swapped arguments.

## Conclusion

**No vulnerability.** The canonical ordering check (`<` not `<=`) alone is sufficient to prevent `mint_a == mint_b`. The explicit `DuplicateMints` check provides defense in depth. Both fire before any state is written or tokens transferred.
