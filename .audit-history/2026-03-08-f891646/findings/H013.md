# H013: Buy Path Tax Math -- Did Fix for H041 Actually Work?

**Status:** RESOLVED (No Regression)
**Severity:** INFORMATIONAL
**Category:** Arithmetic
**Confidence:** HIGH (95%)

## Summary

The previous audit finding H041 flagged incorrect tax fee calculation. The current code in `tax_math.rs` and `swap_sol_buy.rs` correctly uses u128 intermediate arithmetic throughout the tax calculation and distribution pipeline. No regression was found. The fix is complete and well-tested.

---

## Evidence

### calculate_tax (tax_math.rs:34-53)

The function:
1. Validates `tax_bps <= 10_000` (rejects invalid rates, returns None)
2. Widens both operands to u128 before multiplication
3. Uses `checked_mul` and `checked_div` (never panics)
4. Converts result back via `u64::try_from(tax).ok()` -- the safe conversion that returns None on overflow

**Truncation analysis for the `as u64` (actually `try_from`):**
- Maximum possible result: `u64::MAX * 10_000 / 10_000 = u64::MAX`
- The result can never exceed the original input amount (since bps <= 10_000)
- `try_from` is used, not bare `as u64` -- this is the gold standard for u128->u64 conversion

### split_distribution (tax_math.rs:79-110)

The function:
1. Handles micro-tax edge case (< 4 lamports) by routing all to staking
2. Computes staking (71%) and carnage (24%) via u128 intermediates with `checked_mul`/`checked_div`
3. Uses `u64::try_from` for both conversions back to u64
4. Treasury is computed as remainder: `total_tax - staking - carnage` (using `checked_sub`)
5. **Conservation invariant is enforced by construction**: staking + carnage + treasury == total_tax

**No truncation risk:** The division result is always <= total_tax (which is u64), so `try_from` always succeeds.

### Usage in swap_sol_buy.rs (lines 83-118)

The handler chains the functions correctly:
```
calculate_tax(amount_in, tax_bps) -> tax_amount
sol_to_swap = amount_in.checked_sub(tax_amount)
split_distribution(tax_amount) -> (staking, carnage, treasury)
```

All intermediate values use checked arithmetic. The `sol_to_swap > 0` guard (line 94) prevents zero-amount swaps.

### Usage in swap_sol_sell.rs (line 230)

The sell path uses the same `calculate_tax` and `split_distribution` functions, applied to the gross AMM output rather than the input. Same safety properties apply.

### Constants consistency

`tax_math.rs` inline constants (STAKING_BPS=7100, CARNAGE_BPS=2400, BPS_DENOM=10000) match `constants.rs` values (STAKING_BPS=7100, CARNAGE_BPS=2400, BPS_DENOMINATOR=10000). The comment at tax_math.rs:81 documents this intentional duplication (to keep the module free of anchor_lang dependencies for fast testing).

## Attack Scenario

None identified. The H041 fix is complete. The only theoretical concern is the duplication of BPS constants between `tax_math.rs` and `constants.rs`, but these are verified by the audit context (02-arithmetic) and match. A future refactor that changes one but not the other would create a discrepancy, but this is a maintenance concern (informational), not a vulnerability.

---

## Root Cause (of original H041)

The original finding identified that tax calculations were performed using u64 arithmetic, risking overflow on large amounts. The fix promoted all intermediate calculations to u128 with proper checked arithmetic and safe narrowing via `u64::try_from`.

---

## Recommendation

No action required. The fix is correct and well-tested. For defense-in-depth, consider adding a compile-time or CI check that the inline BPS constants in `tax_math.rs` match `constants.rs` (currently verified only by comment).
