# H094 -- Staking: deposit_rewards Zero Amount

**Status:** SAFE
**Severity:** N/A (no issue found)
**Target:** `programs/staking/src/instructions/deposit_rewards.rs`

## Hypothesis

Check 0-amount handling. No div-by-zero when total_staked is 0.

## Analysis

### Zero Amount Handling

`deposit_rewards.rs` line 84 explicitly rejects zero amounts:

```rust
require!(amount > 0, StakingError::ZeroAmount);
```

This prevents empty deposits from inflating event counts or wasting compute.

### Division-by-Zero: Not Applicable in deposit_rewards

`deposit_rewards` does NOT perform any division. It only:
1. Validates `amount > 0`
2. Adds `amount` to `pool.pending_rewards` via `checked_add` (line 90-93)
3. Reconciles escrow balance (line 99-102)
4. Emits event

The division happens later, in `update_cumulative` (called by Epoch Program at epoch end), where `pending_rewards * PRECISION / total_staked` is computed.

### Division-by-Zero in update_cumulative

The `update_cumulative` handler (lines 88-105) guards against div-by-zero:

```rust
if rewards_added > 0 && pool.total_staked > 0 {
    // ... division only happens here
}
```

If `total_staked == 0`, the division is skipped entirely. `pending_rewards` is still cleared to 0 (line 108), and the rewards are reported as `rewards_added` in the event but effectively lost.

### Division-by-Zero in add_to_cumulative (math helper)

The `add_to_cumulative` function in `helpers/math.rs` (lines 91-127) also has an explicit early exit:

```rust
if pool.total_staked == 0 {
    return Ok(0);  // rewards stay in pending
}
```

Note the behavioral difference: `add_to_cumulative` preserves pending_rewards when `total_staked == 0`, while `update_cumulative` clears them. The `update_cumulative` handler is the one called on-chain via CPI, so pending_rewards ARE cleared even with zero stakers.

### Is Clearing Pending with Zero Stakers a Problem?

In practice, this scenario cannot occur due to **dead stake**: `initialize_stake_pool` deposits `MINIMUM_STAKE` (1 PROFIT token = 1,000,000 units) that can never be withdrawn. This ensures `total_staked >= MINIMUM_STAKE > 0` always.

Even if it could occur (e.g., through a future code change), the impact is limited: deposited SOL remains in the escrow vault and is not lost. Only the accounting pointer (`pending_rewards`) is cleared, meaning the SOL becomes unclaimable -- a protocol loss, not a user exploit.

### Overflow Protection

The `checked_add` on line 90-93 prevents `pending_rewards` from overflowing `u64`. The reconciliation check (line 99-102) ensures `pending_rewards` never exceeds actual escrow lamports, providing a second defense against inflation.

## Verdict

No issue. Zero-amount deposits are explicitly rejected. Division-by-zero is impossible due to dead stake (MINIMUM_STAKE permanently staked at initialization). Both `deposit_rewards` and `update_cumulative` use appropriate guards and checked arithmetic.
