## Hypothesis

`trigger_epoch_transition` pays a bounty from `carnage_sol_vault` without accounting for rent-exempt minimum, potentially draining the vault below rent-exempt and destroying the PDA.

## Investigation

### On-Chain Logic (lines 194-227)

The bounty guard at line 195 is:

```rust
let bounty_paid = if vault_balance >= TRIGGER_BOUNTY_LAMPORTS {
    // Transfer TRIGGER_BOUNTY_LAMPORTS (1,000,000 lamports = 0.001 SOL)
    invoke_signed(...)
} else {
    // Skip bounty
    0
};
```

**The "skip when insufficient" guard exists** -- if the vault has fewer lamports than the bounty amount (1M lamports), the bounty is skipped entirely and the epoch transition still proceeds. This is the partial mitigation from H001.

**The gap**: The check is `vault_balance >= 1,000,000` but does NOT subtract `rent_exempt_minimum` (~890,880 for a 0-data SystemAccount). This creates a danger zone:

| Vault Balance | Guard Passes? | Post-Transfer Balance | Survives Rent? |
|--------------|---------------|----------------------|----------------|
| >= 1,890,881 | Yes | >= 890,881 | Yes |
| 1,000,000 -- 1,890,880 | Yes | 0 -- 890,880 | **NO** |
| < 1,000,000 | No (skipped) | Unchanged | Yes |

If the vault is in the 1M-1.89M range, the guard passes, `system_instruction::transfer` executes, and the vault drops below rent-exempt minimum. Solana's runtime would then garbage-collect the PDA at the end of the transaction (or next epoch boundary), destroying the `carnage_sol_vault`.

### Practical Likelihood

**Very low**, for two reasons:

1. **`execute_carnage` / `execute_carnage_atomic` already protect rent**: Both Carnage execution paths compute `available_sol = sol_balance.saturating_sub(rent_exempt_min)` before spending (lines 362-363 in `execute_carnage.rs`, lines 354-355 in `execute_carnage_atomic.rs`). Carnage swaps cannot drain the vault below rent-exempt. Only the bounty path lacks this protection.

2. **Crank-runner auto-tops-up**: `scripts/crank/crank-runner.ts` (lines 62-75, 220-241) monitors vault balance each cycle. If it drops below `MIN_VAULT_BALANCE` (~2M lamports = bounty + rent + 100K buffer), the crank sends 0.005 SOL (5M lamports) from its wallet to the vault. This off-chain mitigation covers ~5 bounties per top-up.

### Remaining Risk Scenarios

1. **Crank not running**: If the crank is offline and the vault is in the danger zone, any permissionless caller triggering an epoch transition would destroy the vault PDA. However, without the crank, VRF flow would also stall, and the vault naturally accumulates 24% of all swap taxes, making the danger zone transient.

2. **Adversarial draining**: An attacker cannot directly drain the vault -- only bounty transfers and Carnage swaps extract SOL, and Carnage swaps already respect rent-exempt. The only scenario reaching the danger zone is many consecutive epochs with no swap activity (no tax inflow) combined with Carnage swaps that drain to near-rent-exempt, followed by exactly one more bounty transfer.

3. **`system_instruction::transfer` runtime behavior**: On current Solana runtime, `system_instruction::transfer` from a PDA that would bring it below rent-exempt minimum will **fail with an error** (the runtime enforces rent-exempt on owned accounts during transfer). This means the bounty transfer would revert, but the epoch transition would also revert -- effectively a DoS on epoch advancement until the vault is topped up. The PDA would NOT be destroyed, but epoch transitions would halt.

### Correction to Initial Analysis

After deeper investigation: Solana's runtime enforces that `system_instruction::transfer` cannot bring an account below its rent-exempt minimum (for non-native programs and PDAs). The transfer itself would fail, not succeed-and-destroy. This means the actual impact is **epoch transition DoS**, not PDA destruction. The epoch transition reverts, and no further epochs can advance until someone sends SOL to the vault.

## Verdict

**LOW severity.** The on-chain code has a logic gap (missing rent-exempt subtraction in the bounty guard), but:

1. The Solana runtime prevents actual PDA destruction -- the transfer would fail, reverting the whole instruction.
2. The impact is epoch transition DoS (recoverable by sending SOL to the vault).
3. The crank-runner mitigates this off-chain by proactively topping up the vault.
4. The danger zone (vault between 1M-1.89M lamports) is narrow and transient given continuous tax inflow.

## Recommended Fix

Trivial on-chain fix that would eliminate the gap entirely:

```rust
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(0);
let bounty_paid = if vault_balance >= TRIGGER_BOUNTY_LAMPORTS.checked_add(rent_exempt_min).unwrap() {
    // Transfer bounty
    ...
} else {
    0
};
```

This matches the pattern already used in `execute_carnage.rs` and `execute_carnage_atomic.rs`, making the protection consistent across all vault spending paths.
