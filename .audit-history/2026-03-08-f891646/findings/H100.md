# H100 -- Epoch Program: Switchboard Version Pinning

**Status:** INFORMATIONAL (version is pinned, no immediate issue)
**Severity:** Informational
**Component:** `programs/epoch-program/Cargo.toml`

## Hypothesis

Check the Switchboard SDK version used by the Epoch Program and note any known issues or pinning concerns.

## Findings

### Pinned Version

From `programs/epoch-program/Cargo.toml`:

```toml
switchboard-on-demand = "=0.11.3"
```

The `=` prefix enforces an **exact version pin** -- Cargo will not auto-upgrade to 0.11.4 or any other version. This is the correct approach for on-chain programs where SDK changes could alter account layouts or CPI behavior.

### Version Context

- **Package:** `switchboard-on-demand` (the Switchboard V3 / On-Demand SDK for Solana)
- **Version:** 0.11.3
- **ARCHITECTURE.md reference:** Assumption A-6 states "Switchboard oracle is honest -- Trusted dependency -- pinned v0.11.3"
- **MEMORY.md notes:** Multiple entries document Switchboard VRF operational issues (gateway rotation failures, timeout recovery patterns) but none related to SDK version bugs.

### Known Considerations

1. **Pre-1.0 crate:** The 0.x version number means the API is not yet considered stable by the Switchboard team. Any minor version bump (0.11.x to 0.12.x) could contain breaking changes.

2. **Exact pin is correct:** Given the pre-1.0 status, the `=0.11.3` pin prevents surprise breakage from `cargo update`. This is best practice for on-chain Solana programs.

3. **No PDA constraint on the account:** The epoch_state account is not constrained to a specific PDA. Note: this is about Switchboard account validation, not epoch_state -- covered separately.

4. **Operational issues documented (not version-related):** The project's MEMORY.md documents gateway rotation failures and VRF timeout recovery. These are operational issues with the Switchboard oracle network, not SDK version bugs. The workarounds (retry with fresh randomness, never rotate gateways) are already implemented.

5. **Audit supply chain note:** The `switchboard-on-demand` crate pulls in significant transitive dependencies. A `cargo audit` check is recommended before mainnet to ensure no known vulnerabilities in the dependency tree.

## Verdict

**INFORMATIONAL.** The Switchboard SDK is correctly pinned at exactly v0.11.3 with `=` syntax. This matches the documented assumption A-6. No version-specific bugs are known from the project's operational history. The pre-1.0 status warrants caution on any future upgrades -- test thoroughly before changing the pin. Consider running `cargo audit` before mainnet deployment to check the transitive dependency tree.
