## Summary

The `held_token` / `held_amount` selection logic in `execute_carnage_atomic.rs` is correctly implemented across all code paths (Burn, Sell, BuyOnly). The encoding mismatch between the `Token` enum (0=CRIME, 1=FRAUD) and the `HeldToken` encoding (0=None, 1=CRIME, 2=FRAUD) is correctly bridged by the `+1` offset at line 452. All edge cases examined -- including same-token burn+buy, same-token sell+buy, zero holdings, and stale state after expiration -- are handled safely.

---

## Analysis

### Encoding Scheme

Two different u8 encodings exist for token identity:

- **`Token` enum** (`enums.rs`): `0=Crime, 1=Fraud`. Used in `EpochState.carnage_target`.
- **`HeldToken` / `CarnageFundState.held_token`**: `0=None, 1=Crime, 2=Fraud`.

The bridge between them is at line 452:
```rust
carnage_state.held_token = target.to_u8() + 1;
```

All reads of `held_token` correctly use the 1=CRIME, 2=FRAUD encoding:
- `burn_held_tokens` (line 514): `match carnage_state.held_token { 1 => true, 2 => false, _ => return Ok(0) }`
- `execute_sell_swap` (line 717): `match held_token { 1 => true, 2 => false, _ => return Ok(()) }`
- `approve_delegate` (line 601): `if held_token == 1 { crime_vault } else { fraud_vault }`
- Burn mint selection (line 273): `if held_token == 1 { crime_mint } else { fraud_mint }`

**Verdict**: Encoding is correct and consistent. The `+1` offset bridge is fragile (relies on `Token::Crime=0, Token::Fraud=1` matching `HeldToken::Crime=1, HeldToken::Fraud=2` via addition), but this is covered by unit tests in both enums.

### Edge Case: Same Token Held and Targeted

The VRF target (`get_carnage_target`) is independent of current holdings. When `held_token == target` (e.g., hold CRIME, target CRIME):

**Burn path**: Burns from crime_vault, then buys into crime_vault. Step 1.5 (line 344-347) reloads the target vault after burn, so `target_vault_before` at line 390-393 reads the post-burn balance. The `tokens_bought` calculation (`target_vault_after - target_vault_before`) is therefore correct. No double-counting.

**Sell path**: Sells CRIME->WSOL via crime_pool, then buys CRIME with WSOL via crime_pool. Two separate swap_exempt CPIs, each using the correct pool. The sell hook accounts and buy hook accounts would both be CRIME's hook PDAs. The hook account partitioning at lines 255-265 allocates `HOOK_ACCOUNTS_PER_MINT * 2` accounts, splitting first 4 for sell, last 4 for buy. When both are CRIME, the client must pass CRIME hook accounts twice (once per swap). This works correctly.

### Edge Case: Zero Holdings (BuyOnly Path)

When `held_amount == 0`, `has_holdings` is false in `consume_randomness`, so `action = CarnageAction::None`. In `execute_carnage_atomic`, the match falls to `CarnageAction::None` at line 333-335, which is a no-op for Step 1. Execution proceeds directly to Step 2 (buy). Correct.

### Edge Case: held_amount > 0 with held_token = 0 (Inconsistent State)

This state should never occur in normal operation (held_token is always set alongside held_amount). If it did occur (e.g., via a bug or corrupted state):
- `burn_held_tokens` returns `Ok(0)` for `held_token` values other than 1 or 2 (line 517)
- `execute_sell_swap` returns `Ok(())` for `held_token` values other than 1 or 2 (line 720)
- The tokens would be silently skipped (not burned or sold)
- The buy step then overwrites `held_token` and `held_amount` with new values (lines 452-453)

This means tokens stuck in a vault with `held_token=0` would be orphaned until the next Carnage trigger buys into that same vault and the new `held_amount` accounts for them. Not exploitable, but tokens could be temporarily untracked.

### Edge Case: Stale Holdings After Expiration

`expire_carnage` and the auto-expire path in `consume_randomness` clear `carnage_pending` but do NOT touch `held_token`/`held_amount`. This is by design -- holdings persist across epochs. The next `consume_randomness` reads `has_holdings = carnage_state.held_amount > 0` to determine the action, correctly handling the persisted state.

### Hook Account Partitioning

The remaining_accounts split (lines 255-265) correctly handles:
- **Sell path**: First 4 accounts = sell token's hooks, remaining = buy token's hooks
- **Burn/BuyOnly**: All remaining_accounts = buy token's hooks (empty slice for sell)
- **Guard**: `ctx.remaining_accounts.len() >= HOOK_ACCOUNTS_PER_MINT * 2` ensures sufficient accounts for Sell path. If fewer are provided, falls through to Burn/BuyOnly partitioning, which would cause the sell swap CPI to fail (insufficient hook accounts). This is a safe failure mode.

## Fragility Observations (Informational)

1. **`+1` offset encoding**: The `target.to_u8() + 1` bridge between `Token` and `HeldToken` encodings is correct but not enforced at the type level. If either enum's discriminant values changed, this would silently produce wrong token selection. Consider using `HeldToken::from(target)` conversion instead of raw arithmetic.

2. **No `HeldToken` type usage in handler**: The handler works entirely with raw `u8` values for `held_token` rather than using the `HeldToken` enum defined in `carnage_fund_state.rs`. Using the typed enum with `from_u8()` validation would catch invalid states at the point of use rather than silently defaulting.

3. **Vault selection defaults to FRAUD on unknown held_token**: In `approve_delegate` (line 601-605), `if held_token == 1 { crime_vault } else { fraud_vault }` -- any value other than 1 (including 0) selects fraud_vault. This is safe because the Sell path only enters when `held_amount > 0` (line 290), and `held_token` should be 1 or 2 in that case. But a defensive `match` with error return would be more robust.
