# H130 -- Tax Program: Event Emission Correctness

**Status:** SAFE
**Severity:** N/A
**Target:** `programs/tax-program/src/events.rs`, `swap_sol_buy.rs`, `swap_sol_sell.rs`

## Hypothesis

If emitted event amounts differ from actual on-chain amounts, off-chain tracking (analytics, frontends) could show incorrect data, potentially enabling discrepancy-based exploits or misleading users.

## Analysis

### Event Structure (`events.rs`)

The `TaxedSwap` event contains: `input_amount`, `output_amount`, `tax_amount`, `tax_rate_bps`, `staking_portion`, `carnage_portion`, `treasury_portion`, `epoch`, `slot`.

### Buy Path (`swap_sol_buy.rs`)

| Field | Source | Correctness |
|-------|--------|-------------|
| `input_amount` | `amount_in` (user parameter) | Correct -- matches the SOL user committed |
| `output_amount` | `tokens_received` (balance-diff after CPI reload, line 318-320) | Correct -- uses post-CPI `user_token_b.reload()` then `amount - before`, reflecting actual AMM output |
| `tax_amount` | `calculate_tax(amount_in, tax_bps)` | Correct -- same value used for actual SOL transfers |
| `tax_rate_bps` | `epoch_state.get_tax_bps(is_crime, true)` | Correct -- same rate used for calculation |
| `staking_portion` | From `split_distribution(tax_amount)` | Correct -- same tuple used for actual transfers |
| `carnage_portion` | From `split_distribution(tax_amount)` | Correct -- same tuple used for actual transfers |
| `treasury_portion` | From `split_distribution(tax_amount)` | Correct -- remainder pattern guarantees sum == tax_amount |

### Sell Path (`swap_sol_sell.rs`)

| Field | Source | Correctness |
|-------|--------|-------------|
| `input_amount` | `amount_in` (user parameter) | Correct -- token amount sent to AMM |
| `output_amount` | `net_output` = `gross_output - tax_amount` (line 466) | Correct -- this is what the user actually retains in WSOL |
| `tax_amount` | `calculate_tax(gross_output, tax_bps)` | Correct -- calculated on actual AMM output (balance-diff) |
| `staking_portion` | From `split_distribution(tax_amount)` | Correct -- same values used for actual SOL transfers |
| `carnage_portion` | From `split_distribution(tax_amount)` | Correct -- same values used for actual SOL transfers |
| `treasury_portion` | From `split_distribution(tax_amount)` | Correct -- remainder absorbs dust, sum == tax_amount |

### Key Design Patterns

1. **Balance-diff for output**: Both paths use `reload()` after CPI followed by subtraction to get the actual output, not a predicted/calculated value. This eliminates AMM rounding discrepancies.
2. **Same variables for action and event**: The code uses the exact same local variables (`tax_amount`, `staking_portion`, etc.) for both the SOL transfers and the event emission. There is no recalculation or separate code path.
3. **Treasury remainder pattern**: `split_distribution` computes staking and carnage as floors, then treasury = total - staking - carnage. This guarantees the invariant `staking + carnage + treasury == tax_amount`, which is verified by 10K proptest cases.

### ExemptSwap Event

The `ExemptSwap` event (for Carnage tax-exempt swaps) emits `amount_a` and `direction`. Since exempt swaps bypass tax, there is no tax breakdown to verify -- the event is purely informational.

## Verdict

**No vulnerability.** All emitted amounts are derived from the same variables used for actual on-chain transfers. The balance-diff pattern for output (FIX-06) ensures the emitted `output_amount` reflects reality, not a prediction. The `split_distribution` remainder pattern guarantees the sum invariant. Off-chain consumers can trust event data matches on-chain state.
