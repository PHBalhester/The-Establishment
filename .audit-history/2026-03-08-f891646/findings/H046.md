# H046 -- Bonding Curve Quadratic Math Overflow

| Field          | Value                                                    |
|----------------|----------------------------------------------------------|
| Hypothesis     | BC quadratic formula discriminant b^2 - 4ac can overflow u128 at extreme price points |
| Target         | `programs/bonding_curve/src/math.rs` (lines 56-110, 133-193) |
| Severity       | **Informational** (no exploit path)                      |
| Status         | **REJECTED -- Safe by checked arithmetic + parameter bounds** |

---

## 1. Analysis

### 1.1 The Quadratic Formula (`calculate_tokens_out`, lines 56-110)

The function solves a quadratic equation for tokens out given SOL input. The discriminant computation is:

```
coef = P_START * TOTAL_FOR_SALE + (P_END - P_START) * current_sold
coef_sq = coef * coef
disc_rhs = 2 * (P_END - P_START) * sol_lamports * TOKEN_DECIMAL_FACTOR * TOTAL_FOR_SALE
discriminant = coef_sq + disc_rhs
```

All multiplications use `checked_mul().ok_or(CurveError::Overflow)?`.

### 1.2 Overflow Boundary Analysis

**Constants:** P_START=900, P_END=3450, b_num=2550, TOTAL_FOR_SALE=460e12, TOKEN_DECIMAL_FACTOR=1e6.

**coef (max at x1=TOTAL_FOR_SALE):**
- `900 * 460e12 + 2550 * 460e12 = 1.587e18`
- `coef^2 = 2.519e36` -- fits in u128 (max 3.4e38)

**disc_rhs (parametric in s=sol_lamports):**
- `2 * 2550 * s * 1e6 * 460e12 = 2.346e24 * s`
- Overflows u128 when `s > 1.45e14 lamports (~145,047 SOL)`
- With `s = TARGET_SOL (1e12)`: `disc_rhs = 2.346e36` -- fits with 70x headroom
- With `s = u64::MAX (1.8e19)`: `disc_rhs = 4.3e43` -- **overflows u128**

**discriminant (with bounded inputs):**
- `coef_sq + disc_rhs = 2.52e36 + 2.35e36 = 4.87e36` -- fits with 70x headroom

### 1.3 Can `sol_lamports` Exceed TARGET_SOL?

Yes. The `purchase` instruction (purchase.rs:116-119) enforces only a **minimum** (`sol_amount >= MIN_PURCHASE_SOL`), not a maximum. A user can pass `sol_amount = u64::MAX` to `calculate_tokens_out`.

**However**, the `checked_mul` chain at lines 83-91 catches the overflow:
```rust
let disc_rhs = 2u128
    .checked_mul(b_num).ok_or(CurveError::Overflow)?   // 5100
    .checked_mul(s).ok_or(CurveError::Overflow)?        // 5100 * s
    .checked_mul(d).ok_or(CurveError::Overflow)?        // 5100 * s * 1e6
    .checked_mul(b_den).ok_or(CurveError::Overflow)?;   // OVERFLOWS for s > 145K SOL
```

The step `.checked_mul(b_den)` overflows when `5100 * s * 1e6 > u128::MAX / 460e12`, which occurs at `s > ~145,047 SOL`. The `checked_mul` returns `None`, which maps to `CurveError::Overflow` -- a clean error, not a panic or silent truncation.

### 1.4 `calculate_sol_for_tokens` (lines 133-193)

This function has much wider headroom because inputs are bounded by protocol state:
- `current_sold <= TARGET_TOKENS` (enforced by state machine)
- `tokens <= current_sold` (enforced by sell instruction balance check)

Worst-case `total_scaled = ~1.001e30`, with u128 headroom of 340 million x. No overflow path exists for valid inputs.

### 1.5 `get_current_price` (lines 203-212)

Uses **unchecked** multiplication at line 210: `price_range * progress`. With current constants, max value is `2550 * 1e12 = 2.55e15`, well within u128. But `checked_mul` at line 207 protects the initial scaling. If it overflows, `unwrap_or(0)` returns 0 silently -- acceptable since this is a display-only function (used in event emissions, not financial calculations).

## 4. Verdict

**REJECTED -- No vulnerability.** The discriminant computation is fully protected:

1. **Within operational range** (s <= TARGET_SOL): Discriminant stays at ~4.87e36, providing 70x headroom below u128::MAX.
2. **Outside operational range** (s > 145K SOL): `checked_mul` returns `CurveError::Overflow` cleanly.
3. **No unchecked arithmetic** on any multiplication or addition that could overflow.
4. **7.5M+ proptest iterations** validate all properties across the operational domain.

The only theoretical concern is that `sol_amount` has no upper bound enforced in `purchase.rs`, meaning a user sending > 145K SOL would get an `Overflow` error rather than a more descriptive `ExceedsMaximum` error. This is cosmetic only -- the user loses nothing (transaction reverts).
