# H015 -- Carnage Two-Account Oracle Manipulation

| Field          | Value                                      |
|----------------|--------------------------------------------|
| **Status**     | REFUTED                                    |
| **Severity**   | Informational                              |
| **Category**   | CPI / Oracle                               |
| **Confidence** | High                                       |

## Summary

The hypothesis posits that an attacker could provide a fake pool `AccountInfo` with inflated reserves for the slippage floor calculation while a different (real) pool is used for the actual CPI swap, rendering the slippage protection ineffective and enabling Carnage fund value extraction.

**This attack is structurally impossible.** There is only one `crime_pool` and one `fraud_pool` field in the `ExecuteCarnageAtomic` accounts struct. The same account reference is used for both the `read_pool_reserves` slippage calculation and the `execute_swap_exempt_cpi` swap. An attacker cannot supply two different accounts for these two operations.

## Evidence

### Single Account Reference for Both Operations (Atomic Path)

In `execute_carnage_atomic.rs` (lines 372-396), the target pool is selected once from `ctx.accounts` and used for both the reserve read and the swap:

```rust
// Lines 372-375: Pool selected for slippage read
let target_pool_info = match target {
    Token::Crime => &ctx.accounts.crime_pool,
    Token::Fraud => &ctx.accounts.fraud_pool,
};
let (reserve_sol, reserve_token) = read_pool_reserves(target_pool_info, ...)?;

// Line 396: execute_buy_swap uses the same ctx.accounts.crime_pool / fraud_pool
execute_buy_swap(&ctx, total_buy_amount, target, ...)?;
```

`execute_buy_swap` (lines 775-788) selects the same `ctx.accounts.crime_pool` or `ctx.accounts.fraud_pool` and forwards it to `execute_swap_exempt_cpi`. Both paths originate from the same struct field.

### Same Pattern in Fallback Path

`execute_carnage.rs` (lines 380-404) follows an identical pattern: the same pool account is read for reserves at line 384 and forwarded to `execute_buy_swap` at line 404.

### AMM Rejects Fake Pools During CPI

Even if a fake account were somehow substituted, the AMM's `SwapSolPool` struct enforces:
1. **Owner check**: `pool.owner == AMM_PROGRAM_ID` (Anchor automatic deserialization)
2. **PDA seeds check**: `seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()]`
3. **Bump check**: `bump = pool.bump`
4. **Reentrancy guard**: `constraint = !pool.locked`

A fake account fails at step 1 or 2, reverting the entire transaction.

### No Remaining-Accounts Substitution Path

The pool accounts are named fields in the Anchor struct (`crime_pool`, `fraud_pool`), not drawn from `remaining_accounts`. There is no code path that could substitute a different account between the reserve read and the swap CPI.

## Attack Scenario (Why It Fails)

1. Attacker constructs a fake account with inflated reserves at byte offsets 137-153.
2. Attacker passes it as `crime_pool` in the `execute_carnage_atomic` instruction.
3. `read_pool_reserves` reads inflated values, computing an artificially low slippage floor.
4. `execute_swap_exempt_cpi` passes the **same** fake account to Tax -> AMM.
5. AMM's Anchor deserialization rejects the fake account (owner check / PDA seeds check fails).
6. CPI reverts. Entire transaction fails.

The two-account attack requires providing account A for slippage and account B for the swap. This is impossible because both operations draw from the same struct field.

## Root Cause (of the Observation, Not a Vulnerability)

The `read_pool_reserves` function does not perform an explicit AMM owner check on the pool `AccountInfo`. This was flagged by multiple Phase 1 agents (CPI: HIGH, Oracle: MEDIUM) as a defense-in-depth gap. However, the structural constraint (single account for both operations) makes the missing owner check unexploitable in the current architecture.

## Recommendation

**No code change required.** The hypothesis is refuted by the single-account structural constraint.

For defense-in-depth hardening (informational), an AMM owner check could be added to `read_pool_reserves` at minimal compute cost (~200 CU):

```rust
require!(
    pool_info.owner == &amm_program_id(),
    EpochError::InvalidCarnageTargetPool
);
```

This would protect against future refactors that might separate the reserve-read and swap-CPI code paths.
