# H110 -- Conversion Vault: Token Account Ownership

**Status:** SAFE
**Severity:** N/A (Informational)
**File:** `programs/conversion-vault/src/instructions/convert.rs`

## Hypothesis

User token accounts in the `Convert` accounts struct lack ownership/mint constraints, potentially allowing misuse.

## Evidence

The `Convert` struct defines six token-related accounts:

| Account | Constraints |
|---------|-------------|
| `user_input_account` | `#[account(mut)]` -- NO mint or authority constraint |
| `user_output_account` | `#[account(mut)]` -- NO mint or authority constraint |
| `vault_input` | `token::authority = vault_config, token::mint = input_mint` |
| `vault_output` | `token::authority = vault_config, token::mint = output_mint` |
| `input_mint` | None (just deserialized as `Mint`) |
| `output_mint` | None (just deserialized as `Mint`) |

### Why this is safe

1. **User input account (line 144-155):** The `transfer_t22_checked` call passes `user.to_account_info()` as the authority/signer. The SPL Token / Token-2022 runtime enforces that the signer must be the token account's owner (or delegate). A mismatched account will fail the on-chain token transfer. The mint check is also enforced by `transfer_checked` which validates the mint matches.

2. **User output account (line 161-171):** Tokens are sent TO this account. The `transfer_checked` CPI validates that the destination account's mint matches `output_mint`. If a user passes a wrong output account, the token program rejects the transfer (mint mismatch). If the user intentionally passes their own account for a different token, `transfer_checked` catches it. The worst case is the user sends tokens to an account they don't control -- self-harm only.

3. **Vault accounts (lines 37-51):** Properly constrained with both `token::authority = vault_config` and `token::mint = input_mint/output_mint`. An attacker cannot substitute a fake vault.

4. **Mint validation (lines 60-113):** `compute_output` validates that the input/output mint pair is one of the allowed combinations (CRIME/FRAUD <-> PROFIT). Arbitrary mints are rejected with `InvalidMintPair`.

## Verdict

**No vulnerability.** The SPL `transfer_checked` CPI provides implicit mint validation (it requires the token account's mint to match the passed mint account). The `user` signer constraint prevents unauthorized transfers from user accounts. Vault accounts have explicit Anchor constraints. The lack of explicit Anchor constraints on user token accounts is a common secure pattern -- the token program itself enforces correctness.
