# H068 -- Bonding Curve consolidate_for_refund: Permissionless Consolidation After Failure

**Auditor:** Token-Economic / State-Machine cross-check
**Severity:** INFORMATIONAL
**Status:** SAFE -- No actionable vulnerability found
**Target:** `programs/bonding_curve/src/instructions/consolidate_for_refund.rs`
**Related:** `claim_refund.rs`, `state.rs` (`is_refund_eligible`), `distribute_tax_escrow.rs`

---

## Summary

The `consolidate_for_refund` instruction moves SOL from the per-curve tax escrow PDA into the SOL vault PDA, making it available for proportional refunds. It is permissionless (anyone can call it) and gated by two conditions: the curve must be refund-eligible, and the `escrow_consolidated` flag must be false. The math and access controls are sound. No blocking vector was identified.

---

## Analysis

### 1. Permissionless Access -- Correctly State-Gated

No `Signer` authority is required. The instruction is gated by:

- **`is_refund_eligible(partner_status)`** (line 89-92): Requires curve status is `Failed`, OR status is `Filled` with partner status `Failed`. This correctly captures both failure modes in the dual-curve system.
- **`!escrow_consolidated`** (line 95-97): Idempotency guard prevents double-consolidation.

This is the correct design for a permissionless refund-enabling instruction. Any participant (or bot) can trigger consolidation once the preconditions are met, preventing a single actor from blocking refunds.

### 2. Partner Curve Validation -- Sufficient for Two-Curve System

The partner_curve_state account has three constraints:
1. Seeds match `[CURVE_SEED, partner_curve_state.token_mint.as_ref()]` -- valid CurveState PDA.
2. Bump matches stored bump -- prevents PDA substitution.
3. `partner_curve_state.key() != curve_state.key()` -- not the same curve.

Since the `Token` enum has exactly two variants (Crime, Fraud) and each CurveState PDA is seeded by token_mint, only two CurveState PDAs can exist. The `!=` constraint guarantees the partner is the _other_ curve. If additional Token variants were added in the future, this constraint would be insufficient -- an attacker could pass any third CurveState whose status satisfies the eligibility check. For the current two-curve design, this is correct.

### 3. Consolidation Math -- Correct

```rust
let rent_exempt = rent.minimum_balance(0);
let escrow_lamports = ctx.accounts.tax_escrow.lamports();
let transferable = escrow_lamports.saturating_sub(rent_exempt);
```

- `rent.minimum_balance(0)` is correct for the tax escrow PDA (a 0-byte SOL-only account, ~890,880 lamports).
- `saturating_sub` ensures no underflow if escrow has less than rent_exempt (would produce 0).
- If `transferable > 0`, lamports are moved from escrow to vault via direct manipulation (both program-owned PDAs).
- The `escrow_consolidated` flag is set to `true` even when transferable is 0. This is correct -- a curve with zero sells would have an empty escrow, and consolidation should still be considered "done" to unblock `claim_refund`.

### 4. Can Blocking Prevent Refunds?

Three potential blocking vectors were examined:

**(a) Refusing to call consolidate_for_refund:** Cannot block -- it is permissionless. Any user, bot, or keeper can call it once conditions are met.

**(b) Preventing mark_failed from being called:** `mark_failed` is also permissionless (no signer authority required), gated only by Active status + deadline + grace period having passed. Cannot be blocked.

**(c) Draining the tax escrow before consolidation:** The tax escrow PDA is owned by the bonding curve program (`seeds = [TAX_ESCROW_SEED, token_mint]`). Only the bonding curve program can subtract lamports from it. The only instructions that touch the escrow are `sell` (adds tax), `consolidate_for_refund` (drains to vault), and `distribute_tax_escrow` (drains to carnage fund). `distribute_tax_escrow` requires `Graduated` status, which is mutually exclusive with the `Failed`/refund-eligible states that `consolidate_for_refund` requires. Therefore, no instruction can drain the escrow before consolidation in the refund path.

**(d) Sending extra lamports to the escrow to manipulate refund pool:** Anyone can credit lamports to any account on Solana. Extra lamports sent to the tax escrow before consolidation would be swept into the sol_vault, increasing the refund pool for all claimers proportionally. This is beneficial to users, not exploitable.

**(e) Sending extra lamports to sol_vault directly:** Same reasoning -- increases refund pool proportionally for all users. Not exploitable.

### 5. Account Validation

- **tax_escrow**: Validated by PDA seeds AND `constraint = tax_escrow.key() == curve_state.tax_escrow`. Double-checked.
- **sol_vault**: Validated by PDA seeds `[CURVE_SOL_VAULT_SEED, curve_state.token_mint.as_ref()]` only. No stored-key cross-check, but PDA derivation is deterministic from the same token_mint, so this is sufficient.
- **curve_state**: PDA seeds + bump check. Correct.

### 6. Interaction with claim_refund

`claim_refund` requires `escrow_consolidated == true` (line 111-113). The refund pool is computed from `sol_vault.lamports() - rent_exempt` at claim time. Since consolidation moves escrow SOL into the vault, the refund pool correctly includes both the raised SOL and the tax escrow SOL. The consolidation is a prerequisite gate, not a value-setting operation -- the actual refund math reads live vault balance.

### 7. Edge Cases

- **Zero tokens sold (no activity):** `claim_refund` checks `total_outstanding > 0`. If nobody bought, nobody can claim. Consolidation would move 0 lamports and set the flag. No harm.
- **All tokens sold back before failure:** `tokens_sold` decreases on sells. If all tokens are sold back, the vault retains the sell tax (in escrow). Consolidation moves that tax into the vault. Nobody holds tokens to claim, so the SOL stays in the vault. This is SOL-locked dust -- acceptable.
- **Race condition between consolidate and claim:** Consolidation sets the flag atomically. Claims cannot proceed until consolidation is complete. No partial-state risk.

---

## Verdict

**SAFE.** The `consolidate_for_refund` instruction is well-designed:
- Permissionless access is correctly gated by compound state conditions
- Math correctly preserves rent-exempt minimum in the escrow PDA
- The idempotency flag prevents double-consolidation
- No blocking vector exists -- all prerequisite instructions are also permissionless
- Partner validation is sufficient for the two-curve system (would need revision if more Token variants are added)

No code changes recommended.
