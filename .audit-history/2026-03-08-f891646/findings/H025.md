## Hypothesis

Verify mint authority burn in `initialize.ts` actually executes and is irreversible for all 3 mints (CRIME, FRAUD, PROFIT).

## Investigation

### 1. Mint Creation (Lines 278-284)

All three mints are created via `createInitializeMintInstruction` with:
- `mintAuthority`: `authority.publicKey` (deployer wallet)
- `freezeAuthority`: `null` (no freeze authority from the start)
- `programId`: `TOKEN_2022_PROGRAM_ID`

Freeze authority is never set, eliminating the freeze-based token lockout vector entirely.

### 2. Mint Authority Burn (Lines 933-960)

The burn loop iterates over `mintKeypairs` which contains all 3 mints:
```typescript
const mintKeypairs = [
  { name: "CRIME", kp: crimeMintKeypair },
  { name: "FRAUD", kp: fraudMintKeypair },
  { name: "PROFIT", kp: profitMintKeypair },
];
```

For each mint, it:
1. **Idempotency check** (lines 936-945): Reads the on-chain mint account, inspects byte 0 of the Token-2022 mint layout (`COption<Pubkey>` tag). If byte 0 is `0` (None), the authority is already burned -- skips with log message.
2. **Burn instruction** (lines 948-955): Calls `createSetAuthorityInstruction` with:
   - `AuthorityType.MintTokens` -- targets mint authority specifically
   - New authority: `null` -- irrevocable removal
   - Program: `TOKEN_2022_PROGRAM_ID` -- correct program
3. **Confirmation** (line 957): Uses `sendAndConfirmTransaction` which waits for confirmation before proceeding.

### 3. Irreversibility Analysis

Once `setAuthority` sets `MintTokens` authority to `null`, the Token-2022 program enforces that no subsequent `setAuthority` call can restore it (there is no current authority to sign the change). This is a one-way operation at the runtime level.

### 4. On-Chain Re-Minting Paths

Searched all 7 production programs (`programs/` directory) for `mint_to`, `MintTo`, `set_authority`, or `setAuthority` instructions. Result: **zero matches in production code**. All `mint_to` references exist exclusively in test files. No on-chain program has the ability to mint tokens or restore mint authority.

### 5. Idempotency Guard Correctness

The guard at line 940 reads byte 0 of the Token-2022 mint layout. For Token-2022 mints, the first 36 bytes are `COption<Pubkey>` for mint_authority:
- Byte 0 = `0`: None (authority burned)
- Byte 0 = `1`: Some (authority present, followed by 32-byte pubkey)

This is the correct byte offset for Token-2022 mint layout. The guard correctly identifies already-burned authorities and skips redundant transactions.

### 6. Devnet Verification

Attempted to verify on-chain state for all 3 mints on devnet. Accounts were not found (`AccountNotFound`), indicating the current devnet deployment cycle may have been wiped or the mints were regenerated. This is expected behavior for devnet and does not affect the code-level finding.

### 7. Ordering Concern

The mint authority burn happens in Step 10 of `initialize.ts`, after vault seeding and token minting. This is the correct ordering: mint all required tokens first, then irrevocably burn the authority. No tokens can be minted after this step completes.

## Verdict

**RESOLVED.** The mint authority burn implementation is correct and complete:

- All 3 mints (CRIME, FRAUD, PROFIT) are covered by the burn loop
- `setAuthority` with `null` new authority is irreversible at the Token-2022 runtime level
- Freeze authority was never set (null from creation)
- No on-chain program contains mint-to or set-authority instructions
- Idempotency guard correctly inspects Token-2022 mint layout byte 0
- Ordering is correct (mint tokens before burning authority)
- `sendAndConfirmTransaction` ensures the burn is confirmed before proceeding

No action required. INV-9 (mint authority burned) is verified.
