# H078 -- AMM Fee BPS Consistency

**Status:** Not Vulnerable
**Severity:** N/A (Informational)
**Confidence:** High

## Summary

The AMM fee is applied consistently and correctly in both buy (AtoB) and sell (BtoA) directions. The fee is deducted from the input amount before the constant-product output calculation, regardless of direction. There is no asymmetry.

## Evidence

### Fee Application Path (swap_sol_pool.rs)

The handler reads `lp_fee_bps` once from pool state (line 73) and applies it identically regardless of direction:

```
// Line 125-126 -- applied BEFORE direction-specific logic
let effective_input =
    calculate_effective_input(amount_in, lp_fee_bps).ok_or(AmmError::Overflow)?;
```

The direction match (lines 101-122) only selects which reserves map to input/output. The fee calculation at line 126 uses the same `amount_in` and `lp_fee_bps` regardless of whether direction is `AtoB` or `BtoA`.

### Fee Math (amm/helpers/math.rs)

```rust
pub fn calculate_effective_input(amount_in: u64, fee_bps: u16) -> Option<u128> {
    let fee_factor = 10_000u128.checked_sub(fee_bps as u128)?;
    amount.checked_mul(fee_factor)?.checked_div(10_000)
}
```

- Single code path for both directions.
- Fee is deducted from input (fee stays in pool as LP yield).
- `fee_bps` is stored once per pool at initialization and is immutable.
- `MAX_LP_FEE_BPS = 500` (5%) prevents excessive fee extraction.
- Underflow protection: `fee_bps > 10000` returns None -> AmmError::Overflow.

### Swap Output Math

```rust
pub fn calculate_swap_output(reserve_in: u64, reserve_out: u64, effective_input: u128) -> Option<u64> {
    let numerator = r_out.checked_mul(effective_input)?;
    let denominator = r_in.checked_add(effective_input)?;
    u64::try_from(numerator.checked_div(denominator)?).ok()
}
```

- Direction-agnostic: `reserve_in` and `reserve_out` are swapped by the match block.
- Integer truncation (floor) always favors the protocol (user gets slightly less output).

### k-Invariant Verification

After reserve updates, `verify_k_invariant` confirms `k_after >= k_before` (line 171-173). This is direction-independent and catches any fee application error.

### Reserve Update Symmetry

Lines 176-185 correctly mirror the reserve updates:
- AtoB: reserve_a += amount_in, reserve_b -= amount_out
- BtoA: reserve_b += amount_in, reserve_a -= amount_out

Note: `amount_in` (pre-fee) is added to reserves. The fee stays in the pool implicitly (effective_input < amount_in, so the output is computed on less, but the full amount_in increases reserves). This is standard Uniswap v2 behavior.

## Recommendation

No action required. The fee mechanism is symmetric, correctly implemented, and verified by the k-invariant check post-swap.
