# H122 -- Staking: Reward Distribution During Unstake

| Field | Value |
|-------|-------|
| Hypothesis | Unstake may modify balance before calculating owed rewards, causing users to lose earned rewards or receive incorrect amounts. |
| Severity | N/A (No finding) |
| Status | **REFUTED** |
| File | `programs/staking/src/instructions/unstake.rs` |
| Supporting | `programs/staking/src/helpers/math.rs` |

## Analysis

The hypothesis asks whether `unstake` triggers reward calculation **before** the balance change. This is critical because in the Synthetix cumulative reward-per-token pattern, the reward delta must be computed against the **old** staked balance. If balance decreases first, the user loses rewards they earned.

### Code Flow (unstake.rs lines 151-187)

```
1. update_rewards(pool, user)?;          // line 156 -- rewards computed on OLD balance
2. rewards_forfeited = user.rewards_earned;  // line 161
3. pool.pending_rewards += rewards_forfeited; // line 163-166
4. user.rewards_earned = 0;              // line 167
5. user.staked_balance -= amount;        // line 171-174  -- balance changes AFTER
6. pool.total_staked -= amount;          // line 176-179
```

### update_rewards (math.rs lines 36-67)

The function computes:
```
pending = user.staked_balance * (pool.rewards_per_token_stored - user.rewards_per_token_paid) / PRECISION
```

This uses `user.staked_balance` which has NOT been decremented yet at the time of the call. The function then:
1. Adds `pending` to `user.rewards_earned`
2. Updates `user.rewards_per_token_paid` to current global checkpoint
3. Updates `user.last_update_slot`

Only after `update_rewards` completes do lines 171-179 modify `staked_balance` and `total_staked`.

### Design Choice: Reward Forfeiture

The protocol intentionally forfeits unclaimed rewards on unstake (lines 161-168). After computing the correct pending rewards, the code:
- Captures `user.rewards_earned` (which includes freshly computed pending)
- Adds it to `pool.pending_rewards` (redistributes to remaining stakers)
- Zeros `user.rewards_earned`

This is a deliberate anti-mercenary-capital mechanism (documented in the file header as "reward forfeiture"). It is NOT a bug -- it means unstakers donate their pending rewards back to the pool.

### CEI Pattern

The code follows Checks-Effects-Interactions correctly:
- **Checks**: amount > 0, ownership, balance, cooldown (lines 108-134)
- **Effects**: reward calculation, forfeiture, balance updates (lines 151-187)
- **Interactions**: token transfer via CPI (lines 199-209)

The mutable borrow scope (lines 151-187) is cleanly separated from the CPI call, with the pool bump captured before the mutable borrow begins (line 148).

### Additional Protections

- **Cooldown gate** (lines 125-134): Prevents rapid stake/unstake to game reward snapshots
- **Partial unstake minimum** (lines 138-145): Auto-full-unstake if remainder < MINIMUM_STAKE, preventing dust positions
- **Checked arithmetic** throughout: `checked_sub`, `checked_add` with explicit error types
- **80K proptest iterations** on the reward math (math.rs property tests T21-T23, lines 576-691)

## Verdict

**SECURE.** The reward calculation correctly executes before any balance mutation. The checkpoint pattern is properly implemented. The forfeiture design is intentional and well-documented.
