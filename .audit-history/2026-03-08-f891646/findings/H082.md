# H082 -- Epoch Skipping Handling

**Status:** VERIFIED -- Epoch skip handled correctly
**Severity:** Informational (confirmed secure)
**Target:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs`

---

## Scenario

If no one calls `trigger_epoch_transition` for multiple epoch durations (e.g., protocol is at epoch 5 stored, but 20 SLOTS_PER_EPOCH periods have elapsed), the system must handle the gap correctly without corrupted state.

---

## Core Logic (lines 134-139)

```rust
let expected_epoch = current_epoch(clock.slot, epoch_state.genesis_slot);
require!(
    expected_epoch > epoch_state.current_epoch,
    EpochError::EpochBoundaryNotReached
);
```

### `current_epoch()` (lines 80-82):
```rust
pub fn current_epoch(slot: u64, genesis_slot: u64) -> u32 {
    ((slot.saturating_sub(genesis_slot)) / SLOTS_PER_EPOCH) as u32
}
```

---

## Skip Analysis

### What happens when multiple epochs are skipped?

If `current_epoch` stored is 5 and the actual epoch based on slot is 20:
- `expected_epoch` = 20
- `expected_epoch > 5` = true (passes boundary check)
- State update (line 176): `epoch_state.current_epoch = expected_epoch` (jumps directly to 20)
- `epoch_start_slot` is set to `epoch_start_slot(20, genesis)` (correct for epoch 20)

**The epoch counter jumps directly to the current epoch.** It does NOT iterate through epochs 6-19. This is correct behavior -- there is nothing to process for missed epochs since no trades occurred during them (the protocol was unattended).

### Are intermediate epochs' tax rates lost?

No impact. Tax rates are VRF-derived per-epoch. During the skipped epochs:
- `taxes_confirmed` was whatever it was set to in epoch 5
- The Tax Program reads `taxes_confirmed` but does not fail when it's stale -- trades use the last confirmed rates (documented design choice, A-10 in ARCHITECTURE.md)
- After the skip trigger, new VRF randomness sets fresh rates for epoch 20

### Is the VRF-pending guard safe across skips?

Yes. Line 149: `require!(!epoch_state.vrf_pending, EpochError::VrfAlreadyPending)`.

If VRF was pending from epoch 5 and nobody called `consume_randomness`, the `vrf_pending` flag stays true. In this case, `trigger_epoch_transition` for epoch 20 would **fail** with `VrfAlreadyPending`.

**Resolution path:** The `retry_epoch_vrf` instruction exists for exactly this case. It checks if `vrf_request_slot + VRF_TIMEOUT_SLOTS` has passed, resets the VRF state, and allows a new trigger. This is the documented VRF timeout recovery mechanism.

### Can the skip cause double-bounty payments?

No. Only one bounty is paid per `trigger_epoch_transition` call. Skipping from epoch 5 to 20 pays one bounty, not 15 bounties.

### Does epoch_start_slot calculation handle the skip?

Yes. `epoch_start_slot(20, genesis)` = `genesis + 20 * SLOTS_PER_EPOCH`. This is the correct start of epoch 20, regardless of what epochs were missed.

### Unit test coverage for skip:

The test `test_epoch_boundary_detection_logic` (lines 329-363) confirms that `current_epoch(10000, 0)` with `current_stored_epoch = 0` correctly triggers (expected > stored). The `test_epoch_calculation_consistency` test verifies epoch arithmetic over 100 epochs.

## Edge Cases

### Skip with pending Carnage
If Carnage was pending from epoch 5 (`carnage_pending = true`), and epochs 6-19 passed:
- `expire_carnage` can be called independently (permissionless, checks `slot > carnage_deadline_slot`)
- After expiring Carnage, the VRF timeout recovery path can reset `vrf_pending`
- Then `trigger_epoch_transition` succeeds

### Skip to u32::MAX epoch
`current_epoch` casts to u32. With SLOTS_PER_EPOCH=4500 (mainnet): max epoch = u32::MAX = 4,294,967,295, requiring ~1.93e13 slots (~24,000 years). Not a practical concern.

### Slot before genesis (saturating_sub)
`saturating_sub` returns 0 if slot < genesis_slot, so epoch is 0. Unit-tested.

---

## Verdict

**CONFIRMED SECURE.** Epoch skipping is handled correctly by computing the current epoch from absolute slot position rather than incrementing. The jump-to-current approach is the right design. The VRF-pending guard prevents re-triggering if a previous VRF cycle was not completed, and the retry mechanism provides recovery. No intermediate epoch state is lost because none was generated. One bounty per trigger call regardless of skip distance.
