# H086 -- Bonding Curve: Deadline Slot vs Clock

| Field | Value |
|-------|-------|
| Hypothesis | Bonding curve uses block timestamp instead of slot for deadline checks |
| Severity | N/A |
| Status | **REJECTED -- Not a finding** |
| Confidence | High |
| Files | `programs/bonding_curve/src/instructions/mark_failed.rs`, `purchase.rs`, `sell.rs`, `start_curve.rs` |

## Analysis

The bonding curve uses `Clock::get()?.slot` (slot-based) exclusively for all deadline logic. No `unix_timestamp` comparisons are used for deadline enforcement.

### Evidence

**start_curve.rs** -- Sets deadline using slot arithmetic:
```rust
curve.deadline_slot = clock.slot + DEADLINE_SLOTS;  // line 66
```

**purchase.rs** -- Checks deadline against current slot:
```rust
clock.slot <= curve.deadline_slot  // line 109
```

**sell.rs** -- Same slot-based check:
```rust
Clock::get()?.slot <= curve.deadline_slot  // line 143
```

**mark_failed.rs** -- Failure eligibility also slot-based:
```rust
let failure_eligible_slot = curve.deadline_slot.checked_add(FAILURE_GRACE_SLOTS)?;
require!(clock.slot > failure_eligible_slot, ...);  // lines 56-62
```

### Constants

- `DEADLINE_SLOTS = 500` (devnet; mainnet would use 432,000 for ~48h)
- `FAILURE_GRACE_SLOTS = 150` (~60 seconds grace buffer)

### Verdict

Slot-based timing is the standard approach on Solana. The `Clock` sysvar's `slot` field is monotonically increasing and cannot be manipulated by validators (unlike `unix_timestamp` which has some validator discretion). The implementation is correct and follows best practices.
