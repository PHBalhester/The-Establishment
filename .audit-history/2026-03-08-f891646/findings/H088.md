# H088 -- Staking: init_if_needed on UserStake

| Field | Value |
|-------|-------|
| Hypothesis | Reinitialization vulnerability on UserStake PDA via `init_if_needed` |
| Severity | N/A |
| Status | **REJECTED -- Not a finding** |
| Confidence | High |
| Files | `programs/staking/src/instructions/stake.rs`, `programs/staking/src/state/user_stake.rs` |

## Analysis

### PDA Seeds Include User Pubkey

The `UserStake` account is derived with seeds `["user_stake", user.key()]` (line 47 of stake.rs):
```rust
#[account(
    init_if_needed,
    payer = user,
    space = UserStake::LEN,
    seeds = [USER_STAKE_SEED, user.key().as_ref()],
    bump,
)]
pub user_stake: Account<'info, UserStake>,
```

Since `user` is a `Signer`, each user can only interact with their own PDA. No user can target another user's `UserStake` account.

### Reinitialization Protection

Anchor's `init_if_needed` will only initialize an account if it does not yet exist (zero lamports / unallocated). Once created, subsequent calls skip the `init` path because the account already has the correct discriminator and data.

The handler adds a second layer of protection (lines 106-114):
```rust
let is_new_user = user.owner == Pubkey::default();
if is_new_user {
    user.owner = ctx.accounts.user.key();
    user.rewards_per_token_paid = pool.rewards_per_token_stored;
    user.first_stake_slot = clock.slot;
    user.bump = ctx.bumps.user_stake;
    user.last_claim_ts = 0;
}
```

This checks `owner == Pubkey::default()` (all zeros), which is the zero-initialized state of a freshly created account. On subsequent `stake` calls, `owner` is already set to the user's pubkey, so the initialization block is skipped entirely.

### Owner Verification on Sensitive Operations

- **unstake.rs** (line 54): `constraint = user_stake.owner == user.key() @ StakingError::Unauthorized`
- **claim.rs** (line 48): `constraint = user_stake.owner == user.key() @ StakingError::Unauthorized`

Both include redundant explicit `require!` checks in the handler body as defense-in-depth.

### Known `init_if_needed` Concern

The classic `init_if_needed` attack vector is when an attacker can create an account at a known PDA address before the legitimate user, pre-filling it with malicious data. This is NOT possible here because:

1. The PDA seeds include `user.key()`, and `user` must be a `Signer`
2. An attacker cannot sign as another user
3. Anchor writes the discriminator atomically during init, so a pre-created system-program account at the same address would fail the discriminator check

### Verdict

No reinitialization vulnerability exists. The PDA seeds bind the account to the signer, Anchor's `init_if_needed` prevents double-initialization, and the `owner == Pubkey::default()` guard provides an additional layer of protection. Owner checks on unstake/claim are properly enforced via both constraints and explicit require statements.
