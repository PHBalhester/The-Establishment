# H043 -- Pool Reserve Overflow (H092 Recheck)

## Hypothesis

Pool reserves are stored as u64. The k-invariant check multiplies `reserve * reserve` -- could this overflow u128 during `verify_k_invariant`?

## Analysis

### verify_k_invariant Implementation

At `programs/amm/src/helpers/math.rs:92-103`:

```rust
pub fn verify_k_invariant(
    reserve_in_before: u64,
    reserve_out_before: u64,
    reserve_in_after: u64,
    reserve_out_after: u64,
) -> Option<bool> {
    let k_before = (reserve_in_before as u128)
        .checked_mul(reserve_out_before as u128)?;
    let k_after = (reserve_in_after as u128)
        .checked_mul(reserve_out_after as u128)?;
    Some(k_after >= k_before)
}
```

### Mathematical Proof: u64 * u64 Cannot Overflow u128

The worst case is `u64::MAX * u64::MAX`:

- `u64::MAX = 2^64 - 1`
- `(2^64 - 1)^2 = 2^128 - 2^65 + 1`
- `u128::MAX = 2^128 - 1`
- `2^128 - 2^65 + 1 < 2^128 - 1` (true, since `2^65 > 2`)

Therefore, the product of any two u64 values always fits in u128. The `checked_mul` is defense-in-depth but will never return `None`.

### Caller Context (swap_sol_pool.rs)

At lines 162-173 of `swap_sol_pool.rs`, `new_reserve_in` is computed via `checked_add` (returns `AmmError::Overflow` on u64 overflow) and `new_reserve_out` via `checked_sub`. Both values remain within u64 bounds before being passed to `verify_k_invariant`. The reserves cannot escape u64 range.

### Test Coverage

The codebase includes an explicit unit test for the extreme case (`math.rs:320-328`):

```rust
fn k_u64_max_both_sides() {
    // u64::MAX * u64::MAX fits in u128:
    // (2^64-1)^2 = 2^128 - 2^65 + 1
    // u128::MAX  = 2^128 - 1
    assert_eq!(
        verify_k_invariant(u64::MAX, u64::MAX, u64::MAX, u64::MAX),
        Some(true)
    );
}
```

Additionally, 10,000 proptest iterations (lines 357-446) exercise random reserve values including u64::MAX and u64::MAX/2 edge cases.

## Verdict

**FALSE POSITIVE.** The u128 intermediate type provides mathematically guaranteed overflow protection for the product of any two u64 values. The `checked_mul` adds a belt-and-suspenders safety net that can never trigger. Reserve values are bounded to u64 by `checked_add`/`checked_sub` before reaching this function. This matches the original H092 conclusion -- no change in risk.
