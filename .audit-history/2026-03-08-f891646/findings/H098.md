# H098 -- Tax Program: EpochState Deserialization Safety

**Status:** SAFE (with known coupling risk noted in ARCHITECTURE.md)
**Severity:** Informational
**Component:** `programs/tax-program/src/state/epoch_state_reader.rs`, `programs/tax-program/src/instructions/swap_sol_buy.rs`, `swap_sol_sell.rs`

## Hypothesis

Check whether the Tax Program correctly validates the EpochState account it reads cross-program: owner check, discriminator validation, and deserialization method.

## Findings

### 1. Account Declaration

The `epoch_state` field in both `SwapSolBuy` and `SwapSolSell` account structs is declared as raw `AccountInfo<'info>` -- no Anchor constraints at the struct level:

```rust
/// CHECK: Validated manually in handler:
/// - Owner check: must be Epoch Program (prevents fake 0% tax)
/// - Deserialization validates discriminator
/// - initialized flag checked
pub epoch_state: AccountInfo<'info>,
```

### 2. Owner Check (PRESENT)

Both `swap_sol_buy` and `swap_sol_sell` handlers perform an explicit owner check before deserialization:

```rust
let epoch_program = epoch_program_id();
require!(
    ctx.accounts.epoch_state.owner == &epoch_program,
    TaxError::InvalidEpochState
);
```

This prevents an attacker from passing a fake account owned by a different program with crafted data (e.g., 0% tax rates).

### 3. Deserialization Method (SAFE)

The code uses `EpochState::try_deserialize(&mut data_slice)`, which is Anchor's standard deserialization. This method:
- Validates the 8-byte discriminator (`sha256("account:EpochState")[0..8]`)
- Deserializes all fields via Borsh
- Returns error on discriminator mismatch or insufficient data

This is the correct approach for cross-program account reads where the account type is defined in a different program.

### 4. Initialized Flag Check (PRESENT)

After deserialization, both handlers verify:
```rust
require!(epoch_state.initialized, TaxError::InvalidEpochState);
```

### 5. Layout Mirror Accuracy (VERIFIED)

The Tax Program's mirror struct at `state/epoch_state_reader.rs` has **identical field order and types** as the Epoch Program's canonical `state/epoch_state.rs`:
- Both: 20 timing bytes + 5 tax config + 8 derived rates + 42 VRF + 23 carnage + 2 protocol = 100 bytes
- Both define `LEN = 108` (8 discriminator + 100 data)
- The Epoch Program has a compile-time assertion: `const _: () = assert!(EpochState::DATA_LEN == 100);`

### 6. Known Risk: No Compile-Time Cross-Crate Check

The ARCHITECTURE.md already documents this at A-4: "EpochState layout matches Tax mirror -- Verified field-by-field -- no compile-time check." If either struct changes independently, deserialization will silently produce wrong values. This is a **coupling risk**, not a bug.

### 7. No PDA Seed Constraint

The `epoch_state` account is not constrained to a specific PDA address at the Anchor struct level. The owner check + discriminator check means an attacker cannot forge a valid-looking account, but they could theoretically pass a *different* valid EpochState PDA if multiple existed. Since EpochState is a singleton PDA (`seeds = ["epoch_state"]`), this is not exploitable -- there is exactly one valid account. However, adding a `seeds` + `seeds::program` constraint at the struct level would be more defensive.

## Verdict

**SAFE.** The three-layer validation (owner check, discriminator via try_deserialize, initialized flag) is sound. The only residual risk is the layout coupling documented in ARCHITECTURE.md Section 7.2 / A-4, which requires manual verification on any struct change. Consider adding a PDA address constraint at the Anchor struct level for defense-in-depth.
