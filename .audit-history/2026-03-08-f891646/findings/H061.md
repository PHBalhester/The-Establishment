# H061 -- Bonding Curve claim_refund Proportionality

**File:** `programs/bonding_curve/src/instructions/claim_refund.rs`
**Severity:** Informational (Low)
**Status:** Safe by design; dust accumulation bounded and favorable to last claimer
**Confidence:** High (1M+ proptest iterations confirm properties)

---

## Hypothesis

Refund uses floor division with a shrinking denominator. Does the last claimer get more or less? How much dust accumulates?

## Mechanism

The refund formula at line 146-149:

```rust
let refund_amount = ((user_balance as u128)
    .checked_mul(refund_pool as u128)
    .ok_or(CurveError::Overflow)?
    / (total_outstanding as u128)) as u64;
```

After each claim, the handler updates state (lines 182-186):
- `tokens_sold -= user_balance` (denominator shrinks)
- `sol_vault -= refund_amount` (numerator shrinks)

Each subsequent claimer sees a smaller `refund_pool` (vault balance minus rent) and a smaller `total_outstanding` (tokens_sold). The floor division at each step truncates fractional lamports, leaving dust in the vault.

## Analysis

### Last Claimer Gets More (Relatively)

The last claimer always receives their exact proportional share with zero dust, because when `user_balance == total_outstanding`, the formula reduces to `refund_pool * 1 / 1 = refund_pool` -- the entire remaining balance. All accumulated dust from prior floor-rounded claims benefits this final claimer.

Concrete example from the unit test at `math.rs:1206-1227`:
- 3 users, each holding 1 token, pool = 10 lamports
- User 1: `floor(1 * 10 / 3) = 3` lamports (fair share: 3.33)
- User 2: `floor(1 * 7 / 2) = 3` lamports (fair share: 3.33)
- User 3: `floor(1 * 4 / 1) = 4` lamports (fair share: 3.33)
- User 3 gets 1 extra lamport of accumulated dust

### Dust Accumulation Is Bounded

Each floor division produces at most 1 lamport of dust. For N claimers, maximum total dust is N-1 lamports (the last claimer absorbs all dust, so only N-1 intermediate claims produce truncation). Since the last claimer always drains the pool exactly, total dust retained in the vault after all claims is **zero**.

The proptest at `math.rs:1461-1509` (R3: refund_vault_exhaustion, 1M cases) proves:
- `total_refunded + remaining == initial_vault_sol` (conservation)
- `remaining <= num_users` (dust bounded by user count)

The proptest at `math.rs:1291-1406` (R1: refund_order_independent, 1M cases) proves:
- Per-user refund differs by at most N-1 lamports between claim orderings
- Each user's refund deviates from fair share by at most N-1 lamports

### Order-Dependency Magnitude

For realistic parameters (460M tokens, 1000 SOL pool, 2-6 claimers), the maximum per-user deviation from fair share is 2 lamports (0.000000002 SOL). This is economically negligible.

### Solvency Guaranteed

The proptest at `math.rs:1408-1458` (R2: refund_solvency_per_claim, 1M cases) proves that no individual claim can overdraw the vault. This holds because `user_balance <= total_outstanding` implies `floor(user_balance * refund_pool / total_outstanding) <= refund_pool`.

### as u64 Cast Safety

The `as u64` cast at line 149 is safe because the result of the division is bounded by `refund_pool` (since `user_balance / total_outstanding <= 1`), and `refund_pool` is derived from a `u64` vault balance. The arithmetic context analysis in `02-arithmetic.md` confirms this.

### Rent-Exempt Handling

The refund pool computation at lines 131-138 correctly subtracts rent-exempt minimum from the vault balance via `checked_sub`. This ensures the vault PDA retains enough lamports to survive. However, after all claims complete, the vault retains exactly `rent_exempt` lamports (all dust is absorbed by the last claimer), which is correct behavior.

## Verdict

**No vulnerability.** The shrinking-denominator pattern is a standard proportional refund design that is well-understood and thoroughly tested:

1. **Last claimer gets slightly more** (by at most N-1 lamports) -- this is the correct and expected behavior of floor-division sequential claims
2. **No dust remains in the vault** after all claims (the last claimer absorbs it)
3. **No individual claim can overdraw** the vault
4. **Order-dependency is bounded** to N-1 lamports per user
5. **All properties validated** by 1M+ proptest iterations across 5 property tests (R1-R5)

The all-or-nothing claim design (line 14-16: "No partial refunds -- prevents rounding exploitation via many small claims") is a critical defense that prevents an attacker from splitting their balance across wallets to manipulate dust accumulation. Since each wallet must claim its entire balance in one shot, the number of claims equals the number of distinct token holders, and dust per claim is at most 1 lamport.
