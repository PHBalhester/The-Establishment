# H001 -- Bonding Curve Authority: withdraw_graduated_sol Theft

**Status:** CONFIRMED
**Severity:** CRITICAL
**Category:** Access Control, Upgrade/Admin
**Confidence:** HIGH

## Summary

The `withdraw_graduated_sol` instruction accepts ANY `Signer<'info>` as the `authority` account with zero on-chain identity verification. There is no `has_one`, no `address =` constraint, no ProgramData upgrade-authority check, and no stored admin comparison. Any wallet that signs the transaction receives all withdrawable SOL (~1000 SOL per graduated curve, ~2000 SOL total for both CRIME and FRAUD curves).

This was flagged by 5 independent agents in audit #2 (Access Control, State Machine, Token/Economic, Upgrade/Admin, Timing) and is the single highest-severity finding across both audits.

## Evidence

**File:** `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs`

**Lines 25-28 -- The authority account has only `Signer` and `mut`, no identity constraint:**
```rust
/// Protocol authority (deployer). Must sign. Receives SOL.
#[account(mut)]
pub authority: Signer<'info>,
```

**Lines 80-81 -- Lamports transferred directly to the unchecked authority signer:**
```rust
**ctx.accounts.sol_vault.try_borrow_mut_lamports()? -= withdrawable;
**ctx.accounts.authority.try_borrow_mut_lamports()? += withdrawable;
```

**Lines 32-36 -- The only guard is that the curve must be in Graduated state:**
```rust
#[account(
    seeds = [CURVE_SEED, curve_state.token_mint.as_ref()],
    bump = curve_state.bump,
    constraint = curve_state.status == CurveStatus::Graduated @ CurveError::CurveNotGraduated,
)]
pub curve_state: Account<'info, CurveState>,
```

**Contrast with AMM's secure pattern at `programs/amm/src/instructions/initialize_admin.rs:44-56`:**
The AMM program correctly validates authority against ProgramData upgrade authority using `program_data.upgrade_authority_address == Some(authority.key())`. The bonding curve program (added in v1.2) did not replicate this pattern.

**All 6 bonding curve admin instructions share the same broken pattern:** `initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault` -- all accept a bare `Signer<'info>` named `authority` with no validation.

## Attack Scenario

1. Attacker monitors on-chain state for both CurveState PDAs (seeds: `["curve", CRIME_MINT]` and `["curve", FRAUD_MINT]`).
2. Once both curves reach `CurveStatus::Filled` (either naturally or by the attacker buying the last tokens), the attacker constructs a single atomic transaction containing:
   - Instruction 1: `prepare_transition` (transitions both curves from Filled to Graduated -- also accepts any signer)
   - Instruction 2: `withdraw_graduated_sol` for CRIME curve (attacker receives ~1000 SOL)
   - Instruction 3: `withdraw_graduated_sol` for FRAUD curve (attacker receives ~1000 SOL)
3. The attacker submits via Jito bundle with a tip to guarantee priority over the legitimate admin.
4. Result: ~2000 SOL stolen in a single atomic transaction. The legitimate admin's subsequent withdrawal attempt returns Ok(()) due to the idempotent check (line 73: `if withdrawable == 0 { return Ok(()) }`), silently yielding zero SOL.

**Cost to attacker:** Only the transaction fee (~0.000005 SOL) plus optional Jito tip.

**Race condition:** The attacker only needs to land their transaction before the admin. On Solana with Jito tips, this is trivially achievable.

## Root Cause

The bonding curve program was added in v1.2 (Phases 70-77) and did not replicate the AMM program's ProgramData upgrade-authority validation pattern. The `authority` field is documented as "Protocol authority (deployer)" in comments but has no on-chain enforcement of this claim. The Graduated status constraint is not a defense -- it is the prerequisite that *enables* the attack.

## Recommendation

**Option A (Recommended): Replicate AMM's ProgramData authority pattern**

Add ProgramData and program accounts to `WithdrawGraduatedSol`, and constrain the authority against the program's upgrade authority:

```rust
#[account(
    constraint = program_data.upgrade_authority_address == Some(authority.key())
        @ CurveError::Unauthorized,
)]
pub program_data: Account<'info, ProgramData>,

#[account(
    constraint = program.programdata_address()? == Some(program_data.key())
        @ CurveError::Unauthorized,
)]
pub program: Program<'info, crate::program::BondingCurve>,
```

Apply the same fix to all 6 admin instructions: `initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault`.

**Option B: Store admin in a BondingCurveAdmin PDA**

Create an `initialize_admin` instruction (gated by ProgramData) that stores an admin pubkey, then add `has_one = admin` constraints to all admin instructions. This mirrors the AMM's AdminConfig pattern and supports admin rotation/burn.
