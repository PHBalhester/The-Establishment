# H033 -- Staking First-Depositor Attack

| Field           | Value                                    |
|-----------------|------------------------------------------|
| **Hypothesis**  | First staker deposits 1 wei, gets rewards, dilutes second staker |
| **Category**    | Token/Economic                           |
| **Target**      | `programs/staking/src/instructions/stake.rs`, `initialize_stake_pool.rs`, `helpers/math.rs`, `constants.rs` |
| **Severity**    | INFORMATIONAL (mitigated)                |
| **Status**      | **NOT VULNERABLE** -- Dead stake mitigation is correctly implemented |

---

## Summary

The classic first-depositor attack (ERC-4626 vault inflation / Synthetix staking inflation) is **fully mitigated** by the dead stake mechanism. The protocol deposits 1 PROFIT token (1,000,000 base units) during `initialize_stake_pool`, and this stake is permanently locked with no associated UserStake account, making it impossible to unstake or claim rewards against it.

---

## Background: The Attack

In unprotected staking systems, the first depositor can:
1. Stake 1 wei (smallest unit)
2. Donate a large reward amount directly to the pool
3. The cumulative reward-per-token becomes astronomically large
4. When a second staker deposits, truncation in the `balance * cumulative / PRECISION` formula causes them to receive far fewer rewards than their proportional share
5. The first depositor claims the excess

This is economically equivalent to the ERC-4626 share inflation attack.

---

## Mitigation Analysis

### 1. Dead Stake Initialization (VERIFIED)

**File:** `programs/staking/src/instructions/initialize_stake_pool.rs:120-138`

During pool initialization:
- `MINIMUM_STAKE` (1,000,000 base units = 1 PROFIT) is transferred from the authority to the stake vault
- `pool.total_staked` is set to `MINIMUM_STAKE`
- **No UserStake PDA is created for the authority** -- the dead stake has no owner

This means the dead stake is permanently locked. No one can unstake it because:
- Unstaking requires a `UserStake` account with `user_stake.owner == user.key()` (line 54 of unstake.rs)
- The dead stake has no corresponding UserStake account
- There is no instruction to create a UserStake retroactively for already-staked tokens

### 2. Division-by-Zero Prevention (VERIFIED)

**File:** `programs/staking/src/helpers/math.rs:97-101`

```rust
if pool.total_staked == 0 {
    return Ok(0);
}
```

If `total_staked` somehow reached 0, `add_to_cumulative` returns early without distributing, and rewards stay in `pending_rewards` for the next epoch. However, with the dead stake, `total_staked` can never reach 0 because:
- Initial: `total_staked = MINIMUM_STAKE = 1,000,000`
- Unstake subtracts user amounts, but the dead stake (1,000,000) is never subtracted
- Even if all human stakers unstake, `total_staked >= 1,000,000`

### 3. Reward Precision (VERIFIED)

**File:** `programs/staking/src/constants.rs:21`

`PRECISION = 1e18` (u128) provides 18 decimal places. With a dead stake of 1,000,000 base units:
- `reward_per_token = pending_rewards * 1e18 / total_staked`
- Even 1 lamport of rewards with 1,000,000 dead stake: `1 * 1e18 / 1,000,000 = 1e12` -- well within u128 range and non-zero
- The dead stake absorbs its proportional share of rewards (these are lost to the protocol, acting as a "dead share" tax)

### 4. Attack Profitability with Dead Stake (CALCULATED)

With dead stake = 1,000,000 (1 PROFIT):

**Scenario:** Attacker stakes 1 base unit, rewards of R lamports are deposited.

- `reward_per_token = R * 1e18 / (1,000,000 + 1)`
- Attacker's reward = `1 * reward_per_token / 1e18 = R / 1,000,001` (truncated to 0 for small R)
- Dead stake absorbs: `1,000,000 * reward_per_token / 1e18 = R * 1,000,000 / 1,000,001`

The attacker gets essentially nothing. To steal meaningful rewards, they'd need to stake more than the dead stake, at which point the attack becomes a normal staking operation with fair proportional rewards.

**Direct donation attack:**
- Attacker sends R SOL directly to escrow vault (not via deposit_rewards CPI)
- `deposit_rewards` reconciliation check (`escrow_vault.lamports() >= pool.pending_rewards`) would pass but `pending_rewards` is not incremented
- Direct donations do NOT inflate `rewards_per_token_stored` -- only `deposit_rewards` CPI (gated by Tax Program PDA) can increment `pending_rewards`
- Therefore, direct donations to the escrow are simply lost, not exploitable

### 5. No Minimum Stake Enforcement on Users (NOTED)

The `stake` instruction accepts any `amount > 0`. Users can stake 1 base unit. However, this is harmless because:
- The dead stake ensures the denominator is always at least 1,000,000
- Small stakers simply receive proportionally small rewards (truncated to 0 for dust amounts)
- There is no inflation vector because `add_to_cumulative` divides by `total_staked`, not by number of stakers

---

## Edge Case: Can Dead Stake Be Drained?

**No.** The dead stake cannot be removed because:
1. There is no `UserStake` PDA for the initializing authority
2. The `unstake` instruction requires `user_stake.owner == user.key()` constraint
3. There is no admin override or emergency withdrawal that bypasses this
4. The stake vault's authority is the `stake_pool` PDA, which only signs in the `unstake` instruction handler

The only way to reduce `total_staked` below `MINIMUM_STAKE` would be a program upgrade that introduces a new instruction -- but that would be a separate security concern (covered by upgrade authority analysis).

---

## Verdict

**NOT VULNERABLE.** The first-depositor attack is comprehensively mitigated by:

1. **Dead stake of 1 PROFIT (1,000,000 base units)** deposited at initialization with no associated UserStake
2. **Division-by-zero guard** in `add_to_cumulative` (defense in depth)
3. **u128 PRECISION (1e18)** preventing meaningful truncation errors
4. **CPI-gated reward deposits** preventing direct donation inflation
5. **80K+ proptest iterations** validating reward conservation invariant (INV-4)

The dead stake amount (1 PROFIT) is reasonable. The reward "leak" to dead stake is negligible: with 20M PROFIT total supply and realistic staking participation (say 5M staked), the dead stake absorbs 1/5,000,001 of rewards -- approximately 0.00002%, economically insignificant.
