# H002 ŌĆö Bonding Curve Authority: prepare_transition Forced Graduation

**Status:** CONFIRMED
**Severity:** CRITICAL
**Category:** Access Control, State Machine
**Confidence:** HIGH

## Summary

The `prepare_transition` instruction accepts ANY `Signer<'info>` as the `authority` account with zero on-chain verification that the signer is the protocol deployer or any specific keypair. Once both CRIME and FRAUD bonding curves reach the `Filled` state (a natural consequence of users purchasing tokens), any attacker can call `prepare_transition` to force graduation, then immediately call `withdraw_graduated_sol` twice in the same transaction to steal all raised SOL (~1000 SOL per curve, ~2000 SOL total).

## Evidence

**prepare_transition.rs:18-21** ŌĆö The `authority` field is a bare `Signer<'info>` with no `has_one`, no `address =`, and no ProgramData constraint:
```rust
pub struct PrepareTransition<'info> {
    /// Protocol authority (deployer). Must sign.
    pub authority: Signer<'info>,
```

**prepare_transition.rs:53-68** ŌĆö The handler only checks that both curves are `Filled`, then sets both to `Graduated`. No authority validation whatsoever:
```rust
require!(ctx.accounts.crime_curve_state.status == CurveStatus::Filled, ...);
require!(ctx.accounts.fraud_curve_state.status == CurveStatus::Filled, ...);
ctx.accounts.crime_curve_state.status = CurveStatus::Graduated;
ctx.accounts.fraud_curve_state.status = CurveStatus::Graduated;
```

**withdraw_graduated_sol.rs:25-28** ŌĆö Same bare `Signer<'info>` pattern, and the signer receives the SOL directly (line 81):
```rust
pub authority: Signer<'info>,
// ...
**ctx.accounts.authority.try_borrow_mut_lamports()? += withdrawable;
```

**withdraw_graduated_sol.rs:35** ŌĆö Only gate is `curve_state.status == CurveStatus::Graduated`, which the attacker just set via `prepare_transition`:
```rust
constraint = curve_state.status == CurveStatus::Graduated @ CurveError::CurveNotGraduated,
```

The `Filled` state is reached naturally when `tokens_sold >= TARGET_TOKENS` during a `purchase` call (state.rs:35, purchase.rs:285-293). This is a permissionless user action ŌĆö the attacker does not need to control the fill.

## Attack Scenario

1. Both CRIME and FRAUD bonding curves reach `Filled` status through normal user purchases.
2. Attacker constructs a single Solana transaction containing 3 instructions:
   - **IX 1:** `prepare_transition` ŌĆö passes attacker's wallet as `authority`, both CurveState PDAs. Both are `Filled`, so the check passes. Both transition to `Graduated`.
   - **IX 2:** `withdraw_graduated_sol` for CRIME curve ŌĆö passes attacker's wallet as `authority`. Curve is now `Graduated` (from IX 1). All SOL minus rent (~1000 SOL) is transferred to attacker.
   - **IX 3:** `withdraw_graduated_sol` for FRAUD curve ŌĆö same pattern. Another ~1000 SOL to attacker.
3. Transaction is atomic ŌĆö all 3 instructions succeed or none do.
4. Attacker extracts ~2000 SOL in a single transaction. The protocol's AMM pool seeding (Phase 74) receives zero SOL, permanently bricking the post-graduation trading system.

The attacker can monitor on-chain state and submit this transaction the moment the second curve reaches `Filled`, potentially front-running the protocol's own graduation orchestration.

## Root Cause

The bonding curve program (added in v1.2) does not implement any authority verification pattern. The AMM program in the same codebase correctly uses a ProgramData upgrade-authority check (`initialize_admin.rs`) and `has_one = admin` constraints, but this pattern was not replicated in the bonding curve program. The `authority` field is documented as "Protocol authority (deployer). Must sign." but the comment is aspirational ŌĆö there is no enforcement.

All 6 bonding curve admin instructions (`initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault`) share this same broken pattern.

## Recommendation

Implement authority verification matching the AMM's proven pattern. Two options:

**Option A (Preferred) ŌĆö ProgramData upgrade-authority check:**
Add a stored authority to a BondingCurveConfig PDA initialized via ProgramData constraint (same as AMM's `initialize_admin`). All 6 admin instructions add `has_one = authority` against this config.

**Option B (Simpler) ŌĆö Hardcoded address:**
Add `address = DEPLOYER_PUBKEY` constraint to the `authority` field, using a feature-gated constant (devnet vs mainnet). Less flexible but immediately effective.

Either fix must be applied to all 6 instructions: `initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault`.
