# H021 -- Epoch Init Front-Running (H057 Recheck)

**Status:** CONFIRMED -- NOT FIXED (unchanged from H057)
**Severity:** LOW (downgraded from original MEDIUM after parameter analysis)
**Category:** Access Control
**Previous Finding:** H057 (Audit #1)
**Target:** `programs/epoch-program/src/instructions/initialize_epoch_state.rs`

---

## Hypothesis

`initialize_epoch_state` accepts any signer with no authority check. A front-runner could initialize the EpochState PDA with malicious parameters.

## Evidence

The instruction's account struct at lines 96-116:

```rust
pub struct InitializeEpochState<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,

    #[account(
        init,
        payer = payer,
        space = EpochState::LEN,
        seeds = [EPOCH_STATE_SEED],
        bump,
    )]
    pub epoch_state: Account<'info, EpochState>,

    pub system_program: Program<'info, System>,
}
```

The `payer` is a bare `Signer<'info>` with no `address =`, `has_one`, or ProgramData upgrade-authority constraint. Any signer can call this instruction.

## Parameter Analysis (Impact Assessment)

Critical question: what can a front-runner actually control?

**All configuration values are HARDCODED -- zero caller-controlled parameters:**

| Field | Value Source | Caller Controls? |
|-------|-------------|-----------------|
| genesis_slot | `Clock::get().slot` | NO -- runtime-determined |
| current_epoch | `0` (literal) | NO |
| epoch_start_slot | `Clock::get().slot` | NO |
| cheap_side | `0` (literal) | NO |
| low_tax_bps | `GENESIS_LOW_TAX_BPS` (300) | NO |
| high_tax_bps | `GENESIS_HIGH_TAX_BPS` (1400) | NO |
| crime_buy_tax_bps | `GENESIS_LOW_TAX_BPS` (300) | NO |
| crime_sell_tax_bps | `GENESIS_HIGH_TAX_BPS` (1400) | NO |
| fraud_buy_tax_bps | `GENESIS_HIGH_TAX_BPS` (1400) | NO |
| fraud_sell_tax_bps | `GENESIS_LOW_TAX_BPS` (300) | NO |
| All VRF fields | Literals (0, false, true, default) | NO |
| All Carnage fields | Literals (false, 0) | NO |
| initialized | `true` | NO |
| bump | `ctx.bumps.epoch_state` | NO |

The handler (lines 32-93) sets every field from constants or Clock. The payer's identity is not stored anywhere in EpochState -- there is no authority field, no admin field, no stored payer.

## Actual Impact of Front-Running

If an attacker front-runs this instruction:

1. **Genesis slot shifts**: The `genesis_slot` would be set to the attacker's transaction slot rather than the deployer's intended slot. Since epoch boundaries are calculated as `(current_slot - genesis_slot) / SLOTS_PER_EPOCH`, this shifts epoch boundaries by however many slots early the front-run lands. In practice this is a few slots at most -- negligible impact on a ~216,000-slot epoch duration.

2. **Attacker pays rent**: The front-runner pays ~0.001 SOL rent for the PDA. No economic harm to the protocol.

3. **PDA is locked**: Anchor's `init` constraint ensures the PDA can only be created once. The `initialized` flag provides defense-in-depth. Re-initialization is impossible.

4. **No stored authority**: Unlike the Transfer Hook's `initialize_authority` (where the first caller becomes the permanent authority), EpochState stores no authority. The front-runner gains zero ongoing privileges.

## Comparison to AMM Pattern

The AMM's `initialize_admin` uses ProgramData upgrade-authority verification:
```rust
constraint = program_data.upgrade_authority_address == Some(authority.key())
```

This pattern could be applied to `initialize_epoch_state` for defense-in-depth, but the actual risk is minimal because no caller-controlled data enters the account.

## Verdict

**CONFIRMED but LOW severity.** The access control gap is real -- any signer can call this instruction. However, the practical impact is negligible because:

1. All parameters are hardcoded constants or runtime-derived values
2. No authority/admin is stored (no privilege capture)
3. The only observable effect is a minor genesis_slot shift (a few slots)
4. PDA uniqueness prevents re-initialization

This is a defense-in-depth concern, not an exploitable vulnerability. Adding ProgramData gating would be best practice but is not security-critical.

## Recommendation

**Optional hardening** (not blocking for mainnet):

Add ProgramData upgrade-authority verification, matching the AMM pattern:
```rust
pub struct InitializeEpochState<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,

    #[account(
        constraint = program_data.upgrade_authority_address == Some(payer.key())
            @ EpochError::Unauthorized
    )]
    pub program_data: Account<'info, ProgramData>,

    #[account(
        init, payer = payer, space = EpochState::LEN,
        seeds = [EPOCH_STATE_SEED], bump,
    )]
    pub epoch_state: Account<'info, EpochState>,

    pub system_program: Program<'info, System>,
}
```

This aligns with the principle of least privilege but does not address a concrete exploit path.
