# H054 -- AMM k-Invariant Verification Bypass via Overflow

## Hypothesis

Can an attacker craft swap amounts where u128 intermediate overflow causes `k_after` to appear >= `k_before`, bypassing the constant-product invariant and draining pool value?

## Analysis

### Full Computation Trace

The swap path in `swap_sol_pool.rs` (lines 125-173) computes:

1. **Fee deduction** (`math.rs:36-40`): `effective_input = amount_in * (10000 - fee_bps) / 10000` in u128
2. **Output calculation** (`math.rs:58-76`): `amount_out = reserve_out * effective_input / (reserve_in + effective_input)` in u128, cast to u64 via `try_from`
3. **New reserves** (`swap_sol_pool.rs:163-168`): `new_reserve_in = reserve_in + amount_in` (u64 checked_add), `new_reserve_out = reserve_out - amount_out` (u64 checked_sub)
4. **k-invariant check** (`math.rs:92-103`): `(new_reserve_in as u128) * (new_reserve_out as u128) >= (reserve_in as u128) * (reserve_out as u128)` via checked_mul

### Overflow Analysis at Each Stage

**Stage 1 -- Fee deduction:**
- Worst case: `u64::MAX * 10000 = 1.84e23` -- fits in u128 (max 3.4e38).
- Division by 10000 reduces the result. Maximum `effective_input` is `u64::MAX` (when fee_bps=0).
- `checked_sub` on `10000 - fee_bps` prevents underflow if fee_bps > 10000.
- No overflow possible.

**Stage 2 -- Output calculation:**
- Numerator: `reserve_out as u128 * effective_input`. Worst case: `u64::MAX * u64::MAX = (2^64-1)^2 = 2^128 - 2^65 + 1 < u128::MAX`. The `checked_mul` cannot return None for any two u64-range values.
- Denominator: `reserve_in as u128 + effective_input`. Worst case: `u64::MAX + u64::MAX = 2^65 - 2`. Fits trivially in u128.
- Division result is at most `reserve_out` (mathematical property of `x*y/(z+y)` where all positive). The `u64::try_from` guard on the result will always succeed for valid inputs since output <= reserve_out which is u64.
- No overflow possible.

**Stage 3 -- Reserve update:**
- `checked_add` on `reserve_in + amount_in` returns `AmmError::Overflow` if the sum exceeds u64::MAX. This is the only place where overflow could theoretically occur, and it is correctly caught -- the transaction aborts.
- `checked_sub` on `reserve_out - amount_out` returns `AmmError::Overflow` if output exceeds reserve. Since `amount_out < reserve_out` by the constant-product formula property, this cannot trigger for valid swaps.
- Both `new_reserve_in` and `new_reserve_out` are guaranteed u64.

**Stage 4 -- k-invariant check:**
- As proven in H043: `u64 * u64` always fits in u128. The `checked_mul` is defense-in-depth and can never return None.
- No overflow possible.

### Mathematical Proof: k_after >= k_before Always Holds

Let `R_in`, `R_out` be reserves before swap. Let `a` = `amount_in`, `e` = `effective_input` where `e <= a` (fee deducted). Let `out` = `floor(R_out * e / (R_in + e))`.

```
k_before = R_in * R_out

k_after  = (R_in + a) * (R_out - out)
```

Since `out <= R_out * e / (R_in + e)` (floor rounds down):

```
R_out - out >= R_out - R_out * e / (R_in + e)
            = R_out * R_in / (R_in + e)
```

Therefore:

```
k_after >= (R_in + a) * R_out * R_in / (R_in + e)
```

Since `a >= e` (fee is non-negative), `R_in + a >= R_in + e`, so:

```
k_after >= (R_in + e) * R_out * R_in / (R_in + e)
        = R_in * R_out
        = k_before
```

Equality holds only when `a = e` (zero fee) AND floor division produces the exact real-valued result (no truncation). In practice, either the fee or the floor division (or both) cause `k_after > k_before` strictly.

### Attack Vector Evaluation

**Vector 1 -- Force u128 overflow in k_after multiplication:** Impossible. Both `new_reserve_in` and `new_reserve_out` are u64 (enforced by checked_add/checked_sub). Their product in u128 cannot overflow (proven in H043).

**Vector 2 -- Force u128 overflow in swap output calculation to get inflated amount_out:** Impossible. The numerator `reserve_out * effective_input` fits in u128 (worst case u64::MAX squared), and the `u64::try_from` guard rejects any result exceeding u64::MAX. Even if the guard were absent, the constant-product formula mathematically guarantees `output < reserve_out`.

**Vector 3 -- Use fee_bps to create negative effective_input:** The `checked_sub(fee_bps as u128)` on the fee factor returns None for fee_bps > 10000, propagated as `AmmError::Overflow`. No path to negative values.

**Vector 4 -- new_reserve_in overflow wraps around to small value, making k_after small but still >= k_before:** The `checked_add` at line 163 catches u64 overflow and aborts the transaction. No wrapping occurs.

**Vector 5 -- Extreme reserve ratio (one near zero) creates precision loss:** When `reserve_in` is very large and `reserve_out` is very small (or vice versa), the floor division in output calculation rounds toward zero, which makes `k_after` even larger relative to `k_before`. No exploitation path.

### Prior Art

H043 established that u128 intermediates in `verify_k_invariant` cannot overflow for u64 inputs. H092 (from audit #1) reached the same conclusion. This analysis extends H043 by examining the entire computation chain from `amount_in` through to the k-invariant check, confirming no overflow or precision-loss vector exists at any stage.

The 10,000-iteration proptest at `math.rs:391-446` exercises random reserve values (including u64::MAX and u64::MAX/2 edge cases) and has never found a k-invariant violation.

## Verdict

**FALSE POSITIVE.** The k-invariant verification cannot be bypassed via overflow at any stage of the swap computation:

1. All u128 intermediates are mathematically bounded below u128::MAX for u64-range inputs.
2. All u64 additions/subtractions use checked arithmetic that aborts on overflow.
3. The `u64::try_from` guard on swap output prevents inflated results.
4. The mathematical relationship between pre-fee input (added to reserves) and post-fee input (used for output calculation) guarantees `k_after >= k_before` for any non-negative fee.
5. Floor division on output only increases k_after further.

No crafted swap amount can cause k_after to falsely appear >= k_before.
