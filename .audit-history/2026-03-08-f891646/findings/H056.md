# H056 — Consume Randomness Edge Cases (H090 Recheck)

## Summary

The auto-expire mechanism added to `consume_randomness` (lines 112-148) correctly handles stale Carnage state at the expiry boundary. No off-by-one errors exist. All four instructions that interact with the Carnage deadline use consistent boundary semantics.

## Analysis

### Boundary Operator Consistency

All instructions use the same operator (`>`) for deadline expiration:

| Instruction | Check | Operator | Meaning |
|---|---|---|---|
| `consume_randomness:115` | Auto-expire guard | `clock.slot > deadline` | Expires after deadline slot |
| `expire_carnage:69` | Expire guard | `clock.slot > deadline` | Expires after deadline slot |
| `execute_carnage:215` | Deadline validity | `clock.slot <= deadline` | Executes at or before deadline |
| `execute_carnage:224` | Lock validity | `clock.slot > lock_slot` | Fallback after lock expires |
| `execute_carnage_atomic` | No deadline check | N/A | No-op guard on `carnage_pending` flag |

At the exact deadline slot:
- Fallback `execute_carnage` CAN execute (`<=` passes)
- `expire_carnage` CANNOT execute (`>` fails)
- Auto-expire in `consume_randomness` CANNOT trigger (`>` fails)

At deadline + 1:
- Fallback `execute_carnage` CANNOT execute (`<=` fails)
- `expire_carnage` CAN execute (`>` passes)
- Auto-expire CAN trigger (`>` passes)

This is correct and gap-free. There is no slot where neither execution nor expiry is possible, and no slot where both are possible.

### State Cleanup After Expiry

Both auto-expire (consume_randomness:126-127) and expire_carnage (lines 88-89) set:
- `carnage_pending = false`
- `carnage_action = CarnageAction::None`

Neither resets `carnage_deadline_slot` to 0. This is safe because all code paths gate on `carnage_pending` before reading `carnage_deadline_slot`.

### Race Condition: expire_carnage vs auto-expire

If both are attempted in the same slot (after deadline):
- If `expire_carnage` executes first: clears `carnage_pending`. `consume_randomness` sees `carnage_pending = false`, skips auto-expire block.
- If `consume_randomness` executes first: auto-expire clears `carnage_pending`. `expire_carnage` fails with `NoCarnagePending` (Anchor constraint at line 33). No double-clear.

### carnage_deadline_slot = 0 Edge Case

At initialization, `carnage_deadline_slot = 0` and `carnage_pending = false`. The auto-expire guard (`carnage_pending && clock.slot > 0`) would be true for any non-zero slot, BUT `carnage_pending = false` short-circuits the check. `carnage_pending` can only become true inside `consume_randomness` itself (line 277), which simultaneously sets a valid deadline. No uninitialized-deadline exploit is possible.

### Atomic Path Has No Deadline Check

`execute_carnage_atomic` uses a no-op guard (`if !carnage_pending { return Ok(()) }`) rather than a deadline check. This is by design — the atomic path is bundled in the same TX as `consume_randomness`, so it executes immediately after Carnage is set pending. Deadline enforcement is only needed for the fallback path where time has elapsed.

### u64 Overflow in Deadline Calculation

`consume_randomness:280-283` uses `checked_add(CARNAGE_DEADLINE_SLOTS)` which returns `Err(Overflow)` if the slot value is near `u64::MAX`. This is correct — no silent overflow.

## Verdict

The auto-expire mechanism introduced to address H090 is correctly implemented with no off-by-one errors, no race conditions, and no boundary gaps. All Carnage deadline checks across the four instructions are consistent and properly aligned.

**No action required.**
