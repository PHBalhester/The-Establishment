# H067 -- Bonding Curve distribute_tax_escrow: One-Shot Guard and Multi-Call Analysis

**Status:** SAFE (Informational observation)
**Severity:** Informational
**Component:** `programs/bonding_curve/src/instructions/distribute_tax_escrow.rs`
**Related:** `consolidate_for_refund.rs`, `sell.rs` (tax accumulation), `state.rs` (CurveState)

---

## Hypothesis

`distribute_tax_escrow` is permissionless after graduation. Can it be called multiple times? Is the escrow amount calculation correct?

## Analysis

### 1. Access Control (Permissionless, State-Gated)

The instruction has no signer authority check. Access is gated solely by:
- `curve_state.status == CurveStatus::Graduated` (line 74-77)
- `transferable > 0` (line 86) -- economic guard against no-op calls

This is correct for a permissionless post-graduation cleanup instruction. Anyone can trigger it, which is desirable since the protocol should not depend on a specific party to move escrowed funds to the carnage fund.

### 2. One-Shot Execution Guard

**There is no persistent boolean flag** (unlike `consolidate_for_refund` which sets `escrow_consolidated = true`). The CurveState account is read-only in this instruction (`seeds`, `bump` constraints only, no `mut`).

The "one-shot" behavior is enforced economically:
1. First call: `transferable = escrow_lamports - rent_exempt` (positive if sells occurred during Active phase)
2. Lamports are moved: escrow drops to exactly `rent_exempt`
3. Second call: `transferable = rent_exempt - rent_exempt = 0` via `saturating_sub`
4. `require!(transferable > 0, CurveError::EscrowAlreadyDistributed)` rejects

**Verdict: Effectively one-shot under normal conditions.** The economic guard is sufficient because:
- No instruction in the Graduated state can add lamports to the tax escrow PDA
- The sell instruction only operates in Active state (line 18 of sell.rs: `CurveNotActiveForSell`)
- The escrow PDA is owned by the bonding curve program, so only program instructions can subtract from it, but no other instruction adds to it post-graduation

### 3. Can It Be Called Multiple Times?

**Yes, but only if someone externally sends SOL to the tax_escrow PDA address.** Any account on Solana can receive lamports via system transfer. If someone sends SOL to the escrow after the first distribution:
- `escrow_lamports` would be `rent_exempt + donated_amount`
- `transferable = donated_amount` (positive)
- The instruction would succeed, forwarding the donation to the carnage fund

**This is benign behavior**, not a vulnerability. The donated SOL simply flows to the carnage fund, which is the protocol's buyback-and-burn mechanism. No value is extracted from the protocol; value is added to it.

### 4. Escrow Amount Calculation

```rust
let rent_exempt = rent.minimum_balance(0);       // ~890,880 lamports for 0-byte account
let escrow_lamports = ctx.accounts.tax_escrow.lamports();
let transferable = escrow_lamports.saturating_sub(rent_exempt);
```

**Correct.** The tax_escrow PDA was created with `init, space = 0` in `initialize_curve.rs` (line 60-69), so `minimum_balance(0)` is the right rent calculation. The `saturating_sub` prevents underflow if the escrow somehow has fewer lamports than rent-exempt (which would mean the account is already being garbage-collected -- impossible for a program-owned PDA that still exists).

The ~890,880 lamports (~0.00089 SOL) of dust left in the escrow is acceptable protocol overhead per the inline documentation (line 70-71).

### 5. Mutual Exclusivity with consolidate_for_refund

- `distribute_tax_escrow` requires: `status == Graduated`
- `consolidate_for_refund` requires: `is_refund_eligible()` which needs `status == Failed` OR `status == Filled && partner == Failed`

These status requirements are **mutually exclusive**. A Graduated curve cannot be Failed, and vice versa. The state machine transitions (Section 5.2 of state.rs) confirm these are terminal/near-terminal states with no path between them:
- `Filled -> Graduated` (via finalize_transition)
- `Active -> Failed` (via mark_failed)
- Once Graduated, status never changes

Therefore, both the distribute and consolidate paths cannot operate on the same curve's escrow. No double-drain is possible.

### 6. Account Validation

- **tax_escrow**: Validated by PDA seeds `[TAX_ESCROW_SEED, curve_state.token_mint]` AND `constraint = tax_escrow.key() == curve_state.tax_escrow` (belt-and-suspenders)
- **carnage_fund**: Validated by PDA derivation against `epoch_program_id()` with `CARNAGE_SOL_VAULT_SEED` -- correct cross-program PDA validation
- **curve_state**: Validated by PDA seeds `[CURVE_SEED, token_mint]` with stored bump

All account validation is sound.

### 7. Lamport Manipulation Safety

The instruction subtracts from `tax_escrow` (owned by bonding curve program -- program can modify its own accounts) and adds to `carnage_fund` (owned by epoch program -- any program can credit any account). This follows the same safe pattern documented in the instruction's comments and used in `sell.rs`.

## Minor Observation

The `CurveError::InvalidStatus` error code is reused for both the tax_escrow constraint (line 40-41) and the carnage_fund constraint (line 49-54). If either validation fails, the error message would be the same generic "Invalid curve status for this operation" which could confuse debugging. A dedicated error variant (e.g., `InvalidCarnageFund`) would improve diagnostics. This is cosmetic only.

## Verdict

**SAFE.** The instruction is correctly implemented:
- Economic one-shot guard is sufficient (no state mutation needed)
- Escrow calculation correctly preserves rent-exempt minimum
- Status-based mutual exclusivity with consolidate prevents double-drain
- Account validation is thorough (PDA seeds + stored key + cross-program derivation)
- Permissionless access is appropriate for this cleanup operation
- Re-callability after external SOL donation is benign (feeds carnage fund)
