# H016 -- Cross-Epoch Tax Arbitrage via VRF Observation

| Field            | Value                                                    |
|------------------|----------------------------------------------------------|
| **ID**           | H016                                                     |
| **Title**        | Cross-Epoch Tax Arbitrage via VRF Observation             |
| **Category**     | Oracle, Economic Model, Timing                           |
| **Origin**       | Novel                                                    |
| **Severity**     | LOW (downgraded from MEDIUM-HIGH)                        |
| **Status**       | Mitigated by Design (atomic bundling closes primary window) |
| **Target**       | `programs/epoch-program/src/instructions/consume_randomness.rs`, `programs/tax-program/src/instructions/swap_sol_buy.rs` |

---

## Hypothesis

An attacker monitors the Switchboard VRF reveal transaction, predicts the new tax rates before `consume_randomness` lands on-chain, and executes trades at old (favorable) rates to capture the tax differential (up to 13% per trade).

## Investigation

### 1. VRF Commit-Reveal Transaction Flow

The epoch transition follows a 3-TX sequence:

1. **TX1**: Create Switchboard randomness account (finalized before proceeding)
2. **TX2**: Bundle `Switchboard.commitIx` + `trigger_epoch_transition` -- binds randomness account, sets `vrf_pending=true`, sets `taxes_confirmed=false`
3. **TX3**: Bundle `Switchboard.revealIx` + `consume_randomness` -- reads VRF bytes, derives new tax rates, sets `taxes_confirmed=true`

The critical question: **Can an attacker observe VRF bytes between TX2 and TX3 and front-run with trades at old rates?**

### 2. Switchboard Reveal Mechanics

The Switchboard On-Demand oracle reveals randomness by writing to the randomness account on-chain. The `revealIx` is a Switchboard instruction that the oracle processes after seeing the commitment. The reveal bytes become readable once the oracle writes them.

**Key insight**: In the current implementation (`scripts/vrf/lib/vrf-flow.ts`), `revealIx` and `consume_randomness` are **bundled in the same transaction (TX3)**. This means:

- The VRF bytes are written to the randomness account AND consumed by `consume_randomness` atomically
- No intermediate state exists where bytes are revealed but not yet consumed
- An observer watching the mempool could see TX3 before it lands, but by the time the TX lands, both the reveal and the tax rate update happen in the same slot

### 3. Tax Rate Reading During VRF Pending Window

Between TX2 and TX3 (while `vrf_pending=true`, `taxes_confirmed=false`):

- `swap_sol_buy.rs:67-78` reads EpochState and calls `get_tax_bps()` without checking `taxes_confirmed`
- `swap_sol_sell.rs:72-93` -- same pattern
- Swaps execute with the **previous epoch's rates** during this window

This is confirmed as an **intentional design choice** (documented in `ARCHITECTURE.md` assumption A-10): the protocol prioritizes liveness over freshness.

### 4. Attack Vector Analysis

**Primary Vector (CLOSED):** Observe VRF reveal bytes before consume_randomness

The bundling of `revealIx + consume_randomness` (+ optionally `executeCarnageAtomic`) in a single TX3 means there is no intermediate on-chain state where bytes are revealed but rates are not updated. The reveal and consumption are atomic.

A mempool observer could see TX3 and attempt to front-run it (landing their swap in the same block but before TX3). However:

- The attacker's swap would execute at **old rates** (which they already know)
- TX3 then updates rates to new values
- The attacker would need to **back-run** TX3 with a second swap at new rates
- This is a standard sandwich, not a VRF-specific exploit

**Secondary Vector (NARROW):** Trade during VRF pending window

Between TX2 (trigger) and TX3 (consume), typically ~3 slots (~1.2 seconds):

- Old rates still apply
- An attacker who is also the crank operator knows VRF is pending
- But the attacker does NOT know the new rates yet (VRF bytes are unrevealed)
- No information asymmetry exists during this window

**Tertiary Vector (THEORETICAL):** Crank operator advantage

The crank operator who builds TX3 knows the VRF result from the Switchboard SDK's `revealIx()` call (which queries the oracle gateway). The operator could:

1. Call `revealIx()` to learn the bytes
2. Compute the new rates from bytes 0-4 using the `derive_taxes` algorithm
3. Submit a swap at old rates if favorable, then submit TX3
4. Maximum value: up to 13% tax differential per trade (e.g., old rate 14%, new rate 1%)

However:
- The crank operator must bundle `revealIx + consume_randomness` in the same TX
- Inserting a personal swap before TX3 in the same block requires validator-level ordering control (Jito bundles or similar)
- The 50% output floor still applies to any swap
- Currently a single Railway-hosted crank runner exists; the information advantage is real but requires additional MEV infrastructure to exploit

### 5. Carnage Bundling Eliminates the Largest MEV Window

The `vrf-flow.ts` implementation bundles `executeCarnageAtomic` in TX3 as well. This closes the CARN-002 MEV window: no `CarnagePending` event is visible on-chain before the Carnage swap executes. The on-chain no-op guard makes this safe when Carnage doesn't trigger.

## Verdict

**LOW severity.** The primary attack vector (observing VRF bytes between reveal and consume) is **closed by atomic TX bundling**. The secondary vector (crank operator information advantage) is real but requires:

1. Running the crank (currently monopolistic -- single Railway instance)
2. MEV infrastructure to insert personal swaps before TX3 in the same block
3. Capital to execute meaningful trades in the ~1.2s VRF pending window

The maximum theoretical extraction is 13% of trade value per epoch transition, but practical extraction requires sophisticated MEV infrastructure and is bounded by the 50% output floor on all swaps.

### Why Downgraded from MEDIUM-HIGH

The original hypothesis assumed VRF bytes would be observable in a separate on-chain state before `consume_randomness` processes them. The actual implementation bundles reveal + consume atomically, eliminating the primary attack window. The residual risk is the crank operator's off-chain information advantage, which is a much narrower vector.

## Mitigating Factors

1. **Atomic TX3 bundling**: `revealIx + consume_randomness` in same transaction -- no intermediate on-chain state
2. **Carnage atomic bundling**: `executeCarnageAtomic` also in TX3 -- no `CarnagePending` visible before execution
3. **50% output floor**: Limits maximum extraction from any single swap
4. **Discrete tax rates**: Only 8 possible values (100-400, 1100-1400 bps) -- limited surprise range
5. **Short VRF pending window**: Typically ~3 slots (~1.2 seconds) between TX2 and TX3

## Residual Risk

The crank operator has a temporal information advantage: they learn VRF bytes off-chain (via Switchboard SDK) before TX3 lands. If the operator has Jito bundle access, they could theoretically insert favorable swaps ahead of TX3. This is an inherent property of any commit-reveal VRF system where the committer also has a financial interest.

**Recommendation**: If crank operator MEV becomes a concern at mainnet scale, consider:
- Competitive crank operation (multiple operators racing reduces any single operator's advantage)
- Higher bounty to attract competing operators
- Time-lock between trigger and first allowed swap in the new epoch (would require Tax Program checking a slot-based window)
