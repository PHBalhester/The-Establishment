# H007: Transfer Hook Init Front-Running: Authority Ransom

**Status:** CONFIRMED
**Severity:** CRITICAL
**Category:** Access Control
**Confidence:** HIGH (code-verified, recheck of S005 from audit #1)

---

## Summary

The Transfer Hook program's `initialize_authority` instruction accepts any signer as the whitelist authority with zero verification against the program's upgrade authority. An attacker who front-runs this instruction during deployment captures permanent, irrevocable control of the whitelist. Since ALL CRIME/FRAUD token transfers require at least one whitelisted party, the attacker can hold the entire protocol hostage (refuse to whitelist required accounts) or permanently brick it (call `burn_authority` with an incomplete whitelist). This finding was reported as S005 in audit #1 and remains **NOT FIXED**.

---

## Evidence

### 1. Vulnerable `initialize_authority` -- No Authority Validation

**File:** `programs/transfer-hook/src/instructions/initialize_authority.rs:15-46`

```rust
pub fn handler(ctx: Context<InitializeAuthority>) -> Result<()> {
    let auth = &mut ctx.accounts.whitelist_authority;
    auth.authority = Some(ctx.accounts.signer.key()); // Any signer becomes authority
    auth.initialized = true;
    // ...
}

#[derive(Accounts)]
pub struct InitializeAuthority<'info> {
    #[account(mut)]
    pub signer: Signer<'info>,  // NO upgrade authority check

    #[account(
        init,
        payer = signer,
        space = 8 + WhitelistAuthority::INIT_SPACE,
        seeds = [WhitelistAuthority::SEED],
        bump
    )]
    pub whitelist_authority: Account<'info, WhitelistAuthority>,

    pub system_program: Program<'info, System>,
}
```

The account struct requires only: (1) a signer, (2) the PDA to init, (3) system program. There is no `program` or `program_data` account, and therefore no constraint tying the signer to the program's upgrade authority.

### 2. Secure Pattern Exists in Same Codebase (AMM)

**File:** `programs/amm/src/instructions/initialize_admin.rs:27-59`

The AMM program correctly gates its initialization against the upgrade authority:

```rust
#[account(
    constraint = program.programdata_address()? == Some(program_data.key())
)]
pub program: Program<'info, crate::program::Amm>,

#[account(
    constraint = program_data.upgrade_authority_address == Some(authority.key())
)]
pub program_data: Account<'info, ProgramData>,
```

This pattern was not replicated in the Transfer Hook program.

### 3. WhitelistAuthority State Is Write-Once, Burn-Only

**File:** `programs/transfer-hook/src/state/whitelist_authority.rs:14-20`

```rust
pub struct WhitelistAuthority {
    pub authority: Option<Pubkey>,  // Some(key) or None (burned)
    pub initialized: bool,
}
```

The `authority` field transitions only: `None` (uninit) -> `Some(attacker_key)` (init) -> `None` (burned). There is no `update_authority` or `transfer_authority` instruction. Once set, the authority pubkey cannot be changed -- only burned.

### 4. Whitelist Controls All Token Transfers

**File:** `programs/transfer-hook/src/instructions/transfer_hook.rs` (hook handler)

Every CRIME/FRAUD transfer invokes the hook, which requires at least one whitelisted party (source or destination). Without whitelist entries for protocol accounts (AMM pools, staking vaults, carnage fund), all swaps, stakes, and transfers fail with `NoWhitelistedParty`.

### 5. No Recovery Path in Instruction Set

The Transfer Hook program exposes exactly 5 instructions:
- `initialize_authority` -- one-shot, already consumed by attacker
- `add_whitelist_entry` -- requires `authority == Some(signer)`, attacker controls
- `burn_authority` -- makes it worse (freezes whitelist permanently)
- `initialize_extra_account_meta_list` -- unrelated to authority
- `transfer_hook` -- read-only validation, no state changes

Zero instructions exist to override, transfer, or reset the authority.

## Attack Scenario

1. **Monitor**: Attacker watches for Transfer Hook program deployment via RPC subscription or validator access.
2. **Front-run**: Attacker submits `initialize_authority` with their wallet as signer, using higher priority fee (Jito tip). Cost: ~0.002 SOL rent + priority fee.
3. **Anchor `init` blocks deployer**: Deployer's `initialize_authority` TX fails ("account already initialized"). The WhitelistAuthority PDA stores `authority = Some(attacker_pubkey)`.
4. **Protocol DOA**: No protocol accounts are whitelisted. All CRIME/FRAUD transfers fail at the hook layer. Swaps, staking, carnage, conversions -- all broken.
5. **Ransom or brick**: Attacker demands payment to whitelist accounts, OR calls `burn_authority` to permanently freeze the whitelist in its empty state.
6. **No on-chain recovery**: Protocol must redeploy the Transfer Hook program, create new token mints (old mints reference old hook program ID), rebuild all 6 other programs with new mint addresses, and migrate all user state.

---

## Root Cause

The `InitializeAuthority` account struct lacks the two accounts needed to verify deployer identity:

1. `program: Program<'info, TransferHook>` -- to resolve the programdata address
2. `program_data: Account<'info, ProgramData>` -- to check `upgrade_authority_address == Some(signer.key())`

This is a first-caller-wins pattern where the first caller's identity is not validated. The AMM program in the same codebase demonstrates the correct pattern, but it was not applied to the Transfer Hook.

---

## Recommendation

### Primary Fix: Add ProgramData Upgrade Authority Gate

Add `program` and `program_data` accounts to `InitializeAuthority` with the same constraint pattern used in `amm/initialize_admin.rs`:

```rust
#[derive(Accounts)]
pub struct InitializeAuthority<'info> {
    #[account(mut)]
    pub signer: Signer<'info>,

    #[account(
        init,
        payer = signer,
        space = 8 + WhitelistAuthority::INIT_SPACE,
        seeds = [WhitelistAuthority::SEED],
        bump
    )]
    pub whitelist_authority: Account<'info, WhitelistAuthority>,

    #[account(
        constraint = program.programdata_address()? == Some(program_data.key())
            @ TransferHookError::InvalidProgramData
    )]
    pub program: Program<'info, crate::program::TransferHook>,

    #[account(
        constraint = program_data.upgrade_authority_address == Some(signer.key())
            @ TransferHookError::UnauthorizedInit
    )]
    pub program_data: Account<'info, ProgramData>,

    pub system_program: Program<'info, System>,
}
```

### Defense-in-Depth: Atomic Deployment Bundle

Even after the fix, use Jito bundles to atomically deploy + initialize in a single slot, eliminating any mempool observation window.

### Future Consideration: Emergency Authority Reset

Consider adding an `emergency_reset_authority` instruction gated by `program_data.upgrade_authority_address`, callable only while the program remains upgradeable. This provides a recovery path if the authority is ever compromised.
