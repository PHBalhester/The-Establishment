# H009 -- Pool Reserve Read Without Owner Check (Carnage)

| Field          | Value                                      |
|----------------|--------------------------------------------|
| **Status**     | REFUTED                                    |
| **Severity**   | Informational                              |
| **Category**   | CPI / Oracle                               |
| **Confidence** | High                                       |

## Summary

The hypothesis posits that `execute_carnage_atomic` and `execute_carnage` read pool reserves at raw byte offsets (137-153) without verifying the account is owned by the AMM program, enabling an attacker to pass a fake AccountInfo with inflated reserves to defeat the slippage floor while the real swap executes against the actual pool.

**This attack is not viable.** The same `AccountInfo` reference is used for both the slippage reserve read and the CPI swap. An attacker cannot provide a different account for each operation. If a fake pool is passed, the AMM rejects it during CPI deserialization; if the real pool is passed, the slippage floor reads correct reserves.

## Evidence

### Same Account Used for Both Operations

In `execute_carnage_atomic.rs` (lines 372-396), the target pool is selected once and used for both the reserve read and the swap CPI:

```rust
// Line 372-375: Select pool for slippage read
let target_pool_info = match target {
    Token::Crime => &ctx.accounts.crime_pool,
    Token::Fraud => &ctx.accounts.fraud_pool,
};
let (reserve_sol, reserve_token) = read_pool_reserves(target_pool_info, ...)?;

// Line 396: Same pool flows into execute_buy_swap -> execute_swap_exempt_cpi
execute_buy_swap(&ctx, total_buy_amount, target, ...)?;
```

`execute_buy_swap` (lines 775-788) selects the same `ctx.accounts.crime_pool` or `ctx.accounts.fraud_pool` and passes it to `execute_swap_exempt_cpi`. The pool reference originates from the same `ctx.accounts` field in both paths.

### AMM Validates Pool During CPI

The AMM's `SwapSolPool` struct (`programs/amm/src/instructions/swap_sol_pool.rs:376-383`) uses Anchor's `Account<'info, PoolState>` type, which enforces:
1. **Owner check**: `pool.owner == AMM_PROGRAM_ID` (automatic Anchor deserialization)
2. **PDA seeds check**: `seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()]`
3. **Bump check**: `bump = pool.bump`
4. **Initialized check**: `constraint = pool.initialized`
5. **Reentrancy guard**: `constraint = !pool.locked`

A fake account would fail step 1 (wrong owner) or step 2 (wrong PDA derivation).

### Identical Pattern in Fallback Path

`execute_carnage.rs` (lines 380-404) follows the same pattern -- same `ctx.accounts.crime_pool`/`fraud_pool` feeds both `read_pool_reserves` and `execute_buy_swap`.

## Attack Scenario (Why It Fails)

1. Attacker constructs a fake account with inflated reserves at byte offsets 137-153.
2. Attacker passes it as `crime_pool` in the `execute_carnage` instruction.
3. `read_pool_reserves` reads inflated values, computing a low slippage floor.
4. `execute_swap_exempt_cpi` passes the same fake account to Tax -> AMM.
5. AMM's Anchor deserialization rejects the fake account (`Account<PoolState>` owner check fails).
6. **CPI reverts. Entire transaction fails.**

The two-account attack (fake for slippage, real for swap) is structurally impossible because there is only one `crime_pool` and one `fraud_pool` field in the accounts struct.

## Root Cause (of the Observation, Not a Vulnerability)

`read_pool_reserves` in both `execute_carnage_atomic.rs:930` and `execute_carnage.rs:863` does not perform an explicit owner check. This is a defense-in-depth gap -- the function trusts its caller to provide a valid pool. In the current codebase this trust is justified because:
- The account is a named field in the Anchor struct (not from `remaining_accounts`)
- The same account is validated by AMM during CPI
- There is no code path where a different account could be substituted between read and swap

## Recommendation

**Informational only -- no code change required.**

For defense-in-depth, consider adding an owner check to `read_pool_reserves`:

```rust
require!(
    pool_info.owner == &amm_program_id(),
    EpochError::InvalidCarnageTargetPool
);
```

This would catch misuse if the function is ever called with an unvalidated account in the future, and costs minimal compute (~200 CU). The same applies to `programs/tax-program/src/helpers/pool_reader.rs:39`.

However, this is a hardening suggestion, not a vulnerability fix. The current code is safe against the hypothesized attack.
