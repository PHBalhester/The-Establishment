# H121 -- Conversion Vault: VaultConfig Bump-Only

## Status: VERIFIED CORRECT (By Design)

## Target
`/Users/mlbob/Projects/Dr Fraudsworth/programs/conversion-vault/src/state.rs`

## Hypothesis
VaultConfig stores only a bump byte. Verify this is intentional and the bump is used correctly for PDA signing.

## Evidence

### VaultConfig Structure (state.rs)
```rust
#[account]
pub struct VaultConfig {
    pub bump: u8,
    #[cfg(feature = "localnet")]
    pub crime_mint: Pubkey,
    #[cfg(feature = "localnet")]
    pub fraud_mint: Pubkey,
    #[cfg(feature = "localnet")]
    pub profit_mint: Pubkey,
}
```

**Production (non-localnet): 9 bytes total** (8 discriminator + 1 bump).
**Localnet: 105 bytes total** (8 + 1 + 32*3) to store mint addresses for test flexibility.

### Why Bump-Only is Intentional

1. **No authority needed.** VaultConfig has no stored admin. It falls under Trust Tier 6 ("No Authority") per ARCHITECTURE.md Section 2 line 49. There is no pause, no admin override, no upgradeable parameters. The vault is fully autonomous after initialization.

2. **No conversion rate stored.** The 100:1 rate is a compile-time constant `CONVERSION_RATE` in constants.rs. Storing it on-chain would add unnecessary deserialization cost and create an attack surface (mutable rate).

3. **Mint addresses are compile-time constants.** Production builds hardcode mint addresses via `constants::crime_mint()`, `constants::fraud_mint()`, `constants::profit_mint()`. The `#[cfg(feature = "localnet")]` fields exist solely for integration tests with random mints.

4. **Vault token accounts use VaultConfig as authority.** The vault's CRIME/FRAUD/PROFIT token accounts are initialized with `token::authority = vault_config` (initialize.rs lines 29, 39, 49). The bump is required to sign CPI transfers from these accounts.

### Bump Usage Verification

**Initialization (initialize.rs line 87):**
```rust
vault_config.bump = ctx.bumps.vault_config;
```
Correctly stores the canonical bump from `#[account(init, seeds = [VAULT_CONFIG_SEED], bump)]`.

**PDA re-derivation (convert.rs lines 16-19):**
```rust
#[account(
    seeds = [VAULT_CONFIG_SEED],
    bump = vault_config.bump,
)]
pub vault_config: Account<'info, VaultConfig>,
```
Uses stored bump for efficient re-derivation (avoids `find_program_address` loop).

**CPI signing (convert.rs lines 158-159):**
```rust
let vault_bump = ctx.accounts.vault_config.bump;
let signer_seeds: &[&[&[u8]]] = &[&[VAULT_CONFIG_SEED, &[vault_bump]]];
```
Correctly constructs signer seeds using the stored bump. Used on line 161 for `transfer_t22_checked` (vault -> user output transfer). The vault_config PDA signs as the token account authority.

### Security Properties
- **One-shot init**: `#[account(init)]` prevents re-initialization (Anchor enforces discriminator check).
- **No stored admin**: Cannot be paused or hijacked. Only program upgrade can modify behavior.
- **Mint validation**: Production builds validate mints against hardcoded constants at init time (initialize.rs lines 63, 70, 77), preventing vault creation for unauthorized token pairs.

## Verdict
**Bump-only VaultConfig is correct and intentional.** The bump is the only runtime-needed datum; everything else (conversion rate, mint addresses, authority model) is either a compile-time constant or enforced structurally via PDA derivation. The bump is stored correctly at init, used correctly for PDA validation, and used correctly for CPI signing in the convert instruction. No issues found.
