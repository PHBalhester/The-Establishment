# H011: EpochState Cross-Program Layout Corruption Risk

**Status:** CONFIRMED (latent -- no active corruption, but no compile-time safety net)
**Severity:** MEDIUM (latent risk; would be HIGH/CRITICAL if layout drift occurs)
**Category:** CPI, State Machine, Arithmetic
**Confidence:** HIGH

## Summary

The Tax Program maintains a read-only mirror of the Epoch Program's `EpochState` struct in `epoch_state_reader.rs`. This mirror must match the canonical layout byte-for-byte for Borsh deserialization to produce correct values. **Currently the layouts match exactly** (20 fields, 100 bytes data, identical field order and types). However, there is **no compile-time enforcement** linking the two structs across crate boundaries. Any future modification to EpochState in the Epoch Program that is not identically replicated in the Tax Program's mirror will silently produce corrupt tax rates.

Additionally, `pool_reader.rs` in the Tax Program and `read_pool_reserves()` in the Epoch Program both read AMM PoolState reserves at hardcoded byte offsets [137..145] and [145..153]. These offsets are currently correct but face the same manual-synchronization risk.

### Safety Measures Present

1. **Owner check**: Tax Program validates `epoch_state.owner == epoch_program_id()` before deserialization (`swap_sol_buy.rs:60-63`, `swap_sol_sell.rs` equivalent). Prevents attacker-crafted fake accounts.
2. **Discriminator check**: `EpochState::try_deserialize()` validates `sha256("account:EpochState")[0..8]`. Prevents type confusion.
3. **Static size assertion in Epoch Program**: `const _: () = assert!(EpochState::DATA_LEN == 100);` at `epoch_state.rs:172`.
4. **Test-level size assertion**: `test_swap_sol_buy.rs:224` and `test_swap_sol_sell.rs:232` both assert `data.len() == 108`.

### Safety Measures ABSENT

1. **No `DATA_LEN` constant or static assertion in the Tax Program mirror.** The canonical EpochState has `DATA_LEN = 100` with a compile-time assertion. The mirror has `LEN = 108` but no equivalent `DATA_LEN` breakdown or `const _: ()` assertion.
2. **No cross-crate compile-time check.** The two structs live in separate Cargo crates with no shared dependency. A field added to one but not the other compiles cleanly.
3. **No runtime size validation beyond Anchor's discriminator+length check.** Anchor's `try_deserialize` will fail if the account data is *shorter* than expected but will silently succeed if data is *longer* (extra bytes ignored). This means: if Epoch Program adds a field and reallocates, the Tax mirror (still 100 bytes) would deserialize successfully using only the first 100 bytes -- correct as long as no existing fields shifted.
4. **No version field** in EpochState to detect schema mismatches.

## Attack Scenario

**Scenario: Silent Tax Rate Corruption After Layout Change**

1. Developer adds a new field to EpochState in the Epoch Program (e.g., `pause_flag: bool` inserted between `high_tax_bps` and `crime_buy_tax_bps`).
2. Epoch Program is redeployed. On-chain account is reallocated to 101+ bytes.
3. Tax Program mirror is NOT updated (oversight, or deployed asynchronously).
4. Tax Program deserializes EpochState. Since the mirror expects `crime_buy_tax_bps` at byte offset 27 but the canonical struct now has it at offset 28, the Tax Program reads misaligned bytes.
5. Tax rates become garbage values. For example, a 1-byte shift could turn a 300 BPS (3%) tax into 76800+ BPS, causing `calculate_tax()` to return `None` (overflow) or an astronomically high tax -- both resulting in either transaction failure or user fund loss.
6. Alternatively, tax could shift to near-zero, allowing untaxed swaps that drain staking rewards and carnage fund accumulation.

**Likelihood of this exact scenario**: LOW for current development (single team controls both programs). MEDIUM-HIGH over protocol lifetime (team changes, emergency patches, V2 migrations).

**Historical precedent**: The project's own `VRF_Migration_Lessons.md` documents this exact failure mode from V3 (EpochState grew from 50 to 82 bytes, breaking deserialization).

## Root Cause

1. **No shared crate**: The two EpochState definitions live in separate Cargo workspaces with no compile-time link.
2. **Borsh serialization is position-dependent**: Unlike protobuf (field tags) or JSON (field names), Borsh serializes fields in declaration order with no framing. Any change to field order, insertion, or type size silently corrupts all subsequent fields.
3. **Anchor's `#[account]` macro generates identical discriminators for identically-named structs** (`sha256("account:EpochState")[0..8]`), but this only validates the struct *name*, not its layout.

## Recommendation

### R1: Add Cross-Program Layout Test (Immediate, Low Cost)

Create a workspace-level integration test that serializes a known EpochState in the Epoch Program and deserializes it using the Tax Program's mirror. This catches layout drift at `cargo test` time:

```rust
// tests/cross_program/epoch_state_layout.rs
#[test]
fn epoch_state_layouts_match() {
    let canonical = epoch_program::state::EpochState { /* all fields */ };
    let bytes = canonical.try_serialize(&mut vec![]).unwrap();
    let mirror = tax_program::state::EpochState::try_deserialize(&mut &bytes[..]).unwrap();
    assert_eq!(canonical.crime_buy_tax_bps, mirror.crime_buy_tax_bps);
    // ... assert all fields
}
```

### R2: Add DATA_LEN Static Assertion to Tax Mirror (Immediate)

Add to `epoch_state_reader.rs`:
```rust
pub const DATA_LEN: usize = 8 + 4 + 8 + 1 + 2 + 2 + 2 + 2 + 2 + 2 + 8 + 1 + 1 + 32 + 1 + 1 + 1 + 8 + 8 + 4 + 1 + 1;
const _: () = assert!(EpochState::DATA_LEN == 100);
```

This doesn't prevent field reordering but catches size mismatches at compile time.

### R3: Add Reserved Padding (Pre-Mainnet)

Per H060 recommendation: add `_reserved: [u8; 32]` to both EpochState definitions. This eliminates the need for reallocation on future field additions and reduces the window where layout drift can occur.

### R4: PoolState Offset Constant Sharing (Long-term)

Extract PoolState byte offsets into a shared constants crate (`protocol-constants`) that both Tax Program and Epoch Program depend on. This makes offset changes a compile-time error across consumers.
