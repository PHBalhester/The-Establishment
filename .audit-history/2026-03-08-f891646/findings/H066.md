# H066 -- Tax Rate Lookup Table Bounds

| Field | Value |
|-------|-------|
| **Hypothesis** | VRF byte % 4 selects index into tax rate lookup tables. Tables may have fewer/more than 4 entries, or values outside [100-1400] BPS, or indexing may go out-of-bounds. |
| **Severity** | N/A |
| **Status** | **REFUTED** -- tables are correctly sized, values are bounded, indexing is safe. |
| **Target** | `programs/epoch-program/src/helpers/tax_derivation.rs:23-26, 89-94` |
| **Secondary** | `programs/epoch-program/src/constants.rs:99-103` (genesis values) |

## Evidence

### Lookup Table Definitions (tax_derivation.rs:23-26)

```rust
const LOW_RATES: [u16; 4] = [100, 200, 300, 400];
const HIGH_RATES: [u16; 4] = [1100, 1200, 1300, 1400];
```

Both arrays have exactly 4 entries. Type annotation `[u16; 4]` enforces this at compile time -- adding or removing an element is a compilation error.

### Index Computation (tax_derivation.rs:89-94)

```rust
let crime_low_bps = LOW_RATES[(vrf_result[1] % 4) as usize];
let crime_high_bps = HIGH_RATES[(vrf_result[2] % 4) as usize];
let fraud_low_bps = LOW_RATES[(vrf_result[3] % 4) as usize];
let fraud_high_bps = HIGH_RATES[(vrf_result[4] % 4) as usize];
```

`vrf_result[n]` is `u8`. The expression `u8 % 4` always yields a value in {0, 1, 2, 3}. Cast to `usize`, this is always a valid index into a 4-element array. Out-of-bounds is impossible.

### Modulo Bias Check

256 (total u8 values) divides evenly by 4 (256 / 4 = 64). Each bucket receives exactly 64 values. Distribution is perfectly uniform with zero bias.

### Value Bounds Check

| Table | Min | Max | Within [100, 1400] BPS? |
|-------|-----|-----|------------------------|
| LOW_RATES | 100 | 400 | Yes |
| HIGH_RATES | 1100 | 1400 | Yes |

All possible derived tax rates are within the protocol's stated [100, 1400] BPS range (1% to 14%).

### Genesis Values (constants.rs:99-103)

```rust
pub const GENESIS_LOW_TAX_BPS: u16 = 300;
pub const GENESIS_HIGH_TAX_BPS: u16 = 1400;
```

Genesis rates (used at initialization before first VRF) are within the same bounded range. These are applied directly to per-token fields at `initialize_epoch_state.rs:55-58`.

### Downstream Consumption

`get_tax_bps()` at `epoch_state_reader.rs:67-73` returns the stored `u16` values directly via match on `(is_crime, is_buy)`. No additional transformation or risk of out-of-range values. The consuming `calculate_tax()` in `tax_math.rs` validates `tax_bps <= 10_000` as defense-in-depth.

## Test Coverage

The existing test suite at `tax_derivation.rs:130-333` covers:
- All 4 LOW_RATES indices (bytes 0-3 and wrap-around bytes 252-255)
- All 4 HIGH_RATES indices (bytes 0-3 and wrap-around bytes 252-255)
- Rate assignment for both cheap side configurations (CRIME cheap, FRAUD cheap)
- Independence of CRIME and FRAUD magnitude rolls
- Flip boundary (byte 191 vs 192)

## Conclusion

No vulnerability. The lookup table mechanism is sound:
1. Tables have exactly 4 entries (enforced by Rust type system)
2. All values are within [100, 1400] BPS
3. `u8 % 4` cannot produce out-of-bounds index
4. Modulo distribution is perfectly uniform (no bias)
5. Genesis values are within range
6. Comprehensive test coverage exists
