# H114 -- Bonding Curve: Purchase Near Maximum Supply

**Status:** SAFE
**Severity:** N/A
**Confidence:** HIGH

## Hypothesis

Edge cases at curve completion could cause problems: user overpays for remaining tokens, partial fill logic is incorrect, or the curve can be pushed past TARGET_TOKENS.

## Analysis

### Partial fill logic (purchase.rs lines 146-162)

The purchase handler computes `remaining = TARGET_TOKENS - curve.tokens_sold` (line 146-148, with checked_sub). Then:

```rust
let actual_tokens = std::cmp::min(tokens_out, remaining);  // line 150
```

If `tokens_out > remaining`, the user only receives `remaining` tokens. The SOL cost is then recalculated:

```rust
let actual_sol = if actual_tokens < tokens_out {
    calculate_sol_for_tokens(curve.tokens_sold, actual_tokens)?  // line 159
} else {
    sol_amount  // line 161
};
```

This correctly reprices using the integral from the current position for only the actual tokens delivered. The user pays the exact integral cost for the tokens they receive, not the full `sol_amount`.

### calculate_tokens_out also caps at remaining (belt-and-suspenders)

In `math.rs:107`:
```rust
let tokens_out = delta_x.min(remaining);
```

The math function itself caps output at remaining supply. So even before the purchase handler's `min()`, the quadratic solver will not return more tokens than available.

### tokens_sold cannot exceed TARGET_TOKENS

After the partial fill, `curve.tokens_sold` is updated with `checked_add(actual_tokens)` (line 272-275). Since `actual_tokens <= remaining = TARGET_TOKENS - tokens_sold`, the result is at most `TARGET_TOKENS`. The status transitions to `Filled` at line 285 when `tokens_sold >= TARGET_TOKENS`, and the CurveNotActive constraint (line 33) prevents further purchases.

### Slippage protection on partial fills

Line 178-181 checks `actual_tokens >= minimum_tokens_out` AFTER the partial fill. If the user set a `minimum_tokens_out` higher than the remaining supply, the transaction correctly fails with `SlippageExceeded`. This protects users from unexpectedly small fills.

### Edge case: remaining == 1 (single base unit)

With 1 base unit remaining:
- `calculate_tokens_out` returns at most 1
- `actual_tokens = min(tokens_out, 1) = 1`
- `calculate_sol_for_tokens(current_sold, 1)` computes the exact integral cost for 1 base unit
- Due to ceil rounding in `calculate_sol_for_tokens` (line 188: `checked_add(denominator - 1)`), the user pays at least 1 lamport

### Edge case: remaining == 0 (already filled)

If somehow `tokens_sold == TARGET_TOKENS` but status is still Active (should not happen due to line 285), the `checked_sub` at line 147 returns 0, `actual_tokens = min(anything, 0) = 0`. But the dust check at line 129 (`require!(tokens_out > 0)`) fires first since `calculate_tokens_out` returns 0 when remaining is 0 (math.rs line 70-72).

### Wallet cap interaction with partial fill

Step 8 (line 167-173) re-checks the wallet cap with `actual_tokens` after partial fill, ensuring the reduced amount still respects the per-wallet limit. This is redundant when `actual_tokens <= tokens_out` (which always holds), but provides defense-in-depth.

## Verdict

**No vulnerability.** The partial fill logic is thorough: tokens are capped at remaining supply, SOL is repriced via the exact integral for only the tokens delivered, slippage protection applies after the cap, and state transitions prevent further purchases once filled. The math layer provides an additional independent cap. Protocol-favored rounding (floor tokens, ceil SOL) ensures the vault is always solvent.
