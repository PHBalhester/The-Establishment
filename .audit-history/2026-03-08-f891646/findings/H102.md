# H102 -- Bonding Curve: SOL Vault Balance Tracking

**Status:** SAFE (with known mitigation)
**Severity:** Informational
**Target:** `programs/bonding_curve/src/state.rs`, `programs/bonding_curve/src/instructions/purchase.rs`, `programs/bonding_curve/src/instructions/sell.rs`, `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs`

## Hypothesis

Does BC track sol_raised in state AND vault balance? If only vault, rent-exempt minimum affects available balance.

## Analysis

### Dual Tracking Confirmed

The bonding curve uses **both** state tracking and vault balance:

1. **State fields** (state.rs:92-93):
   - `sol_raised: u64` -- cumulative SOL from buys (gross, does not decrease on sells)
   - `sol_returned: u64` -- cumulative gross SOL returned via sells
   - `tax_collected: u64` -- cumulative sell tax

2. **Vault balance** (sol_vault): A 0-byte program-owned PDA that holds actual SOL lamports.

### Purchase Flow (purchase.rs)

- Step 10 (line 186): `system_program::transfer` sends `actual_sol` from user to sol_vault.
- Step 13 (line 277): `curve.sol_raised += actual_sol` -- state tracks cumulative input.

### Sell Flow (sell.rs)

- Step 12 (line 241): Direct lamport manipulation: `sol_vault -= sol_net` (net after tax).
- Step 13 (line 247): Direct lamport manipulation: `sol_vault -= tax` to tax_escrow.
- Step 15 (line 262-263): `sol_returned += sol_gross` (gross, before tax). This preserves the identity: `vault_balance = sol_raised - sol_returned` (approximately, modulo rent).

### Rent-Exempt Minimum Handling

The sell instruction includes a **solvency assertion** (sell.rs:284-292):
```rust
let rent_exempt_min = rent.minimum_balance(0);
let expected_from_integral = calculate_sol_for_tokens(0, curve.tokens_sold)?;
let vault_balance = ctx.accounts.sol_vault.lamports();
require!(vault_balance >= expected_from_integral.saturating_sub(rent_exempt_min as u64), ...);
```

This correctly subtracts the rent-exempt minimum from the expected balance. The vault was initialized with rent-exempt lamports (~890,880 for epoch 2024) that are not from purchases, so the assertion accounts for them.

### Withdrawal (withdraw_graduated_sol.rs)

- Line 66-70: Correctly leaves rent-exempt minimum in the vault:
  ```rust
  let rent = Rent::get()?.minimum_balance(0);
  let withdrawable = vault_balance.checked_sub(rent).unwrap_or(0);
  ```
- Only callable when status == Graduated (terminal state). No future sells possible.

### Refund Path (claim_refund)

Not fully analyzed here but the state fields (`sol_raised`, `sol_returned`) provide the accounting needed for proportional refunds.

## Verdict

**No vulnerability.** The bonding curve uses dual tracking: `sol_raised`/`sol_returned` state fields for accounting, and actual vault lamport balance for transfers. The rent-exempt minimum is explicitly handled in the solvency assertion (sell.rs:284-292) and withdrawal (withdraw_graduated_sol.rs:66-70). The design is sound.

The only minor note: the solvency check uses `saturating_sub` for the rent adjustment, which means if `expected_from_integral < rent_exempt_min`, the check trivially passes (threshold becomes 0). This is fine because it only occurs when very few tokens are sold (tiny integral value), where the vault naturally holds more than needed.
