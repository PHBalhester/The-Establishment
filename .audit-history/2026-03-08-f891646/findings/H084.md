# H084 -- Staking: Double Claim Prevention

**Status:** SAFE
**Severity:** N/A
**File:** `programs/staking/src/instructions/claim.rs`, `programs/staking/src/helpers/math.rs`

## Hypothesis

Can a user claim rewards twice for the same epoch or the same accumulated rewards?

## Evidence

### 1. Checkpoint Mechanism (Primary Defense)

In `update_rewards` (math.rs:36-67), after computing the user's pending rewards:

```rust
// Step 4: Update user's checkpoint to current global
user.rewards_per_token_paid = pool.rewards_per_token_stored;
```

On the second call to `claim` (or any instruction that calls `update_rewards`), the delta is:

```
reward_delta = pool.rewards_per_token_stored - user.rewards_per_token_paid
             = X - X
             = 0
```

With delta = 0, `pending = balance * 0 / PRECISION = 0`. No new rewards are added.

### 2. rewards_earned Reset (Secondary Defense)

At claim.rs:116: `user.rewards_earned = 0;`

Even if the checkpoint somehow failed, the second claim would hit the guard at line 95:

```rust
require!(user.rewards_earned > 0, StakingError::NothingToClaim);
```

This returns `NothingToClaim` error, blocking the transaction.

### 3. Epoch-Level Double-Update Prevention

In `update_cumulative` (update_cumulative.rs:78-81):

```rust
require!(
    epoch > pool.last_update_epoch,
    StakingError::AlreadyUpdated
);
```

This prevents the Epoch Program from inflating `rewards_per_token_stored` by calling `update_cumulative` twice for the same epoch. Only CPI-gated (Epoch Program PDA required).

### 4. No Same-TX Reentrancy

The claim uses direct lamport manipulation on program-owned PDAs (not CPI to another program), so there is no reentrancy vector. Solana's runtime also prevents same-account re-entry within a transaction.

### 5. Cross-Instruction Attack (Same TX, Multiple claim IX)

If a user submits a transaction with two `claim` instructions:
- First `claim`: update_rewards sets checkpoint, rewards_earned set to `R`, then zeroed after transfer.
- Second `claim`: update_rewards computes delta = 0 (checkpoint already current), rewards_earned = 0 + 0 = 0. Hits `NothingToClaim` error.

Both the checkpoint and the zero-reset independently prevent double-claiming.

## Conclusion

SAFE. Double-claim is prevented by two independent mechanisms:
1. **Checkpoint pattern**: `rewards_per_token_paid` updated to current cumulative, making delta = 0 on subsequent calls
2. **Zero reset**: `rewards_earned = 0` after claim, guarded by `NothingToClaim` check

Additionally, epoch-level `last_update_epoch` prevents double-distribution at the source.
