# H064 — AMM Pool Locked State (Reentrancy Guard)

## Status: SAFE

## Severity: Informational

## Summary

The `PoolState.locked` field is a reentrancy guard, not an admin-controllable lock. It is set to `true` at the start of `swap_sol_pool` and cleared to `false` at the end. No admin instruction or external code path can permanently lock a pool.

## Mechanism

### Field Definition
`programs/amm/src/state/pool.rs:68`:
```rust
/// Reentrancy guard. Set to true during swap execution, cleared after.
pub locked: bool,
```

### Write Sites (exhaustive — only in `swap_sol_pool.rs`)

1. **Line 84** — Set at swap entry: `ctx.accounts.pool.locked = true;`
2. **Line 322** — Cleared at swap exit: `ctx.accounts.pool.locked = false;`

### Guard Constraint
`swap_sol_pool.rs:381`:
```rust
constraint = !pool.locked @ AmmError::PoolLocked,
```
Anchor evaluates this during account deserialization, before the handler runs.

### Initialization
`initialize_pool.rs` does NOT explicitly set `locked = false`. It relies on Anchor's `init` macro which zero-initializes all account data. Since `false` is represented as `0u8`, this is correct — newly initialized pools start unlocked.

## DoS Analysis

**Can `locked` become permanently `true`?**

No, for three reasons:

1. **Atomic transaction rollback**: If the swap handler errors at any point after setting `locked = true` (lines 87–321), the entire transaction reverts. Anchor only persists account mutations on successful `Ok(())` return. A failed swap never commits the `locked = true` write.

2. **No admin lock instruction**: The AMM program exposes exactly 4 instructions: `initialize_admin`, `burn_admin`, `initialize_pool`, `swap_sol_pool`. None of them set `locked = true` except `swap_sol_pool`, and that instruction always clears it at line 322 before returning `Ok(())`.

3. **No external write access**: The pool account is owned by the AMM program. No other program can write to it. The Tax Program invokes `swap_sol_pool` via CPI but cannot directly mutate pool state.

**Can a CPI callback (transfer hook) cause the lock to stick?**

No. If a Token-2022 transfer hook CPI fails during the INTERACTIONS phase (lines 190–315), the entire transaction reverts, rolling back the `locked = true` written at line 84. On success, execution continues to line 322 which clears the lock.

## Defense-in-Depth Note

As documented in the code comments and `03-state-machine.md` (finding #9), this lock is belt-and-suspenders. Solana's runtime already prevents the same account from being borrowed mutably twice in a CPI call chain (AccountBorrow rules), making same-pool reentrancy impossible at the runtime level. The explicit `locked` guard provides a readable, auditable second layer.

## Conclusion

No vulnerability. The `locked` field cannot be permanently set to `true` through any code path — successful swaps clear it, failed swaps roll back. There is no admin or permissionless instruction that can lock a pool. The field serves purely as defense-in-depth reentrancy protection.
