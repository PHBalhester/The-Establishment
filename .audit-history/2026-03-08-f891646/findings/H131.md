# H131 -- Bonding Curve: Token-2022 Hook Interaction During Purchase

**Status:** SAFE
**Severity:** N/A
**Target:** `programs/bonding_curve/src/instructions/purchase.rs`

## Hypothesis

If Token-2022 Transfer Hook accounts are not properly forwarded during the `purchase` instruction's token transfer, the transfer will fail (or worse, silently skip the hook).

## Analysis

### Transfer Mechanism (lines 207-251)

The purchase instruction uses **manual `invoke_signed`** with `spl_token_2022::instruction::transfer_checked` -- NOT Anchor's built-in CPI helper. This is explicitly documented as necessary because:

> "anchor_spl::token_2022::transfer_checked does NOT properly forward remaining_accounts through the Transfer Hook CPI chain."

### Hook Account Forwarding

**Step 1 -- Build base instruction** (lines 215-224):
```rust
let mut ix = spl_token_2022::instruction::transfer_checked(
    ctx.accounts.token_program.key,
    &ctx.accounts.token_vault.key(),
    &ctx.accounts.token_mint.key(),
    &ctx.accounts.user_token_account.key(),
    &ctx.accounts.curve_state.key(),
    &[],
    actual_tokens,
    TOKEN_DECIMALS,
)?;
```

**Step 2 -- Append hook accounts to instruction** (lines 227-233):
```rust
for account_info in ctx.remaining_accounts {
    ix.accounts.push(AccountMeta {
        pubkey: *account_info.key,
        is_signer: account_info.is_signer,
        is_writable: account_info.is_writable,
    });
}
```

**Step 3 -- Build account_infos with hook accounts** (lines 236-244):
```rust
let mut account_infos = vec![
    ctx.accounts.token_vault.to_account_info(),
    ctx.accounts.token_mint.to_account_info(),
    ctx.accounts.user_token_account.to_account_info(),
    ctx.accounts.curve_state.to_account_info(),
];
for account_info in ctx.remaining_accounts {
    account_infos.push(account_info.clone());
}
```

**Step 4 -- invoke_signed with PDA signer** (lines 247-251):
```rust
anchor_lang::solana_program::program::invoke_signed(
    &ix,
    &account_infos,
    signer_seeds,
)?;
```

### Correctness Verification

1. **Both the instruction AccountMetas AND the AccountInfos include hook accounts**: The code appends `remaining_accounts` to both `ix.accounts` (for the instruction) and `account_infos` (for the runtime). Both must agree for CPI to work.

2. **is_signer and is_writable preserved**: The loop copies each account's `is_signer` and `is_writable` flags faithfully from the passed `remaining_accounts`.

3. **PDA signer correct**: The curve_state PDA signs with seeds `[CURVE_SEED, token_mint_key.as_ref(), &[bump]]`, which matches the token vault's authority (validated by `token::authority = curve_state` on the vault account).

4. **Token-2022 transfer_checked enforces hook**: When a mint has a Transfer Hook extension, `transfer_checked` will invoke the hook program. If the required extra accounts are missing, Token-2022 returns error 3005 (`AccountNotEnoughKeys`). The accounts are client-supplied via `remaining_accounts`.

5. **Client responsibility**: The client must pass the correct hook accounts (extra_account_meta_list, whitelist entries for source/dest, hook program ID -- 4 accounts per the project's `HOOK_ACCOUNTS_PER_MINT` convention). If the client omits them, the TX fails on-chain -- it cannot silently succeed without hook execution.

### Comparison with Sell Path

The `sell` instruction (not shown but referenced in project memory) uses the identical pattern of manual `invoke_signed` + `remaining_accounts` forwarding. This is a proven pattern across the codebase.

## Verdict

**No vulnerability.** The purchase instruction correctly forwards Transfer Hook accounts via `remaining_accounts` to both the instruction's AccountMetas and the CPI's AccountInfos. The manual `invoke_signed` pattern is used precisely because Anchor's built-in helpers do not support hook forwarding. Token-2022 enforces hook execution -- missing accounts cause a hard failure, not a silent skip.
