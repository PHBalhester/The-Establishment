# H112 -- AMM: Pool Fee Range Validation

**Status:** SAFE (with note)
**Severity:** Informational
**File:** `programs/amm/src/instructions/initialize_pool.rs`

## Hypothesis

The `initialize_pool` instruction may not validate `lp_fee_bps` range, allowing extreme values (0 or 10000 BPS) that could break swap math or enable fee-free trading.

## Evidence

### Upper Bound (line 134)

```rust
require!(lp_fee_bps <= MAX_LP_FEE_BPS, AmmError::LpFeeExceedsMax);
```

Where `MAX_LP_FEE_BPS = 500` (5%) in `programs/amm/src/constants.rs:20`. This prevents any fee above 5%.

Values of 10000 (100%) are rejected. The constant comment references "Phase 37 audit finding" as the motivation.

### Lower Bound

There is **no minimum fee check**. `lp_fee_bps = 0` is accepted, creating a zero-fee pool.

### Impact of Zero Fee

In `calculate_effective_input` (math.rs:36-40):
```rust
let fee_factor = 10_000u128.checked_sub(fee_bps as u128)?;  // = 10000
amount.checked_mul(fee_factor)?.checked_div(10_000)          // = amount_in
```

With `fee_bps = 0`, the full input amount passes through. This is **mathematically safe** -- no division by zero, no overflow. The constant-product formula and k-invariant check work correctly with zero fee.

### Access Control

Pool creation is admin-gated (`has_one = admin @ AmmError::Unauthorized` on line 208). Only the verified admin (derived from program upgrade authority) can create pools. A malicious actor cannot create zero-fee pools.

### Fee Immutability

`lp_fee_bps` is set once at pool initialization and never modified (no update instruction exists). The fee cannot be changed after pool creation.

## Verdict

**No vulnerability.** The upper bound of 500 BPS (5%) is enforced. Zero fee is technically allowed but is admin-gated and mathematically safe. Since pool creation requires admin authority (Trust Tier 2), and the admin is derived from the program's upgrade authority, this is a design choice rather than a security gap. The admin is trusted to set reasonable fees.

**Note:** If admin authority is ever burned (via `burn_admin`), no new pools can be created, making this a non-issue for existing pools.
