# H105 -- Carnage Fund: Insufficient Balance for Swap

## Status: INVALID (Handled correctly)

## Hypothesis
If the carnage fund has a very small SOL balance, the swap might fail or return 0 tokens, potentially bricking the carnage state or causing unexpected behavior.

## Analysis

### Available SOL Calculation
The handler computes available SOL conservatively:

```rust
let sol_balance = ctx.accounts.sol_vault.lamports();
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(0);
let available_sol = sol_balance.saturating_sub(rent_exempt_min);
let swap_amount = std::cmp::min(available_sol, MAX_CARNAGE_SWAP_LAMPORTS);
```

- `saturating_sub` ensures `available_sol` is 0 (not underflow) if balance <= rent-exempt minimum.
- `MAX_CARNAGE_SWAP_LAMPORTS` = 1000 SOL caps the upper bound.

### Zero Amount Path
The buy step is gated:

```rust
let total_buy_amount = std::cmp::min(
    swap_amount.checked_add(sol_from_sale).ok_or(EpochError::Overflow)?,
    MAX_CARNAGE_SWAP_LAMPORTS,
);
// ...
let tokens_bought = if total_buy_amount > 0 {
    // ... execute swap ...
    bought
} else {
    0
};
```

If `total_buy_amount == 0` (no SOL available and no sell proceeds), the buy swap is **skipped entirely**. `tokens_bought` = 0. This is not a failure -- execution continues.

### State Update with 0 Tokens
After the buy (or skip), the state is updated:

```rust
carnage_state.held_token = target.to_u8() + 1;
carnage_state.held_amount = tokens_bought; // could be 0
carnage_state.total_sol_spent = carnage_state.total_sol_spent
    .checked_add(total_buy_amount)?; // adds 0
```

The carnage is recorded as executed with 0 tokens bought and 0 SOL spent. The `carnage_pending` flag is cleared. The next epoch proceeds normally.

### Very Small Non-Zero Amount
If `total_buy_amount` is very small (e.g., 1 lamport), the swap CPI to Tax::swap_exempt will execute. The AMM's constant-product formula would compute a proportionally small token output. If the AMM returns 0 tokens due to rounding, the slippage check applies:

```rust
if reserve_sol > 0 && reserve_token > 0 {
    let expected = ...;
    let min_output = (expected as u128)
        .checked_mul(CARNAGE_SLIPPAGE_BPS_ATOMIC as u128)
        .and_then(|n| n.checked_div(10_000))
        .ok_or(EpochError::Overflow)? as u64;
    require!(bought >= min_output, EpochError::CarnageSlippageExceeded);
}
```

If `expected = 0` (due to tiny input), then `min_output = 0`, and `bought >= 0` passes. So even dust swaps succeed without reverting.

### Sell Path with 0 Holdings
The sell path also guards against 0: `if held_amount > 0` gates the sell. If held_amount is 0, sell is skipped.

### Burn Path with 0 Holdings
Similarly: `if amount == 0 { return Ok(0); }` in `burn_held_tokens`.

## Verdict
**No vulnerability.** All paths handle zero and near-zero amounts gracefully. The `total_buy_amount > 0` guard prevents CPI calls with 0 input. Dust amounts produce valid (if tiny) swaps. State is updated consistently regardless of swap size. Carnage execution never bricks -- worst case is a no-op carnage that buys 0 tokens.
