# H012: Staking Escrow Rent Depletion (PDA Destruction)

**Status:** CONFIRMED
**Severity:** HIGH
**Category:** Token/Economic, Staking
**Confidence:** HIGH (previously confirmed as S001 in audit #1, still not fixed)

---

## Summary

The staking program's `claim` instruction transfers SOL rewards from the `escrow_vault` PDA to users via direct lamport manipulation (`try_borrow_mut_lamports`) without reserving the rent-exempt minimum. When the escrow balance is in the danger zone (between 0 and ~890,880 lamports above the last claim amount), two failure modes exist:

1. **Claim freeze**: If a claim would leave the escrow between 0 and rent-exempt (~890,880 lamports for a 0-byte program-owned account), the Solana runtime rejects the transaction. All subsequent claims fail until new tax deposits replenish the escrow.

2. **PDA destruction**: If a claim drains the escrow to exactly 0 lamports, the PDA is garbage-collected. The next `deposit_rewards` CPI from the Tax Program fails because the account no longer exists, permanently halting the 71% staking portion of all swap tax distribution.

This was originally reported as S001 in audit #1 (2026-02-22) and remains **NOT FIXED**.

The transfer at lines 140-145 can drain below rent-exempt:

    .accounts
    .escrow_vault
    .lamports()

### Escrow vault creation: `initialize_stake_pool.rs:49-56`

```rust

    init,
    payer = authority,
    space = 0,        // 0 data bytes
    seeds = [ESCROW_VAULT_SEED],
    bump,

pub escrow_vault: AccountInfo<'info>,
```

The escrow is a 0-data-byte account owned by the staking program. Rent-exempt minimum for this account: `Rent::minimum_balance(0)` = ~890,880 lamports.

### No rent checks anywhere in staking program

A grep for `rent_exempt`, `rent.minimum_balance`, and `RENT` across `programs/staking/src/` returns zero matches. The correct pattern exists elsewhere in the codebase (e.g., `execute_carnage_atomic.rs:351-356`) but was never applied to the staking escrow.

### Deposit path is unaffected but dependent on PDA existence

The Tax Program's `swap_sol_buy.rs:128-140` transfers SOL to the escrow via `system_instruction::transfer`, and then calls `deposit_rewards` CPI which requires the escrow PDA to exist for balance reconciliation (`deposit_rewards.rs:99-101`):

```rust
require!(
    ctx.accounts.escrow_vault.lamports() >= pool.pending_rewards,
    StakingError::InsufficientEscrowBalance
);
```

If the PDA is destroyed, this account cannot be passed, and the entire deposit_rewards CPI fails, halting the staking portion (71%) of all swap tax distribution.

## Attack Scenario

### Scenario 1: Organic Claim Freeze (HIGH likelihood)

1. Protocol operates normally; staking escrow accumulates SOL from swap taxes.
2. Multiple stakers claim rewards over time, draining the escrow.
3. Escrow balance reaches danger zone: e.g., 1,200,000 lamports total.
4. Alice has 500,000 lamports in pending rewards.
5. Alice calls `claim()`.
6. Check passes: `1,200,000 >= 500,000`.
7. Post-transfer balance: `1,200,000 - 500,000 = 700,000` lamports.
8. Runtime rejects: `700,000 < 890,880` (rent-exempt minimum).
9. **All claims are now frozen** until new swap taxes replenish the escrow above `890,880 + next_claim_amount`.

### Scenario 2: PDA Destruction via Exact Drain (MEDIUM likelihood)

1. Escrow balance equals exactly one user's pending rewards (e.g., both are 2,000,000 lamports).
2. User claims, draining escrow to 0.
3. Runtime accepts the transaction (0 lamports is valid -- account is closed).
4. Escrow PDA is garbage-collected.
5. Next swap's `deposit_rewards` CPI fails because the escrow account no longer exists.
6. **71% of all swap tax distribution permanently halted**.
7. Recovery requires program upgrade to recreate the escrow PDA (programs are intended to be non-upgradeable at mainnet).

### Scenario 3: Deliberate Griefing (MEDIUM likelihood)

1. Attacker monitors escrow balance on-chain.
2. Attacker stakes a small amount of PROFIT and waits to accumulate exactly enough rewards.
3. When escrow balance is in the danger zone, attacker claims to trigger freeze or destruction.
4. Cost: near-zero (only needs a small stake and gas).
5. Impact: staking subsystem frozen for all users.

## Root Cause

The `claim` instruction performs a single-threshold check (`escrow_balance >= rewards_to_claim`) without accounting for the rent-exempt minimum that must remain in the account. The Solana runtime enforces that program-owned accounts must either maintain rent-exempt balance OR be drained to exactly 0 -- no in-between state is allowed.

The correct pattern (`available = balance.saturating_sub(rent_exempt_min)`) is already used in `execute_carnage_atomic.rs:351-356` for the carnage SOL vault, demonstrating the developers are aware of the requirement but failed to apply it consistently.

---

## Recommendation

### Fix 1: Reserve rent-exempt minimum in claim.rs (REQUIRED)

```rust
// Replace lines 100-111 in claim.rs:
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(0); // escrow_vault has 0 data bytes
let escrow_balance = ctx.accounts.escrow_vault.lamports();
let available_balance = escrow_balance.saturating_sub(rent_exempt_min);

if available_balance < rewards_to_claim {
    emit!(EscrowInsufficientAttempt {
        user: ctx.accounts.user.key(),
        requested: rewards_to_claim,
        available: available_balance, // Log available, not total
        slot: clock.slot,
    });
    return Err(StakingError::InsufficientEscrowBalance.into());
}
```

### Fix 2: Also guard unstake.rs if it ever adds reward auto-claim

The current `unstake.rs` forfeits rewards rather than claiming them, so it does not transfer SOL from escrow. However, if this ever changes, the same guard must be applied.

### Fix 3: Add regression test

Test that claiming when escrow is at `rent_exempt_min + reward - 1` fails with `InsufficientEscrowBalance`, and that claiming when escrow is at `rent_exempt_min + reward` succeeds with post-balance >= rent_exempt_min.
