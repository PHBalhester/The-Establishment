# H120 -- Tax Program: Buy Path Output Floor

## Status: VERIFIED CORRECT

## Target
`/Users/mlbob/Projects/Dr Fraudsworth/programs/tax-program/src/instructions/swap_sol_buy.rs`

## Hypothesis
Verify the 50% output floor (SEC-10) is applied correctly on the buy path.

## Evidence

### Floor Computation (swap_sol_buy.rs lines 105-111)
```rust
let (reserve_a, reserve_b) = read_pool_reserves(&ctx.accounts.pool)?;
let output_floor = calculate_output_floor(reserve_a, reserve_b, sol_to_swap, MINIMUM_OUTPUT_FLOOR_BPS)
    .ok_or(error!(TaxError::TaxOverflow))?;
require!(
    minimum_output >= output_floor,
    TaxError::MinimumOutputFloorViolation
);
```

### Correctness Checklist

1. **Reserve ordering: CORRECT.** Buy is AtoB (SOL in, token out). `read_pool_reserves` returns `(reserve_a, reserve_b)`. `calculate_output_floor(reserve_in, reserve_out, ...)` receives `reserve_a` as `reserve_in` (SOL) and `reserve_b` as `reserve_out` (token). This matches the constant-product formula for AtoB direction.

2. **Input amount: CORRECT.** Uses `sol_to_swap` (post-tax), NOT `amount_in` (pre-tax). Comment at line 103 explicitly documents why: "Using amount_in would compute a higher expected output than achievable, making the floor too tight." This is correct because the AMM only receives `sol_to_swap` after tax deduction.

3. **Floor BPS: CORRECT.** `MINIMUM_OUTPUT_FLOOR_BPS = 5000` (50%) from constants.rs line 40.

4. **Floor formula (tax_math.rs lines 141-165): CORRECT.**
   - `expected = reserve_out * amount_in / (reserve_in + amount_in)` -- standard constant-product output formula.
   - `floor = expected * floor_bps / 10_000` -- applies 50% threshold.
   - Uses u128 intermediates to prevent overflow.
   - Returns `Some(0)` for empty pools (graceful degradation, not exploitable since AMM would also reject).

5. **Comparison direction: CORRECT.** `minimum_output >= output_floor` -- rejects if user's stated minimum is below the protocol floor. This catches bots/sandwich attackers that set minimum_output=0.

### Sell Path Consistency (swap_sol_sell.rs lines 112-118)
```rust
let (reserve_a, reserve_b) = read_pool_reserves(&ctx.accounts.pool)?;
let output_floor = calculate_output_floor(reserve_b, reserve_a, amount_in, MINIMUM_OUTPUT_FLOOR_BPS)
```
Sell is BtoA, so reserves are swapped: `reserve_b` is `reserve_in` (token), `reserve_a` is `reserve_out` (SOL). **Consistent and correct.**

### Edge Cases
- **Zero reserves**: Returns `Some(0)` floor, so the check passes trivially. Not exploitable because AMM's constant-product invariant would independently reject swaps on empty pools.
- **Tiny swaps**: Floor rounds down to 0, check passes. Acceptable since LP fee already dominates at dust amounts.
- **LP fee not deducted from floor calculation**: Intentional. At 50% floor, the ~1% LP fee is absorbed by the generous threshold (documented in tax_math.rs line 138).

### Test Coverage
- `tax_math.rs` has 12 unit tests for `calculate_output_floor` covering equal reserves, zero inputs, 100% floor, large reserves, small amounts, and realistic pool scenarios.
- 10,000-case proptest suite covers `calculate_tax` and `split_distribution` properties.

## Verdict
**The 50% output floor is correctly implemented on the buy path.** Reserve ordering, input amount selection (post-tax), floor BPS constant, formula math, and comparison direction are all correct. The sell path uses the same function with appropriately swapped reserves. No issues found.
