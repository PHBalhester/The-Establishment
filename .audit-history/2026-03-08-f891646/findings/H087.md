# H087 -- AMM Initialization: Pool Seeding

| Field | Value |
|-------|-------|
| Hypothesis | Pool seeding amounts may set incorrect initial price |
| Severity | **Informational** |
| Status | **CONFIRMED -- Design risk (mitigated by env vars)** |
| Confidence | High |
| Files | `scripts/deploy/initialize.ts`, `tests/integration/helpers/constants.ts`, `.env` |

## Analysis

Pool seeding determines the initial AMM price via the constant-product formula: `price = SOL_reserve / TOKEN_reserve`.

### Constants (from `tests/integration/helpers/constants.ts`)

```typescript
// Test defaults (localnet):
SOL_POOL_SEED_SOL = 10 * LAMPORTS_PER_SOL;           // 10 SOL
SOL_POOL_SEED_TOKEN = 10_000_000_000;                  // 10,000 tokens (6 decimals)

// Overridable via env vars:
SOL_POOL_SEED_SOL = Number(process.env.SOL_POOL_SEED_SOL_OVERRIDE) || 10 SOL;
SOL_POOL_SEED_TOKEN = Number(process.env.SOL_POOL_SEED_TOKEN_OVERRIDE) || 10K tokens;
```

### Production values (from `.env`):

```
SOL_POOL_SEED_SOL_OVERRIDE=2500000000        # 2.5 SOL
SOL_POOL_SEED_TOKEN_OVERRIDE=290000000000000  # 290M tokens (6 decimals)
```

This gives an initial price of: `2.5 SOL / 290,000,000 tokens = 0.00000000862 SOL/token` (~$0.0000017 at $200 SOL).

### Pool Initialization in `initialize.ts`

Both CRIME/SOL and FRAUD/SOL pools are seeded identically (lines 774-802):
- Same `SOL_POOL_SEED_SOL` and `SOL_POOL_SEED_TOKEN` for both pools
- Both use `SOL_POOL_FEE_BPS` for fee configuration
- WSOL amount wraps `2 * SOL_POOL_SEED_SOL + 5e9` (buffer for fees)

### Risk: Missing `.env` Silently Uses Test Defaults

The `MEMORY.md` documents a prior incident: forgetting to source `.env` before `initialize.ts` caused test defaults (10 SOL / 10K tokens) to be used, resulting in a wildly incorrect initial price and wasted ~50 SOL. Pools cannot be re-seeded -- requires full redeploy.

The code uses `Number(process.env.X) || default` which means:
- Missing env var -> falls through to test default silently
- `"0"` or `""` -> also falls through silently (Number("") = 0, which is falsy)

### Verdict

The pool seeding logic itself is correct -- both pools get identical treatment, and the env var mechanism works when properly sourced. The risk is operational: silent fallback to test defaults if `.env` is not loaded. This was already encountered and documented in MEMORY.md. The deploy pipeline (`deploy-all.sh`) should enforce `.env` sourcing, but the constants file does not warn or fail on missing overrides for non-test environments.

### Recommendation

Add a guard in `initialize.ts` when `CLUSTER_URL` contains "devnet" or "mainnet":
```typescript
if (!process.env.SOL_POOL_SEED_SOL_OVERRIDE) {
  throw new Error("SOL_POOL_SEED_SOL_OVERRIDE required for non-localnet deployment");
}
```
