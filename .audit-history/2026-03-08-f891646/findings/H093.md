# H093 -- Bonding Curve: Token Vault PDA Derivation

**Status:** SAFE
**Severity:** N/A (no issue found)
**Target:** `programs/bonding_curve/src/instructions/` (vault account derivation)

## Hypothesis

Check vault PDA seeds for uniqueness per curve. No seed collision.

## Analysis

### PDA Seeds

All bonding curve PDAs include `token_mint.key()` as a seed component, ensuring uniqueness per token:

| Account | Seeds | Defined In |
|---------|-------|-----------|
| CurveState | `["curve", token_mint]` | initialize_curve.rs:26 |
| Token Vault | `["curve_token_vault", token_mint]` | initialize_curve.rs:39 |
| SOL Vault | `["curve_sol_vault", token_mint]` | initialize_curve.rs:52 |
| Tax Escrow | `["tax_escrow", token_mint]` | initialize_curve.rs:65 |

Seed constants from `constants.rs`:
- `CURVE_SEED = b"curve"` (line 101)
- `CURVE_TOKEN_VAULT_SEED = b"curve_token_vault"` (line 104)
- `CURVE_SOL_VAULT_SEED = b"curve_sol_vault"` (line 107)
- `TAX_ESCROW_SEED = b"tax_escrow"` (line 110)

### Uniqueness Guarantee

1. **Per-mint uniqueness**: Each PDA seed includes the full 32-byte `token_mint` pubkey. Different mints produce different PDAs. CRIME and FRAUD curves cannot collide.

2. **Cross-type uniqueness**: The string prefix differs across account types (`"curve"` vs `"curve_token_vault"` vs `"curve_sol_vault"` vs `"tax_escrow"`), so even for the same mint, the four PDAs are distinct.

3. **Cross-program uniqueness**: PDAs are derived from the bonding_curve program ID, so they cannot collide with PDAs from other programs (AMM, Tax, Epoch, etc.) even if seeds overlap.

### Downstream Validation

All instructions that interact with vaults re-derive the PDA via seeds constraints:

- **purchase.rs** (lines 49-58): `seeds = [CURVE_TOKEN_VAULT_SEED, token_mint.key().as_ref()]` with `token::authority = curve_state` and `token::mint = token_mint`.
- **sell.rs** (lines 51-58): Same pattern for token vault, plus `seeds = [CURVE_SOL_VAULT_SEED, token_mint.key().as_ref()]` for SOL vault, and `seeds = [TAX_ESCROW_SEED, token_mint.key().as_ref()]` for tax escrow with stored-key cross-check (`tax_escrow.key() == curve_state.tax_escrow`).
- **fund_curve.rs** (lines 45-53): Token vault re-derived with `token::authority = curve_state`.

### Mint Validation

`initialize_curve.rs` (lines 73-79) has a feature-gated constraint restricting mints to `crime_mint()` or `fraud_mint()` in production. In `localnet` mode, any mint is accepted -- this is test-only flexibility and not a mainnet concern.

### init Prevents Re-initialization

All four accounts use `init` (not `init_if_needed`), so Anchor will fail if the PDA already exists. This prevents a second curve for the same mint from overwriting the first.

## Verdict

No issue. All vault PDAs include the token mint pubkey as a seed, ensuring uniqueness per curve. Cross-type and cross-program collisions are impossible due to distinct seed prefixes and program ID scoping. The `init` constraint prevents re-initialization.
