# H052 -- Bonding Curve Sell During Active State

**Auditor:** Hypothesis Investigator (sos-002)
**Target:** `programs/bonding_curve/src/instructions/sell.rs`
**Severity:** INFORMATIONAL
**Verdict:** SECURE -- No exploitable vulnerability found. All investigated attack vectors are mitigated by the combination of Anchor constraints, integral math properties, solvency assertion, and extensive proptest coverage.

---

## Hypothesis

Can users sell during Active state? Does `tokens_sold` decrease correctly? Can buy-sell cycles manipulate `tokens_sold` or the vault balance to drain SOL or create a profitable round-trip?

## Investigated Attack Vectors

### 1. Selling in Non-Active States

**Finding:** NOT POSSIBLE. The sell instruction is double-gated:

1. **Anchor constraint** (line 36): `constraint = curve_state.status == CurveStatus::Active @ CurveError::CurveNotActiveForSell`
2. **Handler-level require!** (line 121-124): Redundant Active-only check.

Once the curve transitions to `Filled` (via purchase when `tokens_sold >= TARGET_TOKENS`), `Failed` (via `mark_failed`), or `Graduated` (via `prepare_transition`), all sells are permanently blocked. These transitions are forward-only with no mechanism to revert to Active.

### 2. tokens_sold Decrease Correctness

**Finding:** CORRECT. The sell handler computes:

```
x2 = x1.checked_sub(tokens_to_sell)  // line 155-157
...
curve.tokens_sold = x2               // line 254
```

- `checked_sub` prevents underflow (returns `CurveError::Overflow` if `tokens_to_sell > tokens_sold`).
- The balance check at line 129-131 ensures the user actually holds the tokens they claim to sell.
- The Token-2022 `transfer_checked` at line 200-232 atomically moves tokens from user to vault, so the user cannot sell tokens they don't own.
- `tokens_sold` decreases by exactly `tokens_to_sell`, which is consistent with the integral math.

### 3. Buy-Sell Round-Trip Profitability

**Finding:** NOT PROFITABLE. Three independent mechanisms prevent profit extraction:

**A. Integral math symmetry:** Both buy and sell use the same `calculate_sol_for_tokens()` integral function. Buying at position x1 and selling back from (x1 + tokens) to x1 returns the same gross SOL area under the curve. The buy uses `calculate_tokens_out()` which applies floor rounding (user gets fewer tokens), while the sell integral uses ceil rounding (user pays slightly more for the area). This creates a small protocol-favored asymmetry even before tax.

**B. 15% sell tax:** `SELL_TAX_BPS = 1500` (line 91 of constants.rs). Tax is computed with ceil rounding (protocol-favored):
```
tax = (sol_gross * 1500 + 9999) / 10000  // lines 174-179
```
This guarantees at least 15% loss on any round-trip, making buy-sell cycles economically unprofitable.

**C. Solvency assertion (defense-in-depth):** Lines 284-292 verify after every sell:
```
vault_balance >= integral(0, tokens_sold) - rent_exempt_min
```
This post-condition catches any math bug that might allow the vault to become insolvent. It computes the full integral from 0 to the current `tokens_sold` position, ensuring the vault can always cover all remaining obligations.

**Proptest coverage:** 1M iterations of `buy_sell_round_trip_always_loses` (PROPERTY S1) prove no profitable round-trip exists at any curve position. An additional 1M iterations of `vault_solvency_mixed_buy_sell` (PROPERTY S2) prove no sequence of 3-8 mixed buy/sell operations causes vault underflow. `multi_user_solvency` (PROPERTY S5) proves interleaved multi-user operations maintain solvency.

### 4. tokens_sold Manipulation via Concurrent Buy-Sell

**Finding:** NOT EXPLOITABLE. Solana's transaction model provides inherent protection:

- Each transaction that touches `curve_state` holds a write lock on the CurveState PDA.
- Concurrent transactions on the same CurveState are serialized by the runtime.
- Within a single transaction, there is no mechanism to invoke both `purchase` and `sell` on the same curve (they are separate instructions with the same mutable PDA, which Solana serializes).

### 5. Vault Balance Drain via Tax Escrow Split

**Finding:** NOT EXPLOITABLE. The sell moves SOL in two steps:

```
sol_vault -= sol_net      // to user (line 241-242)
sol_vault -= tax          // to tax_escrow (line 247-248)
```

The total SOL leaving the vault is `sol_net + tax = sol_gross`, which exactly equals the integral value. The vault tracks `sol_returned += sol_gross` (line 262-265), preserving the identity `vault_balance = sol_raised - sol_returned`. The solvency assertion then verifies `vault_balance >= integral(0, tokens_sold)`.

The `saturating_sub` at line 290 (`expected_from_integral.saturating_sub(rent_exempt_min)`) correctly accounts for the rent-exempt lamports that were deposited at vault creation (not from purchases). Without this adjustment, the assertion would be too strict (vault has rent lamports + purchase lamports but the integral only accounts for purchase-derived SOL).

### 6. Sell After Deadline (Timing Window)

**Finding:** CORRECTLY BLOCKED. Line 142-145 checks `Clock::get()?.slot <= curve.deadline_slot`. Sells are blocked after the deadline, just like purchases. The `mark_failed` instruction has an additional `FAILURE_GRACE_SLOTS` (150 slots) buffer, but sells are already blocked at the hard deadline. There is no timing window where sells are allowed but the curve is in a transitioning state.

### 7. Sell Zero Tokens

**Finding:** BLOCKED. Line 137 requires `tokens_to_sell > 0`.

### 8. Sell When tokens_sold = 0

**Finding:** BLOCKED. If `tokens_sold = 0`, then `x2 = 0 - tokens_to_sell` would underflow via `checked_sub`, returning an error. Additionally, the user would need to hold tokens (balance check at line 129-131), but no tokens have been sold from this curve, so the user cannot have obtained tokens to sell back.

## Rounding Analysis

The sell path uses three rounding decisions, all protocol-favored:

| Step | Function | Rounding | Direction |
|------|----------|----------|-----------|
| Buy: tokens out | `calculate_tokens_out` | Floor | User gets fewer tokens |
| Sell: SOL gross | `calculate_sol_for_tokens` | Ceil | Area under curve slightly overstated |
| Sell: Tax | `(gross * 1500 + 9999) / 10000` | Ceil | User pays more tax |

**Net effect:** The ceil rounding on `calculate_sol_for_tokens` might seem to favor the user (they get slightly more gross SOL back), but the function is also used on the buy side to compute `actual_sol` for partial fills (purchase.rs line 159). The ceil tax then ensures the protocol retains more. The overall system is protocol-favored in aggregate, as proven by the proptest round-trip loss property.

## Conclusion

The bonding curve sell mechanism during Active state is correctly implemented:

1. **State gating**: Sells are exclusively allowed during Active state, enforced at both constraint and handler levels.
2. **tokens_sold accounting**: Correctly decremented with overflow protection, matching the integral math.
3. **Vault solvency**: Maintained by the mathematical identity between integral areas and SOL flows, with a defense-in-depth assertion after every sell.
4. **No profitable round-trips**: 15% sell tax + rounding asymmetry guarantees loss, proven across 1M+ proptest iterations.
5. **Tax routing**: Correctly splits gross SOL between user (net) and escrow (tax) via direct lamport manipulation.

No finding to report. This hypothesis is **REJECTED** -- the sell mechanism is sound.
