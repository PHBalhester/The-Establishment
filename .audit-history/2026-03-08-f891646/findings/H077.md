# H077 -- Multiple `as u64` Unchecked Casts

**Status:** Confirmed (Informational with one Medium-risk exception)
**Severity:** Low / Medium (one case)
**Confidence:** High

## Summary

Grep across all non-test Rust source found 12 production files containing `as u64` casts from u128 intermediates. Most are provably safe due to domain constraints. One cast in `calculate_refund` (bonding_curve/src/math.rs:236 and claim_refund.rs:149) lacks a `u64::try_from` guard and could theoretically truncate, though domain constraints make exploitation unlikely.

## Evidence

### Category 1: Provably Safe (u128 result bounded by u64 inputs)

| File | Line | Expression | Why Safe |
|------|------|-----------|----------|
| tax_math.rs | 52 | `u64::try_from(tax).ok()` | Uses try_from -- properly guarded |
| tax_math.rs | 97-101 | `u64::try_from(staking_u128).ok()` | Uses try_from -- properly guarded |
| tax_math.rs | 164 | `floor as u64` | floor <= expected <= reserve_out (u64), comment documents proof |
| amm/math.rs | 75 | `u64::try_from(output).ok()` | Uses try_from -- properly guarded |
| trigger_epoch_transition.rs | 99 | `epoch as u64` | epoch is u32, always fits in u64 |
| initialize_extra_account_meta_list.rs | 79 | `account_size as u64` | usize from TLV size calc, realistically tiny |
| bonding_curve/math.rs | 109 | `tokens_out as u64` | Capped by `delta_x.min(remaining)` where remaining <= TOTAL_FOR_SALE = 460e12 < u64::MAX |
| bonding_curve/math.rs | 211 | `price as u64` | price = P_START + range*progress/PRECISION, max = P_END = 3450, trivially fits |
| execute_carnage.rs | 435,440 | `... as u64` | Result of div by (reserve_sol + buy_amount), bounded by reserve_token (u64) |
| execute_carnage_atomic.rs | 428,433 | `... as u64` | Same pattern as execute_carnage.rs |

### Category 2: Medium Risk -- No try_from Guard

| File | Line | Expression | Risk |
|------|------|-----------|------|
| bonding_curve/math.rs | 236 | `Some(result as u64)` in `calculate_refund` | `user_balance * refund_pool / total_outstanding` -- if `user_balance == total_outstanding`, result == refund_pool (u64), safe. But if `total_outstanding` is artificially small (e.g., rounding dust after many claims), result could exceed u64. Domain constraints (tokens_sold decremented atomically with burns) make this unlikely but the cast is unguarded. |
| claim_refund.rs | 149 | Same formula inline: `((user_balance as u128).checked_mul(refund_pool as u128)? / (total_outstanding as u128)) as u64` | Mirrors math.rs:236. Same risk profile. |
| bonding_curve/math.rs | 192 | `sol_lamports as u64` in `calculate_sol_for_tokens` | Result of ceil-division. Worst case: full curve integral ~1000 SOL = 1e12 lamports, well within u64. Safe by economic constraint but not by type system. |

### Category 3: Staking Math

| File | Line | Expression | Analysis |
|------|------|-----------|----------|
| staking/math.rs | 50 | `.checked_div(PRECISION)? as u64` | PRECISION = 1e12. Result = (balance * delta) / PRECISION. balance is u64, delta is u128 (cumulative). If cumulative grows very large (many epochs with small total_staked), this could exceed u64. Mitigated by: cumulative only grows by `rewards * PRECISION / total_staked`, and total rewards are bounded by SOL supply. Still, no try_from guard. |

## Recommendation

1. **Replace bare `as u64` with `u64::try_from(...).ok_or(Error)?`** in:
   - `calculate_refund` (math.rs:236)
   - `claim_refund` handler (claim_refund.rs:149)
   - `calculate_sol_for_tokens` (math.rs:192)
   - `update_rewards` (staking/math.rs:50)

2. The remaining casts are provably safe by domain constraints, but adding `u64::try_from` uniformly would be zero-cost defense-in-depth and eliminate this class of concern entirely.

3. Note: `tax_math.rs` and `amm/math.rs` already use the correct `u64::try_from(...).ok()` pattern throughout -- these should be the model for the rest of the codebase.
