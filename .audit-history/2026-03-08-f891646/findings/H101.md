# H101 -- AMM: compute_swap_output Precision

**Status:** SAFE
**Severity:** N/A (no issue found)
**Target:** `programs/amm/src/helpers/math.rs`, `programs/amm/src/instructions/swap_sol_pool.rs`

## Hypothesis

Check u128 intermediates in swap calculation. Verify floor division favors pool (not trader).

## Analysis

### Math Pipeline

The AMM swap math proceeds in two steps:

1. **Fee deduction** (`calculate_effective_input`, math.rs:36):
   ```
   effective_input = amount_in * (10_000 - fee_bps) / 10_000
   ```
   - Upcast to u128 before multiply -- no overflow possible (u64::MAX * 10_000 fits in u128).
   - Floor division truncates toward zero, meaning the protocol retains fractional fee. Correct: favors pool.

2. **Constant-product output** (`calculate_swap_output`, math.rs:58):
   ```
   output = reserve_out * effective_input / (reserve_in + effective_input)
   ```
   - All operands upcast to u128 before multiply.
   - `numerator = r_out * effective_input` -- max is u64::MAX * u64::MAX ~ 2^128, which fits since u128::MAX = 2^128 - 1.
   - Floor division truncates output downward. Correct: trader receives less, pool retains dust. This is the standard AMM rounding direction.
   - Output downcasted via `u64::try_from` -- fails if output exceeds u64::MAX (returns None -> error).

### K-invariant Post-Check

`verify_k_invariant` (math.rs:92) confirms `k_after >= k_before` using u128 products. Called at swap_sol_pool.rs:171 after reserve updates. This is defense-in-depth: even if the math had a subtle rounding bug, the k-check would catch it.

### Reserve Updates

swap_sol_pool.rs:163-168: `new_reserve_in = reserve_in + amount_in` (pre-fee amount added, not effective_input). The fee stays implicitly in the pool as the difference between amount_in and effective_input. This is correct constant-product AMM behavior.

### Edge Cases Verified

- Zero effective input after fee: caught by `check_effective_input_nonzero` (line 130).
- Zero swap output: caught by `check_swap_output_nonzero` (line 140).
- Denominator zero: `calculate_swap_output` returns None (math.rs:69-71).
- Proptest coverage: 10,000 iterations verify k-invariant holds, output <= reserve_out, fee monotonicity.

## Verdict

**No vulnerability.** All intermediate arithmetic uses u128. Floor division consistently favors the pool over the trader in both fee deduction and output calculation. The k-invariant post-check provides an additional safety net. This is a well-implemented constant-product AMM math module.
