# H005 — Bonding Curve Authority: close_token_vault Rent Extraction

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** Access Control
**Confidence:** HIGH

## Summary

The `close_token_vault` instruction accepts any `Signer<'info>` as `authority` with zero on-chain verification that the signer is the protocol deployer. Any user can call this instruction on a Graduated curve with an empty token vault and receive the rent-exempt lamports (~2.04M lamports / ~0.00204 SOL for a Token-2022 token account). While the direct financial impact is low (rent recovery only), this is part of the broader bonding curve authority gap affecting 6 instructions.

## Evidence

**File:** `programs/bonding_curve/src/instructions/close_token_vault.rs`

- **Line 27-28:** `authority` is a bare `Signer<'info>` with `#[account(mut)]` -- no `has_one`, no `address =`, no ProgramData constraint.
- **Line 35:** State guard: `curve_state.status == CurveStatus::Graduated` -- only callable on graduated curves (terminal state).
- **Line 46:** Balance guard: `token_vault.amount == 0` -- vault must be empty (tokens already withdrawn for AMM seeding).
- **Line 84:** Rent destination: `destination: ctx.accounts.authority.to_account_info()` -- the caller receives the rent lamports.

**No authority verification exists anywhere in the bonding curve program:**
- `Grep` for `has_one`, `address =`, `require_keys_eq` across `programs/bonding_curve/src` returns zero matches.

## Attack Scenario

1. Both bonding curves graduate (status = Graduated via `prepare_transition`).
2. Legitimate admin calls `withdraw_graduated_sol` to extract SOL from vaults and seeds AMM pools.
3. Legitimate admin withdraws tokens from token vaults for AMM pool seeding (vault balance -> 0).
4. Attacker observes the empty, graduated token vaults.
5. Attacker calls `close_token_vault` for each curve, receiving ~0.002 SOL rent per vault (~0.004 SOL total).
6. The token vault PDA account is destroyed.

**Note:** This is the least severe of the 6 unprotected bonding curve instructions. The `withdraw_graduated_sol` vulnerability (H003/similar) is far more critical (~1000 SOL per curve vs ~0.002 SOL here).

## Impact Assessment

- **Direct financial loss:** ~0.004 SOL (2 token vault rent-exempt amounts). Negligible.
- **DoS on refund path:** NOT AFFECTED. The `claim_refund` and `consolidate_for_refund` instructions operate on the SOL vault and tax escrow, not the token vault. Token burns in `claim_refund` use the user's ATA directly.
- **DoS on graduated path:** NOT AFFECTED. By the time `close_token_vault` is callable (Graduated + empty vault), all graduation operations (token withdrawal, AMM seeding) have already completed.
- **Idempotency:** The vault can only be closed once. After closing, the account no longer exists and the instruction would fail on subsequent calls.

## Root Cause

The bonding curve program (added in v1.2) does not implement any on-chain authority verification pattern. All 6 admin-labeled instructions (`initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault`) accept a bare `Signer<'info>` named `authority`. This contrasts with the AMM program in the same codebase, which correctly uses ProgramData upgrade-authority verification.

The state guards (Graduated status + empty vault) are the sole defense, and they are effective at limiting the impact to rent extraction only.

## Recommendation

Apply the same ProgramData upgrade-authority pattern used in `amm/initialize_admin.rs` to the bonding curve program. Add a `BondingCurveAdmin` PDA (or reuse the AMM's `AdminConfig`) and enforce `has_one = admin` on all 6 instructions. This should be addressed as part of the broader bonding curve authority gap fix (same root cause as the CRITICAL `withdraw_graduated_sol` finding).

Specifically for `close_token_vault`, the severity is low enough that fixing the authority check alone would suffice -- the state guards prevent any meaningful abuse.
