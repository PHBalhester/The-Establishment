## Hypothesis

Can the AMM admin delegate pool creation to another address by updating `AdminConfig.admin` after initialization?

## Target Code

- `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/initialize_admin.rs`
- `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/burn_admin.rs`
- `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/initialize_pool.rs`
- `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/state/admin.rs`

## Evidence

### AdminConfig Structure

`AdminConfig` is a PDA with seeds `[b"admin"]` containing two fields:

```rust
pub struct AdminConfig {
    pub admin: Pubkey,  // The admin pubkey authorized to create pools
    pub bump: u8,       // PDA bump seed
}
```

### All Write Paths to `AdminConfig.admin`

Exhaustive grep for `admin_config.admin =` across the entire AMM program yields exactly **two** locations:

1. **`initialize_admin.rs:20`** — `admin_config.admin = admin;`
   - Sets the admin once during initialization.
   - Gated by ProgramData upgrade-authority check (only the deployer can call).
   - `init` constraint ensures the PDA can only be created once.

2. **`burn_admin.rs:22`** — `admin_config.admin = Pubkey::default();`
   - Irreversibly sets admin to `Pubkey::default()` (all zeros).
   - Gated by `has_one = admin` — only the current admin can burn.
   - Since no one can sign as `Pubkey::default()`, this permanently disables pool creation.

### No Update/Transfer Instruction Exists

The AMM program exposes exactly four public instructions (from `lib.rs`):
1. `initialize_admin` — sets admin once
2. `burn_admin` — zeroes admin irreversibly
3. `initialize_pool` — creates pools (requires admin signature)
4. `swap_sol_pool` — executes swaps (CPI-gated, no admin involvement)

There is **no** `update_admin`, `set_admin`, `transfer_admin`, or any other instruction that modifies the `admin` field after initialization. The `AdminConfig` account is only ever `mut` in `initialize_admin` (via `init`) and `burn_admin`.

### Pool Creation Gating

`initialize_pool` enforces admin authorization via:

```rust
#[account(
    seeds = [ADMIN_SEED],
    bump = admin_config.bump,
    has_one = admin @ AmmError::Unauthorized,
)]
pub admin_config: Account<'info, AdminConfig>,

pub admin: Signer<'info>,
```

The `has_one = admin` constraint requires the `admin` signer to match `admin_config.admin` exactly.

### Intentional Delegation at Init Time

The `initialize_admin` instruction accepts an `admin: Pubkey` argument that can differ from the signing `authority` (the upgrade authority). This is by design — the deployer can delegate pool creation to a multisig or operational key at initialization time. This is a one-shot delegation, not an ongoing capability.

## Conclusion

**AdminConfig.admin cannot be updated after initialization.** The only mutation is `burn_admin`, which irreversibly zeroes the field. The admin lifecycle is:

1. **Set once** by upgrade authority during `initialize_admin`
2. **Used** for pool creation via `has_one` check in `initialize_pool`
3. **Burned** (optional, irreversible) via `burn_admin`

There is no authority rotation, no delegation after init, and no escalation path. The initial delegation (deployer sets a different admin pubkey) is intentional and documented. This is the strongest admin pattern in the codebase.

**H124 status: ADDRESSED — no vulnerability present.**
