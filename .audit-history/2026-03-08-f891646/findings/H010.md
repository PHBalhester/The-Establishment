# H010: Graduation MEV Bundle -- Atomic Theft of ~2000 SOL

**Status:** CONFIRMED
**Severity:** CRITICAL
**Category:** Access Control, Timing, State Machine
**Confidence:** HIGH
**Origin:** Novel (ARCHITECTURE.md novel observation #1)

---

## Summary

An attacker can bundle `prepare_transition` and two `withdraw_graduated_sol` calls in a single Solana transaction to atomically graduate both bonding curves and steal all raised SOL (~1000 SOL per curve, ~2000 SOL total). This is possible because: (1) `prepare_transition` accepts ANY `Signer<'info>` with zero authority verification, (2) `withdraw_graduated_sol` likewise accepts any signer AND sends the withdrawn SOL to that signer, and (3) `prepare_transition` writes `CurveStatus::Graduated` to on-chain state within the same transaction, which subsequent instructions in the same TX can immediately read.

## Evidence

### 1. `prepare_transition` accepts any signer

**Location:** `programs/bonding_curve/src/instructions/prepare_transition.rs:18-38`

```rust
pub struct PrepareTransition<'info> {
    /// Protocol authority (deployer). Must sign.
    pub authority: Signer<'info>,  // <-- NO has_one, NO address check, NO ProgramData gate

    #[account(
        mut,
        seeds = [CURVE_SEED, crime_curve_state.token_mint.as_ref()],
        bump = crime_curve_state.bump,
    )]
    pub crime_curve_state: Account<'info, CurveState>,

    #[account(
        mut,
        seeds = [CURVE_SEED, fraud_curve_state.token_mint.as_ref()],
        bump = fraud_curve_state.bump,
    )]
    pub fraud_curve_state: Account<'info, CurveState>,
}
```

The handler sets both curves to `Graduated` if both are `Filled` (lines 53-78). There is no check that `authority` is the deployer.

### 2. `withdraw_graduated_sol` sends SOL to any signer

**Location:** `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs:25-51`

```rust
pub struct WithdrawGraduatedSol<'info> {
    #[account(mut)]
    pub authority: Signer<'info>,  // <-- Receives the SOL. Any signer accepted.

    #[account(
        seeds = [CURVE_SEED, curve_state.token_mint.as_ref()],
        bump = curve_state.bump,
        constraint = curve_state.status == CurveStatus::Graduated @ CurveError::CurveNotGraduated,
    )]
    pub curve_state: Account<'info, CurveState>,

    #[account(
        mut,
        seeds = [CURVE_SOL_VAULT_SEED, curve_state.token_mint.as_ref()],
        bump,
        constraint = sol_vault.key() == curve_state.sol_vault @ CurveError::InvalidStatus,
    )]
    pub sol_vault: UncheckedAccount<'info>,

    pub system_program: Program<'info, System>,
}
```

The handler transfers `vault_balance - rent_exempt_minimum` directly to `authority` via lamport manipulation (line 80-81).

### 3. Same-TX state visibility

Solana's runtime model processes all instructions within a single transaction sequentially against the same in-memory account state. When instruction 1 (`prepare_transition`) writes `CurveStatus::Graduated` to the `CurveState` account, instruction 2 (`withdraw_graduated_sol`) in the same TX reads the updated state and passes the `constraint = curve_state.status == CurveStatus::Graduated` check.

### 4. Transaction fits within compute and size limits

The attack TX contains 3 instructions (optionally 4 with `ComputeBudgetProgram::setComputeUnitLimit`):

| Instruction | Accounts | Est. CU |
|---|---|---|
| `prepare_transition` | 3 (authority, crime_curve, fraud_curve) | ~15,000 |
| `withdraw_graduated_sol` (CRIME) | 4 (authority, crime_curve, crime_sol_vault, system_program) | ~10,000 |
| `withdraw_graduated_sol` (FRAUD) | 4 (authority, fraud_curve, fraud_sol_vault, system_program) | ~10,000 |

**Total unique accounts:** ~7 (authority is shared, system_program is shared, 2 curve states, 2 sol vaults). Well within the 64-account limit.

**Total compute:** ~35,000 CU. Well within the 200,000 default (or 1,400,000 max).

**Transaction size:** ~400 bytes (3 compact instruction headers + account indices + signatures). Well within 1,232-byte limit.

## Attack Scenario

**Prerequisites:** Both CRIME and FRAUD bonding curves have reached `Filled` status (each raised ~1000 SOL from public buyers).

**Steps:**
1. Attacker monitors both `CurveState` PDAs on-chain, waiting for both `status == Filled`.
2. Attacker constructs a single transaction with 3 instructions:
   - IX 1: `prepare_transition` with attacker as `authority`, passing both CurveState PDAs.
   - IX 2: `withdraw_graduated_sol` for CRIME with attacker as `authority`.
   - IX 3: `withdraw_graduated_sol` for FRAUD with attacker as `authority`.
3. Attacker signs and submits the transaction.
4. If the legitimate deployer has not yet called `prepare_transition`, the attacker's TX succeeds atomically:
   - Both curves transition `Filled -> Graduated`.
   - All SOL (minus rent-exempt minimum) from both vaults is transferred to the attacker.
5. The transaction is atomic: if any instruction fails, all three revert. Zero risk to the attacker.

**Attacker profit:** ~2000 SOL (TARGET_SOL = 1,000,000,000,000 lamports = 1000 SOL per curve, minus sell-back refunds and rent).

**Attacker cost:** Transaction fee (~5000 lamports, <0.00001 SOL).

**Race condition:** The attacker races against the legitimate deployer. The attacker can use priority fees and Jito bundles to front-run the deployer's `prepare_transition` call. Since both curves reaching `Filled` is a publicly observable on-chain event, the attacker has full information.

## Root Cause

The bonding curve program (added in v1.2) does not implement any authority verification pattern. The `authority` field on `prepare_transition`, `withdraw_graduated_sol`, and 4 other instructions is a bare `Signer<'info>` with:
- No `has_one` constraint against a stored authority
- No `address = <hardcoded_key>` constraint
- No ProgramData upgrade-authority check (as used by the AMM program's `initialize_admin`)
- No stored admin account anywhere in the bonding curve program

This is in stark contrast to the AMM program in the same codebase, which correctly implements the ProgramData upgrade-authority pattern for admin gating.

The state machine guards (`status == Filled` for prepare_transition, `status == Graduated` for withdraw) are necessary conditions for the legitimate flow, but they do NOT restrict WHO can trigger the transitions -- only WHEN they can be triggered.

## Recommendation

**Option A (Strongest -- matches AMM pattern):** Add a `BondingCurveAdmin` PDA initialized with ProgramData upgrade-authority check, then add `has_one = admin` to all 6 authority-gated instructions.

**Option B (Quick fix):** Add `address = <deployer_pubkey>` constraint on `authority` in all 6 instructions. Less flexible (no key rotation) but immediately effective.

**Option C (Minimum viable):** At minimum, `prepare_transition` and `withdraw_graduated_sol` must verify the authority. These are the highest-impact instructions (SOL theft). The other 4 (`initialize_curve`, `fund_curve`, `start_curve`, `close_token_vault`) have lower impact due to additional state guards.

For all options, the fix should be applied before either bonding curve reaches `Filled` status on mainnet.
