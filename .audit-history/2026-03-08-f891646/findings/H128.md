# H128 — Conversion Vault: Double Initialize Prevention

**Status:** REJECTED (Anchor `init` constraint prevents re-initialization)
**Severity:** N/A
**Confidence:** HIGH
**Target:** `programs/conversion-vault/src/instructions/initialize.rs`

## Hypothesis

The VaultConfig account might be vulnerable to re-initialization, allowing an attacker to overwrite configuration after initial setup.

## Evidence

### Anchor `init` Constraint

From `programs/conversion-vault/src/instructions/initialize.rs:15-21`:
```rust
#[account(
    init,
    payer = payer,
    space = VaultConfig::LEN,
    seeds = [VAULT_CONFIG_SEED],
    bump,
)]
pub vault_config: Account<'info, VaultConfig>,
```

The `init` constraint performs three operations atomically:
1. **Allocates** the account via `system_program::create_account`
2. **Sets the owner** to the current program
3. **Writes the Anchor discriminator** (8-byte hash of `account:VaultConfig`)

A second call to `initialize` will fail at step 1 because `create_account` requires the target account to have zero lamports and zero data. Since the PDA already exists and is funded, the Solana runtime rejects the transaction with `AccountAlreadyInUse`.

### Same Protection for Token Vaults

All three token vault accounts (vault_crime, vault_fraud, vault_profit) also use `init`:
- `vault_crime`: `seeds = [VAULT_CRIME_SEED, vault_config.key().as_ref()]` (line 26-33)
- `vault_fraud`: `seeds = [VAULT_FRAUD_SEED, vault_config.key().as_ref()]` (line 36-46)
- `vault_profit`: `seeds = [VAULT_PROFIT_SEED, vault_config.key().as_ref()]` (line 49-58)

All four PDA accounts are one-shot init — re-calling initialize will fail on every account.

### Mint Validation

The initialize instruction also validates mints against hardcoded addresses (non-localnet builds):
```rust
constraint = cfg!(feature = "localnet") || crime_mint.key() == constants::crime_mint()
```

This prevents initialization with incorrect mints even on the first call.

### Architecture Confirmation

From `.audit/ARCHITECTURE.md` Section 4:
> VaultConfig | ConvVault | `[b"vault_config"]` | bump only

And Trust Tier 6:
> Conversion Vault, Epoch Init, Staking Init: No stored admin. One-shot init, then autonomous.

The architecture correctly classifies this as a one-shot initialization with no admin authority.

### No `init_if_needed`

The instruction uses `init` (not `init_if_needed`), which is the secure pattern. `init_if_needed` would allow re-entry if the account were somehow closed, but `init` strictly requires the account to not exist.

## Conclusion

Anchor's `init` constraint provides strong double-initialization prevention at the Solana runtime level. The VaultConfig PDA can only be created once — any subsequent call fails with `AccountAlreadyInUse`. All four accounts in the initialize instruction (config + 3 token vaults) share this protection.
