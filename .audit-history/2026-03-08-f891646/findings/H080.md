# H080 -- Bonding Curve State Transition Ordering

**Status:** VERIFIED -- No skip-state possible
**Severity:** Informational (confirmed secure)
**Target:** All bonding curve instructions (`programs/bonding_curve/src/instructions/`)

---

## State Machine Definition

```
Initialized -> Active       (start_curve)
Active      -> Filled       (purchase: tokens_sold >= TARGET_TOKENS)
Active      -> Failed       (mark_failed: slot > deadline + grace)
Filled      -> Graduated    (prepare_transition: both curves Filled)
```

Terminal states: **Graduated**, **Failed**.

---

## Per-Instruction State Checks

| Instruction | Required Status | Enforcement Mechanism | Transition |
|---|---|---|---|
| `initialize_curve` | N/A (init) | Anchor `init` constraint (PDA unique) | -> Initialized |
| `fund_curve` | Initialized | `constraint = curve_state.status == CurveStatus::Initialized` | None |
| `start_curve` | Initialized | `constraint = curve_state.status == CurveStatus::Initialized` | -> Active |
| `purchase` | Active | `constraint = curve_state.status == CurveStatus::Active` | -> Filled (conditional) |
| `sell` | Active | `constraint = curve_state.status == CurveStatus::Active` + handler double-check | None |
| `mark_failed` | Active | Handler: `require!(curve.status == CurveStatus::Active)` | -> Failed |
| `prepare_transition` | Filled (both curves) | Handler: `require!(crime == Filled)` + `require!(fraud == Filled)` | -> Graduated |
| `withdraw_graduated_sol` | Graduated | `constraint = curve_state.status == CurveStatus::Graduated` | None |
| `close_token_vault` | Graduated | `constraint = curve_state.status == CurveStatus::Graduated` | None |
| `distribute_tax_escrow` | Graduated | Handler: `require!(status == Graduated)` | None |
| `consolidate_for_refund` | Failed (or Filled+partner Failed) | Handler: `require!(is_refund_eligible(partner))` | None (sets flag) |
| `claim_refund` | Failed (or Filled+partner Failed) | Handler: `require!(is_refund_eligible(partner))` | None (updates tokens_sold) |

## Skip-State Analysis

### Can Initialized skip to Filled?
**No.** Only `purchase` can set status to Filled, and `purchase` requires `status == Active`. There is no instruction that transitions directly from Initialized to Filled.

### Can Initialized skip to Graduated?
**No.** `prepare_transition` requires both curves to be Filled. Getting to Filled requires Active, which requires Initialized -> Active via `start_curve`.

### Can Active skip to Graduated?
**No.** `prepare_transition` requires Filled, not Active. Only `purchase` transitions Active -> Filled.

### Can Filled skip to Failed?
**No.** `mark_failed` explicitly requires `status == Active`. A Filled curve cannot be marked failed. However, `is_refund_eligible()` correctly handles the case where a Filled curve's partner fails -- the Filled curve becomes refund-eligible without transitioning to Failed itself.

### Can Failed revert to Active?
**No.** No instruction sets status from Failed to any other state. Failed is terminal.

### Can Graduated revert?
**No.** No instruction sets status from Graduated to any other state. Graduated is terminal.

### Double-initialization?
**No.** `initialize_curve` uses Anchor `init` constraint which creates the PDA. Re-calling with the same mint would fail because the PDA already exists.

## Identified Concern (Cross-Reference)

The `authority` signer on `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, and `close_token_vault` is ANY signer (no on-chain authority verification). This is the known H-series finding from the architecture document (Trust Tier 4). It does not affect state transition ordering but means unauthorized actors could trigger valid transitions.

Specifically:
- Anyone can call `start_curve` (Initialized -> Active) if vault is funded
- Anyone can call `prepare_transition` (Filled -> Graduated) if both curves are Filled
- Anyone can call `withdraw_graduated_sol` and receive the SOL

The state machine ordering itself is correct. The access control gap is tracked separately.

---

## Verdict

**CONFIRMED SECURE** for state transition ordering. Every instruction enforces its required precondition status via Anchor constraints or handler-level `require!()`. No path allows skipping states. Terminal states (Graduated, Failed) are irreversible. The `is_refund_eligible()` helper correctly handles the cross-curve compound state (Filled + partner Failed).
