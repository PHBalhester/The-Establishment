# S003 -- Staking Escrow: deposit_rewards Failure Cascade

**Audit:** sos-002 | **Severity:** Supplemental (HIGH impact if triggered)
**Status:** CONFIRMED -- escrow destruction bricks all swaps
**Derived from:** Novel Attack Surface #5 (ARCHITECTURE.md Section 9), H012 escrow rent findings

---

## Hypothesis

If the staking escrow vault is destroyed (drained below rent-exempt minimum), the Tax Program's `deposit_rewards` CPI will fail, and since the CPI is not wrapped in error handling, this will cause ALL buy and sell swaps to revert.

## Investigation

### Escrow Architecture

The escrow vault is:
- A 0-byte PDA (`space = 0`) owned by the Staking Program (created via `init` in `initialize_stake_pool.rs`, line 49-55).
- Seeds: `[ESCROW_VAULT_SEED]`, derived from the Staking Program.
- Holds native SOL (lamports only, no token account data).
- Rent-exempt minimum for 0-byte account: **890,880 lamports** (~0.00089 SOL).

### Destruction Vector: claim.rs Lamport Drain

In `claim.rs` (lines 140-145):

```rust
**ctx.accounts.escrow_vault.try_borrow_mut_lamports()? = ctx
    .accounts
    .escrow_vault
    .lamports()
    .checked_sub(rewards_to_claim)
    .ok_or(StakingError::Underflow)?;
```

The claim handler:
1. Checks `escrow_balance >= rewards_to_claim` (line 103) -- but only checks against the claimed amount.
2. **Does NOT check** that `escrow_balance - rewards_to_claim >= rent_exempt_minimum`.
3. Uses direct lamport manipulation (not system transfer), bypassing the runtime's rent-exempt enforcement for system-owned accounts.

Since the escrow is **program-owned** (owned by Staking Program, not System Program), the Solana runtime allows its lamports to be set to any value, including below rent-exempt minimum. Program-owned accounts can be drained to zero.

### Critical Question: Can lamports reach zero?

If `pending_rewards == escrow_balance` (all deposited rewards are claimed), the escrow reaches its rent-exempt minimum only if rewards exactly match deposits. But the escrow was initialized with `space = 0` via `init`, which funds it with rent-exempt lamports. These rent lamports ARE included in `escrow_vault.lamports()`.

The reconciliation check in `deposit_rewards.rs` (line 100):
```rust
require!(
    ctx.accounts.escrow_vault.lamports() >= pool.pending_rewards,
    StakingError::InsufficientEscrowBalance
);
```

This checks `lamports >= pending_rewards`. If all pending rewards are claimed (`pending_rewards` decreased via `update_rewards` -> `distribute` flow before claim), and the initial rent lamports are included in the balance, the escrow can't be drained below rent by `claim` alone... **unless** the claim amount exceeds what was deposited because `rewards_per_token_stored` rounding creates a claim amount that eats into rent lamports.

Actually, re-reading `claim.rs` more carefully: the `escrow_balance < rewards_to_claim` check (line 103) will prevent draining to zero IF the reward calculation doesn't exceed available balance. But the check only prevents underflow -- it doesn't preserve rent-exempt minimum.

**Scenario:** If `rewards_to_claim = escrow_balance` (exact balance claimed), the escrow reaches 0 lamports. The runtime will garbage-collect a 0-lamport program-owned account at the end of the transaction.

### Cascade Analysis: What Happens After Destruction

**Tax Program swap_sol_buy.rs** (lines 126-176):

```rust
if staking_portion > 0 {
    // Step 1: system_instruction::transfer to staking_escrow
    invoke_signed(&system_instruction::transfer(..., staking_portion), ...)?;

    // Step 2: CPI to deposit_rewards
    invoke_signed(&deposit_ix, ...)?;
}
```

If the escrow PDA is garbage-collected:

1. **system_instruction::transfer** to a non-existent account: The System Program will **create** a new account with `staking_portion` lamports, owned by the System Program. This succeeds.

2. **deposit_rewards CPI**: The Staking Program's `DepositRewards` account struct validates `escrow_vault` with `seeds = [ESCROW_VAULT_SEED], bump`. Anchor will derive the PDA and check `escrow_vault.key() == derived_key`. The key matches (same seeds). But now:
   - The account is owned by System Program (from step 1's transfer).
   - The `deposit_rewards` handler calls `ctx.accounts.escrow_vault.lamports()` -- this works (it's just reading lamports).
   - The reconciliation check `lamports() >= pending_rewards` -- depends on accumulated pending vs current balance.
   - **Actually, the PDA validation in Anchor for `AccountInfo` (unchecked) only verifies the address derivation, not ownership.** The escrow_vault in DepositRewards is `AccountInfo<'info>` (line 66), not `Account<>` or `SystemAccount<>`. So ownership is NOT checked.

**Wait** -- re-reading step 1 more carefully: `system_instruction::transfer` sends SOL to an existing account or fails if the account doesn't exist. It does NOT create accounts. If escrow_vault doesn't exist, the transfer will fail with "account not found" or similar.

Actually, on Solana, `system_instruction::transfer` to any valid address works -- it creates the account if needed with data_len=0, owned by System Program. But in `swap_sol_buy.rs`, the transfer is from the **user** (not a PDA), using `invoke_signed` with empty signer seeds `&[]` (line 139). This is actually just `invoke` semantics. The user signs.

So the transfer creates a new system-owned account at the escrow PDA address. Then `deposit_rewards` CPI runs. The PDA address matches, the account now has lamports, the balance check passes. **No immediate failure.**

**But in swap_sol_sell.rs** (lines 318-331), the transfer is from `swap_authority` PDA:
```rust
invoke_signed(
    &system_instruction::transfer(
        ctx.accounts.swap_authority.key,
        ctx.accounts.staking_escrow.key,
        staking_portion,
    ),
    ...
    &[swap_authority_seeds],
)?;
```

Same result: creates/credits a system-owned account at the escrow address.

### Revised Cascade: Insidious State Corruption

If the escrow is destroyed and recreated by system_instruction::transfer:

1. The new account is **owned by System Program**, not Staking Program.
2. `deposit_rewards` succeeds (AccountInfo doesn't check owner).
3. `pending_rewards` counter increments in StakePool.
4. When a user calls `claim`, the claim handler does `try_borrow_mut_lamports()` on the escrow.
5. **The Staking Program cannot modify lamports of a System-owned account.** Only the account's owner can modify lamports. The claim will fail with a runtime error.

**This is worse than a hard revert** -- swaps succeed (depositing rewards that appear claimable), but claims are permanently bricked. Users see rewards accumulating but can never claim them.

### Can the escrow actually be destroyed?

The `claim.rs` handler does:
```rust
**ctx.accounts.escrow_vault.try_borrow_mut_lamports()? = balance - rewards_to_claim
```

If `rewards_to_claim < balance`, the escrow retains lamports. To drain to zero, `rewards_to_claim` must equal `balance`. This requires:
- `rewards_earned` (from `update_rewards`) to exactly equal escrow lamports.
- The rent-exempt lamports (~890,880) would need to be included in a claim.

The `update_rewards` function computes rewards based on `rewards_per_token_stored` and user's staked balance. If `pending_rewards` is distributed, `rewards_per_token_stored` increases. The total claimable rewards across all users = `total_distributed` amount. But `total_distributed` is computed from `pending_rewards`, which is tracked by `deposit_rewards`. So total claimable should equal total deposited (minus rounding).

The escrow balance = initial_rent + total_deposited - total_claimed. If `total_claimed = total_deposited`, the escrow has `initial_rent` remaining. Users cannot claim the initial rent because no rewards are pending.

**But:** if `pending_rewards` somehow includes the rent lamports in its balance check... The reconciliation in `deposit_rewards` checks `lamports() >= pending_rewards`. If pending_rewards is 1000 and lamports is 891,880 (rent + 1000), this passes. Then if all 1000 is claimed, escrow has 890,880 (rent). The user gets their 1000. Escrow is safe.

**Edge case:** What if someone sends additional lamports to the escrow PDA outside the protocol (e.g., airdrop or accidental transfer)? The extra lamports inflate `lamports()` but not `pending_rewards`. This is harmless -- claims cannot exceed `rewards_earned` which derives from `pending_rewards`.

### Revised Conclusion

Under normal protocol operation, the escrow **cannot** be drained below rent-exempt minimum because:
1. Claims are bounded by `rewards_earned`, which derives from `pending_rewards`.
2. `pending_rewards` is bounded by actual deposits.
3. The initial rent lamports are never included in `pending_rewards`.

**However**, there is still a concern: the `claim.rs` handler has no explicit rent-exempt floor check. If a bug in `update_rewards` or a rounding accumulation causes `rewards_earned` to exceed deposited amounts by even a small margin over many epochs, the escrow could be slowly drained.

## Verdict

**CONFIRMED with nuance.** The failure cascade is real and severe, but the trigger condition is harder to reach than initially theorized:

1. **If escrow is destroyed:** Swaps continue (system_instruction::transfer recreates it as system-owned), deposit_rewards CPI succeeds (no owner check on AccountInfo), but all future claims are permanently bricked (staking program cannot modify system-owned lamports).

2. **Trigger difficulty:** Under normal operation, claims cannot exceed deposited amounts, so rent lamports are preserved. The risk is from rounding accumulation in `rewards_per_token_stored` over many epochs with many stakers.

3. **Missing defense-in-depth:** No explicit `escrow_balance - claim_amount >= rent_exempt_minimum` check in `claim.rs`.

## Recommendations

1. **Add rent-exempt floor check in `claim.rs`:**
   ```rust
   let rent_min = Rent::get()?.minimum_balance(0);
   require!(
       escrow_balance.saturating_sub(rewards_to_claim) >= rent_min,
       StakingError::EscrowRentProtection
   );
   ```

2. **Add owner check on escrow_vault in `deposit_rewards.rs`:**
   ```rust
   require!(
       ctx.accounts.escrow_vault.owner == &crate::id(),
       StakingError::InvalidEscrowOwner
   );
   ```
   This would cause a hard revert on swaps if the escrow is destroyed, which is better than silent corruption. The revert signals an emergency requiring protocol intervention rather than silently accumulating unclaimable rewards.

3. **Consider making escrow_vault a `SystemAccount` or adding explicit owner validation** in the Tax Program's account structs as well, so the transfer destination is validated before CPI.
