# H090 -- Bonding Curve: Token Mint Validation

**Status:** SAFE
**Severity:** N/A (confirmed secure)
**File:** `programs/bonding_curve/src/instructions/purchase.rs`

## Hypothesis

Check if the Bonding Curve validates that the `token_mint` account matches the curve's stored `token_mint`. Verify via `has_one` or seeds constraint.

## Evidence

### 1. CurveState PDA Seeds Include token_mint (lines 29-34)

```rust
#[account(
    mut,
    seeds = [CURVE_SEED, token_mint.key().as_ref()],
    bump = curve_state.bump,
    constraint = curve_state.status == CurveStatus::Active @ CurveError::CurveNotActive,
)]
pub curve_state: Account<'info, CurveState>,
```

The `curve_state` PDA is derived from `[b"curve", token_mint.key()]`. This creates a cryptographic binding: the PDA address changes if a different `token_mint` is passed. An attacker cannot pass mint X while loading curve_state for mint Y -- the PDA derivation would produce a different address, and Anchor would reject it because the on-chain account doesn't exist at that address (or has the wrong discriminator).

### 2. Token Vault Validates Against token_mint (lines 50-58)

```rust
#[account(
    mut,
    token::mint = token_mint,
    token::authority = curve_state,
    token::token_program = token_program,
    seeds = [CURVE_TOKEN_VAULT_SEED, token_mint.key().as_ref()],
    bump,
)]
pub token_vault: Box<InterfaceAccount<'info, TokenAccount>>,
```

Triple validation:
- `token::mint = token_mint` -- vault's mint matches the passed mint
- `token::authority = curve_state` -- vault is owned by the correct curve PDA
- Seeds include `token_mint.key()` -- vault PDA is derived from the same mint

### 3. SOL Vault Also Seeded by token_mint (lines 63-68)

```rust
#[account(
    mut,
    seeds = [CURVE_SOL_VAULT_SEED, token_mint.key().as_ref()],
    bump,
)]
pub sol_vault: UncheckedAccount<'info>,
```

### 4. User Token Account Validates Against Mint (lines 39-46)

```rust
#[account(
    init_if_needed,
    payer = user,
    associated_token::mint = token_mint,
    associated_token::authority = user,
    associated_token::token_program = token_program,
)]
pub user_token_account: Box<InterfaceAccount<'info, TokenAccount>>,
```

### 5. CurveState Stores token_mint (state.rs line 81)

The `CurveState` struct stores `token_mint: Pubkey` at field position 2. While `purchase.rs` does not use an explicit `has_one = token_mint` constraint, the seeds-based derivation provides equivalent (stronger) protection -- it is unforgeable at the PDA level.

## Conclusion

Token mint validation is enforced through PDA seed derivation, which is cryptographically stronger than a `has_one` constraint. Every account in the instruction (curve_state, token_vault, sol_vault, user_token_account) is bound to the same `token_mint` either via PDA seeds or `token::mint` constraints. Passing a mismatched mint would produce invalid PDA addresses, causing Anchor deserialization to fail before the handler executes.
