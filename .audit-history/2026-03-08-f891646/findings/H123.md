# H123 -- AMM: PDA Signer for Token Transfers

| Field | Value |
|-------|-------|
| Hypothesis | Pool PDA signer seeds may not match across swap, init, and close, allowing signature failures or cross-pool confusion. |
| Severity | N/A (No finding) |
| Status | **REFUTED** |
| File | `programs/amm/src/instructions/swap_sol_pool.rs` |
| Supporting | `programs/amm/src/instructions/initialize_pool.rs`, `programs/amm/src/constants.rs` |

## Analysis

The hypothesis asks whether the pool PDA seeds are consistent across all instruction contexts (initialize, swap, close/burn). Inconsistent seeds would mean:
- Swap CPI signing would fail (wrong PDA derived)
- Pool state could be substituted across different mint pairs
- Vault authority mismatches during transfers

### Seed Definitions (constants.rs)

```rust
pub const POOL_SEED: &[u8] = b"pool";                    // line 30
pub const SWAP_AUTHORITY_SEED: &[u8] = b"swap_authority"; // line 5
```

### Pool PDA -- Initialize (initialize_pool.rs line 222)

```rust
seeds = [POOL_SEED, mint_a.key().as_ref(), mint_b.key().as_ref()],
bump
```

The pool is `init`-ed with these seeds; bump is stored in `PoolState`.

### Pool PDA -- Swap Constraint (swap_sol_pool.rs lines 377-381)

```rust
seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
bump = pool.bump,
```

Uses stored `pool.mint_a` / `pool.mint_b` and the stored bump. This is **self-referential validation**: the deserialized pool state provides the mint keys used to re-derive the PDA. If anyone passes a different pool account, the seeds won't match and deserialization fails.

### Pool PDA -- Swap Signer Seeds (swap_sol_pool.rs lines 193-197)

```rust
let mint_a_bytes = mint_a_key.to_bytes();    // captured at line 70 before mutation
let mint_b_bytes = mint_b_key.to_bytes();    // captured at line 71
let bump_bytes = [pool_bump];                // captured at line 72
let pool_seeds: &[&[u8]] = &[POOL_SEED, &mint_a_bytes, &mint_b_bytes, &bump_bytes];
```

These are used for `invoke_signed` on vault-to-user output transfers. The values are captured **before** any mutable access to the pool (lines 70-77), preventing borrow conflicts and ensuring the signer seeds match the pool PDA.

### Swap Authority PDA (swap_sol_pool.rs lines 367-371)

```rust
seeds = [SWAP_AUTHORITY_SEED],
bump,
seeds::program = TAX_PROGRAM_ID,
```

This is a separate PDA -- the swap_authority is derived from the **Tax Program**, not the AMM. It gates CPI access so only the Tax Program can invoke swaps. This is correctly distinct from the pool PDA.

### Vault Authority

In `initialize_pool.rs` (lines 233-238), vaults are created with:
```rust
token::authority = pool,
```

The pool PDA is the vault authority. During swaps, the pool PDA signs output transfers using the same seeds. This is consistent.

### Cross-Validation: Vault and Mint Constraints

The swap instruction validates:
- `vault_a.key() == pool.vault_a` (line 389)
- `vault_b.key() == pool.vault_b` (line 395)
- `mint_a.key() == pool.mint_a` (line 401)
- `mint_b.key() == pool.mint_b` (line 405)
- `token_program_a.key() == pool.token_program_a` (line 423)
- `token_program_b.key() == pool.token_program_b` (line 427)

All six account references are validated against stored pool state, preventing any substitution attack.

### Reentrancy Guard

The pool has a `locked` boolean (line 84 sets it, line 322 clears it). The Anchor constraint `!pool.locked` (line 381) prevents reentrancy via Transfer Hook callbacks.

## Verdict

**SECURE.** Pool PDA seeds are consistent across initialize and swap. The seeds `[b"pool", mint_a, mint_b, bump]` are used identically in all contexts. Vault authority matches pool PDA. Swap authority is correctly a separate PDA gated by `seeds::program = TAX_PROGRAM_ID`. All account references are validated against stored pool state.
