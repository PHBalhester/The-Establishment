# H107 -- Staking: UserStake Account Size

**Status:** REJECTED (Not a vulnerability)
**Severity:** Informational
**Component:** `programs/staking/src/state/user_stake.rs`

## Hypothesis

UserStake account size may be insufficient or lack padding for future fields.

## Analysis

### Size Verification

The UserStake struct contains the following fields with a manually computed `LEN`:

| Field | Type | Size |
|-------|------|------|
| Discriminator | [u8; 8] | 8 |
| owner | Pubkey | 32 |
| staked_balance | u64 | 8 |
| rewards_per_token_paid | u128 | 16 |
| rewards_earned | u64 | 8 |
| total_claimed | u64 | 8 |
| first_stake_slot | u64 | 8 |
| last_update_slot | u64 | 8 |
| last_claim_ts | i64 | 8 |
| bump | u8 | 1 |
| **Total** | | **105** |

The `LEN` constant is `8 + 32 + 8 + 16 + 8 + 8 + 8 + 8 + 8 + 1 = 105`, which exactly matches the field layout. A unit test (`test_user_stake_len`) asserts this value at compile time.

The account is allocated with `space = UserStake::LEN` in `instructions/stake.rs` via `init_if_needed`, confirming on-chain allocation matches the constant.

### Padding Assessment

There is **zero padding** reserved for future fields. This is a design observation, not a vulnerability:

1. **Anchor accounts cannot be resized after init** -- `init_if_needed` allocates exactly `LEN` bytes. Adding fields later would require a migration (new PDA seeds or `realloc`).
2. **No unused fields or reserved bytes** -- every byte is accounted for.
3. **This is standard practice** for Anchor programs -- the Synthetix-style staking model used here is well-understood, and the field set (owner, balance, checkpoint, rewards, analytics, cooldown) is complete for the current design.

### Risk if Fields Need Adding

If a future version needs additional fields (e.g., lock duration, boost multiplier), the options are:
- `realloc` with payer in a migration instruction
- New account type with version-tagged seeds

Both are standard Solana upgrade patterns and do not constitute a current vulnerability.

## Verdict

**No finding.** The account size is correct. All fields are properly accounted for. The lack of padding is a standard Anchor pattern, not a security issue. Future extensibility would use `realloc` or versioned accounts.
