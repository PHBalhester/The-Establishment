# H003 — Bonding Curve Authority: initialize_curve Impersonation

**Status:** POTENTIAL
**Severity:** MEDIUM
**Category:** Access Control
**Confidence:** HIGH

## Summary

The `initialize_curve` instruction accepts any `Signer<'info>` as `authority` with no on-chain verification (no `has_one`, no ProgramData upgrade-authority check, no hardcoded address constraint). However, the CurveState PDA is seeded by `[b"curve", token_mint]` with an `init` constraint, meaning each mint can only have one curve. Combined with a hardcoded mint allowlist (CRIME and FRAUD addresses via feature-gated `crime_mint()` / `fraud_mint()`), direct impersonation for production mints is limited to a front-running race during deployment.

## Evidence

**PDA seeds (unique per mint):** `programs/bonding_curve/src/instructions/initialize_curve.rs:22-28`
```rust
#[account(
    init,
    payer = authority,
    space = CurveState::LEN,
    seeds = [CURVE_SEED, token_mint.key().as_ref()],
    bump,
)]
pub curve_state: Account<'info, CurveState>,
```

**Authority — bare signer, no verification:** `initialize_curve.rs:17-18`
```rust
#[account(mut)]
pub authority: Signer<'info>,
```

**Mint allowlist constraint:** `initialize_curve.rs:73-79`
```rust
constraint = cfg!(feature = "localnet")
    || token_mint.key() == crime_mint()
    || token_mint.key() == fraud_mint()
    @ crate::error::CurveError::InvalidStatus,
```

**No authority stored in CurveState:** The handler at lines 93-120 stores token, mint, vault keys, bumps, and counters — but never stores the `authority` pubkey for future verification.

## Attack Scenario

1. Attacker monitors the mempool for the protocol deployment transaction that calls `initialize_curve` for CRIME or FRAUD.
2. Attacker submits a competing `initialize_curve` transaction with the same `token_mint` but with the attacker's wallet as `authority` (and as payer).
3. If the attacker's transaction lands first, the CurveState PDA is created with the attacker as payer. The Anchor `init` constraint prevents the legitimate deployer from re-initializing (PDA already exists).
4. The attacker now controls the curve lifecycle: they paid rent, and downstream instructions (`fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`) also accept any signer as `authority`.
5. If the curve is somehow funded and graduates, the attacker can call `withdraw_graduated_sol` to steal all raised SOL.

**Mitigating factors that reduce severity from CRITICAL to MEDIUM:**
- The mint allowlist (`crime_mint()` / `fraud_mint()`) prevents attackers from creating curves for arbitrary mints — only the two known mints are valid.
- PDA uniqueness (`init` constraint) means the attacker must front-run; they cannot create a parallel curve for the same mint.
- The attacker front-running `initialize_curve` alone is not immediately profitable — they also need the curve to be funded (460M tokens deposited via `fund_curve`), which requires holding the tokens. The real damage comes from the fact that all subsequent admin instructions (`withdraw_graduated_sol`) are equally permissionless, so the front-run is a prerequisite for a larger attack chain.
- Deployment can be done via Jito bundles or priority fees to reduce front-running risk (operational mitigation, not on-chain).

## Root Cause

The bonding curve program (added in v1.2) does not implement any authority verification pattern. Unlike the AMM program which uses ProgramData upgrade-authority gating via `initialize_admin`, the bonding curve accepts a bare `Signer<'info>` named `authority` with no constraints. The authority is not stored in `CurveState`, so even if it were checked at init time, downstream instructions could not reference it.

## Recommendation

Implement the same ProgramData upgrade-authority pattern used by the AMM's `initialize_admin`:

1. Add a `BondingCurveAdmin` PDA (or reuse the AMM's AdminConfig) that stores the authorized admin pubkey, gated by ProgramData upgrade authority at creation.
2. Add `has_one = authority` constraint on `CurveState` (after storing `authority` in the state), OR add `address = bonding_curve_admin.admin` constraint on the `authority` signer.
3. Apply this to all 6 admin instructions: `initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault`.

Alternatively, as a minimal fix, add a hardcoded deployer pubkey constraint:
```rust
#[account(mut, address = deployer_pubkey() @ CurveError::Unauthorized)]
pub authority: Signer<'info>,
```
