# H079 -- AMM Reserve Ordering (mint_a < mint_b)

**Status:** Not Vulnerable
**Severity:** N/A (Informational)
**Confidence:** High

## Summary

Canonical mint ordering (mint_a < mint_b by raw pubkey bytes) is enforced at pool creation and structurally embedded in PDA seeds, making duplicate or misordered pools impossible. The implementation is correct.

## Evidence

### 1. Explicit Ordering Check (initialize_pool.rs:35-38)

```rust
require!(
    ctx.accounts.mint_a.key() < ctx.accounts.mint_b.key(),
    AmmError::MintsNotCanonicallyOrdered
);
```

This is the first validation in the handler, rejecting any pool creation where mints are not in canonical order.

### 2. Duplicate Mint Guard (initialize_pool.rs:41-44)

```rust
require!(
    ctx.accounts.mint_a.key() != ctx.accounts.mint_b.key(),
    AmmError::DuplicateMints
);
```

Redundant with the `<` check above (if a < b then a != b), but provides a clearer error message. Defense-in-depth.

### 3. PDA Seeds Enforce Uniqueness (initialize_pool.rs:218-224)

```rust
#[account(
    init,
    payer = payer,
    space = 8 + PoolState::INIT_SPACE,
    seeds = [POOL_SEED, mint_a.key().as_ref(), mint_b.key().as_ref()],
    bump
)]
pub pool: Account<'info, PoolState>,
```

The pool PDA is derived from `[b"pool", mint_a, mint_b]`. Because:
1. The ordering check guarantees mint_a < mint_b.
2. Anchor's `init` constraint fails if the PDA already exists (account collision).

This means there can be exactly one pool per unordered mint pair. A second call with the same mints would derive the same PDA and fail with `AccountAlreadyInUse`.

### 4. Vault PDAs Are Pool-Scoped (initialize_pool.rs:230-252)

Vault A seeds: `[b"vault", pool.key(), b"a"]`
Vault B seeds: `[b"vault", pool.key(), b"b"]`

Vaults are derived from the pool PDA, not from mints directly. This means vault addresses are deterministic and tied to the canonical pool. No vault confusion is possible.

### 5. Swap Path Validates Pool Seeds (swap_sol_pool.rs:376-382)

```rust
#[account(
    mut,
    seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
    bump = pool.bump,
    constraint = pool.initialized @ AmmError::PoolNotInitialized,
    constraint = !pool.locked @ AmmError::PoolLocked,
)]
pub pool: Account<'info, PoolState>,
```

The swap instruction re-derives the PDA from the stored `pool.mint_a` and `pool.mint_b`. This means:
- A pool with incorrectly ordered mints would have a different PDA and fail deserialization.
- The stored mint ordering is immutable after init.

### 6. Mint Validation on Swap (swap_sol_pool.rs:401-406)

```rust
#[account(constraint = mint_a.key() == pool.mint_a @ AmmError::InvalidMint)]
pub mint_a: InterfaceAccount<'info, Mint>,
#[account(constraint = mint_b.key() == pool.mint_b @ AmmError::InvalidMint)]
pub mint_b: InterfaceAccount<'info, Mint>,
```

Mints passed to swap must match what was stored at pool creation. No mint substitution is possible.

### 7. Known Gotcha (Memory note -- not a vulnerability)

Per project memory: "PROFIT(0x76) < CRIME(0xD1) in raw bytes, so AMM stores CRIME/PROFIT pool as mint_a=PROFIT, mint_b=CRIME." The `is_reversed` detection in swap_profit_buy/sell handles this correctly. SOL pools are safe because NATIVE_MINT(0x06) < everything.

## Recommendation

No action required. The canonical ordering is correctly enforced at creation, embedded in PDA derivation, and validated on every swap. This is a well-implemented pattern.
