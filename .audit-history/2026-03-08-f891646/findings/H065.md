# H065 — VRF Anti-Reroll Bypass

| Field | Value |
|-------|-------|
| **Hypothesis** | Attacker creates multiple Switchboard randomness accounts, triggers epoch transition, observes unfavorable VRF outcomes, and retries with a different account to cherry-pick favorable tax rates or Carnage decisions. |
| **Severity** | N/A — Finding is INVALID |
| **Status** | **REJECTED** |
| **Files** | `programs/epoch-program/src/instructions/trigger_epoch_transition.rs`, `programs/epoch-program/src/instructions/consume_randomness.rs`, `programs/epoch-program/src/instructions/retry_epoch_vrf.rs`, `programs/epoch-program/src/state/epoch_state.rs` |

## Analysis

### Attack Vector Examined

The hypothesis is that an attacker could:
1. Create randomness account A, trigger epoch transition (binding A).
2. Observe the revealed VRF bytes from A (unfavorable outcome).
3. Abandon A, create randomness account B, and call `trigger_epoch_transition` again to bind B instead.
4. Repeat until a favorable outcome is obtained.

### Defense 1: Single Pending VRF Lock (`vrf_pending` flag)

`trigger_epoch_transition` at line 149 enforces:

```rust
require!(!epoch_state.vrf_pending, EpochError::VrfAlreadyPending);
```

Once `trigger_epoch_transition` succeeds, `vrf_pending` is set to `true` (line 181). No second call to `trigger_epoch_transition` can succeed until `vrf_pending` is cleared. The flag is only cleared in two places:
- `consume_randomness` line 206: `epoch_state.vrf_pending = false` (after successfully reading the bound randomness).
- `retry_epoch_vrf`: does NOT clear `vrf_pending` — it only overwrites `pending_randomness_account` and `vrf_request_slot`.

This means an attacker cannot call `trigger_epoch_transition` a second time to bind a different account. The VRF pending state is a lock.

### Defense 2: Anti-Reroll Binding in `consume_randomness`

`consume_randomness` at lines 154-157 enforces:

```rust
require!(
    ctx.accounts.randomness_account.key() == epoch_state.pending_randomness_account,
    EpochError::RandomnessAccountMismatch
);
```

The randomness account passed to `consume_randomness` MUST be the exact same account that was bound during `trigger_epoch_transition` (stored at line 186). An attacker cannot substitute a different randomness account at reveal time.

### Defense 3: Freshness Prevents Pre-Generation

`trigger_epoch_transition` at line 166 enforces:

```rust
require!(slot_diff <= 1, EpochError::RandomnessExpired);
```

The randomness account's `seed_slot` must be within 1 slot of the current slot. This prevents an attacker from pre-generating many randomness accounts and selecting the best one — they can only use a freshly created account.

### Defense 4: Not-Yet-Revealed Check

`trigger_epoch_transition` at lines 170-173 enforces that the randomness has NOT been revealed yet at commit time. This prevents an attacker from creating a randomness account, waiting for the oracle to reveal it, observing the VRF bytes, and only then committing if favorable.

### Retry Path Analysis (`retry_epoch_vrf`)

The `retry_epoch_vrf` instruction does allow replacing the bound randomness account, which is the closest thing to a "reroll." However, it is gated by a 300-slot timeout (line 72):

```rust
require!(
    elapsed_slots > VRF_TIMEOUT_SLOTS,
    EpochError::VrfTimeoutNotElapsed
);
```

An attacker would need to:
1. Trigger epoch transition, binding account A.
2. Deliberately NOT reveal account A (or collude with the oracle to withhold revelation).
3. Wait 300+ slots (~2 minutes).
4. Call `retry_epoch_vrf` with fresh account B.
5. Reveal B and call `consume_randomness`.

**Why this does not constitute a reroll bypass:**
- The attacker never sees A's VRF output before deciding to abandon it. The oracle reveals only after the commit, and if the oracle does reveal A, any third party can call `consume_randomness` with A before the 300-slot timeout expires — the attacker cannot prevent this.
- If the oracle does NOT reveal A (oracle failure), the attacker also has no information about what A's outcome would have been.
- The fresh account B is subject to the same freshness check (seed_slot within 1 slot) and not-yet-revealed check. The attacker gains no predictive advantage.
- Infinite retries are possible in theory (no retry cap), but each retry requires a genuine oracle failure (300-slot window without reveal) and provides no information about future outcomes.

### Edge Case: Colluding with Oracle

If the attacker controls or colludes with the Switchboard oracle:
- They could withhold revelation of unfavorable results and force retries.
- However, this is outside the protocol's threat model (Assumption A-6: "Switchboard oracle is honest").
- Switchboard On-Demand uses decentralized oracle selection; the randomness account is assigned to a specific oracle, and the protocol cannot choose which oracle gets assigned.

## Verdict

**REJECTED — Anti-reroll protection is correctly and comprehensively implemented.**

The four-layer defense (VRF pending lock, account binding, freshness check, not-yet-revealed check) prevents all variations of the reroll attack under the protocol's trust assumptions. The retry path exists only for genuine oracle failure recovery and does not leak information that would enable cherry-picking. This confirms INV-6 from the architecture document.

No code changes required.
