# H109 -- Tax Program: Swap with Locked Pool

**Status:** REJECTED (Not a vulnerability -- graceful failure via CPI error propagation)
**Severity:** Informational
**Component:** `programs/tax-program/src/instructions/swap_sol_buy.rs`

## Hypothesis

If the AMM pool's `locked` reentrancy guard is active when the Tax Program issues a swap CPI, the transaction may fail ungracefully or leave state in an inconsistent condition.

## Analysis

### AMM Lock Mechanism

The AMM's `swap_sol_pool` instruction has an Anchor constraint on the `pool` account:

```rust
constraint = !pool.locked @ AmmError::PoolLocked,
```

If `pool.locked == true`, the AMM instruction fails immediately at account validation (before any state mutation), returning `AmmError::PoolLocked`.

### Tax Program CPI Flow

In `swap_sol_buy.rs`, the Tax Program:
1. Distributes tax via native SOL transfers (steps 5a-5c)
2. Calls AMM `swap_sol_pool` via `invoke_signed` (step 6g)

If the AMM CPI fails (including due to `PoolLocked`), `invoke_signed` returns an error which propagates via `?` to the Tax Program handler, causing the **entire transaction to revert atomically**.

### Atomicity Guarantee

Solana transactions are atomic -- if any instruction (including CPI) fails, all state changes in the entire transaction are rolled back. This means:
- The SOL transfers in steps 5a-5c (tax distribution) are reverted
- The user's SOL balance is restored
- No partial state exists

### Can PoolLocked Actually Trigger?

The `locked` field is a reentrancy guard set to `true` at the start of `swap_sol_pool` and cleared at the end. For it to be `true` when Tax calls AMM:
- A concurrent CPI within the same transaction would need to already be executing `swap_sol_pool` on the same pool account
- Solana's runtime prevents this: you cannot borrow the same account mutably twice in a CPI chain (the runtime enforces exclusive mut borrows)

So `PoolLocked` would only fire if someone crafted a transaction that passed the same pool account to two different Tax swap instructions within a single transaction. Solana's account model prevents this because the pool is `mut` in both paths.

### Pre-CPI State Reading

The Tax Program reads pool reserves via `read_pool_reserves()` (raw byte read at offsets 137-153) **before** the AMM CPI. This reads the `locked` field's neighbor bytes (reserves) but does not check `locked` itself. This is fine because:
- The reserves are read for the output floor calculation, not for lock checking
- The AMM's own constraint handles lock checking
- If the pool were somehow locked, the reserves would still be valid data (the lock doesn't invalidate reserve values)

## Verdict

**No finding.** The Tax Program handles a locked pool gracefully through Solana's atomic transaction model. If the AMM rejects the CPI due to `PoolLocked`, the entire transaction reverts cleanly with no partial state. In practice, `PoolLocked` cannot trigger in normal operation due to Solana's exclusive mutable borrow rules on account data within a transaction.
