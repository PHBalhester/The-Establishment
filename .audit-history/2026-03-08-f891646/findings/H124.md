## Analysis

The hypothesis asks whether token accounting remains balanced after graduation: `tokens_sold + remaining_in_vault == TARGET_TOKENS (460M)`.

### Token Flow Lifecycle

1. **fund_curve**: Deposits exactly `TARGET_TOKENS` (460M with 6 decimals = 460,000,000,000,000) into the token vault (fund_curve.rs line 80).

2. **purchase**: Transfers tokens from vault to buyer. The amount is computed by `calculate_tokens_out()` and capped to `remaining = TARGET_TOKENS - curve.tokens_sold` (purchase.rs lines 146-147). State update: `curve.tokens_sold += actual_tokens` (line 272-274).

3. **sell**: Transfers tokens from seller back to vault. State update: `curve.tokens_sold = x2` where `x2 = x1 - tokens_to_sell` (sell.rs line 254). The sell instruction includes an explicit solvency assertion (sell.rs lines 284-294) that verifies `sol_vault.lamports() >= integral(0, tokens_sold)`.

4. **Filled transition**: When `tokens_sold >= TARGET_TOKENS` (purchase.rs line 285), status transitions to `Filled`. Sells are disabled in Filled status.

5. **prepare_transition**: Simply flips status from `Filled` to `Graduated` for both curves (lines 67-68). No token movement occurs here.

### Invariant: vault_balance + tokens_sold == TARGET_TOKENS

This invariant holds by construction:

- **fund_curve** deposits exactly TARGET_TOKENS. At this point: `vault = TARGET_TOKENS, tokens_sold = 0`. Sum = TARGET_TOKENS.
- **purchase** removes `actual_tokens` from vault and adds `actual_tokens` to `tokens_sold`. The vault transfer and state update use the same `actual_tokens` variable. Sum preserved.
- **sell** adds `tokens_to_sell` back to vault (transfer_checked) and decrements `tokens_sold` by the same amount (`x2 = x1 - tokens_to_sell`). Sum preserved.

When the curve reaches Filled status, `tokens_sold >= TARGET_TOKENS`. Since remaining tokens = `TARGET_TOKENS - tokens_sold`, and tokens_sold can only reach TARGET_TOKENS (capped in purchase), the vault should hold exactly 0 remaining tokens when fully sold.

### Post-Graduation Token Handling

After graduation:
- **withdraw_graduated_sol** extracts SOL from sol_vault (no token impact)
- **close_token_vault** closes the token vault, but only if `token_vault.amount == 0` (close_token_vault.rs line 46 constraint)

The vault must be empty before closing. The 460M tokens in circulation are held by buyers (in their ATAs). The remaining 540M tokens per mint (1B total - 460M for sale) were allocated elsewhere during initialization (pool seeding, staking, etc. per the token model).

### Edge Case: Partial Fill at Target

The purchase instruction caps tokens at remaining (purchase.rs lines 146-148):
```rust
let remaining = TARGET_TOKENS.checked_sub(curve.tokens_sold).ok_or(...)?;
let actual_tokens = tokens_out.min(remaining);
```

If a purchase would overshoot, it's clamped. The SOL is also recalculated for the clamped amount (line 159), and any overpayment is refunded. This prevents tokens_sold from exceeding TARGET_TOKENS.

### Proptest Coverage

The bonding curve math has extensive property-based testing (500K+ proptest iterations per ARCHITECTURE.md INV-3):
- **INV-3**: `vault >= integral(0, tokens_sold)` verified post-sell
- **Property S4**: Sell never increases tokens_sold
- **Property S3**: Multi-operation sequences preserve solvency

## Verdict

**SECURE.** Token accounting is maintained by construction. Every purchase and sell modifies vault balance and `tokens_sold` by the same amount in opposite directions. The invariant `vault_balance + tokens_sold == TARGET_TOKENS` holds throughout the curve lifecycle. At graduation (Filled state), tokens_sold == TARGET_TOKENS and vault == 0 (all tokens distributed to buyers). The `prepare_transition` instruction only flips status -- it performs no token movement and cannot disrupt the accounting.
