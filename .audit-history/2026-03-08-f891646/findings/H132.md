# H132 -- Staking: Dead Stake Initialization

**Status:** SAFE
**Severity:** N/A
**Target:** `programs/staking/src/instructions/initialize_stake_pool.rs`

## Hypothesis

Does the dead stake (first-depositor attack mitigation) get properly created during initialization, and can it be removed afterward?

## Analysis

### Dead Stake Creation (`initialize_stake_pool.rs`)

During `initialize_stake_pool`, the handler:

1. **Creates StakePool PDA** with `init` constraint (lines 35-42). The `init` constraint ensures this can only be called once -- subsequent calls fail because the account already exists.

2. **Transfers MINIMUM_STAKE** (1 PROFIT = 1,000,000 token units at 6 decimals) from the authority's token account to the stake vault (lines 125-135):
   ```rust
   transfer_checked_with_hook(
       &ctx.accounts.token_program.to_account_info(),
       &ctx.accounts.authority_token_account.to_account_info(),
       &ctx.accounts.profit_mint.to_account_info(),
       &ctx.accounts.stake_vault.to_account_info(),
       &ctx.accounts.authority.to_account_info(),
       ctx.remaining_accounts,
       MINIMUM_STAKE,
       PROFIT_DECIMALS,
       &[], // authority is a signer, not a PDA
   )?;
   ```

3. **Sets total_staked = MINIMUM_STAKE** (line 138):
   ```rust
   pool.total_staked = MINIMUM_STAKE;
   ```

### Can Dead Stake Be Removed?

**No.** The dead stake is irremovable by design:

1. **No UserStake account exists for it**: The dead stake is transferred directly to the stake vault during init. No `UserStake` PDA is created for the initializer. The `unstake` instruction requires a `UserStake` account with `constraint = user_stake.owner == user.key()` (unstake.rs line 54). Since no UserStake was created during init, there is no account that could authorize withdrawal of the dead stake.

2. **Unstake only operates on UserStake balances**: The unstake handler decrements `user_stake.staked_balance` and `pool.total_staked` (unstake.rs lines 171-179). Even if an attacker created a UserStake via the `stake` instruction and later unstaked everything, `pool.total_staked` would still retain the MINIMUM_STAKE from init because it was never tracked in any UserStake account.

3. **No admin withdrawal**: There is no instruction in the staking program that allows arbitrary withdrawal from the stake vault. The only outflow path is `unstake`, which is bounded by individual `user_stake.staked_balance`.

4. **Stake vault authority is StakePool PDA**: The vault's authority is the `stake_pool` PDA itself (line 65: `token::authority = stake_pool`). Only instructions that derive this PDA's signer seeds can transfer from the vault, and `unstake` is the only such instruction.

### Accounting Integrity

- `pool.total_staked` starts at MINIMUM_STAKE and can only decrease via `unstake` (which is bounded by individual UserStake balances that sum to `total_staked - MINIMUM_STAKE`).
- The `update_cumulative` instruction divides `pending_rewards` by `total_staked`. Dead stake ensures `total_staked >= MINIMUM_STAKE > 0`, preventing division-by-zero.
- The dead stake's share of rewards is never claimed (no UserStake exists), so those rewards remain in the escrow permanently -- a small ongoing loss (~0.005% of rewards with 20M PROFIT supply) that benefits protocol solvency.

### One-Shot Initialization

The `init` constraint on StakePool guarantees `initialize_stake_pool` can only succeed once. An attacker cannot:
- Call it first with 0 dead stake (MINIMUM_STAKE is a constant, not a parameter)
- Call it again to overwrite state
- Front-run with different parameters (the amount is hardcoded)

**Note:** The initializer can be anyone (Trust Tier 6 per Architecture doc), but this is safe because the dead stake amount is hardcoded and the init is idempotent-by-failure.

## Verdict

**No vulnerability.** Dead stake is correctly created during initialization via a hardcoded MINIMUM_STAKE transfer. It cannot be removed because no UserStake account is associated with it, and the only withdrawal path (unstake) requires a UserStake. The `init` constraint ensures one-shot execution. This effectively prevents the first-depositor inflation attack (verified by 80K proptest iterations per INV-4).
