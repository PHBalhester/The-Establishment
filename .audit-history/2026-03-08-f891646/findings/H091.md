# H091 -- Tax Program: SOL Wrapping/Unwrapping

**Status:** SAFE
**Severity:** N/A (confirmed secure)
**Files:** `programs/tax-program/src/instructions/swap_sol_buy.rs`, `swap_sol_sell.rs`

## Hypothesis

Check WSOL sync_native and close_account handling. Verify no SOL leakage during wrap/unwrap.

## Evidence

### Buy Path (swap_sol_buy.rs) -- No WSOL Wrapping

The buy path does NOT wrap/unwrap WSOL within the Tax Program. Tax is deducted from the user's native SOL input before the AMM CPI:

1. **Tax deducted first** (lines 83-91): `tax_amount = amount_in * tax_bps / 10_000`, `sol_to_swap = amount_in - tax_amount`
2. **Native SOL transfers** (lines 126-209): Tax portions sent via `system_instruction::transfer` directly from user to staking_escrow, carnage_vault, treasury
3. **AMM CPI** (lines 260-308): Only `sol_to_swap` (post-tax) is passed to the AMM

No WSOL intermediary is involved in the buy path. No wrapping/unwrapping occurs. SOL conservation: `amount_in = tax_amount + sol_to_swap`, with `tax_amount = staking_portion + carnage_portion + treasury_portion` (verified by `split_distribution` invariant).

### Sell Path (swap_sol_sell.rs) -- WSOL Intermediary Pattern

The sell path uses a close-and-reinit pattern for WSOL tax extraction:

**Step-by-step flow (lines 265-451):**

1. **AMM CPI executes** (lines 127-215): User sends tokens, receives gross WSOL output into `user_token_a`
2. **Balance diff** (lines 220-225): `gross_output = wsol_after - wsol_before` (reload pattern)
3. **Tax calculation** (line 230): `tax_amount = gross_output * tax_bps / 10_000`
4. **Net output** (lines 238-245): `net_output = gross_output - tax_amount`, checked > 0
5. **SPL Transfer** (lines 269-290): `tax_amount` WSOL transferred from `user_token_a` to `wsol_intermediary` via SPL Token `Transfer` (discriminator 3). User signs.
6. **Close intermediary** (lines 295-312): `close_account` on `wsol_intermediary` with destination = `swap_authority`. This unwraps ALL lamports (token balance + rent-exempt minimum) to `swap_authority` as native SOL.
7. **Distribute tax** (lines 318-398): Native SOL system transfers from `swap_authority` to staking/carnage/treasury. Only `staking_portion + carnage_portion + treasury_portion = tax_amount` is distributed.
8. **Recreate intermediary** (lines 401-451):
   - `create_account` at PDA address, funded by `swap_authority` with exactly `rent_lamports` (line 407-426)
   - `InitializeAccount3` (discriminator 18) sets owner = `swap_authority`, mint = WSOL (lines 431-451)

### SOL Leakage Analysis

**Rent lamport accounting:**
- After close_account (step 6), `swap_authority` holds `tax_amount + rent_lamports`
- After distributing tax (step 7), `swap_authority` holds `rent_lamports` (the exact amount needed for reinit)
- After create_account (step 8), `rent_lamports` is transferred from `swap_authority` to the new intermediary account

This is a closed loop: rent lamports flow from intermediary -> swap_authority -> new intermediary. No SOL leaks.

**Tax amount accounting:**
- `split_distribution` guarantees `staking + carnage + treasury == total_tax` (treasury = remainder pattern, verified by 10K proptest)
- Every lamport of tax is accounted for across exactly 3 destinations

### Edge Case: sync_native Not Used

The Tax Program does NOT use `sync_native` anywhere. This is correct because:
- Buy path: WSOL wrapping is handled by the client (pre-wraps SOL before calling swap_sol_buy)
- Sell path: WSOL is produced by the AMM CPI (already in token account), and unwrapped via close_account

### Edge Case: Sequential Sells

The `wsol_intermediary` account is closed and recreated within each sell transaction. The Anchor constraint uses `seeds = [WSOL_INTERMEDIARY_SEED], bump` without existence checks, allowing the PDA to be in any state at instruction entry (the handler manages its lifecycle).

### Edge Case: Intermediary Initialization

`initialize_wsol_intermediary.rs` is a one-time setup that creates the initial intermediary. After the first sell, the sell handler manages the close-reinit cycle. The initialization has no access control beyond requiring a signer (any signer), but it is idempotent -- `create_account` on an existing account will fail.

## Conclusion

No SOL leakage exists in the wrap/unwrap flow. The sell path's close-and-reinit pattern correctly accounts for both tax lamports (distributed to 3 destinations) and rent lamports (recycled into the new intermediary). The buy path avoids WSOL entirely by deducting tax from native SOL input. The `split_distribution` remainder pattern guarantees exact conservation of tax amounts.
