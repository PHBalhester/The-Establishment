# H081 -- Transfer Hook: Whitelist Check Bypass

**Status:** VERIFIED -- No bypass found
**Severity:** Informational (confirmed secure)
**Target:** `programs/transfer-hook/src/instructions/transfer_hook.rs`

---

## Hook Logic Summary

The transfer hook enforces that **at least one party** (source token account OR destination token account) must be whitelisted. This is an OR-gate whitelist, not AND-gate.

```
File: programs/transfer-hook/src/instructions/transfer_hook.rs

Validation order:
1. Zero amount check (reject amount == 0)
2. Mint owner check (must be Token-2022 program)
3. Transferring flag check (must be in T22 transfer context)
4. Whitelist check: source OR destination must be whitelisted
```

## Source AND Destination Verification

### Source check (line 96-99):
```rust
let source_whitelisted = is_whitelisted(
    &ctx.accounts.whitelist_source.to_account_info(),
    &ctx.accounts.source_token.key(),
);
```

### Destination check (lines 101-108):
```rust
if !source_whitelisted {
    let dest_whitelisted = is_whitelisted(
        &ctx.accounts.whitelist_destination.to_account_info(),
        &ctx.accounts.destination_token.key(),
    );
    require!(dest_whitelisted, TransferHookError::NoWhitelistedParty);
}
```

**Both accounts are checked.** The short-circuit optimization (skip destination if source is whitelisted) does not create a bypass -- it only skips an unnecessary check when the source already passes.

## `is_whitelisted()` Verification (lines 166-178)

```rust
fn is_whitelisted(whitelist_pda: &AccountInfo, token_account: &Pubkey) -> bool {
    // Account must have data (exists)
    if whitelist_pda.data_is_empty() {
        return false;
    }
    // Verify PDA derivation is correct (prevents spoofed accounts)
    let (expected_pda, _bump) = Pubkey::find_program_address(
        &[WhitelistEntry::SEED_PREFIX, token_account.as_ref()],
        &crate::ID
    );
    whitelist_pda.key() == expected_pda
}
```

Two checks:
1. **Existence:** `data_is_empty()` returns false only if the account exists and has data (WhitelistEntry PDA was created via `add_whitelist_entry`).
2. **PDA derivation:** Re-derives the expected PDA from the token account key and verifies it matches. This prevents passing a spoofed account that has data but the wrong derivation.

## Fallback Path Analysis

### Can the hook be bypassed by not passing whitelist accounts?
**No.** The `TransferHook` accounts struct requires both `whitelist_source` and `whitelist_destination` as named accounts. Anchor deserialization will fail if they are missing. The ExtraAccountMetaList resolves these automatically during `transfer_checked`.

### Can the hook be bypassed by direct invocation?
**No.** Validation 3 (`check_is_transferring`) reads the `TransferHookAccount` extension from the source token account. The `transferring` flag is only set by Token-2022 during `transfer_checked`. Direct invocation of the hook program will find this flag as false and reject.

### Can the hook be bypassed by using `transfer` instead of `transfer_checked`?
**No.** Token-2022 mints with a transfer hook extension require `transfer_checked`. Plain `transfer` will fail at the Token-2022 level.

### Can zero-amount transfers slip through?
**No.** Validation 1 explicitly rejects `amount == 0`.

### Can a non-T22 mint bypass the check?
**No.** Validation 2 verifies `mint.owner == spl_token_2022::id()`. The ExtraAccountMetaList PDA is also derived per-mint, adding another layer.

---

## Design Note

The whitelist uses an **OR-gate** model: either source OR destination must be whitelisted. This means:
- Protocol PDAs (AMM pools, bonding curve vaults, staking vaults) are whitelisted
- User-to-user transfers between two non-whitelisted accounts are blocked
- User transfers to/from whitelisted protocol accounts are allowed

This is the intended design for a protocol where all token movement must go through protocol contracts.

---

## Verdict

**CONFIRMED SECURE.** Both source and destination are checked. PDA derivation prevents spoofed whitelist accounts. The `transferring` flag prevents direct invocation. No fallback paths skip the whitelist check. The zero-amount guard is present.
