# S001 -- BC Authority Fix: ProgramData Pattern Completeness

**Audit:** sos-002 | **Severity:** Supplemental (derived from CRITICAL BC authority gap)
**Status:** CONFIRMED -- pattern requires 6 instruction changes, not 1
**Derived from:** H-series BC authority findings (5 agents)

---

## Hypothesis

If the ProgramData authority pattern (used by AMM's `initialize_admin`) is added to the Bonding Curve program, it would need to cover ALL 6 admin-labeled instructions. Partial coverage would leave attack surface.

## Investigation

### AMM Reference Pattern (initialize_admin.rs)

The AMM program uses the following pattern in `InitializeAdmin`:

1. `program: Program<'info, crate::program::Amm>` -- self-reference to look up ProgramData address.
2. `program_data: Account<'info, ProgramData>` -- the BPF loader's ProgramData account.
3. Constraint: `program.programdata_address()? == Some(program_data.key())` -- links program to its ProgramData.
4. Constraint: `program_data.upgrade_authority_address == Some(authority.key())` -- proves caller is deployer.
5. Creates an `AdminConfig` PDA with `has_one = admin` for downstream instructions.

This is a two-tier pattern:
- **Tier A (one-shot):** `initialize_admin` verifies ProgramData upgrade authority, creates AdminConfig PDA.
- **Tier B (ongoing):** `initialize_pool`, `burn_admin` use `has_one = admin` on the AdminConfig PDA.

### Bonding Curve: 6 Instructions Requiring Authority

All 6 instructions currently accept ANY `Signer<'info>` named `authority` with zero verification:

| # | Instruction | Current Access | What It Controls | Impact if Unpatched |
|---|-------------|---------------|-----------------|---------------------|
| 1 | `initialize_curve` | Any signer (pays rent) | Creates CurveState PDA | Attacker creates rogue curves |
| 2 | `fund_curve` | Any signer (needs tokens) | Transfers 460M tokens to vault | Partially gated by token ownership |
| 3 | `start_curve` | Any signer | Sets curve Active, starts deadline | Premature activation |
| 4 | `prepare_transition` | Any signer | Moves both curves to Graduated | Forces graduation, enables withdrawal |
| 5 | `withdraw_graduated_sol` | Any signer (receives SOL) | Withdraws ~1000 SOL per curve | Direct SOL theft |
| 6 | `close_token_vault` | Any signer (receives rent) | Closes empty vault, gets rent | Minor rent theft |

### Required Fix Architecture

Following the AMM pattern, the BC program would need:

**New instruction: `initialize_bc_admin`**
- Verifies ProgramData upgrade authority (same as AMM).
- Creates a `BcAdminConfig` PDA with `admin: Pubkey`.
- One-shot (PDA `init` prevents replay).

**All 6 existing instructions must add:**
```
bc_admin_config: Account<'info, BcAdminConfig>,
// with constraint: has_one = authority (matching admin field)
```

### Instruction-Specific Analysis

| Instruction | Difficulty | Notes |
|-------------|-----------|-------|
| `initialize_curve` | Low | Add `has_one` constraint. No other changes. |
| `fund_curve` | Low | Add `has_one` constraint. Token ownership already provides partial defense. |
| `start_curve` | Low | Add `has_one` constraint. No other changes. |
| `prepare_transition` | Low | Add `has_one` constraint. Both curves checked. |
| `withdraw_graduated_sol` | **CRITICAL** | Add `has_one` constraint. Without this, ~2000 SOL is stealable. |
| `close_token_vault` | Low | Add `has_one` constraint. Minor rent recovery. |

### Partial Fix Danger

If the fix is applied to only some instructions (e.g., only `withdraw_graduated_sol`):

- Attacker calls unprotected `start_curve` to prematurely activate a curve.
- Attacker calls unprotected `prepare_transition` when both curves reach Filled (natural occurrence).
- Attacker cannot call protected `withdraw_graduated_sol`.
- **But:** `prepare_transition` setting Graduated is irreversible. If admin wants to delay graduation for coordination, attacker can force it.

Conversely, if only `prepare_transition` is protected but `withdraw_graduated_sol` is not:
- Admin transitions to Graduated.
- Attacker front-runs `withdraw_graduated_sol` and steals SOL.

**All 6 must be covered.**

## Verdict

**CONFIRMED.** The ProgramData authority pattern requires:
1. One new instruction (`initialize_bc_admin`) adding ~60 lines.
2. Six existing instructions each need a `BcAdminConfig` account and `has_one` constraint (3-5 lines each).
3. A new `BcAdminConfig` state struct (identical to AMM's `AdminConfig`).

Total: ~7 instruction changes, 1 new state struct. This is a medium-size refactor but straightforward since the AMM provides a complete reference implementation.

**Risk of partial fix:** HIGH. Any instruction left unprotected creates an attack path. The most dangerous partial fix is protecting `prepare_transition` without protecting `withdraw_graduated_sol`, or vice versa.

## Recommendation

Apply the full ProgramData pattern to all 6 instructions in a single atomic change. Test that:
1. Non-admin signers are rejected for all 6.
2. Admin can still execute all 6 after `initialize_bc_admin`.
3. `initialize_bc_admin` itself requires the program's upgrade authority.
