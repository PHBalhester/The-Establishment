# H111 -- Epoch Program: VRF Timeout Recovery

**Status:** SAFE
**Severity:** N/A (Informational)
**File:** `programs/epoch-program/src/instructions/retry_epoch_vrf.rs`

## Hypothesis

The `retry_epoch_vrf` instruction may fail to properly replace the old VRF binding or leave stale state, potentially enabling reroll attacks or protocol deadlock.

## Evidence

### State Replacement (lines 98-103)

```rust
epoch_state.vrf_request_slot = clock.slot;
epoch_state.pending_randomness_account = ctx.accounts.randomness_account.key();
```

Both fields are overwritten atomically. The old `pending_randomness_account` pubkey is fully replaced, so `consume_randomness` will only accept the new randomness account going forward (INV-6 maintained).

### Validation Chain (5 checks)

1. **VRF pending guard (line 62):** `epoch_state.vrf_pending` must be true -- cannot retry if no VRF is outstanding.
2. **Timeout elapsed (line 71-74):** `elapsed_slots > VRF_TIMEOUT_SLOTS` (strict greater-than, 301+ slots). Uses `saturating_sub` for underflow safety.
3. **Switchboard owner check (line 37):** Account struct constraint validates `randomness_account.owner == SWITCHBOARD_PROGRAM_ID`.
4. **Freshness check (line 83-90):** New randomness `seed_slot` must be within 1 slot of current -- prevents reusing stale randomness.
5. **Not-yet-revealed check (line 93-96):** Rejects already-revealed randomness -- the new account must be uncommitted.

### State Cleanup

- `vrf_pending` remains `true` (correct -- VRF is still pending, just with a new randomness account).
- `vrf_request_slot` is updated to current slot, resetting the timeout window for the new randomness.
- `current_epoch` is unchanged (correct -- we're retrying the same epoch's VRF).
- No other epoch state fields are modified.

### Anti-Reroll Analysis

An attacker cannot exploit retry to reroll for favorable randomness because:
- The 300-slot timeout must genuinely elapse (permissionless but time-gated).
- The new randomness account must be fresh (seed_slot within 1 slot), meaning the oracle hasn't revealed yet.
- Once revealed, `consume_randomness` binds to the new `pending_randomness_account`, and the old one is discarded.

## Verdict

**No vulnerability.** The retry handler correctly validates all preconditions, fully replaces the stale binding, resets the timeout window, and maintains the anti-reroll invariant (INV-6). The permissionless design is intentional for deadlock recovery.
