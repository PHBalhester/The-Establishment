# H092 -- Carnage: expire_carnage State Cleanup

**Status:** SAFE
**Severity:** N/A (no issue found)
**Target:** `programs/epoch-program/src/instructions/expire_carnage.rs`

## Hypothesis

Check state transition on expire. Verify fund accounting on expiry. No stuck funds.

## Analysis

### State Transition

The `expire_carnage` handler clears exactly two fields on `EpochState`:

```rust
epoch_state.carnage_pending = false;
epoch_state.carnage_action = CarnageAction::None.to_u8();
```

It intentionally does NOT update `last_carnage_epoch` (comment on line 90: "Don't update last_carnage_epoch - Carnage didn't actually execute"). This is correct -- expiry means carnage failed, so it should not count as a completed carnage for frequency tracking.

### Fields NOT Cleared: carnage_target, carnage_deadline_slot, carnage_lock_slot

These fields are left stale. This is safe because:
1. All instructions that read these fields gate on `carnage_pending == true` first (Anchor constraint on line 33).
2. All other clearing paths (execute_carnage, execute_carnage_atomic, consume_randomness auto-expire) follow the same pattern -- they also only clear `carnage_pending` and `carnage_action`.
3. When a new carnage triggers in `consume_randomness` (line 277-284), all four fields (target, action, deadline_slot, lock_slot) are overwritten fresh.

### Fund Accounting: No Stuck Funds

The `sol_vault` PDA (CARNAGE_SOL_VAULT_SEED) is read-only in this instruction -- SOL is retained in the vault. This is by design: the SOL stays available for the next carnage trigger. The architecture doc confirms (Trust Tier 5) that `expire_carnage` is permissionless and state-gated.

Key observations:
- `carnage_state` (CarnageFundState) is also read-only (no `mut` on the account constraint, line 38-42). No fund state is mutated on expiry.
- The SOL vault balance is only reported in the event (`sol_retained`), not modified.
- Next carnage trigger will use the same SOL vault, so funds roll over naturally.

### Access Control

Two guards protect against invalid calls:
1. **Anchor constraint** (line 33): `epoch_state.carnage_pending @ EpochError::NoCarnagePending` -- can only expire if carnage is pending.
2. **Deadline check** (line 68-71): `clock.slot > epoch_state.carnage_deadline_slot` -- can only expire after deadline passes.

These are sufficient. The `>` (strict greater-than) means expiry is not possible at the exact deadline slot, only after it. This gives execute_carnage/execute_carnage_atomic the full window.

### Cross-Reference: consume_randomness Auto-Expire

`consume_randomness` (lines 115-127) also auto-expires stale carnage if `carnage_pending && clock.slot > carnage_deadline_slot` before processing new VRF. This uses the same clearing logic and is consistent with expire_carnage.

## Verdict

No issue. State cleanup is correct and consistent across all four clearing paths. SOL is retained in vault for next trigger -- no stuck funds. Stale fields are harmless behind the `carnage_pending` gate.
