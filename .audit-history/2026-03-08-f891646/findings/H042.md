# H042: Tax Math Rounding (H084 Recheck)

**Audit:** sos-002
**Strategy:** STR-042
**Target:** `programs/tax-program/src/helpers/tax_math.rs`
**Determination:** NOT VULNERABLE
**Severity:** Informational
**Confidence:** HIGH

## Summary

Tax computation rounding in `tax_math.rs` is correctly implemented with no exploitable value leakage. All three functions (`calculate_tax`, `split_distribution`, `calculate_output_floor`) use u128 intermediates, checked arithmetic, and protocol-favored floor rounding. The conservation invariant `staking + carnage + treasury == total_tax` is enforced by construction (treasury absorbs remainder). Proptest coverage (10K iterations per property, 6 properties) validates these guarantees across the full u64 input range.

This is a recheck of H084 from audit #1. The previous finding is **RESOLVED** -- no rounding exploitation path exists.

## Evidence

### 1. `calculate_tax` -- Floor Rounding, No Leakage

```rust
// tax_math.rs:46-48
let tax = amount
    .checked_mul(bps)?          // u128: max u64::MAX * 10000 = 1.84e23, fits u128
    .checked_div(10_000)?;       // floor division
```

- **Rounding direction:** Floor (truncates toward zero). User pays less tax, protocol absorbs the loss. This is the conservative direction -- protocol never over-charges.
- **Maximum rounding loss per operation:** 0.9999 lamports (rounds to 0 lamports lost). For `amount=10, bps=400`: exact tax is 0.4 lamports, floor gives 0. The user pays zero tax on a 10-lamport swap.
- **Overflow safety:** `u64::MAX * 10_000 = 1.84e23`, well within u128 (3.4e38). The `u64::try_from()` cast at line 52 is safe because dividing by 10_000 can never exceed the original u64 input.
- **Edge case:** `calculate_tax(u64::MAX, 10000) = Some(u64::MAX)` -- verified by unit test at line 207.

### 2. `split_distribution` -- Remainder-Based Conservation

```rust
// tax_math.rs:96-107
let staking = floor(total * 7100 / 10000);  // u128 intermediate, try_from to u64
let carnage = floor(total * 2400 / 10000);  // u128 intermediate, try_from to u64
let treasury = total_tax - staking - carnage; // remainder absorbs ALL rounding dust
```

**Conservation invariant (proven):** `staking + carnage + treasury == total_tax` holds for ALL u64 inputs by construction. Treasury is computed as the remainder, not via a third BPS calculation. This is the correct pattern -- it makes the sum-equals-total invariant a mathematical certainty rather than an approximation.

**Maximum dust absorbed by treasury per split:**
- Staking floor loss: up to `(10000 - 1) / 10000` = 0.9999 lamports
- Carnage floor loss: up to 0.9999 lamports
- Combined: treasury can receive up to ~2 extra lamports compared to its "ideal" 5% share

**Micro-tax edge case (total < 4 lamports):** All goes to staking, carnage and treasury get 0. This is correct -- splitting 1-3 lamports three ways would produce mostly zeros. The threshold of 4 is the minimum where `floor(4 * 2400/10000) = 0`, so even at 4 lamports carnage gets 0. At 5 lamports: staking=3, carnage=1, treasury=1 -- first meaningful three-way split.

**Boundary analysis at threshold (total=4):** `split_distribution(4) = (2, 0, 2)`. Staking gets `floor(4*7100/10000) = floor(2.84) = 2`. Carnage gets `floor(4*2400/10000) = floor(0.96) = 0`. Treasury gets `4-2-0 = 2`. This means treasury gets 50% at the boundary, not 5%. This is a quirk but not a vulnerability -- at 4 lamports the absolute amounts are negligible and the conservation invariant still holds.

**u64::MAX test:** `split_distribution(u64::MAX)` returns `Some(...)` and sum equals `u64::MAX` -- verified by unit test at line 280.

### 3. `calculate_output_floor` -- Safe Downcast

```rust
// tax_math.rs:159-164
let floor = expected
    .checked_mul(floor_bps as u128)?
    .checked_div(10_000)?;
Some(floor as u64)               // <-- unchecked cast
```

The `as u64` cast at line 164 is **safe** because:
- `floor <= expected` (multiplying by `floor_bps / 10000` where `floor_bps` is protocol-controlled at 5000)
- `expected <= reserve_out` (constant-product formula output is bounded by the output reserve)
- `reserve_out` is a u64 value

However, there is no `u64::try_from()` guard here, unlike `calculate_tax` and `split_distribution` which use `try_from`. If `floor_bps > 10_000` were ever passed (currently impossible -- hardcoded to 5000), the cast could truncate. This is a **defense-in-depth gap**, not a vulnerability.

### 4. Inline Constants vs Module Constants

`split_distribution` defines BPS constants inline (`const STAKING_BPS: u128 = 7_100; const CARNAGE_BPS: u128 = 2_400;`) rather than importing from `constants.rs`. The comment at line 80-81 explains this is to keep the module free of `anchor_lang` dependencies for fast unit testing. The values **match** `constants.rs` (STAKING_BPS=7100, CARNAGE_BPS=2400). However, these are duplicated values with no compile-time enforcement of consistency. A change to one without the other would silently produce different behavior in tests vs production.

**Risk:** LOW. The integration test suite (LiteSVM) exercises the full program path including `constants.rs` values, so a mismatch would be caught at integration test time.

### 5. Proptest Coverage Assessment

Six proptest properties with 10,000 iterations each:

| Property | Input Range | What It Validates |
|----------|------------|-------------------|
| `tax_never_overflows_valid_bps` | amount: 0..u64::MAX, bps: 0..10000 | No None returns for valid inputs |
| `tax_none_for_invalid_bps` | amount: 0..u64::MAX, bps: 10001..u16::MAX | Always None for invalid bps |
| `tax_monotonic_with_bps` | amount: 1..1e12, bps: 0..9999 | Higher bps = higher or equal tax |
| `split_sum_equals_input` | total: 0..u64::MAX | Conservation invariant |
| `staking_at_least_70_percent` | total: 4..u64::MAX | Staking >= 70% (non-micro) |
| `micro_tax_all_to_staking` | total: 0..3 | Micro-tax rule |

**Coverage gaps:** None significant. The full u64 range is covered for the conservation invariant. The monotonicity test is bounded at 1e12 (not u64::MAX), but this is reasonable since the tax calculation is linear. `calculate_output_floor` has unit tests but **no proptest coverage** -- this is an observation, not a vulnerability, since the function's math is straightforward constant-product with BPS scaling.

### 6. Rounding Direction Summary (All Protocol-Favored)

| Function | Operation | Rounding | Favors |
|----------|-----------|----------|--------|
| `calculate_tax` | `amount * bps / 10000` | Floor | User (less tax) |
| `split_distribution` staking | `total * 7100 / 10000` | Floor | Treasury (gets remainder) |
| `split_distribution` carnage | `total * 2400 / 10000` | Floor | Treasury (gets remainder) |
| `split_distribution` treasury | `total - staking - carnage` | Remainder | Exact (no rounding) |
| `calculate_output_floor` expected | `rOut * aIn / (rIn + aIn)` | Floor | Protocol (lower floor) |
| `calculate_output_floor` floor | `expected * bps / 10000` | Floor | Protocol (lower floor) |

All rounding is floor-based and either protocol-favored or user-favored (never protocol-extractive).

## Observations (Informational)

1. **`calculate_output_floor` uses `as u64` instead of `try_from`** (line 164). Safe with current parameters but inconsistent with the defense-in-depth pattern used in `calculate_tax` and `split_distribution`. Recommend switching to `u64::try_from(floor).ok()?` for consistency.

2. **Duplicated BPS constants** between `tax_math.rs` (inline) and `constants.rs` (module-level). No compile-time cross-check. A `#[cfg(test)]` assertion could verify equality.

3. **No proptest for `calculate_output_floor`**. The function has 8 unit tests covering zero inputs, large reserves, realistic pools, and edge cases. Adding proptest would strengthen confidence but is not critical given the simple math.

## Determination Rationale

- Conservation invariant (`staking + carnage + treasury == total_tax`) is mathematically guaranteed by remainder construction
- All arithmetic uses u128 intermediates with checked operations
- Floor rounding consistently favors protocol or is neutral
- 10K proptest iterations per property across full u64 range
- No value leakage path: every lamport of tax is accounted for in exactly one destination
- Previous finding H084 is confirmed RESOLVED

**Verdict: NOT VULNERABLE. Tax math rounding is sound.**
