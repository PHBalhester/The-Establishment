# H089 -- AMM: swap_sol_pool CPI-only Access

**Status:** SAFE
**Severity:** N/A (confirmed secure)
**File:** `programs/amm/src/instructions/swap_sol_pool.rs`

## Hypothesis

Verify that `seeds::program` constraint on `swap_authority` restricts AMM swap invocation to the Tax Program only, and that no direct invocation is possible.

## Evidence

### 1. swap_authority Constraint (lines 367-372)

```rust
#[account(
    seeds = [SWAP_AUTHORITY_SEED],
    bump,
    seeds::program = TAX_PROGRAM_ID,
)]
pub swap_authority: Signer<'info>,
```

Two constraints work in tandem:

- **`Signer<'info>`**: The account must have actually signed the transaction (or CPI). A regular user cannot produce a valid signature for a PDA.
- **`seeds::program = TAX_PROGRAM_ID`**: The PDA is derived using the Tax Program's program ID as the deriving program. Anchor verifies the passed account matches `Pubkey::find_program_address(&[SWAP_AUTHORITY_SEED], &TAX_PROGRAM_ID)`.

### 2. TAX_PROGRAM_ID is Hardcoded (constants.rs line 10)

```rust
pub const TAX_PROGRAM_ID: Pubkey = pubkey!("Eufdhhek6L1cxrYPvXAgJRVzckuzWVVBLckjNwyggViV");
```

This is a compile-time constant. No admin can change it post-deployment.

### 3. Direct Invocation Impossible

For a direct call (not via Tax Program CPI), the caller would need to pass `swap_authority` as a signer. Since it is a PDA derived from the Tax Program, only the Tax Program can produce a valid `invoke_signed` with the correct seeds. The Solana runtime enforces that PDA signers can only be produced by the program that owns the derivation.

### 4. Cross-verification with Tax Program

Both `swap_sol_buy.rs` (line 217-219) and `swap_sol_sell.rs` (line 127) derive `swap_authority_seeds` from `SWAP_AUTHORITY_SEED` and sign CPI calls with `invoke_signed`. The Tax Program's `SWAP_AUTHORITY_SEED` (constants.rs line 11) is `b"swap_authority"`, matching the AMM's constant exactly.

## Conclusion

The AMM's `swap_sol_pool` instruction is cryptographically restricted to CPI from the Tax Program only. The combination of `Signer` + `seeds::program` with a hardcoded program ID is the strongest access control pattern available on Solana. No direct invocation, no alternative caller, no admin override possible.
