# H115 -- Tax Program: Swap Exempt Path Validation

**Status:** SAFE
**Severity:** N/A
**Confidence:** HIGH

## Hypothesis

`swap_exempt` bypasses tax for Carnage Fund operations. Verify that only the Epoch Program can invoke it via the `seeds::program` gate on the carnage authority PDA.

## Analysis

### Access control mechanism

The `SwapExempt` account struct (`programs/tax-program/src/instructions/swap_exempt.rs:180-255`) enforces access via two PDA constraints:

1. **`carnage_authority`** (line 193-198):
   ```rust
   #[account(
       seeds = [CARNAGE_SIGNER_SEED],
       bump,
       seeds::program = epoch_program_id(),
   )]
   pub carnage_authority: Signer<'info>,
   ```
   This is a `Signer<'info>`, meaning the account must have signed the transaction. The `seeds::program = epoch_program_id()` constraint means Anchor verifies that `carnage_authority` is the PDA derived from `[CARNAGE_SIGNER_SEED]` under the Epoch Program's program ID. Only the Epoch Program can produce a valid `invoke_signed` with this PDA as signer.

2. **`swap_authority`** (line 204-208): Tax Program's own PDA for signing the downstream AMM CPI. Derived from `[SWAP_AUTHORITY_SEED]` under the Tax Program itself.

### Seed constant verification

Both programs define the same seed:
- **Tax Program** (`programs/tax-program/src/constants.rs:66`): `CARNAGE_SIGNER_SEED: &[u8] = b"carnage_signer"`
- **Epoch Program** (`programs/epoch-program/src/constants.rs:91`): `CARNAGE_SIGNER_SEED: &[u8] = b"carnage_signer"`

Cross-program PDA verification tests exist at `programs/tax-program/tests/test_carnage_signer_pda.rs` confirming both sides derive the same PDA.

### Epoch Program usage

The Epoch Program's `execute_carnage` and `execute_carnage_atomic` instructions derive the carnage signer PDA and use `invoke_signed` with `[CARNAGE_SIGNER_SEED, &[bump]]` seeds:
- `execute_carnage.rs:71` -- PDA account constraint
- `execute_carnage.rs:243` -- `invoke_signed` signer seeds
- `execute_carnage_atomic.rs:75` -- PDA account constraint
- `execute_carnage_atomic.rs:235` -- `invoke_signed` signer seeds

### AMM program validation

Line 244 validates the AMM program address:
```rust
#[account(address = amm_program_id() @ TaxError::InvalidAmmProgram)]
pub amm_program: AccountInfo<'info>,
```

This prevents an attacker from substituting a malicious AMM program that could drain funds.

### What swap_exempt bypasses

The handler (line 48-167) performs no tax calculation, no distribution split, and sets `MINIMUM_OUTPUT = 0` (line 111). This is by design -- Carnage accepts market execution per spec. The tax bypass is safe because Carnage is a protocol-owned operation redistributing already-collected taxes.

### Could a malicious program call swap_exempt?

No. The `carnage_authority` must be:
1. A signer on the transaction (enforced by `Signer<'info>`)
2. Equal to the PDA derived from `[b"carnage_signer"]` under the Epoch Program's ID (enforced by `seeds::program`)

Only the Epoch Program can sign with this PDA via `invoke_signed`. No other program can derive the same PDA with a valid signature because PDA derivation is program-ID-specific.

## Verdict

**No vulnerability.** The `seeds::program` constraint cryptographically ensures only the Epoch Program can call `swap_exempt`. The seed constants match across both programs and are covered by dedicated cross-program tests. The AMM program address is also validated, preventing CPI target substitution. This is the same Trust Tier 1 pattern used across all 4 CPI chains in the protocol (Architecture Document Section 2).
