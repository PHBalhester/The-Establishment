# H019: Sell Path AMM Slippage Bypass via minimum_amount_out=0

**Recheck of:** H043 (Audit #1)
**Category:** Arithmetic, Timing
**Determination:** CONFIRMED VULNERABLE
**Severity:** HIGH
**Confidence:** HIGH

## Summary

The AMM's slippage check (`amount_out >= minimum_amount_out` at `swap_sol_pool.rs:145-148`) is correct in isolation. However, the Tax Program's sell path (`swap_sol_sell.rs:147`) hardcodes `amm_minimum = 0` when calling the AMM via CPI, rendering the AMM's slippage protection completely ineffective for sell swaps. The only protections remaining are: (1) a 50% output floor checked against the user's stated `minimum_output` BEFORE the CPI executes, and (2) a post-CPI `net_output >= minimum_output` check. The gap between the 50% floor and the user's actual minimum creates an extractable sandwich window.

This is a shifted observation from H043 (Audit #1), which originally flagged the AMM's internal slippage ordering. The AMM ordering is correct; the vulnerability is in the caller.

## Evidence

### 1. AMM Slippage Check Is Correct

**File:** `programs/amm/src/instructions/swap_sol_pool.rs:144-148`

```rust
// 6. Slippage protection
require!(
    amount_out >= minimum_amount_out,
    AmmError::SlippageExceeded
);
```

The AMM checks `amount_out >= minimum_amount_out` BEFORE reserve updates and token transfers (CEI ordering). This is structurally sound. The issue is what value the caller passes.

### 2. Sell Path Passes minimum_amount_out=0

**File:** `programs/tax-program/src/instructions/swap_sol_sell.rs:146-147`

```rust
// Pass 0 for AMM's minimum_amount_out - we check slippage ourselves after tax
let amm_minimum: u64 = 0;
```

This is an intentional design choice. The comment explains the rationale: the Tax Program checks slippage on the NET output (after tax deduction), not the GROSS output the AMM produces. Since the user's `minimum_output` represents what they expect to receive after tax, passing it directly to the AMM would be too restrictive.

### 3. Buy Path Correctly Passes User's Minimum

**File:** `programs/tax-program/src/instructions/swap_sol_buy.rs:260-262`

```rust
ix_data.extend_from_slice(&sol_to_swap.to_le_bytes());
ix_data.push(0u8); // SwapDirection::AtoB = 0
ix_data.extend_from_slice(&minimum_output.to_le_bytes());
```

The buy path passes the user's `minimum_output` directly to the AMM. This is correct because buy tax is deducted from INPUT (before the swap), so the user's minimum on OUTPUT tokens can be directly enforced by the AMM. The asymmetry between buy and sell is the core issue.

### 4. Output Floor (SEC-10) Provides Partial Protection

**File:** `programs/tax-program/src/instructions/swap_sol_sell.rs:112-118`

```rust
let (reserve_a, reserve_b) = read_pool_reserves(&ctx.accounts.pool)?;
let output_floor = calculate_output_floor(reserve_b, reserve_a, amount_in, MINIMUM_OUTPUT_FLOOR_BPS)
    .ok_or(error!(TaxError::TaxOverflow))?;
require!(
    minimum_output >= output_floor,
    TaxError::MinimumOutputFloorViolation
);
```

`MINIMUM_OUTPUT_FLOOR_BPS = 5000` (50%). This ensures the user's stated `minimum_output` is at least 50% of the expected GROSS output computed from current reserves.

**Critical subtlety:** The floor checks the user's `minimum_output` (which is the NET minimum, after tax) against 50% of expected GROSS output. With tax rates of 1-14%, the effective floor on gross output is approximately:

```
effective_gross_floor = minimum_output / (1 - tax_rate)
```

At 14% tax: if floor = 50% of expected gross, the user's net minimum is ~43% of expected gross. This means the AMM could return as low as ~43% of expected and the transaction would still pass.

### 5. Post-CPI Slippage Check

**File:** `programs/tax-program/src/instructions/swap_sol_sell.rs:238-245`

```rust
let net_output = gross_output
    .checked_sub(tax_amount)
    .ok_or(error!(TaxError::TaxOverflow))?;

require!(net_output > 0, TaxError::InsufficientOutput);
require!(net_output >= minimum_output, TaxError::SlippageExceeded);
```

This check occurs AFTER the AMM CPI completes. If `net_output < minimum_output`, the entire transaction reverts. However, a sandwich attacker does not need the user's transaction to succeed to profit -- the front-run leg moves the price, and if the user's TX reverts, the attacker's back-run restores the price with no loss (they only pay gas).

### 6. TOCTOU on Reserve Reads

**File:** `programs/tax-program/src/helpers/pool_reader.rs:39-57`

The output floor reads pool reserves at byte offsets [137..145] and [145..153] from the raw AccountInfo. This happens BEFORE the AMM CPI. In the same Solana transaction, reserves cannot change between these two points (single instruction execution is atomic). However, within the same BLOCK, a front-runner's prior transaction can manipulate reserves, making the floor calculation based on already-manipulated data.

The floor check uses the manipulated reserves, computes a lower floor, and the user's minimum passes against this lower bar. The actual AMM CPI then executes against the same manipulated reserves with minimum_amount_out=0, producing a poor output.

## Root Cause

The sell path applies tax to the OUTPUT (gross SOL received from AMM), not the input. This creates a design tension:

1. The user's `minimum_output` is denominated in NET SOL (after tax)
2. The AMM's `minimum_amount_out` is denominated in GROSS tokens/SOL (before tax)
3. Converting between them requires knowing the tax rate at the time of CPI construction
4. The protocol chose to pass 0 to the AMM and check slippage post-CPI instead

The correct approach would be to compute a gross minimum from the net minimum: `gross_minimum = minimum_output + expected_tax`. This would give the AMM a meaningful floor.

## Attack Scenario

### Sandwich Attack on Sell Path

1. **Attacker observes** a user's `swap_sol_sell` transaction in the mempool with `amount_in = 1000 CRIME`, `minimum_output = 0.8 SOL` (net)
2. **Front-run:** Attacker sells a large amount of CRIME into the same pool, depressing the SOL/CRIME price
3. **User's TX executes:**
   - Output floor check: `minimum_output (0.8 SOL) >= output_floor (calculated from now-manipulated reserves)` -- passes because reserves are already skewed
   - AMM CPI: `minimum_amount_out = 0` -- accepts ANY gross output
   - AMM returns `gross_output = 0.95 SOL` (worse than expected 1.1 SOL)
   - Tax (e.g., 5%): `tax = 0.0475 SOL`, `net = 0.9025 SOL`
   - Post-CPI check: `0.9025 >= 0.8` -- passes
   - User loses ~0.15 SOL vs fair price
4. **Back-run:** Attacker buys CRIME back at the depressed price, profiting from the spread

### Worst Case Extraction

With the 50% output floor, the theoretical maximum extraction is:
- Expected gross output: X SOL
- Floor allows minimum_output as low as 0.5X
- At 14% tax: net_output as low as 0.5X * (1 - 0.14) = 0.43X
- Maximum extraction: ~57% of expected output

In practice, users set reasonable `minimum_output` values (e.g., 95-99% of expected net), which limits extraction to the gap between fair output and their minimum. The 50% floor only protects against bots/frontends that send `minimum_output = 0`.

## Impact Assessment

**Severity: HIGH**

- Every sell swap is vulnerable to sandwich attacks with no AMM-level protection
- The 50% output floor provides a backstop but allows significant extraction (up to ~50% of expected)
- Buy path is NOT affected (correctly passes user minimum to AMM)
- The attack is profitable for any MEV bot with block-level ordering capability

**Affected users:** All users executing sell swaps (CRIME -> SOL, FRAUD -> SOL)

**Economic impact:** Depends on pool liquidity and MEV competition. With current pool sizes, extraction per sandwich is bounded by:
- Pool depth (larger pools = smaller price impact per unit)
- User's minimum_output setting (tighter minimum = less extractable)
- MEV competition (reduces individual bot profit but does not eliminate user loss)

## Remediation

### Option A: Compute Gross Minimum for AMM (Recommended)

Instead of passing 0, compute the minimum gross output that would satisfy the user's net minimum after tax:

```rust
// In swap_sol_sell handler, before AMM CPI:
let amm_minimum = minimum_output
    .checked_add(calculate_tax(minimum_output, tax_bps).unwrap_or(0))
    .unwrap_or(minimum_output);
```

More precisely, to get the gross amount that yields `minimum_output` after tax:

```rust
// gross - tax(gross) >= minimum_output
// gross - gross * tax_bps / 10000 >= minimum_output
// gross * (10000 - tax_bps) / 10000 >= minimum_output
// gross >= minimum_output * 10000 / (10000 - tax_bps)
let amm_minimum = ((minimum_output as u128)
    .checked_mul(10_000)
    .and_then(|n| n.checked_div(10_000u128.saturating_sub(tax_bps as u128))))
    .and_then(|n| u64::try_from(n).ok())
    .unwrap_or(minimum_output);
```

This gives the AMM a meaningful minimum that rejects sandwiched outputs at the CPI level, causing the entire transaction to revert before any state changes.

**Trade-off:** None. This is strictly better -- the AMM rejects bad outputs earlier, saving compute on the remaining sell-path logic.

### Option B: Tighten the Output Floor

Increase `MINIMUM_OUTPUT_FLOOR_BPS` from 5000 (50%) to 9000+ (90%+). This reduces the extractable window.

**Trade-off:** Legitimate high-volatility swaps may be rejected. Users would need to retry with tighter slippage during volatile periods.

### Option C: Add Transaction Deadline

Add a `deadline_slot` parameter to both swap instructions. Reject if `Clock::get()?.slot > deadline_slot`. This prevents delayed execution via durable nonces.

**Trade-off:** Additional parameter adds instruction size. Minimal cost.

**Recommended combination:** Option A + Option C. Option A eliminates the core vulnerability; Option C adds defense-in-depth against stale execution.

## Conclusion

The AMM's slippage check is correctly implemented and correctly ordered (H043 recheck: ordering is fine). The vulnerability is that the Tax Program's sell path bypasses this protection by passing `minimum_amount_out = 0`. The 50% output floor (SEC-10) provides a partial backstop but leaves a significant extraction window. The recommended fix (Option A) computes a meaningful gross minimum from the user's net minimum and passes it to the AMM, closing the gap with zero trade-offs.
