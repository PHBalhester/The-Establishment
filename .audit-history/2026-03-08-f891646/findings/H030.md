## Hypothesis

VRF reveal bytes are public on Switchboard before `consume_randomness` processes them. An MEV bot observes the reveal, determines whether Carnage triggers (byte 5 < 11), identifies the target pool (byte 7), and front-runs the Carnage swap to extract value.

## Investigation

### 1. VRF Byte Visibility Timeline

The Switchboard On-Demand commit-reveal flow has three transactions:

1. **TX1**: Create randomness account (off-chain, finalized)
2. **TX2**: Bundle `Switchboard.commitIx` + `trigger_epoch_transition` -- binds randomness account, sets `vrf_pending=true`
3. **TX3**: Bundle `Switchboard.revealIx` + `consume_randomness` + `executeCarnageAtomic` -- oracle writes VRF bytes AND protocol consumes them atomically

The critical question is whether VRF bytes are observable between the oracle writing them and `consume_randomness` processing them.

### 2. Atomic Bundling Analysis (Primary Path -- CLOSED)

Verified in `scripts/vrf/lib/vrf-flow.ts:310-318`:

```
const instructions = [
  ComputeBudgetProgram.setComputeUnitLimit({ units: 600_000 }),
  revealIx,       // Switchboard oracle writes VRF bytes
  consumeIx,      // Protocol reads bytes, derives taxes, sets Carnage pending
  carnageIx,      // executeCarnageAtomic executes the swap (no-ops if no trigger)
];
```

All three instructions execute in a single atomic transaction. There is no intermediate on-chain state where:
- VRF bytes are revealed but not consumed, OR
- Carnage is pending but not yet executed

The `executeCarnageAtomic` instruction is designed with a no-op guard -- it returns `Ok(())` when `carnage_pending` is false (`consume_randomness.rs:269-314` sets `carnage_pending` only when triggered). This makes it safe to always include in TX3.

**Verdict on primary path**: No front-running window exists. The reveal, consumption, and Carnage execution are all within one transaction boundary.

### 3. Mempool Observation of TX3

An attacker monitoring the Solana mempool could observe TX3 before it lands. However:

- TX3 is atomic: the attacker cannot insert instructions between reveal and Carnage execution
- The attacker could attempt a same-block sandwich (front-run + back-run around TX3), but they would need to:
  - Parse the bundled TX3 to determine it contains a Carnage trigger
  - Submit a front-run TX in the same block BEFORE TX3
  - This requires validator-level ordering control (Jito bundles or similar)
- Even if sandwiched, the `CARNAGE_SLIPPAGE_BPS_ATOMIC = 8500` (85% floor) limits extraction to at most 15% of the swap amount
- The attacker cannot know IF Carnage triggers just from seeing TX3 (every TX3 includes `executeCarnageAtomic` regardless)

### 4. Fallback Path Analysis (Residual Risk -- NARROW)

If TX3's atomic Carnage execution fails (e.g., compute budget exceeded, slippage floor hit), Carnage enters the fallback path:

- **Slots 0-50** (`CARNAGE_LOCK_SLOTS`): Only atomic path can execute. `execute_carnage.rs:223-226` enforces `clock.slot > carnage_lock_slot`.
- **Slots 50-300** (`CARNAGE_DEADLINE_SLOTS`): Fallback path opens. `execute_carnage` is permissionless.
- **After slot 300**: Carnage expires via `expire_carnage`.

During the fallback window:
1. `carnage_pending = true` is publicly readable on-chain (set by `consume_randomness.rs:277`)
2. `carnage_target` (CRIME or FRAUD) and `carnage_action` (Buy/Sell/Burn) are publicly readable
3. The target pool and swap direction are known
4. The approximate swap amount is predictable (capped at `MAX_CARNAGE_SWAP_LAMPORTS = 1000 SOL`)

An MEV bot can:
- Read `EpochState.carnage_pending`, `carnage_target`, `carnage_action`
- Identify exactly which pool will be traded and in which direction
- Front-run the `execute_carnage` TX with a buy of the target token
- Back-run after Carnage executes to sell at the higher price

**Mitigation**: `CARNAGE_SLIPPAGE_BPS_FALLBACK = 7500` (75% floor) limits the maximum extraction per Carnage swap to 25% of expected output. The fallback path should only be reached when the atomic path fails.

### 5. Crank Operator Information Advantage

As identified in H016, the crank operator who builds TX3 has a temporal information advantage:

1. The crank calls `randomness.revealIx()` via the Switchboard SDK, which queries the oracle gateway
2. The SDK returns the reveal instruction containing the VRF bytes
3. The crank operator could inspect these bytes off-chain to learn:
   - Whether Carnage triggers (byte 5 < 11, ~4.3% chance)
   - Target token (byte 7)
   - New tax rates (bytes 0-4)
4. The operator could insert personal swaps ahead of TX3 using Jito bundles

This is identical to the vector analyzed in H016 and is an inherent property of any commit-reveal VRF system where the committer has financial interest.

### 6. Front-Running Window Calculation

| Path | Window Duration | VRF Bytes Known? | Carnage Intent Known? | Slippage Floor |
|------|----------------|-------------------|-----------------------|----------------|
| Atomic (normal) | 0 slots | No (same TX) | No (same TX) | 85% |
| Mempool observation | ~1 slot (400ms) | Theoretically (TX parsing) | Cannot distinguish trigger vs no-op | 85% |
| Fallback | 50-300 slots (20-120s) | Yes (on-chain) | Yes (on-chain state readable) | 75% |
| Crank operator | Off-chain (pre-TX3) | Yes (SDK response) | Yes (can compute from bytes) | 85% |

### 7. Cross-Reference with H016

H016 analyzed the same VRF visibility issue from the **tax rate arbitrage** angle and concluded LOW severity. The key mitigations identified there apply here as well:

- Atomic TX3 bundling eliminates the on-chain reveal-before-consume gap
- Crank operator advantage requires MEV infrastructure (Jito bundles)
- 50% output floor on user swaps limits collateral damage from rate changes

H030 focuses specifically on Carnage swap front-running. The additional consideration is that Carnage swaps are larger (up to 1000 SOL) and more predictable (known direction, known pool) than user swaps, making them a higher-value MEV target on the fallback path.

## Verdict

**LOW severity.** The primary attack vector (observing VRF bytes before Carnage executes) is closed by the three-instruction atomic bundle in TX3. The residual risk has two components:

1. **Fallback path MEV** (slots 50-300): On-chain Carnage state is publicly readable, allowing sandwich attacks bounded by the 75% slippage floor. This path should rarely be reached -- it requires the atomic path to fail first. Maximum extraction: 25% of Carnage swap amount (up to 250 SOL on a 1000 SOL swap).

2. **Crank operator advantage**: The operator learns VRF bytes off-chain before submitting TX3. Exploiting this requires MEV infrastructure and currently only one Railway-hosted crank exists.

Neither vector enables unbounded extraction. The atomic bundling is the primary defense and it is correctly implemented.

## Mitigating Factors

1. **Atomic TX3 bundle**: `revealIx + consumeIx + carnageIx` in one transaction -- zero on-chain intermediate state
2. **No-op guard on executeCarnageAtomic**: Safe to always include -- does not reveal Carnage intent to mempool observers
3. **85% slippage floor (atomic path)**: Limits extraction even if mempool-sandwiched
4. **75% slippage floor (fallback path)**: Bounds worst-case extraction on the fallback
5. **50-slot lock window**: Prevents fallback execution during the critical post-reveal period
6. **~4.3% Carnage trigger probability**: Most epochs have no Carnage, reducing MEV opportunities
7. **Carnage swaps capped at 1000 SOL**: Bounds maximum value at risk

## Residual Risk

The fallback path (slots 50-300) creates a window where Carnage swap details are fully public. If this path is reached with a large vault balance, an MEV bot can extract up to 25% of the swap amount. This is mitigated by the rarity of reaching the fallback path (requires atomic path failure) and the slippage floor.

**Recommendation**: If fallback path MEV becomes a concern at mainnet:
- Tighten `CARNAGE_SLIPPAGE_BPS_FALLBACK` from 7500 to 8000+ (reduces max extraction from 25% to 20%)
- Add competitive crank operation to reduce single-operator information advantage
- Consider a commit-execute pattern for the fallback path (caller commits to execute, then executes after 1 slot, preventing same-block sandwich)
