# H058 -- CPI Depth at Solana Limit

**Status:** CONFIRMED
**Severity:** MEDIUM
**Category:** CPI/External, Architecture
**Confidence:** HIGH

## Summary

The Carnage execution CPI chain (Epoch -> Tax -> AMM -> Token-2022 -> Transfer Hook) reaches exactly Solana's 4-level CPI depth limit. This is correctly implemented and functional today, but represents a structural fragility: any future code change that adds a CPI call anywhere in this 5-program chain will cause a silent runtime failure (InstructionError::CallDepth). The codebase contains prominent documentation warnings about this constraint, but there is no compile-time or test-time enforcement that depth stays within limits.

All other CPI chains in the protocol operate at depth 3 or below and have headroom.

## Complete CPI Chain Map

### Chain 1: Carnage Swap (DEEPEST -- depth 4)

```
Depth 0: User TX -> Epoch::execute_carnage_atomic (entry point)
  [Pre-swap at depth 0: Token-2022::approve, System::transfer, Token-2022::sync_native]
  Depth 1: Epoch -> Tax::swap_exempt          (invoke_signed, carnage_signer PDA)
    Depth 2: Tax -> AMM::swap_sol_pool        (invoke_signed, swap_authority PDA)
      Depth 3: AMM -> Token-2022::transfer_checked  (invoke_signed, pool PDA)
        Depth 4: Token-2022 -> TransferHook::execute  (runtime CPI)  <-- SOLANA LIMIT
```

Same chain applies to `execute_carnage` (fallback path). The pre-swap operations (burn, approve, SOL wrap) execute at depth 0 before the swap chain begins, so they do not contribute to the maximum depth.

**Key observation:** The AMM performs TWO transfer_checked calls per swap (input transfer + output transfer). In SOL pools, one side is WSOL (SPL Token, no hook) and the other is CRIME/FRAUD (Token-2022, with hook). Only the Token-2022 side triggers the Transfer Hook at depth 4. If both sides were Token-2022 with hooks, the second transfer would also hit depth 4 -- this is safe because they are sequential, not nested.

## Transfer Hook Does NOT Add Extra Depth

The Transfer Hook program (`transfer_hook.rs`) performs **zero outgoing CPI calls**. Its handler:
1. Validates amount > 0
2. Checks mint owner == Token-2022
3. Checks `transferring` flag (anti-direct-invocation)
4. Checks whitelist PDAs via `Pubkey::find_program_address` (pure computation, no CPI)
5. Returns Ok(())

This is critical for Chain 1: if the Transfer Hook made any CPI call, it would exceed depth 4 and the entire Carnage swap path would fail at runtime.

**Lines 77-113 -- Handler has no invoke/invoke_signed calls, only pure computation.**

## Risk Assessment

### Current Risk: LOW
The chain works correctly today. Pre-swap operations (burn, approve, SOL wrap) are deliberately placed at depth 0 before the swap chain. This is a sound architectural decision.

### Future Risk: MEDIUM
1. **No compile-time enforcement:** A developer adding a CPI call inside `swap_exempt` or `swap_sol_pool` would not get a compiler warning. The failure would only manifest at runtime.
2. **No integration test coverage:** The test `test_cpi_access_control.rs:1257` documents that CPI chains complete within limits but does not programmatically assert max depth.
3. **Comment drift:** The depth warnings exist in 2 files (execute_carnage_atomic.rs, swap_exempt.rs) but not in AMM's swap_sol_pool.rs or the Transfer Hook, where a change would also break the chain.
4. **Agave 3.0 consideration:** The upcoming Agave 3.0 runtime increases CPI depth limit from 4 to 8. While this doesn't break existing code, the repeated "SOLANA LIMIT" comments will become misleading, and developers may incorrectly assume they have more room in the Carnage path specifically.

### Fragility Surface
Any of these changes would break the Carnage swap path:
- Adding a CPI call inside Transfer Hook's handler
- Adding a CPI call inside AMM's `swap_sol_pool` (beyond the existing Token-2022 transfers)
- Adding a CPI call inside Tax's `swap_exempt` (beyond the AMM CPI)
- Wrapping the Epoch -> Tax CPI in an additional intermediary program

## Mitigating Factors

1. **Prominent documentation:** Both files in the deepest chain have clear "DO NOT add CPI calls" warnings with depth annotations.
2. **Pre-swap isolation:** Burn/approve/wrap operations execute before the swap chain at depth 0, showing deliberate depth management.
3. **Transfer Hook is pure computation:** The hook program makes no CPI calls, which is correct and necessary for this architecture.
4. **Sequential, not nested:** The AMM's two transfer_checked calls are sequential (input then output), not nested. Each independently reaches depth 3/4 but they don't compound.
5. **Integration tests exist:** Carnage end-to-end tests exercise the full 4-deep chain and would catch depth violations.

## Recommendations

1. **Add a depth-tracking integration test** that explicitly asserts the Carnage CPI chain succeeds, with a companion test that adds a mock CPI at depth 5 and verifies it fails. This creates a regression tripwire.
2. **Add depth comments to AMM's swap_sol_pool.rs** noting that it operates at depth 2 in the Carnage chain, so developers modifying it understand the constraint.
3. **When Agave 3.0 ships**, update the "SOLANA LIMIT" comments to reflect the new limit (8) but keep the architectural constraint documentation (the current chain works at depth 4 regardless of the runtime limit).
4. **Consider a CI lint** that greps for `invoke_signed` or `invoke` additions in the critical path files (swap_exempt.rs, swap_sol_pool.rs, transfer_hook.rs) and flags them for manual review.

## Verdict

**CONFIRMED as MEDIUM severity.** The CPI depth is correctly managed today and the code functions properly. The risk is architectural fragility -- a single misplaced CPI call in any of 4 programs would silently break the entire Carnage execution path at runtime, with no compile-time protection. The existing comment-based warnings are the sole defense. This is an inherent limitation of the protocol's multi-program design and Solana's CPI depth model.
