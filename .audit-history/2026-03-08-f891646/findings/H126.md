# H126 — Staking: StakePool Singleton Collision in Tests

**Status:** REJECTED (Test-only concern, production seeds correct)
**Severity:** N/A (Informational)
**Confidence:** HIGH
**Target:** `programs/staking/src/state/stake_pool.rs`, `programs/staking/src/constants.rs`

## Hypothesis

The StakePool PDA is a singleton derived from a single seed. If production seeds are incorrect or if there is a collision risk, it could compromise the staking system.

## Evidence

### PDA Seed Definition

From `programs/staking/src/constants.rs:53`:
```rust
pub const STAKE_POOL_SEED: &[u8] = b"stake_pool";
```

### Consistent Usage Across All Instructions

Every instruction that touches StakePool uses the same seed constant:

| Instruction | File | Seed Used |
|-------------|------|-----------|
| initialize_stake_pool | `initialize_stake_pool.rs:39` | `seeds = [STAKE_POOL_SEED]` |
| stake | `stake.rs:36` | `seeds = [STAKE_POOL_SEED]` |
| unstake | `unstake.rs:44` | `seeds = [STAKE_POOL_SEED]` |
| claim | `claim.rs:38` | `seeds = [STAKE_POOL_SEED]` |
| deposit_rewards | `deposit_rewards.rs:47` | `seeds = [STAKE_POOL_SEED]` |
| update_cumulative | `update_cumulative.rs:47` | `seeds = [STAKE_POOL_SEED]` |

All 6 instructions reference the same `STAKE_POOL_SEED` constant — no hardcoded strings, no divergent seeds.

### Architecture Confirmation

From `.audit/ARCHITECTURE.md` Section 4:
> StakePool | Staking | `[b"stake_pool"]` | total_staked, rewards_per_token_stored (54 bytes)

The architecture document confirms the singleton seed derivation matches the code.

### Test Isolation (Already Known)

Per project MEMORY.md: "StakePool PDA is singleton. Separate test files (staking.ts, token-flow.ts, security.ts) must run in separate validators to avoid PDA conflicts."

This is a test infrastructure concern only — separate test files sharing a validator will collide on the singleton PDA. This has no production impact since there is exactly one StakePool per deployed program.

### Double-Init Protection

The `initialize_stake_pool` instruction uses Anchor's `init` constraint, which allocates the account. A second call would fail with `AccountAlreadyInUse` since the PDA already exists.

## Conclusion

Production seeds are correct and consistently used across all 6 instructions via a single constant. The singleton collision concern is purely a test infrastructure issue (already documented and mitigated by running tests in separate validators). No production vulnerability.
