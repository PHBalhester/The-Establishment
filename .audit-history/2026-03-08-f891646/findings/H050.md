# H050 — remaining_accounts Forwarding Without Validation

| Field | Value |
|-------|-------|
| **Hypothesis** | Tax Program forwards `remaining_accounts` to AMM without application-level validation; an attacker could inject unexpected accounts that alter AMM or Token-2022 Transfer Hook behavior |
| **Severity** | Informational |
| **Status** | Invalid (False Positive) |
| **File** | `programs/tax-program/src/instructions/swap_sol_buy.rs:242-248` |
| **Also affects** | `swap_sol_sell.rs:211-213`, `swap_exempt.rs` (same pattern) |

## Description

The Tax Program's `swap_sol_buy` handler iterates `ctx.remaining_accounts` and appends each entry to the AMM CPI's `account_metas` and `account_infos` without performing any owner, key, or type validation:

```rust
// swap_sol_buy.rs:242-248
for account in ctx.remaining_accounts.iter() {
    if account.is_writable {
        account_metas.push(AccountMeta::new(account.key(), account.is_signer));
    } else {
        account_metas.push(AccountMeta::new_readonly(account.key(), account.is_signer));
    }
}
```

The AMM (`swap_sol_pool.rs`) receives these as its own `ctx.remaining_accounts` and forwards them again to `transfer_t22_checked` (`transfers.rs:96-102`), which appends them to the `spl_token_2022::transfer_checked` instruction for Transfer Hook resolution.

The hypothesis is that an attacker could inject unexpected accounts to change AMM swap behavior or hijack Transfer Hook execution.

## Analysis

### Full forwarding chain

1. **Tax Program** (`swap_sol_buy.rs:242-248`): Blindly forwards `remaining_accounts` into AMM CPI account metas.
2. **AMM** (`swap_sol_pool.rs:210-315`): Passes its `ctx.remaining_accounts` to `transfer_t22_checked`.
3. **`transfer_t22_checked`** (`transfers.rs:96-123`): Appends hook accounts to both `ix.accounts` and `account_infos` of the `spl_token_2022::instruction::transfer_checked` instruction.
4. **Token-2022 runtime**: Executes `transfer_checked`, resolving Transfer Hook accounts against the mint's `ExtraAccountMetaList` PDA.
5. **Transfer Hook** (`transfer_hook.rs`): Validates `check_is_transferring` flag and whitelist PDA derivation.

### Why injection is not exploitable

**Token-2022 validates hook accounts deterministically.** The `ExtraAccountMetaList` PDA (initialized in `initialize_extra_account_meta_list.rs:36-55`) defines exactly which extra accounts are required for each transfer:

- Whitelist PDA for source: `seeds = ["whitelist", source_token_account.key()]`
- Whitelist PDA for destination: `seeds = ["whitelist", destination_token_account.key()]`

Token-2022's `transfer_checked` implementation resolves these seeds at runtime and validates that the provided accounts match. If the wrong accounts are provided:
- **Wrong hook accounts**: Token-2022 CPI fails with an account mismatch error. The entire transaction reverts.
- **Extra accounts beyond expected**: Token-2022 ignores accounts beyond what `ExtraAccountMetaList` specifies. They are present in the instruction but never read or acted upon.
- **Fewer accounts than expected**: Token-2022 CPI fails because required hook accounts are missing.

**AMM swap logic does not read remaining_accounts.** The AMM's `swap_sol_pool` handler only uses `ctx.remaining_accounts` as a passthrough to `transfer_t22_checked`. It never indexes into, reads from, or branches on remaining_accounts. Swap math, reserve updates, and the k-invariant check are all computed from typed Anchor accounts (`pool`, `vault_a`, `vault_b`), not from remaining_accounts.

**Transfer Hook validates call context.** The `check_is_transferring` function (`transfer_hook.rs:140-153`) reads the `TransferHookAccount` extension from the source token account to verify the `transferring` flag is set. This flag is only set by Token-2022 during a legitimate `transfer_checked` CPI, preventing direct invocation of the hook with crafted accounts.

### Worst-case scenario: DoS only

If a malicious client provides incorrect remaining_accounts, the Token-2022 CPI fails and the entire transaction reverts. The user loses only the transaction fee. No state is corrupted, no funds are moved, and no hook logic is bypassed. This is equivalent to submitting any malformed transaction.

### Industry pattern

This "forward remaining_accounts to Token-2022 for hook resolution" pattern is the standard approach across the Solana DeFi ecosystem (Raydium, Orca, Jupiter) for supporting Token-2022 Transfer Hooks. Application-level validation of hook accounts is intentionally delegated to Token-2022, which has authoritative knowledge of each mint's ExtraAccountMetaList.

## Verdict

**Invalid (False Positive).** The absence of application-level validation on `remaining_accounts` is by design. Token-2022 performs deterministic validation of hook accounts against the mint's `ExtraAccountMetaList` PDA. Wrong accounts cause transaction failure (DoS to self), not exploitation. The AMM never interprets remaining_accounts for swap logic. This matches the industry-standard pattern for Transfer Hook support.
