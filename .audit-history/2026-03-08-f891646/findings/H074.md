# H074 -- AMM burn_admin Irreversibility

**Status:** Confirmed Safe
**Severity:** Informational
**Confidence:** High

## Summary

The `burn_admin` instruction correctly and irreversibly removes admin privileges by setting `admin_config.admin` to `Pubkey::default()`. No instruction in the AMM program can restore admin after burning.

## Evidence

**burn_admin.rs (line 22):**
```rust
admin_config.admin = Pubkey::default();
```

The `has_one = admin` constraint on `BurnAdmin` (line 47) ensures only the current admin can invoke this. After burning, the admin field is `Pubkey::default()` (the 32-byte zero address). No private key exists for this address, so no one can sign as it.

**No reinitialize path exists.** The AMM program has exactly 4 instructions:
1. `initialize_admin` -- uses `init` (Anchor one-shot; fails if PDA already exists)
2. `initialize_pool` -- `has_one = admin` (fails with `Pubkey::default()`)
3. `burn_admin` -- `has_one = admin` (fails with `Pubkey::default()`)
4. `swap_sol_pool` -- CPI-gated, no admin interaction

`initialize_admin` cannot be called again because Anchor's `init` constraint will reject the transaction when the AdminConfig PDA already exists and has a nonzero balance. The PDA is never closed, so reinitialization is impossible.

**Upgrade authority caveat:** The program upgrade authority could deploy new code that adds an `unburn_admin` instruction. This is standard Solana trust model -- program upgradeability is a separate concern (covered by Trust Tier 2 in ARCHITECTURE.md). The on-chain code as written is irreversible.

## Recommendation

None required. The pattern is sound.
