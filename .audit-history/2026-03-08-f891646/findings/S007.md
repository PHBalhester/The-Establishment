# S007: Cross-Program Layout Test Coverage

**Determination:** CONFIRMED (no cross-program layout tests exist)
**Severity:** HIGH
**Confidence:** HIGH

## Hypothesis

No cross-program layout tests exist to verify that struct mirrors and byte-offset reads remain consistent across programs.

## Evidence

### Cross-Program Layout Couplings Identified

The protocol has three critical layout couplings:

**1. EpochState Mirror (Tax Program <-> Epoch Program)**

Tax Program maintains a full copy of EpochState at `programs/tax-program/src/state/epoch_state_reader.rs`. This struct MUST field-for-field match `programs/epoch-program/src/state/epoch_state.rs`. Both declare 100 bytes of data (108 with discriminator).

The mirror is used to deserialize cross-program reads. Any field reordering, addition, or type change in the Epoch Program's EpochState would cause the Tax Program to read corrupted data (wrong tax rates applied to every swap).

**2. AMM PoolState Byte Offsets (Tax Program + Epoch Program -> AMM)**

Two programs read AMM pool reserves at hardcoded byte offsets:

- Tax Program (`programs/tax-program/src/helpers/pool_reader.rs:46-55`): reads `[137..145]` and `[145..153]`
- Epoch Program (`programs/epoch-program/src/instructions/execute_carnage.rs:874-878` and `execute_carnage_atomic.rs:941-945`): same offsets

These offsets assume the AMM PoolState layout:
```
[0..8]     discriminator
[8]        pool_type (1 byte enum)
[9..41]    mint_a (32 bytes)
[41..73]   mint_b (32 bytes)
[73..105]  vault_a (32 bytes)
[105..137] vault_b (32 bytes)
[137..145] reserve_a (u64)
[145..153] reserve_b (u64)
```

If the AMM PoolState struct adds a field before `reserve_a` (e.g., inserting a `u64 total_volume` after `vault_b`), both Tax and Epoch programs would read `total_volume` as `reserve_a`, producing invalid slippage calculations and potentially enabling value extraction via Carnage.

**3. Staking Discriminator (Tax Program -> Staking Program)**

Tax Program hardcodes `DEPOSIT_REWARDS_DISCRIMINATOR: [u8; 8]` at `programs/tax-program/src/constants.rs:152`. This is the sha256 hash of `"global:deposit_rewards"`. If the Staking instruction is renamed, this discriminator becomes invalid and all tax distributions to stakers silently fail.

### Test Coverage Search Results

Searched all test directories for cross-program layout verification:

- `tests/` -- 7 test files, none contain layout assertions
- `tests/integration/` -- `cpi-chains.test.ts` and `smoke.test.ts` test CPI flows but do NOT verify struct layouts
- `tests/cross-program-integration.ts` -- Tests CPI authority gating, not data layout
- Individual program `tests/` directories -- Unit tests for their own logic only

Specific search for layout-related patterns (`layout`, `cross-program`, `byte offset`, `struct match`) returned zero results in any test file that actually verifies struct field ordering or byte positions.

### Existing Safeguards (Partial)

The Epoch Program has a static assertion:
```rust
// programs/epoch-program/src/state/epoch_state.rs:172
const _: () = assert!(EpochState::DATA_LEN == 100);
```

This catches size changes but NOT field reordering. Two structs can both be 100 bytes with completely different field orders.

The Tax Program's mirror has a manual size comment but no `const _:` assertion:
```rust
// programs/tax-program/src/state/epoch_state_reader.rs:51
// Total: 20 + 5 + 8 + 42 + 23 + 2 = 100 bytes
```

## Recommended Test Structure

### Test 1: EpochState Layout Parity (Rust unit test)

Add to a shared test crate or `programs/tax-program/tests/`:

```rust
#[test]
fn epoch_state_layout_parity() {
    // Serialize a known EpochState from the Epoch Program
    let canonical = epoch_program::state::EpochState {
        genesis_slot: 0x1111111111111111,
        current_epoch: 0x22222222,
        epoch_start_slot: 0x3333333333333333,
        cheap_side: 0x44,
        low_tax_bps: 0x5555,
        high_tax_bps: 0x6666,
        // ... all fields with unique sentinel values
    };
    let canonical_bytes = canonical.try_to_vec().unwrap();

    // Deserialize as Tax Program's mirror
    let mirror: tax_program::state::EpochState =
        AnchorDeserialize::deserialize(&mut &canonical_bytes[..]).unwrap();

    // Field-by-field comparison
    assert_eq!(canonical.genesis_slot, mirror.genesis_slot);
    assert_eq!(canonical.current_epoch, mirror.current_epoch);
    assert_eq!(canonical.cheap_side, mirror.cheap_side);
    assert_eq!(canonical.low_tax_bps, mirror.low_tax_bps);
    assert_eq!(canonical.high_tax_bps, mirror.high_tax_bps);
    assert_eq!(canonical.crime_buy_tax_bps, mirror.crime_buy_tax_bps);
    assert_eq!(canonical.crime_sell_tax_bps, mirror.crime_sell_tax_bps);
    assert_eq!(canonical.fraud_buy_tax_bps, mirror.fraud_buy_tax_bps);
    assert_eq!(canonical.fraud_sell_tax_bps, mirror.fraud_sell_tax_bps);
    // ... remaining fields
}
```

### Test 2: AMM PoolState Byte Offset Stability (Rust unit test)

```rust
#[test]
fn pool_state_reserve_offsets() {
    let pool = amm::state::PoolState {
        reserve_a: 0xAAAAAAAAAAAAAAAA,
        reserve_b: 0xBBBBBBBBBBBBBBBB,
        // ... fill other fields
    };
    let bytes = pool.try_to_vec().unwrap();
    // Prepend 8-byte discriminator for full account layout
    let mut account_data = vec![0u8; 8];
    account_data.extend_from_slice(&bytes);

    let ra = u64::from_le_bytes(account_data[137..145].try_into().unwrap());
    let rb = u64::from_le_bytes(account_data[145..153].try_into().unwrap());
    assert_eq!(ra, 0xAAAAAAAAAAAAAAAA);
    assert_eq!(rb, 0xBBBBBBBBBBBBBBBB);
}
```

### Test 3: Staking Discriminator Stability (Rust unit test)

```rust
#[test]
fn deposit_rewards_discriminator_matches_staking() {
    use sha2::{Digest, Sha256};
    let mut hasher = Sha256::new();
    hasher.update(b"global:deposit_rewards");
    let hash = hasher.finalize();
    let expected: [u8; 8] = hash[0..8].try_into().unwrap();
    assert_eq!(
        tax_program::constants::DEPOSIT_REWARDS_DISCRIMINATOR,
        expected
    );
}
```

Note: Test 3 already exists in Tax Program unit tests (`programs/tax-program/src/constants.rs:209-219`), but Tests 1 and 2 do not exist anywhere.

### Implementation Approach

These tests require cross-crate imports. Options:
- **Workspace-level test crate** (`tests/layout/`) with Cargo dependencies on both `epoch-program` and `tax-program` (using `lib` crate type, not `cdylib`)
- **Integration test file** in one program that imports the other via `path` dependency with `features = ["cpi"]`

The workspace approach is cleaner and avoids circular dependencies.

## Impact If Not Fixed

From Architecture Section 7.2: "Silent data corruption on layout changes -> wrong tax rates or broken slippage floors." This is rated HIGH by 4 independent audit agents. Without automated layout tests, the next developer who adds a field to EpochState or PoolState will break cross-program reads with zero compiler warnings.

Estimated effort: 4-6 hours to create the test crate and implement all three test categories.
