# S006: BC Authority + Hook Init -- Combined Deployment Attack

**Determination:** CONFIRMED (combined attack amplifies both H001 and H007)
**Severity:** CRITICAL
**Confidence:** HIGH

## Hypothesis

An attacker combines Transfer Hook authority capture (H007 vector) with Bonding Curve SOL theft (H001-adjacent, specifically the BC authority gap from Architecture Section 7.1) in a coordinated mainnet deployment attack.

## Prerequisite Findings

### BC Authority Gap (Architecture Section 7.1, 4.3-4.4)

The Bonding Curve program's 6 admin instructions accept ANY `Signer<'info>` named `authority` with ZERO on-chain verification:
- `initialize_curve` -- any signer creates curve
- `start_curve` -- any signer starts trading
- `prepare_transition` -- any signer forces graduation
- `withdraw_graduated_sol` -- any signer receives ~1000 SOL per curve
- `close_token_vault` -- any signer receives rent
- `fund_curve` -- any signer (needs tokens)

From `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs:27-28`:
```rust
/// Protocol authority (deployer). Must sign. Receives SOL.
#[account(mut)]
pub authority: Signer<'info>,
```

No `has_one`, no PDA check, no upgrade authority verification. The comment says "Protocol authority (deployer)" but the constraint is merely `Signer<'info>`.

### Transfer Hook Init Front-Running (Architecture Section 7.3)

From `programs/transfer-hook/src/instructions/initialize_authority.rs:15-17`:
```rust
pub fn handler(ctx: Context<InitializeAuthority>) -> Result<()> {
    let auth = &mut ctx.accounts.whitelist_authority;
    auth.authority = Some(ctx.accounts.signer.key());
```

First caller wins. Any signer becomes the authority. The `init` constraint prevents re-calling, but whoever calls first controls the whitelist permanently.

## Combined Attack Scenario

### Attack Timeline

**Pre-deployment monitoring:**
1. Attacker monitors the deployer's wallet and the bonding curve / transfer hook program deployments via Solana RPC or block explorers.

**Phase 1: Hook Authority Capture (seconds after program deploy)**
2. Attacker front-runs `initialize_authority` on the Transfer Hook program.
3. Attacker now controls which addresses can transfer CRIME and FRAUD tokens.
4. Attacker does NOT burn the authority (preserving ongoing control).

**Phase 2: Selective Whitelist Control**
5. Attacker whitelist-blocks the deployer's graduation orchestration addresses.
6. Alternatively, attacker ransoms the whitelist (demands payment to whitelist the protocol's PDAs).

**Phase 3: BC SOL Extraction (after curves fill)**
7. Attacker waits for both bonding curves to fill (~1000 SOL each).
8. Attacker calls `prepare_transition` on both curves (any signer accepted).
9. Attacker calls `withdraw_graduated_sol` on both curves (any signer, SOL sent to attacker).
10. ~2000 SOL extracted.

**Phase 4: Maximum Damage**
11. With hook authority, attacker can freeze all token transfers by removing whitelist entries.
12. With SOL stolen from curves, AMM pools cannot be seeded.
13. Protocol is permanently bricked: no transfers, no liquidity, no recovery.

### Why This Combination Is Worse Than Either Alone

| Attack | Alone | Combined |
|--------|-------|----------|
| Hook authority capture | Ransom/grief (reversible via program upgrade) | Permanent freeze + extraction cover |
| BC SOL theft | ~2000 SOL loss (protocol can still function after re-funding) | ~2000 SOL + protocol permanently bricked |

The combination creates a **kill shot**: hook authority freeze prevents any token movement (including refunds to curve participants), while BC SOL theft drains the graduation funds. The protocol cannot recover without a full redeploy of both programs.

### Attack Cost

- Gas only (~0.01 SOL for the initialization + withdrawal transactions)
- No tokens needed
- No special hardware or MEV infrastructure needed
- Can be executed by a simple bot monitoring for program deployment

## Feasibility Assessment

**On devnet:** Low risk (controlled deployment, no real value).

**On mainnet with current code:** The attack window is the gap between program deployment and the deployer's own initialization transactions. On Solana mainnet with ~400ms block times and priority fees, an attacker monitoring the mempool could land their initialization TX in the same block or within 1-2 blocks of the deployer's deploy TX.

The attack is **deterministic, not probabilistic**. If the attacker's TX lands before the deployer's init TX, the attack succeeds with 100% certainty.

## Mitigation

Both underlying issues must be fixed:

1. **BC Authority:** Replicate AMM's ProgramData upgrade-authority pattern. The `authority` account must be verified against the program's upgrade authority via `ProgramData` account, not accepted as bare `Signer`.

2. **Hook Init:** Either:
   - Gate `initialize_authority` behind ProgramData upgrade authority check (same as AMM admin init)
   - Or use an atomic deployment script that deploys + initializes in a single block (reduces window but does not eliminate it)

3. **Deployment script:** Add initialization transactions to the same block as program deployment using Jito bundles or transaction batching. This minimizes the front-running window but is not a substitute for on-chain access control.

Estimated effort: 4-8 hours for BC authority fix, 2-4 hours for Hook init fix.
