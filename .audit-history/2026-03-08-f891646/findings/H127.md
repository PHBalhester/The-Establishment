# H127 — Bonding Curve: PRECISION Mismatch (1e12 vs 1e18)

**Status:** REJECTED (No cross-program precision sharing)
**Severity:** N/A
**Confidence:** HIGH
**Target:** `programs/bonding_curve/src/constants.rs`, `programs/staking/src/constants.rs`

## Hypothesis

The Bonding Curve program uses `PRECISION = 1e12` while the Staking program uses `PRECISION = 1e18`. If these precision values are shared across program boundaries (via CPI arguments or shared state), arithmetic corruption could result.

## Evidence

### Precision Definitions

**Bonding Curve** (`programs/bonding_curve/src/constants.rs:11`):
```rust
pub const PRECISION: u128 = 1_000_000_000_000; // 1e12
```

**Staking** (`programs/staking/src/constants.rs:21`):
```rust
pub const PRECISION: u128 = 1_000_000_000_000_000_000; // 1e18
```

### Cross-Program Interaction Check

**Bonding Curve references to Staking:** Zero matches. Grep for `staking|stake_pool|STAKING` in `programs/bonding_curve/src/` returned no results.

**Staking references to Bonding Curve:** Zero matches. Grep for `bonding_curve|bonding\.curve` in `programs/staking/src/` returned no results.

### Program Interaction Model

Per the Architecture Document (Section 3), the Bonding Curve program:
- Is a standalone program with no CPI to other protocol programs
- Does not call Tax, AMM, Staking, or Epoch programs
- Has its own self-contained math in `programs/bonding_curve/src/math.rs`
- After graduation, SOL is withdrawn and tokens are seeded into AMM pools — but this is an off-chain orchestration step, not a CPI

The Staking program:
- Is called by Tax Program (deposit_rewards) and Epoch Program (update_cumulative)
- Has no awareness of Bonding Curve at all

### Why Different Precisions Are Appropriate

- **Bonding Curve (1e12):** Used for price curve integration math (`P_START * PRECISION * N`). The intermediate values involve token amounts up to 460M * 1e6 decimals = 4.6e14, multiplied by PRECISION. Using 1e12 keeps products within u128 range while providing adequate precision for lamport-denominated outputs.

- **Staking (1e18):** Used for cumulative reward-per-token tracking (`pending_rewards * PRECISION / total_staked`). The DeFi-standard 1e18 provides 18 decimal places of precision, critical for fair pro-rata distribution over long timeframes where small rounding errors compound.

### No Shared State

These programs share no accounts, no CPI calls, and no serialized precision values. Each program's PRECISION is purely internal to its own arithmetic.

## Conclusion

The precision difference is intentional and safe. Each program uses PRECISION exclusively within its own internal arithmetic — there is zero cross-program precision sharing. No data flows between these two programs at any layer (CPI, shared accounts, or serialized state).
