# H063 -- Staking update_cumulative Timing

**Auditor:** Hypothesis Investigator (sos-002)
**Severity:** INFORMATIONAL
**Status:** SAFE -- No Exploitable Ordering Vulnerability
**Confidence:** HIGH
**Target:** `programs/staking/src/instructions/update_cumulative.rs`, `programs/staking/src/instructions/deposit_rewards.rs`

---

## Hypothesis

`update_cumulative` (called by Epoch Program) updates `rewards_per_token_stored` by consuming `pending_rewards`. If called at the wrong time relative to `deposit_rewards` (called by Tax Program on every swap), rewards could be distributed incorrectly -- either lost, double-counted, or attributed to the wrong epoch's stakers.

## Architecture

### Two-Phase Reward Flow

1. **Accumulation phase** (`deposit_rewards`, called per-swap by Tax Program):
   - Increments `StakePool.pending_rewards += amount`
   - Verifies escrow balance >= pending_rewards (reconciliation)
   - Can be called many times per epoch (once per taxed swap)

2. **Finalization phase** (`update_cumulative`, called once per epoch by Epoch Program via `consume_randomness`):
   - Reads `pending_rewards`
   - Computes `reward_per_token = pending_rewards * PRECISION / total_staked`
   - Adds to `rewards_per_token_stored` (monotonically increasing cumulative)
   - Clears `pending_rewards = 0`
   - Guards via `epoch > pool.last_update_epoch` (prevents double-update)

### Call Sites

- `deposit_rewards`: Called from `tax-program/src/instructions/swap_sol_buy.rs` and `swap_sol_sell.rs` after SOL transfer to staking escrow. CPI-gated via `tax_authority` PDA with `seeds::program = tax_program_id()`.
- `update_cumulative`: Called from `epoch-program/src/instructions/consume_randomness.rs` at step 7.5. CPI-gated via `staking_authority` PDA with `seeds::program = epoch_program_id()`.

## Analysis

### 1. Double-Update Prevention (SAFE)

The guard at `update_cumulative.rs:78-81`:
```rust
require!(
    epoch > pool.last_update_epoch,
    StakingError::AlreadyUpdated
);
```
Prevents the same epoch's rewards from being finalized twice. After successful execution, `pool.last_update_epoch = epoch` (line 111). This is sound.

### 2. Ordering Within consume_randomness (SAFE)

In `consume_randomness.rs`, the flow is:
1. Steps 1-7: Validate VRF, derive new tax rates, clear VRF pending
2. Step 7.5: CPI to `update_cumulative` with `epoch_state.current_epoch`
3. Steps 8-9: Emit event, check Carnage

The `update_cumulative` CPI uses `epoch_state.current_epoch` which was already incremented by `trigger_epoch_transition`. This means it finalizes the rewards accumulated during the PREVIOUS epoch. The ordering (derive new rates -> finalize old epoch yield) is correct per the documented design.

### 3. Same-Slot Race Between deposit_rewards and update_cumulative (INFORMATIONAL)

If a swap transaction (`deposit_rewards`) and the epoch transition transaction (`update_cumulative` inside `consume_randomness`) land in the same slot:

- **Case A: `deposit_rewards` executes first** -- The swap's tax portion is added to `pending_rewards`. Then `update_cumulative` finalizes ALL pending (including this swap's portion) into the cumulative. Correct behavior.

- **Case B: `update_cumulative` executes first** -- `pending_rewards` is finalized and cleared to 0. Then `deposit_rewards` adds the swap's tax portion to the now-cleared `pending_rewards`. These rewards remain in `pending_rewards` until the NEXT epoch's `update_cumulative`. **This creates a 1-epoch delay in reward attribution**, not a loss.

This is an inherent property of the two-phase design. The rewards are never lost -- they simply wait one additional epoch for finalization. The escrow balance reconciliation in `deposit_rewards` (line 99-102) ensures the SOL is actually present regardless of when the state counter catches up.

### 4. Concurrent deposit_rewards Calls (SAFE)

Multiple swaps in the same slot each call `deposit_rewards`. Since Solana transactions within a slot execute sequentially (not in parallel), and each call does `pending_rewards = pending_rewards.checked_add(amount)`, there is no data race. The accumulator is correctly incremented.

### 5. No deposit_rewards During update_cumulative Window (NON-ISSUE)

There is no lock or mutex preventing `deposit_rewards` from being called while `update_cumulative` is "pending." This is fine because:
- `update_cumulative` happens atomically within a single transaction (`consume_randomness`)
- `deposit_rewards` in a different transaction cannot interleave with it
- Solana's single-threaded slot execution prevents true concurrency

### 6. Stale Rewards Attribution (BY DESIGN)

During the VRF transition window (between `trigger_epoch_transition` and `consume_randomness`, potentially 3+ slots), swaps continue using the old epoch's tax rates. Tax proceeds from these swaps go to `pending_rewards`. When `update_cumulative` finally runs, it distributes ALL accumulated `pending_rewards` to stakers at that moment -- regardless of which epoch the individual deposits came from. This is a standard Synthetix-style batching pattern and is correct.

### 7. Forfeited Rewards Re-Entry (SAFE)

When a user unstakes, their `rewards_earned` is forfeited back to `pending_rewards` (`unstake.rs:162-167`). These forfeited rewards join the accumulation pool and are distributed to remaining stakers at the next `update_cumulative`. The forfeiture amount is bounded by what was originally deposited (verified by proptest property T21 in `math.rs`).

## Verdict

**INFORMATIONAL -- No exploitable vulnerability.** The two-phase accumulation-finalization pattern is a well-established design (Synthetix/Quarry) and is correctly implemented here. The ordering constraints are enforced by:
1. CPI gating (only Epoch Program can call `update_cumulative`, only Tax Program can call `deposit_rewards`)
2. Double-update prevention (`epoch > last_update_epoch`)
3. Solana's sequential transaction execution within slots
4. Escrow balance reconciliation as a backstop

The only observable timing effect is that rewards from swaps landing in the same slot as (but after) `consume_randomness` experience a 1-epoch delay before being distributed. This is inherent to the batched design and not exploitable -- the SOL is safely held in the escrow vault throughout.
