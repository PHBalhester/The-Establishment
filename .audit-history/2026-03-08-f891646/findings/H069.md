## Summary

All VRF byte interpretation operations produce mathematically uniform distributions. The only modular operation (`% 4` on u8) divides evenly into 256, and all threshold comparisons use intentional probability targets. There is no modulo bias anywhere in the VRF consumption pipeline.

## Byte Allocation Map

## Detailed Analysis

### Modular Operation: `% 4` on u8 (Bytes 1-4)

**Location:** `tax_derivation.rs:89-94`

```rust
let crime_low_bps  = LOW_RATES[(vrf_result[1] % 4) as usize];
let crime_high_bps = HIGH_RATES[(vrf_result[2] % 4) as usize];
let fraud_low_bps  = LOW_RATES[(vrf_result[3] % 4) as usize];
let fraud_high_bps = HIGH_RATES[(vrf_result[4] % 4) as usize];
```

A u8 has range [0, 255] = 256 values. `256 % 4 == 0`, so each bucket (0, 1, 2, 3) receives exactly 64 values. The distribution is perfectly uniform with zero remainder bias.

For comparison, modulo bias would occur with a modulus that does not evenly divide 256. For example, `% 3` on u8 would give buckets of size 86, 86, 84 (a ~2.3% bias toward values 0 and 1). No such issue exists here.

### Threshold Comparisons (Bytes 0, 5, 6, 7)

Threshold comparisons (`<` against a constant) on a uniformly distributed u8 always produce exact probabilities of `threshold / 256`. There is no modulo bias concern with threshold comparisons -- the only question is whether the intended probability matches the threshold value:

- **Byte 0** (`< 192`): 75% flip probability. `192/256 = 0.75` exactly. Matches spec.
- **Byte 5** (`< 11`): ~4.3% Carnage trigger. `11/256 = 0.04296875`. Matches spec's "~4.3%" target.
- **Byte 6** (`< 5`): ~2% sell action. `5/256 = 0.01953125`. Matches spec's "~2%" target.
- **Byte 7** (`< 128`): 50% CRIME target. `128/256 = 0.5` exactly. Matches spec.

### Byte Selection from VRF Output

**Location:** `consume_randomness.rs:164-171`

```rust
let vrf_result: [u8; 32] = {
    let data = ctx.accounts.randomness_account.try_borrow_data()?;
    let randomness_data = RandomnessAccountData::parse(data)
        .map_err(|_| EpochError::RandomnessParseError)?;
    randomness_data.get_value(clock.slot)
        .map_err(|_| EpochError::RandomnessNotRevealed)?
};
```

Switchboard On-Demand VRF returns 32 bytes of cryptographically uniform randomness. The protocol uses bytes 0-7 sequentially, one per decision. Each byte is:
- Used for exactly one decision (no byte reuse)
- Independent of all other bytes (cryptographic property of VRF output)
- Uniformly distributed over [0, 255]

No byte is used for multiple decisions, eliminating any correlation between tax rates, flip decisions, and Carnage outcomes.

### Boundary Value Verification

All boundary values have been verified through the existing test suite:

- `tax_derivation.rs` tests: Bytes 0, 252-255 tested for `% 4` wrapping behavior
- `carnage.rs` tests: Full range [0, 255] tested for all three Carnage bytes
- `consume_randomness.rs` tests: Boundary values 11 (trigger threshold), 5 (sell threshold), 128 (target threshold) all tested

## Array Bounds Safety

The `% 4` result is used as an array index into `LOW_RATES[4]` and `HIGH_RATES[4]`. Since `u8 % 4` always produces values in [0, 3], array access is always in bounds. The `as usize` cast is safe because u8 values always fit in usize.

## Verdict

**REFUTED.** All VRF byte interpretation operations are mathematically sound:
- The only modular operation (`% 4`) produces a perfectly uniform distribution (256 divides evenly by 4)
- All threshold comparisons produce exact intended probabilities
- Bytes are selected sequentially with no reuse or correlation
- Array bounds are guaranteed safe by the modular arithmetic

No changes recommended.
