# H040: PoolState Struct Manipulation (H067 Recheck)

**Hypothesis:** Pool.rs was modified. Struct changes may affect cross-program byte offset readers, causing silent data corruption in slippage calculations or reserve reads.

**Determination:** NOT VULNERABLE

**Severity:** N/A (Informational -- offsets confirmed correct)

**Confidence:** HIGH

---

## Summary

PoolState (`programs/amm/src/state/pool.rs`) was modified twice since initial creation:

1. **Commit 9c545a5** (feat 11-01): Added `locked: bool` field after `initialized` (byte 156). This is the only structural change.
2. **Commit 7f6972c** (phase-2): Comment-only cleanup -- removed stale PROFIT pool references. No field changes.

The `locked` field was appended after the reserve fields in serialization order. All three cross-program raw byte readers target offsets `[137..145]` (reserve_a) and `[145..153]` (reserve_b), which are **unaffected** by the addition of `locked` at byte 156.

### Pre-Modification Layout (Before 9c545a5)

Same layout but without `locked` at byte 156. Fields after `initialized` shifted down by 1 byte:
- bump was at 155, vault_a_bump at 156, vault_b_bump at 157
- token_program_a at 158..190, token_program_b at 190..222
- INIT_SPACE was 223, total 231

### Key Finding: Reserve Offsets Unchanged

The `locked` field was inserted **after** `reserve_a` (137..145) and `reserve_b` (145..153) in the serialization order. Borsh serializes fields in declaration order. Since `locked` comes after `initialized` (byte 155), and reserves end at byte 153, the addition of `locked` does not shift reserve offsets.

## Structural Risk Assessment

### Why This Modification Was Safe

The `locked` field was added at the **end of the scalar fields section** (after `initialized`, before `bump`). In Borsh serialization, fields are serialized in declaration order without padding or alignment. Since `locked` (1 byte) was inserted between `initialized` (byte 155) and `bump` (previously at byte 155, now at byte 157), only fields AFTER `locked` shifted by 1 byte. The reserve fields at 137-153 were unaffected.

### Future Risk: Adding Fields Before Reserves Would Break Everything

If a field were inserted before `reserve_a` (e.g., between `vault_b` and `reserve_a`), all three cross-program readers would silently read wrong data. There is **no compile-time safety net** for this:

- Tax Program and Epoch Program have no Cargo dependency on the AMM crate
- Byte offsets are hardcoded magic numbers in 3 files across 2 programs
- No shared constant or compile-time assertion validates the offsets

This is documented as assumption A-5 in ARCHITECTURE.md: "AMM PoolState byte offsets (137-153) stable -- Pool.rs MODIFIED -- MUST RE-VERIFY". This finding completes that re-verification.

### Recommendation (Informational)

Add a compile-time assertion in `programs/amm/src/state/pool.rs` to lock the reserve field offsets:

```rust
// Compile-time assertion: reserve_a starts at byte 137 (after discriminator)
// This protects cross-program readers in Tax and Epoch programs.
const _: () = {
    // 8 (disc) + 1 (pool_type) + 32*4 (mints+vaults) = 137
    assert!(8 + 1 + 32 + 32 + 32 + 32 == 137);
    // reserve_a is 8 bytes, so reserve_b starts at 145
    assert!(137 + 8 == 145);
};
```

This would cause a build failure if fields are inserted before reserves, alerting developers to update cross-program readers.

## Pre-Existing Concern: No Owner Check on Pool Account

All three `read_pool_reserves()` call sites accept pool as raw `AccountInfo` without verifying `pool.owner == amm_program_id`. The defense relies on AMM CPI validation downstream, but reserves are read BEFORE the CPI. An attacker could theoretically provide a crafted account with fake reserves to manipulate the slippage floor calculation.

**This is NOT caused by the struct modification** and is tracked separately in ARCHITECTURE.md Section 7.6. It predates all PoolState changes.
