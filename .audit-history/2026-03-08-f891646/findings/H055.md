## Question

Epoch state fields (`cheap_side`, `low_tax_bps`, `high_tax_bps`, `crime_buy_tax_bps`, etc.) have documentary comments indicating valid ranges but no explicit `require!()` range constraints on write. Could a corrupted or out-of-range value be stored?

## Analysis

### 1. Tax Rate Fields Are Write-Constrained by Lookup Table

The only two write paths for tax rate fields are:

**Path A -- Initialization** (`initialize_epoch_state.rs:50-58`):
All values are set from compile-time constants (`GENESIS_LOW_TAX_BPS = 300`, `GENESIS_HIGH_TAX_BPS = 1400`). No user input involved.

**Path B -- VRF Consumption** (`consume_randomness.rs:196-203`):
Tax rates are derived via `derive_taxes()` in `helpers/tax_derivation.rs`. This function selects values exclusively from two hardcoded lookup tables:

```rust
const LOW_RATES: [u16; 4] = [100, 200, 300, 400];
const HIGH_RATES: [u16; 4] = [1100, 1200, 1300, 1400];
```

The VRF byte is used as an index via `vrf_result[n] % 4`, which can only produce values 0, 1, 2, or 3. Array indexing from a 4-element array with a `% 4` modulus makes out-of-bounds indexing mathematically impossible. Therefore the per-token tax BPS fields (`crime_buy_tax_bps`, `crime_sell_tax_bps`, `fraud_buy_tax_bps`, `fraud_sell_tax_bps`) can **only ever** take values from {100, 200, 300, 400, 1100, 1200, 1300, 1400}.

The legacy fields `low_tax_bps` and `high_tax_bps` are set to 0 after VRF consumption (they are no longer used for per-token rates).

### 2. cheap_side Field

- Initialized to `0` (CRIME) at genesis.
- Updated via `tax_config.cheap_side.to_u8()` in consume_randomness. The `Token` enum has only two variants (`Crime = 0`, `Fraud = 1`), and `to_u8()` returns the discriminant. Therefore `cheap_side` can only be 0 or 1.
- Read via `Token::from_u8_unchecked()` which maps 0 -> Crime, anything else -> Fraud. This is a safe fallback; a corrupted value (e.g., 255) would be treated as Fraud rather than causing a panic or undefined behavior.

### 3. carnage_target and carnage_action Fields

- `carnage_target`: Set via `Token::to_u8()`, same analysis as cheap_side -- only 0 or 1.
- `carnage_action`: Set via `CarnageAction::to_u8()`, only values 0 (None), 1 (Burn), 2 (Sell).
- Both are read with `from_u8()` (which returns `Option`, checked) in execute_carnage paths, and with `from_u8_unchecked()` in consume_randomness (safe fallback).

### 4. Downstream Validation in Tax Program

Even if a hypothetically corrupted value somehow reached the tax BPS fields, the tax-program's `calculate_tax()` function (`tax_math.rs:34-53`) includes a guard:

```rust
if tax_bps > 10_000 {
    return None;
}
```

This would cause a `TaxOverflow` error and reject the swap, preventing any economic damage from an impossible-in-practice corrupted rate.

### 5. No External Write Path

EpochState is a PDA owned by the epoch-program. No CPI caller writes to these fields. Only two instructions in the epoch-program modify tax fields: `initialize_epoch_state` and `consume_randomness`. Both are analyzed above.

## Verdict

**No explicit `require!()` range constraints are needed** because the values are structurally constrained:

1. Tax rates are selected from compile-time lookup tables with mathematically bounded indexing -- only 8 discrete values are possible.
2. Enum-to-u8 conversions are bounded by the Rust type system.
3. The downstream consumer (tax-program) has a defense-in-depth guard rejecting rates > 10,000 BPS.

The documented ranges in comments (`low_tax_bps: 100-400`, `high_tax_bps: 1100-1400`) accurately reflect the lookup table outputs. Adding runtime `require!()` assertions would be purely cosmetic -- they could never fire given the current write paths.

## Comparison with H106 (Previous Audit)

H106 flagged "PARTIALLY ADDRESSED" because comments existed but no on-chain range constraints were enforced. This recheck upgrades the status to **SAFE BY DESIGN**: the lookup-table pattern makes out-of-range values structurally impossible, which is stronger than runtime validation (no code path can produce an invalid value, rather than catching one after the fact).
