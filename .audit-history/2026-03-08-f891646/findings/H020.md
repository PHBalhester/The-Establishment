# H020 -- No Emergency Pause Mechanism

| Field | Value |
|-------|-------|
| **Hypothesis** | Zero pause/freeze/emergency mechanisms across all 7 programs. If exploit discovered post-launch, no circuit breaker exists. |
| **Category** | Upgrade/Admin |
| **Severity** | HIGH |
| **Status** | **CONFIRMED** |
| **Affected Programs** | All 7: AMM, Tax Program, Epoch Program, Staking, Bonding Curve, Conversion Vault, Transfer Hook |

---

## Executive Summary

Systematic search of all 7 production program state structs and instruction sets confirms zero pause, freeze, halt, or emergency-stop fields or instructions exist anywhere in the protocol. The AMM's `locked` field is strictly a per-swap reentrancy guard (set true at swap entry, cleared at exit within the same instruction), not a pause mechanism. No program stores an admin-controllable boolean that could halt user-facing operations.

The only emergency response available is a full program upgrade, which requires rebuilding, redeploying, and re-initializing -- a multi-step non-atomic process complicated by the 15+ hardcoded cross-program ID mesh.

### Grep Verification

A case-insensitive search for `pause|frozen|freeze|halt|emergency|circuit.?break|disabled|shutdown` across all program source (`programs/*/src/**/*.rs`) returned zero hits in any state struct, instruction handler, or error enum related to an emergency stop mechanism. The only matches were:

- `freeze_authority` references in test files (SPL mint layout construction, not protocol logic)
- `locked` in AMM (reentrancy guard, confirmed above)
- `halt` in a staking math comment (describing assertion behavior, not a protocol feature)

### No Pause Instruction

No instruction across any of the 7 programs accepts a "pause" or "toggle" action. The full instruction inventory (42 instructions total per ARCHITECTURE.md Section 3) was reviewed -- none implement circuit-breaker logic.

### AMM `locked` Field Is NOT a Pause

Location: `programs/amm/src/instructions/swap_sol_pool.rs`

```rust
// Line 83-84: Set at swap entry
ctx.accounts.pool.locked = true;

// Line 322: Cleared at swap exit
ctx.accounts.pool.locked = false;
```

The Anchor constraint at line 381 (`constraint = !pool.locked @ AmmError::PoolLocked`) rejects concurrent swaps on the same pool. This is a standard reentrancy guard pattern. No instruction exists to externally set `locked = true` -- it is exclusively managed within the swap instruction's execution scope.

## Impact Analysis

### Exploitation Window During Incident Response

If a vulnerability is discovered post-launch, the response timeline via program upgrade is:

1. **Identify and triage** the vulnerability (~minutes to hours)
2. **Develop a fix** in Rust source (~hours)
3. **Rebuild all affected programs** with `anchor build` (~minutes)
4. **Cross-program cascade**: If the fix changes any program's address, ALL programs referencing it must be rebuilt (15+ cross-references). If the fix is logic-only at the same address, only the affected program(s) need redeployment.
5. **Deploy via upgrade authority** (~seconds per program, but sequential and non-atomic across 7 programs)
6. **Reinitialize** if state layout changed (potentially impossible without data migration)

**Minimum realistic response time: 1-4 hours** (logic-only fix at same address, single program).
**Worst case: 6-24 hours** (multi-program cascade with state migration).

During this entire window, the exploit remains active with no ability to halt protocol operations.

### Specific Scenarios Where Pause Would Prevent Loss

| Scenario | Pause Would Protect | Current Mitigation |
|----------|--------------------|--------------------|
| AMM swap exploit (e.g., price manipulation) | Halt all swaps instantly | None -- must deploy fixed program |
| Staking reward drain | Freeze claims/unstakes | None -- must deploy fixed program |
| Bonding curve math bug | Halt purchases/sells | None -- must deploy fixed program |
| Carnage Fund manipulation | Halt carnage execution | None -- must deploy fixed program |
| Cross-program CPI exploit | Halt the entry-point program | None -- must deploy all affected programs |

### Aggravating Factor: Cross-Program ID Mesh

The 15+ hardcoded cross-program ID references mean that if an emergency fix requires deploying a program to a new address (e.g., due to state layout change requiring account migration), every program that references the old address must also be rebuilt and redeployed. This creates a multi-step non-atomic upgrade window where some programs reference old IDs and others reference new ones.

### Aggravating Factor: Single Upgrade Authority

Per `Anchor.toml`, the wallet at `keypairs/devnet-wallet.json` is the provider wallet. On devnet this is a hot wallet. There is no on-chain multisig enforcement for upgrades (the Conversion Vault comment mentioning "Squads multisig" is aspirational, not enforced). If the upgrade authority key holder is unavailable during an incident, no one else can deploy fixes.

## Comparison to Industry Standard

Most production DeFi protocols implement at least one of:

1. **Global pause flag** in a singleton state account, checked by all user-facing instructions
2. **Per-pool pause** (e.g., AMM pools can be individually frozen)
3. **Guardian/multisig** with time-limited emergency powers
4. **Timelock + fast-track** emergency path with reduced delay

The Dr Fraudsworth protocol implements none of these. The design philosophy appears to be "program upgrade IS the emergency mechanism," which is valid for simple single-program protocols but introduces unacceptable latency for a 7-program system with cross-program dependencies.

## Recommendation

Add a `paused: bool` field to the global `EpochState` singleton (or a new `ProtocolConfig` PDA) and check it at the entry point of all user-facing instructions in the Tax Program (which gates all AMM swaps). Since all swap traffic flows through the Tax Program's `swap_sol_buy` and `swap_sol_sell`, a single pause check there would halt all trading.

For the Bonding Curve (which does not flow through Tax), add a separate pause check in `purchase` and `sell` instructions.

For Staking, add a pause check in `stake`, `unstake`, and `claim`.

The pause toggle instruction should be gated by the program's upgrade authority (ProgramData pattern, same as AMM's `initialize_admin`).

**State change required:** Yes -- adds 1 byte to EpochState (100 -> 101 bytes). Requires account reallocation or migration.
