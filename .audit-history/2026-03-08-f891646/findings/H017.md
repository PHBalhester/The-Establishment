# H017 -- Conversion Vault Whitelist Dependency

| Field            | Value                                                    |
|------------------|----------------------------------------------------------|
| **Hypothesis**   | If conversion vault accounts aren't whitelisted before transfer hook authority is burned, all vault conversions permanently brick. |
| **Category**     | Access Control, Token/Economic                           |
| **Origin**       | Novel (assumption A-8)                                   |
| **Severity**     | Initially scoped CRITICAL                                |
| **Verdict**      | **MITIGATED BY DEPLOYMENT SCRIPT -- residual risk LOW**  |

---

## 1. Investigation Summary

The hypothesis asks whether the conversion vault's three token accounts (vault_crime, vault_fraud, vault_profit) could be permanently bricked if they are not whitelisted before the transfer hook authority is irreversibly burned.

The concern is valid in principle: the transfer hook enforces that at least one party (source OR destination token account) in every CRIME/FRAUD transfer must have a WhitelistEntry PDA. The `burn_authority` instruction sets the WhitelistAuthority's `authority` field to `None`, and `add_whitelist_entry` requires `authority.is_some()`. Once burned, no new entries can ever be added. If the vault token accounts were missed, every `convert` instruction involving CRIME or FRAUD tokens would fail at the transfer hook level.

## 2. Evidence: Deployment Script Ordering

The `initialize.ts` script (the sole deployment pipeline) handles this correctly:

- **Step 8** (`initialize.ts:~815-845`): Initializes the Conversion Vault, creating `vault_crime`, `vault_fraud`, and `vault_profit` token accounts.
- **Step 9** (`initialize.ts:~847-884`): Immediately whitelists all three vault token accounts via `addWhitelistEntry()` for each of `vaultCrime`, `vaultFraud`, `vaultProfit`.
- **Step 25** (`initialize.ts:~1652-1707`): Burns the whitelist authority. This is the **final** whitelist operation, occurring after all 15 entries are created.

The script comments at Step 25 explicitly enumerate all 15 whitelist entries:
- Step 6: 3 admin T22 accounts
- Step 9: 3 conversion vault accounts (CRIME, FRAUD, PROFIT)
- Step 11: 4 AMM pool vaults
- Step 14: 1 stake vault
- Step 16: 2 carnage vaults
- Step 19: 1 CRIME curve token vault
- Step 20: 1 FRAUD curve token vault

The verify script (`verify.ts:~553-561`) also explicitly checks VaultCrime, VaultFraud, and VaultProfit whitelist entries exist post-deployment.

## 3. Evidence: Transfer Hook Logic

The transfer hook at `transfer_hook.rs:96-109` requires source OR destination to be whitelisted. For vault conversions:

- **User -> Vault (input transfer)**: The vault token account is the destination. If whitelisted, the transfer succeeds regardless of the user's whitelist status.
- **Vault -> User (output transfer)**: The vault token account is the source. If whitelisted, the transfer succeeds regardless of the user's whitelist status.

This means whitelisting the vault accounts is both necessary and sufficient for conversion operations to work with arbitrary (non-whitelisted) user accounts.

## 4. Evidence: Authority Burn Is Irreversible

- `burn_authority.rs:38`: Sets `auth.authority = None`
- `add_whitelist_entry.rs:52`: Constraint `whitelist_authority.authority.is_some()` rejects all additions after burn
- No `restore_authority` or `set_authority` instruction exists
- No `remove_whitelist_entry` instruction exists (entries are permanent)

Once burned, the whitelist is frozen forever. This is by design.

## 5. Residual Risk Assessment

The deployment script correctly handles the ordering. However, three residual risks remain:

### 5a. Script Interruption Between Step 9 and Step 25 (LOW)

If `initialize.ts` crashes after Step 8 (vault init) but before Step 9 (vault whitelist), and the operator manually runs `burnAuthority` without re-running the full script, the vaults would be permanently bricked. **Mitigated by**: `initialize.ts` is idempotent (re-run safely resumes from any step), and the authority burn is the very last step.

### 5b. Manual Authority Burn (LOW)

An operator or attacker with the whitelist authority key could call `burnAuthority` directly before the script completes all whitelist entries. **Mitigated by**: (a) the authority key is the deployer's key, (b) the script is the canonical deployment path, (c) the `verify.ts` script would catch missing entries before mainnet launch.

### 5c. PROFIT Mint Has No Transfer Hook (INFORMATIONAL)

PROFIT is a plain Token-2022 mint without a transfer hook extension. The vault_profit whitelist entry in Step 9 is technically unnecessary -- PROFIT transfers are not gated by the hook. Only vault_crime and vault_fraud strictly need whitelisting. This is not a bug (extra whitelist entries are harmless) but clarifies that PROFIT conversions are not at risk even if the whitelist were incomplete.

## 6. Conclusion

**The hypothesis is valid in theory but refuted in practice.** The deployment pipeline (`initialize.ts`) correctly whitelists all three conversion vault token accounts (Step 9) well before burning the whitelist authority (Step 25). The script is idempotent, and the verify script independently confirms all whitelist entries exist.

The irreversibility of the authority burn is a deliberate security feature, not a risk, because:
1. All 15 necessary addresses are whitelisted before burn
2. No new protocol accounts need to be created post-burn (the protocol is fully initialized)
3. Whitelist entries cannot be removed (no `remove` instruction), so once set they persist forever

**Severity downgraded from CRITICAL to LOW (residual operational risk only).**
