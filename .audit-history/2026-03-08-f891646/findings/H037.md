# H037 — force_carnage Devnet Gate

## Status: CONFIRMED (Informational / Medium)

## Hypothesis
The `#[cfg(feature = "devnet")]` gate on `force_carnage` correctly prevents the instruction from compiling into mainnet builds.

## Verdict
**The compile-time gate is correctly implemented across all three required locations.** The instruction, its module import, and its re-export are all gated by `#[cfg(feature = "devnet")]`. When the `devnet` feature is not enabled, Rust's conditional compilation completely excludes the code from the binary -- it is not present in the compiled artifact at all, and no instruction discriminator is registered in the Anchor dispatch table.

However, there is a **residual operational risk**: the `devnet` feature is currently the **production deployment feature** for the active devnet environment. The instruction is live and callable on devnet by the hardcoded `DEVNET_ADMIN` key. For mainnet, the feature must be omitted (which it will be by default since the build scripts only pass `--devnet` when the cluster URL contains "devnet").

## Evidence

### 1. Three-layer cfg gate (CORRECT)

**Module declaration** (`programs/epoch-program/src/instructions/mod.rs:7-8`):
```rust
#[cfg(feature = "devnet")]
pub mod force_carnage;
```

**Module re-export** (`programs/epoch-program/src/instructions/mod.rs:18-19`):
```rust
#[cfg(feature = "devnet")]
pub use force_carnage::*;
```

**Instruction entry point** (`programs/epoch-program/src/lib.rs:261-264`):
```rust
#[cfg(feature = "devnet")]
pub fn force_carnage(ctx: Context<ForceCarnage>, target: u8, action: u8) -> Result<()> {
    instructions::force_carnage::handler(ctx, target, action)
}
```

All three layers are gated. Removing any single layer would cause a compile error (e.g., removing the mod.rs gate but keeping the lib.rs gate would fail because `ForceCarnage` struct would not exist without the feature).

### 2. Cargo.toml feature definition (`programs/epoch-program/Cargo.toml:23-25`)
```toml
[features]
default = []
devnet = []
```

The `devnet` feature is **not in the default feature set**. A plain `anchor build -p epoch_program` or `cargo build -p epoch-program` will NOT include `force_carnage`.

### 3. Build script feature injection (`scripts/deploy/build.sh:79-85`)
The `--devnet` flag is only passed when explicitly requested:
```bash
if [[ " $* " == *" --devnet "* ]] || [[ "$1" == "--devnet" ]]; then
  anchor build -p epoch_program -- --features devnet
```

And `deploy-all.sh:146-148` only sets this when the cluster URL contains "devnet":
```bash
if echo "$CLUSTER_URL" | grep -q "devnet"; then
  BUILD_FLAGS="--devnet"
```

A mainnet deploy (non-devnet URL) will not pass the feature flag.

### 4. Admin constraint within the instruction (`force_carnage.rs:17-19,27-29`)
Even when compiled, the instruction requires a specific signer:
```rust
const DEVNET_ADMIN: Pubkey = pubkey!("8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4");

#[account(
    constraint = authority.key() == DEVNET_ADMIN @ EpochError::NotInitialized,
)]
pub authority: Signer<'info>,
```

This is defense-in-depth: even if the feature gate were somehow bypassed, only the devnet wallet holder can invoke it.

### 5. Instruction behavior
The handler directly writes to `epoch_state` fields (`carnage_pending`, `carnage_action`, `carnage_target`, `carnage_deadline_slot`) -- the same fields normally set by `consume_randomness` when VRF triggers Carnage. This bypasses VRF entirely, which is appropriate for testing but would undermine protocol integrity in production.

## Risk Assessment

| Aspect | Assessment |
|--------|-----------|
| Mainnet compile-time exclusion | SECURE -- triple cfg gate + default features exclude it |
| Devnet exposure | EXPECTED -- intentional test helper, admin-gated |
| Operational risk | MEDIUM -- relies on build scripts correctly omitting `--devnet` for mainnet |
| Code comment guidance | GOOD -- "MUST BE REMOVED BEFORE MAINNET DEPLOYMENT" at line 7 |

## Recommendation

The current implementation is **correct and sufficient for devnet**. For mainnet hardening:

1. **Consider removing the file entirely** before mainnet deployment rather than relying solely on feature flags. The comments already call for this ("MUST BE REMOVED BEFORE MAINNET DEPLOYMENT").
2. **Add a mainnet pre-deploy CI check** that verifies `force_carnage` does not appear in the compiled binary (e.g., `nm` or `objdump` on the `.so` file).
3. The error variant used (`EpochError::NotInitialized`) is misleading for an authorization failure. A dedicated error like `Unauthorized` would be clearer, though this is cosmetic.

## Severity: Informational

The compile-time gate works as designed. The instruction cannot appear in a mainnet binary unless the build process is explicitly misconfigured. The hardcoded admin key provides defense-in-depth even if the gate were bypassed.
