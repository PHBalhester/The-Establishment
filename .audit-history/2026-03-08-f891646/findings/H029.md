# H029 -- Carnage Fallback Path MEV Sandwich

| Field            | Value                                                                 |
|------------------|-----------------------------------------------------------------------|
| **Hypothesis**   | After 50-slot atomic lock expires, `execute_carnage` fallback path is sandwichable. 75% slippage floor still leaves 15-25% extractable. |
| **Category**     | Timing, Token/Economic                                                |
| **Severity**     | **MEDIUM** (downgraded from initial MEDIUM-HIGH estimate)             |
| **Status**       | **Confirmed -- with significant mitigating factors**                  |
| **File(s)**      | `programs/epoch-program/src/instructions/execute_carnage.rs`          |

## Summary

The `execute_carnage` fallback path (available after the 50-slot atomic lock window expires) is sandwichable by MEV bots. The 75% slippage floor (`CARNAGE_SLIPPAGE_BPS_FALLBACK = 7500`) caps theoretical extraction at 25% of expected output. However, several design choices significantly constrain the real-world extractable value, making this a MEDIUM rather than HIGH finding.

## Detailed Analysis

### The Sandwich Window

1. **`consume_randomness` sets `carnage_pending = true`** with `carnage_lock_slot` and `carnage_deadline_slot`.
2. **Slots 0-50 (lock window):** Only `execute_carnage_atomic` can execute (bundled with consume_randomness TX). Atomicity makes sandwiching very difficult -- the attacker cannot see the Carnage trigger until the TX lands.
3. **Slots 50-300 (fallback window):** `execute_carnage` becomes callable by anyone. At this point, `carnage_pending`, `carnage_action`, and `carnage_target` are all publicly readable on-chain via `EpochState`. An attacker knows:
   - **Which pool** will be traded (CRIME/SOL or FRAUD/SOL)
   - **The direction** (always a SOL buy on the target pool)
   - **Approximate amount** (bounded by `MAX_CARNAGE_SWAP_LAMPORTS` = 1000 SOL cap, and observable `sol_vault` balance)

### Slippage Protection Chain

The fallback path has a **post-execution** slippage check at `execute_carnage.rs:429-445`:

```rust
let expected = (reserve_token as u128)
    .checked_mul(total_buy_amount as u128)
    .and_then(|n| n.checked_div(
        (reserve_sol as u128).checked_add(total_buy_amount as u128)?
    ))
    .ok_or(EpochError::Overflow)? as u64;

let min_output = (expected as u128)
    .checked_mul(CARNAGE_SLIPPAGE_BPS_FALLBACK as u128)  // 7500
    .and_then(|n| n.checked_div(10_000))
    .ok_or(EpochError::Overflow)? as u64;

require!(bought >= min_output, EpochError::CarnageSlippageExceeded);
```

**Critical detail:** The `reserve_sol` and `reserve_token` are read from pool AccountInfo BEFORE the swap CPI (`read_pool_reserves` at line 384-387). On Solana, AccountInfo data is updated in-place after CPI returns, so the reserves read here reflect the state at the time of the `execute_carnage` instruction entry -- which is AFTER a front-runner's manipulation TX in the same block.

This means a sandwich attacker's front-run transaction moves the reserves to a worse state BEFORE `execute_carnage` reads them. The `expected` calculation uses the already-manipulated reserves, producing a lower floor. The attacker gets their extraction AND the slippage check passes because `expected` was computed on post-manipulation reserves.

### However: The TOCTOU Actually Helps Here

Wait -- re-examining the flow more carefully:

1. Front-runner's TX executes first in the block, moving pool reserves (e.g., buying tokens, raising SOL reserve, lowering token reserve).
2. `execute_carnage` reads reserves AFTER the front-run. The `expected` output is computed on these worse reserves.
3. The swap CPI executes against these same reserves (no further manipulation between read and CPI within the same TX).
4. The `bought >= min_output` check uses `expected` from step 2.

So the reserves read and the reserves used for the actual swap are **the same** (both are post-front-run). The slippage check is internally consistent. The 75% floor applies to the post-front-run expected output, which IS lower than the pre-front-run expected output. The attacker's back-run then restores the pool.

**Net effect:** The attacker extracts value by moving the price before Carnage buys, and restoring it after. The Carnage fund receives fewer tokens than it would have without manipulation. The 75% floor means the Carnage TX will revert if manipulation exceeds 25% of the (pre-manipulation) expected output. But since the floor is computed on POST-manipulation reserves, the actual tolerance is slightly more nuanced.

### Maximum Extractable Value Calculation

For a sandwich attack on a constant-product AMM:

- Let `R_s` = SOL reserve, `R_t` = token reserve, `C` = Carnage buy amount
- Front-run: attacker buys `A` SOL worth of tokens, moving reserves to `R_s + A`, `R_t' = R_t * R_s / (R_s + A)`
- Carnage buys: receives tokens from the worse pool state
- Back-run: attacker sells acquired tokens back

The 75% floor on post-manipulation expected output constrains the front-run size. For typical pool sizes (e.g., 100 SOL / 100M tokens) and Carnage amounts (capped at `MAX_CARNAGE_SWAP_LAMPORTS` = 1000 SOL, but practically limited by `sol_vault` balance which is 24% of taxes):

- If Carnage fund has ~10 SOL to spend against a 100 SOL pool, the maximum front-run that still lets the 75% floor pass is bounded. The attacker can extract roughly **up to 25% of the Carnage swap's expected token output value**.
- At 10 SOL carnage spend, this is ~2.5 SOL equivalent maximum extraction per event.
- Carnage triggers with ~4.3% probability per epoch (per ARCHITECTURE.md), so expected extraction per epoch is very low.

### Mitigating Factors (Why MEDIUM, Not HIGH)

1. **Atomic path gets first priority (50 slots).** The crank runner bundles `execute_carnage_atomic` with `consume_randomness`. If the atomic path succeeds (85% floor, much harder to sandwich), the fallback never opens. The fallback only activates if atomic fails (e.g., compute budget exceeded, CPI depth issue).

2. **Carnage fund size is bounded.** The fund accumulates 24% of swap taxes. For a protocol with moderate volume, each Carnage event might spend 1-50 SOL. The 25% extraction on small amounts yields small absolute profits.

3. **VRF unpredictability of WHEN Carnage triggers.** The ~4.3% per-epoch trigger chance means MEV bots can't guarantee when Carnage will occur. They must monitor on-chain state continuously. (Though once `carnage_pending = true`, the information is public.)

4. **Solana block production.** Solana's leader schedule means the sandwich attacker must either be the block producer or submit transactions with high priority fees. The cost of sandwiching may exceed profit for small Carnage swaps.

5. **The `MAX_CARNAGE_SWAP_LAMPORTS` cap (1000 SOL)** prevents catastrophic single-event extraction.

### Comparison: Atomic vs Fallback

| Property | Atomic | Fallback |
|---|---|---|
| Slippage floor | 85% (`CARNAGE_SLIPPAGE_BPS_ATOMIC = 8500`) | 75% (`CARNAGE_SLIPPAGE_BPS_FALLBACK = 7500`) |
| Max theoretical extraction | 15% | 25% |
| Predictability | Low (bundled with VRF reveal) | High (`carnage_pending` is public) |
| MEV window | Near-zero (same TX as reveal) | 250 slots (~100 seconds) |
| Primary defense | Atomicity + unpredictability | Slippage floor only |

### Additional Concern: Pool AccountInfo Not Owner-Checked

The `crime_pool` and `fraud_pool` in `ExecuteCarnage` are `AccountInfo<'info>` with `#[account(mut)]` but **no owner check** (they are "CPI passthroughs" validated by Tax Program). However, the `read_pool_reserves` function reads raw bytes from these AccountInfos for the slippage floor calculation. This is a separate finding (covered by the architecture doc's section 7.6), but it compounds this issue -- a spoofed pool AccountInfo with inflated reserves would produce a higher `expected`, making the 75% floor tighter. This actually makes spoofing counterproductive for the attacker (it would make the slippage check MORE restrictive), but the lack of owner verification is still a defense-in-depth gap.

**Correction:** If the attacker provides a fake pool with deflated token reserves, the expected output would be lower, and the floor would be correspondingly lower, allowing worse execution. But the actual CPI goes through Tax::swap_exempt which passes the same pool to AMM, where it IS validated. So the pool used for slippage calculation and the pool used for execution are the same AccountInfo. A fake pool would cause the AMM CPI to fail.

## Verdict

**CONFIRMED at MEDIUM severity.** The fallback path is sandwichable with up to 25% theoretical value extraction per Carnage event. Practical extraction is constrained by:
- Atomic path handling most events successfully
- Small fund sizes per event
- Solana MEV infrastructure costs
- The 75% floor providing a hard ceiling

**Estimated realistic impact:** 5-15% extraction on fallback-only events (which are already the minority case), on amounts typically under 50 SOL. Per-event loss likely 0.5-7.5 SOL. Annualized impact depends on trading volume and how often atomic fails.

## Recommendations

1. **Tighten fallback slippage floor.** Consider raising `CARNAGE_SLIPPAGE_BPS_FALLBACK` from 7500 to 8000 (80%), reducing maximum extraction from 25% to 20%. This trades execution reliability for MEV resistance.

2. **Add Jito bundle support for fallback execution.** The crank runner could submit fallback Carnage via Jito bundles with tip, ensuring atomic execution without public mempool visibility. This is a client-side change only.

3. **Consider a two-phase fallback.** Slots 50-150: 80% floor (moderate MEV resistance). Slots 150-300: 70% floor (prioritize execution). This gives honest executors a better window before the floor loosens.

4. **Monitor atomic path failure rate.** If atomic execution rarely fails, the fallback path may never activate in practice, making this finding theoretical. Log and alert on fallback executions.
