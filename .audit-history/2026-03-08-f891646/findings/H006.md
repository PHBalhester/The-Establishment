# H006 -- Bonding Curve Authority: fund_curve Token Injection

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** Access Control, Token/Economic
**Confidence:** HIGH

## Summary

The hypothesis that an attacker could inject unexpected token amounts into the bonding curve via `fund_curve` to manipulate pricing is **not viable**. The instruction transfers a hardcoded constant (`TARGET_TOKENS = 460M`) -- the amount is not caller-controlled. The transfer source is the caller's own token account, meaning any "injection" costs the attacker real tokens with no pricing benefit. Additionally, the `CurveStatus::Initialized` state guard ensures `fund_curve` can only be called once per curve lifecycle (before `start_curve` transitions to `Active`), providing effective idempotency.

The separate issue that ANY signer can call `fund_curve` (the broader bonding curve authority gap) is real but has no exploitable impact for this specific instruction: the caller donates their own tokens to the protocol's vault with no way to recover them.

## Evidence

**Hardcoded transfer amount:** `programs/bonding_curve/src/instructions/fund_curve.rs:80`
```rust
let transfer_amount = TARGET_TOKENS;
```
`TARGET_TOKENS = 460_000_000_000_000` (460M with 6 decimals) at `constants.rs:37`. The caller cannot specify a different amount. Over-funding or under-funding is impossible through this instruction.

**State guard (Initialized only):** `fund_curve.rs:30`
```rust
constraint = curve_state.status == CurveStatus::Initialized @ CurveError::InvalidStatus,
```
Once `start_curve` transitions the curve to `Active`, `fund_curve` is permanently blocked. Since `start_curve` also requires `status == Initialized` (start_curve.rs:25), the lifecycle is strictly: `initialize_curve` -> `fund_curve` -> `start_curve`. Double-funding via `fund_curve` is impossible because `start_curve` (the only way to leave `Initialized`) requires the vault already holds `>= TARGET_TOKENS`.

**Caller pays with own tokens:** `fund_curve.rs:36-41`
```rust
#[account(
    mut,
    token::mint = token_mint,
    token::token_program = token_program,
)]
pub authority_token_account: InterfaceAccount<'info, TokenAccount>,
```
The source is the caller's token account. The transfer uses `invoke` (not `invoke_signed`) with the caller as signer (line 115). The attacker must hold 460M tokens of the correct mint (CRIME or FRAUD) to execute this instruction.

**Vault authority is CurveState PDA:** `fund_curve.rs:48`
```rust
token::authority = curve_state,
```
Tokens deposited into the vault are controlled by the CurveState PDA. The caller cannot withdraw them after funding -- only program instructions (`purchase`, `sell`, `claim_refund`) can move tokens out via PDA signing.

**Mint validation via CurveState seeds:** `fund_curve.rs:28-29`
```rust
seeds = [CURVE_SEED, token_mint.key().as_ref()],
bump = curve_state.bump,
```
The `token_mint` must match the mint used when the CurveState PDA was created. Combined with the `initialize_curve` mint allowlist (crime_mint/fraud_mint), only legitimate CRIME or FRAUD tokens can be deposited.

## Attack Scenario

**Hypothesized attack (does NOT work):**

1. Attacker calls `fund_curve` with their own token account as source.
2. `transfer_amount = TARGET_TOKENS` (hardcoded 460M) -- attacker cannot choose a different amount.
3. Attacker must hold 460M CRIME or FRAUD tokens. Total supply is 1B per token, so this is 46% of total supply.
4. If somehow successful, the attacker has donated 460M tokens to the protocol's vault with no mechanism to retrieve them.
5. `start_curve` can now be called (vault >= TARGET_TOKENS), and the curve operates normally with correct pricing.

**No pricing manipulation is possible** because:
- The curve math (`math.rs:56-237`) depends on `tokens_sold` (starts at 0) and the hardcoded price constants `P_START=900` and `P_END=3450`. The initial vault balance does not enter the price formula.
- Over-funding beyond TARGET_TOKENS via direct SPL transfer to the vault PDA is theoretically possible but has no impact: `start_curve` checks `>= TARGET_TOKENS`, purchase math uses `TOTAL_FOR_SALE` (a constant), and excess tokens would sit unreachable in the vault.

**Double-funding scenario (also does NOT work):**
1. Legitimate deployer calls `fund_curve` (curve is `Initialized`, vault receives 460M tokens).
2. Attacker attempts second `fund_curve` call.
3. If `start_curve` has NOT been called yet: curve is still `Initialized`, so the constraint passes. However, the transfer of another 460M tokens from the attacker's account would succeed, over-funding the vault to 920M tokens. The excess tokens are irrecoverable -- the attacker has donated tokens to the protocol with no benefit.
4. If `start_curve` HAS been called: status is `Active`, constraint fails with `InvalidStatus`.

The double-funding scenario is a self-griefing attack: the attacker loses 460M tokens. The protocol is unaffected because curve math uses `tokens_sold` and constants, not the vault balance.

## Root Cause

No vulnerability exists for the specific token injection scenario. The hardcoded `TARGET_TOKENS` transfer amount, the `Initialized`-only state guard, and the fact that curve pricing is independent of vault balance collectively prevent any meaningful attack via `fund_curve`.

The broader bonding curve authority gap (bare `Signer<'info>` with no identity verification) applies here but is harmless for `fund_curve` specifically: the worst an attacker can do is donate their own tokens to the protocol.

## Recommendation

No specific fix needed for `fund_curve` in isolation. The hardcoded amount and state guard provide effective protection against token injection and double-funding.

The broader authority gap across all 6 bonding curve admin instructions should be addressed as recommended in H001 (ProgramData upgrade-authority pattern or stored admin PDA). When that fix is applied, `fund_curve` will inherit proper authority verification as a side effect.
