# S002 -- Sell Path Gross Minimum Computation Correctness

**Audit:** sos-002 | **Severity:** Supplemental (derived from sell path zero-slippage finding)
**Status:** INFORMATIONAL -- no actionable edge case found
**Derived from:** Sell path AMM minimum_amount_out=0 (Section 7.5 of ARCHITECTURE.md)

---

## Hypothesis

The recommended fix for sell path slippage -- computing `gross_min = minimum_output * 10000 / (10000 - tax_bps)` and passing it to the AMM -- may have edge cases with rounding or extreme tax rates (up to 1400 BPS).

## Investigation

### Current Sell Path Flow

From `swap_sol_sell.rs`:

1. AMM CPI is called with `amm_minimum: u64 = 0` (line 147).
2. AMM returns gross WSOL output to user.
3. Tax computed: `tax_amount = gross_output * tax_bps / 10_000` (floor division).
4. Net: `net_output = gross_output - tax_amount`.
5. Slippage check: `net_output >= minimum_output` (line 245).

The issue: passing `0` to AMM means the AMM provides no sandwich protection. The 50% output floor (MINIMUM_OUTPUT_FLOOR_BPS = 5000) is checked pre-swap against `minimum_output`, but the gap between 50% and the user's actual `minimum_output` is extractable.

### Proposed Fix Analysis

The recommended fix is to compute a gross minimum and pass it to the AMM:

```
gross_min = minimum_output * 10000 / (10000 - tax_bps)
```

This inverts the tax formula: if `net = gross - floor(gross * tax_bps / 10000)`, then `gross >= net * 10000 / (10000 - tax_bps)` should ensure the net output meets the user's minimum.

### Edge Case 1: Rounding Direction

The formula `minimum_output * 10000 / (10000 - tax_bps)` uses integer division (floor).

**Question:** Does flooring `gross_min` risk the AMM returning exactly `gross_min`, which after tax produces `net < minimum_output`?

**Analysis at max tax (1400 BPS = 14%):**

Let `minimum_output = 1_000_000` (0.001 SOL):
- `gross_min = 1_000_000 * 10_000 / (10_000 - 1400) = 10_000_000_000 / 8600 = 1_162_790` (floor)
- Tax on 1_162_790: `floor(1_162_790 * 1400 / 10_000) = floor(162_790.6) = 162_790`
- Net: `1_162_790 - 162_790 = 1_000_000` -- exactly meets minimum.

Now check if `gross_min` was truncated:
- Exact: `10_000_000_000 / 8600 = 1_162_790.697...`
- Floor: `1_162_790`
- What if AMM returns exactly `1_162_790`?
- Tax: `floor(1_162_790 * 1400 / 10_000) = floor(162,790.6) = 162,790`
- Net: `1,162,790 - 162,790 = 1,000,000` -- still meets minimum.

**Edge case where it fails:**

Let `minimum_output = 999_999`:
- `gross_min = 999_999 * 10_000 / 8600 = 9_999_990_000 / 8600 = 1_162_789` (floor)
- Tax on 1_162_789: `floor(1_162_789 * 1400 / 10_000) = floor(162,790.46) = 162,790`
- Net: `1,162,789 - 162,790 = 999,999` -- meets minimum.

Let `minimum_output = 7` (tiny):
- `gross_min = 7 * 10_000 / 8600 = 70_000 / 8600 = 8` (floor)
- Tax on 8: `floor(8 * 1400 / 10_000) = floor(1.12) = 1`
- Net: `8 - 1 = 7` -- meets minimum.

Let `minimum_output = 1`:
- `gross_min = 1 * 10_000 / 8600 = 1` (floor from 1.16...)
- Tax on 1: `floor(1 * 1400 / 10_000) = 0`
- Net: `1 - 0 = 1` -- meets minimum.

**Systematic proof:** For `gross_min = floor(min_out * 10000 / (10000 - bps))`:
- Tax = `floor(gross_min * bps / 10000)`
- Net = `gross_min - floor(gross_min * bps / 10000)`
- Since `gross_min >= min_out * 10000 / (10000 - bps) - 1` (floor truncation), and tax uses floor division (reducing tax), the net is always `>= min_out - 1` at worst.

The concern is the `-1` from double flooring. However:
- `gross_min = floor(min_out * D / (D - bps))` where D = 10000.
- `net = gross_min - floor(gross_min * bps / D)`
- `net >= gross_min - gross_min * bps / D = gross_min * (D - bps) / D`
- `gross_min >= min_out * D / (D - bps) - 1`
- `net >= (min_out * D / (D - bps) - 1) * (D - bps) / D = min_out - (D - bps) / D`
- Since `(D - bps) / D < 1`, `net >= min_out - 1` at worst.

**This means the formula CAN underdeliver by 1 lamport** in adversarial rounding cases.

### Edge Case 2: tax_bps = 0

If `tax_bps = 0`: `gross_min = minimum_output * 10000 / 10000 = minimum_output`. Correct.

### Edge Case 3: Overflow

`minimum_output * 10_000`: max `u64::MAX * 10_000` overflows u64. Must use u128 intermediate.
- `minimum_output` is at most `u64::MAX` in theory, but bounded by pool reserves in practice.
- With u128: no overflow risk.

### Edge Case 4: tax_bps at Protocol Bounds

Tax BPS is bounded to [100, 1400] by the EpochState lookup tables (INV-8). The formula works across this entire range. `(10000 - 1400) = 8600` is well above zero, so no division-by-zero risk.

## Verdict

**INFORMATIONAL.** The proposed gross minimum formula is correct for all practical inputs:

1. **Rounding:** At worst, underdelivers by 1 lamport. For amounts >= 2 lamports with tax rates in [100, 1400] BPS, the formula is exact or net exceeds minimum by 1.
2. **Overflow:** Requires u128 intermediate (standard pattern already used throughout tax_math.rs).
3. **Division by zero:** Impossible with tax BPS capped at 1400 (denominator minimum = 8600).
4. **Zero tax:** Degenerates to `gross_min = minimum_output`. Correct.

The 1-lamport rounding risk is negligible (sub-dust amounts) and can be addressed with `ceiling division` if desired:
```rust
let gross_min = (min_out as u128 * 10_000 + (10_000 - tax_bps as u128) - 1) / (10_000 - tax_bps as u128);
```

This ceiling division guarantees `net >= minimum_output` for all inputs.

## Recommendation

When implementing the sell-path fix:
1. Use u128 intermediate math (consistent with existing `calculate_tax`).
2. Use ceiling division to eliminate the 1-lamport edge case.
3. The formula is safe across the full [100, 1400] BPS range.
