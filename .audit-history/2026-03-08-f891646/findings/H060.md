# H060 — Bonding Curve mark_failed Timing

| Field          | Value                                                        |
|----------------|--------------------------------------------------------------|
| Hypothesis     | mark_failed deadline check or deadline_slot immutability is flawed |
| Target         | `programs/bonding_curve/src/instructions/mark_failed.rs`     |
| Severity       | INFORMATIONAL                                                |
| Status         | **REJECTED — No vulnerability found**                        |

## Summary

`mark_failed` is a permissionless instruction that transitions a bonding curve from `Active` to `Failed` after `deadline_slot + FAILURE_GRACE_SLOTS`. The implementation is correct: the deadline is set exactly once (during `start_curve`), is never modified afterwards, the slot comparison uses strict greater-than with checked arithmetic, and the grace buffer is a compile-time constant.

## Detailed Analysis

### 1. Deadline Setting (Immutability)

`deadline_slot` is written in exactly two places across the entire bonding curve program:

- **`initialize_curve.rs:104`** — Set to `0` at account creation (Initialized status).
- **`start_curve.rs:66`** — Set to `clock.slot + DEADLINE_SLOTS` when transitioning Initialized to Active.

No other instruction writes to `deadline_slot`. Once set in `start_curve`, it is effectively immutable for the lifetime of the curve. The state machine enforces this: `start_curve` requires `status == Initialized` (Anchor constraint at line 25), and no instruction transitions back to Initialized.

**One minor observation**: `start_curve.rs:66` uses unchecked addition (`clock.slot + DEADLINE_SLOTS`). With `DEADLINE_SLOTS = 432,000` and current Solana slot values well below `u64::MAX`, overflow is impossible in practice. However, `mark_failed.rs:57-59` correctly uses `checked_add` for the same type of arithmetic (`deadline_slot.checked_add(FAILURE_GRACE_SLOTS)`). This inconsistency is cosmetic — noted by the timing context agent as well.

### 2. Slot Comparison in mark_failed

```rust
let failure_eligible_slot = curve
    .deadline_slot
    .checked_add(FAILURE_GRACE_SLOTS)
    .ok_or(CurveError::Overflow)?;
require!(
    clock.slot > failure_eligible_slot,
    CurveError::DeadlineNotPassed
);
```

- Uses `checked_add` to prevent overflow (correct).
- Uses **strict greater-than** (`>` not `>=`), meaning the failure-eligible slot itself must have fully passed. This is the conservative (safe) direction.
- `FAILURE_GRACE_SLOTS = 150` (~60 seconds at 400ms/slot) provides buffer for in-flight transactions to finalize after the purchase deadline.

### 3. Status Guard

```rust
require!(
    curve.status == CurveStatus::Active,
    CurveError::InvalidStatus
);
```

Only `Active` curves can be marked failed. This prevents:
- Double-failure (calling `mark_failed` on an already-Failed curve).
- Marking Filled/Graduated/Initialized curves as Failed.

The status check is in the handler (not an Anchor constraint) deliberately, to produce a specific `InvalidStatus` error rather than a generic constraint error.

### 4. Permissionless Access

`mark_failed` requires no signer authority — only the `curve_state` PDA is passed. This is correct by design: failure marking should be permissionless (anyone can observe the deadline has passed). The only gate is the timing condition, which is enforced on-chain via `Clock::get()?.slot`.

### 5. Edge Case: deadline_slot = 0

If `mark_failed` is called on a curve that was initialized but never started (`deadline_slot = 0`), the status check (`Active`) prevents execution because unstarted curves have status `Initialized`. This is safe.

### 6. Grace Period Interaction

- **Purchases** are blocked at `clock.slot > deadline_slot` (hard cutoff in `purchase.rs:109`).
- **Sells** are blocked at the same deadline (`sell.rs:143`).
- **mark_failed** requires `clock.slot > deadline_slot + 150`.

This creates a 150-slot window (~60 seconds) where the curve is Active but no buys/sells can occur. During this window, in-flight transactions can finalize. This is the intended design per the spec comments.

## Verdict

No vulnerability. The `mark_failed` implementation is sound:
- Deadline is set once, never modified.
- Slot comparison uses checked arithmetic and strict greater-than.
- Status guard prevents double-transition.
- Permissionless access is correct (state-gated, not identity-gated).
- Grace buffer provides appropriate safety margin.

The only cosmetic note is the unchecked addition in `start_curve.rs:66` which is inconsistent with the checked arithmetic in `mark_failed`, but poses no practical risk.
