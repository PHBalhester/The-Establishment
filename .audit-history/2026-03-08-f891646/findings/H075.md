# H075 -- Bonding Curve: Purchase with 0 SOL

**Status:** Confirmed Safe
**Severity:** Informational
**Confidence:** High

## Summary

A purchase with `sol_amount = 0` is rejected by the `MIN_PURCHASE_SOL` check at Step 2. Even if this check were bypassed, Step 4 would catch a 0-token output. The instruction is well-defended against zero-amount attacks.

## Evidence

**purchase.rs (lines 116-119) -- Step 2:**
```rust
require!(
    sol_amount >= MIN_PURCHASE_SOL,
    CurveError::BelowMinimum
);
```

**constants.rs (line 53):**
```rust
pub const MIN_PURCHASE_SOL: u64 = 50_000_000; // 0.05 SOL
```

Any `sol_amount < 50_000_000` lamports (including 0) is rejected with `CurveError::BelowMinimum`.

**Defense in depth -- Step 4 (line 129):**
```rust
require!(tokens_out > 0, CurveError::InsufficientTokensOut);
```

Even if `MIN_PURCHASE_SOL` were set to 0, the `calculate_tokens_out(0, _)` call would return 0 tokens (the integral of the price curve over a zero-width SOL interval is zero), and Step 4 would reject it.

**Additional guards:** Step 9 (slippage, line 178-181) would also reject unless caller passes `minimum_tokens_out = 0`, and Step 10 (SOL transfer, line 186-195) would be a no-op system transfer of 0 lamports which Solana permits but is harmless since no state changes would have occurred.

## Recommendation

None required. The 0.05 SOL minimum is a solid first line of defense, with the `tokens_out > 0` check as backup.
