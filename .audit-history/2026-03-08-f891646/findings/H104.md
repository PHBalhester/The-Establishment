# H104 -- Bonding Curve: Concurrent Purchase Race

## Status: INVALID (Not a vulnerability)

## Hypothesis
Two users submit purchase transactions near the Filled threshold simultaneously. Could this cause the curve to oversell beyond TARGET_TOKENS or result in inconsistent state?

## Analysis

### Solana Runtime Account Locking
Solana's runtime enforces **exclusive write locks** on accounts. The `purchase` instruction marks `curve_state` as `mut`:

```rust
#[account(
    mut,
    seeds = [CURVE_SEED, token_mint.key().as_ref()],
    bump = curve_state.bump,
    constraint = curve_state.status == CurveStatus::Active @ CurveError::CurveNotActive,
)]
pub curve_state: Account<'info, CurveState>,
```

Two transactions writing to the same `CurveState` PDA **cannot execute concurrently** within a single slot. The Solana scheduler serializes them. The second transaction sees the first's committed state.

### Partial Fill Logic (Step 6)
Even if two transactions were somehow ordered back-to-back in the same slot, the code handles this correctly:

```rust
let remaining = TARGET_TOKENS
    .checked_sub(curve.tokens_sold)
    .ok_or(CurveError::CurveAlreadyFilled)?;

let actual_tokens = std::cmp::min(tokens_out, remaining);
```

- If TX1 fills the curve to TARGET_TOKENS, TX2 sees `remaining = 0`, and `actual_tokens = 0`.
- Step 4's `require!(tokens_out > 0, ...)` uses the pre-cap value, but Step 9's slippage check `require!(actual_tokens >= minimum_tokens_out, ...)` would reject a 0-token fill if `minimum_tokens_out > 0`.
- If TX1 partially fills, TX2 gets the correct remaining amount via `min(tokens_out, remaining)`.

### State Transition Safety
The Filled transition at Step 14 uses `>=` which is safe:
```rust
if curve.tokens_sold >= TARGET_TOKENS {
    curve.status = CurveStatus::Filled;
}
```

And the Anchor constraint `curve_state.status == CurveStatus::Active` on the account struct means once status is Filled, subsequent purchases are rejected at deserialization time.

### Edge Case: tokens_out > 0 but actual_tokens = 0
If `remaining = 0` (curve exactly at target from prior TX), then `actual_tokens = 0`. The slippage check at Step 9 requires `actual_tokens >= minimum_tokens_out`. If user set `minimum_tokens_out = 0`, the TX would proceed to transfer 0 tokens and 0 SOL -- a no-op that wastes gas but causes no harm. If `minimum_tokens_out > 0` (normal usage), the TX reverts with `SlippageExceeded`.

## Verdict
**No vulnerability.** Solana's account locking model serializes concurrent writes to the same PDA. The partial fill logic correctly handles the boundary case. The `CurveNotActive` constraint prevents post-fill purchases.
