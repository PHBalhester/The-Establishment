# H119 -- CarnageFundState Size (139 bytes), No Padding

## Status: INFORMATIONAL (Confirmed, Low Risk)

## Target
`/Users/mlbob/Projects/Dr Fraudsworth/programs/epoch-program/src/state/carnage_fund_state.rs`

## Hypothesis
CarnageFundState has no padding bytes reserved for schema evolution. Adding any field would require realloc or a new account.

## Evidence

### Account Layout (139 data bytes, 147 total)
```
Pubkey sol_vault:        32 bytes
Pubkey crime_vault:      32 bytes
Pubkey fraud_vault:      32 bytes
u8     held_token:        1 byte
u64    held_amount:       8 bytes
u32    last_trigger_epoch: 4 bytes
u64    total_sol_spent:   8 bytes
u64    total_crime_burned: 8 bytes
u64    total_fraud_burned: 8 bytes
u32    total_triggers:    4 bytes
bool   initialized:       1 byte
u8     bump:              1 byte
                        --------
                        139 bytes
```

### Verification
- Static compile-time assertions at lines 159-162 enforce `DATA_LEN == 139` and `LEN == 8 + DATA_LEN`.
- Runtime unit tests at line 170 re-verify `LEN == 147`.
- `LEN` is used as the `space` parameter in `#[account(init, space = ...)]`, meaning the on-chain account is allocated to exactly 147 bytes with zero slack.

### Schema Evolution Impact
- **Same pattern as EpochState** (100 data bytes, also zero padding). Neither account has reserved bytes.
- No `realloc` usage exists anywhere in the programs directory.
- Adding any field (e.g., a new lifetime counter) would require either:
  1. `realloc` in a migration instruction (requires program upgrade authority), or
  2. A new companion PDA account.
- Since programs are intended to be non-upgradeable at mainnet (Assumption A-7 in ARCHITECTURE.md), realloc-based migration would be impossible post-authority burn.

### Mitigating Factors
- CarnageFundState is a **statistics + coordination** account. Its fields are comprehensive for the current protocol design (3 vaults, holdings tracking, 4 lifetime counters).
- The 3 vault Pubkeys (96 bytes) are set once at init and never change, so there is no pressure to evolve those.
- Adding padding now would increase rent cost marginally (~6960 lamports per 8 bytes of padding on mainnet).

## Verdict
**Informational.** The lack of padding is intentional (matches EpochState pattern). This only becomes a problem if new features require on-chain state changes AND the program has already been made non-upgradeable. Given the protocol is pre-mainnet and upgrade authority is still retained, this is a known design tradeoff, not a vulnerability. Document in mainnet-readiness assessment that schema is frozen once upgrade authority is burned.
