# H027: EpochState No Padding for Schema Evolution (H060 Expanded)

**Status:** CONFIRMED (STILL PRESENT -- no padding added since Audit #1)
**Severity:** MEDIUM (architectural fragility; would escalate to HIGH if upgrade attempted without migration)
**Category:** State Machine, Upgrade/Admin
**Confidence:** HIGH
**Origin:** RECHECK of H060 (Audit #1) -- expanded scope

## Summary

EpochState uses exactly 100 bytes of data (108 with Anchor discriminator) with zero reserved or padding bytes. A compile-time static assertion at `epoch_state.rs:172` enforces this exact size. The account is allocated at exactly `EpochState::LEN = 108` bytes during initialization. Any field addition to EpochState requires either: (a) Solana account reallocation (realloc), (b) a full protocol migration (new PDA, new account), or (c) a coordinated multi-program redeploy. This is complicated by the Tax Program maintaining a mirror struct that must match byte-for-byte, creating a two-program synchronization requirement with no compile-time safety net.

No other state account in the protocol has padding either. This is a protocol-wide pattern, but EpochState is uniquely impacted because it is the only account read cross-program via a duplicated struct definition.

## Evidence

### 1. Exact Size with Zero Headroom

**File:** `programs/epoch-program/src/state/epoch_state.rs:158-172`

```rust
pub const LEN: usize = 8 + 100;

pub const DATA_LEN: usize = 8 + 4 + 8 + 1 + 2 + 2 + 2 + 2 + 2 + 2
    + 8 + 1 + 1 + 32 + 1 + 1 + 1 + 8 + 8 + 4 + 1 + 1;

// Static assertion that DATA_LEN equals expected 100 bytes
const _: () = assert!(EpochState::DATA_LEN == 100);
```

Every byte is accounted for. There is no `_reserved: [u8; N]` field, no `_padding` field, and no gap between the last field (`bump: u8`) and the account boundary.

### 2. Account Allocation at Exact Size

**File:** `programs/epoch-program/src/instructions/initialize_epoch_state.rs:105-111`

```rust
#[account(
    init,
    payer = payer,
    space = EpochState::LEN,  // = 108 bytes exactly
    seeds = [EPOCH_STATE_SEED],
    bump,
)]
```

The account is allocated at exactly 108 bytes. No over-allocation that might accidentally provide headroom.

### 3. Cross-Program Mirror at Same Exact Size

**File:** `programs/tax-program/src/state/epoch_state_reader.rs:54-57`

```rust
pub const LEN: usize = 8 + 100;
```

The Tax Program mirror declares the same 108-byte size. Notably, the mirror does NOT have a `DATA_LEN` constant or a static size assertion (unlike the canonical struct). This means a field count mismatch between the two structs would not be caught at compile time in the Tax crate.

### 4. Protocol-Wide Absence of Padding

A grep for `reserved`, `padding`, `_unused` across all program source returns zero hits in struct definitions. Every state account is tightly packed:

| Account | Program | Data Size | Padding |
|---------|---------|-----------|---------|
| EpochState | Epoch | 100 bytes | 0 |
| CarnageFundState | Epoch | 139 bytes | 0 |
| CurveState | Bonding Curve | 192 bytes | 0 |
| StakePool | Staking | 54 bytes | 0 |
| UserStake | Staking | 97 bytes | 0 |
| PoolState | AMM | 224 bytes | 0 |
| VaultConfig | Conversion Vault | 1 byte | 0 |

### 5. Historical Precedent: V3 Migration Failure

Per H011's citation of `VRF_Migration_Lessons.md`: the project has already experienced this exact failure mode. EpochState grew from 50 to 82 bytes in V3, breaking cross-program deserialization. The lesson was documented but the structural fix (padding) was not applied.

## Impact Analysis

### Direct Impact

Adding any field to EpochState (e.g., a `paused: bool`, a `protocol_version: u8`, or a `last_transition_slot: u64`) requires:

1. **Epoch Program rebuild and redeploy** with the new struct.
2. **On-chain account reallocation** (Solana `realloc` instruction or new PDA).
3. **Tax Program mirror update, rebuild, and redeploy** -- simultaneously.
4. **All client-side code** that deserializes EpochState must be updated.

If step 3 lags behind step 1-2, all swap operations fail because the Tax Program's `EpochState::try_deserialize()` encounters a layout mismatch.

### Affected Cross-Program Readers

| Consumer | File | Access Method | Failure Mode |
|----------|------|---------------|-------------|
| Tax Program (swap_sol_buy) | `swap_sol_buy.rs:67-72` | `EpochState::try_deserialize()` via mirror struct | Tax rate corruption or deserialization failure |
| Tax Program (swap_sol_sell) | `swap_sol_sell.rs:82-87` | Same mirror struct | Same |

Both swap instructions (buy and sell) read EpochState on every single user transaction. A layout mismatch would halt ALL protocol trading.

### Upgrade Scenarios

| Scenario | Padding Would Help? | Without Padding |
|----------|--------------------|-----------------|
| Append field at end | Yes -- no realloc needed | Requires realloc + coordinated redeploy |
| Insert field in middle | No | Both approaches require full migration |
| Change field type | No | Both approaches require full migration |
| Add version/feature flag | Yes -- most common upgrade | Blocked without realloc |

The most common schema evolution pattern (appending new fields) is the one most easily solved by padding.

## Root Cause

1. **No protocol-level schema evolution strategy**: The codebase has no documented approach for handling state account growth. Each new field has been handled ad-hoc (e.g., `carnage_lock_slot` was added in Phase 47, requiring a size bump from 92 to 100 bytes).
2. **Static assertions enforce tight packing**: The `const _: () = assert!(EpochState::DATA_LEN == 100)` assertion, while valuable for catching accidental size mismatches, also codifies the exact size as an invariant. Adding padding would require updating this assertion.
3. **Single-team development model**: With one team controlling all programs, the synchronization burden has been manageable. This assumption breaks with team growth, emergency patches, or third-party integrators.

## Recommendation

### R1: Add Reserved Padding to EpochState (Pre-Mainnet, Low Cost)

Add a `_reserved` field to both the canonical and mirror structs:

```rust
// In both epoch_state.rs and epoch_state_reader.rs:
pub bump: u8,
pub _reserved: [u8; 32],  // Reserved for future schema evolution
```

Update `LEN` to `8 + 132`, `DATA_LEN` to `132`, and the static assertion to `132`. Allocate the account at the new size.

32 bytes provides headroom for 4 u64 fields or 32 bool flags -- sufficient for several major versions of protocol evolution without reallocation.

### R2: Add DATA_LEN and Static Assertion to Tax Mirror (Immediate)

The Tax Program's mirror struct at `epoch_state_reader.rs` lacks a `DATA_LEN` constant and compile-time assertion. Add:

```rust
pub const DATA_LEN: usize = 8 + 4 + 8 + 1 + 2 + 2 + 2 + 2 + 2 + 2
    + 8 + 1 + 1 + 32 + 1 + 1 + 1 + 8 + 8 + 4 + 1 + 1;
const _: () = assert!(EpochState::DATA_LEN == 100);
```

This catches size mismatches at compile time within the Tax Program.

### R3: Add Padding to All Cross-Program-Read State Accounts (Pre-Mainnet)

Apply the same padding strategy to CarnageFundState (also read by multiple epoch instructions) and PoolState (read at raw byte offsets by Tax and Epoch programs). PoolState padding is particularly important because three separate consumers read it at hardcoded offsets.

### R4: Consider Protocol-Wide Over-Allocation Pattern (Long-term)

Allocate state accounts with 32-64 bytes of headroom even without struct padding. Anchor's `realloc` can later consume the headroom without a separate reallocation transaction. This is a one-time rent cost (~0.0003 SOL per 32 bytes per account) that buys significant operational flexibility.
