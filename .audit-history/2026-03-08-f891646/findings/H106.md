# H106 -- Transfer Hook: Authority Burn Completeness

## Status: INVALID (Burn is complete and irreversible)

## Hypothesis
Verify that `burn_authority` properly sets authority to `None`, and that no other instruction can set a new authority after the burn.

## Analysis

### burn_authority Implementation
In `programs/transfer-hook/src/instructions/burn_authority.rs`:

```rust
pub fn handler(ctx: Context<BurnAuthority>) -> Result<()> {
    let auth = &mut ctx.accounts.whitelist_authority;

    // Idempotent: already burned -> return Ok
    if auth.authority.is_none() {
        return Ok(());
    }

    // Verify signer is current authority
    require!(
        auth.authority == Some(ctx.accounts.authority.key()),
        TransferHookError::Unauthorized
    );

    // Burn the authority
    auth.authority = None;
    // ... emit event ...
}
```

The burn sets `authority` to `None`. Only the current authority can invoke this. Once burned, the idempotent guard returns early on subsequent calls.

### Can Authority Be Restored?

**initialize_authority** -- Uses Anchor `init` constraint:
```rust
#[account(
    init,
    payer = signer,
    space = 8 + WhitelistAuthority::INIT_SPACE,
    seeds = [WhitelistAuthority::SEED],
    bump
)]
pub whitelist_authority: Account<'info, WhitelistAuthority>,
```

The `init` constraint fails if the PDA already exists. Since `burn_authority` sets `authority = None` but does NOT close the account (account still exists with data), `initialize_authority` **cannot be called again**. The PDA persists with `authority = None`.

**add_whitelist_entry** -- Has constraint:
```rust
constraint = whitelist_authority.authority.is_some() @ TransferHookError::AuthorityAlreadyBurned
```
This rejects any whitelist additions after burn.

**No other instructions modify WhitelistAuthority.** The complete instruction set is:
1. `initialize_authority` -- one-shot init (blocked by existing PDA)
2. `add_whitelist_entry` -- gated by `authority.is_some()`
3. `burn_authority` -- sets to None (idempotent after first call)
4. `initialize_extra_account_meta_list` -- does not touch WhitelistAuthority
5. `transfer_hook` -- read-only hook execution, does not touch WhitelistAuthority

### Account Closure Attack Vector
Could an attacker close the WhitelistAuthority account to re-enable `initialize_authority`? No -- there is no `close` instruction in the program, and the account is a PDA owned by the Transfer Hook program. Only the program itself can modify or close it, and no instruction provides that capability.

### Program Upgrade Consideration
If the program is upgradeable, a malicious upgrade could add a "restore authority" instruction. This is noted in Architecture doc assumption A-7: "Programs are non-upgradeable at mainnet -- Not yet enforced." This is a deployment concern (S005 already covers the broader upgrade authority risk), not a code bug.

## Verdict
**No vulnerability.** The burn is complete and irreversible at the code level:
1. `burn_authority` sets `authority = None`
2. `initialize_authority` cannot re-init (PDA exists)
3. `add_whitelist_entry` rejects when `authority.is_none()`
4. No other instruction modifies the `WhitelistAuthority` state
5. No close instruction exists to garbage-collect the PDA

The whitelist becomes permanently immutable after burn.
