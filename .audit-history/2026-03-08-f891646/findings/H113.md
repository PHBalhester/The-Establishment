# H113 -- Staking: Zero Stake Edge Case

**Status:** SAFE
**Severity:** N/A
**Confidence:** HIGH

## Hypothesis

If `total_staked == 0` when `deposit_rewards` is called, division-by-zero could occur. Does the dead stake mechanism actually prevent this?

## Analysis

### deposit_rewards does NOT divide by total_staked

The `deposit_rewards` handler (`programs/staking/src/instructions/deposit_rewards.rs`) only increments `pool.pending_rewards` via `checked_add`. It performs zero division operations. The division by `total_staked` occurs later, in two separate places:

1. **`add_to_cumulative()`** (`programs/staking/src/helpers/math.rs:91-127`): Called from `update_cumulative`. Line 99-101 performs an early return `Ok(0)` when `total_staked == 0`, leaving `pending_rewards` unchanged (rewards stay pending for next epoch). Line 110 uses `checked_div` as a belt-and-suspenders guard.

2. **`update_cumulative` handler** (`programs/staking/src/instructions/update_cumulative.rs:88`): The inline version at line 88-93 guards with `if rewards_added > 0 && pool.total_staked > 0` before any division. Division uses `checked_div` with `ok_or(StakingError::DivisionByZero)`.

### Dead stake prevents total_staked == 0 in production

`initialize_stake_pool` (`programs/staking/src/instructions/initialize_stake_pool.rs:138`) sets `pool.total_staked = MINIMUM_STAKE` (1,000,000 = 1 PROFIT token) after transferring dead stake tokens to the vault. This dead stake is irrecoverable -- no instruction allows withdrawing it.

The `unstake` instruction (`programs/staking/src/instructions/unstake.rs:143`) enforces that if remaining balance after unstake < MINIMUM_STAKE, a full unstake occurs (zeroing the user's balance), but the dead stake in the pool is NOT a UserStake account -- it is protocol-owned tokens with no associated UserStake PDA. Users can only unstake their own balances, never the dead stake.

Therefore `total_staked >= MINIMUM_STAKE` is an invariant maintained throughout the protocol lifecycle.

### Defense in depth

Even if the dead stake invariant were violated:
- `add_to_cumulative` returns `Ok(0)` early, keeping rewards pending
- `update_cumulative` skips the division entirely via `&&` guard
- Both paths use `checked_div` as a final safety net

## Evidence

| File | Line | Mechanism |
|------|------|-----------|
| `deposit_rewards.rs` | 90-93 | Only `checked_add`, no division |
| `math.rs` | 99-101 | Early return `Ok(0)` when `total_staked == 0` |
| `math.rs` | 110-111 | `checked_div` fallback |
| `update_cumulative.rs` | 88 | `&& pool.total_staked > 0` guard |
| `initialize_stake_pool.rs` | 138 | Dead stake sets `total_staked = MINIMUM_STAKE` |
| `constants.rs` | 32 | `MINIMUM_STAKE = 1_000_000` |

## Verdict

**No vulnerability.** Three independent layers protect against division-by-zero: (1) dead stake ensures `total_staked >= 1M` at all times, (2) explicit zero-check guards before division, (3) `checked_div` with error propagation. The `deposit_rewards` instruction itself never divides at all.
