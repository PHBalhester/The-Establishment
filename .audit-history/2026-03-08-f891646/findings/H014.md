# H014: Buy Path Slippage -- 50% Output Floor Applied, Symmetric with Sell Path

**Status:** CONFIRMED (Informational -- Mitigated by Design)
**Severity:** LOW
**Category:** Timing, Token/Economic
**Confidence:** HIGH

## Summary

The buy path (`swap_sol_buy`) has the same 50% output floor (MINIMUM_OUTPUT_FLOOR_BPS=5000) as the sell path. The user's `minimum_output` parameter is passed directly to the AMM CPI (unlike the sell path which passes 0). This means the buy path has **two layers** of slippage protection: (1) the Tax Program's output floor requiring `minimum_output >= 50% of expected`, and (2) the AMM's own `amount_out >= minimum_amount_out` check. The sell path has only one effective layer (the AMM's check is bypassed with minimum=0).

The original S010 finding from the previous audit concerned slippage bypass. The 50% floor was added as mitigation. This investigation confirms the floor applies to **both** buy and sell paths, but the 50% threshold still permits up to ~50% value extraction via sandwich attacks on both paths.

## Evidence

### Buy Path Protection (Two Layers)

**Layer 1 -- Tax Program output floor** at `swap_sol_buy.rs:105-111`:
```rust
let (reserve_a, reserve_b) = read_pool_reserves(&ctx.accounts.pool)?;
let output_floor = calculate_output_floor(reserve_a, reserve_b, sol_to_swap, MINIMUM_OUTPUT_FLOOR_BPS)
    .ok_or(error!(TaxError::TaxOverflow))?;
require!(
    minimum_output >= output_floor,
    TaxError::MinimumOutputFloorViolation
);
```

**Layer 2 -- AMM slippage check** at `swap_sol_buy.rs:262`:
```rust
ix_data.extend_from_slice(&minimum_output.to_le_bytes());
```
The user's `minimum_output` is forwarded directly to the AMM, which enforces `amount_out >= minimum_amount_out` at `swap_sol_pool.rs:146`.

### Sell Path Protection (One Effective Layer)

**Layer 1 -- Tax Program output floor** at `swap_sol_sell.rs:112-118` (same as buy).

**Layer 2 -- AMM slippage: BYPASSED** at `swap_sol_sell.rs:147`:
```rust
let amm_minimum: u64 = 0;
```

**Layer 3 -- Post-CPI net check** at `swap_sol_sell.rs:245`:
```rust
require!(net_output >= minimum_output, TaxError::SlippageExceeded);
```

### Output Floor Calculation

`MINIMUM_OUTPUT_FLOOR_BPS = 5000` defined at `tax-program/constants.rs:40`.

`calculate_output_floor()` at `tax_math.rs:141-165` computes:
```
expected = reserve_out * amount_in / (reserve_in + amount_in)
floor = expected * 5000 / 10_000   // 50% of expected
```

### Asymmetry Between Buy and Sell

| Protection | Buy Path | Sell Path |
|---|---|---|
| Output floor (pre-CPI) | `minimum_output >= 50% expected` | `minimum_output >= 50% expected` |
| AMM minimum_amount_out | User's `minimum_output` forwarded | Hardcoded to 0 |
| Post-CPI slippage check | N/A (AMM enforces) | `net_output >= minimum_output` |
| Max extractable by sandwich | ~50% of expected output | ~50% of expected output |

The buy path is marginally stronger (AMM enforces the user's minimum directly), but the effective protection level is identical: both paths allow up to ~50% value extraction.

## Attack Scenario

**Standard sandwich attack on buy path:**

1. Attacker front-runs: large buy moves price up in the pool.
2. User's buy TX executes with `minimum_output` set to just above the 50% floor.
3. User receives fewer tokens than expected (but above their minimum).
4. Attacker back-runs: sells to restore price, capturing the spread.

**Worst case:** User sets `minimum_output` exactly at the 50% floor. Attacker extracts up to ~50% of the user's expected token output. This is identical to the sell path risk.

**Mitigating factors:**
- Rational users/frontends set `minimum_output` much higher than the 50% floor (typically 1-5% slippage tolerance).
- The 50% floor only catches bots/misconfigured frontends that would set minimum_output=0.
- Solana's fast block times (~400ms) reduce sandwich profitability vs Ethereum.

## Root Cause

The 50% output floor is a protocol-level MEV defense, not a per-user slippage protection. It establishes a generous lower bound that prevents catastrophic extraction (e.g., zero-slippage bots) but intentionally does not enforce tight slippage -- that responsibility is delegated to the user's `minimum_output` parameter and the frontend that sets it.

The real protection against sandwich attacks is the user-specified `minimum_output`, which on both paths must be >= 50% floor. The floor prevents the worst-case scenario (minimum_output=0) but permits up to 50% extraction for users who set their minimum at the floor boundary.

## Recommendation

**No on-chain change required.** The current design is intentional and symmetric across buy/sell paths.

1. **Frontend enforcement (client-side):** Ensure the frontend defaults to a tight slippage tolerance (e.g., 1-3%) and computes `minimum_output` accordingly. The 50% floor is a backstop, not the intended user experience.

2. **Consider tightening the floor for mainnet:** The 50% floor (MINIMUM_OUTPUT_FLOOR_BPS=5000) could be tightened to 80-90% (8000-9000 bps) to reduce MEV surface. This would reject more legitimate large-impact swaps against thin liquidity but significantly reduce sandwich profitability. Trade-off depends on expected pool depth at launch.

3. **Document the design decision:** The sell path's `amm_minimum=0` is deliberate (tax is on output, so the AMM can't enforce a post-tax minimum). The buy path correctly forwards `minimum_output` to the AMM. Both paths rely on the 50% floor as the protocol-level defense.
