# S009: Bonding Curve distribute_tax_escrow -- Fund Direction Analysis

**Hypothesis:** `distribute_tax_escrow` has the same bare-signer pattern as other BC instructions. If tax escrow contains SOL, an attacker could direct it to themselves.

**Determination:** NOT VULNERABLE

**Severity:** N/A

**Confidence:** HIGH

---

## Investigation

### Previous Audit Reference

H067 (Audit #1) covered PoolState size mismatch concerns for the AMM, not `distribute_tax_escrow`. The hypothesis incorrectly references H067 -- the relevant BC authority findings are in the H001-series (BC authority gap). However, `distribute_tax_escrow` is explicitly listed in the Architecture document (Section 2, Trust Tier 5) as "permissionless, correctly gated by state conditions and timing, not identity."

### Key Finding: No Authority Account Required

Unlike the 6 vulnerable BC instructions (`initialize_curve`, `fund_curve`, `start_curve`, `prepare_transition`, `withdraw_graduated_sol`, `close_token_vault`), `distribute_tax_escrow` does **NOT accept an `authority` Signer at all**. Its accounts struct is:

```rust
// programs/bonding_curve/src/instructions/distribute_tax_escrow.rs:23-57
pub struct DistributeTaxEscrow<'info> {
    pub curve_state: Account<'info, CurveState>,     // read-only, PDA validated
    pub tax_escrow: UncheckedAccount<'info>,          // PDA validated by seeds + stored key
    pub carnage_fund: UncheckedAccount<'info>,        // PDA validated against epoch program
}
```

There is no `Signer<'info>` in this struct. No one "directs" funds -- the instruction is fully deterministic.

### Fund Flow Analysis

1. **Source:** `tax_escrow` PDA, seeded as `["tax_escrow", token_mint]`, owned by the bonding curve program. Validated via both PDA seeds AND `constraint = tax_escrow.key() == curve_state.tax_escrow` (lines 36-41).

2. **Destination:** `carnage_fund` PDA, validated as the Epoch Program's `carnage_sol_vault` via explicit derivation check (lines 47-56):
   ```rust
   constraint = carnage_fund.key()
       == Pubkey::find_program_address(
           &[CARNAGE_SOL_VAULT_SEED],
           &epoch_program_id(),
       ).0
   ```

3. **Transfer mechanism:** Direct lamport manipulation (lines 91-92):
   ```rust
   **ctx.accounts.tax_escrow.try_borrow_mut_lamports()? -= transferable;
   **ctx.accounts.carnage_fund.try_borrow_mut_lamports()? += transferable;
   ```
   The bonding curve program owns `tax_escrow`, so it can subtract lamports. Any program can add lamports to any account (Solana runtime rule). The destination is hardcoded to the validated `carnage_fund` PDA.

### State Guards

- **Graduation check (line 75):** `curve_state.status == CurveStatus::Graduated` -- only graduated curves can distribute
- **Non-zero check (line 86):** `transferable > 0` -- prevents empty distribution
- **Rent preservation (line 83):** `escrow_lamports.saturating_sub(rent_exempt)` -- keeps escrow PDA alive

### Attack Scenario Analysis

**Can an attacker redirect funds?**
No. The destination is a PDA derived from fixed seeds (`CARNAGE_SOL_VAULT_SEED`) and a fixed program (`epoch_program_id()`). An attacker cannot substitute a different destination because:
1. The `carnage_fund` account is validated via `Pubkey::find_program_address` in the constraint
2. There is no parameter or signer that influences where funds go
3. The transfer uses direct lamport manipulation, not a system transfer with a user-specified destination

**Can an attacker call this prematurely?**
Only if the curve is `Graduated`. The `CurveStatus::Graduated` state is reached via `prepare_transition` (which has the BC authority vulnerability). However, even if an attacker forces graduation, the tax escrow SOL goes to the carnage fund -- a protocol-owned PDA -- not to the attacker.

**Is this the "bare signer" pattern?**
No. This instruction has NO signer account. It is purely permissionless and deterministic. The hypothesis confused this with the 6 instructions that DO have the bare `authority: Signer<'info>` pattern.

## Recommendation

No action required. This instruction is correctly designed as a permissionless, state-gated operation with hardcoded PDA destinations. The Architecture document correctly classifies it under Trust Tier 5 (Permissionless, State-Gated).

## Determination Rationale

- No authority/signer account: instruction is fully permissionless
- Destination is a hardcoded PDA validated via `find_program_address` constraint
- State guard requires `Graduated` status
- Direct lamport manipulation with program-owned source -- no redirection possible
- This is NOT the "bare signer" pattern -- hypothesis was based on incorrect assumption
