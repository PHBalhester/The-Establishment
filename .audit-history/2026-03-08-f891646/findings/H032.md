# H032 -- WSOL Intermediary DoS Vector

## Status: REJECTED (False Positive)

## Hypothesis
If swap_authority PDA lamports are drained below rent-exempt for intermediary recreation, all sell operations halt.

## Category
CPI, Token/Economic

## Target Code
- `programs/tax-program/src/instructions/swap_sol_sell.rs` (lines 292-451)
- `programs/tax-program/src/instructions/initialize_wsol_intermediary.rs`

## Investigation

### 1. WSOL Intermediary Lifecycle

The sell path uses a close-and-recreate cycle for a WSOL intermediary token account:

1. **Transfer** (line 269-290): User's WSOL tax amount transferred into intermediary
2. **Close** (line 295-312): Intermediary closed to swap_authority, unwrapping WSOL to native SOL
3. **Distribute** (lines 318-398): swap_authority sends tax portions to staking/carnage/treasury
4. **Recreate** (lines 409-451): swap_authority funds new intermediary with rent-exempt lamports

### 2. Lamport Balance Analysis

Tracing the swap_authority PDA balance through one sell instruction:

```
Before sell:    swap_authority has B lamports

Step 2 (close): +rent_exempt + tax_amount   (from intermediary)
Step 3 (dist):  -tax_amount                 (staking + carnage + treasury = tax_amount via remainder pattern)
Step 4 (reinit):-rent_lamports              (fund new intermediary)

After sell:     B + rent_exempt - rent_lamports = B + 0 = B
```

The `rent_exempt` received from closing and the `rent_lamports` spent on recreation are computed from the same `Rent::get()?.minimum_balance(165)` call within the same instruction. They are identical. The swap_authority balance is **invariant across sells**.

### 3. Can swap_authority Be Drained Externally?

The swap_authority is a PDA derived from `[SWAP_AUTHORITY_SEED]` under the Tax Program. Key properties:

- **System-owned** (never initialized with `init`), starts at 0 lamports
- **Only the Tax Program** can sign for it via `invoke_signed` with the correct seeds
- No instruction in any program transfers lamports OUT of swap_authority except `swap_sol_sell` (which is self-balancing as shown above)
- In the buy path (`swap_sol_buy.rs`), swap_authority is **not mutable** -- it only acts as a CPI signer, no lamport flow
- In `swap_exempt.rs`, swap_authority is also **not mutable** -- read-only CPI signer

Since no external party can produce a valid PDA signature for swap_authority, no one can drain it via `system_instruction::transfer` or any other signed operation.

### 4. Initial Funding

The intermediary is created during deployment via `initialize_wsol_intermediary`, where the **admin pays the rent** directly. The swap_authority never needs to hold pre-existing lamports. On the first sell, close gives swap_authority `rent + tax`, distribute takes `tax`, recreate takes `rent`. Net change: 0.

### 5. Edge Case: Accumulated Lamports

If someone sends SOL to the swap_authority PDA directly (anyone can credit any account on Solana), those lamports would accumulate harmlessly. They cannot be extracted (no instruction drains swap_authority beyond the self-balancing sell cycle), but they also do not interfere with operation.

## Conclusion

The hypothesis is **false**. The swap_authority PDA cannot be drained because:

1. Only the Tax Program can sign for it
2. The only instruction that moves lamports out (swap_sol_sell) is self-balancing -- it receives exactly as much as it spends within each invocation
3. The intermediary recreation is funded by the intermediary's own rent recovered from the close step
4. No external actor can produce the PDA signature needed to transfer lamports out

The WSOL intermediary close-reinit pattern is architecturally sound. The rent lamports flow in a closed loop (intermediary -> swap_authority -> new intermediary) with zero leakage.

## Severity: N/A (Not a vulnerability)

## Evidence
- `swap_sol_sell.rs:292-312`: Close intermediary to swap_authority (receives rent + tax)
- `swap_sol_sell.rs:318-398`: Distribute tax_amount only (staking + carnage + treasury via remainder)
- `swap_sol_sell.rs:405-407`: `rent_lamports = rent.minimum_balance(165)` -- same calculation as implicit close amount
- `swap_sol_sell.rs:409-426`: Recreate intermediary funded by swap_authority
- `swap_sol_sell.rs:510-515`: swap_authority PDA constraint (`seeds = [SWAP_AUTHORITY_SEED], bump`) -- Tax Program owns the signature
- `swap_sol_buy.rs:370-376`: swap_authority is NOT mutable in buy path -- no lamport flow
- `swap_exempt.rs:204-208`: swap_authority is NOT mutable in exempt path -- no lamport flow
