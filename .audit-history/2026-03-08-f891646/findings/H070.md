## Hypothesis

A single crank operator running the epoch transition pipeline sees VRF results off-chain before submitting TX3 (reveal + consume + carnage). This gives the operator an information monopoly: they can compute upcoming tax rates and Carnage trigger status before any other market participant, and sequence personal trades to extract value from the tax differential (up to 13% per trade) or front-run Carnage swaps (up to 1000 SOL).

## Investigation

### 1. Current Crank Architecture -- Single Operator

The crank runner (`scripts/crank/crank-runner.ts`) is a single process deployed on Railway (per project MEMORY). It runs an infinite loop:

1. Wait for epoch boundary (~750 slots devnet / ~4500 slots mainnet)
2. Call `advanceEpochWithVRF()` which executes the 3-TX flow
3. Log results, repeat

There is no on-chain mechanism to enforce multiple competing crank operators. The bounty (0.001 SOL per epoch, `TRIGGER_BOUNTY_LAMPORTS = 1_000_000` at `constants.rs:81`) is designed to be small -- approximately 66x the 3-TX base cost. At mainnet (30-minute epochs), this translates to ~0.048 SOL/day, which is insufficient to attract competitive operators purely on bounty economics.

### 2. Information Flow Timeline

The critical information advantage occurs during TX3 construction in `vrf-flow.ts:269-341`:

```
Timeline:
  T+0:    Crank calls randomness.revealIx() via Switchboard SDK
          SDK queries oracle gateway, receives VRF bytes in the instruction data
          >>> CRANK NOW KNOWS ALL 32 VRF BYTES OFF-CHAIN <<<

  T+~0s:  Crank builds consumeIx + carnageIx
          Crank assembles and signs TX3

  T+~1s:  TX3 submitted to Solana RPC
          TX3 enters leader's transaction queue

  T+~2s:  TX3 lands on-chain
          EpochState updated, new tax rates public
```

Between T+0 and T+~2s, the crank operator exclusively knows:

| Information | VRF Bytes | Derivation |
|---|---|---|
| New cheap_side | byte 0 | `< 192` = flip (75% probability) |
| CRIME low tax | byte 1 | `LOW_RATES[byte % 4]` = 100/200/300/400 bps |
| CRIME high tax | byte 2 | `HIGH_RATES[byte % 4]` = 1100/1200/1300/1400 bps |
| FRAUD low tax | byte 3 | Same as CRIME low |
| FRAUD high tax | byte 4 | Same as CRIME high |
| Carnage triggers? | byte 5 | `< 11` = yes (~4.3% chance) |
| Carnage action | byte 6 | `< 5` = sell (2%), else burn (98%) |
| Carnage target | byte 7 | `< 128` = CRIME, else FRAUD |

### 3. Exploitation Strategies

**Strategy A: Tax Rate Front-Running**

The crank operator knows the new tax rates ~2 seconds before TX3 lands. If current rates are unfavorable (e.g., CRIME buy tax = 14%) and new rates are favorable (e.g., CRIME buy tax = 1%), the operator could:

1. Learn VRF bytes from `revealIx()` SDK call
2. Submit a buy-CRIME swap at old (14%) rate -- wait, this is the WRONG direction
3. Actually: delay TX3 submission, execute a swap at OLD rates if OLD rates are more favorable, then submit TX3
4. Or: submit TX3 + a personal swap in a Jito bundle, ordering the personal swap AFTER TX3 to get the new favorable rate

Maximum tax differential: 14% - 1% = 13% per trade. However, this requires:
- Jito bundle infrastructure for transaction ordering
- Sufficient capital and AMM liquidity for meaningful extraction
- The specific rate transition to be favorable (not every epoch)

**Strategy B: Carnage Awareness**

When byte 5 < 11 (4.3% chance), the operator knows Carnage will trigger and which pool/direction. Since Carnage is bundled atomically in TX3, the operator cannot sandwich the Carnage swap itself within TX3. However, they could:

1. Buy the Carnage target token BEFORE submitting TX3 (front-run)
2. TX3 lands, Carnage buys the same token (pushing price up)
3. Sell the target token (back-run)

This requires splitting the flow: personal buy TX -> TX3 (with Carnage) -> personal sell TX. The atomic bundling prevents intra-TX3 extraction but not cross-TX extraction.

**Strategy C: Epoch Delay**

The crank operator can delay TX3 to extend the VRF pending window. During this window (`taxes_confirmed = false`), swaps execute at OLD rates (`swap_sol_buy.rs:78` calls `get_tax_bps()` without checking `taxes_confirmed`, per assumption A-10). The operator could:

1. Learn VRF bytes, see that new rates favor a specific trade
2. Execute personal trades at old rates
3. Then submit TX3 to update rates

The on-chain code does not enforce a maximum delay between TX2 (commit+trigger) and TX3 (reveal+consume). The only timeout is VRF_TIMEOUT_SLOTS = 300 (~2 minutes), after which `retry_epoch_vrf` becomes callable by anyone. So the operator has up to ~2 minutes of exclusive information advantage per epoch.

### 4. Economic Impact Assessment

**Per-epoch maximum extraction (Strategy A):**
- Tax differential: up to 13% (1400 bps -> 100 bps)
- Trade size limited by: AMM pool liquidity, 50% output floor, operator capital
- Realistic trade: 10 SOL at 13% differential = 1.3 SOL profit
- Epoch frequency: every ~5 min (devnet) / ~30 min (mainnet)
- Daily mainnet ceiling: 48 epochs * 1.3 SOL = 62.4 SOL/day (extremely optimistic upper bound)
- Realistic daily extraction: Much lower -- favorable rate transitions don't happen every epoch; the 75% flip probability means the cheap_side changes most epochs but the magnitude differential varies

**Per-Carnage extraction (Strategy B):**
- Carnage probability: ~4.3% per epoch
- Carnage swap: up to 1000 SOL
- Front-run profit bounded by: pool depth, operator capital, slippage
- Realistic: 5-15% of Carnage amount if pools are thin = 50-150 SOL per trigger
- Expected frequency: ~2 triggers/day at mainnet cadence
- Daily ceiling: ~100-300 SOL/day (extremely optimistic, assumes large Carnage fund balance)

**Practical constraints reducing extraction:**
1. AMM pool liquidity limits trade size before slippage eats profits
2. 50% output floor caps maximum price impact
3. Jito bundle fees reduce net profit
4. Not every epoch transition produces a favorable rate swing
5. Carnage fund balance may be small (especially early)

### 5. On-Chain Enforcement Gap

There is no on-chain mechanism to:
- Require multiple crank operators
- Enforce minimum time between VRF reveal and personal trades
- Detect or penalize crank operator self-dealing
- Rate-limit swaps around epoch boundaries

The `trigger_epoch_transition` instruction is permissionless (`payer: Signer<'info>` at `trigger_epoch_transition.rs:30`), meaning anyone CAN run a crank. But the low bounty (0.001 SOL) and operational complexity (3-TX VRF flow, Switchboard SDK, error recovery) create a natural monopoly.

### 6. Comparison with H016 and H030

H016 analyzed the same information asymmetry from the tax-rate perspective and concluded LOW severity because:
- Atomic TX3 bundling closes the on-chain VRF visibility window
- The remaining risk is crank-operator-only, requiring MEV infrastructure

H030 analyzed it from the Carnage front-running perspective and also concluded LOW because:
- Atomic bundling prevents intra-TX Carnage sandwich
- Fallback path MEV is bounded by slippage floors

H070 synthesizes both vectors into the broader "information monopoly" framing: the single crank operator has exclusive off-chain knowledge of ALL VRF-derived outcomes (tax rates AND Carnage trigger) before any other participant. The compound advantage across both vectors is greater than either alone, but the practical extraction remains bounded by the same mitigations.

## Verdict

**LOW severity.** The crank operator's information monopoly is real and confirmed. A single Railway-hosted process exclusively knows VRF results ~2 seconds before they land on-chain (or up to ~2 minutes if the operator deliberately delays TX3). This enables:

1. **Tax rate front-running**: Up to 13% differential per trade, bounded by pool liquidity and 50% output floor
2. **Carnage front-running**: Cross-TX sandwich of Carnage swaps (~4.3% of epochs), bounded by atomic bundling preventing intra-TX extraction
3. **Epoch delay**: Extending the stale-rate window to trade at known-favorable old rates

The severity is LOW rather than MEDIUM because:
- Atomic TX3 bundling eliminates the highest-value intra-transaction extraction vectors
- Practical extraction requires additional MEV infrastructure (Jito bundles) beyond just running the crank
- The 50% output floor and discrete rate bands (only 8 possible values) limit per-trade extraction
- The bounty economics naturally attract automated bots at mainnet scale, which would dilute any single operator's monopoly

### Why Not INFORMATIONAL

Unlike a pure design observation, this finding describes a concrete, exploitable information advantage with quantifiable extraction bounds. The single-operator deployment is a confirmed operational reality (Railway crank runner), not a theoretical concern.

## Mitigating Factors

1. **Atomic TX3 bundling**: `revealIx + consumeIx + carnageIx` in one transaction prevents on-chain intermediate state leakage
2. **50% output floor**: All swaps capped at 50% maximum price impact (`MINIMUM_OUTPUT_FLOOR_BPS = 5000`)
3. **VRF timeout (300 slots)**: Caps maximum delay before anyone can retry, limiting the stale-rate window
4. **Discrete rate bands**: Only 8 possible tax rates reduce surprise range
5. **Permissionless trigger**: Anyone can run a competing crank -- monopoly is economic, not cryptographic
6. **Low per-epoch value**: 0.001 SOL bounty means operator's primary revenue must come from other sources, incentivizing legitimate operation

## Residual Risk

The information monopoly is inherent to any single-operator VRF commit-reveal system where the operator also has trading access. This cannot be fully eliminated without one of:

1. **Threshold VRF / MPC**: Multiple parties must collaborate to reveal, eliminating single-party information advantage
2. **Enforced operator isolation**: On-chain mechanism preventing the crank wallet from trading (impractical -- operator can use a separate wallet)
3. **Competitive crank market**: Multiple operators racing to trigger transitions, reducing any single operator's timing window

**Recommendations (ordered by effort):**
1. **Increase bounty**: Higher bounty (e.g., 0.01 SOL) attracts competing operators, reducing monopoly. Cost: ~0.48 SOL/day at mainnet cadence.
2. **Swap cooldown at epoch boundary**: Tax Program checks `clock.slot - epoch_state.epoch_start_slot < COOLDOWN_SLOTS` and rejects swaps during the first N slots of an epoch. Prevents front-running the rate change. Cost: minor latency for all traders at epoch boundaries.
3. **Log VRF bytes hash in TX2**: Emit a hash of the expected VRF bytes in the commit TX so observers can verify the crank didn't delay TX3 to trade. Doesn't prevent exploitation but enables detection.
4. **Competitive crank incentive redesign**: Dutch-auction bounty that increases over time after epoch boundary, incentivizing fastest possible transition.
