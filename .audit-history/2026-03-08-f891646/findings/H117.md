# H117 -- Carnage: Dual Pool Execution Order

## Status: NOT A BUG -- Low Risk, Mitigated by Design

## Target
`programs/epoch-program/src/instructions/execute_carnage_atomic.rs`

## Hypothesis
Carnage operates on two pools (CRIME/SOL and FRAUD/SOL). Does the execution order of sell-then-buy across pools create exploitable inter-pool arbitrage?

## Analysis

### Execution Flow

Carnage has three action types:
1. **Burn**: Burns held tokens, then buys target token. No pool interaction for disposal.
2. **Sell**: Sells held tokens on Pool A (held token's pool), then buys target token on Pool B (target token's pool).
3. **BuyOnly (None)**: No disposal step, just buys target token.

Only the **Sell** action touches two pools sequentially.

### Sell Path Order

When Carnage holds (e.g.) CRIME and targets FRAUD:
1. **Sell CRIME on CRIME/SOL pool** (BtoA direction): Receives WSOL.
2. **Buy FRAUD on FRAUD/SOL pool** (AtoB direction): Spends WSOL (from sell + tax accumulation).

The sell depresses CRIME's price (adds CRIME supply, removes SOL from pool). The buy inflates FRAUD's price (adds SOL, removes FRAUD from pool). These are two independent pools with no shared reserves.

### Inter-Pool Arbitrage Assessment

**Could an attacker front-run to extract value?**

- The sell on Pool A and buy on Pool B happen in the SAME transaction (atomic). An attacker cannot insert transactions between them.
- An attacker could sandwich the entire Carnage transaction, but:
  - Carnage is triggered by VRF (unpredictable which epoch triggers it, ~4.3% chance).
  - The action (Burn/Sell/BuyOnly) and target (CRIME/FRAUD) are determined by VRF randomness consumed in the same transaction or the immediately prior `consume_randomness` call.
  - `execute_carnage_atomic` is designed to be bundled atomically with `consume_randomness` in a single TX, so the attacker cannot observe the VRF result before Carnage executes.

**Could execution order itself leak value?**

- The sell on Pool A does not affect Pool B's reserves. The constant-product formula on each pool operates independently.
- The order (sell A then buy B) vs (buy B then sell A) would produce identical net results because the pools have no shared state.
- The combined buy amount (`total_buy_amount`) is capped at `MAX_CARNAGE_SWAP_LAMPORTS` (1000 SOL), preventing the sell proceeds from causing an outsized buy.

### Slippage Protection

The buy step has an 85% slippage floor (`CARNAGE_SLIPPAGE_BPS_ATOMIC = 8500`):
```rust
let min_output = expected * 8500 / 10_000;
require!(bought >= min_output, CarnageSlippageExceeded);
```

The sell step has NO explicit slippage floor -- it relies on the AMM's constant-product formula and the fact that sell proceeds are immediately used for the buy (no value sitting idle between TXs).

### Pool Reserve Read Without Owner Check

The `read_pool_reserves` function reads raw bytes at offsets [137..153] from the pool AccountInfo without verifying the account is owned by the AMM program. This is flagged in ARCHITECTURE.md Section 7.6. A spoofed pool account could feed false reserves to the slippage calculation. However, the pool accounts are validated by the Tax Program during the actual swap CPI, so a spoofed pool would cause the swap itself to fail.

## Verdict

**Inter-pool arbitrage from execution order: NOT EXPLOITABLE.** The two pool operations are independent (no shared reserves), atomic (same TX), and the order doesn't matter for the final outcome. VRF unpredictability + atomic bundling provide primary MEV defense.

**Residual risk:** The sell step lacks its own slippage floor. If pool A is manipulated before the TX, the sell could receive less WSOL than expected. The buy step's 85% floor only protects the buy, not the sell. This is a known design tradeoff documented in Phase 47 -- the primary defense is VRF unpredictability making pre-positioning unprofitable.

## Severity: Informational
