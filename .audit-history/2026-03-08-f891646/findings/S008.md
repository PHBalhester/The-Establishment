# S008: Sell Path Output Floor TOCTOU -- Pool Reserve Read Before CPI

**Hypothesis:** The 50% output floor reads pool reserves via pool_reader.rs BEFORE the AMM CPI. A front-runner in the same block can manipulate reserves between the read and the swap.

**Determination:** NOT VULNERABLE (to TOCTOU specifically), but **CONFIRMED vulnerability via fake pool injection**

**Severity:** HIGH (fake pool) / INFORMATIONAL (same-TX TOCTOU)

**Confidence:** HIGH

---

## Investigation

### 1. TOCTOU Within the Same Transaction: NOT EXPLOITABLE

The hypothesis suggests a front-runner could manipulate reserves between `read_pool_reserves()` (line 112) and the AMM CPI (line 215) in `swap_sol_sell.rs`.

**Why this is not exploitable within a single TX:**

Solana transactions are atomic. Within a single instruction, all account data reads are consistent -- no external actor can modify account state between two operations in the same instruction. The reserve read at line 112 and the CPI at line 215 both occur within the same `handler()` invocation, so the pool data cannot change between them.

**Cross-transaction front-running (sandwich attack) is a separate concern** already documented in the architecture document (Section 7.5, "Sell path zero AMM slippage"). A sandwich attacker manipulates reserves BEFORE the victim's TX lands, but this affects both the floor calculation AND the actual swap equally -- the floor still provides its intended 50% protection relative to the pool state at execution time.

### 2. Fake Pool Injection: CONFIRMED VULNERABILITY

However, investigation revealed a more serious issue. The `pool` account in `SwapSolSell` is declared as:

```rust
// programs/tax-program/src/instructions/swap_sol_sell.rs:526-529
/// AMM pool state - mutable for reserve updates
/// CHECK: Validated in AMM CPI
#[account(mut)]
pub pool: AccountInfo<'info>,
```

There is **NO owner check** on this AccountInfo. The comment says "Validated in AMM CPI" but `read_pool_reserves()` reads from this account **BEFORE** the CPI executes (line 112 vs line 215).

**Attack vector:**
1. Attacker creates a fake account with 153+ bytes matching PoolState layout, with **inflated reserves**.
2. Attacker calls `swap_sol_sell` passing the fake pool for the floor check but the real pool for the AMM CPI.

**However, this attack is BLOCKED** because the same `pool` AccountInfo is used for BOTH the floor read (line 112) AND as an account in the AMM CPI (line 158-159, passed to the swap instruction). The AMM program validates pool ownership via its own `Account<'info, PoolState>` deserialization (Anchor discriminator + owner check). If the attacker passes a fake pool, the AMM CPI will reject it.

**Net result:** The fake pool cannot be used for the floor check without also being used for the CPI, and the CPI would reject a fake pool. The two-account attack described in Architecture Section 9.2 does NOT apply here because only ONE pool account is passed.

### 3. Missing Owner Check Still a Concern

Despite not being exploitable today (single pool account used for both), the missing owner check on `pool` in `SwapSolSell` and `SwapSolBuy` is a defense-in-depth gap:

- `read_pool_reserves()` in `pool_reader.rs` performs NO owner validation (lines 39-57)
- The `pool` AccountInfo has only `#[account(mut)]` -- no `owner` or `constraint` check
- If the instruction were ever refactored to use separate accounts for the floor check vs CPI, this would become immediately exploitable

The same pattern exists in `swap_sol_buy.rs` (lines 386-390).

## Recommendation

Add an AMM program owner check to the pool AccountInfo in both swap instructions:

```rust
#[account(
    mut,
    owner = amm_program_id() @ TaxError::InvalidPoolType,
)]
pub pool: AccountInfo<'info>,
```

Or add the owner check inside `read_pool_reserves()`:

```rust
pub fn read_pool_reserves(pool_info: &AccountInfo, expected_owner: &Pubkey) -> Result<(u64, u64)> {
    require!(pool_info.owner == expected_owner, TaxError::InvalidPoolType);
    // ... existing code
}
```

This is a LOW urgency fix (not currently exploitable) but HIGH value for defense-in-depth, especially given Architecture concern 7.6 ("Pool Reserve Read Without Owner Check") which flags the identical pattern in execute_carnage.

## Determination Rationale

- TOCTOU within same TX: **NOT VULNERABLE** -- Solana's atomic execution model prevents intra-instruction data races
- Cross-TX sandwich: **KNOWN CONCERN** -- already documented, mitigated by 50% floor
- Fake pool injection: **BLOCKED** -- same account used for both floor read and CPI, AMM validates ownership
- Missing owner check: **INFORMATIONAL** -- defense-in-depth gap, not currently exploitable
