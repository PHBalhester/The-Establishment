# H049 -- Cross-Program Upgrade Cascade

**Severity:** MEDIUM
**Category:** Upgrade/Admin
**Status:** CONFIRMED
**Audit:** sos-002-20260307-f891646

---

## Summary

The protocol's 7 production programs form a tightly coupled dependency mesh via 15 hardcoded cross-program ID references and 4 shared PDA seed constants. Redeploying any single program to a new address requires rebuilding and redeploying every program that references it. This multi-transaction upgrade is not atomic on Solana, creating an inconsistency window where some programs reference stale IDs and core protocol operations (swaps, staking rewards, epoch transitions, carnage) will fail.

## Root Cause

All cross-program references are compile-time constants (`pubkey!()` / `Pubkey::from_str()`) embedded in each program's binary. There is no on-chain registry or indirection layer. Changing one program's address means updating the constant in every dependent, rebuilding, and redeploying each -- a multi-step non-atomic operation.

## Inconsistency Window Assessment

### In-Place Upgrade (same address, standard Solana BPF upgrade)

No cascade required. The program ID stays the same, so all references remain valid. This is the expected upgrade path.

**Risk:** If a struct layout changes (e.g., adding a field to EpochState), the Tax Program's mirror struct becomes invalid. The in-place upgrade of Epoch Program takes effect immediately, but Tax Program still deserializes with the old layout until it too is upgraded. During this window:
- Tax reads stale/corrupt fields from EpochState -> wrong tax rates applied to all swaps.
- `read_pool_reserves()` at wrong byte offsets -> slippage calculation corrupt -> possible sandwich attacks bypass floor.

**Window duration:** Minutes to hours (depends on how fast all programs can be rebuilt and upgraded sequentially). Cannot be reduced to zero because Solana program upgrades are individual transactions.

### New-Address Deployment (program keypair changes)

Full cascade required. The old program at the old address continues operating. Programs not yet updated still call the old address. Programs already updated call the new address which may not have state initialized.

**Risk:** During the cascade:
1. Swaps halt (Tax -> AMM CPI fails if either side has mismatched ID).
2. Staking reward deposits halt (Tax -> Staking CPI fails).
3. Epoch transitions halt (Epoch -> Staking CPI fails).
4. Carnage execution halts (Epoch -> Tax CPI fails).

**Window duration:** Potentially hours. Each program must be built, deployed, and verified in the correct order.

### Required Upgrade Ordering (new-address scenario)

Due to circular dependencies (AMM <-> Tax, Tax <-> Staking, Tax <-> Epoch, Epoch <-> Staking), a single-pass ordered deployment is **impossible**. The protocol requires a two-pass deploy:

1. **Pass 1:** Deploy all programs with new addresses (using pre-generated keypairs so IDs are known).
2. **Pass 2:** Rebuild all programs with the new cross-references, redeploy.

This is already the established pattern (`deploy-all.sh` two-pass deploy), but it means the protocol is fully non-functional between passes.

## Existing Mitigations

1. **In-place upgrades preserve addresses.** Standard Solana BPF upgrades do not change the program ID, so the cascade only applies to struct layout changes or complete redeployments.
2. **Unit tests verify ID consistency.** All `constants.rs` files have tests asserting cross-program IDs match expected strings.
3. **Two-pass deploy pipeline exists.** `deploy-all.sh` already handles the chicken-and-egg problem with pre-generated keypairs.
4. **Shared seed constants are byte-literal.** Simple string constants (`b"swap_authority"`) are easy to verify visually.

## Residual Risk

1. **Struct layout change during in-place upgrade** has no atomic solution. Even with in-place upgrades, changing EpochState or PoolState layout creates a deserialization mismatch window. There is no compile-time enforcement that the Tax Program's mirror matches the Epoch Program's source of truth.
2. **No circuit breaker during upgrade.** If a partially-upgraded state causes incorrect behavior (e.g., wrong tax rates), there is no pause mechanism to halt swaps while the remaining programs are upgraded.
3. **Emergency new-address deploy** (e.g., if a program's upgrade authority is compromised) requires full cascade with hours of downtime and no atomic guarantee.

## Recommendation

**SHORT-TERM (pre-mainnet):**
- Document the exact upgrade ordering and test it in a devnet dry-run.
- Add a protocol-wide pause flag (checked by Tax Program's swap entry points) to freeze operations during multi-program upgrades.

**MEDIUM-TERM:**
- Move cross-program ID references to a shared on-chain registry PDA, allowing runtime resolution instead of compile-time constants. This eliminates the rebuild cascade for new-address deployments.
- Add compile-time layout assertions (e.g., `static_assertions::const_assert_eq!` on struct sizes) between the Epoch Program's EpochState and the Tax Program's mirror.

**LONG-TERM:**
- Adopt the Anchor CPI crate pattern where programs import each other's types directly, giving compile-time guarantees on struct layout compatibility.
