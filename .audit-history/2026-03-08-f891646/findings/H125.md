# H125 — Tax Program: Compute Budget Exhaustion

**Status:** REJECTED (Not a vulnerability)
**Severity:** N/A
**Confidence:** HIGH
**Target:** Full swap path (Tax -> AMM -> Token-2022 -> Transfer Hook)

## Hypothesis

The full CPI chain (Tax -> AMM -> T22 -> Hook) may exhaust the default 200,000 CU compute budget, causing swap transactions to fail.

## Evidence

### Measured CU Consumption (Docs/Compute_Budget_Profile.md, Phase 32)

| CPI Path | Measured CU | % of 200k Default |
|----------|------------|-------------------|
| swap_sol_buy (CRIME/SOL) | 97,901 | 49% |
| swap_sol_buy (FRAUD/SOL) | 121,910 | 61% |
| swap_sol_sell (CRIME/SOL) | 98,585 | 49% |
| swap_sol_sell (FRAUD/SOL) | 122,586 | 61% |
| execute_carnage_atomic (depth-5) | 105,017 | 52.5% |

**Highest utilization is 61.3%** — well below the 80% WARNING threshold.

### CPI Depth Map

| Chain | Depth |
|-------|-------|
| User swap (Tax -> AMM -> T22 -> Hook) | 4 |
| Carnage (Epoch -> Tax -> AMM -> T22 -> Hook) | 5 (Solana max) |

Depth-5 carnage chain is at Solana's hard limit but CU consumption is moderate at 105k.

### Client-Side Compute Budget Usage

The client already sets explicit CU limits:
- `scripts/e2e/lib/swap-flow.ts`: `setComputeUnitLimit({ units: 200_000 })` for swaps
- `scripts/e2e/lib/carnage-flow.ts`: `setComputeUnitLimit({ units: 400_000 })` for carnage
- `app/lib/swap/multi-hop-builder.ts`: Strips per-step CU instructions and replaces with combined set
- `app/lib/staking/staking-builders.ts`: Explicit CU limits per staking operation

### On-Chain Programs

No on-chain programs request elevated compute budgets (confirmed via grep) — this is correct, as CU limits are a client-side concern set via `ComputeBudgetProgram`.

## Conclusion

All swap paths consume under 62% of the default 200k CU limit. The carnage path at depth-5 uses only 52.5%. Client code sets explicit CU limits. The 20% padding recommended in the compute budget profile (120k-150k depending on token) provides adequate headroom for production liquidity variance.

No compute exhaustion risk exists under current architecture.
