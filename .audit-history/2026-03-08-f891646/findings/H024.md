# H024 -- Bonding Curve Sell Tax u64 Precision

| Field            | Value                                                    |
|------------------|----------------------------------------------------------|
| **Hypothesis**   | Bonding curve sell tax uses u64 (not u128 like other calculations). For large values, multiplication overflow possible. |
| **Category**     | Arithmetic                                               |
| **Target**       | `programs/bonding_curve/src/instructions/sell.rs:174-179` |
| **Severity**     | Hypothesized HIGH                                        |
| **Verdict**      | **INVALID -- Not exploitable**                           |
| **Confidence**   | HIGH                                                     |

---

## 1. Hypothesis Restated

The bonding curve sell instruction computes tax using u64 arithmetic (`sol_gross.checked_mul(SELL_TAX_BPS)`), while every other tax calculation in the protocol uses u128 intermediates (e.g., `tax_math.rs`). If an attacker can produce a `sol_gross` large enough to overflow the u64 multiplication, `checked_mul` would return `None`, mapped to `CurveError::Overflow` -- but the hypothesis asks whether this could somehow wrap to a small value, yielding near-zero tax.

## 2. Attack Vector Analyzed

1. Purchase maximum tokens across one or many wallets.
2. Sell the largest possible token amount in a single transaction.
3. If `sol_gross * SELL_TAX_BPS` overflows u64, tax wraps to a small value, and the seller receives nearly all SOL gross with minimal tax.

## 3. Code Under Test

```rust
// sell.rs:174-179
let tax = sol_gross
    .checked_mul(SELL_TAX_BPS)     // u64 * u64
    .ok_or(CurveError::Overflow)?  // returns Err on overflow
    .checked_add(BPS_DENOMINATOR - 1) // ceil rounding
    .ok_or(CurveError::Overflow)?
    / BPS_DENOMINATOR;
```

Constants from `programs/bonding_curve/src/constants.rs`:
- `SELL_TAX_BPS: u64 = 1_500` (15%)
- `BPS_DENOMINATOR: u64 = 10_000`
- `TARGET_SOL: u64 = 1_000_000_000_000` (1000 SOL)
- `TARGET_TOKENS: u64 = 460_000_000_000_000` (460M tokens)
- `MAX_TOKENS_PER_WALLET: u64 = 20_000_000_000_000` (20M tokens)

## 4. Overflow Analysis

**u64 overflow threshold for `checked_mul`:**
- `u64::MAX / SELL_TAX_BPS = 18_446_744_073_709_551_615 / 1_500 = 12_297_829_382_473_034`
- This equals ~12,298 SOL in lamports.
- Overflow occurs only when `sol_gross > 12,297,829,382,473,034 lamports`.

**Maximum possible `sol_gross` from the curve:**
- The full curve integral (selling all 460M tokens from position 460M back to 0) yields ~1000.5 SOL (~1.0005e12 lamports).
- `TARGET_SOL = 1e12 lamports` (1000 SOL).
- No single sell can produce more SOL than the entire curve holds.

**Per-wallet maximum:**
- `MAX_TOKENS_PER_WALLET = 20M tokens` (20e12 base units).
- Even selling 20M tokens from the highest price point on the curve, `sol_gross` is a fraction of TARGET_SOL.

**Safety margin:**
- Maximum `sol_gross` (~1e12) vs overflow threshold (~1.23e16) = **12,298x safety margin**.
- The product `1e12 * 1500 = 1.5e15`, well within u64 (max ~1.8e19).

## 5. Wrapping vs Error Behavior

The hypothesis specifically asks whether overflow could "wrap to small value." It cannot:

- `checked_mul` returns `None` on overflow (no wrapping).
- `.ok_or(CurveError::Overflow)?` converts `None` to an error and aborts the transaction.
- Even if unchecked (wrapping) multiplication were used, the attacker could never produce a `sol_gross` large enough to trigger it, given the curve's economic bounds.

This is fundamentally different from an unchecked `*` operator, which would silently wrap. The code correctly uses checked arithmetic.

## 6. Design Inconsistency (Informational)

While not exploitable, this is a legitimate design inconsistency:

| Location | Tax Computation | Intermediate Width |
|----------|----------------|-------------------|
| `tax_math.rs:46-48` (AMM swaps) | `(amount as u128) * bps / 10000` | **u128** |
| `sell.rs:174-179` (bonding curve) | `sol_gross.checked_mul(SELL_TAX_BPS)` | **u64** |

The u128 pattern in `tax_math.rs` would never overflow for any u64 input with any valid BPS value. The u64 pattern in `sell.rs` relies on the economic bound (`sol_gross <= ~1000 SOL`) for safety rather than type-level guarantees. If `TARGET_SOL` were ever increased above ~12,298 SOL, the u64 multiplication would start rejecting legitimate sells with `CurveError::Overflow`.

## 7. Verdict

**INVALID.** The hypothesis is not exploitable for three independent reasons:

1. **checked_mul prevents wrapping**: Overflow returns an error, not a wrapped value. The transaction would abort, not produce zero tax.
2. **Economic bounds provide 12,298x safety margin**: Maximum `sol_gross` from the curve (~1e12 lamports) is far below the overflow threshold (~1.23e16 lamports).
3. **Curve supply is finite**: `tokens_sold` cannot exceed `TARGET_TOKENS` (460M), and selling all tokens yields at most ~1000 SOL gross.

The use of u64 instead of u128 is a minor design inconsistency (informational) but poses no security risk with current or any reasonable future parameters. The 13.5M+ proptest iterations on bonding curve math further validate that no overflow path exists within the curve's operational range.

## 8. Recommendation (Informational)

For defense-in-depth consistency with the rest of the codebase, consider upgrading to u128 intermediates:

```rust
let tax = (sol_gross as u128)
    .checked_mul(SELL_TAX_BPS as u128)
    .ok_or(CurveError::Overflow)?
    .checked_add(BPS_DENOMINATOR as u128 - 1)
    .ok_or(CurveError::Overflow)?
    / (BPS_DENOMINATOR as u128);
let tax = u64::try_from(tax).map_err(|_| CurveError::Overflow)?;
```

This eliminates the dependency on economic parameter bounds for arithmetic safety. Priority: low.
