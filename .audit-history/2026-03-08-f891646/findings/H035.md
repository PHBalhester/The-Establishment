# H035 -- Tax Split BPS Mismatch (Stale Comments)

**Severity:** INFORMATIONAL
**Status:** CONFIRMED (comment-only, no code bug)

## Summary

The hypothesis claims "comments say 75/24/1 but code uses 71/24/5." This is confirmed: two locations still reference the old 75/24/1 split while all on-chain code correctly implements 71/24/5. The code is correct; only comments are stale.

## Evidence

### On-Chain Constants (CORRECT -- 71/24/5)

`programs/tax-program/src/constants.rs:17-25`:
```rust
pub const STAKING_BPS: u128 = 7_100;   // 71%
pub const CARNAGE_BPS: u128 = 2_400;   // 24%
pub const TREASURY_BPS: u128 = 500;    // 5%  (remainder)
```

### On-Chain Split Logic (CORRECT -- 71/24/5)

`programs/tax-program/src/helpers/tax_math.rs:79-110` -- `split_distribution()` uses inline constants matching the above (STAKING_BPS=7100, CARNAGE_BPS=2400) and computes treasury as `total - staking - carnage` (remainder pattern).

### BPS Sum Verification

```
7100 + 2400 + 500 = 10000  (exactly 100%)
71% + 24% + 5% = 100%
```

The sum is correct. No dust leakage in percentage space.

### Treasury = Remainder Pattern (CORRECT)

```rust
let staking = floor(total * 7100 / 10000);
let carnage = floor(total * 2400 / 10000);
let treasury = total - staking - carnage;   // absorbs rounding dust
```

This guarantees `staking + carnage + treasury == total_tax` for all inputs. The invariant is enforced by construction and validated by 10K+ proptest iterations.

### Prior Audit Reference

## Verdict

**No on-chain vulnerability.** All deployed code uses the correct 71/24/5 split with proper u128 arithmetic, remainder-based treasury allocation, and micro-tax edge case handling. The only issue is two stale comments that reference the pre-change 75/24/1 ratios.

**Risk:** Stale comments could mislead future auditors or developers into thinking the split is wrong, or could cause confusion during mainnet deployment verification.

## Recommendation

Update the two stale locations:
1. `programs/tax-program/src/lib.rs:5-7` -- change to 71%/24%/5%
2. `scripts/e2e/lib/swap-flow.ts:112` -- change comment to "71/24/5 split"
