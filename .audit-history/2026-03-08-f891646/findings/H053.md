# H053 -- Flash-Stake: Deposit Large PROFIT Before Rewards, Claim After

| Field          | Value                                                                                    |
|----------------|------------------------------------------------------------------------------------------|
| **Hypothesis** | An attacker can deposit a large PROFIT position just before `deposit_rewards` / `update_cumulative`, then claim immediately after, capturing a disproportionate share of rewards. |
| **Severity**   | Informational                                                                            |
| **Status**     | **Not Vulnerable** -- Synthetix cumulative model + forfeiture + cooldown prevent this.   |
| **Target**     | `programs/staking/src/instructions/unstake.rs`, `deposit_rewards.rs`, `stake.rs`, `claim.rs`, `helpers/math.rs` |

---

## 1. Attack Scenario

A "flash-staker" or "mercenary capital" attacker attempts the following:

1. Observe that `deposit_rewards` has just been called (SOL added to escrow, `pending_rewards` counter incremented).
2. Stake a very large PROFIT position just BEFORE `update_cumulative` moves `pending_rewards` into `rewards_per_token_stored`.
3. After `update_cumulative` executes, the attacker's large share dilutes existing stakers and captures most of the newly distributed rewards.
4. Claim rewards, then unstake.

## 2. Defense Analysis

### Defense 1: Checkpoint Pattern (PRIMARY -- Prevents Past-Reward Capture)

`stake.rs:110` -- On first stake, the user's checkpoint is set to the CURRENT global cumulative:
```rust
user.rewards_per_token_paid = pool.rewards_per_token_stored;
```

`math.rs:39-41` -- `update_rewards()` computes pending based on the DELTA since last checkpoint:
```rust
let reward_delta = pool.rewards_per_token_stored
    .checked_sub(user.rewards_per_token_paid)
    .ok_or(StakingError::Underflow)?;
```

**Effect**: A new staker earns ZERO from any `rewards_per_token_stored` that existed before they staked. They only earn from future cumulative increases. This eliminates retroactive reward capture entirely.

### Defense 2: Two-Phase Reward Pipeline (STRUCTURAL)

Rewards flow through two distinct stages:

| Stage | Instruction | Caller | Effect |
|-------|------------|--------|--------|
| 1. Deposit | `deposit_rewards` | Tax Program CPI | Increments `pending_rewards` counter |
| 2. Distribute | `update_cumulative` | Epoch Program CPI | Moves `pending_rewards` into `rewards_per_token_stored` |

**Critical timing**: If an attacker stakes between Stage 1 and Stage 2, they WILL capture a proportional share of that epoch's rewards -- but only proportional to their share of `total_staked`. This is the correct Synthetix behavior: staking before distribution means you were "there" for that epoch.

### Defense 3: Forfeiture on Unstake (ANTI-HIT-AND-RUN)

`unstake.rs:160-168`:
```rust
let rewards_forfeited = user.rewards_earned;
if rewards_forfeited > 0 {
    pool.pending_rewards = pool.pending_rewards
        .checked_add(rewards_forfeited)...;
    user.rewards_earned = 0;
}
```

**Effect**: If the attacker stakes, waits for `update_cumulative`, and then unstakes -- all earned rewards are forfeited back to the pool and redistributed to remaining stakers. The attacker gets ZERO SOL yield from this path.

### Defense 4: Cooldown Gate (ANTI-CLAIM-EXIT)

`claim.rs:119`: Claims set `last_claim_ts = clock.unix_timestamp`.

`unstake.rs:125-133`: Unstake requires `elapsed >= COOLDOWN_SECONDS` (12 hours production, 2 seconds test).

**Effect**: An attacker who claims rewards must wait 12 hours before they can unstake their PROFIT. During this 12-hour window, the PROFIT is locked and exposed to any price movement, conversion vault drainage, or protocol risk -- eliminating the "risk-free" aspect of the attack.

### Defense 5: Dead Stake (PREVENTS TOTAL-POOL CAPTURE)

`initialize_stake_pool.rs:138`: Pool always has `MINIMUM_STAKE` (1 PROFIT = 1,000,000 base units) as dead stake that nobody can withdraw. This prevents an attacker from being the only staker and capturing 100% of rewards.

## 3. Remaining Attack Surface

The only viable "flash-stake" path is:

1. Stake large PROFIT before `update_cumulative`
2. Wait for cumulative update
3. Claim rewards (captures proportional share)
4. Wait 12-hour cooldown
5. Unstake

**This is not an exploit** -- it is the intended Synthetix staking behavior. The attacker's reward share is exactly proportional to their share of `total_staked` during the epoch update. The 12-hour lockup means the capital is genuinely at risk during that period.

**Economic analysis**: With COOLDOWN_SECONDS = 43,200 (12 hours) and epoch transitions every ~15 minutes, an attacker would need to lock capital for ~48 epoch transitions worth of time to capture 1 epoch's rewards. The opportunity cost far exceeds the proportional yield.

## 4. `update_rewards` Checkpoint Correctness

Verified that `update_rewards()` is called BEFORE balance changes in all paths:

| Instruction | Call Site | Correct? |
|------------|-----------|----------|
| `stake` | Line 118, before `staked_balance += amount` at line 121 | Yes |
| `unstake` | Line 156, before `staked_balance -= amount` at line 171 | Yes |
| `claim` | Line 92, before `rewards_earned = 0` at line 116 | Yes |

If `update_rewards` were called AFTER a balance change, the new balance would be used against the old checkpoint, producing incorrect rewards. All three paths correctly call it first.

## 5. Verdict

**Not Vulnerable.** The Synthetix cumulative reward-per-token model inherently prevents flash-stake attacks through checkpoint synchronization. The additional defenses (forfeiture on unstake, 12-hour cooldown after claim, dead stake) provide defense-in-depth that makes any mercenary capital strategy economically irrational.

No code changes recommended.
