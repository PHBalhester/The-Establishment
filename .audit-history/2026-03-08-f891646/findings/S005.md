# S005: Emergency Pause -- Program Upgrade Timing

**Determination:** CONFIRMED (no pause mechanism; upgrade is sole emergency response)
**Severity:** HIGH
**Confidence:** HIGH

## Hypothesis

Map the minimum steps and time for a single-program emergency upgrade. Consider cross-program cascade effects.

## Evidence

### No Pause Mechanism Exists

Architecture document Section 7.4 confirms: "Zero pause/freeze/emergency mechanisms across all 7 programs." The only emergency response is a full program upgrade.

### Minimum Steps for Single-Program Emergency Upgrade

Based on analysis of `scripts/deploy/deploy-all.sh` and `scripts/deploy/build.sh`:

| Step | Command | Time Estimate | Notes |
|------|---------|---------------|-------|
| 1 | Write fix in Rust source | 5-60 min | Depends on complexity |
| 2 | `anchor build -p <program>` | 30-90 sec | Single program compile |
| 3 | Feature rebuild (if devnet/mainnet) | 30-90 sec | `anchor build -p <prog> -- --features devnet` |
| 4 | `solana program deploy target/deploy/<prog>.so` | 10-30 sec | Single program deploy |
| 5 | Verify on-chain state | 10-30 sec | Manual check |

**Best case (single program, trivial fix): ~6 minutes**
**Realistic case (complex fix, testing): 30-120 minutes**

### Cross-Program Cascade Analysis

The protocol has tight cross-program dependencies that force multi-program rebuilds:

**Dependency chain:**
```
Tax Program depends on:
  - AMM (Cargo dependency for CPI)
  - Epoch Program (constants: program ID, PDA seeds)
  - Staking Program (constants: program ID, PDA seeds, discriminator)

Epoch Program depends on:
  - Tax Program (CPI for swap_exempt)
  - Staking Program (CPI for update_cumulative)
  - Switchboard (external)

Bonding Curve depends on:
  - Epoch Program (constants: program ID)
  - Token mints (compiled-in addresses)
```

**If you change a program's on-chain layout or program ID (redeployment to new address), every program referencing it must be rebuilt and redeployed.**

### Worst Case: Cross-Program Upgrade Cascade

From Architecture Section 9.8: "Fixing one program requires rebuilding all programs that reference its ID. Multi-step non-atomic upgrade creates inconsistency window."

If the AMM needs an emergency fix that changes its program ID:
1. Rebuild AMM (new keypair/address)
2. Rebuild Tax Program (references AMM program ID in `amm_program_id()`)
3. Rebuild Epoch Program (if it references AMM indirectly via Tax)
4. Deploy all 3 programs -- **non-atomically**

**Inconsistency window:** Between deploying the first and last program, cross-program CPI calls using the old program ID will fail. This means:
- All swaps fail (Tax -> AMM CPI broken)
- Epoch transitions may fail (Epoch -> Tax -> AMM chain broken)
- Tax distribution halts

Estimated inconsistency window: 1-5 minutes (deploy 3 programs sequentially).

### Non-Disruptive Single-Program Upgrade (Same Address)

If the program address stays the same (upgrade authority writes new bytecode to existing program account):
1. Build fix
2. `solana program deploy --program-id <existing-keypair> target/deploy/<prog>.so`
3. No cascade required -- other programs' hardcoded IDs still valid

**This is the normal case.** Solana program upgrades replace bytecode at the same address. Cross-program cascade only occurs if you need a fresh program address (extremely rare).

### Current Upgrade Authority Status

Architecture Section 6 Assumption A-7: "Programs are non-upgradeable at mainnet -- Not yet enforced -- devnet is upgradeable."

On devnet, all programs are upgradeable (standard for development). For mainnet, the decision to make programs immutable has not been finalized. If programs are made immutable, emergency upgrades become impossible.

### Initialization Race During Upgrade

Programs with "first caller wins" initialization (Transfer Hook, Epoch, Staking, Conversion Vault) have a subtle risk: if an upgrade changes the initialization PDA seeds or space, the existing PDA persists. The upgrade cannot re-initialize without adding migration logic.

## Recommendation

1. **Keep upgrade authority on mainnet** for at least the first 3-6 months. Immutability can be added later via authority burn.
2. **Add a lightweight pause flag** to EpochState and/or Tax Program. A single `paused: bool` checked at the top of `swap_sol_buy` and `swap_sol_sell` would halt all trading instantly without a full redeploy. Toggle via admin/upgrade-authority-gated instruction.
3. **Document the emergency runbook**: Exact commands, expected timing, who has authority keys. The current `deploy-all.sh` is for full fresh deploys, not surgical upgrades.
4. **Pre-build emergency patches**: For known risks (staking escrow drain, bounty rent issue), have pre-written patches ready to compile and deploy.

Estimated effort for pause mechanism: 2-4 hours per program. Affects Tax Program and Epoch Program (the two programs through which all value flows).
