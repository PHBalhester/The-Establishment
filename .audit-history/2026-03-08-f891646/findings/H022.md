# H022: Pool State Byte Offset Stability (A-5 Recheck)

**Status:** CONFIRMED SAFE (offsets correct; modification was comment-only)
**Severity:** INFORMATIONAL (no active vulnerability; latent risk persists)
**Category:** CPI, Arithmetic
**Confidence:** HIGH

## Summary

ARCHITECTURE.md flagged assumption A-5 as requiring re-verification because `pool.rs` was marked as MODIFIED since audit #1. Investigation reveals the modification (commit `7f6972c`) was **comment-only** -- it cleaned up stale PROFIT pool references in documentation comments. The `PoolState` struct definition has not changed since its last structural modification (commit `9c545a5`, which added the `locked` field).

The byte offsets [137..145] for `reserve_a` and [145..153] for `reserve_b` are **correct** across all three consumers:

1. `programs/tax-program/src/helpers/pool_reader.rs:46-55`
2. `programs/epoch-program/src/instructions/execute_carnage.rs:874-880`
3. `programs/epoch-program/src/instructions/execute_carnage_atomic.rs:941-947`

Wait -- the pool.rs comment says "8 + 224 = 232". Let me recount data bytes:
1 + 32 + 32 + 32 + 32 + 8 + 8 + 2 + 1 + 1 + 1 + 1 + 1 + 32 + 32 = **216**.

**Discrepancy found**: The inline comment in pool.rs (line 34) claims "8 + 224 = 232 bytes total (224 INIT_SPACE)" but the actual INIT_SPACE is **216 bytes** (total 224 with discriminator). This is a **documentation error** in the pool.rs comment, not a layout error. The Anchor `InitSpace` derive computes the correct value at compile time, so `init` allocations are correct regardless of the comment.

### Verification of PoolType Borsh Size

`PoolType` is a Borsh-serialized enum with two unit variants (`MixedPool`, `PureT22Pool`). Borsh encodes unit-variant enums as a single `u8` variant index. The `InitSpace` derive confirms this (the comment says 1 byte). This is critical because if PoolType serialized as more than 1 byte, all downstream offsets would shift.

## Residual Risks

### 1. Documentation Error in pool.rs (LOW)

The comment at line 30-34 claims INIT_SPACE is 224 and total is 232, but actual INIT_SPACE is 216 (total 224). This is purely a comment discrepancy -- the Anchor derive macro computes the correct value. However, if someone manually allocates space based on the comment rather than `InitSpace`, they would over-allocate by 8 bytes (harmless but wasteful).

### 2. Latent Layout Coupling (MEDIUM, same as H011)

The three consumers hardcode byte offsets with no compile-time link to the PoolState struct definition. If a field were added before `reserve_a` (e.g., between `vault_b` and `reserve_a`), all three consumers would silently read wrong data. This is the same structural risk documented in H011 for EpochState.

### 3. No Owner Check on Pool AccountInfo (HIGH, pre-existing)

As documented in the CPI context (04-cpi-external.md, risk observation #1), none of the `read_pool_reserves` callers verify that the pool AccountInfo is owned by the AMM program before reading bytes. This is independent of the offset correctness but amplifies the impact if offsets were ever wrong -- an attacker could provide a crafted account.

## Verdict

**A-5 is RE-VERIFIED: SAFE.** The pool.rs modification was comment-only. Byte offsets [137..145] and [145..153] correctly correspond to `reserve_a` and `reserve_b` in the current PoolState layout. No code change is required.

**Minor finding:** The INIT_SPACE comment in pool.rs is wrong (says 224, actual is 216). This should be corrected to prevent future confusion.

## Recommendations

### R1: Fix pool.rs INIT_SPACE Comment (Trivial)

Update the comment at `programs/amm/src/state/pool.rs:30-34` to reflect the actual sizes:
- INIT_SPACE = 216 bytes
- Total with discriminator = 224 bytes

### R2: Add Compile-Time Size Assertion (Low Cost, from H011)

Add to pool.rs:
```rust
const _: () = assert!(std::mem::size_of::<PoolState>() == 216);
```

Or use Anchor's INIT_SPACE:
```rust
const _: () = assert!(PoolState::INIT_SPACE == 216);
```

This catches any future field changes at compile time.

### R3: Extract Offset Constants (Medium Cost, from H011-R4)

Create shared constants for the byte offsets used by Tax and Epoch programs, ideally in a shared crate or as verified constants with compile-time assertions.
