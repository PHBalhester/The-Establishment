# H038 -- Carnage Atomic Execution Lock

**Auditor:** Hypothesis Investigator (sos-002)
**Target:** `programs/epoch-program/src/instructions/execute_carnage_atomic.rs`
**Supporting:** `consume_randomness.rs:270-287`, `execute_carnage.rs:210-226`, `expire_carnage.rs:63-71`, `constants.rs:138`
**Severity:** INFORMATIONAL
**Status:** SECURE (with caveats noted)

---

## Hypothesis

`execute_carnage_atomic` has a 50-slot lock window. Investigate whether TOCTOU or double-execution vulnerabilities exist in the lock flag set/check ordering, and verify that Solana runtime single-threaded-per-account prevents concurrent execution.

## Analysis

### 1. Lock Window Mechanism

The Carnage execution uses a tiered timing design:

- **Lock window (slots 0-50):** Only `execute_carnage_atomic` can execute. The fallback `execute_carnage` is blocked by `carnage_lock_slot`.
- **Fallback window (slots 50-300):** Both atomic and fallback paths can execute.
- **Expiry (after slot 300):** `expire_carnage` clears the pending state.

**Lock slot set** in `consume_randomness.rs:284-287`:
```rust
epoch_state.carnage_lock_slot = clock
    .slot
    .checked_add(CARNAGE_LOCK_SLOTS)  // +50
    .ok_or(EpochError::Overflow)?;
```

**Lock slot checked** in `execute_carnage.rs:223-226`:
```rust
require!(
    clock.slot > ctx.accounts.epoch_state.carnage_lock_slot,
    EpochError::CarnageLockActive
);
```

The fallback path (`execute_carnage`) also requires `carnage_pending == true` at the Anchor constraint level (line 55), plus `clock.slot <= carnage_deadline_slot` (line 215).

### 2. TOCTOU / Double Execution Analysis

**Question:** Can `execute_carnage_atomic` execute twice for the same Carnage event?

**Answer: No.** The `carnage_pending` flag provides the single-execution guarantee:

1. `execute_carnage_atomic` reads `carnage_pending` at line 215: `if !ctx.accounts.epoch_state.carnage_pending { return Ok(()); }`
2. At the end of successful execution, it clears the flag at line 465: `epoch_state.carnage_pending = false;`
3. Any subsequent call finds `carnage_pending == false` and no-ops.

This is NOT a TOCTOU vulnerability because:
- The read (line 215) and write (line 465) occur within a single Solana transaction.
- Solana's runtime enforces that **a single account can only be write-locked by one transaction at a time within a block**. Two transactions that both write to `epoch_state` cannot execute concurrently -- they are serialized.
- Within a single transaction, instructions execute sequentially. If `execute_carnage_atomic` is included twice in the same TX, the second invocation would see the already-cleared flag and no-op.

**Question:** Can both `execute_carnage_atomic` and `execute_carnage` (fallback) execute for the same event?

**Answer: No, for the same reason.** Both clear `carnage_pending = false`. Whichever executes first clears the flag. The fallback additionally has a hard Anchor constraint (`constraint = epoch_state.carnage_pending`) that would fail at deserialization time if the flag is already false.

### 3. Lock Check Ordering

The ordering is correct:

1. `consume_randomness` sets `carnage_pending = true`, `carnage_lock_slot`, and `carnage_deadline_slot` atomically in one TX.
2. `execute_carnage_atomic` is designed to be bundled in the **same transaction** as `consume_randomness` (as the next instruction). It reads the just-set `carnage_pending` flag. Because Anchor persists account writes between instructions in the same TX, the flag is visible.
3. `execute_carnage` (fallback) checks `clock.slot > carnage_lock_slot` -- during the lock window, this fails, correctly blocking the fallback.
4. `expire_carnage` checks `clock.slot > carnage_deadline_slot` -- during both lock and fallback windows, this fails.

### 4. Solana Runtime Concurrency Guarantees

Solana's Sealevel runtime provides the critical guarantee:

- **Write lock exclusivity:** When a transaction declares a writable account, it acquires an exclusive write lock. No other transaction in the same block can write to that account concurrently.
- `execute_carnage_atomic` declares `epoch_state` as `#[account(mut)]`, acquiring a write lock.
- This means two concurrent transactions both attempting to execute Carnage on the same `epoch_state` are impossible -- the runtime serializes them.

### 5. Atomic Path No-Op Design

The `execute_carnage_atomic` uses a **soft guard** (runtime `if` check returning `Ok(())`) rather than a hard Anchor constraint for `carnage_pending`. This is a deliberate design choice documented in comments: the instruction is meant to be bundled in every `consume_randomness` TX. When Carnage does not trigger, it no-ops gracefully (~100 CU). A hard constraint would cause the entire TX to fail when Carnage doesn't trigger, breaking the bundling pattern.

### 6. Pool Reserve Read Without Owner Check (Cross-reference)

The `read_pool_reserves` function (lines 930-956) reads pool data at raw byte offsets without verifying that the pool account is owned by the AMM program. This is already documented as a separate finding (ARCHITECTURE.md Section 7.6). For the atomic path specifically, this is lower risk because the atomic path is bundled in the same TX as `consume_randomness`, giving less opportunity for reserve manipulation. However, the concern stands for the fallback path.

## Verdict

**SECURE.** The Carnage lock mechanism is correctly implemented:

1. **No double execution:** `carnage_pending` flag acts as a one-shot latch, cleared on first successful execution. Solana write-lock exclusivity prevents concurrent execution.
2. **Correct lock ordering:** `consume_randomness` sets `carnage_lock_slot` before `execute_carnage_atomic` can run (same TX, sequential instruction ordering).
3. **No TOCTOU:** Read and write of `carnage_pending` occur within a single atomic transaction. Cross-TX races are prevented by Solana's write-lock serialization.
4. **Tiered window design is sound:** Atomic (0-50 slots) -> Fallback (50-300 slots) -> Expiry (300+ slots) with each tier correctly gated.

## Caveats (Informational)

1. **`force_carnage` (devnet-only) does not set `carnage_lock_slot`** (documented in state-machine context, finding #7). This means the fallback path is immediately available after `force_carnage`, bypassing the atomic-only window. This is devnet-only (`#[cfg(feature = "devnet")]`) and has no mainnet impact, but could mask lock-timing bugs during testing.

2. **Pool reserve read lacks AMM owner check** in `read_pool_reserves` (line 930-956). The pool `AccountInfo` is passed as an unchecked `#[account(mut)]` at lines 122-123. A malicious caller could pass a crafted account with fake reserves to manipulate the 85% slippage floor. This is mitigated by: (a) the CPI to `Tax::swap_exempt` validates the pool, so the actual swap uses the real pool; (b) if the slippage floor is set too low via fake reserves, the actual swap output still must pass. However, if fake reserves make the floor appear satisfied when real reserves would not, the swap proceeds and could yield fewer tokens than expected. **This is a pre-existing concern (ARCHITECTURE.md 7.6), not specific to the lock mechanism.**
