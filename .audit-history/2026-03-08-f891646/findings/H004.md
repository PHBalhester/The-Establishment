# H004 — Bonding Curve Authority: start_curve Premature Activation

**Status:** NOT VULNERABLE
**Severity:** INFORMATIONAL
**Category:** Access Control, Timing
**Confidence:** HIGH

## Summary

The hypothesis that an attacker could call `start_curve` on an unfunded curve to enable purchases at incorrect prices is **not viable**. The instruction contains an explicit funding guard (`token_vault.amount >= TARGET_TOKENS`) that rejects activation unless the vault holds the full 460M tokens. The separate issue that ANY signer can call `start_curve` (once the vault IS funded) is a real access control gap, but it does not enable pricing manipulation -- the curve math is identical regardless of who activates it.

## Evidence

**State guard (status):** `programs/bonding_curve/src/instructions/start_curve.rs:25`
```rust
constraint = curve_state.status == CurveStatus::Initialized @ CurveError::InvalidStatus,
```
The curve must be in `Initialized` status. This is set only by `initialize_curve` and cannot be reverted to from any other state.

**Funding guard:** `programs/bonding_curve/src/instructions/start_curve.rs:56-59`
```rust
require!(
    ctx.accounts.token_vault.amount >= TARGET_TOKENS,
    CurveError::CurveNotFunded
);
```
`TARGET_TOKENS = 460_000_000_000_000` (460M with 6 decimals) at `constants.rs:37`.

**Vault authority:** `start_curve.rs:32-33`
```rust
token::authority = curve_state,
```
The token vault's authority is the CurveState PDA itself, meaning only the bonding curve program can move tokens out via PDA signing. An attacker cannot partially drain the vault to manipulate pricing.

**fund_curve transfers exact amount:** `fund_curve.rs:80`
```rust
let transfer_amount = TARGET_TOKENS;
```
The funding instruction transfers exactly `TARGET_TOKENS`. Combined with the `>= TARGET_TOKENS` check in `start_curve`, activation requires a fully funded vault.

## Attack Scenario

The hypothesized attack does NOT work:

1. Attacker calls `start_curve` on an `Initialized` curve before `fund_curve` is called.
2. `token_vault.amount` is 0 (vault created empty in `initialize_curve`).
3. `require!(0 >= 460_000_000_000_000)` fails with `CurveError::CurveNotFunded`.
4. Transaction reverts. No state change occurs.

Even a partially-funded vault (e.g., someone sends tokens directly to the vault PDA without using `fund_curve`) would fail unless the exact threshold is met.

## Root Cause

No vulnerability exists for the specific premature activation scenario. The `TARGET_TOKENS` guard is a sufficient defense against unfunded curve activation.

The broader issue -- that `authority` is a bare `Signer<'info>` with no identity verification -- means anyone can call `start_curve` once the vault IS funded. However, this produces identical on-chain behavior to the deployer calling it: same `start_slot`, same `deadline_slot`, same curve math. The timing of activation (immediately after funding vs. some delay) is the only variable, and the attacker gains no pricing advantage.

## Recommendation

No specific fix needed for premature activation. The `TARGET_TOKENS` guard is correct and sufficient.

The broader bonding curve authority gap (ANY signer accepted across 6 instructions) is tracked separately and is the real concern -- particularly for `withdraw_graduated_sol` and `prepare_transition` where the authority identity determines who receives funds or controls graduation timing.
