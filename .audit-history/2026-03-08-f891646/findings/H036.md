# H036 -- Init Front-Running: General (Beyond Hook + Epoch)

**Severity:** HIGH (composite; individual ratings below)
**Category:** Access Control / Deployment Security
**Status:** CONFIRMED
**Programs:** Staking, Epoch (Carnage Fund), Conversion Vault, Tax Program (WSOL Intermediary)
**Related:** H007 (Transfer Hook init), H021 (Epoch State init), S005 (Hook authority ransom)

---

## Summary

Five programs use open initialization (any signer). H007 and H021 cover the Transfer Hook and Epoch State cases respectively. This finding covers the remaining four: **Staking init**, **Carnage Fund init**, **Vault Config init**, and **WSOL Intermediary init**. Two of these accept arbitrary mint parameters with no on-chain validation, creating a protocol-bricking vector if front-run.

---

## Detailed Analysis

### 1. Staking `initialize_stake_pool` -- MEDIUM-HIGH

**File:** `programs/staking/src/instructions/initialize_stake_pool.rs:27-90`

**Access control:** `authority: Signer<'info>` -- any signer, no ProgramData check.

**Re-init protection:** Anchor `init` constraint on PDA `seeds = ["stake_pool"]`. One-shot.

**Arbitrary parameters:** YES -- `profit_mint: InterfaceAccount<'info, Mint>` is unconstrained. The attacker can pass any Token-2022 mint as "PROFIT". The init handler:
1. Creates `stake_vault` as a token account for the attacker-chosen mint (line 64: `token::mint = profit_mint`)
2. Transfers `MINIMUM_STAKE` (1 token) from `authority_token_account` as dead stake
3. Stores vault addresses in `StakePool` PDA

**Impact:** If front-run with a fake mint:
- The stake_vault PDA is permanently bound to the wrong mint
- `stake` instruction validates `token::mint = profit_mint` against the vault, so users would need to stake the attacker's fake token, not PROFIT
- `deposit_rewards` (SOL deposits from Tax Program) still works (SOL-based, no mint check), but rewards accumulate against zero real stakers
- **Protocol staking is permanently bricked.** No re-initialization possible (PDA already exists).
- Requires full program redeployment to fix.

**Attacker cost:** ~0.003 SOL (rent for StakePool + stake_vault + escrow_vault) + 1 token of any T22 mint.

### 2. Carnage Fund `initialize_carnage_fund` -- MEDIUM

**File:** `programs/epoch-program/src/instructions/initialize_carnage_fund.rs:69-131`

**Access control:** `authority: Signer<'info>` -- any signer, no ProgramData check.

**Re-init protection:** Anchor `init` on PDA `seeds = ["carnage_fund"]` + defense-in-depth `!carnage_state.initialized` check.

**Arbitrary parameters:** YES -- `crime_mint` and `fraud_mint` are unconstrained `InterfaceAccount<'info, Mint>`. The handler creates:
- `crime_vault` at PDA `seeds = ["carnage_crime_vault"]` with `token::mint = crime_mint` (line 102)
- `fraud_vault` at PDA `seeds = ["carnage_fraud_vault"]` with `token::mint = fraud_mint` (line 114)

**Impact:** If front-run with fake mints:
- Carnage token vaults are bound to wrong mints
- `execute_carnage_atomic` and `execute_carnage` attempt to buy/burn tokens into these vaults via CPI
- The CPI swap would fail because the AMM pools use the real mints, creating a mint mismatch
- **Carnage Fund is permanently bricked** -- 24% of all swap taxes accumulate in the SOL vault but can never be spent on buyback-and-burn
- SOL accumulates indefinitely in carnage_sol_vault with no way to use it
- The `expire_carnage` fallback returns SOL to the carnage vault, not to users

**Attacker cost:** ~0.005 SOL (rent for CarnageFundState + 2 token vaults).

### 3. Conversion Vault `initialize` -- LOW

**File:** `programs/conversion-vault/src/instructions/initialize.rs:8-99`

**Access control:** `payer: Signer<'info>` -- any signer, no ProgramData check.

**Re-init protection:** Anchor `init` on PDA `seeds = ["vault_config"]`. One-shot.

**Arbitrary parameters:** NO (in production builds). The instruction has hardcoded mint validation:
```rust
constraint = cfg!(feature = "localnet") || crime_mint.key() == constants::crime_mint()
constraint = cfg!(feature = "localnet") || fraud_mint.key() == constants::fraud_mint()
constraint = cfg!(feature = "localnet") || profit_mint.key() == constants::profit_mint()
```

In non-localnet builds, the mints MUST match the hardcoded addresses. The attacker can only become the payer (losing rent SOL) but cannot influence any stored state beyond the bump byte.

**Impact:** Attacker pays ~0.004 SOL in rent. No protocol damage. The VaultConfig stores only a bump (production) or bump + mints (localnet). No authority stored post-init.

**Caveat:** If a localnet build were accidentally deployed to mainnet, the mint constraints are bypassed. This overlaps with the mainnet placeholder concern (A-9 in ARCHITECTURE.md).

### 4. WSOL Intermediary `initialize_wsol_intermediary` -- LOW

**File:** `programs/tax-program/src/instructions/initialize_wsol_intermediary.rs:31-127`

**Access control:** `admin: Signer<'info>` -- any signer despite the name, no ProgramData check.

**Re-init protection:** Implicit -- uses `system_instruction::create_account` which fails if the PDA already has lamports/data. Solana runtime rejects account creation at an existing address.

**Arbitrary parameters:** PARTIALLY -- the `mint` account is passed as `AccountInfo` with `/// CHECK: Must be native mint. Validated by SPL Token InitializeAccount`. SPL Token's `InitializeAccount3` instruction validates the mint is a real SPL token mint but does NOT enforce it is NATIVE_MINT specifically. However:
- The `swap_authority` PDA is deterministically derived (seeds = ["swap_authority"]), so the token account owner is correct regardless
- The `wsol_intermediary` PDA is deterministically derived (seeds = ["wsol_intermediary"]), so the address is fixed
- If initialized with a non-WSOL mint, the sell flow's `SyncNative` + close-and-distribute pattern would fail because `SyncNative` only works on native mint accounts

**Impact:** If front-run with wrong mint:
- The intermediary token account is created for a non-WSOL mint at the expected PDA
- `swap_sol_sell` calls `SyncNative` on it, which fails (not a native mint account)
- **All sell operations are bricked** until program redeployment
- Buy operations are unaffected (different code path)

**Revised severity for wrong-mint front-run:** MEDIUM (sells bricked, buys work).
**Practical severity:** LOW -- an attacker would need to pass a non-NATIVE_MINT, and SPL Token's InitializeAccount3 would succeed for any valid mint. However, the PDA address and owner are fixed, so the only variable is the mint. If the attacker passes NATIVE_MINT (the obvious choice since it's the only valid option that doesn't brick things), the init succeeds identically to legitimate initialization. The attacker gains nothing except paying rent.

## Root Cause

The AMM's `initialize_admin` instruction demonstrates the correct pattern: require ProgramData account and verify `program_data.upgrade_authority_address == Some(authority.key())`. This pattern was not replicated in any of the other 6 programs (Staking, Epoch, Tax, Conversion Vault, Transfer Hook, Bonding Curve).

For Staking and Carnage Fund specifically, the additional issue is that mint parameters are accepted without on-chain validation against hardcoded addresses (unlike Conversion Vault which does validate).

---

## Recommended Fixes

### Priority 1: Add mint validation to Staking init and Carnage Fund init

For `initialize_stake_pool`, add a constraint like Conversion Vault:
```rust
#[account(
    constraint = profit_mint.key() == constants::profit_mint() @ StakingError::InvalidMint,
)]
pub profit_mint: InterfaceAccount<'info, Mint>,
```

For `initialize_carnage_fund`, add:
```rust
#[account(
    constraint = crime_mint.key() == constants::crime_mint() @ EpochError::InvalidMint,
)]
pub crime_mint: InterfaceAccount<'info, Mint>,
#[account(
    constraint = fraud_mint.key() == constants::fraud_mint() @ EpochError::InvalidMint,
)]
pub fraud_mint: InterfaceAccount<'info, Mint>,
```

This does NOT fix the front-running itself but ensures that even a front-runner must use the correct mints, neutralizing the protocol-bricking vector.

### Priority 2: Add ProgramData upgrade-authority gating to all init instructions

Replicate the AMM pattern across all programs. This is the comprehensive fix that prevents front-running entirely.

### Priority 3 (Operational): Bundle all init instructions in a single transaction

If init instructions are sent in separate transactions, there is a mempool observation window between each. Bundling all inits into one atomic transaction eliminates the front-running window. This is a deployment-time mitigation, not an on-chain fix.

### Priority 4: Validate WSOL intermediary mint

Add a check that `mint.key() == spl_token::native_mint::id()` in the WSOL intermediary init.
