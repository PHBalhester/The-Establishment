# H057: Epoch Constants Tuning (H119 Recheck)

**Hypothesis:** Epoch timing constants may be miscalibrated for mainnet, creating liveness risks or economic vulnerabilities.
**Determination:** CONFIRMED LOW — Constants are reasonable but bounty-vs-rent gap remains unpatched.
**Severity:** LOW
**Confidence:** HIGH
**Target:** `programs/epoch-program/src/constants.rs`

---

## Summary

The epoch program's timing and economic constants were reviewed for mainnet readiness. All timing relationships are internally consistent: VRF timeout fits within an epoch, carnage deadlines are proportioned correctly, and slippage floors match their documented intent. The one persistent issue (carried from audit #1 as a known TODO) is that `trigger_epoch_transition` checks `vault_balance >= TRIGGER_BOUNTY_LAMPORTS` without accounting for the rent-exempt minimum of the `carnage_sol_vault` SystemAccount PDA, meaning a bounty transfer could theoretically reduce the vault below rent-exempt and destroy the PDA.

---

## Evidence

### 1. Timing Constant Relationships (REASONABLE)

| Constant | Devnet | Mainnet | Real Time (mainnet, ~400ms/slot) |
|----------|--------|---------|----------------------------------|
| `SLOTS_PER_EPOCH` | 750 | 4,500 | ~30 minutes |
| `VRF_TIMEOUT_SLOTS` | 300 | 300 | ~2 minutes |
| `CARNAGE_DEADLINE_SLOTS` | 300 | 300 |  ~2 minutes |
| `CARNAGE_LOCK_SLOTS` | 50 | 50 | ~20 seconds |

**Key relationships verified:**

- **VRF_TIMEOUT_SLOTS (300) < SLOTS_PER_EPOCH (4,500)**: The VRF timeout is 6.7% of an epoch. A failed VRF can be retried ~15 times within a single epoch window. This is generous and appropriate -- an oracle failure will not skip multiple epochs.

- **CARNAGE_LOCK_SLOTS (50) < CARNAGE_DEADLINE_SLOTS (300)**: The atomic-only lock window is 16.7% of the total carnage deadline. The fallback window (slots 50-300) is 250 slots (~100 seconds), giving ample time for multiple retry attempts. Proportionally sound.

- **VRF_TIMEOUT_SLOTS == CARNAGE_DEADLINE_SLOTS (both 300)**: These serve different purposes (VRF retry vs carnage expiry) but happen to share the same value. No conflict -- VRF timeout gates `retry_epoch_vrf` while carnage deadline gates `expire_carnage`. They operate on independent state fields (`vrf_request_slot` vs `carnage_trigger_slot`).

- **Devnet SLOTS_PER_EPOCH (750) vs VRF_TIMEOUT_SLOTS (300)**: On devnet, VRF timeout is 40% of an epoch. This is tighter but still allows 2+ retries per epoch. Acceptable for testing.

### 2. Bounty Constant (KNOWN ISSUE)

**File:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:194-227`

```rust
let vault_balance = ctx.accounts.carnage_sol_vault.lamports();
let bounty_paid = if vault_balance >= TRIGGER_BOUNTY_LAMPORTS {
    // Transfer bounty from carnage_sol_vault PDA to triggerer
    ...
    TRIGGER_BOUNTY_LAMPORTS
} else {
    // Skip bounty if insufficient
    0
};
```

`TRIGGER_BOUNTY_LAMPORTS = 1,000,000` (0.001 SOL). The check `vault_balance >= TRIGGER_BOUNTY_LAMPORTS` does not account for the rent-exempt minimum (~890,880 lamports for a 0-byte SystemAccount). If the vault holds exactly 1,890,880 lamports (rent + bounty), the transfer succeeds but leaves exactly rent-exempt. If the vault holds 1,500,000 lamports (above bounty but below rent+bounty), the transfer would reduce the vault to 500,000 lamports -- below rent-exempt.

**Contrast with carnage execution**, which correctly handles this:

**File:** `programs/epoch-program/src/instructions/execute_carnage.rs:359-363`
```rust
let rent_exempt_min = rent.minimum_balance(0);
let available_sol = sol_balance.saturating_sub(rent_exempt_min);
```

The carnage swap logic properly subtracts rent-exempt before calculating available SOL. The bounty logic does not.

**Practical impact:** The crank auto-tops-up when below a danger zone (per project MEMORY), so this is operationally mitigated. However, without the crank, a series of bounty payouts on an underfunded vault could destroy the PDA.

### 3. Carnage Slippage Constants (REASONABLE)

| Constant | Value | Purpose |
|----------|-------|---------|
| `CARNAGE_SLIPPAGE_BPS_ATOMIC` | 8500 (85%) | Tight floor for same-TX atomic execution |
| `CARNAGE_SLIPPAGE_BPS_FALLBACK` | 7500 (75%) | Looser floor for recovery/fallback execution |

The 85%/75% split is sensible: the atomic path has MEV protection from transaction atomicity + VRF unpredictability, so a tighter floor is appropriate. The fallback path (exposed to front-running after the lock window) uses a more lenient floor to prioritize execution. The 10-point spread between the two is small enough that switching from atomic to fallback does not create a significant MEV extraction opportunity.

### 4. VRF Probability Constants (REASONABLE)

| Constant | Value | Probability | Purpose |
|----------|-------|-------------|---------|
| `CARNAGE_TRIGGER_THRESHOLD` | 11 | ~4.3% (11/256) | Carnage trigger per epoch |
| `CARNAGE_SELL_THRESHOLD` | 5 | ~2.0% (5/256) | Sell action vs burn |
| `MAX_CARNAGE_SWAP_LAMPORTS` | 1,000 SOL | N/A | Per-swap cap |

At mainnet's ~30-minute epochs, carnage triggers on average once every ~11.6 hours (48 epochs/day * 4.3%). The 1,000 SOL cap prevents single carnage events from moving thin pools excessively. These are game-design parameters, not security constants -- they are reasonable for their intended purpose.

### 5. Genesis Tax Rates (REASONABLE)

`GENESIS_LOW_TAX_BPS = 300` (3%) and `GENESIS_HIGH_TAX_BPS = 1400` (14%). These are within the valid range of the VRF-derived rate tables (LOW_RATES: 100-400 BPS, HIGH_RATES: 1100-1400 BPS). Genesis rates are used only for the first epoch before VRF derives new rates, so they have minimal lasting economic impact.

### 6. Cross-Program Seed Consistency (VERIFIED)

Both epoch-program and tax-program define matching seeds:
- `CARNAGE_SOL_VAULT_SEED = b"carnage_sol_vault"` -- matches in both
- `EPOCH_STATE_SEED = b"epoch_state"` -- matches in both
- `CARNAGE_SIGNER_SEED = b"carnage_signer"` -- matches in both

No discrepancies found. However, these are string-matched at compile time with no compile-time cross-crate verification (flagged separately in architecture as A-3).

## Impact

**Timing constants**: No security impact. All timing relationships are well-proportioned for mainnet slot times. VRF timeout provides adequate retry windows. Carnage deadlines leave sufficient fallback room.

**Bounty rent-exempt gap**: LOW severity. The `system_instruction::transfer` will fail at the Solana runtime level if it would reduce the account below rent-exempt minimum (runtime enforces this for SystemAccounts). The real risk is not PDA destruction but bounty non-payment: the check `vault_balance >= TRIGGER_BOUNTY_LAMPORTS` passes, the transfer instruction fails, and the entire `trigger_epoch_transition` reverts. This creates a narrow band where the vault has enough lamports for the bounty check but not enough for the actual transfer (between ~890,880 and ~1,890,880 lamports). In this band, epoch transitions silently fail until more SOL enters the vault.

**Correction to prior analysis**: The Solana runtime prevents actual PDA destruction via rent-exempt enforcement on `system_instruction::transfer`. The real concern is transaction reversion in the narrow band described above, not PDA loss.

---

## Recommendation

1. **Fix bounty check to account for rent-exempt minimum** (LOW priority):
```rust
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(0);
let available_for_bounty = vault_balance.saturating_sub(rent_exempt_min);
let bounty_paid = if available_for_bounty >= TRIGGER_BOUNTY_LAMPORTS {
    // ... transfer
} else {
    0
};
```
This eliminates the narrow revert band and makes the skip behavior consistent with intent.

2. **No changes needed for timing constants**. All values are well-calibrated for mainnet.

3. **Note that VRF_TIMEOUT_SLOTS and CARNAGE_DEADLINE_SLOTS are not feature-gated** -- they use the same value (300) on both devnet and mainnet. This is acceptable because slot timing (~400ms) is similar across networks, but worth documenting as an explicit design choice.

## Conclusion

Epoch constants are well-tuned for mainnet. Timing relationships are internally consistent and proportionally sound. The only actionable finding is the minor bounty rent-exempt gap in `trigger_epoch_transition`, which is operationally mitigated by the crank top-up but should be fixed to match the pattern already used in `execute_carnage`. This is a LOW severity finding consistent with the H119 recheck verdict of "NO NEW CONCERNS" from the architecture document.
