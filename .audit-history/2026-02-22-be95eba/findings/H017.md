# H017: AMM Pool Reentrancy via Hook Callback

**Category:** CPI / Reentrancy
**Severity:** NOT VULNERABLE
**Status:** CLEAR
**Date:** 2026-02-22
**Auditor:** STR-017 Investigator (Stronghold of Security)

---

## HYPOTHESIS

The AMM swap instruction triggers Token-2022 `transfer_checked`, which invokes the transfer hook program. If the hook (or a malicious hook on a different mint) could call back into the AMM, the reentrancy guard (`pool.locked` flag) might not protect against cross-instruction reentrancy if the flag is set AFTER the CPI that triggers the hook.

**Attack Vector:** Craft a scenario where the hook callback re-enters `AMM::swap` before the first swap completes. Success would depend on:
1. The `pool.locked` flag NOT being set before the CPI
2. Token-2022 allowing arbitrary program invocation during hook execution
3. Solana runtime allowing same-program re-entry during CPI

**Impact:** CRITICAL if exploitable — Double-spend on pool reserves via reentrancy

### Account Constraint Enforcement

**File:** `swap_sol_pool.rs` line 381

```rust
#[account(
    mut,
    seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
    bump = pool.bump,
    constraint = pool.initialized @ AmmError::PoolNotInitialized,
    constraint = !pool.locked @ AmmError::PoolLocked,  // ← Checked at deserialization
)]
pub pool: Account<'info, PoolState>,
```

The `constraint = !pool.locked` check happens during **account deserialization** (before the handler executes). Any re-entrant call would fail this check because the lock is already set.

### Transfer Hook Cannot Re-enter AMM

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/transfer_hook.rs`

The transfer hook program is **read-only** and **CPI-free**:

1. **Line 78-113:** The `handler` function only performs validation:
   - Zero-amount check (line 80)
   - Mint owner check (line 86)
   - Transferring flag check (line 90)
   - Whitelist validation (lines 96-109)

2. **No CPI calls:** Grep for `invoke|CpiContext` in the transfer hook instruction shows zero CPI calls in the `transfer_hook.rs` handler. The only CPI in the entire hook program is in `initialize_extra_account_meta_list.rs` (initialization only, not executed during transfers).

3. **Direct invocation prevented:** Line 90 checks `check_is_transferring()` which validates the Token-2022 runtime set the `transferring` flag. Direct calls would fail this check.

### Solana Runtime Reentrancy Prevention

Per Solana's runtime design (documented in [Solana Runtime Spec](https://docs.solana.com/implemented-proposals/cross-program-invocations)):

> **A program cannot be invoked recursively.** During a CPI, if the target program's ID matches the calling program's ID, the transaction fails with an error.

**What this means:**
- If the AMM calls Token-2022 (depth 1)
- Token-2022 calls Transfer Hook (depth 2)
- Transfer Hook **cannot** call back into AMM (would be depth 3 with same program ID as depth 1)
- The runtime enforces this at the SVM level — it's not a contract-level check

This is distinct from the lock flag (which is application-level defense-in-depth). The runtime provides a hard guarantee.

## ATTACK SCENARIOS EVALUATED

### Scenario 1: Malicious Hook on Protocol Tokens

**Attack:** Deploy a malicious transfer hook for CRIME/FRAUD/PROFIT that tries to re-enter AMM.

**Defense:**
1. **Impossible:** The protocol mints are already deployed with the legitimate hook program address baked into the mint's TransferHook extension. This cannot be changed.
2. **Token-2022 validates:** During `transfer_checked`, Token-2022 reads the hook program ID from the mint extension and only invokes that specific program.
3. **ExtraAccountMetaList binding:** The hook accounts are resolved from the ExtraAccountMetaList PDA, which is derived from the mint and owned by the hook program.

**Result:** NOT EXPLOITABLE — Cannot substitute a different hook for protocol tokens.

### Scenario 2: User Provides Malicious Token

**Attack:** User creates a malicious Token-2022 mint with a custom hook that attempts AMM re-entry, then tries to swap it in a pool.

**Defense:**
1. **Pool initialization:** Pools are initialized by the admin with specific mint addresses. Per `swap_sol_pool.rs` line 401-405:
   ```rust
   #[account(constraint = mint_a.key() == pool.mint_a @ AmmError::InvalidMint)]
   pub mint_a: InterfaceAccount<'info, Mint>,
   ```
   The mint addresses are validated against the pool's stored `mint_a` and `mint_b`.

2. **No user-controlled mints:** Users cannot substitute arbitrary mints — the pool determines which mints are valid.

**Result:** NOT EXPLOITABLE — User cannot introduce arbitrary tokens into the swap.

### Scenario 3: Cross-Instruction Reentrancy (Same Transaction)

**Attack:** User includes two `swap` instructions in the same transaction, attempting to re-enter while the first swap is in progress.

**Defense:**
1. **Anchor constraint:** `constraint = !pool.locked` at line 381 is evaluated **at account deserialization time** (before the instruction handler runs).
2. **Lock persistence:** The lock is set in the first swap (line 84) and only cleared at the end (line 322). If a second swap instruction tries to deserialize the pool account while locked, the constraint check fails and the transaction is rejected.

**Result:** NOT EXPLOITABLE — Anchor constraints prevent cross-instruction reentrancy in the same transaction.

### Scenario 4: Reentrancy via Different Pool

**Attack:** Attacker creates two pools sharing a vault, tries to re-enter the AMM via a different pool while the first is locked.

**Defense:**
1. **PDA vault isolation:** Pool vaults are PDAs derived from `[POOL_SEED, mint_a, mint_b, bump]`. Each pool has unique vaults.
2. **Vault validation:** Line 389-397 validates `vault_a.key() == pool.vault_a` and `vault_b.key() == pool.vault_b`. Vaults are hardcoded in pool state during initialization.

**Result:** NOT EXPLOITABLE — Cannot share vaults across pools.

## CONCLUSION

**STATUS: NOT VULNERABLE**

The AMM pool reentrancy hypothesis is **REFUTED** by multiple layers of defense:

1. **Application-level lock:** `pool.locked` flag is set BEFORE any CPI and cleared AFTER all CPIs.
2. **CEI ordering:** State updates (reserves) happen BEFORE interactions (token transfers).
3. **Anchor constraints:** `!pool.locked` check at deserialization prevents cross-instruction reentrancy.
4. **Hook isolation:** The transfer hook program is read-only and makes no CPI calls.
5. **Runtime enforcement:** Solana's SVM prevents same-program recursive invocation.
6. **Account validation:** Mints and vaults are validated against pool state, preventing substitution attacks.

**Impact Assessment:** No exploitable reentrancy vector exists. The AMM follows secure CEI patterns with defense-in-depth.
