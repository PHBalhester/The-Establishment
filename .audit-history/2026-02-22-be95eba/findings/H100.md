# H100: Staking Checkpoint Correctly Uses Clock::slot

**Strategy:** STR-100: Staking Checkpoint Uses Wrong Clock Field

**Determination:** HYPOTHESIS REJECTED — Staking checkpoint implementation is correct.

**Severity:** N/A (No Vulnerability)

**Confidence:** 100% (Code audit with full trace)

---

## Summary

The hypothesis proposed that staking checkpoint state might use `Clock::unix_timestamp` instead of `Clock::slot`, which would break comparisons with other slot-based values and create timing inconsistencies. Full code audit confirms that the staking program correctly uses `Clock::slot` exclusively for all checkpoint and timing operations. No vulnerability exists.

### 6. Checkpoint Pattern Architecture

Per **08-timing-ordering.md Section 6.1 & 6.3:**
- The checkpoint pattern uses `rewards_per_token_paid` (a cumulative u128), not slot numbers
- Slot numbers (`first_stake_slot`, `last_update_slot`) are stored but only used for informational/audit purposes
- The actual reward calculation in `update_rewards()` uses:
  - `reward_delta = pool.rewards_per_token_stored - user.rewards_per_token_paid` (u128 arithmetic)
  - `pending = balance * reward_delta / PRECISION` (u128 with protection against overflow)
- No timing window comparisons occur on slot numbers in the checkpoint logic itself

### 7. Verification: Cumulative Snapshot Does Not Depend on Slot

**math.rs lines 36-67:**

The `update_rewards()` function:
1. Reads current pool state
2. Calculates reward delta using only u128 cumulative values
3. Updates user checkpoint to current cumulative (not timestamp)
4. Sets `last_update_slot` for audit trail only

The checkpoint (`rewards_per_token_paid`) is a snapshot of cumulative rewards, not a time-based reference point. Flash-loan protection comes from the fact that the cumulative only advances at epoch boundaries (controlled by Epoch Program CPI), not from time-based enforcement.

## Impact

No impact — this is not a vulnerability. The protocol correctly separates concerns:
- **Epoch boundaries:** Determined by slot number (epoch = (current_slot - genesis_slot) / SLOTS_PER_EPOCH)
- **Reward snapshots:** Stored as cumulative u128 values (rewards_per_token_paid, rewards_per_token_stored)
- **Audit trail:** Last interaction slot stored for observability

All timing-critical operations use `Clock::slot`. No timestamp-based logic exists in the staking module.

---

## Recommendation

No action required. The hypothesis was based on a valid concern pattern (slot vs. timestamp confusion) but does not apply to this codebase. The staking implementation follows Solana best practices per SP-053 and solana-runtime-quirks.md: slot-only timing for ordering-critical logic.

**Documentation note:** Consider adding a comment in the staking state struct to explicitly document that checkpoint uses cumulative rewards, not time, for flash-loan resistance:

```rust
pub struct UserStake {
    // ...
    /// Snapshot of global cumulative rewards at user's last interaction.
    /// Flash-loan resistance: users who stake and unstake in same epoch
    /// get reward_delta = 0 because cumulative only advances at epoch
    /// boundaries, not per-slot. This field is NOT a time-based reference.
    pub rewards_per_token_paid: u128,
    // ...
}
```

This clarifies the architecture for future auditors and maintainers.
