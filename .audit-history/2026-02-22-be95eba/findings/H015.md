# H015: Re-Initialization of Already-Initialized Accounts

**Strategy:** STR-015
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

All initialization instructions across the 5 programs correctly use Anchor's `init` constraint, which prevents re-initialization by checking that the account has zero discriminator (i.e., has never been initialized). The protocol is **NOT VULNERABLE** to re-initialization attacks.

### How Anchor's `init` Prevents Re-Initialization

Per the Anchor documentation and FP-004 from the knowledge base:

1. When `#[account(init)]` is used, Anchor creates the account with an 8-byte discriminator prepended to the account data
2. The discriminator is computed as `hash("account:<StructName>")[0..8]`
3. On subsequent attempts to call the same initialization instruction, Anchor's account deserialization checks the discriminator
4. If the discriminator is non-zero (meaning the account already exists), the `init` constraint **fails** and returns an error
5. This prevents the account from being re-initialized

**Code reference**: All `init` constraints observed use the full pattern:
```rust
#[account(
    init,
    payer = <payer>,
    space = <size>,
    seeds = [<seeds>],
    bump
)]
```

This pattern ensures:
- Account is created with correct discriminator
- Account can only be created once (discriminator check)
- Account is created at the deterministic PDA address (seeds + bump)
- Re-calling the instruction will fail at the Anchor constraint level

### Edge Case: `init_if_needed` Usage

The **only** use of `init_if_needed` is in the staking program:

**File**: `/Users/mlbob/Projects/Dr Fraudsworth/programs/staking/src/instructions/stake.rs:44-49`

```rust
#[account(
    init_if_needed,
    payer = user,
    space = UserStake::LEN,
    seeds = [USER_STAKE_SEED, user.key().as_ref()],
    bump,
)]
pub user_stake: Account<'info, UserStake>,
```

**Analysis per FP-004**:
- `init_if_needed` is safe here because the PDA seeds include the user's pubkey (`user.key().as_ref()`)
- Each user gets a unique UserStake account (one per user address)
- New user detection uses `user.owner == Pubkey::default()` (line 106), which is correct because Anchor zero-initializes account data when creating the account
- If the UserStake already exists (returning user), the `init_if_needed` loads the existing state and skips the initialization block
- No authority or critical state can be overwritten because the owner check ensures initialization only happens once (when `owner == Pubkey::default()`)

**Verdict**: This is the **correct and safe** use of `init_if_needed` per the common false positives guide.
