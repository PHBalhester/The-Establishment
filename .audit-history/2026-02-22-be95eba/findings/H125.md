# H125: AMM Pool Creation Lacks Mint Validation

**Strategy:** STR-125: AMM Pool Supports Any Token Pair

**Determination:** CONFIRMED

**Severity:** MEDIUM

**Confidence:** HIGH (95%)

---

## Summary

The `initialize_pool` instruction in the AMM program accepts any two token mints without validating them against an allowed list. While the protocol architecture specifies only 4 intended pools (CRIME/SOL, FRAUD/SOL, CRIME/PROFIT, FRAUD/PROFIT), there is no on-chain enforcement mechanism to prevent creation of unauthorized pools. An attacker or misconfigured deployment could create pools between:

- Arbitrary external tokens and protocol tokens (enabling rug pulls)
- Arbitrary non-protocol tokens (polluting the AMM with junk pairs)
- Same token paired with itself (creating nonsensical markets)

The protection currently relies entirely on **off-chain admin discipline** rather than on-chain validation.

## Impact

### Severity: MEDIUM (not HIGH or CRITICAL)

**Why not higher:**
- Pool creation requires **admin signature** — not permissionless
- Existing swap instructions (`swap_sol_pool`, `swap_profit_pool`) are gated to authorized pools
- Cross-program calls from Tax Program validate pool addresses deterministically via PDA seeds

**Why not lower:**
- A compromised or malicious admin could create **honeypot pools** (e.g., `EvilToken/CRIME`) to extract value from users who accidentally trade on the wrong pair
- A misconfigured deployment could lock liquidity in unintended pools
- Unauthorized pools could confuse protocol analytics and monitoring systems

### Exploit Scenarios

#### Scenario A: Honeypot Pool (Requires Compromised Admin)
1. Attacker compromises admin keypair
2. Attacker calls `initialize_pool(ExternalToken, CRIME)` with `amount_a=0.1 SOL` seeding
3. Pool PDA created: `hash([b"pool", ExternalToken, CRIME])`
4. Attacker publishes fake swap path: "Trade CRIME for ExternalToken on AMM"
5. Users connect to the unauthorized pool, send CRIME
6. Attacker drains the pool and mints more ExternalToken

#### Scenario B: Misconfiguration Lock
1. Deployment script has typo in mint address
2. Admin accidentally creates pool with wrong mint (e.g., test mint instead of production)
3. Liquidity is seeded and locked forever (no pool closure mechanism exists)
4. Protocol cannot reclaim the stranded liquidity

#### Scenario C: Supply Chain Attack
1. Attacker modifies deployment script (post-clone, pre-execution)
2. Creates pool between CRIME and a rug-pull token
3. Adds pool address to client-side UI routing
4. Users unknowingly swap CRIME for worthless token

## Recommendation

### P1 (Pre-Mainnet) — Add Mint Allowlist

Implement on-chain validation:

```rust
// In initialize_pool.rs handler

// Define approved mints (could be stored in ProtocolConfig PDA)
const APPROVED_MINTS: &[Pubkey] = &[
    CRIME_MINT,
    FRAUD_MINT,
    PROFIT_MINT,
    WSOL_MINT,
];

// Validate both mints are in approved list
require!(
    APPROVED_MINTS.contains(&ctx.accounts.mint_a.key()),
    AmmError::UnauthorizedMint
);
require!(
    APPROVED_MINTS.contains(&ctx.accounts.mint_b.key()),
    AmmError::UnauthorizedMint
);

// Validate pairing rules (optional: enforce CRIME/FRAUD must pair with PROFIT or SOL)
// This prevents CRIME/FRAUD pairing directly
```

### P2 (Enhancement) — Dynamic Allowlist via ProtocolConfig

Replace hardcoded constants with a ProtocolConfig PDA:

```rust
pub struct ProtocolConfig {
    pub allowed_mints: Vec<Pubkey>,
    pub admin: Pubkey,
}

// In initialize_pool:
require!(
    ctx.accounts.protocol_config.allowed_mints.contains(&ctx.accounts.mint_a.key()),
    AmmError::UnauthorizedMint
);
```

This allows mainnet mint additions without redeployment (e.g., adding wrapped BTC).

### P3 (Nice-to-Have) — Pool Registry

Add a PoolRegistry PDA to track all authorized pool pairs:

```rust
pub struct PoolRegistry {
    pub authorized_pairs: BTreeMap<(Pubkey, Pubkey), bool>,
}

// Checked before accepting a new pool
require!(
    ctx.accounts.pool_registry.authorized_pairs.get(&(mint_a, mint_b)) == Some(&true),
    AmmError::UnauthorizedPoolPair
);
```

This would allow audit trails and deprecation of pools.

## False Positives Addressed

**Q: "Canonical ordering prevents duplicate pools"**
A: Canonical ordering prevents the same *unordered* pair from being created twice, but does NOT prevent creation of unauthorized mint pairs.

**Q: "Swap instructions validate pools via PDA"**
A: True. `swap_sol_pool` and `swap_profit_pool` are correctly gated. However, this is **defense-in-depth only**. The weakness is at pool *creation* time, not swap time.

**Q: "Admin is trusted, no validation needed"**
A: Partially true on devnet. On mainnet, the admin key should burn after initialization or be held by a tiered-timelock multisig. Without mainnet governance, a single admin keypair is a critical risk.
