# S008: Combined Admin Key Compromise: Bounty Drain + Whitelist Manipulation

**Strategy:** S008
**Determination:** MITIGATED
**Severity:** MEDIUM
**Confidence:** HIGH

## Summary

While individual admin capabilities are limited by design, a combined attack exploiting both the Transfer Hook whitelist authority and the epoch bounty mechanism could extract value and disrupt protocol operations. However, the attack is significantly constrained by: (1) the intentional deploy-and-lock admin model where authorities are meant to be burned post-initialization, (2) the limited scope of admin powers (cannot drain pools or modify state), and (3) the economic limits on bounty extraction. The primary risk is **operational disruption** rather than fund theft.

### Attack Vector 2: Whitelist Manipulation

**Location:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/add_whitelist_entry.rs:12-42`

```rust
pub fn handler(ctx: Context<AddWhitelistEntry>) -> Result<()> {
    let auth = &ctx.accounts.whitelist_authority;
    let address = ctx.accounts.address_to_whitelist.key();

    // Authority check
    require!(
        auth.authority == Some(ctx.accounts.authority.key()),
        TransferHookError::Unauthorized
    );

    // Address validation: reject system program and null pubkey
    require!(
        address != Pubkey::default() && address != anchor_lang::system_program::ID,
        TransferHookError::InvalidWhitelistPDA
    );

    // Populate entry
    let entry = &mut ctx.accounts.whitelist_entry;
    entry.address = address;
    entry.created_at = Clock::get()?.unix_timestamp;

    Ok(())
}
```

**Constraints:**
- Authority check: Only whitelist authority can add entries
- One-way operation: **Cannot remove** whitelist entries (no `remove_whitelist_entry` instruction)
- Burn mechanism: Authority can permanently burn itself via `burn_authority.rs:38`

**Admin Can:**
- Whitelist arbitrary addresses (except system program and null key)
- Grant unauthorized accounts ability to send/receive CRIME/FRAUD/PROFIT tokens

**Admin Cannot:**
- Remove existing whitelist entries
- Bypass whitelist check (enforced by Token-2022 runtime on every transfer)
- Access token vaults directly (vaults are PDA-owned)

## Combined Attack Scenario

### Preconditions
1. Admin controls both Transfer Hook authority AND has access to devnet admin key
2. Protocol has NOT burned whitelist authority yet (still in initialization/early deployment phase)
3. Either:
   - `force_carnage` is present (devnet binary deployed to mainnet — H004), OR
   - Admin operates during the narrow post-deployment/pre-authority-burn window

**Step 1: Monitor Carnage Vault**

```typescript
// Trigger epochs as fast as possible to drain bounty
setInterval(async () => {
  // Check if epoch boundary reached
  const epochState = await epochProgram.account.epochState.fetch(epochStatePDA);
  const currentSlot = await connection.getSlot();

  if (currentSlot >= epochState.lastEpochSlot + SLOTS_PER_EPOCH) {
```

**Step 4: Operational Disruption**
- Whitelisted attacker accounts can now transfer CRIME/FRAUD tokens freely
- Protocol operators cannot revoke whitelist entries (no remove instruction)
- Bounty accumulation is drained at maximum rate (once per epoch)

**Maximum Extraction:**
- Bounty rate: 0.001 SOL per epoch
- Mainnet epoch: 4500 slots ≈ 30 minutes (48 epochs/day)
- Daily extraction: 48 × 0.001 = **0.048 SOL/day** (≈$9.60 at $200/SOL)
- Annual extraction: 17.52 SOL/year (≈$3,504/year)

**Assessment:** Economic impact is **MINIMAL**. Bounty drain is negligible compared to protocol revenue. The real damage is **operational disruption** from unauthorized whitelist entries.

## Impact Assessment

### Scenario 1: Deploy-and-Lock Model (Intended)

**Timeline:**
1. Deploy all programs
2. Initialize with admin/authority keys
3. Create all pools (AMM admin)
4. Whitelist all protocol addresses (Transfer Hook authority)
5. **Burn AMM admin** (pool creation disabled)
6. **Burn Transfer Hook authority** (whitelist frozen)
7. Protocol operates autonomously

**Attack window:** Hours to days (initialization phase only)

**Maximum damage during window:**
- Whitelist pollution: Attacker addresses gain transfer privileges (cannot be revoked)
- Bounty drain: Negligible (≈$9.60/day)
- Pool creation: Attacker could create malicious pools with 5% fee cap (cannot drain existing pools)

**Impact after authority burn:**
- **ZERO** — admins have no capabilities post-burn

### Scenario 2: Authority Not Burned (Deviation from Design)

**If protocol operates indefinitely with active admin:**
- Bounty drain: 0.048 SOL/day (17.52 SOL/year) — **NEGLIGIBLE**
- Whitelist pollution: Ongoing risk of unauthorized addresses
- No direct fund theft (cannot withdraw from PDA vaults)

**Impact:** **MEDIUM** operational disruption, **LOW** economic damage

### Scenario 3: Devnet Binary Deployed to Mainnet (H004)

**If `force_carnage` is present:**
- Market manipulation: 10-20% profit per Carnage
- Carnage Fund drain: 100% of accumulated fees (5% of protocol volume)

**Impact:** **CRITICAL** economic damage, protocol integrity destroyed

## Mitigating Factors

### 1. Design Intent: Authorities Are Temporary

From `.audit/context/07-upgrade-admin.md:626-632`:

> **Operational Security Assessment**
>
> Role Separation (Mainnet Recommended):
> - AMM Admin: Separate multisig (or burn immediately)
> - Transfer Hook Authority: Same as upgrade authority (or burn)

The protocol is **designed** for authorities to be burned post-initialization. This is not oversight — it's intentional architecture.

### 2. Limited Admin Scope

From `.audit/ARCHITECTURE.md` Section 2.4:

> Maximum Damage Assessment (Key Compromise):
> - Admin key compromised: Can update treasury (redirect tax revenue), modify whitelist, force carnage (devnet). **Cannot drain pools, cannot modify program logic.**
> - All keys compromised: Tax revenue redirect + off-schedule carnage. **Pool funds and staked tokens remain safe (PDA-controlled).**

**Critical constraint:** Admin powers do NOT include:
- Withdrawing from pool vaults (no `withdraw` instruction)
- Modifying pool parameters post-creation (no `update_pool_fee`)
- Changing tax rates (VRF-driven, not admin-controlled)
- Accessing staking rewards (CPI-gated to Tax/Epoch programs)

### 3. Bounty Economics Are Non-Viable

**Bounty accumulation rate:**
- Carnage vault funded by 5% of 15% tax = 0.75% of trading volume
- At $100K/day volume: $750/day flows to vault
- Bounty extraction: 0.048 SOL/day ≈ $9.60/day (at $200/SOL)
- **Attacker extracts 1.28% of vault inflow**

The remaining 98.72% accumulates for legitimate Carnage operations. Bounty drain is economically insignificant.

### 4. Whitelist Cannot Be Used to Steal Funds

**What whitelist grants:**
- Ability to send/receive CRIME/FRAUD/PROFIT tokens
- Bypass transfer hook validation

**What whitelist does NOT grant:**
- Access to pool vaults (PDA-owned)
- Ability to withdraw staking rewards (CPI-gated)
- Minting authority (mints are separate PDAs)
- Ability to drain carnage vault (no withdraw instruction)

An attacker with whitelisted addresses can **participate in the protocol** but cannot **extract protocol-owned funds**.

### 5. force_carnage is Feature-Gated

**Double-gated protection:**
1. Module level: `#[cfg(feature = "devnet")]` at `mod.rs:7-8`
2. Instruction level: `#[cfg(feature = "devnet")]` at `lib.rs:261`

**Build verification** (from H004 recommendation):
```bash
# Mainnet build MUST NOT use --features devnet
anchor build  # Standard build excludes force_carnage

# Verify IDL has no force_carnage
if grep -q "force_carnage" target/idl/epoch_program.json; then
  echo "ERROR: force_carnage found in mainnet build!"
  exit 1
fi
```

**Risk:** Human deployment error, not code vulnerability.

## Determination Rationale: MITIGATED (not CONFIRMED)

**Why MITIGATED instead of CONFIRMED:**

1. **Intended Design:** The deploy-and-lock model with authority burning is documented across multiple files (mainnet-checklist.md, security-model.md, upgrade/admin analysis)

2. **Economic Non-Viability:** Bounty drain alone is negligible (1.28% of vault inflow). Whitelist manipulation cannot directly steal funds.

3. **Critical Dependency on Deployment Error:** The high-value attack (H004 force_carnage manipulation) only works if devnet binary deployed to mainnet — this is a **deployment process failure**, not a code vulnerability.

4. **Narrow Attack Window:** For non-devnet scenarios, the attack only works during the hours/days between deployment and authority burning.

**Why MEDIUM severity (not LOW):**

1. **Operational Impact:** Whitelist pollution is permanent (cannot remove entries), creating long-term operational burden

2. **Deployment Process Risk:** While authorities are meant to be burned, there's no **on-chain enforcement** — a tired developer could deploy and forget to burn

3. **Combined Effect:** While individual vectors are low-impact, the combination creates a **nuisance attack** that disrupts protocol operations

**Why not HIGH/CRITICAL:**

1. **No Direct Fund Theft:** Attacker cannot drain pools, vaults, or staking rewards
2. **Economic Damage is Minimal:** $9.60/day bounty drain is negligible
3. **Depends on Process Deviation:** Attack requires protocol team to deviate from documented deployment process

**Add automated authority burn verification:**

File: `scripts/deploy/post-deploy-checks.sh`
```bash
#!/bin/bash
# Post-deployment verification script

echo "Verifying authorities are burned..."

# Check AMM admin
AMM_ADMIN=$(solana program show $AMM_PROGRAM_ID | grep "Admin:" | awk '{print $2}')
if [ "$AMM_ADMIN" != "11111111111111111111111111111111" ]; then
  echo "WARNING: AMM admin not burned! Current admin: $AMM_ADMIN"
  echo "Run: anchor run burn-amm-admin"
  exit 1
fi

# Check Transfer Hook authority
HOOK_AUTH=$(anchor account WhitelistAuthority $HOOK_AUTHORITY_PDA | grep "authority:" | awk '{print $2}')
if [ "$HOOK_AUTH" != "None" ]; then
  echo "WARNING: Transfer Hook authority not burned! Current authority: $HOOK_AUTH"
  echo "Run: anchor run burn-hook-authority"
  exit 1
fi

echo "✓ All authorities properly burned"
```
