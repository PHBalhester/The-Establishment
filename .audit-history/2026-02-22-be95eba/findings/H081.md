# H081: Staking User Account Not Validated Against Pool

**Strategy:** STR-081
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** High

## Summary

STR-081 hypothesizes that a `UserStake` account for Pool A could be passed to an instruction for Pool B, causing cross-pool accounting corruption. Investigation of the staking program reveals this vulnerability is **not possible** because the staking protocol implements a **global singleton pool architecture**—there is exactly one StakePool per deployment, eliminating the possibility of cross-pool account substitution.

## Evidence

### 1. Singleton Pool Architecture

The staking program defines a **global singleton StakePool** with deterministic seeds:

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/staking/src/state/stake_pool.rs:9,17`
```rust
/// Seeds: ["stake_pool"]
#[account]
pub struct StakePool {
    pub total_staked: u64,
    pub rewards_per_token_stored: u128,
    pub pending_rewards: u64,
    pub last_update_epoch: u32,
    pub total_distributed: u64,
    pub total_claimed: u64,
    pub initialized: bool,
    pub bump: u8,
}
```

The StakePool is derived with **fixed seeds** `["stake_pool"]` with no discriminating parameters. This produces exactly **one valid PDA address per program deployment**. There are no pool variants, no factory patterns, and no way to create multiple pools.

### 2. UserStake Architecture

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/staking/src/state/user_stake.rs:9,17-25`
```rust
/// PDA Derivation: seeds = ["user_stake", user_pubkey]
pub struct UserStake {
    pub owner: Pubkey,           // User who owns this stake position
    pub staked_balance: u64,     // Amount staked by this user
    pub rewards_per_token_paid: u128,
    pub rewards_earned: u64,
    pub total_claimed: u64,
    pub first_stake_slot: u64,
    pub last_update_slot: u64,
    pub bump: u8,
}
```

**Key observation:** UserStake contains NO field linking it to a specific pool. It has an `owner` field (the user's pubkey), but no `stake_pool` reference.

### 3. Account Validation in User Operations

All user-facing instructions derive accounts with fixed, deterministic seeds that resolve to the singleton pool:

**stake.rs (line 34-39):**
```rust
#[account(
    mut,
    seeds = [STAKE_POOL_SEED],
    bump = stake_pool.bump,
)]
pub stake_pool: Account<'info, StakePool>,
```

**unstake.rs (line 40-44):**
```rust
#[account(
    mut,
    seeds = [STAKE_POOL_SEED],
    bump = stake_pool.bump,
)]
pub stake_pool: Account<'info, StakePool>,
```

**claim.rs (line 36-40):**
```rust
#[account(
    mut,
    seeds = [STAKE_POOL_SEED],
    bump = stake_pool.bump,
)]
pub stake_pool: Account<'info, StakePool>,
```

Every instruction that accepts a StakePool validates it via the **same fixed seed derivation**. No caller can substitute an arbitrary pool account—Anchor's constraint system enforces the PDA match at the instruction level.

### 4. No Pool Validation on UserStake

The UserStake account struct does NOT include a `has_one = stake_pool` constraint:

**stake.rs (line 43-50):**
```rust
#[account(
    init_if_needed,
    payer = user,
    space = UserStake::LEN,
    seeds = [USER_STAKE_SEED, user.key().as_ref()],
    bump,
)]
pub user_stake: Account<'info, UserStake>,
```

**unstake.rs (line 48-54):**
```rust
#[account(
    mut,
    seeds = [USER_STAKE_SEED, user.key().as_ref()],
    bump = user_stake.bump,
    constraint = user_stake.owner == user.key() @ StakingError::Unauthorized,
)]
pub user_stake: Account<'info, UserStake>,
```

The absence of a `has_one` constraint is **not a vulnerability** because:
1. The UserStake stores per-user state only (no pool pointer field exists)
2. The StakePool is globally singleton—there's no Pool B to confuse it with
3. The instruction's StakePool argument is validated separately via its own PDA derivation

### 5. UserStake Ownership Validation

All user-facing instructions validate that the UserStake account belongs to the caller via the `owner` field:

**unstake.rs (line 48-54):**
```rust
constraint = user_stake.owner == user.key() @ StakingError::Unauthorized,
```

**claim.rs (line 44-50):**
```rust
constraint = user_stake.owner == user.key() @ StakingError::Unauthorized,
```

This prevents **user-to-user** account confusion (Alice accessing Bob's stake), but cross-pool confusion is moot because there is only one pool.

## Impact

**Zero impact.** The singleton pool architecture fundamentally eliminates the attack surface. Even if an attacker could manufacture a malicious UserStake account:

1. It would still reference the same global StakePool (no Pool B exists to switch to)
2. The user_stake PDA derivation includes the user's pubkey, so only the user can create/access their own account
3. The stake pool's `total_staked` and `rewards_per_token_stored` are global, not per-UserStake

Any UserStake operation (stake, unstake, claim) interacts with the **one, true StakePool**—no cross-pool routing is possible.

## Recommendation

**No action required.** This vulnerability class is architecturally impossible in a singleton pool design. The absence of `has_one = stake_pool` on UserStake is not a defect—it's correct, because:

1. The constraint would be redundant (only one pool exists to link to)
2. Including it would consume additional compute and account space without security benefit

**For future work:** If the protocol ever transitions to a multi-pool staking architecture (different tokens, different reward streams, etc.), the `has_one = stake_pool` constraint MUST be added to UserStake at that time.
