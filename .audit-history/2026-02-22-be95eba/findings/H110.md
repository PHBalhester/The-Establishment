# H110: Carnage Execution Without VRF (Direct Call)

**Strategy:** STR-110

## Summary

The hypothesis that `execute_carnage` on the Tax Program might be callable directly with a crafted `carnage_signer`, bypassing VRF requirements, is **disproved**. The protocol's signer validation is correct and strict. The `carnage_signer` is a PDA derived exclusively by the Epoch Program, and the Tax Program validates it through cryptographic constraints that **cannot be forged**.

### 2. How Seeds::Program Constraint Works (Cryptographic Security)

The Anchor constraint `seeds::program = epoch_program_id()` enforces:

```rust
// What Anchor verifies at runtime:
actual_pda = Pubkey::find_program_address(
    &[CARNAGE_SIGNER_SEED],
    &EPOCH_PROGRAM_ID
)
require!(actual_pda == carnage_authority.key())
```

**Why this cannot be forged:**
- The PDA is derived from `find_program_address`, which uses **SHA-256 hashing** of seeds + program ID
- Solana's `find_program_address` iterates bump values (0-255) until it finds one where the derived address is **not a valid ED25519 public key** (off-curve)
- To forge a valid PDA address:
  1. Attacker must produce an account with a valid ED25519 public key address (only ~50% of 256-bit values are on-curve)
  2. That address must be the output of `find_program_address([b"carnage_signer"], EPOCH_PROGRAM_ID)` for some bump value
  3. For the account to be callable as a signer, it must also have a valid keypair (private key) — **which only the Epoch Program can produce** via `invoke_signed`

**Mathematical impossibility:** The attacker cannot provide two things simultaneously:
- A valid Solana address (ED25519 public key) on the signature curve
- That address is the exact PDA hash output for the Epoch Program's seeds

The constraints form a **mathematical lock** that only the legitimate PDA owner can satisfy.

**Tax Program validates the signer:**
- When Tax receives the CPI from Epoch, the `carnage_authority` in the Accounts struct is validated by Anchor
- Anchor verifies:
  1. The account is a `Signer<'info>` (it signed the CPI call)
  2. The PDA address matches `find_program_address([CARNAGE_SIGNER_SEED], EPOCH_PROGRAM_ID)`
  3. Both conditions must be true simultaneously

### 4. Direct Call Impossibility

An attacker attempting to call `Tax::swap_exempt` directly would:

1. **Create a fake account** with some address X
2. Pass X as the `carnage_authority` argument
3. Sign the transaction with X's keypair

**Result:** Anchor's constraint validation would fail:
- Is X a valid PDA derived from `[b"carnage_signer"]` and Epoch Program ID? **No** (wrong program ID or wrong seeds)
- OR: If X = a valid PDA, does X have a keypair to sign? **Only if X is the legitimate Epoch-derived PDA**, in which case only the Epoch Program can invoke_signed it

**Exploit path blocked.** ✗
