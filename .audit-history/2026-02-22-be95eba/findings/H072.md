# H072: Transfer Hook Whitelist Entry PDA Not Closed on Remove

**Strategy:** STR-072
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis assumes that a `remove_from_whitelist` instruction exists in the transfer hook program and that it fails to close the PDA account when removing entries, potentially leaving stale data. However, **no such instruction exists in the codebase**. The transfer hook program implements an immutable whitelist model where:

1. Whitelist entries can be added via `add_whitelist_entry` instruction
2. The whitelist authority can be permanently burned via `burn_authority` instruction
3. **There is NO remove/delete mechanism for individual entries**

### Whitelist Entry Structure

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/state/whitelist_entry.rs`

```rust
/// A whitelisted address entry.
///
/// Whitelist uses existence-based PDA pattern: if this PDA exists for an address,
/// that address is whitelisted. PDA non-existence = not whitelisted.
///
/// Seeds: [b"whitelist", address.as_ref()]
/// Space: 8 (discriminator) + 32 (Pubkey) + 8 (i64) = 48 bytes
#[account]
#[derive(InitSpace)]
pub struct WhitelistEntry {
    pub address: Pubkey,
    pub created_at: i64,
}
```

The whitelist uses an **existence-based PDA pattern**: presence indicates whitelist status. Absence indicates not whitelisted. There is no need to explicitly "close" entries because non-existence is the natural state representing "not whitelisted."

### Add-Only Model Documentation

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/add_whitelist_entry.rs`

The handler and constraints document that:
1. Only the whitelist authority can add entries (Anchor constraint checks `authority.is_some()`)
2. Entries are created with `#[account(init, ...)]` constraint — can only be created once
3. Authority can be burned (making the whitelist immutable) via `burn_authority`

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/lib.rs` — `burn_authority` documentation:
```rust
/// Permanently burn the whitelist authority.
///
/// Sets authority to None, making the whitelist immutable. This is
/// idempotent: calling on an already-burned authority succeeds silently.
```

The design choice is explicit: **once an entry is added, it cannot be removed individually. The entire whitelist can only be made immutable via authority burn**, not edited entry-by-entry.

### Transfer Validation

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/transfer_hook.rs`

The transfer hook checks for whitelist entry existence:
```rust
// Whitelist check uses is_whitelisted() which derives the PDA and checks existence
// If the PDA exists (has lamports > 0), the address is whitelisted
```

This is the **inverse pattern**: checking if an account exists, not requiring it to be non-zero or carry specific data. An account that was closed (lamports zeroed, data cleared) would no longer exist and would not be whitelisted.

## Impact

**No impact.** The hypothesis does not apply because:

1. **No removal instruction exists** — There is no code path that removes entries
2. **Intentional design** — The add-only model with authority burn is a deliberate design choice to ensure whitelist immutability
3. **Existence-based checking** — The whitelist validation logic checks PDA existence, not data content. A closed PDA would return zero lamports and fail the existence check (correctly) without ever reading stale data
4. **Authority burn is the control** — If an entry should never have been added, the whitelist authority can be burned to freeze the list and prevent future changes

## Recommendation

**No action required.** The transfer hook whitelist is designed as immutable once authority is burned. If the project needs a removable whitelist in the future, add an explicit `remove_whitelist_entry` instruction with:

1. Authority check (only signer can remove)
2. Account closure using `close =` Anchor constraint to zero lamports and data
3. Event emission documenting removal
4. Test coverage for the removal path

However, this was not implemented in the current design, suggesting immutability was intentional.
