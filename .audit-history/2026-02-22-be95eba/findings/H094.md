# H094: Staking Pool Not Faction-Specific — Universal Pool Accepts All Staking Tokens

**Strategy:** STR-094: Staking Pool Not Validated as Correct Faction

**Determination:** NOT A FINDING — Design is intentional and correctly implemented

**Severity:** INFORMATIONAL (design verification only)

**Confidence:** HIGH (100% code + docs verification)

---

## Summary

**Hypothesis:** Staking pools might be faction-specific (separate pools for CRIME and FRAUD stakers). If pool validation doesn't check faction, a user could stake CRIME in the FRAUD pool or vice versa.

**Finding:** The staking system uses a **single universal StakePool PDA** that accepts only **PROFIT tokens**. Faction tokens (CRIME/FRAUD) cannot be staked at all. The staking pool is faction-agnostic by architectural design, not a validation gap.

This is **not a vulnerability** — it is the intended tokenomic model where:
- CRIME and FRAUD are swappable as user preference
- PROFIT is the shared yield token earned by all stakers regardless of faction choice
- Rewards are distributed pro-rata to PROFIT stakers, not faction members

## Impact

**Positive security property:** Users staking CRIME cannot accidentally deposit into a misconfigured FRAUD pool (or vice versa) because:
1. Only PROFIT can be deposited (enforced by `token::mint = profit_mint`)
2. There is only one staking pool (singleton PDA)
3. All stakers earn from a unified reward distribution

**Economic design:** The universal pool ensures:
- Reward distribution is fair and faction-agnostic
- Stakers who switch factions can redeploy PROFIT without restriction
- Tax splitting (75/24/1) flows to a single reward pool regardless of swap faction
- No liquidity fragmentation (no separate CRIME staker pool vs FRAUD staker pool)

```markdown
## Decision: Universal Staking Pool vs Faction-Specific Pools

**Choice:** Single global StakePool for all PROFIT stakers, faction-agnostic.

**Rationale:**
1. Reduces PDA proliferation (1 pool, not 2)
2. Eliminates liquidity fragmentation (all SOL rewards in one escrow)
3. Enables faction switching (users can redeploy PROFIT without unstaking)
4. Aligns with soft peg model (factions are interchangeable, not separate classes)

**Implementation:** StakePool PDA seeds = ["stake_pool"] (no faction parameter)
                    Stake::token::mint = profit_mint (only PROFIT accepted)
                    Single escrow_vault for all pending rewards

**Verification:** Anchor constraints enforce PROFIT-only deposits; init constraint
                 prevents duplicate pools.
```
