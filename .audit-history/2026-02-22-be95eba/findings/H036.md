# H036: VRF Timeout Recovery Does Not Invalidate Old Randomness

**Strategy:** STR-036
**Determination:** NOT VULNERABLE
**Severity:** INFO
**Confidence:** HIGH

## Summary

The `retry_epoch_vrf` instruction allows replacing a stale randomness account after VRF timeout (300 slots). Initial concern was that if the old randomness account eventually gets revealed by the oracle, an attacker could consume both the old and new randomness, potentially causing double epoch transitions or VRF manipulation. However, after thorough analysis, the anti-reroll protection mechanisms prevent this attack vector entirely. The old randomness account is effectively invalidated by overwriting the `pending_randomness_account` key binding, and `consume_randomness` enforces exact key matching.

## Attack Scenario Analysis

### Hypothetical Attack: Consume Stale Randomness After Retry

**Preconditions:**
- Oracle A is assigned to randomness account R1
- Oracle A experiences downtime (VRF timeout occurs)
- Retry is called with new randomness account R2 (assigned to working Oracle B)
- Oracle A comes back online and reveals R1

**Attack Steps:**
1. Epoch boundary reached at slot 1000
2. User calls `trigger_epoch_transition` with randomness R1 (Oracle A)
3. `epoch_state.pending_randomness_account = R1` (key binding)
4. `epoch_state.vrf_pending = true`
5. Oracle A is offline, 300 slots pass (timeout at slot 1300+)
6. Admin calls `retry_epoch_vrf` with fresh randomness R2 (Oracle B)
7. **Key replacement occurs:** `epoch_state.pending_randomness_account = R2` (R1 is now invalid)
8. `epoch_state.vrf_request_slot = 1301` (reset)
9. Oracle B reveals R2 at slot ~1304
10. User calls `consume_randomness` with R2
11. **Anti-reroll check:** `R2 == epoch_state.pending_randomness_account` ✓ PASS
12. VRF consumed, `epoch_state.vrf_pending = false`
13. **Oracle A belatedly reveals R1 at slot 1350**
14. Attacker attempts: `consume_randomness` with R1
15. **Check 1 FAILS:** `epoch_state.vrf_pending = false` → `EpochError::NoVrfPending`

**Result:** Attack is **blocked at instruction level**. The `vrf_pending` flag is already cleared after consuming R2, so any attempt to consume R1 is rejected immediately.

### Alternative Attack: Consume Old Before New

**Attack Steps:**
1. Epoch boundary reached, randomness R1 committed (Oracle A offline)
2. 300 slots pass, timeout window opens
3. Oracle A belatedly comes online at slot 1305 and reveals R1
4. Attacker races to call `consume_randomness` with R1 before anyone calls retry
5. **Anti-reroll check:** `R1 == epoch_state.pending_randomness_account` ✓ PASS (R1 is still bound)
6. VRF consumed successfully using R1
7. `epoch_state.vrf_pending = false`
8. Admin later calls `retry_epoch_vrf` with R2
9. **Check FAILS:** `epoch_state.vrf_pending = false` → `EpochError::NoVrfPending`

**Result:** This is **not an attack** — it's the intended recovery path. If the original oracle eventually reveals within the deadline, consuming that randomness is valid. The retry mechanism is a fallback for *permanent* oracle failure, not a way to get "better" randomness.

### Attack Attempt 3: Double-Consume via Forced Retry

**Attack Steps:**
1. Randomness R1 committed and consumed successfully (epoch advanced)
2. Attacker calls `retry_epoch_vrf` with R2 to get fresh randomness
3. **Check FAILS:** `epoch_state.vrf_pending = false` → `EpochError::NoVrfPending` (line 62)

**Result:** Cannot retry after successful consumption. The `vrf_pending` flag must be true.

## Root Cause Analysis: Why This Is Secure

### Defense Layer 1: Single-Key Binding (Anti-Reroll)

The `pending_randomness_account` field can only hold **one** Pubkey at a time. When `retry_epoch_vrf` executes:

```rust
epoch_state.pending_randomness_account = ctx.accounts.randomness_account.key();
```

This is a **destructive overwrite**, not an append. The old randomness account key is discarded from on-chain state. There is no array or history of randomness accounts.

**Implication:** After retry, the protocol has "forgotten" the old randomness account's identity. `consume_randomness` will reject any account that doesn't match the current binding.

### Defense Layer 2: Exclusive VRF Pending State

The `vrf_pending` boolean is a **mutual exclusion lock**:

- `trigger_epoch_transition`: Sets `vrf_pending = true` (line 181)
- `consume_randomness`: Requires `vrf_pending = true` (line 151), then sets `vrf_pending = false` (line 206)
- `retry_epoch_vrf`: Requires `vrf_pending = true` (line 62), does NOT clear it (intentional)

**State Transition Table:**

| Action | Precondition | Postcondition |
|--------|--------------|---------------|
| `trigger_epoch_transition` | `vrf_pending == false` | `vrf_pending = true`, `pending_randomness_account = R1` |
| `retry_epoch_vrf` | `vrf_pending == true` | `vrf_pending = true`, `pending_randomness_account = R2` (overwrites R1) |
| `consume_randomness` | `vrf_pending == true` | `vrf_pending = false`, `taxes_confirmed = true` |

**Critical Observation:** After `consume_randomness` clears `vrf_pending`, both retry and consume become invalid until the next `trigger_epoch_transition`. This prevents:
- Double consumption (can't consume twice in one epoch cycle)
- Retry after consumption (retry requires pending VRF)
- New epoch without consuming (next trigger requires `!vrf_pending`)

### Defense Layer 3: Timeout Window Correctness

The timeout check in `retry_epoch_vrf` is:

```rust
let elapsed_slots = clock.slot.saturating_sub(epoch_state.vrf_request_slot);
require!(
    elapsed_slots > VRF_TIMEOUT_SLOTS,  // Strict > not >=
    EpochError::VrfTimeoutNotElapsed
);
```

**VRF_TIMEOUT_SLOTS = 300** (~2 minutes at 400ms/slot).

This gives the original oracle 300 slots to reveal before retry is permitted. If the oracle reveals within this window, the randomness can be consumed normally. If the oracle fails for >300 slots, retry provides a recovery path by binding a fresh randomness account.

**Nuance from MEMORY.md:**
> "VRF gateway rotation DOES NOT WORK: Each randomness account is assigned to a specific oracle. Alternative gateways serve different oracles whose signatures fail on-chain (0x1780). Only retry the default gateway. If down, use VRF timeout recovery with fresh randomness."

This means:
- The retry mechanism creates a **new randomness account** (not just a gateway switch)
- The new account may be assigned to a different oracle by Switchboard
- The old randomness account remains assigned to the original (failed) oracle
- Even if the old oracle later reveals, it cannot be consumed after retry overwrites the key binding

## Theoretical Edge Cases Examined

### Case 1: Retry Called While Old Oracle Is Mid-Reveal

**Timeline:**
- Slot 1000: Trigger with R1 (Oracle A)
- Slot 1301: Timeout expires, retry with R2 (Oracle B)
- Slot 1302: Oracle A's reveal transaction for R1 lands on-chain
- Slot 1303: User calls `consume_randomness` with R1

**Result:**
- R1 is now revealed (has valid VRF bytes)
- But `consume_randomness` requires `ctx.accounts.randomness_account.key() == R2` (the current binding)
- Check fails: `R1 != R2` → `EpochError::RandomnessAccountMismatch`

**Verdict:** Secure. The reveal of R1 is valid at the Switchboard program level, but the Epoch Program rejects it because the key binding changed.

### Case 2: Multiple Sequential Retries

**Timeline:**
- Slot 1000: Trigger with R1
- Slot 1301: Retry with R2 (Oracle B also fails)
- Slot 1602: Retry again with R3 (Oracle C)

**State After Each Step:**
1. `pending_randomness_account = R1`, `vrf_pending = true`
2. `pending_randomness_account = R2`, `vrf_pending = true` (R1 invalidated)
3. `pending_randomness_account = R3`, `vrf_pending = true` (R1 and R2 invalidated)

**Valid Consume Target:** Only R3. Attempts to consume R1 or R2 will fail the key match check.

**Verdict:** Secure. Each retry invalidates all previous randomness accounts by overwriting the single binding slot.

### Case 3: Malicious Retry to Get "Better" Randomness

**Scenario:**
- Attacker sees R1 has been revealed with unfavorable VRF bytes (e.g., Carnage not triggered)
- Attacker waits for 300 slots to elapse
- Attacker calls `retry_epoch_vrf` with R2 hoping for more favorable randomness

**Defense:**
1. **Cost**: Attacker must pay for creating R2 (Switchboard randomness account creation cost)
2. **Unpredictability**: R2's VRF output is equally unpredictable as R1 (cryptographic randomness)
3. **No Advantage**: The attacker cannot know R2's result before committing to it (not-yet-revealed check in retry)
4. **Original Invalidated**: R1 can no longer be consumed after retry, so this is a **commitment change**, not a reroll

**Verdict:** This is the intended recovery mechanism, not an exploit. The attacker gains no advantage because:
- R2 is as random as R1 (no cherry-picking)
- R1 is invalidated (can't fall back to it)
- Cost is non-zero (randomness account creation)

**From `.audit/context/06-oracle-data.md:229`:**
> "The attacker cannot predict what the new oracle will reveal, so this is not an effective reroll vector. The retry mechanism correctly uses a fresh randomness account with unknown future value."

## Why This Is NOT a Finding

### 1. Old Randomness Is Cryptographically Invalidated

The binding overwrite (`pending_randomness_account = new_key`) is equivalent to revoking the old randomness account's authorization. Even though the old Switchboard account still exists on-chain and may contain valid VRF bytes, the Epoch Program will reject it due to key mismatch.

**Analogy:** It's like changing a door lock. The old key still exists physically, but the door won't open with it.

### 2. State Machine Prevents Double Execution

The `vrf_pending` flag acts as a **consumed nonce**:
- Set on commit (trigger or retry)
- Required for consume
- Cleared after consume

This ensures exactly-once consumption per epoch cycle, regardless of how many randomness accounts exist.

### 3. No State Corruption on Retry

When `retry_epoch_vrf` executes:
- **Does NOT** create duplicate randomness bindings
- **Does NOT** skip state machine validation
- **Does NOT** allow consuming without clearing pending flag
- **Only** replaces the key binding and resets the timeout clock

The epoch state remains consistent: one pending randomness account at a time.

### 4. Timeout Recovery Is By Design, Not a Bug

From VRF implementation docs and code comments:
- Timeout recovery prevents protocol deadlock if oracle fails
- Fresh randomness ensures new oracle assignment (different from failed oracle)
- This is a **liveness mechanism**, not a security hole

**From MEMORY.md:**
> "VRF timeout recovery: wait VRF_TIMEOUT_SLOTS (300 slots), create fresh randomness, call retry_epoch_vrf, then commit+reveal+consume. Fresh randomness may get a different (working) oracle."

**Enhancement Suggestion:** Add `original_randomness_account` and `new_randomness_account` fields to the event for clearer audit trail:

```rust
pub struct VrfRetryRequested {
```

**Benefit:** Off-chain services can track which randomness accounts were invalidated via retry, making forensic analysis easier.

### 3. Add Integration Test for Retry Invalidation (Testing)

**New Test Case:**
```rust
#[tokio::test]
async fn test_retry_invalidates_old_randomness() {
    // Setup: Trigger with R1, wait for timeout
    let r1 = create_randomness().await;
    trigger_epoch_transition(r1.pubkey).await?;

    // Wait for timeout
    advance_clock(VRF_TIMEOUT_SLOTS + 1).await;

    // Retry with R2
    let r2 = create_randomness().await;
    retry_epoch_vrf(r2.pubkey).await?;

    // Simulate: Oracle A belatedly reveals R1
    reveal_randomness(r1.pubkey).await?;

    // Attempt to consume R1 (should fail)
    let result = consume_randomness(r1.pubkey).await;
    assert_eq!(
        result.unwrap_err(),
        EpochError::RandomnessAccountMismatch,
        "Old randomness should be rejected after retry"
    );

    // Consume R2 (should succeed)
    reveal_randomness(r2.pubkey).await?;
    consume_randomness(r2.pubkey).await?;

    // Verify epoch advanced
    let state = fetch_epoch_state().await?;
    assert!(state.taxes_confirmed);
    assert!(!state.vrf_pending);
}
```
