# H022: Carnage Direction Prediction from On-Chain State (NOT VULNERABLE)
**Strategy:** STR-022
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis proposed that before `consume_randomness` is called, the VRF result is on-chain (after reveal), allowing an observer to compute which faction Carnage will buy and front-run by buying the same faction (profiting from Carnage-driven price increase) or selling the winning faction. After analyzing the VRF lifecycle and Carnage execution flow, this attack vector is **not viable** due to atomic bundling of reveal + consume + execute_carnage_atomic instructions.

## Evidence

### VRF + Carnage Transaction Flow

**Actual implementation** (scripts/vrf/lib/vrf-flow.ts:246-318):

```
TX1: Client creates Switchboard randomness account (off-chain SDK call)

TX2: commitIx + trigger_epoch_transition (bundled atomically)
  - Binds randomness account key to epoch_state.pending_randomness_account
  - Sets vrf_pending = true, taxes_confirmed = false
  - Pays bounty to triggerer
  - Emits EpochTransitionTriggered (does NOT reveal VRF bytes yet)

[~3 slots pass while oracle computes VRF off-chain]

TX3: revealIx + consume_randomness + execute_carnage_atomic (bundled atomically as v0 VersionedTransaction)
  - Oracle writes VRF bytes to randomness account (revealIx)
  - consume_randomness reads bytes, derives tax rates, applies to EpochState
  - Sets vrf_pending = false, taxes_confirmed = true
  - IF vrf_result[5] < 11: Sets carnage_pending = true + carnage_target + carnage_action
  - execute_carnage_atomic executes Carnage swap immediately (or no-ops if not triggered)
  - All state changes commit atomically in same transaction
```

### No-Op Guard (Makes Atomic Bundling Safe)

The `execute_carnage_atomic` instruction contains a no-op guard that allows it to be bundled in every TX3 without harming transactions where Carnage doesn't trigger. When `carnage_pending = false`, the instruction returns `Ok(())` immediately after account validation (~100 CU cost).

This design pattern means the crank runner can ALWAYS include `execute_carnage_atomic` in TX3, ensuring that when Carnage DOES trigger (4.3% probability), the swap executes atomically with zero MEV gap.

## Root Cause (Why This Is Not Exploitable)

The attack hypothesis assumes a **sequential** flow:
1. VRF reveals → bytes visible on-chain (including byte 7 = target faction)
2. (Gap window - attacker reads VRF bytes and computes target)
3. Protocol calls consume_randomness → sets carnage_pending flag
4. (Gap window - attacker front-runs execute_carnage with buy TX)
5. execute_carnage executes → buys target token at inflated price

The **actual** implementation uses **atomic bundling**:
1. VRF reveal + consume + execute happen in same VersionedTransaction v0
2. No observable gap between reveal and execution
3. By the time VRF bytes are visible on-chain, Carnage swap has already executed
4. The `CarnagePending` event is emitted in the SAME transaction as execution

Per Solana's runtime model, all instructions in a transaction execute atomically. Within a single transaction, all state changes (randomness account data written by oracle, EpochState updated by consume_randomness, pool reserves updated by execute_carnage_atomic) become visible to other transactions only after the entire transaction commits.

## Detailed Timeline Analysis

### Observability Timeline

| Slot | Observer Can See | Observer CANNOT See |
|------|------------------|---------------------|
| N | Old epoch state (previous tax rates, no VRF pending) | Future VRF bytes |
| N+1 | TX2 commits: vrf_pending=true, pending_randomness_account bound | VRF bytes (not revealed yet) |
| N+2-4 | Oracle computes VRF off-chain | VRF bytes (not on-chain yet) |
| N+5 | **TX3 commits atomically**: VRF bytes, carnage_pending, carnage_target, carnage_action, **AND swap result** all visible simultaneously | Nothing - all state changes are visible together |

**Key insight**: There is no slot where VRF bytes are visible but Carnage has not executed. The attacker would need to observe TX3 in the mempool, extract VRF bytes from the reveal instruction data, compute Carnage target, construct a front-running transaction, and get it included in the SAME slot BEFORE TX3. This is infeasible on Solana:

1. **Gulf Stream forwarding**: Solana validators receive transactions via UDP and process them immediately. There is no traditional mempool where pending transactions sit waiting.
2. **Transaction ordering**: Within a block, transaction ordering is determined by the leader. An attacker cannot reliably insert their transaction before a specific transaction.
3. **Atomic execution**: Even if an attacker could see TX3 before it executes, by the time TX3 commits, all three instructions have already executed. The VRF bytes and the swap result appear on-chain simultaneously.

## Attack Scenario (Infeasible)

For this attack to work, the attacker would need to:

1. **Monitor for TX3 in "mempool"**: Solana's Gulf Stream does forward transactions to validators, but the transaction is not pending in a traditional sense. By the time a validator includes the transaction in a block, it has already executed.

2. **Extract VRF bytes from reveal instruction**: The `revealIx` instruction contains the oracle signature over the VRF proof. The attacker could theoretically parse this, but:
   - The signature is over the randomness seed, not the output
   - The output is only written to the randomness account DURING execution
   - The attacker cannot predict the output before the reveal instruction executes

3. **Compute Carnage target before consume_randomness reads it**: Even if the attacker could extract VRF byte 7 from the reveal instruction data, they would need to:
   - Parse the byte (< 128 = CRIME, >= 128 = FRAUD)
   - Construct a buy transaction for the target token
   - Submit it to the SAME validator
   - Get it included in the SAME slot BEFORE TX3
   - This requires block-building level access or validator collusion

4. **Front-run with buy before execute_carnage_atomic**: Impossible - the Carnage swap executes in the same transaction as the reveal. A subsequent buy transaction would execute in a LATER slot and see the Carnage-inflated price (buying high, not profiting).

## Fallback Path MEV Window (Documented, Separate Issue)

There IS a documented MEV window, but it's in the **fallback execution path**, not the atomic path:

**Gap window:** Between TX3 (consume_randomness sets carnage_pending) and fallback `execute_carnage` call (slots 50-300).

**During this gap:**
- `carnage_pending = true` (visible on-chain)
- `carnage_target` is known (CRIME or FRAUD)
- `carnage_action` is known (Burn or Sell)
- Anyone can call the fallback `execute_carnage` instruction

**Why this gap exists:**
- The fallback path is a recovery mechanism for when the atomic execution fails (e.g., insufficient compute units, pool liquidity issues)
- The atomic path executes during slots 0-50 (CARNAGE_LOCK_SLOTS)
- After slot 50, if execution hasn't completed, the fallback becomes callable by anyone
- The fallback has a 75% slippage floor (vs 85% atomic) to account for potential manipulation

**Why this gap is acceptable:**
- The atomic path is the primary path — crank runner always attempts it first
- The fallback is only needed if atomic execution fails (rare edge case)
- The 75% slippage floor limits MEV extraction to 25% of Carnage buy amount
- MAX_CARNAGE_SWAP_LAMPORTS caps at 1000 SOL
- Maximum extractable: 25% * 1000 SOL = 250 SOL per fallback event
- Fallback frequency: ~0.1% of epochs (assuming 98% atomic success rate, 4.3% trigger rate)

**This is a separate finding** from STR-022 (atomic path prediction). The fallback MEV window is documented in context/06-oracle-data.md:472-489 as an accepted design trade-off.

## Impact Assessment

**Actual Impact:** NONE - The atomic bundling pattern prevents the hypothesized attack entirely.

**Theoretical Maximum Extractable Value (if attack were possible):**

Assume an attacker could predict Carnage target before execution:

1. **Attack pattern**: Buy target faction token before Carnage buy → Carnage buy pushes price up → Sell at inflated price
2. **Carnage buy amount**: Max 1000 SOL per trigger (MAX_CARNAGE_SWAP_LAMPORTS)
3. **Price impact**: Depends on pool liquidity. For a 100 SOL pool (thin memecoin), a 1000 SOL buy would move price ~10x (constant product). For a 10,000 SOL pool, ~1.1x.
4. **Attacker profit**: Requires front-running with ~2000 SOL to push price up 20%, then Carnage buys at inflated price receiving ~17% fewer tokens (within 85% floor), then attacker sells for ~15 SOL profit MINUS ~40 SOL LP fees = **net loss**

Even if the attack were possible, it's not economically profitable due to:
- 85% slippage floor rejects extreme manipulation
- 1% LP fee on SOL pools (charged on both buy and sell)
- Carnage buy is capped at 1000 SOL (limits total extraction)

However, this is **purely theoretical** since the attack is structurally prevented by atomic bundling.

## Remediation

**No remediation required.** The current implementation is secure against this attack vector.

**Existing defense (adequate):**
- Atomic bundling of `revealIx + consume_randomness + execute_carnage_atomic` in TX3
- No-op guard in `execute_carnage_atomic` allows safe bundling in every TX3
- Crank runner (scripts/crank/crank-runner.ts) enforces correct bundling pattern
- Anti-reroll protection prevents VRF account substitution
- 85% slippage floor on atomic path limits manipulation even if MEV gap existed

**Fallback path defense-in-depth (already implemented):**
- 75% slippage floor on fallback `execute_carnage` (slots 50-300)
- MAX_CARNAGE_SWAP_LAMPORTS caps at 1000 SOL
- Fallback is recovery mechanism, not primary path
