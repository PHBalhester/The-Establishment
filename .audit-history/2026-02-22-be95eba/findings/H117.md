# H117: Transfer Hook Has No Rate Limiting

**Strategy:** STR-117: Transfer Hook Has No Rate Limiting
**Determination:** NOT A VULNERABILITY — Design is Intentional, System Cost-Bounded
**Severity:** INFO
**Confidence:** HIGH

## Summary

**Hypothesis Claim:** The transfer hook executes on every transfer. If an attacker sends many tiny transfers, the hook executes for each, consuming CU. This could DoS other operations in the same slot.

**Finding:** The transfer hook executes on EVERY transfer (as designed by Token-2022 spec), but per-execution CU cost is negligible (~2,500 CU), and system-level safeguards make this a non-vulnerability:

1. **Per-hook execution cost:** ~2,500 CU (whitelist PDA existence check + derivation validation)
2. **Solana slot CU budget:** 1.4M maximum (mainnet 150K default per TX, but entire slot = 1.4M shared)
3. **Max concurrent transfers per slot:** ~560 (1.4M ÷ 2.5K), each consuming negligible budget
4. **Economic barrier:** Gas cost (5,000 lamports per transfer on devnet) + tax (15% on swaps) makes spam economically irrational
5. **Token-2022 runtime enforcement:** All transfers MUST pass through the hook; there is no bypass path

**Verdict:** Not exploitable. The design is correct and intentional.

## Evidence

### 1. Hook Execution Path (Code Review)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/transfer_hook.rs:77-112`

The transfer hook handler is invoked by Token-2022 runtime on EVERY `transfer_checked` call:

```rust
pub fn handler(ctx: Context<TransferHook>, amount: u64) -> Result<()> {
    // Validation 1: Zero amount check (cheapest, fail fast) — ~50 CU
    require!(amount > 0, TransferHookError::ZeroAmountTransfer);

    // Validation 2: Mint owner check (defense-in-depth) — ~100 CU
    check_mint_owner(&ctx.accounts.mint.to_account_info())?;

    // Validation 3: Transferring flag check (security) — ~200 CU
    check_is_transferring(&ctx.accounts.source_token.to_account_info())?;

    // Validation 4: Whitelist check with short-circuit — ~2,000 CU
    let source_whitelisted = is_whitelisted(
        &ctx.accounts.whitelist_source.to_account_info(),
        &ctx.accounts.source_token.key(),
    );

    if !source_whitelisted {
        // Conditional destination check only if source fails
        let dest_whitelisted = is_whitelisted(
            &ctx.accounts.whitelist_destination.to_account_info(),
            &ctx.accounts.destination_token.key(),
        );
        require!(dest_whitelisted, TransferHookError::NoWhitelistedParty);
    }

    Ok(())
}
```

**CU Breakdown:**
- **Validation 1 (amount > 0):** ~50 CU (single comparison)
- **Validation 2 (mint owner check):** ~100 CU (account info read, single comparison)
- **Validation 3 (transferring flag):** ~200 CU (account data borrow, extension parse, flag read)
- **Validation 4 (whitelist check × 2):** ~2,000 CU (two `Pubkey::find_program_address` calls + 2 key comparisons + 2 empty checks per account)

**Total per hook execution:** ~2,350 CU (conservative estimate with overhead)

### 2. Whitelist Check Cost (is_whitelisted function)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/transfer_hook.rs:166-179`

```rust
fn is_whitelisted(whitelist_pda: &AccountInfo, token_account: &Pubkey) -> bool {
    // Account must have data (exists) — ~50 CU
    if whitelist_pda.data_is_empty() {
        return false;
    }

    // Verify PDA derivation is correct — ~1,000 CU
    let (expected_pda, _bump) = Pubkey::find_program_address(
        &[WhitelistEntry::SEED_PREFIX, token_account.as_ref()],
        &crate::ID
    );

    // Single key comparison — ~50 CU
    whitelist_pda.key() == expected_pda
}
```

Per invocation: ~1,100 CU (dominated by `find_program_address`, which is a linear hash search)

This is called up to 2× per transfer (source + destination if source not whitelisted).

### 3. Slot CU Budget Constraint

**Reference:** `/Users/mlbob/Projects/Dr Fraudsworth/docs/architecture.md:38-39`

```
| Compute budget | 200K CU default / 1.4M max | Heaviest path: swap_sol_sell(FRAUD) at 122,586 CU (61% of 200K). No scaling risk. |
```

**Solana Limits:**
- **Per-transaction limit:** 200,000 CU (default), adjustable up to 1,400,000 via `SetComputeBudgetInstruction`
- **Per-slot limit (global):** 1,400,000 CU total for ALL transactions in the slot

**Attack Scenario Math:**

Attacker wishes to spam transfers to exhaust the slot CU budget:
- Each transfer: 2,500 CU (hook + base Token-2022 overhead)
- Max transfers per slot: 1,400,000 ÷ 2,500 = **560 concurrent transfers**

**Reality Check:**
- Attacker would need 560 transaction signatures in a single slot
- Each transaction requires a unique signer (or sequence number)
- Solana slots are ~400ms; validator would need to pack 560 TXs in one block
- More realistically: ~50-100 TXs per slot under normal conditions

**Verdict:** The attacker can add ~2.5K CU per transfer, but the slot budget of 1.4M constrains total DoS impact. Any spam would compete with legitimate traffic for the same 1.4M pool, not uniquely exhaust it.

### 4. Economic Barriers

**Devnet SOL Cost per Transfer:**
- Signature verification: ~1,000 lamports
- Transaction fee: ~5,000 lamports
- **Total:** ~5,000–6,000 lamports per transfer

For a meaningful DoS (100 transfers to consume 250K CU = 18% of slot budget):
- Cost: 100 × 5,000 = **500,000 lamports = 0.005 SOL**
- Devnet faucet limit: 2 SOL per day
- **Feasibility on devnet:** Theoretically possible but requires fund exhaustion

**Mainnet Cost per Transfer:**
- Current (Feb 2026): ~100–1,000 lamports depending on network load
- For 100 transfers: 10,000–100,000 lamports = 0.00001–0.0001 SOL
- **Feasibility on mainnet:** Very low cost but competitive with other users for slot space

**With Tax (swaps):**
- User swaps trigger 15% tax, reducing net trading volume per SOL
- Pure transfers (whitelist-only, e.g., bridge operations) incur no tax
- Attacker would need to sustain pure transfers at scale

**Verdict:** Cost is low but not zero. Economic barrier is weak compared to intentional DDoS attacks on other networks, but present.

### 5. Token-2022 Runtime Enforcement

**Architecture:** All three custom tokens (CRIME, FRAUD, PROFIT) have the TransferHook extension pointing to the hook program.

**Per ARCHITECTURE.md, Section 5.6:**
```
Any CRIME/FRAUD/PROFIT transfer → Token-2022 → Hook::execute
                                                    │
                                                    ├─ Source whitelisted? → Allow
                                                    ├─ Dest whitelisted? → Allow
                                                    └─ Neither? → REJECT
```

**No Bypass Paths:**
1. Direct token account manipulation is prevented by Token-2022's owner validation
2. `transfer_checked` MUST invoke the hook (enforced by Token-2022 runtime)
3. `transfer` (non-checked variant) is disabled by `RequiredMemoTransfers` extension (if configured)
4. Program-derived swaps (Tax, AMM, Staking) all use whitelisted accounts, so they pass the hook

**Verdict:** The hook cannot be bypassed. Every transfer pays the cost, but the cost is low and distributed across the entire slot.

## Impact

### Threat Model

**Who Benefits from Hook Spam?**
1. **MEV bot:** No benefit. Hook validation is deterministic (either pass or fail). Spamming transfers doesn't manipulate prices.
2. **Griever:** Low-value attack. Consuming 18% of a slot's CU budget costs 0.005 SOL on devnet, 0.00001 SOL on mainnet. Legitimate traffic is not significantly impacted.
3. **Network attacker:** The Solana network operator (validators + leader) has stronger DoS vectors (e.g., transaction flooding), rendering this obsolete.

### Realistic Impact Estimates

**Scenario 1: Devnet Spam (100 transfers)**
- Attacker cost: 0.005 SOL
- Slots affected: 2 (assuming 50 TXs/slot packing)
- CU consumed: 250,000 (18% of 1.4M budget)
- Legitimate impact: Competing traffic delayed by ~2 slots, recovers after attacker funds exhaust

**Scenario 2: Mainnet Spam (1,000 transfers over 1 hour)**
- Attacker cost: 0.00001–0.0001 SOL (negligible)
- Slots affected: ~10 (assuming 100 TXs/slot packing)
- CU consumed per slot: 2.5M ÷ 1.4M = **exceeds slot limit**, so validator drops excess TXs
- Legitimate impact: None (excess TXs don't land on-chain)

**Verdict:** The attack scales poorly. Mainnet validators reject excess-budget TXs; devnet can be spammed but at negligible cost and temporary impact.

### Security Implications

1. **Not a vulnerability in the hook:** The hook is correctly implemented. It does what Token-2022 requires.
2. **Not a protocol vulnerability:** The slot-level CU budget is enforced by the Solana runtime, not the protocol.
3. **Not exploitable for value extraction:** The attacker does not gain any economic benefit (cannot sandwich, cannot front-run, cannot manipulate prices).

**Verdict:** INFO level. Design is sound; no remediation needed.

---

## Recommendation

**No action required.** The transfer hook design correctly implements the Token-2022 TransferHook extension specification. The low per-execution cost (~2.5K CU) is unavoidable given the requirement for whitelist validation on every transfer.

### Defense-in-Depth Considerations (Optional, Future)

If future Solana hard-forks increase the per-slot CU budget significantly (e.g., to 10M+), the economic feasibility of sustained transfer spam could increase. At that point, consider:

1. **Rate limiting per account:** Skip whitelist check for frequently-used accounts (cache-based). Trades security for performance.
2. **Per-signer transfer limits:** Admin enforces a quota (e.g., max 100 transfers/signer/block). Requires state mutation and governance.
3. **Transfer batching:** Move multiple transfers into a single instruction. Reduces hook invocations. Requires client-side batching logic.

None of these are recommended now. The current design is optimal.
