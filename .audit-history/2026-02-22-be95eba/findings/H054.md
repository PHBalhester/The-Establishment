# H054 — Switchboard Randomness Account Substitution

## Determination
**NOT VULNERABLE**

## Severity
N/A (Theoretical: CRITICAL)

## Confidence
HIGH

## Summary
Investigation into whether an attacker could substitute a fake Switchboard randomness account with known values between the commit and consume phases. The protocol implements robust anti-reroll protection that prevents this attack.

## Evidence

### 1. Randomness Account Binding (Commit Phase)
**File:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:186`
```rust
// === 6. Bind randomness account (anti-reroll protection) ===
// This is critical: consume_randomness MUST use the same account
epoch_state.pending_randomness_account = ctx.accounts.randomness_account.key();
msg!(
    "Bound randomness account: {} at slot {}",
    epoch_state.pending_randomness_account,
    clock.slot
);
```

The randomness account's public key is stored in `EpochState.pending_randomness_account` during `trigger_epoch_transition`.

### 2. Anti-Reroll Verification (Consume Phase)
**File:** `programs/epoch-program/src/instructions/consume_randomness.rs:154-161`
```rust
// === 2. Anti-reroll: verify SAME randomness account that was committed ===
require!(
    ctx.accounts.randomness_account.key() == epoch_state.pending_randomness_account,
    EpochError::RandomnessAccountMismatch
);
msg!(
    "Anti-reroll verified: {} matches bound account",
    ctx.accounts.randomness_account.key()
);
```

The `consume_randomness` instruction validates that the randomness account matches the one stored during commit.

## Root Cause (Hypothetical)
If the protocol did NOT bind the randomness account during commit, an attacker could:
1. Create multiple Switchboard randomness accounts with different seeds
2. Wait for all to be revealed
3. Select the account with the most favorable tax rates/Carnage outcomes
4. Pass that account to `consume_randomness`

However, this is prevented by the binding mechanism.

## Attack Scenario (Prevented)
**Attacker's Goal:** Control tax rates and Carnage direction by using a pre-selected randomness account.

**Attack Steps (BLOCKED):**
1. Attacker calls `trigger_epoch_transition` with randomness account A
   - Account A's address is stored in `epoch_state.pending_randomness_account`
2. Attacker creates additional randomness accounts B, C, D with different seeds
3. Attacker waits for all randomness accounts to be revealed
4. Attacker analyzes revealed values and finds account C has favorable rates
5. **BLOCKED:** Attacker attempts to call `consume_randomness` with account C
   - Fails with `RandomnessAccountMismatch` error
   - Only account A (the bound account) is accepted

**Protection:**
```rust
require!(
    ctx.accounts.randomness_account.key() == epoch_state.pending_randomness_account,
    EpochError::RandomnessAccountMismatch
);
```

## Impact (If Vulnerable)
**CRITICAL** - Complete protocol compromise:

1. **Tax Rate Manipulation:**
   - Attacker selects randomness that produces low cheap-side tax rates
   - Enables profitable tax arbitrage

2. **Carnage Control:**
   - Attacker selects randomness that prevents Carnage triggers (byte 5 >= 11)
   - Or selects favorable Carnage actions/targets
   - Protects attacker's positions from burns

3. **MEV Extraction:**
   - Predictable tax rates enable frontrunning opportunities
   - Large holders can time trades around known rates

## Remediation
**Status:** ALREADY IMPLEMENTED

The protocol has robust protections:
1. Randomness account binding at commit time
2. Anti-reroll verification at consume time
3. Owner validation (Switchboard program)

**Additional Recommendations:**
1. ✅ Document anti-reroll mechanism in spec (already done in code comments)
2. ✅ Emit events with bound randomness account address for monitoring
3. Consider: Add randomness account address to `EpochTransitionTriggered` event for enhanced transparency
