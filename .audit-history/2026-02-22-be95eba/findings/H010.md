# H010: CPI Depth at Exact Runtime Limit (Potential DoS Risk)
**Strategy:** STR-010
**Determination:** POTENTIAL
**Severity:** MEDIUM
**Confidence:** HIGH

## Summary

The Carnage execution path reaches exactly CPI depth 4 on the current Solana runtime, which is the maximum limit prior to Agave 3.0. While this currently executes successfully, it represents a **latent architectural fragility** with two potential failure modes:

1. **Token-2022 runtime upgrade risk**: If Token-2022 adds any internal CPI call to the `transfer_checked` flow (e.g., for a new extension), Carnage execution would fail permanently
2. **Transfer Hook extension risk**: If the transfer hook implementation or any new Token-2022 extension adds nested CPI within the hook execution, the depth limit would be exceeded

This is **not actively exploitable** but creates a dependency on external runtime behavior outside the protocol's control. The risk is mitigated on Agave 3.0+ (depth limit 8) but remains a concern on current validators.

### Token-2022 Transfer Hook Architecture

From Solana documentation search, Token-2022's transfer hook is invoked **by the runtime during `transfer_checked`**, not via explicit CPI from the caller:

> "During transfer, Token-2022 calls into the program with the accounts specified at a well-defined program-derived address for that mint and program id. This call happens after all other transfer logic, so the accounts reflect the _end_ state of the transfer."

The hook execution is currently a **single CPI from Token-2022 to the hook program** (depth 3 → 4). However, there is no architectural guarantee that:

1. Token-2022 will never add internal CPI before invoking the hook (e.g., for a new extension requiring cross-program validation)
2. Future transfer hook extensions won't require nested CPI within the hook execution itself

### Runtime Limit History

From web search ([Helius Agave 3.0 blog](https://www.helius.dev/blog/agave-v3-0)):

> "SIMD-0268: Raise CPI Nesting Limit increases the maximum depth of Cross-Program Invocation (CPI) calls from 4 to 8. This effectively doubles the number of times a Solana program can invoke other programs within a single transaction."

**Timeline**:
- **Pre-Agave 3.0**: CPI depth limit = 4 (Dr Fraudsworth is at the limit)
- **Post-Agave 3.0**: CPI depth limit = 8 (provides headroom)

**Current Status** (as of Feb 2026): Agave 3.0 is deployed but not yet universally adopted across all validators. The protocol must assume some users are executing on depth-4 validators.

## Root Cause

The Carnage architecture chains four distinct CPI calls:

1. **Entry**: `execute_carnage_atomic` (user-invoked or bundled with `consume_randomness`)
2. **Depth 1**: Tax Program's `swap_exempt` (signed by Epoch's `carnage_signer` PDA)
3. **Depth 2**: AMM's `swap_sol_pool` (signed by Tax's `swap_authority` PDA)
4. **Depth 3**: Token-2022's `transfer_checked` (runtime call from AMM)
5. **Depth 4**: Transfer Hook's `execute` (runtime call from Token-2022)

This chain is **architecturally necessary**:
- Epoch Program holds the Carnage logic and must invoke Tax Program for tax-exempt swaps
- Tax Program must invoke AMM for actual swap execution
- AMM must invoke Token-2022 for token transfers
- Token-2022 must invoke Transfer Hook for whitelist enforcement

The code comments explicitly warn against adding any CPI to this path. However, the protocol **cannot control** whether Token-2022 or the transfer hook runtime adds internal CPI in future updates.

## Attack Scenario

This is **not an active exploit**. The potential failure modes are:

### Scenario 1: Token-2022 Runtime Change (External)

1. Token-2022 maintainers add a new extension (e.g., "TransferValidator") that requires cross-program validation during `transfer_checked`
2. The extension adds an internal CPI call within `transfer_checked` (before invoking the transfer hook)
3. Carnage swap path becomes: Epoch (0) → Tax (1) → AMM (2) → Token-2022 (3) → Validator CPI (4) → Transfer Hook (5)
4. **Depth 5 exceeds the limit** → All Carnage executions fail with `CallChainTooDeep` error
5. The protocol's core mechanic is permanently broken until redesign

### Scenario 2: Transfer Hook Nested CPI (Less Likely)

1. A new transfer hook implementation (or protocol fork) adds nested CPI within the hook execution
2. Example: Hook calls an oracle program for dynamic whitelist validation
3. Depth becomes: Epoch (0) → Tax (1) → AMM (2) → Token-2022 (3) → Hook (4) → Oracle (5)
4. Same failure as Scenario 1

### Scenario 3: Pre-Agave 3.0 Validator Execution

1. User executes Carnage on a validator still running pre-Agave 3.0 runtime
2. Depth limit is 4, protocol is at 4 → **executes successfully** (current state)
3. If Token-2022 were upgraded with Scenario 1, it would fail on pre-Agave validators even if Agave 3.0 validators support depth 8

## Impact Assessment

**Severity: MEDIUM** (downgraded from hypothesis HIGH)

**Justification**:
- **Not currently exploitable**: The protocol executes correctly on current runtime
- **Requires external change**: Failure requires Token-2022 or hook program to add CPI, which is outside the protocol's control
- **Agave 3.0 provides headroom**: The limit increase to 8 makes Scenario 1 far less likely (Token-2022 would need to add 4 nested CPI calls to break Carnage)
- **High impact if triggered**: Carnage is a core game mechanic. Permanent failure would be critical, but the likelihood is low

**Impact if triggered**:
- **Carnage permanently disabled** until protocol redesign
- **No user funds at risk** (atomic TX failure, no state corruption)
- **Game mechanic broken**: Losing faction tokens cannot be redistributed
- **Requires program redeployment**: Since all 5 programs are non-upgradeable, fixing this would require full protocol migration

```rust
    invoke(&validator_instruction, &validator_accounts)?; // Depth +1
```


## Remediation

### Option 1: Architectural Redesign (Breaking Change)

Reduce CPI depth by collapsing program boundaries:

1. **Merge Tax + AMM into single program**: Eliminate one CPI level
   - New chain: Epoch (0) → TaxAMM (1) → Token-2022 (2) → Hook (3) = **Depth 3**
   - **Trade-off**: Loses separation of concerns, increases program complexity
   - **Deployment**: Requires full protocol redeployment (all programs are non-upgradeable)

2. **Direct AMM call from Epoch**: Skip Tax Program for Carnage-only path
   - New chain: Epoch (0) → AMM (1) → Token-2022 (2) → Hook (3) = **Depth 3**
   - **Trade-off**: Duplicates swap logic, loses tax-exempt semantics
   - **Deployment**: Requires Epoch + AMM redeployment

### Option 2: Monitor + Contingency Plan (Recommended)

Since the risk is external and low-probability:

1. **Monitor Token-2022 upgrades**: Watch for any changes to `transfer_checked` implementation
2. **Monitor Agave adoption**: Track validator upgrades to depth-8 runtime
3. **Document contingency plan**: If Token-2022 adds CPI:
   - Deploy patched programs using Option 1 redesign
   - Migrate liquidity and state to new program addresses
   - Provide migration UI for users

4. **Add explicit runtime check** (informational only):
   ```rust
   // In execute_carnage_atomic, add log message
   msg!("CRITICAL: Carnage CPI depth is 4/4. Monitor Token-2022 for changes.");
   ```

### Option 3: Require Agave 3.0+ (Partial Mitigation)

Add documentation requiring Carnage execution on Agave 3.0+ validators:

- **Headroom on depth 8**: Token-2022 would need to add 4 internal CPI calls to break Carnage (unlikely)
- **Trade-off**: Excludes users on older validators (if any remain by mainnet launch)

### Recommended Path Forward

1. **Immediate**: Document the depth constraint in mainnet checklist
2. **Pre-mainnet**: Verify all target validators are running Agave 3.0+
3. **Post-mainnet**: Monitor Token-2022 GitHub for PRs adding CPI to `transfer_checked`
4. **Contingency**: Prepare Option 1 redesign as emergency fix (pre-audit code, ready to deploy)
