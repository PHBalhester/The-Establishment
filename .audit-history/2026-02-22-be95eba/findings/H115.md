# H115: Epoch Program Does Not Read Stale Tax Rates — Architecture is Correctly Designed

**Strategy:** STR-115: Epoch Program Reads Stale Tax Rate

**Determination:** HYPOTHESIS REJECTED — No vulnerability found

**Severity:** N/A (informational)

**Confidence:** HIGH (100% — confirmed by code review and architecture analysis)

---

## Summary

The hypothesis posits that when the Epoch Program triggers tax operations via CPI, it might pass stale tax rates to the Tax Program. Investigation reveals this hypothesis is **incorrect**: the actual architecture is more robust than the hypothesis assumes.

**Key Finding:** The Epoch Program does **NOT** pass tax rates as parameters to the Tax Program. Instead, the Tax Program **reads the tax rates directly from EpochState** using fresh deserialization, ensuring rates are always current.

---

## Evidence

### 1. Epoch Program Does NOT Pass Tax Rates

The Epoch Program writes tax rates to EpochState via `consume_randomness`:

**File:** `/programs/epoch-program/src/instructions/consume_randomness.rs`, lines 196-203

```rust
// === 6. Update EpochState with new tax configuration ===
epoch_state.cheap_side = tax_config.cheap_side.to_u8();
epoch_state.low_tax_bps = tax_config.low_tax_bps;
epoch_state.high_tax_bps = tax_config.high_tax_bps;
epoch_state.crime_buy_tax_bps = tax_config.crime_buy_tax_bps;
epoch_state.crime_sell_tax_bps = tax_config.crime_sell_tax_bps;
epoch_state.fraud_buy_tax_bps = tax_config.fraud_buy_tax_bps;
epoch_state.fraud_sell_tax_bps = tax_config.fraud_sell_tax_bps;
```

The tax rates are **written to EpochState, not passed as instruction arguments** to any downstream program.

### 2. Tax Program Reads Fresh Tax Rates from EpochState

When the Tax Program executes swaps, it deserializes the EpochState account and reads the current tax rates:

**File:** `/programs/tax-program/src/instructions/swap_sol_buy.rs`, lines 54-78

```rust
// 1. Read and validate EpochState
//    (CRITICAL: This prevents attackers from passing a fake EpochState with 0% tax.)

// Owner check: EpochState must be owned by Epoch Program.
let epoch_program = epoch_program_id();
require!(
    ctx.accounts.epoch_state.owner == &epoch_program,
    TaxError::InvalidEpochState
);

// Deserialize EpochState data.
let epoch_state = {
    let data = ctx.accounts.epoch_state.try_borrow_data()?;
    let mut data_slice: &[u8] = &data;
    EpochState::try_deserialize(&mut data_slice)
        .map_err(|_| error!(TaxError::InvalidEpochState))?
};

// Validate EpochState is initialized (defense-in-depth).
require!(epoch_state.initialized, TaxError::InvalidEpochState);

// Get the appropriate tax rate (is_buy = true for buy direction).
let tax_bps = epoch_state.get_tax_bps(is_crime, true);
```

**Critical Pattern:**
- The Tax Program reads the entire EpochState account from the Solana runtime
- It deserializes the fresh state data using Anchor's `try_deserialize`
- It validates the EpochState account is owned by the Epoch Program
- It then reads the tax rate using `get_tax_bps()`, which simply returns the cached derived rates from EpochState

### 3. Tax Rates Are Cached (Not Derived On-Demand)

**File:** `/programs/tax-program/src/state/epoch_state_reader.rs`, lines 67-74

```rust
pub fn get_tax_bps(&self, is_crime: bool, is_buy: bool) -> u16 {
    match (is_crime, is_buy) {
        (true, true) => self.crime_buy_tax_bps,
        (true, false) => self.crime_sell_tax_bps,
        (false, true) => self.fraud_buy_tax_bps,
        (false, false) => self.fraud_sell_tax_bps,
    }
}
```

The rates are cached in EpochState (not passed as parameters), and the Tax Program reads them on every swap invocation. This design is intentional and correct:

1. **Derive once per epoch** (in `consume_randomness`)
2. **Read many times per epoch** (in every swap)
3. **Always read from fresh state** (Solana runtime ensures latest account data)

### 4. Timeline: When Rates Are Updated vs. When They're Used

The Epoch Program follows a three-transaction VRF flow:

- **TX 1 (off-chain):** Client creates Switchboard randomness account
- **TX 2:** Client bundles Switchboard's `commitIx` + Epoch's `trigger_epoch_transition`
  - Sets `vrf_pending = true`
  - Binds randomness account for anti-reroll
  - **Tax rates are NOT updated yet**
- **TX 3:** Client bundles Switchboard's `revealIx` + Epoch's `consume_randomness`
  - Reads revealed VRF bytes
  - **Updates all 4 tax rates in EpochState** (lines 196-203)
  - Sets `taxes_confirmed = true`

**After TX 3 completes:**
- EpochState now contains the new tax rates
- Any subsequent user swap via Tax Program will read these fresh rates
- There is no timing gap where old rates are applied

### 5. All Swap Instructions Read EpochState Independently

Every tax-related swap instruction in the Tax Program performs the same pattern:

| Instruction | File | Behavior |
|-------------|------|----------|
| `swap_sol_buy` | `swap_sol_buy.rs:54-78` | Deserializes EpochState, reads fresh tax rate |
| `swap_sol_sell` | `swap_sol_sell.rs` | Same pattern |
| `swap_profit_buy` | `swap_profit_buy.rs` | Same pattern |
| `swap_profit_sell` | `swap_profit_sell.rs` | Same pattern |

The exempt swap path (`swap_exempt`) does NOT apply taxes (used by Carnage), so it skips the EpochState read entirely.

## Impact

**Positive Impact:** The actual design is MORE secure than the hypothesis assumes.

- Tax rates are never passed as parameters (eliminates parameter-tampering attack surface)
- Tax rates are always read from fresh state (eliminates stale-state risk)
- Owner validation prevents fake EpochState account substitution
- Anchor's discriminator validation prevents data corruption

---

## Architectural Rationale

This design is optimal for a multi-program system:

1. **Single Source of Truth:** EpochState is the canonical source for all tax configuration
2. **No CPI Parameter Coupling:** Tax Program doesn't depend on Epoch passing parameters
3. **Atomic Epoch Transitions:** All rates update atomically in one transaction (`consume_randomness`)
4. **Defense-in-Depth:** Multiple validation layers:
   - Owner check (EpochState must be Epoch Program)
   - Discriminator check (Anchor automatic)
   - Initialization check (prevents re-initialization)
