# H132: PRECISION Constant Is Staking-Only, Not Cross-Program

**Strategy:** STR-132: Staking PRECISION Constant Inconsistent Across Programs

**Determination:** FINDING REJECTED — Hypothesis is invalid. PRECISION is correctly isolated to the Staking program and does NOT interact with the Tax program.

**Severity:** NONE (Hypothesis invalidated)

**Confidence:** 99% (Exhaustive search completed)

## Summary

The hypothesis that "PRECISION (1e18) is defined differently in Staking vs Tax Program, causing reward computations to produce inconsistent results" is **factually incorrect**.

**Key Finding:**
- **Staking Program** defines `PRECISION: u128 = 1_000_000_000_000_000_000` (1e18) in `constants.rs:21`
- **Tax Program** does NOT define PRECISION and does NOT reference it anywhere
- **No cross-program reward calculation dependency** exists between Tax and Staking

The Staking program's PRECISION constant is used **exclusively** within the Staking program's own reward calculations, following the Synthetix/Quarry cumulative reward-per-token pattern. The Tax program performs no precision-scaled reward calculations — it simply extracts a fixed 75% of tax for staking distribution and deposits it as raw lamports to the Staking escrow via `deposit_rewards`.

### 3. Staking Program Math Uses PRECISION Consistently

**File:** `/programs/staking/src/helpers/math.rs`

Two core functions use PRECISION:

#### Function 1: `update_rewards()` (lines 36-67)
```rust
// Formula: pending = (global_cumulative - user_checkpoint) * balance / PRECISION
let pending = (user.staked_balance as u128)
    .checked_mul(reward_delta)
    .ok_or(StakingError::Overflow)?
    .checked_div(PRECISION)  // <-- PRECISION = 1e18
    .ok_or(StakingError::DivisionByZero)? as u64;
```

**Purpose:** Calculate individual user's pending rewards from cumulative reward-per-token
**PRECISION usage:** Denominator to scale back cumulative (which is scaled up by PRECISION)

#### Function 2: `add_to_cumulative()` (lines 91-127)
```rust
// Formula: reward_per_token = pending_rewards * PRECISION / total_staked
let reward_per_token = (pool.pending_rewards as u128)
    .checked_mul(PRECISION)  // <-- PRECISION = 1e18
    .ok_or(StakingError::Overflow)?
    .checked_div(pool.total_staked as u128)
    .ok_or(StakingError::DivisionByZero)?;

pool.rewards_per_token_stored = pool.rewards_per_token_stored
    .checked_add(reward_per_token)
    .ok_or(StakingError::Overflow)?;
```

**Purpose:** Calculate per-epoch reward distribution scaled by PRECISION
**PRECISION usage:** Numerator to scale up cumulative per-token before dividing by stakers

### 4. Tax Program's Tax Distribution (No PRECISION)

**File:** `/programs/tax-program/src/helpers/tax_math.rs:79-104`

```rust
pub fn split_distribution(total_tax: u64) -> Option<(u64, u64, u64)> {
    // Staking: floor(total * 75 / 100)
    let staking_u128 = total.checked_mul(75)?.checked_div(100)?;
    let staking = u64::try_from(staking_u128).ok()?;

    // Carnage: floor(total * 24 / 100)
    let carnage_u128 = total.checked_mul(24)?.checked_div(100)?;
    let carnage = u64::try_from(carnage_u128).ok()?;

    // Treasury: remainder
    let treasury = total_tax
        .checked_sub(staking)?
        .checked_sub(carnage)?;

    Some((staking, carnage, treasury))
}
```

**PRECISION:** NOT used
**Mathematics:** Simple basis-point calculation (75/100, 24/100)
**Output:** Raw lamports to be deposited as `pending_rewards` in Staking escrow

### 5. CPI Flow: Tax → Staking (No Precision Mismatch)

**Tax Program deposits tax as raw lamports:**
1. Calculate tax amount: `amount * tax_bps / 10_000`
2. Split distribution: `(staking_portion, carnage_portion, treasury_portion)`
3. Call Staking's `deposit_rewards(staking_portion)` via CPI
4. Staking receives the raw lamport amount and adds to `pending_rewards`

**Staking Program later processes it:**
1. At epoch end, calls `add_to_cumulative()`
2. Scales pending rewards by PRECISION: `pending * PRECISION / total_staked`
3. Each user claims their share: `balance * reward_delta / PRECISION`

**Result:** No inconsistency. PRECISION is applied once by Staking, not contested by Tax.

## Recommendation

**CLOSE finding as INVALID.**

The STR-132 hypothesis was overly broad and assumed a cross-program arithmetic dependency that does not exist. This is a common pattern in DeFi audits where different layers operate at different precision scales:
- **Tax Layer** (Tax Program): Simple integer arithmetic with basis points (10,000 denominator)
- **Reward Layer** (Staking Program): Cumulative reward-per-token with PRECISION scaling (1e18)

Both layers maintain their own arithmetic integrity without conflict.
