# H055: Pool Authority Can Swap Without Going Through Tax Program

**Status:** SAFE
**Severity:** N/A (not exploitable)
**Component:** AMM swap_authority validation
**Related:** Tax Program swap authority, PDA derivation

## Summary

The swap_authority PDA is the only entity allowed to call AMM::swap_sol_pool and AMM::swap_profit_pool. This investigation confirms that **only the Tax Program can derive a valid signature** for this PDA, making it impossible for another program to bypass tax collection by calling the AMM directly.

## Investigation

### Hypothesis
Could another program derive the same swap_authority PDA and call AMM::swap directly to bypass taxes?

### Code Analysis

#### 1. AMM Swap Authority Validation

Both swap instructions validate swap_authority identically:

**File:** `programs/amm/src/instructions/swap_sol_pool.rs` (lines 359-372)
**File:** `programs/amm/src/instructions/swap_profit_pool.rs` (lines 283-296)

```rust
#[account(
    seeds = [SWAP_AUTHORITY_SEED],
    bump,
    seeds::program = TAX_PROGRAM_ID,
)]
pub swap_authority: Signer<'info>,
```

**Key constraint:** `seeds::program = TAX_PROGRAM_ID`

This constraint has two critical effects:

1. **PDA Derivation Source:** The PDA is derived from the **Tax Program**, not the AMM
2. **Signature Validation:** Only the Tax Program can produce a valid `invoke_signed` signature for this PDA

#### 2. Tax Program Swap Authority Derivation

**File:** `programs/tax-program/src/instructions/swap_sol_buy.rs` (lines 217-220)

```rust
let swap_authority_seeds: &[&[u8]] = &[
    SWAP_AUTHORITY_SEED,
    &[ctx.bumps.swap_authority],
];
```

The Tax Program derives the PDA from its own program ID when signing CPI calls to the AMM.

#### 3. Constants Validation

**AMM constants:** `programs/amm/src/constants.rs` (lines 3-10)
```rust
pub const SWAP_AUTHORITY_SEED: &[u8] = b"swap_authority";
pub const TAX_PROGRAM_ID: Pubkey = pubkey!("DRjNCjt4tfTisSJXD1VrAduKbgA7KHuGPoYwxrUQN8uj");
```

**Tax Program constants:** `programs/tax-program/src/constants.rs` (lines 9-11)
```rust
pub const SWAP_AUTHORITY_SEED: &[u8] = b"swap_authority";
```

Both programs use identical seeds, but the critical difference is the **program ID** used for derivation.

### Why This Is Safe

#### PDA Derivation Mathematics

Solana PDAs are derived using:
```
PDA = findProgramAddress([seeds...], program_id)
```

For the swap_authority:
- **Seeds:** `["swap_authority"]`
- **Program ID:** Tax Program (`DRjNCjt4...`)

The resulting PDA address is:
```
swap_authority = findProgramAddress(["swap_authority"], TAX_PROGRAM_ID)
```

#### Signature Requirements

For a CPI to succeed with a PDA signer:

1. The calling program must **own** the PDA (derived from the calling program's ID)
2. The program must use `invoke_signed` with the correct seeds and bump

**Attack scenario analysis:**

If another program tried to:
```rust
// Attacker program trying to call AMM
let seeds = &[b"swap_authority", &[bump]];
invoke_signed(
    &amm_swap_instruction,
    &accounts,
    &[seeds]  // ❌ This produces PDA derived from ATTACKER_PROGRAM_ID, not TAX_PROGRAM_ID
)?;
```

The AMM would reject this because:
- AMM expects: `findProgramAddress(["swap_authority"], TAX_PROGRAM_ID)`
- Attacker provides: `findProgramAddress(["swap_authority"], ATTACKER_PROGRAM_ID)`
- These are **different addresses** (PDA derivation includes program_id in hash)

#### Anchor's `seeds::program` Constraint

The Anchor constraint:
```rust
#[account(
    seeds = [SWAP_AUTHORITY_SEED],
    bump,
    seeds::program = TAX_PROGRAM_ID,  // ← This is the critical line
)]
pub swap_authority: Signer<'info>,
```

Validates that:
1. The account is a `Signer` (has signed the transaction)
2. The account matches `findProgramAddress([SWAP_AUTHORITY_SEED], TAX_PROGRAM_ID)`

**Only the Tax Program can satisfy both requirements** because:
- Only the Tax Program can use `invoke_signed` to sign for PDAs derived from its own program ID
- Any other program's `invoke_signed` would produce a different PDA address

## Conclusion

**SAFE** - The swap_authority PDA design is cryptographically secure:

1. ✅ **PDA includes Tax Program ID in derivation** - via `seeds::program = TAX_PROGRAM_ID`
2. ✅ **Only Tax Program can sign** - only the program that owns a PDA can sign for it
3. ✅ **No collision possible** - different program IDs produce different PDAs even with identical seeds
4. ✅ **Anchor validates both seed match AND signature** - attacker cannot fake either

The `seeds::program` pattern is a standard Solana security practice for cross-program authorization. The swap_authority PDA acts as an unforgeable capability token that only the Tax Program can wield.

## Recommendation

**No action required.** This is the correct pattern for controlled cross-program invocations.

For documentation purposes, consider adding a comment in AMM explaining why `seeds::program = TAX_PROGRAM_ID` is critical:

```rust
/// SECURITY: seeds::program constraint ensures only Tax Program can sign.
/// PDAs are unique per (seeds, program_id) tuple. This prevents other
/// programs from deriving the same address even with identical seeds.
#[account(
    seeds = [SWAP_AUTHORITY_SEED],
    bump,
    seeds::program = TAX_PROGRAM_ID,
)]
pub swap_authority: Signer<'info>,
```
