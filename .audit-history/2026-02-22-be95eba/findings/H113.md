# H113: Mint Authorities Not Revoked (Infinite Mint Risk)

**Strategy:** STR-113: Mint Authority Still Active (Infinite Mint)

**Determination:** CONFIRMED

**Severity:** CRITICAL

**Confidence:** 100%

---

## Summary

The three custom token mints (CRIME, FRAUD, PROFIT) retain active mint authorities controlled by a single EOA (the deployment authority). There is no mechanism in the protocol to disable or revoke these authorities post-initialization. This allows the authority holder to mint unlimited tokens of any faction or the shared PROFIT token, directly violating the token supply assumption and crashing both trading prices and the staking reward system.

**Impact Zone:** Token supply conservation, economic security, price floors
**Affected Components:** CRIME mint, FRAUD mint, PROFIT mint (all Token-2022)
**Exploitability:** Trivial (requires only a single keypair signature)
**Detection Difficulty:** Trivial (public blockchain query of mint authority field)

## Impact

### A. Infinite Minting Attack

**Execution:**
1. Authority holder calls `mintTo` on CRIME mint with 1 billion tokens
2. Tokens appear in a vault account controlled by the attacker
3. Protocol is now insolvent (reserves < liabilities)

**Cascade Effects:**

1. **Price Collapse (immediate):**
   - Attacker swaps newly minted tokens for SOL via AMM
   - Constant-product formula: `output = reserve_out * input / (reserve_in + input)`
   - With 1B input and 1B reserve, output approaches 50% of reserve
   - CRIME/SOL pool price crashes from ~X SOL per token to ~0.000X
   - FRAUD/SOL price follows (depeg from CRIME suggests protocol failure)

2. **Staking System Collapse (within minutes):**
   - New PROFIT tokens minted to attacker's stake
   - Staking reward exchange rate is denominator-based: `rewards_per_token = pending_rewards / total_staked`
   - Attacker unstakes, claiming all pending protocol revenue
   - Legitimate stakers receive zero rewards despite legitimate tax contributions

3. **Treasury Theft (automatic):**
   - If attacker mints CRIME and sells it, 1% tax goes to treasury
   - Treasury is controlled by the same authority (mainnet uses `Pubkey::default()` per UA-001)
   - Attacker controls tax revenue

4. **Carnage System Distortion:**
   - Carnage buys at lower prices due to depressed reserves
   - `CRIME/PROFIT` and `FRAUD/PROFIT` pools used in Carnage path also collapse
   - Cross-faction arbitrage becomes trivial

### B. Supply-Based Invariant Violations

The protocol relies on implicit invariants:
- Total token supply = staked tokens + in-circulation tokens
- Pool reserves ≈ liquidity-provider-supplied tokens (no authorization to mint into reserves directly)
- `k-invariant` (constant-product) holds because input amounts are user-generated, not unlimited mints

**All three are violated** if mint authority is active.

### C. Whitelist Bypass

The transfer hook whitelist enforces protocol-approved transfers. However:
- Whitelist does NOT prevent `mintTo` operations
- Attacker can mint directly to their vault, bypassing transfer hook checks
- The attacker's vault could be whitelisted separately (they control the whitelist authority if they control the upgrade authority)

## Detailed Walkthrough

### Step 1: Authority Control Chain

```
Upgrade Authority (deployer keypair)
  ├─ Controls all program deployments
  ├─ Can re-initialize AMM AdminConfig
  └─ Implicitly controls mint creation

Mint Authority (same deployer keypair)
  ├─ Authorized to call mintTo on CRIME/FRAUD/PROFIT mints
  ├─ No on-chain constraints on minting amount or frequency
  └─ [REVOCATION MISSING] No mechanism to disable this permission
```

### Step 2: On-Chain Mint Authority Validation

When a Token-2022 mint is queried on-chain:

```
MintState {
  authority: Some(Pubkey::from_str("8kPzh...")),  // ← ACTIVE
  supply: 1_000_000_000_000_000,                   // 1B tokens @ 6 decimals
  decimals: 6,
  is_initialized: true,
  ...
}
```

The `authority` field is **non-None**, meaning `MintTo` is allowed.

To disable minting, a call to `setAuthority(None)` is required. This call is **absent** from all deployment scripts.

### Step 3: Exploit Execution

**Client-side (JavaScript):**

```javascript

```

**On-chain validation:** Token-2022 runtime checks `mint.authority == signer`. This passes. No program-level gates apply.

### Step 4: Post-Mint Protocol State

**CRIME vault balance:** 1B tokens
**CRIME/SOL pool reserves:** 100,000 SOL, 1,000,000 CRIME (initial liquidity)
**k-invariant:** `1_000_000_000_000_000 * 100_000_000_000 = 1e20` (should hold at 1e14 before mint)

**Attacker sells 500M CRIME for SOL:**

```

       = 50_000_000_000 / 500_000_000
       ≈ 100 SOL
```

**Attacker receives 100 SOL despite supplying 500M tokens.** Pool now has:
- CRIME: 1.5B (including minted tokens)
- SOL: 99,900 (drained)

**Staking impact:**

- Attacker's dead stake (1M PROFIT) and their 1B newly-staked PROFIT tokens now share the reward pool
- Reward per token drops by 1000x
- Legitimate user's pending rewards = 50K / 1001 ≈ 50 SOL (99.9% stolen)

## Recommendation

### Immediate (Devnet Testing)

1. **Disable all mint authorities post-initialization:**

   Add to `initialize.ts` after metadata initialization (Step 1, end of minting phase):

   ```typescript
   import { setAuthority } from "@solana/spl-token";

   for (const { name, kp } of mintKeypairs) {
     await setAuthority(
       connection,
       authority,           // payer + authority holder
       kp.publicKey,        // mint
       null,                // new_authority (None disables minting)
       "MintTokens",        // authority_type
       [authority],         // signers
       undefined,           // confirmOptions
       TOKEN_2022_PROGRAM_ID
     );
     log.info(`✓ Mint authority revoked: ${name}`);
   }
   ```

   This call is idempotent if run twice. Add to Step 1 of the initialization sequence.

2. **Verify post-revocation:**

   Query on-chain and confirm `mint.authority == None`:

   ```typescript

   ```

## Why This Was Missed

1. **Implicit assumption:** Mints are created once, authorities never need to be managed again.
2. **Deployment script mentality:** The initialize.ts script focuses on account creation, not authority lifecycle management.
3. **No defensive invariant check:** The protocol doesn't validate mint authority status at runtime.
4. **Devnet testing habit:** On devnet, an active mint authority is convenient for testing (re-minting, fund recovery). This pattern leaked into the production initialization script.

## False Positive Assessment

**Q: Could the devnet use case justify keeping authorities active?**

A: No. The initialize.ts script is the **production deployment script**. Devnet-specific testing should use a separate test helper (e.g., `tests/integration/helpers/protocol-init.ts`) that can preserve authorities. Keeping them active in the mainnet-bound script is a footgun.

**Q: Could a timelock on minting substitute for revocation?**

A: No. The protocol has zero timelock infrastructure. Any timelock would require on-chain governance (new program, new authority, new instructions), which doesn't exist.

**Q: Is there a freeze authority that prevents minting?**

A: No. The initialize.ts sets `freeze_authority: null` explicitly (line 260). Freeze authority is different from mint authority (it restricts transfers, not minting).
