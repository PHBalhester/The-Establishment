# H091: Carnage Sell Proceeds Combined with Buy Step

**Strategy:** STR-091: Carnage Sell Proceeds Not Added to Buy Step

**Determination:** VERIFIED FIXED

**Severity:** CRITICAL (if unfixed) / RESOLVED

**Confidence:** 100% (code inspection + event tracking)

---

## Summary

The Carnage Fund executes a sell-then-buy sequence when the Sell action is triggered. Previously, this operation had a critical bug: SOL proceeds from selling held tokens were not being combined with the tax fund SOL, causing them to be stranded and wasted. The sell proceeds would sit in the carnage_wsol account but not be used for the subsequent buy step.

**Status:** This bug has been identified and **fixed correctly** in both execution paths (`execute_carnage_atomic.rs` and `execute_carnage.rs`). The fix combines sell proceeds with tax SOL before calculating the buy amount.

### The Fix

Both files implement identical logic for the Sell path that:

1. **Measures sell proceeds** (lines 307-323 in atomic, lines 315-331 in fallback):
   ```rust
   let wsol_before = ctx.accounts.carnage_wsol.amount;
   execute_sell_swap(...)?;
   ctx.accounts.carnage_wsol.reload()?;
   let wsol_after = ctx.accounts.carnage_wsol.amount;
   sol_from_sale = wsol_after
       .checked_sub(wsol_before)
       .ok_or(EpochError::Overflow)?;
   ```

2. **Calculates available tax SOL** (lines 352-356 in atomic, lines 360-364 in fallback):
   ```rust
   let sol_balance = ctx.accounts.sol_vault.lamports();
   let rent = Rent::get()?;
   let rent_exempt_min = rent.minimum_balance(0);
   let available_sol = sol_balance.saturating_sub(rent_exempt_min);
   let swap_amount = std::cmp::min(available_sol, MAX_CARNAGE_SWAP_LAMPORTS);
   ```

3. **Combines both amounts for the buy step** (lines 358-366 in atomic, lines 366-374 in fallback):
   ```rust
   let total_buy_amount = std::cmp::min(
       swap_amount.checked_add(sol_from_sale).ok_or(EpochError::Overflow)?,
       MAX_CARNAGE_SWAP_LAMPORTS,
   );
   // Only wrap the portion not already in carnage_wsol from the sell
   let wrap_amount = total_buy_amount.saturating_sub(sol_from_sale);
   ```

4. **Wraps only the new SOL** (lines 381-386 in atomic, lines 389-394 in fallback):
   ```rust
   if wrap_amount > 0 {
       let sol_vault_bump = ctx.bumps.sol_vault;
       wrap_sol_to_wsol(&ctx, wrap_amount, sol_vault_bump)?;
   }
   ```
   The sell proceeds already sit in `carnage_wsol` as WSOL, so they don't need to be wrapped again.

5. **Executes buy with combined amount** (line 396 in atomic, line 404 in fallback):
   ```rust
   execute_buy_swap(&ctx, total_buy_amount, target, carnage_signer_seeds, buy_hook_accounts)?;
   ```

6. **Emits event with both values** (lines 478-487 in atomic, lines 484-493 in fallback):
   ```rust
   emit!(CarnageExecuted {
       epoch: current_epoch,
       action: action.to_u8(),
       target: target.to_u8(),
       sol_spent: total_buy_amount,
       tokens_bought,
       tokens_burned,
       sol_from_sale,          // ← Tracked in event
       atomic: true,
   });
   ```

## Impact

### If Bug Had Persisted
The unfixed bug would have caused:
- **Economic loss**: SOL proceeds from selling held tokens would be stranded in `carnage_wsol` and not deployed
- **Reduced buy volume**: Each Carnage execution would spend only tax SOL, not tax SOL + sale proceeds
- **Protocol inefficiency**: The Carnage Fund would accumulate larger balances but execute smaller buys, wasting the opportunity to amplify market impact
- **Compounding over time**: This would reduce the deflationary pressure of burns and support for token price

### With Fix Applied
- **Proceeds are fully utilized**: Sell proceeds are combined with tax SOL for maximum buy impact
- **Single wrapping call**: Only new SOL needs to be wrapped (sell proceeds already in WSOL)
- **Slippage check applies to combined amount**: The 85% (atomic) or 75% (fallback) slippage floor protects the entire combined buy
- **Event transparency**: Every Carnage execution transparently reports `sol_from_sale`, enabling off-chain monitoring

## Why This Matters

The Carnage mechanism is the **only consistent buyer** in the protocol. When Carnage is triggered:
1. It buys faction tokens at market-driven prices
2. It burns or holds those tokens
3. It creates deflationary pressure and demand support

The sell-then-buy pattern is essential for Carnage rotation:
- **Epoch N**: Buy CRIME, hold
- **Epoch N+1**: Sell CRIME, buy FRAUD
- **Epoch N+2**: Sell FRAUD, buy CRIME

If proceeds aren't combined, each cycle loses value: Sell CRIME for SOL, then buy FRAUD with only tax SOL, leaving sold-CRIME-proceeds stranded. This breaks the rotation efficiency and wastes the opportunity to deploy more capital.

The fix ensures that the Carnage Fund operates at **maximum efficiency**, deploying all available SOL (both accumulated tax and sale proceeds) on each buy operation.
