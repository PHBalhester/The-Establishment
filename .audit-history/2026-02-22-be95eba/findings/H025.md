# H025: Whitelist Allows Wrong Token Account for Hook Resolution

**Category:** Transfer Hook / Token-2022
**Severity:** NOT VULNERABLE
**Status:** CLEARED
**Auditor:** Security Hypothesis Investigator
**Date:** 2026-02-22

---

## Summary

**Hypothesis:** Transfer hook resolves `extra_account_meta_list` PDA per mint. If the wrong `meta_list` PDA is passed, the hook might check the wrong whitelist, allowing whitelist bypass across different token types (CRIME/FRAUD/PROFIT).

**Result:** NOT VULNERABLE. Token-2022 runtime enforces the correct `extra_account_meta_list` PDA derivation, and whitelist PDAs do NOT include the mint in their seeds, making cross-token whitelist bypass impossible.

---

## Investigation

### Hypothesis Basis

The hypothesis was based on exploit patterns:
- **EP-042 (Arbitrary CPI Program Substitution)**: CPI target program ID not validated
- **EP-053**: Account substitution patterns

The concern was that if:
1. `extra_account_meta_list` PDA can be substituted with one from a different mint
2. Whitelist PDAs don't include the mint in their seeds
3. Then an attacker could whitelist an account for CRIME and use it to transfer FRAUD

### Code Analysis

#### 1. ExtraAccountMetaList PDA Derivation

**File:** `/programs/transfer-hook/src/instructions/initialize_extra_account_meta_list.rs`

The `extra_account_meta_list` PDA is derived as:
```rust
seeds = [b"extra-account-metas", mint.key().as_ref()]
```

This PDA is **mint-specific**. Each mint (CRIME, FRAUD, PROFIT) has its own `extra_account_meta_list`.

**File:** `/programs/transfer-hook/src/instructions/transfer_hook.rs`

The hook validates this PDA:
```rust
#[account(
    seeds = [b"extra-account-metas", mint.key().as_ref()],
    bump
)]
pub extra_account_meta_list: UncheckedAccount<'info>,
```

Anchor's `seeds` constraint verifies the passed account matches the derived address. If an attacker passes the `extra_account_meta_list` from CRIME when transferring FRAUD, Anchor rejects the transaction.

#### 2. Whitelist PDA Seeds

**File:** `/programs/transfer-hook/src/state/whitelist_entry.rs`

Whitelist PDAs are derived as:
```rust
seeds = [b"whitelist", token_account.as_ref()]
```

**Critical observation:** Whitelist PDAs are keyed by **token account address**, NOT by mint. This means:
- A whitelist entry for token account `ABC123...` applies to that specific token account
- The whitelist does NOT distinguish between mints

#### 3. ExtraAccountMetaList Configuration

**File:** `/programs/transfer-hook/src/instructions/initialize_extra_account_meta_list.rs`

The `ExtraAccountMetaList` for each mint is configured with:
```rust
let extra_metas = vec![
    // Source whitelist PDA: ["whitelist", source_token_account]
    ExtraAccountMeta::new_with_seeds(
        &[
            Seed::Literal { bytes: b"whitelist".to_vec() },
            Seed::AccountKey { index: 0 },  // source_token_account
        ],
        false, false,
    )?,
    // Destination whitelist PDA: ["whitelist", destination_token_account]
    ExtraAccountMeta::new_with_seeds(
        &[
            Seed::Literal { bytes: b"whitelist".to_vec() },
            Seed::AccountKey { index: 2 },  // destination_token_account
        ],
        false, false,
    )?,
];
```

Both CRIME, FRAUD, and PROFIT mints use the **same** `ExtraAccountMetaList` configuration. They all derive whitelist PDAs from `["whitelist", token_account]`.

### Token-2022 Enforcement

Per Solana documentation ([source](https://www.solana-program.com/docs/transfer-hook-interface/configuring-extra-accounts)):

> The validation account is a PDA off of the transfer hook program derived from the following seeds:
> `"extra-account-metas" + <mint-address>`

Token-2022 **automatically derives** the `extra_account_meta_list` address from the mint being transferred. The runtime:
1. Reads the mint's `TransferHook` extension to get the hook program ID
2. Derives `extra_account_meta_list = PDA(["extra-account-metas", mint], hook_program)`
3. Fetches the account data at that address
4. Resolves additional accounts per the configurations stored in that PDA
5. Calls the hook with the resolved accounts

**Key security property:** The client cannot substitute a different `extra_account_meta_list`. Token-2022 derives it deterministically from the mint address.

### Attack Scenario Analysis

**Scenario:** Attacker whitelists their CRIME vault, then tries to transfer FRAUD using the CRIME `extra_account_meta_list`.

**Step-by-step failure:**

1. **Client builds transfer:** Attacker calls `transfer_checked(fraud_mint, ...)`
2. **Token-2022 derives meta list:** Runtime computes `PDA(["extra-account-metas", fraud_mint], hook_program)`
3. **Token-2022 resolves accounts:** Runtime reads FRAUD's `extra_account_meta_list`, which specifies `["whitelist", source_token_account]` and `["whitelist", dest_token_account]`
4. **Token-2022 invokes hook:** Calls `hook::execute` with:
   - Account 0: source_token (FRAUD token account)
   - Account 1: mint (FRAUD mint)
   - Account 2: destination_token (FRAUD token account)
   - Account 3: owner
   - Account 4: FRAUD's `extra_account_meta_list`
   - Account 5: Whitelist PDA for source FRAUD token account
   - Account 6: Whitelist PDA for destination FRAUD token account
5. **Hook validates:** `transfer_hook.rs` checks:
   - Mint owner is Token-2022 ✓
   - `extra_account_meta_list` seeds match `["extra-account-metas", fraud_mint]` ✓
   - Whitelist PDA existence for source or destination FRAUD token account ✗ (attacker only whitelisted CRIME vault)

**Result:** Transfer fails. The attacker cannot use a CRIME whitelist entry for FRAUD transfers because:
- Token-2022 enforces the correct `extra_account_meta_list` (per FRAUD mint)
- The whitelist PDAs are derived from the FRAUD token account addresses, which are different from CRIME token account addresses

### Cross-Token Whitelist Sharing (Design Feature)

**Important clarification:** The whitelist PDAs are keyed by token account address, not mint. This means:
- If an account holds CRIME at address `ABC123...`, the whitelist entry is for `PDA(["whitelist", ABC123])`
- If the same account later holds FRAUD at address `ABC123...` (impossible on Solana—token accounts are mint-specific)
- The whitelist would apply

However, **Token-2022 token accounts are mint-specific**. A token account for CRIME (mint A) cannot hold FRAUD (mint B). The mint is stored in the token account state and validated during transfers.

Therefore, even though whitelist PDAs don't include the mint, there is no cross-mint whitelist bypass because token accounts themselves are mint-specific.

### Protocol-Specific Whitelisting

In Dr Fraudsworth, the whitelisted accounts are:
- AMM pool vaults (one vault per mint)
- Staking vault (PROFIT only)
- Carnage vaults (one per faction)
- Tax Program intermediary accounts (one per mint)

Each of these is a **distinct token account** tied to a specific mint. The whitelist entries are:
- `PDA(["whitelist", crime_pool_vault_b])` — for CRIME pool vault
- `PDA(["whitelist", fraud_pool_vault_b])` — for FRAUD pool vault
- `PDA(["whitelist", profit_pool_vault_a])` — for PROFIT in CRIME/PROFIT pool
- `PDA(["whitelist", profit_pool_vault_b])` — for PROFIT in FRAUD/PROFIT pool
- etc.

There is no overlap. A CRIME vault whitelist entry does not apply to FRAUD vaults because they are different token accounts.

## Conclusion

**Severity:** NOT VULNERABLE

**Reason:** Token-2022 runtime enforces deterministic `extra_account_meta_list` derivation from the mint address. Clients cannot substitute a different mint's meta list. While whitelist PDAs don't include the mint in their seeds, token accounts are mint-specific at the Token-2022 level, preventing cross-token whitelist reuse.

**Architecture observation:** The whitelist design is mint-agnostic by intention—it whitelists token accounts, not mints. This allows a single whitelist entry to cover an account across operations (stake, swap, claim) without mint-specific entries. Token-2022's mint-to-account binding provides the security boundary.

**Verification:**
- ✓ `extra_account_meta_list` includes mint in seeds: `["extra-account-metas", mint]`
- ✓ Token-2022 derives meta list from mint (not client-controlled)
- ✓ Anchor validates meta list PDA derivation in `transfer_hook.rs:41-44`
- ✓ Token accounts are mint-specific (Token-2022 invariant)
- ✓ Protocol whitelists distinct token accounts per mint

**Recommendation:** No changes required. The design is secure. Consider documenting that whitelist PDAs are token-account-specific (not mint-specific) for future maintainers.
