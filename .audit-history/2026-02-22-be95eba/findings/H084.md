# H084: Carnage Fund Accumulation Without Spending (No Carnage Trigger)

**Strategy:** STR-084
**Determination:** POTENTIAL (Medium Risk)
**Severity:** MEDIUM
**Confidence:** HIGH

## Summary

The Carnage Fund receives 24% of all protocol trading tax but only spends when a VRF-driven trigger event occurs (~4.3% probability per epoch). If Carnage is never triggered (e.g., due to VRF oracle failures, network issues, or extended unlucky streaks), the fund grows unbounded. When finally triggered, the massive accumulated fund executes a single buy/sell/burn operation up to `MAX_CARNAGE_SWAP_LAMPORTS` (1000 SOL). The issue is not that the fund grows—that's intentional—but rather that the protocol lacks mechanisms to:

1. Cap maximum fund growth to bounded levels
2. Force spending or recovery if fund exceeds reasonable bounds
3. Gracefully handle the case where accumulated fund far exceeds available SOL in pools

The investigation reveals the fund IS bounded by the 1000 SOL per-swap limit, but accumulation can still reach economically-significant levels that may exceed pool liquidity or create extreme market impact when finally deployed.

## Evidence

### 1. Accumulation Mechanism (Verified)

**File:** `programs/tax-program/src/helpers/tax_math.rs:79-103`

The tax split function distributes:
```rust
staking = floor(total_tax * 75 / 100)
carnage = floor(total_tax * 24 / 100)
treasury = total_tax - staking - carnage
```

Every trade contributes 24% of tax to Carnage. With 1-14% protocol tax and potentially high trading volume, the fund accumulates:

**Example accumulation scenario:**
- Daily volume: $1M equivalent (~1e15 lamports)
- Average tax rate: 7.5%
- Daily tax: 75B lamports
- Daily carnage allocation: 18B lamports (~0.018 SOL)
- Monthly carnage: ~540B lamports (~0.54 SOL)
- **Annual carnage accumulation: ~6.5 SOL** (at baseline volume)

At 10x volume or higher tax rates, annual accumulation could reach **65+ SOL**, approaching the 1000 SOL per-swap limit within 15-20 years of continuous operation.

### 2. Spending Trigger (VRF-Dependent)

**File:** `programs/epoch-program/src/constants.rs:140-146`

```rust
pub const CARNAGE_TRIGGER_THRESHOLD: u8 = 11;      // ~4.3% per epoch
pub const CARNAGE_SELL_THRESHOLD: u8 = 5;          // 2% probability
pub const MAX_CARNAGE_SWAP_LAMPORTS: u64 = 1_000_000_000_000;  // 1000 SOL
pub const SLOTS_PER_EPOCH: u64 = 750 / 4500;       // devnet/mainnet
```

Carnage spending is triggered by:
- VRF byte 5 < 11: Carnage pending (11/256 ≈ 4.3% per epoch)
- VRF byte 6 < 5: Sell action (5/256 ≈ 2% per epoch)
- Otherwise: Burn action (98% of triggers)

**Critical vulnerability:** If VRF fails or oracle is unavailable, Carnage is never triggered. The fund grows indefinitely.

### 3. Spending Limit (per Execution)

**File:** `programs/epoch-program/src/instructions/execute_carnage_atomic.rs:27-31, 400-450`

```rust
const MAX_CARNAGE_SWAP_LAMPORTS: u64 = 1_000_000_000_000;  // 1000 SOL limit
```

Each Carnage execution is capped at 1000 SOL. The handler respects this:
- Read available SOL in vault: `available = sol_vault.lamports() - rent_min`
- Cap swap amount: `swap_amount = min(available, MAX_CARNAGE_SWAP_LAMPORTS)`
- Estimate output with 85% slippage floor for atomic path

**Key limitation:** The 1000 SOL cap is PER EXECUTION, not per epoch. If accumulated fund exceeds 1000 SOL, it takes multiple epochs to spend.

### 4. No Upper Bound or Recovery Valve

**Files searched:**
- `programs/epoch-program/src/instructions/*.rs`
- `programs/epoch-program/src/state/carnage_fund_state.rs`
- `programs/epoch-program/src/constants.rs`

**Finding:** There is NO:
- Maximum fund balance constant
- Alarm or emergency spending trigger if fund exceeds X SOL
- Forced liquidation if fund grows too large
- Admin override to forcibly spend accumulated funds
- Recovery mechanism if VRF fails for N epochs

The only limit is physical: the rent-exempt minimum prevents spending below ~0.89 SOL on devnet.

### 5. Pool Liquidity Risk

**From 09-economic-model.md (Section 3.5):**

Carnage buy amounts up to 1000 SOL could exceed the depth of initial pools. For a memecoin:
- Initial CRIME/SOL pool: typically 100-500 SOL equivalent
- Max Carnage buy: 1000 SOL
- **Result:** Single Carnage execution could attempt to acquire more tokens than available, even with 85% slippage floor

The 85% floor prevents price collapse but doesn't prevent the operation from failing if pool depth is insufficient. The fallback path (75% floor) is more lenient but still fails if slippage exceeds available liquidity.

### 6. Accumulation Timeline (Realistic)

Based on the economic model from 09-economic-model.md:

**Baseline scenario (devnet):**
- Volume: $10K/day at 7.5% average tax
- Daily carnage: $75
- Annual accumulation: $27,375
- In lamports: ~274 SOL/year
- **Time to reach 1000 SOL: 3.6 years**

**High-volume scenario:**
- Volume: $100K/day at 10% tax
- Daily carnage: $2,400
- Annual accumulation: $876K
- In lamports: ~8,760 SOL/year
- **Time to reach 1000 SOL: ~5 weeks**

In either scenario, if VRF fails during this accumulation window, the fund grows unchecked.

### 7. VRF Failure Likelihood

**From MEMORY.md and context files:**

VRF failures are documented:
- Gateway rotation doesn't work (forces retry on default gateway)
- Timeout recovery required after 300 slots
- Oracle assignment is per-randomness-account; switching oracles fails
- Switchboard outages (though rare) can last days

**Mitigation in place:** `retry_epoch_vrf` allows recovery after VRF_TIMEOUT_SLOTS, but if the oracle remains unavailable, Carnage is never triggered during the outage.

## Impact

### Direct Impact: Market Volatility

When accumulated Carnage finally executes:
- **Fund size:** 50+ SOL (realistic 1-2 year accumulation)
- **Market impact:** Potentially 50-100% of initial pool liquidity
- **Result:** Extreme slippage, possible price crash

Example: If Carnage holds 100 SOL but FRAUD/SOL pool has only 150 SOL reserve:
- Carnage buys 100 SOL worth of FRAUD tokens
- At constant-product, this pushes price up ~3x
- Slippage check (85%) caps acceptable impact
- If actual impact > 85%, execution fails, and bounty system prevents retry

### Secondary Impact: Staking Ecosystem

The Carnage Fund is designed to:
1. Buy tokens (create demand)
2. Burn 98% (reduce supply)
3. Create deflationary pressure

If Carnage is idle for extended periods:
- **No demand support** → Token prices decline
- **No burning** → Supply remains high
- **Staking yield collapse** (tied to trading volume, not Carnage action directly)

### Tertiary Impact: Protocol Incentives

Users who deposit into Carnage-based ecosystem expect:
- Regular buy support
- Regular supply reduction
- Predictable market interventions

If VRF is down for weeks/months:
- No buy support
- No burning
- User confidence deteriorates
- "Carnage is broken" narrative emerges

## Scenarios

### Scenario 1: Extended VRF Outage (5-7 days)

1. Switchboard oracle for current randomness account becomes unavailable
2. VRF timeout allows retry after 300 slots (~2 minutes devnet)
3. Fresh randomness created, new oracle assigned
4. If new oracle also unavailable, cycle repeats
5. **Result:** Carnage never triggers for 5-7 days. Fund accumulates 0.1-1 SOL.
6. **When oracle returns:** Massive execute triggers, burns/sells accumulated fund
7. **Risk:** Moderate. Accumulation is small; market impact is contained.

### Scenario 2: Protocol Pause (Network Rollback/Restart)

1. Solana network experiences disruption, validators rollback state
2. Epoch clock resets or skips; VRF state corrupted
3. VRF timeout recovery is blocked because randomness account invalid
4. **Result:** Carnage stuck in "pending" state for recovery window
5. **When fixed:** Stale pending carnage expires after `CARNAGE_DEADLINE_SLOTS` (300 slots)
6. `expire_carnage` clears the flag, but fund remains unspent
7. **Risk:** Low (fund remains safe, but yield is lost). However, if cycle repeats multiple times, fund accumulates significantly.

### Scenario 3: Unlucky VRF Streaks (Statistical)

1. VRF functions normally but produces "unlucky" bytes 5-7
2. For 100 consecutive epochs, byte 5 >= 11 (never triggers)
3. **Duration:** 100 epochs * 750 slots/epoch = 75,000 slots ≈ 8.3 hours on devnet
4. **Accumulation:** 8.3 hours * 24B lamports/hour (hypothetical volume) ≈ ~200B lamports (~0.2 SOL)
5. **Trigger on epoch 101:** Probability resets; next 10-50 epochs likely contain a trigger
6. **Risk:** Very low. Statistical streaks up to 1 week are normal for 4.3% events.

### Scenario 4: Long-Term High-Volume (Years)

1. Protocol gains traction; $1M/day volume sustained
2. Carnage accumulates 8-10 SOL per year
3. After 2 years: 16-20 SOL accumulated
4. After 5 years: 40-50 SOL accumulated
5. When Carnage finally executes:
   - Buys up to 1000 SOL of target token (capped)
   - If fund = 50 SOL, executes single 50 SOL buy
   - Market impact depends on pool depth
   - 50 SOL into thin memecoin pool = 2-3x price movement
   - 85% slippage floor may reject the swap if impact exceeds 15%
6. **Risk:** Medium. Fund can grow to levels causing execution failures.

## Recommendation

### Immediate (No Code Change Required)

1. **Document accumulation expectations** in Carnage_Fund_Spec.md:
   - Provide accumulation rate formula
   - Show expected fund balance at 1yr/5yr/10yr timelines
   - Clarify that fund growth is intentional and bounded only by per-swap limit

2. **Monitor Carnage fund balance on-chain:**
   - Track `carnage_sol_vault.lamports()` in metrics
   - Alert if fund exceeds 500 SOL (warning) or 750 SOL (critical)
   - Alerts trigger operational review, not automatic action

3. **Establish VRF monitoring:**
   - Log all VRF triggers and failures
   - Track trigger probability against expected 4.3%
   - Alert if 100+ consecutive epochs without trigger
   - Alert if randomness account creation fails

### Short-Term (Phase 53 Planning)

4. **Add max fund balance constant** (optional enhancement):
   ```rust
   pub const CARNAGE_FUND_MAX_BALANCE: u64 = 500_000_000_000; // 500 SOL
   ```
   Use for monitoring only, not enforcement (prevents surprises but doesn't change behavior).

5. **Add "fund health" event logging:**
   - Emit `CarnageFundBalance` event on every `execute_carnage` showing current/max
   - Helps client-side dashboards track fund growth
   - No consensus impact (events are informational only)

6. **Enhanced `expire_carnage` flow (consider for next phase):**
   - When carnage_pending expires without executing, optionally attempt to spend remaining fund
   - Requires new instruction: `force_carnage_expiry` that executes pending action before clearing flag
   - More aggressive but ensures fund doesn't get stranded
   - Would need careful design to avoid breaking existing recovery flows

### Long-Term (Post-Mainnet)

7. **Consider forced spending mechanism** for mainnet+ production:
   - Add instruction: `spend_carnage_if_exceeds(max_balance)`
   - Permissionless, callable by anyone when fund exceeds threshold
   - Executes single buy with available SOL (up to MAX_CARNAGE_SWAP_LAMPORTS)
   - Bounds maximum fund size
   - Requires careful governance decision (could create artificial buy pressure)

8. **VRF redundancy:**
   - Evaluate fallback oracle mechanism
   - Currently: only default gateway accepted; alternatives fail
   - Future: maintain whitelist of backup oracles
   - Reduces single-point-of-failure risk on Switchboard integration

## Root Cause

The protocol correctly implements:
- Tax collection and distribution (24% to Carnage, verified)
- Per-swap spending limits (1000 SOL max, verified)
- VRF-driven spending triggers (~4.3% probability, verified)

But it lacks:
- **Upper bound on fund balance** — No maximum, only per-swap limits
- **Forced spending mechanism** — Fund can be idle indefinitely
- **Recovery valve** — No way to force spending if fund exceeds economically-reasonable bounds

This is a **design choice**, not a bug. The Carnage Fund is intentionally unbounded to allow year-over-year accumulation for large market operations. However, the lack of visibility or emergency controls creates operational risk.

## Conclusion

**Status:** POTENTIAL VULNERABILITY

The Carnage Fund accumulation without spending is not an immediate security risk because:
1. The 1000 SOL per-swap limit bounds each execution
2. 85% slippage protection prevents extreme market crashes
3. Realistic accumulation timelines are measured in years

However, it represents an **operational risk** and **incentive design weakness** because:
1. No visibility into expected accumulation rates
2. No recovery mechanism if VRF fails for extended periods
3. Fund can grow to levels exceeding pool liquidity
4. Execution failures are possible (fail-silently if slippage exceeds floor)

**Recommended action:** Document accumulation expectations and add monitoring. Consider forced spending mechanism if fund exceeds 500 SOL (mainnet phase).
