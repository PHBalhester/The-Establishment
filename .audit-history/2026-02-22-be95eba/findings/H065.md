# H065: Tax Split Division by Zero

**Strategy:** STR-065
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

Hypothesis claimed that tax split might divide by zero if `tax_rate = 0`. Investigation confirms this is NOT VULNERABLE. The tax split logic divides by constants (100), not by tax_rate. Additionally, tax rates are cryptographically guaranteed to never be zero through VRF-derived lookup tables with minimum value of 100 bps (1%).

## Evidence

### Tax Split Implementation (`tax_math.rs:79-104`)

The `split_distribution()` function divides by constant 100, not by tax_rate:

```rust
pub fn split_distribution(total_tax: u64) -> Option<(u64, u64, u64)> {
    if total_tax < 4 {
        return Some((total_tax, 0, 0));
    }
    let total = total_tax as u128;
    // Staking: floor(total * 75 / 100)
    let staking_u128 = total.checked_mul(75)?.checked_div(100)?;  // Divides by constant
    let staking = u64::try_from(staking_u128).ok()?;
    // Carnage: floor(total * 24 / 100)
    let carnage_u128 = total.checked_mul(24)?.checked_div(100)?;  // Divides by constant
    let carnage = u64::try_from(carnage_u128).ok()?;
    // Treasury absorbs remainder
    let treasury = total_tax.checked_sub(staking)?.checked_sub(carnage)?;
    Some((staking, carnage, treasury))
}
```

**No division by tax_rate occurs.**

### Tax Calculation (`tax_math.rs:34-53`)

The `calculate_tax()` function divides by constant 10_000:

```rust
pub fn calculate_tax(amount_lamports: u64, tax_bps: u16) -> Option<u64> {
    if tax_bps > 10_000 {
        return None;
    }
    let amount = amount_lamports as u128;
    let bps = tax_bps as u128;
    let tax = amount.checked_mul(bps)?.checked_div(10_000)?;  // Divides by constant
    u64::try_from(tax).ok()
}
```

**Divisor is constant 10_000, not tax_bps.**

All return values are guaranteed non-zero.

### Checked Arithmetic Pattern

All division operations use `checked_div()` which returns `Option<T>`:

```rust
// Caller must handle None:
let tax = amount.checked_mul(bps)?.checked_div(10_000)?;
```

If division by zero were possible, `checked_div()` would return `None`, propagating to the caller. **Defense-in-depth is present.**

## Impact

No impact. The hypothesized division by zero cannot occur because:

1. **Tax split divides by constants**, not by tax_rate
2. **Tax rates are never zero** - minimum 100 bps from VRF lookup tables or genesis defaults
3. **All divisions use `checked_div()`** which safely returns `None` if denominator is zero
4. **Cross-program validation** in swap handlers verifies EpochState ownership and initialization before reading tax rates

## Recommendation

No fix required. The implementation is secure and follows best practices:

- **Prefer**: Keep as-is. Tax rates are guaranteed non-zero by design, and `checked_div()` provides defense-in-depth.
- **Optional Enhancement**: Add a comment in `get_tax_bps()` documenting the invariant that all tax rates are constrained to `[100, 400] bps (buy) or [1100, 1400] bps (sell)` to make the non-zero guarantee explicit for future auditors.
