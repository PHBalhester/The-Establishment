# H053: Tax Rate Clamping Edge Case

**Strategy:** STR-053
**Determination:** NOT VULNERABLE
**Severity:** INFO
**Confidence:** HIGH

## Summary

The hypothesis presumed a clamping mechanism that constrains VRF-derived tax rates to a [500, 2500] bps range using `<` vs `<=` comparisons, potentially making boundary values (500 or 2500 bps) unreachable or doubled. However, investigation reveals **no such clamping mechanism exists** in the current implementation.

The tax derivation system uses **discrete lookup tables** with modulo-4 indexing, producing only pre-defined rates: LOW_RATES `[100, 200, 300, 400]` bps and HIGH_RATES `[1100, 1200, 1300, 1400]` bps. The constants `MIN_TAX_BPS`, `MAX_TAX_BPS`, and `TOTAL_TAX_BPS` referenced in the strategy do not exist in the codebase. Boundary-exact values of 500 or 2500 bps are **mathematically impossible** to produce.

## Root Cause

**Hypothesis invalidated by architecture.** The strategy assumed a continuous VRF-to-rate transformation (e.g., `vrf_modulo → raw_rate → clamp(min, max)`) where boundary handling would matter. The actual implementation uses a **discrete two-stage lookup**:

1. **VRF byte → index**: `vrf_byte % 4` produces index ∈ {0, 1, 2, 3}
2. **Index → rate**: `LOW_RATES[index]` or `HIGH_RATES[index]` yields a pre-defined constant

This design eliminates:
- **Arithmetic drift**: No addition/subtraction that could produce off-by-one errors
- **Boundary ambiguity**: No continuous range requiring `<=` vs `<` decisions
- **Intermediate values**: Only 8 total possible tax rates (4 low + 4 high)

The Phase 37 refactor (documented in lines 12-18) replaced the legacy shared-byte system with independent per-token magnitude rolls, but retained the discrete lookup approach that inherently prevents boundary-clamping bugs.

**None.** The strategy correctly identified a vulnerability pattern (EP-017 Off-by-One precedent) but the codebase does not implement the vulnerable pattern. ### Tax Rate Reachability Verification
Comprehensive unit tests (lines 170-249) verify all discrete rates are reachable:
- ✅ All 4 CRIME low rates (100, 200, 300, 400 bps) reachable via `vrf_result[1] % 4`
- ✅ All 4 CRIME high rates (1100, 1200, 1300, 1400 bps) reachable via `vrf_result[2] % 4`
- ✅ All 4 FRAUD low rates (100, 200, 300, 400 bps) reachable via `vrf_result[3] % 4`
- ✅ All 4 FRAUD high rates (1100, 1200, 1300, 1400 bps) reachable via `vrf_result[4] % 4`

**Statistical properties:**
- Each low rate has **25% probability** (uniform over 256 possible byte values)
- Each high rate has **25% probability**
- No modulo bias (256 ÷ 4 = 64 exactly, per EP-020 precedent check)
- Boundary values 500/2500 bps have **0% probability** (not in lookup tables)

**Investigator Notes:**

The hypothesis was based on outdated assumptions about the tax derivation mechanism. The constants `MIN_TAX_BPS = 500`, `MAX_TAX_BPS = 2500`, and `TOTAL_TAX_BPS = 3000` appear to reference a legacy continuous-range design that may have existed prior to Phase 37 or in a different codebase variant.

The current implementation demonstrates **defense-in-depth through architectural simplification**: by restricting the solution space to 8 discrete values rather than a continuous range, the code eliminates entire vulnerability classes (clamping bugs, floating-point precision, arithmetic overflow in boundary calculations) at the expense of slightly reduced granularity (300 bps between levels vs continuous). This is a well-reasoned security trade-off for a DeFi protocol where predictable behavior is paramount.
