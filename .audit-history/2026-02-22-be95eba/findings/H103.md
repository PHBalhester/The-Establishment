# H103: Pool k-Value Not Stored — Recomputed Each Swap

**Strategy:** STR-103: Pool k-Value Not Stored (Recomputed Each Swap) — Hypothesis: If k is recomputed as `reserve_a * reserve_b` at swap start rather than stored from initialization, a rounding error in reserves propagates to k.

## Summary

The Dr Fraudsworth AMM does **not store the k-invariant value** in the PoolState struct. Instead, k is recomputed from current reserves during every swap verification. This is not a vulnerability because:

1. **Computation precision is identical for both k values** — Both k_before and k_after are computed using u128 multiplication from the same u64 reserve values.
2. **No rounding error propagation** — The k-invariant check compares two values computed at the same precision level: `k_after >= k_before` where both are u128.
3. **Architectural choice is correct** — Not storing k eliminates a secondary state variable that would need to be kept in sync with reserves, reducing attack surface.

```rust
pub fn verify_k_invariant(
    reserve_in_before: u64,
    reserve_out_before: u64,
    reserve_in_after: u64,
    reserve_out_after: u64,
) -> Option<bool> {
    let k_before = (reserve_in_before as u128)
        .checked_mul(reserve_out_before as u128)?;
    let k_after = (reserve_in_after as u128)
        .checked_mul(reserve_out_after as u128)?;
    Some(k_after >= k_before)
}
```

**Analysis:**
- k_before is computed fresh from `reserve_in_before` and `reserve_out_before` using u128 multiplication
- k_after is computed fresh from `reserve_in_after` and `reserve_out_after` using the same u128 multiplication
- Both multiplications are lossless (u64 × u64 fits exactly in u128: (2^64-1)^2 < 2^128-1)
- Comparison is `>=`, correctly allowing k to increase (due to fee accumulation) or stay equal

## Detailed Analysis: Why Recomputing k Is Not a Vulnerability

### 1. Precision Equivalence

The hypothesis assumes that computing k from reserves introduces rounding error. However:

- **u64 reserves are exact values** — They represent the actual token counts in vault accounts. No rounding.
- **u128 multiplication is lossless** — Two 64-bit values multiplied fit exactly in 128 bits (no truncation).
- **Both k_before and k_after use identical computation** — Same precision, same path, so any "rounding" (none actually occurs) affects both equally.

Example: If `reserve_a = 1,000,000,000` and `reserve_b = 999,999,999`:
- `k_before = 1_000_000_000 × 999_999_999 = 999_999_999_000_000_000` (u128, exact)
- After swap: `reserve_a = 1,000,001,000`, `reserve_b = 999,998,999`
- `k_after = 1_000_001_000 × 999_998_999 = 999_999_999_000_000_000` (u128, exact)
- Both values are computed with identical precision (u128), so `k_after >= k_before` is a valid invariant check.

### 2. Stored k Would Be Fragile

If k were stored at initialization, it would need to be:

- Recomputed and cached on every liquidity add/remove operation
- Verified after every external liquidity modification
- Protected against deserialization attacks (someone manually writing a stale k value to disk)

Not storing k is the correct security pattern — reduce attack surface by eliminating secondary state.

### 3. Constant-Product Formula Is Self-Verifying

The k-invariant is a **mathematical property**, not an **external input**. For any valid constant-product swap:

```
k_before = reserve_in_before × reserve_out_before
k_after = reserve_in_after × reserve_out_after

where: reserve_in_after = reserve_in_before + amount_in
       reserve_out_after = reserve_out_before - amount_out
       amount_out = (reserve_out_before × effective_input) / (reserve_in_before + effective_input)
```

If the math is correct (which it is, verified by proptest), then k_after >= k_before is guaranteed. There is no rounding error in reserves themselves — they are exact u64 values representing actual token balances.

### 4. No Initialization Failure Path

The hypothesis concerns rounding error "propagating from initialization." Analysis:

- **Pool initialization** (programs/amm/src/instructions/initialize_pool.rs): Sets `reserve_a` and `reserve_b` to the values provided by the pool creator.
- **No k is stored**, so no k initialization happens.
- **First swap verification**: Calls `verify_k_invariant()` with the initial reserves. Since the pool hasn't executed any swaps, k typically equals k_before (first swap may increase k due to fees).

There is no "initialization misalignment" because there is no stored k to be misaligned.
