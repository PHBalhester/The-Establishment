# STR-034: Missing Explicit Mint Validation in Staking Unstake

**Severity:** INFO
**Status:** Mitigated by Design
**Category:** Defense-in-Depth
**Subsystem:** Staking Program

---

## Summary

The staking program's `unstake` instruction does not explicitly validate that the `profit_mint` account matches a hardcoded constant. However, this does NOT create a practical vulnerability because the `stake_vault` PDA is initialized with a specific mint during deployment, and Solana token accounts cannot have their mint changed after creation. The constraint `token::mint = profit_mint` implicitly enforces that only the correct mint can be used.

---

## Vulnerability Details

### Location
- **File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/staking/src/instructions/unstake.rs`
- **Lines:** 66-74, 86

### Current Implementation

```rust
/// Stake vault PDA (source for unstaked tokens).
#[account(
    mut,
    token::mint = profit_mint,
    token::authority = stake_pool,
    token::token_program = token_program,
    seeds = [STAKE_VAULT_SEED],
    bump,
)]
pub stake_vault: InterfaceAccount<'info, TokenAccount>,

/// PROFIT token mint.
pub profit_mint: InterfaceAccount<'info, Mint>,
```

The `profit_mint` account has NO address constraint. It only validates:
- It's a valid Mint account (via `InterfaceAccount<'info, Mint>`)
- The stake_vault's mint field matches this mint (via `token::mint = profit_mint`)

### Why This Is NOT Exploitable

**Implicit Protection via Token Account Invariants:**

1. **Initialization Sets Mint:** The `stake_vault` is created during `initialize_stake_pool` with `init` constraint:
   ```rust
   #[account(
       init,
       payer = authority,
       token::mint = profit_mint,  // Mint is set at creation
       seeds = [STAKE_VAULT_SEED],
       bump,
   )]
   pub stake_vault: InterfaceAccount<'info, TokenAccount>,
   ```

2. **Mint is Immutable:** Solana token accounts store their mint in the account data. Once created, this field CANNOT be changed. The `stake_vault` at PDA `["stake_vault"]` will ALWAYS have the mint that was specified during initialization.

3. **Constraint Provides Implicit Validation:** During `unstake`, the constraint `token::mint = profit_mint` validates that:
   - The passed `profit_mint` matches the vault's stored mint
   - Since the vault's mint is the REAL PROFIT mint (set at initialization), only passing the REAL PROFIT mint satisfies this constraint
   - Passing a fake mint causes the constraint to fail: `Account stake_vault mint does not match expected`

**Attack Scenarios Considered:**

| Attack | Feasibility | Why It Fails |
|--------|-------------|--------------|
| Pass fake mint during unstake | ❌ BLOCKED | Constraint `token::mint = profit_mint` fails because vault's mint != fake mint |
| Create second stake_vault with fake mint | ❌ BLOCKED | PDA collision - only ONE account can exist at `["stake_vault"]` |
| Initialize stake_vault with fake mint | ⚠️ POSSIBLE | This is an initialization front-running issue (already documented as TE-008) |

### Comparison with Other Programs

**Tax Program** (has explicit validation):
```rust
pub fn profit_mint() -> Pubkey {
    Pubkey::from_str("8y7Mati78NNAn6YfGqiFeSP9mtnThkFL2AGwGpxmtZ11").unwrap()
}

// Used in constraints like:
constraint = profit_mint.key() == profit_mint() @ TaxError::InvalidMint
```

**Staking Program** (relies on implicit validation):
- No hardcoded PROFIT mint constant
- No explicit address constraint
- Relies on stake_vault's immutable mint field

## Impact Assessment

**Severity Rationale: INFO**

- ✅ No direct exploit vector once properly initialized
- ✅ Token account mint immutability prevents runtime mint substitution
- ✅ PDA uniqueness prevents multiple vaults
- ⚠️ Initialization front-running is ALREADY documented (TE-008, finding 7.8)
- ⚠️ Missing defense-in-depth (explicit validation would be better)

**Real Risk:**
- **During Initialization:** If an attacker front-runs `initialize_stake_pool` with a fake mint, the stake_vault is created with the wrong mint. Legitimate users cannot stake PROFIT tokens. This is the initialization front-running issue already documented.
- **After Initialization:** NO RISK. The unstake flow is safe.

**User Impact:**
- Users cannot receive wrong tokens on unstake (original STR-034 concern)
- Worst case: Initialization front-run causes DoS (re-deploy required)

## Evidence

### Constraint Validation Test

The constraint `token::mint = profit_mint` is enforced by Anchor's constraint system:

```rust
// From Anchor's generated code (simplified):
if stake_vault.mint != profit_mint.key() {
    return Err(ErrorCode::ConstraintTokenMint.into());
}
```

This happens BEFORE any instruction logic executes, so an attacker passing a fake mint fails immediately.

### Token Account Structure

```rust
// Solana Token Account data layout (SPL/Token-2022):
pub struct Account {
    pub mint: Pubkey,          // Offset 0-32: IMMUTABLE after creation
    pub owner: Pubkey,         // Offset 32-64
    pub amount: u64,           // Offset 64-72
    // ... other fields
}
```

The `mint` field at offset 0-32 is written ONCE during initialization and never modified by ANY token program instruction.

### Initialization Flow

```
1. Admin calls initialize_stake_pool(profit_mint = REAL_PROFIT)
2. Anchor creates stake_vault with:
   - PDA address from ["stake_vault"]
   - mint field set to REAL_PROFIT (from passed account)
3. stake_vault now permanently linked to REAL_PROFIT

Later:
4. User calls unstake(profit_mint = FAKE_MINT)
5. Anchor validates: stake_vault.mint == FAKE_MINT? NO
6. Transaction fails with ConstraintTokenMint error
```

## Recommendations

### 1. Add Explicit Mint Validation (Defense-in-Depth)

**Priority:** Low
**Effort:** Minimal

Add a hardcoded PROFIT mint constant and explicit constraint:

```rust
// In constants.rs:
pub fn profit_mint() -> Pubkey {
    #[cfg(feature = "devnet")]
    { pubkey!("8y7Mati78NNAn6YfGqiFeSP9mtnThkFL2AGwGpxmtZ11") }

    #[cfg(not(feature = "devnet"))]
    { Pubkey::default() } // Replace before mainnet
}

// In unstake.rs:
use crate::constants::profit_mint;

#[account(
    mut,
    token::mint = profit_mint_account,
    token::authority = stake_pool,
    token::token_program = token_program,
    seeds = [STAKE_VAULT_SEED],
    bump,
)]
pub stake_vault: InterfaceAccount<'info, TokenAccount>,

#[account(
    constraint = profit_mint_account.key() == profit_mint() @ StakingError::InvalidMint
)]
pub profit_mint_account: InterfaceAccount<'info, Mint>,
```

**Benefits:**
- Explicit validation readable in code
- Fails fast with clear error message
- Aligns with tax program's pattern
- Defense-in-depth against future refactoring mistakes

**Drawbacks:**
- Marginal compute cost (one pubkey comparison)
- Requires maintaining mint constant across programs

### 2. Include Mint in Stake Vault PDA Seeds (Structural Fix)

**Priority:** Low (requires re-deployment)
**Effort:** Medium

Derive stake_vault PDA from `["stake_vault", profit_mint]`:

```rust
#[account(
    init,
    payer = authority,
    token::mint = profit_mint,
    token::authority = stake_pool,
    seeds = [STAKE_VAULT_SEED, profit_mint.key().as_ref()],
    bump,
)]
pub stake_vault: InterfaceAccount<'info, TokenAccount>,
```

**Benefits:**
- Makes mint part of vault identity
- Allows multiple vaults for different tokens (if ever needed)
- PDA itself encodes which mint is correct

**Drawbacks:**
- Requires program re-deployment (changes PDA address)
- Breaks existing client code
- Overkill for single-mint use case

### 3. Document Initialization Security (No Code Change)

**Priority:** Medium
**Effort:** Minimal

Update deployment documentation to emphasize:
1. Initialize stake pool in same transaction bundle as program deployment
2. Verify stake_vault mint matches expected PROFIT address post-deployment
3. Use multi-sig or secure key management for initialization

## Conclusion

**STR-034 is a FALSE POSITIVE with respect to unstake returning wrong tokens.** The constraint `token::mint = profit_mint` provides sufficient protection once the stake_vault is properly initialized. The only risk is during initialization (front-running), which is already documented.

However, adding an explicit mint address constraint is recommended as a **defense-in-depth** measure to:
1. Make the validation explicit and auditable
2. Provide clearer error messages
3. Align with best practices used in the tax program
4. Protect against future refactoring errors

**Recommended Action:** Add explicit mint validation per Recommendation #1. Implement during next planned program update (non-urgent).
