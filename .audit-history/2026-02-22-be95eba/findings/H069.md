## Impact

**None.** This is a security-positive finding. Correct PDA seed matching is essential for access control.

### Why This Matters

If seeds were mismatched:
1. **Wrong PDA derived:** The signer would not be recognized by the receiving program
2. **CPI would fail:** `InvalidSeeds` error (Anchor constraint validation would reject the signer)
3. **Protocol would be broken:** No swaps, no staking, no carnage would execute

The fact that the protocol is functioning (as evidenced by live deployments on devnet and the 299 passing tests) proves seeds are correct.

---

## Recommendation

**No action required.** This is a confirmation of correct implementation.

For future reference:
1. PDA seeds are correctly documented in constants files with cross-program references
2. Unit tests verify seed values and lengths
3. All `invoke_signed` calls use Anchor's `ctx.bumps` auto-derived bumps
4. No hardcoded seeds are present — all use named constants

**Defense-in-depth observation:** The use of `seeds::program = <function>()` constraints (rather than hardcoded program IDs in seeds) is excellent. It ensures the validation logic matches the derivation logic at compile-time.
