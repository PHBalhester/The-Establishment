# H003: Initialization Front-Running Claims Admin Authority

**Strategy:** STR-003
**Determination:** POTENTIAL
**Severity:** HIGH
**Confidence:** MEDIUM

## Summary

Three initialization instructions across the protocol accept any signer without verifying they are the deployer/upgrade authority, creating a front-running window where an attacker could claim critical protocol authority by monitoring the mempool and submitting their initialization transaction before the legitimate deployer.

## Root Cause

These initialization instructions use Anchor's `init` constraint which prevents **re-initialization** (the PDA can only be created once via discriminator check), but they do NOT verify the **caller identity**. Per EP-076 (Front-Runnable Init), an attacker monitoring the mempool can front-run the deployer's initialization transaction by submitting their own with higher priority fees.

The vulnerable pattern:
```rust
#[account(init, payer = signer, ...)] // Prevents re-init
pub config: Account<'info, Config>,
pub signer: Signer<'info>,  // But accepts ANY signer!
```

The secure pattern (from AMM):
```rust
#[account(constraint = program_data.upgrade_authority_address == Some(authority.key()))]
pub program_data: Account<'info, ProgramData>,
```

## Attack Scenario

### Phase 1: Monitoring

1. Attacker monitors Solana mainnet mempool for program deployments
2. Attacker identifies the 5 Dr Fraudsworth programs being deployed (recognizable by program size, transaction structure, or leaked deployment plans)
3. Attacker observes the deployment transaction containing the Transfer Hook program ID

### Phase 2: Front-Running

4. Deployer submits transaction calling `initialize_authority` with their intended authority
5. Attacker sees this transaction in the mempool (before block inclusion)
6. Attacker crafts an identical `initialize_authority` transaction with:
   - Same program ID
   - Higher priority fee (to be processed first)
   - Attacker's own pubkey as the signer
7. Both transactions land in the same block, attacker's transaction executes first
8. Deployer's transaction fails with "account already initialized" error

### Phase 3: Protocol Capture

**Transfer Hook Authority Scenario:**
9. Attacker now controls the whitelist authority
10. Deployer attempts to whitelist protocol vaults (AMM pools, staking vaults, carnage fund)
11. Attacker refuses to execute whitelist additions
12. ALL token transfers fail at the Transfer Hook execution layer
13. Protocol is completely non-functional — swaps, staking, carnage all blocked
14. Attacker demands ransom or permanently DoSes the protocol

**Epoch State Scenario:**
9. Attacker initialized epoch state with their chosen `genesis_slot`
10. If future parameters were configurable (currently hardcoded), attacker could set favorable rates
11. Timing manipulation could affect epoch boundary calculations, though impact is limited

## Impact Assessment

### Critical: Transfer Hook Authority
- **Attacker gains:** Complete control over token transfer permissions for CRIME, FRAUD, and PROFIT
- **Protocol loses:** Ability to function — all token movement depends on whitelist
- **Recovery:** Requires redeploying the entire Transfer Hook program (and potentially all tokens if mints reference the old hook program ID)
- **Economic damage:** Complete protocol failure, all liquidity locked, user funds inaccessible

### Moderate: Epoch State
- **Attacker gains:** Control over genesis timing, potential future parameter manipulation
- **Protocol loses:** Canonical genesis state, timing predictability
- **Recovery:** Requires redeploying Epoch Program, resetting all VRF state
- **Economic damage:** Medium (protocol functions but with manipulated timing)

### Low: Staking Pool
- **Attacker gains:** Minimal (pool parameters are constants, attacker pays dead stake)
- **Protocol loses:** Canonical pool initialization
- **Recovery:** Requires redeploying Staking Program
- **Economic damage:** Low (attacker paid 1 PROFIT for dead stake, parameters are safe)

## Remediation

### Recommended Fix: Add Upgrade Authority Verification

Apply the AMM's `initialize_admin` pattern to all three vulnerable initialization instructions:

#### 1. Transfer Hook `initialize_authority.rs`

```rust
#[derive(Accounts)]
pub struct InitializeAuthority<'info> {
    /// The program's upgrade authority. Must sign to prove deployer identity.
    #[account(mut)]
    pub authority: Signer<'info>,

    #[account(
        init,
        payer = authority,  // Changed from 'signer'
        space = 8 + WhitelistAuthority::INIT_SPACE,
        seeds = [WhitelistAuthority::SEED],
        bump
    )]
    pub whitelist_authority: Account<'info, WhitelistAuthority>,

    // ADD THESE TWO ACCOUNTS:
    /// The Transfer Hook program itself
    #[account(
        constraint = program.programdata_address()? == Some(program_data.key())
    )]
    pub program: Program<'info, crate::program::TransferHook>,

    /// The program's ProgramData account (created by BPF loader on deploy)
    /// Constraint: upgrade_authority_address must equal the signing authority
    #[account(
        constraint = program_data.upgrade_authority_address == Some(authority.key())
            @ TransferHookError::UnauthorizedInit
    )]
    pub program_data: Account<'info, ProgramData>,

    pub system_program: Program<'info, System>,
}
```

### Alternative Mitigations (Lower Security)

1. **Atomic Deployment Bundle:** Use Jito bundles or similar MEV protection to deploy all programs and call all initialization instructions atomically in a single transaction. This reduces but does not eliminate the front-running window.

2. **PDA Seeds with Deployer Key:** Change PDA seeds to include the deployer's pubkey:
   ```rust
   seeds = [WhitelistAuthority::SEED, DEPLOYER_PUBKEY.as_ref()]
   ```
   This makes the PDA specific to the deployer, but requires hardcoding the deployer key at compile time.

3. **Post-Initialization Authority Transfer:** Initialize with upgrade authority, then transfer authority to a different multisig in a second transaction. This doesn't prevent front-running but limits the initial authority scope.

## Mitigating Factors

1. **Anchor `init` Constraint:** All three instructions use `init`, which means the PDA can only be created once. The front-runner cannot re-initialize after the legitimate deployer initializes. This limits the attack to a **one-time race condition** during deployment.

2. **Deployment Process:** In practice, deployment scripts typically bundle all initialization calls in the same transaction or submit them immediately after deployment with minimal mempool exposure.

3. **Economic Cost (Staking Only):** The staking pool front-runner must pay `MINIMUM_STAKE` (1 PROFIT token) as dead stake, which acts as a mild deterrent.

4. **Limited Mainnet Mempool Visibility:** Solana's leader schedule and transaction forwarding architecture means not all pending transactions are visible in a public mempool. However, RPC providers and validators can observe pending transactions.

5. **Parameters Are Hardcoded:** The Epoch Program's genesis parameters (`cheap_side`, `low_tax_bps`, `high_tax_bps`) are compile-time constants, not caller-provided arguments. This limits the manipulation scope compared to if the front-runner could set arbitrary values.

## Determination Rationale: POTENTIAL (not CONFIRMED)

**Why POTENTIAL instead of CONFIRMED:**

1. The vulnerability requires **perfect timing** — the attacker must observe the deployment transaction and submit a front-running transaction within the same slot (400ms on Solana).

2. The attack assumes **mempool visibility**, which is not guaranteed on Solana due to Gulf Stream and transaction forwarding to leaders. Many deployment transactions may not be visible to external observers before block inclusion.

3. The protocol team likely uses **deployment automation** (scripts/CI) that bundles program deployment and initialization in rapid succession, minimizing exposure.

4. The `init` constraint prevents **re-exploitation** — if the deployer wins the race, the attacker cannot try again.

**Why still HIGH severity:**

1. The **Transfer Hook authority** is critical infrastructure — a successful front-run would completely brick the protocol.

2. The attack is **low-cost** for the attacker (just priority fees + rent for PDA creation).

3. The attack is **undetectable until too late** — the deployer only discovers they lost the race when their init transaction fails.

4. **Recovery is expensive** — requires redeploying programs and potentially reissuing tokens.
