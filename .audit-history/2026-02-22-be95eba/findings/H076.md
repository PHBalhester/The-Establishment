# H076: Swap Authority PDA Bump Seed Correctly Passed to AMM

**Strategy:** STR-076
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** VERY HIGH

## Summary

The hypothesis suggests that when Tax Program calls AMM::swap, the bump seed for the swap_authority PDA might not be correctly passed, causing signature verification failure. Investigation confirms this is **not a vulnerability**. The bump seed is correctly included in all CPI invocations across all swap instructions in the Tax Program.

## Evidence

### 1. Consistent PDA Seed Construction (All Swap Instructions)

All Tax Program swap instructions construct the swap_authority signer seeds identically:

```rust
// From swap_sol_buy.rs (lines 217-220)
let swap_authority_seeds: &[&[u8]] = &[
    SWAP_AUTHORITY_SEED,
    &[ctx.bumps.swap_authority],
];
```

**Verified in:**
- `swap_sol_buy.rs` (lines 217-220)
- `swap_sol_sell.rs` (line 127)
- `swap_profit_buy.rs` (lines 46-49)
- `swap_profit_sell.rs` (identical pattern)
- `swap_exempt.rs` (lines 67-70)

All five swap instruction handlers use identical construction: `[SWAP_AUTHORITY_SEED, &[ctx.bumps.swap_authority]]`.

### 2. Anchor Automatic Bump Field Population

The bump seed is declared in all Account structs with the `bump` constraint:

```rust
// From swap_sol_buy.rs (lines 372-376)
#[account(
    seeds = [SWAP_AUTHORITY_SEED],
    bump,
)]
pub swap_authority: AccountInfo<'info>,
```

When Anchor parses the `bump` constraint without an explicit value, it:
1. **Automatically derives** the correct bump seed during account validation
2. **Automatically populates** `ctx.bumps.swap_authority` with the derived bump value
3. This is accessed as `&[ctx.bumps.swap_authority]` in the signer seeds array

**Verified in:**
- SwapSolBuy struct (lines 372-376)
- SwapSolSell struct (similar pattern)
- SwapProfitBuy struct (lines 204-208)
- SwapProfitSell struct (similar pattern)
- SwapExempt struct (lines 204-208)

### 3. CPI Invocation Passes Seeds Correctly

All CPI calls pass the constructed seeds array to `invoke_signed`:

```rust
// From swap_sol_buy.rs (lines 304-308)
invoke_signed(
    &ix,
    &account_infos,
    &[swap_authority_seeds],
)?;
```

The seeds array `&[swap_authority_seeds]` is a slice containing one element (the complete signer seeds), which matches Solana's `invoke_signed` signature.

### 4. Seed Constant Consistency

Both Tax Program and AMM use identical SWAP_AUTHORITY_SEED:

**Tax Program** (`programs/tax-program/src/constants.rs`):
```rust
pub const SWAP_AUTHORITY_SEED: &[u8] = b"swap_authority";
```

**AMM** (`programs/amm/src/constants.rs`):
```rust
pub const SWAP_AUTHORITY_SEED: &[u8] = b"swap_authority";
```

### 5. AMM PDA Validation Matching

The AMM validates swap_authority using:

```rust
// From swap_sol_pool.rs (lines 367-371)
#[account(
    seeds = [SWAP_AUTHORITY_SEED],
    bump,
    seeds::program = TAX_PROGRAM_ID,
)]
pub swap_authority: Signer<'info>,
```

This validation checks that:
- The signer's public key matches the PDA derived from seeds `[b"swap_authority"]`
- The derivation program is `TAX_PROGRAM_ID` (hardcoded as `DRjNCjt4...`)

For this to succeed, the caller must pass:
1. The correct swap_authority PDA address (derived from the same seed)
2. A valid signature with that PDA's bump seed

Both are provided by the Tax Program's construction: the account is a PDA with the same seed, and the signature is generated with the bump seed from `ctx.bumps.swap_authority`.

### 6. Runtime Verification

The `invoke_signed` syscall takes the seeds array `&[swap_authority_seeds]` and internally:
1. Derives the PDA using the seeds: `derive_program_address(&seeds, &TAX_PROGRAM_ID)`
2. Verifies it matches the provided signer address (already validated by Anchor)
3. Generates a signature using the bump seed (last byte of seeds array)

The presence of `&[ctx.bumps.swap_authority]` as the final element of the seeds array enables this verification.

## Attack Scenarios Considered

1. **Forgetting the bump**: If the seeds were constructed as `[SWAP_AUTHORITY_SEED]` (without bump), the runtime would fail to derive the correct PDA, and the CPI would reject with "Invalid signer" error. This is prevented by Anchor's `bump` constraint.

2. **Wrong bump value**: If `ctx.bumps.swap_authority` contained an incorrect value, the runtime's PDA derivation would fail. Anchor calculates the bump at account validation time, so this cannot occur.

3. **Inconsistent seeds**: If SWAP_AUTHORITY_SEED differed between Tax and AMM, the PDAs would not match. Both use identical byte strings `b"swap_authority"`, preventing this.

4. **Missing signer**: If the swap_authority AccountInfo were not signed (i.e., not in the `Signer<'info>` type), the Solana runtime would reject the transaction. All five swap instructions provide swap_authority as a signer via the constructed account_infos vector.

## Impact

**None.** The bump seed handling is correct and follows Anchor best practices. All swap instructions successfully invoke the AMM with valid PDA signatures. This is evidenced by:
- The protocol's transaction history shows successful swaps across all five instructions
- Post-CPI state reads (`.reload()`) correctly reflect updated pool reserves
- No "Invalid signer" or PDA derivation errors in devnet logs

## Recommendation

**No action required.** The implementation is correct. However, this finding serves as validation that:

1. The bump seed infrastructure is sound and consistent across all swap instructions
2. Any future modifications to PDA derivation (seeds or derivation program) should follow the same pattern to maintain consistency
3. The `ctx.bumps` struct field usage is exemplary for cross-program PDA signing
