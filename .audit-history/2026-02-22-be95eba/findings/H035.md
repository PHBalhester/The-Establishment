# H035: Carnage WSOL Account Drained via Direct Transfer

**Severity:** SAFE
**Status:** Not Vulnerable
**Category:** Access Control / Fund Security
**Threat:** STR-035

---

## Summary

Investigation into whether the Carnage WSOL account can be drained via unauthorized direct transfers. Analysis shows the account is **properly protected** by PDA-based authority controls. The carnage_wsol account is owned by the `carnage_signer` PDA (derived from Epoch Program), and only authorized instructions can sign transfers from it.

---

## Vulnerability Analysis

### Account Structure

The Carnage WSOL account is created during protocol initialization with the following properties:

```typescript
// scripts/deploy/initialize.ts:1045-1053
const carnageWsolAccount = await createWrappedNativeAccount(
  connection,
  authority,                     // payer
  carnageSignerPda,             // owner (CarnageSigner PDA)
  0,                            // 0 lamports initial
  carnageWsolKeypair,           // explicit keypair
  { commitment: "confirmed" },
  TOKEN_PROGRAM_ID              // WSOL uses SPL Token, NOT Token-2022
);
```

**Key security properties:**
1. **Owner:** `carnage_signer` PDA derived from `seeds = ["carnage_signer"], program = EpochProgram`
2. **Token Type:** WSOL (SPL Token, not Token-2022)
3. **Not a PDA:** Uses explicit keypair (because ATA rejects off-curve PDA owners)
4. **Authority:** Only the owner PDA can authorize transfers

### Authority Derivation

```rust
// epoch-program/src/constants.rs:91
pub const CARNAGE_SIGNER_SEED: &[u8] = b"carnage_signer";
```

The `carnage_signer` PDA is derived deterministically:
```rust
PDA = find_program_address(["carnage_signer"], EPOCH_PROGRAM_ID)
```

Only the Epoch Program can produce a valid signature with this PDA using `invoke_signed`.

### Access Control Validation

Both Carnage execution instructions enforce strict owner validation:

#### execute_carnage_atomic.rs
```rust
// Line 92-97
#[account(
    mut,
    constraint = carnage_wsol.owner == carnage_signer.key()
        @ EpochError::InvalidCarnageWsolOwner,
)]
pub carnage_wsol: Box<InterfaceAccount<'info, TokenAccount>>,
```

#### execute_carnage.rs
```rust
// Line 86-92
#[account(
    mut,
    constraint = carnage_wsol.owner == carnage_signer.key()
        @ EpochError::InvalidCarnageWsolOwner,
)]
pub carnage_wsol: Box<InterfaceAccount<'info, TokenAccount>>,
```

Both instructions explicitly validate that `carnage_wsol.owner == carnage_signer.key()`, rejecting any account with a different owner.

### All Transfer Paths

Comprehensive search of all programs shows carnage_wsol is only accessed in 3 instructions:

#### 1. execute_carnage_atomic (Epoch Program)
- **Entry:** Permissionless (but no-ops when `carnage_pending = false`)
- **WSOL operations:**
  - **Receive:** Native SOL wrapped to WSOL via `system_program::transfer + sync_native` (lines 651-693)
  - **Send (buy):** WSOL -> faction token via Tax::swap_exempt CPI (lines 760-806)
  - **Receive (sell):** Faction token -> WSOL via Tax::swap_exempt CPI (lines 704-755)
- **Authority:** All transfers use `carnage_signer` PDA signing via `invoke_signed`
- **Validation:** Owner constraint on carnage_wsol account

#### 2. execute_carnage (Epoch Program - fallback)
- **Entry:** Permissionless fallback (after 50-slot atomic lock expires, before 300-slot deadline)
- **WSOL operations:** Identical to execute_carnage_atomic
- **Authority:** Same carnage_signer PDA signing
- **Validation:** Same owner constraint

#### 3. swap_exempt (Tax Program)
- **Entry:** CPI-only from Epoch Program via carnage_signer
- **WSOL operations:** Uses carnage_wsol as `user_token_a` in AMM swaps
- **Authority:** Receives carnage_authority (carnage_signer PDA) as signer parameter
- **Validation:**
  ```rust
  // tax-program/src/instructions/swap_exempt.rs:193-198
  #[account(
      seeds = [CARNAGE_SIGNER_SEED],
      bump,
      seeds::program = epoch_program_id(),
  )]
  pub carnage_authority: Signer<'info>,
  ```

#### No Other Access Paths

Search results confirm carnage_wsol is **not referenced** in:
- AMM Program (only receives it via Tax Program CPI passthrough)
- Staking Program (does not interact with WSOL)
- Transfer Hook (only enforces whitelist on Token-2022 tokens, not SPL Token WSOL)
- Tax Program user-facing instructions (only swap_exempt, which is CPI-only)

### Attack Vector Analysis

**Attempted Attack:** Attacker calls execute_carnage_atomic or execute_carnage with a malicious carnage_wsol account they control.

**Defense Mechanism:**
1. Attacker passes their own token account as `carnage_wsol`
2. Instruction constraint checks: `carnage_wsol.owner == carnage_signer.key()`
3. Attacker's account owner ≠ carnage_signer PDA
4. Transaction fails with `EpochError::InvalidCarnageWsolOwner`

**Why PDA Signing Cannot Be Forged:**
- The carnage_signer PDA is derived with `program = EPOCH_PROGRAM_ID`
- Only the Epoch Program can call `invoke_signed` with these seeds
- Solana runtime validates that the signing program matches the seeds::program
- An attacker cannot produce a valid signature for the PDA from outside the Epoch Program

**Why Direct SPL Token Transfer Cannot Work:**
- WSOL uses SPL Token program (not Token-2022)
- SPL Token enforces that only the account owner can authorize transfers
- carnage_wsol owner is carnage_signer PDA (not a user keypair)
- Only the Epoch Program can sign for this PDA
- Direct transfer_checked calls require the owner signature
- Attacker cannot provide this signature

## Defense Layers

The Carnage WSOL account has **4 layers of defense**:

### Layer 1: PDA Ownership
- Account owner is `carnage_signer` PDA, not a user keypair
- Only Epoch Program can sign for this PDA via `invoke_signed`

### Layer 2: Explicit Owner Validation
- Both execution instructions check `carnage_wsol.owner == carnage_signer.key()`
- Rejects any substitute account with different owner

### Layer 3: Cross-Program CPI Gating
- swap_exempt validates carnage_authority via `seeds::program = epoch_program_id()`
- Only Epoch Program can provide valid carnage_signer signature
- Tax Program rejects calls from any other program

### Layer 4: SPL Token Runtime Enforcement
- SPL Token program requires owner signature for all transfers
- Runtime validates signatures before executing token operations
- Cannot be bypassed by passing different account parameters

## Conclusion

**SAFE - Not Vulnerable**

The Carnage WSOL account is properly protected against unauthorized drainage via:

1. **PDA Authority:** Account owned by carnage_signer PDA, not a user keypair
2. **Explicit Validation:** Owner constraint in both execution instructions
3. **CPI Gating:** swap_exempt requires carnage_signer signature via seeds::program
4. **Runtime Enforcement:** SPL Token program validates owner signatures

No instruction exists in any of the 5 programs that can transfer from carnage_wsol without proper carnage_signer PDA authority. The account can only be accessed through:
- execute_carnage_atomic (validates owner)
- execute_carnage (validates owner)
- swap_exempt (validates carnage_authority via seeds::program)

All three paths enforce that only the Epoch Program can sign for the carnage_signer PDA, preventing external drainage.

**Impact:** None. Funds are secure.

**Recommendation:** Consider adding explicit negative test cases to document and verify constraint enforcement.
