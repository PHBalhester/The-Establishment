# H056 — Staking total_staked Underflow on Unstake

**Status:** PASS ✅
**Severity:** N/A (vulnerability not present)
**Category:** Arithmetic Safety
**Subsystem:** Staking Program
**Strategy:** STR-056

---

## Hypothesis

If `total_staked` doesn't use checked_sub, unstaking more than total_staked causes underflow to u64::MAX, breaking all subsequent reward calculations.

**Potential Impact:** HIGH — Staking reward pool permanently corrupted

## Findings

### ✅ Defense Layer 1: Input Validation
- **Line 128-131:** Validates `user_stake.staked_balance >= amount` before any state changes
- Triggers `StakingError::InsufficientBalance` if user attempts to unstake more than their balance
- This prevents the underflow scenario at the user level

### ✅ Defense Layer 2: Checked Arithmetic on User Balance
- **Line 182-185:** Uses `checked_sub(amount)` when updating `user.staked_balance`
- Returns `StakingError::Underflow` if subtraction would underflow
- Defense-in-depth: even if validation fails, arithmetic is still safe

### ✅ Defense Layer 3: Checked Arithmetic on Pool Total
- **Line 187-190:** Uses `checked_sub(amount)` when updating `pool.total_staked`
- Returns `StakingError::Underflow` if subtraction would underflow
- **Critical:** This prevents the exact vulnerability described in the hypothesis

### Architecture Analysis

The code follows a **defense-in-depth** pattern:

1. **Check Phase (Lines 128-131):** Validates sufficient user balance
2. **Effects Phase (Lines 182-190):** Updates state with checked arithmetic
3. **Interactions Phase (Lines 214-232):** Performs token transfers

Even if a logic bug allowed `amount > user.staked_balance` to pass validation, the checked arithmetic would still prevent underflow to `u64::MAX`.

### Edge Case: Concurrent Unstakes

**Scenario:** Two unstake transactions in the same block for the same user.

**Protection:**
- Each transaction reads the current `user.staked_balance`
- The first transaction succeeds and decrements balance
- The second transaction will fail at **Line 128** validation because the updated balance is already committed
- Solana's account write locks prevent race conditions within a single user's stake account

---

## Conclusion

**Verdict:** The vulnerability is **NOT PRESENT**.

The staking program implements three layers of defense:
1. Pre-validation of user balance
2. Checked subtraction on user state
3. Checked subtraction on pool state

All arithmetic operations use `checked_sub` with proper error handling. The `StakingError::Underflow` variant is defined and would trigger if any subtraction attempted to wrap.

**No remediation required.**
