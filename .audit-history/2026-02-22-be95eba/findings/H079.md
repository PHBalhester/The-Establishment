# H079: AMM Locked Flag Persists Across Instructions in Same TX

**Strategy:** STR-079

**Determination:** NOT VULNERABLE

**Severity:** N/A

**Confidence:** HIGH

## Summary

This finding investigates whether the AMM pool's reentrancy guard (`pool.locked` boolean) can persist in a locked state across multiple instructions within a single transaction, causing subsequent instructions to fail.

**Verdict:** The hypothesis is **not viable** due to Solana's fundamental transaction atomicity guarantees. The locked flag cannot persist in a locked state across instructions because:

1. Failed transactions roll back all state mutations atomically
2. The Solana runtime prevents a program from re-entering itself during CPI
3. The lock is always cleared before transaction finality (or the entire TX reverts)

## Evidence

### Code Review: Lock Lifecycle

**swap_sol_pool.rs (lines 84, 322):**
```rust
// Line 84: SET LOCK at start of CHECKS phase
ctx.accounts.pool.locked = true;

// ... all swap computation and transfers ...

// Line 322: CLEAR LOCK at end of POST-INTERACTION phase
ctx.accounts.pool.locked = false;
```

**swap_profit_pool.rs (lines 64, 247):**
```rust
// Line 64: SET LOCK
ctx.accounts.pool.locked = true;

// ... all swap computation and transfers ...

// Line 247: CLEAR LOCK
ctx.accounts.pool.locked = false;
```

Both handlers follow identical CEI (Checks-Effects-Interactions) ordering:
- CHECKS (lines 86-149): Validate inputs, compute swap math
- EFFECTS (lines 162-185): Update pool reserves in place
- INTERACTIONS (lines 188-315): Execute token transfers via CPI
- POST-INTERACTION (lines 321-340): Clear lock, emit event

### Account Constraint Guard

Both swap instructions use Anchor's constraint validation:

**swap_sol_pool.rs (line 381):**
```rust
#[account(
    mut,
    seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
    bump = pool.bump,
    constraint = pool.initialized @ AmmError::PoolNotInitialized,
    constraint = !pool.locked @ AmmError::PoolLocked,  // ← PREVENTS RE-ENTRY
)]
pub pool: Account<'info, PoolState>,
```

**swap_profit_pool.rs (line 305):**
```rust
constraint = !pool.locked @ AmmError::PoolLocked,  // ← PREVENTS RE-ENTRY
```

These constraints are **evaluated at account deserialization time**, before the handler runs. If the pool account's `locked` field is `true` when deserializing, the instruction fails with `PoolLocked` error.

### Solana Runtime Atomicity

**Property 1: Failed Transactions Revert Completely**
- If any CPI in the token transfer chain fails (lines 210-315), the `?` operator propagates the error
- Anchor immediately aborts the transaction
- The Solana runtime rolls back **all state changes** from this instruction, including `locked = true`
- The pool is never left in a locked state on-chain

**Property 2: No Self-Reentrancy During CPI**
- The Solana runtime prevents a program from calling itself (recursively) during a CPI
- Even if the AMM tried to call itself via `invoke_signed`, the runtime would reject it
- The lock flag provides defense-in-depth, but Solana's design already prevents the scenario

**Property 3: Instruction Atomicity**
- Each instruction either succeeds completely or fails completely
- There is no "partial state" persisted to the blockchain
- Multiple instructions in a single transaction are atomic at the transaction level; if instruction N+1 fails, instruction N's state changes are still committed, but if instruction N fails, all state is rolled back

### Verified Invariant

From the State Machine context document (Section 10, INV-03):

> **INV-03:** `pool.locked` is false at rest (not stuck)
> - Enforced By: Transaction atomicity -- failed TX rolls back
> - Verified: Yes (by Solana design)

## Impact

**None.** The locked flag cannot persist across instructions because:
- If a swap completes successfully, the flag is explicitly cleared (line 322/247)
- If a swap fails, the transaction reverts and the flag never persists to the blockchain
- Subsequent instructions in the same TX see a clean pool state (either the flag was cleared, or the entire TX failed before they ran)

## Recommendation

**No action required.** The implementation is sound. The reentrancy guard is implemented correctly and benefits from both explicit code logic (clearing the flag) and Solana's runtime guarantees (transaction atomicity).

This is a **false alarm** common to blockchain systems where developers worry about state persistence across failed instructions. Solana's atomicity model eliminates this class of vulnerability entirely.
