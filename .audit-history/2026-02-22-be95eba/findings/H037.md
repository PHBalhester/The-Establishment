# H037: Combined Admin Whitelist + Force Carnage Attack (Devnet Build Only)

**Strategy:** STR-037
**Determination:** POTENTIAL (conditional on deployment error)
**Severity:** HIGH
**Confidence:** HIGH

## Summary

If the Epoch Program is accidentally deployed to mainnet with `--features devnet` enabled, a compromised admin key could execute a combined attack leveraging **both** whitelist control and forced Carnage triggering to extract maximum value from the protocol. While H004 documented the standalone `force_carnage` backdoor, this finding demonstrates the **multiplicative severity** when combined with Transfer Hook whitelist manipulation.

**Key Insight:** Treasury redirection (hypothesized in strategy) is **NOT POSSIBLE** (confirmed by H005 - no `update_treasury` instruction exists). However, whitelist + forced Carnage is a more potent combination attack.

### Carnage Vault Architecture

**Carnage Fund Vaults** (all PDA-controlled by Epoch Program):
- `carnage_sol_vault` (seeds: `[b"carnage_sol_vault"]`) - Accumulates 5% of all swap taxes
- `carnage_crime_vault` (seeds: `[b"carnage_crime_vault"]`) - Holds CRIME tokens from sells
- `carnage_fraud_vault` (seeds: `[b"carnage_fraud_vault"]`) - Holds FRAUD tokens from sells

**Carnage Execution Flow** (from `execute_carnage_atomic` / `execute_carnage`):
1. Read `carnage_pending` state (set by VRF or `force_carnage`)
2. Use accumulated SOL vault balance to buy losing faction token via tax-exempt swap
3. Optionally burn portion of purchased tokens (if `carnage_action = 1`)
4. Optionally sell remainder for winning faction token (if `carnage_action = 2`)

**Critical:** Carnage swaps use `swap_exempt` (Tax Program) which bypasses the 15% tax and uses the `carnage_signer` PDA authority. The AMM validates this via `seeds::program = EPOCH_PROGRAM_ID`.

## Root Cause

The vulnerability exists due to **multiple independent admin capabilities** that can be **combined** when a single attacker controls both keys:

1. **Whitelist Control:** Admin can add arbitrary addresses to bypass transfer restrictions
2. **Force Carnage:** Admin can trigger Carnage execution at any time (bypassing VRF)
3. **No Treasury Update:** Admin CANNOT redirect tax revenue (H005 proves this is impossible)

The root cause is a **process gap in mainnet deployment verification**, not a code flaw. The code correctly gates `force_carnage` with `#[cfg(feature = "devnet")]`, but there's no enforcement to prevent deploying the devnet-featured binary to mainnet.

## Attack Scenario

### Preconditions
1. **Epoch Program deployed to mainnet WITH `--features devnet`** (accidental or malicious)
2. **Attacker compromises BOTH:**
   - DEVNET_ADMIN key (`8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4`) - in git
   - Whitelist authority key (deployment wallet)
3. **Carnage Fund has accumulated significant SOL** (e.g., 10+ SOL from protocol fees)

## Impact Assessment

### Direct Impact
- **Market Manipulation:** Attacker controls Carnage timing and direction (predictable price pumps)
- **Whitelist Bypass:** Attacker can freely move tokens without hook restrictions
- **Front-Running:** Attacker knows Carnage will execute before public (zero competition)

### Combined Impact (Whitelist + Force Carnage)
The **multiplicative effect** exceeds standalone capabilities:

| Attack Type | Standalone Severity | Combined Severity |
|-------------|-------------------|------------------|
| Force Carnage alone (H004) | HIGH - Market manipulation, ~$500/day | CRITICAL when combined |
| Whitelist manipulation alone | MEDIUM - Can bypass transfers, but can't profit directly | CRITICAL when combined |
| **Combined Attack** | **N/A** | **HIGH** - Front-run + unrestricted exit = max extraction |

**Why Combined is Worse:**
1. **Whitelist control** removes the transfer hook barrier, enabling free token movement
2. **Force Carnage** creates predictable price pumps to exploit
3. **Together:** Attacker can enter positions, force price pump, exit without restrictions
4. **Profit amplification:** Whitelist bypass means attacker can use multiple wallets (Sybil attack)

### Economic Impact

Assume protocol processes $100K/day in volume:
- 15% tax = $15K/day
- 5% to Carnage = $5K/day in WSOL accumulation
- Carnage Fund balance grows to 10-20 SOL before execution
- **At 10 SOL per Carnage event:**
  - 15% price pump on target pool (conservative)
  - Attacker uses 5 SOL to front-run
  - 15% gain on 5 SOL = 0.75 SOL profit per event
  - **4 events/day (mainnet: 30min epochs) = 3 SOL/day = $450/day profit**
- **Annualized:** $164K/year extraction from Carnage manipulation alone

**Worse:** Attacker could add multiple accomplice wallets to whitelist, distribute tokens, and coordinate exits after each forced Carnage for multiplied gains.

## Remediation

### Primary Fix: Remove force_carnage (CRITICAL)
**Same as H004:** Delete `force_carnage.rs` module entirely before mainnet deployment.

### Secondary Fix: Admin Key Separation (HIGH)
Even if `force_carnage` is removed, whitelist control is still powerful. Mitigate by:

**1. Burn whitelist authority after all legitimate addresses are whitelisted:**
```rust
// After deployment initialization:
await hookProgram.methods
  .burnAuthority()
  .accounts({ authority: whitelistAuthority.publicKey })
  .signers([whitelistAuthority])
  .rpc();
// Whitelist is now permanently frozen
```

**2. Use separate keys for different admin roles (defense-in-depth):**
```bash
# Deployment key: Program upgrade authority (3-of-5 multisig)
# AMM admin: Separate 2-of-3 multisig (pool creation only)
# Whitelist authority: Separate cold wallet (burn after setup)
```

**3. Add timelock to whitelist additions (mainnet hardening):**
```rust
// In add_whitelist_entry.rs:
pub struct WhitelistEntry {
    pub address: Pubkey,
    pub created_at: i64,
    pub active_at: i64, // created_at + 24h timelock
}

// In transfer_hook.rs handler:
let entry = WhitelistEntry::try_from_slice(/* whitelist_source data */)?;
require!(
    Clock::get()?.unix_timestamp >= entry.active_at,
    TransferHookError::WhitelistEntryNotActive
);
```

This prevents instant whitelist -> attack -> de-whitelist cycles.

### Tertiary Fix: Mainnet Deployment Checklist Enforcement

**Add automated verification to `scripts/verify-mainnet-build.sh`:**
```bash
#!/bin/bash
set -e

echo "=== Mainnet Deployment Security Verification ==="

# 1. Verify force_carnage absent from IDL
if grep -q "force_carnage" target/idl/epoch_program.json; then
  echo "FAIL: force_carnage present in epoch_program IDL"
  echo "      This means the binary was compiled with --features devnet"
  exit 1
fi

# 2. Verify admin keys are multisigs (not single EOAs)
echo "Verify AMM admin is multisig (manual check):"
echo "  Admin pubkey: <display from AdminConfig PDA>"
echo "  Is this a Squads/Realms multisig? (y/n)"
read -r IS_MULTISIG
if [[ "$IS_MULTISIG" != "y" ]]; then
  echo "WARN: AMM admin should be a multisig for mainnet"
fi

# 3. Verify no devnet keypairs in directory
if ls keypairs/*.json 2>/dev/null; then
  echo "FAIL: Keypair JSON files found in keypairs/ directory"
  echo "      Mainnet keys should NEVER be committed to git"
  exit 1
fi

echo "PASS: Mainnet deployment verification complete"
```

## Notes for Auditor

### Why HIGH (not CRITICAL) Severity?

**Despite the severe combined impact, severity remains HIGH because:**

1. **Conditional on deployment error:** Requires accidental mainnet deployment with `--features devnet`
2. **Key compromise required:** Attacker needs BOTH devnet admin + whitelist authority keys
3. **Detectable:** Whitelist additions emit events; force_carnage calls are on-chain visible
4. **Mitigatable post-deploy:** Admin can burn whitelist authority to cut off one attack vector

**If severity were based purely on impact:** CRITICAL (protocol-level manipulation)
**If severity were based purely on likelihood:** MEDIUM (requires deployment error + key compromise)
**Composite severity:** HIGH (significant impact, moderate likelihood in adversarial scenario)

### Cross-Finding Relationships

- **H004 (force_carnage backdoor):** Standalone severity HIGH → remains HIGH
- **H037 (combined attack):** Leverages H004 + whitelist → severity HIGH (multiplicative, but still conditional)
- **H005 (treasury redirect):** NOT VULNERABLE → removed from combined attack surface

The key insight is that **whitelist control amplifies force_carnage impact** by removing exit friction, but doesn't create new attack primitives beyond what H004 already documented.

## Conclusion

The combined admin whitelist + force Carnage attack represents a **HIGH severity** risk when:
1. Epoch Program is deployed with `--features devnet` (accidental deployment)
2. Admin keys are compromised (likely if using devnet wallet keypair from git)
3. Whitelist authority is not burned after initialization

**Critical mitigations:**
- Remove `force_carnage` entirely (as documented in H004)
- Burn whitelist authority after protocol initialization
- Use separate multisig keys for all admin roles
- Enforce automated verification in deployment pipeline

The protocol's deploy-and-lock philosophy is sound, but requires **strict discipline** in the deployment process to prevent devnet-only backdoors from reaching mainnet.
