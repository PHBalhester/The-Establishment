# H014: Carnage Buy Amount Capped — Pool Imbalance Risk Mitigated

**Strategy:** STR-014
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis that Carnage's buy step could use an uncapped fund balance to catastrophically imbalance pools is **NOT CONFIRMED**. The protocol implements a hard cap of `MAX_CARNAGE_SWAP_LAMPORTS = 1000 SOL` (line 151 of constants.rs) on every Carnage buy operation, regardless of the accumulated fund balance. This cap, combined with the 85% slippage floor, prevents the scenario where a large Carnage fund dumps its entire balance into a single pool.

## Evidence

### Cap Enforcement (execute_carnage_atomic.rs:356-364)

```rust
// Calculate swap amount: min of available SOL balance and cap.
let available_sol = sol_balance.saturating_sub(rent_exempt_min);
let swap_amount = std::cmp::min(available_sol, MAX_CARNAGE_SWAP_LAMPORTS);

// Combine tax SOL with sell proceeds (if any)...
let total_buy_amount = std::cmp::min(
    swap_amount.checked_add(sol_from_sale).ok_or(EpochError::Overflow)?,
    MAX_CARNAGE_SWAP_LAMPORTS,
);
```

**Critical observation:** The cap is applied TWICE:
1. First on `swap_amount` — limits fresh SOL from the vault
2. Second on `total_buy_amount` — caps the combined SOL + sell proceeds

Even if the Carnage fund accumulates 10,000 SOL over many epochs, the maximum per-swap is 1000 SOL.

### Slippage Floor (execute_carnage_atomic.rs:422-438)

```rust
// 85% slippage floor (15% tolerance) for atomic Carnage path
let expected = (reserve_token as u128)
    .checked_mul(total_buy_amount as u128)
    .and_then(|n| n.checked_div(
        (reserve_sol as u128).checked_add(total_buy_amount as u128)?
    ))
    .ok_or(EpochError::Overflow)? as u64;

let min_output = (expected as u128)
    .checked_mul(CARNAGE_SLIPPAGE_BPS_ATOMIC as u128)
    .and_then(|n| n.checked_div(10_000))
    .ok_or(EpochError::Overflow)? as u64;

require!(
    bought >= min_output,
    EpochError::CarnageSlippageExceeded
);
```

The 85% floor rejects swaps where the actual output is less than 85% of the constant-product expected output. This catches extreme pool manipulation attempts.

### Economic Model (context/09-economic-model.md)

From Section 2.4:
- **Carnage accumulation rate:** 24% of all tax revenue (context line 266)
- **Carnage trigger probability:** 4.3% per epoch (~11/256 via VRF byte 5)
- **Max spend per trigger:** 1000 SOL (enforced by constant)
- **Fund can grow faster than it spends:** With $10,000 daily volume, fund accumulates ~$180/day but spends ~$87 per trigger (context lines 562-571)

**Implication:** The fund WILL accumulate over time, potentially reaching 10,000+ SOL. However, the per-swap cap prevents dumping the entire balance at once.

## Root Cause (Non-Issue)

The original hypothesis was that Carnage would use its ENTIRE balance for a single buy:

> "Carnage's buy step uses the entire carnage fund WSOL balance"

This is **false**. The code explicitly limits each buy to:
- MIN(available_sol, 1000 SOL) for fresh vault SOL
- MIN(that + sell_proceeds, 1000 SOL) for total buy amount

## Attack Scenario (Prevented)

**Hypothetical attack (if no cap existed):**
1. Wait for Carnage fund to accumulate 10,000 SOL over 6 months
2. Trigger epoch when VRF favors buying CRIME
3. Carnage dumps all 10,000 SOL into CRIME/SOL pool
4. For a pool with 500 SOL / 500M CRIME (example):
   - Input: 10,000 SOL (20x reserve)
   - Constant product: output ≈ 476M CRIME (95% of pool drained)
   - Pool becomes 10,500 SOL / 24M CRIME
   - Price skewed 20x in favor of SOL sellers
5. Pool permanently imbalanced, subsequent sells execute at terrible prices
6. Protocol's own mechanism destroys the market

**Actual mitigation:**
- Max 1000 SOL per buy (2x reserve in example, not 20x)
- Output ≈ 333M CRIME (67% extraction, not 95%)
- Pool becomes 1,500 SOL / 167M CRIME (3x skew, not 20x)
- 85% floor catches extreme cases: if expected 333M, actual must be ≥ 283M
- **Still significant impact**, but not catastrophic pool destruction

## Impact Assessment

### With the 1000 SOL Cap

**Moderate pool impact:** A 1000 SOL buy into a 500 SOL pool is significant but not catastrophic:
- k-invariant ensures pools remain functional after swaps
- The 85% slippage floor prevents execution if manipulation pushes output below 283M
- Subsequent trades can rebalance the pool via arbitrage
- The fund spends at most 1000 SOL per trigger (~4.3% chance per epoch)

**Accumulation analysis:**
- At 24% of tax from $10K daily volume: $180/day accumulation
- At 4.3% trigger rate (48 epochs/day): ~2 triggers/day
- Expected spend: 2 × 1000 SOL × (current SOL price)
- If SOL = $100: Expected spend $200K/day vs accumulation $180/day
- **Fund depletes unless volume is MUCH higher**

**Equilibrium:** The fund reaches equilibrium when:
```
accumulation_rate = trigger_rate × avg_spend_per_trigger
24% × total_tax = 4.3% × 1000_SOL
```

For the fund to grow to 10,000 SOL, volume would need to sustain high enough to accumulate faster than the ~4.3% trigger burn rate.

### Without the Cap (Counterfactual)

If `total_buy_amount = available_sol` (no cap):
- 10,000 SOL into 500 SOL pool = 95% extraction
- Price skew 20x+
- k-invariant still holds but pool effectively destroyed
- **CRITICAL severity** — protocol self-sabotage

## Remediation

**No remediation required.** The existing implementation correctly caps Carnage swaps.

**Optional enhancement (for consideration):**
- Add a **dynamic cap** based on pool reserves: `MAX_CARNAGE = min(1000 SOL, pool_reserve_sol × 0.5)`
- This would limit buys to 50% of pool depth, preventing even the moderate 2x-3x skew
- Trade-off: Reduces Carnage market impact (good for pool health) but also reduces burn effectiveness (bad for tokenomics)

**Economic model note:**
The 1000 SOL cap assumes pools will maintain sufficient depth. If a pool shrinks to 100 SOL, a 1000 SOL Carnage buy is still 10x the reserve. The protocol should monitor pool depth and consider:
1. Adjusting the cap dynamically
2. Splitting large Carnage actions across multiple epochs
3. Implementing minimum pool depth requirements before Carnage execution

## Auditor Notes

**Why this was investigated:**
- The hypothesis was plausible: Carnage fund accumulates 24% of all tax over time
- Without reading the code, "use entire balance" seemed reasonable
- AMM Playbook (Large Swap Impact) and EP-059 (Liquidity Drain) set precedent

**Why it's NOT VULNERABLE:**
- Explicit cap at TWO points in the code path
- Cap is a constant (1000 SOL), not dynamic or bypassable
- Slippage floor provides secondary defense
- k-invariant mathematically guarantees pools survive (though skewed)

**Economic concern (separate from security):**
- The 1000 SOL cap may still be too large relative to pool depth in early/low-liquidity phases
- This is a **protocol design/economic question**, not a security vulnerability
- Recommend pool depth monitoring and dynamic cap consideration for mainnet

**Cross-validation:**
- Checked both `execute_carnage_atomic.rs` (atomic path) and `execute_carnage.rs` (fallback)
- Both enforce the same cap (lines 356 and 363 in atomic, similar in fallback)
- No bypass path found via VRF manipulation or privileged access
- Carnage signer PDA is derived by Epoch Program, not bypassable

---

**Determination:** The protocol correctly implements a hard cap on Carnage buy amounts. The hypothesis that uncapped fund accumulation could destroy pools is **NOT CONFIRMED**. The existing 1000 SOL cap, combined with 85% slippage floors and k-invariant preservation, prevents catastrophic pool imbalance. This is a **non-finding**.
