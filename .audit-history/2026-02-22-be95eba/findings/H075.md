# H075: Reward Per Token Precision Loss With Large total_staked
**Strategy:** STR-075
**Determination:** POTENTIAL
**Severity:** MEDIUM
**Confidence:** MEDIUM

## Summary

The staking program calculates cumulative reward-per-token using the formula:
```
reward_per_token = (pending_rewards * PRECISION) / total_staked
```

Where `PRECISION = 1e18` (u128), and both numerator and denominator are u64 values.

When `total_staked` grows very large (approaching u64::MAX ≈ 1.84×10^19 units), the division can produce results that round down to zero even when rewards are distributed. This causes the cumulative reward-per-token to cease growing, meaning newly staked users receive no rewards from subsequent epochs.

**Critical scenario:** If `total_staked ≈ u64::MAX` and `pending_rewards < total_staked / 1e18`, the division floors to zero, violating the economic invariant that all rewards are distributed.

## Evidence

### Code Locations
1. **update_cumulative.rs:89-93** (instruction handler)
2. **math.rs:107-111** (helper function)
3. **constants.rs:21** (`PRECISION = 1_000_000_000_000_000_000`)

### The Calculation Path

```rust
// update_cumulative.rs:89-93
let reward_per_token = (rewards_added as u128)
    .checked_mul(PRECISION)                    // Safe: u64 * u128 -> u128
    .ok_or(StakingError::Overflow)?
    .checked_div(pool.total_staked as u128)    // ⚠️ Integer division truncates
    .ok_or(StakingError::DivisionByZero)?;
```

The `checked_div` performs **integer division**, which **floors** the result. No `rounded()` or remainder handling occurs.

### Minimum Reward Threshold

For `reward_per_token` to be non-zero:
```
(pending_rewards * PRECISION) / total_staked >= 1
pending_rewards >= total_staked / PRECISION
```

At various stake levels:
| total_staked | Min non-zero reward | Realizable? |
|---|---|---|
| 1e6 (MINIMUM_STAKE) | 1e-12 lamports | ✓ Always |
| 1e9 | 1e-9 lamports | ✓ Always |
| 1e15 | 1 lamport | ✓ Common (tax > 1 sat) |
| 1e18 | 1e3 lamports | ⚠️ Borderline |
| u64::MAX (1.84e19) | ~18.4 lamports | ❌ Below typical rewards |

**At maximum total_staked:** If `total_staked = u64::MAX` and `pending_rewards < 19`, the result is zero even though rewards exist.

### Historical Test Evidence

From update_cumulative.rs:186-200, the test explicitly verifies edge case handling:
```rust
#[test]
fn test_small_rewards_precision() {
    let pending_rewards: u64 = 1;                    // 1 lamport
    let total_staked: u64 = 1_000_000_000_000;      // 1e12 units

    let reward_per_token = (pending_rewards as u128)
        .checked_mul(PRECISION)
        .unwrap()
        .checked_div(total_staked as u128)
        .unwrap();

    // 1 * 1e18 / 1e12 = 1e6 (non-zero) ✓
    assert_eq!(reward_per_token, 1_000_000u128);
}
```

This test passes, confirming 1 lamport on 1e12 staked yields 1e6 (non-zero). But it does **NOT** test the maximum stake scenario.

### No Upper Bound Test

The proptest suite in math.rs:493-527 (reward_conservation) bounds inputs to:
```rust
total_staked in 1u64..=1_000_000_000_000u64,    // Max 1e12 units, not u64::MAX
pending in 1u64..=1_000_000_000_000u64,         // Max 1e12 lamports
```

This is well below the u64::MAX threshold where the edge case manifests. **No test covers the true maximum stake scenario.**

### Mathematical Proof of Risk

For any `reward >= 1` and `total_staked > PRECISION`:
```
floor(reward * PRECISION / total_staked) = 0
when reward * PRECISION < total_staked
when reward < total_staked / PRECISION
```

Example failure case:
```
total_staked = 5e18 (5 million PROFIT)
pending_rewards = 4e9 (4000 SOL)
reward_per_token = (4e9 * 1e18) / 5e18 = 4e9 / 5 = 8e8 (non-zero) ✓

total_staked = 5e18
pending_rewards = 1e9 (1 SOL)
reward_per_token = (1e9 * 1e18) / 5e18 = 1e9 / 5 = 2e8 (non-zero) ✓

total_staked = 5e18
pending_rewards = 1 (1 lamport)
reward_per_token = (1 * 1e18) / 5e18 = 1/5 = 0 (ZERO!) ❌
```

## Impact

### Functional Impact

When `reward_per_token` remains at its prior value (due to zero increment):
1. **For new stakers:** They checkpoint at the current cumulative. Since cumulative doesn't advance, they claim zero rewards from that epoch.
2. **For existing stakers:** Their rewards_delta remains constant, preventing additional rewards.
3. **Over multiple epochs:** The protocol's reward distribution breaks — no rewards flow despite positive pending_rewards.

### Economic Impact

- **Yield becomes unpredictable:** Stakers cannot reliably expect pro-rata returns when total_staked reaches certain thresholds.
- **User experience degrades:** At high TVL, marginal rewards truncate away, discouraging further deposits.
- **Unfair distribution:** Users staked when total_staked was low get all rewards; users staked at high TVL get none.

### Probability Assessment

**Low probability in current protocol state:**
- Current testnet has ~30M PROFIT staked (well below the 1e18 threshold)
- Mainnet would require ~1M PROFIT (1e12 units) to hit the risk zone
- This is achievable but requires significant TVL adoption

**Becomes critical at scale:**
- If protocol reaches billions in TVL (1e18+ units staked), even modest per-epoch rewards (< thousands of SOL) would truncate
- This is a **latent defect** that appears only at high adoption

## Recommendation

### Immediate (Low-Effort)

**Extend proptest bounds to cover full u64 range:**

In math.rs (reward_conservation proptest), change:
```rust
// Current (inadequate):
total_staked in 1u64..=1_000_000_000_000u64,

// Recommended:
total_staked in MINIMUM_STAKE..=u64::MAX,
pending in 1u64..=u64::MAX,
```

This will expose any precision loss in the CI/CD pipeline. The test may fail at the extreme bounds, validating the hypothesis.

### Short-Term (Documentation)

Add explicit invariant documentation to `math.rs:add_to_cumulative`:
```rust
/// INVARIANT: reward_per_token > 0 for all pending_rewards > 0 and total_staked > 0.
///
/// This holds when: pending_rewards > total_staked / PRECISION
/// At u64::MAX scale (1.84e19 units), requires pending >= ~18.4 lamports.
/// Protocol tax (~1% of swaps) ensures this in realistic scenarios, but
/// users with extreme dust-level pending rewards may see zero increments.
```

### Long-Term (Robust Solution)

**Implement residual handling:**

Store remainder-per-token separately and accumulate it over epochs:
```rust
// Instead of:
reward_per_token = (pending * PRECISION) / total_staked;

// Do:
(reward_per_token, remainder) = div_with_remainder(pending * PRECISION, total_staked);
cumulative_remainder += remainder;
if cumulative_remainder >= total_staked {
    overflow_reward_per_token = cumulative_remainder / total_staked;
    cumulative_remainder %= total_staked;
    reward_per_token += overflow_reward_per_token;
}
```

This prevents dust accumulation and ensures all rewards eventually distribute.

Alternatively, **batch-scale the PRECISION:** If the protocol caps total_staked to a known maximum (e.g., 1e15 units), reduce PRECISION proportionally to maintain numerical stability while preventing overflow.
