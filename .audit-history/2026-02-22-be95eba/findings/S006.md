# S006: Comprehensive Devnet Feature Gate Audit — No Additional Leakage Beyond force_carnage

**Strategy:** S006
**Determination:** VERIFIED SAFE
**Severity:** INFO
**Confidence:** HIGH

## Summary

Following the discovery of `force_carnage` as a devnet-gated backdoor (H004), this investigation performed an exhaustive search for **all** `#[cfg(feature = "devnet")]` blocks across all 5 programs to determine if other test helpers, debug logging, or bypass logic could leak into mainnet builds. **Result: The devnet feature gate is used correctly and conservatively.** Only 3 subsystems use devnet features:

1. **force_carnage instruction** (Epoch Program) — Admin backdoor for rapid Carnage testing. Already flagged as H004.
2. **SWITCHBOARD_PROGRAM_ID** (Epoch Program) — Selects between devnet/mainnet Switchboard oracle addresses. **Safe** — required for correct operation.
3. **treasury_pubkey() and profit_mint()** (Tax Program) — Devnet uses test wallet, mainnet uses placeholder. **Safe** — required for deployment.
4. **SLOTS_PER_EPOCH** (Epoch Program) — Devnet uses 750 slots (~5 min), mainnet uses 4500 (~30 min). **Safe** — required for testing.

No other devnet-gated code exists. No debug logging, no test bypass logic, no backdoors beyond `force_carnage`.

#### 1. force_carnage Instruction (Epoch Program)

**Location:** `programs/epoch-program/src/instructions/force_carnage.rs` (entire file)

**Gate 1 — Module level** (`instructions/mod.rs:7-8`):

**Gate 2 — Instruction level** (`lib.rs:261-264`):

**Purpose:** Admin-gated test helper to trigger Carnage without waiting for VRF randomness. Allows rapid testing of all Carnage execution paths (Burn, Sell, BuyOnly).

**Security Impact:** HIGH (already flagged as H004)
- Bypasses VRF security guarantees
- Enables arbitrary Carnage timing and direction if key is compromised
- Double-gated (module + instruction) for defense-in-depth

**Mainnet Action Required:** MUST NOT be compiled into mainnet binary
- Build script must run WITHOUT `--devnet` flag
- Post-build verification: `grep force_carnage target/idl/epoch_program.json` must return empty

#### 2. SWITCHBOARD_PROGRAM_ID (Epoch Program)

**Location:** `programs/epoch-program/src/constants.rs:45-49`

```rust
#[cfg(feature = "devnet")]

#[cfg(not(feature = "devnet"))]
pub const SWITCHBOARD_PROGRAM_ID: Pubkey = switchboard_on_demand::ON_DEMAND_MAINNET_PID;
```

**Purpose:** Compile-time selection of Switchboard oracle program ID based on deployment environment.

**Security Impact:** SAFE — Required for Correct Operation
- Devnet VRF randomness accounts are owned by `ON_DEMAND_DEVNET_PID`
- Mainnet VRF randomness accounts are owned by `ON_DEMAND_MAINNET_PID`
- Used as `owner` constraint on randomness accounts to prevent fake-randomness injection attacks (line `epoch-program/src/instructions/trigger_epoch_transition.rs:101`)
- **If wrong PID is used:** All VRF transactions fail with `ConstraintOwner` error (safe failure, not exploitable)

**Evidence of Safe Usage:**
```rust

#[account(
    mut,
    owner = constants::SWITCHBOARD_PROGRAM_ID,  // ← Uses feature-gated constant
)]
pub randomness: AccountInfo<'info>,
```

**Mainnet Action Required:** Build without `--devnet` flag → automatically uses mainnet PID

#### 3. SLOTS_PER_EPOCH (Epoch Program)

**Location:** `programs/epoch-program/src/constants.rs:57-61`


**Purpose:** Devnet uses shorter epochs (~5 minutes) for rapid testing. Mainnet uses 30-minute epochs for better VRF cost amortization and reduced epoch transition spam.

**Security Impact:** SAFE — Testing Convenience Only

- Used in `trigger_epoch_transition` to validate slot boundaries (line `trigger_epoch_transition.rs:134`)
- **If wrong value is used on mainnet:** Epochs would be too frequent (more VRF costs, more bounty payouts) but not a security vulnerability
- No bypass logic, no admin override, no attack surface

**Evidence of Safe Usage:**
```rust

    epoch_state.epoch_end_slot <= current_slot,
    EpochError::EpochNotExpired

// epoch_end_slot is set to epoch_start + SLOTS_PER_EPOCH
```

**Mainnet Action Required:** Build without `--devnet` flag → automatically uses 4500 slots

## Recommendation

### 1. Mainnet Build Process Hardening (CRITICAL)

**Problem:** No enforcement mechanism prevents accidental mainnet deployment of devnet-featured binaries.

**Solution:** Add post-build verification to `scripts/deploy/build.sh`:

```bash
# After anchor build, verify force_carnage is NOT in IDL
echo "[SECURITY CHECK] Verifying force_carnage is excluded..."
if grep -q "force_carnage" target/idl/epoch_program.json; then
  echo "  ERROR: force_carnage found in epoch_program.json"
  echo "  This binary was built with --features devnet and MUST NOT be deployed to mainnet."
  echo "  Rebuild with: ./scripts/deploy/build.sh (no --devnet flag)"
  exit 1
fi
echo "  OK: force_carnage not found (mainnet-safe build)"
```

**Why in build.sh and not deploy.sh?**
- Build verification catches errors before deployment (fail-fast)
- Deploy script might run on CI with pre-built artifacts
- Build script is the single source of truth for compilation flags

### 2. Mainnet Deployment Checklist Enhancement (MEDIUM)

**Current:** `Docs/mainnet-checklist.md` lists treasury/mint placeholders but doesn't emphasize the build flag risk.

**Add to checklist:**

## Conclusion

This exhaustive audit of all `#[cfg(feature = "devnet")]` blocks across all 5 programs confirms that **no additional security risks exist beyond force_carnage (H004)**. The devnet feature gate is used correctly:

1. **Critical backdoor:** force_carnage (already flagged)

3. **Deployment placeholders:** treasury_pubkey(), profit_mint() (tracked in checklist)

The primary risk is **process failure** (deploying devnet-built binary to mainnet), not **code structure**. Recommendations focus on build verification, deployment checklists, and post-deployment validation.

**Status:** No new vulnerabilities discovered. Existing H004 finding is comprehensive. This investigation provides defense-in-depth confirmation that no other devnet-gated code could leak into production.
