# H006: Carnage Slippage Floor is NOT Bypassable via Pool Manipulation

**Strategy:** STR-006
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis that Carnage's 85% slippage floor can be bypassed via pool manipulation is **DISPROVEN**. The Carnage execution flow uses a **same-instruction atomicity pattern** where pool reserves are read immediately before the swap CPI, with all operations occurring within a single instruction handler. This eliminates any gap where an attacker could sandwich the reserve read and swap execution.

## Evidence

**Atomicity Guarantee:** All these operations occur within a **single Solana instruction**. Solana's execution model guarantees that instructions are atomic — no other transaction can modify account state between operations within the same instruction.

This reads from the **same pool account** that will be mutated by the `execute_buy_swap` CPI. Because both operations are in the same instruction, Solana guarantees:
1. No other transaction can execute between the reserve read and the swap
2. The reserves read are the exact reserves that will be used for the swap calculation
3. Any prior manipulation in a different transaction would be visible to BOTH the read and the swap

### Attack Scenario Analysis

**Proposed attack:**
1. Attacker observes `carnage_pending = true` on-chain
2. Attacker front-runs with large buy to inflate token price
3. Carnage executes with manipulated reserves, failing slippage check or receiving fewer tokens
4. Attacker back-runs to sell

**Why it fails:**

**Front-run timing:** The attacker's manipulation transaction would execute in a **prior slot/block**. By the time Carnage's `execute_carnage_atomic` instruction runs:
- Pool reserves reflect the manipulated state
- **Both** `read_pool_reserves` (line 376) AND the AMM swap (line 396) see the SAME manipulated reserves
- The slippage floor calculation uses `reserve_token / (reserve_sol + total_buy_amount)` — if reserves are skewed, the expected output is ALSO skewed proportionally
- The 85% floor checks: `actual_output >= 0.85 * expected_output_from_current_reserves`

**Mathematical proof of proportionality:**

Assume normal reserves: `R_sol = 1000 SOL`, `R_token = 10,000 CRIME`
Carnage buys 100 SOL worth.

**Expected output (normal):**
`expected = (10,000 * 100) / (1000 + 100) = 909 CRIME`
`min_output = 909 * 0.85 = 772 CRIME`

Now assume attacker front-runs with 500 SOL buy, removing ~3,333 CRIME:
- New reserves: `R_sol = 1500`, `R_token = 6,667`

**Expected output (manipulated reserves):**
`expected_manip = (6,667 * 100) / (1500 + 100) = 416 CRIME`
`min_output_manip = 416 * 0.85 = 354 CRIME`

**Actual AMM output with manipulated reserves:**
The AMM swap will output approximately 416 CRIME (constant product formula).

**Slippage check:** `416 >= 354` ✓ **PASSES**

**The attack produces no profit:**
- Attacker spent 500 SOL to remove 3,333 CRIME (paying LP fees + slippage)
- Carnage bought 100 SOL of CRIME at the new (worse) price, receiving ~416 CRIME
- Attacker now must sell 3,333 CRIME back to restore the pool
- Net result: Attacker paid LP fees twice (in and out) for no gain
- Carnage received proportionally correct output for the current reserve state

**Why the slippage floor doesn't help the attacker:** The floor is calculated against **post-manipulation reserves**, not pre-manipulation. The attacker cannot force Carnage to accept a worse rate than what the current reserves dictate because the "expected" value in the slippage formula updates with the reserves.

### Jito Bundle Analysis

**Could bundling help?** Even if the attacker uses Jito bundles to guarantee transaction ordering within the same slot:
1. The manipulation transaction (TX1) executes first, updating pool reserves
2. Carnage transaction (TX2) executes second, reading the post-TX1 reserves
3. The slippage floor in TX2 is calculated against the post-TX1 reserves
4. TX2's AMM swap executes against the post-TX1 reserves
5. Both the floor calculation and swap use the SAME reserve state

The attacker gains nothing because they cannot create a gap between the reserve read and the swap — both happen in the same instruction of TX2.

## Root Cause

**No vulnerability exists.** The concern would be valid if:
1. Reserves were read in a prior transaction/instruction, OR
2. The slippage floor was hardcoded or derived from external data, OR
3. Multiple instructions were required (read in IX1, swap in IX2)

None of these apply. The implementation uses correct **same-instruction atomicity**.
