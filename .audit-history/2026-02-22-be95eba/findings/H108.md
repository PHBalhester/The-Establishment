# H108: All Amount Parameters Use Unsigned Integers

**Strategy:** STR-108: Signed Integer Used for Unsigned Amount

**Determination:** NOT EXPLOITABLE — All amount parameters across the protocol use `u64`. No signed integers (`i64`, `i128`, `i32`, etc.) are used for any financial amounts, balances, or reserves.

**Severity:** INFORMATIONAL

**Confidence:** 100% — Complete source code review verified across all 5 programs.

---

## Summary

This finding investigates whether any amount field is `i64` instead of `u64`, which could allow negative amounts to cause unexpected behavior. The audit confirms:

1. **All amount parameters are `u64`**: Every instruction parameter, state field, and event field representing financial quantities uses unsigned 64-bit integers.
2. **Signed integers reserved for timestamps only**: The only signed integers in the codebase are `i64` fields used for Unix timestamps (via Solana's `Clock::unix_timestamp`).
3. **No negative amount attack surface**: Since all amounts are unsigned, negative values cannot be passed to instructions, preventing underflow attacks via signed-to-unsigned casting.


**Epoch Program Events (`events.rs:18, 91`)**:

All `i64` usage is explicitly for Unix timestamps sourced from Solana's `Clock::unix_timestamp`, which is defined as `i64` in the Solana SDK. This is standard practice and not a vulnerability.

### Search Methodology

1. **Global grep for signed types**: `grep -r "i64|i128|i32|i16|i8" programs/`
2. **Function signature scan**: All handler functions verified for amount parameters
3. **State struct analysis**: All `pub struct` definitions checked for financial fields
4. **Balance field audit**: Specific search for `balance`, `amount`, `reserve` with signed types
5. **Result**: 6 files contain signed integers; all are events with timestamps, not financial data

## Impact

**Security Impact: NONE**

- No negative amounts can be passed to instructions (type system enforces)
- No integer casting vulnerabilities from `i64` to `u64` (never occurs)
- No underflow via negative subtraction (all amounts unsigned)
- No fee bypass via negative tax calculations (not applicable)

**Protocol Impact: POSITIVE**

The exclusive use of `u64` for amounts is a security best practice:
1. Prevents entire classes of signed/unsigned casting bugs (EP-020)
2. Aligns with Rust's type safety for unsigned quantities
3. Consistent with Solana and Anchor conventions (all token amounts are `u64`)
