## Summary

STR-024 hypothesized that the tax split computation (75% staking, 24% carnage, remainder treasury) could leave 1-2 lamports per swap locked permanently in the Tax Program due to integer rounding. **Investigation confirms this does NOT occur.** The tax split uses a **remainder pattern** where the treasury explicitly absorbs all rounding dust, ensuring `staking + carnage + treasury == total_tax` always. Furthermore, all three portions are transferred out to their respective destinations (staking escrow, carnage vault, treasury account), so no lamports remain in the Tax Program.

## Evidence

### Tax Split Implementation

File: `programs/tax-program/src/helpers/tax_math.rs:79-104`

```rust
pub fn split_distribution(total_tax: u64) -> Option<(u64, u64, u64)> {
    // Handle micro-tax edge case: if tax < 4 lamports, all to staking
    if total_tax < 4 {
        return Some((total_tax, 0, 0));
    }

    let total = total_tax as u128;

    // Staking: floor(total * 75 / 100)
    let staking_u128 = total.checked_mul(75)?.checked_div(100)?;
    let staking = u64::try_from(staking_u128).ok()?;

    // Carnage: floor(total * 24 / 100)
    let carnage_u128 = total.checked_mul(24)?.checked_div(100)?;
    let carnage = u64::try_from(carnage_u128).ok()?;

    // Treasury: remainder (absorbs rounding dust)
    // This ensures the invariant: staking + carnage + treasury == total_tax
    let treasury = total_tax
        .checked_sub(staking)?
        .checked_sub(carnage)?;

    Some((staking, carnage, treasury))
}
```

**Key observation:** Line 99-101 computes `treasury = total_tax - staking - carnage`. This is a **remainder pattern** that guarantees conservation: all rounding dust goes to treasury, not to the Tax Program.

## Root Cause Analysis

The hypothesis was based on a common anti-pattern in naive tax split implementations:

```rust
// NAIVE (BROKEN) PATTERN:
let staking = total * 75 / 100;
let carnage = total * 24 / 100;
let treasury = total * 1 / 100;
// Problem: staking + carnage + treasury < total if total is not divisible by 100
// The missing lamports are "lost" to integer rounding
```

**Dr Fraudsworth uses the correct pattern:**

```rust
// CORRECT PATTERN (REMAINDER ABSORPTION):
let staking = total * 75 / 100;   // floor
let carnage = total * 24 / 100;   // floor
let treasury = total - staking - carnage;  // absorbs all dust
// Result: staking + carnage + treasury == total (always)
```

This is a **best practice** for multi-way splits with fractional percentages. The last recipient absorbs all rounding dust, ensuring perfect conservation.

## Attack Scenario

**None.** The original hypothesis was:

1. Attacker makes many small swaps
2. Each swap produces 1-2 lamports of rounding dust
3. Dust accumulates in Tax Program, locked permanently
4. Over time, this creates accounting drift

**Why this doesn't work:**
- The tax is transferred FROM the user, not FROM the Tax Program
- The split is applied BEFORE any transfers
- All three portions (staking, carnage, treasury) are transferred OUT immediately
- The Tax Program holds zero lamports at the end of the swap
- The treasury explicitly receives the rounding dust (by design)

## Impact Assessment

**Impact:** NONE (false positive)

- **Accounting drift:** Does not occur. Treasury absorbs dust.
- **Permanent lockup:** Does not occur. All tax portions are transferred out.
- **Value extraction:** Not possible. The dust goes to the treasury, which is a legitimate protocol destination.

**Where the "dust" actually goes:**

For a 100-lamport tax:
- Staking: `floor(100 * 75 / 100)` = 75 lamports
- Carnage: `floor(100 * 24 / 100)` = 24 lamports
- Treasury: `100 - 75 - 24` = **1 lamport** (the "dust")

The 1 lamport is transferred to the treasury account, which is controlled by the protocol admin. This is the **intended behavior** — the treasury is designed to receive the remainder after the 75/24 split.

## Remediation

**No remediation required.** The implementation is correct as-is.

### Why the Remainder Pattern is Optimal

1. **Conservation guarantee:** `sum == total` is guaranteed mathematically, not verified post-hoc.
2. **Gas efficient:** One subtraction instead of percentage multiplication for the last recipient.
3. **No edge cases:** Works for any input value, including dust amounts.
4. **Fair to protocol:** The treasury receives slightly more than 1% on average, compensating for its administrative role.

### Alternative (NOT recommended)

Some protocols use a "dust collection" account that accumulates rounding errors and is periodically distributed. This adds complexity with no benefit. The remainder pattern is simpler and equally correct.
