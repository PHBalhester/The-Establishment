# S001: Staking Escrow Vault Can Be Drained Below Rent-Exempt Minimum

**Strategy:** S001
**Determination:** CONFIRMED
**Severity:** HIGH
**Confidence:** HIGH

## Summary

The Staking Program's `claim` and `unstake` instructions transfer SOL rewards from the `escrow_vault` PDA to users via direct lamport manipulation (`try_borrow_mut_lamports`). While these instructions verify that `escrow_balance >= rewards_to_claim` before the transfer, they **do not account for the rent-exempt minimum** that must remain in the account. This mirrors the bounty vault bug (H001) but affects a different subsystem with different operational impact.

When the escrow vault balance is between `rent_exempt_minimum` (890,880 lamports) and `rewards_to_claim`, the check passes but the transfer drains the account below its rent floor. The Solana runtime will either reject the transaction or mark the account for garbage collection, preventing future reward distributions.

## Root Cause

Both `claim` and `unstake` instructions use a **single-threshold check** that only validates the transfer amount can be paid:

```rust
if escrow_balance >= rewards_to_claim  // Only checks reward amount, not reward + rent
```

The correct check should be:

```rust
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(8);  // escrow_vault Account type has 8-byte discriminator
let available_balance = escrow_balance.saturating_sub(rent_exempt_min);
if available_balance >= rewards_to_claim
```

## Attack Scenario

### Scenario 1: Organic Exhaustion (High Likelihood)

**Preconditions:**
- Protocol operates normally for extended period
- Multiple users stake and accumulate rewards
- User reward claims drain the escrow vault over time
- Escrow vault balance approaches rent-exempt minimum (< 1 SOL)

**Exploitation Steps:**
1. Alice has 500,000 lamports (0.0005 SOL) in pending rewards
2. Escrow vault balance: 1,200,000 lamports (0.0012 SOL)
3. Rent-exempt minimum for Account<EscrowVault>: ~897,120 lamports
4. Alice calls `claim_rewards()`
5. Check at line 103 passes: `1,200,000 >= 500,000` ✓
6. Transfer executes: `escrow_vault.lamports = 1,200,000 - 500,000 = 700,000`
7. **Post-transfer balance (700K) < rent-exempt (897K)**
8. Solana runtime behavior:
   - **Option A (strict):** Transaction reverts with "insufficient lamports for rent" error
   - **Option B (lenient):** Transaction succeeds but account flagged for garbage collection
   - **Option C (actual):** Depends on Solana version; modern clients reject in simulation

**Impact:**
- If transaction reverts: Claim/unstake operations freeze for all users when escrow is low
- If garbage collection: Escrow account deleted, all pending rewards permanently lost
- Either way: Requires emergency admin intervention to replenish escrow

**Recovery:**
```bash
# Admin must manually transfer SOL to escrow PDA to bring above threshold
solana transfer <escrow_vault_pda> 0.01 --from admin.json
```

### Scenario 2: Griefing Attack (Medium Likelihood)

**Preconditions:**
- Attacker monitors on-chain escrow vault balance
- Attacker has staked tokens with accumulated rewards
- Escrow vault balance in danger zone: `[rent_exempt_min, rent_exempt_min + attacker_rewards]`

**Exploitation Steps:**
1. Attacker waits for escrow balance to drop to ~900,000 lamports (via organic claims)
2. Attacker has 100,000 lamports pending rewards (easily achievable via small stake)
3. Attacker calls `claim_rewards()` or `unstake()`
4. Transaction either:
   - Fails (blocking all future claims/unstakes for everyone)
   - Succeeds but triggers garbage collection (permanent fund loss)
5. **Cost to attacker:** Zero (if TX fails) or minimal (100K lamports = $0.0001)
6. **Impact:** Protocol staking subsystem frozen until admin intervention

## Impact Assessment

### Severity: HIGH (Not CRITICAL)

**Why HIGH, not CRITICAL:**
- Unlike H001 (bounty bug which freezes **epoch transitions** = core protocol function), this only affects the **staking rewards subsystem**
- Epoch transitions, carnage execution, and trading continue unaffected
- Admin can restore by depositing SOL to escrow PDA
- Staked PROFIT tokens remain safe (in separate stake_vault, not escrow)

**Why not MEDIUM:**
- **All stakers** (not just one user) are affected — no claims/unstakes possible
- Pending rewards are at risk of permanent loss (if garbage collection occurs)
- Requires manual admin intervention to fix (no auto-recovery)
- Likely to manifest organically (not just theoretical)

### User Impact

| User Type | Impact |
|-----------|--------|
| Stakers trying to claim rewards | Transaction fails with InsufficientEscrowBalance error |
| Stakers trying to unstake | Cannot unstake if they have pending rewards |
| New stakers | Can stake (only deposits PROFIT, no SOL transfer from escrow) |
| Traders | Unaffected (trading swaps work, but tax->staking deposits also blocked) |

### Protocol Impact

| Subsystem | Impact |
|-----------|--------|
| Staking Rewards | **FROZEN** — Cannot claim or distribute |
| Epoch Transitions | **WORKS** — Unaffected |
| Carnage Execution | **WORKS** — Uses separate sol_vault |
| Tax Collection | **DEGRADED** — Tax→staking CPI in swap_sol_buy.rs calls deposit_rewards, which WILL SUCCEED (only updates counter), but the SOL transfer happens before CPI. The freeze only affects OUTBOUND from escrow, not INBOUND. |

**Correction:** Tax deposits to escrow are **user→escrow transfers** (see swap_sol_buy.rs:128-140), which increase escrow balance. The rent bug only affects **escrow→user withdrawals**. Tax collection continues working, which means escrow can self-heal over time from trading volume.

### Financial Impact

- **Pending rewards at risk:** Total `StakePool.pending_rewards - total_claimed` (could be 10s of SOL in production)
- **Recovery cost:** Admin gas + manual transfer (minimal)
- **Reputational cost:** Medium (staking frozen, but not total protocol failure)

## Proof of Concept

```rust
// Scenario: Escrow has exactly rent + 0.0001 SOL
// User tries to claim 0.0001 SOL reward

let escrow_balance = 897_220;  // rent_exempt (897,120) + 100 lamports
let rewards_to_claim = 100_000; // 0.0001 SOL

// Vulnerable check (claim.rs:103)
assert!(escrow_balance >= rewards_to_claim);  // ✓ PASSES (897,220 >= 100,000)

// Post-transfer balance
let post_transfer = escrow_balance - rewards_to_claim;
// = 897,220 - 100,000
// = 797,220 lamports

// Runtime check (implicit in Solana)
let rent_exempt_min = 897_120;  // For Account<EscrowVault> with 8-byte discriminator
assert!(post_transfer >= rent_exempt_min);  // ✗ FAILS (797,220 < 897,120)

// Transaction ABORTED with error: "insufficient lamports for rent"
```

**Critical Range:**
- Danger zone when `escrow_balance ∈ [rent_exempt_min, rent_exempt_min + typical_reward]`
- For typical rewards of 0.001-0.01 SOL (1M-10M lamports), danger zone: **[897K, 10.9M lamports]**
- Much wider danger zone than bounty bug's [1M, 1.89M]

## Remediation

### Fix 1: Reserve Rent-Exempt Minimum (Recommended)

**File:** `programs/staking/src/instructions/claim.rs`

```rust
// Line 100-111 BEFORE:
let escrow_balance = ctx.accounts.escrow_vault.lamports();
if escrow_balance < rewards_to_claim {
    emit!(EscrowInsufficientAttempt { ... });
    return Err(StakingError::InsufficientEscrowBalance.into());
}

// AFTER (defensive pattern):
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(8);  // escrow_vault is Account<EscrowVault> with 8-byte discriminator
let escrow_balance = ctx.accounts.escrow_vault.lamports();
let available_balance = escrow_balance.saturating_sub(rent_exempt_min);

if available_balance < rewards_to_claim {
    emit!(EscrowInsufficientAttempt {
        user: ctx.accounts.user.key(),
        requested: rewards_to_claim,
        available: available_balance,  // Log available, not total
        slot: clock.slot,
    });
    return Err(StakingError::InsufficientEscrowBalance.into());
}
```

**Same fix in unstake.rs:162-166.**

### Fix 2: Add Constant (Alternative)

Define in `staking/src/constants.rs`:

```rust
/// Rent-exempt minimum for Account<EscrowVault> (8 data bytes for discriminator).
/// Calculated via: Rent::minimum_balance(8) ≈ 897,120 lamports on mainnet.
/// Hardcoded to avoid runtime Rent sysvar access overhead on every check.
pub const ESCROW_RENT_EXEMPT_MIN: u64 = 897_120;
```

Then use in both claim and unstake:

```rust
let available_balance = escrow_balance.saturating_sub(ESCROW_RENT_EXEMPT_MIN);
```

**Trade-off:** Hardcoding rent value risks staleness if Solana rent schedule changes. Using `Rent::get()?` is more future-proof but costs ~400 CU per call.
