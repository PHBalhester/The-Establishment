# H119: VRF Oracle Single Point of Failure

**Strategy:** STR-119: VRF Oracle Single Point of Failure
**Determination:** CONFIRMED — Single oracle dependency with timeout recovery
**Severity:** LOW
**Confidence:** HIGH

---

## Summary

Dr Fraudsworth depends exclusively on **Switchboard On-Demand VRF** for randomness-driven game mechanics (tax rates, Carnage triggers). While this creates a single oracle dependency, the protocol includes a timeout recovery mechanism (300-slot window, ~2 minutes) that prevents permanent deadlock. If Switchboard experiences an outage, epoch transitions stall but the protocol remains functional with frozen tax rates and suspended Carnage. No funds are at risk from oracle unavailability.

### 2. VRF Lifecycle: Timeout Recovery Mechanism

**Files**: `trigger_epoch_transition.rs:1-246`, `retry_epoch_vrf.rs:1-176`, `consume_randomness.rs:1-420`

The protocol implements a 3-phase VRF lifecycle with explicit timeout recovery:

```
PHASE 1: COMMIT (TX 2)
├─ Client bundles SDK commitIx + trigger_epoch_transition
├─ Epoch program binds pending_randomness_account to epoch_state
└─ Sets vrf_pending=true, vrf_request_slot=current_slot

PHASE 2: REVEAL + CONSUME (TX 3, ~3 slots later)
├─ Oracle reveals randomness on-chain
├─ Client bundles SDK revealIx + consume_randomness
├─ consume_randomness reads VRF bytes via get_value(clock.slot)
├─ Derives tax rates, checks Carnage trigger
└─ Sets vrf_pending=false, taxes_confirmed=true

PHASE 3: TIMEOUT RECOVERY (if oracle fails after 300 slots)
├─ Anyone can call retry_epoch_vrf (permissionless)
├─ Validates elapsed_slots > 300 (line 72: retry_epoch_vrf.rs)
├─ Binds fresh randomness account
└─ Resets vrf_request_slot, re-enters COMMIT state
```

### 3. Switchboard Oracle Assignment and Retry Behavior

**Per MEMORY.md context:**

> "VRF gateway rotation DOES NOT WORK": Each randomness account is assigned to a specific oracle. If that oracle goes offline, the epoch is stuck until timeout recovery. Alternative gateways serve different oracles whose signatures fail on-chain (0x1780). Only retry the default gateway. If down, use VRF timeout recovery with fresh randomness.

**Analysis:**
- Each randomness account created by the client is assigned to a **specific Switchboard oracle**.
- If that oracle is offline or experiencing issues, the VRF will not reveal within expected timeframe (~3 slots).
- The timeout mechanism allows creation of a **fresh randomness account** after 300 slots, which **may be assigned to a different (working) oracle**.
- This is the intended recovery path. Fresh randomness is not guaranteed to succeed, but it provides an opportunity to reach a different oracle.

### 4. Timeout Constants and Validation

**File**: `epoch-program/src/instructions/retry_epoch_vrf.rs:71-74`

```rust
require!(
    elapsed_slots > VRF_TIMEOUT_SLOTS,
    EpochError::VrfTimeoutNotElapsed
);
```

**From constants.rs** (via grep of audit context):
```
VRF_TIMEOUT_SLOTS: 300 slots (~2 minutes at 400ms/slot)
```

The timeout window is **strictly enforced**: `elapsed_slots > 300` (not `>=`). This means:
- At exactly 300 slots elapsed: NOT allowed (continues waiting)
- At 301 slots elapsed: ALLOWED (can retry)
- Anyone can call `retry_epoch_vrf` permissionlessly — recovery is not gated

### 5. Oracle Dependency Tracking in Code

**File**: `epoch-program/src/instructions/trigger_epoch_transition.rs:151-173`

```rust
// === 3. Validate randomness account (On-Demand freshness checks) ===
let randomness_data = {
    let data = ctx.accounts.randomness_account.try_borrow_data()?;
    RandomnessAccountData::parse(data).map_err(|_| EpochError::RandomnessParseError)?
};

// Freshness check: seed_slot must be within 1 slot of current
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
require!(slot_diff <= 1, EpochError::RandomnessExpired);

// Not-yet-revealed check: must still be in commit phase
if randomness_data.get_value(clock.slot).is_ok() {
    msg!("Randomness already revealed - cannot use for new commit");
    return Err(EpochError::RandomnessAlreadyRevealed.into());
}
```

The protocol calls `randomness_data.get_value(clock.slot)` only after Switchboard has written the reveal. This is a **passive dependency check**; the protocol does not create or manage the oracle itself, only consumes Switchboard's output.

### 6. Impact Analysis: Oracle Outage Scenario

**If Switchboard experiences a complete outage:**

1. **At trigger_epoch_transition**: Succeeds (only reads seed_slot, doesn't require reveal yet)
2. **At consume_randomness** (~3 slots later): Fails with `RandomnessNotRevealed` because oracle never revealed VRF bytes
3. **Tax rates**: Remain at previous epoch's values (frozen, no randomization)
4. **Carnage**: Does not trigger for the pending epoch (Carnage state not updated)
5. **Staking**: Continues normally (yield distribution is independent of VRF)
6. **Swaps**: Continue at old tax rates (not broken, just static)
7. **After 300 slots**: Anyone can call `retry_epoch_vrf` with fresh randomness to a potentially different oracle
8. **Fund security**: Zero impact — all pool reserves, staked tokens, and user balances remain secure (VRF only affects tax rates and Carnage triggers, not fund movements)

### 7. No Fallback Mechanism

**Evidence**: No alternative oracle provider exists in the codebase. Grep of all epoch-program files reveals:
- No Pyth oracle integration
- No chainlink dependency
- No on-chain RNG as fallback
- No off-chain signature accumulation
- No multi-oracle voting

If Switchboard is unavailable, the only recovery path is `retry_epoch_vrf` with a fresh randomness account, hoping it routes to a working oracle.

## Impact

### Liveness Impact: MEDIUM
- **Epoch advancement**: Halts if Switchboard oracle is offline (cannot consume randomness)
- **Tax rate updates**: Freeze at previous epoch values
- **Carnage triggering**: Suspended (no new Carnage events)
- **Duration**: Up to 300 slots (~2 minutes) until retry mechanism can be triggered
- **Recovery**: Permissionless (anyone can call `retry_epoch_vrf`)

### Fund Safety Impact: NONE
- **Pool reserves**: Secured by AMM invariants and PDA authority (VRF-independent)
- **Staked PROFIT**: Withdrawn/claimed independent of VRF
- **User tokens**: Transferable (whitelisted only, not VRF-dependent)
- **Oracle absence does not enable theft or manipulation**

### User Experience Impact: LOW-MEDIUM
- **Sophisticated traders**: Can exploit frozen tax rates if aware of outage (sandwich attack on rate change, but no rate change if oracle down)
- **Carnage expectation**: Game mechanic becomes static
- **Yield**: Staking rewards continue (VRF-independent)

## Mitigating Factors

1. **Timeout Recovery Mechanism**: Protocol does not deadlock permanently. After 300 slots, fresh randomness can retry with a potentially different oracle.

2. **Switchboard Mainnet Availability**: Switchboard has strong operational track record on Solana mainnet. On-Demand VRF (unlike legacy aggregators) is designed for low-latency oracle operations.

3. **No Collateral at Risk**: VRF is used only for game mechanics (tax rates, Carnage triggers), not for collateral valuation, lending decisions, or liquidation logic. Unlike lending protocols, oracle failure does not cascade into insolvency.

4. **Passive Dependency**: The protocol does not call Switchboard APIs; it only reads randomness accounts that the client creates and Switchboard populates. This reduces integration complexity.

5. **Permissionless Recovery**: Anyone can trigger `retry_epoch_vrf` — recovery is not gated to admin or a specific caller.

6. **Protocol Remains Functional**: Without VRF, swaps continue at frozen tax rates. Funds cannot be stolen. The protocol degrades gracefully.

## Recommendation

**Risk Level**: ACCEPTABLE

This is a **liveness concern, not a safety concern**. The single-oracle dependency is a known trade-off in VRF-driven protocols. The 300-slot timeout recovery mechanism is the industry-standard mitigation.

**Operational Mitigations**:

1. **Monitor Switchboard Health**: Add off-chain monitoring to track oracle reveal latency. Alert if reveals consistently exceed 10 slots.

2. **Document Dependency**: Update operational runbook to explicitly state: "Protocol depends on Switchboard On-Demand VRF. If Switchboard experiences extended outage (>300 slots), epoch advancement will stall. Tax rates will freeze and Carnage will not trigger. Call `retry_epoch_vrf` to attempt recovery with fresh randomness."

3. **Test Failure Scenarios**: Add integration test simulating oracle reveal timeout. Verify `retry_epoch_vrf` transitions state correctly when called after 300 slots.

4. **No Actionable Code Changes**: The retry mechanism is correctly implemented. Switchboard is the appropriate oracle choice for Solana randomness. Do not add secondary oracle provider (introduces complexity; Switchboard monopoly is by design on Solana).

---

## No False Positives

The following patterns were examined and found NOT to be security issues:

- **"No confidence interval on VRF"**: VRF outputs are deterministic and cryptographically verified by Switchboard. Unlike price oracles, there is no "uncertainty" — randomness is either revealed (cryptographically valid) or not. No confidence threshold needed.

- **"VRF timeout causes permanent deadlock"**: False. The 300-slot timeout recovery explicitly prevents deadlock by allowing retry with fresh randomness.

- **"Fresh randomness may not reach a working oracle"**: True, but acceptable. The retry mechanism provides an **opportunity** to reach a different oracle. If both original and fresh randomness hit offline oracles, the cycle can repeat (300-slot windows). This is probabilistically unlikely on mainnet and is the cost of a single-oracle design.

- **"Admin could call retry_epoch_vrf to reroll"**: Incorrect. `retry_epoch_vrf` is permissionless — anyone can call it, not just admin. It requires timeout to have elapsed (>300 slots), so it cannot be used for fast rerolling.

## Conclusion

Dr Fraudsworth's reliance on Switchboard On-Demand VRF is a **known architectural trade-off**. The protocol makes an explicit choice to use a single oracle provider for simplicity and cost. The 300-slot timeout recovery mechanism is the appropriate mitigation and is correctly implemented. No funds are at risk from oracle unavailability. Operational monitoring and clear documentation of the dependency is the recommended path forward.

**This is not a vulnerability. It is a documented design pattern with appropriate compensating controls.**
