# H096: Token Account Rent Exemption Not Checked on Create

**Strategy:** STR-096: Token Account Rent Exemption Not Checked on Create

**Hypothesis:** New token accounts created during swaps might not be rent-exempt, causing future failures.

**Target:** Tax Program and AMM Program — token account creation and usage during swap operations.

---

## Determination

**SAFE — No Vulnerability**

The protocol correctly handles token account rent exemption. User-supplied token accounts are pre-created and rent-exempt before being passed to swap instructions. The swap programs themselves do NOT create token accounts; they only perform transfers to pre-existing accounts. Tests verify that all token accounts meet rent-exemption requirements.

---

## Severity

**INFO** (No action required for security)

---

## Confidence

**HIGH (95%)**

Based on:
1. Direct code inspection of account validation constraints in swap handlers
2. Test helper function demonstrating rent-exemption calculation
3. Absence of any `init_if_needed` or manual account creation in critical swap paths
4. Explicit rent calculation in all test fixtures

---

## Summary

The hypothesis proposed checking whether new token accounts created during swaps are rent-exempt. Investigation reveals:

1. **Swap handlers do NOT create token accounts** — The Tax Program's `swap_sol_buy`, `swap_sol_sell`, `swap_profit_buy`, `swap_profit_sell` instructions and the AMM's `swap_sol_pool`, `swap_profit_pool` instructions all accept token accounts as input parameters rather than creating them.

2. **Token accounts are pre-created by callers** — User-facing tests demonstrate that token accounts are created with exact rent-exemption requirements before being passed to swap instructions:
   ```rust
   // From test_swap_sol_pool.rs
   let rent = solana_sdk::rent::Rent::default();
   let space = 165u64;  // TokenAccount size
   let lamports = rent.minimum_balance(space as usize);
   // Then create_account_ix with lamports to SystemProgram
   ```

3. **No `init_if_needed` in critical paths** — Grep search across all swap instruction files found zero instances of `init_if_needed` on token accounts, eliminating the risk of zero-initialized accounts.

4. **Account validation is comprehensive** — Token account constraints use `InterfaceAccount<'info, TokenAccount>`, which Anchor auto-validates:
   - Owner matches token program
   - Discriminator valid
   - No explicit rent check needed (enforced by Solana runtime during transfers)

**Analysis:** Every test creates token accounts with `rent.minimum_balance(space)` where `space = 165` (the standard TokenAccount size). This is the definition of rent-exemption.

### 4. Runtime Validation by Token Program

The Solana Token and Token-2022 programs validate rent-exemption during `transfer_checked`:
- The source account must have balance = data_size + rent_exempt_minimum
- If the account drops below this threshold after transfer, the runtime rejects the transaction
- This is a **runtime check**, not enforced by the on-chain program code

The protocol's swap handlers delegate this validation to the token program via CPI, which is the correct design pattern.

---

## Impact

**None.** The protocol cannot experience a failure due to non-rent-exempt token accounts because:

1. **Token accounts are caller-provided** — The protocol does not create them, eliminating account-creation bugs entirely.

2. **Token program validates at transfer time** — If a caller passes a non-rent-exempt account, the `transfer_checked` CPI will fail at the token program level with a rent-related error.

3. **Tests all pre-fund accounts** — Every integration test creates accounts with exact rent-exemption, demonstrating the expected usage pattern.

---

## Recommendation

**No action required.** The current design is correct:
- Keep token account creation responsibility with callers (clients)
- Callers must use AssociatedTokenProgram or manually create rent-exempt accounts
- Swap handlers correctly accept pre-existing accounts without re-validation

**Optional:** If increased clarity is desired, add a documentation comment to swap instruction account structs:
```rust
/// User's token account for token A.
/// Must be rent-exempt. Created and funded by caller before swap.
/// The token program validates rent-exemption during transfer_checked.
#[account(mut)]
pub user_token_a: InterfaceAccount<'info, TokenAccount>,
```
