# H016: Whitelist Check Bypass via Delegate/Authority Transfer

**Status:** NOT VULNERABLE
**Category:** Token / Transfer Hook
**Severity:** N/A (Hypothesis Disproven)
**Investigated:** 2026-02-22

---

## Hypothesis

The transfer hook checks source and destination token account owners against the whitelist. But Token-2022 allows delegate authorities who can transfer on behalf of the owner. If the hook doesn't check the actual transfer initiator (delegate), a non-whitelisted delegate could transfer tokens.

**Attack Vector:** Whitelisted account A delegates to non-whitelisted account B. B initiates transfer from A to non-whitelisted C. Hook sees source owner=A (whitelisted) → allows.

---

## Investigation

### 1. Understanding Transfer Hook Validation

The transfer hook program validates transfers at the **token account** level, not the wallet/authority level.

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/transfer_hook.rs`

#### What the Hook Checks (lines 96-108):

```rust
// WHTE-06: Transfer allowed if source OR destination is whitelisted
let source_whitelisted = is_whitelisted(
    &ctx.accounts.whitelist_source.to_account_info(),
    &ctx.accounts.source_token.key(),  // ← TOKEN ACCOUNT address, not owner
);

if !source_whitelisted {
    let dest_whitelisted = is_whitelisted(
        &ctx.accounts.whitelist_destination.to_account_info(),
        &ctx.accounts.destination_token.key(),  // ← TOKEN ACCOUNT address, not owner
    );
    require!(dest_whitelisted, TransferHookError::NoWhitelistedParty);
}
```

#### Whitelist PDA Derivation (lines 166-179):

```rust
fn is_whitelisted(whitelist_pda: &AccountInfo, token_account: &Pubkey) -> bool {
    if whitelist_pda.data_is_empty() {
        return false;
    }

    // Verify PDA derivation is correct (SECU-04: prevents spoofed accounts)
    let (expected_pda, _bump) = Pubkey::find_program_address(
        &[WhitelistEntry::SEED_PREFIX, token_account.as_ref()],  // ← seeds = ["whitelist", TOKEN_ACCOUNT_PUBKEY]
        &crate::ID
    );

    whitelist_pda.key() == expected_pda
}
```

**Key Insight:** The whitelist check validates the **TOKEN ACCOUNT ADDRESS** (`source_token.key()` and `destination_token.key()`), not the `owner` field in the token account and not the signer/delegate of the transaction.

### 3. Transfer Hook Account Context

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/transfer-hook/src/instructions/transfer_hook.rs` (lines 21-55)

The hook receives from Token-2022:
- `source_token` (InterfaceAccount<TokenAccount>) — the source token account
- `destination_token` (InterfaceAccount<TokenAccount>) — the destination token account
- `owner` (UncheckedAccount) — the authority/delegate who signed the transfer
- `mint` — the token mint

The hook validates `source_token` and `destination_token` addresses against the whitelist. It does **NOT** validate the `owner` field.

### 4. Why Delegate Transfers Don't Bypass the Whitelist

Token-2022 delegate transfers work as follows:
1. User A owns token account TA
2. User A approves delegate B for X tokens on TA
3. Delegate B calls `transfer_checked` specifying:
   - **Source:** TA (owned by A)
   - **Destination:** TC (owned by C)
   - **Authority:** B (the delegate)

When Token-2022 invokes the transfer hook:
- `source_token` = TA (the actual token account being debited)
- `destination_token` = TC (the actual token account being credited)
- `owner` = B (the delegate, passed but not checked by hook)

The hook checks if **TA** or **TC** is whitelisted. It does NOT check if B is whitelisted.

**Result:** If neither TA nor TC is whitelisted, the transfer is **REJECTED** regardless of who initiated it (owner A, delegate B, or anyone else).

The `approve_delegate` function:

2. The vaults are **owned by `carnage_state` PDA**
3. The delegate is **`carnage_signer` PDA** (a different PDA)

When the subsequent transfer occurs via `swap_exempt`:

- **Authority:** `carnage_signer` (the delegate, NOT checked by hook)

The transfer succeeds because **both the source vault and destination vault token accounts are whitelisted**, not because the delegate is whitelisted.

### 6. Whitelist Design Rationale

From the architectural context (`.audit/context/05-token-economic.md`, lines 109-126):

> **Security assessment:** Sound design. The OR-based whitelist means protocol vaults are whitelisted, and any user can transfer to/from them. Direct user-to-user transfers of CRIME/FRAUD/PROFIT fail unless one side is whitelisted. The `is_transferring` check prevents attackers from calling the hook directly.

The whitelist is designed to:
- Allow any user to transfer **TO** protocol vaults (buy, stake)
- Allow any user to transfer **FROM** protocol vaults (sell, unstake, claim)
- Block user-to-user transfers (prevents secondary markets)

This works because:
- Protocol vault **token accounts** are whitelisted (not their owners/PDAs)
- User token accounts are NOT whitelisted
- User → Protocol vault: destination is whitelisted ✓
- Protocol vault → User: source is whitelisted ✓
- User A → User B: neither is whitelisted ✗

The whitelist operates at the **token account level** (the actual SPL Token account address), not the wallet/authority/delegate level. This is the correct security boundary for Token-2022 transfer hooks.

## Conclusion

**NOT VULNERABLE**

The transfer hook correctly validates token account addresses, not authorities/delegates. This is by design:

1. **Token Account vs Authority:** The hook checks the **source and destination token account addresses**, not the `owner` field or transaction signer.

2. **Delegate Transfers Don't Bypass:** When a delegate initiates a transfer, the source and destination token accounts remain the same. The hook validates these accounts, not the delegate.

3. **Real-World Usage:** The protocol itself uses delegate transfers (Carnage execution), demonstrating that the whitelist works correctly with delegated authority.

4. **Correct Security Boundary:** Token-2022 transfer hooks receive the token accounts being debited/credited. Checking these addresses (not the transaction signer) is the correct approach because:
   - Token accounts are the actual custody locations
   - Delegates can change, but the token account address is immutable
   - Whitelisting at the token account level prevents secondary markets regardless of who initiates the transfer

### Architectural Correctness

This is actually a **strength** of the design, not a weakness. By whitelisting token accounts (not wallets), the protocol ensures:
- Protocol vaults can be operated by multiple authorities (swap_authority, carnage_signer, tax_authority) without requiring all PDAs to be whitelisted
- Users cannot circumvent the whitelist by delegating to a third party
- The whitelist remains simple and focused on the actual custody boundary

**Investigator Notes:**

The hypothesis stemmed from confusion between "what initiates the transfer" (authority/delegate) and "what is transferred" (token accounts). Token-2022 transfer hooks execute in the context of the **token accounts**, not the transaction signer. The `owner` parameter passed to the hook (account index 3) is informational and not used for whitelist validation.

This is architecturally sound. If the hook checked authorities instead of accounts, the whitelist would be ineffective — users could create new wallets, delegate, and bypass restrictions. By checking the token account addresses, the protocol ensures that only protocol-controlled vaults can participate in transfers, regardless of who signs the transaction.
