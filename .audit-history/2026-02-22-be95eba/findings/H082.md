# H082: Epoch Transition Bounty Griefing via Rent-Exempt Vault Draining

**Strategy:** STR-082

**Determination:** MITIGATED (ON-CHAIN BUG CONFIRMED, OFF-CHAIN WORKAROUND DEPLOYED)

**Severity:** CRITICAL (on-chain bug) / MEDIUM (post-mitigation)

**Confidence:** VERY HIGH

---

## Summary

The epoch transition mechanism incentivizes validators/crank runners to advance epochs by paying a 1 SOL bounty from the Carnage SOL vault. However, the on-chain bounty payment logic contains a critical bug: it checks `vault_balance >= TRIGGER_BOUNTY_LAMPORTS (1M lamports)` but fails to account for the rent-exempt minimum (~890K lamports) required to keep the SystemAccount alive.

**Attack Vector:** If the vault balance falls into the "danger zone" (1M to ~1.89M lamports), the bounty check passes but the actual `system_program::transfer` fails at runtime because the remaining balance would drop below rent-exempt. This breaks epoch advancement entirely—a MEV griefing attack.

**Current Status:** The on-chain bug remains unfixed (requires program redeploy). However, the off-chain crank runner has been hardened with a preventive top-up mechanism that keeps the vault above the danger zone, eliminating the attack surface in practice.

### On-Chain Vulnerability (trigger_epoch_transition.rs:194-227)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/epoch-program/src/instructions/trigger_epoch_transition.rs`

```rust
// Line 194-195: Bounty check (VULNERABLE)
let vault_balance = ctx.accounts.carnage_sol_vault.lamports();
let bounty_paid = if vault_balance >= TRIGGER_BOUNTY_LAMPORTS {

    invoke_signed(
        &system_instruction::transfer(
            ctx.accounts.carnage_sol_vault.to_account_info().key,
            ctx.accounts.payer.to_account_info().key,
            TRIGGER_BOUNTY_LAMPORTS,
        ),
        &[ ... ],
        &[signer_seeds],
    )?;
    // ...
} else {
    msg!("Carnage vault balance insufficient for bounty: {} < {} (skipped)", vault_balance, TRIGGER_BOUNTY_LAMPORTS);
    0
}
```

**The Bug:**
1. Line 195 checks: `vault_balance >= 1_000_000 lamports`
2. If true, transfers 1M lamports via `system_instruction::transfer`
3. The remaining balance becomes: `vault_balance - 1_000_000`
4. **Missing check:** If remaining < 890,880 lamports (rent-exempt minimum), Solana runtime rejects the transfer with `InsufficientFundsForRent`

**Danger Zone:** If `vault_balance ∈ [1_000_000, 1_890_880)`, the check passes but the transfer fails. Any bounty-triggerer calling this in the danger zone experiences a failed transaction, preventing epoch advancement.

**Constants Verified:**
- `TRIGGER_BOUNTY_LAMPORTS = 1_000_000` (line 16 / constants.rs:81)
- Rent-exempt for 0-data SystemAccount: ~890,880 lamports (Solana standard)

### Off-Chain Mitigation (crank-runner.ts:63-75, deployed 2026-02-22)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/scripts/crank/crank-runner.ts`

Commit `6214fb3` ("fix: crank auto-tops-up carnage vault to prevent rent-exempt bug") deployed the following hardening:

```typescript
// Lines 63-75: Mitigation constants and logic
const TRIGGER_BOUNTY_LAMPORTS = 1_000_000;           // Must match on-chain
const RENT_EXEMPT_MINIMUM = 890_880;                 // 0-data SystemAccount
const MIN_VAULT_BALANCE = TRIGGER_BOUNTY_LAMPORTS + RENT_EXEMPT_MINIMUM + 100_000; // ~2M
const VAULT_TOP_UP_LAMPORTS = 5_000_000;            // 0.005 SOL — covers ~5 bounties
```

**Mitigation Flow (each epoch cycle):**
1. Read current vault balance from chain
2. **If balance < MIN_VAULT_BALANCE (2M lamports):**
   - Send 5M lamports (0.005 SOL) from crank operator wallet to vault
   - Wait for confirmation
   - Log the top-up with TX signature
3. Proceed with epoch advancement

**Safety Margin:** The crank maintains a 2M lamport buffer = `1M (bounty) + 890K (rent-exempt) + 100K (cushion)`, ensuring the vault never enters the danger zone during normal operation.

**Deployed State:**
- Railway crank runner (railway-crank.toml) runs 24/7 with auto-topup enabled
- Mainnet checklist (Docs/mainnet-checklist.md) documents the known on-chain bug
- Crank operator account must have sufficient SOL balance (~0.01 SOL buffer for multiple top-ups)

## Impact

### Without Mitigation (Pre-Fix Scenario)

An attacker who can observe the Carnage SOL vault balance and time transactions could:
1. **Griefing Attack:** Monitor the vault; when it enters the danger zone, call `trigger_epoch_transition` repeatedly
2. **Epoch Lock:** Each failed transaction wastes the attacker's fees but more importantly, prevents the crank from advancing the epoch
3. **DoS Duration:** The vault stays in the danger zone until:
   - New trade tax revenue accumulates (unpredictable, volume-dependent), OR
   - Admin manually sends SOL (requires action, not automated)
4. **Staking Halt:** While epochs can't advance, staking rewards can't be distributed, breaking yield for users

### With Current Mitigation

The auto-top-up crank eliminates the attack surface entirely:
- Vault is kept safely above 2M lamports at all times
- Even if an attacker drains the vault down to 1.5M in a single epoch (impossible; max tax revenue ~0.2M/epoch), the next crank cycle tops it back up
- The attacker would need to repeatedly drain 0.005 SOL per crank cycle (~1 minute), consuming more in fees than damage caused

## Root Cause Analysis

The vulnerability stems from conflating two concepts:
1. **Balance sufficiency for a transfer:** Account has enough lamports
2. **Post-transfer account liveness:** Account remains above rent-exempt

The on-chain logic only checks (1) but Solana runtime enforces (2). This is a common pattern in account-based blockchain systems where system accounts have minimum balance requirements.

The fix is straightforward: `require!(vault_balance - bounty >= rent_exempt_minimum)` before paying out.
