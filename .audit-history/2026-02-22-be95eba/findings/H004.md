# H004: `force_carnage` Backdoor Survives to Mainnet if Built with Devnet Features

**Strategy:** STR-004
**Determination:** POTENTIAL
**Severity:** CRITICAL
**Confidence:** HIGH

## Summary
The `force_carnage` instruction allows an admin to trigger Carnage at will, bypassing the VRF randomness mechanism. While correctly gated by `#[cfg(feature = "devnet")]`, the instruction **is present in the current IDL** (`target/idl/epoch_program.json`), proving it was compiled in the most recent build. If this binary were deployed to mainnet, a key compromise would enable arbitrary Carnage execution for market manipulation and pool draining.

## Root Cause

The vulnerability exists due to a **process gap**, not a code flaw:

1. The code correctly uses `#[cfg(feature = "devnet")]` to exclude the backdoor from mainnet builds
2. The build script correctly conditionalizes the devnet feature rebuild
3. **However**, there is no **enforcement mechanism** to prevent accidental mainnet deployment of a devnet-featured binary

The risk scenarios:
- **Human error**: Developer runs `./build.sh --devnet` then deploys to mainnet
- **Copy-paste error**: Devnet `.so` file copied to mainnet deployment directory
- **CI/CD misconfiguration**: Mainnet deployment pipeline uses `--devnet` flag
- **Key reuse**: If mainnet uses the same keypair as devnet for initial deployment

## Attack Scenario

### Preconditions
1. Epoch program deployed to mainnet **with** `--features devnet` (either accidentally or maliciously)
2. Attacker compromises the devnet admin private key (`8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4`)
   - Key is currently in `keypairs/devnet-wallet.json` (committed to git per mainnet checklist)
   - If mainnet reuses this key or its seed phrase, compromise is highly likely

## Remediation

### Primary Fix: Remove Module Entirely (RECOMMENDED)
Delete the entire `force_carnage.rs` module before mainnet build:

```bash
# In build.sh, add verification step:
if [[ " $* " != *" --devnet "* ]]; then
  echo "Verifying force_carnage is absent..."
  if grep -q "force_carnage" target/idl/epoch_program.json; then
    echo "ERROR: force_carnage found in mainnet build!"
    echo "Run: git rm programs/epoch-program/src/instructions/force_carnage.rs"
    exit 1
  fi
fi
```

Alternatively, move test logic to integration test harness:
```bash
mv programs/epoch-program/src/instructions/force_carnage.rs \
   tests/helpers/force_carnage_test_helper.rs
```

### Secondary Fix: Build Pipeline Enforcement
Add compile-time assertion in `lib.rs`:

```rust
// At top of epoch_program/src/lib.rs
#[cfg(all(not(feature = "devnet"), feature = "force-carnage-check"))]
compile_error!("force_carnage must not be present in mainnet builds");

// In Cargo.toml [features]:
force-carnage-check = []  // Enabled by CI for mainnet builds
```

### Tertiary Fix: Mainnet Checklist Automation
Update `Docs/mainnet-checklist.md` Item 10 with verification script:

```bash
# scripts/verify-mainnet-build.sh
#!/bin/bash
set -e

echo "Verifying mainnet-safe build..."

# Check IDL has no force_carnage
if grep -q "force_carnage" target/idl/epoch_program.json; then
  echo "FAIL: force_carnage present in IDL"

# Check binary size (devnet binary is larger due to extra instruction)

if [ $ACTUAL_SIZE -gt $EXPECTED_SIZE ]; then
  echo "WARN: Binary larger than expected (possible devnet build)"
fi

echo "PASS: Build is mainnet-safe"
```

## Notes for Auditor

**Why POTENTIAL instead of CONFIRMED:**
The vulnerability only exists **if** the mainnet deployment uses the devnet-featured binary. The code itself is correctly gated. The risk is in the **deployment process**, which is harder to verify via static analysis.

**Why CRITICAL severity despite POTENTIAL:**
1. **Key is in git**: `keypairs/devnet-wallet.json` is committed, making compromise trivial if reused
2. **100% of Carnage Fund at risk**: Attacker can drain protocol's most valuable accumulated balance
3. **Perpetual drain**: Attack repeats every epoch indefinitely
4. **No mitigation post-deployment**: Programs are immutable — if deployed with backdoor, only option is full redeployment
