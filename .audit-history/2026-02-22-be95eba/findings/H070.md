# H070: Staking Reward Token Account Mint Validation

**Strategy:** STR-070
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

Investigation of hypothesis: "Staking reward distribution transfers PROFIT tokens but the reward vault might hold a different token."

**Finding:** The staking reward system does NOT transfer PROFIT tokens. Staking rewards are distributed exclusively as native SOL (lamports), not as PROFIT token transfers. The hypothesis is based on a misunderstanding of the reward mechanism architecture.

- **PROFIT tokens:** Held in `stake_vault` PDA with explicit mint constraint
- **SOL rewards:** Held in `escrow_vault` PDA (native account, not a token account)
- **No reward vault token mismatch possible:** Because no token account exists for reward distribution

### 5. No Token Transfer in Reward Distribution

The `deposit_rewards` instruction (called by Tax Program on every swap) only updates a counter, no token transfers:

Per `/Users/mlbob/Projects/Dr Fraudsworth/programs/staking/src/instructions/deposit_rewards.rs` (lines 83-119):

```rust
pub fn handler(ctx: Context<DepositRewards>, amount: u64) -> Result<()> {
    require!(amount > 0, StakingError::ZeroAmount);

    let pool = &mut ctx.accounts.stake_pool;
    let clock = Clock::get()?;

    // Add to pending (SOL already transferred by Tax Program)
    pool.pending_rewards = pool
        .pending_rewards
        .checked_add(amount)
        .ok_or(StakingError::Overflow)?;

    // Reconcile: verify escrow vault actually holds enough SOL.
    require!(
        ctx.accounts.escrow_vault.lamports() >= pool.pending_rewards,
        StakingError::InsufficientEscrowBalance
    );

    // ... emit event ...
    Ok(())
}
```

**Analysis:** No token account is passed to this instruction. It only updates the `pending_rewards` counter. The SOL is already in `escrow_vault` (a system account) from the Tax Program's prior `system_instruction::transfer`.

## Impact

**No impact.** The hypothesis was based on a misconception about the reward distribution mechanism. The staking system architecture is sound:

1. **PROFIT tokens** (staked by users) are held in `stake_vault` with explicit mint constraint
2. **SOL rewards** (distributed to users) are held in `escrow_vault` (native system account)
3. **No token mint mismatch possible** because no token account is used for reward distribution

The separation of concerns is intentional and correct:
- Staking: PROFIT tokens (Token-2022) → `stake_vault`
- Rewards: Native SOL (lamports) → `escrow_vault`

## Recommendation

**No action required.** The staking reward architecture correctly separates PROFIT token custody (for staking) from SOL reward custody (for distribution). All token account constraints are properly enforced.

**Documentation note:** The term "reward token account" in architectural diagrams or specifications should be clarified as "reward SOL escrow" (a system account, not a token account) to prevent future confusion about mint validation.
