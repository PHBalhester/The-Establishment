# H104: Whitelist Covers Token Account Addresses, Not Wallet Addresses

## Strategy
**STR-104**: Whitelist Covers Account Addresses Not Owner Addresses

## Determination
**HYPOTHESIS CONFIRMED** — The whitelist covers **token account addresses** (specific), not **wallet/owner addresses** (broad). This is by design and correctly implemented.

## Severity
**LOW** (Design pattern, not a vulnerability)

## Confidence
**HIGH** — Code review confirms consistent implementation across whitelist creation, transfer hook enforcement, and documentation.

---

## Summary

The Dr Fraudsworth Transfer Hook uses an **address-specific whitelist** that checks token account addresses rather than wallet owner addresses. When a user creates a new token account (new ATA) for CRIME, FRAUD, or PROFIT tokens, it is NOT automatically whitelisted based on the owner's identity.

**Design intent:** Protocol vaults are explicitly whitelisted by their token account addresses. User-to-user transfers of these tokens fail unless one party's token account is whitelisted. This creates a gated transfer model where only whitelisted parties can move these tokens.

**Implication:** If a user wants to send CRIME/FRAUD to another user, either:
1. The sender's token account must be whitelisted (protocol vault scenario), OR
2. The recipient's token account must be whitelisted (receiving from a protocol operation)

A regular user cannot send faction tokens to another regular user unless one of them already has a whitelisted token account.

fn is_whitelisted(whitelist_pda: &AccountInfo, token_account: &Pubkey) -> bool {
    // Account must have data (exists)
    if whitelist_pda.data_is_empty() {

## Impact

### Normal Scenario (Expected Behavior)
1. **Protocol vaults (AMM, Staking, Epoch)** have their token account addresses explicitly whitelisted at initialization
2. **Users can send tokens to protocol vaults** — destination vault is whitelisted
3. **Users can receive tokens from protocol vaults** — source vault is whitelisted
4. **User-to-user transfers fail** — neither party's token account is whitelisted, and ownership identity doesn't matter

### Edge Case: User Creates New Token Account
If a user creates a new Associated Token Account (ATA) for CRIME/FRAUD/PROFIT:
- The new ATA is a **different token account address** than any previously whitelisted ATA they may own
- The new ATA is **not automatically whitelisted** based on the owner's identity
- Any transfer involving the new ATA (as sender or receiver) will fail unless the new ATA is explicitly whitelisted
- The owner of the account is **irrelevant** to the whitelist check

**This is not a vulnerability** — it's consistent with the protocol's design goal of gated token movement. However, it does create UX friction if users try to transfer between their own token accounts (e.g., if they create a second ATA for the same token).

---

## Recommendation

**No fix required** — this is correct-by-design. The address-specific whitelist is the intended security model.

**However, for user education:**

1. **Documentation for integrators:** Clearly document that whitelist entries are **token account addresses**, not wallet addresses. Users creating new ATAs must request whitelisting or use existing whitelisted ATAs.

2. **Optional enhancement (post-mainnet):** Consider a convenience instruction that allows whitelisting all ATAs owned by a wallet address, derived via `find_associated_token_address(wallet, mint)`. This would improve UX without compromising security (whitelisting is permissioned to admin anyway).

3. **Test coverage:** Ensure test suite covers the scenario where a user creates a second ATA and verifies it is not automatically whitelisted based on ownership.

## Architecture Context

This design aligns with Dr Fraudsworth's broader architecture:

- **Protocol-owned liquidity:** AMM pools are protocol-owned PDAs, with explicit vault addresses whitelisted
- **Permissioned token movement:** Only explicitly whitelisted parties (protocol vaults + authorized users) can transfer faction tokens
- **Single whitelist authority:** One admin controls all whitelist entries (per design, with ability to burn authority irreversibly)

The address-specific approach is safer than owner-based whitelisting because:
1. **No wallet substitution:** An attacker cannot create a new wallet and inherit whitelisting
2. **Explicit control:** Each token account must be individually whitelisted, preventing accidental scope creep
3. **Audit clarity:** Protocol can enumerate all whitelisted token accounts deterministically
