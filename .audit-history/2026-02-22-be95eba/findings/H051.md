# H051: Carnage Burn Authority Correctly Configured

**Strategy:** STR-051
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The Carnage burn authority is correctly configured. The token vaults that receive bought tokens during Carnage execution are owned by the `carnage_state` PDA (seeds: `["carnage_fund"]`), and the burn instruction properly uses this same PDA as the signing authority. The burn operation will succeed because the authority matches the token account owner.

The hypothesis that Carnage might lack burn authority over purchased tokens is **not confirmed**. The implementation correctly establishes authority relationships during initialization and maintains them throughout the burn execution flow.

    carnage_state: &mut Account<'info, CarnageFundState>,

    token_program: &Interface<'info, TokenInterface>,

    burn_mint: AccountInfo<'info>,

    // Build burn instruction data manually for Token-2022
    let mut burn_data = vec![8u8]; // Burn instruction discriminator

            burn_mint,

## Architecture Analysis

The Carnage burn flow follows this authority chain:

1. **Initialization:** `initialize_carnage_fund` creates token vaults with `authority = carnage_state` PDA
2. **Buy Operation:** Tokens purchased via `execute_swap_exempt_cpi` land in these vaults (owned by `carnage_state`)
3. **Burn Operation:** `burn_held_tokens` uses `carnage_state` PDA as the signing authority via `invoke_signed`
4. **Token-2022 Validation:** The Token-2022 program verifies the burn instruction's authority matches the vault's owner

**Result:** The authority chain is complete and correct. The PDA that owns the vaults is the same PDA that signs the burn instruction.

## Comparison to Sell Path

For comparison, the Sell path (lines 891-943) uses a **delegate approval** mechanism:
- `carnage_state` PDA owns the vaults
- `carnage_signer` PDA is approved as a delegate
- `carnage_signer` signs the swap CPI (which internally calls `transfer_checked`)

This delegation is necessary because the swap CPI requires a user authority signer. However, for burn operations, Token-2022 accepts the direct owner signature without requiring delegation, which is why the burn path is simpler and more direct.

## Impact Assessment

**No vulnerability exists.** The Carnage burn authority is correctly configured and will execute successfully. The implementation demonstrates:

1. ✅ Correct PDA derivation and ownership
2. ✅ Proper use of `invoke_signed` with matching seeds
3. ✅ Runtime validation to prevent vault substitution
4. ✅ Clear separation between `carnage_state` (vault owner) and `carnage_signer` (swap authority)

The burn instruction will not fail due to authority issues. Tokens bought during Carnage execution will be successfully burned when the action is `CarnageAction::Burn`.
