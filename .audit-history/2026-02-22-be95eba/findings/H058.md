# H058 — Transfer Hook Whitelist PDA Derivation Mismatch

**Status:** SAFE ✅
**Severity:** N/A
**Strategy:** STR-058
**Date:** 2026-02-22

## Summary
Investigation into potential PDA seed mismatch between whitelist creation and verification. **No vulnerability found** — seeds are consistent across all codepaths.

## Hypothesis
The whitelist entry PDA might be derived with different seeds during `add_to_whitelist` vs during transfer hook execution, causing whitelist checks to always fail or always pass incorrectly.

## Investigation

### 1. Whitelist Entry Creation (add_whitelist_entry.rs)

**PDA derivation at line 60:**
```rust
seeds = [WhitelistEntry::SEED_PREFIX, address_to_whitelist.key().as_ref()],
```

**Constant definition (state/whitelist_entry.rs:24):**
```rust
pub const SEED_PREFIX: &'static [u8] = b"whitelist";
```

**Expanded seeds:**
- `b"whitelist"` (literal prefix)
- `address_to_whitelist.key()` (the token account being whitelisted)

### 2. Whitelist Check During Transfer (transfer_hook.rs)

**PDA verification at line 173-176:**
```rust
let (expected_pda, _bump) = Pubkey::find_program_address(
    &[WhitelistEntry::SEED_PREFIX, token_account.as_ref()],
    &crate::ID
);
```

**Expanded seeds:**
- `WhitelistEntry::SEED_PREFIX` = `b"whitelist"` (same constant)
- `token_account` = the source or destination token account being checked

### 3. ExtraAccountMetaList Configuration

**Source whitelist (initialize_extra_account_meta_list.rs:38-43):**

## Analysis

### Seed Consistency ✅

All three locations use **identical seed derivation**:

1. **Creation**: `[b"whitelist", address_to_whitelist.key()]`
2. **Verification**: `[b"whitelist", token_account]`
3. **Resolution**: `[b"whitelist", AccountKey{index: 0 or 2}]`

Where:
- `address_to_whitelist` is the token account passed during whitelist creation
- `token_account` is the source/destination account during transfer
- `AccountKey{index: 0}` resolves to source_token_account at runtime
- `AccountKey{index: 2}` resolves to destination_token_account at runtime

### Call Flow Verification

**When adding source account to whitelist:**
```
add_to_whitelist(address = TokenAccount_A)
  → Creates PDA with seeds [b"whitelist", TokenAccount_A]
```

**When transfer occurs with that source:**
```
transfer_hook(source = TokenAccount_A, dest = TokenAccount_B)
  → Token-2022 resolves extra account using seeds [b"whitelist", source]
  → Passes resolved PDA as whitelist_source
  → is_whitelisted() verifies PDA matches [b"whitelist", source]
  → ✅ Matches the PDA created during add_to_whitelist
```

### Additional Safety Features

1. **PDA Ownership**: All PDAs derived with `&crate::ID` (transfer hook program)
2. **Existence Check**: `is_whitelisted()` checks `data_is_empty()` first (line 168)
3. **Derivation Verification**: Explicit `find_program_address()` check prevents spoofed accounts (SECU-04)
4. **Short-circuit Logic**: Only checks destination if source fails (optimization at line 101)

## Conclusion

**SAFE** — No PDA derivation mismatch exists. The whitelist system uses consistent seeds across:
- PDA creation during `add_to_whitelist`
- PDA resolution by Token-2022 via ExtraAccountMetaList
- PDA verification during `transfer_hook` execution

The seeds `[b"whitelist", token_account_pubkey]` are identical in all contexts, ensuring whitelist entries are correctly matched during transfer validation.
