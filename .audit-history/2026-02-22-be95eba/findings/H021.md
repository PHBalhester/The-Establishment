# H021: Staking First-Depositor Attack
**Strategy:** STR-021
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis that the first staker could deposit a minimal amount (1 token) and then donate large rewards to capture disproportionate rewards from subsequent stakers was investigated. The attack vector suggested that due to precision loss in the `reward_per_token` calculation, the first depositor could inflate the reward rate to such a degree that subsequent depositors would receive zero or near-zero rewards despite their deposits.

After comprehensive code review and mathematical analysis, **this vulnerability is NOT present**. The protocol implements a robust dead stake protection mechanism that makes this attack economically infeasible.

### Key Protections

1. **MINIMUM_STAKE dead stake** — The protocol locks 1 PROFIT (1,000,000 base units) during `initialize_stake_pool`, ensuring the attacker cannot be the first depositor
2. **Checked arithmetic throughout** — All multiplication and addition operations use `checked_mul`, `checked_add`, returning `Err(StakingError::Overflow)` rather than wrapping
3. **Economic infeasibility** — Even with MINIMUM_STAKE as the only stake, manipulating reward distribution would require donating >1,000,000 SOL to the escrow vault

pub fn update_rewards(pool: &StakePool, user: &mut UserStake) -> Result<()> {
    // Step 1: Calculate delta between global cumulative and user's checkpoint
    // This represents rewards-per-token since user last interacted
    let reward_delta = pool.rewards_per_token_stored
        .checked_sub(user.rewards_per_token_paid)
        .ok_or(StakingError::Underflow)?;

    // Step 2: Calculate user's pending rewards
    // pending = balance * reward_delta / PRECISION
    // Note: multiply before divide to preserve precision
    let pending = (user.staked_balance as u128)
        .checked_mul(reward_delta)
        .ok_or(StakingError::Overflow)?
        .checked_div(PRECISION)
        .ok_or(StakingError::DivisionByZero)? as u64;

    // Step 3: Add to user's accumulated rewards
    user.rewards_earned = user.rewards_earned
        .checked_add(pending)
        .ok_or(StakingError::Overflow)?;

    // Step 4: Update user's checkpoint to current global
    // This prevents claiming the same rewards twice
    user.rewards_per_token_paid = pool.rewards_per_token_stored;

}

### Mathematical Analysis

**Attack scenario (prevented by dead stake):**

```
Hypothetical (if MINIMUM_STAKE didn't exist):
1. Attacker deposits 1 lamport of PROFIT as the first depositor
2. Attacker donates 1000 SOL (1e12 lamports) directly to escrow_vault
3. Attacker calls deposit_rewards with amount = 1e12
4. reward_per_token = (1e12 * 1e18) / 1 = 1e30
5. Second depositor stakes 1M PROFIT (1e6 units)
6. Second depositor's reward = (1e6 * 1e30) / 1e18 = 1e18 lamports
7. Second depositor's reward exceeds total deposited → accounting corruption
```

**Actual scenario (with MINIMUM_STAKE = 1,000,000):**

```
Protocol initialization:
1. initialize_stake_pool transfers MINIMUM_STAKE (1,000,000) to stake vault
2. pool.total_staked = 1,000,000 (permanently, cannot be removed)
3. Attacker stakes additional 1 unit (now total = 1,000,001)
4. Attacker donates 1000 SOL to escrow_vault and calls deposit_rewards
5. reward_per_token = (1e12 * 1e18) / 1,000,000 = 1e24
6. Second depositor stakes 1M PROFIT
7. Second depositor's share = 1,000,000 / 2,000,001 = ~50%
8. Attacker's share = 1 / 2,000,001 = ~0.00005%
9. Attacker loses nearly all the donated SOL to second depositor
```

**To meaningfully manipulate the distribution, the attacker would need to donate:**

```
Attack cost analysis:
- To capture 90% of rewards with 1 unit staked against MINIMUM_STAKE dead stake:
  - Attacker stake: 1 unit
  - Dead stake: 1,000,000 units
  - Required donation to shift distribution: ~10,000,000 SOL (absurd)

- Even if attacker stakes 1M PROFIT (equal to dead stake):
  - Attacker share before donation: 50%
  - Donation required to capture 90%: Still millions of SOL
  - Any second depositor immediately claims ~50% of that donation

Conclusion: Attack is economically impossible. The attacker loses the donation.
```

## Root Cause Analysis

**This is a false positive.** The classic first-depositor attack pattern from EVM DeFi protocols assumes:

1. Attacker can be the FIRST depositor with a tiny amount (1 wei)
2. No minimum deposit/liquidity on first stake
3. Exchange rate or reward rate calculated from `total_assets / total_shares`
4. Direct donation to vault inflates the exchange rate
5. Second depositor receives zero shares due to rounding

**Why this fails in Dr Fraudsworth's staking protocol:**

1. **Attacker cannot be first depositor** — The protocol itself is the first depositor during `initialize_stake_pool`, staking 1,000,000 units as dead stake
2. **Dead stake is permanent** — It is never withdrawable. `pool.total_staked` can never drop below MINIMUM_STAKE
3. **Economic infeasibility** — To shift reward distribution meaningfully against the 1M dead stake requires donating millions of SOL, which would immediately be captured by other stakers
4. **Checked arithmetic prevents overflow** — Even if an attacker tried to overflow `reward_per_token_stored`, the `checked_mul` would return `Err(Overflow)`, halting the transaction

## Attack Scenario

**Hypothetical attack flow (all steps fail):**

```
Step 1: Attacker tries to initialize the pool without dead stake
  ❌ FAILS: initialize_stake_pool.rs lines 125-138 enforce MINIMUM_STAKE transfer
  ❌ FAILS: Anchor's `init` constraint means pool can only be created once

Step 2: Attacker stakes 1 lamport to become the smallest staker
  ❌ FAILS: Pool already has 1,000,000 dead stake from initialization
  ❌ FAILS: Attacker's 1 lamport is 0.0001% of total, not 100%

Step 3: Attacker donates 1000 SOL to escrow_vault
  ✅ SUCCESS: Anyone can transfer SOL to the vault PDA

Step 4: Attacker calls deposit_rewards(amount = 1e12)
  ❌ FAILS: deposit_rewards is CPI-gated (seeds::program = tax_program_id())
  ❌ FAILS: Only Tax Program's tax_authority PDA can call it
  ❌ Alternative: Attacker waits for legitimate deposit_rewards call

Step 5: Attacker's rewards calculated as 100% due to donation inflation
  ❌ FAILS: Attacker's share = attacker_stake / (MINIMUM_STAKE + attacker_stake)
  ❌ FAILS: Even with 1M staked, attacker gets 50%, not 100%
  ❌ FAILS: Second depositor immediately claims ~50% of the donation

Conclusion: Attack fails at multiple steps. No exploitation path exists.
```

## Remediation

**No remediation required.** The protocol's existing defenses are comprehensive:

1. ✅ **Dead stake protection** — 1 PROFIT (1,000,000 units) locked permanently at initialization
2. ✅ **Checked arithmetic** — All financial operations use `checked_*` methods, overflow returns `Err`
3. ✅ **CPI-gated reward deposits** — Only Tax Program can call `deposit_rewards`, preventing direct manipulation
4. ✅ **Escrow reconciliation** — `deposit_rewards` verifies `escrow_vault.lamports() >= pending_rewards` (line 100 of deposit_rewards.rs)
5. ✅ **Comprehensive testing** — Unit tests confirm reward distribution correctness (H012 investigation)

**Design Pattern Analysis:**

This implementation follows **best practices from battle-tested DeFi protocols**:

- **Uniswap V2**: Locks 1000 wei of LP tokens on first mint (MINIMUM_LIQUIDITY)
- **Synthetix StakingRewards**: Uses time-weighted cumulative pattern (prevents snapshot gaming)
- **Quarry (Solana)**: Uses similar cumulative reward-per-token with dead stake

**Optional hardening (defense-in-depth, not necessary):**

If the protocol team wants additional assurance, they could add:

```rust
// In add_to_cumulative, add explicit bounds check on reward_per_token
const MAX_REWARD_PER_TOKEN_INCREMENT: u128 = 1e30; // Generous upper bound

let reward_per_token = (pool.pending_rewards as u128)
    .checked_mul(PRECISION)
    .ok_or(StakingError::Overflow)?
    .checked_div(pool.total_staked as u128)
    .ok_or(StakingError::DivisionByZero)?;

// Sanity check: reward increment per epoch should never be astronomical
require!(
    reward_per_token < MAX_REWARD_PER_TOKEN_INCREMENT,
    StakingError::SuspiciousRewardRate
);
```

**This is NOT recommended** because:
- It adds unnecessary complexity
- The existing checked arithmetic already provides complete protection
- The economic infeasibility makes the attack impossible even without overflow
- Adding magic number bounds creates maintenance burden and potential false positives
