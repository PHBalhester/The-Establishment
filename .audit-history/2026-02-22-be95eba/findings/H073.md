# H073: Switchboard SDK Version Stability & Dependency Management

**Strategy:** STR-073
**Determination:** NOT VULNERABLE
**Severity:** MEDIUM (Operational Risk)
**Confidence:** HIGH

## Summary

The protocol uses `switchboard-on-demand = "=0.11.3"` (pinned version) for VRF-driven epoch randomness. Investigation confirms that:

1. **Version choice is well-justified:** v0.11.3 is the working version validated in January 2026 v3 implementation
2. **No known security vulnerabilities** identified in v0.11.3 itself or its dependencies
3. **SDK API is stable** for the specific usage patterns employed (RandomnessAccountData parsing, get_value() reading)
4. **Dependency chain is minimal and healthy:** Dependencies (anyhow, bincode, borsh, solana-program, etc.) are standard, widely-used crates with no flagged vulnerabilities
5. **Feature flag pinning is correct:** The code uses feature-flagged Switchboard PIDs (devnet vs mainnet) with proper compilation guards

**However, two operational risks exist:**

- **Mainnet transition requirement:** When upgrading to mainnet, the build process MUST use the non-devnet feature flag to get the correct Switchboard PID, or ConstraintOwner errors will occur
- **Future upgrade path:** The Switchboard SDK is actively developed with new feature flags (solana-v2, solana-v3) suggesting SDK version transitions may be needed as Solana SDK versions advance

## Evidence

### Version Selection Rationale (Per VRF_Migration_Lessons.md)

The project evaluated three Switchboard approaches before settling on v0.11.3:

1. **`solana-randomness-service-lite`** — ABANDONED
   - Referenced non-existent account: `DCe143sY8nC6SNwZWwi7qeFco7FxSvKrGrZ62vsufTJa`
   - Service infrastructure shut down by Switchboard
   - No deprecation warning on crates.io

2. **`solana-randomness-service` + `switchboard-v2`** — OUTDATED
   - Built with Anchor 0.29; incompatible with project's Anchor 0.32
   - Required 276 instructions across ~48 transactions for single proof
   - Version mismatch caused build failures

3. **`switchboard-on-demand` v0.11.3** — ADOPTED
   - Proven working on devnet (January 2026)
   - Client-side commit-reveal pattern using SGX TEE
   - Only requires 3 transactions for complete VRF flow
   - SDK provides `RandomnessAccountData` for safe parsing

### SDK Usage Pattern (Safe)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/epoch-program/src/instructions/consume_randomness.rs:12,164-171`

```rust
use switchboard_on_demand::RandomnessAccountData;

// Usage is restricted to:
let vrf_result: [u8; 32] = {
    let data = ctx.accounts.randomness_account.try_borrow_data()?;
    let randomness_data =
        RandomnessAccountData::parse(data).map_err(|_| EpochError::RandomnessParseError)?;
    randomness_data
        .get_value(clock.slot)
        .map_err(|_| EpochError::RandomnessNotRevealed)?
};
```

This usage pattern:
- Borrows randomness account data
- Parses it via SDK's `RandomnessAccountData::parse()`
- Reads revealed value via `get_value(current_slot)` — only succeeds if oracle has revealed
- Returns error if parse fails or reveal not ready

All are stable SDK methods with no known breaking changes in v0.11.3 or reported vulnerabilities.

### Feature Flag Correctness

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/epoch-program/src/constants.rs:45-49`

```rust
#[cfg(feature = "devnet")]
pub const SWITCHBOARD_PROGRAM_ID: Pubkey = switchboard_on_demand::ON_DEMAND_DEVNET_PID;

#[cfg(not(feature = "devnet"))]
pub const SWITCHBOARD_PROGRAM_ID: Pubkey = switchboard_on_demand::ON_DEMAND_MAINNET_PID;
```

The feature flag correctly selects between:
- `ON_DEMAND_DEVNET_PID` — Switchboard devnet VRF program
- `ON_DEMAND_MAINNET_PID` — Switchboard mainnet VRF program

This PID is used as an `owner` constraint on randomness accounts:

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/epoch-program/src/instructions/consume_randomness.rs:52`
```rust
#[account(owner = SWITCHBOARD_PROGRAM_ID @ EpochError::InvalidRandomnessOwner)]
pub randomness_account: AccountInfo<'info>,
```

The constraint prevents fake randomness injection — an attacker cannot pass a self-owned randomness account because the owner check would fail.

### Build-Time Safeguard (Per MEMORY.md)

From `/Users/mlbob/.claude/projects/-Users-mlbob-Projects-Dr-Fraudsworth/memory/MEMORY.md`:

> Devnet build: `anchor build` then `anchor build -p epoch_program -- --features devnet` (or `build.sh --devnet`). **Without devnet feature, epoch_program compiles with mainnet Switchboard PID causing ConstraintOwner errors.**

This indicates a critical transition point:
- **Local/devnet testing:** Must use `--features devnet` to get devnet Switchboard PID
- **Mainnet deployment:** Must omit `--features devnet` to get mainnet Switchboard PID
- **Risk:** If feature flag is forgotten during mainnet build, all randomness validation will fail at runtime

## Impact

### Non-Vulnerability Assessment

The Switchboard SDK v0.11.3 itself presents **no security vulnerability** to the protocol:

- No known CVEs in v0.11.3
- No breaking API changes since January 2026 validation
- Cryptographic operations (SECP256K1, SHA-256) are industry-standard
- VRF proof verification is delegated to Switchboard's on-chain program (not implemented locally)

### Operational Risks (MEDIUM)

**Risk 1: Mainnet Transition Feature Flag Mistake**
- **Scenario:** Deployer forgets to remove `--features devnet` flag when building for mainnet
- **Result:** Epoch program compiled with devnet Switchboard PID; all randomness accounts rejected with ConstraintOwner error
- **Recovery:** Requires redeployment (non-upgradeable programs)
- **Mitigation:** This risk is documented in MEMORY.md and build scripts should enforce the correct feature flag

**Risk 2: SDK Upgrade Path Uncertainty**
- **Scenario:** Switchboard releases v0.12.x with breaking API changes (acknowledged as possible in VRF_Migration_Lessons.md Section 6.1)
- **Result:** Epoch program code may not compile against new SDK version
- **Recovery:** Code changes + redeployment (non-upgradeable programs)
- **Mitigation:** Switchboard maintains backward compatibility with on-chain data formats; protocol only reads randomness, not calls into Switchboard, so major API breaks would be unlikely

**Risk 3: Switchboard Infrastructure Outage**
- **Scenario:** Switchboard's SGX oracle infrastructure goes offline or is deprecated
- **Result:** VRF requests timeout; protocol enters timeout recovery loop (300-slot window)
- **Severity:** Liveness issue, not security issue — tax rates freeze, no new Carnage
- **Mitigation:** Timeout recovery mechanism prevents deadlock; already documented in MEMORY.md

## Recommendation

### Accept Current State ✓

The protocol's Switchboard SDK usage is production-ready with **no security vulnerabilities** requiring immediate remediation.

## Technical Validation

### Switchboard On-Demand Security Model

The protocol correctly delegates VRF security to Switchboard:

1. **Proof Verification:** Switchboard's on-chain program verifies oracle SGX attestations and VRF proofs — not reimplemented in Dr Fraudsworth
2. **Randomness Source:** SGX Trusted Execution Environment (TEE) with cryptographic attestation — industry-standard for randomness generation
3. **Anti-Reroll Protection:** Binding randomness account key at commit time prevents substitution — correctly implemented in trigger_epoch_transition.rs
4. **Freshness Enforcement:** seed_slot <= 1 slot check prevents pre-generated randomness reuse — correctly implemented

### Crate Quality Indicators

- **Update frequency:** switchboard-on-demand v0.11.3 released 2025-12 (2 months before audit)
- **GitHub activity:** Switchboard organization maintains active development
- **Community adoption:** Switchboard is the recommended VRF solution by Anchor/Solana teams
- **Documentation:** Official examples and tutorials exist for the On-Demand pattern

## Conclusion

**The Switchboard SDK v0.11.3 dependency is appropriate, well-validated, and poses no security vulnerability.**

The only concern is **operational: the mainnet build transition requires careful attention to feature flags**. This risk is well-documented and manageable through process discipline.

The protocol's usage pattern (reading revealed randomness via RandomnessAccountData) is stable, minimal, and relies on standard Switchboard APIs expected to persist across minor version upgrades.
