# H086: VRF Commit Without Proper Slot Validation

**Strategy:** STR-086: VRF Commit Without Proper Slot Validation — Hypothesis: `commit_vrf` might accept a randomness account created for a past or future slot, not the current epoch. Target: `epoch_program/commit_vrf` — slot validation. Investigation: Check slot freshness validation on VRF commit.

---

## Determination

**FINDING DISPROVEN / NOT APPLICABLE**

The hypothesis was based on the assumption that `commit_vrf` (implemented as `trigger_epoch_transition`) lacked proper slot validation. Upon thorough code review, **comprehensive slot freshness validation is present and correctly implemented**.

---

## Severity

**N/A** — No vulnerability found.

---

## Confidence

**HIGH** — Code inspection is definitive. The validation checks are explicitly present in the handler and verified by unit tests.

---

## Summary

The `trigger_epoch_transition` instruction (which handles the VRF commit phase) implements **multi-layer slot validation** that prevents acceptance of randomness accounts created outside the current epoch context:

1. **Epoch Boundary Validation** (lines 136-139): Enforces that the current slot has advanced past the expected epoch boundary before accepting a commit.

2. **Seed-Slot Freshness Check** (lines 159-166): Validates that `randomness_data.seed_slot` is within 1 slot of the current slot:
   ```rust
   let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
   require!(slot_diff <= 1, EpochError::RandomnessExpired);
   ```
   This prevents use of pre-generated or stale randomness accounts from past epochs.

3. **Not-Yet-Revealed Check** (lines 170-173): Ensures the randomness is still in the commit phase and has not been revealed:
   ```rust
   if randomness_data.get_value(clock.slot).is_ok() {
       return Err(EpochError::RandomnessAlreadyRevealed.into());
   }
   ```
   This prevents reuse of randomness accounts whose output is already observable.

4. **Timeout Recovery with Fresh Slot Binding** (retry_epoch_vrf.rs lines 82-90): The retry mechanism applies the same freshness checks to new randomness accounts, ensuring that even in recovery paths, only current-epoch randomness is accepted.

Ensures the current slot is past the epoch boundary, preventing premature triggers.

**Freshness Check (lines 157-166):**
```rust
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
msg!(
    "Randomness freshness: current_slot={}, seed_slot={}, diff={}",
    clock.slot,
    randomness_data.seed_slot,
    slot_diff
);
require!(slot_diff <= 1, EpochError::RandomnessExpired);
```
Strictly enforces that `seed_slot` is the current slot or immediately previous slot. A randomness account created 2+ slots ago will be rejected.

**Not-Yet-Revealed Check (lines 168-173):**
```rust
if randomness_data.get_value(clock.slot).is_ok() {
    msg!("Randomness already revealed - cannot use for new commit");
    return Err(EpochError::RandomnessAlreadyRevealed.into());
}
```
Prevents binding to a randomness account that Switchboard has already revealed. The protocol enforces the commit-reveal ordering.

## Impact

**None** — No vulnerability exists. The protocol correctly validates that:
- Randomness accounts are created in the current slot or immediately previous slot
- Randomness accounts are not yet revealed at commit time
- Epoch boundaries are correctly enforced before accepting commits
- Retry paths apply the same validation as initial commits

The three-layer validation (epoch boundary + seed-slot freshness + not-yet-revealed) prevents accepting randomness from past or future epoch contexts.
