# H123: Epoch Duration Tight Enough to Stress VRF Completion Window

**Strategy:** STR-123 — Epoch Duration Too Short Allows VRF Spam
**Determination:** NOT EXPLOITABLE — Timing is adequate
**Severity:** LOW (architectural observation)
**Confidence:** HIGH (timing constants verified, flow analyzed)

## Summary

The protocol uses feature-flagged epoch durations: **750 slots (devnet, ~5 min)** and **4,500 slots (mainnet, ~30 min)**. The VRF (Verifiable Random Function) commit-reveal cycle requires **~3-5 slots** (1-2 seconds) to complete after `trigger_epoch_transition` bundles with Switchboard's commit instruction.

**Finding:** The devnet epoch duration (750 slots) is **NOT tight** relative to the VRF completion window. However, a potential spam vector was identified: if an attacker could submit multiple VRF commit transactions within the same epoch, they could theoretically waste Carnage Fund SOL on failed reveal attempts. This is **structurally prevented** by the "no double-commit" check.

**Conclusion:** Epoch durations are appropriate. The protocol's anti-reroll protection and single-commit-per-epoch enforcement make VRF spam impossible.

### 2. VRF Commit-Reveal Flow Timing

**File:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:106-128`

The VRF flow is a 3-transaction sequence:

```
TX1: Client creates randomness account (off-chain, prior)
TX2: Client bundles:
     - Switchboard SDK commitIx (tells oracle: "here's randomness account X, commit to seed S")
     - trigger_epoch_transition (binds account X for anti-reroll)
     → Oracle receives commit and starts computing randomness

TX3: ~3-5 slots later (1-2 seconds)
     - Switchboard SDK revealIx (oracle posts the randomness value)
     - consume_randomness (reads VRF bytes, derives tax rates, checks Carnage)
```

**Key timing details:**

1. **Commit phase duration:** Oracle has up to 30 slots to reveal before `VRF_TIMEOUT_SLOTS = 300` fires
2. **Typical reveal window:** 3-5 slots (~1.2-2 seconds) on Switchboard devnet
3. **Freshness check at trigger:** `seed_slot` must be within 1 slot of current (prevents pre-aged randomness)
4. **Not-yet-revealed check:** `randomness_data.get_value(clock.slot)` must fail, confirming oracle is still computing

**File:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:157-166`

```rust
// Freshness check: seed_slot must be within 1 slot of current
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
require!(slot_diff <= 1, EpochError::RandomnessExpired);

// Not-yet-revealed check: must still be in commit phase
if randomness_data.get_value(clock.slot).is_ok() {
    msg!("Randomness already revealed - cannot use for new commit");
    return Err(EpochError::RandomnessAlreadyRevealed.into());
}
```

### 3. Anti-Double-Commit Protection

**File:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:147-149`

```rust
// === 2. Validate no VRF already pending ===
// Prevents double-commit attacks and state inconsistency
require!(!epoch_state.vrf_pending, EpochError::VrfAlreadyPending);
```

This check **enforces exactly one VRF commit per epoch**. Once `vrf_pending = true` is set, no new trigger can occur until `consume_randomness` clears it. Attackers cannot submit multiple commits.

**File:** `programs/epoch-program/src/instructions/consume_randomness.rs:150-151`

```rust
// === 1. Validate VRF is pending ===
require!(epoch_state.vrf_pending, EpochError::NoVrfPending);
```

The consume handler requires `vrf_pending = true`, and after processing, sets it back to `false`.

### 4. Timing Margin Analysis

**Devnet epoch (750 slots) breakdown:**

| Phase | Duration | Slots | % of Epoch |
|-------|----------|-------|-----------|
| Epoch starts | 0 | 0 | 0% |
| Trigger allowed (anytime after boundary) | ~1-2 slots | 1-2 | 0.1% |
| VRF commit → reveal | ~3-5 slots | 3-5 | 0.4-0.7% |
| Carnage atomic lock window | 50 slots | 50 | 6.7% |
| Carnage fallback window | 250 slots | 250 | 33% |
| Total consumed | ~304-306 slots | 304-306 | 40.5% |
| **Remaining slack** | **~444-446 slots** | 444-446 | **59%** |

The devnet epoch has **substantial slack**. Even if VRF reveal took 10x longer (50 slots), the protocol would still function.

**Mainnet epoch (4,500 slots) is even more generous:**
- Total consumed for typical flow: ~350 slots
- Remaining slack: ~4,150 slots (92% of epoch)

### 5. Bounty Incentive & Spam Resistance

**File:** `programs/epoch-program/src/constants.rs:77-81`

```rust
/// Bounty paid to epoch trigger caller (0.001 SOL).
/// Incentivizes timely epoch transitions.
/// ~66x actual 3-TX base cost -- generous but treasury-efficient.
pub const TRIGGER_BOUNTY_LAMPORTS: u64 = 1_000_000;  // 0.001 SOL
```

**Spam cost analysis:**
- Each `trigger_epoch_transition` TX costs ~5,000 lamports in fees (0.000005 SOL)
- Bounty reward: 1,000,000 lamports (0.001 SOL) = **200x the TX fee**
- To spam Carnage Fund with failed reveals, attacker would need to repeatedly trigger **but can't** due to anti-double-commit

An attacker who submits a TX that fails (e.g., randomness already revealed) pays 5,000 lamports in fees and wastes compute. The bounty is irrelevant because the instruction reverts. This is not economically viable spam.

### 6. VRF Timeout Recovery

**File:** `programs/epoch-program/src/constants.rs:67-70`

```rust
/// VRF timeout in slots (~2 minutes).
/// If oracle doesn't reveal within this window, retry is permitted.
pub const VRF_TIMEOUT_SLOTS: u64 = 300;
```

If Switchboard oracle fails to reveal within 300 slots:
1. After 300 slots, `retry_epoch_vrf` (permissionless) creates a fresh randomness
2. The fresh randomness account may be assigned to a different (working) oracle
3. The cycle restarts

**File:** `programs/epoch-program/src/instructions/retry_epoch_vrf.rs:64-74` (from 08-timing-ordering.md)

```rust
let elapsed_slots = clock.slot.saturating_sub(epoch_state.vrf_request_slot);
require!(elapsed_slots > VRF_TIMEOUT_SLOTS, EpochError::VrfTimeoutNotElapsed);
```

This uses strict `>`, so timeout fires at slot `vrf_request_slot + 301`. The 300-slot window is adequate for:
- Normal oracle reveal (~3-5 slots)
- Network delays and reorg recovery (~10-20 slots)
- Oracle fallback retries (multiple attempts within 300 slots)

## Impact

### Attack Vector: VRF Spam (REJECTED)

**Hypothesis:** If an attacker could submit unlimited VRF commits per epoch, they could:
1. Create multiple randomness accounts
2. Submit multiple `trigger_epoch_transition` TXs
3. Only one would succeed; others would fail on reveal
4. Failed reveals would waste Carnage Fund SOL if bounties were paid before validation

**Why it fails:**

The `!epoch_state.vrf_pending` check prevents any second trigger until the first VRF completes. Once a TX succeeds, `vrf_pending = true`. A second submit **reverts** before any state change or bounty transfer.

The bounty is only paid at the **end** of `trigger_epoch_transition` (lines 194-227), after all validations. If the anti-double-commit check fails, the instruction reverts and no bounty is transferred.

**Mitigation effectiveness:** STRONG (compile-time enforced)

### Architectural Observation: Tight but Adequate

The devnet 750-slot epoch is not tight in absolute terms. A typical VRF cycle takes 3-5 slots, leaving 745+ slots for:
- Carnage execution (atomic + fallback)
- User swaps
- Staking rewards computation
- Next epoch preparation

However, on mainnet with 30-minute epochs, the ratio is even more generous.

**No exploitable timing race identified.**

## Appendix: Slot Timing Sensitivity

### Solana Slot Skipping Behavior

Per solana-runtime-quirks.md: validators may skip slots during outages or high load. Slot numbers are **not guaranteed continuous**.

**Impact on epoch calculations:**
- If slots are skipped across epoch boundary, `current_epoch()` still calculates correctly: `((slot - genesis_slot) / SLOTS_PER_EPOCH)` uses division, not increment
- Example: if slot jumps from 1,749 to 1,760 (skipping 10 slots), and epoch boundary is at 1,750, the protocol still detects the boundary correctly on the next valid slot

**Impact on VRF timeout:**
- 300-slot timeout could expire faster in wall-clock time if slots are skipped
- With heavy slot skipping, 300 slots might represent 60-90 seconds instead of 120 seconds
- This is acceptable because the timeout is a safety valve, not a precise timer

**Conclusion:** Slot skipping does not create exploitable timing gaps.
