## Summary

Hypothesis H127 postulated that Tax Program's Carnage fund SOL deposits might not call `syncNative` when wrapping to WSOL, causing the WSOL token balance to not reflect the SOL deposit. Investigation confirms the hypothesis is **false**. The Epoch Program's `wrap_sol_to_wsol()` function correctly implements the full two-step wrapping pattern:

1. **System Program Transfer**: SOL lamports from `carnage_sol_vault` PDA → `carnage_wsol` token account
2. **SyncNative**: SPL Token `sync_native` instruction to update WSOL token balance

The implementation is sound and no vulnerability exists.

### Implementation Details

```rust
fn wrap_sol_to_wsol<'info>(
    ctx: &Context<'_, '_, 'info, 'info, ExecuteCarnageAtomic<'info>>,
    amount: u64,
    sol_vault_bump: u8,
) -> Result<()> {
    // 1. Transfer native SOL from sol_vault PDA to carnage_wsol token account.
    let sol_vault_seeds: &[&[u8]] = &[CARNAGE_SOL_VAULT_SEED, &[sol_vault_bump]];

    let transfer_ix = anchor_lang::solana_program::system_instruction::transfer(
        &ctx.accounts.sol_vault.key(),
        &ctx.accounts.carnage_wsol.key(),
        amount,
    );

    invoke_signed(
        &transfer_ix,
        &[
            ctx.accounts.sol_vault.to_account_info(),
            ctx.accounts.carnage_wsol.to_account_info(),
            ctx.accounts.system_program.to_account_info(),
        ],
        &[sol_vault_seeds],
    )?;

    // 2. SyncNative: tells the SPL Token program to update the WSOL token balance
    //    to match the account's lamport balance (minus rent-exempt minimum).
    let sync_native_ix = Instruction {
        program_id: ctx.accounts.token_program_a.key(),
        accounts: vec![
            AccountMeta::new(ctx.accounts.carnage_wsol.key(), false),
        ],
        data: vec![17u8], // SyncNative instruction discriminator
    };

    invoke_signed(
        &sync_native_ix,
        &[
            ctx.accounts.carnage_wsol.to_account_info(),
            ctx.accounts.token_program_a.to_account_info(),
        ],
        &[], // SyncNative is permissionless -- no signer needed
    )?;

    msg!("Wrapped {} lamports SOL -> WSOL in carnage_wsol", amount);
    Ok(())
}
```

### Why syncNative is Required

The WSOL (Wrapped SOL) token is a special account owned by the SPL Token program. When SOL (lamports) are directly transferred to it via `system_program::transfer`, the account's **lamport balance increases**, but the **SPL token balance** (tracked in the token metadata) does NOT update automatically.

`syncNative` (instruction discriminator 17) is a permissionless SPL Token instruction that:
- Reads the account's lamport balance
- Subtracts the rent-exempt minimum
- Updates the token balance to match remaining lamports
- Enables the WSOL to be used in subsequent `transfer_checked` instructions

Without this call, the WSOL token account would have zero token balance despite containing SOL, breaking the swap that depends on the WSOL balance.
