# H019: Hardcoded Byte Offsets Drift After Anchor Upgrade (MEDIUM)
**Strategy:** STR-019
**Determination:** VULNERABLE
**Severity:** MEDIUM
**Confidence:** HIGH

## Summary

The `read_pool_reserves()` function in the Tax Program and the dual Carnage execution paths in the Epoch Program use **hardcoded byte offsets** `[137..145]` and `[145..153]` to read `reserve_a` and `reserve_b` from the AMM's PoolState account. While these offsets are **currently correct** for Anchor 0.32.1's serialization layout, they have **no compile-time or runtime verification** against the actual PoolState struct definition.

If Anchor's serialization format changes in a future version (e.g., Borsh encoding modifications, discriminator changes, or field ordering adjustments), or if the PoolState struct is modified to add fields before the reserves, these offsets will **silently read incorrect data**, causing:

1. **Wrong slippage calculations** (reading garbage as reserves)
2. **Wrong swap direction** (reversed swap if reserves are flipped)
3. **Protocol-wide swap failures** (all tax swaps, PROFIT swaps, and Carnage executions broken)

Unlike H013 (canonical mint ordering), which compares against a known constant and fails safely if offsets are wrong, the reserve offsets are used **directly in arithmetic calculations** with no validation that the data represents valid reserves.

### 2. Offset Calculation (Currently Correct)

PoolState struct layout per Anchor 0.32.1:

```
[0..8]     Anchor discriminator (8 bytes)
[8..9]     pool_type (1 byte enum)
[9..41]    mint_a (Pubkey, 32 bytes)
[41..73]   mint_b (Pubkey, 32 bytes)
[73..105]  vault_a (Pubkey, 32 bytes)
[105..137] vault_b (Pubkey, 32 bytes)
[137..145] reserve_a (u64, 8 bytes)  ✓ MATCHES pool_reader.rs
[145..153] reserve_b (u64, 8 bytes)  ✓ MATCHES pool_reader.rs
[153..155] lp_fee_bps (u16, 2 bytes)
[155..156] initialized (bool, 1 byte)
[156..157] locked (bool, 1 byte)
[157..158] bump (u8, 1 byte)
[158..159] vault_a_bump (u8, 1 byte)
[159..160] vault_b_bump (u8, 1 byte)
[160..192] token_program_a (Pubkey, 32 bytes)
[192..224] token_program_b (Pubkey, 32 bytes)

Total: 8 + 224 = 232 bytes
```

**Verified:** All four test files in `programs/amm/tests/` independently calculate these same offsets using incremental offset tracking (see test_swap_profit_pool.rs:730-747, test_swap_sol_pool.rs:681-696, etc.), confirming the layout is correct **for the current Anchor version**.

### 3. Lack of Compile-Time Verification

**No static assertions:**
```rust
// DOES NOT EXIST anywhere in the codebase:
const _: () = assert!(offset_of!(PoolState, reserve_a) == 137);
const _: () = assert!(core::mem::size_of::<PoolState>() == 224);
```

**No runtime validation:**
- Length check `data.len() >= 153` only verifies the account is big enough to read 153 bytes
- Does NOT verify that bytes [137..145] actually contain `reserve_a` (could be garbage if layout shifted)
- Does NOT verify the data is a valid PoolState account (discriminator is not checked)
- Does NOT verify reserve values are sane (e.g., non-zero, within expected bounds)

**Comparison to H013 (safe pattern):**
- H013 reads `mint_a` at offset [9..41] and compares it to `profit_mint()` constant
- If offset is wrong, comparison yields garbage, `is_reversed` gets wrong value, AMM swap **fails with direction mismatch error**
- Reserve offsets have **no such safety check** — wrong data is used directly in arithmetic

### 4. AMM Test Suite Provides NO Protection

The AMM test suite (299 passing tests) verifies swap logic **using the AMM crate's PoolState struct**, which Anchor deserializes automatically. The tests do NOT verify:
- Cross-program raw byte reading correctness
- Offset stability across Anchor versions
- Tax/Epoch program's hardcoded offset assumptions

The four test helper functions that manually parse PoolState bytes (test_swap_profit_pool.rs:730-760, etc.) are **test utilities only** and do not run as assertions against production code paths.

### 5. Anchor Serialization Stability Assumptions

**Assumption (currently true):** Anchor's `#[account]` macro generates deterministic Borsh serialization with stable field ordering.

**Risk vectors:**

| Scenario | Likelihood | Impact |
|----------|-----------|--------|
| Anchor 0.33+ changes Borsh encoding | LOW | CRITICAL |
| PoolState struct modified (new field before reserves) | MEDIUM | CRITICAL |
| Discriminator calculation changes | VERY LOW | CRITICAL |
| Developer adds field to PoolState during refactor | MEDIUM | CRITICAL |
| Cross-program dependency on wrong AMM version | LOW | CRITICAL |

**Historical precedent:**
- Anchor 0.27 → 0.28 changed discriminator calculation for events (not accounts)
- Anchor 0.29 → 0.30 changed IDL generation (not serialization)
- No known breaking changes to `#[account]` Borsh layout **yet**

**Future risk:**
- Solana's account versioning proposals (ongoing) could introduce optional metadata before account data
- Anchor's `InitSpace` derive macro could be enhanced to pad/align fields differently
- Multi-version protocol upgrades (where AMM upgrades to new Anchor but Tax/Epoch lag behind) could create layout mismatch

## Root Cause

The Tax and Epoch programs avoid Cargo dependencies on the AMM crate to reduce coupling and compile times. This architectural decision forces cross-program state reads to use raw byte parsing instead of importing and deserializing the PoolState struct.

**Design tradeoff:**
- **Benefit:** No circular dependencies, faster builds, smaller program sizes
- **Cost:** Fragile byte offset coupling with no compile-time validation

**Why not deserialize PoolState directly?**

From `pool_reader.rs:35-38`:
> The Tax Program has no Cargo dependency on the AMM crate and does not import PoolState. Raw byte reads at known offsets avoid cross-crate coupling. This pattern is proven in Carnage code.

This justification assumes offsets will remain stable indefinitely, which is **not guaranteed** by Anchor or Solana's stability promises.

## Attack Scenario

### Scenario 1: Anchor 0.33 Changes Borsh Encoding

**Timeline:**
1. Dr Fraudsworth v1.0 deployed to mainnet with Anchor 0.32.1
2. Programs are non-upgradeable (immutable)
3. Anchor 0.33 releases with a Borsh optimization that reorders boolean fields to pack bytes
4. PoolState layout changes:
   ```
   OLD [137..145] reserve_a
   NEW [135..143] reserve_a (shifted 2 bytes left due to bool packing)
   ```
5. Tax Program's hardcoded `[137..145]` now reads **2 bytes of reserve_a + 6 bytes of lp_fee_bps/initialized/locked**

**Exploitation:**
- User calls `swap_buy_sol` with 1000 SOL
- Tax Program reads "reserve_a" = `0x00640001000000FF` (partial reserve_a + partial lp_fee_bps)
- This interprets as u64 = 18,374,686,479,671,623,680 (garbage)
- Slippage calculation uses wrong reserves
- AMM CPI executes with wrong expected output
- User receives far less tokens than expected (or transaction reverts with slippage error)

**Impact:** All tax swaps broken until Tax/Epoch programs are redeployed (but they're non-upgradeable!)

### Scenario 2: PoolState Struct Modified (Add Field Before Reserves)

**Timeline:**
1. Phase 60 adds a new feature: `pub pool_owner: Pubkey` field to PoolState for admin tracking
2. Developer inserts field **before reserves** to keep related fields grouped:
   ```rust
   pub vault_a: Pubkey,
   pub vault_b: Pubkey,
   pub pool_owner: Pubkey,  // ← NEW, 32 bytes
   pub reserve_a: u64,      // NOW at offset [169..177] instead of [137..145]
   pub reserve_b: u64,      // NOW at offset [177..185] instead of [145..153]
   ```
3. AMM program is upgraded (or redeployed)
4. Tax/Epoch programs still use old offsets [137..145]

**Exploitation:**
- Old offset [137..145] now reads **the last 8 bytes of pool_owner pubkey**
- Interpreted as u64 reserve_a = random value (public key bytes)
- If pubkey bytes happen to be small, reserve_a reads as tiny value
- Slippage calculation: `min_output = reserve_out * amount_in / (reserve_in + amount_in)`
- With `reserve_in` = garbage (could be 0, causing division by zero), or huge (causing underflow), or tiny (causing massive output expectation)

**Outcome:** Protocol-wide swap failure, user fund loss, or pool drainage (if min_output is set to 0 due to overflow/underflow in tax calculation)

## Impact Assessment

**Severity:** MEDIUM (not HIGH) because:
- **Current deployment is safe:** Offsets are correct for Anchor 0.32.1
- **Programs are non-upgradeable:** No risk of accidental struct modification post-deploy
- **Requires external trigger:** Anchor upgrade or deliberate struct change

**Elevated to HIGH if:**
- Mainnet deployment uses upgradeable programs
- PoolState struct changes are planned
- Cross-version compatibility is required (e.g., testing new AMM against old Tax program)

**Impact scope:**
- **User funds:** Incorrect swaps could cause users to receive wrong token amounts (loss of funds)
- **Protocol integrity:** All tax swaps, PROFIT swaps, and Carnage executions broken
- **Availability:** Protocol becomes unusable until Tax/Epoch programs are patched and redeployed
- **Reputation:** Silent failures are worse than reverts — users lose funds without understanding why

## Remediation

### Option 1: Add Compile-Time Assertions (RECOMMENDED)

Add to `programs/amm/src/state/pool.rs`:
```rust
#[cfg(test)]
mod layout_tests {
    use super::*;
    use core::mem::offset_of;  // Requires Rust 1.77+

    #[test]
    fn test_reserve_offsets_for_cross_program_reads() {
        // These offsets are hardcoded in tax-program/pool_reader.rs
        // and epoch-program/execute_carnage*.rs
        const EXPECTED_RESERVE_A_OFFSET: usize = 137 - 8;  // -8 for discriminator
        const EXPECTED_RESERVE_B_OFFSET: usize = 145 - 8;

        assert_eq!(
            offset_of!(PoolState, reserve_a),
            EXPECTED_RESERVE_A_OFFSET,
            "reserve_a offset changed! Update Tax/Epoch programs."
        );
        assert_eq!(
            offset_of!(PoolState, reserve_b),
            EXPECTED_RESERVE_B_OFFSET,
            "reserve_b offset changed! Update Tax/Epoch programs."
        );
    }
}
```

**Benefit:** Any PoolState struct modification that changes reserve offsets **breaks the build**, forcing developer awareness.

### Option 2: Runtime Sanity Checks (Defense-in-Depth)

Add to `pool_reader.rs:read_pool_reserves()`:
```rust
pub fn read_pool_reserves(pool_info: &AccountInfo) -> Result<(u64, u64)> {
    let data = pool_info.data.borrow();
    require!(data.len() >= 153, TaxError::InvalidPoolType);

    // VERIFY ANCHOR DISCRIMINATOR (first 8 bytes)
    const POOL_STATE_DISCRIMINATOR: [u8; 8] = [
        0x9d, 0x84, 0x91, 0x2e, 0x5c, 0xa1, 0x8f, 0x1a  // sha256("account:PoolState")[0..8]
    ];
    require!(
        data[0..8] == POOL_STATE_DISCRIMINATOR,
        TaxError::InvalidPoolType
    );

    let reserve_a = u64::from_le_bytes(data[137..145].try_into()?);
    let reserve_b = u64::from_le_bytes(data[145..153].try_into()?);

    // SANITY CHECK: Reserves should be non-zero and within reasonable bounds
    require!(reserve_a > 0 && reserve_a < u64::MAX / 2, TaxError::InvalidPoolType);
    require!(reserve_b > 0 && reserve_b < u64::MAX / 2, TaxError::InvalidPoolType);

    Ok((reserve_a, reserve_b))
}
```

**Benefit:**
- Discriminator check ensures we're reading a PoolState account (not garbage)
- Sanity checks on reserve values catch offset drift (would read garbage values outside expected range)
- Graceful failure (revert with error) instead of silent corruption

**Tradeoff:** Adds ~300 compute units per call (negligible vs total swap cost ~50k CU)

### Option 3: Import PoolState (Ideal but Breaking Change)

**Add to `programs/tax-program/Cargo.toml`:**
```toml
[dependencies]
amm = { path = "../amm", features = ["cpi"] }
```

**Modify pool_reader.rs:**
```rust
use amm::state::PoolState;

pub fn read_pool_reserves(pool_info: &AccountInfo) -> Result<(u64, u64)> {
    let pool: Account<PoolState> = Account::try_from(pool_info)?;
    Ok((pool.reserve_a, pool.reserve_b))
}
```

**Benefit:**
- Type-safe, no hardcoded offsets
- Automatic deserialization with Anchor's validation
- Compiler enforces layout compatibility

**Tradeoff:**
- Circular dependency (Tax depends on AMM, AMM depends on Tax for integration tests)
- Larger program size (includes AMM state definitions)
- Requires full redeploy of Tax/Epoch programs

### Option 4: Shared State Crate (Mainnet Refactor)

Create `programs/shared-state/` crate with PoolState definition. Both AMM and Tax/Epoch import it.

**Benefit:** Single source of truth, no circular deps, compile-time safety
**Tradeoff:** Significant refactor, requires redeploy of all programs
