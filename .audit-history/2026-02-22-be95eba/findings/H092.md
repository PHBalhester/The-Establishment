# H092: Pool Reserves Overflow on Large Deposit

**Strategy:** STR-092: Pool Reserves Overflow on Large Deposit

**Determination:** NOT EXPLOITABLE in practice, but THEORETICAL VULNERABILITY exists

**Severity:** INFORMATIONAL (upgrading finding from MEDIUM due to practical constraints)

**Confidence:** HIGH (95%)

---

## Summary

Pool reserves in the AMM program are stored as `u64` (18-byte unsigned integer, max ~1.84 × 10^19). Hypothesis STR-092 proposes that a deposit sufficiently large to push reserves above `u64::MAX` could trigger an overflow, corrupting pool state or enabling fund theft.

Investigation confirms:
1. **Reserve updates DO use `checked_add`** — overflow is caught and transaction reverts (no silent corruption)
2. **Practical overflow is impossible** — Solana SPL Token mint supply (u64 max ~18.4 billion tokens) at 6 decimal precision yields ~18.4 million whole tokens. Even Solana's entire SOL supply (~425M SOL) fits comfortably in u64
3. **Theoretical vulnerability path:** A hypothetical Token-2022 mint with supply > 10^18 tokens (base units) could theoretically overflow reserves if all of it were deposited into a single pool
4. **No initialization deposit limit** — The `initialize_pool` instruction does not validate `amount_a` and `amount_b` cannot sum to u64::MAX

## Evidence

### 1. Reserve Update Code (Checked Arithmetic)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/swap_sol_pool.rs:163-168`

```rust
// 8. Compute new reserves (amount_in is pre-fee -- fee stays in pool)
let new_reserve_in = reserve_in
    .checked_add(amount_in)
    .ok_or(AmmError::Overflow)?;
let new_reserve_out = reserve_out
    .checked_sub(amount_out)
    .ok_or(AmmError::Overflow)?;
```

**Analysis:** Both `checked_add` and `checked_sub` are used. If `reserve_in + amount_in > u64::MAX`, the `.checked_add()` returns `None`, which is immediately mapped to `AmmError::Overflow` via `.ok_or()`. The transaction reverts — no wrapping, no silent data corruption.

**Identical pattern in:** `swap_profit_pool.rs:131-136`, confirming consistent implementation across all swap handlers.

### 2. K-Invariant Verification (Overflow Detection)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/helpers/math.rs:92-103`

```rust
pub fn verify_k_invariant(
    reserve_in_before: u64,
    reserve_out_before: u64,
    reserve_in_after: u64,
    reserve_out_after: u64,
) -> Option<bool> {
    let k_before = (reserve_in_before as u128)
        .checked_mul(reserve_out_before as u128)?;
    let k_after = (reserve_in_after as u128)
        .checked_mul(reserve_out_after as u128)?;
    Some(k_after >= k_before)
}
```

The k-invariant check widens reserves to u128 for multiplication. If a reserve had overflowed during `checked_add`, the k-invariant would detect the arithmetic error (on line 171 of swap_sol_pool.rs: `require!(k_valid, AmmError::KInvariantViolation)`). However, this is redundant since `checked_add` would have already prevented the overflow.

### 3. Practical Token Supply Constraints

**Solana Token Standards:**
- SPL Token mint: supply stored as u64, max ~18.4 × 10^18 base units
- Token-2022 mint: same constraint, enforced at the token program level
- Typical token decimals: 6-9 (SOL = 9, USDC = 6)
- Dr Fraudsworth tokens (CRIME, FRAUD, PROFIT): 6 decimals

**Reserve capacity analysis:**
- Max supply of a 6-decimal token: u64::MAX base units = 18,446,744,073,709,551,615 units = 18,446,744,073 whole tokens (~18 billion)
- Solana's entire SOL supply: ~425 million SOL (9 decimals) = 4.25 × 10^17 base units
- Dr Fraudsworth protocol: Initial mints are modest (millions of tokens), far below token program max

**Conclusion:** For overflow to occur, a token supply would need to exceed u64::MAX at the base-unit level. This violates the SPL Token standard itself — token programs reject mint amounts > u64::MAX.

### 4. No Initialization Deposit Validation

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/initialize_pool.rs:25-48`

```rust
pub fn handler<'info>(
    ctx: Context<'_, '_, 'info, 'info, InitializePool<'info>>,
    lp_fee_bps: u16,
    amount_a: u64,
    amount_b: u64,
) -> Result<()> {
    // --- Validation ---

    // Canonical ordering: mint_a must be the "smaller" pubkey.
    require!(
        ctx.accounts.mint_a.key() < ctx.accounts.mint_b.key(),
        AmmError::MintsNotCanonicallyOrdered
    );

    // Duplicate mints would create a nonsensical self-pairing pool.
    require!(
        ctx.accounts.mint_a.key() != ctx.accounts.mint_b.key(),
        AmmError::DuplicateMints
    );

    // Zero seed amounts would cause division-by-zero in the CPMM formula
    require!(amount_a > 0 && amount_b > 0, AmmError::ZeroSeedAmount);
```

**Observation:** The initialization validates:
- ✓ Canonical mint ordering
- ✓ No duplicate mints
- ✓ Non-zero seed amounts

But does **NOT** validate:
- ✗ `amount_a + amount_b <= u64::MAX` (though both are u64 individually)
- ✗ Maximum reasonable liquidity per pool

However, this is **not exploitable** because:
1. `amount_a` and `amount_b` are stored directly as reserves without addition (lines 132-133: `pool.reserve_a = amount_a; pool.reserve_b = amount_b`)
2. To overflow a reserve, a subsequent deposit (via swap) would add to these initial amounts
3. All deposits use `checked_add` and revert on overflow

## Attack Surface Analysis

### Scenario 1: Direct Overflow via Large Swap (BLOCKED)
**Attack:** Execute a swap with `amount_in` such that `reserve_in + amount_in > u64::MAX`
**Defense:** `checked_add` on line 164 of swap_sol_pool.rs
**Outcome:** Transaction reverts with `AmmError::Overflow`
**Feasibility:** IMPOSSIBLE

### Scenario 2: Cumulative Deposits (BLOCKED)
**Attack:** Execute many large swaps, each individually valid but totaling to overflow
**Defense:** Each swap uses `checked_add`; cumulative additions are safe as long as final sum ≤ u64::MAX
**Outcome:** Transaction reverts when any swap would cause overflow
**Feasibility:** IMPOSSIBLE (requires total supply > u64::MAX, violates SPL Token)

### Scenario 3: Token-2022 with Non-Standard Supply Cap (BLOCKED)
**Attack:** Create custom Token-2022 mint with supply > u64::MAX, deposit into pool
**Defense:** Token-2022 enforces u64 supply cap at the token program level — custom mints cannot exceed this
**Outcome:** Token program rejects mint creation/transfer exceeding u64::MAX
**Feasibility:** IMPOSSIBLE (enforced by Solana runtime)

### Scenario 4: Adversarial Initialization (THEORETICAL, NOT EXPLOITABLE)
**Attack:** Initialize pool with maximum valid amounts: `amount_a = amount_b = u64::MAX / 2 + 1`
**Result:** Both reserves set to ~9.2 × 10^18. Any subsequent swap that adds to either reserve would overflow.
**Defense:** `checked_add` prevents overflow. Also unrealistic — no token has supply > 2^63.
**Feasibility:** BLOCKED by practical token supply constraints + checked_add defense

## Impact

**If overflow occurred (hypothetically):**
1. **Silent data corruption:** NO — `checked_add` returns Option, forcing error handling
2. **Fund theft:** NO — checked arithmetic prevents silent wrapping
3. **Pool drain:** NO — k-invariant verification (lines 171-173) would catch anomalies
4. **User loss:** NO — transaction reverts on any arithmetic error

**Actual impact:** ZERO (mitigated by checked_add)

## Why This Remains Informational

1. **Checked arithmetic is present and correct** — The codebase uses Rust's built-in `checked_*` methods, which are the gold standard for overflow protection
2. **Practical constraints prevent exploitation** — SPL Token/Token-2022 supply limits (~2^64 base units) are orders of magnitude below any realistic DeFi scenario
3. **No defense bypass identified** — Every code path that modifies reserves uses `checked_*` methods
4. **Test coverage:** The codebase includes unit tests for `calculate_effective_input` with u64::MAX (math.rs lines 192-200), confirming developers considered this boundary


2. **Optional: Hardening** (not required, but defensive):
   - Add explicit validation in `initialize_pool` to reject initialization with `amount_a > MAX_REASONABLE_RESERVE`:
     ```rust
     const MAX_REASONABLE_RESERVE: u64 = 10_000_000_000_000_000; // 1 billion tokens at 6 decimals
     require!(amount_a <= MAX_REASONABLE_RESERVE && amount_b <= MAX_REASONABLE_RESERVE,
              AmmError::ExcessiveInitialLiquidity);
     ```
     This provides **defense-in-depth** (not strictly necessary, but makes intent explicit).
