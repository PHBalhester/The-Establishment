# H120: Pool Token Account Authority Properly Secured to Pool PDA

**Strategy:** STR-120: AMM Pool Token Account Authority Not Pool PDA

**Determination:** NOT A FINDING — Pool token account authorities are correctly set to the pool PDA. No vulnerability exists.

**Severity:** N/A (Cleared)

**Confidence:** Very High (100%)

## Summary

The hypothesis that pool token accounts might have incorrect authorities was **disproven**. Pool vault token accounts (vault_a and vault_b) are properly initialized with the pool PDA as the authority. This prevents unauthorized token drainage and is enforced at multiple levels through constraints and validation.

### 1. Vault Initialization (Pool Creation)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/initialize_pool.rs:230-252`

```rust
/// Vault A: PDA-owned token account for reserve A.
/// The pool PDA is the authority, ensuring only the program can move funds.
/// Seeds: [b"vault", pool.key().as_ref(), b"a"]
#[account(
    init,
    payer = payer,
    token::mint = mint_a,
    token::authority = pool,          // <-- Pool PDA as authority
    token::token_program = token_program_a,
    seeds = [VAULT_SEED, pool.key().as_ref(), VAULT_A_SEED],
    bump
)]
pub vault_a: InterfaceAccount<'info, TokenAccount>,

/// Vault B: PDA-owned token account for reserve B.
/// Seeds: [b"vault", pool.key().as_ref(), b"b"]
#[account(
    init,
    payer = payer,
    token::mint = mint_b,
    token::authority = pool,          // <-- Pool PDA as authority
    token::token_program = token_program_b,
    seeds = [VAULT_SEED, pool.key().as_ref(), VAULT_B_SEED],
    bump
)]
pub vault_b: InterfaceAccount<'info, TokenAccount>,
```

**Key Points:**
- Anchor's `token::authority = pool` constraint ensures the vault's on-chain authority field is set to the pool PDA at initialization
- This is enforced by Anchor's TokenAccount constraint validation during account creation
- The pool PDA is derived deterministically: `seeds = [b"pool", mint_a, mint_b]` with bump constraint

### 3. Vault Account Validation During Swaps

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/swap_sol_pool.rs:385-398`

Every swap instruction validates the vault accounts against the pool state:

```rust
/// Vault A: PDA-owned token account holding reserve A.
/// Validated against pool state to prevent vault substitution attacks.
#[account(
    mut,
    constraint = vault_a.key() == pool.vault_a @ AmmError::VaultMismatch,
)]
pub vault_a: InterfaceAccount<'info, TokenAccount>,

/// Vault B: PDA-owned token account holding reserve B.
#[account(
    mut,
    constraint = vault_b.key() == pool.vault_b @ AmmError::VaultMismatch,
)]
pub vault_b: InterfaceAccount<'info, TokenAccount>,
```

**Why This Matters:**
- The constraint `vault_a.key() == pool.vault_a` prevents an attacker from passing a different token account
- If an attacker tries to pass a vault account they control, the transaction fails with `VaultMismatch` error
- The vault passed must be the same PDA created during pool initialization

### 4. Transfer Authorization via Pool PDA Signer

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/helpers/transfers.rs:53-126`

When the AMM transfers tokens FROM vault TO user (output transfer), the pool PDA signs:

```rust
pub fn transfer_t22_checked<'info>(
    token_program: &AccountInfo<'info>,
    from: &AccountInfo<'info>,           // vault_a or vault_b
    mint: &AccountInfo<'info>,
    to: &AccountInfo<'info>,             // user token account
    authority: &AccountInfo<'info>,      // pool PDA (for vault-to-user)
    amount: u64,
    decimals: u8,
    signer_seeds: &[&[&[u8]]],           // pool PDA seeds passed here
    hook_accounts: &[AccountInfo<'info>],
) -> Result<()> {
    // ... validation ...

    // Invoke with signer_seeds so Token program validates pool PDA is authority
    anchor_lang::solana_program::program::invoke_signed(
        &ix,
        &account_infos,
        signer_seeds,  // Pool PDA derives signer from these seeds
    )?;
    Ok(())
}
```

**Security Model:**
- User transfers are unsigned (user provides authority)
- Vault transfers are signed by the pool PDA via `invoke_signed`
- The Token program validates the signer is the vault's on-chain authority
- If the vault authority were NOT the pool PDA, these transfers would fail

### 5. Cross-Reference: Pool Creation Sequence

The initialization instruction enforces the chain of custody:
1. Pool PDA is created with seeds: `[b"pool", mint_a, mint_b]`
2. Vault A PDA is created with seeds: `[b"vault", pool.key(), b"a"]` and authority = pool PDA
3. Vault B PDA is created with seeds: `[b"vault", pool.key(), b"b"]` and authority = pool PDA
4. Initial liquidity transferred IN to vaults (admin signs)
5. PoolState stores vault_a and vault_b addresses permanently

Later swaps:
6. Swap instruction validates passed vaults match PoolState
7. Output transfers OUT from vault (pool PDA signs via `invoke_signed`)
8. Token program validates signer is vault's authority (pool PDA)

## Impact Assessment

**If authorities were wrong (hypothetical):**
- Any token account with write authority could drain pool reserves
- Swaps would succeed but vaults would become inconsistent with PoolState
- Attackers could steal liquidity without triggering AMM constraints

**Actual implementation:**
- Pool PDA is the sole authority for vault accounts
- Only the pool PDA can sign transfers moving tokens out of vaults
- Only the AMM program (via hardcoded swap_authority PDA) can invoke pool operations
- No vulnerability exists

## Verification Checklist

✅ **Vault authority field:** Set to pool PDA at initialization via `token::authority = pool` constraint

✅ **Vault account validation:** Every swap validates `vault.key() == pool.vault_{a|b}` to prevent substitution

✅ **PDA-signed transfers:** Vault-to-user transfers require pool PDA signature via `invoke_signed`

✅ **No owner check needed:** Token program validates the signer is the vault's authority during `transfer_checked` CPI

✅ **No bypass vectors:** Users cannot invoke transfers directly (Anchor constraint + Token program validation)

✅ **Immutable storage:** Vault addresses stored in PoolState and validated on every access
