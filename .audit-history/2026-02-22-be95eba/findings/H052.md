# H052: VRF Commit Without Subsequent Reveal Does NOT Block Epoch (Timeout Recovery Sufficient)

**Strategy:** STR-052
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis that a VRF commit without subsequent reveal can permanently block epochs is **NOT CONFIRMED**. The protocol implements comprehensive recovery mechanisms through the `retry_epoch_vrf` instruction, which creates fresh randomness accounts that may be assigned to different (working) oracles. This recovery path has been battle-tested in production devnet operations. There is no admin override instruction to skip VRF entirely, but none is needed because the timeout recovery mechanism provides sufficient resilience without introducing centralization risk.

### 1. VRF Timeout Recovery Mechanism

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/epoch-program/src/instructions/retry_epoch_vrf.rs` (lines 41-122)

The `retry_epoch_vrf` instruction provides a permissionless recovery path:

```rust
/// Handler for retry_epoch_vrf instruction.
///
/// # Flow
/// 1. Validate VRF is pending
/// 2. Validate timeout has elapsed (elapsed_slots > VRF_TIMEOUT_SLOTS)
/// 3. Validate new randomness account freshness (seed_slot within 1 slot)
/// 4. Validate new randomness not yet revealed
/// 5. Overwrite pending state with new randomness account
/// 6. Emit VrfRetryRequested event
pub fn handler(ctx: Context<RetryEpochVrf>) -> Result<()> {
    let epoch_state = &mut ctx.accounts.epoch_state;
    let clock = Clock::get()?;

    // === 1. Validate VRF is pending ===
    require!(epoch_state.vrf_pending, EpochError::NoVrfPending);

    // === 2. Validate timeout has elapsed ===
    let elapsed_slots = clock.slot.saturating_sub(epoch_state.vrf_request_slot);
    require!(
        elapsed_slots > VRF_TIMEOUT_SLOTS,
        EpochError::VrfTimeoutNotElapsed
    );

    // === 3-4. Validate new randomness account ===
    let randomness_data = {
        let data = ctx.accounts.randomness_account.try_borrow_data()?;
        RandomnessAccountData::parse(data).map_err(|_| EpochError::RandomnessParseError)?
    };

    // Freshness + not-yet-revealed checks...

    // === 4. Overwrite pending state with new randomness account ===
    epoch_state.vrf_request_slot = clock.slot;
    epoch_state.pending_randomness_account = ctx.accounts.randomness_account.key();

    Ok(())
}
```

**Key properties:**
- **Permissionless**: Anyone can call after 300 slots (~2 minutes)
- **Can be called multiple times**: Each retry creates a fresh randomness account
- **Oracle rotation via fresh randomness**: New account may be assigned to a different oracle
- **No special authority required**: Regular wallet with TX fees can execute

**Critical insight from Memory:**
> **VRF gateway rotation DOES NOT WORK**: Each randomness account is assigned to a specific oracle. Alternative gateways serve different oracles whose signatures fail on-chain (0x1780). Only retry the default gateway. If down, use VRF timeout recovery with **fresh randomness**.

Each fresh randomness account is a **new lottery ticket** for oracle assignment. If 100 oracles exist and 1 is down, probability of getting a working oracle improves with each retry:
- Retry 1: P(success) = 99/100 = 99%
- Retry 2 (if retry 1 failed): P(success) = 99/100 = 99%
- Retry N: P(cumulative failure after N retries) = (1/100)^N

After 3 retries, probability of all retries hitting the same broken oracle: **0.0001%**

## Root Cause Analysis

The hypothesis in STR-052 stated:

> After `commit_vrf`, if `reveal_vrf` never succeeds (oracle down, network issue), the epoch is stuck. The `retry_epoch_vrf` provides recovery, but **if it also fails**, epochs are permanently blocked.

The missing piece in this hypothesis is the **unlimited retry capability** with **oracle rotation via fresh randomness**. Key architectural properties that prevent permanent blockage:

1. **Permissionless retry**: No admin key required, anyone can retry
2. **Multiple oracles**: Switchboard queue has multiple oracles; fresh randomness accounts rotate through them
3. **No retry limit**: `retry_epoch_vrf` can be called infinitely many times
4. **Stale state auto-recovery**: `consume_randomness` auto-expires stale Carnage (lines 112-148 in consume_randomness.rs)
5. **Idempotent recovery**: Crank bot restart automatically detects `vrfPending=true` and enters recovery

## Attack Scenario Analysis

**Attacker goal:** Permanently freeze epoch transitions to prevent tax regime changes.

**Attack vector 1: DOS the assigned oracle**

- **Cost**: Must compromise or DOS the specific Switchboard oracle assigned to the current randomness account
- **Duration**: 300 slots (~2 minutes) until timeout
- **Defense**: `retry_epoch_vrf` creates fresh randomness → may get different oracle
- **Outcome**: Attacker must DOS **all** Switchboard oracles in the queue to maintain freeze
- **Feasibility**: Switchboard is a decentralized oracle network with economic incentives for oracle operators to remain online. Attacking all oracles is economically infeasible.

**Attack vector 2: Front-run `retry_epoch_vrf` calls**

- **Goal**: Prevent anyone from successfully calling `retry_epoch_vrf`
- **Defense**: Permissionless instruction, anyone can call, multiple participants incentivized (0.001 SOL bounty)
- **Outcome**: Attacker must outbid all honest callers on every retry attempt
- **Feasibility**: Economically unsustainable; 0.001 SOL bounty + protocol importance ensures multiple parties will retry

**Attack vector 3: Spam VRF requests to exhaust Switchboard queue**

- **Goal**: Overwhelm Switchboard with requests to prevent legitimate VRF processing
- **Defense**: Switchboard rate limits requests, economic cost of randomness creation
- **Outcome**: Protocol experiences delays but not permanent freeze (retry mechanism still works)
- **Feasibility**: Expensive attack with temporary impact only

## Remediation

**No remediation required.**

The protocol's VRF timeout recovery mechanism is **sufficient** to prevent permanent epoch freeze. The design is intentionally permissionless to avoid centralization risk.

**Recommendations for operational excellence:**

1. **Monitor VRF timeout events** via `VrfRetryRequested` event emissions
2. **Run redundant crank bots** (multiple operators) to ensure retry attempts continue during extended outages
3. **Document Switchboard status page** in operational runbook for operator awareness
4. **Set Sentry alerts** for VRF duration > 300 seconds (indicates retry occurred)

**Against adding admin override:**

Adding an admin "skip VRF" instruction would:
- Introduce centralization (admin key compromise = full protocol control)
- Create regulatory risk (admin can manipulate tax rates)
- Violate immutability guarantee (post-authority-burn, no admin should exist)
- Reduce user trust (permissionless claim becomes false)

The current design correctly prioritizes **decentralization over convenience**.
