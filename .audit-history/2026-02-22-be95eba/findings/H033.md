# H033: Pool State Desync After Failed Carnage

**Status:** NOT EXPLOITABLE
**Severity:** INFORMATIONAL
**Category:** State Machine / Atomicity
**Impact:** NONE (Theoretical risk mitigated by Solana transaction model)
**CWE:** CWE-662 (Improper Synchronization)

---

## Summary

**Investigation Target:** Verify that Carnage execution is atomic and cannot leave pool state modified while `carnage_pending` remains `true` if any step fails.

**Conclusion:** The protocol is NOT vulnerable to pool state desync. Solana's transaction atomicity model guarantees that all state changes (including pool reserve modifications and flag updates) either commit together or roll back together. Carnage execution cannot partially succeed.

---

## Technical Analysis

### Carnage Execution Flow

Both atomic (`execute_carnage_atomic`) and fallback (`execute_carnage`) paths follow this sequence:

1. **Entry guard**: Read `carnage_pending` flag from `EpochState`
   - Atomic path: soft no-op if `false` (line 215)
   - Fallback path: hard constraint requiring `true` (line 55)

2. **Multi-step execution**:
   - **Step 1**: Dispose of held tokens (burn or sell via CPI to Tax→AMM)
   - **Step 2**: Wrap SOL to WSOL
   - **Step 3**: Buy target token (CPI to Tax→AMM)
   - **Step 4**: Verify slippage constraints

3. **State updates**:
   - Update `CarnageFundState` holdings (lines 448-462)
   - **Clear `carnage_pending = false`** (line 465 atomic, line 471 fallback)
   - Update statistics and emit event

### Critical Observation: Effects Happen AFTER Interactions

The code follows an **inverted CEI pattern**:

```rust
// execute_carnage_atomic.rs
execute_sell_swap(...)?;          // INTERACTION (line 310)
execute_buy_swap(...)?;           // INTERACTION (line 396)
// ... state updates happen here ...
epoch_state.carnage_pending = false;  // EFFECT (line 465)
```

**Normally this would be dangerous** — if the interaction fails, you want state already updated so reentrancy sees the "completed" state. However, this pattern is **safe in Solana** because:

1. **Transaction atomicity**: Solana's runtime guarantees that if ANY instruction in the call stack returns an error, ALL state changes are rolled back. Pool reserves modified by AMM swap CPIs and the `carnage_pending` flag clear happen in the same transaction — they commit or revert together.

2. **Reentrancy prevention**: The Anchor constraint `constraint = epoch_state.carnage_pending` (fallback path) checks at **account deserialization time** (before the handler executes). Even if the program could call itself during a CPI (it can't — Solana runtime prevents self-reentrancy), the constraint would fail on re-entry.

3. **AMM locked flag**: The AMM's `pool.locked` boolean (checked at deserialization, set before swaps, cleared after) provides defense-in-depth against concurrent modifications.

### Transaction Failure Scenarios

| Scenario | Outcome | State After Failure |
|----------|---------|---------------------|
| Burn CPI fails | Entire TX reverts | `carnage_pending` remains `true`, pool untouched |
| Sell swap fails | Entire TX reverts | `carnage_pending` remains `true`, pool untouched |
| Buy swap fails | Entire TX reverts | `carnage_pending` remains `true`, pool untouched |
| Slippage check fails | TX reverts at line 435-438 | `carnage_pending` remains `true`, pool untouched |
| State update fails (overflow) | TX reverts at line 454-461 | `carnage_pending` remains `true`, pool untouched |

**The problematic scenario does not exist**: There is no execution path where:
- Pool reserves are modified (swaps succeeded)
- BUT `carnage_pending` is not cleared (flag update failed)

If any step errors, the `?` operator propagates the error up the call stack. Solana's runtime catches the error and discards the entire transaction, including:
- All CPI state changes (pool reserves, token balances)
- All local state changes (`carnage_pending`, `held_amount`, statistics)

## Attack Vectors Considered

### 1. Partial Execution via Failed Slippage Check
**Hypothesis**: Buy swap succeeds but slippage check fails, leaving pool modified.

**Analysis**: The slippage check (lines 422-438 atomic, 429-446 fallback) happens AFTER the swap CPI but BEFORE state updates. If it fails:
1. `require!` macro throws `EpochError::CarnageSlippageExceeded`
2. Solana runtime catches error and rolls back the entire TX
3. Pool reserves modified by the swap CPI are reverted
4. `carnage_pending` flag clear never executes

**Verdict**: Not exploitable. Transaction atomicity prevents partial state.

### 2. Reentrancy During Multi-Step Execution
**Hypothesis**: Between sell and buy swaps, an attacker re-enters `execute_carnage` to cause double-execution.

**Analysis**:
1. **Solana runtime prevents self-reentrancy**: A program cannot CPI back to itself during instruction execution. The Epoch Program cannot call `execute_carnage` while already executing it.
2. **Anchor constraint blocks re-entry**: Even if reentrancy were possible, the fallback path's `constraint = epoch_state.carnage_pending` checks the flag at deserialization time (before handler runs). If the flag were cleared mid-execution, re-entry would fail the constraint.
3. **Atomic path no-op guard**: The atomic path uses `if !carnage_pending { return Ok() }` which would exit immediately if the flag were cleared.

**Verdict**: Not exploitable. Runtime prevents self-reentrancy, constraints provide defense-in-depth.

### 3. Concurrent Execution in Same Transaction
**Hypothesis**: Two `execute_carnage` instructions in the same transaction could race.

**Analysis**:
1. **AMM locked flag**: The AMM's `pool.locked` boolean is checked via Anchor constraint `constraint = !pool.locked` and set to `true` before any swap. A second swap in the same TX would fail at account validation.
2. **Single-threaded transaction execution**: Solana executes instructions sequentially, not concurrently.

**Verdict**: Not exploitable. AMM reentrancy guard prevents concurrent swaps.

### 4. State Corruption via Account Reload Failure
**Hypothesis**: `.reload()` calls fail, causing stale data reads.

**Analysis**:
1. All `.reload()` calls use `?` error propagation (lines 319, 345-346, 400-407 atomic; 327, 352-354, 408-415 fallback)
2. If reload fails (e.g., account closed unexpectedly), the `?` propagates the error and the entire TX reverts
3. No scenario where stale data is used after a failed reload

**Verdict**: Not exploitable. Reload failures cause transaction abort.

## False Positive Indicators

This finding was flagged as **STR-033** (Strategic Thread — pool state desync) based on multi-step execution pattern recognition. However, the concern assumes an Ethereum-style execution model where external calls can succeed while local state changes fail.

**Why the pattern triggered**:
- Multi-step CPI chain (burn/sell → buy)
- State updates (flag clear) happen after interactions
- Complex state machine with boolean flags

**Why it's a false positive**:
- Solana's transaction atomicity makes partial execution impossible
- All state changes (CPI results + local updates) commit or revert atomically
- The protocol correctly relies on this runtime guarantee

**Similar False Positives**:
- Any Solana program using inverted CEI with CPI
- Multi-step operations split across CPIs (staking deposit → reward distribution)
- State machines where flags are cleared after external calls

## Mitigation (None Required)

**Recommendation**: No code changes needed. The current implementation is secure.

**Documentation Enhancement**: Consider adding inline comments explaining the atomicity guarantee:

```rust
// State updates happen after CPIs (inverted CEI pattern).
// This is SAFE in Solana: if any CPI fails, the entire transaction
// reverts, including all pool reserve changes. Partial execution
// (modified pools + stale carnage_pending flag) cannot occur.
epoch_state.carnage_pending = false;
```

**Defensive Pattern (Optional)**: For defense-in-depth, the code already uses:
1. `.reload()` after every state-mutating CPI to prevent stale reads
2. Checked arithmetic with `.ok_or(EpochError::Overflow)?` to catch edge cases
3. Hard constraints on critical invariants (e.g., `carnage_pending` check in fallback)
