# H126: Staking User Account Creation Race Condition

**Strategy:** STR-126

**Determination:** SAFE — No exploitable race condition exists

**Severity:** INFORMATIONAL

**Confidence:** VERY HIGH (95%)

---

## Summary

The hypothesis posited that two concurrent transactions attempting to create a `UserStake` account for the same user could collide, with one overwriting the other's initial state and corrupting the reward checkpoint. Investigation confirms this is **not possible** due to:

1. **Anchor's `init_if_needed` discriminator validation** — Once created, subsequent loaders see the discriminator and load the account (not recreate it)
2. **Solana's atomic `SystemProgram::create_account`** — Multiple concurrent create attempts against the same account address fail serially; only one succeeds
3. **Solana's serialized transaction execution** — The runtime processes transactions sequentially by shard, preventing parallel mutations of the same account
4. **PDA uniqueness** — Each user's `UserStake` derives to a unique, deterministic address based on `[b"user_stake", user.key()]`

The protocol is **safe against this attack vector**.

### 2. How `init_if_needed` Works (Anchor Framework)

Anchor's `init_if_needed` macro performs the following sequence:

1. **Derivation**: Computes the PDA address using the specified seeds
2. **Account Existence Check**: Reads the account from the blockchain state
3. **Discriminator Validation**: Checks for the 8-byte Anchor discriminator
   - **If account exists with valid discriminator** → Load it (LOAD path)
   - **If account doesn't exist (zero-initialized)** → Create it via `SystemProgram::create_account` (CREATE path)
4. **Initialization**: If CREATE path taken, writes the discriminator and returns an empty account
5. **Handler Execution**: The handler receives the account (either newly created or previously existing)

**Key Insight**: The discriminator acts as a state machine flag. Once written during account creation, subsequent transactions see it and take the LOAD path.

### 3. Scenario: Two Concurrent Transactions from Same User

**Setup**: User Alice attempts to stake twice in the same block (TX1 and TX2).

#### Timeline (Solana Runtime Execution)

```
Block N, Slot S

Validator processes transactions serially:

1. TX1 (Alice stakes 1000 PROFIT):
   ├─ Anchor derives UserStake(alice) PDA address
   ├─ Reads blockchain state: Account doesn't exist yet
   ├─ SystemProgram::create_account succeeds ✓
   ├─ Anchor writes discriminator (8 bytes)
   ├─ Handler runs:
   │  ├─ is_new_user = (alice.owner == Pubkey::default()) = TRUE
   │  ├─ Initializes: owner=alice, rewards_per_token_paid=pool_value
   │  └─ Updates staked_balance += 1000
   └─ TX1 committed to ledger

2. TX2 (Alice stakes 500 PROFIT):
   ├─ Anchor derives UserStake(alice) PDA address (SAME ADDRESS)
   ├─ Reads blockchain state: Account exists with discriminator ✓
   ├─ Anchor loads existing account (LOAD path, not CREATE)
   ├─ Handler runs:
   │  ├─ is_new_user = (alice.owner == Pubkey::default())
   │  │              = (alice != Pubkey::default()) [set by TX1]
   │  │              = FALSE
   │  ├─ Skips initialization block (correct — already initialized)
   │  ├─ Calls update_rewards() with existing checkpoint
   │  │  → reward_delta = pool.rewards_per_token_stored - alice.rewards_per_token_paid
   │  │  → In same epoch: reward_delta ≈ 0 (checkpoint prevents flash-loan)
   │  └─ Updates staked_balance += 500 (now 1500 total)
   └─ TX2 committed to ledger
```

**Result**: Both transactions complete successfully. The second transaction correctly loads the account created by the first and applies its own stake update without re-initializing and overwriting state.

### 4. Why Concurrent Create Attempts Fail (Solana Guarantee)

If hypothetically both TX1 and TX2 attempted to create the account simultaneously:

**Solana's Serialized Runtime** (per solana-runtime-quirks.md and Solana consensus docs):
- Transactions are processed **sequentially** by the leader (single thread per shard)
- Account mutations are serialized via account-level locking
- Duplicate mutable accounts in a transaction are rejected (cannot have two instructions mutating the same account in one TX)

**SystemProgram::create_account Behavior**:
- Syscall 1 (TX1): Account doesn't exist → creates it → returns OK (0)
- Syscall 2 (TX2): Account now exists → returns `SystemError::AccountAlreadyInUse`
- TX2's entire transaction fails, reverts, or is dropped from the block

**Conclusion**: The second transaction cannot overwrite the first's state because:
- The first transaction's effects are committed and visible to the second
- The second transaction's create_account syscall fails atomically
- No partial state corruption is possible

### 5. Checkpoint State Integrity

The reward checkpoint (`rewards_per_token_paid`) is critical for the staking system's security.

**First Stake (TX1)**:
- Initializes `rewards_per_token_paid = pool.rewards_per_token_stored` (correct)
- This is the cumulative reward rate at the time of first stake
- Prevents the user from earning retroactive rewards from before they staked

**Second Stake (TX2, same epoch)**:
- Does NOT re-initialize (is_new_user = FALSE)
- Reuses the checkpoint from TX1
- Calls `update_rewards()`:
  ```rust
  let reward_delta = pool.rewards_per_token_stored - user.rewards_per_token_paid;
  // In same epoch: pool.rewards_per_token_stored hasn't changed
  // (It only updates at epoch boundaries via update_cumulative CPI)
  // Therefore: reward_delta ≈ 0
  ```
- Result: No spurious rewards earned; state is correct

**If TX2 were to incorrectly re-initialize the checkpoint**:
- Would set `rewards_per_token_paid = pool.rewards_per_token_stored` (current value)
- Would reset `first_stake_slot = clock.slot` (new value)
- Would lose the original first-stake timestamp
- But this **cannot happen** because the discriminator prevents re-initialization

### 6. Control Flow: is_new_user Detection

The handler uses a simple but effective detection mechanism:

```rust
let is_new_user = user.owner == Pubkey::default();
```

**Why this is safe**:
- Fresh Anchor accounts are zero-initialized
- Pubkey::default() is the zero address (11111...111111)
- On first stake, the handler immediately sets `owner = user.key()`
- On subsequent stakes, `owner` is never reset to default (no code path does this)
- Therefore: After first initialization, `is_new_user` is always FALSE

**Could an attacker inject a zero-owner account?**
- No: The PDA is derived deterministically. An attacker cannot create a second account with the same address.
- The PDA uniqueness guarantee prevents substitution.

### 7. Anchor Discriminator as Safety Valve

Anchor's discriminator provides defense-in-depth:

```
Account Layout (UserStake):
[8 bytes: discriminator] [32 bytes: owner] [8 bytes: staked_balance] ...
```

When Anchor loads an account with `Account<'info, UserStake>`:
1. Reads the first 8 bytes
2. Compares against expected discriminator for UserStake
3. If mismatch → deserialization fails with error
4. If match → proceeds with deserialization and returns the account

**What this prevents**:
- Account type confusion (loading a different struct type)
- Corruption of account data between invocations
- Loading a raw system account instead of a program-derived account

**In the race condition scenario**:
- TX1 creates the account and Anchor writes the discriminator
- TX2 reads the account and Anchor validates the discriminator
- If discriminator were somehow corrupted, TX2 would fail before executing the handler
- No state would be overwritten

## Impact

**Likelihood of Exploitation**: None. The attack is not feasible.

**Severity**: INFORMATIONAL — This is a non-issue.

**Why No Impact**:
1. Solana's atomicity guarantees prevent concurrent account creation
2. Anchor's discriminator prevents state corruption
3. The handler's `is_new_user` check is reliable
4. Even if the check failed, the second initialization would write the same values (idempotent)

## Recommendation

**No remediation required.** The protocol is safe against this attack vector.

**Optional Enhancement** (for code clarity):
The handler could add a redundant defensive check for documentation:

```rust
// Optional: Re-verify discriminator (Anchor already does this)
require!(user.discriminator == UserStake::DISCRIMINATOR, StakingError::InvalidAccount);

let is_new_user = user.owner == Pubkey::default();
if is_new_user {
    // Initialize fields
    user.owner = ctx.accounts.user.key();
    user.rewards_per_token_paid = pool.rewards_per_token_stored;
    // ...
}
```

However, this is redundant because:
- Anchor's `Account<'info, T>` struct already validates the discriminator before returning
- Adding manual checks increases compute costs without security benefit
- The current implementation is idiomatic Anchor

**Conclusion**: Leave the code as-is. It is correct and efficient.

## Appendix: Why This Attack Is Theoretically Possible in Other Contexts

Some blockchain systems (Ethereum, other non-serialized systems) might be vulnerable to this if:
1. Accounts could be created in parallel (Ethereum uses Merkle-tree state roots, allowing concurrent execution)
2. Create operations were not atomic (Ethereum uses nonces to serialize account creation)
3. No discriminator/type validation existed (Ethereum's contract code layout uses different mechanisms)

**Solana's serialized execution model eliminates this class of vulnerability entirely.**
