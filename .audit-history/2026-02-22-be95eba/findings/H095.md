# H095: AMM LP Fee Recipient is Pool Not Treasury

**Strategy:** STR-095: AMM Fee Recipient is Pool Not Treasury

**Determination:** Confirmed as designed and intentional.

**Severity:** Info

**Confidence:** High

---

## Summary

The 30-50 basis point LP fee collected by the AMM on every swap stays in the pool (increases reserve balances) rather than being extracted to the protocol treasury. This is the standard design for constant-product AMMs (e.g., Uniswap v2) where fees accrue to liquidity providers proportionally to their stake. In Dr Fraudsworth, since there are **no LP tokens and no depositor mechanism**, the "liquidity providers" are effectively the **protocol admin** (who seeds all pools with initial capital and owns the vaults via PDAs).

**Key Design Facts:**
- **SOL pool fee:** 100 bps (1.0%) — Constants.rs line 15
- **PROFIT pool fee:** 50 bps (0.5%) — Constants.rs line 20
- **Fee mechanism:** Deducted before swap output calculation (swap_sol_pool.rs lines 125-126)
- **Fee stays in pool:** `new_reserve_in = reserve_in + amount_in` (line 163-165) — the full `amount_in` (pre-fee) is added to the reserve, so the fee portion never leaves
- **No LP token mechanism:** No way to track depositor shares or claim fees
- **Protocol-owned liquidity:** All vaults are PDAs controlled by the AMM program

**This is correct for this architecture.** The alternative—extracting fees to a separate treasury account—would require LP token tracking (or equivalent share accounting) to distribute fees fairly. Since the protocol chose swap-only AMM with protocol-owned liquidity, fees staying in the pool is the natural consequence.

## Evidence

### 1. Fee Calculation (Pre-Swap)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/helpers/math.rs` lines 36-40

```rust
pub fn calculate_effective_input(amount_in: u64, fee_bps: u16) -> Option<u128> {
    let amount = amount_in as u128;
    let fee_factor = 10_000u128.checked_sub(fee_bps as u128)?;
    amount.checked_mul(fee_factor)?.checked_div(10_000)
}
```

The fee is subtracted from the input **before** the swap calculation. For example:
- Input: 1000 lamports
- SOL pool fee: 100 bps
- Effective input: 1000 * 9900 / 10000 = 990
- Fee amount: 1000 - 990 = 10 lamports

This fee amount (10 lamports) **never leaves the pool**.

### 2. Reserve Update (Fee Stays in Pool)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/swap_sol_pool.rs` lines 162-168

```rust
// 8. Compute new reserves (amount_in is pre-fee -- fee stays in pool)
let new_reserve_in = reserve_in
    .checked_add(amount_in)  // <-- full input (includes fee) added to reserve
    .ok_or(AmmError::Overflow)?;
let new_reserve_out = reserve_out
    .checked_sub(amount_out)
    .ok_or(AmmError::Overflow)?;
```

The key is that `new_reserve_in` includes the **full `amount_in`**, not just the effective input. The fee portion is never transferred out; it remains in the vault as part of the reserve.

**Example walk-through:**
- Before: reserve_a = 1M, reserve_b = 1M
- User swaps 1000 tokens (token A) to token B
- Fee calculated: 100 bps → 10 tokens (stays in pool)
- Effective input for swap: 990 tokens
- Swap output: 990 * 1M / (1M + 990) ≈ 989 tokens (token B)
- After: reserve_a = 1M + 1000 = 1,001,000
- After: reserve_b = 1M - 989 = 999,011
- **k before:** 1M * 1M = 1e12
- **k after:** 1,001,000 * 999,011 = 1,000,010,011,000
- **Fee effect:** k increased by 10,011,000 (from the fee and slippage)

The 10-token fee stayed in reserve_a, increasing the pool's total value.

### 5. Protocol-Owned Liquidity (Vaults are PDAs)

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/initialize_pool.rs` lines 59-123

Pool initialization transfers seed liquidity from the **admin's accounts** into **PDA-owned vault accounts**. The vaults are controlled by the AMM program, not by any LP token holders.

```rust
// Transfer token A from admin's source account into vault A.
if tp_a_key == anchor_spl::token_2022::ID {
    transfer_t22_checked(
        &ctx.accounts.token_program_a.to_account_info(),
        &ctx.accounts.source_a.to_account_info(),      // Admin's token account
        &ctx.accounts.mint_a.to_account_info(),

```

The admin provides all seed liquidity, and the vaults are PDAs (Pubkey::default() as authority is implicit in PDA signing). This means:
- Fees cannot be distributed to LPs (no LPs exist)
- Fees can only accrue to the protocol (in the form of increased reserve balances)
- The only way for the protocol to extract value is through admin withdrawal (not implemented in current AMM)

## Impact

**Positive Aspects:**
1. **Correct CPMM behavior**: Keeping fees in the pool is the standard design for constant-product AMMs. Uniswap v2, Orca, and most DEXes work this way.
2. **Simplified on-chain logic**: Without LP tokens or share tracking, the swap code is simpler and cheaper computationally.
3. **k-invariant benefit**: Fees increase k, which compounds to the protocol's benefit over time.

**Economic Consequences:**
1. **Protocol treasury receives no direct AMM fee revenue**: The 100-50 bps that would normally go to a treasury/governance fund instead stay in the pool. Over time, this increases pool reserves and improves the constant-product formula, benefiting all future swaps.
2. **No LP token staking/emissions possible**: Without LP shares, the protocol cannot incentivize liquidity provision with rewards (though this may be intentional—all liquidity is protocol-owned).
3. **Value extracted via pool growth, not fee distribution**: If the protocol ever needs to extract value from the AMM, it would be through admin withdrawal of vault tokens, which now include both the original seed liquidity and accumulated fees.

**Example: Accumulated Value Over Time**
- Initial: reserve_a = 1M, reserve_b = 1M, k = 1e12
- After 1000 swaps with 100 bps fee, ~1000 fee tokens accumulated
- New state: reserve_a ≈ 1M + 1000, reserve_b ≈ 1M (simplified), k ≈ 1.001e12
- If the protocol ever withdraws, it extracts 1000 extra tokens (the accumulated fees) on top of the original seed

## Recommendation

**Status:** No action required. This is correct design for a protocol-owned AMM.

**If the protocol later wants to:**

1. **Extract fees to a separate treasury account:**
   - Implement fee withdrawal instruction (admin-gated)
   - Requires tracking fee-to-date separately from reserve (add `accumulated_fee_a` and `accumulated_fee_b` fields to `PoolState`)
   - Alternative: track fee coefficient as a multiplier to avoid constant updates

2. **Support multiple liquidity providers with LP tokens:**
   - Add LP token mint to each pool (major redesign)
   - Track total LP shares, mint/burn LP tokens on deposit/withdraw
   - Distribute accumulated fees to LP token holders
   - This would require an `add_liquidity` and `remove_liquidity` instruction in addition to `swap`
   - Current codebase has no deposit/withdraw logic

3. **Allow the protocol to benefit from fee accumulation:**
   - Document that reserve increases compound to the protocol's benefit (already happening)
   - Implement an admin withdrawal instruction if value extraction is needed later
   - Track total fee revenue off-chain from SwapEvent logs for reporting

## Conclusion

The hypothesis is **confirmed and correct**. The 30-50 bps AMM fee stays in the pool (increases reserves) rather than being extracted to the protocol treasury. This is the correct and standard behavior for a constant-product AMM with protocol-owned liquidity and no LP token mechanism. The design is intentional and well-reasoned—it simplifies the on-chain logic, provides stronger security (no LP extraction attack surface), and still allows the protocol to benefit from fee revenue through reserve accumulation.
