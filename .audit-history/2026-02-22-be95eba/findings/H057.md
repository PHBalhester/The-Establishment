# H057 — Carnage Lock Slot Not Initialized in Genesis

**Status:** VERIFIED
**Severity:** MEDIUM
**Category:** State Management / Timing Logic
**Location:** `programs/epoch-program/src/instructions/initialize_epoch_state.rs`

---

## Summary

The `carnage_lock_slot` field in `EpochState` is not initialized in the `initialize_epoch_state` instruction. This field defaults to 0 (Anchor zero-initializes account memory), which breaks the atomic execution window protection for the first Carnage event.

---

## Vulnerability Details

### Root Cause

In `initialize_epoch_state.rs` (lines 60-76), the Carnage state initialization sets:
- `carnage_pending = false`
- `carnage_target = 0`
- `carnage_action = 0`
- `carnage_deadline_slot = 0`
- `last_carnage_epoch = 0`

But **`carnage_lock_slot` is never initialized**. Since Anchor zero-initializes account memory, this field defaults to 0.

### Attack Vector

The fallback execution path in `execute_carnage.rs` validates the lock window has expired:

```rust
// Line 224 in execute_carnage.rs
require!(
    clock.slot > ctx.accounts.epoch_state.carnage_lock_slot,
    EpochError::CarnageLockActive
);
```

**When carnage_lock_slot = 0:**
- Any slot > 0 passes this check
- The 50-slot atomic-only window is bypassed
- MEV bots can front-run the atomic execution during the first Carnage event

### Expected Behavior

When Carnage triggers in `consume_randomness.rs` (lines 280-287):

```rust
epoch_state.carnage_lock_slot = clock
    .slot
    .checked_add(CARNAGE_LOCK_SLOTS)  // 50 slots
    .ok_or(EpochError::Overflow)?;
```

This sets `carnage_lock_slot = current_slot + 50`, creating a window where only `execute_carnage_atomic` (bundled with `consume_randomness`) can execute.

After the lock expires (but before the 300-slot deadline), the fallback `execute_carnage` becomes callable.

### Actual Behavior

**First Carnage Event:**
- `carnage_lock_slot` starts at 0 (uninitialized)
- When Carnage triggers at slot N, `carnage_lock_slot` is set to N + 50
- But before the trigger, `carnage_lock_slot = 0`
- The check `clock.slot > 0` always passes (except at genesis slot 0)
- The atomic lock window is ineffective

**Subsequent Carnage Events:**
- After the first trigger, `carnage_lock_slot` is properly set
- The lock window works as intended

## Impact Assessment

**Severity: MEDIUM**

### Impact
- **MEV Front-Running:** During the first Carnage event, MEV bots can execute the fallback path before the atomic bundled transaction
- **Slippage Difference:** Fallback uses 1000 BPS slippage vs atomic's 500 BPS, potentially resulting in worse execution
- **One-Time Event:** Only affects the first Carnage trigger, not subsequent events

### Likelihood
- **Moderate:** The first Carnage event is guaranteed to occur at some point
- **Devnet Already Deployed:** Current devnet deployment may have already experienced the first Carnage

### Mitigating Factors
1. **Limited Window:** The issue only exists from protocol initialization until first Carnage trigger
2. **Same Execution Logic:** Both atomic and fallback paths use identical burn/sell/buy logic
3. **Slippage Still Applied:** Fallback has higher slippage tolerance (1000 BPS vs 500 BPS) but still has protection
4. **Permissionless Execution:** The fallback path is permissionless and incentivized via bounties

## Proof of Concept

### Scenario: First Carnage Event

1. **Protocol Initialization (Slot 1000):**
   - `initialize_epoch_state` runs
   - `carnage_lock_slot = 0` (zero-initialized, not explicitly set)
   - `carnage_deadline_slot = 0`
   - `carnage_pending = false`

2. **First Epoch Transition (Slot 5500):**
   - `trigger_epoch_transition` runs successfully
   - `vrf_pending = true`
   - Still no Carnage pending

3. **First Carnage Trigger (Slot 5502):**
   - `consume_randomness` runs at slot 5502
   - VRF triggers Carnage (byte 5 < 11)
   - Sets:
     - `carnage_pending = true`
     - `carnage_lock_slot = 5502 + 50 = 5552`
     - `carnage_deadline_slot = 5502 + 300 = 5802`

4. **MEV Bot Front-Run (Slot 5503):**
   - MEV bot calls `execute_carnage` (fallback path)
   - Check: `5503 > 0` ✓ PASSES (should FAIL during lock window)
   - Check: `5503 <= 5802` ✓ PASSES (within deadline)
   - **Carnage executes with 1000 BPS slippage instead of atomic's 500 BPS**

5. **Atomic Attempt Fails (Slot 5503):**
   - Crank bundles `reveal + consume_randomness + execute_carnage_atomic`
   - `execute_carnage_atomic` sees `carnage_pending = false` (already cleared)
   - No-ops gracefully (returns Ok())

## Recommendations

### Fix: Initialize carnage_lock_slot in Genesis

Add initialization in `initialize_epoch_state.rs`:

```rust
// Add after line 74
epoch_state.carnage_lock_slot = 0;
```

While this doesn't change the default value (Anchor already zero-initializes), it makes the intent explicit and ensures consistency with other Carnage fields.

### Alternative Fix: Use u64::MAX for Uninitialized State

Initialize to maximum value to ensure lock is "always active" until first trigger:

```rust
epoch_state.carnage_lock_slot = u64::MAX;
```

This prevents any premature fallback execution until Carnage actually triggers.

**Recommended approach:** Use `u64::MAX` to be defensive against edge cases.
