# H078: Carnage Buy and Sell Use the Same Pool (Not Dual Pools)

**Strategy:** STR-078
**Determination:** CONFIRMED NOT VULNERABLE
**Severity:** INFORMATIONAL
**Confidence:** HIGH

## Summary

The hypothesis proposes that Carnage buys the losing token in a SOL/LOSING pool while selling remainder in a LOSING/PROFIT pool — using two different pools. Investigation reveals this is **incorrect**: Carnage buy and sell operations use **the same pool** (either SOL/CRIME or SOL/FRAUD), not dual pools. This actually **strengthens** the protocol's economic security rather than weakening it.

## Evidence

### Pool Architecture (4 Total Pools)

From ARCHITECTURE.md and verified in code:
- **SOL/CRIME pool** (mixed: WSOL + CRIME Token-2022)
- **SOL/FRAUD pool** (mixed: WSOL + FRAUD Token-2022)
- **CRIME/PROFIT pool** (pure T22: CRIME Token-2022 + PROFIT Token-2022)
- **FRAUD/PROFIT pool** (pure T22: FRAUD Token-2022 + PROFIT Token-2022)

These are different pools (one is SOL/CRIME, other is SOL/FRAUD), but they are **both SOL pools**, not "dual-pool interaction" in the sense of different asset pair types.

## Impact Assessment

### Why This Design is Sound (Not a Vulnerability)

1. **Unified Liquidity Model**: Both Carnage buy and sell operations interact with the same faction's SOL pool. This creates **interdependent slippage**: selling CRIME into SOL/CRIME pool **improves** the price for the subsequent FRAUD buy (less CRIME supply competes with the buy), or if buying the same token, selling and buying use the same pool with natural sequencing (sell depresses price, buy happens at better rate).

2. **Price Impact Symmetry**: Unlike the hypothesis's "different pool" concern, operating on the same SOL pool means:
   - If Carnage sells FRAUD for SOL, the FRAUD pool's reserves shift: less FRAUD, more SOL
   - When Carnage then buys CRIME with that SOL, the CRIME pool is unaffected by the previous FRAUD sell
   - This is **correct behavior** for a VRF-random target mechanism

3. **No Reserve Separation Risk**: The hypothesis feared that dual pools would allow one operation to not impact the other. In reality, **all SOL liquidity is shared** across all SOL/* pools through the common WSOL token. Any SOL added/removed from one SOL pool affects the aggregate SOL supply available to the system.

4. **Slippage Floor Protection**: Both operations are protected by the 85% atomic slippage floor, calculated using **pre-swap reserves** (line 422-439). This prevents extreme manipulation regardless of which pool is used.

## Economic Implication

The fact that Carnage uses **the same SOL pool for both sell and buy** creates a natural **price stabilization mechanism**:

1. If Carnage holds CRIME and target flips to FRAUD, it must sell CRIME (at whatever price the SOL/CRIME pool offers) and buy FRAUD
2. The CRIME sell increases SOL in the SOL/CRIME pool, making CRIME cheaper for the next epoch's regular traders
3. The FRAUD buy reduces SOL in the SOL/FRAUD pool, making FRAUD more expensive

This creates a **rebalancing pressure** that is economically sound, not a vulnerability.

## Recommendation

**No action required.** The single-pool design is correct and well-implemented. The hypothesis was based on an incorrect assumption about the pool topology. Carnage correctly:
1. Selects pools based on token type (held for sell, target for buy)
2. Both operate on SOL pools (not PROFIT pools)
3. Executes with consistent 85% slippage protection
4. Shares liquidity across all SOL pools via the WSOL intermediary

The code correctly enforces this pattern via `execute_sell_swap()` and `execute_buy_swap()` helper functions that take token type as parameter and unconditionally select the appropriate SOL pool.
