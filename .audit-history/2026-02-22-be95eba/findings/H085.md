# H085: Transfer Hook Whitelist Check Validation

**Strategy:** STR-085

**Determination:** NOT VULNERABLE

**Severity:** N/A (Hypothesis rejected)

**Confidence:** HIGH

---

## Summary

STR-085 hypothesized that the transfer hook's whitelist validation could be bypassed if a non-existent whitelist PDA is passed, with the assumption that Anchor would deserialize it as zeroed data. Investigation confirms the hook uses an **existence-based check**, not deserialization-based validation, eliminating this attack vector.

---

## Evidence

### Hook Implementation Pattern

The `is_whitelisted()` function in `transfer_hook.rs` (lines 166-179) validates whitelist PDAs via:

```rust
fn is_whitelisted(whitelist_pda: &AccountInfo, token_account: &Pubkey) -> bool {
    // Check 1: Account must have data (exists)
    if whitelist_pda.data_is_empty() {
        return false;
    }

    // Check 2: Verify PDA derivation (prevents spoofed accounts)
    let (expected_pda, _bump) = Pubkey::find_program_address(
        &[WhitelistEntry::SEED_PREFIX, token_account.as_ref()],
        &crate::ID
    );

    whitelist_pda.key() == expected_pda
}
```

**Key Security Properties:**

1. **No Deserialization:** The function never deserializes the `WhitelistEntry` struct. Account data is not read or validated for structure. This eliminates the "zeroed data deserialization" attack vector.

2. **Existence-Based Check:** The function returns `false` immediately if `data_is_empty()` is true. Non-existent PDAs have empty data and fail the first check.

3. **Address Derivation Verification:** Even if an account has non-empty data, the function verifies its public key matches the expected PDA derivation. This prevents substituting a different account.

4. **UncheckedAccount Pattern:** The whitelist PDAs are passed as `UncheckedAccount<'info>` in the accounts struct (lines 49-54 of transfer_hook.rs), meaning Anchor does NOT attempt automatic deserialization. The hook controls all validation.

### Account Structure

```rust
// From transfer_hook.rs struct definition
pub struct TransferHook<'info> {
    // ... other accounts ...

    /// CHECK: Existence and derivation checked in handler
    pub whitelist_source: UncheckedAccount<'info>,

    /// CHECK: Existence and derivation checked in handler
    pub whitelist_destination: UncheckedAccount<'info>,
}
```

The `UncheckedAccount` variant means:
- Anchor does NOT deserialize the account
- Anchor does NOT validate the account's discriminator
- Anchor does NOT validate the account's owner
- Only the handler's manual checks apply

### Test Coverage

The test file (`programs/transfer-hook/tests/test_transfer_hook.rs`) documents that whitelist validation covers:
- SECU-04: "Check whitelist PDAs with derivation verification"
- Requirements: "WHTE-06, WHTE-07, SECU-01, SECU-02, SECU-03, SECU-04"

## Why the Hypothesis Fails

**STR-085 Assumption:** "Anchor might deserialize [the whitelist PDA] as zeroed data rather than erroring"

**Reality:** Anchor does not deserialize the whitelist PDAs at all. The accounts are:
- Declared as `UncheckedAccount<'info>` (not `Account<'info, WhitelistEntry>`)
- Explicitly documented with `/// CHECK:` comments indicating manual validation in the handler
- Never accessed via `.load()`, `.try_borrow_data()`, or any Anchor deserialization method

The hook uses a **cryptographic commitment pattern**: whitelist existence is proven by:
1. Non-empty account data (implies account was created and funded)
2. Public key matches the derived PDA (proves the account is the specific whitelist for this token)

A zero-initialized account (address all-zeros) would fail the derivation check. A non-existent account would fail the `data_is_empty()` check.

---

## Attack Surface Analysis

**Could a non-existent PDA be exploited?**

No. The only way to pass the `is_whitelisted()` check is:

1. **Account exists** (`!data_is_empty()`)
2. **Account's public key matches the derived PDA** (`whitelist_pda.key() == expected_pda`)

An attacker cannot forge condition 2 without:
- Knowing the full PDA derivation (requires `WhitelistEntry::SEED_PREFIX` and the exact token_account pubkey)
- Controlling a private key that derives to that PDA (computationally infeasible without owning the program)

---

## Impact

None. The whitelist check is cryptographically sound.

---

## Recommendation

**No action required.** The transfer hook's whitelist validation is correctly implemented using an existence-based cryptographic commitment pattern. The use of `UncheckedAccount` is justified and safe because:

1. All validation is performed manually in the handler
2. The handler does not rely on deserialization
3. The account's public key is the only security-critical data point, and it's validated via PDA derivation

The implementation is a **secure pattern** (SP-027: Cryptographic Commitment Pattern).

## Conclusion

STR-085's hypothesis was based on the assumption that Anchor would deserialize non-existent PDAs as zeroed data. The transfer hook bypasses this entirely by using existence-based validation without deserialization. The whitelist check is **SECURE**.
