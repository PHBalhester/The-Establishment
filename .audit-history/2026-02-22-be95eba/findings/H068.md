# H068: Carnage Signer Bump Not Stored
**Strategy:** STR-068
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis posits that if the `carnage_signer` PDA bump is derived at runtime but not stored, a different bump could be used in re-derivation, yielding a different PDA that could bypass access control in `Tax::swap_exempt`.

**Finding:** The protocol is NOT VULNERABLE to this attack. The `carnage_signer` bump is:
1. **Always derived from `ctx.bumps.carnage_signer`** — obtained from Anchor's account deserialization
2. **Never stored in any state struct** — no cached copy that could become stale or incorrect
3. **Always canonical** — Anchor's constraint validation ensures only the correct bump passes validation
4. **Used consistently across all CPI calls** — same bump passed to every `invoke_signed` in both `execute_carnage_atomic` and `execute_carnage`

## Attack Scenario Analysis

### Hypothetical Attack: Use a Different Bump

**Attacker's goal:** Call `Tax::swap_exempt` as if authorized by Epoch Program.

**Attempt 1: Hardcode a different bump**
- Call `execute_carnage_atomic` with `carnage_signer` account = PDA derived with bump 255 (invalid)
- Tax Program's `swap_exempt` constraint validates `seeds::program = epoch_program_id()`
- Anchor derives the canonical bump for this seed/program combo (not 255)
- Constraint validation fails ❌ TX rejected

**Attempt 2: Re-derive the bump at a different slot**
- PDAs are deterministic: same seeds + same program = same address + same canonical bump
- Slot number doesn't affect PDA derivation
- Any re-derivation of `(b"carnage_signer", epoch_program_id)` yields the same canonical bump ❌

**Attempt 3: Exploit a "stale bump" from stored state**
- Epoch Program doesn't store `carnage_signer_bump` anywhere
- Every call to `execute_carnage_atomic` fetches `ctx.bumps.carnage_signer` fresh from Anchor
- No stale state vector ❌

## Why Anchor's `ctx.bumps` Is Always Canonical

Anchor's account derivation process:
1. When deserializing an account with `seeds = [X], bump` in constraints:
   - Anchor calls `Pubkey::find_program_address(&[X], &program_id)`
   - This function returns the **canonical** bump (the first bump that produces a valid PDA)
   - Solana's PDA spec ensures there is exactly one canonical bump per seed+program pair
2. Anchor validates that the passed-in account address matches the derived PDA
3. If validation passes, `ctx.bumps.carnage_signer` contains the canonical bump
4. If validation fails (wrong bump provided), the transaction is rejected before handler logic runs

Thus, `ctx.bumps` is cryptographically guaranteed to be canonical.

## Impact

**None.** The vulnerability does not exist. The protocol correctly uses Anchor's constraint system to enforce PDA derivation at the account level, ensuring the canonical bump is always used.

## Recommendation

No action required. The implementation is correct. This is a good example of:
- Proper use of Anchor's built-in PDA derivation (never manual `find_program_address`)
- Cross-program validation via `seeds::program` constraints
- No caching of bumps in state (avoid stale data)

Consider documenting this pattern in a security runbook for future developers as a best-practice reference.
