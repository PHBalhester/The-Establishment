**Strategy:** STR-116: Carnage Fund WSOL Not Synced

**Determination:** NOT A FINDING — Hypothesis rejected. The Carnage WSOL account is properly synchronized before being used in swaps.

**Severity:** N/A

## Summary

The hypothesis posited that the Carnage Fund's WSOL account (`carnage_wsol`) might not be synchronized via `syncNative`, causing balance reads to return zero even though SOL was deposited. Investigation reveals:

1. **WSOL wrapping is explicitly called** with `wrap_sol_to_wsol()` before every buy swap operation
2. **syncNative is invoked inside wrap_sol_to_wsol()** immediately after native SOL transfer
3. **Execution flow ensures sync before read**: The WSOL balance is only read AFTER wrap_sol_to_wsol() returns successfully
4. **Multiple paths all sync**: BuyOnly, Burn+Buy, and Sell+Buy paths all follow the same wrapping procedure

### 1. Wrap Function Explicitly Calls SyncNative

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/epoch-program/src/instructions/execute_carnage_atomic.rs:646-694`

```rust
fn wrap_sol_to_wsol<'info>(
    ctx: &Context<'_, '_, 'info, 'info, ExecuteCarnageAtomic<'info>>,
    amount: u64,
    sol_vault_bump: u8,
) -> Result<()> {
    // ... sol_vault_seeds setup ...

    // Step 1: Transfer native SOL from sol_vault PDA to carnage_wsol token account
    let transfer_ix = anchor_lang::solana_program::system_instruction::transfer(
        &ctx.accounts.sol_vault.key(),
        &ctx.accounts.carnage_wsol.key(),
        amount,
    );

    invoke_signed(
        &transfer_ix,
        &[...],
        &[sol_vault_seeds],
    )?;

    // Step 2: SyncNative — UPDATE WSOL TOKEN BALANCE TO MATCH LAMPORT BALANCE
    let sync_native_ix = Instruction {
        program_id: ctx.accounts.token_program_a.key(),
        accounts: vec![
            AccountMeta::new(ctx.accounts.carnage_wsol.key(), false),
        ],
        data: vec![17u8], // SyncNative instruction discriminator
    };

    invoke_signed(
        &sync_native_ix,
        &[
            ctx.accounts.carnage_wsol.to_account_info(),
            ctx.accounts.token_program_a.to_account_info(),
        ],
        &[], // SyncNative is permissionless
    )?;

    msg!("Wrapped {} lamports SOL -> WSOL in carnage_wsol", amount);
    Ok(())
}
```

**Key observation:** The function performs:
1. `system_program::transfer` — adds lamports to the WSOL token account
2. `spl_token::sync_native` — updates the SPL token balance to reflect those lamports

## Impact

**No impact — the hypothesis is refuted.** The code correctly synchronizes the WSOL account before using it in swaps:
- Every deposit of SOL is followed by an immediate `syncNative` call
- No path reads the WSOL balance without first syncing

## Root Cause Analysis

The hypothesis may have arisen from:
1. **General WSOL patterns**: In typical WSOL workflows, forgetting `syncNative` is a common bug
2. **Account balance confusion**: Native SOL (`lamports()`) and SPL token balance (`amount`) are separate fields
3. **Code visibility**: The sync call is buried inside the `wrap_sol_to_wsol()` helper, not visible at the call site
