# STR-030: Swap Authority PDA Derivation Mismatch

**Status:** CLEAR
**Severity:** N/A (No vulnerability found)
**Subsystem:** Cross-Program CPI (Tax Program ↔ AMM)
**Threat Model Reference:** STR-030
**Audit Date:** 2026-02-22

---

## Summary

**Investigation Target:** Verify that the `swap_authority` PDA derivation is identical between Tax Program (derivation) and AMM (validation) to prevent CPI authorization bypass.

**Finding:** The `swap_authority` PDA derivation is **correctly implemented and fully consistent** across both programs. The seeds and program IDs match exactly, preventing any possibility of PDA mismatch or unauthorized access.

---

### 4. Attack Vector Analysis

**Hypothetical Attack Scenario (STR-030 Threat Model):**

An attacker tries to bypass AMM access control by deriving a PDA from a different program ID:

```rust
// Attacker's malicious program (ID: "EVIL...")
let fake_swap_authority = Pubkey::find_program_address(
    &[b"swap_authority"],
    &EVIL_PROGRAM_ID  // Different program ID
);
// Result: PDA_Y ≠ PDA_X
```

**Defense Mechanism:**

The AMM's `seeds::program = TAX_PROGRAM_ID` constraint validates:
1. The PDA seeds are `[b"swap_authority"]` ✅
2. The PDA was derived from `TAX_PROGRAM_ID` ✅
3. The account has the `Signer` flag set (can only be set by the deriving program during CPI) ✅

**Failure Point:**
If `EVIL_PROGRAM_ID ≠ TAX_PROGRAM_ID`, then `PDA_Y ≠ PDA_X`, and the AMM rejects the transaction at deserialization time with:
```
Error: ConstraintSeeds (seeds constraint violated)
```

**Result:** ✅ **Attack prevented by Anchor's PDA validation framework**

### 5. Code Quality Observations

**Strengths:**

1. **Explicit Documentation:**
   - Tax Program constants include comment: *"Must match AMM's SWAP_AUTHORITY_SEED for CPI compatibility"* (`constants.rs:4`)
   - AMM includes detailed comment explaining the `seeds::program` constraint (`swap_sol_pool.rs:362-366`)

2. **Defense-in-Depth:**
   - AMM uses `Signer<'info>` type (runtime signature validation) **AND** `seeds::program` constraint (program ownership validation)
   - This double-checks both that the account signed the TX and that it was derived from the correct program

## Conclusion

**STR-030 Status:** **CLEAR**

The `swap_authority` PDA derivation is **correctly implemented** with:
- ✅ Identical seed strings across both programs (`b"swap_authority"`)
- ✅ Matching program IDs (Tax Program `crate::ID` == AMM `TAX_PROGRAM_ID`)
- ✅ Proper validation via `seeds::program` constraint in AMM
- ✅ Defense-in-depth with both `Signer` type and PDA validation
- ✅ Comprehensive test coverage including negative access control tests
- ✅ Explicit documentation of cross-program dependency

**No vulnerability exists.** The AMM can only be called by the legitimate Tax Program, and no PDA derivation mismatch is possible.
