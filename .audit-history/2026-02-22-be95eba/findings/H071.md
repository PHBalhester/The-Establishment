# H071: AMM Swap With Same Input/Output Mint

**Strategy:** STR-071
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis proposed that swapping token A for token A (same mint both sides) might pass validation in the AMM swap handlers and corrupt reserves via the constant-product formula.

**Result:** The protocol **prevents same-mint swaps at the pool initialization level**, making this vulnerability impossible. A pool cannot be created with identical input and output mints.

## Evidence

### 1. Pool Initialization Validation (initialize_pool.rs:40-44)

The `initialize_pool` instruction validates that mint_a and mint_b are distinct:

```rust
// Duplicate mints would create a nonsensical self-pairing pool.
require!(
    ctx.accounts.mint_a.key() != ctx.accounts.mint_b.key(),
    AmmError::DuplicateMints
);
```

This check occurs **before** any pool state is created or vaults are initialized. The error `AmmError::DuplicateMints` is returned immediately.

### 2. Canonical Ordering Constraint (initialize_pool.rs:35-38)

Additionally, pools enforce canonical mint ordering:

```rust
// Canonical ordering: mint_a must be the "smaller" pubkey.
// This guarantees exactly one PDA per unordered mint pair.
require!(
    ctx.accounts.mint_a.key() < ctx.accounts.mint_b.key(),
    AmmError::MintsNotCanonicallyOrdered
);
```

If mint_a == mint_b, the first check catches this. If somehow the first check failed, the second check would also fail because no pubkey is strictly less than itself (`x < x` is always false).

### 3. PDA Derivation with Mint Pair (initialize_pool.rs:216-223)

The pool PDA is derived from the canonical mint pair:

```rust
seeds = [POOL_SEED, mint_a.key().as_ref(), mint_b.key().as_ref()],
```

With identical mints, this would derive a PDA: `[b"pool", mint, mint]`. The constraint check prevents reaching this point.

### 4. Test Coverage

The test suite validates this protection in `test_pool_initialization.rs:1172-1206`:

```rust
fn test_duplicate_mints_rejected() {
    // ...
    let ix = build_initialize_pool_ix(
        // ...
        &ctx.mint_a,
        &ctx.mint_a,  // SAME mint as mint_a
        // ...
    );

    let result = ctx.svm.send_transaction(tx);
    assert!(result.is_err(),
        "Duplicate mints should be rejected");
}
```

This test confirms that attempting to create a pool with duplicate mints fails.

### 5. Swap Handler Assumptions

The swap handlers (`swap_sol_pool.rs` and `swap_profit_pool.rs`) assume they operate on valid pools created by `initialize_pool`. They do not re-validate that mint_a != mint_b at swap time. This is safe because:

1. Invalid pools cannot be created
2. The swap handlers are CPI-only (require `swap_authority` PDA from Tax Program)
3. The pool PDA seed derivation with canonical mints prevents any workarounds

## Impact

**No exploitable impact.** Same-mint swaps are impossible by design.

- **Reserve Corruption:** Cannot occur because same-mint pools cannot exist
- **k-Invariant Violation:** Moot; no pool exists to violate it
- **Dust Accumulation:** Not applicable

## Recommendation

**No action required.** The protection is already in place and tested.

**Rationale:** The protocol correctly prevents this class of attack at the lowest level (pool creation) rather than at the swap handler level. This is defense-in-depth best practice: fail fast at the data structure initialization stage.

**Defense-in-depth note:** While not required, a re-validation at swap time (`require!(pool.mint_a != pool.mint_b)`) would provide an additional safety layer, but it is not necessary given the pool creation constraints.

## Forensic Path Taken

1. **Hypothesis:** Swap same token for itself with AtoB direction
   - Potential outcome: `reserve_a` increases (input), `reserve_b` decreases (output from same reserve)
   - k-invariant: `(reserve_a + input) * (reserve_a - output) >= reserve_a^2`?

2. **Investigation Step 1:** Search for swap validation logic
   - Found: `swap_sol_pool.rs` and `swap_profit_pool.rs` handlers
   - Neither validates mint_a != mint_b at swap time

3. **Investigation Step 2:** Trace backwards to pool creation
   - Found: `initialize_pool.rs` with explicit `require!(mint_a != mint_b)`
   - This is the control point that prevents the vulnerability

4. **Investigation Step 3:** Verify test coverage
   - Found: `test_duplicate_mints_rejected()` confirms rejection
   - Test uses SVM (Solana Virtual Machine) to confirm on-chain behavior

5. **Conclusion:** Vulnerability prevented at pool initialization, not at swap execution
