# H046: Epoch Transition During Active Swap - No Race Condition

**Strategy:** STR-046
**Determination:** NOT VULNERABLE
**Severity:** INFO
**Confidence:** HIGH

## Summary

The hypothesis that epoch transitions create unexpected tax rate changes for in-flight swaps is **NOT a vulnerability**. The system behaves correctly and safely:

1. **Tax rates are read atomically** at the start of each swap instruction
2. **Users are protected by slippage checks** (`minimum_output`)
3. **Solana's sequential transaction processing** ensures deterministic ordering
4. **Tax Program never commits to future rates** - it always reads current state

This is expected blockchain behavior, not a race condition. Users submitting swaps near epoch boundaries accept the risk that rates may change, but their slippage protection (`minimum_output`) guards them against adverse outcomes.

## Evidence

### 1. Tax Rate Reading is Atomic (Swap Instructions)

Both `swap_sol_buy.rs` and `swap_sol_sell.rs` follow the same pattern:

**swap_sol_buy.rs (lines 67-78):**
```rust
// Deserialize EpochState data.
let epoch_state = {
    let data = ctx.accounts.epoch_state.try_borrow_data()?;
    let mut data_slice: &[u8] = &data;
    EpochState::try_deserialize(&mut data_slice)
        .map_err(|_| error!(TaxError::InvalidEpochState))?
};

// Validate EpochState is initialized (defense-in-depth).
require!(epoch_state.initialized, TaxError::InvalidEpochState);

// Get the appropriate tax rate (is_buy = true for buy direction).
let tax_bps = epoch_state.get_tax_bps(is_crime, true);
```

**Key observations:**
- Tax rate is read **once** at instruction start (line 78)
- The value is **stored in a local variable** (`tax_bps`)
- All subsequent calculations use the cached `tax_bps` value
- No re-reading of EpochState during instruction execution

**swap_sol_sell.rs (lines 82-93):**
```rust
let epoch_state = {
    let data = ctx.accounts.epoch_state.try_borrow_data()?;
    let mut data_slice: &[u8] = &data;
    EpochState::try_deserialize(&mut data_slice)
        .map_err(|_| error!(TaxError::InvalidEpochState))?
};

require!(epoch_state.initialized, TaxError::InvalidEpochState);

// Get the appropriate tax rate (is_buy = false for sell direction).
let tax_bps = epoch_state.get_tax_bps(is_crime, false);
```

Identical pattern - atomic read, no re-checks.

### 2. Tax Rates Are Written Atomically (Epoch Program)

**consume_randomness.rs (lines 196-207):**
```rust
// === 6. Update EpochState with new tax configuration ===
epoch_state.cheap_side = tax_config.cheap_side.to_u8();
epoch_state.low_tax_bps = tax_config.low_tax_bps;
epoch_state.high_tax_bps = tax_config.high_tax_bps;
epoch_state.crime_buy_tax_bps = tax_config.crime_buy_tax_bps;
epoch_state.crime_sell_tax_bps = tax_config.crime_sell_tax_bps;
epoch_state.fraud_buy_tax_bps = tax_config.fraud_buy_tax_bps;
epoch_state.fraud_sell_tax_bps = tax_config.fraud_sell_tax_bps;

// === 7. Clear VRF pending state ===
epoch_state.vrf_pending = false;
epoch_state.taxes_confirmed = true;
```

Tax rates are updated **all at once** within a single instruction. There's no intermediate state where some rates are old and some are new.

**trigger_epoch_transition.rs (lines 176-182):**
```rust
// === 4. Advance epoch number ===
epoch_state.current_epoch = expected_epoch;
epoch_state.epoch_start_slot = epoch_start_slot(expected_epoch, epoch_state.genesis_slot);

// === 5. Set VRF pending state ===
epoch_state.vrf_request_slot = clock.slot;
epoch_state.vrf_pending = true;
epoch_state.taxes_confirmed = false;
```

**Critical finding:** `trigger_epoch_transition` does **NOT** change tax rates. It only:
- Advances the epoch counter
- Sets `vrf_pending = true`
- Sets `taxes_confirmed = false`

Tax rates remain **unchanged** until `consume_randomness` is called (typically 3+ slots later).

### 3. Transaction Ordering Guarantees

Solana processes transactions **sequentially** within a slot:
- If `trigger_epoch_transition` lands in slot N
- And a user swap lands in slot N
- The transactions execute in arrival order (deterministic)
- Each transaction sees the state **as modified by all prior transactions** in that slot

This is not a race condition - it's deterministic state machine execution.

### 4. User Protection Via Slippage Checks

**swap_sol_buy.rs (lines 108-111):**
```rust
require!(
    minimum_output >= output_floor,
    TaxError::MinimumOutputFloorViolation
);
```

**swap_sol_sell.rs (line 245):**
```rust
require!(net_output >= minimum_output, TaxError::SlippageExceeded);
```

Users specify `minimum_output` in their swap transaction. If the swap executes after an epoch transition with higher taxes:
- Tax calculation uses the **new (higher) rate**
- User receives **less output** than expected
- Slippage check **fails** (`net_output < minimum_output`)
- Transaction **reverts** atomically

User loses nothing - the swap simply fails to execute.

## Root Cause / Analysis

This is **not a bug** - it's expected behavior in a dynamic-rate system:

1. **No Time-of-Check-Time-of-Use (TOCTOU) gap**: The swap instruction reads tax rates once at start and uses that value consistently throughout. There's no vulnerability window.

2. **No stale data**: Solana's account model ensures all reads within an instruction see consistent state. Once a transaction begins executing, the account data it sees cannot be modified by other transactions until it completes.

3. **Two-phase epoch transition**: The protocol wisely separates epoch advancement (`trigger_epoch_transition`) from tax rate changes (`consume_randomness`). This gives a 3-slot buffer between "epoch number changed" and "tax rates changed."

4. **Users control their own risk**: By setting `minimum_output`, users decide their acceptable slippage. If they want protection against epoch transitions, they set tighter slippage bounds.

## Impact Assessment

**No user funds at risk.**

Worst case scenario:
- User submits swap with 5% slippage tolerance
- Epoch transition occurs before their swap executes
- New tax rate is 2% higher than when they built the transaction
- Swap uses new rate, produces less output
- Slippage check fails, transaction reverts
- User retains original tokens (pays only transaction fee)

**This is identical to any AMM slippage scenario** (price movement, front-running, etc.). The protocol does not introduce new attack surface beyond standard DeFi risks.

## Remediation

**No remediation required.** The system is working as designed.

### Optional Improvements for User Experience (not security fixes):

1. **Frontend Warning**: Display a warning when submitting swaps within 100 slots of expected epoch boundary:
   ```javascript
   const nextBoundary = genesisSlot + ((currentEpoch + 1) * SLOTS_PER_EPOCH);
   const slotsUntilBoundary = nextBoundary - currentSlot;
   if (slotsUntilBoundary < 100) {
     warn("Epoch transition imminent - consider tighter slippage or wait");
   }
   ```

2. **SDK Helper**: Provide a method to check if `taxes_confirmed === false` (VRF pending state):
   ```typescript
   async function shouldWaitForEpoch(epochState: EpochState): Promise<boolean> {
     return !epochState.taxesConfirmed; // Wait if rates uncertain
   }
   ```

3. **Documentation**: Clarify in user docs that:
   - Tax rates can change between transaction submission and execution
   - Slippage protection (`minimum_output`) guards against adverse rate changes
   - For maximum certainty, avoid swaps when `taxes_confirmed = false`
