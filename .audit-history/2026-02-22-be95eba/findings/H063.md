# H063: Hook Initialize Re-callable / Initialization Front-Running

**Strategy:** STR-063
**Determination:** POTENTIAL
**Severity:** MEDIUM
**Confidence:** HIGH

## Summary

The Transfer Hook program's `initialize_authority` instruction is not re-callable due to Anchor's `init` constraint, but it is **front-runnable**. Any account can call the instruction before the legitimate deployer, and the first caller becomes the whitelist authority permanently. This is a variant of EP-075/EP-076 (Double Initialization / Front-Runnable Initialization). The instruction accepts any `Signer<'info>` without verifying the caller holds the program's upgrade authority. While re-initialization is prevented by the `init` constraint (which creates the account with a discriminator), the initialization itself is unguarded.

## Root Cause Analysis

The `init` constraint in Anchor prevents **re-initialization** of the same PDA, but does NOT prevent front-running during the **initial** deployment. The constraint ensures that if the account already exists with a discriminator, the `init` will fail. However, during deployment, if any attacker transaction is included in the same block BEFORE the legitimate initializer, that attacker becomes the authority.

**Comparison to Secure Pattern (AMM's initialize_admin.rs):**
The AMM program correctly implements initialization protection:

```rust
// AMM's initialize_admin.rs (SECURE PATTERN)
#[account(
    constraint = program_data.upgrade_authority_address == Some(authority.key())
)]
pub program_data: Account<'info, ProgramData>,
```

This validates that only the program's upgrade authority can initialize. The Transfer Hook program does NOT include this check.

## Impact

**Exploit Scenario:**
1. Attacker monitors the blockchain for deployment transactions of Transfer Hook program
2. Attacker constructs `initialize_authority` transaction with their own signer key
3. Attacker front-runs the legitimate deployer's initialize call in the same block
4. Attacker becomes the permanent whitelist authority for CRIME/FRAUD/PROFIT tokens
5. Attacker can then:
   - Refuse to whitelist protocol vaults → Complete DoS of swaps/transfers
   - Whitelist only attacker-controlled accounts → Allow unauthorized token movement
   - Remove legitimate vault whitelist entries → Freeze transfers

**Severity Justification:**
- **MEDIUM** (not HIGH) because Anchor's `init` prevents immediate re-exploitation after legitimate initialization
- **MEDIUM** (not LOW) because whitelist authority controls all CRIME/FRAUD/PROFIT transfers, which are critical protocol infrastructure
- The attack window is limited to deployment block, but deployment is a high-risk event (not a continuous vulnerability)
- Mitigating factor: Modern deployment practices typically bundle initialization in the same atomic transaction as deployment, reducing attack window

## Recommendation

Add upgrade authority verification to the initialization instruction, following the pattern used by the AMM program:

```rust
// Transfer Hook: initialize_authority.rs (PROPOSED FIX)
#[derive(Accounts)]
pub struct InitializeAuthority<'info> {
    #[account(mut)]
    pub signer: Signer<'info>,

    /// NEW: Verify signer is the program's upgrade authority
    #[account(
        constraint = program_data.upgrade_authority_address == Some(signer.key())
    )]
    pub program_data: Account<'info, ProgramData>,

    #[account(
        init,
        payer = signer,
        space = 8 + WhitelistAuthority::INIT_SPACE,
        seeds = [WhitelistAuthority::SEED],
        bump
    )]
    pub whitelist_authority: Account<'info, WhitelistAuthority>,

    pub system_program: Program<'info, System>,
}
```

**Alternative Mitigation (if upgrade_authority is not available):**
Ensure deployment script initializes the authority in the same transaction as the program deployment, preventing any mempool-based front-running.
