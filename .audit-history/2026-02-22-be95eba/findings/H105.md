# H105: No Swap Direction Detection Bug — AMM Direction Handling Verified Correct

**Strategy:** STR-105: AMM Swap Direction Detection Bug Hypothesis
**Determination:** HYPOTHESIS REJECTED (No Finding)
**Severity:** N/A
**Confidence:** Very High (95%)

---

## Summary

Investigation of the hypothesis "AMM determines swap direction based on which account is input, and if direction detection is wrong, reserves are updated backwards" yielded **NO VULNERABILITY**. The AMM's swap direction handling is explicit, caller-supplied, and correctly implemented across all swap paths (SOL buy, SOL sell, PROFIT buy, PROFIT sell). Direction is not inferred from account order; instead, callers pass an explicit `SwapDirection` enum variant (AtoB=0 or BtoA=1) as instruction data.

## Evidence

### 1. Canonical Mint Ordering (Verified)

Pool initialization enforces strict canonical ordering via byte comparison:
```rust
// programs/amm/src/instructions/initialize_pool.rs:35-38
require!(
    ctx.accounts.mint_a.key() < ctx.accounts.mint_b.key(),
    AmmError::MintsNotCanonicallyOrdered
);
```

**On-chain mint ordering (all devnet pools):**
- WSOL (bytes: 0x06...) < FRAUD (0x68...) < PROFIT (0x76...) < CRIME (0xd1...)

**Therefore all devnet pools store:**
- CRIME/SOL pool: mint_a=WSOL, mint_b=CRIME
- FRAUD/SOL pool: mint_a=WSOL, mint_b=FRAUD
- CRIME/PROFIT pool: mint_a=PROFIT, mint_b=CRIME (reversed from semantic convention)
- FRAUD/PROFIT pool: mint_a=FRAUD, mint_b=PROFIT

### 2. Explicit Direction Specification (Verified)

The AMM uses an explicit `SwapDirection` enum (not implicit account-order detection):

```rust
// programs/amm/src/instructions/swap_sol_pool.rs:15-32
#[derive(AnchorSerialize, AnchorDeserialize, Clone, Copy, PartialEq, Eq)]
pub enum SwapDirection {
    AtoB,   // variant 0: Token A in, Token B out
    BtoA,   // variant 1: Token B in, Token A out
}
```

**Callers pass direction as instruction data (1 byte):**
- swap_sol_buy.rs:261: `ix_data.push(0u8)` → AtoB (WSOL in, CRIME/FRAUD out)
- swap_sol_sell.rs:144: `direction: u8 = 1` → BtoA (CRIME/FRAUD in, WSOL out)
- swap_profit_buy.rs:114: `if is_reversed { 1 } else { 0 }` → detects pool ordering
- swap_profit_sell.rs: Similar is_reversed logic

### 3. Direction-Based Account Selection (Verified)

The swap handler matches on direction to bind reserves and accounts:

```rust
// programs/amm/src/instructions/swap_sol_pool.rs:89-122
let (
    reserve_in,
    reserve_out,
    input_mint_key,
    output_mint_key,
    ...
) = match direction {
    SwapDirection::AtoB => (
        reserve_a,      // WSOL reserve
        reserve_b,      // CRIME/FRAUD reserve
        mint_a_key,     // WSOL
        mint_b_key,     // CRIME/FRAUD
        ...
    ),
    SwapDirection::BtoA => (
        reserve_b,      // CRIME/FRAUD reserve
        reserve_a,      // WSOL reserve
        mint_b_key,     // CRIME/FRAUD
        mint_a_key,     // WSOL
        ...
    ),
};
```

This binding is **directionally correct** — input and output reserves are correctly swapped.

### 4. Reserve Update Logic (Verified)

Reserves are updated using the direction:

```rust
// programs/amm/src/instructions/swap_sol_pool.rs:176-185
match direction {
    SwapDirection::AtoB => {
        ctx.accounts.pool.reserve_a = new_reserve_in;   // WSOL +
        ctx.accounts.pool.reserve_b = new_reserve_out;  // CRIME/FRAUD -
    }
    SwapDirection::BtoA => {
        ctx.accounts.pool.reserve_b = new_reserve_in;   // CRIME/FRAUD +
        ctx.accounts.pool.reserve_a = new_reserve_out;  // WSOL -
    }
}
```

**Verification by example (CRIME/SOL buy):**
- Before: reserve_a=1000 SOL, reserve_b=1000 CRIME
- User input: 100 SOL (direction=AtoB)
- After fee: effective_input=99 SOL
- Expected CRIME: (1000 * 99) / (1000 + 99) ≈ 90 CRIME
- After: reserve_a=1100 SOL, reserve_b=910 CRIME
- k-invariant: 1000*1000=1M, 1100*910=1.001M ✓ (fee increases k)

This is **correct**.

### 5. PROFIT Pool Direction Detection (Verified)

PROFIT pools implement `is_reversed` detection to handle canonical mint ordering:

```rust
// programs/tax-program/src/instructions/swap_profit_buy.rs:54-114
let pool_mint_a = read_pool_mint_a(&ctx.accounts.pool)?;
let is_reversed = pool_mint_a == profit_mint();

let (reserve_in, reserve_out) = if is_reversed {
    // A=PROFIT, B=faction → BtoA reads reserves as (B, A)
    (reserve_b, reserve_a)
} else {
    // A=faction, B=PROFIT → AtoB reads reserves as (A, B)
    (reserve_a, reserve_b)
};

let direction: u8 = if is_reversed { 1 } else { 0 };
```

**This logic is correct.** When pool stores mint_a=PROFIT (reversed), the direction must be BtoA to maintain the semantic "input=faction, output=PROFIT" relationship.

### 6. Transfer Routing (Verified)

Direction correctly routes transfer accounts:

```rust
// programs/amm/src/instructions/swap_sol_pool.rs:210-315
match direction {
    SwapDirection::AtoB => {
        // Input: user_token_a -> vault_a
        transfer_t22_checked(..., &ctx.accounts.user_token_a, &ctx.accounts.vault_a, ...);
        // Output: vault_b -> user_token_b
        transfer_t22_checked(..., &ctx.accounts.vault_b, &ctx.accounts.user_token_b, ...);
    }
    SwapDirection::BtoA => {
        // Input: user_token_b -> vault_b
        transfer_t22_checked(..., &ctx.accounts.user_token_b, &ctx.accounts.vault_b, ...);
        // Output: vault_a -> user_token_a
        transfer_t22_checked(..., &ctx.accounts.vault_a, &ctx.accounts.user_token_a, ...);
    }
}
```

**This is correct.** Input and output routing matches the direction binding.

## Counter-Evidence Analysis

**Hypothesis assumption:** "AMM determines direction from account ordering"

This is explicitly contradicted by code:
1. Direction is passed as a **1-byte enum variant** in instruction data
2. Direction is **not derived** from account order
3. Direction is **explicitly validated** at swap handler entry (set reentrancy guard)
4. Callers pass correct direction values (0=AtoB for buy, 1=BtoA for sell)

**Hypothesis assumption:** "Wrong direction causes reserve updates backwards"

This is contradicted by:
1. Match statement on direction governs ALL operations (reserve selection, transfer routing, final update)
2. A wrong direction would require the **caller to deliberately pass the wrong enum variant**, which would be caught by tests
3. All tests pass, proving actual execution matches expected behavior

**Phase 52.1 "canonical mint ordering fix":**
The MEMORY.md references an `is_reversed` detection for PROFIT pools. This is **not a bug fix**, but rather a **caller-side adaptation** to handle the fact that mint_a and mint_b are in canonical (byte-sorted) order, not semantic order. The on-chain code correctly implements the is_reversed pattern, and no direction inversion bugs exist.
