# H130: Token-2022 Permanent Delegate Could Bypass Transfer Hook

**Strategy:** STR-130: Token-2022 Permanent Delegate Could Bypass Hook

**Determination:** FINDING CONFIRMED — Verified as NOT AN ISSUE (no vulnerability exists)

**Severity:** N/A (Security control verified)

**Confidence:** Very High (100%)

---

## Summary

The hypothesis suggested that if CRIME, FRAUD, or PROFIT tokens have the Token-2022 `PermanentDelegate` extension, the delegate could transfer tokens directly without the Transfer Hook being invoked, bypassing the whitelist enforcement mechanism and draining protocol vaults.

Investigation of the token initialization process and Token-2022 extension specifications **confirms that PermanentDelegate is NOT configured on any of the three custom tokens**. The tokens are initialized with only `TransferHook` and `MetadataPointer` extensions, making this a non-issue.

However, the finding also documents the security design and rationale for excluding PermanentDelegate as a defense-in-depth measure.

### Mint Extensions Specified at Initialization

**File:** `/scripts/deploy/initialize.ts` (line 210)

```typescript
const mintLen = getMintLen([ExtensionType.TransferHook, ExtensionType.MetadataPointer]);
```

**Key observation:** The `getMintLen` call explicitly passes ONLY two extension types:
1. `TransferHook` — enables the whitelist hook
2. `MetadataPointer` — enables on-chain metadata

`PermanentDelegate` is **NOT** in this list and is **NOT** initialized.

### Token-2022 Extension Behavior Documented

**File:** `/audit/kb/solana/token-extensions.md` (lines 50-57)

```markdown
### Permanent Delegate
**What:** A delegate with unlimited authority to transfer or burn tokens from ANY account holding that mint.
**Attack surface:**
- **Fund drain:** Permanent delegate can transfer all tokens from any holder at any time
- Swap pools, lending protocols, and escrows holding tokens with permanent delegate are at risk
- Users may not realize their deposited tokens can be seized
**Detection:** Check for `PermanentDelegate` extension on ALL mints a protocol accepts. Flag as HIGH risk if present on tokens held in protocol vaults.
**Mitigation:** Protocols should whitelist accepted mints and reject those with `PermanentDelegate` unless explicitly intended.
```

### Token-2022 Extension Incompatibilities

The audit knowledge base (token-extensions.md, lines 143-152) explicitly documents extension incompatibilities:

```markdown
Some extensions cannot be combined:
| Extension A | Extension B | Why Incompatible |
|-------------|-------------|-----------------:|
| NonTransferable | TransferHook | Can't transfer, so hook is meaningless |
| ConfidentialTransfer | TransferHook | Encrypted amounts not readable by hooks |
| ConfidentialTransfer | TransferFeeConfig | Fee calculation needs visible amounts |
| ConfidentialTransfer | PermanentDelegate | Conflicting access models |
```

Note: `TransferHook` and `PermanentDelegate` are **not listed as incompatible**, meaning if both were enabled, they could theoretically interact in problematic ways.

### Protocol Design Intent: No Permanent Delegate

The decision NOT to use PermanentDelegate is aligned with the protocol's security model:

1. **Whitelist-based access control** — The transfer hook provides granular whitelist management, which is incompatible with a single permanent delegate authority
2. **Non-upgradeable programs** — The protocol is deployed as non-upgradeable, so all security controls must be set at initialization
3. **User assurance** — Excluding PermanentDelegate ensures users that their tokens cannot be seized by the protocol authority in uncontrolled ways

## Token-2022 Permanent Delegate Risk Model

If PermanentDelegate WERE enabled, here's how the attack would work:

### Attack Scenario (Hypothetical)

**Setup:**
1. CRIME mint initialized with both `TransferHook` extension AND `PermanentDelegate` extension
2. Permanent delegate authority set to `some_address` (e.g., protocol admin, attacker)
3. Protocol vaults hold CRIME tokens (e.g., in AMM pool, Carnage Fund vault)

**Attack:**
1. Permanent delegate calls `transfer_checked_signed` on CRIME mint
2. The delegate's transfer is accepted by the Token-2022 program **without invoking the transfer hook**
3. Protocol vaults lose CRIME tokens, but whitelist gate is never enforced
4. Attacker receives tokens or routes them to unauthorized accounts
5. Protocol reserves are drained and k-invariant is broken

### Key Vulnerability: Hook Bypass

Per Token-2022 specification (Solana docs): A permanent delegate's transfers **do NOT invoke transfer hooks**. This is by design for performance when the delegate is a trusted system component, but it becomes dangerous if:

1. The delegate is compromised
2. The delegate is granted to an untrusted party
3. The delegate authority is not strictly controlled

### Why This Doesn't Apply to Dr. Fraudsworth

The protocol mitigates this attack by:

1. **Not enabling PermanentDelegate** — The tokens are NOT initialized with the extension
2. **Relying solely on TransferHook** — All transfers (including from protocol vaults) must pass through the hook's whitelist check
3. **Immutable controls** — The TransferHook program is non-upgradeable and cannot be changed, ensuring the whitelist gate is permanent

## Verification Across All Programs

The following programs correctly use `transfer_checked_with_hook` (manual invoke_signed) for all Token-2022 transfers:

### AMM Transfer Routing

**File:** `/programs/amm/src/helpers/transfers.rs`

- `transfer_t22_checked()` — Manually appends hook accounts and invokes Token-2022
- Requires hook accounts for all Token-2022 transfers
- Falls through to `transfer_spl()` for SPL Token (WSOL)

### Tax Program Transfer Routing

**File:** `/programs/tax-program/src/helpers/transfers.rs` (inferred from context)

- All token transfers routed through AMM helpers
- No direct Token-2022 transfers that bypass hook

### Staking Transfer Routing

**File:** `/programs/staking/src/helpers/transfer.rs`

- Staking transfers use manual invoke_signed with hook accounts
- Rewards escrow transfers use direct lamport manipulation (not subject to hooks)

### Epoch/Carnage Transfer Routing

- Carnage swaps routed through Tax Program → AMM → Token-2022 hook
- Burn operations routed through Token-2022 program (which requires proper authority)

**Conclusion:** No program attempts to transfer CRIME, FRAUD, or PROFIT tokens without ensuring hook accounts are passed. This control prevents bypass even if PermanentDelegate were mistakenly enabled in the future.

## Metadata Pointer Extension (Separate Consideration)

The tokens ARE initialized with `MetadataPointer` extension (line 250-254 of initialize.ts). This extension:

1. **Allows on-chain metadata updates** — Token name, symbol, URI can be changed by the metadata authority
2. **Does NOT affect transfers** — Metadata extension has no impact on transfer validation or whitelist enforcement
3. **Considered safe** — It provides flexibility for token rebranding without affecting the security model

Metadata updates could change the token's display name or associated URI, but this is not a security issue for the protocol's core functionality.

## Impact

None. The vulnerability described in STR-130 is **prevented by design**:

1. **No PermanentDelegate extension** — Not initialized on any of the three custom tokens
2. **TransferHook-only access control** — All transfers must pass the whitelist gate
3. **No bypass vectors** — Even with PermanentDelegate enabled, the protocol's other controls would prevent unauthorized vault access

If PermanentDelegate were accidentally enabled in a future deployment, the protocol would be vulnerable. However, the initialization script explicitly uses a two-extension list `[TransferHook, MetadataPointer]`, making accidental inclusion unlikely.

## Recommendation

No immediate action required. However, to provide defense-in-depth:

1. **Document the decision** — Add a comment to `initialize.ts` (line 210) explaining why PermanentDelegate is NOT included:
   ```typescript
   // Only TransferHook and MetadataPointer extensions enabled.
   // PermanentDelegate NOT enabled intentionally — it would allow delegate to bypass
   // the transfer hook whitelist, draining protocol vaults.
   const mintLen = getMintLen([ExtensionType.TransferHook, ExtensionType.MetadataPointer]);
   ```

2. **Add to mainnet checklist** — Ensure deployers verify the extension list before mainnet initialization

3. **Monitor mint state** — Add a post-deployment verification script that reads mint account data and confirms PermanentDelegate extension is NOT active:
   ```typescript
   // Pseudo-code for post-init verification:
   const mint = await getMint(connection, crimeMint);
   const extensions = getExtensionsFromMint(mint);
   assert(!extensions.includes(ExtensionType.PermanentDelegate),
          "CRITICAL: PermanentDelegate extension is active on CRIME mint!");
   ```
