# H049: AMM Fee Correctly Deducted Before k-Invariant Check

**Strategy:** STR-049
**Determination:** NOT VULNERABLE
**Severity:** N/A (Secure Implementation)
**Confidence:** HIGH

## Summary

The Dr Fraudsworth AMM correctly implements fee deduction BEFORE computing swap output and checking the k-invariant. The fee calculation follows the canonical constant-product AMM pattern:

1. **Fee deduction:** `effective_input = amount_in * (10000 - fee_bps) / 10000`
2. **Output calculation:** `amount_out = reserve_out * effective_input / (reserve_in + effective_input)`
3. **Reserve update:** `new_reserve_in = reserve_in + amount_in` (pre-fee amount)
4. **k-invariant verification:** `k_after >= k_before`

This ordering ensures that the LP fee stays in the pool as increased reserves, causing k to increase slightly on each swap (integer truncation dust). The implementation prevents both pool draining and pool bricking attacks.

## Evidence

### 1. Fee Calculation (programs/amm/src/helpers/math.rs:36-40)

```rust
/// Calculate effective input after LP fee deduction.
///
/// Formula: `amount_in * (10_000 - fee_bps) / 10_000`
pub fn calculate_effective_input(amount_in: u64, fee_bps: u16) -> Option<u128> {
    let amount = amount_in as u128;
    let fee_factor = 10_000u128.checked_sub(fee_bps as u128)?;
    amount.checked_mul(fee_factor)?.checked_div(10_000)
}
```

### 2. Swap Output Calculation (programs/amm/src/helpers/math.rs:58-76)

```rust
/// Calculate swap output using constant-product formula.
///
/// Formula: `reserve_out * effective_input / (reserve_in + effective_input)`
pub fn calculate_swap_output(
    reserve_in: u64,
    reserve_out: u64,
    effective_input: u128,  // ← Uses post-fee input
) -> Option<u64> {
    let r_in = reserve_in as u128;
    let r_out = reserve_out as u128;

    let numerator = r_out.checked_mul(effective_input)?;
    let denominator = r_in.checked_add(effective_input)?;

    if denominator == 0 {
        return None;
    }

    let output = numerator.checked_div(denominator)?;
    u64::try_from(output).ok()
}
```

### 4. k-Invariant Verification (programs/amm/src/helpers/math.rs:92-103)

```rust
/// Verify the constant-product invariant: k_after >= k_before.
///
/// k = reserve_in * reserve_out (computed in u128 to avoid overflow).
/// A valid swap must never decrease k. Integer truncation in the output
/// calculation means k typically increases slightly (protocol profit).
pub fn verify_k_invariant(
    reserve_in_before: u64,
    reserve_out_before: u64,
    reserve_in_after: u64,
    reserve_out_after: u64,
) -> Option<bool> {
    let k_before = (reserve_in_before as u128)
        .checked_mul(reserve_out_before as u128)?;
    let k_after = (reserve_in_after as u128)
        .checked_mul(reserve_out_after as u128)?;
    Some(k_after >= k_before)  // ← Enforces k cannot decrease
}
```

## Root Cause

**N/A — secure implementation**

The code follows the canonical constant-product AMM pattern established by Uniswap V2. The fee is deducted from the input before the swap calculation, ensuring:

1. **Correct pricing:** The output is calculated based on what the user actually contributes to the pool (after fee)
2. **Fee accrual:** The difference between `amount_in` and `effective_input` stays in the pool, increasing LP value
3. **k-invariant safety:** Since fee stays in reserves, k increases slightly on each swap (from integer truncation)

## Attack Scenario

**N/A — no vulnerability**

If the fee were deducted AFTER output calculation (the vulnerability this strategy tests for), it would cause:

1. **k-ratcheting (pool bricking):** Output calculated using full `amount_in`, but only `effective_input` added to reserves
   - Result: `k_after = (reserve_in + effective_input) * (reserve_out - amount_out)` would be LESS than `k_before`
   - The `verify_k_invariant` check would fail, making all swaps revert
   - Pool becomes permanently bricked

2. **Or if k-check was removed (worse):** Pool would slowly drain as each swap reduces k

The current implementation prevents both scenarios.

## Impact Assessment

**No impact — secure design**

The AMM correctly implements the constant-product formula with fee deduction before output calculation. The k-invariant check at line 171-173 (swap_sol_pool.rs) and 139-141 (swap_profit_pool.rs) provides a safety net that would catch any deviation from the constant-product curve.

### Additional Security Properties

1. **Property-based testing:** The codebase includes 10,000-iteration proptest validating the k-invariant holds for all valid swaps (math.rs:400-446)
2. **Explicit k-check:** The `verify_k_invariant` function explicitly enforces `k_after >= k_before`, preventing any implementation bugs from draining the pool
3. **Checked arithmetic:** All math uses `checked_add`, `checked_mul`, `checked_div`, `checked_sub` — no overflow risks
4. **Integer truncation direction:** Division rounds down in `calculate_swap_output`, giving less output to swapper (protocol keeps dust) — this is the correct direction per AMM best practices

**N/A — no changes needed**

The implementation is secure and follows industry best practices. The code comments and documentation clearly explain the fee deduction order and k-invariant checking.
