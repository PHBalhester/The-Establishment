# H039: AMM `add_liquidity` Dilutes Existing Reserves

**Finding ID:** H039
**Category:** AMM / Admin
**Severity:** NOT APPLICABLE (Instruction Does Not Exist)
**Status:** CLOSED - FALSE POSITIVE
**Date:** 2026-02-22
**Auditor:** Stronghold of Security (Sonnet 4.5)

---

## Executive Summary

**Investigation Result:** The hypothesized `add_liquidity` instruction **does not exist** in the AMM program. The protocol uses a **"protocol-owned liquidity"** design where liquidity is seeded once during pool initialization and then managed exclusively through swap operations. There is no instruction to add or remove liquidity after initialization, eliminating the entire attack surface described in STR-039.

**Architectural Pattern:** This is an **intentional design choice** (not a missing feature) documented in ARCHITECTURE.md Section 9.7: "Protocol-Owned Liquidity with No LP Tokens — AMM pools have no LP token mechanism — liquidity is admin-seeded and protocol-owned. This eliminates LP extraction attacks but creates dependency on admin for liquidity management."

**Conclusion:** This finding is a **false positive** based on incorrect assumptions about the AMM's capabilities. The threat model in STR-039 assumed the existence of an `add_liquidity` instruction that was never implemented.


**Context:** Inside `initialize_pool` handler, executed once during pool creation. The pool PDA uses Anchor's `init` constraint, preventing re-initialization.

Reserves updated according to constant-product formula (x * y = k), not arbitrary admin input.

## Why This Design Choice Was Made

### Advantages of Protocol-Owned Liquidity

1. **No LP extraction attacks** — Classic DeFi vulnerability where privileged accounts drain LP tokens (e.g., EP-101, EP-064 JIT liquidity manipulation)
2. **Simplified tokenomics** — No need to manage LP token minting, burning, fee distribution to LPs
3. **Deterministic pool state** — Reserves only change via swaps, never via arbitrary admin deposits
4. **Reduced attack surface** — Eliminates entire class of "admin dilutes reserves" attacks
5. **Gas efficiency** — No LP token mint/burn operations on every liquidity operation

### Trade-Offs

1. **No liquidity provision incentive** — External users cannot add liquidity to earn fees
2. **Admin bears impermanent loss** — Protocol (not LP holders) absorbs IL from swap activity
3. **One-time seeding** — If initial liquidity is insufficient, pools cannot be topped up (must redeploy)
4. **Rent locked forever** — Vault accounts persist with no recovery mechanism

## Why the Hypothesis Was Reasonable

### Sources of Confusion

1. **ARCHITECTURE.md mentions `add_liquidity`** (line 145) — The document was synthesized from 9 Phase 1 context analyses and includes this as a hypothetical capability, not an audited fact.

2. **STRATEGIES.md includes STR-039** (line 388-395) — The threat model was generated from AMM playbook patterns and correctly identifies this as a common risk in AMMs with admin-controlled liquidity. The strategy correctly asks "Check if add_liquidity enforces ratio matching" without assuming it exists.

3. **Standard AMM pattern** — Uniswap V2, Raydium, Orca all have `addLiquidity` instructions. It's a natural assumption for any AMM.

4. **`initialize_pool` transfers tokens** — The pool initialization DOES transfer liquidity into vaults, which superficially resembles `add_liquidity`. However, it's one-time and gated by Anchor `init`.
