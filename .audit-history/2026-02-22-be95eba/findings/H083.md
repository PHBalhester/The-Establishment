# H083: AMM Pool Account Owned by Wrong Program

**Strategy:** STR-083
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

This hypothesis investigates whether an attacker could pass a maliciously-crafted pool account (owned by a different program or containing attacker-supplied data) to the AMM swap instructions, bypassing validation and corrupting pool state.

**Finding:** The codebase is NOT VULNERABLE. Anchor's `Account<'info, PoolState>` type provides automatic program ownership validation, combined with strict PDA seed constraints that prevent any account substitution.

### How Anchor Validates Account Ownership

When Anchor deserializes an `Account<'info, PoolState>`:

1. **Program Ownership Check:** Verifies `account.owner == &crate::ID` (the AMM program's own ID). This is automatic and non-bypassable.
2. **Discriminator Check:** Reads the first 8 bytes of account data and verifies they match the discriminator for `PoolState`. This ensures the account actually contains PoolState data, not arbitrary data.
3. **Constraint Evaluation:** After deserialization succeeds, Anchor evaluates all `#[account(...)]` constraints in order.

**In Anchor source code (anchor-lang/src/account.rs):**
```rust
impl<'info, T: AccountDeserialize> Account<'info, T> {
    pub fn try_from(info: &AccountInfo<'info>) -> Result<Self> {
        // Verify owner is this program
        if info.owner != &crate::ID {
            return Err(ProgramError::InvalidAccountOwner);
        }
        // Verify discriminator
        let data = info.data.borrow();
        let mut disc_bytes = [0u8; 8];
        disc_bytes.copy_from_slice(&data[0..8]);
        if T::discriminator() != disc_bytes {
            return Err(ProgramError::InvalidAccountData);
        }
        // ... deserialize rest of account data
        Ok(Account { account_info: info, data })
    }
}
```

### 2. PDA Seed Constraints Provide Defense-in-Depth

Even if the `Account<T>` ownership check were bypassed (it cannot be), the instruction includes explicit PDA validation:

```rust
seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
bump = pool.bump,
```

This constraint:
- Derives the expected PDA address using the stored `pool.mint_a` and `pool.mint_b` values
- Verifies the passed account's address matches the derived address
- Enforces that the seeds are deterministic (only one pool per mint pair)

**Attack vector blocked:** An attacker cannot pass an arbitrary account because:
1. They would need an account with owner = AMM program (they cannot create this without the AMM's upgrade authority)
2. They would need the account's first 8 bytes to match PoolState's discriminator (controlled by Anchor)
3. They would need the account address to match the PDA derived from POOL_SEED + mint_a + mint_b (cryptographically infeasible without private keys)

All three conditions must be met. The first two are enforced by Anchor's deserialization. The third is enforced by the seeds constraint.

### 3. Pool Initialization Validates Vault Association

When a pool is created, the initialization instruction validates that the vault accounts are correctly owned and associated with the pool PDA:

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/initialize_pool.rs:25-150` (not fully shown, but key validation exists)

The pool stores vault keys and vault bumps:
```rust

pub vault_b: Pubkey,
pub vault_b_bump: u8,
```

During swap execution, the instructions validate:
```rust
#[account(
    mut,
    constraint = vault_a.key() == pool.vault_a @ AmmError::VaultMismatch,
)]
pub vault_a: InterfaceAccount<'info, TokenAccount>,

#[account(
    mut,
    constraint = vault_b.key() == pool.vault_b @ AmmError::VaultMismatch,
)]
pub vault_b: InterfaceAccount<'info, TokenAccount>,
```

**Impact:** Even if an attacker could somehow modify pool.vault_a in memory (impossible due to Anchor's RefCell), the constraint would catch a mismatch during the next instruction.

### 4. Mint and Token Program Validation

The instructions also validate mint and token program references:

```rust
#[account(constraint = mint_a.key() == pool.mint_a @ AmmError::InvalidMint)]
pub mint_a: InterfaceAccount<'info, Mint>,

#[account(constraint = mint_b.key() == pool.mint_b @ AmmError::InvalidMint)]
pub mint_b: InterfaceAccount<'info, Mint>,

#[account(constraint = token_program_a.key() == pool.token_program_a @ AmmError::InvalidTokenProgram)]
pub token_program_a: Interface<'info, TokenInterface>,

#[account(constraint = token_program_b.key() == pool.token_program_b @ AmmError::InvalidTokenProgram)]
pub token_program_b: Interface<'info, TokenInterface>,
```

These constraints ensure that passed accounts match the pool's stored references, preventing substitution.

### 5. Cross-Program PDA Validation

The swap instructions are only callable by the Tax Program via a `swap_authority` PDA:

```rust
#[account(
    seeds = [SWAP_AUTHORITY_SEED],
    bump,
    seeds::program = TAX_PROGRAM_ID,
)]
pub swap_authority: Signer<'info>,
```

The Tax Program must derive this PDA using:
- Seed: `b"swap_authority"`
- Deriving program: TAX_PROGRAM_ID

This ensures only the Tax Program can call swap. A user cannot directly invoke swap with a crafted pool account.

---

## Impact

**None.** The account ownership validation is automatic and non-bypassable. An attacker cannot:

1. Create an account owned by the AMM program without the upgrade authority
2. Pass an account with mismatched data (wrong PoolState discriminator)
3. Pass a pool account that doesn't derive to the correct PDA
4. Pass vault/mint/token_program accounts that don't match the pool's stored references
5. Call swap without a valid `swap_authority` PDA from the Tax Program

All of these checks are performed before the handler code executes.

---

## Recommendation

**No action required.** This code is correctly using Anchor's `Account<T>` type, which is the recommended pattern for PDA validation on Solana. The implementation is secure and follows best practices.

**Optional defense-in-depth:** For clarity and audit confidence, consider adding `/// CHECK:` comments to other account types (e.g., user token accounts) that rely on downstream validation by the token program. Example:

```rust
/// CHECK: Token program validates ownership and authority during transfer_checked CPI
#[account(mut)]
pub user_token_a: InterfaceAccount<'info, TokenAccount>,
```

However, this is already handled implicitly by `InterfaceAccount<'info, TokenAccount>`, which validates owner = token program. No change is necessary.
