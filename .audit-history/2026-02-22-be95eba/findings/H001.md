# H001: Bounty Transfer Can Drain Vault Below Rent-Exempt Minimum

**Strategy:** STR-001
**Determination:** CONFIRMED
**Severity:** CRITICAL
**Confidence:** HIGH

## Summary

The `trigger_epoch_transition` instruction checks that the carnage SOL vault has sufficient balance to pay the bounty (`vault_balance >= TRIGGER_BOUNTY_LAMPORTS`) but fails to account for the Solana rent-exempt minimum. When the vault balance is between 1,890,880 lamports (bounty + rent minimum) and 1,000,000 lamports (bounty only), the transfer will succeed in depleting the vault below the rent-exempt threshold of 890,880 lamports, causing the Solana runtime to reject the transaction. This creates a permanent deadlock where epoch transitions cannot proceed until the vault is manually replenished.


The `carnage_sol_vault` is a `SystemAccount` (native SOL holder) derived from PDA seeds `["carnage_sol_vault"]`.

## Root Cause

The balance check at line 195 uses a **single-threshold comparison** that only validates the bounty amount can be paid:

```rust
if vault_balance >= TRIGGER_BOUNTY_LAMPORTS  // Only checks bounty, not bounty + rent
```

The correct check should be:

```rust
if vault_balance >= TRIGGER_BOUNTY_LAMPORTS + RENT_EXEMPT_MINIMUM
```

The Solana runtime enforces that **all accounts must maintain their rent-exempt minimum**. When a transfer would cause an account to drop below this threshold, the transaction is rejected with a runtime error.

### Why This Wasn't Caught by Integration Tests

The carnage vault is funded by 24% of all trading taxes (per the tax distribution split documented in `tax_math.rs`). In typical testing scenarios with multiple trades, the vault accumulates well above the danger zone. The vulnerability only manifests when:

1. The vault is nearly depleted (e.g., after multiple carnage executions with low trading volume between epochs)
2. Exactly one more bounty payment is possible by balance alone
3. That payment would violate the rent-exempt invariant

This is a **timing-dependent edge case** that requires simulating extended low-liquidity periods.

## Attack Scenario

### Scenario 1: Organic Deadlock (No Malicious Actor)

**Preconditions:**
- Protocol has been running for extended period
- Multiple carnage events have drained the vault
- Trading volume is low (minimal tax replenishment)
- Current vault balance: between 1,000,000 and 1,890,880 lamports

**Steps:**
1. Epoch boundary is reached (permissionless trigger condition met)
2. Honest user or crank service calls `trigger_epoch_transition`
3. Check at line 195 passes: `vault_balance >= 1_000_000` ✓
4. System instruction attempts transfer of 1,000,000 lamports
5. **Solana runtime rejects**: post-transfer balance < rent-exempt minimum
6. Transaction fails, epoch transition DOES NOT occur
7. Protocol is deadlocked:
   - Cannot advance epochs (triggers fail)
   - Cannot execute carnage (requires epoch transition)
   - Cannot accumulate more vault funds (carnage deposits taxes)
   - **Permanent freeze** until manual admin intervention

**Likelihood:** MEDIUM-HIGH in production with sustained usage and volatile trading patterns.

### Scenario 2: Griefing Attack

**Preconditions:**
- Adversary monitors vault balance
- Vault balance approaches danger zone

**Steps:**
1. Attacker waits for vault balance to be in range [1,000,000, 1,890,879]
2. Attacker triggers epoch transition (permissionless instruction, anyone can call)
3. Transaction fails due to rent violation
4. **Result**: Epoch advancement is blocked. Protocol cannot auto-recover without external SOL injection into the vault.
5. **Cost to attacker**: Zero (failed transactions don't deduct lamports from `payer`)
6. **Impact**: Protocol operational freeze, loss of user confidence

**Likelihood:** HIGH if vault depletion becomes public knowledge (on-chain state is transparent).

## Impact Assessment

### User Impact
- **Stakers**: Cannot claim rewards (reward distribution requires epoch transitions)
- **Traders**: Cannot trigger new tax rate randomization (stuck with current rates)
- **Liquidity Providers**: Pool operations continue but yield mechanisms freeze

### Protocol Impact
- **Epoch VRF**: Permanently stuck in current epoch, no new randomness consumed
- **Carnage**: Cannot execute (requires epoch transition to set carnage window)
- **Tax Rates**: Frozen at current values (new rates derive from VRF at epoch transitions)
- **Staking Rewards**: Cannot be distributed to pools (distribution is epoch-gated)

### Financial Impact
- **Direct Loss**: None (funds remain in vault, just inaccessible for bounties)
- **Indirect Loss**:
  - Loss of trading fees if users migrate to working protocols
  - Reputational damage requiring potential redeployment
  - Emergency admin intervention costs

### Recovery Path
**Manual Intervention Required:**
1. Admin must transfer SOL directly to `carnage_sol_vault` PDA to bring balance above threshold
2. After replenishment, normal operations resume
3. **Risk**: Programs are non-upgradeable, so a code fix requires full redeployment

## Proof of Concept Sketch

**Transaction Simulation:**

```rust
// Initial state
vault_balance = 1_500_000 lamports  // 0.0015 SOL
TRIGGER_BOUNTY = 1_000_000
RENT_EXEMPT_MIN = 890_880

// Vulnerable check
assert!(vault_balance >= TRIGGER_BOUNTY);  // ✓ passes (1.5M >= 1M)

// Runtime check (implicit in Solana system_instruction::transfer)
post_transfer_balance = vault_balance - TRIGGER_BOUNTY;
// = 1_500_000 - 1_000_000
// = 500_000 lamports

assert!(post_transfer_balance >= RENT_EXEMPT_MIN);  // ✗ FAILS (500K < 890K)

// Transaction aborted with error: "insufficient lamports for rent"
```

**Critical Range:** Vault balance in `[1_000_000, 1_890_879]` lamports triggers the bug.

**Exploitation Code (pseudocode):**
```typescript
// Monitor carnage vault balance on-chain
const vaultBalance = await connection.getBalance(carnageVaultPDA);
```

## Remediation

### Fix 1: Account for Rent-Exempt Minimum (Recommended)

**File:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:195`

**Change:**
```rust
// Before (vulnerable)
let bounty_paid = if vault_balance >= TRIGGER_BOUNTY_LAMPORTS {

// After (secure)
const RENT_EXEMPT_MINIMUM: u64 = 890_880;  // Or calculate via Rent sysvar
let bounty_paid = if vault_balance >= TRIGGER_BOUNTY_LAMPORTS + RENT_EXEMPT_MINIMUM {
```

**Better (using Rent sysvar):**
```rust
let rent = Rent::get()?;
let rent_exempt_min = rent.minimum_balance(0);  // 0 data bytes for SystemAccount
let bounty_paid = if vault_balance >= TRIGGER_BOUNTY_LAMPORTS
    .checked_add(rent_exempt_min)
    .ok_or(EpochError::Overflow)?
{
```

### Fix 2: Use `saturating_sub` for Available Calculation

**Alternative pattern** (more defensive):
```rust
let rent_exempt_min = Rent::get()?.minimum_balance(0);
let available_balance = vault_balance.saturating_sub(rent_exempt_min);
let bounty_paid = if available_balance >= TRIGGER_BOUNTY_LAMPORTS {
    // Transfer bounty
    // ... existing code
    TRIGGER_BOUNTY_LAMPORTS
} else {
    msg!(
        "Insufficient available balance: {} lamports available (after rent), {} required",
        available_balance,
        TRIGGER_BOUNTY_LAMPORTS
    );
    0
};
```

**Why this is better:**
- Explicitly separates "available" from "reserved for rent"
- Uses `saturating_sub` which cannot underflow (if vault < rent, available = 0)
- More readable and maintainable

### Fix 3: Add Integration Test

**New test case** to ensure the fix works:
```rust
#[tokio::test]
async fn test_trigger_bounty_respects_rent_exempt_minimum() {
    // Setup: Fund carnage vault with exactly bounty + rent + 1 lamport
    let rent_min = program.rpc().get_minimum_balance_for_rent_exemption(0)?;
    let vault_balance = TRIGGER_BOUNTY_LAMPORTS + rent_min + 1;

    // Trigger should succeed
    program.methods().triggerEpochTransition()
        .accounts({ carnage_sol_vault, ... })
        .rpc()?;

    // Verify vault still rent-exempt
    let post_balance = program.rpc().get_balance(carnage_sol_vault)?;
    assert!(post_balance >= rent_min);
}

#[tokio::test]
async fn test_trigger_bounty_skips_when_insufficient() {
    // Setup: Fund vault with bounty amount but NOT bounty + rent
    let rent_min = program.rpc().get_minimum_balance_for_rent_exemption(0)?;
    let vault_balance = TRIGGER_BOUNTY_LAMPORTS + (rent_min / 2);  // Danger zone

    // Trigger should succeed BUT skip bounty payment
    let tx = program.methods().triggerEpochTransition()
        .accounts({ carnage_sol_vault, ... })
        .rpc()?;

    // Verify bounty was NOT paid (vault balance unchanged)
    let post_balance = program.rpc().get_balance(carnage_sol_vault)?;
    assert_eq!(post_balance, vault_balance);

    // Verify epoch still advanced (core function worked)
    let epoch_state = program.account::<EpochState>(epoch_state_pda)?;
    assert!(epoch_state.current_epoch > initial_epoch);
}
```
