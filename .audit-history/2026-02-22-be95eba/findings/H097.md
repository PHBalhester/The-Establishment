# H097: Dual Hook Account Ordering for PROFIT Pools (VERIFIED CORRECT)

**Strategy:** STR-097
**Determination:** CLEAR — No vulnerability
**Severity:** N/A
**Confidence:** 95%

## Summary

Investigated the hypothesis that PROFIT pool swaps (CRIME/PROFIT and FRAUD/PROFIT) might have incorrect remaining_accounts ordering for dual transfer hooks. Both token sides in PROFIT pools are Token-2022 with transfer hooks, requiring separate hook accounts per mint. The protocol's critical invariant is: remaining_accounts must be ordered as **[INPUT hooks, OUTPUT hooks]** matching the actual swap direction, NOT as [side A, side B] by pool position.

**Finding:** The implementation is CORRECT across all three entry points (swap_profit_buy, swap_profit_sell, execute_carnage_atomic). The remaining_accounts splitting logic correctly partitions based on swap direction, ensuring each transfer receives the correct hook accounts for its corresponding mint.

## Evidence

### 1. AMM Handler (swap_profit_pool.rs:171-180)

```rust
// 12. Split remaining_accounts for dual-hook CPI.
let hook_account_count = ctx.remaining_accounts.len() / 2;
let (input_hook_accounts, output_hook_accounts) =
    ctx.remaining_accounts.split_at(hook_account_count);

// Direction-aware transfer routing:
// - Input transfer: user -> vault (user signs, no PDA signer)
// - Output transfer: vault -> user (pool PDA signs)
```

The split is at the midpoint, then each transfer receives its semantically correct half:
- Input transfer (user -> vault) gets `input_hook_accounts`
- Output transfer (vault -> user) gets `output_hook_accounts`

Both transfers execute immediately after the split, ensuring no cross-contamination.

**Critical:** The names are semantic (input/output based on swap direction), not positional (A/B). This is correct.

### 2. Direction-Specific Transfer Routing (swap_profit_pool.rs:187-240)

For **AtoB direction** (user sends A, receives B):
```rust
// Input: A (user_token_a -> vault_a)
transfer_t22_checked(..., input_hook_accounts)?;

// Output: B (vault_b -> user_token_b)
transfer_t22_checked(..., output_hook_accounts)?;
```

For **BtoA direction** (user sends B, receives A):
```rust
// Input: B (user_token_b -> vault_b)
transfer_t22_checked(..., input_hook_accounts)?;

// Output: A (vault_a -> user_token_a)
transfer_t22_checked(..., output_hook_accounts)?;
```

Each direction correctly routes hook accounts: first half to input (regardless of A/B position), second half to output. This matches the formula: for any direction, remaining = [INPUT hooks, OUTPUT hooks].

### 3. Tax Program Callers (swap_profit_buy.rs:141-150, swap_profit_sell.rs:133-142)

Both callers forward remaining_accounts unchanged to the AMM CPI:

```rust
// swap_profit_buy.rs: lines 141-150
// Forward remaining_accounts for dual transfer hook support
// PROFIT pools have hooks on BOTH tokens (CRIME/FRAUD and PROFIT)
// AMM splits these at midpoint: first half for input, second half for output
for account in ctx.remaining_accounts.iter() {
    if account.is_writable {
        account_metas.push(AccountMeta::new(account.key(), account.is_signer));
    } else {
        account_metas.push(AccountMeta::new_readonly(account.key(), account.is_signer));
    }
}
```

**Client-side responsibility verified:** The comments explicitly state the expectation that clients will order remaining_accounts as [input_hooks, output_hooks]. The handler relies on this, and spot-checking the TypeScript client (carnage.test.ts in project memory) confirms it does provide them in this order.

### 4. Carnage Execution (execute_carnage_atomic.rs:255-265)

For the Sell action (the most complex case requiring dual-hook partitioning):

```rust
const HOOK_ACCOUNTS_PER_MINT: usize = 4;

let (sell_hook_accounts, buy_hook_accounts) = if matches!(action, CarnageAction::Sell)
    && ctx.remaining_accounts.len() >= HOOK_ACCOUNTS_PER_MINT * 2
{
    (
        &ctx.remaining_accounts[..HOOK_ACCOUNTS_PER_MINT],
        &ctx.remaining_accounts[HOOK_ACCOUNTS_PER_MINT..],
    )
} else {
    // Burn/BuyOnly: all remaining_accounts are for the buy hook
    (&ctx.remaining_accounts[..0], ctx.remaining_accounts)
};
```

For the Sell path, the Epoch Program:
1. Performs a **sell CPI** (user=carnage_signer, direction=BtoA if normal) with `sell_hook_accounts`
2. Performs a **buy CPI** (user=carnage_signer, direction=AtoB if normal) with `buy_hook_accounts`

The Sell transfer is INPUT (removes held tokens), Buy transfer is OUTPUT (adds bought tokens). Partitioning at [..4, 4..] correctly assigns sell hooks to the first swap, buy hooks to the second swap.

### 5. Cross-Reference Against Hook Account Structure

Per context document 05-token-economic.md (Section 1.5):

> **HOOK_ACCOUNTS_PER_MINT = 4** across all programs:
> 1. `extra_account_meta_list` PDA
> 2. Source whitelist PDA
> 3. Destination whitelist PDA
> 4. Transfer Hook program ID

Both CRIME and FRAUD mints use the **same hook program** (Transfer Hook program) with identical PDA derivation. The split-at-midpoint strategy works because:
- Each mint's 4 hook accounts are indistinguishable in structure
- The split doesn't depend on order within the 4 accounts (Token-2022 validates the ExtraAccountMetas PDA internally)
- Both transfers get exactly 4 accounts each

### 6. Token-2022 Runtime Validation

The Transfer Hook program validates:
- `extra_account_meta_list` must be the correct PDA for the mint (derived from mint pubkey)
- `transferring` flag must be set (only Token-2022 runtime sets this during transfer_checked)
- Source OR destination must be whitelisted

If the wrong 4 accounts are passed to a transfer (e.g., mint B's hooks to a mint A transfer), Token-2022 will fail to match the ExtraAccountMetas PDA derived from mint A, returning error 3005 (AccountNotEnoughKeys). The protocol doesn't validate hook accounts itself; it delegates to Token-2022.

## Impact

**No vulnerability.** The dual-hook partitioning is correct:

1. **Correctness of ordering:** [INPUT, OUTPUT] matches the actual control flow (input transfer executes first, receives first half of remaining_accounts; output transfer executes second, receives second half).

2. **Robustness:** If a client were to provide hooks in the wrong order, Token-2022 would reject the transfer with error 3005, preventing silent failure.

3. **Test coverage:** Project memory notes 10,000+ proptest iterations and integration tests covering both directions (AtoB, BtoA) and both swap types (profit_buy, profit_sell). Carnage has 6/6 path tests.

## Recommendation

No action required. The implementation correctly solves the previously known dual-hook ordering bug that was fixed in Phase 51 (per project memory: "PROFIT pool swaps involve tokens on both sides with transfer hooks. The remaining_accounts must be ordered as [INPUT hooks, OUTPUT hooks].")

**For future auditors:** If this code is modified:
- Verify remaining_accounts splitting logic continues to separate based on swap **direction** (input/output), not pool position (A/B)
- Verify each transfer receives exactly HOOK_ACCOUNTS_PER_MINT accounts
- Confirm client-side hook account resolution provides them in [INPUT, OUTPUT] order
- Test both swap directions (AtoB, BtoA) for each pool type
