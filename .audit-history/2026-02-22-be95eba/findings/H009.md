# H009: Dust Swap Accumulation for Zero-Tax Trading

**Strategy:** STR-009
**Determination:** NOT VULNERABLE
**Severity:** INFO
**Confidence:** HIGH

## Summary

Integer division in the tax calculation function (`calculate_tax`) rounds small amounts to zero. This creates a theoretical attack vector where a user could split a large trade into many dust-sized swaps to bypass taxation entirely. However, **Solana's transaction fee structure makes this attack economically impossible** — the attacker would lose more in transaction fees than they would save in taxes.

The dust bypass threshold exists but is protected by a natural economic barrier. This finding is classified as **NOT VULNERABLE** with **INFO** severity to document the behavior for completeness.

## Evidence

### Tax Calculation Code

**File:** `programs/tax-program/src/helpers/tax_math.rs:34-53`

```rust
pub fn calculate_tax(amount_lamports: u64, tax_bps: u16) -> Option<u64> {
    // Validate bps range: 0-10000 is valid (0% to 100%)
    if tax_bps > 10_000 {
        return None;
    }

    // Use u128 intermediates to prevent overflow
    let amount = amount_lamports as u128;
    let bps = tax_bps as u128;

    // tax = amount * tax_bps / 10_000
    // Multiply first for precision (no early truncation)
    let tax = amount
        .checked_mul(bps)?
        .checked_div(10_000)?;

    // Convert back to u64
    u64::try_from(tax).ok()
}
```

### Test Case Confirming Truncation

**File:** `programs/tax-program/src/helpers/tax_math.rs:191-195`

```rust
#[test]
fn tax_4pct_rounds_down() {
    // 4% of 10 lamports = 0.4 -> 0 (floor)
    assert_eq!(calculate_tax(10, 400), Some(0));
}
```

This test explicitly verifies that integer division truncates to zero for small amounts.

## Root Cause

The tax formula is `tax = (amount * tax_bps) / 10_000`, implemented with integer division that truncates fractional results. For small input amounts where `(amount * tax_bps) < 10_000`, the division produces zero:

**Mathematical threshold:** `tax = 0` when `amount < 10_000 / tax_bps`

### Exact Dust Thresholds by Tax Rate

| Tax Rate | BPS Value | Zero-Tax Threshold | Example |
|----------|-----------|-------------------|---------|
| 1% (cheap buy) | 100 | < 100 lamports | 99 lamports × 1% = 0.99 → **0** |
| 2% (cheap buy) | 200 | < 50 lamports | 49 lamports × 2% = 0.98 → **0** |
| 4% (cheap buy) | 400 | < 25 lamports | 24 lamports × 4% = 0.96 → **0** |
| 11% (expensive buy) | 1100 | < 10 lamports | 9 lamports × 11% = 0.99 → **0** |
| 14% (max buy tax) | 1400 | < 8 lamports | 7 lamports × 14% = 0.98 → **0** |

**Key observation:** At the **minimum tax rate (1%)**, amounts up to **99 lamports** produce zero tax.

## Attack Scenario

### Theoretical Attack

1. **Attacker wants to buy 1 SOL worth of CRIME tokens (1,000,000,000 lamports)**
2. **Normal tax at 1% (cheap buy):** `1,000,000,000 × 0.01 = 10,000,000 lamports` (0.01 SOL tax)
3. **Dust bypass strategy:**
   - Split into 10,101,011 swaps of 99 lamports each (total = 999,999,989 lamports ≈ 1 SOL)
   - Each swap: `calculate_tax(99, 100) = 0` (zero tax per swap)
   - Total tax: **0 lamports** instead of 10,000,000 lamports

### Economic Reality Check

**Solana transaction fee:** ~5,000 lamports per transaction (base fee for signature verification)

**Attack cost analysis:**
- **Tax saved:** 10,000,000 lamports (0.01 SOL)
- **Number of transactions required:** 10,101,011
- **Transaction fee cost:** 10,101,011 × 5,000 = **50,505,055,000 lamports** (50.5 SOL)
- **Net loss:** 50.5 SOL in fees to save 0.01 SOL in tax = **-50.49 SOL**

**Break-even calculation:**

For the attack to be profitable, the tax saved must exceed transaction fees:

```
tax_saved > num_transactions × tx_fee
(amount × tax_rate) > (amount / dust_threshold) × 5000
tax_rate > 5000 / dust_threshold

At 1% tax (100 bps), dust_threshold = 100:
0.01 > 5000 / 100
0.01 > 50
FALSE — Attack is never profitable
```

**Conclusion:** The attacker would need to save **50+ lamports per transaction** in tax to break even. At 1% tax on 99-lamport swaps, they save **0 lamports**. The attack loses ~5,000× more than it saves.

### Maximum Viable Dust Amount

Even if we consider the absolute maximum dust (99 lamports at 1% tax), each transaction costs 5,000 lamports in fees. The attacker is paying **50.5× the swap amount** just in transaction fees, making this approach absurd.

## Proof of Concept Sketch

```rust
// Theoretical attack (economically impossible)
const DUST_AMOUNT: u64 = 99; // Max zero-tax amount at 1% rate
const TAX_BPS: u16 = 100;     // 1% tax (cheap buy)
const TX_FEE: u64 = 5_000;     // Solana base fee

let target_amount = 1_000_000_000; // 1 SOL worth of tokens
let num_swaps = target_amount / DUST_AMOUNT; // ~10,101,011 swaps

let tax_per_swap = calculate_tax(DUST_AMOUNT, TAX_BPS); // = 0
let total_tax = tax_per_swap * num_swaps;                 // = 0

let total_tx_fees = num_swaps * TX_FEE; // = 50,505,055,000 lamports (50.5 SOL)

// Profitability check
let normal_tax = calculate_tax(target_amount, TAX_BPS); // = 10,000,000 (0.01 SOL)
let net_profit = normal_tax as i64 - total_tx_fees as i64; // = -50,495,055,000

assert!(net_profit < 0, "Attack loses 50.49 SOL");
```

**Result:** The "attack" costs 5,050× more than it saves.

## Remediation

### Status: NO ACTION REQUIRED

**Rationale:**
1. **Natural economic barrier:** Solana's base transaction fee (5,000 lamports) exceeds any possible dust-level tax savings by orders of magnitude.
2. **Attack is irrational:** No economically motivated actor would execute this attack.
3. **Protocol-favoring rounding:** The existing truncation pattern (floor division) is consistent across the codebase and favors the protocol in all other calculations (swap output, staking rewards).
4. **Code simplicity:** Adding a minimum swap amount check would add complexity for zero security benefit.

### If Remediation Were Desired (Not Recommended)

Two theoretical mitigation approaches (both unnecessary):

#### Option 1: Minimum Tax Enforcement

```rust
pub fn calculate_tax(amount_lamports: u64, tax_bps: u16) -> Option<u64> {
    if tax_bps > 10_000 {
        return None;
    }

    let tax = (amount_lamports as u128)
        .checked_mul(tax_bps as u128)?
        .checked_div(10_000)?;

    let tax_u64 = u64::try_from(tax).ok()?;

    // If amount > 0 and tax_bps > 0, enforce minimum 1 lamport tax
    if amount_lamports > 0 && tax_bps > 0 && tax_u64 == 0 {
        return Some(1); // Minimum 1 lamport tax
    }

    Some(tax_u64)
}
```

**Downside:** Changes rounding behavior, could create edge cases where users pay more tax than expected on tiny transfers.

#### Option 2: Minimum Swap Amount

```rust
// In swap handlers
const MINIMUM_SWAP_AMOUNT: u64 = 100_000; // 0.0001 SOL

require!(
    amount_in >= MINIMUM_SWAP_AMOUNT,
    TaxError::AmountTooSmall
);
```

**Downside:** Blocks legitimate small swaps (e.g., users testing the protocol, small cleanup transfers).

### Recommendation

**Keep the existing implementation.** The dust bypass behavior is:
- Economically impossible to exploit
- Consistent with protocol-wide truncation patterns
- Simpler and cleaner than any mitigation
- Already covered by extensive property-based testing (10,000 iterations)

Document this as a known non-issue in the codebase to prevent future auditors from flagging it unnecessarily.
