# H107: Tax Program Accepts Any Token Mint for Swaps — But Upstream Validation Prevents Exploitation

**Strategy:** STR-107: Tax Program Allows Swap of Non-Faction Token

**Determination:** FALSE POSITIVE — Mint validation is correctly enforced at the AMM layer.

**Severity:** N/A (Mitigated)

**Confidence:** HIGH (verified in code + defense-in-depth analysis)

---

## Summary

The hypothesis proposed that `swap_buy_sol` and other tax program swap functions might accept any token mint (e.g., a worthless custom token) without validation, allowing users to swap arbitrary tokens against CRIME/FRAUD/PROFIT.

**Verdict:** This attack surface does NOT exist. While the Tax Program does not directly validate mints, the downstream AMM program enforces strict mint matching via account constraints:

```rust
// AMM SwapSolPool struct (lines 401-406)
#[account(constraint = mint_a.key() == pool.mint_a @ AmmError::InvalidMint)]
pub mint_a: InterfaceAccount<'info, Mint>,

#[account(constraint = mint_b.key() == pool.mint_b @ AmmError::InvalidMint)]
pub mint_b: InterfaceAccount<'info, Mint>,
```

Any mismatch causes the AMM CPI to fail during deserialization, reverting the transaction. The protocol is secure through **layered validation**: Tax Program trusts the AMM to validate mints, and the AMM enforces the constraint.

---

## Evidence

### 1. Tax Program Account Structs (No Direct Mint Validation)

All Tax Program swap functions accept `mint_a` and `mint_b` as `InterfaceAccount<'info, Mint>` with **NO constraints**:

**`swap_sol_buy.rs` (lines 403-406):**
```rust
/// WSOL mint
pub mint_a: InterfaceAccount<'info, Mint>,

/// CRIME or FRAUD mint (Token-2022)
pub mint_b: InterfaceAccount<'info, Mint>,
```

**`swap_profit_buy.rs` (lines 292-295):**
```rust
/// Mint A (canonical position -- must match pool.mint_a)
pub mint_a: InterfaceAccount<'info, Mint>,

/// Mint B (canonical position -- must match pool.mint_b)
pub mint_b: InterfaceAccount<'info, Mint>,
```

**`swap_profit_sell.rs` (lines 284-287):**
```rust
pub mint_a: InterfaceAccount<'info, Mint>,
pub mint_b: InterfaceAccount<'info, Mint>,
```

**`swap_sol_sell.rs` (lines 542-545):**
```rust
/// WSOL mint
pub mint_a: InterfaceAccount<'info, Mint>,

/// CRIME or FRAUD mint (Token-2022)
pub mint_b: InterfaceAccount<'info, Mint>,
```

The lack of constraints on mint parameters is intentional: the Tax Program passes accounts to the AMM via CPI and relies on the AMM to validate them.

### 2. AMM Enforces Strict Mint Matching

The AMM's `swap_sol_pool.rs` struct (lines 401-406) uses Anchor constraints to verify mint accounts match the pool state:

```rust
/// Mint A: used for decimals in transfer_checked and token program routing.
#[account(constraint = mint_a.key() == pool.mint_a @ AmmError::InvalidMint)]
pub mint_a: InterfaceAccount<'info, Mint>,

/// Mint B: used for decimals in transfer_checked and token program routing.
#[account(constraint = mint_b.key() == pool.mint_b @ AmmError::InvalidMint)]
pub mint_b: InterfaceAccount<'info, Mint>,
```

This constraint is evaluated by Anchor during account deserialization. If a caller passes a mint address that doesn't match `pool.mint_a` or `pool.mint_b`, the CPI fails with `AmmError::InvalidMint` and the transaction reverts.

### 3. Pool State Stores Expected Mints

The PoolState PDA seeds include both mints (lines 378):
```rust
seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
```

The pool address is deterministically derived from `(mint_a, mint_b)` pubkeys. Attempting to access a pool with wrong mints yields the wrong pool address, which fails validation.

### 4. Attack Scenario — Blocked

**Hypothetical attack:**
1. User calls `swap_buy_sol` with `mint_b = CustomToken::new()`
2. Tax Program accepts the mint (no constraint)
3. Tax Program builds CPI to AMM with the custom mint
4. AMM deserialization fails: `custom_mint.key() != pool.mint_b`
5. CPI fails, transaction reverts

**Result:** User cannot swap a custom token. The attack is blocked at the AMM boundary.

### 5. Defense-in-Depth

The protocol employs layered validation:
- **Layer 1 (AMM pool constraints):** Mints must match pool state
- **Layer 2 (Pool PDA derivation):** Pool address is derived from canonical mints, preventing pool substitution
- **Layer 3 (Token program constraints):** Both AMM and Tax Program validate token program IDs match pool state

Any single layer would be sufficient. Multiple layers ensure robustness.

## Impact

**None.** The attack cannot occur. An attacker cannot:
1. Use a non-CRIME/FRAUD/PROFIT token in a swap
2. Trick the protocol into accepting arbitrary tokens
3. Bypass mint validation through CPI manipulation

## Recommendation

**No remediation required.** The mint validation is correct and enforced at the appropriate layer (downstream CPI handler).

**Code quality note:** The Tax Program accounts could include explicit documentation comments noting that mint validation is deferred to the AMM, but this is a minor improvement only. Current design is secure.

## Conclusion

**Status:** CLOSED — No vulnerability present

The hypothesis was based on the valid observation that Tax Program swap functions don't directly validate mints. However, this is a secure design pattern where validation is delegated to the downstream CPI handler (AMM). The AMM enforces strict mint constraints that make the attack impossible.

This is a good example of **correct trust boundary design**: each program is responsible for validating its own inputs, and Solana's synchronous CPI ensures failures propagate correctly.
