# S010: Epoch Freeze Recovery Requires Admin Intervention — No Autonomous Replenishment Path

**Strategy:** S010
**Determination:** CONFIRMED
**Severity:** HIGH
**Confidence:** HIGH

## Summary

After a bounty deadlock (H001), the protocol cannot autonomously recover from the frozen state. All paths that deposit SOL to the carnage vault are **permissionless user-facing swaps** (swap_sol_buy, swap_sol_sell) or **protocol-automated carnage executions**. There is **no admin-privileged instruction** to manually deposit SOL to the carnage_sol_vault, and there is **no emergency recovery mechanism**. If the admin key is lost, compromised, or unavailable, a bounty deadlock results in **permanent protocol freeze** with no path to resume epoch transitions.

This creates a **single point of failure** dependency on:
1. **Admin key availability** to manually transfer SOL to the PDA
2. **External SOL injection** via raw Solana CLI or custom tooling (not protocol-native)

The protocol lacks **redundancy** and **fail-safe mechanisms** for critical operational continuity.

### Fallback Recovery Mechanism: External Tooling

The **only** way to replenish the vault after a deadlock is:

**Option A: Raw Solana CLI Transfer**
```bash
solana transfer <carnage_sol_vault_pda> 0.002 --keypair admin.json
```
- **Requirement:** Admin key must be available and secure
- **Risk:** If admin key is lost/compromised, this path is blocked

**Option B: Custom Script via Web3.js/Anchor**
```typescript
await provider.sendAndConfirm(
  new Transaction().add(
    SystemProgram.transfer({
      fromPubkey: admin.publicKey,
      toPubkey: carnageSolVaultPda,
      lamports: 2_000_000, // 0.002 SOL
    })
  ),
  [adminKeypair]
);
```
- **Same limitation:** Admin key dependency

**Critical Gap:** Both fallback paths are **external to the protocol**. They are not:
- Part of the protocol's instruction set
- Auditable as protocol logic
- Protected by program-level constraints
- Accessible to non-admin parties (no permissionless recovery)

## Root Cause Analysis

### Why Autonomous Recovery is Impossible

1. **All inflows are user-driven:** Deposits to carnage_sol_vault come from swap taxes (24% of trading volume). During a deadlock:
   - Epoch transitions freeze → stale tax rates → reduced trading incentives
   - Users may migrate to competitors → trading volume drops
   - **Feedback loop:** Less trading → less vault funding → prolonged deadlock

2. **No protocol-automated top-up:** Unlike some DeFi protocols that have:
   - Emergency reserve funds (a separate vault that auto-drains to cover shortfalls)
   - Dynamic fee routing (redirect 100% of fees to critical vaults during emergencies)
   - Fallback staking rewards (mint tokens to cover operational costs)
   - **This protocol has NONE of these mechanisms**

3. **No permissionless fail-safe:** Unlike DAO-governed protocols where a quorum can vote to trigger emergency actions, this protocol:
   - Has no governance module
   - Is non-upgradeable (no emergency upgrade path)
   - Has a single admin key (no multisig, no key rotation)

### Comparison to H001 (Bounty Deadlock)

| Aspect | H001 (Rent-Exempt Bug) | S010 (Recovery Dependency) |
|--------|------------------------|---------------------------|
| **Root Cause** | Incomplete balance check in trigger_epoch_transition | No admin deposit instruction in protocol |
| **Trigger** | Vault balance in range [1M, 1.89M] lamports | H001 occurs AND admin key unavailable |
| **Impact** | Protocol freeze (epoch transitions fail) | **Permanent freeze** (no recovery) |
| **Severity** | CRITICAL (can occur organically) | HIGH (requires key compromise/loss) |
| **Recovery** | Admin sends 0.002 SOL to vault (external) | **Same** — no autonomous path |

**Key Insight:** H001 and S010 are **layered vulnerabilities**. H001 is the **trigger**, S010 is the **amplifier**. H001 creates the deadlock; S010 makes it **irrecoverable** without admin intervention.

## Attack Scenarios

### Scenario 1: Admin Key Loss (Organic Failure)

**Preconditions:**
- Admin wallet private key lost (disk failure, lost seed phrase, hardware failure)
- No backup key exists (single point of failure)
- H001 bounty bug occurs (vault depleted below threshold)

**Timeline:**
1. **Day 0:** Bounty deadlock triggers (H001). Protocol freezes.
2. **Day 1:** Community reports frozen epochs. Team attempts recovery.
3. **Day 2:** Admin realizes private key is unrecoverable (hardware wallet died, no seed backup).
4. **Day 3+:** **Permanent protocol freeze.** No one can deposit to vault.
   - All epoch transitions fail forever
   - Carnage never executes again
   - Tax rates frozen at stale values
   - Staking rewards distribution halted
   - Protocol is **dead**

**Likelihood:** LOW (assumes poor key management practices), but **catastrophic** if it occurs.

### Scenario 2: Admin Key Compromise (Adversarial Failure)

**Preconditions:**
- Admin key compromised (phishing, malware, social engineering)
- Attacker does NOT immediately drain funds (stealth mode)
- Attacker **waits** for H001 to organically trigger

**Timeline:**
1. **Week 1:** Attacker gains admin key access, does **nothing** (avoids detection).
2. **Week 4:** H001 bounty bug triggers naturally (vault depleted from carnage + low trading volume).
3. **Hour 0:** Protocol freezes. Community expects admin to fix it.
4. **Hour 6:** Legitimate admin attempts recovery. **Attacker front-runs** every replenishment transaction:
   - Admin sends 0.002 SOL to vault → Attacker immediately triggers `trigger_epoch_transition` with a fresh randomness account that will fail to reveal (force another bounty payment)
   - Vault drains again below threshold
   - Repeat griefing
5. **Day 2:** Community suspects compromise. Admin attempts to revoke key.
   - **Problem:** There is **no key rotation mechanism** in the protocol
   - Only option: **Redeploy entire protocol** (lose all existing state, liquidity, positions)

**Likelihood:** MEDIUM (phishing attacks on DeFi admins are common), **high impact**.

### Scenario 3: Regulatory Seizure / Admin Unavailability

**Preconditions:**
- Government seizes admin's devices (regulatory action, legal dispute)
- Admin is incarcerated, incapacitated, or otherwise unavailable
- H001 triggers during admin's absence

**Timeline:**
1. **Day 0:** Legal action begins. Admin's keys are in custody, inaccessible.
2. **Day 7:** H001 bounty bug triggers.
3. **Day 8+:** Protocol frozen. Community has no recourse.
   - No multisig to rotate control
   - No governance to vote on emergency recovery
   - **Protocol is abandoned**

**Likelihood:** LOW (depends on jurisdiction, admin's legal exposure), but **not zero**.

## Recommendation

### Fix 1: Add Admin Emergency Deposit Instruction (Immediate Term)

**Location:** `programs/epoch-program/src/instructions/admin_deposit_carnage.rs` (new file)

**Implementation:**
```rust
use anchor_lang::prelude::*;
use anchor_lang::solana_program::system_instruction;

use crate::constants::{CARNAGE_SOL_VAULT_SEED, DEVNET_ADMIN};
use crate::errors::EpochError;
use crate::events::AdminDepositCarnage;

/// Emergency admin deposit to carnage_sol_vault.
///
/// Allows admin to manually replenish the vault if organic trading volume
/// is insufficient to recover from a bounty deadlock (H001).
///
/// This is a fail-safe mechanism for operational continuity, not a routine
/// operation. Normal vault funding comes from swap taxes (24% of all trades).
///
/// # Arguments
/// * `amount` - Lamports to deposit (must be > 0)
///
/// # Errors
/// * `Unauthorized` if caller is not the admin pubkey
/// * `InvalidAmount` if amount is 0
pub fn handler(ctx: Context<AdminDepositCarnage>, amount: u64) -> Result<()> {
    require!(amount > 0, EpochError::InvalidAmount);

    // Transfer SOL from admin to carnage_sol_vault
    invoke(
        &system_instruction::transfer(
            ctx.accounts.admin.key,
            ctx.accounts.carnage_sol_vault.key,
            amount,
        ),
        &[
            ctx.accounts.admin.to_account_info(),
            ctx.accounts.carnage_sol_vault.to_account_info(),
            ctx.accounts.system_program.to_account_info(),
        ],
    )?;

    msg!("Admin emergency deposit: {} lamports to carnage_sol_vault", amount);

    emit!(AdminDepositCarnage {
        admin: ctx.accounts.admin.key(),
        amount,
        new_balance: ctx.accounts.carnage_sol_vault.lamports(),
        timestamp: Clock::get()?.unix_timestamp,
    });

    Ok(())
}

#[derive(Accounts)]
pub struct AdminDepositCarnage<'info> {
    /// Admin authority (must sign and pay)
    #[account(
        mut,
        constraint = admin.key() == DEVNET_ADMIN @ EpochError::Unauthorized
    )]
    pub admin: Signer<'info>,

    /// Carnage SOL vault PDA (receives deposit)
    /// CHECK: PDA derived from known seeds, holds native SOL
    #[account(
        mut,
        seeds = [CARNAGE_SOL_VAULT_SEED],
        bump,
    )]
    pub carnage_sol_vault: SystemAccount<'info>,

    /// System program
    pub system_program: Program<'info, System>,
}
```

**Tradeoffs:**
- **Pro:** Immediate operational continuity fail-safe
- **Pro:** No change to core protocol logic (append-only instruction)
- **Con:** Does NOT eliminate admin key as single point of failure
- **Con:** Mainnet deployment still requires secure admin key custody

### Fix 2: Implement Multisig Admin Authority (Medium Term)

**Recommendation:** Replace single admin key with a Squads multisig or similar M-of-N threshold wallet.

**Benefits:**
- **Key Loss Resilience:** Losing 1 key (in a 3-of-5 setup) does not freeze the protocol
- **Key Compromise Protection:** Attacker needs M keys, not 1
- **Operational Continuity:** Multiple team members can authorize emergency actions

**Implementation Path:**
1. Deploy multisig wallet (e.g., Squads, Realms, custom PDA-based)
2. Update `DEVNET_ADMIN` constant to multisig PDA
3. All admin instructions (admin_deposit_carnage, force_carnage, update_treasury) validate against multisig

**Tradeoff:** Increases operational complexity (coordination among signers), but drastically reduces single-point-of-failure risk.

### Fix 3: Add Protocol-Owned Reserve Fund (Long Term)

**Recommendation:** Create a **reserve vault** that auto-drains to carnage_sol_vault when balance drops critically low.

**Design:**
1. Initialize `reserve_sol_vault` PDA at deployment
2. Route **1% of carnage deposits** to reserve (e.g., split 23% carnage → 22.77% carnage, 0.23% reserve)
3. Add instruction `auto_replenish_carnage` (permissionless):
   - **Guard:** `carnage_sol_vault.lamports() < EMERGENCY_THRESHOLD` (e.g., 3M lamports)
   - **Action:** Transfer from reserve to carnage vault (capped at MAX_AUTO_REPLENISH per call)
   - **Rate Limit:** Once per epoch to prevent abuse

**Benefits:**
- **Autonomous Recovery:** No admin intervention required for small shortfalls
- **Gas Efficiency:** Can be bundled with trigger_epoch_transition (caller gets bounty + protocol gets replenishment)
- **Permissionless:** Anyone can call this when threshold is met

**Tradeoff:** Reduces carnage fund growth rate by ~1% (acceptable for resilience gain).

### Fix 4: Remove Bounty Payment (Alternative Mitigation)

**Radical Option:** Eliminate the 0.001 SOL bounty payment from `trigger_epoch_transition`.

**Rationale:**
- Epoch transitions are already incentivized by protocol liveness (traders benefit from tax rate updates)
- Removing the bounty eliminates H001 entirely (no outflow = no rent-exempt deadlock)
- Carnage vault becomes **inflow-only** (all taxes in, only carnage execution spends)

**Tradeoff:**
- **Pro:** Simplifies logic, removes H001 attack surface
- **Con:** Reduces incentive for third-party crank operators to trigger epochs (may rely on altruistic/protocol-owned crank)
- **Pro:** Cranks can still be run by the team or a dedicated service (Railway crank already exists per MEMORY.md)

**Verdict:** If the team already runs a crank, this is the **simplest fix**. Otherwise, keep the bounty but fix H001 + add admin_deposit_carnage.
