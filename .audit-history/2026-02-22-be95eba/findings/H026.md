# H026: AMM Swap Output Calculation Off-By-One

**Strategy:** STR-026
**Determination:** NOT PRESENT
**Severity:** N/A (hypothetical CRITICAL if present)
**Confidence:** HIGH

## Summary

The constant-product swap formula in the AMM program was investigated for potential off-by-one errors that could allow pool draining through repeated swaps. The hypothesis was that the denominator in the output calculation might be missing the input amount, which would allow output to equal or exceed reserve_out.

**Result:** The implementation is **CORRECT**. The formula properly includes the input amount in the denominator, and multiple layers of verification (checked arithmetic, k-invariant checks, property-based testing) confirm the swap math cannot drain pools.

## Investigation

### Formula Under Investigation

The constant-product AMM formula should be:
```
output = reserve_out * input / (reserve_in + input)
```

Where the denominator **MUST** include the input amount to preserve the k-invariant (reserve_in * reserve_out = k).

### Code Analysis

**Location:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/helpers/math.rs:58-76`

**Actual Implementation:**
```rust
pub fn calculate_swap_output(
    reserve_in: u64,
    reserve_out: u64,
    effective_input: u128,
) -> Option<u64> {
    let r_in = reserve_in as u128;
    let r_out = reserve_out as u128;

    let numerator = r_out.checked_mul(effective_input)?;       // Line 66
    let denominator = r_in.checked_add(effective_input)?;      // Line 67 - CRITICAL

    if denominator == 0 {
        return None;
    }

    let output = numerator.checked_div(denominator)?;          // Line 73

    u64::try_from(output).ok()
}
```

**Key Finding (Line 67):** The denominator is computed as `r_in.checked_add(effective_input)`, which **correctly includes the input amount**. This is the proper constant-product formula.

### Mathematical Verification

For a swap where `amount_in = 1000`, `reserve_in = 1_000_000`, `reserve_out = 1_000_000`:

**Correct Formula (implemented):**
```
numerator = 1_000_000 * 1000 = 1_000_000_000
denominator = 1_000_000 + 1000 = 1_001_000
output = 1_000_000_000 / 1_001_000 = 999 (truncated)
```

**k-invariant check:**
```
k_before = 1_000_000 * 1_000_000 = 1_000_000_000_000
k_after = 1_001_000 * 999_001 = 1_000_000_999_001
k_after > k_before ✓ (invariant holds)
```

**Incorrect Formula (hypothetical bug):**
```
numerator = 1_000_000 * 1000 = 1_000_000_000
denominator = 1_000_000 (MISSING input)
output = 1_000_000_000 / 1_000_000 = 1000
```

**k-invariant check (if bug existed):**
```
k_before = 1_000_000 * 1_000_000 = 1_000_000_000_000
k_after = 1_001_000 * 999_000 = 999_999_000_000
k_after < k_before ✗ (invariant violated - would fail on-chain)
```

With the hypothetical bug, `output = input` for equal reserves, which allows draining the pool by repeatedly swapping equal amounts. The k-invariant check at line 171-173 in `swap_sol_pool.rs` would catch this and reject the transaction.
