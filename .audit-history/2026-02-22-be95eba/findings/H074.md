# H074: AMM Pool Not Marked Burnable Before Carnage Burn
**Strategy:** STR-074
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The strategy hypothesizes that Carnage burns tokens from a pool, and if the pool (or specifically, the token mint) isn't marked as "burnable", the burn CPI would fail. Investigation into the code and Token-2022 specification reveals this hypothesis is based on a misunderstanding of Token-2022's burn mechanism.

**Conclusion:** The hypothesis is invalid. Token-2022 does not require any "burnable" flag or extension on the mint to perform burns. The Burn instruction is always available to the token owner/authority. Therefore, this is NOT VULNERABLE.

## Evidence

### 1. Token-2022 Burn Mechanism

Token-2022's Burn instruction (instruction discriminator 8) is a standard operation available on all Token-2022 mints by default. Unlike the optional Pausable extension (which gates burn/transfer/mint) or the Permissioned Burn extension (which requires an additional authority), the base Burn instruction requires only:
- The source token account (writable)
- The mint (writable)
- The account owner/authority (signer)

**Source:** Solana Token-2022 documentation lists mint extensions as optional (Transfer Fees, Permanent Delegate, Pausable, Permissioned Burn, etc.). The Burn instruction is NOT gated behind any extension.

### 2. Carnage Burn Implementation

The `burn_held_tokens` function manually constructs a Burn instruction:

```rust
let mut burn_data = vec![8u8]; // Burn instruction discriminator
burn_data.extend_from_slice(&amount.to_le_bytes());

let burn_ix = Instruction {
    program_id: token_program.key(),
    accounts: vec![
        AccountMeta::new(vault.key(), false),        // source account
        AccountMeta::new(mint, false),               // mint
        AccountMeta::new_readonly(carnage_state.key(), true), // authority
    ],
    data: burn_data,
};

invoke_signed(&burn_ix, &[vault, burn_mint, carnage_state_info], &[signer_seeds])?;
```

This construction is correct:
- Accounts: [source, mint, authority]
- Authority: `carnage_state` PDA (the vault owner)
- Signature: carnage_state signs via `invoke_signed`

No "burnable" flag check is performed because none is required.

### 4. Token-2022 Extension Hierarchy

Token-2022 provides optional extensions:
- **Pausable:** Requires initialization before InitializeMint. Blocks burn/transfer/mint when paused.
- **Permissioned Burn:** Requires initialization before InitializeMint. Requires an additional authority signature.
- **Standard Burn:** Always available. Requires owner/authority signature only.

The CRIME and FRAUD mints use the standard burn mechanism (no extensions initialized), which is always functional.

## Impact

There is no impact because the burn mechanism does not require a "burnable" flag. The test execution (299 Rust tests, 0 failures) confirms that burns execute successfully across all Carnage paths.

Test coverage includes:

## Recommendation

**No action required.** The code is correct. This hypothesis was based on assumptions about Token-2022 design that do not match the actual specification. Token-2022 burn is always enabled on standard mints and requires no prior flag configuration.
