# STR-040: Staking Reward Distribution Without Sufficient Balance

**Severity:** MITIGATED (Originally assessed as MEDIUM)
**Status:** NOT VULNERABLE
**Subsystem:** Staking Program (deposit_rewards, claim_rewards)
**Category:** Economic Integrity / Accounting

---

## Summary

Investigation into whether `deposit_rewards` could inflate `pending_rewards` without actual SOL being transferred to the escrow vault, potentially leaving later claimers unable to withdraw their earned rewards.

**Conclusion:** The vulnerability does NOT exist. Multiple defensive layers prevent accounting inflation.

---

## Attack Scenario (Theoretical)

An attacker (or buggy Tax Program call) could:
1. Call `deposit_rewards` with a large `amount` value
2. Increment `pool.pending_rewards` without transferring SOL
3. `update_cumulative` would distribute these phantom rewards into `rewards_per_token_stored`
4. Users would claim based on inflated cumulative value
5. Escrow vault would run dry
6. Last claimers would receive `InsufficientEscrowBalance` errors and lose their rewards

**Expected Impact:** Complete loss of staking rewards for late claimers, potential protocol insolvency.

---

## Mitigations Found (ROBUST)

### 1. deposit_rewards Balance Reconciliation (PRIMARY DEFENSE)

**Location:** `programs/staking/src/instructions/deposit_rewards.rs:99-102`

```rust
// Reconcile: verify escrow vault actually holds enough SOL.
// Tax Program should have transferred SOL before this CPI call.
// If the transfer failed silently or the amount was short-changed,
// pending_rewards would be inflated beyond available balance.
require!(
    ctx.accounts.escrow_vault.lamports() >= pool.pending_rewards,
    StakingError::InsufficientEscrowBalance
);
```

**How it works:**
- After incrementing `pending_rewards`, the instruction reads the ACTUAL lamport balance of the escrow vault
- Compares `vault.lamports() >= pool.pending_rewards` (note: cumulative sum of ALL pending, not just this deposit)
- If the vault balance is insufficient to cover pending_rewards, the instruction REVERTS
- This catches:
  - Missing SOL transfers before CPI
  - Partial/short-changed transfers
  - Race conditions or out-of-order execution
  - Any accounting drift over time

**Why it's effective:**
- The check happens in the SAME TRANSACTION as the pending_rewards increment
- Solana transactions are atomic - if the check fails, the entire state change (including the pending_rewards increment) is rolled back
- The escrow vault is a PDA owned by the Staking Program, so its lamport balance cannot be manipulated by the Tax Program

### 2. Tax Program Enforces Transfer-Before-CPI (DEFENSE-IN-DEPTH)

**Location:** `programs/tax-program/src/instructions/swap_sol_buy.rs:126-176`

```rust
// 5a. Transfer staking portion (75%) and notify Staking Program
if staking_portion > 0 {
    // Transfer SOL to staking escrow
    invoke_signed(
        &system_instruction::transfer(
            ctx.accounts.user.key,
            ctx.accounts.staking_escrow.key,
            staking_portion,
        ),
        &[...],
        &[], // User signs
    )?;

    // CPI to Staking Program's deposit_rewards to update pending_rewards counter.
    // The SOL is already in escrow; this just updates the state.
    invoke_signed(&deposit_ix, &[...], &[tax_authority_seeds])?;
}
```

**How it works:**
- The Tax Program FIRST executes `system_instruction::transfer` to move SOL into the escrow vault
- Only AFTER the transfer succeeds does it call `deposit_rewards` via CPI
- If the transfer fails (e.g., insufficient user balance), the entire transaction reverts before `deposit_rewards` is reached
- This ordering ensures the balance reconciliation check in `deposit_rewards` will always pass in normal operation

**Why it's effective:**
- Solana instruction execution is sequential within a transaction
- The `?` operator propagates errors - if the transfer fails, execution halts before the CPI
- This design pattern (transfer-then-notify) is the canonical safe approach for deposit operations

### 3. claim_rewards Escrow Solvency Check (LAST LINE OF DEFENSE)

**Location:** `programs/staking/src/instructions/claim.rs:100-111`

```rust
// Verify escrow has sufficient balance
// Emit EscrowInsufficientAttempt event before returning error for monitoring
let escrow_balance = ctx.accounts.escrow_vault.lamports();
if escrow_balance < rewards_to_claim {
    emit!(EscrowInsufficientAttempt {
        user: ctx.accounts.user.key(),
        requested: rewards_to_claim,
        available: escrow_balance,
        slot: clock.slot,
    });
    return Err(StakingError::InsufficientEscrowBalance.into());
}
```

**How it works:**
- Before transferring SOL to the user, `claim_rewards` checks the escrow vault balance
- If the vault cannot cover the claim, the instruction:
  1. Emits an `EscrowInsufficientAttempt` event for monitoring/alerting
  2. Returns an error (transaction reverts, no state change)
- The user's `rewards_earned` remains intact - they can retry later if the vault is refilled

**Why it's effective:**
- Even if accounting somehow became inflated (e.g., via a hypothetical bug), claims would FAIL rather than drain the vault
- The event emission-before-error pattern ensures monitoring systems can detect anomalies even when transactions revert
- This prevents the "last claimer gets nothing" scenario - ALL claimers would fail if the vault is underfunded

### 4. Protocol-Favoring Truncation (MATHEMATICAL GUARANTEE)

**Location:** `programs/staking/src/helpers/math.rs:49-50`

```rust
let pending = (user.staked_balance as u128)
    .checked_mul(reward_delta)
    .ok_or(StakingError::Overflow)?
    .checked_div(PRECISION)
    .ok_or(StakingError::DivisionByZero)? as u64;
```

**How it works:**
- The Synthetix cumulative reward formula uses integer division
- Division truncates (floors) the result, discarding fractional lamports
- This truncation ALWAYS favors the protocol, meaning the sum of all user claims is LESS THAN the total deposited

**Mathematical proof:**
- `reward_per_token = (pending * PRECISION) / total_staked` (truncated)
- `user_reward = (balance * reward_per_token) / PRECISION` (truncated)
- Each truncation loses a fractional lamport
- With N users, up to N lamports of dust remain in the escrow
- Therefore: `sum(user_claims) <= total_deposited`

**Property test evidence:** `programs/staking/src/helpers/math.rs:492-527` (proptest with 10,000 iterations validates this property)

## Attack Viability: NOT FEASIBLE

### Why Direct Exploitation Fails

An attacker would need to call `deposit_rewards` without transferring SOL. However:

1. **Access Control:** `deposit_rewards` requires the caller to be the Tax Program's `tax_authority` PDA:
   ```rust
   #[account(
       seeds = [TAX_AUTHORITY_SEED],
       bump,
       seeds::program = tax_program_id(),
   )]
   pub tax_authority: Signer<'info>,
   ```
   Only the Tax Program can produce a valid PDA signer. External attackers cannot call this instruction directly.

2. **Balance Reconciliation:** Even if an attacker compromised the Tax Program's logic to skip the SOL transfer, the `escrow_vault.lamports() >= pool.pending_rewards` check in `deposit_rewards` would catch it and revert.

3. **No Bypass Path:** There is no alternative instruction that increments `pending_rewards` or `rewards_per_token_stored`. The only paths are:
   - `deposit_rewards` (guarded by balance check)
   - `update_cumulative` (moves pending to cumulative, no external input)

### Why Accounting Drift Cannot Occur

**Scenario:** Could repeated deposits with rounding errors cause drift?

**Analysis:** No, because:
- `deposit_rewards` checks `vault.lamports() >= pending_rewards` (the CUMULATIVE sum)
- Not just `vault.lamports() >= amount` (the incremental deposit)
- Any drift (even 1 lamport) would cause the check to fail
- This prevents cumulative error accumulation

**Scenario:** Could race conditions cause the vault balance to decrease between deposit and check?

**Analysis:** No, because:
- Solana transactions execute atomically within a single slot
- The escrow vault is a PDA owned by the Staking Program
- No other instruction can withdraw from the vault while `deposit_rewards` is executing
- The only valid withdrawal path is `claim_rewards`, which also checks solvency

## Cross-Program Dependencies

### deposit_rewards CPI Flow

```
User initiates swap
  → Tax Program (swap_sol_buy handler)
    → system_instruction::transfer (User → Escrow Vault) [SOL moved]
    → Staking Program (deposit_rewards CPI)
      → pool.pending_rewards += amount
      → require!(vault.lamports() >= pending_rewards) [Balance check]
      → emit!(RewardsDeposited)
```

**Critical ordering:** The balance check in `deposit_rewards` happens AFTER the increment but BEFORE the transaction commits. If the check fails, Solana's runtime rolls back the entire transaction, including the pending_rewards increment.

### claim_rewards Flow

```
User calls claim
  → Staking Program (claim handler)
    → update_rewards() [Finalize pending based on cumulative]
    → escrow_balance = vault.lamports()
    → if escrow_balance < rewards_to_claim { emit + revert }
    → user.rewards_earned = 0 [State update]
    → Transfer SOL (Escrow Vault → User) [SOL moved]
```

**CEI Pattern:** Checks and state updates happen before the external transfer, preventing reentrancy.

## Recommendations

Despite the finding being MITIGATED, consider these enhancements for defense-in-depth:

### 1. Add Invariant Monitoring (LOW PRIORITY)

Emit a periodic event comparing `pool.total_distributed` vs. `escrow_vault.lamports()`:

```rust
// In update_cumulative or a dedicated health-check instruction
emit!(EscrowHealthCheck {
    total_distributed: pool.total_distributed,
    total_claimed: pool.total_claimed,
    escrow_balance: escrow_vault.lamports(),
    expected_unclaimed: total_distributed - total_claimed,
    slot: clock.slot,
});
```

**Benefit:** Off-chain monitoring can alert if the invariant `escrow_balance >= (total_distributed - total_claimed)` is violated.

### 2. Add Rent-Exempt Minimum to Balance Checks (MEDIUM PRIORITY)

The `deposit_rewards` balance check should account for rent-exempt minimum:

```rust
let rent_exempt_min = Rent::get()?.minimum_balance(0); // For empty account
require!(
    ctx.accounts.escrow_vault.lamports() >= pool.pending_rewards + rent_exempt_min,
    StakingError::InsufficientEscrowBalance
);
```

**Rationale:** Similar to the known bounty rent bug (ECON-07), withdrawing the last lamport would make the account rent-ineligible. This is unlikely for the escrow vault (it will always have dust due to truncation), but belt-and-suspenders.

### 3. Document the Balance Reconciliation Pattern (INFO)

Add a design doc explaining the transfer-before-CPI + balance-check pattern for future developers. This is a critical security invariant that must be preserved across refactors.

## Comparison with Similar Vulnerabilities

### Known Issue: Bounty Rent Bug (ECON-07)

**Location:** `epoch_program/trigger_epoch_transition`

**Issue:** Checks `vault_balance >= TRIGGER_BOUNTY_LAMPORTS` but doesn't subtract rent-exempt minimum, potentially making the vault rent-ineligible after transfer.

**Similarity:** Both involve checking vault balance before transfer.

**Difference:** The staking balance check happens AFTER the increment, not before. This is actually SAFER - it validates the POST-STATE rather than PRE-STATE.

## Conclusion

**STR-040 is MITIGATED** by robust defensive coding:

1. **deposit_rewards balance reconciliation** (primary defense) prevents phantom deposits
2. **Tax Program transfer-before-CPI ordering** (defense-in-depth) ensures normal operation safety
3. **claim_rewards solvency check** (last line of defense) prevents vault overdraft even if accounting bugs exist
4. **Protocol-favoring truncation** (mathematical guarantee) ensures total claims cannot exceed deposits

**No code changes required.** The existing implementation is secure against the described attack scenario.

**Risk Level:** NONE (was theoretically MEDIUM, but all attack paths are blocked)
