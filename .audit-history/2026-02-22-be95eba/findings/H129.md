# H129: Epoch Program Validates Tax Program Address

**Strategy:** STR-129: Epoch Program Doesn't Validate Tax Program Address

**Determination:** FINDING REJECTED — No vulnerability exists

**Severity:** N/A

**Confidence:** Very High (100%)

---

## Summary

The hypothesis suggested that the Epoch Program might not validate the Tax Program address when CPIing into Tax for Carnage execution, potentially allowing program substitution. Investigation of the Epoch Program's `execute_carnage_atomic` and `execute_carnage` instructions confirms that **all Tax Program account references are properly validated** using hardcoded address constraints.

---

## Evidence

### Direct Address Validation

**File:** `/programs/epoch-program/src/instructions/execute_carnage_atomic.rs` (lines 166-170)

```rust
// === Programs ===
/// Tax Program (for swap_exempt CPI)
/// CHECK: Address validated against known Tax program ID
#[account(address = tax_program_id() @ EpochError::InvalidTaxProgram)]
pub tax_program: AccountInfo<'info>,
```

**File:** `/programs/epoch-program/src/instructions/execute_carnage.rs` (lines 161-165)

```rust
// === Programs ===
/// Tax Program (for swap_exempt CPI)
/// CHECK: Address validated against known Tax program ID
#[account(address = tax_program_id() @ EpochError::InvalidTaxProgram)]
pub tax_program: AccountInfo<'info>,
```

### Validation Mechanism

The `#[account(address = tax_program_id() @ EpochError::InvalidTaxProgram)]` constraint:

1. **Hardcoded check**: Compares the passed `tax_program` account against `tax_program_id()` at runtime
2. **Anchor-enforced**: This constraint is validated by the Anchor framework during account deserialization, before the handler executes
3. **Enforces equality**: The account MUST exactly match the Tax Program ID, or the transaction is rejected with `InvalidTaxProgram` error
4. **Cannot be bypassed**: The constraint is enforced at the Accounts struct level, not in user-provided logic

### CPI Usage

Both instructions use the validated `tax_program` account when building the CPI instruction:

**execute_carnage_atomic.rs, line 877:**
```rust
let ix = Instruction {
    program_id: ctx.accounts.tax_program.key(),  // Uses validated account
    accounts: account_metas,
    data: ix_data,
};
```

**execute_carnage.rs, line 810:**
```rust
let ix = Instruction {
    program_id: ctx.accounts.tax_program.key(),  // Uses validated account
    accounts: account_metas,
    data: ix_data,
};
```

The CPI instruction uses the validated account's public key as the `program_id`, ensuring only the legitimate Tax Program can be invoked.

### Cross-Program Validation Pattern

Per the architecture document (ARCHITECTURE.md, Section 3.2 "CPI Validation"):

> All CPI targets validated via **hardcoded address constraints** (no dynamic program ID)

This is consistent with the exploit pattern **EP-042 (Arbitrary CPI Program Substitution)**, which is marked **CLEAR** in the context document at line 103:

> Arbitrary CPI Program Substitution | EP-042 | CLEAR | All CPI targets validated via `address =` constraints with hardcoded program IDs
