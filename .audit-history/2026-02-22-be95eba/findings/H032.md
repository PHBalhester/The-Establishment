## Executive Summary

**FINDING STATUS: CLEARED — No vulnerability exists.**

Investigation into potential "faction confusion" where Carnage might burn the WINNING faction token instead of the LOSING faction token revealed that **the protocol does NOT use a "losing faction" concept**. The Carnage mechanism is VRF-driven and completely independent of tax rates, cheap side, or any notion of "winning" or "losing" factions.

The target token is determined purely by VRF byte 7 with 50/50 probability (CRIME if < 128, FRAUD if >= 128), and this target is the token that is **BOUGHT**, not burned. The burn operation applies to **previously held tokens**, not to the target.

## Investigation Methodology

### 1. Threat Model Analysis

**Initial hypothesis:** If the "which faction lost" logic is inverted or mint accounts swapped, Carnage burns the WINNING faction token instead of losing.

**Assumption to validate:** Does Carnage have a concept of "losing faction" tied to tax rates or cheap side?

### 2. Code Flow Traced

#### VRF Byte Interpretation (`helpers/carnage.rs`)

```rust
/// Determine Carnage buy target from VRF byte 7.
///
/// - If byte 7 < 128: returns Token::Crime (50%)
/// - Else: returns Token::Fraud (50%)
///
/// Note: Target is VRF-determined regardless of which token is currently held.
/// This means after a 2% sell, Carnage could immediately rebuy the same token.
pub fn get_carnage_target(vrf_result: &[u8; 32]) -> Token {
    if vrf_result[7] < 128 {
        Token::Crime
    } else {
        Token::Fraud
    }
}
```

**Key finding:** VRF byte 7 determines the **BUY target** with 50/50 probability. There is NO reference to:
- Current cheap side
- Tax rates (cheap/expensive)
- Previous epoch performance
- Pool reserves
- Any "losing faction" concept

## Carnage Lifecycle (Correct Behavior)

### Epoch N (Carnage Triggers, VRF byte 7 = 50 → CRIME target)

1. **VRF reveals:** byte 7 = 50 (< 128) → `target = Token::Crime`
2. **Held state before execution:** `held_token = 2` (FRAUD), `held_amount = 1000`
3. **Action determination:** VRF byte 6 = 200 → `action = Burn` (≥ 5)

**Execution flow:**
1. **Burn held tokens:** Burns 1000 FRAUD tokens (from `held_token = 2`)
2. **Buy target tokens:** Buys CRIME tokens (from `target = Token::Crime`)
3. **Update held state:** `held_token = 1` (CRIME), `held_amount = 5000` (newly bought)

### Epoch N+1 (Carnage Triggers, VRF byte 7 = 200 → FRAUD target)

1. **VRF reveals:** byte 7 = 200 (≥ 128) → `target = Token::Fraud`
2. **Held state before execution:** `held_token = 1` (CRIME), `held_amount = 5000` (from previous epoch)
3. **Action determination:** VRF byte 6 = 100 → `action = Burn`

**Execution flow:**
1. **Burn held tokens:** Burns 5000 CRIME tokens (from `held_token = 1`)
2. **Buy target tokens:** Buys FRAUD tokens (from `target = Token::Fraud`)
3. **Update held state:** `held_token = 2` (FRAUD), `held_amount = 3000` (newly bought)

## False Positive Analysis

The threat model assumed that Carnage implements a "losing faction burns" mechanism tied to tax rates or cheap side. This assumption is **incorrect**.

**Actual mechanism:**
- Carnage buys tokens based on VRF byte 7 (50/50 random)
- Carnage burns PREVIOUSLY HELD tokens (from the last buy)
- There is no concept of "winning" or "losing" factions in Carnage

**Economic impact:**
The VRF-driven 50/50 buy targeting creates balanced long-term demand for both CRIME and FRAUD. Over many epochs, the protocol buys approximately equal amounts of each token. The burn mechanism applies to held tokens, which cycle through both factions, creating symmetric deflationary pressure.

**Design rationale (inferred from spec):**
This design prevents predictable patterns. If Carnage always burned the "losing" faction (expensive side), attackers could:
1. Observe which faction is cheap
2. Know that Carnage will NOT burn it (it's winning)
3. Front-run Carnage buy with confidence

The VRF-driven random targeting prevents this MEV opportunity.

## Conclusion

**CLEARED — NO VULNERABILITY EXISTS.**

The protocol correctly implements VRF-driven Carnage with the following verified properties:

1. **Target selection:** VRF byte 7 determines buy target (50/50 CRIME/FRAUD)
2. **Burn target:** Burns previously held tokens from last Carnage buy
3. **No faction confusion:** Token enum mappings are consistent across all code paths
4. **No dependency on cheap side:** Carnage is independent of tax configuration

**The threat model's "losing faction" assumption does not apply to this protocol.**
