# H023: Pool Lock Not Released on Swap Error

**Category:** State Machine / Reentrancy
**Severity:** NOT VULNERABLE
**Status:** CLEAR
**Date:** 2026-02-22
**Auditor:** STR-023 Investigator (Stronghold of Security)

---

## HYPOTHESIS

If the AMM swap instruction sets `pool.locked = true` early in execution but then panics or errors before setting it back to `false`, the pool could become permanently locked. This would prevent all future swaps on that pool, including Carnage operations, effectively bricking the protocol.

**Attack Vector:** Trigger a failure condition after the lock is set but before it's released. Success depends on:
1. The lock being set via a state write operation
2. An error/panic occurring after lock=true but before lock=false
3. The failed transaction persisting the locked state on-chain

**Impact:** CRITICAL if exploitable — Permanent pool freeze (DoS), Carnage cannot execute, protocol halted

### Solana Transaction Atomicity Guarantee

#### Runtime Behavior

Per Solana's transaction execution model ([docs.solana.com](https://docs.solana.com/developing/programming-model/transactions)):

> **Transactions are atomic.** If any instruction in a transaction fails, the entire transaction fails and all account changes are discarded.

**What this means for state mutations:**

1. When a transaction begins execution, the Solana runtime creates a **snapshot** of all account states
2. Programs make changes to their in-memory copies of account data
3. **Only if the transaction succeeds** are the changes written back to the ledger
4. **If any instruction fails** (panic, `Err(...)`, constraint violation), the runtime:
   - Discards all state changes
   - Reverts accounts to their pre-transaction state
   - Marks the transaction as failed
   - **Does NOT persist any mutations to the blockchain**

#### Expert Confirmation

From [Solana StackExchange #9149](https://solana.stackexchange.com/questions/9149):

> "Since the Solana transactions are atomic in nature, the transaction you posted will fail if the transfer instruction doesn't go through."

From [Solana StackExchange #787](https://solana.stackexchange.com/questions/787):

> "The reason for this is that in the solana runtime if any part of a transaction fails or raises an error, the entire transaction fails. [...] **The transaction never gets included, so there will be no balance changes.**"

#### Applied to Pool Lock Flag

**Scenario 1: Error in CHECKS phase (e.g., line 87 zero-amount check fails)**
- Lock set: Line 84 executes, `pool.locked = true` in transaction context
- Error raised: Line 87 `require!` fails
- Runtime action: Discards transaction context, reverts `pool.locked` to `false`
- On-chain result: Pool remains unlocked, transaction fails, no state persisted

**Scenario 2: Error in EFFECTS phase (e.g., line 173 k-invariant violation)**
- Lock set: Line 84 executed
- Reserves computed: Lines 163-168 executed
- Error raised: Line 173 `require!` fails
- Runtime action: Discards transaction context (lock flag + reserve updates)
- On-chain result: Pool state unchanged, still unlocked

**Scenario 3: Error in INTERACTIONS phase (e.g., CPI transfer fails at line 214)**
- Lock set: Line 84 executed
- Reserves updated: Lines 178-179 executed
- CPI fails: Line 214 `transfer_t22_checked` returns `Err`
- Runtime action: Discards transaction context (all state changes)
- On-chain result: Pool state unchanged, lock still `false`, reserves unchanged

**Scenario 4: Successful execution**
- Lock set: Line 84
- All checks pass: Lines 86-157
- Reserves updated: Lines 176-185
- Transfers complete: Lines 210-315
- Lock cleared: Line 322
- Runtime action: Commits all changes atomically
- On-chain result: Pool unlocked, reserves updated, transfers persisted

### Anchor Constraint Defense-in-Depth

**File:** `swap_sol_pool.rs` line 381

```rust
#[account(
    mut,
    seeds = [POOL_SEED, pool.mint_a.as_ref(), pool.mint_b.as_ref()],
    bump = pool.bump,
    constraint = pool.initialized @ AmmError::PoolNotInitialized,
    constraint = !pool.locked @ AmmError::PoolLocked,  // ← Checked at deserialization
)]
pub pool: Account<'info, PoolState>,
```

The `constraint = !pool.locked` check is evaluated **during account deserialization** (before the handler executes). This provides two layers of protection:

1. **Runtime atomicity:** Failed transactions don't persist state changes (primary defense)
2. **Constraint guard:** Even if the lock somehow got stuck (e.g., via a hypothetical Solana runtime bug), subsequent swap attempts would fail at the constraint check instead of executing with corrupted state

The constraint check is **redundant** given Solana's atomicity guarantee, but provides valuable defense-in-depth.

## ATTACK SCENARIOS EVALUATED

### Scenario 1: Overflow During Reserve Calculation

**Attack:** Provide swap parameters that cause `checked_add` or `checked_sub` to overflow.

**Code Path:**
```rust
let new_reserve_in = reserve_in.checked_add(amount_in).ok_or(AmmError::Overflow)?;  // Line 164
```

**Result:**
- `checked_add` returns `None`
- `ok_or()` converts to `Err(AmmError::Overflow)`
- Handler returns error
- Solana runtime reverts transaction
- Pool lock flag change discarded
- **NOT EXPLOITABLE** - Lock does not persist

### Scenario 2: K-Invariant Violation

**Attack:** Craft a swap that violates `k_after >= k_before`.

**Code Path:**
```rust
require!(k_valid, AmmError::KInvariantViolation);  // Line 173
```

**Result:**
- `require!` macro returns `Err(AmmError::KInvariantViolation)`
- Transaction fails
- All state changes (including lock flag) reverted
- **NOT EXPLOITABLE**

### Scenario 3: CPI Transfer Failure

**Attack:** Trigger a transfer failure (insufficient balance, frozen account, hook rejection).

**Code Path:**
```rust
transfer_t22_checked(...)?;  // Line 214 (or 239, 266, 291)
```

**Result:**
- CPI to Token-2022 fails
- `?` operator propagates error up
- Transaction fails
- All state changes reverted
- **NOT EXPLOITABLE**

### Scenario 4: Slippage Exceeded

**Attack:** Front-run a swap to move the price, causing slippage check to fail.

**Code Path:**
```rust
require!(amount_out >= minimum_amount_out, AmmError::SlippageExceeded);  // Line 145
```

**Result:**
- `require!` fails
- Transaction aborted
- Lock flag change not persisted
- **NOT EXPLOITABLE** (and this is intended behavior - slippage protection)

### Scenario 5: Panic in Solana Runtime

**Hypothetical:** What if the Solana runtime itself panics during transaction execution?

**Solana Design:**
- Runtime panics are caught by the validator's outer execution layer
- Transaction is marked as failed
- No state changes are committed to the ledger
- Validator continues processing other transactions

**Result:** Even runtime panics don't persist partial state changes.

## CONCLUSION

**STATUS: NOT VULNERABLE**

The pool lock flag **cannot** get stuck in a `true` state due to Solana's **transaction atomicity guarantee**.

**Why the hypothesis is refuted:**

1. **Atomicity is absolute:** Solana's runtime does not allow partial state commits. When a transaction fails (for any reason), **all** account mutations are discarded, including the `pool.locked = true` change.

2. **Multiple error paths are intentional:** The swap handler has multiple validation points (zero-amount check, slippage check, k-invariant check, CPI failures). These are **security features**, not bugs. If any validation fails, the transaction aborts and state is reverted.

3. **Lock is never persisted without being cleared:** The only way `pool.locked = true` reaches the ledger is if the transaction completes successfully to line 322, which sets it back to `false`. There is no code path where:
   - Transaction succeeds (commits state), AND
   - Lock is `true` at end of transaction

4. **Defense-in-depth via constraints:** The Anchor `constraint = !pool.locked` check at deserialization provides an additional safety layer. Even in a hypothetical scenario where a Solana runtime bug allowed partial state commits, the constraint would prevent subsequent swaps from executing with corrupted state.

**Impact Assessment:** No exploitable vector exists. The pool cannot be permanently locked via transaction failures.

## RECOMMENDATIONS

**None required.** The implementation correctly relies on Solana's transaction atomicity.

**Positive observations:**

1. ✅ The CEI pattern (lock set first, cleared last) is exemplary
2. ✅ The Anchor constraint provides defense-in-depth
3. ✅ Explicit comments (lines 79-82, 159-160, 317-319) demonstrate security-aware design
4. ✅ The lock flag pattern is consistent across both swap instructions (sol_pool, profit_pool)

**Educational note:**

The lock flag's primary purpose is **reentrancy prevention** (addressed in H017), not error recovery. Solana's atomicity guarantee makes error recovery automatic - failed transactions are simply discarded. The lock flag would only be necessary in a system without atomic transactions (e.g., Ethereum pre-revert), where partial execution could persist.

In Solana's model, the lock flag serves three roles:
1. **Reentrancy defense-in-depth** (Solana runtime already prevents same-program recursion)
2. **Intent signaling** (makes the non-reentrant design explicit to auditors)
3. **Cross-instruction guard** (prevents multiple swap instructions in same transaction)

All three roles are fulfilled correctly.
