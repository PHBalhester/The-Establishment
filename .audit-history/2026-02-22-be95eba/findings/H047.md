# H047: Staking Zero-Amount Deposit - Guard Present

**Strategy:** STR-047
**Determination:** NOT VULNERABLE
**Severity:** N/A (Protection confirmed)
**Confidence:** HIGH

## Summary
The `deposit_rewards` instruction properly validates that `amount > 0` before processing any state updates. Zero-amount deposits are rejected at the instruction entry point, preventing any accounting anomalies or reward calculation issues.

## Evidence

### Primary Guard (deposit_rewards.rs:84)
```rust
pub fn handler(ctx: Context<DepositRewards>, amount: u64) -> Result<()> {
    require!(amount > 0, StakingError::ZeroAmount);

    let pool = &mut ctx.accounts.stake_pool;
    let clock = Clock::get()?;

    // Add to pending (SOL already transferred by Tax Program)
    pool.pending_rewards = pool
        .pending_rewards
        .checked_add(amount)
        .ok_or(StakingError::Overflow)?;
```

## Root Cause / Analysis

The hypothesis was that `deposit_rewards(0)` might:
1. Update accounting without adding rewards
2. Cause `reward_per_token` calculation to skip a period
3. Set `reward_rate` to zero, stopping all reward accrual

**Analysis shows:**
- **Primary defense:** `amount > 0` guard at instruction entry (line 84)
- **Secondary defense:** `update_cumulative` conditional logic (lines 88-105)
- **Tertiary defense:** CPI access control via `seeds::program` constraint prevents unauthorized callers from even attempting this

The Synthetix-style reward math pattern is implemented correctly:
```
reward_per_token = stored + (pending_rewards * PRECISION / total_staked)
```

With the `amount > 0` guard, `pending_rewards` can never be incremented by zero, and the conditional in `update_cumulative` ensures that even if `pending_rewards == 0`, the cumulative calculation is skipped entirely (no state change).

## Impact Assessment

**Impact if vulnerable:** LOW - Accounting anomaly, no direct theft possible

**Actual impact:** NONE - Protection is in place and tested

**Attack complexity:** Would require bypassing multiple layers:
1. CPI access control (`seeds::program` constraint limiting caller to Tax Program)
2. Zero-amount validation (`require!(amount > 0, ...)`)
3. Conditional update logic in `update_cumulative`
