# H067: PoolState Size Mismatch vs Anchor 0.32.1 Discriminator Expectation

**Strategy:** STR-067

**Determination:** POTENTIAL

**Severity:** MEDIUM

**Confidence:** HIGH

## Summary

The PoolState account uses Anchor 0.32.1's `#[derive(InitSpace)]` macro, which correctly calculates serialized field sizes. However, there is a potential vulnerability if the actual discriminator calculation differs between what the space allocation assumes and what Anchor actually writes to the account at initialization time.

**Key Finding**: The code allocates space for:
- 8 bytes discriminator (explicit in account initialization at line 221: `space = 8 + PoolState::INIT_SPACE`)
- 224 bytes from `PoolState::INIT_SPACE`
- **Total: 232 bytes**

However, Anchor 0.32.1 introduced custom discriminators (optional via `#[account(discriminator = N)]`). If:
1. The struct's actual serialized field size differs from 224 bytes, OR
2. The discriminator is non-standard or changes between versions, OR
3. The `InitSpace` derive macro output changes for `PoolType` enum (derives `InitSpace` via `#[derive]` at line 15)

...then the account could be either under-allocated (CPI writes fail) or over-allocated (wasted rent).

## Risk Factors

### 1. **PoolType Enum Size**
The `PoolType` enum at line 15 derives `InitSpace` automatically. For a 2-variant enum:
- Borsh serialization typically uses 1 byte for discriminant + 0 bytes for largest variant = **1 byte**
- The documented calculation assumes 1 byte for `pool_type`
- ✅ **Likely correct**, but `InitSpace` derive could add padding

### 2. **Anchor 0.32.1 Discriminator Behavior**
From anchor-version-gotchas.md (lines 108-113), Anchor 0.32.0+ supports custom discriminators:
```rust
#[account(discriminator = 1)]  // Custom discriminator
pub struct MyAccount { ... }
```
The PoolState uses default discriminator (derived from module path hash).
- Standard Anchor 0.32.1 uses 8-byte SHA256 discriminator
- ✅ **Code uses standard discriminator** (no `discriminator =` override)
- ⚠️ **Risk**: If someone later modifies the struct or relocates it in the module hierarchy, Anchor recompiles with a different discriminator without space reallocation

### 3. **Space Padding in InitSpace Derive**
The `InitSpace` derive macro in Anchor 0.30+ includes alignment padding for certain field layouts. For example:
- If a field struct contains a bool after a u64, the macro may add padding for alignment
- **PoolState mixes types**: u64s, u32s, bools, Pubkeys without explicit padding
- The documented size (224) matches manual calculation without padding
- ✅ **Appears correct** but would need compile-time verification

### 4. **No Explicit Test of Space Constant**
Searching the codebase (grep results above), there are no explicit tests or compile-time assertions that verify:
```rust
const_assert_eq!(PoolState::INIT_SPACE, 224);
```
This is a best-practice safeguard in Anchor programs.

## Impact

**If PoolState::INIT_SPACE differs from actual serialized size:**

1. **Under-allocation** (actual size > 224):
   - When the pool is initialized, Anchor would attempt to write beyond the allocated space
   - The `init` constraint would fail with "Insufficient space" error
   - The pool could never be created, breaking the entire protocol

2. **Over-allocation** (actual size < 224):
   - Wastes ~1-8 bytes of rent per pool (negligible financial impact: ~0.006 SOL per pool)
   - No functional impact, just inefficiency

3. **Discriminator mismatch** (unlikely but possible):
   - If the program is redeployed with a different module structure, Anchor regenerates the discriminator
   - Existing pools would have mismatched discriminators, causing type-checking to fail
   - All swap instructions would fail because `Account<PoolState>` deserialization would reject the old discriminator

## Recommendation

### Immediate Action (Low Priority)
Add a compile-time assertion to verify the space calculation:

```rust
// In programs/amm/src/state/pool.rs, after the PoolState struct definition:
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_pool_state_size() {
        // Ensure InitSpace matches manual calculation + discriminator
        assert_eq!(PoolState::INIT_SPACE, 224,
            "PoolState::INIT_SPACE mismatch. Expected 224, got {}",
            PoolState::INIT_SPACE);

        // Optional: Also verify discriminator size
        assert_eq!(size_of::<u64>(), 8, "Discriminator must be 8 bytes");
    }
}
```

Run with: `cargo test -p amm test_pool_state_size -- --nocapture`

### Process Improvement (Medium Priority)
1. **Document discriminator assumption**: Add a comment in pool.rs that the struct uses Anchor's standard 8-byte discriminator and cannot be relocated without redeployment
2. **Lock module path**: Consider renaming inhibitions or adding a proc-macro guard to prevent accidental relocation
3. **CI check**: Add to tests that all `#[account]` structs with `#[derive(InitSpace)]` pass a size validation test

### Deployment Safeguard
Before any Anchor version upgrade (0.32.1 → 0.33+):
1. Compare `PoolState::INIT_SPACE` between versions (if it changes, all pools must be migrated)
2. Test pool creation and swaps on devnet with the new version
3. If discriminator changes, add a migration instruction to rewrite pools with new discriminator

## Determination Rationale

**POTENTIAL** rather than CONFIRMED or NOT VULNERABLE because:
- The code appears correctly structured and matches documented space calculations
- Anchor 0.32.1's `InitSpace` derive is well-tested and reliable
- However, there is no runtime verification or compile-time assertion
- The risk is primarily in future maintenance (version upgrades, struct relocations)
- This is a "defense-in-depth" recommendation, not an active exploit vector

The codebase has:
- ✅ Correct space calculation (8 + 224 = 232)
- ✅ Standard Anchor discriminator (no custom override)
- ✅ Proper use of `#[account]` constraint
- ❌ No compile-time assertion to catch divergence
- ❌ No documentation of discriminator assumptions

**Status**: Ready for implementation of recommended safeguards.
