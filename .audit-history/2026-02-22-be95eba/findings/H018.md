# H018: Epoch Transition Double-Trigger in Same Slot

**Category:** State Machine / Timing
**Severity:** NOT VULNERABLE
**Status:** Solana runtime protections prevent this attack
**Date:** 2026-02-22
**Investigator:** Stronghold of Security Audit

---

## HYPOTHESIS

If two `trigger_epoch_transition` transactions land in the same slot, both might pass the epoch number check simultaneously, causing double epoch advancement, double reward distribution, or double Carnage execution.

**Attack Vector:**
1. Epoch boundary reached at slot S
2. Attacker submits two `trigger_epoch_transition` transactions in rapid succession
3. Both transactions read `epoch_state.vrf_pending == false` before either writes
4. Both transactions pass the guard `require!(!epoch_state.vrf_pending)`
5. Both transactions increment `epoch_state.current_epoch`
6. Result: Double epoch advancement, potential double Carnage

**Precedent:** EP-037 (Race Condition), EP-038 (Replay), EP-090 (Simultaneous Operation Race)

### Solana Runtime Protections

#### Account Locking Model

Solana's Sealevel execution engine uses **read/write locks** to manage transaction parallelization:

1. **Write Lock Exclusivity:** When a transaction declares an account as **writable** (via `mut`), the Solana runtime acquires an **exclusive write lock** on that account for the duration of transaction execution.

2. **Lock Acquisition Before Execution:** Account locks are acquired **before** the transaction begins executing any instructions. This happens during the transaction scheduling phase, not during instruction execution.

3. **Sequential Execution for Conflicting Writes:** If two transactions both require a write lock on the same account (in this case, `epoch_state`), they **cannot execute in parallel**. The scheduler queues them sequentially.

4. **Atomic State Visibility:** The second transaction attempting to acquire the write lock will **block** until the first transaction completes and releases the lock. When the second transaction finally executes, it will see the **updated state** from the first transaction (i.e., `vrf_pending = true`).

**Source:** [Solana: Architecture & Parallel Transactions](https://chainstack.com/solana-architecture-parallel-transactions/)

#### Intra-Slot Execution Order

Per Solana's parallel processing model:

> "Transactions touching the same writable account must run sequentially. Account locks, compute availability, and thread timing decide who runs first—not submission order."

**Source:** [RPC Fast - Same-Block Execution on Solana](https://rpcfast.com/blog/solana-same-block-execution-rpc-fast)

This means:
- **No simultaneous execution:** Two transactions modifying `epoch_state` cannot execute at the exact same time.
- **Sequential state visibility:** The second transaction sees the state changes from the first.
- **Guard enforcement:** When the second transaction executes, `epoch_state.vrf_pending` will be `true`, causing the `require!(!epoch_state.vrf_pending)` check to fail with `EpochError::VrfAlreadyPending`.

#### Multi-Threaded Scheduler Ordering

From Solana's scheduler implementation:

> "Each thread maintains a queue of transactions waiting for execution. Transactions are randomly assigned to a thread's queue, ordered by priority fee and time, but there is no global ordering—just local ordering in each thread's queue."

**Source:** [Helius - Solana vs Sui Transaction Lifecycle](https://www.helius.dev/blog/solana-vs-sui-transaction-lifecycle)

**Implication:** While the **order** in which the two transactions execute is non-deterministic (either could go first depending on thread assignment and priority), they will **always execute sequentially** due to the write lock on `epoch_state`. The first to execute wins; the second fails the `vrf_pending` guard.

## VERDICT: NOT VULNERABLE

### Why the Attack Fails

The Solana runtime's account locking mechanism prevents the race condition at the **scheduler level**, before either transaction executes:

1. **TX1** submitted at slot S → Scheduler acquires **exclusive write lock** on `epoch_state` → Executes:
   - Checks `!vrf_pending` → **PASS** (false)
   - Sets `vrf_pending = true`
   - Sets `current_epoch = expected_epoch`
   - Releases write lock

2. **TX2** submitted at slot S → Scheduler attempts to acquire **exclusive write lock** on `epoch_state` → **BLOCKED** until TX1 completes → Executes:
   - Reads updated state: `vrf_pending = true`
   - Checks `!vrf_pending` → **FAIL**
   - Transaction aborts with `EpochError::VrfAlreadyPending`

The race condition described in **EP-090 (Simultaneous Operation Race)** applies to protocols where:
- Multiple operations are batched in the **same transaction** (e.g., withdraw + borrow in same IX list), OR
- Health checks are deferred until **after** all operations complete

In Solana, **transactions are atomic units**. Two separate transactions cannot read the same account state and both write to it simultaneously—the runtime enforces sequential execution via write locks.

### Comparison to EP-090 Precedent

**EP-090 (Cyclos, Hubble exploits):**
- Withdraw collateral and borrow in **same transaction** (multiple instructions).
- Health check executed **once** at end of transaction, **after** both operations.
- Both operations see the **same initial state** within the transaction.
- **Mitigation:** Check health **per-instruction**, not per-transaction.

**Dr Fraudsworth:**
- Two **separate transactions**, each containing `trigger_epoch_transition`.
- Solana's write lock ensures they execute **sequentially**.
- The second transaction sees the **updated state** from the first.
- **No mitigation needed:** Solana runtime already enforces sequential consistency.

## ADDITIONAL OBSERVATIONS

### 1. No Same-Transaction Multi-Trigger

The protocol does not expose a mechanism to call `trigger_epoch_transition` multiple times within the **same transaction**. The instruction is permissionless and standalone—there's no composable entry point that could batch multiple triggers.

### 2. Epoch Monotonicity

Even if the double-trigger were somehow possible, the epoch increment logic is based on **slot-derived calculation**, not simple increment:

```rust
let expected_epoch = current_epoch(clock.slot, epoch_state.genesis_slot);
require!(
    expected_epoch > epoch_state.current_epoch,
    EpochError::EpochBoundaryNotReached
);
epoch_state.current_epoch = expected_epoch;
```

If a double-trigger occurred:
- **TX1:** `expected_epoch = 5`, `current_epoch` advances from 4 to 5
- **TX2:** `expected_epoch = 5` (same slot), check `5 > 5` → **FAIL** (`EpochBoundaryNotReached`)

The epoch number is **deterministically derived** from `(slot - genesis_slot) / SLOTS_PER_EPOCH`, not incremented. Two transactions in the same slot would calculate the **same** `expected_epoch`, causing the second to fail the boundary check.

### 3. VRF Anti-Reroll Binding

The VRF commit flow includes an anti-reroll mechanism (line 186):

```rust
epoch_state.pending_randomness_account = ctx.accounts.randomness_account.key();
```

Even if two triggers somehow advanced the epoch twice, the second trigger would bind a **different randomness account**, which would fail validation in `consume_randomness` (line 154-157):

```rust
require!(
    ctx.accounts.randomness_account.key() == epoch_state.pending_randomness_account,
    EpochError::RandomnessAccountMismatch
);
```

The protocol has **defense-in-depth** against double-triggering:
1. **Runtime-level:** Solana's write locks (scheduler enforcement)
2. **Instruction-level:** `vrf_pending` boolean guard
3. **Epoch-level:** Slot-derived epoch calculation
4. **VRF-level:** Randomness account binding

## SECURITY POSTURE

**Strengths:**
- ✅ Solana's account locking provides automatic protection against simultaneous writes
- ✅ Multiple layers of defense (runtime, boolean guard, epoch math, VRF binding)
- ✅ Epoch calculation is deterministic from slot number, not stateful increment

**False Positive Pattern:**
- **FP-008 from common-false-positives.md:** "Reentrancy possible via CPI" is marked as partially safe because Solana limits CPI depth and requires upfront account declaration. The same principles apply here—Solana's transaction model enforces sequential execution for conflicting writes.

**Not a Finding:**
This is a **common false positive** when auditing Solana programs with traditional smart contract race condition patterns. The Solana runtime's account locking model fundamentally prevents this class of attack at the scheduler level.

## CONCLUSION

**Status:** NOT VULNERABLE

The hypothesis of double epoch transition via simultaneous same-slot transactions is **prevented by Solana's runtime architecture**. The write lock on `epoch_state` enforces sequential execution, ensuring the second transaction always sees the updated `vrf_pending = true` state and fails the guard check.

The protocol's multi-layered defense (runtime locks + boolean guards + deterministic epoch math + VRF binding) provides robust protection against this attack vector.

**No remediation required.**
