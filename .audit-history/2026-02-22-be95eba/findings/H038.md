# STR-038: Token-2022 Transfer Fee Extension Interaction

**Strategy ID:** STR-038
**Severity:** N/A
**Status:** Not Applicable
**Investigated:** 2026-02-22

---

## Summary

Investigation into whether CRIME/FRAUD/PROFIT mints use the Token-2022 TransferFee extension, which could cause tax calculation discrepancies if fees are deducted before the transfer hook executes.

**Result:** The protocol's Token-2022 mints do NOT have the TransferFee extension enabled. This attack vector is **Not Applicable**.

---

## Strategy Description

**Target:** Token mint configurations, tax calculation logic
**Impact (if applicable):** MEDIUM
**Hypothesis:** If tokens have the TransferFee extension, fees would be deducted BEFORE the transfer hook sees the amount, causing the tax program's calculations to be off. This could lead to:
- Tax calculations based on pre-fee amounts instead of post-fee amounts
- Accounting mismatches between expected and actual received amounts
- User confusion about effective tax rates

**Attack scenario (if extension were present):**
1. User initiates transfer of 1000 tokens
2. TransferFee extension deducts 5% (50 tokens) → recipient receives 950 tokens
3. Transfer hook executes tax logic on the 950 tokens (post-fee amount)
4. Tax program expects to calculate tax on 1000 tokens (pre-fee amount)
5. Result: Tax is calculated on the wrong base amount

### 4. Token-2022 Extension Interaction Analysis

From `.audit/kb/solana/token-extensions.md`:

> **Transfer Fee Config**
> **What:** Mint authority can charge a fee on every transfer. Fee is deducted from the transferred amount.
> **Attack surface:**
> - Fees are deducted from the RECIPIENT's received amount, not the sender's sent amount.
> - Programs that assume `amount_sent == amount_received` will have accounting errors
> - Must use `TransferCheckedWithFee` or precalculate with `calculate_inverse_fee` for exact amounts

**Tax program's current assumption:**
From `.audit/context/05-token-economic.md`:

> "All AMM swaps follow checks-effects-interactions: validate, update state, then CPI transfers"

The tax program calculates tax amounts BEFORE the transfer occurs and assumes the transfer amount matches the calculated amount. This is correct for mints WITHOUT TransferFee, but would break if the extension were present.

## Risk Assessment

**Current Risk:** NONE
**Future Risk:** LOW (if mints are recreated with TransferFee on mainnet)

### Why N/A
The TransferFee extension is a **mint-level extension** that must be set at initialization and cannot be added later (Token-2022 extensions are immutable after initialization). The current mints do not have this extension and never will.

### Future Deployment Considerations
If the protocol decides to redeploy with TransferFee extension on mainnet:

**Required changes:**
1. Use `transfer_checked_with_fee` instead of `transfer_checked` in all transfer helper functions
2. Calculate tax on post-fee amounts using `calculate_inverse_fee` for exact amounts
3. Update AMM swap output calculations to account for transfer fees
4. Add fee validation logic to ensure fees don't exceed expected bounds
5. Update documentation to explain effective tax rates include both protocol tax AND transfer fees

**Example fix (if extension were added):**
```rust
// Current: calculate tax on full amount
let tax = calculate_tax(amount, tax_bps)?;
let net_amount = amount - tax;

// With TransferFee: calculate tax on post-fee amount
let transfer_fee = calculate_fee(amount, &mint_account)?;
let post_fee_amount = amount - transfer_fee;
let tax = calculate_tax(post_fee_amount, tax_bps)?;
let net_amount = post_fee_amount - tax;
```

---

## Conclusion

**STR-038 is NOT APPLICABLE to the current protocol implementation.**

The CRIME, FRAUD, and PROFIT mints do NOT have the `TransferFeeConfig` extension enabled. The protocol's tax calculation logic correctly assumes `amount_sent == amount_received`, which is valid for the current mint configuration.

**No remediation required.**

### Mainnet Checklist Item
This finding should be added to the mainnet deployment checklist:

- [ ] Verify CRIME/FRAUD/PROFIT mints on mainnet do NOT have `TransferFeeConfig` extension (unless intentionally added with corresponding code changes)
- [ ] If TransferFee extension is desired, implement fee-aware tax calculations before deployment
