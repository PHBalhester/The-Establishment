# H031: Epoch Number Overflow Analysis

**Finding ID:** STR-031
**Severity:** INFO
**Status:** Not Exploitable
**Category:** Integer Overflow
**Location:** `programs/epoch-program/src/state/epoch_state.rs:57`, `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:81`
**Auditor:** Security Investigator
**Date:** 2026-02-22

---

## Summary

The epoch number is stored as `u32` in `EpochState.current_epoch` and computed via downcast from `u64` in `trigger_epoch_transition.rs`. After 2^32 epochs (4,294,967,296 epochs), the epoch number would wrap to 0, potentially resetting the state machine to its initial configuration.

**Impact Assessment:** INFO (not exploitable in practice)

The time required to reach u32 overflow is astronomically long under both devnet and mainnet configurations, making this a theoretical concern only.

---

## Technical Details

### Vulnerable Code

**EpochState definition** (`epoch_state.rs:57`):
```rust
/// Current epoch number (0-indexed).
/// Increments each time trigger_epoch_transition succeeds.
pub current_epoch: u32,
```

**Epoch calculation** (`trigger_epoch_transition.rs:81`):
```rust
pub fn current_epoch(slot: u64, genesis_slot: u64) -> u32 {
    ((slot.saturating_sub(genesis_slot)) / SLOTS_PER_EPOCH) as u32
}
```

### Constants

**Devnet:**
```rust
pub const SLOTS_PER_EPOCH: u64 = 750;
```

**Mainnet:**
```rust
pub const SLOTS_PER_EPOCH: u64 = 4_500;
```

## Exploitation Analysis

### Time to Overflow (Devnet)

- **u32::MAX:** 4,294,967,295 epochs
- **Slots per epoch:** 750
- **Total slots:** 4,294,967,295 × 750 = 3,221,225,471,250 slots
- **Slot time:** ~400ms (0.4 seconds)
- **Total time:** 3,221,225,471,250 × 0.4s = 1,288,490,188,500 seconds
- **In years:** 1,288,490,188,500s ÷ (60 × 60 × 24 × 365.25) ≈ **40,843 years**

### Time to Overflow (Mainnet)

- **u32::MAX:** 4,294,967,295 epochs
- **Slots per epoch:** 4,500
- **Total slots:** 4,294,967,295 × 4,500 = 19,327,352,827,500 slots
- **Slot time:** ~400ms (0.4 seconds)
- **Total time:** 19,327,352,827,500 × 0.4s = 7,730,941,131,000 seconds
- **In years:** 7,730,941,131,000s ÷ (60 × 60 × 24 × 365.25) ≈ **245,059 years**

### State Machine Impact if Overflow Occurred

If the epoch counter wrapped to 0, the following would occur:

1. **`current_epoch` resets to 0** — the protocol would appear to restart from epoch 0
2. **`epoch_start_slot` recalculation** — would become incorrect (genesis_slot + 0 × SLOTS_PER_EPOCH = genesis_slot), causing epoch boundary detection to fail
3. **VRF state confusion** — `last_carnage_epoch` (also u32) could wrap independently or in sync, causing carnage frequency tracking to break
4. **Staking distribution** — The staking program uses `update_cumulative` with epoch comparison: `require!(epoch > pool.last_update_epoch)`. If epoch wraps to 0 while `last_update_epoch` is high, this check would fail, blocking all reward distributions permanently

**Severity if exploitable:** CRITICAL (complete protocol lockup)
**Actual severity:** INFO (unreachable in practice)

## Architectural Context

### Why u32 Instead of u64?

The epoch number uses `u32` to save account storage space. The `EpochState` account is exactly 108 bytes (8-byte discriminator + 100 bytes data). Upgrading `current_epoch` from u32 to u64 would increase the account size by 4 bytes, requiring rent recalculation and redeployment.

Given that:
1. The protocol programs are **non-upgradeable** (no upgrade authority)
2. The overflow time exceeds the heat death of the universe
3. Account storage costs are permanent on Solana (rent-exempt minimum)

The u32 choice is architecturally sound.

### Other u32 Fields

The protocol also uses u32 for `last_carnage_epoch` in `EpochState` (line 140). This field tracks when carnage last occurred and is compared to `current_epoch`. If both wrap simultaneously, the comparison logic remains consistent. If they wrap at different times (impossible given the timeframe), the comparison `current_epoch > last_carnage_epoch` could temporarily fail, preventing carnage for one epoch.

## False Positive Classification

**Pattern:** EP-033 (Integer Truncation in State Transition)
**Verdict:** FALSE POSITIVE (theoretical only)

The `as u32` cast is silent truncation, which violates the checked-arithmetic principle. However, the value being truncated is `(slot - genesis_slot) / SLOTS_PER_EPOCH`, which is bounded by:

- Maximum slot number in Solana: u64::MAX = 18,446,744,073,709,551,615
- Minimum slots per epoch: 750
- Maximum epoch number: 18,446,744,073,709,551,615 ÷ 750 = 24,595,658,764,946,068

This exceeds u32::MAX by ~5.7 million times. However, reaching even u32::MAX would take 40,000+ years.

**Mitigation:** The protocol's `saturating_sub` prevents underflow if `genesis_slot > current_slot` (impossible after initialization), ensuring the division input is always valid.

## Recommendations

### For Current Deployment

**No action required.** The overflow condition is unreachable within any conceivable operational timeframe.

### For Future Protocols

If designing a similar system with shorter epoch durations (e.g., hourly epochs = 9,000 slots/epoch on mainnet), u32 overflow becomes reachable in ~4,760 years. For protocols intended to operate on century+ timescales (e.g., on-chain time capsules, multi-generational trusts), use `u64` for all epoch counters.

**Pattern:** Checked cast with explicit overflow handling:
```rust
pub fn current_epoch(slot: u64, genesis_slot: u64) -> Result<u32> {
    let epoch_u64 = (slot.saturating_sub(genesis_slot)) / SLOTS_PER_EPOCH;
    u32::try_from(epoch_u64).map_err(|_| EpochError::EpochOverflow)
}
```

This would make the overflow explicit rather than silent, allowing graceful degradation (e.g., freezing at u32::MAX instead of wrapping).

## Conclusion

**Verdict:** INFO — Not Exploitable

The epoch number overflow condition exists in theory but is unreachable in practice. The time required to trigger the overflow (40,000+ years on devnet, 240,000+ years on mainnet) exceeds the operational lifetime of any blockchain system. The u32 storage choice is appropriate for the protocol's design constraints.

If this protocol were expected to operate for geological timescales, upgrading to u64 would be advisable. For the current use case, the INFO severity is appropriate and no remediation is required.
