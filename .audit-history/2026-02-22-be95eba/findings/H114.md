# H114: Phantom add_liquidity Instruction Does Not Exist

**Strategy:** STR-114: AMM Add Liquidity at Arbitrary Price

**Determination:** INSTRUCTION DOES NOT EXIST — The claimed `add_liquidity` instruction in ARCHITECTURE.md has no implementation in the actual codebase.

**Severity:** INFORMATIONAL

**Confidence:** 100% (verified via codebase analysis)

---

## Summary

The security architecture document (`.audit/ARCHITECTURE.md`) lists `add_liquidity` as an admin-gated instruction in the AMM program (line 145 of Instruction Map). However, a complete audit of the AMM program reveals:

1. **No `add_liquidity` instruction exists** in any source file
2. **The only liquidity provision mechanism is `initialize_pool`**, which creates a pool with seed liquidity in a single atomic transaction
3. **No post-initialization liquidity management exists** — pools cannot be topped up, drained, or rebalanced by any admin action

This finding is a **documentation error** rather than a security vulnerability. However, it highlights a critical architectural constraint that deserves documentation: **protocol-owned pools are immutable after creation**.

## Impact

### Positive: No add_liquidity Vulnerability

The **absence** of `add_liquidity` eliminates the hypothesized attack vector (STR-114):
- Users cannot add liquidity at arbitrary price points
- Pool reserves cannot be manipulated post-creation
- No LP token inflation attacks are possible (no LP tokens exist)
- No slippage exploitation via malicious liquidity additions

### Negative: Architectural Constraint

The immutability of pools creates an operational constraint:
- **Pool drains over time** due to protocol-owned reserves being consumed by Carnage swaps cannot be refilled
- If a pool's reserves drop too low, the swaps may fail due to output constraints
- The protocol does not support liquidity replenishment

This is mitigated by:
1. **Protocol-owned pools** — Admin can redeploy with new seed amounts if needed (requires program upgrade)
2. **Conservative Carnage slippage floors** — 85% atomic, 75% fallback prevents excessive reserve consumption
3. **Known crank operation** — The crank runner manages carnage execution and has incentive to monitor pool health

### Documentation Issue: Architecture Document Mismatch

The `.audit/ARCHITECTURE.md` synthesized by audit agents claims `add_liquidity` as a valid instruction (line 145). This is a **first-order error** in the document synthesis process, indicating that:
- The agent analyzed outdated documentation or previous design iterations
- The current codebase reflects a **final design** where pools are immutable after seed
- **No vulnerability exists**, but the discrepancy suggests the architecture document is not authoritative

1. **Update `.audit/ARCHITECTURE.md` line 145** to remove `add_liquidity`:
   ```diff

   ```

2. **Add a design note** to document the immutability decision:
   ```markdown
   ### Pool Immutability (by design)

   Pools created by `initialize_pool` are immutable. No instruction exists to:
   - Add post-initialization liquidity
   - Modify LP fees
   - Withdraw reserves
   - Close pools

   This design eliminates LP token extraction attacks and ensures pools
   behave as stable market-making infrastructure. If pool health degrades,
   the protocol must redeploy with fresh pools (requiring upgrade authority).
   ```

3. **Document the pool drain concern** — Carnage execution periodically consumes reserves. This is by design but should be explicitly noted:
   ```markdown
   ### Carnage Impact on Reserves

   Epoch Carnage executes swaps against all pools. With long-running protocols,
   reserves may deplete over time. The conservative slippage floors (85%-75%)
   reduce this risk. Pool monitoring is a crank operator responsibility.
   ```

The `add_liquidity` instruction claimed in ARCHITECTURE.md is **phantom**. The actual AMM program implements a **pool-immutability-first** design where:
