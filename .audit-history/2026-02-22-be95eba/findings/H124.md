# H124: Admin Key is Deployer Wallet (Hot Wallet Risk)

**Strategy:** STR-124

---

## Determination

**CONFIRMED WITH MITIGATION FACTOR**

The hypothesis is technically correct: the admin key used for protocol initialization is the same keypair as the deployer wallet (the one that signs the `initializeAdmin` instruction). However, the **architectural mitigation is strong**:

1. The admin's only power post-deployment is pool creation (a one-time activity)
2. The admin can be set to ANY pubkey (including a multisig) via the `admin` parameter of `initializeAdmin`
3. The admin **must be burned** after all pools are created, permanently freezing pool creation
4. The design explicitly separates upgrade authority (deployer) from admin (can be different keys)

**Current Status (Devnet):** The devnet wallet (`8kPzh...`) is used as the upgrade authority signer in `initialize.ts:165`, and is passed directly as the `admin` parameter in `initialize.ts:372`. This means deployer = admin on devnet, which is acceptable for a development environment.

**Mainnet Risk:** If the same pattern is replicated on mainnet without burning the admin key, or if the mainnet keypairs are committed to git, this becomes a **HIGH** severity finding.

**HIGH (if replicated on mainnet without mitigation)**

## Summary

The Dr. Fraudsworth protocol delegates pool creation authority to an `AdminConfig` PDA initialized by the program's upgrade authority. The initialization script (`initialize.ts:165-372`) uses the provider wallet (derived from `Anchor.toml:29`) as both:

1. The authority that calls `initializeAdmin` (must be upgrade authority)
2. The admin parameter passed to `initializeAdmin` (who gates pool creation)

This means the deployer wallet becomes the admin by default on devnet. This is a typical pattern for development but creates a hot wallet risk if:

- The same approach is used on mainnet
- The mainnet keypair is committed to git (like devnet keypairs currently are)
- The admin is not burned after pools are created
- The admin key is compromised

**The protocol's mitigation is strong:** the admin has extremely limited powers (pool creation only), can be set to a multisig at initialization time, and **must be burned** before mainnet to prevent ongoing privileges.

## Impact

### On Devnet (Current)
- **ACCEPTABLE**: The devnet wallet is a test key with no real value. The admin's limited powers (pool creation only) are appropriate for a test environment. Once pools are created and admin is burned, the key's compromise has zero impact.

### On Mainnet (If Deployer Key Model Replicated)
- **HIGH SEVERITY**: If the mainnet deployer wallet is used as the admin, and:
  1. The keypair is committed to git or leaked in any way → Attacker can create pools with malicious fee structures (up to 5%)
  2. The admin is not burned after deployment → Ongoing risk of pool creation attacks
  3. The deployer key is a single EOA, not a multisig → Single point of failure for entire upgrade authority chain

- **CRITICAL if combined with other risks**: If the admin key is the same as the upgrade authority key and that key is compromised, an attacker can:
  1. Create pools with manipulated fee structures
  2. Upgrade any program via the upgrade authority
  3. Disable swaps and staking until admin is burned

### Missing Operational Security Measures
- The mainnet checklist (`Docs/mainnet-checklist.md:123-124`) notes that mainnet keypairs should NOT be committed to git, but this is not enforced or verified in the deployment process
- No deployment validation that admin is burned before going live
- No monitoring to detect if admin is still active post-deployment

## Recommendation

### Phase 52 (Current Development)
1. **Document the intended deployment flow** in `Docs/deployment-phases.md`:
   - Step A: Deploy programs (upgrade authority = Squads multisig)
   - Step B: Call `initializeAdmin` with multisig address (not deployer wallet)
   - Step C: Create all pools using the multisig
   - Step D: Burn admin (irreversible)
   - Step E: Verify admin is burned before mainnet cutover

2. **Create a mainnet-deployment checklist item**:
   ```
   [ ] Upgrade authorities: Set to 3-of-5 Squads multisig for all 5 programs
   [ ] Admin key: Set to separate multisig (different from upgrade authority)
   [ ] Keypairs: All mainnet keypairs loaded from env vars, NEVER committed to git
   [ ] Verification: Query AdminConfig.admin on all programs before going live
   [ ] Burn step: Add post-pool-creation task to burn admin via multisig
   ```

3. **Add deployment validation** to `initialize.ts`:
   ```typescript
   // Add option: --init-admin <multisig-address>
   // Require env var MAINNET_ADMIN_MULTISIG (fail if not set on non-devnet)
   // Log the admin pubkey prominently after init
   const initAdminAddress = process.env.MAINNET_ADMIN_MULTISIG || authority.publicKey;
   if (process.env.CLUSTER_URL?.includes("mainnet") && initAdminAddress === authority.publicKey) {
     throw new Error("MAINNET_ADMIN_MULTISIG not set! Using deployer wallet as admin is unsafe.");
   }
   ```

### Before Mainnet Launch
1. **Separate upgrade authority from admin authority**:
   - Upgrade Authority → Squads multisig (3-of-5 trusted team members)
   - Admin Authority → Separate Squads multisig (or burn immediately post-deployment)

2. **Enforce .gitignore**:
   ```
   # Add to .gitignore before mainnet branch
   keypairs/
   *.json  # Or more specifically: keypairs/*.json
   .env*
   ```

3. **Add pre-deployment checks**:
   - Verify no mainnet keypairs in git history
   - Verify upgrade authority is a multisig (check via chain), not a single key
   - Verify admin is burned (AdminConfig.admin == Pubkey::default()) before cutover

4. **Add post-deployment monitoring**:
   - Alert if AdminConfig.admin changes (should be immutable after burn)
   - Alert if upgrade authority changes (should be immutable)
   - Monitor for unexpected pool creation attempts

### False Positive Mitigation

This is NOT a false positive because:
- The deployer wallet IS used as admin on devnet (confirmed in code)
- The mainnet checklist has a TODO to decide upgrade authority strategy (UA-005)
- No evidence exists that mainnet deployment will use a multisig (yet)

However, this is EASILY MITIGATED by:
- Passing a multisig address to `initializeAdmin` instead of the deployer wallet
- Burning the admin after pool creation (already supported in code)

## Key Takeaway

The admin key being the deployer wallet is a **pattern choice, not a flaw**. The architecture supports separating them (and strongly recommends it for mainnet). The mitigation is built into the code (burn mechanism is irreversible and tested). The risk is **operational**: forgetting to burn the admin or using a single-key deployer wallet for mainnet launch.

**Action Required:** Define and document the intended separation of roles before mainnet. The code is ready to support multisig admins; the deployment scripts just need to be configured to use them.
