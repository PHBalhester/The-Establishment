# H045: Carnage Sell Step Protected Against Zero Output by Buy Step Pool Skew

**Strategy:** STR-045
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary

The hypothesis that Carnage's sell step could produce zero output after buying the losing token is **NOT VULNERABLE**. The buy step creates pool conditions that are the opposite of those required for zero-output swaps. Additionally, multiple layers of protection exist: AMM-level zero-output checks, slippage floors (75%/85%), and the constant-product formula's inherent properties.

### 3. Mathematical Analysis: Buy Step Prevents Zero-Output Sell

**Zero-output condition (from AMM tests):**
A swap produces zero output when: `reserve_out * effective_input < reserve_in`

Example from test (`test_swap_profit_pool.rs` lines 1609-1673):
- Pool: 1,000,000,000,000 tokens in : 1,000 SOL out
- Swap: 1,000 tokens → 0 SOL output (integer division truncates)

**Carnage flow creates the OPPOSITE pool state:**

1. **Initial pool (balanced):** 100 SOL : 100M tokens
2. **After Carnage buy (1000 SOL):**
   - SOL reserve: 100 + 1000 = 1,100 SOL (INCREASES)
   - Token reserve: 100M - 90.8M = 9.1M tokens (DECREASES)
   - Pool skew: 11x more SOL, only 9% tokens remaining

3. **Sell step (tokens → SOL):**
   - reserve_in = 9.1M tokens (SMALL)
   - reserve_out = 1,100 SOL (LARGE)
   - Selling 45M tokens → 915 SOL output

**This is the inverse of zero-output conditions.** The buy step creates a pool where:
- The output reserve (SOL) is MUCH larger than initial
- The input reserve (tokens) is MUCH smaller than initial
- The denominator is small, numerator is large → guaranteed significant output

## Root Cause / Analysis

The vulnerability hypothesis assumed that:
1. Carnage buys heavily, skewing the pool
2. The sell step then operates in an unfavorable pool state
3. The sell produces zero or near-zero output

**This assumption is incorrect.** The Carnage buy step skews the pool in a way that **FAVORS** the subsequent sell operation:

- **Buy direction:** SOL → tokens (increases SOL reserve, decreases token reserve)
- **Sell direction:** tokens → SOL (small input reserve, large output reserve)

For a sell to produce zero output, we would need:
- Large token reserve (what we're selling FROM)
- Small SOL reserve (what we're buying)

But Carnage's buy step creates:
- Small token reserve (Carnage removed most tokens)
- Large SOL reserve (Carnage added 1-1000 SOL)

**The buy step creates the exact opposite pool state needed for zero-output sells.**

Additionally, three layers of protection exist:

1. **AMM-level check:** `check_swap_output_nonzero` rejects any zero-output swap
2. **Carnage slippage floor:** 75-85% minimum output requirement
3. **MAX_CARNAGE_SWAP_LAMPORTS cap:** Limits buy to 1000 SOL, preventing unbounded pool manipulation

## Impact Assessment

**Actual impact:** NONE - The vulnerability does not exist. The Carnage buy step creates pool conditions that prevent zero-output sells, and multiple protection layers exist.

## Remediation

**No remediation required.** The current implementation is secure against this attack vector.

**Existing protections are sufficient:**

1. AMM zero-output check (`AmmError::ZeroSwapOutput`)
2. Carnage slippage floors (75%/85%)
3. MAX_CARNAGE_SWAP_LAMPORTS cap (1000 SOL)
4. Pool state dynamics (buy step creates favorable sell conditions)

**Optional enhancement (informational):**
If paranoid about edge cases, could add an explicit check in the sell step to verify `sol_from_sale > 0` before proceeding. However, this is redundant given the AMM's existing protection and would add unnecessary compute overhead.

**Mathematical proof:**
For zero output: `reserve_out * effective_input < reserve_in`

After Carnage buy of X SOL:
- reserve_in (tokens) = initial_tokens - bought_tokens (DECREASES)
- reserve_out (SOL) = initial_sol + X (INCREASES, X up to 1000 SOL)

Selling Y tokens back:
- output = reserve_out * Y / (reserve_in + Y)
- Since reserve_out is LARGE and reserve_in is SMALL, output cannot be zero unless Y itself rounds to zero after fees

The AMM's `check_effective_input_nonzero` prevents Y from being zero when input is non-zero.

**Conclusion:** The attack vector is mathematically impossible given the Carnage execution flow.
