# S009: Carnage SOL Vault is NOT Shared — Dedicated Single-Purpose Account

**Strategy:** S009
**Determination:** FALSE POSITIVE
**Severity:** N/A
**Confidence:** HIGH

## Summary

**Hypothesis:** If the Carnage SOL vault is also used for other purposes (staking rewards, treasury), draining it for bounty affects those functions too. Conversely, other deposits may mask the bounty danger zone.

**Finding:** The Carnage SOL vault (`carnage_sol_vault` PDA with seeds `["carnage_sol_vault"]`) is a **dedicated, single-purpose account** that is NOT shared with any other protocol function. The protocol uses **three separate vault PDAs** for different purposes:

1. **Carnage SOL Vault** (`["carnage_sol_vault"]`) — Epoch Program — receives 24% of trading tax, funds epoch trigger bounties, funds carnage buy operations
2. **Staking Escrow Vault** (`["escrow_vault"]`) — Staking Program — receives 75% of trading tax, pays staking rewards to users
3. **Treasury** (externally-controlled address) — receives 1% of trading tax

There is **no cross-contamination** between these vaults. The hypothesis that draining the carnage vault affects staking or that staking deposits mask carnage bounty issues is **FALSE**.

```rust
pub fn split_distribution(total_tax: u64) -> Result<TaxDistribution> {

        .checked_mul(75)
        .ok_or(TaxError::Overflow)?
        .checked_div(100)

        .checked_div(100)

        .checked_sub(staking_portion)
        .ok_or(TaxError::Underflow)?
        .checked_sub(carnage_portion)
        .ok_or(TaxError::Underflow)?;
    // ... (returns TaxDistribution struct)
}
```

**Distribution flow:**
```
User pays tax → Tax Program splits →
  ├─ 75% → staking_escrow (Staking Program PDA)
  ├─ 24% → carnage_vault (Epoch Program PDA)
  └─  1% → treasury (external wallet)
```

Each portion is transferred via separate `system_instruction::transfer` calls to **three distinct addresses**. There is no commingling of funds.

## Impact Assessment

**Original Concern:**
> "Draining carnage vault for bounty affects staking functions"

**Reality:** Staking functions use `escrow_vault` (seeds `["escrow_vault"]`), which has a completely separate balance. Draining `carnage_sol_vault` has **zero impact** on staking rewards.

**Original Concern:**
> "Other deposits may mask the bounty danger zone"

**Reality:** The carnage vault only receives 24% of trading taxes. Staking receives 75%, but to a **different PDA**. There is no "masking" because the accounts are separate. The known bounty rent-exempt bug (H001) is real, but it's isolated to the carnage vault's own balance, not hidden by staking deposits.

## Related Findings

**H001: Bounty Transfer Can Drain Vault Below Rent-Exempt Minimum**
- Status: Confirmed CRITICAL vulnerability
- Location: `trigger_epoch_transition.rs:194-227`
- Issue: Bounty check doesn't account for rent-exempt minimum
- **Relevance to S009:** H001 is a real issue with the carnage vault, but it is NOT caused by or related to vault sharing. The bug exists because the balance check is incomplete, not because the vault is shared.

**TE-001: `constraint = true` Placeholders on carnage_vault in Tax Program**
- Status: Confirmed MEDIUM vulnerability
- Location: `swap_sol_buy.rs:447`, `swap_sol_sell.rs:586`
- Issue: Tax Program accepts any account as `carnage_vault` (no PDA seed validation)
- **Relevance to S009:** This is a validation bug, not a sharing bug. An attacker could redirect the 24% tax to a wrong account, but that doesn't mean the legitimate carnage vault is shared with other functions.

## Conclusion

The hypothesis that the Carnage SOL vault is shared with other protocol functions is **FALSE**. The protocol uses a clean separation-of-concerns architecture with three independent vaults:

- **Carnage vault** (24% of tax) → Epoch trigger bounties + carnage operations
- **Staking escrow** (75% of tax) → User staking rewards
- **Treasury** (1% of tax) → Protocol revenue

There is **no cross-contamination**, **no deposit masking**, and **no shared-vault risk**. The known issues (H001 rent-exempt bug, TE-001 validation bug) are isolated to the carnage vault's own logic and do not involve other protocol vaults.

**Determination:** Strategy S009 yields no new findings. The carnage vault is architecturally sound as a dedicated account.
