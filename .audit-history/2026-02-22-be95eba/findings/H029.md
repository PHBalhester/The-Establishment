# H029: consume_randomness Cannot Be Called Multiple Times Per Epoch

**Severity:** NOT VULNERABLE (False Positive)
**Category:** Oracle / VRF / State Machine
**Program:** Epoch Program
**Finding Type:** Security Analysis - No Exploit Found

---

## Summary

The investigation target STR-029 hypothesized that `consume_randomness` could be called multiple times per epoch to re-roll tax rates if a `randomness_consumed` flag was not properly set atomically. **This attack vector does not exist.** The protocol uses a different, more comprehensive anti-reroll protection mechanism based on the `vrf_pending` flag and randomness account binding.

---

## Technical Analysis

### VRF Lifecycle State Machine

The protocol uses a three-transaction VRF flow:

```
TX 1: Create Switchboard randomness account
TX 2: commitIx + trigger_epoch_transition
TX 3: revealIx + consume_randomness
```

### Anti-Reroll Protection Mechanisms

The protocol implements **five layers** of protection against randomness re-rolling:

#### 1. VRF Pending Flag (Primary Protection)

**File:** `epoch-program/src/instructions/consume_randomness.rs:151`
```rust
// === 1. Validate VRF is pending ===
require!(epoch_state.vrf_pending, EpochError::NoVrfPending);
```

**File:** `epoch-program/src/instructions/consume_randomness.rs:206-207`
```rust
// === 7. Clear VRF pending state ===
epoch_state.vrf_pending = false;
epoch_state.taxes_confirmed = true;
```

The `vrf_pending` flag is:
- **Set to `true`** in `trigger_epoch_transition` (line 181)
- **Checked at entry** in `consume_randomness` (line 151)
- **Cleared to `false`** in `consume_randomness` after tax derivation (line 206)

This creates an **atomic once-per-epoch consumption guarantee**. After `consume_randomness` executes once, `vrf_pending = false`, and any subsequent call to `consume_randomness` will fail at line 151 with `EpochError::NoVrfPending`.

#### 2. Randomness Account Binding (Anti-Reroll)

**File:** `epoch-program/src/instructions/trigger_epoch_transition.rs:186`
```rust
// === 6. Bind randomness account (anti-reroll protection) ===
epoch_state.pending_randomness_account = ctx.accounts.randomness_account.key();
```

**File:** `epoch-program/src/instructions/consume_randomness.rs:154-157`
```rust
// === 2. Anti-reroll: verify SAME randomness account that was committed ===
require!(
    ctx.accounts.randomness_account.key() == epoch_state.pending_randomness_account,
    EpochError::RandomnessAccountMismatch
);
```

The randomness account public key is bound at commit time (TX 2) and verified at consume time (TX 3). An attacker cannot substitute a different randomness account to get different random bytes.

#### 3. Double-Commit Prevention

**File:** `epoch-program/src/instructions/trigger_epoch_transition.rs:149`
```rust
// === 2. Validate no VRF already pending ===
require!(!epoch_state.vrf_pending, EpochError::VrfAlreadyPending);
```

Cannot trigger a new epoch transition while a VRF request is pending. This prevents creating a new randomness request while the current one is uncommitted.

#### 4. Freshness Enforcement

**File:** `epoch-program/src/instructions/trigger_epoch_transition.rs:159-166`
```rust
let slot_diff = clock.slot.saturating_sub(randomness_data.seed_slot);
require!(slot_diff <= 1, EpochError::RandomnessExpired);
```

The randomness account must have been created within 1 slot of the commit transaction. Cannot use pre-generated randomness accounts with known values.

#### 5. Not-Yet-Revealed Check

**File:** `epoch-program/src/instructions/trigger_epoch_transition.rs:170-173`
```rust
if randomness_data.get_value(clock.slot).is_ok() {
    return Err(EpochError::RandomnessAlreadyRevealed.into());
}
```

Cannot use a randomness account whose value is already known. The VRF output is only knowable after the oracle reveals it (post-commit).

## Attack Scenario Analysis

### Scenario 1: Call consume_randomness Twice in Same Epoch

**Attack Flow:**
1. Attacker calls `trigger_epoch_transition` (sets `vrf_pending = true`)
2. Oracle reveals randomness
3. Attacker calls `consume_randomness` (clears `vrf_pending = false`)
4. Attacker attempts second call to `consume_randomness`

**Result:** **BLOCKED**
The second call fails at line 151 with `EpochError::NoVrfPending` because `vrf_pending` was cleared to `false` in step 3.

### Scenario 2: Trigger New Epoch Without Consuming Current VRF

**Attack Flow:**
1. Attacker calls `trigger_epoch_transition` (sets `vrf_pending = true`)
2. Without consuming, attacker attempts to call `trigger_epoch_transition` again

**Result:** **BLOCKED**
The second `trigger_epoch_transition` fails at line 149 with `EpochError::VrfAlreadyPending` because `vrf_pending` is still `true`.

### Scenario 3: Substitute Different Randomness Account

**Attack Flow:**
1. Attacker creates randomness account A
2. Attacker calls `trigger_epoch_transition` with account A (binds `pending_randomness_account = A`)
3. Attacker creates randomness account B with potentially different oracle
4. Attacker calls `consume_randomness` with account B

**Result:** **BLOCKED**
The `consume_randomness` call fails at line 154-157 with `EpochError::RandomnessAccountMismatch` because account B's key doesn't match the bound account A.

### Scenario 4: Wait for Multiple Epochs and Re-Consume

**Attack Flow:**
1. Epoch N: Trigger and consume randomness (sets `vrf_pending = false`)
2. Epoch N+1 boundary passes
3. Attacker attempts to call `consume_randomness` again with old randomness account

**Result:** **BLOCKED**
The `consume_randomness` call fails at line 151 with `EpochError::NoVrfPending` because a new epoch transition must call `trigger_epoch_transition` first to set `vrf_pending = true`.

## State Invariants

The VRF lifecycle maintains these invariants:

| Invariant | Enforcement |
|-----------|-------------|
| INV-VRF-1: Each randomness can be consumed at most once | `vrf_pending` flag checked at entry, cleared after use |
| INV-VRF-2: Cannot trigger new epoch while VRF pending | `!vrf_pending` required by `trigger_epoch_transition` |
| INV-VRF-3: Randomness account is bound at commit time | `pending_randomness_account` set in `trigger_epoch_transition` |
| INV-VRF-4: Same randomness account must be used at consume time | Key equality check in `consume_randomness` |
| INV-VRF-5: One VRF cycle per epoch transition | Epoch advances only in `trigger_epoch_transition`, which sets `vrf_pending = true` |

---

## Why No randomness_consumed Flag?

The protocol uses `vrf_pending` as the consumption flag, but with **inverted logic**:
- `vrf_pending = true` means "randomness committed, awaiting consumption"
- `vrf_pending = false` means "no pending randomness" (can be because: never triggered, or already consumed)

This is combined with `taxes_confirmed`:
- `taxes_confirmed = false` during the commit-reveal window
- `taxes_confirmed = true` after consumption

The combination `(vrf_pending = false) AND (taxes_confirmed = true)` indicates successful consumption.

This design is **equivalent to but more expressive than** a simple `randomness_consumed` flag.

## Code Quality Observations

### Strengths

1. **Multiple defense layers**: The protocol doesn't rely on a single flag but combines multiple checks (pending flag, account binding, freshness, not-yet-revealed).

2. **Clear state machine**: The `vrf_pending` and `taxes_confirmed` flags create an unambiguous state machine with no invalid states.

3. **Comprehensive logging**: All state transitions are logged with `msg!()` for off-chain monitoring.

4. **Unit test coverage**: The VRF lifecycle has extensive inline tests verifying the state transitions.

### Minor Documentation Opportunity

The `.audit/ARCHITECTURE.md` reference to "consumed flag + slot check" is slightly misleading. The actual mechanism is:
- **"Consumed" is tracked via `vrf_pending = false`** (not a separate `randomness_consumed` flag)
- **Slot check** refers to `vrf_request_slot` for timeout recovery, not consumption tracking

This is a documentation clarity issue, not a security issue.

## Conclusion

**STR-029 is a FALSE POSITIVE.**

The `consume_randomness` instruction **CANNOT** be called multiple times per epoch. The protocol implements comprehensive anti-reroll protection through:

1. **Atomic flag-based once-per-epoch enforcement** (`vrf_pending`)
2. **Randomness account binding** (key verification)
3. **Double-commit prevention** (cannot trigger while pending)
4. **Freshness enforcement** (1-slot window)
5. **Not-yet-revealed check** (cannot use known randomness)

No exploit vector exists for re-rolling tax rates by calling `consume_randomness` multiple times.

// === 1. Validate VRF is pending ===
// This check, combined with clearing vrf_pending=false after consumption,
// ensures consume_randomness can only be called ONCE per epoch.
// After the first successful call, vrf_pending=false and subsequent calls

require!(epoch_state.vrf_pending, EpochError::NoVrfPending);

This finding demonstrates the importance of understanding the protocol's actual implementation rather than making assumptions based on theoretical attack patterns.
