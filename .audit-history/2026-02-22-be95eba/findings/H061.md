# H061: AMM Pool Initialization With Zero Reserves

**Strategy:** STR-061
**Determination:** NOT VULNERABLE
**Severity:** N/A
**Confidence:** HIGH

## Summary
The `initialize_pool` instruction enforces a mandatory non-zero reserves check at lines 46-48, preventing creation of pools with zero initial seed amounts. The check occurs before any pool state initialization, making first-swap manipulation attacks impossible.

## Evidence

**File:** `/Users/mlbob/Projects/Dr Fraudsworth/programs/amm/src/instructions/initialize_pool.rs`

**Vulnerable Code Pattern (NOT present):**
```rust
// This would be vulnerable:
pub fn handler(
    ctx: Context<InitializePool>,
    amount_a: u64,
    amount_b: u64,
) -> Result<()> {
    // ❌ MISSING: require!(amount_a > 0 && amount_b > 0, ...);
    // Pool created with zero reserves...
}
```

**Actual Implementation (SECURE):**
```rust
// Line 46-48: ENFORCES non-zero reserves
require!(amount_a > 0 && amount_b > 0, AmmError::ZeroSeedAmount);
```

**Execution Flow:**
1. **Line 35-38:** Canonical mint ordering validation (prevents duplicate pools)
2. **Line 41-44:** Duplicate mint check (prevents self-pairing)
3. **Line 46-48:** **Non-zero reserves requirement** ← PRIMARY DEFENSE
4. **Line 50-57:** Pool type inference
5. **Line 59-123:** Initial liquidity transfer into vaults
6. **Line 125-141:** Pool state populated with `reserve_a = amount_a` and `reserve_b = amount_b`

The reserves are set directly from the validated input parameters (lines 132-133):
```rust
pool.reserve_a = amount_a;  // amount_a is guaranteed > 0
pool.reserve_b = amount_b;  // amount_b is guaranteed > 0
```

## Attack Surface Analysis

**First-Swap Manipulation (EP-075 / Playbook Precedent):**
The vulnerability hypothesizes that creating a pool with zero reserves would allow a first depositor to manipulate exchange rates. This requires the ability to initialize a pool with:
- `reserve_a = 0` and `reserve_b = 0`, OR
- `reserve_a = 0` (some amount for B), OR
- `reserve_b = 0` (some amount for A)

**Code prevents all three scenarios:**
1. Both zero: `require!(amount_a > 0 && amount_b > 0)` blocks this directly
2. One zero: Same `&&` operator requires BOTH to be > 0; either zero fails the check
3. Underflow in transfers: Initial transfers at lines 73-123 would fail if amounts were zero (token program would reject zero-amount transfer), but this doesn't matter because the arithmetic requirement catches it first

## Impact
**None.** The vulnerability cannot be exploited. An attacker cannot:
- Create a pool with zero reserves (blocked by require!)
- Perform a first-swap manipulation (requires at least one zero reserve)
- Drain a pool via k-invariant violation (k-invariant check at swap time prevents this even if somehow a zero pool existed)
