# Finding: H041 - No Compute Budget on Bonding Curve Transactions

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

Both `BuyForm.tsx` and `SellForm.tsx` build transactions without `ComputeBudgetProgram.setComputeUnitLimit()` or `ComputeBudgetProgram.setComputeUnitPrice()` instructions. Without an explicit compute unit limit, the transaction requests the default 200,000 CU allocation. Without a priority fee, the transaction has zero priority in the validator's scheduler. This contrasts with the AMM swap path (`swap-builders.ts`) and staking path (`staking-builders.ts`), both of which include ComputeBudget instructions.

The bonding curve instructions involve Transfer Hook CPI (4 extra accounts per transfer), which adds compute overhead. While 200K CU is likely sufficient for a single bonding curve instruction, the lack of priority fees means transactions may be deprioritized during congestion (e.g., at launch when many users buy simultaneously). The combination of no compute budget + no priority fee + `skipPreflight: true` means transactions silently fail during congestion with poor error feedback.

## Hypothesis Investigated

BC TXs don't set compute budget, risking compute exhaustion failures.

## Investigation Path

### Step 1: Check BuyForm transaction construction
**Location:** `app/components/launch/BuyForm.tsx:181-185`
**Finding:** Transaction contains only the purchase instruction. No ComputeBudget instructions.
**Evidence:**
```typescript
const tx = new Transaction();
tx.add(instruction);
tx.feePayer = publicKey;
tx.recentBlockhash = blockhash;
```

### Step 2: Check SellForm transaction construction
**Location:** `app/components/launch/SellForm.tsx:190-194`
**Finding:** Same pattern -- only the sell instruction, no ComputeBudget.
**Evidence:**
```typescript
const tx = new Transaction();
tx.add(instruction);
tx.feePayer = publicKey;
tx.recentBlockhash = blockhash;
```

### Step 3: Verify curve-tx-builder doesn't add compute budget
**Location:** `app/lib/curve/curve-tx-builder.ts:92-128, 143-180`
**Finding:** `buildPurchaseInstruction` and `buildSellInstruction` return a single `TransactionInstruction`. No ComputeBudget instructions are added.

### Step 4: Confirm AMM swap builders DO include compute budget
**Location:** `app/lib/swap/swap-builders.ts:176-178, 211-213`
**Finding:** AMM swap builders explicitly add compute budget.
**Evidence:**
```typescript
tx.add(ComputeBudgetProgram.setComputeUnitLimit({ units: computeUnits }));
// ... and priority fee if configured
tx.add(ComputeBudgetProgram.setComputeUnitPrice({ microLamports: priorityFeeMicroLamports }));
```

### Step 5: Confirm staking builders also include compute budget
**Location:** `app/lib/staking/staking-builders.ts:150-178`
**Finding:** All 3 staking builders include ComputeBudget instructions.

## If CONFIRMED

### Severity Assessment
- **Severity:** LOW
- **Likelihood:** MEDIUM (especially during launch when congestion is expected)
- **Impact:** Failed/delayed transactions during congestion. Not a fund loss vector, but degrades launch experience.

### Attack Scenario
```
Step 1: Token launch begins. Many users attempt to buy simultaneously.
  Target: BuyForm.tsx
  Prerequisite: High demand during bonding curve phase

Step 2: All buy TXs have 0 priority fee (no ComputeBudgetProgram.setComputeUnitPrice).
  Result: TXs sit at bottom of validator queue

Step 3: Bots and other protocols with priority fees get processed first.
  Result: User TXs delayed, some expire (blockhash becomes invalid)

Step 4: Users retry, creating more congestion.
  Result: Cascade of failed TXs, poor launch experience

Final Result: Degraded UX during the highest-traffic period (launch).
No fund loss, but reputational damage and user frustration.
```

### Recommended Fix

Update `BuyForm.tsx` and `SellForm.tsx` to include ComputeBudget instructions, following the same pattern used by `swap-builders.ts`:

```typescript
import { ComputeBudgetProgram } from '@solana/web3.js';

// In handleSubmit, before tx.add(instruction):
const BONDING_CURVE_CU = 150_000; // Profile and adjust
tx.add(ComputeBudgetProgram.setComputeUnitLimit({ units: BONDING_CURVE_CU }));

// Add priority fee based on settings
const priorityFee = getPriorityFeeMicroLamports(settings.priorityFeePreset);
if (priorityFee > 0) {
  tx.add(ComputeBudgetProgram.setComputeUnitPrice({ microLamports: priorityFee }));
}

tx.add(instruction);
```

Alternatively, move compute budget logic into `curve-tx-builder.ts` to return a full `Transaction` (matching `swap-builders.ts` pattern) rather than a bare instruction.

## Related Findings
- Combines with H039 (skipPreflight) -- without preflight, compute exhaustion failures won't be caught before submission
- The `settings.priorityFeePreset` value is already available in both forms (`useSettings()`) but unused for compute budget

## Incidental Discoveries
- The forms access `settings.slippageBps` but not `settings.priorityFeePreset`, despite it being available through the same `useSettings()` hook. This suggests priority fees were overlooked when building the bonding curve forms. Suggested hypothesis: verify whether the priority fee preset from SettingsProvider is respected across all transaction types.
