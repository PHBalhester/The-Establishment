# Finding: H025 - CSP unsafe-inline Enabling XSS Escalation

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The hypothesis posits that `script-src 'unsafe-inline'` in the CSP allows inline script execution, negating XSS protection if any injection point exists. The CSP configuration at `app/next.config.ts:9` does indeed include `'unsafe-inline'` for `script-src`, which means CSP would not block an inline `<script>` tag if an attacker could inject HTML into the page. However, after exhaustive analysis, **no injection vector exists in the current codebase**. The application uses React's JSX (which auto-escapes by default), has zero uses of `dangerouslySetInnerHTML`, `innerHTML`, `eval()`, `document.write()`, `DOMParser`, `insertAdjacentHTML`, or `postMessage` listeners. No URL parameters or user input are rendered into HTML. All API routes return JSON only. Without an injection point, `unsafe-inline` cannot be exploited.

The finding is classified as NOT VULNERABLE for the **current** codebase, but the CSP gap is a genuine defense-in-depth weakness that would become exploitable if any future code introduces an HTML injection vector.

## Hypothesis Investigated

`script-src 'unsafe-inline'` in CSP allows inline script execution, negating XSS protection if any injection point exists.

## Investigation Path

### Step 1: Confirm the CSP configuration
**Location:** `app/next.config.ts:7-23`
**Finding:** CSP is defined as a template literal and applied to all routes via `headers()` at lines 58-86. The `script-src` directive includes `'self' 'unsafe-inline'`, confirming the hypothesis target.
**Evidence:**
```typescript
const cspHeader = `
  default-src 'self';
  script-src 'self' 'unsafe-inline';
  ...
`;
```
The comment at line 6 says `'unsafe-inline' needed for Next.js style injection` -- but `unsafe-inline` is on `script-src`, not just `style-src`. The comment appears to conflate the two directives.

### Step 2: Search for HTML injection vectors (dangerouslySetInnerHTML, innerHTML)
**Location:** All `app/**/*.{ts,tsx,js,jsx}` files
**Finding:** Zero matches for `dangerouslySetInnerHTML`, `innerHTML`, `__html`, or `v-html`. React's JSX auto-escaping is the sole rendering mechanism.
**Evidence:** Grep across entire `app/` directory returned no matches.

### Step 3: Search for dynamic code execution sinks
**Location:** All `app/**/*.{ts,tsx,js,jsx}` files
**Finding:** No uses of `eval()`, `Function()`, `document.write()`, `DOMParser`, `createContextualFragment`, or `insertAdjacentHTML`. `setTimeout`/`setInterval` usage is limited to safe patterns (function references, not string evaluation).
**Evidence:** All `setTimeout` calls pass function references (e.g., `setTimeout(resetForm, AUTO_RESET_MS)` at `app/hooks/useSwap.ts:864`), never strings.

### Step 4: Search for reflected URL parameters in page rendering
**Location:** `app/app/**/*.tsx` page components
**Finding:** No page component uses `useSearchParams`, `useParams`, or `router.query`. URL parameters are only consumed in API routes (`app/app/api/candles/route.ts:186-196`) and are used exclusively for parameterized Drizzle ORM queries that return JSON. No URL parameter content is rendered into HTML.

### Step 5: Search for postMessage listeners
**Location:** All `app/**/*.{ts,tsx,js,jsx}` files
**Finding:** No `postMessage` handlers or `addEventListener('message', ...)` found anywhere. No cross-origin message processing surface.

### Step 6: Search for DOM manipulation patterns
**Location:** All `app/**/*.{ts,tsx,js,jsx}` files
**Finding:** Only two DOM `src` assignments found: `app/lib/audio-manager.ts:417` and `:458`, both setting `audio.src` to URLs from a hardcoded playlist. No user-controlled values reach DOM properties.

### Step 7: Check for nonce-based CSP alternative
**Location:** `app/` directory
**Finding:** No `middleware.ts` exists. No nonce generation or injection anywhere. The current architecture uses static CSP headers via `next.config.ts` `headers()` function, which cannot support per-request nonces without middleware.

## Invariant Analysis

### Target Invariant
CSP should prevent execution of attacker-injected scripts, even if an HTML injection vector is discovered.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `app/next.config.ts:9` | `script-src 'self' 'unsafe-inline'` | **No** -- `unsafe-inline` allows inline scripts, weakening the invariant |
| React JSX auto-escaping | All dynamic content is escaped before DOM insertion | **Yes** -- no bypasses (`dangerouslySetInnerHTML`) found |

| JSON-only API responses | API routes cannot serve HTML injection | **Yes** -- all return `NextResponse.json()` |

## Blocking Factors
- **React JSX auto-escaping**: All components use JSX, which escapes `<`, `>`, `"`, `'`, `&` in dynamic expressions -- prevents HTML injection without explicit `dangerouslySetInnerHTML`
- **No `dangerouslySetInnerHTML`**: Zero instances across the entire `app/` directory
- **No DOM mutation APIs**: No `innerHTML`, `insertAdjacentHTML`, `document.write`, `DOMParser` usage
- **No URL parameter rendering**: Pages do not read or render URL parameters
- **No `postMessage` listeners**: No cross-origin message handling surface
- **JSON-only API routes**: All 6 API routes return JSON via `NextResponse.json()`, never HTML

## Enabling Factors
- **`unsafe-inline` in script-src**: `app/next.config.ts:9` -- If an injection vector were found, CSP would not block inline `<script>` execution
- **No middleware for nonce-based CSP**: No `middleware.ts` exists, making it impossible to use per-request nonces as an alternative to `unsafe-inline`
- **Comment mismatch**: Line 6 comment says `unsafe-inline` is for "style injection" but it is applied to `script-src` too, suggesting the `script-src` inclusion may be accidental

**Overall counterargument confidence: HIGH**

All five challenges are refuted with high confidence. The core protection is React's JSX auto-escaping combined with the absence of any bypass mechanism (`dangerouslySetInnerHTML`, DOM mutation APIs, URL param rendering). Even if `unsafe-inline` remains in the CSP, there is no path to inject executable HTML into the page.

## Related Findings
- The `unsafe-inline` gap is a defense-in-depth weakness. If a future code change introduces `dangerouslySetInnerHTML` or a third-party component with an XSS vulnerability, the CSP would fail to provide its intended second line of defense.
- The comment at `app/next.config.ts:6` incorrectly attributes `unsafe-inline` to "Next.js style injection." The `style-src 'unsafe-inline'` is the one needed for Next.js -- `script-src 'unsafe-inline'` may be removable without breaking the application, though this needs testing.

## Incidental Discoveries

1. **Potentially unnecessary `unsafe-inline` on `script-src`**: `app/next.config.ts:9` -- The comment says it is for "style injection" but applies to `script-src`. Next.js App Router with Turbopack may not require `unsafe-inline` for scripts. Removing it (or switching to nonce-based CSP via middleware) would restore CSP's XSS protection as a defense-in-depth layer. **Suggested hypothesis: Test whether the application functions correctly with `script-src 'self'` only (no `unsafe-inline`), and if not, implement nonce-based CSP via middleware.**

2. **No `Strict-Transport-Security` (HSTS) header**: `app/next.config.ts:58-86` -- Missing HSTS means the browser does not enforce HTTPS on subsequent visits. While Railway enforces HTTPS at the edge, HSTS would prevent SSL stripping on first visit. This was already identified in context file `19-WEB-02` but is worth reiterating as a related hardening item.
