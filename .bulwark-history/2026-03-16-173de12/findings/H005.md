# Finding: H005 - Keypairs Committed to Git Repository

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

Twelve keypair files in the `keypairs/` directory are tracked by git, including the devnet wallet (admin/crank/upgrade authority), all 7 program deploy keypairs, a carnage WSOL keypair, and test keypairs. The `keypairs/` directory is NOT excluded by `.gitignore` -- only `scripts/deploy/mint-keypairs/` is excluded. These keypairs have been in git history since the earliest commits (v4 rebuild). The repository is hosted on GitHub at `https://github.com/MetalLegBob/dr-fraudsworth.git`.

The deployment scripts (`deploy.sh`, `deploy-all.sh`) use these committed keypair files directly for program deployment with **no guard preventing reuse on mainnet**. The `WALLET` variable defaults to `keypairs/devnet-wallet.json` with no cluster-aware check. While the mainnet checklist documents the intent to add `keypairs/` to `.gitignore` before mainnet and generate fresh keys, this is a purely procedural control -- no code enforces it.

## Hypothesis Investigated

Program deploy keypairs and devnet wallet committed to the git repository could be used to impersonate program authorities if the same keys are reused on mainnet.

## Blocking Factors
- **Devnet only (currently):** The project is currently on devnet where these keypairs have no real financial value. -- `Docs/mainnet-readiness-assessment.md`
- **Vanity keypairs untracked:** The mainnet vanity address keypairs (`FraUdp6Y...json`, `cRiMEhAx...json`) in the project root are NOT tracked by git, suggesting awareness of the issue for future keypairs.

## Enabling Factors
- **No .gitignore protection:** `keypairs/` is NOT in `.gitignore` -- `.gitignore`
- **Deploy scripts hardcode keypair paths:** `deploy.sh:137-143` hardcodes all 7 program keypair paths from `keypairs/` with no env var override for program keypairs (only the wallet has an override).
- **No mainnet guard in deploy scripts:** No code checks whether the target cluster is mainnet and refuses to use committed keypairs -- `deploy.sh:97` only checks cluster for airdrop logic.
- **Git history is permanent:** Even if `keypairs/` is added to `.gitignore` now, the keypair bytes are permanently in git history (since commit `a5235b7`).
- **Same keypair = same program address:** On Solana, the keypair determines the program address. If the same keypairs are used on mainnet, the programs deploy to the same addresses, AND the deploying wallet is the upgrade authority.

## Severity Assessment
- **Severity:** HIGH (CRITICAL if keypairs are reused on mainnet)
- **Likelihood:** MEDIUM -- requires mainnet deployment mistake (procedural failure) OR repo access for devnet exploitation
- **Impact:** Complete protocol takeover. Attacker with program deploy keypairs + wallet keypair can: (1) upgrade all 7 programs to malicious code, (2) call any admin-gated instruction, (3) drain protocol funds via malicious program upgrades.

## Recommended Fix

### Immediate (pre-mainnet, defense in depth):

1. **Add `keypairs/` to `.gitignore`:**
```gitignore
# Before (vulnerable)
scripts/deploy/mint-keypairs/

# After (fixed)
keypairs/
scripts/deploy/mint-keypairs/
```

2. **Add mainnet guard to deploy.sh:**
```bash
# After (fixed) - add after line 56
if echo "$CLUSTER_URL" | grep -qE "mainnet"; then
  if [ "$WALLET" = "keypairs/devnet-wallet.json" ]; then
    echo "ERROR: Cannot use committed devnet keypairs for mainnet deployment."
    echo "Set WALLET env var to a mainnet-specific keypair."
    exit 1
  fi
  for entry in "${PROGRAMS[@]}"; do
    IFS=':' read -r name keypair <<< "$entry"
    if echo "$keypair" | grep -q "^keypairs/"; then
      echo "ERROR: Cannot use committed keypairs/ for mainnet. Override via PROGRAM_KEYPAIRS env."
      exit 1
    fi
  done
fi
```

3. **Generate completely new keypairs for mainnet** -- the committed keypairs must be considered compromised for anyone with current or historical repo access.

4. **Purge keypairs from git history** using `git filter-repo` or BFG Repo-Cleaner (only effective if repo is private and history hasn't been cloned by others).

### Long-term (mainnet readiness):
- Use Squads multisig as program upgrade authority (per mainnet checklist B4)
- Support env-var-based program keypair paths in deploy.sh (not just the wallet)
- Implement startup validation in crank that checks loaded public key matches expected address

## Incidental Discoveries
- **carnage-wsol.json file permissions inconsistency:** `keypairs/carnage-wsol.json` has permissions `0644` (world-readable) while other keypairs have `0600`. This file was created by `initialize.ts` which uses `fs.writeFileSync` with default permissions. -- `keypairs/carnage-wsol.json`, `scripts/deploy/initialize.ts:1174`
  - Suggested hypothesis: Investigate whether any deployment scripts create files with insecure default permissions.
- **Deploy script program keypairs not overridable:** Unlike the wallet (overridable via `WALLET` env var), the 7 program keypair paths in `deploy.sh:137-143` are hardcoded with no env var override mechanism. Even if a mainnet operator wanted to use different keypairs, they'd need to modify the script.
  - Suggested hypothesis: Investigate deploy script inflexibility as a mainnet operational risk.
