# Finding: H029 - Crank Infinite Retry Without Backoff

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The crank runner's error handling at `scripts/crank/crank-runner.ts:302-314` uses a fixed 30-second retry delay for ALL errors, regardless of error type, frequency, or consecutive failure count. There is no exponential backoff, no maximum retry count, no error classification, and no circuit breaker. This means:

1. **Persistent failures burn SOL indefinitely**: If the RPC returns errors, each retry attempt may still consume transaction fees (especially with `skipPreflight: true` in VRF paths). At 30-second intervals, this is ~2,880 failed attempts per day.
2. **RPC rate limiting is not respected**: If Helius returns HTTP 429 (rate limited), the crank retries at the same fixed 30s interval instead of backing off.
3. **No alerting or escalation**: The operator has no way to know the crank is stuck in an error loop short of manually checking Railway logs.
4. **No auto-shutdown on persistent failure**: A fundamentally broken state (wallet drained, program upgraded, key revoked) results in infinite retry with no recovery possible.

The crank is the sole mechanism for epoch advancement. When it enters an error loop, the protocol stalls -- but the crank continues burning SOL on fees without advancing any epochs.

## Hypothesis Investigated

Crank retries every 30s without exponential backoff, risking SOL drain and RPC ban during persistent failures.

## Investigation Path

### Step 1: Error catch block analysis
**Location:** `scripts/crank/crank-runner.ts:302-315`
**Finding:** All errors are caught, truncated to 300 chars, logged, and retried after a fixed 30s sleep.
**Evidence:**
```typescript
} catch (err) {
    const errStr = String(err).slice(0, 300);
    console.error(
      `[crank] ERROR cycle ${cycleCount}: ${errStr}`
    );

    // Don't exit — sleep and retry next cycle
    if (!shutdownRequested) {
      console.log(
        `[crank] Retrying in ${ERROR_RETRY_DELAY_MS / 1000}s...`
      );
      await sleep(ERROR_RETRY_DELAY_MS);
    }
  }
```

### Step 2: Error classification check
**Location:** `scripts/crank/crank-runner.ts:302`
**Finding:** No error classification. A transient RPC timeout, a permanent "wallet has insufficient funds" error, a 429 rate limit, and an on-chain constraint violation all receive identical treatment: 30s sleep, retry.

### Step 3: Consecutive failure tracking
**Location:** `scripts/crank/crank-runner.ts:191-316`
**Finding:** No consecutive failure counter. The `cycleCount` variable increments on every loop iteration (both success and failure) but is only used for logging. There is no `consecutiveErrors` counter, no `MAX_CONSECUTIVE_ERRORS` threshold, and no auto-shutdown logic.

### Step 4: VRF inner retry vs outer loop
**Location:** `scripts/vrf/lib/vrf-flow.ts:229-254` (tryReveal)
**Finding:** The VRF reveal retry uses linear backoff (3s, 6s, 9s... up to 10 attempts). This is a bounded inner retry. However, if the entire `advanceEpochWithVRF` call fails (including after exhausting inner retries), the outer crank loop catches it and retries the full cycle after 30s -- creating a nested retry pattern where the inner bounded retry is wrapped by an outer unbounded retry.

### Step 5: Fee impact analysis
**Location:** `scripts/crank/crank-runner.ts:201-300`
**Finding:** Each cycle makes 3-5 RPC calls (readEpochState, getBalance, getBalance for vault, getSlot) before attempting the epoch transition. These read calls are free. However, if the error occurs during `advanceEpochWithVRF` (which submits transactions), failed TXs with `skipPreflight: true` land on-chain and consume fees. At 30s intervals, this is approximately 0.00001 SOL * 2880 attempts/day = ~0.03 SOL/day in fees from failed attempts. While not catastrophic, combined with vault top-ups (0.005 SOL each, potentially every cycle), the drain accelerates.

### Step 6: Context file cross-reference
**Location:** `.bulwark/context/09-BOT-01-keeper-crank.md`
**Finding:** The context file identifies this exact issue at OC-249 ("Infinite Retry on Failed Operations") and R-05/R-06/R-07. The context also notes the vault top-up has no cumulative limit, meaning error-loop cycles that trigger vault reads could also trigger continuous top-ups.

## Recommended Fix
```typescript
// Before (vulnerable)
} catch (err) {
    const errStr = String(err).slice(0, 300);
    console.error(`[crank] ERROR cycle ${cycleCount}: ${errStr}`);
    if (!shutdownRequested) {
      console.log(`[crank] Retrying in ${ERROR_RETRY_DELAY_MS / 1000}s...`);
      await sleep(ERROR_RETRY_DELAY_MS);
    }
  }

// After (fixed)
const MAX_CONSECUTIVE_ERRORS = 10;
const BASE_RETRY_DELAY_MS = 30_000;
const MAX_RETRY_DELAY_MS = 300_000; // 5 minutes
let consecutiveErrors = 0;

// ... in the catch block:
} catch (err) {
    consecutiveErrors++;
    const errStr = String(err).slice(0, 500);
    // Preserve Anchor program logs if available
    const logs = (err as any)?.logs?.join("\n") ?? "";
    console.error(`[crank] ERROR cycle ${cycleCount} (${consecutiveErrors} consecutive): ${errStr}`);
    if (logs) console.error(`[crank] Program logs:\n${logs}`);

    if (consecutiveErrors >= MAX_CONSECUTIVE_ERRORS) {
      console.error(`[crank] CIRCUIT BREAKER: ${MAX_CONSECUTIVE_ERRORS} consecutive errors. Shutting down.`);
      // TODO: Send alert webhook/notification here
      process.exit(2);
    }

    if (!shutdownRequested) {
      // Exponential backoff: 30s, 60s, 120s, 240s, 300s (capped)
      const delay = Math.min(
        BASE_RETRY_DELAY_MS * Math.pow(2, consecutiveErrors - 1),
        MAX_RETRY_DELAY_MS
      );
      console.log(`[crank] Retrying in ${(delay / 1000).toFixed(0)}s (backoff level ${consecutiveErrors})...`);
      await sleep(delay);
    }
  }

// Reset on success (inside the try block, after advanceEpochWithVRF):
consecutiveErrors = 0;
```
