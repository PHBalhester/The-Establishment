# Finding: S004 - Mainnet Launch Day Attack Bundle

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

This strategy chains four confirmed findings -- committed keypairs (H005), any-signer bonding curve authority (H010), first-caller-wins hook init (H016), and devnet fallback confusion (H009) -- into a comprehensive mainnet launch day attack that exploits the deployment window. The chain is devastating: (1) an attacker obtains all program deploy keypairs and the admin wallet from the git repository (H005), (2) they monitor for the mainnet deploy transaction, (3) they immediately front-run `initialize_authority` on the transfer hook to capture whitelist control (H016), (4) they wait for bonding curves to reach `Filled` status and call `prepare_transition` + `withdraw_graduated_sol` to steal ~2000 SOL (H010), (5) meanwhile, if the team fails to set `NEXT_PUBLIC_RPC_URL`, the frontend silently falls back to devnet, causing user confusion during the critical launch window (H009).

Each individual finding is independently exploitable, but the combination on launch day creates a time-pressured scenario where the team must defend against all four vectors simultaneously. The attacker needs only one to succeed.

## Hypothesis Investigated

Committed keypairs (H005) + any-signer BC authority (H010) + first-caller-wins hook init (H016) + devnet fallback confusion (H009) = comprehensive launch day attack.

### Link 3: First-Caller-Wins Hook Init (H016)

## Enabling Factors
- **Committed keypairs:** `keypairs/` directory in git -- attacker knows all program addresses
- **No authority on bonding curve:** `CurveState` has no `authority` field
- **First-caller-wins init:** `initialize_authority` accepts bare `Signer<'info>`
- **Deploy-init gap:** `deploy-all.sh` has observable time between deploy and init
- **No mainnet env validation:** `connection.ts:33` silently falls back to devnet
- **No deployment monitoring:** No alerting on unexpected authority initialization

## Blocking Factors
- **Repository must be accessible:** If truly private and never compromised, keypairs are less exposed (but git history is permanent)
- **Mainnet keypair rotation:** If team generates fresh mainnet keypairs, H005 devnet keys are useless for program impersonation (but deploy-all.sh has no enforcement)
- **Bonding curve fill takes time:** H010 requires curves to reach Filled status first

## Severity Assessment
- **Severity:** CRITICAL
- **Likelihood:** HIGH (H016 hook front-running is trivially executable on launch day; H010 is a guaranteed steal once curves fill)
- **Impact:** ~2000+ SOL direct theft (H010), permanent protocol bricking (H016), or both simultaneously

### Combined Severity Justification
Individual findings are HIGH (H005), CRITICAL (H010), CRITICAL (H016), MEDIUM (H009). The chain is CRITICAL because:
1. **Multiple independent kill shots:** H010 and H016 are each independently CRITICAL; together they guarantee at least one succeeds
2. **Time-pressured defense:** Launch day creates chaos; H009 devnet fallback adds confusion; team must defend all vectors simultaneously
3. **Irreversible consequences:** Hook authority capture is permanent (no re-init). Bonding curve SOL withdrawal is immediate and irreversible.
4. **Attacker advantage:** Attacker prepares scripts in advance, team must react in real-time
5. **H005 enables reconnaissance:** Committed keypairs let attacker predict program addresses and pre-build transactions

## Incidental Discoveries
- The `deploy-all.sh` script has no `--mainnet` or `--production` guard. It accepts any cluster URL including mainnet without additional confirmation or safety checks. Suggested hypothesis: Can an operator accidentally deploy to mainnet with devnet keypairs and configuration?
- The graduation script (`scripts/graduation/graduate.ts`) also uses the admin keypair from `keypairs/devnet-wallet.json` with no cluster-aware guard. If the same script is run on mainnet, it would attempt to use the compromised devnet keypair.
