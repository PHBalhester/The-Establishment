# Finding: H052 - Version Mismatch @solana/web3.js Between Workspaces

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The hypothesis proposes that different `@solana/web3.js` versions between root (`^1.95.5`) and app (`1.98.4`) workspaces could produce different PDA derivations. After investigation, this is not a practical vulnerability:

1. PDA derivation (`PublicKey.findProgramAddressSync`) uses SHA-256 hashing of seeds, which has been stable across all 1.x versions of `@solana/web3.js`. The algorithm is defined by the Solana runtime specification and has not changed.

2. The root workspace (`^1.95.5` with caret) will resolve to the latest compatible 1.x version. npm's workspace hoisting means the app's pinned `1.98.4` takes precedence within the app workspace, while scripts at root use whatever version was installed.

3. The shared package (`@dr-fraudsworth/shared`) declares `@solana/web3.js: "^1.95.5"` as a `peerDependency`, meaning it uses whatever version the consuming workspace provides -- no separate installation.

4. PDA derivation in the codebase uses `PublicKey.findProgramAddressSync()` which is a pure cryptographic function (SHA-256 loop) that has been identical across all 1.x releases.

The version mismatch is a code hygiene issue but not a security vulnerability.

## Hypothesis Investigated

Root vs app different @solana/web3.js could produce different PDA derivations.

## Investigation Path

### Step 1: Version declarations
**Location:** `package.json:22` (root) and `app/package.json:25` (app)
**Finding:** Root uses `^1.95.5` (range), app uses `1.98.4` (pinned). Shared uses `^1.95.5` as peerDependency.
**Evidence:**
```json
// Root package.json
"@solana/web3.js": "^1.95.5"

// App package.json
"@solana/web3.js": "1.98.4"

// Shared package.json
"peerDependencies": {
  "@solana/web3.js": "^1.95.5"
}
```

### Step 2: PDA derivation stability
**Finding:** `PublicKey.findProgramAddressSync()` implements the Solana PDA spec: SHA-256 hash of `[seeds..., programId, "ProgramDerivedAddress"]` with a bump byte search. This algorithm is defined by the Solana runtime, not the SDK version. It has been unchanged across all 1.x releases (verified by checking the function signature and behavior across changelog entries).

### Step 3: Workspace hoisting behavior
**Finding:** npm workspaces hoist shared dependencies. The app workspace pins `1.98.4`, so all code within `app/` uses exactly that version. Root-level scripts use whatever `^1.95.5` resolved to at install time. With `package-lock.json` gitignored, the root version could vary between builds.

### Step 4: Cross-workspace PDA usage
**Finding:** PDA derivation happens in both workspaces:
- App: `app/hooks/useRoutes.ts`, `app/lib/swap/`, PDA helpers in React hooks
- Scripts: `scripts/deploy/`, `scripts/crank/`, `scripts/e2e/`
- Shared: `shared/constants.ts` exports `PublicKey` addresses (not derivations)

The shared package exports pre-computed `PublicKey` constants, not derivation functions. So the version used for derivation only matters during script execution, not at the shared-constants level.

## Invariant Analysis

### Target Invariant
PDA derivations must produce identical results across all workspaces.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| Solana runtime spec | SHA-256 PDA algorithm | Yes (specification level) |
| `shared/constants.ts` | Pre-computed PublicKey exports | Yes (not derived at runtime per-workspace) |

## Blocking Factors
- Solana PDA algorithm is spec-level: identical SHA-256 across all 1.x versions
- Shared package uses `peerDependencies`: defers to consuming workspace's version
- Pre-computed constants in `shared/` avoid runtime derivation divergence

## Enabling Factors
- Version range (`^1.95.5`) in root + gitignored lockfile = non-deterministic resolution
- If a future 2.x release changes PDA derivation (breaking change), the caret range would not auto-update

## If NOT VULNERABLE

### Devil's Advocate Analysis (MANDATORY)
| Challenge | Argument | Refutation | Evidence | Confidence |
|-----------|----------|------------|----------|------------|
| Crafted input | What if @solana/web3.js 1.95.x has a BN.js bug that produces different serialization? | PDA derivation uses `Buffer.from(seed)` not BN.js; seeds are byte arrays | `PublicKey.findProgramAddressSync` source | HIGH |
| Multiple accounts | What if root scripts deploy with one PDA and app uses another? | Both derive from same seeds + same SHA-256 = same result | Solana PDA specification | HIGH |
| External API MITM | What if npm registry serves a malicious 1.x version with backdoored PDA? | This is a supply chain attack (covered separately by lockfile finding), not a version mismatch issue | Package integrity is a separate concern | MEDIUM |
| Race condition | What if concurrent npm installs resolve different versions? | npm workspaces resolve deterministically per `npm install` run | npm documentation | HIGH |
| Frontend chain | What if XSS modifies PDA derivation in the browser? | `PublicKey.findProgramAddressSync` is called in module scope, not from user input; XSS can't modify the function | Module system isolation | HIGH |

**Overall counterargument confidence:** HIGH. PDA derivation is cryptographically stable across versions. The real risk is the gitignored lockfile (a separate, already-documented finding), not the version mismatch itself.

## Incidental Discoveries
- `shared/package.json` has `@solana/spl-token: "^0.4.14"` as a regular dependency (not peer). This means the shared package installs its own copy, potentially creating duplicate spl-token instances in the bundle. Suggested hypothesis: "Duplicate @solana/spl-token installations between shared and app workspaces could cause type incompatibilities or incorrect token program references."
