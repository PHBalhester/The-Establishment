# Finding: H067 - Railway Migration Injection

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The Railway `preDeployCommand` at `railway.toml:7` runs `npx tsx app/db/migrate.ts` before each deployment, executing SQL migration files from `app/db/migrations/`. The hypothesis suggests malicious migration files could execute arbitrary SQL. Investigation shows this is NOT exploitable because: (1) migration files are committed to git and reviewed as part of the normal development process -- they are not dynamically generated or user-supplied; (2) the Drizzle ORM migration runner (`drizzle-orm/postgres-js/migrator`) reads only from the `migrationsFolder` directory and applies files sequentially by name; (3) writing a migration file requires git push access to the repository; and (4) the migration runner uses the same Drizzle safety model as all other database operations.

## Hypothesis Investigated

preDeployCommand runs migrations; malicious migration files could execute SQL.

## Investigation Path

### Step 1: Examine Railway pre-deploy command
**Location:** `railway.toml:7`
**Finding:** Pre-deploy runs the migration script.
**Evidence:**
```toml
preDeployCommand = "npx tsx app/db/migrate.ts"
```

### Step 2: Examine the migration script
**Location:** `app/db/migrate.ts`
**Finding:** The script is straightforward: connect to Postgres, run pending migrations from `app/db/migrations/` directory, exit. Uses Drizzle ORM's programmatic migrator.
**Evidence:**
```typescript
const client = postgres(databaseUrl, { max: 1 });
const db = drizzle(client);
await migrate(db, { migrationsFolder: path.resolve(__dirname, "migrations") });
```

### Step 3: Examine migration files
**Location:** `app/db/migrations/0000_simple_squadron_supreme.sql`
**Finding:** Only one migration file exists. It contains standard DDL: `CREATE TABLE`, `CREATE INDEX`. No dynamic SQL, no user input interpolation.
**Evidence:**
```sql
CREATE TABLE "candles" (
  "id" integer PRIMARY KEY GENERATED ALWAYS AS IDENTITY ...
  ...
);
CREATE TABLE "swap_events" ( ... );
CREATE UNIQUE INDEX "candle_unique_idx" ON "candles" ...;
```

### Step 4: Analyze migration file supply chain
**Location:** git history / development workflow
**Finding:** Migration files are generated by `drizzle-kit generate` (dev tool) and committed to git. They are reviewed as part of normal PR/commit flow. An attacker would need git push access to inject a malicious migration.

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Add malicious .sql file to migrations/ | Requires git push access | Migration files are committed code |
| 2 | Migration runs on Railway deploy | Yes (automatically) | `railway.toml:7` |
| 3 | Arbitrary SQL executed against production DB | Yes (if step 1 succeeds) | Drizzle migrator executes .sql files |

## Invariant Analysis

### Target Invariant
Database migrations should only execute trusted, reviewed SQL statements.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| Git access control | Only authorized users can push migrations | Yes |
| `migrate.ts:46` | Reads only from `migrations/` directory | Yes |
| Drizzle migrator | Sequential execution by filename order | Yes |

## Blocking Factors
- **Git access control**: Migration files require push access to add/modify
- **Static file source**: `migrate.ts:46` reads from committed `migrations/` directory only
- **No user input**: Migration runner takes no runtime parameters (only `DATABASE_URL` env var for connection)
- **Drizzle migration tracking**: Drizzle tracks applied migrations; already-applied files are skipped

## Enabling Factors
- **Auto-execution**: `railway.toml:7` -- migrations run automatically on every deploy, no manual gate
- **Full DB access**: `DATABASE_URL` grants full database access to the migration script
- **No migration review gate**: No CI/CD check specifically validates migration SQL (H055)

## Incidental Discoveries
- The `DATABASE_URL` connection in `migrate.ts:42` uses `postgres(databaseUrl, { max: 1 })` without explicit `ssl: true` or `sslmode=require`. Per ARCHITECTURE.md assumption A-OC7, Railway Postgres may enforce TLS by default, but this is unvalidated in application code. This is a separate concern from the migration injection hypothesis.
