# Finding: H034 - Double-Submit in useSwap Without Guard

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The `useSwap` hook has no explicit double-submit guard. The `executeSwap` and `executeRoute` functions can be called multiple times concurrently -- there is no check of the current `status` state before proceeding with transaction building. The only guards are `!wallet.publicKey || !wallet.connected || !quote` (line 674), none of which prevent a second call while the first is in-flight. The `status` state variable transitions through `building -> signing -> sending -> confirming -> confirmed`, but neither `executeSwap` nor `executeRoute` check whether `status !== "idle"` before proceeding.

A rapid double-click on the swap button (or programmatic invocation) could initiate two identical transactions simultaneously. Both would reach the wallet for signing, and if the user confirms both (or if a programmatic caller auto-signs), two identical swaps would execute on-chain, doubling the user's intended trade. The second swap would execute at a worse price due to the first swap's price impact.

## Hypothesis Investigated

No double-submit guard allows accidental duplicate swaps.

## Investigation Path

### Step 1: Search for submission guards
**Location:** `app/hooks/useSwap.ts` (full file)
**Finding:** No `isSubmitting` ref, no mutex, no status check at the top of `executeSwap` or `executeRoute`. Searched for patterns: `isSubmitting`, `submitting`, `isSwapping`, `swapping`, `double.submit`, `guard` -- zero matches.

### Step 2: Check executeSwap entry guards
**Location:** `app/hooks/useSwap.ts:673-676`
**Finding:** The only checks are wallet connection and quote existence. No status check.
**Evidence:**
```typescript
const executeSwap = useCallback(async () => {
  if (!wallet.publicKey || !wallet.connected || !quote) {
    return;
  }
  // ... proceeds to build and sign transaction
```

### Step 3: Check executeRoute entry guards
**Location:** `app/hooks/useSwap.ts:822-826`
**Finding:** Same pattern -- checks smart routing state and route existence, but no status guard.
**Evidence:**
```typescript
const executeRoute = useCallback(async () => {
  if (!smartRouting || !routesResult.selectedRoute) {
    return executeSwap();
  }
  // ...
  if (!wallet.publicKey || !wallet.connected) return;
```

### Step 4: Check if status is checked anywhere before submission
**Location:** `app/hooks/useSwap.ts` (full file)
**Finding:** `setStatus("building")` is called at lines 683 and 842, but the status is never READ as a precondition. The status is only used for UI display and for the `clearTerminalStatus` function (which clears `confirmed`/`failed` states when the user starts editing).

### Step 5: Check UI-level protection
**Finding:** The swap button in the UI likely uses `status` to disable itself, but this is a UI-only protection. The `executeSwap`/`executeRoute` functions themselves have no guard, meaning any caller (including React effects, keyboard shortcuts, or rapid clicks that beat React's re-render cycle) can invoke them concurrently.

## Invariant Analysis

### Target Invariant
Each user-initiated swap action should result in exactly one on-chain transaction.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `useSwap.ts:674` | Wallet connected + quote exists | No -- doesn't prevent concurrent submissions |
| UI button (assumed) | Disabled during non-idle status | Partial -- React re-render race window exists |

## Blocking Factors
- Wallet signing prompt is a user-facing blocker (most wallets show a modal)
- Most wallet adapters serialize signing requests (second prompt appears after first is dismissed)
- On-chain slippage check (`minimumOutput`) would cause the second TX to fail IF price moved enough

## Enabling Factors
- `useSwap.ts:673-676`: No status guard in executeSwap
- `useSwap.ts:822-836`: No status guard in executeRoute
- React state updates are batched/async: two rapid calls both see the pre-update status
- Some wallets auto-approve or have fast approval flows

### Attack Scenario
```

Step 2: User double-clicks the "Swap" button rapidly
Step 3: Both click handlers invoke executeSwap() before React re-renders
Step 4: Two identical transactions are built and sent to the wallet
Step 5: Wallet shows two signing prompts (or auto-signs both)
Step 6: First TX executes: 1 SOL -> 10,000 CRIME (OK)
Step 7: Second TX executes: 1 SOL -> 9,800 CRIME (worse price due to first TX's impact)

```

### Recommended Fix
```typescript
// Before (vulnerable)
const executeSwap = useCallback(async () => {
  if (!wallet.publicKey || !wallet.connected || !quote) {
    return;
  }
  // ... proceeds immediately

// After (fixed) -- use a ref-based guard (survives React batching)
const isSubmittingRef = useRef(false);

const executeSwap = useCallback(async () => {
  if (isSubmittingRef.current) return; // Guard against double-submit
  if (!wallet.publicKey || !wallet.connected || !quote) return;

  isSubmittingRef.current = true;
  try {
    setStatus("building");
    // ... rest of swap logic
  } finally {
    isSubmittingRef.current = false;
  }
}, [/* deps */]);
```

Note: Using a `useRef` rather than `useState` is critical because refs update synchronously, closing the React batching race window.

## Related Findings
- The `status` state machine (idle -> building -> signing -> ...) provides the infrastructure for a guard but doesn't use it as one.

## Incidental Discoveries
- `executeRoute` at line 824-826 falls through to `executeSwap()` when smart routing is off. If both `executeRoute` and `executeSwap` were called concurrently (e.g., by different UI elements), they could also produce duplicate submissions. Suggested hypothesis: investigate whether both `executeSwap` and `executeRoute` are exposed to the UI and could be triggered independently.
