# Finding: H020 - IDL Supply Chain Attack via Build Pipeline

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The hypothesis posits that a compromised Rust dependency could generate tampered IDL files during `anchor build`, which `sync-idl.mjs` would then copy into the frontend, causing malicious transaction construction. Investigation reveals this attack is not viable in the production deployment path because: (1) `target/` is gitignored, so IDL files never reach Railway from the build output -- the production deployment uses only the git-committed `app/idl/*.json` files; (2) the sync script gracefully exits when `target/idl/` is absent (Railway builds); and (3) on the developer's machine, Cargo.lock is committed, pinning the Rust supply chain. The IDL sync hook only runs locally during `npm run dev` or local `npm run build`, where the developer can diff changes before committing.

The attack path requires compromising the Rust toolchain on the developer's local machine AND having the developer commit the tampered IDL files without review -- a plausible but low-probability scenario already mitigated by standard git review practices and Cargo.lock pinning.

## Hypothesis Investigated

The `sync-idl.mjs` predev/prebuild hook copies IDL files from `target/idl/` to the frontend. A compromised Rust dependency could generate tampered IDL files that cause the frontend to construct malicious transactions.

## Investigation Path

### Step 1: Examined the sync-idl.mjs script
**Location:** `app/scripts/sync-idl.mjs:1-89`
**Finding:** The script copies 7 named program IDL JSON files from `target/idl/` to `app/idl/`, and TypeScript type files from `target/types/` to `app/idl/types/`. It only copies files matching a hardcoded allowlist of 7 program names (line 29-37). If `target/idl/` does not exist, it warns and exits 0 (line 40-44).
**Evidence:**
```javascript
// Line 40-44: Graceful exit when target/ absent
if (!existsSync(IDL_SOURCE)) {
  console.warn("target/idl/ not found -- run 'anchor build' first");
  process.exit(0); // Don't fail dev server for missing IDLs
}
```

### Step 2: Verified IDL files are committed to git
**Location:** `app/idl/*.json` (7 files, all git-tracked)
**Finding:** All 7 IDL JSON files and their TypeScript type counterparts are committed to git. The git log shows explicit commits for IDL syncs (e.g., `0c09f23`, `c359402`). Since `target/` is gitignored (`.gitignore:4`), the `target/idl/` directory never exists in Railway's build environment.
**Evidence:**
```
$ git ls-files --error-unmatch app/idl/amm.json
app/idl/amm.json   # tracked
```

### Step 3: Traced the Railway production build path
**Location:** `railway.toml:3`, `app/package.json:8`
**Finding:** Railway's build command is `npm run --workspace app build`, which triggers `prebuild` -> `node scripts/sync-idl.mjs` -> `next build`. Since `target/` is gitignored and Railway only has git-committed files, `sync-idl.mjs` hits the early exit at line 40. The committed `app/idl/*.json` files are used directly by Next.js's bundler.
**Evidence:**
```toml
# railway.toml:3
buildCommand = "npm run --workspace app build"
```

### Step 4: Analyzed IDL consumption in the frontend
**Location:** `app/lib/anchor.ts:17-23`, `app/lib/curve/curve-tx-builder.ts:34-53`
**Finding:** IDL JSON files are imported by Anchor's `Program` constructor, which uses the `address` field from the IDL to set `program.programId`. This programId drives PDA derivation (e.g., `curve-tx-builder.ts:35-37`) and instruction discriminator generation. A tampered IDL could: (a) change the `address` field, causing PDAs to derive incorrectly (transactions would fail on-chain), (b) change instruction discriminators (transactions would fail), or (c) change account ordering (could potentially send funds to wrong accounts if the on-chain program accepted the instruction).
**Evidence:**
```typescript
// app/lib/curve/curve-tx-builder.ts:34-38
function deriveCurveState(mint: PublicKey, programId: PublicKey): PublicKey {
  return PublicKey.findProgramAddressSync(
    [SEEDS.CURVE, mint.toBuffer()],
    programId,  // From IDL address field
  )[0];
}
```

### Step 5: Verified Cargo.lock is committed
**Location:** `Cargo.lock` (git-tracked, confirmed by DEP-01 context)
**Finding:** The Rust dependency tree is pinned via committed Cargo.lock. A compromised Rust crate would need to be an exact version already in the lockfile, or the attacker would need to modify Cargo.lock (visible in git diff).

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Compromise a Rust crate used by Anchor build | Unlikely | Cargo.lock pins exact versions; attacker needs to compromise a specific pinned version |
| 2 | Tampered crate generates modified IDL during `anchor build` | Possible | Anchor's IDL generation runs as part of build; a proc-macro crate could theoretically interfere |
| 3 | `sync-idl.mjs` copies tampered IDL to `app/idl/` | Yes (local only) | Script copies unconditionally if `target/idl/` exists (`sync-idl.mjs:62`) |
| 4 | Developer commits tampered IDL to git | Requires inattention | `git diff` would show changes; no automated integrity check |
| 5 | Railway deploys with tampered IDL | Yes (if committed) | Railway uses committed files directly |
| 6 | Frontend constructs malicious transactions | Partially | Tampered address/discriminators cause TX failures; tampered account ordering could be dangerous |

**Verdict:** Steps 1 and 4 are the blocking factors. Step 1 is mitigated by Cargo.lock. Step 4 is mitigated by standard git review. Step 6 is partially blocked because most IDL tampering causes on-chain transaction failures rather than successful malicious transactions.

## Invariant Analysis

### Target Invariant
IDL files consumed by the frontend must accurately represent the on-chain program's instruction interface (account layout, discriminators, program address).

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `Cargo.lock` (git-tracked) | Rust dependency versions are pinned | Yes |
| `.gitignore:4` (`target/`) | Build artifacts don't reach production directly | Yes |
| `app/idl/*.json` (git-tracked) | IDL changes are visible in git diffs | Yes |
| `sync-idl.mjs:40-44` | Graceful exit if no build output | Yes |
| On-chain program validation | Mismatched discriminators/accounts cause TX failure | Yes (for most tampering) |

## Blocking Factors
- **Cargo.lock committed**: `Cargo.lock` (git-tracked) -- Rust supply chain is pinned to exact versions
- **target/ gitignored**: `.gitignore:4` -- Build artifacts never reach Railway; only committed IDL files are deployed
- **IDL changes visible in git**: `app/idl/*.json` are tracked -- any sync-induced changes appear in `git diff`
- **On-chain discriminator validation**: Tampered instruction discriminators would cause all transactions to fail on-chain
- **Hardcoded program allowlist**: `sync-idl.mjs:29-37` -- Only 7 named programs are synced; cannot inject arbitrary new IDL files

## Enabling Factors
- **No integrity verification**: `sync-idl.mjs:62` -- `cpSync` copies files without checksum or signature validation
- **Silent overwrite**: The script overwrites `app/idl/*.json` without comparing to previous content or alerting on changes
- **prebuild hook runs automatically**: `app/package.json:8` -- Runs on every `npm run build` locally, no opt-out

### Devil's Advocate Analysis (MANDATORY)
| Challenge | Argument | Refutation | Evidence | Confidence |
|-----------|----------|------------|----------|------------|
| Crafted input | A malicious IDL JSON could contain extra fields or oversized data that exploits JSON parsing bugs in Anchor's Program constructor | Anchor's Program constructor only reads known fields (address, instructions, accounts, types). Extra fields are ignored. JSON.parse is battle-tested. The IDL is static JSON imported at build time, not runtime-parsed from user input. | `app/lib/anchor.ts:40` -- `new Program(ammIdl as unknown as Amm, ...)` | HIGH |
| Multiple accounts | An attacker with repo write access could modify both Cargo.lock and IDL files simultaneously | This requires compromising the git repository itself (not just the npm or crate supply chain). With repo write access, far simpler attacks exist (modifying any source code). This is outside the scope of a supply chain attack. | Git access control is the mitigation | HIGH |
| External API MITM | An attacker MITMs the crates.io registry to serve a compromised crate during `cargo update` | Cargo.lock is committed and `anchor build` uses `cargo build`, which respects the lockfile. `cargo update` would need to be run explicitly, and the lockfile diff would be visible. Crates.io uses TLS and package signing. | `Cargo.lock` committed to git | HIGH |
| Race condition | Attacker times a compromised IDL sync between the developer running `anchor build` and `git add` | The developer would see the changed files in `git diff` before staging. The window is between build and commit, during which the developer is presumably reviewing changes. No automated process commits IDL changes. | Manual git workflow (no auto-commit) | MEDIUM |
| Frontend chain | XSS in the frontend could modify IDL data at runtime before Anchor processes it | IDL JSON is imported as static modules at build time (webpack/turbopack bundles them). At runtime, the IDL objects are already in memory. An XSS attacker would need to monkey-patch the Anchor Program instances, but at that point they could directly modify transaction construction without touching IDLs. The IDL is not a useful attack surface for XSS chaining. | `app/lib/anchor.ts:17-23` -- static imports | HIGH |

**Overall counterargument confidence: HIGH**

The race condition challenge has MEDIUM confidence because a developer in a rush could miss IDL changes in a large diff. However, this requires the Rust supply chain to already be compromised (past Cargo.lock), making it a compound-probability scenario.

## Related Findings
- Relates to DEP-01 context (package dependency analysis): The npm supply chain (`package-lock.json` gitignored) is a far more viable supply chain attack vector than the Rust/IDL path. The Rust side is properly pinned.
- May combine with a hypothetical "compromised developer workstation" scenario, but that is out of scope for a supply chain analysis.

## Incidental Discoveries
- **No IDL checksum verification**: `sync-idl.mjs` does not compare source and destination files before overwriting. A simple SHA-256 comparison and warning on changes would provide an additional layer of visibility. This is defense-in-depth, not a vulnerability. Suggested hypothesis: "Could a CI/CD step verify IDL integrity against a known-good baseline?"
- **IDL address field as implicit program ID source**: The `address` field in each IDL JSON (e.g., `tax_program.json:2`) is the canonical program ID used by Anchor's `Program` constructor. If this field were changed (e.g., to point to an attacker-controlled program), PDA derivation would produce wrong addresses and transactions would fail. However, if an attacker deployed a fake program at the tampered address with matching instruction layouts, transactions could succeed against the wrong program. This scenario requires both IDL tampering AND deploying a malicious program -- very high attacker effort. Suggested hypothesis: "H-NEW: Could a tampered IDL address field redirect transactions to a look-alike malicious program?"
