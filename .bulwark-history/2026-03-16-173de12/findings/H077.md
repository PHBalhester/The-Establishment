# Finding: H077 - Railway No Resource Limits

## Status: CONFIRMED
## Confidence Score: 7

## Rationale
Neither `railway.toml` nor `railway-crank.toml` specify memory limits, CPU limits, or replica counts. Railway PaaS does provide default limits per plan tier, but without explicit configuration, the services rely on Railway's implicit defaults. The web service has a healthcheck (`healthcheckPath = "/api/health"`, timeout 120s) and restart policy (ON_FAILURE, max 3 retries). The crank has ON_FAILURE with max 10 retries but no healthcheck. A memory leak in either service would consume resources until Railway's plan limit is hit.

## Code Evidence
```toml
# railway.toml -- no memory/CPU limits
[deploy]
startCommand = "npm run --workspace app start"
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 3

# railway-crank.toml -- no memory/CPU limits, no healthcheck
[deploy]
startCommand = "npx tsx scripts/crank/crank-runner.ts"
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 10
```

## Severity Estimate
LOW -- Railway enforces plan-level limits regardless of config. The crank's lack of healthcheck is the more material concern -- if it enters a hung state (not crashing, just not processing), Railway won't restart it.
