# Finding: H090 - Client Fetch Without Timeouts

## Status: CONFIRMED
## Confidence Score: 7

## Rationale
Client-side hooks that make fetch/RPC calls do not set explicit timeouts. In `app/hooks/useChartData.ts:140`, a bare `fetch(url)` call has no `AbortController` or timeout. In `app/hooks/useCarnageEvents.ts:102`, `fetch("/api/carnage-events")` also has no timeout. The Anchor `program.account.*.fetch()` calls in `useEpochState.ts:93`, `useStaking.ts:238`, `useCarnageData.ts:79`, and `useCurveState.ts:251` rely on the underlying RPC connection timeout which defaults to ~30 seconds in `@solana/web3.js`. While browser fetch has its own built-in timeout (browser-dependent, typically 300s), this is far too long for a responsive UI -- a hung RPC node would leave the UI in a loading state for minutes.

## Code Evidence
```typescript
// app/hooks/useChartData.ts:140
fetch(url)
  .then((res) => res.json())
  .then((data: CandleData[]) => { ... })

// app/hooks/useCarnageEvents.ts:102
const res = await fetch("/api/carnage-events");
```
No `AbortController`, no `signal`, no timeout wrapper in any client hook.

## Severity Estimate
LOW -- This is a UX/reliability issue. A slow or hung server will leave the UI in a perpetual loading state. No security impact since all financial enforcement is on-chain, but it degrades user experience and could mask actual outages.
