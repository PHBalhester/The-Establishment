# Finding: H045 - No Server-Side Error Reporting

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The server-side instrumentation file (`app/instrumentation.ts`) is an explicit no-op. The `register()` function is empty, with a comment stating that all @sentry/* packages conflict with Turbopack's SSR runtime. This means all server-side errors (API route crashes, SSR failures, webhook handler errors, database connection failures) are only visible in Railway's stdout logs. There is no structured error reporting, no alerting, no error aggregation, and no stack trace correlation for server-side code.

This is a deliberate architectural decision (Turbopack compatibility) but represents a real operational gap. The API routes handling financial-adjacent data (webhook ingestion, SSE broadcasting) have no error reporting pipeline. If the webhook handler crashes silently, chart data stops updating with no alert.

## Hypothesis Investigated

Server-side instrumentation is a no-op. Errors only in Railway logs.

## Investigation Path

### Step 1: Examine server-side instrumentation
**Location:** `app/instrumentation.ts:1-9`
**Finding:** Empty `register()` function. Explicit no-op.
**Evidence:**
```typescript
// Server-side instrumentation hook for Next.js.
// Currently a no-op -- error reporting is handled client-side via
// lib/sentry.ts (zero-dependency fetch-based reporter).
// Server-side errors surface through Railway's logging.

export async function register() {
  // No-op. All @sentry/* npm packages conflict with Turbopack's SSR
  // runtime, so we don't import any Sentry code on the server.
}
```

### Step 2: Check if lib/sentry.ts is imported server-side anywhere
**Location:** `app/lib/sentry.ts:26`
**Finding:** `captureException` works in both browser and Node.js (uses `fetch` which is available in both). However, it is only imported in `instrumentation-client.ts` (client-side), not in any API route or server component.
**Evidence:** `captureException` is imported at `instrumentation-client.ts:35` with the guard `typeof window !== "undefined"`. No server-side imports found.

### Step 3: Check API routes for error reporting
**Finding:** Searched all API routes -- none import or call `captureException`. Server-side errors are caught by try/catch blocks that return HTTP error responses but do not report to any monitoring system.

### Step 4: Verify the Turbopack constraint is real
**Finding:** Per MEMORY.md: "ALL @sentry/* npm packages break Turbopack SSR: @sentry/nextjs, @sentry/browser, @sentry/node all cause HTTP 500 on page routes." The zero-dependency `lib/sentry.ts` was built specifically to work around this. The same `captureException` function could theoretically be called server-side (it uses `fetch`, available in Node.js), but no one wired it up.

### Recommended Fix

Wire up `lib/sentry.ts` server-side. The `captureException` function already works with Node.js `fetch`:

```typescript
// app/instrumentation.ts
export async function register() {
  // Zero-dependency Sentry works server-side too (uses fetch, not @sentry/*)
  const { captureException } = await import("@/lib/sentry");

  // Capture unhandled rejections on the server
  process.on("unhandledRejection", (reason) => {
    captureException(reason);
  });

  process.on("uncaughtException", (error) => {
    captureException(error);
  });
}
```

Additionally, add `captureException` calls to catch blocks in critical API routes (webhook handler, SSE manager).

For the Sentry DSN, use `SENTRY_DSN` (server-side only, not `NEXT_PUBLIC_`) to avoid leaking a separate server key. The existing `lib/sentry.ts:30` already checks for `process.env.SENTRY_DSN` as a fallback.
