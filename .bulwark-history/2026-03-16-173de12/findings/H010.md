# Finding: H010 - Bonding Curve Authority Theft (Cross-Boundary)

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The bonding curve program's `prepare_transition` and `withdraw_graduated_sol` instructions accept ANY signer as `authority`. There is no stored authority pubkey in the `CurveState` struct, no `has_one` constraint, and no hardcoded admin check. Once both curves reach `Filled` status, any wallet can call `prepare_transition` to graduate both curves, then call `withdraw_graduated_sol` twice to drain ~1000 SOL per curve (~2000 SOL total) to their own wallet. The off-chain graduation script (`scripts/graduation/graduate.ts`) provides zero protection -- it is merely a convenience wrapper that calls these permissionless instructions.

This is a CRITICAL on-chain vulnerability that the off-chain layer cannot mitigate. The graduation script's use of an admin keypair is an operational convention, not a security enforcement. The same instructions are callable by anyone with a Solana wallet.

## Hypothesis Investigated

The SOS audit found that bonding curve instructions accept ANY signer. An attacker can call `withdraw_graduated_sol` directly, stealing ~1000 SOL per curve (~2000 SOL total). The off-chain graduation script provides no protection.

## Investigation Path

### Step 1: Examined CurveState struct for stored authority
**Location:** `programs/bonding_curve/src/state.rs:76-105`
**Finding:** The `CurveState` struct has NO `authority` field. It stores `token`, `token_mint`, `token_vault`, `sol_vault`, `tokens_sold`, `sol_raised`, `status`, `start_slot`, `deadline_slot`, `participant_count`, and `bump`. There is no record of who initialized or owns the curve.
**Evidence:**
```rust
pub struct CurveState {
    pub token: Token,                   // 1 byte
    pub token_mint: Pubkey,             // 32 bytes
    pub token_vault: Pubkey,            // 32 bytes
    pub sol_vault: Pubkey,              // 32 bytes
    pub tokens_sold: u64,               // 8 bytes
    pub sol_raised: u64,                // 8 bytes
    pub status: CurveStatus,            // 1 byte
    pub start_slot: u64,                // 8 bytes
    pub deadline_slot: u64,             // 8 bytes
    pub participant_count: u32,         // 4 bytes
    // ... bump
}
```

### Step 2: Examined prepare_transition authority check
**Location:** `programs/bonding_curve/src/instructions/prepare_transition.rs:17-38`
**Finding:** The `authority` field is `Signer<'info>` with NO constraints. No `has_one`, no `constraint = authority.key() == EXPECTED_ADMIN`, nothing. Any signer is accepted.
**Evidence:**
```rust
pub struct PrepareTransition<'info> {
    /// Protocol authority (deployer). Must sign.
    /// Matches the initialize_curve pattern: just a Signer, no hardcoded pubkey.
    pub authority: Signer<'info>,
    // ...
}
```

### Step 3: Examined withdraw_graduated_sol authority check
**Location:** `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs:25-51`
**Finding:** Same pattern. `authority: Signer<'info>` is mutable (receives SOL) but unconstrained. The SOL is transferred directly to whoever signs: `**ctx.accounts.authority.try_borrow_mut_lamports()? += withdrawable;` (line 81).
**Evidence:**
```rust
pub struct WithdrawGraduatedSol<'info> {
    #[account(mut)]
    pub authority: Signer<'info>,
    // ...
}
```
```rust
// Line 81: SOL goes to whoever signed
**ctx.accounts.authority.try_borrow_mut_lamports()? += withdrawable;
```

### Step 4: Examined off-chain graduation script for compensating controls
**Location:** `scripts/graduation/graduate.ts:327-366`
**Finding:** The script uses `authority.publicKey` from the loaded provider wallet. It calls `.signers([authority])` which signs with the admin keypair. However, this is entirely voluntary -- the on-chain instruction does not validate that this specific pubkey signed. Any keypair that signs the transaction will be accepted.
**Evidence:**
```typescript
// graduate.ts:354-362 -- just passes authority as signer, no on-chain validation
const sig = await programs.bondingCurve.methods
    .prepareTransition()
    .accountsStrict({
      authority: authority.publicKey,
      crimeCurveState,
      fraudCurveState,
    })
    .signers([authority])
    .rpc();
```

### Step 5: Verified same pattern across all 6 admin instructions
**Location:** Multiple files in `programs/bonding_curve/src/instructions/`
**Finding:** All admin-gated bonding curve instructions use the same unconstrained `Signer<'info>` pattern:
- `initialize_curve.rs:18` -- `pub authority: Signer<'info>` (no constraint)
- `fund_curve.rs:23` -- `pub authority: Signer<'info>` (no constraint)
- `start_curve.rs:18` -- `pub authority: Signer<'info>` (no constraint)
- `prepare_transition.rs:21` -- `pub authority: Signer<'info>` (no constraint)
- `withdraw_graduated_sol.rs:28` -- `pub authority: Signer<'info>` (no constraint)
- `close_token_vault.rs:28` -- `pub authority: Signer<'info>` (no constraint)

### Step 6: Confirmed the IDL is publicly available
**Location:** `app/idl/bonding_curve.json:877, 1510`
**Finding:** Both `prepare_transition` and `withdraw_graduated_sol` are in the published IDL (`app/idl/bonding_curve.json`), which is bundled in the client-side JavaScript. Any attacker can read the IDL, derive the correct PDAs, and construct the transactions.

### Step 7: Confirmed distribute_tax_escrow is intentionally permissionless
**Location:** `programs/bonding_curve/src/instructions/distribute_tax_escrow.rs:11-12`
**Finding:** `distribute_tax_escrow` is explicitly documented as permissionless ("anyone can call once the curve has graduated"), and it sends SOL to a hardcoded PDA (carnage fund), not to the caller. This is not exploitable -- the destination is fixed. This contrasts with `withdraw_graduated_sol` where SOL goes to the mutable `authority` (caller).

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Wait for both curves to reach `Filled` status | Yes | Public on-chain state; `CurveState.status` is readable by anyone |
| 2 | Call `prepare_transition` with attacker wallet as authority | Yes | `prepare_transition.rs:21` -- `authority: Signer<'info>` with no constraint |
| 3 | Call `withdraw_graduated_sol` for CRIME curve with attacker as authority | Yes | `withdraw_graduated_sol.rs:28,81` -- SOL sent to unconstrained `authority` |
| 4 | Call `withdraw_graduated_sol` for FRAUD curve with attacker as authority | Yes | Same as step 3 for second curve |
| 5 | Bundle all 3 calls in a single atomic transaction | Yes | All 3 instructions fit in one TX; no cross-TX dependencies |

## Invariant Analysis

### Target Invariant
Only the protocol deployer/admin should be able to withdraw graduated SOL from bonding curve vaults.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `prepare_transition.rs:21` | Requires a signer | No -- any signer accepted |
| `withdraw_graduated_sol.rs:28` | Requires a signer | No -- any signer accepted |

## Blocking Factors
- **Curves must be Filled**: Both curves must have sold 460M tokens each to reach `Filled` status. This is a natural precondition, not an attack blocker -- the attacker just waits.

## Enabling Factors
- **No stored authority on CurveState**: `programs/bonding_curve/src/state.rs:76-105` -- The struct lacks an authority pubkey field, so there is no reference point for access control.
- **Unconstrained Signer**: `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs:28` -- `Signer<'info>` with no `has_one` or constraint.
- **SOL destination is the signer**: `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs:81` -- `authority.try_borrow_mut_lamports()? += withdrawable` sends SOL to whoever signed.
- **IDL publicly available**: `app/idl/bonding_curve.json` -- Instruction schemas and account layouts are bundled in the frontend JavaScript.
- **PDA seeds are deterministic**: `["curve", mint]` and `["curve_sol_vault", mint]` -- anyone who knows the mint addresses (public) can derive the PDAs.

### Attack Scenario

```
Step 1: Monitor both CurveState accounts for status == Filled
  Target: On-chain CurveState PDAs (derived from ["curve", crime_mint] and ["curve", fraud_mint])
  Payload: getAccountInfo RPC calls
  Prerequisite: Mint addresses are public (in IDL, in client bundle)
  Result: Attacker knows when curves are ready

Step 2: Build atomic transaction with 3 instructions:
  Instruction 1: prepare_transition
    - authority: attacker_wallet (Signer)
    - crime_curve_state: PDA(["curve", crime_mint])
    - fraud_curve_state: PDA(["curve", fraud_mint])

  Instruction 2: withdraw_graduated_sol (CRIME)
    - authority: attacker_wallet (Signer, mut -- receives SOL)
    - curve_state: PDA(["curve", crime_mint])
    - sol_vault: PDA(["curve_sol_vault", crime_mint])
    - system_program

  Instruction 3: withdraw_graduated_sol (FRAUD)
    - authority: attacker_wallet (Signer, mut -- receives SOL)
    - curve_state: PDA(["curve", fraud_mint])
    - sol_vault: PDA(["curve_sol_vault", fraud_mint])
    - system_program

Step 3: Submit transaction with high priority fee
  Target: Solana RPC
  Prerequisite: Curves are Filled
  Result: All 3 instructions execute atomically

Final Result: ~2000 SOL transferred to attacker's wallet. Protocol is in Graduated
state with empty SOL vaults. Legitimate graduation script cannot proceed (no SOL
for pool seeding). Protocol is bricked -- users cannot trade via AMM.
```

### Recommended Fix

```rust
// Before (vulnerable) -- prepare_transition.rs:18-21
pub struct PrepareTransition<'info> {
    pub authority: Signer<'info>,
    // ...
}

// After (fixed) -- Option A: Store authority in CurveState
pub struct PrepareTransition<'info> {
    #[account(
        constraint = authority.key() == crime_curve_state.authority
            @ CurveError::UnauthorizedAuthority,
    )]
    pub authority: Signer<'info>,
    // ...
}

// CurveState must add: pub authority: Pubkey
```

```rust
// Before (vulnerable) -- withdraw_graduated_sol.rs:27-28
pub struct WithdrawGraduatedSol<'info> {
    #[account(mut)]
    pub authority: Signer<'info>,
    // ...
}

// After (fixed)
pub struct WithdrawGraduatedSol<'info> {
    #[account(
        mut,
        constraint = authority.key() == curve_state.authority
            @ CurveError::UnauthorizedAuthority,
    )]
    pub authority: Signer<'info>,
    // ...
}
```

The fix requires:
1. Add `pub authority: Pubkey` to `CurveState` struct.
2. Set `authority` during `initialize_curve`.
3. Add `has_one = authority` or equivalent constraint to all 6 admin instructions.

## Incidental Discoveries

- **close_token_vault also has no authority check** (`programs/bonding_curve/src/instructions/close_token_vault.rs:28`): The `authority: Signer<'info>` receives rent-recovered SOL. While the amount is small (rent for one token account, ~2M lamports / 0.002 SOL), this is exploitable by the same pattern. Since the token vault authority is the CurveState PDA (not the signer), the close CPI uses PDA signer seeds, making it executable by anyone. Suggested hypothesis: investigate all 6 bonding curve admin instructions for the same authority gap.

- **Race condition with legitimate admin**: If the admin is running the graduation script while an attacker monitors for `Filled` status, the attacker can front-run the admin's `prepare_transition` TX with a higher priority fee, graduating the curves and withdrawing SOL before the admin's TX lands. The admin's TX then fails (curves already graduated), and the attacker has the SOL. Suggested hypothesis: investigate whether the graduation script handles this failure gracefully or leaves partial state.

- **distribute_tax_escrow is safely permissionless**: Unlike the admin instructions, `distribute_tax_escrow` sends SOL to a hardcoded PDA (`carnage_sol_vault` derived from the epoch program). This is the correct pattern for permissionless instructions -- the destination is fixed, not caller-controlled.
