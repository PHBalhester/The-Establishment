# Finding: H062 - Candle Aggregator Float Price Precision

## Status: NOT VULNERABLE
## Confidence Score: 7

## Executive Summary

The candle aggregator at `app/db/candle-aggregator.ts:170-175` computes prices using JavaScript `number` (float64) division. The database schema stores prices as `real` (PostgreSQL float4, 32-bit). This introduces two layers of precision loss: float64 division in JS, then truncation to float32 in Postgres. However, this is NOT exploitable because: (1) candle prices are display-only data for TradingView charts -- they do not influence any transaction parameters; (2) the price ratios involved (lamports/token_units) are well within float64 safe range for display purposes; (3) even the float32 truncation only affects the 7th+ significant digit, which is irrelevant for chart display; and (4) no trading decisions are programmatically driven by candle data.

## Hypothesis Investigated

Float division accumulates rounding errors in candle prices.

## Investigation Path

### Step 1: Examine price computation
**Location:** `app/db/candle-aggregator.ts:159-209`
**Finding:** Prices are computed as ratios of integer amounts (lamports, base units). For buy: `price = netInput / outputAmount`. For sell: `price = grossOutput / inputAmount`. Both use JS `number` division.
**Evidence:**
```typescript
// candle-aggregator.ts:171-172
const netInput = swap.inputAmount - swap.taxAmount;
price = swap.outputAmount > 0 ? netInput / swap.outputAmount : 0;
```

### Step 2: Check database storage type
**Location:** `app/db/migrations/0000_simple_squadron_supreme.sql:6-8`
**Finding:** OHLCV price columns use `real` (Postgres float4, 32-bit IEEE 754).
**Evidence:**
```sql
"open" real NOT NULL,
"high" real NOT NULL,
"low" real NOT NULL,
"close" real NOT NULL,
```

### Step 3: Assess precision loss impact
**Location:** `app/db/candle-aggregator.ts` and chart display pipeline
**Finding:** Float64 has ~15 significant digits. Float32 has ~7 significant digits. For typical token prices (e.g., 0.0000025 SOL per token), float32 preserves 7 significant digits of the mantissa -- more than sufficient for chart display. The prices are never used for transaction construction, minimumOutput calculation, or any financial enforcement.

### Step 4: Verify candle data is display-only
**Location:** `app/app/api/candles/route.ts` (API endpoint serving candles)
**Finding:** Candle data flows: webhook -> aggregator -> Postgres -> API route -> TradingView chart component. No code path feeds candle prices back into swap transactions or slippage calculations.

## Blocking Factors
- **Display-only data**: Candle prices are never used for transaction construction
- **Adequate precision**: Float32 provides 7 significant digits -- sufficient for price display
- **On-chain authority**: Swap execution uses real-time AMM reserves, not historical prices

## Enabling Factors
- **Double truncation**: Float64 -> Float32 loses precision at ~7th digit
- **No NUMERIC/DECIMAL type**: Using `real` instead of `numeric` for financial-adjacent data
