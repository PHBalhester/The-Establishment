# Finding: S009 - Graduation Race -- Authority + Keypair Combo

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

This chain attack combines H010 (bonding curve authority theft -- ANY signer accepted), H005 (keypairs committed to git), and H016 (transfer hook init front-running) into a devastating graduation-phase attack that could steal ~2000 SOL and brick the protocol's token transfer system. The bonding curve's `prepare_transition` and `withdraw_graduated_sol` instructions accept ANY signer (H010). The committed keypair files in `keypairs/` (H005) give an attacker deterministic knowledge of all program IDs and PDA addresses before deployment. The transfer hook's `initialize_authority` instruction uses a first-caller-wins pattern (H016). Together, an attacker who monitors for `Filled` curve status can atomically: (a) graduate both curves, (b) withdraw ~2000 SOL to their wallet, and (c) front-run the transfer hook authority initialization during the inevitable redeploy, bricking all token transfers.

This is a multi-phase attack: Phase 1 (graduation theft) is immediately executable once curves fill. Phase 2 (hook authority capture) exploits the team's forced response to the theft -- they must redeploy, and the attacker already knows the program IDs and can pre-build front-running transactions.

## Hypothesis Investigated

Committed keypairs provide deterministic PDA knowledge. Attacker monitors for Filled curves. Atomically withdraws ~2000 SOL (H010). Then front-runs hook init on the inevitable redeploy (H016), using pre-computed program addresses from committed keypairs (H005).

## Investigation Path

### Step 1: Confirm Bonding Curve Authority is Unchecked
**Location:** `programs/bonding_curve/src/instructions/prepare_transition.rs:17-21`, `programs/bonding_curve/src/instructions/withdraw_graduated_sol.rs:25-28`
**Finding:** Both instructions use `pub authority: Signer<'info>` with NO `has_one` constraint, NO stored authority field in `CurveState`, and NO hardcoded admin check. SOL goes directly to the signer at `withdraw_graduated_sol.rs:81`.
**Evidence:**
```rust
// prepare_transition.rs:21
pub authority: Signer<'info>,  // No constraint

// withdraw_graduated_sol.rs:81
**ctx.accounts.authority.try_borrow_mut_lamports()? += withdrawable;  // SOL to caller
```

### Step 2: Confirm Keypair Files are Committed and Provide PDA Knowledge
**Location:** `keypairs/` directory (12 files tracked by git)
**Finding:** All program deploy keypairs are committed: `amm-keypair.json`, `bonding-curve-keypair.json`, `transfer-hook-keypair.json`, `tax-program-keypair.json`, `epoch-program.json`, `staking-keypair.json`, `vault-keypair.json`. This gives an attacker deterministic knowledge of all program IDs (program ID = keypair's public key on Solana). With program IDs known, all PDAs are derivable: `["curve", mint]`, `["curve_sol_vault", mint]`, `["authority"]` (hook whitelist), etc.
**Evidence:**
```
$ git ls-files keypairs/
keypairs/bonding-curve-keypair.json
keypairs/transfer-hook-keypair.json
keypairs/amm-keypair.json
... (12 files total)
```

### Step 3: Confirm Graduation Script Has No Race Protection
**Location:** `scripts/graduation/graduate.ts:354-362`
**Finding:** The script calls `prepareTransition()` with the admin's keypair, but since the on-chain instruction accepts any signer, an attacker can front-run the admin's TX with a higher priority fee. The script does NOT check if the curve has already been graduated by someone else before calling `withdrawGraduatedSol`.
**Evidence:**
```typescript
// graduate.ts:354-362
const sig = await programs.bondingCurve.methods
    .prepareTransition()
    .accountsStrict({
      authority: authority.publicKey,  // Anyone can be authority
      crimeCurveState,
      fraudCurveState,
    })
    .signers([authority])
    .rpc();
```

### Step 4: Confirm Atomic 3-Instruction Attack is Feasible
**Location:** Bonding curve IDL analysis
**Finding:** All three instructions (`prepare_transition`, `withdraw_graduated_sol` x2) can fit in a single Solana transaction. The accounts are:
- `prepare_transition`: authority (signer), crimeCurveState, fraudCurveState
- `withdraw_graduated_sol` (CRIME): authority (signer, mut), curveState, solVault, systemProgram
- `withdraw_graduated_sol` (FRAUD): authority (signer, mut), curveState, solVault, systemProgram

Total unique accounts: ~8 (authority shared across all 3). Well within the 64-account transaction limit.

### Step 5: Confirm Transfer Hook Init Front-Running on Redeploy
**Location:** `programs/transfer-hook/src/instructions/initialize_authority.rs:27-33`
**Finding:** `initialize_authority` uses `#[account(init, ...)]` with a bare `Signer<'info>`. First caller wins. The PDA seeds are `[b"authority"]` -- public knowledge. The deploy pipeline (`deploy-all.sh:155-168`) has a multi-minute gap between program deployment and initialization.
**Evidence:**
```rust
// initialize_authority.rs
pub signer: Signer<'info>,  // No constraint
#[account(
    init,
    payer = signer,
    space = 8 + WhitelistAuthority::INIT_SPACE,
    seeds = [WhitelistAuthority::SEED],
    bump
)]
pub whitelist_authority: Account<'info, WhitelistAuthority>,
```

### Step 6: Confirm Initialize Script Silently Skips Captured Authority
**Location:** `scripts/deploy/initialize.ts:321-336`
**Finding:** The init script checks `accountExists(whitelistAuthority)` and SKIPs initialization if the account already exists -- without verifying the stored authority matches the deployer's pubkey. An attacker who captures the authority during step 5 would not be detected by the init script.
**Evidence:**
```typescript
if (await accountExists(connection, whitelistAuthority)) {
    log.step(stepNum, TOTAL_STEPS, "Initialize WhitelistAuthority", "SKIPPED");
    skipped++;
}
```

## Blocking Factors
- **Curves must reach Filled**: Both curves must sell 460M tokens each. This is a natural precondition -- the attacker simply waits. At bonding curve prices, this represents ~2000 SOL of real demand (1000 per curve).
- **Repo must be accessible**: If the repo is private, keypair files are not extractable (unless attacker has contributor access, CI token, or gains access through other means).

## Enabling Factors
- **Unconstrained authority on bonding curve**: `prepare_transition.rs:21`, `withdraw_graduated_sol.rs:28` -- H010

### Severity Escalation from Individual Findings
H010 alone = CRITICAL (~2000 SOL theft). H005 alone = HIGH (keypair exposure). H016 alone = CRITICAL (protocol brick). The chain is CRITICAL with maximum conviction because:
1. H005 (committed keypairs) provides intelligence that makes BOTH H010 and H016 pre-computable -- the attacker knows exact program IDs, PDAs, and account layouts before any deployment occurs.
2. H010's theft forces a team response (redeploy), which creates the window for H016.
3. The attacker can prepare Phase 2 transactions BEFORE Phase 1 even executes, because all addresses are deterministic.

### Recommended Fix

**Fix 1: On-chain authority check on bonding curve (blocks H010):**
```rust
// Add to CurveState:
pub authority: Pubkey,

// Add to prepare_transition and withdraw_graduated_sol:
#[account(
    constraint = authority.key() == curve_state.authority
        @ CurveError::UnauthorizedAuthority,
)]
pub authority: Signer<'info>,
```

**Fix 2: ProgramData upgrade-authority check on hook init (blocks H016):**
```rust
// initialize_authority.rs -- gate with upgrade authority
#[account(
    constraint = program.programdata_address()? == Some(program_data.key()),
)]
pub program: Program<'info, TransferHook>,
#[account(
    constraint = program_data.upgrade_authority_address == Some(signer.key()),
)]
pub program_data: Account<'info, ProgramData>,
```

**Fix 3: Add keypairs/ to .gitignore (blocks H005 intelligence):**
```gitignore
keypairs/
```
Generate completely new keypairs for mainnet. Purge from git history via `git filter-repo`.

**Fix 4: Atomic deploy+init (closes time window):**
Use Jito bundles to atomically deploy the program and call `initializeAuthority` in the same block, or at minimum, add ownership verification to the init script:
```typescript
// initialize.ts -- verify authority ownership
if (whitelistAuthorityData &&
    !whitelistAuthorityData.authority.equals(authority.publicKey)) {
  log.error("WhitelistAuthority captured by attacker! ABORT.");
  process.exit(1);
}
```

## Incidental Discoveries
- **Graduation state file has no integrity protection**: `scripts/graduation/graduate.ts` uses `graduation-state.json` for checkpoint/resume with no HMAC or signature. If an attacker can modify this file (e.g., via filesystem access on the admin machine), they could skip steps or replay steps. However, this requires local access, making it a lower-severity concern. Location: graduate.ts state persistence. Suggested hypothesis: Investigate whether graduation-state.json tampering could cause the script to skip critical steps or re-execute harmful operations.
- **All 6 bonding curve admin instructions have the same authority gap**: Not just `prepare_transition` and `withdraw_graduated_sol`, but also `initialize_curve`, `fund_curve`, `start_curve`, and `close_token_vault` accept any signer. A pre-graduation attack on `start_curve` could start curves early/improperly. Suggested hypothesis: Investigate whether `initialize_curve` or `start_curve` could be exploited before curves are meant to be active.
