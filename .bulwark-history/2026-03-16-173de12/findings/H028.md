# Finding: H028 - Health Endpoint Information Disclosure

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The `/api/health` endpoint at `app/app/api/health/route.ts:28-56` is publicly accessible with no authentication and returns dependency topology information: whether Postgres is reachable and whether the Solana RPC is reachable, along with an overall status of "ok" or "degraded". While the endpoint does not expose connection strings, credentials, or version numbers, it does reveal the existence and current availability of backend dependencies to any unauthenticated caller.

This is a low-severity information disclosure. An attacker gains: (1) confirmation that the application uses PostgreSQL, (2) confirmation that it connects to a Solana RPC, and (3) real-time availability status of each. This topology information aids reconnaissance and can help time attacks (e.g., attacking when RPC is degraded, knowing the app relies on Postgres for data integrity).

## Hypothesis Investigated

`/api/health` exposes dependency topology to unauthenticated callers.

## Investigation Path

### Step 1: Health endpoint response structure
**Location:** `app/app/api/health/route.ts:28-56`
**Finding:** Returns JSON with `status`, `checks: { postgres, solanaRpc }`, and `timestamp`. Always returns HTTP 200 (by design, for Railway liveness).
**Evidence:**
```typescript
export async function GET() {
  let postgres = false;
  let solanaRpc = false;

  try {
    await db.execute(sql`SELECT 1`);
    postgres = true;
  } catch (err) {
    console.error("[health] Postgres check failed:", err);
  }

  try {
    const connection = getConnection();
    await connection.getSlot();
    solanaRpc = true;
  } catch (err) {
    console.error("[health] Solana RPC check failed:", err);
  }

  const status = postgres && solanaRpc ? "ok" : "degraded";

  return NextResponse.json({
    status,
    checks: { postgres, solanaRpc },
    timestamp: new Date().toISOString(),
  });
}
```

### Step 2: Authentication check
**Location:** `app/app/api/health/route.ts:28`
**Finding:** No authentication, no rate limiting, no IP restriction. Any anonymous HTTP GET returns the full response.

### Step 3: Error logging check
**Location:** `app/app/api/health/route.ts:37,47`
**Finding:** Failed checks log the full error to `console.error` (server-side only, Railway logs). The error object is NOT included in the HTTP response -- only the boolean pass/fail is exposed to the caller. This is the correct pattern.

### Step 4: What is NOT exposed
**Finding:** The endpoint does NOT expose:
- Database connection strings or credentials
- RPC URLs or API keys
- Software versions
- Internal IP addresses
- Stack traces or error details

Only boolean connectivity status per dependency.

## Invariant Analysis

### Target Invariant
Internal infrastructure topology should not be disclosed to unauthenticated external callers.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `route.ts:28` | No auth gate | **No** -- endpoint is fully public |
| `route.ts:37,47` | Error details server-side only | Yes -- errors not in response |
| `route.ts:51-55` | Response structure | Partial -- reveals dependency names |

## Blocking Factors
- Error details are kept server-side: `route.ts:37,47` -- actual error objects only go to `console.error`, not the HTTP response
- No credentials, URLs, or versions exposed in the response

## Enabling Factors
- No authentication: `route.ts:28` -- any anonymous caller gets the response
- No rate limiting: the endpoint can be polled continuously for real-time monitoring
- Dependency names revealed: `postgres` and `solanaRpc` confirm the technology stack

## Recommended Fix
```typescript
// Option A: Return only liveness status to unauthenticated callers
// Add detailed checks only when an admin token is provided
export async function GET(req: NextRequest) {
  const authHeader = req.headers.get("authorization");
  const isAdmin = authHeader === `Bearer ${process.env.HEALTH_CHECK_SECRET}`;

  // Always return 200 for Railway liveness
  if (!isAdmin) {
    return NextResponse.json({ status: "ok", timestamp: new Date().toISOString() });
  }

  // Full checks only for authenticated monitoring
  // ... existing postgres + solanaRpc checks ...
}

// Option B: Genericize the response (minimal change)
return NextResponse.json({
  status,
  timestamp: new Date().toISOString(),
  // Remove: checks: { postgres, solanaRpc }
});
```

Option B is the minimal fix -- remove the `checks` object from the public response. Railway's liveness check only needs the HTTP 200, not the body details.

## Incidental Discoveries
- **Health endpoint as DoS amplifier**: Each health check call triggers a Postgres query (`SELECT 1`) AND a Solana RPC call (`getSlot`). Without rate limiting, an attacker could poll the health endpoint at high frequency to amplify load on both Postgres and the Solana RPC, potentially contributing to rate limit exhaustion on Helius. Suggested hypothesis: investigate rate limiting across all API endpoints, particularly those that trigger upstream service calls.
