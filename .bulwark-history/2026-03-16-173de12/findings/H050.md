# Finding: H050 - Webhook No Body Size Limit

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The Helius webhook handler at `app/app/api/webhooks/helius/route.ts` has no explicit body size limit on incoming POST requests. The handler calls `await req.json()` which reads the entire request body into memory before parsing. Next.js App Router (runtime: "nodejs") does not impose a body size limit by default for route handlers -- the `bodyParser` limit from Pages Router does not apply to App Router API routes.

An attacker can send a very large JSON payload (e.g., 100MB+ array of fake transactions) that would be fully buffered into Node.js heap memory. Combined with the fail-open authentication, this enables a memory exhaustion attack against the Railway-hosted Next.js process.

The `next.config.ts` has no `serverBodyLimit` configuration, and the route handler has no manual size check before calling `req.json()`.

## Hypothesis Investigated

No explicit body size limit on webhook POST; large payloads exhaust memory.

## Investigation Path

### Step 1: Request body handling
**Location:** `app/app/api/webhooks/helius/route.ts:146`
**Finding:** `await req.json()` buffers the entire body into memory with no size check.
**Evidence:**
```typescript
const transactions: HeliusTransaction[] = await req.json();
```

### Step 2: Next.js body size defaults
**Location:** `app/next.config.ts:25-91`
**Finding:** No `serverBodyLimit` or equivalent configuration. Next.js App Router route handlers do not have a default body size limit (unlike Pages Router's 1MB default).
**Evidence:** Full config read -- no body limit anywhere.

### Step 3: Pre-parse size check
**Location:** `app/app/api/webhooks/helius/route.ts:131-146`
**Finding:** No `Content-Length` check or streaming parser before `req.json()`.

### Step 4: Auth bypass
**Location:** `app/app/api/webhooks/helius/route.ts:135-141`
**Finding:** Fail-open auth means unauthenticated requests can send large payloads.

## Blocking Factors
- Railway may impose a maximum request size at the load balancer level (typically 100MB) -- but this is still far too large for a webhook handler that expects small JSON arrays.

## Enabling Factors
- No body size check before `req.json()`: `route.ts:146`
- No `serverBodyLimit` in Next.js config: `next.config.ts`
- Fail-open auth: `route.ts:135-141`
- No rate limiting on the endpoint
- `runtime = "nodejs"` (not Edge, which has a 4MB limit): `route.ts:48`

```
  Payload: ~100MB allocated on V8 heap (JSON parsing doubles memory briefly)
  Prerequisite: Step 1

  Payload: 5 x 100MB

```

### Recommended Fix
```typescript
// Before (vulnerable)
const transactions: HeliusTransaction[] = await req.json();

// After (fixed) -- check Content-Length before parsing
export async function POST(req: NextRequest): Promise<NextResponse> {
  // Body size limit: 5MB (Helius batches are typically < 500KB)
  const MAX_BODY_SIZE = 5 * 1024 * 1024;
  const contentLength = parseInt(req.headers.get("content-length") || "0", 10);
  if (contentLength > MAX_BODY_SIZE) {
    return NextResponse.json({ error: "Payload too large" }, { status: 413 });
  }

  // Also read body as text first to check actual size
  const body = await req.text();
  if (body.length > MAX_BODY_SIZE) {
    return NextResponse.json({ error: "Payload too large" }, { status: 413 });
  }

  const transactions: HeliusTransaction[] = JSON.parse(body);
  // ... rest of handler
}
```
