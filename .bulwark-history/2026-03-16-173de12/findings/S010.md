# Finding: S010 - VRF Recovery Path Non-Atomic Carnage Window

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The VRF recovery path in `scripts/vrf/lib/vrf-flow.ts:388-534` explicitly does NOT bundle Carnage execution atomically with the reveal+consume transaction. When the recovery path completes an epoch transition that triggers Carnage, it returns `carnageExecutedAtomically: false` (line 533). The `CarnagePending` flag is set on-chain by `consume_randomness`, but the Carnage swap is not executed in the same transaction. This creates an observable on-chain state where the market knows Carnage is about to execute (visible `carnage_pending = true` on the EpochState account), which pool will be targeted, and approximately how much SOL will be spent. MEV bots can front-run the Carnage swap by buying tokens from the target pool, forcing the Carnage to execute at a worse rate, then selling after Carnage pushes the price up.

The normal (non-recovery) VRF path correctly bundles reveal+consume+carnage atomically in TX3 (line 310-326), closing this MEV window. The recovery path was intentionally not given atomic bundling (comment at line 533: "Recovery path does not attempt atomic Carnage"). While the recovery path is only triggered when VRF is stale (oracle timeout or failure), an attacker may be able to FORCE VRF recovery by DoSing the Switchboard oracle gateway during the critical reveal window.

## Hypothesis Investigated

VRF recovery path (`vrf-flow.ts:533`) does NOT bundle Carnage atomically. If VRF timeout is forced, recovery creates a window for MEV extraction by front-running the Carnage swap.

## Investigation Path

### Step 1: Confirm Normal Path is Atomic (Baseline)
**Location:** `scripts/vrf/lib/vrf-flow.ts:297-328`
**Finding:** The normal VRF path bundles `revealIx + consumeIx + carnageIx` in a single v0 VersionedTransaction. This is the CARN-002 fix -- no `CarnagePending` event is visible on-chain before the swap completes.
**Evidence:**
```typescript
// vrf-flow.ts:310-326
if (accounts.carnageAccounts && accounts.alt) {
  const carnageIx = await buildExecuteCarnageAtomicIx(...);
  const instructions = [
    ComputeBudgetProgram.setComputeUnitLimit({ units: 600_000 }),
    revealIx,
    consumeIx,
    carnageIx,
  ];
  const sig = await sendV0Transaction(...);  // Atomic
}
```

### Step 2: Confirm Recovery Path is NOT Atomic
**Location:** `scripts/vrf/lib/vrf-flow.ts:388-534`
**Finding:** The recovery path uses `sendRevealAndConsume()` at line 419 and 492, which DOES attempt atomic carnage bundling (if `accounts.carnageAccounts && accounts.alt` is truthy at line 297). However, examining the recovery code flow more carefully:

Line 419: `recoveryConsumeSig = await sendRevealAndConsume(provider, epochProgram, accounts, staleRevealIx, stalePubkey, wallet)` -- This calls the same `sendRevealAndConsume` function that bundles carnage. BUT the return value at line 533 explicitly says `carnageExecutedAtomically: false`.

Let me re-examine. The `sendRevealAndConsume` function at line 269-341 DOES bundle carnage if `accounts.carnageAccounts && accounts.alt` are present. The crank runner provides these at `crank-runner.ts:150-171`. So `sendRevealAndConsume` in the recovery path WOULD include carnage atomically...

**Re-examination at line 533:** The return value `carnageExecutedAtomically: false` is a signal to the CALLER, not a statement about what happened. Let me trace whether the carnage actually executes atomically.

### Step 3: Deep Trace of Recovery Path Carnage Bundling
**Location:** `scripts/vrf/lib/vrf-flow.ts:269-341`, `419`, `492`
**Finding:** Both recovery sub-paths call `sendRevealAndConsume()`:
- Line 419: Stale reveal succeeded path
- Line 492: Retry with fresh randomness path

`sendRevealAndConsume()` at line 297 checks `if (accounts.carnageAccounts && accounts.alt)` and if true, bundles `revealIx + consumeIx + carnageIx` in a v0 TX. The crank runner at `crank-runner.ts:150-171` provides `carnageAccounts` and `alt` in the `VRFAccounts` struct. So the recovery path DOES attempt atomic carnage bundling through the shared `sendRevealAndConsume` function.

However, the `carnageExecutedAtomically: false` flag at line 533 suggests the developer believes recovery does NOT bundle atomically. There may be a case where `accounts.carnageAccounts` or `accounts.alt` is not set in the recovery context, OR the developer set this flag as a documentation marker even though the code does bundle.

### Step 4: Identify When Atomic Bundling Fails in Recovery
**Location:** `scripts/vrf/lib/vrf-flow.ts:331-340`
**Finding:** If `accounts.carnageAccounts` OR `accounts.alt` is falsy, `sendRevealAndConsume` falls through to the "legacy path" at line 331-340, which sends ONLY `revealIx + consumeIx` WITHOUT carnage:
```typescript
// Legacy path: reveal + consume only (no carnageAccounts provided)
const consumeTx = new Transaction().add(
  ComputeBudgetProgram.setComputeUnitLimit({ units: 400_000 }),
  revealIx,
  consumeIx
);
```

In the crank runner context, `carnageAccounts` and `alt` ARE provided. However, the `carnageExecutedAtomically: false` flag suggests the developer intentionally set this to `false` for the recovery path, possibly because:
1. The recovery path's carnage bundling may fail (compute budget exceeded with recovery overhead)
2. The developer chose not to rely on it working in recovery scenarios
3. The flag is used by the crank runner to decide if a SEPARATE carnage execution is needed

### Step 5: Trace How Crank Runner Handles non-atomic Carnage
**Location:** `scripts/crank/crank-runner.ts:280-300`
**Finding:** The crank runner logs the `carnageAtomic` result but does NOT have explicit code to execute Carnage separately if `carnageExecutedAtomically === false`. It relies on `sendRevealAndConsume` having bundled it.

However, if the v0 bundled TX fails due to compute limits (recovery path + carnage may exceed 600K CU), the entire TX would fail, and the recovery path would throw, hitting the catch block at line 302 and retrying after 30 seconds. During this 30-second retry window, the epoch state has `carnage_pending = true` on-chain (if consume succeeded in a previous attempt without carnage).

### Step 6: Identify the MEV Window
**Location:** On-chain `EpochState.carnage_pending` field
**Finding:** The MEV window exists in the following scenario:
1. Normal VRF flow: TX2 (trigger) succeeded, but oracle didn't respond in time
2. Recovery path activates (VRF timeout after 300 slots)
3. Recovery calls `sendRevealAndConsume` with carnage bundling
4. If the bundled TX FAILS (compute exceeded, network issue):
   - `consume_randomness` may have succeeded in a partial/separate attempt
   - `carnage_pending = true` is now visible on-chain
   - Carnage execution happens in a separate TX (manual retry or next crank cycle)
5. During the gap between consume and carnage execution, MEV bots can front-run

Even without TX failure, there is a more subtle window: between TX2 (trigger_epoch_transition, which may set Carnage hints via VRF bytes) and TX3 (recovery reveal+consume), an observer knows an epoch transition is in progress and can position trades.

### Step 7: Assess Forced VRF Recovery
**Finding:** An attacker can potentially force the recovery path by:
1. Identifying when the crank submits TX2 (commit + trigger) -- observable on-chain
2. DoSing the Switchboard oracle gateway to prevent the oracle from responding
3. After 300 slots (~2 minutes), the crank enters recovery
4. Recovery creates fresh randomness and reveals/consumes

The oracle DoS is the weakest link in this attack. Switchboard uses multiple gateways and oracles. However, the project's own MEMORY.md notes: "VRF gateway rotation DOES NOT WORK: Each randomness account is assigned to a specific oracle." If the assigned oracle's gateway is DoSed, the VRF timeout is forced.

## Invariant Analysis

### Target Invariant
The Carnage execution should never be observable on-chain before it completes. The `CarnagePending` state should not be visible in a separate transaction from the Carnage swap.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `vrf-flow.ts:310-326` | Normal path: atomic reveal+consume+carnage | Yes |
| `vrf-flow.ts:297-328` | `sendRevealAndConsume` bundles carnage if accounts present | Yes (but may fail) |
| `vrf-flow.ts:533` | Recovery path: marks non-atomic | Documents the gap |
| `crank-runner.ts:302-314` | Error retry after 30s | Yes, but creates temporal gap |
| (MISSING) | Separate carnage execution if bundled TX fails | Not implemented |

## Blocking Factors
- **Normal path is atomic**: `vrf-flow.ts:310-326` -- In the normal (non-recovery) VRF flow, Carnage is always bundled. This is the common path.
- **Oracle DoS is difficult**: Switchboard oracles are professionally operated infrastructure. DoSing them requires sustained effort against hosted services.
- **Recovery is rare**: The recovery path only activates on VRF timeout (oracle failure), which is uncommon under normal conditions.

## Enabling Factors
- **Recovery path non-atomic flag**: `vrf-flow.ts:533` -- `carnageExecutedAtomically: false`
- **VRF gateway single-oracle assignment**: MEMORY.md -- "Each randomness account is assigned to a specific oracle"
- **300-slot timeout window**: `vrf-flow.ts:433` -- known delay before recovery activates
- **No MEV protection on crank transactions**: Standard Helius RPC, no Jito bundles
- **CarnagePending is publicly observable**: `EpochState.carnage_pending` is an on-chain boolean
- **Carnage target is predictable**: VRF bytes determine which pool Carnage targets; after consume, the target is known
- **30-second error retry**: `crank-runner.ts:310-313` -- if bundled TX fails, 30s gap before retry

## If CONFIRMED

### Severity Assessment
- **Severity:** MEDIUM
- **Likelihood:** LOW-MEDIUM (requires oracle DoS AND recovery TX failure AND MEV positioning)
- **Impact:** MEV extraction from Carnage swap. The Carnage fund typically holds a fraction of protocol fees (staking/carnage split). At devnet pool sizes, the extractable value is modest. At mainnet scale with larger fee accumulation, it could be significant.

### Severity Rationale
The normal path (>99% of epochs) is correctly atomic. The recovery path has a theoretical MEV window, but exploiting it requires: (1) forcing VRF recovery via oracle DoS, (2) the bundled recovery TX failing, (3) MEV positioning during the gap. This multi-condition chain reduces likelihood. However, the attack is profitable when it succeeds (MEV extraction from a known buy), and the project's own code acknowledges the gap (`carnageExecutedAtomically: false`).

### Attack Scenario
```
Step 1: Monitor for epoch transition TX2
  Target: EpochState account subscription
  Action: Watch for vrf_pending = true
  Result: Know that VRF commit has been made, reveal expected within ~3 slots

Step 2: DoS the assigned oracle gateway
  Target: Switchboard oracle HTTP gateway
  Action: Flood the gateway to prevent oracle response
  Prerequisite: Identify gateway URL (Switchboard uses known endpoints)
  Result: Oracle doesn't respond within VRF timeout window

Step 3: Wait for recovery path activation (~2 minutes)
  Result: Crank waits VRF_TIMEOUT_SLOTS (300 slots), enters recovery

Step 4: Recovery creates fresh randomness and sends bundled TX
  Result: Recovery TX either:
    (a) Succeeds with atomic carnage -> attack fails (no window)
    (b) Fails due to compute/network -> carnage_pending visible on-chain

Step 5: If (b), front-run Carnage swap
  Target: Pool that Carnage will buy from (determined by VRF/epoch state)
  Action: Buy tokens from target pool before Carnage executes
  Prerequisite: carnage_pending = true on-chain; target pool known
  Result: Attacker holds tokens; pool price moved up slightly

Step 6: Crank retries Carnage execution (after 30s or next cycle)
  Result: Carnage buys tokens at higher price (attacker's front-run raised it)

Step 7: Attacker sells tokens
  Result: MEV profit = Carnage slippage due to front-running

Final Result: MEV extraction proportional to Carnage fund size.
At devnet scale: small. At mainnet scale: potentially significant
(Carnage fund accumulates from 60% of all tax revenue).
```

### Recommended Fix

**Fix 1: Explicit carnage execution in recovery error handling:**
```typescript
// crank-runner.ts -- after line 295, check for non-atomic carnage
if (result.carnageTriggered && !result.carnageExecutedAtomically) {
  console.log("[crank] Carnage triggered but not atomic. Executing separately...");
  try {
    await executeCarnageSeparately(provider, programs, vrfAccounts);
  } catch (carnErr) {
    console.error(`[crank] Separate Carnage execution failed: ${String(carnErr).slice(0, 200)}`);
    // Will retry on next cycle
  }
}
```

**Fix 2: Force atomic bundling in recovery path:**
```typescript
// vrf-flow.ts recovery path -- ensure carnage is always bundled
// If sendRevealAndConsume fails with carnage bundled, retry with
// higher compute budget (800K CU) before falling back to non-atomic
```

**Fix 3: Use Jito bundles for Carnage execution:**
Submit Carnage transactions via Jito to prevent front-running regardless of atomicity:
```typescript
// Use jito-ts to submit carnage as a private transaction bundle
import { searcherClient } from "jito-ts/dist/sdk/block-engine/searcher";
```

**Fix 4: On-chain Carnage slippage protection:**
The on-chain `executeCarnageAtomic` instruction already reads pool reserves and computes output. Ensure it has a minimum output check that limits MEV extraction even if front-run.

## Incidental Discoveries
- **VRF recovery is the only non-atomic Carnage path**: All other Carnage execution paths go through `sendRevealAndConsume` with the atomic bundling check. The recovery path is the sole exception, making it a high-value target for MEV attackers who can force its activation. Location: `vrf-flow.ts:297-341` vs `vrf-flow.ts:388-534`. Suggested hypothesis: Investigate whether there are other code paths that could cause `consume_randomness` to execute without atomic Carnage (e.g., manual admin scripts, test flows).
- **30-second error retry is a fixed, predictable interval**: `crank-runner.ts:310-313` uses `ERROR_RETRY_DELAY_MS = 30_000`. If an MEV bot forces a TX failure, they know exactly when the retry will happen (30s later). A randomized backoff would make timing-based attacks harder. Location: `crank-runner.ts:57`. Suggested hypothesis: Investigate whether the predictable retry interval enables timing-based MEV strategies beyond Carnage front-running.
- **No alerting on VRF recovery**: When recovery activates, it logs to stdout but doesn't trigger any external alert (Discord webhook, PagerDuty). The team doesn't know the oracle failed or that recovery was needed. Location: `vrf-flow.ts:401`. Suggested hypothesis: Investigate whether sustained oracle failures could go undetected and whether an attacker could exploit repeated forced recoveries.
