# Finding: H114 - globalThis Singleton HMR Leak

## Status: NOT VULNERABLE
## Confidence Score: 8

## Rationale
Both `app/db/connection.ts` and `app/lib/sse-manager.ts` use the `globalThis` singleton pattern specifically to prevent HMR resource leaks -- this is the standard Next.js recommended pattern, not a vulnerability. In `connection.ts` (lines 28-31, 50-55), the pgClient is cached on globalThis only in non-production mode, and the Proxy pattern ensures lazy connection creation. In `sse-manager.ts` (lines 84-92), the SSEManager singleton is similarly cached on globalThis only in development. In production, there is exactly one module load, so globalThis caching is harmless. The pattern correctly prevents the exact problem it was designed for: each HMR cycle reusing the existing connection pool instead of creating new ones. The subscriber Set is correctly cleaned up when callbacks throw (line 63-64 of sse-manager.ts).

## Code Evidence
```typescript
// connection.ts:53-55 - Only cache in dev
if (process.env.NODE_ENV !== "production") {
  globalForDb.pgClient = client;
}

// sse-manager.ts:90-92 - Same pattern
if (process.env.NODE_ENV !== "production") {
  globalForSSE.sseManager = sseManager;
}
```

This is the correct implementation of the Next.js singleton pattern. No leak in production.
