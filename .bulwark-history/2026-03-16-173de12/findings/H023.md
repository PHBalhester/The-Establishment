# Finding: H023 - SSE Connection Exhaustion DoS

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The SSE endpoint at `/api/sse/candles` accepts unlimited connections without any authentication, rate limiting, or subscriber cap. The `SSEManager` class uses an unbounded `Set<SSECallback>` with no maximum size check. An attacker can open thousands of concurrent SSE connections, each of which holds a `ReadableStream`, a `setInterval` timer (15-second heartbeat), and a subscriber callback in the in-memory Set. This exhausts server memory and file descriptors on the Railway container, causing denial of service for all users.

Additionally, each `broadcast()` call iterates the entire subscriber Set (O(n)), meaning that as connections accumulate, webhook processing latency degrades proportionally, creating a feedback loop where Helius webhook deliveries timeout and retry, amplifying the load further.

## Hypothesis Investigated

SSE endpoint accepts unlimited connections without auth or rate limiting, enabling memory/FD exhaustion.

## Investigation Path

### Step 1: SSEManager subscriber storage
**Location:** `app/lib/sse-manager.ts:29-30`
**Finding:** The subscribers Set has no size limit. The `subscribe()` method unconditionally adds callbacks.
**Evidence:**
```typescript
export class SSEManager {
  private subscribers = new Set<SSECallback>();
  // ...
  subscribe(callback: SSECallback): () => void {
    this.subscribers.add(callback);  // No size check
    return () => { this.subscribers.delete(callback); };
  }
}
```

### Step 2: SSE route handler has no connection guard
**Location:** `app/app/api/sse/candles/route.ts:38-95`
**Finding:** The GET handler creates a new `ReadableStream` and subscribes to `sseManager` for every incoming request. There is no check on `sseManager.subscriberCount`, no authentication, no IP-based throttling, and no `middleware.ts` exists in the app.
**Evidence:**
```typescript
export async function GET(req: NextRequest): Promise<Response> {
  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    start(controller) {
      // No auth check, no subscriber limit check
      controller.enqueue(encoder.encode("event: connected\ndata: {}\n\n"));
      const unsubscribe = sseManager.subscribe((payload: string) => {
        // ...
      });
      const heartbeat = setInterval(() => { /* ... */ }, 15_000);
      // ...
    },
  });
  return new Response(stream, { /* SSE headers */ });
}
```

### Step 3: No middleware or rate limiting exists
**Location:** Searched `app/` directory for `middleware.ts`, rate limit patterns, and connection limit patterns.
**Finding:** No `middleware.ts` file exists. No rate limiting is applied to any API route. The `next.config.ts` has no body size or connection limits configured.

### Step 4: Resource consumption per connection
**Location:** `app/app/api/sse/candles/route.ts:41-80`
**Finding:** Each SSE connection allocates:
1. A `ReadableStream` with a `ReadableStreamDefaultController` (held open indefinitely)
2. A `TextEncoder` instance
3. A subscriber callback closure in `sseManager.subscribers`
4. A `setInterval` timer firing every 15 seconds
5. An abort signal event listener on `req.signal`
6. The underlying HTTP response / TCP socket (file descriptor)

### Step 5: Broadcast amplification
**Location:** `app/lib/sse-manager.ts:58-67`
**Finding:** `broadcast()` iterates the entire Set for every event. With N subscribers, each webhook-triggered broadcast is O(N) work. The webhook handler (`app/app/api/webhooks/helius/route.ts:222`) calls broadcast inline, meaning webhook processing time grows linearly with subscriber count.
**Evidence:**
```typescript
broadcast(event: string, data: unknown): void {
  const payload = `event: ${event}\ndata: ${JSON.stringify(data)}\n\n`;
  for (const callback of this.subscribers) {  // O(N)
    try { callback(payload); } catch { this.subscribers.delete(callback); }
  }
}
```

### Step 6: Cleanup reliability
**Location:** `app/app/api/sse/candles/route.ts:72-80`
**Finding:** Cleanup depends on `req.signal` abort event firing. If an attacker holds connections open (keeps reading the stream), the abort signal never fires and connections persist indefinitely. The heartbeat catch block only cleans up if `controller.enqueue` throws, which only happens if the ReadableStream is closed -- not if the connection is simply idle/open.

### Step 7: Railway infrastructure limits
**Location:** `railway.toml:1-12`
**Finding:** Railway's container has finite memory and file descriptor limits (typically 1024 FDs default for Linux containers). The config shows `restartPolicyMaxRetries = 3`, meaning after 3 OOM kills the service stays down until manually restarted.

## If CONFIRMED

### Severity Assessment
- **Severity:** MEDIUM
- **Likelihood:** HIGH (trivial to execute with a simple script)
- **Impact:** Service disruption -- complete denial of service for all users. No direct fund loss (financial operations are on-chain), but chart data streaming, price display, and overall UX are degraded for all connected users. If Railway exhausts restart retries, manual intervention required.

### Attack Scenario

```
Step 1: Open SSE connections in a loop
  Target: GET /api/sse/candles
  Payload: Standard HTTP request with Accept: text/event-stream
  Prerequisite: None (no auth required)
  Tool: Simple script: `for i in $(seq 1 5000); do curl -sN https://dr-fraudsworth-production.up.railway.app/api/sse/candles &; done`
  Result: 5000 concurrent SSE connections, each holding a ReadableStream + setInterval + subscriber callback

Step 2: Hold connections open
  Action: Read heartbeat data every 15 seconds (curl -N does this automatically)
  Result: Connections persist, no cleanup triggered

Step 3: Resource exhaustion
  Result: Node.js process hits EMFILE (too many open file descriptors) or OOM.
  - EMFILE: New connections fail, including legitimate user SSE and webhook POST
  - OOM: Process crashes, Railway restarts (up to 3 times)

Step 4: Broadcast amplification (secondary effect)
  When a legitimate webhook delivery arrives, sseManager.broadcast() iterates
  5000+ subscribers, taking significant time. Helius webhook delivery timeout
  triggers retry, adding more load.

Final Result: Complete denial of service for the frontend (no chart updates,
potentially no webhook processing). After 3 OOM restarts, Railway stops
restarting the service until manual intervention.
```

### Recommended Fix
```typescript
// Before (vulnerable) -- app/lib/sse-manager.ts
export class SSEManager {
  private subscribers = new Set<SSECallback>();

  subscribe(callback: SSECallback): () => void {
    this.subscribers.add(callback);
    return () => { this.subscribers.delete(callback); };
  }
}

// After (fixed) -- app/lib/sse-manager.ts
const MAX_SSE_SUBSCRIBERS = 500;

export class SSEManager {
  private subscribers = new Set<SSECallback>();

  subscribe(callback: SSECallback): (() => void) | null {
    if (this.subscribers.size >= MAX_SSE_SUBSCRIBERS) {
      return null; // Reject connection
    }
    this.subscribers.add(callback);
    return () => { this.subscribers.delete(callback); };
  }
}
```

```typescript
// Before (vulnerable) -- app/app/api/sse/candles/route.ts
export async function GET(req: NextRequest): Promise<Response> {
  // ... creates stream and subscribes unconditionally
}

// After (fixed) -- app/app/api/sse/candles/route.ts
export async function GET(req: NextRequest): Promise<Response> {
  // Reject if at capacity
  if (sseManager.subscriberCount >= MAX_SSE_SUBSCRIBERS) {
    return new Response("Too many connections", { status: 503 });
  }
  // ... rest of handler
}
```

Additionally, consider:
1. Adding a `middleware.ts` with IP-based rate limiting for SSE connections
2. Adding a connection TTL (e.g., force-close after 5 minutes, let client reconnect)
3. Moving broadcast to async/non-blocking to decouple webhook processing from subscriber count

## Incidental Discoveries
- **Heartbeat timer leak on error path**: At `route.ts:48-53`, if the subscriber callback's `controller.enqueue` throws, it calls `unsubscribe()` but does NOT call `clearInterval(heartbeat)`. The heartbeat timer continues firing and silently failing. This is a minor resource leak per disconnected client. Suggested hypothesis: investigate timer/resource leak accumulation across many client disconnect/reconnect cycles.
- **No CORS restriction on SSE endpoint**: The SSE endpoint returns standard headers without explicit CORS restrictions beyond Next.js defaults. While `connect-src` in CSP restricts which origins the *browser* can connect from, a non-browser attacker (curl, script) is unaffected. This is expected but worth noting -- CORS is not a server-side defense.
