# Finding: S006 - Webhook Data Poisoning + Number Overflow for Chart Manipulation

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

This chain attack combines H001 (webhook auth bypass / fail-open data injection) with H014 (quote-engine Number overflow) to create a compounded chart manipulation and user exploitation scenario. An attacker who injects fabricated swap events via the unauthenticated webhook can craft extreme prices that exploit JavaScript `number` precision boundaries. The OHLCV candle aggregator uses `GREATEST`/`LEAST` SQL upserts with `number`-typed price values, meaning injected prices near `Number.MAX_SAFE_INTEGER` or at float precision boundaries produce permanently corrupted candles that cannot be corrected by subsequent legitimate trades. The SSE broadcast amplifies the corrupted data to all connected browsers in real-time. Users viewing manipulated charts may make irrational trading decisions, and the widened minimumOutput from H014's Number overflow means their "protective" slippage bound is already looser than intended -- compounding the damage.

The chain is more severe than either finding alone: H001 provides the injection vector, H014 ensures the injected data has maximum impact because the display layer cannot accurately represent extreme values, and the candle aggregation's one-directional `GREATEST`/`LEAST` pattern makes corruption permanent.

## Hypothesis Investigated

Fake webhook events (H001) with extreme values are injected into the database. Frontend Number overflow (H014) compounds display errors, creating persistent chart corruption that cannot self-heal and that misleads users into accepting worse execution.

## Investigation Path

### Step 1: Webhook Injection of Extreme Price Values
**Location:** `app/app/api/webhooks/helius/route.ts:135-141`
**Finding:** The fail-open auth pattern allows unauthenticated POSTs when `HELIUS_WEBHOOK_SECRET` is unset. The handler parses Anchor events from `logMessages` and derives price at line 219 using `derivePrice()`. An attacker crafting a TaxedSwap event with `inputAmount: 9007199254740991` (MAX_SAFE_INTEGER) and `outputAmount: 1` produces price = 9007199254740991 -- a value that pushes JavaScript `number` to its precision limit.
**Evidence:**
```typescript
// route.ts:92-105 -- derivePrice uses number arithmetic
function derivePrice(
  direction: string,
  inputAmount: number,
  outputAmount: number,
  taxAmount: number = 0,
): number {
  if (direction === "buy") {
    const netInput = inputAmount - taxAmount;
    return outputAmount > 0 ? netInput / outputAmount : 0;
  }
  const grossOutput = outputAmount + taxAmount;
  return inputAmount > 0 ? grossOutput / inputAmount : 0;
}
```

### Step 2: Candle Aggregator Permanent Corruption with Extreme Values
**Location:** `app/db/candle-aggregator.ts:121-122`
**Finding:** The `GREATEST`/`LEAST` upsert pattern means a single injected extreme price permanently sets the candle high to an unreachable value. All 6 resolutions are corrupted simultaneously via `Promise.all`. Legitimate trades cannot "heal" the corrupted high -- `GREATEST(9007199254740991, legitimate_price)` always returns the extreme value.
**Evidence:**
```typescript
// candle-aggregator.ts:118-126
.onConflictDoUpdate({
  target: [candles.pool, candles.resolution, candles.openTime],
  set: {
    high: sql`GREATEST(${candles.high}, ${update.price})`,
    low: sql`LEAST(${candles.low}, ${update.price})`,
    close: sql`${update.price}`,
    volume: sql`${candles.volume} + ${update.volume}`,
    tradeCount: sql`${candles.tradeCount} + 1`,
  },
});
```

### Step 3: Number Precision Loss on Corrupted Candle Display
**Location:** `app/lib/swap/quote-engine.ts:37-62`
**Finding:** The frontend chart components read candle data from the `/api/candles` endpoint and display it using JavaScript `number`. When the corrupted high value (e.g., 9.007e15) is rendered alongside legitimate prices (e.g., 8.6e-6 SOL/token), the chart Y-axis scale becomes astronomically distorted, making all legitimate price action invisible -- compressed into a flat line at the bottom. This compounds with H014's existing precision issues: the quote engine already produces imprecise `minimumOutput` values at current pool sizes, so a user who sees fake chart data and decides to trade gets BOTH the visual manipulation AND the widened slippage window.

### Step 4: SSE Broadcast Amplification of Corrupted Data
**Location:** `app/app/api/webhooks/helius/route.ts:221-234`
**Finding:** Each injected swap event triggers 6 SSE broadcasts (one per candle resolution) to all connected clients. The SSE manager has no subscriber cap, no authentication, and no data validation. A single webhook POST with 100 fake transactions produces 600 SSE broadcasts to N subscribers.
**Evidence:**
```typescript
// route.ts:221-234
const RESOLUTIONS = ["1m", "5m", "15m", "1h", "4h", "1d"];
for (const resolution of RESOLUTIONS) {
  sseManager.broadcast("candle-update", {
    pool: poolAddress,
    resolution,
    price,         // <-- extreme price from attacker's fabricated event
    volume: ...,
    timestamp: Math.floor(timestamp.getTime() / 1000),
  });
}
```

### Step 5: No Data Range Validation Anywhere in Pipeline
**Location:** `app/app/api/webhooks/helius/route.ts:180-243`, `app/db/candle-aggregator.ts:159-208`
**Finding:** Neither the webhook handler nor the candle aggregator validates that price > 0, price < some economic maximum, volume within realistic bounds, or that the derived price is within any plausible range relative to existing candle data. An attacker can inject any value that passes Borsh deserialization.

## Blocking Factors
- **Anchor BorshCoder parsing** (`event-parser.ts:221-229`): Attacker must craft valid Borsh-encoded events. IDL is public, so this is a speed bump, not a barrier.
- **On-chain slippage check**: Prevents total loss on swaps, but does NOT prevent the widened slippage window from H014's precision loss.

## Enabling Factors
- **Fail-open webhook auth**: `route.ts:135-141` -- H001 vector
- **number-typed price derivation**: `route.ts:92-105` -- `derivePrice()` uses `number`, loses precision for extreme values
- **One-directional candle corruption**: `candle-aggregator.ts:121-122` -- `GREATEST`/`LEAST` cannot self-heal
- **Number overflow in quote engine**: `quote-engine.ts:61` -- intermediates exceed MAX_SAFE_INTEGER (H014)
- **No data validation**: No price bounds, volume bounds, or economic sanity checks
- **No rate limiting**: Unlimited webhook POSTs
- **Public SSE with no cap**: Amplification to all connected browsers

### Recommended Fix

**Fix 1: Fail-closed webhook auth (addresses H001 injection vector):**
```typescript
// route.ts:131-141 -- make auth mandatory
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (!webhookSecret) {
  return NextResponse.json({ error: "Webhook not configured" }, { status: 503 });
}
const authHeader = req.headers.get("authorization") ?? "";
const secretBuf = Buffer.from(webhookSecret, "utf-8");
const headerBuf = Buffer.from(authHeader, "utf-8");
if (secretBuf.length !== headerBuf.length ||
    !timingSafeEqual(secretBuf, headerBuf)) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

**Fix 2: Data range validation (defense-in-depth):**
```typescript
// After derivePrice() call in route.ts, before candle upsert:
const MAX_REASONABLE_PRICE = 1e12; // 1 trillion -- well above any plausible price
if (price > MAX_REASONABLE_PRICE || price < 1e-18) {
  console.warn(`[webhook] Suspicious price ${price} in tx ${signature}, skipping candle`);
  continue;
}
```

**Fix 3: Migrate quote-engine to BigInt (addresses H014 precision loss):**
```typescript
// quote-engine.ts -- follow curve-math.ts pattern
export function calculateSwapOutput(
  reserveIn: bigint, reserveOut: bigint, effectiveInput: bigint
): bigint {
  const denominator = reserveIn + effectiveInput;
  if (denominator === 0n) return 0n;
  return (reserveOut * effectiveInput) / denominator;
}
```

## Incidental Discoveries
- **No candle data integrity audit/repair mechanism**: Once candle data is corrupted, there is no automated way to detect or repair it. No background job compares candle data against on-chain state. Manual SQL `UPDATE` statements are the only recovery path. Suggested hypothesis: Investigate whether a periodic candle reconciliation job (comparing DB candles against on-chain getSignaturesForAddress data) could detect and repair data poisoning.
- **SSE broadcast includes raw price without sanitization**: `route.ts:225` broadcasts the derived `price` directly. Even if candle DB writes were validated, the SSE broadcast would still send extreme values to clients. Both paths need independent validation. Location: `route.ts:222-234`.
