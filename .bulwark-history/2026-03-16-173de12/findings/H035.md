# Finding: H035 - DB Connection Pool Exhaustion

## Status: NOT VULNERABLE
## Confidence Score: 7

## Executive Summary

The hypothesis suggests that 6 parallel candle writes could exhaust the 10-connection pool. Investigation reveals that while the pool size is indeed 10 (`app/db/connection.ts:51`), the candle aggregator's 6 parallel upserts (`candle-aggregator.ts:102`) are unlikely to exhaust it in practice. `postgres.js` (the driver used) queues requests when all connections are busy rather than failing -- it has an internal queue with no hard limit. The 6 parallel upserts from one webhook event leave 4 connections available for other concurrent operations (candle reads, swap storage, other webhook processing). Under extreme webhook flood conditions, the queue would grow and latency would increase, but connections would not be "exhausted" in a way that causes errors.

However, the fixed pool size of 10 with no runtime configuration is a mild operational concern that could become an issue at scale.

## Hypothesis Investigated

Max 10 connections with 6 parallel candle writes could exhaust pool.

## Investigation Path

### Step 1: Check pool configuration
**Location:** `app/db/connection.ts:48-51`
**Finding:** Pool is hardcoded to `max: 10`. Comment acknowledges Railway free tier limit.
**Evidence:**
```typescript
// postgres.js client: max 10 connections (Railway free tier limit).
// In production, increase this based on your Postgres plan.
const client =
  globalForDb.pgClient ?? postgres(connectionString, { max: 10 });
```

### Step 2: Check parallel writes in candle aggregator
**Location:** `app/db/candle-aggregator.ts:101-130`
**Finding:** 6 upserts run in `Promise.all`, each targeting a different resolution row. These are independent DB operations that each acquire a connection from the pool.
**Evidence:**
```typescript
export async function upsertCandles(update: CandleUpdate): Promise<void> {
  await Promise.all(
    RESOLUTIONS.map((resolution) => {
      // ... each resolution gets its own upsert query
    }),
  );
}
```

### Step 3: Check postgres.js queuing behavior
**Finding:** `postgres.js` (used via Drizzle) uses an internal connection queue. When all 10 connections are in use, additional queries wait in the queue rather than throwing an error. This means 6 concurrent candle writes + 4 other queries = all 10 connections busy, and the 11th query would queue (with added latency) rather than fail.

### Step 4: Check concurrent webhook access
**Location:** `app/app/api/webhooks/helius/route.ts:160-211`
**Finding:** Within a single webhook POST, transactions are processed sequentially (for-loop). Each swap triggers `upsertCandlesForSwap` which calls `upsertCandles` (6 parallel writes). If two webhook POSTs arrive simultaneously, we'd have 12 concurrent upserts competing for 10 connections -- 2 would queue.

### Step 5: Check singleton pattern
**Location:** `app/db/connection.ts:28-31, 50-54`
**Finding:** The globalThis singleton ensures all routes share one pool. The `globalForDb.pgClient` caching is only applied in non-production (dev HMR protection). In production, a new client is created but only once (the first call caches in `globalForDb.drizzleDb`).
**Evidence:**
```typescript
if (process.env.NODE_ENV !== "production") {
  globalForDb.pgClient = client;
}
const instance = drizzle(client, { schema });
globalForDb.drizzleDb = instance;
```

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Flood webhook endpoint with many concurrent POSTs | Yes | Webhook has no rate limiting |
| 2 | Each POST triggers 6 parallel DB operations | Yes | `Promise.all` over 6 resolutions |
| 3 | Connection pool becomes fully utilized | Yes | 2 concurrent webhooks = 12 queries for 10 connections |
| 4 | Queries start queuing | Yes | postgres.js queues overflow requests |
| 5 | Connection errors or timeouts | Unlikely | postgres.js queues indefinitely by default; timeout only if query itself times out |

## Related Findings
- The webhook endpoint lacks rate limiting (separate architecture concern), which is the actual enabler for any pool pressure scenario.

## Incidental Discoveries
- `connection.ts:53-54` only caches `pgClient` in non-production mode. In production, `pgClient` is not cached on `globalForDb`, but `drizzleDb` is. Since `drizzleDb` wraps the client, this is fine -- the pool is effectively singleton. However, if `getDb()` were ever called after `globalForDb.drizzleDb` was cleared (e.g., hot module replacement in production), a second pool could be created. This is theoretical and unlikely in practice. Suggested hypothesis: investigate whether production hot-reload scenarios could create duplicate connection pools.
