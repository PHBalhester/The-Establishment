# Finding: H031 - No unhandledRejection Handler in Crank

## Status: POTENTIAL
## Confidence Score: 7

## Executive Summary

The crank runner at `scripts/crank/crank-runner.ts` has no `process.on("unhandledRejection")` or `process.on("uncaughtException")` handlers. While the main loop's `try/catch` block at line 201-315 catches most errors and retries after a 30-second delay, any unhandled promise rejection originating outside this scope (e.g., in the `advanceEpochWithVRF` internals that fire-and-forget async operations, or in the `loadCrankProvider`/`getOrCreateProtocolALT` initialization) would cause Node.js to terminate the process with no structured log entry. The `main().catch()` at line 328 catches top-level rejections but not mid-loop unhandled promises from detached async paths.

The risk is operational: epoch transitions would halt until Railway auto-restarts the service, leaving the protocol in a stale epoch state. This is not a direct fund-loss vulnerability but could cascade into missed carnage executions and stalled staking rewards.

## Hypothesis Investigated

No global error handlers; unhandled async errors crash silently.

## Investigation Path

### Step 1: Check for process error handlers
**Location:** `scripts/crank/crank-runner.ts:79-89`
**Finding:** Only SIGINT and SIGTERM handlers exist. No `unhandledRejection` or `uncaughtException` handlers.
**Evidence:**
```typescript
process.on("SIGINT", () => {
  shutdownRequested = true;
  console.log("\n[crank] Shutdown requested (SIGINT). Finishing current cycle...");
});

process.on("SIGTERM", () => {
  shutdownRequested = true;
  console.log("\n[crank] Shutdown requested (SIGTERM). Finishing current cycle...");
});
```

### Step 2: Check main loop error handling
**Location:** `scripts/crank/crank-runner.ts:201-315`
**Finding:** The main while-loop body is wrapped in a try/catch that catches synchronous and awaited async errors. Errors cause a 30-second sleep and retry. However, if `advanceEpochWithVRF` internally spawns detached promises (fire-and-forget), those rejections would escape this catch.
**Evidence:**
```typescript
} catch (err) {
  const errStr = String(err).slice(0, 300);
  console.error(`[crank] ERROR cycle ${cycleCount}: ${errStr}`);
  if (!shutdownRequested) {
    console.log(`[crank] Retrying in ${ERROR_RETRY_DELAY_MS / 1000}s...`);
    await sleep(ERROR_RETRY_DELAY_MS);
  }
}
```

### Step 3: Check top-level catch
**Location:** `scripts/crank/crank-runner.ts:328-331`
**Finding:** `main().catch()` handles top-level promise rejection (e.g., initialization failure) and exits with code 1. This is good for startup errors but does not protect against mid-execution unhandled rejections from detached async operations.
**Evidence:**
```typescript
main().catch((err) => {
  console.error(`[crank] FATAL: ${String(err)}`);
  process.exit(1);
});
```

### Step 4: Search entire scripts directory for unhandledRejection
**Location:** `scripts/` directory (full grep)
**Finding:** Zero matches for "unhandledRejection" or "uncaughtException" anywhere in the scripts directory.

## Blocking Factors
- Main loop try/catch at `crank-runner.ts:201` catches most runtime errors
- Railway auto-restarts crashed processes, limiting downtime

## Enabling Factors
- No `process.on("unhandledRejection")`: `crank-runner.ts` has zero global rejection handlers
- No `process.on("uncaughtException")`: synchronous exceptions in callbacks would also crash
- Node.js v15+ default: unhandled rejections terminate the process

## If CONFIRMED

### Severity Assessment
- **Severity:** LOW
- **Likelihood:** LOW (most async operations are properly awaited)
- **Impact:** Service disruption -- epoch transitions halt until Railway restart (seconds to minutes)

### Attack Scenario
This is not an externally exploitable attack but an operational reliability gap:
```
Step 1: RPC connection drops during a non-awaited callback inside advanceEpochWithVRF
Step 2: Promise rejection propagates with no handler
Step 3: Node.js terminates the crank process
Step 4: Railway detects exit, restarts service (30-60 second gap)
Step 5: During the gap, no epoch transitions occur
```

### Recommended Fix
```typescript
// Before (vulnerable)
// No global handlers

// After (fixed)
process.on("unhandledRejection", (reason, promise) => {
  console.error("[crank] UNHANDLED REJECTION:", reason);
  // Don't exit -- log and continue. The main loop will retry.
});

process.on("uncaughtException", (err) => {
  console.error("[crank] UNCAUGHT EXCEPTION:", err);
  // Exit with non-zero for Railway to restart cleanly
  process.exit(1);
});
```
