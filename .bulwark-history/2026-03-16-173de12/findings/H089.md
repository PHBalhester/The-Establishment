# Finding: H089 - Error Truncation to 300 Characters

## Status: CONFIRMED
## Confidence Score: 8

## Rationale
Multiple error handlers across the codebase truncate error messages to 300 characters via `String(err).slice(0, 300)`. The most impactful instance is in the crank runner at `scripts/crank/crank-runner.ts:303`, where the sole error diagnostic for production epoch failures is truncated. Anchor errors include a `.logs` property with program-level debug messages that is completely discarded by `String(err)`. Other instances appear in `scripts/e2e/security-verification.ts` (lines 171, 270, 337, 557, 652, 773), `scripts/e2e/lib/carnage-flow.ts` (lines 598, 881, 976), and `scripts/e2e/carnage-hunter.ts` (line 405). The E2E script instances are less critical since those are test utilities.

## Code Evidence
```typescript
// scripts/crank/crank-runner.ts:303
const errStr = String(err).slice(0, 300);
console.error(`[crank] ERROR cycle ${cycleCount}: ${errStr}`);
```
Anchor errors have `.logs` arrays with program-level diagnostics, but `String(err)` only captures the top-level message. A 300-char limit can cut off even the basic error description for verbose Solana RPC errors.

## Severity Estimate
LOW -- This is an operational/debugging concern, not a security vulnerability. Truncated errors make production incident diagnosis harder but do not enable any attack vector. The crank continues operating regardless of log quality.
