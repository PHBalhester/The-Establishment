# Finding: H066 - Dependency Confusion on @dr-fraudsworth/shared

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The `@dr-fraudsworth/shared` package is referenced at `app/package.json:18` as version `0.0.1`. The hypothesis suggests an attacker could publish a malicious `@dr-fraudsworth/shared` to the public npm registry, which would be installed instead of the local workspace package. However, this is NOT exploitable because: (1) the package is resolved via npm workspaces -- the root `package.json` declares `"workspaces": ["shared", "app"]`, which means npm resolves `@dr-fraudsworth/shared` from the local `shared/` directory, not from the registry; (2) npm workspaces create a symlink in `node_modules/@dr-fraudsworth/shared` pointing to `./shared/`, which takes priority over any registry package; and (3) even if the workspace resolution failed, npm's scope resolution requires explicit registry configuration for scoped packages.

## Hypothesis Investigated

Attacker publishes @dr-fraudsworth/shared to public npm.

## Investigation Path

### Step 1: Verify workspace configuration
**Location:** Root `package.json:4-7`
**Finding:** The root package.json declares npm workspaces including `shared` and `app`.
**Evidence:**
```json
{
  "workspaces": [
    "shared",
    "app"
  ]
}
```

### Step 2: Verify the shared package identity
**Location:** `shared/package.json:1-3`
**Finding:** The `shared/` directory contains a package.json with `"name": "@dr-fraudsworth/shared"`.
**Evidence:**
```json
{
  "name": "@dr-fraudsworth/shared",
  "version": "0.0.1"
}
```

### Step 3: Verify how app references the shared package
**Location:** `app/package.json:18`
**Finding:** The app depends on `"@dr-fraudsworth/shared": "0.0.1"`.
**Evidence:**
```json
"@dr-fraudsworth/shared": "0.0.1",
```

### Step 4: Analyze npm workspace resolution behavior
**Location:** npm workspace documentation
**Finding:** When npm workspaces are configured, packages listed in the `workspaces` field are symlinked into `node_modules/` and take absolute priority over registry packages with the same name and version. The resolution order is: (1) workspace package, (2) registry package. Since `shared/` is a workspace member with matching name and version, it will always be resolved locally.

### Step 5: Check for .npmrc registry overrides
**Location:** Project root
**Finding:** No `.npmrc` file exists, so no custom registry configuration that could redirect scoped packages.

## Invariant Analysis

### Target Invariant
The `@dr-fraudsworth/shared` package must always resolve to the local workspace, not a public registry.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| Root `package.json:4-7` | Workspace declaration includes `shared/` | Yes |
| npm workspace resolution | Local workspace takes priority over registry | Yes |

## Blocking Factors
- **npm workspaces**: Root `package.json:4-7` -- `shared` is a workspace member, always resolved locally
- **Symlink resolution**: npm creates `node_modules/@dr-fraudsworth/shared -> ../../shared/`
- **Version match**: Local package version `0.0.1` matches the dependency requirement exactly

## Enabling Factors
- **Unclaimed scope**: The `@dr-fraudsworth` scope may not be claimed on npm, allowing an attacker to register it
- **No `.npmrc` pinning**: No explicit registry configuration for the scope
