# Finding: H037 - Mixed-Denomination Fee Display

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The sell fee percentage computation in `useSwap.ts:433-435` mixes denominations in a mathematically incorrect way. For a token-to-SOL sell swap, the LP fee is denominated in **tokens** (deducted from the token input) while the tax is denominated in **SOL** (deducted from the SOL output). The code computes `totalFeePct` by adding `(taxAmount / grossSolOutput) * 100` (a SOL-denominated percentage) to `(lpFee / baseUnits) * 100` (a token-denominated percentage). These are percentages of different bases -- adding them produces a number that is not a meaningful percentage of anything.

The route engine (`route-engine.ts:331-332`) correctly handles this by using BPS summation from step-level rates, which is denomination-independent. The direct swap path in `useSwap.ts` does not use this approach.

This is a display-only issue -- on-chain enforcement is unaffected. Users see an inaccurate total fee percentage, which could mislead trading decisions.

## Hypothesis Investigated

Sell fee sums SOL-denominated tax and token-denominated LP fee.

## Investigation Path

### Step 1: Examine the sell fee computation
**Location:** `app/hooks/useSwap.ts:433-435`
**Finding:** The sell path computes totalFeePct by mixing token and SOL denominators.
**Evidence:**
```typescript
const totalFeeAmount = result.taxAmount;
const totalFeePct = result.grossSolOutput > 0
  ? ((totalFeeAmount / result.grossSolOutput) * 100 + (result.lpFee / baseUnits) * 100).toFixed(1) + "%"
  : "0%";
```

### Step 2: Verify denomination of each fee component
**Location:** `app/lib/swap/quote-engine.ts:187-210`
**Finding:** For `quoteSolSell`:
- `result.lpFee` = token base units (deducted from token input at line 196)
- `result.taxAmount` = lamports (deducted from SOL output at line 202)
- `result.grossSolOutput` = lamports
- `baseUnits` (from useSwap) = token base units (the input amount)
**Evidence:**
```typescript
// LP fee on token input
const effectiveInput = calculateEffectiveInput(tokenAmountBaseUnits, lpFeeBps);
const lpFee = tokenAmountBaseUnits - effectiveInput;  // TOKEN denomination

// AMM output in SOL
const grossSolOutput = calculateSwapOutput(reserveToken, reserveWsol, effectiveInput);

// Tax on SOL output
const taxAmount = calculateTax(grossSolOutput, sellTaxBps);  // SOL denomination
```

### Step 3: Verify the math is wrong
**Concrete example:**
- User sells 10,000 CRIME (baseUnits = 10,000,000 at 6 decimals)
- LP fee = 1% of token input = 100,000 token units
- grossSolOutput = 50,000,000 lamports (0.05 SOL)
- taxAmount = 10% of SOL output = 5,000,000 lamports

The code computes:
```
taxPct = (5,000,000 / 50,000,000) * 100 = 10%
lpPct = (100,000 / 10,000,000) * 100 = 1%
totalFeePct = 10% + 1% = "11.0%"
```

The 11% is actually correct by coincidence here because each percentage is computed against its own base. But this is only correct when interpreted as "sum of individual fee rates" -- not as a percentage of any single quantity. If displayed as "Total fee: 11.0% of input", it would be misleading because 11% of 10M tokens is 1.1M tokens, but the actual fees are 100K tokens + 5M lamports (different units entirely).

### Step 4: Verify the route engine does it correctly
**Location:** `app/lib/swap/route-engine.ts:331-332`
**Finding:** The route engine uses BPS summation from step-level rates, which is denomination-independent and correct.
**Evidence:**
```typescript
const totalFeeBps = steps.reduce((sum, s) => sum + s.lpFeeBps + s.taxBps, 0);
const totalFeePct = `${(totalFeeBps / 100).toFixed(1)}%`;
```

### Step 5: Check the buy path for comparison
**Location:** `app/hooks/useSwap.ts:408-411`
**Finding:** The buy path correctly uses a single denominator (SOL input) for both fees:
```typescript
const totalFeeAmount = result.taxAmount + result.lpFee;
const totalFeePct = baseUnits > 0
  ? ((totalFeeAmount / baseUnits) * 100).toFixed(1) + "%"
  : "0%";
```
For buys, both `taxAmount` and `lpFee` are in lamports (SOL denomination), so this addition is valid.

### Step 6: Check reverse-quote sell path
**Location:** `app/hooks/useSwap.ts:503-505`
**Finding:** Same mixed-denomination issue in the reverse sell quote:
```typescript
const totalFeePct = desiredOutput > 0
  ? ((totalFeeAmount / (desiredOutput + result.taxAmount)) * 100 + (result.lpFee / result.inputTokensNeeded) * 100).toFixed(1) + "%"
  : "0%";
```
Here `desiredOutput` and `result.taxAmount` are SOL (lamports), while `result.lpFee` and `result.inputTokensNeeded` are tokens.

## Blocking Factors
- On-chain enforcement is independent of displayed fee percentage
- The actual output amount shown is correct (it's computed from the quote engine, which is accurate)
- Smart routing enabled by default uses the route engine's correct BPS computation

## Enabling Factors
- `useSwap.ts:433-435`: Adds SOL-based and token-based percentages
- `useSwap.ts:503-505`: Same issue in reverse sell path
- Direct swap path (smart routing OFF) uses the flawed computation

## If CONFIRMED

### Severity Assessment
- **Severity:** LOW
- **Likelihood:** HIGH (every sell swap through direct path shows this)
- **Impact:** Misleading fee display. No fund loss -- display-only issue.

### Attack Scenario
This is not an exploitable attack but a display accuracy issue:
```
Step 1: User toggles smart routing OFF (uses direct swap path)
Step 2: User initiates a CRIME -> SOL sell
Step 3: Fee display shows "11.0%" (mixed-denomination sum)
Step 4: User interprets this as "I lose 11% of my input to fees"
Step 5: Actual fee structure is 1% LP (token side) + 10% tax (SOL side)
Step 6: The displayed percentage doesn't correspond to any single meaningful metric
```

### Recommended Fix
```typescript
// Before (mixed denominations)
const totalFeeAmount = result.taxAmount;
const totalFeePct = result.grossSolOutput > 0
  ? ((totalFeeAmount / result.grossSolOutput) * 100 + (result.lpFee / baseUnits) * 100).toFixed(1) + "%"
  : "0%";

// After (BPS-based, denomination-independent -- matches route-engine approach)
const lpFeeBps = SOL_POOL_FEE_BPS;
const totalFeeBps = lpFeeBps + sellTaxBps;
const totalFeePct = `${(totalFeeBps / 100).toFixed(1)}%`;
```

This matches the route engine's approach at `route-engine.ts:331-332` and provides a consistent, meaningful fee percentage.

## Incidental Discoveries
- The buy path at `useSwap.ts:408-411` adds `result.taxAmount + result.lpFee` which works because both are in lamports for buy swaps. But this sum is the absolute fee amount, and dividing by `baseUnits` (SOL input) gives a correct percentage. The asymmetry between buy (correct) and sell (incorrect) suggests the sell path was written by analogy without accounting for the denomination difference.
