# Finding: H019 - Crank No Kill Switch or Circuit Breaker

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The crank runner (`scripts/crank/crank-runner.ts`) has no emergency shutdown mechanism beyond OS-level SIGINT/SIGTERM signals, which require Railway dashboard or SSH access. There is no remote kill switch, no circuit breaker pattern (max consecutive errors before auto-halt), no spending cap, and no health check endpoint. If the crank enters a pathological loop -- whether from a bug, a compromised RPC returning manipulated state, or an on-chain exploit draining the vault -- the only way to stop it is to manually access the Railway dashboard and terminate the process. Combined with infinite retry on errors and unbounded vault top-ups, this creates a concrete fund-drain risk during adversarial conditions.

This is not theoretical: the crank's catch-all retry at lines 302-315 treats every error identically (30s sleep, retry forever), and the vault top-up at lines 225-241 has no cumulative limit. A stale RPC returning artificially low vault balances would cause the crank to transfer 0.005 SOL per cycle indefinitely.

## Hypothesis Investigated

The crank runner has no emergency shutdown mechanism. If a bug or exploit causes the crank to repeatedly submit harmful transactions, there's no way to stop it without Railway access.

## Investigation Path

### Step 1: Shutdown Mechanisms
**Location:** `scripts/crank/crank-runner.ts:79-89`
**Finding:** The only shutdown mechanism is SIGINT/SIGTERM signal handlers that set a `shutdownRequested` boolean flag. The main loop checks this flag at `while (!shutdownRequested)` (line 197) and in the catch block (line 309).
**Evidence:**
```typescript
let shutdownRequested = false;

process.on("SIGINT", () => {
  shutdownRequested = true;
  console.log("\n[crank] Shutdown requested (SIGINT). Finishing current cycle...");
});

process.on("SIGTERM", () => {
  shutdownRequested = true;
  console.log("\n[crank] Shutdown requested (SIGTERM). Finishing current cycle...");
});
```

There is no:
- Remote kill switch (HTTP endpoint, on-chain flag, env var check per cycle)
- Health check endpoint for external monitoring
- Config-based toggle (file-based or env-var-based pause)
- Dead man's switch (heartbeat requirement)

### Step 2: Circuit Breaker / Error Counting
**Location:** `scripts/crank/crank-runner.ts:302-315`
**Finding:** The catch block retries unconditionally after a fixed 30s delay. There is no consecutive error counter, no maximum error threshold, no error classification (transient vs permanent), and no escalating backoff.
**Evidence:**

### Step 3: Spending Limits
**Location:** `scripts/crank/crank-runner.ts:225-241`
**Finding:** The vault top-up transfers 0.005 SOL per cycle when vault balance is below ~2M lamports. There is no per-cycle, per-day, or cumulative spending limit. The `cycleCount` variable exists but is used only for logging, not for budget enforcement.
**Evidence:**

### Step 4: Process-Level Crash Handlers
**Location:** `scripts/crank/crank-runner.ts` (entire file)
**Finding:** No `process.on("unhandledRejection")` or `process.on("uncaughtException")` handlers. The `main().catch()` at line 328 catches top-level errors, but unhandled rejections from detached async paths (e.g., dynamic imports in `vrf-flow.ts:299,312`) bypass the main try/catch and could crash the process silently.

### Step 5: Balance Check Enforcement
**Location:** `scripts/crank/crank-runner.ts:212-217`
**Finding:** The wallet balance check at <1 SOL only logs a WARNING. It does not pause operations, reduce spending, or trigger any protective action.
**Evidence:**

### Step 6: waitForSlotAdvance Infinite Loop
**Location:** `scripts/vrf/lib/vrf-flow.ts:182-194`
**Finding:** The `waitForSlotAdvance` function contains a `while (true)` loop with no absolute timeout or maximum iteration count. If the RPC returns a stuck slot, this blocks the crank indefinitely -- it will never reach the shutdown check in the main loop.
**Evidence:**

## Invariant Analysis

### Target Invariant
The crank runner should have bounded failure modes: it should automatically cease operations when detecting anomalous conditions (excessive errors, rapid balance drain, persistent failures) to protect the crank wallet from fund loss.

### Enforcement Points

## Blocking Factors
- Railway process management provides auto-restart on crash (but this is also an enabling factor -- a crashed crank restarts and resumes spending)

## Enabling Factors
- **No error counting**: `crank-runner.ts:302-315` -- all errors treated identically, no max consecutive error threshold
- **No spending limit**: `crank-runner.ts:225-241` -- vault top-up has no per-day cap
- **Advisory-only balance check**: `crank-runner.ts:213-217` -- low balance triggers log, not operational pause
- **Infinite slot wait**: `vrf-flow.ts:182-194` -- blocks shutdown check, crank hangs unresponsive
- **No process crash handlers**: `crank-runner.ts` -- missing `unhandledRejection`/`uncaughtException` means silent death with no logging
- **No remote control**: No HTTP endpoint, no on-chain flag, no config file check for pause/kill

## Severity Assessment
- **Severity:** HIGH
- **Likelihood:** MEDIUM (requires either adversarial RPC, sustained error condition, or operator unavailability during incident)
- **Impact:** Fund loss (crank wallet drain through repeated failed TXs and unbounded vault top-ups); protocol stall (if crank hangs in `waitForSlotAdvance`); delayed incident response (no alerting, only Railway stdout logs)

## Attack Scenario

```
Step 1: Adversary compromises or spoofs the RPC endpoint
  Target: CLUSTER_URL environment variable or DNS resolution
  Payload: Return artificially low vault balance (e.g., 0 lamports) for every getBalance(vaultPubkey) call
  Prerequisite: Ability to MITM the RPC connection (no TLS pinning)
  Result: Crank enters vault top-up branch every cycle

Step 2: Crank transfers 0.005 SOL per cycle (~every 5 minutes)
  Target: crank-runner.ts:225-241
  Result: 0.005 SOL * 288 cycles/day = 1.44 SOL/day drained to the vault address

Step 3: Simultaneously, return manipulated epoch state to cause VRF failures
  Target: readEpochState and advanceEpochWithVRF
  Result: Each failed TX costs ~0.005 SOL in fees; additional drain

Step 4: Operator receives no alert
  Target: No alerting integration
  Result: Drain continues until operator manually checks Railway logs or wallet balance

Step 5: Operator must access Railway dashboard to stop the crank
  Target: No remote kill switch
  Result: If operator is unavailable (sleeping, traveling, etc.), drain continues for hours

Final Result: Crank wallet drained of operational SOL. Even at conservative rates
(0.005 SOL/cycle vault + 0.005 SOL/cycle fees = 0.01 SOL/cycle, 288 cycles/day),
this is ~2.88 SOL/day. A well-funded crank wallet (10+ SOL) would be drained in 3-4 days
without operator intervention.
```

## Incidental Discoveries
- **Missing process crash handlers**: `crank-runner.ts` has no `process.on("unhandledRejection")` or `process.on("uncaughtException")`. Dynamic imports in `vrf-flow.ts:299,312` create async paths outside the main try/catch. An unhandled rejection crashes the process silently without logging. Suggested hypothesis: Investigate whether Railway auto-restart + silent crash + no alerting creates a "zombie crank" scenario where the crank repeatedly crashes and restarts without anyone knowing.
- **Error truncation hides diagnostics**: `crank-runner.ts:303` truncates errors to 300 chars and does not log Anchor program logs (`err.logs`). This makes production incident diagnosis extremely difficult. Suggested hypothesis: Investigate whether error truncation could mask a security-relevant error condition (e.g., a constraint violation that indicates an attack) from operator awareness.
- **RPC URL potentially logged with API key**: `crank-runner.ts:177` logs `process.env.CLUSTER_URL` which may contain `?api-key=...`. The devnet VRF validation script masks this (`vrf-flow.ts` area), but crank-runner does not. Suggested hypothesis: Investigate Railway log exposure of the Helius API key via crank startup logs.
