# Finding: H124 - BuyForm BigInt via Number Intermediate

## Status: POTENTIAL
## Confidence Score: 7

## Rationale
At `app/components/launch/BuyForm.tsx:84-86`, the code computes `currentHoldings` as `BigInt(Math.floor(crimeBalance * Number(TOKEN_DECIMAL_FACTOR)))`. `TOKEN_DECIMAL_FACTOR` is `1_000_000n` (from curve-constants.ts:51), so `Number(TOKEN_DECIMAL_FACTOR)` = 1,000,000. The `crimeBalance` comes from `useTokenBalances` which returns a human-readable `number`. The multiplication `crimeBalance * 1_000_000` could lose precision if `crimeBalance` exceeds `Number.MAX_SAFE_INTEGER / 1_000_000 = 9,007,199,254` (9 billion tokens). With a max per-wallet cap of 20M tokens, the maximum value is `20_000_000 * 1_000_000 = 2e13`, which is safely below `Number.MAX_SAFE_INTEGER` (9.007e15). The same pattern appears in SellForm.tsx:94.

## Code Evidence
```typescript
// app/components/launch/BuyForm.tsx:84-86
const currentHoldings = tokenSymbol === 'CRIME'
  ? BigInt(Math.floor(crimeBalance * Number(TOKEN_DECIMAL_FACTOR)))
  : BigInt(Math.floor(fraudBalance * Number(TOKEN_DECIMAL_FACTOR)));
```

## Severity Estimate
LOW -- While the pattern is fragile (relies on MAX_TOKENS_PER_WALLET being << 9 billion), the actual per-wallet cap of 20M tokens keeps values well within Number safe range. The intermediate `2e13` is ~450x below `MAX_SAFE_INTEGER`. Would become a real issue if token caps were increased significantly.
