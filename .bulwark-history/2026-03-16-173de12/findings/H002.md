# Finding: H002 - Helius API Key Extraction from Client Bundle

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The Helius API key `[REDACTED-DEVNET-HELIUS-KEY]` is hardcoded in `shared/programs.ts:22` (embedded in the RPC URL) and `shared/constants.ts:474` (exported as a named constant). The `shared/programs.ts` module is directly imported by `app/lib/connection.ts:16` and `app/providers/providers.tsx:4`, which are client-side modules bundled into the browser JavaScript. Any user can extract this key by viewing page source or inspecting network requests.

The key grants access to the Helius webhook management API (`https://api.helius.xyz/v0/webhooks?api-key=...`), which supports full CRUD operations (list, create, update, delete). The CSP in `app/next.config.ts:19` explicitly whitelists `https://api.helius.xyz` in `connect-src`, meaning even an XSS payload or browser console command can call the webhook management API from the page context. An attacker can: (a) exhaust RPC rate limits for all users, (b) delete the production webhook to halt the data pipeline, (c) redirect the webhook to an attacker-controlled server to intercept transaction data, or (d) register malicious webhooks. This is not a theoretical concern -- the `scripts/webhook-manage.ts` file demonstrates all four CRUD operations using the same key.

## Hypothesis Investigated

An attacker extracts the Helius API key from the client JavaScript bundle and uses it to exhaust RPC rate limits or manipulate webhook registrations.

## Investigation Path

### Step 1: Verify API key presence in source code
**Location:** `shared/programs.ts:22`, `shared/constants.ts:474`
**Finding:** The key `[REDACTED-DEVNET-HELIUS-KEY]` is hardcoded in plaintext at both locations. Comments explicitly say "not a secret" and "free-tier API key."
**Evidence:**
```typescript
// shared/programs.ts:21-22
export const DEVNET_RPC_URL =
  "https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]";

// shared/constants.ts:474
export const HELIUS_API_KEY = "[REDACTED-DEVNET-HELIUS-KEY]";
```

### Step 2: Verify key reaches client bundle
**Location:** `app/lib/connection.ts:16`, `app/providers/providers.tsx:4,35`
**Finding:** Both files import `DEVNET_RPC_URL` from `@dr-fraudsworth/shared` and use it as the fallback RPC URL. `providers.tsx` passes it directly to `<ConnectionProvider endpoint={endpoint}>`, and `connection.ts` uses it in the singleton `Connection` constructor. Both are client-side modules -- the URL (with embedded API key) is inlined into the production JavaScript bundle by Next.js.
**Evidence:**
```typescript
// app/lib/connection.ts:16,32-33
import { DEVNET_RPC_URL } from "@dr-fraudsworth/shared";
const url = rpcUrl ?? process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL;

// app/providers/providers.tsx:4,35
import { DEVNET_RPC_URL } from "@dr-fraudsworth/shared";
const endpoint = process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL;
```

### Step 3: Verify key grants webhook management API access
**Location:** `scripts/webhook-manage.ts:28-29,49-54`
**Finding:** The same API key is used to call `https://api.helius.xyz/v0/webhooks?api-key=...` for full CRUD operations. The script demonstrates list, create, update, and delete functionality. This proves the key has webhook management permissions beyond simple RPC access.
**Evidence:**
```typescript
// scripts/webhook-manage.ts:28-29
const HELIUS_API_KEY = process.env.HELIUS_API_KEY ?? "[REDACTED-DEVNET-HELIUS-KEY]";
const HELIUS_API_BASE = "https://api.helius.xyz/v0";

// scripts/webhook-manage.ts:54
const url = `${HELIUS_API_BASE}${path}?api-key=${HELIUS_API_KEY}`;
```

### Step 4: Verify CSP allows webhook API calls from browser
**Location:** `app/next.config.ts:19`
**Finding:** The Content Security Policy's `connect-src` directive explicitly whitelists `https://api.helius.xyz`, meaning client-side JavaScript (including browser console or XSS payloads) can make fetch requests to the Helius webhook management API directly from the page.
**Evidence:**
```
connect-src 'self' ... https://api.helius.xyz ...
```

### Step 5: Verify no additional authentication on Helius webhook API
**Location:** `scripts/webhook-manage.ts:49-71`
**Finding:** The Helius webhook management API authenticates solely via the `api-key` query parameter. There is no additional OAuth token, HMAC signature, or IP allowlist. The API key alone is sufficient for all operations.

### Step 6: Verify key appears in 6+ locations total
**Finding:** The key appears in: `shared/constants.ts:474`, `shared/programs.ts:22`, `scripts/webhook-manage.ts:28`, `scripts/backfill-candles.ts:47`, `scripts/deploy/pda-manifest.json:3`, `.claude/settings.local.json:193,360,374-375`. It is also permanently in git history.

## Enabling Factors
- **Hardcoded key in client-bundled module**: `shared/programs.ts:22` -- imported by client-side code, inlined by bundler
- **CSP whitelists Helius API**: `app/next.config.ts:19` -- `connect-src` includes `https://api.helius.xyz`
- **Same key for RPC and webhook management**: `scripts/webhook-manage.ts:28` -- no separation of read-only RPC key vs management key

- **Webhook production URL in source code**: `scripts/webhook-manage.ts:43` -- `https://dr-fraudsworth-production.up.railway.app/api/webhooks/helius`
- **Key permanently in git history**: Even if rotated, the current key is compromised forever in git

## Blocking Factors
- None identified. There is no protection preventing this attack.

### Recommended Fix

```typescript
// BEFORE (vulnerable) -- shared/programs.ts:21-22
export const DEVNET_RPC_URL =
  "https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]";

// AFTER (fixed) -- Option A: Server-side RPC proxy
// 1. Create /api/rpc proxy route that forwards to Helius with server-side key
// 2. Frontend connects to /api/rpc (no API key in client)
// 3. Rate limit the proxy route
// app/app/api/rpc/route.ts:
export async function POST(req: NextRequest) {
  const body = await req.json();
  const heliusKey = process.env.HELIUS_API_KEY; // server-only env var
  const resp = await fetch(`https://devnet.helius-rpc.com/?api-key=${heliusKey}`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(body),
  });
  return new NextResponse(resp.body, { status: resp.status });
}

// AFTER (fixed) -- Option B: Separate keys (simpler, less secure)
// 1. Create a separate Helius API key with RPC-only permissions for client use
// 2. Keep the management key server-side only
// 3. Rotate the current compromised key
// shared/programs.ts:
export const DEVNET_RPC_URL = process.env.NEXT_PUBLIC_RPC_URL ?? "https://devnet.helius-rpc.com/?api-key=<client-rpc-only-key>";

// AFTER (fixed) -- Option C: Remove api.helius.xyz from CSP (defense-in-depth)
// app/next.config.ts -- remove https://api.helius.xyz from connect-src
// This prevents browser-side webhook API calls even if key is exposed
```

**Additional remediation steps:**
1. Rotate the current Helius API key (it is permanently compromised via git history)
2. Use separate API keys: one for client-side RPC (read-only, rate-limited), one for webhook management (server-only, never in source code)
3. Remove `https://api.helius.xyz` from CSP `connect-src` -- the frontend has no legitimate need to call the Helius management API
4. Move webhook management key to a server-only env var (`HELIUS_MANAGEMENT_KEY`), never `NEXT_PUBLIC_`
5. Consider BFG Repo-Cleaner to purge key from git history (or accept it is compromised and rely on rotation)

Even if the primary locations are fixed, this file leaks the key. Suggested hypothesis: Audit all non-obvious locations where secrets may be committed (settings files, build configs, CI configs).
