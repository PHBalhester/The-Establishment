# Finding: H024 - No Rate Limiting on Any API Endpoint

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

All 6 API routes in the Dr. Fraudsworth Next.js application lack any form of rate limiting. There is no Next.js middleware.ts file, no per-route throttling logic, no IP-based limiting, and no connection cap on the SSE endpoint. The Railway hosting configuration (`railway.toml`) provides no application-layer rate limiting either. This creates multiple abuse vectors: database resource exhaustion via the candle/carnage read endpoints, upstream API rate-limit burning via the sol-price proxy, memory exhaustion via unbounded SSE connections, and data injection amplification via the webhook endpoint. The absence is not an oversight in one route -- it is a systemic architectural gap across the entire API surface.

## Hypothesis Investigated

All 6 API routes lack rate limiting, enabling resource exhaustion or abuse.

### Recommended Fix

The recommended approach is a Next.js middleware with IP-based rate limiting using an in-memory sliding window. For production, this should be backed by Redis or a proper WAF.

```typescript
// Before (vulnerable) -- no middleware.ts exists

// After (fixed) -- app/middleware.ts
import { NextRequest, NextResponse } from "next/server";

// Simple in-memory sliding window rate limiter
// For production: use Redis-backed solution (e.g., @upstash/ratelimit)
const windowMs = 60_000; // 1 minute window
const limits: Record<string, number> = {
  "/api/webhooks/helius": 30,   // Helius sends in batches
  "/api/sse/candles": 5,        // Long-lived -- few connections per IP
  "/api/candles": 60,           // Chart data reads
  "/api/sol-price": 30,         // Cached anyway
  "/api/carnage-events": 30,    // Simple reads
  "/api/health": 10,            // Health checks
};

const ipRequests = new Map<string, { count: number; resetAt: number }>();

export function middleware(req: NextRequest) {
  const ip = req.headers.get("x-forwarded-for")?.split(",")[0] ?? "unknown";
  const path = req.nextUrl.pathname;

  // Find matching limit
  const matchedPath = Object.keys(limits).find((p) => path.startsWith(p));
  if (!matchedPath) return NextResponse.next();

  const maxRequests = limits[matchedPath];
  const key = `${ip}:${matchedPath}`;
  const now = Date.now();

  let entry = ipRequests.get(key);
  if (!entry || now > entry.resetAt) {
    entry = { count: 0, resetAt: now + windowMs };
    ipRequests.set(key, entry);
  }

  entry.count++;
  if (entry.count > maxRequests) {
    return NextResponse.json(
      { error: "Too many requests" },
      { status: 429, headers: { "Retry-After": "60" } }
    );
  }

  return NextResponse.next();
}

export const config = {
  matcher: "/api/:path*",
};
```

Additionally, the SSE manager should enforce a connection cap:

```typescript
// Before (vulnerable) -- app/lib/sse-manager.ts:38
subscribe(callback: SSECallback): () => void {
    this.subscribers.add(callback);
    return () => { this.subscribers.delete(callback); };
}

// After (fixed)
private static readonly MAX_SUBSCRIBERS = 1000;

subscribe(callback: SSECallback): (() => void) | null {
    if (this.subscribers.size >= SSEManager.MAX_SUBSCRIBERS) {
        return null; // Reject -- caller should return 503
    }
    this.subscribers.add(callback);
    return () => { this.subscribers.delete(callback); };
}
```

## Incidental Discoveries
- **Sol-price cache race condition**: `app/app/api/sol-price/route.ts:80-100` -- Multiple concurrent requests arriving after cache expiry will ALL make upstream API calls (no mutex/lock on cache refresh). While this is a minor issue with the 60s cache, it means a coordinated burst at the exact cache expiry moment hits upstream APIs harder than expected. Suggested hypothesis: Investigate whether the sol-price cache refresh has a thundering herd problem.
- **Health endpoint as DoS amplifier**: `app/app/api/health/route.ts:28-47` -- Each health check request makes both a Postgres query (`SELECT 1`) and a Solana RPC call (`getSlot()`). Rapid health check requests would consume both DB connections and RPC quota. The Railway health check at `railway.toml:8` only hits this every 120 seconds, but the endpoint is publicly accessible to anyone. Suggested hypothesis: Investigate health endpoint as an RPC/DB resource amplification vector.
