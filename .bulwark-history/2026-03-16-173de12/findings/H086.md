# Finding: H086 - No Crank Health Check in Railway

## Status: CONFIRMED
## Confidence Score: 9

## Rationale
The `railway-crank.toml` configures the crank service with `restartPolicyType = "ON_FAILURE"` and `restartPolicyMaxRetries = 10`, but defines no `healthcheckPath` or `healthcheckTimeout`. The crank runner (`scripts/crank/crank-runner.ts`) is a long-running loop that catches all errors internally (line 302-315) and retries after 30 seconds. This means the process never exits on error -- it sleeps and retries. Railway's ON_FAILURE restart only triggers on process exit, so a crank that hangs (e.g., stuck RPC call, deadlocked await, infinite retry loop on a permanent error) will never be detected or restarted.

## Code Evidence
```toml
# railway-crank.toml (complete file)
[build]
builder = "NIXPACKS"
buildCommand = "npm install"

[deploy]
startCommand = "npx tsx scripts/crank/crank-runner.ts"
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 10
```
No `healthcheckPath`, no `healthcheckTimeout`, no `healthcheckInterval` configured. The crank also has no internal watchdog or heartbeat mechanism.

## Severity Estimate
MEDIUM -- A hung crank means no epoch advancement, no VRF, no Carnage execution. The protocol stalls silently. Since the crank manages the entire protocol lifecycle, this is an availability risk for the entire system.
