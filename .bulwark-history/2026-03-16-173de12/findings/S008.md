# Finding: S008 - Helius Key + Webhook Deletion for Data Pipeline Kill

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

This chain attack combines H002 (Helius API key exposed in client bundle) with H001 (webhook auth bypass) to execute a full data pipeline takeover. The Helius API key `[REDACTED-DEVNET-HELIUS-KEY]` is hardcoded in `shared/constants.ts:474` and `shared/programs.ts:22`, bundled into the client JavaScript, and accessible to any user via browser DevTools. This key grants full CRUD access to the Helius webhook management API. An attacker extracts the key, lists all webhooks (`GET /v0/webhooks?api-key=...`), deletes the legitimate production webhook, and registers a new webhook pointing to the same production endpoint but WITHOUT the `authHeader` field -- exploiting H001's fail-open pattern to ensure their replacement webhook's deliveries are accepted without authentication.

The combined effect is devastating: the attacker controls WHAT data flows into the chart pipeline. They can halt data flow entirely (delete webhook, register none), redirect data to their own server (surveillance), or -- most insidiously -- register a webhook for DIFFERENT programs or accounts, causing the pipeline to ingest irrelevant transaction data while missing legitimate swap events.

## Hypothesis Investigated

Extract Helius API key (H002), delete the legitimate webhook via management API, and register an attacker-controlled replacement webhook -- either redirecting data flow or creating a data vacuum that freezes all charts.

## Investigation Path

### Step 1: Confirm API Key is Extractable from Client Bundle
**Location:** `shared/constants.ts:474`, `shared/programs.ts:22`
**Finding:** The key `[REDACTED-DEVNET-HELIUS-KEY]` is hardcoded in two locations. `shared/programs.ts` is imported by `app/lib/connection.ts:16` and `app/providers/providers.tsx:4`, both client-side modules. The key is visible in the browser JS bundle via DevTools Sources tab or Network tab (in RPC request URLs).
**Evidence:**
```typescript
// shared/constants.ts:474
export const HELIUS_API_KEY = "[REDACTED-DEVNET-HELIUS-KEY]";

// shared/programs.ts:22
export const DEVNET_RPC_URL =
  "https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]";
```

### Step 2: Confirm Key Grants Webhook Management Access
**Location:** `scripts/webhook-manage.ts:28-54`
**Finding:** The same API key authenticates to `https://api.helius.xyz/v0/webhooks?api-key=...` for all CRUD operations. The script demonstrates list, create, update, and delete functionality. No additional OAuth, HMAC, or IP allowlist protects the management API.
**Evidence:**
```typescript
// webhook-manage.ts:28-29
const HELIUS_API_KEY = process.env.HELIUS_API_KEY ?? "[REDACTED-DEVNET-HELIUS-KEY]";
const HELIUS_API_BASE = "https://api.helius.xyz/v0";

// webhook-manage.ts:54
const url = `${HELIUS_API_BASE}${path}?api-key=${HELIUS_API_KEY}`;
```

### Step 3: Confirm CSP Allows Webhook API Calls from Browser
**Location:** `app/next.config.ts:19`
**Finding:** The CSP `connect-src` directive explicitly whitelists `https://api.helius.xyz`, meaning the attacker can execute the entire attack from the browser console on the production site -- no external tools needed.

### Step 4: Confirm Webhook Deletion and Re-Registration Flow
**Location:** `scripts/webhook-manage.ts:77-167`
**Finding:** The management API flow is:
1. `GET /v0/webhooks` -- returns all webhooks with their IDs and URLs
2. `DELETE /v0/webhooks/{id}` -- removes the production webhook
3. `POST /v0/webhooks` -- creates a new webhook

The `create` endpoint accepts an optional `authHeader` field. If omitted, Helius delivers without any authorization header, triggering H001's fail-open pattern on the receiving end.

### Step 5: Confirm Webhook Handler Accepts Unauthenticated Deliveries
**Location:** `app/app/api/webhooks/helius/route.ts:135-141`
**Finding:** Even if `HELIUS_WEBHOOK_SECRET` IS set in Railway, the attacker's replacement webhook is registered WITHOUT an `authHeader`. Helius delivers WITHOUT the `Authorization` header. The handler checks `if (webhookSecret) { if (authHeader !== webhookSecret) { reject } }`. Since `authHeader` is `null` (Helius didn't send one), `null !== webhookSecret` evaluates to `true`, and the request IS rejected. However, this protection ONLY works if the secret is set AND the comparison catches `null`.

Wait -- re-examining: `req.headers.get("authorization")` returns `null` when the header is absent. `null !== "some-secret"` is `true`, so the handler returns 401. This means the fail-open attack path requires `HELIUS_WEBHOOK_SECRET` to be UNSET.

**Revised analysis:** If the secret IS set, the attacker's replacement webhook deliveries would be rejected (401). The attacker can instead register the webhook pointing to their OWN server (intercepting all data while the legitimate pipeline gets nothing), OR register the webhook with a DIFFERENT set of `accountAddresses` to cause the pipeline to miss swap events.

### Step 6: Confirm No Webhook Monitoring or Alerting
**Location:** Project-wide search
**Finding:** No automated check verifies that the production webhook exists and points to the correct URL. No health check endpoint queries the Helius webhook list. No alerting on webhook deletion. If the webhook is deleted, charts silently freeze (no new candle data) and the team has no automated notification.

## Blocking Factors
- **Webhook auth when secret IS set**: If `HELIUS_WEBHOOK_SECRET` is configured in Railway, the handler rejects deliveries without the correct `Authorization` header (`route.ts:138`). This blocks attack path 4c but not 4a (data intercept) or 4b (data vacuum) or the deletion itself.

## Enabling Factors
- **API key in client bundle**: `shared/programs.ts:22` -- trivially extractable
- **Same key for RPC + management**: `scripts/webhook-manage.ts:28` -- no role separation
- **CSP whitelists `api.helius.xyz`**: `app/next.config.ts:19` -- attack executable from browser console
- **Production webhook URL in source**: `scripts/webhook-manage.ts:43` -- `https://dr-fraudsworth-production.up.railway.app/api/webhooks/helius`
- **No webhook monitoring**: No automated check that webhook exists and is correctly configured
- **No alerting on data pipeline freeze**: Charts silently stop updating; no health check detects missing data

### Recommended Fix

**Fix 1: Rotate and separate API keys:**
```typescript
// Before (vulnerable) -- shared/programs.ts:22
export const DEVNET_RPC_URL =
  "https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]";

// After -- use env var for RPC, server-only key for management
export const DEVNET_RPC_URL =
  process.env.NEXT_PUBLIC_RPC_URL ?? "https://devnet.helius-rpc.com/?api-key=<CLIENT_RPC_ONLY_KEY>";
```
Create two Helius API keys: one for client-side RPC (read-only), one for webhook management (server-only, never in source code).

**Fix 2: Remove `api.helius.xyz` from CSP:**
```
connect-src 'self' https://devnet.helius-rpc.com ...
// Remove: https://api.helius.xyz
```
The frontend has no legitimate need to call the Helius management API.

**Fix 3: Webhook configuration health check:**
```typescript
// /api/health/webhook route -- check webhook exists and is correct
async function checkWebhookHealth() {
  const key = process.env.HELIUS_MANAGEMENT_KEY; // server-only
  const resp = await fetch(`https://api.helius.xyz/v0/webhooks?api-key=${key}`);
  const webhooks = await resp.json();
  const ours = webhooks.find(w => w.webhookURL.includes("dr-fraudsworth"));
  if (!ours) {
    alert("WEBHOOK DELETED -- data pipeline down");
  }
}
```

**Fix 4: Webhook delivery monitoring:**
Add a "last webhook received" timestamp to the health check. If no webhook delivery in 10+ minutes during active trading hours, alert.

## Incidental Discoveries
- **Webhook re-registration race**: If the team detects the deletion and re-creates the webhook using the same compromised API key, the attacker can immediately delete it again. This creates a cat-and-mouse loop that only ends when the API key is rotated. The key is permanently in git history, so rotation requires creating a new Helius account or contacting Helius support. Location: `scripts/webhook-manage.ts:28`. Suggested hypothesis: Investigate whether Helius supports IP allowlisting or OAuth-based management API access that would survive key rotation delays.
- **Multiple webhooks for same program**: Helius allows registering multiple webhooks for the same account addresses. An attacker could register ADDITIONAL webhooks (not just replace the existing one) pointing to their server, achieving surveillance without disrupting the legitimate pipeline. This is stealthier than deletion. Suggested hypothesis: Investigate whether Helius has a per-account webhook limit and whether duplicate webhooks could be used for passive surveillance.
