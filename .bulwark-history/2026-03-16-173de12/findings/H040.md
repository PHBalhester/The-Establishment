# Finding: H040 - Default 5% Slippage Too High for Thin Pools

## Status: POTENTIAL
## Confidence Score: 6

## Executive Summary

The `SettingsProvider` defaults `slippageBps` to 500 (5%). While this default applies during the bonding curve phase where slippage is mathematically bounded by the linear curve, it will persist into post-graduation AMM trading where pools start with ~290M tokens + ~1000 SOL. At these thin liquidity levels, a 5% slippage tolerance creates a significant sandwich attack surface. The UI provides presets at 0.5%, 1%, and 2% (in `SlippageConfig.tsx`), but the default loaded from localStorage on first visit is 500 BPS. The SettingsProvider validates slippage to 0-10000 BPS range but does not warn on first use or auto-adjust based on pool depth.

However, this is POTENTIAL rather than CONFIRMED because: (1) during the bonding curve phase, MEV sandwich attacks are not applicable (deterministic curve pricing, no AMM), and (2) post-graduation, the user's default may have been changed via the settings UI. The risk materializes specifically for users who never touch settings and trade AMM pools post-graduation.

## Hypothesis Investigated

500 BPS default is excessive for thin liquidity post-graduation.

## Investigation Path

### Step 1: Verify default slippage
**Location:** `app/providers/SettingsProvider.tsx:77`
**Finding:** Default is 500 BPS (5%).
**Evidence:**
```typescript
return {
  slippageBps: 500,
  priorityFeePreset: 'medium',
```

### Step 2: Verify validation bounds
**Location:** `app/providers/SettingsProvider.tsx:101-106`
**Finding:** Validates 0-10000 BPS. 500 is within range.
**Evidence:**
```typescript
slippageBps:
  typeof parsed.slippageBps === 'number' &&
  parsed.slippageBps >= 0 &&
  parsed.slippageBps <= 10_000
    ? parsed.slippageBps
    : defaults.slippageBps,
```

### Step 3: Check UI presets
**Location:** `app/components/swap/SlippageConfig.tsx:43-47`
**Finding:** Presets are 0.5%, 1%, 2% -- all significantly lower than the 5% default. The default doesn't match any preset, so on first visit the UI shows 5% as a custom value.
**Evidence:**
```typescript
const SLIPPAGE_PRESETS = [
  { label: "0.5%", bps: 50 },
  { label: "1%", bps: 100 },
  { label: "2%", bps: 200 },
] as const;
```

### Step 4: Check how slippage is used in bonding curve forms
**Location:** `app/components/launch/BuyForm.tsx:168-169`, `SellForm.tsx:177-178`
**Finding:** Both forms apply slippage via BigInt arithmetic. At 500 BPS, minimumOutput = tokensOut * 9500/10000. This is a 5% buffer on the bonding curve.
**Evidence:**
```typescript
const slippageBps = BigInt(settings.slippageBps);
const minimumTokensOut = tokensOut * (10000n - slippageBps) / 10000n;
```

### Step 5: Check AMM swap path slippage usage
**Location:** `app/hooks/useSwap.ts:407, 432`
**Finding:** AMM path also uses `slippageBps` from settings. Same 500 BPS default applies.
**Evidence:**
```typescript
const minimumOutput = Math.floor(result.outputTokens * (10_000 - slippageBps) / 10_000);
```

### Step 6: Assess post-graduation pool depth
**Finding:** Per graduation script (`scripts/graduation/graduate.ts:102-107`), pools will be seeded with ~290M tokens + ~1000 SOL. At these reserve sizes, a 5% slippage tolerance on a 1 SOL trade means the user would accept receiving 5% fewer tokens than quoted. An MEV bot can insert a buy before and sell after, extracting up to 5% of the trade value.

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | User has default 500 BPS slippage (never changed settings) | Yes | First-visit default in SettingsProvider.tsx:77 |
| 2 | User submits AMM swap post-graduation | Yes | useSwap.ts uses same slippageBps |
| 3 | MEV bot observes pending TX in mempool | Yes | No Jito/MEV protection (ARCHITECTURE.md) |

## Invariant Analysis

### Target Invariant
Users should not be exposed to excessive MEV extraction due to default settings.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `SettingsProvider.tsx:77` | Default slippage | Set to 500 BPS -- too high |
| `SlippageConfig.tsx:88-89` | Warning on high slippage | Warns > 500 BPS -- exactly AT the default, no warning |
| On-chain `minimumOutput` | Slippage floor | Enforced, but at the 5% tolerance |

## Blocking Factors
- `SlippageConfig.tsx:88`: `highSlippage = slippageBps > 500` -- warning triggers above 500, but NOT at exactly 500 (the default)

## Enabling Factors
- `SettingsProvider.tsx:77`: 500 BPS default
- No MEV protection (standard Helius RPC, no Jito bundles)
- Post-graduation pools have thin liquidity (~1000 SOL per pool)

## If CONFIRMED

### Severity Assessment
- **Severity:** MEDIUM
- **Likelihood:** MEDIUM (only affects users who never change settings + trade AMM pools)
- **Impact:** Up to 5% MEV extraction per swap for affected users

### Attack Scenario
```
Step 1: New user connects wallet, never opens settings
  Prerequisite: Default 500 BPS slippage active

Step 2: User swaps 10 SOL for CRIME on AMM pool post-graduation
  Target: useSwap.ts
  Result: minimumOutput = 95% of quoted output

Step 3: MEV bot front-runs with large buy, inflating price
  Result: User's TX executes at worse rate (up to 5% worse)

Step 4: MEV bot back-runs with sell, pocketing the difference
  Result: Bot extracts up to ~0.5 SOL from a 10 SOL swap

Final Result: User loses up to 5% of swap value to MEV extraction
```

### Recommended Fix
```typescript
// Before
return {
  slippageBps: 500,  // 5% -- too high for AMM

// After
return {
  slippageBps: 100,  // 1% -- standard DeFi default
```

Additionally, change the warning threshold:
```typescript
// Before
const highSlippage = slippageBps > 500; // > 5%

// After
const highSlippage = slippageBps > 200; // > 2%
```
