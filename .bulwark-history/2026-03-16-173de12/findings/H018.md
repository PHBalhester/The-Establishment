# Finding: H018 - Graduation Script State File Tampering

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The graduation state file (`graduation-state.json`) has no cryptographic integrity protection (no HMAC, no signature, no checksum). An attacker with filesystem write access to the admin's machine could modify the file to mark steps as completed, causing the script to skip them on the next resume. However, this attack is effectively mitigated by two independent factors: (1) every step that matters has on-chain idempotency checks that would either catch the inconsistency or cause the step to safely no-op, and (2) filesystem write access to the admin machine is itself a catastrophic compromise that grants far more powerful attack paths (direct access to the admin keypair). The state file tampering adds no incremental attack value beyond what filesystem access already provides.

## Hypothesis Investigated

The graduation state file (`graduation-state.json`) has no integrity protection. An attacker with filesystem access could mark steps as completed to skip critical graduation steps.

## Investigation Path

### Step 1: State file read/write mechanism
**Location:** `scripts/graduation/graduate.ts:147,172-193`
**Finding:** The state file is a plain JSON file at `scripts/graduation/graduation-state.json`. It is read with `fs.readFileSync` and parsed with `JSON.parse`. There is no HMAC, checksum, signature, or any integrity verification. Any process with write access to the file can modify it.
**Evidence:**
```typescript
// Line 147
const STATE_FILE = path.resolve(__dirname, "graduation-state.json");

// Lines 172-189
function loadState(): GraduationState {
  if (fs.existsSync(STATE_FILE)) {
    const raw = fs.readFileSync(STATE_FILE, "utf8");
    const state: GraduationState = JSON.parse(raw);
    return state;
  }
  // ... create fresh state
}
```

### Step 2: Step skip mechanism via checkpoint
**Location:** `scripts/graduation/graduate.ts:885-891`
**Finding:** The execution loop iterates over `state.steps` and skips any step where `step.completed === true`. If an attacker sets `completed: true` for any step, that step is skipped entirely.
**Evidence:**
```typescript
// Lines 885-891
for (let i = 0; i < state.steps.length; i++) {
  const step = state.steps[i];
  if (step.completed) {
    console.log(`  SKIP: ${step.name} (completed at ${step.timestamp})`);
    continue;
  }
```

### Step 3: Impact analysis of skipping each step
**Location:** `scripts/graduation/graduate.ts:898-950` (switch statement)
**Finding:** I analyzed the consequence of skipping each step:

| Step | Name | Skip Consequence |
|------|------|-----------------|
| 1 | `verify_curves_filled` | Verification-only; step 2 would fail on-chain if curves aren't Filled |
| 2 | `prepare_transition` | **Most critical**: curves stay in Filled, not Graduated. Steps 3-4 `withdrawGraduatedSol` would fail on-chain (requires Graduated status). Pool creation (7-8) would proceed but with wrong state. |
| 3-4 | `withdraw_*_sol` | SOL stays in curve vaults. Pool creation (7-8) proceeds with admin SOL instead. Curve SOL remains recoverable later. |
| 5-6 | `close_*_token_vault` | Non-critical -- vault stays open. Token balance check prevents data loss. |
| 7-8 | `create_*_pool` | Pools not created. `accountExists` check in the function itself would catch this on resume, but if marked complete, no resume. Protocol stuck without AMM pools. |
| 9 | `seed_conversion_vault` | Vault empty. Conversion feature non-functional. |
| 10-11 | `distribute_*_tax_escrow` | Permissionless operation -- anyone can call later. Low impact. |

### Step 4: On-chain idempotency as defense layer
**Location:** `scripts/graduation/graduate.ts:343-349,393-399,444-448,509-512,632-637,700-705`
**Finding:** Most step implementations have their own on-chain state checks that detect whether the operation has actually been completed, independent of the state file:

- Step 2 (`prepareTransition`): Checks if curves are already `graduated` (line 346)
- Steps 3-4 (`withdrawSol`): Checks vault balance <= rent-exempt minimum (line 396)
- Steps 5-6 (`closeTokenVault`): Checks if vault account is null (line 445)
- Steps 7-8 (`createAmmPool`): Checks if pool already exists (line 509)
- Step 9 (`seedConversionVault`): Checks if vault has tokens (line 634)
- Steps 10-11 (`distributeTaxEscrow`): Checks escrow balance <= rent (line 702)

However, these checks only run IF the step executes. The state file skip at line 888 happens BEFORE the step function is called, so these checks are bypassed by tampering.

### Step 5: Prerequisite analysis -- filesystem access
**Location:** `scripts/graduation/graduate.ts:206-217` (mint keypair loading), `scripts/deploy/lib/connection.ts:64-101` (wallet loading)
**Finding:** An attacker with filesystem write access to the admin machine already has access to:
1. The admin wallet keypair (`keypairs/devnet-wallet.json` or the `WALLET` env var source)
2. All mint keypairs (`scripts/deploy/mint-keypairs/`)
3. The `.env` file with `CLUSTER_URL` containing the RPC URL

With the admin keypair, the attacker can directly call ANY on-chain instruction (including `prepareTransition`, `withdrawGraduatedSol`, pool creation) without needing the graduation script at all. State file tampering is strictly less powerful than direct keypair theft.

## Invariant Analysis

### Target Invariant
Graduation steps execute in order, completely, and each step actually performs its intended on-chain operation before being marked complete.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `graduate.ts:885-891` | Sequential iteration over steps | Yes (for execution order) |
| `graduate.ts:952-957` | Step marked complete only AFTER successful execution | Yes (for honest execution) |
| `graduate.ts:191-193` | State persisted to disk | Yes (but no integrity protection) |
| On-chain programs | Each instruction validates preconditions | Yes (but bypassed if step skipped) |

## Blocking Factors
- **Filesystem access prerequisite**: `graduate.ts:147` -- state file lives on admin's local machine. Requires prior compromise of the machine.
- **Admin keypair co-located**: `scripts/deploy/lib/connection.ts:92` -- admin keypair is on the same machine. Filesystem access grants direct keypair theft, which is strictly more powerful than state file tampering.
- **On-chain state verification at step 2**: `graduate.ts:346` -- if step 2 is skipped via tamper, steps 3-4 would fail on-chain because `withdrawGraduatedSol` requires Graduated status.
- **Post-graduation verification**: `graduate.ts:982` -- runs after all steps, checking on-chain state matches expectations. Would surface inconsistencies.

## Enabling Factors
- **No integrity check on state file**: `graduate.ts:173-176` -- plain JSON.parse with no HMAC or signature verification.
- **Skip logic before execution**: `graduate.ts:888-891` -- state file `completed` flag bypasses the step function entirely, skipping the on-chain idempotency checks.
- **Default file permissions**: `graduate.ts:192` -- `fs.writeFileSync` with no explicit mode, likely 0644 on most systems.

**Overall counterargument confidence: HIGH**

All five challenges are refuted with high confidence. The core defense is that filesystem access to the admin machine is a prerequisite that subsumes state file tampering -- an attacker with that access has the admin keypair and can do anything without touching the state file.

## Incidental Discoveries
- **State file default permissions**: `graduate.ts:192` uses `fs.writeFileSync` without an explicit mode parameter. On most Unix systems this creates the file with 0644 permissions (world-readable). While not exploitable for this hypothesis (reading the file has no value, writing requires separate access), it leaks graduation progress information. Suggested hypothesis: H-NEW -- investigate whether any state files or logs written by admin scripts use overly permissive file modes.
- **No schema validation on loaded state**: `graduate.ts:175` does `JSON.parse(raw)` and casts directly to `GraduationState` with no runtime validation. A corrupted or malformed state file (e.g., from disk error, not attack) could cause the script to crash with an unhelpful error or silently misbehave. This is a robustness concern, not a security concern.
