# Finding: H064 - ALT Cache Stale Data

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The ALT (Address Lookup Table) cache at `scripts/e2e/lib/alt-helper.ts:48` persists the ALT address to `scripts/deploy/alt-address.json`. If the ALT were recreated (e.g., during a redeploy with new program addresses), the cached address would point to the old ALT. However, the `getOrCreateProtocolALT()` function has built-in staleness detection: it verifies the cached ALT exists on-chain AND contains all required addresses, extending it with missing addresses or creating a new ALT if the cached one is invalid. This makes the cache self-healing. The only scenario where this could cause issues is if the old ALT is deactivated/closed on-chain AND the cache file still references it, which the code handles by catching the error and creating a new ALT.

## Hypothesis Investigated

Cached ALT address could become stale if ALT recreated.

## Investigation Path

### Step 1: Examine ALT cache mechanism
**Location:** `scripts/e2e/lib/alt-helper.ts:48,218-264`
**Finding:** ALT address cached in JSON file. On load, the code verifies the ALT exists on-chain and has all needed addresses.
**Evidence:**
```typescript
const ALT_CACHE_PATH = path.resolve(__dirname, "../../deploy/alt-address.json");
// ...
if (fs.existsSync(ALT_CACHE_PATH)) {
  const cached = JSON.parse(fs.readFileSync(ALT_CACHE_PATH, "utf-8"));
  const altAddress = new PublicKey(cached.altAddress);
  const altAccount = await connection.getAddressLookupTable(altAddress);
  if (altAccount.value) {
    // Verify ALL needed addresses are present
    const altKeys = new Set(altAccount.value.state.addresses.map((a) => a.toBase58()));
    const missing = allAddresses.filter((addr) => !altKeys.has(addr.toBase58()));
    if (missing.length === 0) {
      return altAccount.value;  // Cache is valid
    }
    // Extend with missing addresses
    await extendALT(provider, altAddress, missing);
```

**Location:** `scripts/e2e/lib/alt-helper.ts:259-263`
**Finding:** If the cached ALT doesn't exist on-chain or any error occurs during loading, the code falls through to create a new ALT.
**Evidence:**

### Step 3: Verify completeness check
**Location:** `scripts/e2e/lib/alt-helper.ts:228-246`
**Finding:** The code computes ALL needed protocol addresses fresh on every call via `collectProtocolAddresses()`, then compares against what the cached ALT has. Any missing addresses trigger an extension. This means even if new PDAs are added to the manifest, the cached ALT gets updated.
