# Finding: H007 - Cross-Epoch Tax Arbitrage via VRF Observation

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The hypothesis posits that an attacker can monitor Switchboard VRF reveal transactions, pre-compute new epoch tax rates from the revealed VRF bytes, and front-run `consume_randomness` to execute swaps at the old (favorable) rates before the new rates take effect.

This attack is not viable because the crank bundles `revealIx + consumeIx + executeCarnageAtomicIx` in a single atomic v0 VersionedTransaction (TX3). VRF bytes are never observable on-chain in a revealed-but-unconsumed state. The reveal and consume happen in the same transaction, so there is no window between VRF revelation and tax rate update. Furthermore, between TX2 (trigger) and TX3 (reveal+consume), the old tax rates remain active and the new rates are unknowable because the VRF has not been revealed yet.

## Hypothesis Investigated

An attacker monitors the Switchboard VRF reveal transaction, predicts new tax rates, and front-runs `consume_randomness` to execute trades at old (favorable) rates.

## Investigation Path

### Step 1: VRF Lifecycle Timing Analysis
**Location:** `scripts/vrf/lib/vrf-flow.ts:269-341`
**Finding:** TX3 bundles reveal + consume + executeCarnageAtomic in a single v0 VersionedTransaction. The three instructions execute atomically -- either all succeed or all fail. There is no on-chain state where VRF bytes are revealed but not yet consumed.
**Evidence:**
```typescript
// vrf-flow.ts:310-327
const instructions = [
  ComputeBudgetProgram.setComputeUnitLimit({ units: 600_000 }),
  revealIx,
  consumeIx,
  carnageIx,
];

const sig = await sendV0Transaction(
  connection,
  wallet.publicKey,
  instructions,
  [wallet.payer],
  accounts.alt
);
```

### Step 2: Tax Rate Update Timing
**Location:** `programs/epoch-program/src/instructions/trigger_epoch_transition.rs:176-186`
**Finding:** `trigger_epoch_transition` (TX2) advances the epoch number and sets `vrf_pending = true`, `taxes_confirmed = false`, but does NOT modify any tax rate fields. Old tax rates remain active between TX2 and TX3.
**Evidence:**
```rust
// trigger_epoch_transition.rs:176-182
epoch_state.current_epoch = expected_epoch;
epoch_state.epoch_start_slot = epoch_start_slot(expected_epoch, epoch_state.genesis_slot);
epoch_state.vrf_request_slot = clock.slot;
epoch_state.vrf_pending = true;
epoch_state.taxes_confirmed = false;
// NOTE: No tax rate fields are modified here
```

### Step 3: Tax Program Rate Reading
**Location:** `programs/tax-program/src/instructions/swap_sol_buy.rs:67-78`
**Finding:** The Tax Program reads tax rates from on-chain EpochState at execution time. It does NOT check `taxes_confirmed`. This is design choice A-10 and means swaps execute with whatever rates are stored in EpochState at the moment of execution.
**Evidence:**
```rust
// swap_sol_buy.rs:67-78
let epoch_state = {
    let data = ctx.accounts.epoch_state.try_borrow_data()?;
    let mut data_slice: &[u8] = &data;
    EpochState::try_deserialize(&mut data_slice)
        .map_err(|_| error!(TaxError::InvalidEpochState))?
};
require!(epoch_state.initialized, TaxError::InvalidEpochState);
let tax_bps = epoch_state.get_tax_bps(is_crime, true);
```

### Step 4: taxes_confirmed Not Enforced by Tax Program
**Location:** `programs/tax-program/src/state/epoch_state_reader.rs:33-36`
**Finding:** The Tax Program's EpochState reader includes `taxes_confirmed` in the struct (to match byte layout) but never references it in any swap handler. Confirmed via grep: zero references to `taxes_confirmed` in `programs/tax-program/src/instructions/`.
**Evidence:**
```rust
// epoch_state_reader.rs:33
// VRF State - Tax Program doesn't use but must match layout (8 + 1 + 1 + 32 = 42 bytes)
pub vrf_request_slot: u64,
pub vrf_pending: bool,
pub taxes_confirmed: bool,
```

### Step 5: consume_randomness Updates Rates Atomically
**Location:** `programs/epoch-program/src/instructions/consume_randomness.rs:108-207`
**Finding:** `consume_randomness` reads VRF bytes, derives new tax rates, and writes them to EpochState all within a single instruction. Since this instruction is bundled with `revealIx` in the same transaction (TX3), there is no intermediate state where VRF bytes exist on-chain but rates haven't been updated.

### Step 6: Crank Log Output Analysis
**Location:** `scripts/crank/crank-runner.ts:286-300`
**Finding:** The crank logs VRF bytes (`vrfBytes.slice(0, 8)`) and tax rates to stdout AFTER TX3 completes. This is post-facto logging, not a pre-reveal information leak. Railway logs are accessible only to project operators, not public.

### Step 7: Frontend Tax Rate Display
**Location:** `app/hooks/useEpochState.ts:140-178`
**Finding:** The frontend subscribes to EpochState via WebSocket (`onAccountChange`) at "confirmed" commitment. It receives updated tax rates only AFTER `consume_randomness` has written them on-chain. There is no mechanism for the frontend to observe VRF bytes before they are consumed.

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Monitor Switchboard oracle TX for VRF reveal | No | Reveal is bundled atomically with consume in TX3 (`vrf-flow.ts:310-327`). No separate reveal TX exists on-chain. |
| 2 | Read VRF bytes before consume_randomness | No | VRF bytes are read and consumed in the same transaction. No intermediate state. |
| 3 | Pre-compute new tax rates from VRF bytes | No (blocked by step 2) | Cannot observe VRF bytes before they are consumed. |
| 4 | Execute swap at old rates before new rates apply | Partially possible | Between TX2 and TX3 (~3 slots / ~1.2s), old rates are active. But attacker cannot know new rates, so cannot determine if old rates are favorable. |
| 5 | Exploit `taxes_confirmed` being unchecked | N/A | This flag being unchecked does not help -- old rates remain valid and unchanged between TX2 and TX3. |

## Invariant Analysis

### Target Invariant
Tax rates applied to swaps must match the rates stored in on-chain EpochState at the moment of swap execution. No party should be able to trade using rates that differ from the current on-chain state.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `swap_sol_buy.rs:67-78` | Reads tax rates from on-chain EpochState at execution time | Yes |
| `swap_sol_sell.rs` (equivalent) | Same pattern for sell direction | Yes |
| `vrf-flow.ts:310-327` | Atomic bundling prevents VRF reveal without simultaneous consume | Yes |
| `consume_randomness.rs:207` | Sets `taxes_confirmed = true` after rate update | Yes |
| `trigger_epoch_transition.rs:176-182` | Does NOT change rates during trigger -- old rates stay active | Yes |

## Blocking Factors
- **Atomic TX3 bundling**: `vrf-flow.ts:310-327` -- reveal + consume + carnage in single v0 TX eliminates the window between VRF revelation and rate update
- **On-chain rate reading**: `swap_sol_buy.rs:78` -- Tax Program reads rates from EpochState at execution time, not from client parameters
- **Anti-reroll protection**: `consume_randomness.rs:88` -- randomness account must match the one bound during trigger, preventing manipulation of which VRF bytes are consumed

## Enabling Factors
- **`taxes_confirmed` unchecked (A-10)**: `epoch_state_reader.rs:36` -- swaps are not gated during the VRF pending window. However, this only allows trading at OLD rates (which are known to all participants), not at predicted NEW rates.
- **TX2 observable**: `vrf-flow.ts:573-597` -- TX2 (commit+trigger) is visible on-chain, signaling that an epoch transition is imminent. An observer knows rates WILL change but cannot predict HOW.

## If NOT VULNERABLE

### Protection Analysis
| Protection | Location | How It Works |
|------------|----------|--------------|
| Atomic TX3 bundling | `vrf-flow.ts:310-327` | Reveal + consume + carnage are in one VersionedTransaction. VRF bytes never exist in a revealed-but-unconsumed state on-chain. |
| On-chain rate enforcement | `swap_sol_buy.rs:67-78` | Tax Program reads rates from EpochState account at execution time. Client-side rate display is informational only. |
| Anti-reroll protection | `consume_randomness.rs:88` | Bound randomness account prevents an attacker from substituting a different VRF source. |
| VRF unpredictability | Switchboard oracle design | VRF output is cryptographically unpredictable before the oracle reveals it. |

### Devil's Advocate Analysis (MANDATORY)

| Challenge | Argument | Refutation | Evidence | Confidence |
|-----------|----------|------------|----------|------------|
| Crafted input | Attacker sends a crafted `consume_randomness` TX that separates reveal from consume, creating a window. | `consume_randomness` reads from the randomness account that was already revealed by `revealIx` in the same TX. Even if called separately, the attacker still cannot observe the VRF bytes before the oracle reveals them. The oracle only reveals to the randomness account owner. Moreover, anyone can call consume_randomness (caller is just a signer, line 39), but they'd need to wait for the oracle to reveal first, and by then the crank's atomic TX3 has already consumed the bytes. | `consume_randomness.rs:37-53` -- no authority check on caller, but anti-reroll ensures same randomness account. `vrf-flow.ts:310-327` -- crank submits TX3 immediately after oracle responds. | HIGH |
| Multiple accounts/Sybil | Attacker runs their own crank, calling trigger+commit with their own randomness, then separating reveal from consume. | `trigger_epoch_transition` requires `vrf_pending == false` (`trigger_epoch_transition.rs:149`). Only one VRF can be pending at a time. The crank's TX2 sets `vrf_pending = true`, blocking any other trigger. The attacker would need to wait for VRF timeout (300 slots) to retry, and the crank would have completed the cycle by then. | `trigger_epoch_transition.rs:149` -- `require!(!epoch_state.vrf_pending, EpochError::VrfAlreadyPending)` | HIGH |
| External API MITM | Attacker compromises or monitors the Switchboard oracle gateway to see VRF bytes before the reveal TX is submitted. | Switchboard VRF uses verifiable random functions -- the output is deterministic from the oracle's private key + seed, but unpredictable without the private key. Even if the attacker monitors the gateway HTTP response, the crank immediately bundles reveal+consume atomically. The attacker would need to front-run TX3 with their own consume TX, but consume requires the same randomness account (anti-reroll), and the crank already holds the reveal instruction. The attacker cannot construct a valid reveal instruction without the oracle's response. | `tryReveal` at `vrf-flow.ts:229-254` fetches from oracle; TX3 bundled immediately at `vrf-flow.ts:310-327`. | MEDIUM |
| Race condition / timing | Attacker monitors the mempool for TX3 (reveal+consume), reads the VRF bytes from the reveal instruction data, and front-runs with swaps that exploit the new rates. | Even if the attacker could parse VRF bytes from TX3's instruction data in the mempool, they cannot execute swaps at the OLD rates because: (a) their swap TX would need to land BEFORE TX3, in which case old rates still apply to everyone, or (b) their swap TX lands AFTER TX3, in which case new rates apply to everyone. The attacker gains no advantage from knowing the VRF bytes because both the rate change and the attacker's swap would use whatever rates are on-chain at execution time. The attacker's optimal strategy is simply to react quickly after TX3 confirms -- which is standard DeFi behavior, not an exploit. | `swap_sol_buy.rs:78` -- rates read at execution time from on-chain state, not from TX parameters. | HIGH |
| Cross-layer chaining | Attacker combines this with the crank log output (Railway stdout) that includes VRF bytes and rates, predicting future rate patterns. | Crank logs are post-facto (`crank-runner.ts:286-300` -- logged AFTER TX3 completes). Railway logs require project access (authenticated dashboard). Even if logs were accessible, they show historical data, not future VRF outputs. VRF outputs are cryptographically independent across epochs. | `crank-runner.ts:296` -- `vrfBytes: result.vrfBytes.slice(0, 8)` logged after `advanceEpochWithVRF` returns. | HIGH |

**Overall counterargument confidence: HIGH**

One row (External API MITM) has MEDIUM confidence due to the theoretical possibility of an attacker with oracle gateway access and extremely fast TX construction. However, even in this scenario, the atomic bundling means the attacker cannot separate reveal from consume, and front-running TX3 with swaps only executes at old rates (which offer no information advantage). Upgrading to HIGH overall because the economic argument holds regardless: the attacker cannot profit from knowing future rates if they can only trade at current rates.

## Incidental Discoveries
- **Recovery path Carnage gap**: `vrf-flow.ts:533` -- The VRF recovery path (stale VRF handling) explicitly sets `carnageExecutedAtomically: false`. If Carnage triggers during a recovery cycle, there would be a CarnagePending window where external actors could observe and front-run the Carnage swap. This is a known limitation acknowledged in the code but worth a separate hypothesis if not already covered. Suggested hypothesis: "Can an attacker force VRF recovery (e.g., by DoSing the oracle gateway) to create a non-atomic Carnage window?"
- **`taxes_confirmed` could serve as a UX guard**: The flag is set to `false` between TX2 and TX3 but is never checked by the Tax Program. While this is an accepted design choice (A-10), the frontend could use this flag to warn users that rates are in flux during the ~1.2s VRF window. Currently `useEpochState.ts` exposes epoch data but does not expose `taxes_confirmed` -- it is not in the `EpochStateData` interface (`useEpochState.ts:35-49`). Low priority, informational only.
