# Finding: H107 - Dual-Curve Grief Attack

## Status: NOT VULNERABLE
## Confidence Score: 7

## Rationale
The hypothesis posits that an attacker could prevent one bonding curve from filling while the other fills, causing a deadlock. However, the bonding curve design includes explicit failure handling: each curve has a `deadline_slot` (set at activation via `start_slot + DEADLINE_SLOTS`). If a curve is not filled by the deadline + `FAILURE_GRACE_SLOTS`, anyone can call the permissionless `mark_failed` instruction. When either curve is marked as Failed, `is_refund_eligible()` returns true for both curves -- users on the Filled curve can also claim refunds (the state machine handles `(Filled, Failed)` as refund-eligible per `state.rs:134`). Additionally, per-wallet caps (`MAX_TOKENS_PER_WALLET`) prevent a single actor from filling one curve entirely. The grief vector is economically irrational: an attacker buying one curve to fill and blocking the other would lose all their SOL when the deadline triggers failure and refunds.

## Code Evidence

The deadline + refund mechanism prevents permanent griefing.
