# Finding: H054 - Carnage Fallback MEV Sandwich (Cross-Boundary)

## Status: POTENTIAL
## Confidence Score: 7

## Executive Summary

The Carnage execution system has two paths: atomic (bundled in the same TX as consume_randomness, slots 0-50) and fallback (permissionless, slots 50-300). The hypothesis asks whether the fallback path is sandwichable.

After investigation, the off-chain crank runner (`scripts/crank/crank-runner.ts`) always uses the atomic path -- it bundles `reveal + consume + executeCarnageAtomic` in a single v0 VersionedTransaction via `vrf-flow.ts`. This means the crank never calls the permissionless `execute_carnage` fallback instruction under normal operation.

However, the on-chain `execute_carnage` (fallback) instruction exists and is permissionless (caller can be anyone after slot 50). If the atomic path fails (TX dropped, compute exceeded, ALT stale), `carnage_pending` remains true and the fallback window opens at slot 50. Any MEV bot monitoring the chain can then:

1. Observe `carnage_pending = true` + `carnage_target` + `carnage_action` on-chain
2. Wait for slot 50 (lock window expiry)
3. Sandwich the `execute_carnage` call: front-run to manipulate pool reserves, execute carnage, back-run to profit

The on-chain slippage floor for fallback is 75% (`CARNAGE_SLIPPAGE_BPS_FALLBACK = 7500`), meaning up to 25% slippage is tolerated. This is a wide window for sandwich extraction.

The key question is: how often does the atomic path fail? The off-chain code is well-designed with bundled v0 transactions, but failures are possible (compute limit, blockhash expiry, RPC timeout). The recovery path in `vrf-flow.ts:533` explicitly sets `carnageExecutedAtomically: false` and does NOT attempt atomic Carnage -- leaving the fallback window open.

## Hypothesis Investigated

After 50-slot lock, Carnage fallback is permissionless and sandwichable.

## Investigation Path

### Step 1: Crank atomic bundling
**Location:** `scripts/vrf/lib/vrf-flow.ts:293-328`
**Finding:** TX3 bundles `reveal + consume + executeCarnageAtomic` in one v0 TX. The on-chain no-op guard makes this safe even when Carnage doesn't trigger.
**Evidence:**
```typescript
// Bundle all three: reveal + consume + executeCarnageAtomic in ONE v0 TX
const instructions = [
  ComputeBudgetProgram.setComputeUnitLimit({ units: 600_000 }),
  revealIx,
  consumeIx,
  carnageIx,
];
const sig = await sendV0Transaction(connection, wallet.publicKey, instructions, [wallet.payer], accounts.alt);
```

### Step 2: Recovery path does NOT attempt atomic Carnage
**Location:** `scripts/vrf/lib/vrf-flow.ts:492-534`
**Finding:** The VRF recovery path (stale VRF) completes the epoch transition but explicitly does NOT attempt atomic Carnage. `carnageExecutedAtomically: false`.
**Evidence:**
```typescript
return {
  // ...
  carnageExecutedAtomically: false, // Recovery path does not attempt atomic Carnage
};
```

### Step 3: On-chain fallback exists and is permissionless
**Location:** `programs/epoch-program/src/instructions/execute_carnage.rs:3,46`
**Finding:** The on-chain `execute_carnage` instruction is permissionless (caller is just `Signer`, no authority check beyond that). It's available after the 50-slot lock window.
**Evidence:**
```rust
/// Caller (anyone - permissionless)
pub caller: Signer<'info>,
// ...
require!(
    clock.slot > ctx.accounts.epoch_state.carnage_lock_slot,
    EpochError::CarnageLockActive
);
```

### Step 4: Fallback slippage tolerance
**Location:** `programs/epoch-program/src/constants.rs:132`
**Finding:** Fallback uses `CARNAGE_SLIPPAGE_BPS_FALLBACK = 7500` (75% floor), allowing up to 25% price deviation.
**Evidence:**
```rust
pub const CARNAGE_SLIPPAGE_BPS_FALLBACK: u64 = 7500;
```

### Step 5: Crank runner does NOT call fallback
**Location:** `scripts/crank/crank-runner.ts:271-283`
**Finding:** Crank only calls `advanceEpochWithVRF()` which uses the atomic path. If atomic fails and `carnage_pending` remains true, the crank does NOT separately call `execute_carnage` (fallback).
**Evidence:**
```typescript
const result = await advanceEpochWithVRF(provider, programs.epochProgram, vrfAccounts);
// No separate fallback carnage execution
```

### Step 6: Carnage state visibility
**Finding:** After `consume_randomness` sets `carnage_pending = true`, the on-chain state reveals `carnage_target` (CRIME or FRAUD) and `carnage_action` (BuyOnly/Burn/BurnAndSell). This is readable by anyone, giving MEV bots full information about what the fallback Carnage will do.

## Invariant Analysis

### Target Invariant
Carnage execution should not be extractable by MEV bots.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| Atomic bundling | `vrf-flow.ts:293-328` | Carnage in same TX as reveal -- no MEV window | Yes (happy path) |
| 50-slot lock | On-chain `carnage_lock_slot` | Prevents fallback during atomic-only period | Yes |

## Blocking Factors
- Atomic bundling prevents MEV in happy path: `vrf-flow.ts:293-328`
- 50-slot lock window prevents early fallback: `execute_carnage.rs:223-225`
- On-chain slippage floor limits extraction: `constants.rs:132` (but 25% is large)

## Enabling Factors
- Recovery path skips atomic Carnage: `vrf-flow.ts:533`
- Fallback is fully permissionless: `execute_carnage.rs:46`
- Carnage target/action visible on-chain: EpochState account
- 25% slippage tolerance on fallback: `constants.rs:132`
- `MAX_CARNAGE_SWAP_LAMPORTS` could be significant SOL amounts (extractable value)
- Crank has no separate fallback retry: `crank-runner.ts:271-283`

```
  Target: AMM pool for the Carnage target token (e.g., CRIME/SOL)
  Payload: Large buy to inflate token price before Carnage buys
  Prerequisite: Sufficient capital in target pool

Step 4: Call execute_carnage (permissionless fallback)
  Target: Epoch program execute_carnage instruction
  Prerequisite: Step 3 completed in same slot
  Result: Carnage executes at inflated price, 75% slippage floor allows it

Step 5: Back-run: reverse the price manipulation
  Target: Same AMM pool
  Payload: Sell the tokens bought in step 3
  Prerequisite: Step 4 completed
  Result: Bot profits from the price delta (up to 25% of Carnage value)

Final Result: MEV bot extracts up to 25% of the Carnage swap value. Carnage fund receives fewer tokens than optimal.
```

## Incidental Discoveries
- `vrf-flow.ts:533` recovery path sets `carnageExecutedAtomically: false` but the crank runner at `crank-runner.ts:281` only logs `result.carnageTriggered` -- it does not take any action when `carnageTriggered=true && carnageExecutedAtomically=false`. This is the exact gap that enables the MEV window. Suggested hypothesis: "Crank runner ignores failed atomic Carnage, leaving no off-chain fallback attempt."
- The `carnage-flow.ts` E2E test code has a `testForcedCarnage` function that CAN call the fallback, but this is test code, not production crank code. The production crank lacks this fallback.
