# Finding: H053 - Pool Reserve Read Without Owner Check (Cross-Boundary)

## Status: NOT VULNERABLE
## Confidence Score: 7

## Executive Summary

The hypothesis targets off-chain pool reserve reading at raw byte offsets without owner checks. Investigation reveals that the off-chain code does NOT read pool reserves at raw byte offsets. Instead, it uses two safe patterns:

1. **Frontend (`app/hooks/usePoolPrices.ts`)**: Uses Anchor's `program.coder.accounts.decode("poolState", accountInfo.data)` which validates the 8-byte Anchor discriminator before decoding. If a spoofed account has the wrong discriminator, the decode throws an error.

2. **Frontend (`app/hooks/useRoutes.ts`)**: Consumes already-decoded `PoolData` from `usePoolPrices`, not raw bytes.

3. **Crank runner**: Does not read pool reserves at all -- it passes pool account addresses to on-chain instructions which validate ownership via Anchor constraints.

The `route-types.ts:105` comment mentioning "byte offsets 137 and 145" refers to the on-chain Rust code in `programs/tax-program/src/helpers/pool_reader.rs`, which is out of scope for this off-chain audit. The off-chain code exclusively uses Anchor's typed deserialization with discriminator checks.

The one weakness: `getMultipleAccountsInfo` at `usePoolPrices.ts:118` does not check `accountInfo.owner` before decoding. An attacker who could make a fake account appear at the pool PDA address (impossible without program upgrade authority) could inject false reserves. But since pool PDAs are derived deterministically from the AMM program, an attacker cannot create accounts at those addresses without controlling the AMM program itself.

## Hypothesis Investigated

execute_carnage reads pool reserves at raw byte offsets without owner check.

## Investigation Path

### Step 1: Frontend pool reserve reading
**Location:** `app/hooks/usePoolPrices.ts:118-146`
**Finding:** Uses `getMultipleAccountsInfo` + `program.coder.accounts.decode("poolState", ...)` -- Anchor discriminator validated.
**Evidence:**
```typescript
const accounts = await connection.getMultipleAccountsInfo(poolPubkeys, "confirmed");
// ...
const decoded = program.coder.accounts.decode("poolState", accountInfo.data);
```

### Step 2: WebSocket update path
**Location:** `app/hooks/usePoolPrices.ts:217-241`
**Finding:** Same Anchor decode for WebSocket updates: `program.coder.accounts.decode("poolState", accountInfo.data)`.

### Step 3: Route engine reserve consumption
**Location:** `app/hooks/useRoutes.ts:79-97`
**Finding:** `toPoolReserves()` converts already-decoded `PoolData` objects (from `usePoolPrices`) to `PoolReserves`. No raw byte access.

### Step 4: Crank runner
**Location:** `scripts/crank/crank-runner.ts:273`
**Finding:** Crank calls `advanceEpochWithVRF()` which passes pool addresses to on-chain instructions. No off-chain pool reserve reading.

### Step 5: Owner check absence
**Location:** `app/hooks/usePoolPrices.ts:129-140`
**Finding:** `accountInfo.owner` is not checked before decode. However, this is mitigated by: (a) pool addresses are PDAs derived from the AMM program, and (b) only the AMM program can create accounts at those addresses.

## Invariant Analysis

### Target Invariant
Pool reserve data used for off-chain quotes must come from legitimate AMM PoolState accounts.

### Enforcement Points
| Location | What It Enforces | Complete? |

| PDA derivation | Only AMM program can create accounts at pool addresses | Yes |

## Blocking Factors
- Anchor `coder.accounts.decode` validates 8-byte discriminator: `usePoolPrices.ts:143`
- Pool addresses are PDAs derived from AMM program: cannot be spoofed without AMM authority
- Off-chain code does NOT use raw byte offsets (that's the on-chain Rust code)

## Enabling Factors
- `accountInfo.owner` is not explicitly checked in `usePoolPrices.ts:129` (defense-in-depth gap)

## If NOT VULNERABLE

### Devil's Advocate Analysis (MANDATORY)
| Challenge | Argument | Refutation | Evidence | Confidence |
|-----------|----------|------------|----------|------------|
| Crafted input | Attacker crafts data with matching Anchor discriminator at a different account | PDA addresses are hardcoded from `DEVNET_POOLS`; attacker can't change which address is queried | `usePoolPrices.ts:57` (`POOL_ENTRIES`) | HIGH |
| Multiple accounts | Attacker creates many fake pool accounts | Can't create accounts at PDA addresses without the AMM program's signing authority | Solana PDA rules | HIGH |
| External API MITM | Attacker intercepts RPC response to inject false accountInfo | RPC MITM would need to modify `getMultipleAccountsInfo` response; Helius uses HTTPS | TLS on RPC connection | MEDIUM |
| Race condition | Attacker front-runs pool state to show stale reserves | This is a standard MEV concern, not a reserve-reading vulnerability; on-chain slippage check protects | On-chain `minimumOutput` enforcement | HIGH |
| Frontend chain | XSS modifies the pool address constants | `DEVNET_POOLS` imported from compiled shared package; XSS would need to override module exports | Module system | MEDIUM |

**Overall counterargument confidence:** HIGH. The combination of Anchor discriminator validation and PDA address security makes this attack infeasible through the off-chain layer.

## Incidental Discoveries
- `usePoolPrices.ts:93-96` `toNum()` helper uses `val.toNumber()` for BN values. For very large reserves (> Number.MAX_SAFE_INTEGER = 9e15), this loses precision. Current pool sizes (290M tokens = 2.9e14 base units with 6 decimals) are safe, but future growth could hit this limit. Suggested hypothesis: "BN.toNumber() precision loss in usePoolPrices for pools with reserves exceeding Number.MAX_SAFE_INTEGER."
