# Finding: H093 - NEXT_PUBLIC_SENTRY_DSN Spam

## Status: POTENTIAL
## Confidence Score: 6

## Rationale
The `NEXT_PUBLIC_SENTRY_DSN` environment variable is exposed to the browser (Next.js `NEXT_PUBLIC_` prefix makes it available client-side). At `app/lib/sentry.ts:30`, the DSN is read and used to construct the Sentry ingest URL. A Sentry DSN is considered semi-public (Sentry docs say DSNs are not secrets), but an attacker who extracts it from the client bundle could spam the Sentry project with fake error envelopes, potentially exhausting the event quota. The custom zero-dependency reporter at `app/lib/sentry.ts:86-92` does fire-and-forget POSTs with no rate limiting on the client side. However, Sentry itself has server-side rate limiting and quota controls.

## Code Evidence
```typescript
// app/lib/sentry.ts:27-31
const sentryDsn =
  dsn ??
  (typeof process !== "undefined"
    ? process.env.NEXT_PUBLIC_SENTRY_DSN ?? process.env.SENTRY_DSN
    : undefined);
```
DSN exposed via NEXT_PUBLIC_ prefix. No client-side rate limiting on error submissions.

## Severity Estimate
LOW -- Sentry DSNs are designed to be semi-public and Sentry has server-side rate limiting. The worst case is quota exhaustion causing legitimate errors to be dropped, which is an operational nuisance rather than a security vulnerability.
