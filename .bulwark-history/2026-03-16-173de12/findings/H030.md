# Finding: H030 - VRF Wait Loop Potential Infinite Loop

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The `waitForSlotAdvance` function at `scripts/vrf/lib/vrf-flow.ts:159-195` contains a `while (true)` loop that polls the Solana RPC for slot advancement. The loop has no maximum iteration count, no wall-clock timeout, and no escape condition other than the slot reaching the target value. If the RPC consistently returns the same slot number (due to cluster halt, RPC caching bug, network partition, or stale RPC proxy), the loop runs forever, blocking the crank process entirely.

This is distinct from H029 (crank retry without backoff) because the infinite loop occurs INSIDE the try block -- it never reaches the catch handler. The crank is alive but permanently stuck, not retrying. SIGINT/SIGTERM handlers would still work (the `sleep` calls yield the event loop), but no automated mechanism detects or recovers from this state.

## Hypothesis Investigated

`waitForSlotAdvance` could loop infinitely if slot never advances.

## Investigation Path

### Step 1: waitForSlotAdvance implementation
**Location:** `scripts/vrf/lib/vrf-flow.ts:159-195`
**Finding:** Pure `while (true)` loop with only one exit: `if (currentSlot >= targetSlot) return;`. RPC errors are caught and retried (line 190-193), but the case where the RPC succeeds but returns a stale/unchanging slot is NOT handled.
**Evidence:**
```typescript
export async function waitForSlotAdvance(
  connection: Connection,
  targetSlots: number
): Promise<void> {
  const startSlot = await connection.getSlot();
  const targetSlot = startSlot + targetSlots;

  // For long waits, sleep most of the estimated time first
  if (targetSlots > 30) {
    const estimatedMs = targetSlots * 400;
    const sleepMs = estimatedMs - 15_000;
    if (sleepMs > 0) {
      await sleep(sleepMs);
    }
  }

  const pollMs = targetSlots > 30 ? 10_000 : 500;

  while (true) {           // <-- No escape condition other than slot reaching target
    await sleep(pollMs);
    try {
      const currentSlot = await connection.getSlot();
      if (currentSlot >= targetSlot) {
        console.log(`  [slot] Advanced to ${currentSlot}`);
        return;
      }
    } catch (e) {
      // Transient RPC error -- retry after brief delay
      await sleep(2000);
    }
  }
}
```

### Step 3: Missing timeout analysis
**Location:** `scripts/vrf/lib/vrf-flow.ts:182`
**Finding:** No `Date.now()` wall-clock check, no iteration counter, no `AbortController`, no `setTimeout` race. The function signature returns `Promise<void>` with no cancellation mechanism.

### Step 4: Scenarios causing non-advancing slots
**Finding:** Multiple realistic scenarios:
1. **Solana cluster halt**: Has occurred historically (Feb 2023, Feb 2024). All slots stop advancing.
2. **RPC caching/proxy bug**: A CDN or proxy in front of the RPC returns cached `getSlot()` responses.
3. **Network partition**: Crank can reach RPC but RPC has stale view of the cluster.
4. **Devnet instability**: Devnet has frequent slot skips and occasionally stalls.

### Step 5: Recovery mechanisms
**Finding:** The SIGINT/SIGTERM handlers (crank-runner.ts:79-89) set `shutdownRequested = true`, but `waitForSlotAdvance` does NOT check this flag. It would only be checked after `waitForSlotAdvance` returns (which it never does in the stuck case). However, since `await sleep()` yields the event loop, a SIGTERM would eventually cause Railway to send SIGKILL after its grace period.

## Recommended Fix
```typescript
// Before (vulnerable)
while (true) {
    await sleep(pollMs);
    try {
      const currentSlot = await connection.getSlot();
      if (currentSlot >= targetSlot) {
        console.log(`  [slot] Advanced to ${currentSlot}`);
        return;
      }
    } catch (e) {
      await sleep(2000);
    }
  }

// After (fixed)
const MAX_WAIT_MS = 20 * 60 * 1000; // 20 minutes absolute max
const startTime = Date.now();
let lastObservedSlot = startSlot;
let staleCount = 0;

while (true) {
    // Wall-clock timeout
    if (Date.now() - startTime > MAX_WAIT_MS) {
      throw new Error(
        `waitForSlotAdvance timed out after ${MAX_WAIT_MS / 1000}s. ` +
        `Start: ${startSlot}, target: ${targetSlot}, last observed: ${lastObservedSlot}`
      );
    }

    await sleep(pollMs);
    try {
      const currentSlot = await connection.getSlot();
      if (currentSlot >= targetSlot) {
        console.log(`  [slot] Advanced to ${currentSlot}`);
        return;
      }

      // Detect stale RPC (slot not advancing)
      if (currentSlot === lastObservedSlot) {
        staleCount++;
        if (staleCount > 60) { // ~30s at 500ms poll, ~10min at 10s poll
          throw new Error(
            `waitForSlotAdvance: slot stuck at ${currentSlot} for ${staleCount} polls. ` +
            `Possible cluster halt or stale RPC.`
          );
        }
      } else {
        staleCount = 0;
        lastObservedSlot = currentSlot;
      }
    } catch (e) {
      if ((e as Error).message?.includes("waitForSlotAdvance")) throw e; // Re-throw our own timeouts
      await sleep(2000);
    }
  }
```

This adds:
1. **Wall-clock timeout** (20 minutes) -- absolute upper bound on wait time
2. **Stale slot detection** -- throws if the same slot is returned N times consecutively
3. **Re-throw of timeout errors** -- prevents the inner catch from swallowing our own timeout errors

The thrown error propagates to the crank's outer catch block, which handles it as a normal error (log + retry with backoff, per H029 fix).

## Incidental Discoveries
- **`shutdownRequested` not checked inside `waitForSlotAdvance`**: At `vrf-flow.ts:182`, the `while (true)` loop does not check the crank's `shutdownRequested` flag. This means a SIGTERM during a long wait (e.g., 750-slot epoch boundary) will not be processed until the wait completes. The flag should either be passed as a parameter or `waitForSlotAdvance` should accept an `AbortSignal`. Suggested hypothesis: investigate whether graceful shutdown is actually graceful during long slot waits.
- **No stale slot logging**: When `getSlot()` returns the same value repeatedly, there is no log output. The operator sees only the initial "waiting for +N" message and then silence. Adding periodic progress logs (every 30s) would make it possible to distinguish "waiting normally" from "stuck".
