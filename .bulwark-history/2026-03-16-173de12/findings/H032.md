# Finding: H032 - WebSocket Reconnection Loss of Events

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

Both `useEpochState.ts` and `useCurveState.ts` use `connection.onAccountChange()` WebSocket subscriptions for real-time updates. The hypothesis suggests that dropped WebSocket connections could cause permanent loss of on-chain events. Investigation reveals that while the hooks do not implement explicit WebSocket reconnection logic, they have a robust multi-layer recovery mechanism: (1) visibility-aware pause/resume that unsubscribes and re-subscribes WS on tab hide/show, (2) burst-refresh RPC fetches on every tab return, and (3) the underlying `@solana/web3.js` Connection object handles WS reconnection internally. The combination means that even if a WS connection drops, the user gets fresh data within seconds of returning to the tab, and the hooks re-subscribe on visibility transitions.

The key insight is that these hooks track **account state** (not events). Missing a WS notification doesn't lose data -- the next notification or RPC fetch delivers the full current state, not an incremental delta.

## Hypothesis Investigated

No WebSocket reconnection logic; dropped connections lose on-chain events.

## Investigation Path

### Step 1: Check WebSocket subscription pattern
**Location:** `app/hooks/useEpochState.ts:136-183`, `app/hooks/useCurveState.ts:289-360`
**Finding:** Both hooks use `connection.onAccountChange()` which subscribes via WebSocket. The subscription returns a subscription ID stored in a ref for cleanup.
**Evidence:**
```typescript
const subId = connection.onAccountChange(
  DEVNET_PDAS.EpochState,
  (accountInfo: AccountInfo<Buffer>) => { /* decode and setState */ },
  "confirmed"
);
wsSubIdRef.current = subId;
```

### Step 2: Check for explicit reconnection logic
**Location:** Both hook files
**Finding:** No explicit WebSocket reconnection, keepalive, or health-check logic exists in either hook. However, this is by design -- see Step 3.

### Step 3: Check visibility-aware recovery
**Location:** `useEpochState.ts:217-229`, `useCurveState.ts:394-406`
**Finding:** Both hooks implement a visibility-aware lifecycle: when the tab becomes hidden, WS subscriptions are removed. When the tab becomes visible again, the hooks (1) fire a burst-refresh RPC fetch for immediate fresh data, then (2) create new WS subscriptions.
**Evidence:**
```typescript
if (wasActive && !isActive) {
  unsubscribe();
} else if (!wasActive && isActive) {
  fetchEpochState();  // Burst-refresh via RPC
  subscribe();         // New WS subscription
}
```

### Step 4: Check @solana/web3.js internal reconnection
**Finding:** The `@solana/web3.js` Connection class manages WebSocket reconnection internally. When the WS connection drops, it automatically reconnects and re-subscribes. This is the primary reconnection mechanism that the hooks rely on.

### Step 5: Verify state vs event model
**Finding:** Both hooks track **full account state** snapshots, not incremental events. `onAccountChange` delivers the complete `AccountInfo<Buffer>` on every change. Missing one notification means you miss one intermediate state, but the next notification delivers the current state in full. There is no "event loss" -- only temporary staleness.
