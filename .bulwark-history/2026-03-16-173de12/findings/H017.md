# Finding: H017 - Staking Escrow Rent Depletion (Cross-Boundary)

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The off-chain code provides zero protection against the confirmed on-chain staking escrow rent depletion vulnerability (SOS H012). The `useStaking.ts` claim flow does not check the escrow vault's SOL balance before building and submitting a claim transaction. The crank runner monitors and auto-tops-up the Carnage SOL Vault for rent depletion (`crank-runner.ts:220-241`) but does NOT monitor or top up the staking escrow vault. There is no off-chain circuit breaker, balance guard, or escrow health check anywhere in the codebase.

This means the last claimer can drain the escrow below rent-exempt minimum (~890,880 lamports), causing either a claim freeze (runtime rejects sub-rent balance) or PDA destruction (drained to exactly 0), which halts all `deposit_rewards` CPI from the Tax Program -- permanently blocking 71% of swap tax distribution to stakers.

## Hypothesis Investigated

The last staking reward claimer could drain the escrow PDA below rent-exempt minimum, destroying it. Next `deposit_rewards` CPI fails, halting all swap tax distribution.

## Investigation Path

### Step 1: Claim Transaction Construction
**Location:** `app/lib/staking/staking-builders.ts:345-380`
**Finding:** `buildClaimTransaction` passes the hardcoded `DEVNET_PDAS_EXTENDED.EscrowVault` PDA with zero balance validation. No RPC call checks escrow balance before building the transaction.
**Evidence:**
```typescript
// staking-builders.ts:366-375
const claimIx = await stakingProgram.methods
    .claim()
    .accountsStrict({
      user: userPublicKey,
      stakePool: DEVNET_PDAS_EXTENDED.StakePool,
      userStake: userStakePda,
      escrowVault: DEVNET_PDAS_EXTENDED.EscrowVault,
      systemProgram: SystemProgram.programId,
    })
    .instruction();
```

### Step 2: useStaking Claim Flow
**Location:** `app/hooks/useStaking.ts:541-553`
**Finding:** The claim case in `execute()` calls `buildClaimTransaction` without any pre-flight check on escrow balance. The only protection is the on-chain error 6002 (InsufficientEscrowBalance), which is caught post-failure and mapped to a user-friendly string.
**Evidence:**
```typescript
// useStaking.ts:541-553
case "claim": {
    tx = await buildClaimTransaction({
        connection,
        userPublicKey: wallet.publicKey,
        priorityFeeMicroLamports: priorityMicroLamports,
    });
    setLastResult({
        action: "claim",
        claimedAmount: pendingRewards,
    });
    break;
}
```

### Step 3: Crank Runner Escrow Monitoring - ABSENT
**Location:** `scripts/crank/crank-runner.ts:220-241`
**Finding:** The crank monitors `carnageSolVault` balance and auto-tops-up when it falls below `MIN_VAULT_BALANCE` (~2M lamports). There is NO equivalent monitoring for the staking `EscrowVault`. The escrow vault is not mentioned anywhere in the crank runner logic.
**Evidence:**
```typescript
// crank-runner.ts:220-241 -- Carnage vault monitored, escrow NOT
const vaultPubkey = vrfAccounts.carnageAccounts!.carnageSolVault;
const vaultBalance = await provider.connection.getBalance(vaultPubkey);
if (vaultBalance < MIN_VAULT_BALANCE) {
    // ... top-up logic
}
// No equivalent check for EscrowVault
```

### Step 4: Error Handling (Reactive Only)
**Location:** `app/lib/staking/error-map.ts:25`
**Finding:** Error code 6002 is mapped to `"The reward escrow doesn't have enough SOL. Please try again later."` This is purely reactive -- the user discovers the problem only after the transaction fails on-chain. No proactive warning or prevention exists.

### Step 5: On-Chain Confirmation (Cross-Reference)
**Location:** `.audit/findings/H012.md` (SOS audit, CONFIRMED)
**Finding:** The on-chain `claim.rs` checks `escrow_balance >= rewards_to_claim` but does NOT reserve rent-exempt minimum (~890,880 lamports). The Solana runtime rejects transfers that leave a program-owned account between 0 and rent-exempt. Draining to exactly 0 destroys the PDA.

### Recommended Fix

**Fix 1: Off-chain pre-flight escrow balance check in useStaking.ts**
```typescript
// Before (vulnerable) -- useStaking.ts:541-553
case "claim": {
    tx = await buildClaimTransaction({ ... });
    break;
}

// After (fixed) -- add escrow balance check before building TX
case "claim": {
    // Pre-flight: check escrow can cover claim without rent depletion
    const RENT_EXEMPT_MIN = 890_880; // 0-data program-owned account
    const escrowBalance = await connection.getBalance(
        DEVNET_PDAS_EXTENDED.EscrowVault
    );
    if (escrowBalance - pendingRewards < RENT_EXEMPT_MIN && pendingRewards > 0) {
        setStatus("failed");
        setErrorMessage(
            "Escrow balance too low to claim safely. " +
            "Please wait for more swap tax deposits."
        );
        return;
    }
    tx = await buildClaimTransaction({ ... });
    break;
}
```

**Fix 2: Crank escrow monitoring (mirror carnage vault pattern)**
```typescript
// crank-runner.ts -- add after carnage vault check (~line 241)
const escrowPubkey = manifest.EscrowVault; // or DEVNET_PDAS_EXTENDED.EscrowVault
const escrowBalance = await provider.connection.getBalance(escrowPubkey);
if (escrowBalance < MIN_VAULT_BALANCE) {
    console.log(`[crank] Escrow balance low: ${escrowBalance}. Topping up...`);
    const topUpTx = new Transaction().add(
        SystemProgram.transfer({
            fromPubkey: provider.wallet.publicKey,
            toPubkey: escrowPubkey,
            lamports: VAULT_TOP_UP_LAMPORTS,
        })
    );
    await provider.sendAndConfirm(topUpTx, []);
}
```

**Fix 3: On-chain rent guard (PRIMARY FIX -- requires program update)**
See SOS H012 recommendation: reserve `Rent::minimum_balance(0)` in `claim.rs` before transfer.

## Incidental Discoveries
- **Asymmetric rent protection**: The crank runner has explicit rent-depletion protection for the Carnage SOL Vault (`crank-runner.ts:62-75`) but zero protection for the staking Escrow Vault, despite both being 0-data program-owned PDAs with identical rent-exempt requirements (~890,880 lamports). This asymmetry suggests the escrow vault was simply overlooked when the carnage vault protection was implemented. **Suggested hypothesis**: Audit all PDA accounts for rent-depletion protection consistency.
- **No claim amount cap**: The off-chain `useStaking.ts` allows claiming the full `pendingRewards` amount with no partial claim option. If the protocol added a "claim up to X" feature, users could safely claim smaller amounts that leave the escrow above rent-exempt. Currently the on-chain `claim()` takes no amount parameter -- it claims all accrued rewards. **Suggested hypothesis**: Investigate whether adding partial claim functionality would mitigate rent depletion risk.
