# Finding: H026 - Missing HSTS Header

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The Next.js security headers configuration at `app/next.config.ts:58-86` defines five security headers (CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy) but does not include a `Strict-Transport-Security` (HSTS) header. This means on first visit, a user's browser may connect over HTTP before being redirected to HTTPS, creating a window for SSL stripping attacks. While Railway enforces HTTPS at the load balancer level, the browser does not know this -- without HSTS, a network-level attacker (coffee shop WiFi, compromised router) can intercept the initial HTTP request and downgrade the connection.

The `upgrade-insecure-requests` CSP directive (line 22) partially mitigates this for subresources loaded by the page, but it does NOT protect the initial navigation request itself. HSTS is the only mechanism that tells the browser "never connect to this domain over HTTP."

## Hypothesis Investigated

No HSTS header means first-visit SSL stripping attacks are possible.

## Investigation Path

### Step 1: Security Headers Configuration
**Location:** `app/next.config.ts:58-86`
**Finding:** Five security headers are set, but HSTS is absent.
**Evidence:**
```typescript
async headers() {
    return [
      {
        source: "/(.*)",
        headers: [
          { key: "Content-Security-Policy", value: cspHeader.replace(/\s{2,}/g, " ").trim() },
          { key: "X-Frame-Options", value: "DENY" },
          { key: "X-Content-Type-Options", value: "nosniff" },
          { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
          { key: "Permissions-Policy", value: "camera=(), microphone=(), geolocation=()" },
        ],
      },
    ];
  },
```

No `Strict-Transport-Security` header is present.

### Step 2: CSP upgrade-insecure-requests
**Location:** `app/next.config.ts:22`
**Finding:** The CSP includes `upgrade-insecure-requests`, which instructs the browser to upgrade HTTP subresource requests (images, scripts, stylesheets) to HTTPS. However, this directive does NOT protect the top-level navigation. A user typing `http://dr-fraudsworth-production.up.railway.app` or clicking an HTTP link would still make the initial request over HTTP.

### Step 3: Railway Platform Check
**Finding:** Railway provides automatic HTTPS termination at its load balancer and typically redirects HTTP to HTTPS. However, HSTS is a defense-in-depth mechanism that works at the browser level -- even if the server redirects, without HSTS the initial HTTP request is vulnerable to interception before the redirect is received.

## Invariant Analysis

### Target Invariant
All browser-to-server communication must occur over HTTPS, with no window for HTTP interception.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| Railway load balancer | HTTP -> HTTPS redirect | Partial (redirect is interceptable) |
| `next.config.ts:22` (CSP `upgrade-insecure-requests`) | Subresource HTTPS upgrade | Partial (nav requests excluded) |
| HSTS header | Browser refuses HTTP entirely | **MISSING** |

## Blocking Factors
- Railway's HTTPS redirect provides server-side protection, but the redirect response itself travels over HTTP and can be intercepted.

## Enabling Factors
- No HSTS header: `app/next.config.ts:58-86` -- browser has no instruction to refuse HTTP connections
- DeFi application with wallet connections makes this higher-impact than a typical web app

## Recommended Fix
```typescript
// In next.config.ts headers() function, add:
{
  key: "Strict-Transport-Security",
  value: "max-age=63072000; includeSubDomains; preload",
},
```

This tells browsers to:
- Only connect via HTTPS for 2 years (`max-age=63072000`)
- Apply to all subdomains (`includeSubDomains`)
- Eligible for browser HSTS preload lists (`preload`)

After deploying, submit the domain to https://hstspreload.org/ for inclusion in browser preload lists (eliminates even the first-visit window).
