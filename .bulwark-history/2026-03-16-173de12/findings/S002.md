# Finding: S002 - Crank Drain Amplification Loop

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

This strategy chains three confirmed findings -- unbounded vault top-up (H013), no circuit breaker (H019), and no alerting (H004) -- into a continuous SOL drain loop that empties the crank wallet without detection. The chain is: (1) an RPC MITM or stale cache causes the vault balance query to persistently return below `MIN_VAULT_BALANCE` (~2M lamports), (2) every crank cycle transfers 0.005 SOL to the vault (or attacker-controlled address if PDA_MANIFEST is poisoned), (3) with no circuit breaker, the loop continues indefinitely until the wallet is empty, (4) with no alerting, the drain is invisible until someone manually checks Railway logs or the wallet balance.

The combined severity is HIGH. Under normal conditions the crank cycle takes ~5 minutes on devnet (750 slots at ~0.4s each), yielding a drain rate of ~1.44 SOL/day. However, if the epoch boundary wait is shortened (by RPC reporting advanced slots) or the retry delay is triggered repeatedly (30s cycles instead of ~5min), the drain rate accelerates to ~14.4 SOL/day. On mainnet with 4500-slot epochs (~30 min), the baseline is lower but the per-cycle amount could be increased by manipulating the MIN_VAULT_BALANCE comparison.

## Hypothesis Investigated

RPC MITM triggers vault top-up without limit (H013). No circuit breaker (H019) ensures continuous drain. No alerting (H004) means no detection.

## Enabling Factors
- **Single RPC source:** `crank-runner.ts:222` -- vault balance from one `getBalance` call, no quorum
- **No cumulative guard:** No counter tracking how many top-ups have occurred
- **Infinite retry:** `crank-runner.ts:308-314` -- errors cause 30s sleep then full retry
- **Advisory-only warning:** `crank-runner.ts:213-217` -- console.log, not a halt condition

## Blocking Factors
- **Top-up amount is small:** 0.005 SOL per cycle limits drain speed under normal timing
- **HTTPS transport:** RPC MITM requires significant capability (DNS hijack, cert compromise)
- **Devnet impact limited:** Current devnet wallet has limited SOL; mainnet impact would be higher

## Severity Assessment
- **Severity:** HIGH
- **Likelihood:** MEDIUM (requires RPC compromise or manifest poisoning, but combined with H003 supply chain risk, likelihood increases)
- **Impact:** Complete crank wallet drain. At 0.005 SOL/cycle with ~5min cycles = ~1.44 SOL/day baseline. With error-loop acceleration (30s cycles) = ~14.4 SOL/day. If wallet holds 10 SOL, full drain in ~0.7-7 days.

### Combined Severity Justification
Individual findings are HIGH (H013), HIGH (H019), HIGH (H004). The chain maintains HIGH because:
1. The drain mechanism already exists in code -- no new capability needed beyond manipulated RPC responses
2. The attack is silent and persistent (no alerting, no circuit breaker)
3. The chain is self-sustaining once initiated (no attacker interaction needed after initial RPC manipulation)
4. On mainnet with a funded crank wallet, the financial impact scales directly

### Attack Scenario

```

  Target: Helius RPC endpoint used by crank (CLUSTER_URL env var)
  Method: DNS hijack, cache poisoning, or supply chain compromise of RPC provider module
  Prerequisite: Network-level access or npm supply chain entry (H003)
  Result: crank-runner.ts:222 getBalance() returns artificially low value

  Target: crank-runner.ts:225-241

Step 3: Loop continues with no cap
  Target: Main while loop at crank-runner.ts:197
  Condition: shutdownRequested === false (nobody knows to shut it down)
  Result: 0.005 SOL/cycle * ~288 cycles/day = ~1.44 SOL/day

Step 4: No detection
  Target: console.log at crank-runner.ts:215
  Observation: Low balance warning appears in Railway logs
  Detection: Nobody is monitoring -- no alerting pipeline exists
  Result: Drain continues until wallet is empty

Step 5 (variant): Error loop acceleration
  Target: crank-runner.ts:302-314
  If: advanceEpochWithVRF fails (RPC manipulation can cause this too)
  Result: 30s retry cycle instead of ~5min, drain rate 10x faster

Final Result: Crank wallet drained of all SOL.
  - Protocol stops: no more epoch transitions, VRF, or carnage
  - If PDA_MANIFEST poisoned: SOL sent to attacker directly
  - No recovery: crank must be manually funded and restarted
```

### Recommended Fix

```typescript
// crank-runner.ts -- Add spending limits and circuit breaker

// 1. Cumulative top-up tracking
let topUpCountThisCycle = 0;
const MAX_TOP_UPS_PER_HOUR = 5;
let topUpResetTime = Date.now();

// 2. Before top-up
if (Date.now() - topUpResetTime > 3600_000) {
  topUpCountThisCycle = 0;
  topUpResetTime = Date.now();
}
if (topUpCountThisCycle >= MAX_TOP_UPS_PER_HOUR) {
  console.error("[crank] CIRCUIT BREAKER: Max top-ups/hour exceeded. Halting.");
  shutdownRequested = true;
  break;
}

// 3. Consecutive error circuit breaker
let consecutiveErrors = 0;
const MAX_CONSECUTIVE_ERRORS = 5;

// In catch block:
consecutiveErrors++;
if (consecutiveErrors >= MAX_CONSECUTIVE_ERRORS) {
  console.error("[crank] CIRCUIT BREAKER: Too many consecutive errors. Halting.");
  shutdownRequested = true;
}

// 4. External alerting (webhook to Discord/Slack/PagerDuty)
async function sendAlert(message: string) {
  const webhookUrl = process.env.ALERT_WEBHOOK_URL;
  if (webhookUrl) {
    await fetch(webhookUrl, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ content: `[CRANK ALERT] ${message}` }),
    }).catch(() => {}); // Don't crash on alert failure
  }
}
```
