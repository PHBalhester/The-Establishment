# Finding: H110 - No Timelock on Admin Actions

## Status: CONFIRMED
## Confidence Score: 9

## Rationale
Searching all 7 programs for "timelock", "time_lock", "delay", "governance", or "multisig" in instruction logic yields no timelock mechanisms. Admin actions are executed immediately upon signing. The AMM admin can be set to a multisig (`amm/src/state/admin.rs:14` -- "Can be a multisig address"), but this is optional and there is no enforced delay between proposal and execution. Key admin actions that execute immediately include: AMM admin initialization, whitelist authority operations, epoch state initialization, staking pool setup, and curve activation. While the AMM admin key is eventually burned (making those admin actions permanently unavailable), the upgrade authority for all 7 programs remains active and has no timelock. A compromised upgrade authority could deploy malicious program code instantly.

No `delay_slots`, `timelock_expiry`, or governance voting mechanism exists in any program.

## Severity Estimate
MEDIUM -- Standard DeFi practice includes timelocks on admin actions (24-48h delay) so users can exit before malicious changes take effect. The upgrade authority is the most critical vector. Squads multisig is mentioned in conversion-vault comments but not enforced on-chain.
