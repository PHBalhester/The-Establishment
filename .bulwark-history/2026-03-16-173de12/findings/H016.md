# Finding: H016 - Transfer Hook Init Front-Running (Cross-Boundary)

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The transfer hook program's `initialize_authority` instruction uses Anchor's `init` constraint with a bare `Signer<'info>` -- a first-caller-wins pattern where ANY wallet can become the whitelist authority. The off-chain deployment pipeline (`deploy-all.sh`) creates a multi-minute gap between program deployment (Phase 2) and initialization (Phase 3), during which an attacker monitoring on-chain program deployments can call `initializeAuthority` and capture the whitelist authority. This would give the attacker control over all token transfers (via the whitelist), effectively bricking the protocol or enabling a ransom attack.

This is the off-chain dimension of the SOS on-chain finding S005 (CRITICAL, NOT FIXED). The off-chain deployment pipeline provides zero mitigation for the on-chain vulnerability and in fact exacerbates it by introducing an observable time window between deploy and init.

## Hypothesis Investigated

The transfer hook's `initialize_authority` instruction uses a first-caller-wins pattern. An attacker who front-runs deployment could capture the authority, holding all token transfers hostage.

## Investigation Path

### Step 1: On-Chain Constraint Verification (for off-chain context)
**Location:** `programs/transfer-hook/src/instructions/initialize_authority.rs:27-46`
**Finding:** The `InitializeAuthority` accounts struct uses `#[account(init, ...)]` on the WhitelistAuthority PDA with a bare `Signer<'info>` -- no `has_one`, no address constraint, no ProgramData upgrade-authority check. Whoever calls this first becomes the authority.
**Evidence:**
```rust
#[account(mut)]
pub signer: Signer<'info>,

#[account(
    init,
    payer = signer,
    space = 8 + WhitelistAuthority::INIT_SPACE,
    seeds = [WhitelistAuthority::SEED],
    bump
)]
pub whitelist_authority: Account<'info, WhitelistAuthority>,
```

### Step 2: Deployment Pipeline Time Window Analysis
**Location:** `scripts/deploy/deploy-all.sh:155-168`
**Finding:** The pipeline runs Phase 2 (deploy all 7 programs) then Phase 3 (initialize). These are separate shell commands with no atomicity. Phase 2 deploys programs sequentially (`solana program deploy` x7), taking several minutes on devnet. Phase 3 runs `npx tsx scripts/deploy/initialize.ts`, which must compile/load TypeScript before executing. The gap between the transfer_hook program being deployed (2nd program in the array at `deploy.sh:138`) and `initializeAuthority` being called (Step 2 of `initialize.ts:311-336`) is substantial -- on the order of minutes.
**Evidence:**
```bash
# deploy-all.sh:155-168
# Phase 2: Deploy -- deploys all 7 programs sequentially
bash scripts/deploy/deploy.sh "$CLUSTER_URL"

# Phase 3: Initialize -- runs TypeScript init (compiles + executes)
npx tsx scripts/deploy/initialize.ts
```

### Step 3: Initialization Idempotency Check
**Location:** `scripts/deploy/initialize.ts:321-336`
**Finding:** The initialization script checks if the WhitelistAuthority account already exists before calling `initializeAuthority`. If an attacker has already initialized it, the script sees the account exists and SKIPs the step -- silently proceeding as if initialization succeeded. It does NOT verify that the authority stored in the account matches the deployer's pubkey.
**Evidence:**
```typescript
if (await accountExists(connection, whitelistAuthority)) {
    log.step(stepNum, TOTAL_STEPS, "Initialize WhitelistAuthority", "SKIPPED");
    skipped++;
} else {
    const sig = await programs.transferHook.methods
      .initializeAuthority()
      .accountsStrict({
        signer: authority.publicKey,
        whitelistAuthority,
        systemProgram: SystemProgram.programId,
      })
      .signers([authority])
      .rpc();
    log.step(stepNum, TOTAL_STEPS, "Initialize WhitelistAuthority", "done", sig);
    completed++;
}
```

### Step 4: Verification Script Gap
**Location:** `scripts/deploy/verify.ts` (Phase 4 of deploy-all.sh)
**Finding:** The verify script (Phase 4) runs AFTER initialization. It checks that accounts exist but the question is whether it validates the authority field. Even if it does, by Phase 4 the damage is done -- the attacker's authority is already set, and the `init` constraint prevents re-initialization.

### Step 5: No Jito Bundling or Atomic Init
**Location:** `scripts/deploy/deploy.sh:171-176`, `scripts/deploy/deploy-all.sh:168`
**Finding:** Program deployment uses `solana program deploy` (standard CLI), not Jito bundles. Initialization uses `npx tsx` (TypeScript via Node.js), not bundled with the deploy transaction. There is no mechanism to atomically deploy + initialize in a single transaction or Jito bundle.

## If CONFIRMED

### Severity Assessment
- **Severity:** CRITICAL
- **Likelihood:** MEDIUM (requires attacker to monitor deployments; only exploitable during deployment events, not ongoing)
- **Impact:** Protocol-killing -- attacker controls all token transfers. Can ransom the authority or permanently brick the protocol by refusing to whitelist required accounts (pools, vaults, staking). The `burnAuthority` instruction could make the damage permanent.

### Attack Scenario

```
Step 1: Reconnaissance
  Target: GitHub repo / on-chain monitoring
  Action: Attacker reads keypairs/transfer-hook-keypair.json from the public repo
          to know the program ID. Derives WhitelistAuthority PDA:
          findProgramAddressSync([Buffer.from("authority")], hookProgramId)
  Prerequisite: Open-source repo with committed keypair files
  Result: Attacker knows exact program ID and PDA address

Step 2: Monitor for deployment
  Target: Solana RPC / program account subscription
  Action: Watch for the transfer_hook program account to become executable
  Prerequisite: deploy.sh runs solana program deploy for transfer_hook
  Result: Attacker detects program deployment within ~1 slot (400ms)

Step 3: Front-run initialization
  Target: transfer_hook::initializeAuthority instruction
  Action: Submit initializeAuthority TX with attacker's wallet as signer,
          using high priority fee (e.g., 1M microLamports/CU)
  Prerequisite: Program is deployed but initializeAuthority not yet called
  Result: WhitelistAuthority PDA created with attacker as authority

Step 4: Legitimate deployer silently compromised
  Target: scripts/deploy/initialize.ts Step 2
  Action: initialize.ts checks accountExists(whitelistAuthority) -> true -> SKIPPED
  Prerequisite: Attacker's TX landed first
  Result: Deployer proceeds, unaware that authority is compromised

Step 5: Ransom or brick
  Target: Protocol operations
  Action: Attacker refuses to add whitelist entries, or adds only their own addresses.
          Alternatively, calls burnAuthority to permanently lock the whitelist in its
          current (empty) state.
  Result: No token transfers can occur. Protocol is bricked.

Final Result: Complete protocol denial-of-service. If attacker calls burnAuthority
before any whitelist entries exist, ALL token transfers are permanently blocked.
Recovery requires redeploying the transfer_hook program with new program ID,
which breaks all mints that reference the old hook program ID.
```

### Recommended Fix

**Off-chain fix (immediate -- reduces window):**
```typescript
// Before (vulnerable) -- initialize.ts:321-336
if (await accountExists(connection, whitelistAuthority)) {
    log.step(stepNum, TOTAL_STEPS, "Initialize WhitelistAuthority", "SKIPPED");
    skipped++;
} else {
    // ... call initializeAuthority
}

// After (fixed) -- verify authority ownership even when account exists
const whitelistAuthorityData = await programs.transferHook.account.whitelistAuthority.fetchNullable(
    whitelistAuthority
);
if (whitelistAuthorityData) {
    // CRITICAL: Verify the authority is US, not an attacker
    if (whitelistAuthorityData.authority === null) {
        log.error("WhitelistAuthority already burned by unknown party! ABORT.");
        process.exit(1);
    }
    if (!whitelistAuthorityData.authority.equals(authority.publicKey)) {
        log.error(`WhitelistAuthority captured by attacker: ${whitelistAuthorityData.authority.toBase58()}`);
        log.error("Protocol deployment is COMPROMISED. Redeploy transfer_hook with new program ID.");
        process.exit(1);
    }
    log.step(stepNum, TOTAL_STEPS, "Initialize WhitelistAuthority", "SKIPPED (verified ours)");
    skipped++;
} else {
    // ... call initializeAuthority
}
```

**On-chain fix (recommended -- eliminates the vulnerability):**
Gate `initialize_authority` with ProgramData upgrade-authority check, matching the AMM's `initialize_admin` pattern. This ensures only the program's upgrade authority can initialize the whitelist authority.

**Operational fix (belt-and-suspenders):**
Use Jito bundles to atomically deploy the transfer_hook program and call `initializeAuthority` in the same block, eliminating the time window entirely.

## Incidental Discoveries

- **Silent skip without ownership verification across ALL init steps**: The `accountExists()` pattern used in `initialize.ts` is repeated for every initialization step (ExtraAccountMetaLists at line 350, AdminConfig, EpochState, StakePool, etc.). None of these checks verify that the existing account was initialized by the legitimate deployer. If ANY of these programs have a first-caller-wins init pattern, the same silent-skip vulnerability applies. -> Suggested hypothesis: H016-B: Audit all `accountExists()` checks in `initialize.ts` for ownership verification gaps across all 7 programs.

- **Committed keypair files enable pre-deployment reconnaissance**: The keypair files in `keypairs/` directory are referenced in `deploy.sh` and appear to be committed to git (or at least accessible in the working directory). If the repo is public (or becomes public), program IDs are pre-computable, meaning attackers can prepare front-running transactions before deployment even begins. -> Suggested hypothesis: Verify whether `keypairs/` is in `.gitignore` and whether program IDs are predictable from public information.
