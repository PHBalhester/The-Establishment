# Finding: H027 - Iframe Sandbox Weakness

## Status: NOT VULNERABLE
## Confidence Score: 7

## Executive Summary

Both `DocsModal.tsx` and `DocsStation.tsx` use iframes with `sandbox="allow-scripts allow-same-origin allow-popups"` to embed documentation. The hypothesis is that the combination of `allow-scripts` and `allow-same-origin` on a same-origin source creates a sandbox escape. However, investigation reveals that neither iframe actually loads same-origin content:

- **DocsModal** (`app/components/launch/DocsModal.tsx:107`) loads from `https://dr-fraudsworth-production.up.railway.app/docs` -- this is the production Railway URL, which IS the same origin as the main app in production. This is the concerning case.
- **DocsStation** (`app/components/station/DocsStation.tsx:66`) loads from `NEXT_PUBLIC_DOCS_URL` (defaulting to `http://localhost:3001` in dev, or `https://docs-drfraudsworth.up.railway.app` in production). The production docs URL is on a different Railway subdomain, making it cross-origin. This is safe.

For DocsModal, the iframe IS same-origin in production, meaning `allow-scripts + allow-same-origin` effectively negates the sandbox -- the iframe can access the parent page's cookies, localStorage, and DOM via `window.parent`. However, the content loaded (`/docs`) is self-hosted documentation (Nextra), not user-controlled. The risk requires compromise of the docs content itself (XSS in Nextra, supply chain attack on docs dependencies) to exploit.

## Hypothesis Investigated

DocsModal iframe has `allow-scripts + allow-same-origin` with same-origin source, potentially allowing sandbox escape.

## Investigation Path

### Step 1: DocsModal iframe source analysis
**Location:** `app/components/launch/DocsModal.tsx:16,103-108`
**Finding:** Hardcoded URL `https://dr-fraudsworth-production.up.railway.app/docs` which is the same origin as the main app.
**Evidence:**
```typescript
const DOCS_URL = 'https://dr-fraudsworth-production.up.railway.app/docs';
// ...
<iframe
  src={DOCS_URL}
  sandbox="allow-scripts allow-same-origin allow-popups"
/>
```

### Step 2: DocsStation iframe source analysis
**Location:** `app/components/station/DocsStation.tsx:22,61-68`
**Finding:** Uses `NEXT_PUBLIC_DOCS_URL` env var, defaulting to `http://localhost:3001` (dev). In production, the CSP `frame-src` includes `https://docs-drfraudsworth.up.railway.app` which is a different subdomain (cross-origin).
**Evidence:**
```typescript
const DOCS_URL = process.env.NEXT_PUBLIC_DOCS_URL || 'http://localhost:3001';
// ...
<iframe
  src={DOCS_URL}
  sandbox="allow-scripts allow-same-origin allow-popups"
/>
```

### Step 3: CSP frame-src verification
**Location:** `app/next.config.ts:17-18`
**Finding:** CSP allows frames from `http://localhost:3001` and `https://docs-drfraudsworth.up.railway.app` -- the docs site is deployed as a separate Railway service.

### Step 4: Sandbox escape analysis
**Finding:** Per the HTML spec, `allow-scripts + allow-same-origin` on a same-origin iframe means the iframe can use `window.parent` to access the parent document, read/write cookies, access localStorage, and effectively execute arbitrary JavaScript in the parent context. For DocsModal (same origin), this means an XSS in the `/docs` route would be equivalent to XSS in the main app. For DocsStation (cross-origin), the sandbox provides meaningful isolation.

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Compromise docs content (XSS in Nextra, supply chain) | Unlikely but possible | `/docs` is self-hosted Nextra |
| 2 | Injected script in iframe accesses parent window | Yes (DocsModal only) | Same-origin + allow-scripts + allow-same-origin |
| 3 | Access parent cookies, localStorage, DOM | Yes | Sandbox is effectively negated |
| 4 | Steal wallet session or inject malicious transactions | Possible | Depends on what's accessible in parent |

## Invariant Analysis

### Target Invariant
Embedded iframe content cannot access or modify the parent application's state, cookies, or DOM.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `DocsModal.tsx:107` sandbox attr | Iframe sandbox | **No** -- `allow-scripts + allow-same-origin` on same-origin negates sandbox |
| `DocsStation.tsx:66` sandbox attr | Iframe sandbox | **Yes** -- cross-origin source means sandbox provides real isolation |
| `next.config.ts:16-17` X-Frame-Options: DENY | Prevents app from being framed | Yes (but this is the inverse direction) |
| `next.config.ts:17` frame-src CSP | Restricts what can be framed | Partial -- allows the docs URLs |

## Blocking Factors
- Docs content is self-hosted, not user-controlled: exploitation requires compromising the docs content first
- The docs route (`/docs`) serves static documentation (Nextra) with limited attack surface
- X-Frame-Options: DENY prevents the MAIN app from being framed, but does not prevent the app from framing other content

## Enabling Factors
- `allow-scripts + allow-same-origin` on same-origin iframe: `DocsModal.tsx:107` -- sandbox is effectively negated
- If docs content is ever compromised (Nextra vulnerability, dependency attack), attacker gets full parent access

## Protection Analysis
| Protection | Location | How It Works |
|------------|----------|--------------|
| Cross-origin docs (DocsStation) | `DocsStation.tsx:22` | Docs URL is a separate Railway service, real sandbox isolation |
| Self-hosted docs content | Nextra deployment | Docs are static, limited injection surface |
| CSP frame-src | `next.config.ts:17-18` | Restricts which URLs can be framed |

## Devil's Advocate Analysis (MANDATORY)
| Challenge | Argument | Refutation | Evidence | Confidence |
|-----------|----------|------------|----------|------------|
| Crafted input | Attacker finds XSS in Nextra docs (e.g., search query reflected unsafely) | Nextra is a well-maintained framework with SSG output; most pages are pure static HTML with no user input reflection | Nextra architecture (static generation) | MEDIUM |
| Multiple accounts | N/A -- iframe sandbox is not account-dependent | N/A | N/A | HIGH |
| External API MITM | Attacker MITM's the docs URL and injects scripts into the iframe response | For DocsModal, HTTPS prevents MITM. For DocsStation dev mode, localhost is HTTP but only in development | `DocsModal.tsx:16` uses HTTPS | HIGH |
| Race condition | N/A -- iframe loading is not race-sensitive | N/A | N/A | HIGH |
| Frontend chain | If attacker has XSS in docs, `allow-scripts + allow-same-origin` lets them pivot to the parent app (DocsModal case) | This IS the vulnerability. The mitigation is that getting XSS in docs requires a separate vulnerability | Docs are static Nextra | MEDIUM |

**Overall counterargument confidence:** MEDIUM -- The DocsModal case is genuinely concerning from a defense-in-depth perspective. The sandbox attribute provides zero additional security when both `allow-scripts` and `allow-same-origin` are set on same-origin content. However, exploitation requires a prerequisite vulnerability in the docs content.

## Related Findings
- May combine with H026 (Missing HSTS) -- if MITM is achieved and docs are loaded over downgraded HTTP, attacker can inject scripts into the iframe which then escapes the sandbox.

## Incidental Discoveries
- **DocsModal hardcoded production URL**: `DocsModal.tsx:16` hardcodes `https://dr-fraudsworth-production.up.railway.app/docs` rather than using an env var. This means the component breaks in development/staging environments and ties the component to a specific deployment URL. Suggested hypothesis: investigate whether the hardcoded URL creates issues in non-production environments or could be redirected.
- **DocsStation localhost fallback**: `DocsStation.tsx:22` falls back to `http://localhost:3001` when `NEXT_PUBLIC_DOCS_URL` is unset. In production, if the env var is missing, the iframe would attempt to load from localhost -- which would fail silently but waste a connection attempt. This is not a security issue but is a reliability concern.
