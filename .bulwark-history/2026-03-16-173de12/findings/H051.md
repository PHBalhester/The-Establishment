# Finding: H051 - CustomEvent Balance Sync as RPC DoS

## Status: NOT VULNERABLE
## Confidence Score: 7

## Executive Summary

The hypothesis proposes that rapid `token-balances-refresh` CustomEvent dispatches could trigger massive RPC bursts. After investigation, the attack surface is significantly limited by the design of `useTokenBalances.ts`: (1) the `isDispatchingRef` guard prevents self-triggered fetches, (2) `fetchBalances` is a standard `useCallback` with no debounce but also no amplification -- each call makes exactly 2 RPC requests (`getBalance` + `getParsedTokenAccountsByOwner`), and (3) the event is only dispatched on `window`, limiting it to browser-local scope.

However, the lack of debouncing means a malicious browser extension or XSS could dispatch thousands of events per second, each triggering 2 RPC calls per hook instance. With N instances on the page, this creates 2*N RPC calls per dispatch. The practical impact is limited because Helius rate-limits at the API key level, and the browser's network stack has concurrent connection limits (~6 per origin). The attack would cause 429 errors from Helius but not a true DoS -- just degraded balance display for the single affected user session.

## Hypothesis Investigated

Rapid `token-balances-refresh` events trigger massive RPC bursts.

## Investigation Path

### Step 1: Event dispatch mechanism
**Location:** `app/hooks/useTokenBalances.ts:166-174`
**Finding:** `refresh()` dispatches `CustomEvent("token-balances-refresh")` on `window`. No throttling.
**Evidence:**
```typescript
const refresh = useCallback(() => {
  fetchBalances();
  isDispatchingRef.current = true;
  window.dispatchEvent(new CustomEvent(BALANCE_REFRESH_EVENT));
  isDispatchingRef.current = false;
}, [fetchBalances]);
```

### Step 2: Event listener (cross-instance)
**Location:** `app/hooks/useTokenBalances.ts:151-162`
**Finding:** Other instances of the hook listen for the event and call `fetchBalances()` directly. The `isDispatchingRef` guard prevents self-triggered fetches, but externally dispatched events always trigger a fetch.
**Evidence:**
```typescript
const handleRefreshEvent = () => {
  if (isDispatchingRef.current) return; // Skip self-dispatched
  fetchBalances();
};
window.addEventListener(BALANCE_REFRESH_EVENT, handleRefreshEvent);
```

### Step 3: RPC call count per fetch
**Location:** `app/hooks/useTokenBalances.ts:85-90`
**Finding:** Each `fetchBalances()` makes exactly 2 RPC calls via `Promise.all`.
**Evidence:**
```typescript
const [solBalance, tokenAccounts] = await Promise.all([
  connection.getBalance(key),
  connection.getParsedTokenAccountsByOwner(key, {
    programId: TOKEN_2022_PROGRAM_ID,
  }),
]);
```

### Step 4: External dispatch capability
**Location:** Browser global `window` object
**Finding:** Any script running in the same browsing context (XSS, browser extension) can dispatch the event.
**Evidence:** `window.dispatchEvent(new CustomEvent("token-balances-refresh"))` -- standard DOM API, no authentication.

### Step 5: Impact assessment
**Finding:** The attack scope is limited to a single user's browser session. Helius rate-limits per API key, and browser connection limits cap concurrent HTTP requests at ~6. The user would see balance fetch errors, but the Helius endpoint would not be overwhelmed globally.

## Invariant Analysis

### Target Invariant
Balance refresh events should not cause unbounded RPC request volume.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `useTokenBalances.ts:153-154` | `isDispatchingRef` prevents self-triggered fetch | Yes (self only) |
| N/A | Debounce/throttle on event handler | **Not implemented** |
| N/A | Rate limit on `fetchBalances` | **Not implemented** |
| Browser | ~6 concurrent connections per origin | Yes (implicit) |

## Blocking Factors
- `isDispatchingRef` prevents cascade: `useTokenBalances.ts:153-154`
- Browser concurrent connection limit (~6 per origin) caps effective RPC throughput
- Attack is per-user session only, not global
- `fetchBalances` uses `Promise.all` for 2 calls, not N separate calls

## Enabling Factors
- No debounce on event handler: `useTokenBalances.ts:152-162`
- `window` global is accessible to any in-page script
- CSP `unsafe-inline` allows some XSS injection vectors

## If NOT VULNERABLE

### Protection Analysis
| Protection | Location | How It Works |
|------------|----------|--------------|
| Self-dispatch guard | `useTokenBalances.ts:153-154` | `isDispatchingRef` prevents the dispatching instance from re-fetching |
| Browser connection limit | Browser engine | ~6 concurrent HTTP connections per origin |
| Per-user scope | DOM API | `window` events only affect the current browser tab |
| Helius rate limit | External service | API key rate limit prevents unbounded requests |

### Devil's Advocate Analysis (MANDATORY)
| Challenge | Argument | Refutation | Evidence | Confidence |
|-----------|----------|------------|----------|------------|
| Crafted input | Attacker dispatches 10,000 events/sec via `setInterval` | Browser connection limit (~6) caps actual RPC calls; most `fetchBalances` calls queue behind pending ones | Browser HTTP/1.1 connection pooling | MEDIUM |
| Multiple accounts | Attacker opens 100 tabs to amplify | Each tab is independent; Helius rate limit is per-API-key, so 100 tabs share the same limit | Helius API key level rate limiting | HIGH |
| External API MITM | Attacker intercepts RPC to return slow responses, keeping connections busy | Would slow down balance display but not cause fund loss; just UI degradation | `useTokenBalances.ts:107-111` error handling | HIGH |
| Race condition | Rapid dispatches cause concurrent fetchBalances calls that overlap | No state corruption risk -- `setBalances` is React setState (last-write-wins), and each fetch returns independent data | `useTokenBalances.ts:101-106` | HIGH |
| Frontend chain | XSS + rapid event dispatch = RPC exhaustion for the victim user | CSP `unsafe-inline` enables XSS; combined with event spam = victim's balance display broken. But no fund loss. | `next.config.ts:9` | MEDIUM |

**Overall counterargument confidence:** MEDIUM. The attack is possible via XSS but the impact is limited to UI degradation for a single user. No fund loss, no protocol-wide impact. The lack of debounce is a code quality concern rather than a security vulnerability.

## Incidental Discoveries
- `useTokenBalances.ts:130` has a 30-second polling interval (`setInterval(fetchBalances, REFRESH_INTERVAL_MS)`). This interval is NOT cleared when `fetchBalances` is called manually via the event handler. If 100 manual dispatches overlap with the interval timer, they stack. Suggested hypothesis: "Overlapping manual refresh + interval timer in useTokenBalances creates redundant RPC calls during normal operation."
