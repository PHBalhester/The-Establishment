# Finding: H033 - Candle Close Price Ordering from Out-of-Order Webhooks

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The candle aggregator unconditionally sets `close = new price` on every upsert, with no ordering guarantee. When Helius delivers webhook events out of order (which is documented behavior for concurrent transaction processing), the "close" price for a candle may reflect an earlier transaction rather than the chronologically last one in the bucket. The SQL upsert at `candle-aggregator.ts:123` sets `close: sql'${update.price}'` without comparing timestamps. Two concurrent webhook POSTs for transactions at T=14:37:50 and T=14:37:55 in the same 1m candle bucket could arrive in reverse order, making the close price reflect the 14:37:50 trade instead of the 14:37:55 trade.

This is a data integrity issue affecting chart display accuracy. It does not enable fund theft (on-chain enforcement prevents that), but could mislead users about price trends, particularly on low-timeframe candles (1m, 5m) where fewer trades per bucket amplify the effect.

## Hypothesis Investigated

Concurrent webhooks may deliver events out of order, corrupting close price.

## Investigation Path

### Step 1: Examine the upsert close logic
**Location:** `app/db/candle-aggregator.ts:118-127`
**Finding:** The `onConflictDoUpdate` sets `close` unconditionally to the new price. There is no timestamp comparison.
**Evidence:**
```typescript
.onConflictDoUpdate({
  target: [candles.pool, candles.resolution, candles.openTime],
  set: {
    high: sql`GREATEST(${candles.high}, ${update.price})`,
    low: sql`LEAST(${candles.low}, ${update.price})`,
    close: sql`${update.price}`,  // Always overwrites -- no ordering
    volume: sql`${candles.volume} + ${update.volume}`,
    tradeCount: sql`${candles.tradeCount} + 1`,
  },
})
```

### Step 2: Verify timestamp source
**Location:** `app/app/api/webhooks/helius/route.ts:174-177`
**Finding:** The timestamp is derived from `tx.blockTime` (Solana on-chain block time in unix seconds). This is monotonically increasing per slot but does NOT guarantee ordering across concurrent webhook deliveries.
**Evidence:**
```typescript
const timestamp = blockTime
  ? new Date(blockTime * 1000)
  : new Date();
```

### Step 3: Verify concurrent webhook processing
**Location:** `app/app/api/webhooks/helius/route.ts:160-211`
**Finding:** The webhook handler processes transactions sequentially within a single POST body (for-loop at line 160). However, Helius may send multiple webhook POSTs concurrently, and Next.js processes these requests in parallel. Two POSTs arriving simultaneously would race through `upsertCandlesForSwap`.

### Step 4: Verify high and low are safe
**Location:** `app/db/candle-aggregator.ts:121-122`
**Finding:** `high` and `low` use SQL `GREATEST` and `LEAST` which are commutative (order-independent). These are correctly computed regardless of webhook delivery order. Only `close` is affected.

### Step 5: Check if open is safe
**Location:** `app/db/candle-aggregator.ts:97` (comment), `candle-aggregator.ts:118-127` (upsert)
**Finding:** The `open` column is NOT included in the `set` object of `onConflictDoUpdate`, so it's preserved from the first INSERT. However, if two concurrent inserts race, the one that wins the INSERT sets the open -- this could be the chronologically later trade. This is a secondary ordering issue.

## Invariant Analysis

### Target Invariant
Candle close price should reflect the chronologically last trade in the bucket.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `candle-aggregator.ts:123` | Sets close to latest received price | No -- "latest received" != "chronologically latest" |

## Blocking Factors
- Within a single webhook POST body, transactions are processed sequentially (`for...of` loop at line 160)
- High/low are order-independent (GREATEST/LEAST)

## Enabling Factors
- `candle-aggregator.ts:123`: close price has no timestamp comparison -- last-writer-wins
- Helius webhooks provide no delivery ordering guarantee across concurrent POSTs
- Next.js processes concurrent HTTP requests in parallel
- No database-level `last_trade_timestamp` column exists for comparison

## If CONFIRMED

### Severity Assessment
- **Severity:** LOW
- **Likelihood:** MEDIUM (concurrent swaps in the same bucket are common during active trading)
- **Impact:** Chart data integrity -- users see incorrect close prices on candles. No fund loss.

### Attack Scenario
```
Step 1: Two swaps occur at T=14:37:50 (price=0.0001) and T=14:37:55 (price=0.00012) in the same 1m candle
Step 2: Helius sends webhook POST A (T=14:37:55 swap) and POST B (T=14:37:50 swap) concurrently
Step 3: POST A arrives first, upserts close=0.00012
Step 4: POST B arrives second, overwrites close=0.0001
Step 5: Candle displays close=0.0001 instead of the correct 0.00012
Step 6: Users see a bearish candle when the actual last trade was bullish
```

### Recommended Fix
```sql
-- Before (vulnerable)
close = ${update.price}

-- After (fixed): Add a last_trade_timestamp column and use conditional update
close = CASE
  WHEN ${candles.lastTradeTimestamp} IS NULL
    OR ${update.timestamp} >= ${candles.lastTradeTimestamp}
  THEN ${update.price}
  ELSE ${candles.close}
END,
last_trade_timestamp = GREATEST(
  COALESCE(${candles.lastTradeTimestamp}, ${update.timestamp}),
  ${update.timestamp}
)
```

Alternatively, use the Solana slot number (monotonically increasing) as the ordering key:
```typescript
// In CandleUpdate interface, add:
slot: number;

// In upsert:
close: sql`CASE WHEN ${candles.lastSlot} IS NULL OR ${update.slot} >= ${candles.lastSlot} THEN ${update.price} ELSE ${candles.close} END`,
lastSlot: sql`GREATEST(COALESCE(${candles.lastSlot}, 0), ${update.slot})`,
```

## Incidental Discoveries

Suggested hypothesis: investigate open price race condition on concurrent new-bucket inserts.
