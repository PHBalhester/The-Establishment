# Finding: H046 - RPC Response Not Validated for Financial Data

## Status: CONFIRMED
## Confidence Score: 7

## Executive Summary

Pool reserve data from RPC responses is used to compute swap quotes, price displays, and `minimumOutput` slippage parameters without validation. In `usePoolPrices.ts`, the Anchor decoder converts on-chain `u64` reserve values using `toNumber()`, which silently loses precision for values exceeding `Number.MAX_SAFE_INTEGER` (2^53). The quote engine (`quote-engine.ts`) then uses JavaScript `number` arithmetic on these reserves, producing imprecise results. This is already flagged in ARCHITECTURE.md as assumption A-OC3: "Pool reserves will stay within Number.MAX_SAFE_INTEGER -- ALREADY VIOLATED at current pool sizes."

The specific concern: `reserveA` and `reserveB` values from RPC are passed through `toNum()` which calls `val.toNumber()` on BN objects. For reserves of 290M tokens at 6 decimals (290,000,000,000,000 = 2.9e14), this is within safe integer range. However, at scale or with different pool configurations, this could overflow. More critically, there is NO validation that the RPC response is well-formed (non-zero reserves, account owned by correct program, etc.) before it drives financial computations.

## Hypothesis Investigated

Pool reserve data from RPC used directly without validation.

## Investigation Path

### Step 1: Trace RPC data ingestion in usePoolPrices
**Location:** `app/hooks/usePoolPrices.ts:142-160`
**Finding:** Anchor's coder decodes the raw account data, then `toNum()` converts reserves to JavaScript `number`. No validation of the decoded values.
**Evidence:**
```typescript
const decoded = program.coder.accounts.decode("poolState", accountInfo.data);
setPools((prev) => ({
  ...prev,
  [entry.label]: {
    // ...
    reserveA: toNum(decoded.reserveA),
    reserveB: toNum(decoded.reserveB),
    // ...
  },
}));
```

### Step 2: Examine toNum helper
**Location:** `app/hooks/usePoolPrices.ts:93-96`
**Finding:** Calls `.toNumber()` on BN objects, which silently truncates values above 2^53.
**Evidence:**
```typescript
const toNum = useCallback(
  (val: number | { toNumber(): number }): number =>
    typeof val === "number" ? val : val.toNumber(),
  []
);
```

### Step 3: Check how reserves flow to quote engine
**Location:** `app/lib/swap/quote-engine.ts:37-62`
**Finding:** All quote functions accept `number` parameters. `calculateSwapOutput` computes `floor(reserveOut * effectiveInput / (reserveIn + effectiveInput))` using JavaScript `number` multiplication, which overflows for large values.
**Evidence:**
```typescript
export function calculateSwapOutput(
  reserveIn: number,
  reserveOut: number,
  effectiveInput: number,
): number {
  const denominator = reserveIn + effectiveInput;
  if (denominator === 0) return 0;
  return Math.floor((reserveOut * effectiveInput) / denominator);
}
```

### Step 4: Assess current reserve magnitudes
**Finding:** Current pools have ~290M tokens at 6 decimals = 290,000,000,000,000 (2.9e14). A swap of 2.5 SOL (2.5e9 lamports) produces intermediate: `2.9e14 * 2.5e9 = 7.25e23`, which exceeds `Number.MAX_SAFE_INTEGER` (9.007e15) by 8 orders of magnitude. The intermediate multiplication `reserveOut * effectiveInput` is already imprecise.

### Step 5: Check for validation of RPC response integrity
**Finding:** No checks for:
- Zero reserves (would cause division by zero -- handled by `if (denominator === 0) return 0`)
- Negative reserves (impossible on-chain but possible if deserialization is corrupted)
- Account owner verification (Anchor decoder checks discriminator but not program owner)
- Staleness detection (no slot number comparison)

### Step 6: Contrast with bonding curve math
**Location:** `app/lib/curve/curve-math.ts` (referenced in ARCHITECTURE.md)
**Finding:** The bonding curve math correctly uses `BigInt` for all arithmetic. This inconsistency means the BC forms compute accurate `minimumOutput` while the AMM forms compute imprecise values.

### Attack Scenario
```
Step 1: User swaps 2.5 SOL for CRIME on AMM pool
  Pool reserves: 2.5e9 WSOL, 2.9e14 CRIME tokens

Step 2: Quote engine computes: floor(2.9e14 * effective_input / denominator)
  Intermediate: 2.9e14 * ~2.47e9 = ~7.16e23 (exceeds MAX_SAFE_INTEGER)
  Result: outputTokens is off by some amount due to precision loss

Step 3: minimumOutput = Math.floor(outputTokens * 9500 / 10000) -- further imprecision
  Result: minimumOutput is lower than what on-chain would optimally produce

Step 4: MEV bot sandwiches: extracts the gap between off-chain minimum and on-chain optimal
  Result: User loses small amount per trade (could be a few lamports to a few cents)

Final Result: Systematic but small value extraction across many trades.
Precision loss accumulates over protocol lifetime.
```

### Recommended Fix

Convert `quote-engine.ts` to use `BigInt` arithmetic (matching `curve-math.ts`):

```typescript
// Before (imprecise)
export function calculateSwapOutput(
  reserveIn: number, reserveOut: number, effectiveInput: number
): number {
  return Math.floor((reserveOut * effectiveInput) / (reserveIn + effectiveInput));
}

// After (precise)
export function calculateSwapOutput(
  reserveIn: bigint, reserveOut: bigint, effectiveInput: bigint
): bigint {
  const denominator = reserveIn + effectiveInput;
  if (denominator === 0n) return 0n;
  return (reserveOut * effectiveInput) / denominator; // BigInt division truncates
}
```

And update `usePoolPrices.ts` to use `BigInt` for reserves:

```typescript
// Before
reserveA: toNum(decoded.reserveA),

// After
reserveA: BigInt(decoded.reserveA.toString()),
```

This is a larger refactor (all consumers of reserve data need to handle BigInt), but it eliminates the precision loss entirely.

## Incidental Discoveries
- `usePoolPrices.ts:217-254` (WebSocket callback) has the same `toNum()` precision issue for real-time updates. Every WebSocket-triggered pool update also loses precision. The fix must cover both the batch fetch path AND the WebSocket callback path.
- The `toNum` helper at line 93 has a subtle issue: if Anchor returns a plain `number` for small u64 values (which it sometimes does depending on version), the `typeof val === "number"` branch passes through without any validation. A corrupted or manipulated RPC response could inject a non-finite number (NaN, Infinity) that would propagate through the entire quote pipeline. Adding `isFinite(val)` checks would harden this path.
