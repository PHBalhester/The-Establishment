# Finding: H068 - BuyForm Cap Check Validation Race

## Status: NOT VULNERABLE
## Confidence Score: 8

## Rationale
The BuyForm validation at `app/components/launch/BuyForm.tsx:122-144` runs sequentially within a single `setTimeout` callback. The SOL balance check (line 123) sets `validationError` but does NOT early-return -- it falls through to the cap check (line 140), which can overwrite the error. However, this is not a race condition: both checks run synchronously in the same tick. The `handleSubmit` at line 157-158 gates on `if (validationError) return;`, so any validation error (balance OR cap) blocks submission. The on-chain bonding curve program also enforces the per-wallet cap independently, making this a defense-in-depth scenario.

## Code Evidence
```typescript
// line 157-158: Submit is gated on ANY validation error
if (!publicKey || !connected || !tokensOut || tokensOut === 0n || solLamports === 0n) return;

```

The cap check at line 140 runs after the balance check in the same synchronous callback -- no race is possible. On-chain enforcement is the final backstop regardless.
