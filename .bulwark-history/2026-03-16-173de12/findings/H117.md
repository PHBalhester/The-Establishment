# Finding: H117 - Webhook TX Signature Uniqueness

## Status: NOT VULNERABLE
## Confidence Score: 8

## Rationale
The hypothesis suggests that `onConflictDoNothing` without on-chain validation could allow data poisoning. However, this concern conflates two separate issues. The `onConflictDoNothing` in `app/app/api/webhooks/helius/route.ts:337` uses the TX signature as the primary key for `swap_events`, meaning duplicate deliveries of the same signature are silently dropped (correct idempotency). The real vulnerability is not in the conflict strategy but in the **fail-open webhook authentication** (lines 26-28: auth is skipped if `HELIUS_WEBHOOK_SECRET` is unset), which is already tracked as a separate finding. If an attacker can POST to the webhook endpoint, they can inject fabricated events with any signature -- but that is an authentication gap, not a uniqueness gap. The `onConflictDoNothing` strategy is actually correct for idempotent webhook handling.

// app/app/api/webhooks/helius/route.ts:20-22

The conflict handling is sound. The `candle-aggregator.ts` uses `onConflictDoUpdate` with `GREATEST/LEAST` SQL for atomic upserts, which is also correct. No on-chain validation is needed because this data is display-only (charts), not used for financial enforcement.
