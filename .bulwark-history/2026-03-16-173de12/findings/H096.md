# Finding: H096 - BN to Number Conversion for Event Amounts

## Status: POTENTIAL
## Confidence Score: 7

## Rationale
The `bnToNumber` function at `app/lib/event-parser.ts:161-174` converts Anchor BN (arbitrary-precision integer) values to JavaScript `number` type. JavaScript `Number.MAX_SAFE_INTEGER` is 2^53 (~9.007 * 10^15). Token amounts with 6 decimals can reach 1 billion tokens = 10^15 base units, which is within safe range. SOL amounts in lamports max at ~500M SOL = 5 * 10^17, which exceeds MAX_SAFE_INTEGER. However, the protocol's token supply is 1B CRIME + 1B FRAUD + 20M PROFIT, so token amounts stay well under 2^53. The comment at line 159 acknowledges this: "safe for our use case (lamport amounts under 2^53 = ~9 billion SOL, well above any realistic amount)." The risk is theoretical: if pool reserves or event amounts ever approach 2^53 base units, precision loss would corrupt displayed data.

## Code Evidence
```typescript
// app/lib/event-parser.ts:161-174
function bnToNumber(value: unknown): number {
  if (typeof value === "number") return value;
  if (typeof value === "object" && value !== null && "toNumber" in value) {
    return (value as { toNumber: () => number }).toNumber();
  }
  return Number(value);
}
```
BN.toNumber() throws if value exceeds MAX_SAFE_INTEGER, providing a safety valve. But `Number(value)` on line 174 silently truncates.

## Severity Estimate
LOW -- Current token supplies and SOL amounts are well within safe range. The event parser is used for display purposes only (chart data, event history), not for transaction construction. Any precision loss would affect display accuracy, not on-chain execution.
