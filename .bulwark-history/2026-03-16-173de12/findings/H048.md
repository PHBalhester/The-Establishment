# Finding: H048 - Sign-Then-Send Bypasses Wallet Simulation

## Status: POTENTIAL
## Confidence Score: 6

## Executive Summary

The `useProtocolWallet.ts` hook uses a sign-then-send pattern: `signTransaction()` obtains the wallet signature, then `sendRawTransaction()` submits via the app's Helius RPC. This intentionally bypasses the wallet provider's built-in transaction simulation (e.g., Phantom/Blowfish). The code was written this way because Phantom's `signAndSendTransaction` routes through Phantom's own RPC which silently drops devnet transactions.

On mainnet, this means users lose a significant safety net: wallet-side simulation that warns about suspicious transactions (token approval traps, unexpected account changes). While the code itself constructs legitimate transactions, if a future code change or XSS introduces a malicious transaction, users would not see a Blowfish warning before it executes.

The `skipPreflight` behavior is partially mitigated: `useSwap.ts:747` explicitly sets `skipPreflight: false` for standard swap transactions. However, `sendRawTransaction` in `useProtocolWallet.ts:111` passes through `sendOptions.skipPreflight` from the caller -- callers that set `skipPreflight: true` bypass both wallet simulation AND RPC preflight.

## Hypothesis Investigated

signTransaction() + sendRawTransaction() bypasses wallet's built-in simulation.

## Investigation Path

### Step 1: Sign-then-send pattern
**Location:** `app/hooks/useProtocolWallet.ts:87-121`
**Finding:** Confirmed: uses `signTransaction()` + `connection.sendRawTransaction()` instead of `wallet.sendTransaction()`.
**Evidence:**
```typescript
// 1. Wallet signs the transaction (single popup, Blowfish-compatible)
const signed = await signTransaction(tx);
// ...
// 3. Submit via OUR connection (Helius RPC) -- we control the endpoint
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: sendOptions.skipPreflight,
  // ...
});
```

### Step 2: Caller usage of skipPreflight
**Location:** `app/hooks/useSwap.ts:746-749`
**Finding:** Standard swaps use `skipPreflight: false`, preserving RPC-side preflight simulation.
**Evidence:**
```typescript
const signature = await wallet.sendTransaction(tx, connection, {
  skipPreflight: false,
  maxRetries: 2,
});
```

### Step 3: Blowfish/wallet simulation
**Location:** `app/hooks/useProtocolWallet.ts:101`
**Finding:** `signTransaction()` still presents the transaction for user approval in the wallet popup. Wallets like Phantom run Blowfish simulation at the signing stage. So users DO see simulation warnings before signing.
**Evidence:** The comment on line 101 says "single popup, Blowfish-compatible" -- this is correct. Phantom's Blowfish simulation runs during `signTransaction()`, not `signAndSendTransaction()`.

### Step 4: Verification of wallet simulation behavior
**Finding:** After deeper analysis, modern wallets (Phantom v24+, Solflare) run transaction simulation during the `signTransaction()` call, not just `signAndSendTransaction()`. The Blowfish integration in Phantom evaluates the transaction at signing time regardless of submission method. This significantly reduces the risk.

## Blocking Factors
- Modern wallets (Phantom 24+, Solflare) simulate during `signTransaction()`: `useProtocolWallet.ts:102`
- Standard swaps set `skipPreflight: false`: `useSwap.ts:747`

## Enabling Factors
- Sign-then-send pattern changes the simulation timing and context: `useProtocolWallet.ts:102-116`
- Older wallet versions may not simulate during `signTransaction()`
- Some callers might pass `skipPreflight: true` (bonding curve TXs noted in ARCHITECTURE.md)
- MEMORY.md notes: "skipPreflight: true on bonding curve TXs -- unnecessary for legacy TXs"

**Overall counterargument confidence:** MEDIUM. The `unsafe-inline` CSP + sign-then-send creates a theoretical attack chain. Modern wallets mitigate this significantly, but older wallet versions and the `unsafe-inline` CSP weaken the defense.

## Incidental Discoveries
- `useProtocolWallet.ts:110` destructures `signers` from opts but discards it (`_signers`). If a caller passes additional signers (e.g., for a partially-signed TX), they are silently dropped. Suggested hypothesis: "Additional signers silently dropped by useProtocolWallet could cause TX failures or unexpected behavior for multi-signer flows."
