# Finding: H095 - Deploy Scripts Source .env with set -a

## Status: CONFIRMED
## Confidence Score: 8

## Rationale
At `scripts/deploy/deploy-all.sh:80-82`, the script uses `set -a` to auto-export all variables from `.env`, then `set +a` to stop auto-exporting. This exports every variable in `.env` to the environment of all child processes (npm, npx, tsx, anchor, solana CLI). Any variable in `.env` -- including `WALLET_KEYPAIR`, `DATABASE_URL`, `HELIUS_WEBHOOK_SECRET` -- becomes available to every subprocess. While this is the standard pattern for `.env` loading in shell scripts (and is documented in MEMORY.md as required), it means sensitive credentials are exposed to the full process tree. If any child process logs environment variables or a dependency has a telemetry leak, these secrets could be exposed.

## Code Evidence
```bash
# scripts/deploy/deploy-all.sh:79-82
if [ -f .env ]; then
  set -a  # auto-export all variables
  source .env
  set +a
```
All `.env` variables become environment variables for the entire script execution, including `npx tsx scripts/deploy/initialize.ts` and all other subprocesses.

## Severity Estimate
LOW -- This is standard practice for deployment scripts and is required for the deployment pipeline to work correctly (initialize.ts reads pool seed amounts from env). The risk is limited to the local operator's machine during deployment. No production exposure since deploy scripts don't run on Railway.
