# Finding: H056 - Deprecated npm Packages with Known Vulnerabilities

## Status: POTENTIAL
## Confidence Score: 6

## Executive Summary

The `package-lock.json` contains 7 deprecated transitive dependencies, most notably two versions of `glob` with the deprecation message explicitly stating "widely publicized security vulnerabilities." However, these are transitive dev/build dependencies (pulled in by `rimraf`, `ts-node`, etc.), not runtime production dependencies. The practical exploitability is limited because (1) `glob` vulnerabilities are ReDoS (Denial of Service via regex), not RCE, and (2) these packages run only during build/dev, not in the deployed Railway application. The risk is POTENTIAL because the gitignored lockfile (H003) means fresh `npm install` could resolve to different vulnerable versions at any time.

## Hypothesis Investigated

7 deprecated transitive packages including `glob` with security issues.

## Investigation Path

### Step 1: Identify deprecated packages in lockfile
**Location:** `package-lock.json`
**Finding:** 7 deprecated entries found:
1. `@esbuild-kit/core-utils` - "Merged into tsx"
2. `@esbuild-kit/esm-loader` - "Merged into tsx"
3. `glob@7.x` (via rimraf) - "widely publicized security vulnerabilities"
4. `glob@7.x` (via ts-node) - same
5. `glob@7.x` (via another path) - same
6. `inflight` - "leaks memory... Do not use it"
7. `rimraf@3.x` - "prior to v4 are no longer supported"

**Evidence:**
```json
"deprecated": "Old versions of glob are not supported, and contain widely publicized security vulnerabilities, which have been fixed in the current version."
```

### Step 2: Determine if deprecated packages are runtime dependencies
**Location:** `package-lock.json` dependency trees
**Finding:** `glob` is pulled in by `rimraf` (used by build tooling), `ts-node` (dev tool), and similar build-time packages. None appear in the production runtime dependency tree of the Next.js app. The `inflight` package is a transitive dependency of `glob` and inherits the same scope.

### Step 3: Assess vulnerability severity
**Location:** npm advisory database (known from deprecation messages)
**Finding:** The `glob` vulnerabilities are primarily ReDoS (Regular Expression Denial of Service) -- an attacker would need to control file path inputs to `glob()` calls. In this project, `glob` is only invoked during build/development, never in production request handlers.

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | Exploit glob ReDoS in production | No | glob only runs at build time, not in request paths |
| 2 | Exploit inflight memory leak in production | No | inflight is a transitive of glob, same scope |

## Blocking Factors
- **Build-time only**: `glob`, `rimraf`, `inflight` are not in the production runtime path
- **ReDoS scope**: Vulnerabilities require attacker-controlled file path inputs to `glob()` calls

## Enabling Factors
- **No lockfile pinning**: `.gitignore:9` excludes `package-lock.json` -- each build resolves fresh
- **No CI scanning**: No `npm audit` or equivalent runs (H055)
- **Transitive depth**: Deprecated packages are 2-3 levels deep, easy to miss in manual review

## Severity Assessment
- **Severity:** LOW
- **Likelihood:** LOW (build-time only, ReDoS not RCE)
- **Impact:** Build-time DoS at worst; no direct fund loss or data breach

### Recommended Fix
```bash
# Add to CI pipeline (when H055 is fixed)
npm audit --audit-level=moderate

# Update transitive dependencies
npm update rimraf glob
# Or add overrides in package.json:
"overrides": {
  "glob": "^10.0.0",
  "rimraf": "^5.0.0"
}
```
