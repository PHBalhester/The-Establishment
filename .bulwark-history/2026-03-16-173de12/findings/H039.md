# Finding: H039 - skipPreflight on Bonding Curve Transactions

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

Both `BuyForm.tsx` and `SellForm.tsx` pass `skipPreflight: true` when submitting bonding curve transactions. Unlike the AMM multi-hop path (which uses v0 versioned transactions that genuinely break devnet simulation), bonding curve transactions are legacy `Transaction` objects with no such constraint. The `skipPreflight` flag disables RPC-side simulation before submission, meaning users send transactions that may fail on-chain without getting a fast client-side error. This wastes transaction fees and causes confusing UX where transactions are "confirmed" but actually failed.

The AMM swap path (`useSwap.ts:747`) correctly uses `skipPreflight: false`, and the staking path (`useStaking.ts:564`) also uses `skipPreflight: false`. Only the bonding curve forms deviate.

## Hypothesis Investigated

BuyForm/SellForm use `skipPreflight: true` unnecessarily for legacy TXs.

## Investigation Path

### Step 1: Verify BuyForm skipPreflight usage
**Location:** `app/components/launch/BuyForm.tsx:188-190`
**Finding:** BuyForm passes `skipPreflight: true` to `sendTransaction`.
**Evidence:**
```typescript
const signature = await sendTransaction(tx, connection, {
  skipPreflight: true,
});
```

### Step 2: Verify SellForm skipPreflight usage
**Location:** `app/components/launch/SellForm.tsx:197-199`
**Finding:** SellForm also passes `skipPreflight: true`.
**Evidence:**
```typescript
const signature = await sendTransaction(tx, connection, {
  skipPreflight: true,
});
```

### Step 3: Confirm these are legacy transactions (not v0)
**Location:** `app/components/launch/BuyForm.tsx:182-185`, `SellForm.tsx:190-194`
**Finding:** Both forms construct a `new Transaction()` (legacy), not a `VersionedTransaction`. Legacy TXs do not suffer from the devnet v0 simulation bug documented in MEMORY.md.
**Evidence:**
```typescript
const tx = new Transaction();
tx.add(instruction);
tx.feePayer = publicKey;
tx.recentBlockhash = blockhash;
```

### Step 4: Verify AMM path uses skipPreflight: false
**Location:** `app/hooks/useSwap.ts:747`
**Finding:** The AMM swap path correctly uses `skipPreflight: false` for the same sign-then-send pattern.
**Evidence:**
```typescript
const signature = await wallet.sendTransaction(tx, connection, {
  skipPreflight: false,
  maxRetries: 2,
});
```

### Step 5: Trace through useProtocolWallet
**Location:** `app/hooks/useProtocolWallet.ts:111-112`
**Finding:** The `sendTransaction` wrapper faithfully passes the `skipPreflight` option to `connection.sendRawTransaction`. So the `true` value propagates to the RPC call.
**Evidence:**
```typescript
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: sendOptions.skipPreflight,
```

## Invariant Analysis

### Target Invariant
Transactions that will fail on-chain should be caught before submission to avoid wasting user SOL on TX fees.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `BuyForm.tsx:105-143` | Client-side validation (balance, min, cap) | Partial -- catches obvious errors |
| On-chain program | Slippage, state gates, caps | Yes -- but costs TX fee |
| Preflight simulation | Would catch stale state, compute exhaustion | **Disabled** |

## Blocking Factors
- None -- the flag is explicitly set to `true`.

## Enabling Factors
- `BuyForm.tsx:189` -- `skipPreflight: true`
- `SellForm.tsx:198` -- `skipPreflight: true`
- Legacy transactions have no technical reason to skip preflight

## If CONFIRMED

### Severity Assessment
- **Severity:** LOW
- **Likelihood:** MEDIUM (any user whose state is stale will hit this)
- **Impact:** Wasted TX fees + confusing UX (TX "confirmed" but failed). Not a fund loss vector.

### Attack Scenario
```
Step 1: User opens BuyForm, sees quote for 1 SOL -> 100K CRIME
  Target: BuyForm.tsx
  Prerequisite: Curve state has changed since quote (another user bought)

Step 2: User submits TX. No preflight catches the stale state.
  Target: useProtocolWallet.ts -> connection.sendRawTransaction
  Result: TX submitted to validator without simulation

Step 3: On-chain slippage check fails. TX is confirmed but errored.
  Result: User loses ~5000 lamports in TX fee, gets confusing error message

Final Result: User wastes SOL on failed TXs. Not exploitable for fund theft,
but degrades UX and wastes gas. In high-frequency scenarios (rapid price
movement during launch), this could cause repeated failed TXs.
```

### Recommended Fix
```typescript
// Before (vulnerable)
const signature = await sendTransaction(tx, connection, {
  skipPreflight: true,
});

// After (fixed)
const signature = await sendTransaction(tx, connection, {
  skipPreflight: false,
  maxRetries: 2,
});
```

Apply to both `BuyForm.tsx:188-190` and `SellForm.tsx:197-199`.
