# Finding: H015 - Sandwich Attack on User Swaps (No MEV Protection)

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

User swap transactions across all three submission paths (direct SOL swaps, atomic multi-hop routes, and bonding curve buy/sell) are submitted via standard Helius RPC without any MEV protection. There are no Jito bundles, no private mempool submission, and no MEV-protected RPC endpoint anywhere in the codebase. Combined with a default slippage tolerance of 500 BPS (5%), this creates a highly profitable sandwich attack surface on mainnet. An MEV bot can extract up to ~4.9% of every swap's value minus negligible gas costs (~0.0001 SOL).

This is not a theoretical concern -- Solana MEV extraction is industrialized, with searchers running automated sandwich bots that monitor all standard RPC-submitted transactions. The protocol's default configuration maximizes attacker profit by setting a generous 5% slippage tolerance out of the box.

## Hypothesis Investigated

User swap transactions are submitted via standard Helius RPC without MEV protection, enabling sandwich attacks especially given the default 5% slippage tolerance.

## Investigation Path

### Step 1: Transaction Submission Path (All User Swaps)
**Location:** `app/hooks/useProtocolWallet.ts:87-121`
**Finding:** All user transactions flow through a single code path: `signTransaction()` then `connection.sendRawTransaction()`. The `connection` is a standard Helius RPC connection with no MEV protection.
**Evidence:**
```typescript
// app/hooks/useProtocolWallet.ts:111
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: sendOptions.skipPreflight,
  preflightCommitment: sendOptions.preflightCommitment ?? connection.commitment ?? undefined,
  maxRetries: sendOptions.maxRetries,
  minContextSlot: sendOptions.minContextSlot,
});
```

### Step 2: RPC Endpoint Configuration
**Location:** `app/lib/connection.ts:31-52`, `shared/programs.ts:21-22`
**Finding:** The connection uses standard Helius devnet RPC (`devnet.helius-rpc.com`). No MEV-protected endpoint (e.g., Jito block engine, Helius staked connections) is configured anywhere.
**Evidence:**
```typescript
// shared/programs.ts:21-22
export const DEVNET_RPC_URL =
  "https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]";
```

### Step 3: Default Slippage Tolerance
**Location:** `app/providers/SettingsProvider.tsx:77`
**Finding:** Default slippage is 500 BPS (5%). This is 5-10x higher than industry standard (Jupiter defaults to 0.5%, Raydium to 0.5-1%).
**Evidence:**
```typescript
// app/providers/SettingsProvider.tsx:76-77
return {
  slippageBps: 500,  // 5% default
```

### Step 4: Slippage Applied to minimumOutput
**Location:** `app/hooks/useSwap.ts:407,432`
**Finding:** The slippage BPS directly determines the `minimumOutput` parameter sent on-chain. With 500 BPS default, the minimum output is 95% of expected output, giving an MEV bot a 5% extraction window.
**Evidence:**
```typescript
// app/hooks/useSwap.ts:407
const minimumOutput = Math.floor(result.outputTokens * (10_000 - slippageBps) / 10_000);
// app/hooks/useSwap.ts:432
const minimumOutput = Math.floor(result.outputSol * (10_000 - slippageBps) / 10_000);
```

### Step 5: No Jito/MEV Libraries in Codebase
**Location:** Entire `app/` directory
**Finding:** Grep for `jito`, `mev`, `private.*mempool`, `bundle`, `protect` across the app directory returns zero relevant results. No MEV protection library is imported or referenced anywhere in the frontend code.

### Step 6: Bonding Curve Swaps Doubly Exposed
**Location:** `app/components/launch/BuyForm.tsx:188-189`, `app/components/launch/SellForm.tsx:197-198`
**Finding:** Bonding curve transactions use `skipPreflight: true` AND have no priority fee instructions. This makes them even more visible to MEV bots (immediate broadcast without simulation) and easier to front-run (no priority fee competition).
**Evidence:**
```typescript
// app/components/launch/BuyForm.tsx:188-189
const signature = await sendTransaction(tx, connection, {
  skipPreflight: true,
```

### Step 7: Slippage Upper Bound Allows 100%
**Location:** `app/providers/SettingsProvider.tsx:103-104` vs `app/components/swap/SlippageConfig.tsx:133`
**Finding:** SettingsProvider validates `slippageBps <= 10_000` (100%), while UI custom input caps at `parsed <= 50` (50%). A malicious browser extension or XSS could set `localStorage['dr-fraudsworth-settings']` to `{"slippageBps": 10000}`, making `minimumOutput = 0` and removing all slippage protection entirely.
**Evidence:**
```typescript
// app/providers/SettingsProvider.tsx:101-106
slippageBps:
  typeof parsed.slippageBps === 'number' &&
  parsed.slippageBps >= 0 &&
  parsed.slippageBps <= 10_000  // allows up to 100%
    ? parsed.slippageBps
    : defaults.slippageBps,
```

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |
|------|-----------------|-----------|----------|
| 1 | MEV bot monitors Helius RPC for pending swap TXs | Yes | Standard Helius RPC, no private mempool. Transactions visible to validators and searchers. |

## Invariant Analysis

### Target Invariant
Users should receive swap output within a reasonable tolerance of the quoted amount, and transaction submission should not expose users to preventable value extraction.

### Enforcement Points
| Location | What It Enforces | Complete? |

## Blocking Factors
- None. There are zero MEV protections in the off-chain codebase.

## Enabling Factors
- **Standard RPC submission**: `app/hooks/useProtocolWallet.ts:111` -- `sendRawTransaction` through Helius with no private mempool
- **5% default slippage**: `app/providers/SettingsProvider.tsx:77` -- gives MEV bot a 5% extraction ceiling
- **100% slippage allowed**: `app/providers/SettingsProvider.tsx:104` -- localStorage manipulation can set slippage to 100%
- **No priority fees on bonding curve**: `app/components/launch/BuyForm.tsx:188-189` -- no `ComputeBudgetProgram.setComputeUnitPrice` instruction
- **skipPreflight on multiple paths**: `app/lib/swap/multi-hop-builder.ts:381`, `app/components/launch/BuyForm.tsx:189`, `app/components/launch/SellForm.tsx:198` -- immediate broadcast makes TXs visible faster

## Severity Assessment
- **Severity:** HIGH
- **Likelihood:** HIGH (MEV bots are ubiquitous on Solana mainnet; automated extraction is industrialized)
- **Impact:** Direct value extraction from every user swap. For a $1,000 swap with default 5% slippage, attacker profit ceiling is ~$49 minus ~$0.01 gas. Scales linearly with trade volume.

### Attack Scenario

```
Step 1: MEV bot monitors Helius RPC for pending transactions
  Target: Standard Helius RPC mempool / leader schedule
  Prerequisite: Know Dr Fraudsworth program IDs (public)
  Result: Bot sees user's swap TX before it lands in a block

Step 2: Bot decodes user's swap instruction
  Target: TX instruction data
  Payload: Reads inputAmount, minimumOutput, pool accounts
  Result: Bot knows the user's swap size and maximum acceptable slippage

Step 3: Bot front-runs with a buy TX (higher priority fee)
  Target: Same AMM pool
  Payload: Buy TX with ComputeUnitPrice=1,000,000 (turbo)
  Result: Pool price moves up; bot's TX lands before user's

Step 4: User's TX lands at worse rate
  Target: User's original swap TX
  Result: User receives output between minimumOutput and expectedOutput
         (up to 5% worse than expected with default settings)

Step 5: Bot back-runs with a sell TX
  Target: Same AMM pool
  Payload: Sell the tokens just bought, capturing the spread
  Result: Bot profits from the difference between front-run price and back-run price

Final Result: Bot extracts up to ~4.9% of user's swap value per transaction.
With bonding curve swaps (no priority fees + skipPreflight), extraction is even easier.
```

### Recommended Fix

**Fix 1: Integrate MEV-protected transaction submission (PRIMARY)**
```typescript
// Before (vulnerable) -- app/hooks/useProtocolWallet.ts:111
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: sendOptions.skipPreflight,
  ...
});

// After (fixed) -- use Jito bundle submission for swap TXs
import { searcherClient } from "jito-ts/dist/sdk/block-engine/searcher";

// Option A: Jito bundle submission
const bundle = new Bundle([signedTx], tipLamports);
await client.sendBundle(bundle);

// Option B: Use Helius priority fee API + staked connections
// Helius offers staked connections that skip the public mempool
const signature = await connection.sendRawTransaction(serialized, {
  skipPreflight: false,
  // Use Helius staked connection endpoint (separate URL)
});
```

**Fix 2: Reduce default slippage (IMMEDIATE)**
```typescript
// Before (vulnerable) -- app/providers/SettingsProvider.tsx:77
slippageBps: 500, // 5%

// After (improved) -- reduce to industry standard
slippageBps: 50, // 0.5% (matches Jupiter default)
```

**Fix 3: Cap maximum slippage in SettingsProvider (IMMEDIATE)**
```typescript
// Before (vulnerable) -- app/providers/SettingsProvider.tsx:104
parsed.slippageBps <= 10_000 // allows 100%

// After (capped)
parsed.slippageBps <= 3_000 // cap at 30% -- still generous, prevents total extraction
```

**Fix 4: Add priority fees to bonding curve TXs (IMMEDIATE)**
```typescript
// Before (vulnerable) -- app/components/launch/BuyForm.tsx (no ComputeBudget)
const tx = new Transaction().add(purchaseIx);

// After (improved)
const tx = new Transaction()
  .add(ComputeBudgetProgram.setComputeUnitLimit({ units: 200_000 }))
  .add(ComputeBudgetProgram.setComputeUnitPrice({ microLamports: priorityFee }))
  .add(purchaseIx);
```

## Incidental Discoveries
- **localStorage slippage injection**: `app/providers/SettingsProvider.tsx:104` allows up to 100% slippage via localStorage manipulation. A malicious browser extension or XSS could silently set `slippageBps: 10000`, making `minimumOutput = 0` for all subsequent swaps. This is a separate attack vector from MEV (social engineering / extension-based) but compounds the sandwich risk. Suggested hypothesis: "Malicious browser extension or XSS sets slippage to 100% via localStorage, enabling complete value extraction on next swap."
- **No dynamic priority fee estimation**: `app/hooks/useSwap.ts:94-100` uses static priority fee presets. On mainnet during congestion, even "turbo" (1M microLamports) may be insufficient for timely inclusion, while MEV bots use dynamic fee estimation to always outbid users. Suggested hypothesis: "Static priority fee presets cause user TXs to be consistently deprioritized vs MEV bots during network congestion."
