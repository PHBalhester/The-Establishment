# Finding: H011 - Database Connection Without TLS

## Status: POTENTIAL
## Confidence Score: 6

## Executive Summary

The PostgreSQL connections established by the Next.js application (`app/db/connection.ts:51`) and migration runner (`app/db/migrate.ts:42`) do not explicitly configure TLS/SSL. The `postgres()` driver is called with only `{ max: N }` options -- no `ssl` parameter is passed. The same pattern appears in the backfill script (`scripts/backfill-candles.ts:225`).

Whether TLS is actually used depends entirely on what the `DATABASE_URL` connection string contains (e.g., `?sslmode=require`) and whether Railway's Postgres proxy enforces TLS at the infrastructure level. The application code provides zero enforcement or verification. This means a configuration change to `DATABASE_URL` (accidental removal of `sslmode` parameter, Railway infrastructure change, or migration to a different Postgres host) would silently downgrade to plaintext without any application-level warning.

## Hypothesis Investigated

The PostgreSQL connection from the Next.js app and migration runner does not enforce TLS, potentially allowing eavesdropping on database traffic.

## Investigation Path

### Step 1: Connection.ts analysis
**Location:** `app/db/connection.ts:50-51`
**Finding:** The postgres.js client is created with only a connection string and `max` option. No `ssl` configuration object is passed.
**Evidence:**
```typescript
const client =
    globalForDb.pgClient ?? postgres(connectionString, { max: 10 });
```

### Step 2: Migrate.ts analysis
**Location:** `app/db/migrate.ts:42`
**Finding:** Identical pattern -- no SSL options.
**Evidence:**
```typescript
const client = postgres(databaseUrl, { max: 1 });
```

### Step 3: Backfill script analysis
**Location:** `scripts/backfill-candles.ts:225`
**Finding:** Same pattern in the backfill script.
**Evidence:**
```typescript
return postgres(connectionString, { max: 5 });
```

### Step 4: Drizzle config analysis
**Location:** `app/drizzle.config.ts:17-24`
**Finding:** The Drizzle Kit config also passes `DATABASE_URL` without any SSL options via `dbCredentials.url`.

### Step 5: Codebase-wide SSL search
**Location:** Entire `app/` directory and all `*.ts` files
**Finding:** Grep for `ssl`, `sslmode`, `tls`, `certificate`, `rejectUnauthorized` found zero database-related TLS configuration anywhere in the codebase. The only `tls` references are Next.js webpack module aliasing (`app/next.config.ts:51`) for browser builds.

### Step 6: Railway configuration check
**Location:** `railway.toml`
**Finding:** No database configuration in Railway TOML. DATABASE_URL is provided via Railway dashboard environment variables. No SSL enforcement at the application config level.

### Step 7: postgres.js default behavior analysis
**Finding:** The postgres.js library (the `postgres` npm package) has the following SSL behavior by default:
- If no `ssl` option is provided, it does NOT negotiate TLS unless the connection string contains `sslmode=require` (or similar).
- If `sslmode=require` is in the connection string, TLS is negotiated but certificate verification depends on the `sslmode` value (`require` = encrypt but don't verify cert; `verify-full` = encrypt + verify cert).
- Railway's auto-provisioned DATABASE_URL format typically includes `?sslmode=require`, but this is an infrastructure assumption, not an application guarantee.

## Blocking Factors
- **Railway internal networking**: If Railway's Postgres and Next.js service run on the same internal network with TLS enforced at the proxy, network-level eavesdropping may be impractical. However, this is an infrastructure assumption (A-OC7 in ARCHITECTURE.md), not a code-level guarantee.
- **Railway DATABASE_URL format**: Railway typically provisions URLs with `?sslmode=require`. If present, postgres.js will negotiate TLS even without explicit `ssl` options. But this is not verified by the application.

## Enabling Factors
- **No application-level SSL enforcement**: `app/db/connection.ts:51` -- the `postgres()` call passes no `ssl` option
- **No startup validation**: There is no code that checks `DATABASE_URL` contains `sslmode=require` before connecting
- **No certificate verification**: Even if TLS is negotiated via the connection string, there is no `ssl: { rejectUnauthorized: true }` to verify the server certificate, making MITM possible
- **Three separate connection sites**: The pattern is repeated in `connection.ts`, `migrate.ts`, and `backfill-candles.ts`, all with the same gap
- **DATABASE_URL is the sole control**: Any change to this env var (Railway rotation, migration to different provider, accidental edit) could silently disable TLS

## Severity Assessment
- **Severity:** MEDIUM
- **Likelihood:** LOW (Railway likely enforces TLS at proxy level, but unvalidated)
- **Impact:** Credential exposure (DATABASE_URL contains password), query data exposure (swap events, candle data, potentially connection strings in error messages)

## Attack Scenario

```
Precondition: DATABASE_URL is changed/rotated without sslmode parameter,
              OR project migrates to a non-Railway Postgres host

Step 1: Attacker gains network position (e.g., compromised Railway instance,
        cloud VPC misconfiguration, or non-Railway deployment without TLS)
  Target: Network path between Next.js process and Postgres
  Result: Can observe Postgres wire protocol traffic

Step 2: Passive eavesdropping on Postgres connections
  Target: app/db/connection.ts:51 connection establishment
  Result: Captures DATABASE_URL credentials (username/password),
          all SQL queries (swap events, candle data),
          and potentially sensitive data in error responses

Step 3: Active MITM (if no cert verification)
  Target: Same network path
  Result: Can modify query results, inject fake data,
          or harvest credentials for direct DB access
```

## Recommended Fix

```typescript
// Before (vulnerable) -- app/db/connection.ts:51
const client =
    globalForDb.pgClient ?? postgres(connectionString, { max: 10 });

// After (fixed) -- enforce TLS with certificate verification
const client =
    globalForDb.pgClient ?? postgres(connectionString, {
      max: 10,
      ssl: {
        rejectUnauthorized: true,  // Verify server certificate
      },
    });
```

Apply the same fix to:
- `app/db/migrate.ts:42`
- `scripts/backfill-candles.ts:225`

Additionally, add a startup validation check:

```typescript
// In getDb() before creating the client:
if (!connectionString.includes('sslmode=')) {
  console.warn(
    '[db] WARNING: DATABASE_URL does not contain sslmode parameter. ' +
    'Connections may not be encrypted. Set sslmode=require in your connection string.'
  );
}
```

**Note:** Railway Postgres with `sslmode=require` + `ssl: { rejectUnauthorized: true }` may fail if Railway uses self-signed certificates. In that case, use `ssl: 'require'` (encrypt without cert verify) as a minimum, which is still better than no explicit SSL config.

## Incidental Discoveries
- **No connection timeouts configured**: All three `postgres()` call sites pass no `idle_timeout`, `connect_timeout`, or `max_lifetime` options. A hung Postgres connection could hold a pool slot indefinitely. Location: `app/db/connection.ts:51`, `app/db/migrate.ts:42`, `scripts/backfill-candles.ts:225`. Suggested hypothesis: "Database connection pool exhaustion via hung connections."
- **Same credentials for app and migrations**: `app/db/connection.ts` and `app/db/migrate.ts` both use the same `DATABASE_URL`, meaning the runtime application has DDL-level database privileges. Suggested hypothesis: "Privilege escalation via application-level SQL if ORM is bypassed."
