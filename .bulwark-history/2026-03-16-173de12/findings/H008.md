# Finding: H008 - SSE Amplification DoS via Webhook Flooding

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The hypothesized SSE amplification DoS attack is viable. An attacker can combine two independently weak surfaces -- an unauthenticated SSE endpoint with unbounded subscriber count and a fail-open webhook endpoint -- to create a multiplicative resource exhaustion attack. Each forged webhook transaction generates 6 candle resolution broadcasts to every connected SSE subscriber. With no rate limiting, no connection caps, no body size enforcement, and no authentication on either endpoint, the amplification factor is N_transactions x 6_resolutions x M_subscribers. At modest scale (100 fake TXs, 1000 SSE connections), this produces 600,000 `controller.enqueue()` calls in a single webhook request lifecycle, likely exhausting the Railway container's memory and CPU.

The attack requires no authentication, no insider knowledge beyond the publicly discoverable webhook URL, and can be scripted trivially. Railway's `restartPolicyMaxRetries = 3` means sustained attack causes permanent downtime until manual intervention.

## Hypothesis Investigated

An attacker combines unauthenticated webhook access with unbounded SSE subscribers to create a novel amplification attack that exhausts server resources. Specifically: open 1000 SSE connections, POST 100 fake transactions to the webhook, triggering 6 candle upserts x 1000 subscriber broadcasts = 600,000 messages per webhook POST.

### Step 3: Broadcast Amplification Factor
**Location:** `app/lib/sse-manager.ts:58-67`

**Evidence:**

    try {
      callback(payload);
    } catch {
      this.subscribers.delete(callback);
    }
  }

## Enabling Factors
- **No SSE connection limit:** `app/lib/sse-manager.ts:39` -- subscribers.add() with no size guard
- **No SSE authentication:** `app/app/api/sse/candles/route.ts:38` -- bare GET handler
- **Fail-open webhook auth:** `app/app/api/webhooks/helius/route.ts:135-136` -- skipped when env var unset
- **6x resolution fan-out:** `app/app/api/webhooks/helius/route.ts:216-221` -- broadcasts per swap per resolution
- **No rate limiting anywhere:** No `middleware.ts` exists in the project
- **No body size limit:** `app/app/api/webhooks/helius/route.ts:146` -- `req.json()` reads unbounded body
- **Limited restart budget:** `railway.toml:11` -- only 3 retries before permanent death

## Blocking Factors
- **If HELIUS_WEBHOOK_SECRET is set:** Webhook auth is enforced (`route.ts:136-140`). This blocks Step 2 unless the attacker knows the secret. However, the SSE connection flooding (Step 1) is still possible independently, and the timing-unsafe comparison (`!==` at line 138) allows gradual secret recovery.
- **Railway proxy limits:** Railway may impose connection limits at the proxy/load-balancer level that are not visible in application code. This is unvalidated and cannot be relied upon.
- **parseSwapEvents filtering:** Forged webhook payloads must contain valid-looking Anchor log messages that pass `parseSwapEvents()`. However, the parser uses BorshCoder which would reject malformed data -- the attacker needs correctly-formatted fake events. This raises the bar slightly but is achievable with Anchor IDL access (which is committed to the repo at `app/idl/`).

## Severity Assessment
- **Severity:** HIGH
- **Likelihood:** POSSIBLE (requires webhook secret to be unset OR attacker to crack it via timing; SSE flooding alone is PROBABLE)
- **Impact:** Service outage (complete unavailability of the web frontend, API routes, and SSE streaming). Does not directly cause fund loss, but prevents all users from interacting with the protocol through the web interface.

### Attack Scenario

```
Step 1: Open 1000 SSE connections
  Target: GET /api/sse/candles
  Payload: None (standard EventSource connections)
  Prerequisite: None (no auth, no limit)
  Tool: Simple script with 1000 EventSource() or HTTP clients
  Result: sseManager.subscribers.size === 1000

Step 2: Craft webhook payload with fake swap events
  Prerequisite: HELIUS_WEBHOOK_SECRET unset in production, OR secret leaked/cracked
  Payload: JSON array of 100 fake transactions, each containing:
    - Unique signature (random base58 string)
    - logMessages array with valid Anchor TaxedSwap event encoding
    - blockTime, slot fields
    - No meta.err field
  Note: Attacker can derive correct Borsh encoding from committed IDL files

Step 3: POST crafted payload to webhook
  Target: POST /api/webhooks/helius
  Headers: Authorization: <secret> (if required)
  Body: 100-element JSON array of fake transactions
  Result: 100 swap events x 6 resolutions x 1000 subscribers = 600,000 enqueue() calls

Step 4: Repeat POST every few seconds
  Each POST triggers 600,000 synchronous operations
  Node.js event loop is blocked during broadcast iterations
  Memory accumulates from ReadableStream buffers
  Railway container exceeds memory limit -> OOM kill

Step 5: Container restart exhaustion
  Railway restarts container (restartPolicyType = "ON_FAILURE")
  Attacker reopens SSE connections during restart window
  After 3 OOM kills, restartPolicyMaxRetries exhausted
  Service is permanently down until manual intervention

Final Result: Complete denial of service of the web frontend, all API routes,
and SSE streaming. Protocol on-chain operations continue (crank is separate
Railway service) but users cannot interact through the web UI.
```

### Recommended Fix

**Fix 1: Add subscriber cap to SSEManager**
```typescript
// Before (vulnerable)
subscribe(callback: SSECallback): () => void {
  this.subscribers.add(callback);
  return () => { this.subscribers.delete(callback); };
}

// After (fixed)
private static readonly MAX_SUBSCRIBERS = 500;

subscribe(callback: SSECallback): (() => void) | null {
  if (this.subscribers.size >= SSEManager.MAX_SUBSCRIBERS) {
    return null; // Reject connection
  }
  this.subscribers.add(callback);
  return () => { this.subscribers.delete(callback); };
}
```

**Fix 2: Make webhook auth fail-closed**
```typescript
// Before (vulnerable)
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (webhookSecret) {
  // auth only if secret is configured
}

// After (fixed)
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (!webhookSecret) {
  console.error("[webhook] HELIUS_WEBHOOK_SECRET not configured -- rejecting all requests");
  return NextResponse.json({ error: "Webhook not configured" }, { status: 503 });
}
const authHeader = req.headers.get("authorization");
if (!crypto.timingSafeEqual(Buffer.from(authHeader ?? ""), Buffer.from(webhookSecret))) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

**Fix 3: Add rate limiting middleware**
Create `app/middleware.ts` with IP-based rate limiting on `/api/webhooks/*` and connection counting on `/api/sse/*`.

**Fix 4: Add body size check on webhook**
```typescript
// Before req.json()
const contentLength = parseInt(req.headers.get("content-length") ?? "0", 10);
if (contentLength > 5_000_000) { // 5MB max
  return NextResponse.json({ error: "Payload too large" }, { status: 413 });
}
```

## Related Findings
- Combines with **webhook auth fail-open** (the primary enabler for the amplification path)
- Combines with **timing-unsafe secret comparison** (`route.ts:138` uses `!==`) which could allow secret recovery over time
- Combines with **no body size limit** on webhook POST for additional OOM vector

## Incidental Discoveries
- **SSE subscriber zombie accumulation**: Even without active attack, if SSE clients disconnect through proxies that don't properly propagate the abort signal, subscribers accumulate forever in the in-memory Set. The heartbeat catch block (`route.ts:62-68`) only removes subscribers that throw on `enqueue()`, but a buffering proxy that silently accepts data would persist indefinitely. Suggested hypothesis: Long-term memory leak from SSE subscriber zombies under normal operation.
- **Webhook processes DB writes synchronously within request lifecycle**: The `for (const tx of transactions)` loop at `route.ts:160` processes each TX sequentially with `await`. A batch of 100 TXs each requiring candle upserts creates a long-running request. If Helius has a delivery timeout, large batches could cause timeout -> retry -> re-processing loops. Suggested hypothesis: Webhook timeout-retry feedback loop causing duplicate processing load.
