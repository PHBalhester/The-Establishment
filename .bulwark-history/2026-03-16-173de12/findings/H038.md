# Finding: H038 - Split Route Zero Fee Display

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The `buildSplitRoute` function in `app/hooks/useRoutes.ts:176-177` hardcodes `totalLpFee: 0` and `totalTax: 0` in the Route object for split routes. The comments explain this is intentional because "mixed denominations across legs" make absolute fee amounts meaningless. However, the Route type's `totalLpFee` and `totalTax` fields are consumed by `useSwap.ts:808-809` to populate the `SwapQuote` object, which is then displayed in the UI's fee breakdown.

When smart routing selects a split route, the UI's `quote.lpFee` and `quote.taxAmount` will show 0, giving users the false impression that split routes have no fees. The `totalFeePct` field IS correctly computed from BPS summation (line 160-161), so the percentage display is accurate -- but any UI component that displays absolute fee amounts (e.g., "LP Fee: 0.005 SOL") would show zero.

## Hypothesis Investigated

Split routes hardcode `totalLpFee: 0` and `totalTax: 0`.

## Investigation Path

### Step 1: Verify the hardcoded zeros
**Location:** `app/hooks/useRoutes.ts:176-177`
**Finding:** Both `totalLpFee` and `totalTax` are hardcoded to 0 with explanatory comments.
**Evidence:**
```typescript
return {
  steps,
  inputToken,
  outputToken,
  inputAmount: totalInput,
  outputAmount: splitOutput,
  totalLpFee: 0, // Not meaningful for split routes (mixed denominations across legs)
  totalTax: 0,   // Callers use step-level BPS or totalFeePct for display
  totalPriceImpactBps,
  totalFeePct,
  hops: 2,
  isSplit: true,
  splitRatio: [splitRatioA, splitRatioB],
  label,
  minimumOutput,
};
```

### Step 2: Trace how these values are consumed
**Location:** `app/hooks/useSwap.ts:797-816`
**Finding:** When smart routing is active and a route is selected, `useSwap` populates `quote` from the route's fields, directly mapping `totalLpFee` and `totalTax` to `lpFee` and `taxAmount`.
**Evidence:**
```typescript
useEffect(() => {
  if (smartRouting && routesResult.selectedRoute) {
    const route = routesResult.selectedRoute;
    // ...
    setQuote({
      outputAmount: route.outputAmount,
      lpFee: route.totalLpFee,        // <-- 0 for split routes
      taxAmount: route.totalTax,       // <-- 0 for split routes
      priceImpactBps: route.totalPriceImpactBps,
      minimumOutput: route.minimumOutput,
      totalFeePct: route.totalFeePct,  // <-- Correct BPS-based value
    });
  }
}, [smartRouting, routesResult.selectedRoute]);
```

### Step 3: Verify totalFeePct is correct
**Location:** `app/hooks/useRoutes.ts:160-161`
**Finding:** `totalFeePct` IS correctly computed using BPS summation from step-level rates.
**Evidence:**
```typescript
const totalFeeBps = steps.reduce((sum, s) => sum + s.lpFeeBps + s.taxBps, 0);
const totalFeePct = `${(totalFeeBps / 100).toFixed(1)}%`;
```

### Step 4: Check if non-split multi-hop routes have the same issue
**Location:** `app/lib/swap/route-engine.ts:300-301, 349-351`
**Finding:** Non-split multi-hop routes DO accumulate `totalLpFee` and `totalTax` from step results. However, these are also mixed-denomination for multi-hop routes (e.g., SOL->CRIME->PROFIT: LP fee in lamports + vault fee of 0). The route engine accumulates them anyway, which is inconsistent with the split route's decision to zero them out.
**Evidence:**
```typescript
let totalLpFee = 0;
let totalTax = 0;
for (const edge of path) {
  totalLpFee += result.lpFeeAmount;
  totalTax += result.taxAmount;
}
```

## Blocking Factors
- `totalFeePct` is correctly computed and displayed
- Steps array contains per-step fee data (any UI that reads steps directly would see correct values)
- The comment explicitly documents the design choice

## Enabling Factors
- `useRoutes.ts:176-177`: Hardcoded zeros for `totalLpFee` and `totalTax`
- `useSwap.ts:808-809`: Propagates zeros to the SwapQuote consumed by the UI
- No fallback computation in useSwap to derive absolute fees from steps when route values are 0

### Severity Assessment
- **Severity:** LOW

### Attack Scenario
```
Step 1: User selects SOL -> PROFIT swap with smart routing ON (default)
Step 2: Route engine computes split route: 60% via CRIME, 40% via FRAUD
Step 3: Split route wins best-output comparison, auto-selected
Step 4: UI fee breakdown shows:
  - LP Fee: 0 SOL       (incorrect -- should show some value)
  - Tax: 0 SOL           (incorrect -- should show some value)
  - Total Fee: 11.0%     (correct from BPS summation)
Step 5: User sees "0 fee" but 11% total, creating confusion
```

### Recommended Fix
```typescript
// Option 1: Compute an approximate absolute fee from the percentage
// Since totalFeePct is correct, derive absolute fees:
const totalFeeBps = steps.reduce((sum, s) => sum + s.lpFeeBps + s.taxBps, 0);
const totalLpFeeBps = steps.reduce((sum, s) => sum + s.lpFeeBps, 0);
const totalTaxBps = steps.reduce((sum, s) => sum + s.taxBps, 0);

// Express fees as fraction of input (single denomination):
const totalLpFee = Math.floor(totalInput * totalLpFeeBps / 10_000);
const totalTax = Math.floor(totalInput * totalTaxBps / 10_000);

// Option 2: Explicitly mark as unavailable
totalLpFee: null,  // Change type to number | null
totalTax: null,    // UI checks for null and shows "N/A" instead of "0"
```

Option 2 is more honest -- the absolute amounts in a single denomination are genuinely not meaningful for cross-token split routes. The UI should reflect this rather than showing misleading zeros.
