# Finding: H061 - No Negative Amount Guards on Quote Primitives

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The `quoteSolBuy` and `quoteSolSell` functions in `app/lib/swap/quote-engine.ts` do not explicitly check for negative inputs. However, this is not exploitable because: (1) the functions are client-side display computations only -- they do NOT construct or submit transactions; (2) the actual transaction submission uses `toBaseUnits()` which includes `parseFloat() > 0` validation; (3) negative amounts would produce obviously wrong UI values (negative outputs) that a user would not sign; and (4) on-chain programs enforce all constraints regardless of client-side quotes. The quote engine's role is preview display only -- the on-chain slippage check (INV-OC1) is the real safety net.

## Hypothesis Investigated

quoteSolBuy/quoteSolSell don't validate positive inputs.

## Investigation Path

### Step 1: Check quote function input validation
**Location:** `app/lib/swap/quote-engine.ts:132-156` (quoteSolBuy), `187-211` (quoteSolSell)
**Finding:** Neither function validates that `solAmountLamports` or `tokenAmountBaseUnits` is positive. However, the reverse quote functions DO check: `reverseQuoteSolBuy` returns `null` for `desiredOutputTokens <= 0` (line 285), and `reverseQuoteSolSell` returns `null` for `desiredOutputSol <= 0` (line 331). The vault convert function checks `inputAmount <= 0` (line 240).
**Evidence:**
```typescript
// quoteSolBuy -- no input validation
export function quoteSolBuy(
  solAmountLamports: number,  // no check for <= 0
  reserveWsol: number,
  reserveToken: number,
  buyTaxBps: number,
  lpFeeBps: number,
): SolBuyQuote {
  const taxAmount = calculateTax(solAmountLamports, buyTaxBps);
  // ...
```

### Step 2: Trace callers of quoteSolBuy/quoteSolSell
**Location:** `app/hooks/useSwap.ts:322-521`
**Finding:** The `computeQuote` callback always passes through `toBaseUnits()` first, which filters out non-positive values.
**Evidence:**
```typescript
// useSwap.ts:391-396
const baseUnits = toBaseUnits(amount, inputToken);
if (baseUnits <= 0) {
  setQuote(null);
  setOutputAmountRaw("");
  return;
}
```

And `toBaseUnits` itself:
```typescript
// useSwap.ts:295-297
const parsed = parseFloat(amount);
if (isNaN(parsed) || parsed <= 0) return 0;
```

### Step 3: Verify quote results don't influence transaction construction
**Location:** `app/hooks/useSwap.ts:673-791`
**Finding:** The `executeSwap` function re-derives `amountInLamports` from `inputAmount` via `toBaseUnits()` (line 713), not from the quote. The `minimumOutput` comes from the quote (line 718: `quote.minimumOutput`), but this is sent as on-chain instruction data where the program enforces it independently.

## Invariant Analysis

### Target Invariant
Quote engine should never produce values that lead to worse execution than intended.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `useSwap.ts:296-297` | `parseFloat > 0` in `toBaseUnits()` | Yes (all callers) |
| `useSwap.ts:391-396` | Early return if `baseUnits <= 0` | Yes |
| On-chain program | `minimumOutput` slippage check | Yes (INV-OC1) |

## Blocking Factors
- **Caller validation**: `useSwap.ts:296-297` -- `toBaseUnits()` returns 0 for negative/NaN
- **Early exit**: `useSwap.ts:391-396` -- exits before calling quote functions if amount <= 0
- **On-chain enforcement**: INV-OC1 -- slippage check is on-chain, independent of client

## Enabling Factors
- **Missing defensive check**: `quote-engine.ts:132,187` -- functions accept any number without validation
