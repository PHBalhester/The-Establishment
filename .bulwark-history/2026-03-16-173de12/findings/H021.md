# Finding: H021 - Patch-Mint-Addresses as Trust Amplifier

## Status: POTENTIAL
## Confidence Score: 7

## Executive Summary

The `patch-mint-addresses.ts` script reads keypair files from disk, derives their public keys, and writes those addresses directly into Rust `constants.rs` source files before `anchor build`. This creates a trust amplification chain: whoever controls the keypair files on disk controls what addresses get compiled into the on-chain programs. The script performs **no integrity verification** (no checksums, no expected-value assertions, no signing) on the keypair files it reads.

While the post-build `verify-program-ids.ts` script catches **program ID** mismatches across keypairs/declare_id!/Anchor.toml, it does **not** verify the mint addresses or treasury address that `patch-mint-addresses.ts` writes into `constants.rs` files. This means a compromised mint keypair file would pass all existing verification checks and produce programs that reference attacker-controlled mint addresses.

The practical exploitability is limited by the fact that this is a local developer workflow (not CI/CD), and the operator would need to notice the wrong addresses during the subsequent `initialize.ts` and `verify.ts` phases. However, the attack surface is real and the gap in verification is concrete.

## Hypothesis Investigated

`patch-mint-addresses.ts` modifies Rust source code at build time based on keypair files on disk. A compromised keypair file would cause wrong mint addresses to be compiled into production programs.

## Investigation Path

### Step 1: Patch script keypair loading
**Location:** `scripts/deploy/patch-mint-addresses.ts:35-42`
**Finding:** The `loadKeypair()` function reads a JSON file from disk, parses it as a secret key byte array, and returns a `Keypair`. There is zero validation that the keypair matches any expected value. No checksum, no hash comparison, no allowlist.
**Evidence:**
```typescript
function loadKeypair(filePath: string): Keypair {
  const resolved = path.resolve(PROJECT_ROOT, filePath);
  if (!fs.existsSync(resolved)) {
    throw new Error(`Keypair not found: ${resolved}`);
  }
  const secretKey = JSON.parse(fs.readFileSync(resolved, "utf8"));
  return Keypair.fromSecretKey(new Uint8Array(secretKey));
}
```

### Step 2: What gets patched and where
**Location:** `scripts/deploy/patch-mint-addresses.ts:141-200`
**Finding:** Nine patch operations are defined across three Rust files:
1. `programs/conversion-vault/src/constants.rs` — crime_mint, fraud_mint, profit_mint
2. `programs/tax-program/src/constants.rs` — epoch_program_id, staking_program_id, treasury_pubkey
3. `programs/bonding_curve/src/constants.rs` — crime_mint, fraud_mint, epoch_program_id

The `newAddress` values come directly from `keypair.publicKey.toBase58()` with no validation.

### Step 3: File write with no integrity protection
**Location:** `scripts/deploy/patch-mint-addresses.ts:233-236`
**Finding:** Modified file contents are written back to disk unconditionally via `fs.writeFileSync`. No backup is created, no diff is shown for operator review, no checksum is stored for later verification.
**Evidence:**
```typescript
for (const [relPath, content] of fileContents) {
    const filePath = path.resolve(PROJECT_ROOT, relPath);
    fs.writeFileSync(filePath, content);
}
```

### Step 4: Build script invocation
**Location:** `scripts/deploy/build.sh:52-58`
**Finding:** The build script calls the patch script before `anchor build`, but only if the `mint-keypairs` directory exists. The patch script's exit code is not explicitly checked (though `set -e` at line 28 would catch a non-zero exit).
**Evidence:**
```bash
if [ -d "scripts/deploy/mint-keypairs" ]; then
  npx tsx scripts/deploy/patch-mint-addresses.ts
  echo "  done"
else
  echo "  SKIP: No mint-keypairs directory (first deploy will create them)"
fi
```

### Step 5: Post-build verification gap
**Location:** `scripts/verify-program-ids.ts:133-164` (CROSS_REFS array)
**Finding:** The `verify-program-ids.ts` script checks cross-program references but only for **program IDs** (epoch_program_id, staking_program_id, tax_program_id, TAX_PROGRAM_ID). It does **NOT** verify:
- Mint addresses (crime_mint, fraud_mint, profit_mint) in vault or bonding curve constants
- Treasury address (treasury_pubkey) in tax program constants

This is the critical gap: the patched mint addresses and treasury address bypass all post-build verification.

### Step 6: Treasury address from environment
**Location:** `scripts/deploy/patch-mint-addresses.ts:138`
**Finding:** The treasury address comes from `TREASURY_PUBKEY` env var with a hardcoded fallback to the devnet wallet. If the env var is set to an attacker-controlled address, the tax program's treasury would route funds to the attacker.
**Evidence:**
```typescript
const treasuryAddress = process.env.TREASURY_PUBKEY || "8kPzhQoUPx7LYM18f9TzskW4ZgvGyq4jMPYZikqmHMH4";
```

### Step 7: Mint keypair storage security
**Location:** `.gitignore:13`
**Finding:** `scripts/deploy/mint-keypairs/` is gitignored (good — not committed to git). However, the files are created with default permissions (0644 per SEC-01 context finding R-06) and have no file integrity monitoring. The `keypairs/` directory (containing program keypairs) IS committed to git.

## Invariant Analysis

### Target Invariant
Compiled-in mint addresses in on-chain programs must correspond to the legitimate protocol mints (CRIME, FRAUD, PROFIT) and not to attacker-controlled mints.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `scripts/deploy/patch-mint-addresses.ts:35-42` | Keypair file exists | No — existence != integrity |
| `scripts/deploy/build.sh:52-58` | Patch runs before build | Yes — ordering is correct |
| `scripts/verify-program-ids.ts:133-164` | Cross-program ID consistency | **No** — only checks program IDs, not mint addresses |
| `scripts/deploy/verify.ts` | Post-deploy verification | Partial — verifies mints exist on-chain but not that they match compiled-in values |

## Blocking Factors
- **Local-only workflow**: `scripts/deploy/patch-mint-addresses.ts:22` — This runs on the operator's local machine, not in CI/CD. Requires local filesystem compromise.
- **Same keypair files used for init**: `scripts/deploy/initialize.ts:129-131` — The same mint keypair files are used to create the on-chain mints. If an attacker replaces the files, the on-chain mints would match the compiled-in addresses (but they'd be attacker-controlled mints, which the attacker could mint additional supply from before authority burn).
- **set -e in build.sh**: `scripts/deploy/build.sh:28` — If the patch script crashes (e.g., malformed JSON), the build stops.

## Enabling Factors
- **No keypair integrity verification**: `scripts/deploy/patch-mint-addresses.ts:35-42` — `loadKeypair()` accepts any valid Solana keypair file with no expected-value check.
- **No mint address post-build verification**: `scripts/verify-program-ids.ts:133-164` — CROSS_REFS array omits all mint address checks.
- **Treasury from env var with no validation**: `scripts/deploy/patch-mint-addresses.ts:138` — `TREASURY_PUBKEY` env var is used directly with no format validation beyond what `Pubkey::from_str` does at Rust compile time.
- **Default file permissions on mint keypairs**: `scripts/deploy/initialize.ts:168` — Files written with 0644 permissions (world-readable on shared systems).
- **npm supply chain exposure**: `.gitignore:9` — `package-lock.json` is gitignored. A compromised npm dependency's postinstall script could swap keypair files before the build runs.

## Severity Assessment
- **Severity:** HIGH
- **Likelihood:** LOW (requires local machine compromise or supply chain attack)
- **Impact:** If exploited, attacker-controlled mint addresses compiled into vault, tax, and bonding curve programs. Combined with the fact that `initialize.ts` uses the same keypair files, an attacker could create mints they control, deploy programs that reference those mints, and extract value before the operator notices.

## Attack Scenario

```
Step 1: Compromise a keypair file
  Target: scripts/deploy/mint-keypairs/crime-mint.json
  Payload: Replace with attacker-generated keypair JSON
  Prerequisite: Write access to build machine filesystem (via npm postinstall, dev machine compromise, or insider)
  Result: Attacker's keypair is on disk

Step 2: Operator runs deploy-all.sh
  Target: scripts/deploy/build.sh → patch-mint-addresses.ts
  Result: Attacker's crime mint address written into:
    - programs/conversion-vault/src/constants.rs (crime_mint())
    - programs/bonding_curve/src/constants.rs (crime_mint())

Step 3: Programs built and deployed
  Target: anchor build → anchor deploy
  Result: On-chain programs reference attacker's mint address
  Note: verify-program-ids.ts passes (doesn't check mint addresses)

Step 4: Initialize creates attacker-controlled mint
  Target: scripts/deploy/initialize.ts
  Result: On-chain mint created from attacker's keypair. Attacker holds the secret key.
  Note: Mint authority is burned AFTER minting, but attacker could pre-mint or race.

Step 5: Attacker mints additional supply before authority burn
  Target: On-chain mint (attacker has secret key)
  Prerequisite: Attacker submits mint TX between mint creation and authority burn in initialize.ts
  Result: Attacker holds excess token supply

Final Result: Attacker has unbounded supply of one or more protocol tokens. Protocol programs accept these tokens as legitimate because the compiled-in mint addresses match.
```

## Recommended Fix

```typescript
// Before (vulnerable) — patch-mint-addresses.ts:35-42
function loadKeypair(filePath: string): Keypair {
  const resolved = path.resolve(PROJECT_ROOT, filePath);
  if (!fs.existsSync(resolved)) {
    throw new Error(`Keypair not found: ${resolved}`);
  }
  const secretKey = JSON.parse(fs.readFileSync(resolved, "utf8"));
  return Keypair.fromSecretKey(new Uint8Array(secretKey));
}

// After (fixed) — Add integrity verification
function loadKeypair(filePath: string, expectedPubkey?: string): Keypair {
  const resolved = path.resolve(PROJECT_ROOT, filePath);
  if (!fs.existsSync(resolved)) {
    throw new Error(`Keypair not found: ${resolved}`);
  }
  const secretKey = JSON.parse(fs.readFileSync(resolved, "utf8"));
  const keypair = Keypair.fromSecretKey(new Uint8Array(secretKey));

  if (expectedPubkey && keypair.publicKey.toBase58() !== expectedPubkey) {
    throw new Error(
      `Keypair integrity check failed for ${filePath}: ` +
      `expected ${expectedPubkey}, got ${keypair.publicKey.toBase58()}`
    );
  }
  return keypair;
}
```

Additionally, `verify-program-ids.ts` should be extended to verify mint addresses in the CROSS_REFS array:
```typescript
// Add to CROSS_REFS in verify-program-ids.ts
{
  sourceFile: "programs/conversion-vault/src/constants.rs",
  referenceName: "crime_mint()",
  expectedProgram: "crime_mint", // Need to add mint keypairs to PROGRAMS registry
  pattern: /Pubkey::from_str\("([A-Za-z0-9]+)"\)/,
},
// ... same for fraud_mint, profit_mint in vault and bonding_curve
```

## Incidental Discoveries

1. **TREASURY_PUBKEY env var injection**: `scripts/deploy/patch-mint-addresses.ts:138` — The treasury address is taken from an environment variable with a devnet fallback. If `TREASURY_PUBKEY` is set to an attacker address (e.g., via `.env` file tampering), all tax program distributions route to the attacker. No validation that the address is a known/expected treasury. Suggested hypothesis: Investigate env var injection vectors for `TREASURY_PUBKEY` across the deploy pipeline.

2. **Verify script does not check mint addresses**: `scripts/verify-program-ids.ts:133-164` — The CROSS_REFS array only covers program-to-program ID references. Mint address references in vault and bonding curve constants are entirely unverified post-build. This is a verification coverage gap independent of the patch script. Suggested hypothesis: Enumerate all compiled-in addresses that are NOT covered by verify-program-ids.ts.

3. **Race window in initialize.ts between mint creation and authority burn**: If an attacker has the mint keypair (from the file on disk), there is a window between when `initialize.ts` creates the mint and when it burns the mint authority where the attacker could submit a mint transaction. This is an on-chain concern but the attack vector starts off-chain (keypair file access). Suggested hypothesis: Measure the timing window between mint creation and authority burn in initialize.ts.
