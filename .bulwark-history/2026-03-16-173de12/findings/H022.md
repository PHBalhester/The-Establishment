# Finding: H022 - Sell Path Zero AMM Slippage (Cross-Boundary)

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The Tax Program's on-chain `swap_sol_sell` handler passes `minimum_amount_out=0` to the AMM CPI call (line 147 of `swap_sol_sell.rs`), while the buy path correctly forwards the user's `minimum_output` to the AMM (line 262 of `swap_sol_buy.rs`). This creates an asymmetric slippage protection gap on the sell path.

The on-chain 50% output floor (SEC-10, `MINIMUM_OUTPUT_FLOOR_BPS=5000`) checks the user's `minimum_output` parameter before the CPI, not the AMM's actual execution output. The Tax Program then checks `net_output >= minimum_output` after the CPI and tax deduction. However, during the AMM CPI execution itself, the AMM accepts ANY output above 0. This means an MEV bot can sandwich the AMM swap step to extract value in the gap between the 50% floor and the user's actual slippage tolerance.

The off-chain code correctly computes and passes a proper `minimumOutput` value. The vulnerability is that the on-chain Tax Program discards this value when calling the AMM, and the off-chain layer cannot compensate for this on-chain design choice. The user's slippage protection is enforced post-swap (after MEV extraction has already occurred within the AMM), and the 50% floor is the only pre-swap guardrail.

## Hypothesis Investigated

Tax Program passes `minimum_amount_out=0` to AMM on the sell path. The 50% output floor is the only protection, creating a gap between 50% and the user's actual minimum that is extractable by MEV.

## Investigation Path

### Step 1: Off-chain minimumOutput computation (sell path)
**Location:** `app/hooks/useSwap.ts:432`
**Finding:** The client correctly computes minimumOutput using the user's slippage tolerance.
**Evidence:**
```typescript
const minimumOutput = Math.floor(result.outputSol * (10_000 - slippageBps) / 10_000);
```
For a user with default 5% slippage (500 BPS), minimumOutput = 95% of expected output.

### Step 2: Off-chain transaction construction (sell path)
**Location:** `app/lib/swap/swap-builders.ts:365-369`
**Finding:** The client correctly passes minimumOutput to the on-chain instruction.
**Evidence:**
```typescript
const swapIx = await taxProgram.methods
  .swapSolSell(
    new BN(amountInBaseUnits),
    new BN(minimumOutput),  // User's slippage-adjusted minimum
    isCrime,
  )
```

### Step 3: On-chain sell path -- floor check
**Location:** `programs/tax-program/src/instructions/swap_sol_sell.rs:112-118`
**Finding:** The on-chain code enforces the 50% output floor on the user's `minimum_output` parameter BEFORE the AMM CPI. This prevents submitting `minimum_output=0` but allows anything >= 50% of expected output.
**Evidence:**
```rust
let output_floor = calculate_output_floor(reserve_b, reserve_a, amount_in, MINIMUM_OUTPUT_FLOOR_BPS)
    .ok_or(error!(TaxError::TaxOverflow))?;
require!(
    minimum_output >= output_floor,
    TaxError::MinimumOutputFloorViolation
);
```

### Step 4: On-chain sell path -- AMM CPI with zero slippage
**Location:** `programs/tax-program/src/instructions/swap_sol_sell.rs:146-154`
**Finding:** The Tax Program passes `amm_minimum: u64 = 0` to the AMM CPI. The AMM will accept any output above zero.
**Evidence:**
```rust
// Pass 0 for AMM's minimum_amount_out - we check slippage ourselves after tax
let amm_minimum: u64 = 0;

let mut ix_data = Vec::with_capacity(25);
ix_data.extend_from_slice(&discriminator);
ix_data.extend_from_slice(&amount_in.to_le_bytes());
ix_data.extend_from_slice(&[direction]);
ix_data.extend_from_slice(&amm_minimum.to_le_bytes());
```

### Step 5: On-chain sell path -- post-swap slippage check
**Location:** `programs/tax-program/src/instructions/swap_sol_sell.rs:238-245`
**Finding:** After the AMM CPI executes (with 0 minimum), the Tax Program calculates gross output from balance difference, deducts tax, and THEN checks `net_output >= minimum_output`. This check is correct but comes after the AMM has already executed at whatever price the MEV bot manipulated the pool to.
**Evidence:**
```rust
let net_output = gross_output
    .checked_sub(tax_amount)
    .ok_or(error!(TaxError::TaxOverflow))?;
require!(net_output > 0, TaxError::InsufficientOutput);
require!(net_output >= minimum_output, TaxError::SlippageExceeded);
```

### Step 6: Comparison with buy path
**Location:** `programs/tax-program/src/instructions/swap_sol_buy.rs:262`
**Finding:** The buy path correctly passes the user's `minimum_output` to the AMM CPI. This is an asymmetry -- buy has two layers of protection (AMM check + Tax Program check), sell has only one (Tax Program post-check).
**Evidence:**
```rust
ix_data.extend_from_slice(&minimum_output.to_le_bytes());
```

### Step 7: Off-chain route engine (multi-hop sell)
**Location:** `app/lib/swap/multi-hop-builder.ts:315-323`
**Finding:** Multi-hop routes also compute per-step minimumOutput, which is passed to `buildStepTransaction` and ultimately to `buildSolSellTransaction`. The same on-chain vulnerability applies to multi-hop sell legs.

## Invariant Analysis

### Target Invariant
User should receive at least `minimumOutput` SOL after tax on every sell transaction. The AMM execution step should be constrained to prevent unnecessary value extraction.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `useSwap.ts:432` | Computes minimumOutput = expectedOutput * (10000 - slippageBps) / 10000 | Yes (off-chain) |
| `swap-builders.ts:368` | Passes minimumOutput to on-chain instruction | Yes (off-chain) |
| `swap_sol_sell.rs:116` | Rejects if minimumOutput < 50% of expected AMM output | Yes (on-chain, pre-CPI) |
| `swap_sol_sell.rs:147` | Passes 0 to AMM CPI | **NO** -- AMM has no slippage protection during execution |
| `swap_sol_sell.rs:245` | Checks net_output >= minimumOutput after tax | Yes (on-chain, post-CPI) |

The post-CPI check at line 245 is the actual enforcement. The attack extracts value between the "no AMM minimum" and the user's minimum, constrained by the fact that the post-check must still pass.

## Blocking Factors
- **Post-CPI slippage check:** `swap_sol_sell.rs:245` -- The Tax Program checks `net_output >= minimum_output` AFTER the swap. This prevents the user from receiving less than their stated minimum.
- **50% output floor:** `swap_sol_sell.rs:113-118` -- Prevents `minimum_output=0` from being submitted at all.

## Enabling Factors
- **Zero AMM minimum:** `swap_sol_sell.rs:147` -- The AMM accepts any output, giving MEV bots room to manipulate the pool during the CPI.
- **No MEV protection:** `app/hooks/useProtocolWallet.ts:111` -- All transactions go through standard Helius RPC (confirmed in H015).
- **Default 5% slippage:** `app/providers/SettingsProvider.tsx:77` -- 500 BPS default means the extractable gap is up to ~5% of the expected output.
- **Sell path asymmetry:** Buy path passes real minimum to AMM (`swap_sol_buy.rs:262`), sell path passes 0 (`swap_sol_sell.rs:147`).

## If CONFIRMED

### Severity Assessment
- **Severity:** MEDIUM
- **Likelihood:** HIGH (MEV bots are automated and ubiquitous on Solana)
- **Impact:** Partial fund loss -- user receives their stated minimum but not the optimal execution they would have received if the AMM had proper slippage protection. The extractable amount is bounded by `expectedOutput - minimumOutput` (e.g., 5% with default slippage). This is the same sandwich exposure as H015 but with the additional nuance that the sell path has weaker AMM-level protection than the buy path.

### Why MEDIUM not HIGH
The post-CPI slippage check (`swap_sol_sell.rs:245`) ensures the user always receives at least their stated `minimum_output`. The user is not losing funds beyond their expressed slippage tolerance. The concern is that the AMM executes with 0 minimum during the CPI, creating an unnecessary MEV extraction window that would not exist if the Tax Program calculated and passed an appropriate AMM-level minimum. On the buy path, this is handled correctly.

### Attack Scenario

```
Step 1: User submits CRIME -> SOL sell for 100,000 CRIME tokens
  Target: Tax Program swap_sol_sell
  Expected output: 1.0 SOL after tax
  User's slippage: 5% (default)
  minimumOutput: 0.95 SOL (950,000,000 lamports)

Step 2: On-chain floor check passes
  output_floor = 50% of expected AMM output ~ 0.526 SOL
  0.95 SOL >= 0.526 SOL -> PASS

Step 3: MEV bot observes TX in mempool, front-runs with pool manipulation
  Bot buys SOL from pool, moving the price against the seller

Step 4: AMM CPI executes with minimum_amount_out=0
  Due to pool manipulation, gross AMM output = 1.06 SOL (instead of ~1.11 SOL)
  After tax deduction: net_output = 1.06 - 0.106 = 0.954 SOL

Step 5: Post-CPI check passes
  0.954 SOL >= 0.95 SOL -> PASS

Step 6: MEV bot back-runs to restore pool price, profits ~0.05 SOL

Final Result: User receives 0.954 SOL instead of ~1.0 SOL (4.6% extraction)
```

### Recommended Fix

The off-chain code is correct -- it computes and passes a proper `minimumOutput`. The fix is on-chain in the Tax Program's sell handler. The Tax Program should compute an appropriate AMM-level minimum to pass to the CPI, rather than 0.

**Off-chain mitigation (partial):** Lower the default slippage from 500 BPS to 100 BPS (1%) to reduce the extractable window:

```typescript
// Before (app/providers/SettingsProvider.tsx:77)
const DEFAULT_SLIPPAGE_BPS = 500; // 5%

// After (recommended)
const DEFAULT_SLIPPAGE_BPS = 100; // 1%
```

**Off-chain mitigation (complementary):** Integrate MEV-protected transaction submission (Jito bundles or private mempool) as recommended in H015.

**On-chain fix (definitive):** Calculate and pass a gross output minimum to the AMM CPI:

```rust
// Before (swap_sol_sell.rs:147)
let amm_minimum: u64 = 0;

// After: compute AMM minimum from user's minimum_output + tax
// If user wants at least minimum_output after tax,
// then gross AMM output must be at least:
// gross_min = ceil(minimum_output * 10000 / (10000 - tax_bps))
let gross_min = minimum_output
    .checked_mul(10000)
    .and_then(|v| v.checked_add(10000u64.checked_sub(tax_bps as u64).unwrap_or(1) - 1))
    .and_then(|v| v.checked_div(10000u64.checked_sub(tax_bps as u64).unwrap_or(1)))
    .unwrap_or(minimum_output);
let amm_minimum: u64 = gross_min;
```

## Related Findings
- **H015 (Sandwich Attack on User Swaps):** This finding is a deepening of H015's analysis. H015 identifies the lack of MEV protection on all swap paths. H022 identifies the specific on-chain mechanism that makes the sell path MORE vulnerable than the buy path -- the AMM has zero slippage protection during sell CPI execution.
- The combination of H015 (no MEV-protected RPC) + H022 (zero AMM minimum on sell) + default 5% slippage creates a particularly exploitable sell path.

## Incidental Discoveries
- **Buy/sell asymmetry is intentional but poorly documented:** The on-chain code comments at `swap_sol_sell.rs:146` say "we check slippage ourselves after tax" as justification for passing 0. This is a conscious design decision but creates an unnecessary MEV window. The buy path demonstrates the correct pattern (passing the user's minimum to the AMM), proving it is technically feasible to do the same for sell.
- **E2E test scripts use minimumOutput=1 as fallback:** `scripts/e2e/lib/swap-flow.ts:408` falls back to `minimumOutput = 1` when reserve reads fail. This is test-only code but demonstrates the mindset around loose minimums. No production impact since the code is in `scripts/`.
