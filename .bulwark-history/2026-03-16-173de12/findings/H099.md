# Finding: H099 - No Max Trade Size Limit

## Status: POTENTIAL
## Confidence Score: 7

## Rationale
Neither `app/hooks/useSwap.ts` nor `app/lib/swap/quote-engine.ts` enforce a maximum trade size limit. The quote engine accepts any `number` value for input amounts and computes output using the constant-product formula. There is no client-side cap like "max 10 SOL per swap" or "max 50% of pool reserves per trade." The on-chain programs enforce slippage (via `minimum_output` parameter) but not trade size. A user could submit a swap for the entire pool reserve, causing massive price impact. However, the constant-product formula naturally punishes large trades with diminishing returns, and the slippage protection set by the user would likely cause the transaction to fail if price impact is too high. The `useSwap.ts` only has a `maxRetries: 2` reference (line 748) but no input size validation beyond balance checks.

## Code Evidence
```typescript
// app/lib/swap/quote-engine.ts:54-62 - no size guard
export function calculateSwapOutput(
  reserveIn: number,
  reserveOut: number,
  effectiveInput: number,
): number {
  const denominator = reserveIn + effectiveInput;
  if (denominator === 0) return 0;
  return Math.floor((reserveOut * effectiveInput) / denominator);
}
```
Accepts any effectiveInput value, no upper bound check.

## Severity Estimate
LOW -- The constant-product AMM formula naturally limits extraction (you can never get more than the reserve). On-chain slippage protection prevents unexpectedly bad execution. The absence of a max trade size is standard for AMM protocols. A whale draining a pool is an economic concern, not a bug.
