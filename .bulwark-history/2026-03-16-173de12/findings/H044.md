# Finding: H044 - Sentry DSN Spam via Client Exposure

## Status: POTENTIAL
## Confidence Score: 6

## Executive Summary

The Sentry DSN is exposed via `NEXT_PUBLIC_SENTRY_DSN` (a Next.js public env var, meaning it's bundled into client JavaScript). The DSN contains the public key, project ID, and ingest hostname. Any attacker can extract this from the browser bundle and flood the Sentry project with garbage error events using simple HTTP POST requests to the envelope API.

This is a known, accepted pattern in all Sentry deployments -- DSNs are designed to be public. However, the custom zero-dependency Sentry reporter in `lib/sentry.ts` lacks client-side rate limiting, and there's no evidence of server-side Sentry rate limits or abuse detection being configured. For a project not yet at scale, the practical impact is minimal: Sentry's free tier has event quotas, so excessive spam would simply exhaust the monthly quota, causing legitimate errors to be dropped.

## Hypothesis Investigated

Exposed Sentry DSN allows flooding with garbage events.

## Investigation Path

### Step 1: Verify DSN exposure
**Location:** `app/instrumentation-client.ts:37`
**Finding:** DSN is read from `process.env.NEXT_PUBLIC_SENTRY_DSN` -- the `NEXT_PUBLIC_` prefix means it's embedded in the client bundle.
**Evidence:**
```typescript
if (typeof window !== "undefined" && process.env.NEXT_PUBLIC_SENTRY_DSN) {
```

### Step 2: Examine Sentry reporter
**Location:** `app/lib/sentry.ts:26-96`
**Finding:** Zero-dependency reporter. Fire-and-forget `fetch()` to `https://{host}/api/{projectId}/envelope/?sentry_key={key}&sentry_version=7`. No client-side rate limiting, no deduplication, no session tracking.
**Evidence:**
```typescript
const ingestUrl = `https://${host}/api/${projectId}/envelope/?sentry_key=${key}&sentry_version=7`;
fetch(ingestUrl, {
  method: "POST",
  body: envelope,
  headers: { "Content-Type": "text/plain;charset=UTF-8" },
}).catch(() => {});
```

### Step 3: Assess what an attacker can do with the DSN
**Finding:** With the DSN, an attacker can:
1. Extract the ingest URL, public key, and project ID from the client bundle
2. POST arbitrary error envelopes to the Sentry ingest API
3. Flood the project with fake errors, exhausting the event quota
4. This causes legitimate errors to be dropped (quota exceeded)

### Step 4: Check for rate limiting or abuse prevention
**Location:** `app/lib/sentry.ts` (entire file)
**Finding:** No client-side rate limiting. No deduplication. No CORS restrictions (Sentry ingest accepts any origin). No server-side configuration visible in the codebase.

## If CONFIRMED

### Severity Assessment
- **Severity:** LOW

### Attack Scenario
```
Step 1: Attacker views page source or browser devtools
  Target: Client JS bundle
  Result: Extracts NEXT_PUBLIC_SENTRY_DSN value

Step 2: Attacker parses DSN: https://<key>@<host>/<project_id>
  Result: Has ingest URL, public key, project ID

Step 3: Attacker runs: for i in $(seq 1 100000); do curl -X POST ...; done
  Target: Sentry ingest API
  Result: 100K garbage events submitted

Step 4: Sentry monthly quota exhausted
  Result: All subsequent legitimate errors are dropped for the billing period

Final Result: Team loses visibility into production errors. Real bugs go unnoticed.
```

### Recommended Fix

Add client-side rate limiting to `lib/sentry.ts`:

```typescript
// Before: no rate limiting
export function captureException(error: unknown, dsn?: string) {
  // ... immediately sends

// After: simple rate limiter
let eventCount = 0;
let windowStart = Date.now();
const MAX_EVENTS_PER_MINUTE = 10;

export function captureException(error: unknown, dsn?: string) {
  const now = Date.now();
  if (now - windowStart > 60_000) {
    eventCount = 0;
    windowStart = now;
  }
  if (++eventCount > MAX_EVENTS_PER_MINUTE) return;
  // ... rest of function
```

Additionally, configure server-side Sentry rate limits and abuse detection in the Sentry project settings.

## Related Findings

## Incidental Discoveries
- `lib/sentry.ts:76` includes the full DSN in every envelope header: `dsn: sentryDsn`. While this is standard Sentry protocol, it means every error event payload contains the full DSN string. If error events are ever logged or forwarded, the DSN leaks through those channels too. Low concern since DSNs are public by design.
