# Finding: H009 - Devnet Fallback Causing Mainnet Transaction Failure

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The frontend RPC connection factory at `app/lib/connection.ts:33` and the root provider at `app/providers/providers.tsx:35` both use the same fallback chain: `NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL`. The `DEVNET_RPC_URL` is a hardcoded Helius devnet endpoint exported from `shared/programs.ts:22`. If `NEXT_PUBLIC_RPC_URL` is not set in a mainnet Railway deployment, the entire frontend silently connects to devnet. This affects all transaction submissions (via `useProtocolWallet.ts` sign-then-send through `connection.sendRawTransaction`), all on-chain state reads (pool reserves, balances, epoch state), and all WebSocket subscriptions. There is no startup validation, no runtime check, and no user-visible indicator that the wrong network is in use.

This is not a theoretical concern. There are zero guardrails preventing this misconfiguration. The project has no centralized env var validation (no envalid, no Zod schema), and the `NEXT_PUBLIC_RPC_URL` variable is documented as "unset, falls back to shared" in the mainnet checklist -- confirming the fallback path is the current default behavior.

## Hypothesis Investigated

If `NEXT_PUBLIC_RPC_URL` is not set in a mainnet Railway deployment, the frontend silently falls back to devnet RPC, causing mainnet-signed transactions to be sent to the wrong network.

## Investigation Path

### Step 1: Traced the RPC URL resolution chain
**Location:** `app/lib/connection.ts:31-33`
**Finding:** The `getConnection()` function resolves the RPC URL via nullish coalescing: `rpcUrl ?? process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL`. No callers pass `rpcUrl` in production -- all 20+ call sites use `getConnection()` with no arguments.
**Evidence:**
```typescript
// app/lib/connection.ts:31-33
export function getConnection(rpcUrl?: string): Connection {
  const url =
    rpcUrl ?? process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL;
```

### Step 2: Confirmed the same pattern in the provider tree
**Location:** `app/providers/providers.tsx:35`
**Finding:** The root `ConnectionProvider` (wallet adapter) uses the identical fallback: `process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL`. This means the wallet adapter's connection context AND the standalone `getConnection()` singleton both resolve to devnet when the env var is missing.
**Evidence:**
```typescript
// app/providers/providers.tsx:34-35
  // MAINNET TODO: Switch to WalletAdapterNetwork.MainnetBeta + mainnet RPC URL.
  const endpoint = process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL;
```

### Step 3: Confirmed the hardcoded devnet URL
**Location:** `shared/programs.ts:21-22`
**Finding:** `DEVNET_RPC_URL` is a string literal pointing to `https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-KEY]-...`. This is a devnet-specific Helius endpoint. Mainnet-signed transactions sent here would be rejected by the devnet validator cluster (wrong genesis hash).
**Evidence:**
```typescript
// shared/programs.ts:21-22
export const DEVNET_RPC_URL =
  "https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-HELIUS-KEY]";
```

### Step 4: Confirmed transaction submission uses this connection
**Location:** `app/hooks/useSwap.ts:678`, `app/hooks/useSwap.ts:838`, `app/components/launch/BuyForm.tsx:164`, `app/components/launch/SellForm.tsx:172`
**Finding:** All swap and bonding curve operations call `getConnection()` to obtain the Connection, then pass it to `wallet.sendTransaction(tx, connection, ...)`. The `useProtocolWallet.ts:111` sign-then-send flow calls `connection.sendRawTransaction(serialized, ...)`. If the connection points to devnet, the signed mainnet transaction is sent to devnet validators, which reject it (wrong cluster).

### Step 5: Confirmed no startup validation exists
**Location:** Entire `app/` directory
**Finding:** No centralized env var validation library (envalid, Zod) is used. No startup check verifies that `NEXT_PUBLIC_RPC_URL` is set. No runtime check validates the RPC endpoint responds on the expected cluster. The `NEXT_PUBLIC_` prefix means this is a build-time Next.js variable baked into the client bundle -- if missing at build time, the fallback is permanently compiled in.

### Step 6: Confirmed no network mismatch detection
**Finding:** The wallet adapter signs transactions against whatever network the wallet is configured for (mainnet). The Connection object sends those transactions to whatever RPC endpoint it was given (devnet, if fallback). There is no check comparing the wallet's cluster to the Connection's cluster. The transaction simply fails silently on the devnet RPC.

## Enabling Factors
- **Silent fallback**: `app/lib/connection.ts:33` -- nullish coalescing silently resolves to devnet without logging or warning
- **No env validation**: No centralized schema validates required env vars at build or startup time
- **Build-time baking**: `NEXT_PUBLIC_` vars are replaced at build time by Next.js. If not set during `npm run build` on Railway, the fallback is permanently compiled into the JS bundle
- **No cluster indicator in UI**: No visual indicator shows which cluster the app is connected to
- **Documented as TODO**: `app/providers/providers.tsx:34` has `// MAINNET TODO` comment, confirming this is a known gap

### Attack Scenario
This is not an adversarial attack but an operational misconfiguration with severe impact:

```
Step 1: Admin deploys to Railway for mainnet
  Target: Railway deployment configuration
  Action: Sets DATABASE_URL, WALLET_KEYPAIR, etc. but forgets NEXT_PUBLIC_RPC_URL
  Prerequisite: Normal human error during deployment
  Result: Next.js build compiles DEVNET_RPC_URL into client bundle

Step 2: Users visit the mainnet site
  Target: Frontend at production URL
  Action: Page loads, hooks call getConnection() which returns devnet Connection
  Result: All pool reserves, balances, epoch state read from devnet (stale/empty)

Step 3: Users attempt a swap
  Target: useSwap.ts -> useProtocolWallet.ts -> connection.sendRawTransaction
  Action: Wallet signs a mainnet TX, frontend submits to devnet RPC
  Result: Devnet rejects TX (wrong genesis hash). User sees "Transaction failed"

Step 4: Users see devnet prices as if they were real
  Target: usePoolPrices, chart data
  Action: Devnet pool reserves (likely empty or test values) displayed as mainnet prices
  Result: Users make decisions based on completely wrong data

Final Result: Protocol appears broken. No fund loss, but complete service disruption
and user confusion. Reputational damage. Requires redeployment with correct env var.
```

### Recommended Fix

**Fix 1: Fail-fast on missing RPC URL (immediate)**
```typescript
// Before (vulnerable) -- app/lib/connection.ts:31-33
export function getConnection(rpcUrl?: string): Connection {
  const url =
    rpcUrl ?? process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL;

// After (fixed) -- app/lib/connection.ts:31-37
export function getConnection(rpcUrl?: string): Connection {
  const url = rpcUrl ?? process.env.NEXT_PUBLIC_RPC_URL;
  if (!url) {
    throw new Error(
      "NEXT_PUBLIC_RPC_URL is not set. Cannot connect to Solana. " +
      "Set this env var before building the app."
    );
  }
```

**Fix 2: Same pattern in providers.tsx**
```typescript
// Before (vulnerable) -- app/providers/providers.tsx:35
const endpoint = process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL;

// After (fixed) -- app/providers/providers.tsx:35-40
const endpoint = process.env.NEXT_PUBLIC_RPC_URL;
if (!endpoint) {
  throw new Error(
    "NEXT_PUBLIC_RPC_URL is not set. The app cannot determine which Solana cluster to use."
  );
}
```

**Fix 3 (defense-in-depth): Runtime cluster validation**
```typescript
// Add to app startup or getConnection():
// After creating the connection, verify the genesis hash matches expected cluster
const genesisHash = await connection.getGenesisHash();
const MAINNET_GENESIS = "5eykt4UsFv8P8NJdTREpY1vzqKqZKvdpKuc147dw2N9d";
const DEVNET_GENESIS = "EtWTRABZaYq6iMfeYKouRu166VU2xqa1wcaWoxPkrZBG";
if (expectedCluster === "mainnet-beta" && genesisHash !== MAINNET_GENESIS) {
  throw new Error(`RPC endpoint returned ${genesisHash}, expected mainnet genesis hash`);
}
```

**Fix 4: Keep DEVNET_RPC_URL only for local development**
Move the devnet fallback to a `.env.local` file that is never deployed to Railway, rather than hardcoding it in shared source code.
