# Finding: H104 - EpochState Layout Coupling

## Status: POTENTIAL
## Confidence Score: 7

## Rationale
The Tax Program reads AMM PoolState reserves using raw byte offsets rather than deserializing the struct (`programs/tax-program/src/helpers/pool_reader.rs`). Offsets are hardcoded: `data[137..145]` for reserve_a and `data[145..153]` for reserve_b. The same pattern is used in the Epoch Program's `execute_carnage_atomic.rs` (referenced in the pool_reader comment at line 7). While the pool_reader.rs comments document the expected layout and there is a minimum size check (`data.len() >= 153`), any change to the AMM PoolState struct layout (adding/removing/reordering fields before the reserve fields) would silently produce incorrect reserve values without any compile-time or runtime error. This is cross-program coupling via magic numbers rather than shared types or versioned layouts.

## Code Evidence
```rust
// tax-program/src/helpers/pool_reader.rs:46-55
let reserve_a = u64::from_le_bytes(
    data[137..145]
        .try_into()
        .map_err(|_| error!(TaxError::TaxOverflow))?,
);
let reserve_b = u64::from_le_bytes(
    data[145..153]
        .try_into()
        .map_err(|_| error!(TaxError::TaxOverflow))?,
);
```

## Severity Estimate
MEDIUM -- Silent data corruption if AMM struct layout changes. The AMM is controlled by the same team, but this creates a fragile coupling. A layout version byte or discriminator check would reduce risk.
