# Finding: H006 - Webhook Timing Attack on Secret Comparison

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The webhook endpoint at `app/app/api/webhooks/helius/route.ts:138` uses JavaScript's `!==` operator for secret comparison rather than `crypto.timingSafeEqual`. While this is technically a non-constant-time comparison, the attack is not practically exploitable over HTTP. Network jitter (typically 1-50ms on Railway's infrastructure) dwarfs the nanosecond-level timing differences produced by V8's string comparison short-circuiting. Statistical extraction of the secret would require millions of requests with sub-microsecond timing precision, which is infeasible over a TLS-encrypted HTTP connection through Railway's reverse proxy.

This is a defense-in-depth best-practice improvement, not an exploitable vulnerability. The fix is trivial and recommended regardless.

## Hypothesis Investigated

The webhook secret comparison uses `!==` (not timing-safe), potentially allowing an attacker to progressively leak the secret byte-by-byte by measuring response timing differences, eventually reconstructing the full webhook secret and using it to forge authenticated webhook requests.

## Investigation Path

### Step 1: Confirm the vulnerable comparison pattern
**Location:** `app/app/api/webhooks/helius/route.ts:135-141`
**Finding:** The `!==` comparison is confirmed. The secret is read from `process.env.HELIUS_WEBHOOK_SECRET` and compared directly to the `Authorization` header value.
**Evidence:**
```typescript
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (webhookSecret) {
  const authHeader = req.headers.get("authorization");
  if (authHeader !== webhookSecret) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
}
```

### Step 2: Analyze V8's string comparison behavior
**Location:** V8 engine internals (JavaScript runtime)
**Finding:** V8's `!==` for strings performs: (1) reference equality check, (2) length comparison, (3) byte-by-byte content comparison. The length check short-circuits immediately if lengths differ -- this is the primary information leak. Content comparison uses memcmp-like semantics that short-circuit on the first differing byte. In theory, an attacker could determine: (a) the secret length, then (b) each byte position.

### Step 3: Assess network noise vs. signal
**Location:** Infrastructure analysis (Railway PaaS hosting)
**Finding:** The timing signal from string comparison is on the order of nanoseconds (1-10ns per byte difference). The noise sources include:
- **TLS handshake variance**: 5-50ms
- **Railway reverse proxy**: 1-10ms variance per request
- **Node.js event loop**: 0.1-5ms variance depending on garbage collection, other request handling
- **Internet RTT variance**: 1-50ms depending on path

The signal-to-noise ratio is approximately 1:1,000,000. Standard statistical methods (Crosby's box test, Brumley-Boneh) require the signal to be at least 1:1000 of the noise to be extractable with ~10,000 samples. At 1:1,000,000, extracting a single byte would require on the order of 10^12 measurements -- infeasible.

### Step 4: Evaluate the early-return path
**Location:** `app/app/api/webhooks/helius/route.ts:138-139`
**Finding:** The comparison failure returns immediately (`NextResponse.json`), while a successful comparison proceeds to `req.json()` (body parsing), database operations, SSE broadcasts, etc. This means the timing difference between auth-fail and auth-success is huge (microseconds vs. potentially seconds). However, this only leaks whether auth succeeded or failed -- not byte-by-byte information. An attacker who gets a 401 already knows the header was wrong. The useful timing signal is only between different wrong values (e.g., "aaa" vs "aab"), which would both take the fast-fail path with only nanosecond-level differences.

### Step 5: Evaluate length leakage specifically
**Location:** V8 string comparison internals
**Finding:** The length check is the most practically extractable signal. If the attacker varies the length of their Authorization header, the comparison against a wrong-length string terminates slightly faster (one fewer branch) than a same-length-but-wrong-content string. However, the timing difference is still < 10ns, well below the noise floor.

## Recommended Fix (Defense-in-Depth)

Despite being NOT VULNERABLE, the fix is trivial and should be applied as defense-in-depth:

```typescript
// Before (technically non-constant-time)
if (authHeader !== webhookSecret) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}

// After (constant-time comparison)
import { timingSafeEqual } from "crypto";

function safeCompare(a: string, b: string): boolean {
  if (a.length !== b.length) {
    // Length difference leaks length info, but we need to handle it.
    // Compare against a dummy to maintain constant time for the compare step.
    const dummy = Buffer.alloc(Buffer.byteLength(a));
    timingSafeEqual(Buffer.from(a), dummy);
    return false;
  }
  return timingSafeEqual(Buffer.from(a), Buffer.from(b));
}

if (!authHeader || !safeCompare(authHeader, webhookSecret)) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```
