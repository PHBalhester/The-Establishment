# Finding: H047 - Single RPC Provider No Failover

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The entire protocol -- frontend, crank runner, and all scripts -- depends exclusively on Helius as its sole RPC provider with zero failover logic. The frontend (`app/lib/connection.ts`) creates a singleton Connection to a single Helius URL. The crank runner (`scripts/crank/crank-provider.ts`) similarly connects to a single `CLUSTER_URL`. If Helius experiences downtime, rate-limits the API key, or the API key is revoked (already exposed in client JS), the entire protocol halts: no user swaps, no epoch transitions, no Carnage execution, no chart data.

This is a confirmed operational vulnerability. Helius outages have occurred historically, and the exposed API key (documented in ARCHITECTURE.md) makes rate-limit exhaustion a trivially achievable attack.

## Hypothesis Investigated

Entire protocol depends on Helius with no failover.

## Investigation Path

### Step 1: Frontend RPC Connection
**Location:** `app/lib/connection.ts:31-52`
**Finding:** Single URL, no failover, no retry logic. Falls back to devnet URL if env var unset.
**Evidence:**
```typescript
export function getConnection(rpcUrl?: string): Connection {
  const url =
    rpcUrl ?? process.env.NEXT_PUBLIC_RPC_URL ?? DEVNET_RPC_URL;
  // ... returns singleton Connection to this single URL
}
```

### Step 2: Crank Runner RPC Connection
**Location:** `scripts/crank/crank-provider.ts:35`
**Finding:** Single URL from `CLUSTER_URL` env var, defaults to localhost. No failover.
**Evidence:**
```typescript
const url = process.env.CLUSTER_URL || "http://localhost:8899";
```

### Step 3: Crank Runner Error Handling
**Location:** `scripts/crank/crank-runner.ts:302-314`
**Finding:** On error, the crank retries after 30 seconds with the same single RPC. No circuit breaker, no alternate endpoint.
**Evidence:**
```typescript
} catch (err) {
  // Don't exit -- sleep and retry next cycle
  if (!shutdownRequested) {
    await sleep(ERROR_RETRY_DELAY_MS); // 30 seconds, same RPC
  }
}
```

### Step 4: WebSocket Subscriptions
**Location:** `app/hooks/usePoolPrices.ts:217`
**Finding:** WebSocket subscriptions also go through the same singleton Connection. No reconnect logic beyond what `@solana/web3.js` provides internally.

## Blocking Factors
- None -- there are no failover mechanisms anywhere in the codebase.

## Enabling Factors
- Helius API key exposed in client bundle: allows targeted rate-limit exhaustion
- Single env var (`NEXT_PUBLIC_RPC_URL` / `CLUSTER_URL`) with no secondary/tertiary fallback
- Crank infinite retry on same endpoint: wastes compute without switching providers
