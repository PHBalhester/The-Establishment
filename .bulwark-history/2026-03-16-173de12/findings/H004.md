# Finding: H004 - Crank Wallet Key Compromise via Railway Env Var

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The crank wallet private key is stored as a plaintext JSON byte array in the Railway environment variable `WALLET_KEYPAIR`. This key is the sole signing authority for all crank operations: epoch transitions, VRF commit/reveal, Carnage execution, and vault SOL top-ups. The key is accessible to anyone with Railway dashboard access, could leak through Railway build/deploy logs, and is vulnerable to supply chain attacks on the `@solana/web3.js` or `@coral-xyz/anchor` packages that process it. There are no mitigations: no spending limits, no address allowlist, no multi-sig, no HSM/KMS integration, and no alerting if the key is used from an unexpected source.

The finding is CONFIRMED because the attack vector is real and the impact (fund loss from crank wallet + ability to sign malicious epoch transitions) is concrete. The only question is the likelihood of Railway dashboard compromise, which is a platform-level concern outside the codebase's control -- but the codebase provides zero defense-in-depth against it.

## Hypothesis Investigated

The crank wallet private key stored as a Railway environment variable could be extracted via Railway dashboard access compromise, log leakage, or supply chain attack.

## Investigation Path

### Step 1: WALLET_KEYPAIR Parsing and Storage
**Location:** `scripts/crank/crank-provider.ts:41-57`
**Finding:** The full 64-byte secret key (32 bytes private + 32 bytes public) is stored as a JSON array string in the `WALLET_KEYPAIR` environment variable. It is parsed directly into a `Keypair` object with no intermediate encryption, no key derivation, and no HSM integration.
**Evidence:**
```typescript
const keypairEnv = process.env.WALLET_KEYPAIR;
// ...
const secretKey = JSON.parse(keypairEnv);
wallet = new anchor.Wallet(
  Keypair.fromSecretKey(new Uint8Array(secretKey))
);
```

The parsed `Uint8Array` containing the secret key is never zeroed after the `Keypair` is constructed. It persists in the Node.js heap for the lifetime of the crank process.

### Step 2: What the Compromised Key Can Do
**Location:** `scripts/crank/crank-runner.ts:197-316`
**Finding:** The crank wallet signs ALL of the following operations:
- Epoch transitions (`trigger_epoch_transition`)
- VRF commit/reveal/consume (creates randomness, commits, reveals)
- Carnage execution (`executeCarnageAtomic`)
- Vault SOL top-ups (`SystemProgram.transfer` at line 230-236)
- Any future operations added to the crank loop

The vault top-up at line 225-241 is particularly dangerous: it transfers SOL from the crank wallet to the CarnageSolVault with no per-cycle or per-day limit. An attacker with the key could drain the wallet via direct SOL transfers to any address.

### Step 3: Log Leakage Vectors
**Location:** `scripts/crank/crank-runner.ts:176-177`
**Finding:** The crank logs the RPC URL raw to stdout:
```typescript
console.log(`  Wallet: ${provider.wallet.publicKey.toBase58()}`);
console.log(`  RPC: ${process.env.CLUSTER_URL || "localhost"}`);
```

Line 177 logs `CLUSTER_URL` which contains the Helius API key (confirmed in `.env:5`: `CLUSTER_URL=https://devnet.helius-rpc.com/?api-key=[REDACTED-DEVNET-KEY]...`). While this does not directly leak the wallet key, it demonstrates that sensitive environment variables are logged without redaction. If a code change or error handler were to log `process.env` or iterate env vars, the wallet key would appear in Railway logs.

The error handler at line 303 truncates errors to 300 chars: `String(err).slice(0, 300)`. While this prevents full stack trace leakage, if the `WALLET_KEYPAIR` parsing fails (line 55), the error message is truncated to 100 chars: `String(err).slice(0, 100)`. The JSON.parse error for a malformed keypair would not include the keypair value itself (JSON.parse errors reference the parse position, not the input), so this specific path is safe.

### Step 4: Error Path Analysis for Key Leakage
**Location:** `scripts/crank/crank-provider.ts:53-56`, `scripts/crank/crank-runner.ts:328-330`
**Finding:** Two error paths handle the keypair:
1. Parse failure at `crank-provider.ts:55`: `Failed to parse WALLET_KEYPAIR env var: ${String(err).slice(0, 100)}` -- Safe. `JSON.parse` errors don't include the input string.
2. Fatal error at `crank-runner.ts:329`: `console.error(\`[crank] FATAL: ${String(err)}\`)` -- This logs the full error string (no truncation). If a downstream error includes the keypair in its message, it would leak. However, the keypair is consumed by `Keypair.fromSecretKey` which throws generic errors (e.g., "bad secret key size"), not including the key bytes.

### Step 5: No Key Zeroization
**Location:** `scripts/crank/crank-provider.ts:46-48`
**Finding:** After parsing, the `secretKey` array (line 46) and the `Uint8Array` (line 48) are never zeroed. The `Keypair` object also holds an internal reference to the key. In the long-running crank process (runs 24/7), the secret key bytes persist in the V8 heap indefinitely, increasing the window for memory dump attacks.

### Step 6: No Address Validation on Loaded Key
**Location:** `scripts/crank/crank-provider.ts:44-57`
**Finding:** The loaded keypair's public key is logged (truncated to 12 chars) but never validated against an expected address. If `WALLET_KEYPAIR` is replaced with a different keypair (via Railway env injection), the crank silently operates with the attacker's key -- or more critically, the attacker holds the original key and can use it independently.

## Blocking Factors
- None that prevent exploitation. Once the key is extracted, the attacker has full signing authority with no defense-in-depth.

## Enabling Factors
- **Plaintext env var storage**: `crank-provider.ts:41` -- WALLET_KEYPAIR is stored as plaintext JSON in Railway, not encrypted at rest by the application
- **No spending limits**: `crank-runner.ts:225-241` -- vault top-up has no cumulative cap; arbitrary SOL transfers possible with the key
- **No address allowlist**: `crank-provider.ts:44-57` -- no validation that the loaded key matches an expected pubkey
- **No multi-sig**: All operations use a single signer
- **No alerting**: No mechanism to detect unauthorized key usage
- **Long-running process**: Crank runs 24/7, so secret key bytes persist in memory indefinitely
- **RPC URL logged unredacted**: `crank-runner.ts:177` -- demonstrates pattern of env var logging without sanitization

### Recommended Fix

**Tier 1 (Immediate -- defense-in-depth within current architecture):**

```typescript
// Before (vulnerable) -- crank-provider.ts:44-57
if (keypairEnv) {
  const secretKey = JSON.parse(keypairEnv);
  wallet = new anchor.Wallet(
    Keypair.fromSecretKey(new Uint8Array(secretKey))
  );
}

// After (improved) -- add expected address validation
const EXPECTED_CRANK_PUBKEY = process.env.EXPECTED_CRANK_PUBKEY;
if (keypairEnv) {
  const secretKeyArr = JSON.parse(keypairEnv);
  const secretKey = new Uint8Array(secretKeyArr);
  const kp = Keypair.fromSecretKey(secretKey);
  // Zero the intermediate array
  secretKeyArr.fill(0);
  secretKey.fill(0);

  // Validate address matches expected
  if (EXPECTED_CRANK_PUBKEY && kp.publicKey.toBase58() !== EXPECTED_CRANK_PUBKEY) {
    throw new Error(
      `Loaded keypair ${kp.publicKey.toBase58().slice(0, 12)}... does not match ` +
      `expected ${EXPECTED_CRANK_PUBKEY.slice(0, 12)}...`
    );
  }

  wallet = new anchor.Wallet(kp);
}
```

**Tier 2 (Spending limits -- crank-runner.ts):**

```typescript
// Add daily spending cap
const MAX_DAILY_SPEND_LAMPORTS = 500_000_000; // 0.5 SOL
let dailySpend = 0;
let lastResetDate = new Date().toDateString();

// Before each operation that spends SOL:
const today = new Date().toDateString();
if (today !== lastResetDate) {
  dailySpend = 0;
  lastResetDate = today;
}
if (dailySpend + VAULT_TOP_UP_LAMPORTS > MAX_DAILY_SPEND_LAMPORTS) {
  console.error("[crank] CIRCUIT BREAKER: Daily spend limit reached. Stopping.");
  process.exit(2);
}
dailySpend += VAULT_TOP_UP_LAMPORTS;
```

**Tier 3 (Mainnet -- migrate away from raw env var):**
- Use a KMS (AWS KMS, GCP KMS) for signing instead of raw keypair in env var
- Or use Squads multi-sig with a crank co-signer
- Implement external alerting (PagerDuty, Discord webhook) on balance changes

**Log redaction fix (crank-runner.ts:177):**
```typescript
// Before
console.log(`  RPC: ${process.env.CLUSTER_URL || "localhost"}`);

// After
const rpcDisplay = (process.env.CLUSTER_URL || "localhost")
  .replace(/api-key=[^&]+/, "api-key=***");
console.log(`  RPC: ${rpcDisplay}`);
```

## Incidental Discoveries

1. **Carnage WSOL secret key loaded unnecessarily for pubkey extraction**: `crank-runner.ts:110-111` reads the full keypair file just to call `.publicKey`. The `CARNAGE_WSOL_PUBKEY` env var path (line 98-100) correctly avoids this, but the file fallback loads secret key bytes into memory unnecessarily. Suggested hypothesis: Investigate whether the carnage WSOL keypair file could be exploited to close/drain the WSOL token account.

2. **No process isolation between env vars**: All Railway env vars for the crank service (WALLET_KEYPAIR, CLUSTER_URL, PDA_MANIFEST, CARNAGE_WSOL_PUBKEY) are in the same scope. Compromising one (e.g., via a log leak of CLUSTER_URL showing the approach to env var management) provides intelligence for targeting others. Suggested hypothesis: Investigate whether CLUSTER_URL manipulation (RPC MITM) could be used as an alternative attack vector to achieve the same fund drainage without needing the private key.
