# Finding: H013 - Crank Vault Top-Up Without Spending Limit

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

The crank runner's vault top-up mechanism at `scripts/crank/crank-runner.ts:225-241` transfers 0.005 SOL from the crank wallet to the CarnageSolVault every cycle where the vault balance reads below ~2M lamports. There is no per-cycle counter, no per-day cumulative cap, no maximum top-up count, and no anomaly detection. The wallet balance warning at line 213-217 is advisory only (console.log) and does not halt operations.

Under normal operation the drain rate is low (~1.44 SOL/day at 288 cycles/day). However, the mechanism becomes a continuous siphon if: (a) the RPC returns stale data showing persistently low vault balance, (b) an on-chain vulnerability drains the vault between cycles, or (c) the vault address in PDA_MANIFEST is manipulated to point to an attacker-controlled account. In all three scenarios, the crank transfers SOL indefinitely until the wallet is empty.

This is distinct from H004 (keypair compromise), which requires stealing the private key. H013 exploits the crank's own logic path without needing keypair theft -- only RPC data manipulation or manifest poisoning (which H003/H004 also flag as enabling factors).

## Hypothesis Investigated

The crank runner's vault top-up mechanism has no upper spending limit per operation or per epoch, enabling unlimited SOL drainage if the crank logic is exploited.

## Investigation Path

### Step 1: Vault top-up trigger condition
**Location:** `scripts/crank/crank-runner.ts:220-224`
**Finding:** Every crank cycle reads the vault balance via RPC. No caching, no rate-limiting on the check itself.
**Evidence:**
```typescript
const vaultPubkey = vrfAccounts.carnageAccounts!.carnageSolVault;
const vaultBalance = await provider.connection.getBalance(vaultPubkey);
await sleep(RPC_DELAY_MS);
```

### Step 2: Transfer with no cumulative guard
**Location:** `scripts/crank/crank-runner.ts:225-241`
**Finding:** If `vaultBalance < MIN_VAULT_BALANCE` (~2M lamports), the crank transfers `VAULT_TOP_UP_LAMPORTS` (5M lamports / 0.005 SOL) unconditionally. No counter tracks how many top-ups have occurred. No daily budget check exists.
**Evidence:**
```typescript
if (vaultBalance < MIN_VAULT_BALANCE) {
  // ... log ...
  const topUpTx = new Transaction().add(
    SystemProgram.transfer({
      fromPubkey: provider.wallet.publicKey,
      toPubkey: vaultPubkey,
      lamports: VAULT_TOP_UP_LAMPORTS,
    })
  );
  const sig = await provider.sendAndConfirm(topUpTx, []);
  // ... log ...
}
```

### Step 3: Balance warning is advisory only
**Location:** `scripts/crank/crank-runner.ts:213-217`
**Finding:** When wallet balance drops below 1 SOL, the crank logs a WARNING but continues operating, including performing vault top-ups. There is no `if (balance < threshold) { shutdownRequested = true; }` guard.
**Evidence:**
```typescript
if (balance < 1 * LAMPORTS_PER_SOL) {
  console.log(
    `[crank] WARNING: Wallet balance low: ${(balance / LAMPORTS_PER_SOL).toFixed(3)} SOL`
  );
}
```

### Step 4: No top-up-after-transfer verification
**Location:** `scripts/crank/crank-runner.ts:237-240`
**Finding:** After the top-up transaction confirms, the code logs success but does NOT re-read the vault balance to verify it actually increased. On the next cycle, if the RPC still returns a low balance (stale read, vault drained between cycles, or MITM), the top-up fires again.

### Step 5: Vault address comes from unvalidated manifest
**Location:** `scripts/crank/crank-runner.ts:221`
**Finding:** `vaultPubkey` is derived from `vrfAccounts.carnageAccounts.carnageSolVault`, which comes from `manifest.pdas.CarnageSolVault` (line 153). The manifest is loaded from the PDA_MANIFEST env var with zero integrity verification (see `crank-provider.ts`). A poisoned manifest would redirect all top-up SOL to an attacker address.

## Invariant Analysis

### Target Invariant
The crank wallet's SOL balance should not decrease beyond a predictable, bounded rate from vault maintenance operations.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `crank-runner.ts:213-217` | Logs warning when wallet < 1 SOL | No -- advisory only, does not halt |
| `crank-runner.ts:72-75` | Constants define top-up amount (0.005 SOL) | Partially -- per-transfer cap exists, but no cumulative cap |

## Blocking Factors
- **Fixed per-transfer amount**: `crank-runner.ts:75` -- each top-up is only 0.005 SOL, limiting drain speed to ~1.44 SOL/day under normal cycle timing
- **On-chain transaction fees**: Each failed/successful top-up costs ~5000 lamports in fees, which is negligible but does add up
- **Solana tx will fail at 0 balance**: When the wallet has insufficient SOL for the transfer + fees, the `sendAndConfirm` throws, caught by the outer catch at line 302, and the crank retries after 30s (burning no more SOL but also never stopping)

## Enabling Factors
- **No cumulative cap**: `crank-runner.ts:225-241` -- no daily/weekly/total spending tracker
- **Advisory-only balance warning**: `crank-runner.ts:213-217` -- logs but does not halt
- **No post-transfer verification**: `crank-runner.ts:237` -- does not confirm vault balance actually rose
- **Unvalidated manifest**: `crank-provider.ts` -- vault address from PDA_MANIFEST with no integrity check
- **Infinite retry loop**: `crank-runner.ts:302-315` -- errors from failed top-ups are caught and retried, never escalated

## Severity Assessment
- **Severity:** MEDIUM
- **Likelihood:** MEDIUM (requires either RPC manipulation, on-chain vault drain, or manifest poisoning -- all flagged by other findings)
- **Impact:** Crank wallet SOL drain (up to total balance). At 0.005 SOL per ~5-minute cycle, a 10 SOL wallet drains in ~3.5 days. The crank becoming insolvent halts all epoch transitions, stalling the protocol.

### Attack Scenario

**Scenario A: RPC data manipulation (most likely)**
```
Step 1: Attacker compromises or MITMs the RPC endpoint (CLUSTER_URL)
  Target: provider.connection.getBalance(vaultPubkey)
  Payload: Always return 0 lamports for vault balance query
  Prerequisite: DNS hijack, BGP hijack, or compromised RPC provider
  Result: Crank sees vault as empty every cycle

Step 2: Crank transfers 0.005 SOL every ~5 minutes
  Target: SystemProgram.transfer to vaultPubkey
  Result: Legitimate vault receives unnecessary SOL (waste), OR if vault
          address is also spoofed via manifest, SOL goes to attacker

Step 3: Wallet drains over hours/days
  Blocking: Balance warning logs to stdout only
  Result: Wallet reaches 0 SOL, crank becomes insolvent, protocol halts

Final Result: Crank wallet drained of all SOL. Protocol epoch advancement stops.
```

**Scenario B: On-chain vault drain (chained attack)**
```
Step 1: Exploit an on-chain vulnerability to drain CarnageSolVault
  Prerequisite: On-chain bug in vault authority (out of off-chain scope)
  Result: Vault balance is genuinely 0 after each drain

Step 2: Crank auto-replenishes vault every cycle
  Result: Attacker drains vault again, crank tops up again -- infinite loop

Step 3: Crank wallet is siphoned through the vault
  Result: All crank SOL transferred to vault and then extracted by attacker

Final Result: Crank wallet drained via the vault as a passthrough.
```

### Recommended Fix

```typescript
// Before (vulnerable) -- crank-runner.ts:225-241
if (vaultBalance < MIN_VAULT_BALANCE) {
  const topUpTx = new Transaction().add(
    SystemProgram.transfer({
      fromPubkey: provider.wallet.publicKey,
      toPubkey: vaultPubkey,
      lamports: VAULT_TOP_UP_LAMPORTS,
    })
  );
  const sig = await provider.sendAndConfirm(topUpTx, []);
}

// After (fixed)
const MAX_TOPUPS_PER_DAY = 50;       // ~0.25 SOL/day max
const MIN_WALLET_BALANCE = 0.5 * LAMPORTS_PER_SOL;  // Hard floor

// Track cumulative top-ups (reset daily)
let dailyTopUpCount = 0;
let lastResetDate = new Date().toDateString();

// Inside the loop, before the top-up:
const today = new Date().toDateString();
if (today !== lastResetDate) {
  dailyTopUpCount = 0;
  lastResetDate = today;
}

if (vaultBalance < MIN_VAULT_BALANCE) {
  // Circuit breaker: daily cap
  if (dailyTopUpCount >= MAX_TOPUPS_PER_DAY) {
    console.error(
      `[crank] CIRCUIT BREAKER: Daily top-up limit (${MAX_TOPUPS_PER_DAY}) reached. ` +
      `Possible vault drain or stale RPC. Halting.`
    );
    process.exit(2);
  }

  // Circuit breaker: wallet floor
  if (balance < MIN_WALLET_BALANCE) {
    console.error(
      `[crank] CIRCUIT BREAKER: Wallet below minimum (${MIN_WALLET_BALANCE / LAMPORTS_PER_SOL} SOL). Halting.`
    );
    process.exit(2);
  }

  const topUpTx = new Transaction().add(
    SystemProgram.transfer({
      fromPubkey: provider.wallet.publicKey,
      toPubkey: vaultPubkey,
      lamports: VAULT_TOP_UP_LAMPORTS,
    })
  );
  const sig = await provider.sendAndConfirm(topUpTx, []);
  dailyTopUpCount++;

  // Post-transfer verification
  const newVaultBalance = await provider.connection.getBalance(vaultPubkey);
  if (newVaultBalance <= vaultBalance) {
    console.error(
      `[crank] WARNING: Vault balance did not increase after top-up ` +
      `(before: ${vaultBalance}, after: ${newVaultBalance}). Possible drain.`
    );
  }
}
```

## Incidental Discoveries
- **No `process.on("unhandledRejection")` handler**: `crank-runner.ts` has SIGINT/SIGTERM handlers (lines 81-89) but no `unhandledRejection` or `uncaughtException` handlers. An unhandled promise rejection from dynamic imports (`vrf-flow.ts:299,312`) or the Switchboard SDK could crash the process silently. Suggested hypothesis: "Crank silent crash via unhandled promise rejection leading to protocol stall."
- **RPC URL logged without API key masking**: `crank-runner.ts:177` logs `process.env.CLUSTER_URL` which may contain `?api-key=...`. The VRF validation script at `devnet-vrf-validation.ts:94` masks this, but the crank runner does not. Suggested hypothesis: "API key leakage in Railway logs via crank RPC URL logging."
