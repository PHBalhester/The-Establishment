# Finding: H065 - WSOL ATA Race Condition

## Status: NOT VULNERABLE
## Confidence Score: 8

## Executive Summary

The hypothesis suggests that WSOL ATA (Associated Token Account) creation and closure could race with concurrent swaps. Investigation reveals this is not exploitable because: (1) WSOL ATA creation, the swap instruction, and WSOL unwrap are all bundled into a SINGLE atomic transaction -- they either all succeed or all fail together; (2) a user can only submit one transaction at a time from their wallet (wallet adapter enforces sequential signing); (3) even if two transactions were submitted in rapid succession, Solana's transaction ordering ensures each transaction is internally atomic; and (4) the on-chain program's account constraints would reject a transaction referencing a non-existent or incorrect WSOL ATA.

## Hypothesis Investigated

WSOL ATA creation/close could race with concurrent swaps.

## Investigation Path

### Step 1: Examine WSOL ATA lifecycle in buy transaction
**Location:** `app/lib/swap/swap-builders.ts:216-224`
**Finding:** WSOL wrap instructions (create ATA if needed + SOL transfer + syncNative) are added to the SAME transaction as the swap instruction. This is atomic.
**Evidence:**
```typescript
// swap-builders.ts:216-224
// 2. WSOL wrap instructions (create ATA if needed + SOL transfer + syncNative)
const wsolInstructions = await buildWsolWrapInstructions(
  connection,
  userPublicKey,
  amountInLamports,
);
for (const ix of wsolInstructions) {
  tx.add(ix);
}
```

### Step 2: Examine WSOL ATA lifecycle in sell transaction
**Location:** `app/lib/swap/swap-builders.ts:337-401`
**Finding:** Sell transaction: (1) creates WSOL ATA if needed, (2) executes swap, (3) unwraps WSOL ATA -- all in one transaction.
**Evidence:**
```typescript
// swap-builders.ts:337-350 (create)
const userWsolAta = await getWsolAta(userPublicKey);
const wsolAtaInfo = await connection.getAccountInfo(userWsolAta);
if (!wsolAtaInfo) {
  tx.add(createAssociatedTokenAccountInstruction(...));
}
// ... swap instruction ...
// swap-builders.ts:399-401 (unwrap)
const unwrapIx = await buildWsolUnwrapInstruction(userPublicKey);
tx.add(unwrapIx);
```

### Step 3: Check for concurrent swap possibility
**Location:** `app/hooks/useSwap.ts:673-791`
**Finding:** The `executeSwap` function manages state via `setStatus("building")` etc. While there's no explicit lock preventing double-clicks, the wallet adapter's `signTransaction` call blocks the UI until the user approves or rejects. A second swap cannot be initiated while the wallet prompt is open.

### Step 4: Analyze Solana transaction atomicity
**Location:** Solana runtime (inherent property)
**Finding:** All instructions within a single Solana transaction execute atomically. If the WSOL ATA creation succeeds but the swap fails, the entire transaction is rolled back. There is no intermediate state where the ATA exists without the swap completing (or vice versa).

## Attack Path Analysis

| Step | Required Action | Possible? | Evidence |

| 1 | Race WSOL ATA creation with swap | No | Same transaction = atomic |
| 2 | Race WSOL ATA close with concurrent swap | No | Close is in same TX as swap |

## Invariant Analysis

### Target Invariant
WSOL ATA state must be consistent throughout the swap lifecycle.

### Enforcement Points
| Location | What It Enforces | Complete? |
|----------|-----------------|-----------|
| `swap-builders.ts:216-224` | ATA create + swap in one TX | Yes |
| `swap-builders.ts:399-401` | Swap + ATA close in one TX | Yes |

## Blocking Factors
- **Transaction atomicity**: `swap-builders.ts` -- all WSOL lifecycle + swap in one TX
- **Wallet signing gate**: User must approve each transaction sequentially
- **Solana runtime**: Transactions are atomic; no partial execution

## Enabling Factors
- **RPC check for ATA existence**: `swap-builders.ts:338-339` -- `getAccountInfo` check is a TOCTOU in theory, but the ATA creation is idempotent (createAssociated... is a no-op if ATA exists)
