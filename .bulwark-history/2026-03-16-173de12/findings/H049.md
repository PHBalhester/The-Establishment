# Finding: H049 - Webhook No Replay Protection

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The Helius webhook handler at `app/app/api/webhooks/helius/route.ts` has no timestamp validation on incoming payloads. An attacker who captures a legitimate webhook payload (via network inspection, log access, or the fail-open auth bypass) can replay it indefinitely. While the swap_events table uses `onConflictDoNothing` for idempotent deduplication on the `txSignature` primary key, the replay still triggers:

1. Full JSON parsing and event extraction for every replayed transaction
2. 6x candle upsert operations per swap event (one per time resolution)
3. N SSE broadcasts to all connected clients per candle upsert

This creates an amplification attack: replaying a batch of 100 transactions triggers up to 600 candle upserts and 600*N SSE broadcasts, consuming database connections, CPU, and bandwidth -- even though no new rows are inserted in swap_events.

The idempotency on swap_events prevents data duplication but does NOT prevent the candle upsert or SSE broadcast from executing. The candle aggregator uses upsert logic that re-computes OHLCV even for duplicate data, and SSE broadcasts fire unconditionally.

## Hypothesis Investigated

No timestamp validation allows replay of captured webhook payloads.

## Investigation Path

### Step 1: Authentication check
**Location:** `app/app/api/webhooks/helius/route.ts:135-141`
**Finding:** Auth is optional (fail-open when `HELIUS_WEBHOOK_SECRET` is unset).
**Evidence:**
```typescript
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (webhookSecret) {
  const authHeader = req.headers.get("authorization");
  if (authHeader !== webhookSecret) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
}
```

### Step 2: Timestamp validation
**Location:** `app/app/api/webhooks/helius/route.ts:131-290`
**Finding:** No timestamp validation anywhere in the handler. The `blockTime` field from the transaction is used for database storage but never checked for staleness.
**Evidence:** The handler reads `tx.blockTime` at line 166 and uses it for DB insertion, but never checks `Date.now() - blockTime < MAX_AGE`.

### Step 3: Candle upsert on replay
**Location:** `app/app/api/webhooks/helius/route.ts:209-241`
**Finding:** Candle upsert and SSE broadcast execute BEFORE the `onConflictDoNothing` check on swap_events. Even if the swap event already exists, the candle aggregator and SSE broadcast still fire.
**Evidence:**
```typescript
// Candle upsert fires regardless of whether swap_events insert succeeded
if (swap.type === "TaxedSwap" || swap.type === "UntaxedSwap") {
  try {
    await upsertCandlesForSwap(swap, poolAddress, timestamp);
    // SSE broadcast also fires unconditionally
    for (const resolution of RESOLUTIONS) {
      sseManager.broadcast("candle-update", { ... });
    }
  }
}
```

### Step 4: Swap event idempotency
**Location:** `app/app/api/webhooks/helius/route.ts:337`
**Finding:** `onConflictDoNothing` prevents duplicate rows in `swap_events` table.
**Evidence:**
```typescript
.onConflictDoNothing();
```

### Severity Assessment

### Attack Scenario
```
Step 1: Capture a legitimate webhook payload
  Target: POST /api/webhooks/helius
  Payload: Copy the JSON body from a previous delivery (network logs, Helius dashboard)
  Prerequisite: Access to webhook URL (hardcoded in source)
  Result: Valid payload in hand

Step 2: Replay at high rate
  Target: POST /api/webhooks/helius (no auth if secret unset)
  Payload: Same captured payload, 100x per second
  Prerequisite: No rate limiting
  Result: 100 * N_swaps * 6_resolutions = candle upserts + SSE broadcasts per second

Step 3: Database and SSE saturation
  Target: PostgreSQL + SSE connections
  Prerequisite: Step 2 sustained for >30 seconds
  Result: DB connection pool exhausted, SSE clients receive flood of duplicate updates, chart UI flickers

Final Result: Service degradation -- charts become unusable, DB connections saturated, legitimate webhooks may be rejected with 500 errors
```

### Recommended Fix
```typescript
// Before (vulnerable)
const transactions: HeliusTransaction[] = await req.json();

// After (fixed) -- add timestamp validation
const MAX_WEBHOOK_AGE_SEC = 300; // 5 minutes
const transactions: HeliusTransaction[] = await req.json();

// Reject stale payloads (replay protection)
for (const tx of transactions) {
  if (tx.blockTime) {
    const ageSec = Math.floor(Date.now() / 1000) - tx.blockTime;
    if (ageSec > MAX_WEBHOOK_AGE_SEC || ageSec < -60) {
      // Skip stale or future-dated transactions
      continue;
    }
  }
}

// Also add: request-level dedup (hash of payload -> Redis/memory set with TTL)
```
