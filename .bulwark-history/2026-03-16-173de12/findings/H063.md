# Finding: H063 - Demo Mode Hook Violation

## Status: CONFIRMED
## Confidence Score: 10

## Executive Summary

The `useCurveState` hook at `app/hooks/useCurveState.ts:209-218` contains a conditional early return BEFORE the `useState` and `useRef` hook calls when `DEMO_MODE` is `true`. This violates React's Rules of Hooks, which require that hooks are called in the same order on every render. When `NEXT_PUBLIC_DEMO_MODE=true`, the function returns on line 217 before reaching `useState` calls on lines 220-223 and `useRef`/`useCallback`/`useEffect` calls below. This is a confirmed React bug that will cause a runtime error if the component ever re-renders with `DEMO_MODE` changing (unlikely since it's a build-time constant), but more importantly, React's static analysis (eslint-plugin-react-hooks) would flag this as a violation and React's internal hook tracking will produce a corrupted fiber state.

In practice, because `DEMO_MODE` is a build-time constant (`process.env.NEXT_PUBLIC_DEMO_MODE === "true"` is evaluated at build/bundle time by Next.js), the condition is effectively dead code elimination -- the hooks below are ALWAYS present OR NEVER present in a given build. This means the rule-of-hooks violation is technically present in source but does not cause runtime errors in practice.

## Hypothesis Investigated

useCurveState returns before useState when DEMO_MODE true, violating rules-of-hooks.

## Investigation Path

### Step 1: Verify the hook violation
**Location:** `app/hooks/useCurveState.ts:209-223`
**Finding:** Conditional return before hooks. DEMO_MODE check at line 211 returns before useState at lines 220-223.
**Evidence:**
```typescript
export function useCurveState(): UseCurveStateResult {
  // Demo mode: return mock data immediately, no RPC calls
  if (DEMO_MODE) {                          // line 211
    return {                                 // line 212 -- EARLY RETURN
      crime: getDemoCurveState("crime"),
      fraud: getDemoCurveState("fraud"),
      loading: false,
      error: null,
    };
  }                                          // line 218

  const [crime, setCrime] = useState<CurveStateData | null>(null);  // line 220
  const [fraud, setFraud] = useState<CurveStateData | null>(null);  // line 221
  const [loading, setLoading] = useState(true);                      // line 222
  const [error, setError] = useState<string | null>(null);           // line 223
```

### Step 2: Check if DEMO_MODE is truly static
**Location:** `app/hooks/useCurveState.ts:168`
**Finding:** `DEMO_MODE` is a module-level constant derived from `process.env.NEXT_PUBLIC_DEMO_MODE`. In Next.js, `NEXT_PUBLIC_*` env vars are replaced at build time by the bundler.
**Evidence:**
```typescript
const DEMO_MODE = process.env.NEXT_PUBLIC_DEMO_MODE === "true";
```

### Step 3: Assess runtime impact
**Location:** React runtime hook tracking
**Finding:** Because `DEMO_MODE` is a build-time constant, in a DEMO_MODE=true build, the function ALWAYS returns early and the hooks below are dead code. In a DEMO_MODE=false build, the condition is always false and hooks always execute. The rule-of-hooks violation exists in source code analysis but does NOT manifest at runtime because the condition never changes between renders.

## Blocking Factors
- **Build-time constant**: `DEMO_MODE` cannot change between renders
- **Dead code elimination**: Next.js/webpack replaces `process.env.NEXT_PUBLIC_DEMO_MODE` at build time

## Enabling Factors
- **Source-level violation**: `useCurveState.ts:211-218` -- early return before useState calls
- **ESLint suppression risk**: If eslint-plugin-react-hooks is not configured, the violation goes undetected

## Severity Assessment
- **Severity:** LOW
- **Likelihood:** LOW (build-time constant prevents runtime issue)
- **Impact:** Code quality issue; potential for future bugs if pattern is copied with dynamic conditions

### Attack Scenario
This is not an externally exploitable vulnerability. It is a code quality defect that could become a runtime bug if the DEMO_MODE check were ever changed to a dynamic condition (e.g., a React state variable or prop).

### Recommended Fix
```typescript
// Before (violates rules-of-hooks in source)
export function useCurveState(): UseCurveStateResult {
  if (DEMO_MODE) {
    return { crime: getDemoCurveState("crime"), fraud: getDemoCurveState("fraud"), loading: false, error: null };
  }
  const [crime, setCrime] = useState<CurveStateData | null>(null);
  // ...

// After (hooks always called, demo mode handled inside)
export function useCurveState(): UseCurveStateResult {
  const [crime, setCrime] = useState<CurveStateData | null>(
    DEMO_MODE ? getDemoCurveState("crime") : null
  );
  const [fraud, setFraud] = useState<CurveStateData | null>(
    DEMO_MODE ? getDemoCurveState("fraud") : null
  );
  const [loading, setLoading] = useState(!DEMO_MODE);
  const [error, setError] = useState<string | null>(null);

  // Skip RPC/WS setup in demo mode
  useEffect(() => {
    if (DEMO_MODE) return;
    // ... existing RPC + WebSocket logic
  }, []);

  return { crime, fraud, loading, error };
}
```
