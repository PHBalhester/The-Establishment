# Finding: H001 - Webhook Auth Bypass via Missing Env Var

## Status: CONFIRMED
## Confidence Score: 9

## Executive Summary

The Helius webhook endpoint at `/api/webhooks/helius` uses a fail-open authentication pattern: if the `HELIUS_WEBHOOK_SECRET` environment variable is not set in Railway, authentication is silently skipped and any HTTP client can POST fabricated transaction data. The webhook URL is hardcoded in source code (`scripts/webhook-manage.ts:43`), making it trivially discoverable. An attacker who POSTs crafted JSON with valid Anchor-encoded log messages can inject fake swap events into PostgreSQL, permanently corrupt OHLCV candle data via the one-directional `GREATEST`/`LEAST` aggregation, and broadcast false price data to all connected SSE chart clients in real-time.

Even if `HELIUS_WEBHOOK_SECRET` IS set, the string comparison uses `!==` (non-constant-time), enabling theoretical timing-based secret extraction over many requests. There is also no validation that transaction signatures actually exist on-chain, no body size limit, and no rate limiting.

## Hypothesis Investigated

An attacker can inject fabricated swap events into the database by POSTing to the unauthenticated webhook endpoint, poisoning price charts and SSE broadcasts.

## Investigation Path

### Step 1: Webhook Authentication Logic
**Location:** `app/app/api/webhooks/helius/route.ts:131-141`
**Finding:** Authentication is entirely opt-in. The guard is `if (webhookSecret) { ... }` -- when the env var is unset, the entire auth block is skipped. No fail-closed behavior, no startup warning, no `NODE_ENV` gate.
**Evidence:**
```typescript
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (webhookSecret) {
  const authHeader = req.headers.get("authorization");
  if (authHeader !== webhookSecret) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
}
```

### Step 2: Webhook URL Discoverability
**Location:** `scripts/webhook-manage.ts:43`
**Finding:** The production webhook URL is hardcoded as a string literal in source code.
**Evidence:**
```typescript
const DEFAULT_WEBHOOK_URL = "https://dr-fraudsworth-production.up.railway.app/api/webhooks/helius";
```

### Step 3: Event Parsing Requirements
**Location:** `app/lib/event-parser.ts:221-298`
**Finding:** The Anchor `EventParser` with `BorshCoder` performs Borsh deserialization against the IDL schema. An attacker must craft valid Anchor event log messages (base64-encoded Borsh data with correct discriminator bytes). However, the IDL is publicly available in `app/idl/tax_program.json` and `app/idl/epoch_program.json` (shipped in the client bundle), so an attacker can construct valid events offline.

### Step 4: Idempotency Guard Analysis
**Location:** `app/app/api/webhooks/helius/route.ts:337` (swap_events), `app/db/schema.ts:33`
**Finding:** `onConflictDoNothing` uses `txSignature` as the primary key. This ONLY prevents exact duplicate signatures. An attacker generating novel fake signatures (any unique 88-character base58 string) will pass this guard every time. There is no validation that the signature corresponds to a real on-chain transaction.

### Step 5: Candle Corruption Permanence
**Location:** `app/db/candle-aggregator.ts:118-126`
**Finding:** The `GREATEST`/`LEAST` upsert pattern is one-directional -- once a high price is set, it can only go higher; once a low is set, it can only go lower. A single injected extreme price permanently corrupts that candle's range. Recovery requires manual SQL `UPDATE` statements.
**Evidence:**
```typescript
set: {
  high: sql`GREATEST(${candles.high}, ${update.price})`,
  low: sql`LEAST(${candles.low}, ${update.price})`,
  close: sql`${update.price}`,
  volume: sql`${candles.volume} + ${update.volume}`,
  tradeCount: sql`${candles.tradeCount} + 1`,
},
```

### Step 6: SSE Broadcast to All Clients
**Location:** `app/app/api/webhooks/helius/route.ts:222-234`, `app/lib/sse-manager.ts:58-68`
**Finding:** After storing the fake swap event, the webhook handler immediately broadcasts `candle-update` events to ALL connected SSE subscribers across all 6 time resolutions. The SSE manager has no authentication, no filtering, and no data validation. One fake webhook POST triggers 6 SSE broadcasts per injected swap event to every connected browser.

### Step 7: No On-Chain Signature Verification
**Location:** `app/app/api/webhooks/helius/route.ts:160-268`
**Finding:** The webhook handler never calls `getTransaction()` or any RPC method to verify that the submitted transaction signature exists on-chain. It fully trusts the incoming payload once it passes the (optional) auth check.

## Blocking Factors
- **Anchor BorshCoder parsing** (`event-parser.ts:221-229`): Attacker must craft valid Borsh-encoded event data. This raises the bar slightly but is not a real barrier since the IDL is public.

## Enabling Factors
- **Fail-open auth**: `route.ts:135-141` -- Entire auth block skipped when `HELIUS_WEBHOOK_SECRET` unset
- **Hardcoded URL**: `webhook-manage.ts:43` -- Webhook endpoint trivially discoverable
- **Public IDL**: `app/idl/tax_program.json`, `app/idl/epoch_program.json` -- Shipped in client bundle
- **No TX verification**: No RPC call to confirm signature exists on-chain
- **No rate limiting**: No middleware.ts, no per-route throttling
- **No body size limit**: `route.ts:146` -- `req.json()` reads unlimited body
- **No data range validation**: No checks that price > 0, amounts are within realistic bounds, etc.
- **One-directional candle corruption**: `candle-aggregator.ts:121-122` -- `GREATEST`/`LEAST` makes corruption permanent

## Severity Assessment
- **Severity:** HIGH
- **Likelihood:** HIGH (if env var unset) / MEDIUM (if set, timing attack needed)
- **Impact:** Data integrity corruption -- manipulated price charts visible to all users, persistent database pollution, potential to socially engineer users into bad trades

### Severity Rationale
Per the calibration guide: "State corruption: Database/cache poisoned" = Integrity Impact. Combined with HIGH likelihood (fail-open is a probable misconfiguration, especially given no startup validation), this maps to HIGH severity. Not CRITICAL because no direct fund loss occurs -- the attack manipulates display data, not on-chain state. However, manipulated charts can indirectly cause fund loss through user trading decisions.

## Recommended Fix

```typescript
// Before (vulnerable) -- app/app/api/webhooks/helius/route.ts:131-141
const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (webhookSecret) {
  const authHeader = req.headers.get("authorization");
  if (authHeader !== webhookSecret) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
}

// After (fixed)
import { timingSafeEqual } from "crypto";

const webhookSecret = process.env.HELIUS_WEBHOOK_SECRET;
if (!webhookSecret) {
  // FAIL CLOSED: Reject all requests if secret is not configured
  console.error("[webhook] HELIUS_WEBHOOK_SECRET not set -- rejecting request");
  return NextResponse.json({ error: "Webhook not configured" }, { status: 503 });
}

const authHeader = req.headers.get("authorization") ?? "";
// Timing-safe comparison to prevent secret leakage
const secretBuf = Buffer.from(webhookSecret, "utf-8");
const headerBuf = Buffer.from(authHeader, "utf-8");
if (secretBuf.length !== headerBuf.length || !timingSafeEqual(secretBuf, headerBuf)) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

**Additional hardening (defense-in-depth):**
1. Add a startup health check that warns/fails if `HELIUS_WEBHOOK_SECRET` is unset in production
2. Validate that `blockTime` is within a reasonable window (e.g., +/- 5 minutes of server time) for replay protection
3. Consider async on-chain signature verification for high-value events (getTransaction RPC call)
4. Add basic data range validation (price > 0, amounts within u64 range, etc.)
5. Add rate limiting on the webhook endpoint

## Related Findings
- Timing-unsafe comparison (`route.ts:138`) is a secondary vector even when the secret IS set
- No body size limit (`route.ts:146`) enables OOM DoS independent of auth bypass
- No SSE subscriber cap (`sse-manager.ts:30`) amplifies the broadcast impact
- Hardcoded Helius API key (`webhook-manage.ts:28`) could allow an attacker to delete/redirect the legitimate webhook via Helius API, then register their own

## Incidental Discoveries
- **No data range validation on webhook-ingested values**: The webhook handler stores whatever the event parser returns without checking if price > 0, amounts are within realistic bounds, or epoch numbers are sequential. Even with auth fixed, a compromised Helius delivery could inject anomalous data. Location: `route.ts:180-243`. Suggested hypothesis: "Can a compromised Helius account or MITM on the Helius delivery pipeline inject subtly wrong data that passes auth but corrupts charts?"
- **Candle volume double-counting risk**: If the same swap event is processed with a different signature (e.g., via a fork/re-org where Helius delivers the same swap under two different signatures), the candle `tradeCount` and `volume` would be inflated. This is an edge case inherent to the `GREATEST`/`LEAST` aggregation pattern. Location: `candle-aggregator.ts:124-125`.
