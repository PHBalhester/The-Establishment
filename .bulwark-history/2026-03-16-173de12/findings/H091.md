# Finding: H091 - No Distributed Lock for Crank

## Status: POTENTIAL
## Confidence Score: 7

## Rationale
The crank runner at `scripts/crank/crank-runner.ts` has no distributed locking mechanism. If two instances run simultaneously (e.g., Railway scaling, manual local run alongside Railway, or accidental double-deploy), both would attempt epoch transitions, VRF commits/reveals, and Carnage execution concurrently. While most on-chain instructions would fail for the second caller (epoch already advanced, VRF already committed), the race creates wasted transactions, wasted SOL on fees, and potential for confusing error logs. The vault top-up at lines 225-241 could double-top-up if both instances detect low balance simultaneously. Currently Railway runs a single crank instance, but there is no enforcement preventing multiple.

## Code Evidence
```typescript
// scripts/crank/crank-runner.ts:197 - main loop, no lock acquisition
while (!shutdownRequested) {
  cycleCount++;
  // ... directly proceeds to read state and advance epoch
```
No Redis lock, no filesystem lock, no on-chain singleton check before proceeding.

## Severity Estimate
LOW -- Currently mitigated by Railway's single-instance deployment. The on-chain programs have their own state gates (epoch boundary checks, VRF pending flags) that prevent double-execution. The main risk is wasted SOL from duplicate failed transactions if a second instance runs.
