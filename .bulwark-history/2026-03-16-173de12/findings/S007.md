# Finding: S007 - Supply Chain -> Crank Key Exfiltration

## Status: CONFIRMED
## Confidence Score: 8

## Executive Summary

This chain attack combines H003 (npm supply chain attack via gitignored lockfile) with H004 (crank wallet keypair in Railway env var) into a direct fund-loss scenario. The `package-lock.json` is explicitly gitignored (`.gitignore:9`), meaning every Railway deploy runs `npm install` against the public npm registry, resolving caret-ranged dependencies fresh. An attacker who publishes a compromised patch version of `@coral-xyz/anchor`, `@solana/web3.js`, or `@switchboard-xyz/on-demand` (all three use `^` ranges) will have their malicious code installed on the next crank deploy. The crank process loads `WALLET_KEYPAIR` from `process.env` at `crank-provider.ts:44-48` -- a full 64-byte Solana secret key as a JSON array string. A single `fetch()` call from compromised code exfiltrates this key, giving the attacker complete signing authority over the crank wallet. The attacker can then drain all SOL, sign malicious epoch transitions, and disrupt Carnage execution.

The chain is more severe than either individual finding: H003 provides persistent code execution in the crank process, and H004 provides the high-value secret that H003's code can steal. Neither finding alone results in fund loss -- H003 needs a target secret, and H004 needs a vector to extract it. Together, they produce a CRITICAL fund-loss chain.

## Hypothesis Investigated

npm supply chain compromise gets malicious code into the crank runner, which exfiltrates `WALLET_KEYPAIR` from `process.env`, enabling the attacker to drain the crank wallet and sign arbitrary protocol transactions.

## Investigation Path

### Step 1: Confirm Lockfile is Gitignored
**Location:** `.gitignore:9`
**Finding:** `package-lock.json` is explicitly excluded from version control. Every Railway deploy resolves dependencies fresh.
**Evidence:**
```
# .gitignore line 9
package-lock.json
```

### Step 2: Confirm Crank Build Uses npm install Without Lockfile
**Location:** `railway-crank.toml:3`
**Finding:** The crank Railway service explicitly runs `npm install` as its build command. Without a committed lockfile, this resolves all `^`-ranged versions from the public npm registry.
**Evidence:**
```toml
[build]
buildCommand = "npm install"
```

### Step 3: Confirm Caret-Ranged Dependencies in Crank Import Tree
**Location:** `package.json:20-23`
**Finding:** Three runtime dependencies use caret ranges that the crank transitively imports:
- `@coral-xyz/anchor: ^0.32.1` -- imported by `crank-provider.ts:19`
- `@solana/web3.js: ^1.95.5` -- imported by `crank-runner.ts:32`
- `@switchboard-xyz/on-demand: ^3.7.3` -- imported transitively via `vrf-flow.ts`

A compromised `@coral-xyz/anchor@0.32.2` would be auto-resolved within the `^0.32.1` range.

### Step 4: Trace WALLET_KEYPAIR Access in Crank Process
**Location:** `scripts/crank/crank-provider.ts:41-57`
**Finding:** The full secret key is loaded from `process.env.WALLET_KEYPAIR`, parsed via `JSON.parse()`, and constructed into a `Keypair` object. The parsed `Uint8Array` is never zeroed. Any code running in the same Node.js process has access to `process.env` and the V8 heap.
**Evidence:**
```typescript
// crank-provider.ts:41-48
const keypairEnv = process.env.WALLET_KEYPAIR;
if (keypairEnv) {
  const secretKey = JSON.parse(keypairEnv);
  wallet = new anchor.Wallet(
    Keypair.fromSecretKey(new Uint8Array(secretKey))
  );
}
```

### Step 5: Confirm No Egress Filtering
**Location:** Railway deployment configuration
**Finding:** Railway containers have unrestricted outbound network access. No egress filtering, no firewall rules documented. A compromised dependency can `fetch()` to any external endpoint.

### Step 6: Confirm No Compensating Controls
**Location:** Project-wide search
**Finding:** No `--ignore-scripts` flag. No `.npmrc` with integrity enforcement. No CI/CD pipeline with `npm audit`. No Dependabot or Snyk. No runtime monitoring for unexpected outbound connections. 11 packages with install scripts run unrestricted during `npm install`.

### Step 7: Confirm No Spending Limits on Crank Wallet
**Location:** `scripts/crank/crank-runner.ts:225-241`
**Finding:** The vault top-up at line 230 uses `SystemProgram.transfer` with no cumulative cap. An attacker with the private key can construct arbitrary `SystemProgram.transfer` instructions to any destination with no on-chain guard.

## Enabling Factors
- **Gitignored lockfile**: `.gitignore:9` -- fresh dependency resolution on every deploy
- **Caret-ranged deps**: `package.json:20-23` -- accepts minor/patch version bumps
- **npm install without flags**: `railway-crank.toml:3` -- no `--ignore-scripts`, no integrity check
- **Plaintext secret key in process.env**: `crank-provider.ts:41-48` -- accessible to any code in the process
- **No key zeroization**: `crank-provider.ts:46-48` -- parsed key bytes persist in V8 heap
- **No egress filtering**: Railway allows arbitrary outbound connections
- **No spending limits**: `crank-runner.ts:225-241` -- unlimited SOL transfers from wallet
- **No runtime monitoring**: No detection of unexpected outbound connections or key usage from new IPs

## Incidental Discoveries
- **All Railway env vars exfiltrated simultaneously**: A supply chain compromise exfiltrates the entire `process.env`, not just `WALLET_KEYPAIR`. This includes `DATABASE_URL` (full Postgres access), `CLUSTER_URL` (Helius API key embedded), `PDA_MANIFEST` (protocol topology), and `CARNAGE_WSOL_PUBKEY`. The blast radius extends beyond the crank wallet. Location: `crank-provider.ts:41`, `crank-runner.ts:177`. Suggested hypothesis: Investigate whether `DATABASE_URL` exfiltration enables a separate attack path (direct database manipulation of candle data, bypassing webhook auth entirely).
- **postinstall scripts are a pre-runtime vector**: 11 packages have install scripts that execute during `npm install`, before the crank process even starts. A compromised postinstall script runs with full filesystem and network access during the build phase, before any application-level monitoring could detect it. This is a separate attack surface from module-level code injection. Location: `railway-crank.toml:3` (`npm install` runs all scripts).
